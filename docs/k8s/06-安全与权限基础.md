## 1. K8s 安全体系整体介绍

### 1.1 K8s 安全核心原则与分层防御模型

#### 概念定义

K8s 安全并不是单一的密码或权限配置，而是一套**分层纵深防御体系**。类比小区安保体系：门禁认证、人员权限、楼栋隔离、房间锁、监控审计，多层防护，单一防线破损不会导致整体安全崩溃。

K8s 官方核心安全三原则：

- **最小权限原则**：所有用户、服务、组件仅授予完成工作的最低权限，无多余权限；

- **纵深防御原则**：多维度、多层级防护，避免单点防护失效；

- **默认拒绝原则**：所有访问、通信、权限默认禁止，按需主动放行。

标准六层分层防御模型（由外到内）：

1. 传输层安全：所有组件通信HTTPS加密；

2. API认证层：校验访问者身份是否合法；

3. API授权层：校验合法身份是否拥有操作权限；

4. 准入控制层：校验资源配置是否合规、安全；

5. 网络隔离层：限制Pod、节点之间网络通信；

6. 运行时安全层：限制容器权限、文件系统、系统调用。

#### 核心原理

K8s 所有外部、内部操作全部经过 APIServer 统一入口，所有请求必须逐层经过**认证→授权→准入控制**校验，再写入ETCD。分层模型的设计目的是：避免单一安全漏洞导致集群沦陷，每一层都是独立的防护屏障，层层拦截非法操作与攻击行为。

#### 场景与价值

生产集群裸奔部署时，一旦任意账号泄露、容器被入侵、内网被渗透，攻击者可直接接管整个集群。分层防御模型可以做到：攻击穿透一层防护后，无法突破剩余防线，最大程度降低集群被入侵、数据泄露、服务篡改的风险。

#### 实操示例

无独立命令，该模型为集群底层安全架构，所有安全配置均基于此模型实现。可通过以下命令查看集群安全校验链路日志：

```bash
# 查看apiserver审计日志（部分集群默认开启）
journalctl -u kube-apiserver -f
# 查看集群准入插件列表
kubectl api-resources | grep admission

```

#### 验证与排错

**验证方式**：使用无权限用户操作集群资源，会逐层触发认证失败、授权失败报错，证明分层校验生效。

**常见报错**：

- 401 Unauthorized：认证失败，身份不合法；

- 403 Forbidden：认证成功，授权失败，无操作权限；

- admission webhook denied：准入控制拦截不合规资源。

#### ⚠️避坑指南/💡最佳实践

- ❌ 禁止集群仅配置单一权限防护，极易被绕过攻击；

- ❌ 禁止默认开放所有权限、所有网络通信；

- ✅ 严格遵循最小权限、默认拒绝、纵深防御三大原则做集群安全加固；

- ✅ 生产集群必须完整落地六层安全防御模型。

#### 📌面试/高频场景题

**Q：简述K8s六层安全防御模型？**

**答案**：从外到内分为传输加密、API认证、API授权、准入控制、网络隔离、容器运行时安全六层，所有集群操作逐层校验，实现纵深防御，防止单点安全失效。

**Q：K8s三大安全核心原则是什么？**

**答案**：最小权限原则、纵深防御原则、默认拒绝原则，是所有集群安全配置的基准规范。

### 1.2 集群常见安全威胁与攻击链路

#### 概念定义

K8s集群常见安全威胁指攻击者通过账号泄露、容器漏洞、内网渗透、配置不当等方式，入侵集群、篡改业务、窃取数据、植入木马的攻击手段。集群大部分安全事故，均来自通用的标准化攻击链路。

#### 核心原理

K8s集群组件多、资源复杂、微服务数量多，一旦任意一个微服务被入侵，若无网络与权限隔离，攻击者可横向渗透，遍历集群所有Pod、节点、密钥资源，最终接管整个集群。

#### 场景与价值

梳理生产高频攻击链路，针对性做安全加固，提前规避数据泄露、集群沦陷、业务篡改等高危生产事故。

#### 实操示例

整理生产最常见的5类高危攻击链路：

1. **低权限账号泄露**：开发账号权限过大，账号泄露后可篡改线上资源；

2. **容器漏洞逃逸**：基础镜像存在漏洞，攻击者突破容器权限，获取宿主机权限；

3. **内网横向渗透**：单个微服务被入侵，通过内网访问其他业务、数据库；

4. **敏感配置泄露**：Secret、配置文件明文存储，导致账号密码泄露；

5. **组件未加固**：Kubelet、APIServer对外开放，被恶意扫描攻击。

#### 验证与排错

通过安全审计日志、kube-bench工具扫描集群高危漏洞，定位潜在攻击风险。

常见隐患排查：权限泛滥、特权容器、网络全通、镜像高危漏洞、密钥明文。

#### ⚠️避坑指南/💡最佳实践

- ❌ 禁止开发、测试账号拥有集群全局权限；

- ❌ 禁止运行特权容器、挂载宿主机根目录；

- ✅ 所有微服务之间按需放行网络通信，默认隔离；

- ✅ 定期扫描镜像漏洞与集群安全漏洞。

#### 📌面试/高频场景题

**Q：K8s集群最常见的安全风险有哪些？**

**答案**：权限泛滥、容器逃逸、内网横向渗透、敏感信息泄露、集群组件未加固、镜像高危漏洞，以上风险会导致集群被接管、业务篡改、数据泄露。

### 1.3 安全防护整体目标与建设思路

#### 概念定义

K8s集群安全建设是一套标准化落地体系，并非零散的配置修改。整体目标是：**身份可信、权限可控、通信安全、运行安全、操作可追溯、漏洞可闭环**。

#### 核心原理

从访问入口、权限管控、网络通信、容器运行、日志审计、漏洞修复六个维度，全方位补齐集群安全短板，形成闭环安全体系，杜绝单点安全漏洞。

#### 场景与价值

适用于企业生产集群安全合规建设、等保测评、安全整改，保障集群满足企业安全规范与行业合规要求。

#### 实操示例

企业标准化集群安全建设流程：

1. 加固APIServer、kubelet、etcd核心组件；

2. 落地RBAC最小权限体系，划分开发、运维、自动化账号权限；

3. 配置准入插件，拦截高危Pod配置；

4. 配置NetworkPolicy实现微服务网络隔离；

5. 统一容器安全上下文，收紧容器权限；

6. 开启集群审计日志，实现操作可追溯；

7. 定期扫描镜像与集群漏洞，迭代修复。

#### 验证与排错

通过kube-bench、trivy工具自动化检测集群合规性，输出安全整改报告，验证加固效果。

#### ⚠️避坑指南/💡最佳实践

- ❌ 不要只做表面加固，忽略容器运行时与网络底层安全；

- ✅ 安全建设必须标准化、常态化，而非一次性整改；

- ✅ 所有安全变更必须留存记录，适配安全审计与等保要求。

#### 📌面试/高频场景题

**Q：企业K8s集群安全建设整体思路是什么？**

**答案**：基于纵深防御原则，从组件加固、权限管控、准入校验、网络隔离、容器安全、审计监控、漏洞修复全方位落地，实现身份可信、权限可控、通信安全、操作可追溯的安全目标。

## 2. API 安全三大核心机制

### 2.1 认证、授权、准入控制 执行顺序与整体流程

#### 概念定义

K8s 所有集群操作（kubectl操作、服务调用、组件通信）都会请求 **APIServer**，所有请求必须严格经过三道关卡：**认证（Authentication）→ 授权（Authorization）→ 准入控制（Admission）**，三道关卡全部通过，资源才会写入ETCD生效。

通俗类比：小区进门流程：身份核验（认证）→ 校验通行权限（授权）→ 禁止违规物品进入（准入控制）。

#### 核心原理

1. **认证**：解决「你是谁」的问题，校验访问者身份合法性；

2. **授权**：解决「你能干什么」的问题，校验身份对应的操作权限；

3. **准入控制**：解决「资源是否合规」的问题，校验创建/修改的资源配置是否符合集群安全规范。

三道流程**顺序固定不可颠倒**，前置流程失败直接拒绝请求，不会执行后续流程。

#### 场景与价值

是集群所有安全能力的底层基石，所有RBAC权限、准入规则、安全拦截均依赖该流程实现，没有三道机制，集群完全无安全防护。

#### 实操示例

模拟完整请求链路：普通用户创建特权Pod

1. 用户携带token请求APIServer创建Pod；

2. 认证模块校验token合法性；

3. 授权模块校验用户是否拥有创建Pod权限；

4. 准入插件检测Pod开启特权权限，拦截本次创建；

5. 请求失败，资源不写入集群。

#### 验证与排错

- 401：认证失败，身份无效；

- 403：认证成功、授权失败，权限不足；

- admission failed：准入控制拦截，资源不合规。

#### ⚠️避坑指南/💡最佳实践

- ❌ 禁止跳过准入控制，直接修改集群资源；

- ✅ 生产集群必须完整开启三道安全机制，缺一不可。

#### 📌面试/高频场景题

**Q：简述K8s APIServer三大安全机制与执行顺序？**

**答案**：执行顺序为认证→授权→准入控制。认证校验访问者身份，授权校验操作权限，准入控制校验资源配置合规性，三层校验全部通过后资源才能写入集群。

### 2.2 认证机制：多种身份认证方式详解

#### 概念定义

认证机制是APIServer的第一道关卡，负责识别所有访问集群的客户端身份。K8s不内置用户体系，所有用户、服务账号、组件通信身份都通过外部认证方式校验。

#### 核心原理

APIServer启动时加载多种认证插件，自上而下匹配客户端身份，任意一种认证成功即视为身份合法，全部匹配失败则返回401。

集群主流四种认证方式：

1. **客户端证书认证**：集群组件（kubelet、scheduler）默认认证方式，安全性最高；

2. **Bearer Token认证**：ServiceAccount服务账号默认认证方式；

3. **基础账号密码认证**：老旧版本方案，生产已淘汰；

4. **外部OIDC认证**：对接企业LDAP、单点登录系统。

#### 场景与价值

区分集群管理员、开发人员、集群组件、业务Pod的访问身份，实现身份统一管控，杜绝匿名访问集群。

#### 实操示例

```bash
# 查看当前用户认证配置
kubectl config view
# 查看Pod内部默认的token认证文件（ServiceAccount认证）
cat /var/run/secrets/kubernetes.io/serviceaccount/token

```

#### 验证与排错

认证失败大概率是token过期、证书过期、kubeconfig配置错误，优先检查证书有效期与token合法性。

#### ⚠️避坑指南/💡最佳实践

- ❌ 禁止使用基础密码认证，安全性极低；

- ✅ 集群组件统一使用证书认证，业务Pod使用Token认证；

- ✅ 企业统一对接OIDC实现员工账号统一登录。

#### 📌面试/高频场景题

**Q：Pod内部访问APIServer使用什么认证方式？**

**答案**：Pod通过挂载ServiceAccount的Token文件，使用Bearer Token方式完成APIServer身份认证。

### 2.3 授权机制：主流授权模式对比

#### 概念定义

授权机制是认证通过后的第二道关卡，用于控制合法身份的操作权限，决定用户可以对哪些资源执行增删改查操作。

#### 核心原理

K8s内置四种授权模式，集群可同时开启多种模式，任意一种授权通过即可放行请求：

1. **AlwaysAllow**：默认模式，全部放行，无权限校验（裸奔集群）；

2. **AlwaysDeny**：全部拒绝，仅用于测试；

3. **ABAC**：基于属性授权，老旧方案，配置复杂，已淘汰；

4. **RBAC**：基于角色授权，生产唯一标准授权模式。

#### 场景与价值

替换默认的全放行模式，精细化管控集群操作权限，解决权限泛滥问题。

#### 实操示例

```bash
# 查看集群开启的授权模式
kubectl get cm kube-apiserver -n kube-system -o yaml | grep authorization-mode

```

#### 验证与排错

生产集群必须仅开启RBAC，若开启AlwaysAllow代表集群无权限防护，存在极大安全风险。

#### ⚠️避坑指南/💡最佳实践

- ❌ 生产禁止开启AlwaysAllow、ABAC；

- ✅ 所有生产集群**强制开启RBAC授权模式**。

#### 📌面试/高频场景题

**Q：K8s四种授权模式区别，生产用哪种？**

**答案**：AlwaysAllow全部放行、AlwaysDeny全部拒绝、ABAC属性授权老旧复杂、RBAC角色授权精细化灵活。生产环境唯一标准为RBAC授权模式。

### 2.4 准入控制：变异型、校验型插件原理

#### 概念定义

准入控制是APIServer第三道关卡，属于资源写入前的**最后校验屏障**。准入插件分为两大类：**变异型（Mutating）**和**校验型（Validating）**。

通俗理解：变异型是自动改错，校验型是违规拦截。

#### 核心原理

执行顺序：**变异型插件优先执行 → 修改补齐资源配置 → 校验型插件校验合规性**，校验失败直接拒绝资源创建。

- 变异型：主动修改、补充资源配置（自动注入标签、sidecar、资源配额）；

- 校验型：仅做校验，不修改配置，不合规直接拦截报错。

#### 场景与价值

统一集群资源规范，自动补齐安全配置，拦截高危、不合规的资源配置，从源头杜绝安全隐患。

#### 实操示例

```bash
# 查看集群所有准入插件
kubectl get admissionregistration

```

#### 验证与排错

资源创建报错admission webhook deny，代表校验型插件拦截；资源配置自动新增字段，代表变异型插件生效。

#### ⚠️避坑指南/💡最佳实践

- ❌ 禁止关闭核心准入插件，会导致集群配置混乱、高危资源泛滥；

- ✅ 根据业务场景自定义准入规则，统一集群安全规范。

#### 📌面试/高频场景题

**Q：变异型和校验型准入插件的区别？**

**答案**：变异型准入插件可以修改、补齐资源配置，执行优先级更高；校验型仅校验配置合规性，不修改资源，不合规直接拦截，是集群安全校验的最后屏障。

### 2.5 集群必开核心准入插件介绍

#### 概念定义

K8s官方内置多个准入插件，部分插件为生产集群**强制开启**，用于保障集群基础安全与配置规范。

#### 核心原理

核心准入插件覆盖资源校验、安全拦截、配额管控、默认配置补齐能力，是集群稳定运行的基础保障。

#### 场景与价值

规避默认配置漏洞、拦截高危Pod、防止资源滥用，保障集群基础安全性。

#### 实操示例

生产必开五大准入插件：

1. **NamespaceLifecycle**：禁止删除正在使用的命名空间，防止资源错乱；

2. **LimitRanger**：补齐Pod默认资源配额，防止资源滥用；

3. **ResourceQuota**：限制命名空间总资源上限；

4. **DefaultStorageClass**：自动补充默认存储类；

5. **PodSecurity**：校验Pod安全配置，拦截高危容器。

#### 验证与排错

通过apiserver启动参数查看已开启准入插件，缺失核心插件需手动补齐配置。

#### ⚠️避坑指南/💡最佳实践

所有生产集群必须完整开启以上五大准入插件，缺一都会存在安全隐患或资源管控漏洞。

#### 📌面试/高频场景题

**Q：生产集群必须开启哪些准入插件？各自作用？**

**答案**：包含NamespaceLifecycle、LimitRanger、ResourceQuota、DefaultStorageClass、PodSecurity，分别用于命名空间保护、资源默认配额、总资源限制、存储默认配置、容器安全校验。

## 3. RBAC 权限体系（重中之重）

### 3.1 RBAC 核心概念、设计模型与四大资源

#### 概念定义

**RBAC（基于角色的访问控制）**是K8s生产唯一标准的授权体系，核心思想是：**不给用户直接分配权限，先定义角色权限，再将用户绑定角色**。类比公司权限：先定义岗位权限，再给员工分配岗位。

RBAC包含四大核心资源：**Role、ClusterRole、RoleBinding、ClusterRoleBinding**。

#### 核心原理

RBAC将「权限」和「身份」解耦，角色统一定义权限规则，绑定操作关联身份与角色，实现权限复用、统一管控，完美适配企业多用户、多团队、多环境权限体系。

#### 场景与价值

解决集群全员最高权限、权限泛滥、无法分权管理的问题，实现管理员、运维、开发、自动化账号权限分层隔离。

#### 实操示例

RBAC四大资源关系模型：

1. Role：命名空间内局部权限；

2. ClusterRole：集群全局权限；

3. RoleBinding：命名空间内身份绑定；

4. ClusterRoleBinding：集群全局身份绑定。

#### 验证与排错

```bash
# 查看集群所有角色与绑定关系
kubectl get role,clusterrole,rolebinding,clusterrolebinding -n kube-system

```

#### ⚠️避坑指南/💡最佳实践

- ❌ 禁止直接给用户绑定集群管理员权限；

- ✅ 严格分层定义角色，遵循最小权限原则。

#### 📌面试/高频场景题

**Q：RBAC四大资源的作用与区别？**

**答案**：Role命名空间内定义权限，ClusterRole全局定义权限；RoleBinding命名空间内绑定身份，ClusterRoleBinding全局绑定身份，实现权限与身份解耦复用。

### 3.2 Role 与 ClusterRole 权限角色详解

#### 概念定义

**Role**：命名空间级别角色，权限仅作用于**单个命名空间**，属于局部权限；

**ClusterRole**：集群级别角色，权限作用于**全集群所有命名空间、集群资源**，属于全局权限。

#### 核心原理

Role仅能管控命名空间资源（Pod、Deployment、Service），无法管控集群级资源（节点、命名空间、PV）；ClusterRole可以管控所有集群资源，权限范围更大。

#### 场景与价值

开发人员使用Role命名空间权限，仅管理自己业务环境；运维人员按需使用ClusterRole全局权限，实现权限分层。

#### 实操示例

命名空间只读Role配置：

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: dev-read-role # 开发只读角色
  namespace: test # 仅作用于test命名空间
rules:
- apiGroups: [""] # 核心资源组
  resources: ["pods","services"] # 管控资源类型
  verbs: ["get","list","watch"] # 仅允许查询，禁止修改
```

#### 验证与排错

绑定角色后，用户仅能操作对应命名空间资源，跨命名空间、集群级资源操作报403。

#### ⚠️避坑指南/💡最佳实践

- ❌ 普通开发账号禁止使用ClusterRole全局角色；

- ✅ 局部资源操作优先使用Role，最小化权限范围。

#### 📌面试/高频场景题

**Q：Role和ClusterRole核心区别？**

**答案**：Role是命名空间级别，仅管控当前命名空间资源；ClusterRole是集群级别，管控全集群资源、集群级对象，权限范围更大。

### 3.3 RoleBinding 与 ClusterRoleBinding 绑定规则

#### 概念定义

角色仅定义权限规则，无法直接生效，必须通过**绑定资源**将权限赋予用户/服务账号。

RoleBinding：在指定命名空间，将用户绑定Role/ClusterRole；

ClusterRoleBinding：全局绑定用户，权限覆盖全集群。

#### 核心原理

ClusterRole可以被RoleBinding局部绑定，也可以被ClusterRoleBinding全局绑定；Role仅能被RoleBinding绑定，无法全局生效。

#### 场景与价值

实现全局角色局部复用，无需重复创建角色，简化权限管理。

#### 实操示例

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: dev-read-bind
  namespace: test
subjects:
- kind: User
  name: dev-user # 绑定开发用户
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: dev-read-role # 关联自定义只读角色
  apiGroup: rbac.authorization.k8s.io

```

#### 验证与排错

绑定失败大概率是命名空间不匹配、角色名称错误、用户不存在。

#### ⚠️避坑指南/💡最佳实践

优先使用RoleBinding做局部绑定，严格限制ClusterRoleBinding全局绑定的使用场景。

#### 📌面试/高频场景题

**Q：ClusterRole能否通过RoleBinding绑定？有什么效果？**

**答案**：可以，ClusterRole通过RoleBinding绑定后，权限仅作用于当前命名空间，实现全局角色局部复用。

### 3.4 权限规则语法：apiGroups / resources / verbs

#### 概念定义

RBAC权限规则由三大核心字段组成，精准定义权限范围：

- **apiGroups**：资源所属API组；

- **resources**：需要管控的资源对象；

- **verbs**：可执行的操作动作。

#### 核心原理

K8s所有资源归属不同API组，不同资源支持不同操作动作，三者组合精准锁定权限边界。

#### 场景与价值

精细化自定义权限，避免权限过大，完美落地最小权限原则。

#### 实操示例

```yaml
rules:
- apiGroups: ["apps"] # Deployment所属API组
  resources: ["deployments"] # 管控部署资源
  verbs: ["get","list","watch","update"] # 允许查询、更新，禁止删除创建

```

#### 验证与排错

权限不生效大概率是apiGroups填写错误，不同资源API组不可混用。

#### ⚠️避坑指南/💡最佳实践

禁止使用verbs: ["*"]全部权限，必须按需拆分操作权限。

#### 📌面试/高频场景题

**Q：RBAC规则三大字段作用？**

**答案**：apiGroups指定资源分组，resources指定管控资源，verbs指定操作权限，三者组合定义精准的资源操作权限。

### 3.5 ServiceAccount 原理 & 容器内 RBAC 授权

#### 概念定义

**ServiceAccount（服务账号）**：专门给**Pod内部业务**使用的账号，区别于人工用户账号，用于Pod内部程序调用APIServer、操作集群资源。

#### 核心原理

Pod绑定ServiceAccount后，会自动挂载对应的Token文件，容器内程序通过Token认证，结合RBAC规则获取集群操作权限。默认ServiceAccount权限极低，无法操作集群资源。

#### 场景与价值

适配监控、CI/CD、自动扩缩容等需要Pod内部调用集群API的业务场景。

#### 实操示例

```yaml
# 创建服务账号
apiVersion: v1
kind: ServiceAccount
metadata:
  name: pod-sa
  namespace: test
---
# 绑定权限
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: sa-bind
  namespace: test
subjects:
- kind: ServiceAccount
  name: pod-sa
roleRef:
  kind: Role
  name: dev-read-role
  apiGroup: rbac.authorization.k8s.io

```

#### 验证与排错

进入Pod内部，通过kubectl测试集群资源访问权限，验证授权生效。

#### ⚠️避坑指南/💡最佳实践

禁止给业务Pod绑定高权限ServiceAccount，防止容器被入侵后接管集群。

#### 📌面试/高频场景题

**Q：User账号和ServiceAccount账号区别？**

**答案**：User用于人工登录操作集群，ServiceAccount用于Pod内部程序访问集群API，是容器专属账号。

### 3.6 常用内置 ClusterRole 介绍

#### 概念定义

K8s内置大量通用ClusterRole，无需手动创建，开箱即用，覆盖集群大部分权限场景。

#### 核心原理

内置角色由官方标准化定义，权限边界清晰，适配常规运维、只读、编辑场景。

#### 场景与价值

简化权限配置，无需重复自定义角色，统一企业权限标准。

#### 实操示例

四大高频内置角色：

- **cluster-admin**：超级管理员，最高权限；

- **admin**：命名空间管理员，可修改命名空间所有资源；

- **edit**：资源编辑权限，无权限管理能力；

- **view**：纯只读权限，无法修改任何资源。

#### 验证与排错

```bash
# 查看所有内置集群角色
kubectl get clusterrole

```

#### ⚠️避坑指南/💡最佳实践

生产仅运维账号使用cluster-admin，普通开发仅使用view/edit角色。

#### 📌面试/高频场景题

**Q：简述四大内置ClusterRole权限区别？**

**答案**：cluster-admin全局超级权限，admin命名空间管理权限，edit资源编辑权限，view全局只读权限。

### 3.7 RBAC 生产配置、最小权限原则

#### 概念定义

RBAC生产标准化配置，核心遵循**最小权限原则**：任何账号仅拥有完成工作必需的最低权限，无任何多余权限。

#### 核心原理

权限越大，泄露风险越高。最小权限可以最大程度降低账号泄露、容器入侵后的集群风险。

#### 场景与价值

统一企业权限规范，杜绝权限泛滥，加固集群安全边界。

#### 实操示例

企业三层权限模型：

1. 运维：cluster-admin超级权限；

2. 测试开发：命名空间view/edit权限；

3. 业务Pod：自定义最小权限ServiceAccount。

#### 验证与排错

定期巡检集群账号权限，清理多余绑定、高权限普通账号。

#### ⚠️避坑指南/💡最佳实践

禁止批量赋予用户全局管理员权限，所有权限按需单独配置。

#### 📌面试/高频场景题

**Q：什么是RBAC最小权限原则？生产为什么要遵守？**

**答案**：最小权限原则即账号仅拥有必需的操作权限，无多余权限；可以有效降低账号泄露、服务被入侵后的集群安全风险。

### 3.8 权限 403 报错排查与 auth can-i 工具

#### 概念定义

403 Forbidden是集群最高频报错，代表身份认证成功，但权限不足。kubectl auth can-i是官方自带权限检测工具，用于快速校验账号权限、排查权限报错。

#### 核心原理

auth can-i工具模拟当前用户操作，调用RBAC授权模块校验权限，直接返回是否允许，无需手动测试资源操作。

#### 场景与价值

快速定位权限不足问题，大幅提升权限故障排查效率。

#### 实操示例

```bash
# 检测当前用户是否可以创建Pod
kubectl auth can-i create pods -n test
# 检测指定用户权限
kubectl auth can-i update deployments -n test --as=dev-user

```

#### 验证与排错

403排查标准流程：确认用户身份→使用auth can-i校验权限→检查Role/RoleBinding配置→补齐权限。

#### ⚠️避坑指南/💡最佳实践

所有权限变更前，优先使用auth can-i预校验权限，避免业务操作失败。

#### 📌面试/高频场景题

**Q：集群403报错排查思路是什么？**

**答案**：403代表认证成功授权失败，使用auth can-i校验当前账号权限，检查角色规则与绑定关系，补充对应资源操作权限。

## 4. 容器运行时安全

### 4.1 容器高危风险：特权、逃逸、挂载越权

#### 概念定义

容器运行时风险是集群最致命的安全隐患，指容器配置不当导致攻击者突破容器隔离，获取宿主机权限、篡改主机文件、接管集群节点。三大高危风险：**特权容器、容器逃逸、挂载越权**。

#### 核心原理

容器默认共享宿主机内核，若开启特权权限、挂载主机目录、放开系统调用，攻击者可通过漏洞突破容器沙箱隔离，直接操作宿主机系统文件、执行主机命令。不同于独立虚拟机，容器没有完整操作系统隔离，一旦配置放松，隔离机制彻底失效，是生产集群最高危、最容易被忽视的安全漏洞。

#### 场景与价值

线上微服务大多基于开源镜像构建，镜像大概率自带漏洞。一旦业务漏洞被利用，若容器存在特权、目录挂载越权配置，攻击者可直接逃逸控制节点，接管整集群。补齐容器运行时安全配置，可以从底层筑牢容器隔离边界，杜绝容器逃逸风险。

#### 实操示例

三类高危违规Pod配置示例（生产禁止使用）：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: danger-pod
spec:
  # 开启特权容器，拥有宿主机所有权限
  securityContext:
    privileged: true
  containers:
  - name: nginx
    image: nginx:latest
    # 挂载宿主机根目录，完全越权读写主机文件
    volumeMounts:
    - name: host-root
      mountPath: /host
  volumes:
  - name: host-root
    hostPath:
      path: /

```

#### 验证与排错

**验证方式**：通过kubectl查询集群特权容器、主机挂载容器

```bash
# 筛选集群所有特权Pod
kubectl get pods --all-namespaces -o yaml | grep privileged: true
# 筛选挂载宿主机目录的Pod
kubectl get pods --all-namespaces -o yaml | grep hostPath

```

**常见问题**：老旧业务为适配硬件、日志采集、挂载配置，大量开启特权与主机挂载，存在极高逃逸风险。

#### ⚠️避坑指南/💡最佳实践

- ❌ 生产绝对禁止业务Pod开启privileged特权模式；

- ❌ 禁止业务Pod随意挂载hostPath宿主机目录；

- ✅ 硬件设备、特殊运维场景必需特权容器时，单独白名单放行，严格审计；

- ✅ 统一通过安全上下文收紧容器系统权限，缩小攻击面。

#### 📌面试/高频场景题

**Q：什么是容器逃逸？生产如何预防？**

**答案**：容器共享宿主机内核，若容器开启特权权限、挂载主机目录、放开系统调用，攻击者可利用镜像漏洞突破容器隔离，控制宿主机。预防方案：关闭特权容器、禁止随意主机挂载、配置安全上下文、遵循Pod安全标准、镜像漏洞扫描。

### 4.2 SecurityContext 安全上下文全解

#### 概念定义

**SecurityContext（安全上下文）**是K8s专属的容器安全配置单元，用于精准定义Pod与容器的运行权限、文件系统权限、系统调用权限、用户权限，相当于给容器加一层安全枷锁，限制容器的最大权限，杜绝越权逃逸。

安全上下文分为两个粒度：**Pod级别全局配置**、**容器级别单独配置**，容器级配置优先级高于Pod级。

#### 核心原理

容器默认以root用户运行、拥有完整系统调用权限、可读写文件系统，权限极其宽松。通过SecurityContext可以：修改运行用户、禁止特权权限、只读挂载文件系统、禁用高危系统能力、屏蔽核心敏感目录，从运行时层面收紧容器权限。

#### 场景与价值

是生产集群容器安全加固的基础配置，无需改造业务代码，仅通过YAML配置即可大幅降低容器逃逸、越权风险，适配所有微服务业务。

#### 实操示例

生产标准安全上下文配置模板：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  # Pod级别安全配置
  securityContext:
    runAsNonRoot: true # 禁止root用户运行
    runAsUser: 1000 # 使用普通用户UID运行
    fsGroup: 1000 # 文件系统归属普通用户
  containers:
  - name: nginx
    image: nginx:latest
    # 容器级别安全配置（优先级更高）
    securityContext:
      privileged: false # 关闭特权模式
      readOnlyRootFilesystem: true # 根文件系统只读
      allowPrivilegeEscalation: false # 禁止权限提升

```

#### 验证与排错

```bash
# 查看Pod安全配置
kubectl get pod secure-pod -o yaml | grep securityContext

```

常见问题：开启只读文件系统后，业务需要临时写入目录会启动失败，需单独挂载emptyDir临时目录解决。

#### ⚠️避坑指南/💡最佳实践

- ❌ 所有业务容器禁止默认root运行、禁止权限提升；

- ✅ 所有生产Pod统一配置标准化安全上下文；

- ✅ 区分Pod级与容器级配置，特殊容器单独差异化收紧权限。

#### 📌面试/高频场景题

**Q：SecurityContext核心作用是什么？Pod级和容器级区别？**

**答案**：安全上下文用于收紧容器运行权限，防止越权逃逸。Pod级作用于所有容器，容器级仅作用于当前容器，容器级配置优先级更高。

### 4.3 非root运行、只读文件系统、能力收紧

#### 概念定义

容器三大核心安全加固配置：**非root运行、根文件系统只读、关闭权限提升、收紧Linux Capabilities能力**，是生产必配的基础安全项。

- 非root运行：容器放弃最高权限用户，使用普通用户运行业务；

- 只读文件系统：禁止容器修改系统目录与业务镜像文件；

- 权限关闭：禁止容器运行中提升自身权限；

- 能力收紧：剥离Linux高危系统调用能力。

#### 核心原理

容器root用户不等于宿主机root，但拥有容器内最高权限，一旦业务被入侵，攻击者可修改容器配置、植入木马。通过以上配置，即使容器被攻破，攻击者无法修改文件、无法提升权限、无法调用高危系统指令，彻底锁死容器逃逸路径。

#### 场景与价值

适配所有Java、Go、Python微服务，无需业务改造，零侵入加固容器安全，是企业等保测评必查项。

#### 实操示例

完整能力收紧安全配置：

```yaml
securityContext:
  runAsNonRoot: true
  runAsUser: 1000
  readOnlyRootFilesystem: true
  allowPrivilegeEscalation: false
  # 清空所有Linux系统能力
  capabilities:
    drop:
      - ALL

```

#### 验证与排错

非root运行报错：镜像未预置普通用户，需重构基础镜像；只读文件系统报错：业务存在本地写入逻辑，需挂载临时存储目录。

#### ⚠️避坑指南/💡最佳实践

禁止保留多余Linux系统能力，默认全部丢弃，仅按需添加必需能力。

#### 📌面试/高频场景题

**Q：生产容器为什么要开启只读文件系统？**

**答案**：防止容器被入侵后篡改系统文件、植入木马、持久化恶意程序，限制攻击者在容器内的操作权限，阻断逃逸与驻留攻击。

### 4.4 PodSecurityPolicy 历史方案介绍

#### 概念定义

**PodSecurityPolicy（PSP）**是K8s旧版本用于全局管控Pod安全配置的准入资源，用于统一限制集群所有Pod的特权、用户、文件系统权限，实现集群容器安全统一管控。该资源在 **K8s 1.21废弃，1.25彻底移除**。

#### 核心原理

PSP作为准入插件，拦截所有Pod创建请求，校验Pod的SecurityContext配置是否符合集群安全策略，不合规直接拒绝创建，实现全局统一安全标准。

#### 场景与价值

旧版本集群用于统一容器安全规范，替代人工逐个配置安全上下文，统一管控集群容器风险，现已被全新方案替代。

#### 实操示例

老旧PSP策略模板（仅学习，生产禁止使用）：

```yaml
apiVersion: policy/v1beta1
kind: PodSecurityPolicy
metadata:
  name: restricted-psp
spec:
  privileged: false
  readOnlyRootFilesystem: true
  runAsUser:
    rule: MustRunAsNonRoot

```

#### 验证与排错

高版本集群创建PSP资源会直接报错，资源不存在。

#### ⚠️避坑指南/💡最佳实践

- ❌ 新版本集群禁止使用PSP；

- ✅ 统一使用Pod Security Standards替代。

#### 📌面试/高频场景题

**Q：PSP是什么？目前是否还推荐使用？**

**答案**：PSP是旧版Pod安全准入策略，用于全局管控容器安全配置，K8s 1.25版本已彻底移除，官方废弃，生产统一使用PSS标准。

### 4.5 Pod Security Standards 三级安全标准

#### 概念定义

**Pod Security Standards（PSS）**是官方替代PSP的全新容器安全标准，分为三级安全规范，配合PodSecurity准入插件使用，是当前生产唯一标准的容器全局安全管控方案。三级标准：**Privileged、Baseline、Restricted**。

#### 核心原理

通过命名空间打标签的方式，为不同命名空间绑定不同安全等级，自动校验当前命名空间下所有Pod配置，不合规直接拦截，无需逐个编写策略YAML，轻量化、官方标准化。

三级等级说明：

1. **Privileged（特权级）**：无任何限制，兼容所有Pod配置，用于运维特殊组件；

2. **Baseline（基础级）**：拦截高危逃逸配置，兼容绝大多数业务，适合测试环境；

3. **Restricted（受限级）**：最严格安全标准，强制所有安全加固配置，生产环境标准。

#### 场景与价值

统一集群全局容器安全规范，替代老旧PSP，适配新版K8s集群，满足生产安全合规要求。

#### 实操示例

```bash
# 为命名空间绑定受限级安全策略（生产标准）
kubectl label namespace prod pod-security.kubernetes.io/enforce=restricted

```

#### 验证与排错

命名空间绑定受限级策略后，创建特权Pod、root运行Pod会直接准入拦截，创建失败。

#### ⚠️避坑指南/💡最佳实践

生产业务命名空间统一使用restricted级别，运维组件命名空间按需使用privileged级别。

#### 📌面试/高频场景题

**Q：简述PSS三级安全标准区别与生产选型？**

**答案**：Privileged无限制、Baseline基础安全拦截、Restricted最严格限制。生产业务命名空间统一使用Restricted受限级。

### 4.6 容器镜像安全与基础镜像选型规范

#### 概念定义

容器镜像作为容器运行的基础，镜像自带的系统漏洞、预装高危组件、多余工具是集群最隐蔽的安全隐患。镜像安全即通过**基础镜像选型、漏洞扫描、精简镜像、禁止多余工具**，从源头杜绝容器漏洞。

#### 核心原理

镜像层级越多、预装组件越多、使用通用系统镜像（centos/ubuntu），漏洞数量越多。精简镜像、专用镜像可以大幅缩小攻击面，从源头减少可被利用的漏洞。

#### 场景与价值

绝大部分容器逃逸、入侵漏洞均来自镜像高危漏洞，规范镜像选型与构建标准，是容器安全的第一道防线。

#### 实操示例

镜像生产规范：

1. 优先使用官方精简镜像：alpine、slim版本；

2. 禁止使用最新标签latest，固定镜像版本；

3. 镜像构建清理yum/apt、卸载多余工具；

4. 通过Trivy扫描镜像高危漏洞。

```bash
# trivy扫描本地镜像漏洞
trivy image nginx:latest

```

#### 验证与排错

定期扫描业务镜像高危漏洞，修复高危、严重级漏洞，禁止带高危漏洞镜像上线生产。

#### ⚠️避坑指南/💡最佳实践

- ❌ 禁止使用latest动态标签、禁止使用完整版系统镜像；

- ✅ 统一企业私有镜像仓库，拦截高危镜像；

- ✅ 镜像分层精简，不预装调试、黑客工具。

#### 📌面试/高频场景题

**Q：如何保障容器镜像安全？**

**答案**：选用精简官方镜像、固定镜像版本、精简镜像组件、自动化扫描漏洞、私有仓库准入拦截、定期修复迭代漏洞。

## 5. NetworkPolicy 网络安全隔离

### 5.1 NetworkPolicy 核心概念与使用前提

#### 概念定义

**NetworkPolicy（网络策略）**是K8s原生网络隔离资源，用于管控Pod之间、Pod与外部的网络访问权限，实现微服务之间网络白名单隔离，防止内网横向渗透攻击。通俗理解：给每个微服务配置防火墙规则。

#### 核心原理

K8s集群默认**所有Pod内网互通**，无任何隔离。NetworkPolicy基于标签匹配机制，通过白名单规则控制Pod入站、出站流量，默认拒绝所有陌生通信，仅放行指定服务。

#### 场景与价值

微服务架构下服务数量多，单一服务被入侵极易横向扩散。网络策略实现服务隔离，攻击无法横向扩散，保障内网安全。

#### 实操示例

使用前提：集群CNI插件必须支持NetworkPolicy（Calico、Weave），原生Flannel不支持。

```bash
# 校验集群是否支持网络策略
kubectl get networkpolicy --all-namespaces

```

#### 验证与排错

创建网络策略不生效，大概率是CNI插件不支持，需更换Calico插件。

#### ⚠️避坑指南/💡最佳实践

所有生产集群必须部署Calico，开启NetworkPolicy能力。

#### 📌面试/高频场景题

**Q：NetworkPolicy作用与使用前提？**

**答案**：用于Pod网络隔离，管控内外网络流量，防止内网横向渗透。必须依赖支持网络策略的CNI插件（Calico）。

### 5.2 白名单机制、Ingress / Egress 规则

#### 概念定义

NetworkPolicy 严格遵循**默认拒绝、白名单放行**机制，包含两类核心规则：

- **Ingress（入站）**：管控外部访问当前Pod的流量；

- **Egress（出站）**：管控当前Pod访问外部的流量。

#### 核心原理

未配置任何规则时，所有流量放行；配置任意一条白名单规则后，默认拒绝所有未匹配流量，仅放行白名单内通信。

#### 场景与价值

精准控制微服务调用关系，仅允许上下游依赖服务互通，阻断多余内网通信。

#### 实操示例

基础网络策略模板：仅允许指定Pod访问当前服务

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: test-np
  namespace: test
spec:
  podSelector:
    matchLabels:
      app: nginx # 匹配目标Pod
  policyTypes:
  - Ingress # 开启入站管控
  - Egress # 开启出站管控
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: client # 仅client标签Pod可以访问

```

#### 验证与排错

配置后非白名单Pod访问超时断开，代表策略生效；业务访问异常优先检查网络策略拦截。

#### ⚠️避坑指南/💡最佳实践

开启Ingress/Egress管控后，需按需放行DNS、外网依赖接口，防止业务断连。

#### 📌面试/高频场景题

**Q：NetworkPolicy白名单机制是什么？**

**答案**：网络策略默认全部放行，配置任意白名单规则后，默认拒绝所有陌生流量，仅放行规则内的Pod通信。

### 5.3 标签选择器、命名空间选择器、IP 段控制

#### 概念定义

网络策略支持三种流量匹配方式，适配复杂微服务隔离场景：**Pod标签选择器、命名空间选择器、IP网段选择器**。

#### 核心原理

通过多维度选择器，可以精准匹配：同命名空间指定服务、跨命名空间服务、指定内网IP段、外网IP，实现精细化网络隔离。

#### 场景与价值

适配多环境、多命名空间集群，实现测试环境与生产环境网络隔离、内外网隔离。

#### 实操示例

```yaml
ingress:
# 放行指定命名空间所有Pod
- namespaceSelector:
    matchLabels:
      env: dev
# 放行指定内网IP段
- ipBlock:
    cidr: 10.244.0.0/16

```

#### 验证与排错

跨命名空间访问失败，优先检查命名空间标签与网络策略规则。

#### ⚠️避坑指南/💡最佳实践

禁止使用宽泛IP段放行，精准限制业务依赖地址。

#### 📌面试/高频场景题

**Q：NetworkPolicy三种匹配方式？**

**答案**：Pod标签选择器匹配单个服务、命名空间选择器匹配整个环境、IP段选择器匹配指定网段。

### 5.4 通用生产规则：全局默认隔离、按需放行

#### 概念定义

生产标准网络安全规范：**全局默认网络隔离，所有Pod默认禁止互通，仅按需放行上下游依赖服务**。

#### 核心原理

先阻断所有内网横向通信，再根据业务依赖关系逐个添加白名单，从根源杜绝横向渗透攻击。

#### 场景与价值

适配所有生产微服务集群，是企业网络安全合规标配。

#### 实操示例

全局默认隔离策略：

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: prod
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress

```

#### 验证与排错

配置后所有Pod网络中断，需及时配置业务白名单规则。

#### ⚠️避坑指南/💡最佳实践

上线全局隔离前，提前梳理业务依赖拓扑，避免大规模业务故障。

#### 📌面试/高频场景题

**Q：生产网络策略最佳实践是什么？**

**答案**：遵循默认拒绝原则，全局默认隔离所有Pod通信，基于业务依赖按需放行白名单。

### 5.5 微服务隔离、跨命名空间访问限制实战

#### 概念定义

企业集群通常拆分dev/test/prod多命名空间，需要实现**环境隔离、服务单独隔离、禁止跨环境访问**。

#### 核心原理

通过命名空间标签+Pod标签双层限制，实现测试环境无法访问生产环境，不同微服务仅保留必要调用链路。

#### 场景与价值

防止测试环境漏洞渗透生产环境，保障线上环境网络安全。

#### 实操示例

禁止dev命名空间访问prod命名空间资源，仅放行网关访问后端服务。

#### 验证与排错

跨环境访问拦截正常，符合安全预期；业务调用异常核对调用拓扑与白名单。

#### ⚠️避坑指南/💡最佳实践

严格禁止测试环境主动访问生产数据库、核心服务。

#### 📌面试/高频场景题

**Q：如何实现多命名空间环境网络隔离？**

**答案**：通过命名空间选择器配置网络策略，禁止测试环境命名空间访问生产命名空间，实现环境隔离。

### 5.6 网络策略排错与 CNI 兼容性说明

#### 概念定义

网络策略生效依赖底层CNI插件，不同CNI对网络策略支持度不同，排错需要结合插件特性与规则配置。

#### 核心原理

Flannel不支持NetworkPolicy，Calico完整支持所有规则，是生产唯一标准CNI插件。

#### 场景与价值

快速定位网络策略不生效、业务访问异常问题。

#### 实操示例

排错流程：检查CNI插件→检查策略标签匹配→检查Ingress/Egress配置→检查全局默认策略。

#### 验证与排错

策略配置无误但不生效，100%为CNI插件不兼容，需替换Calico。

#### ⚠️避坑指南/💡最佳实践

生产集群初始化必须部署Calico CNI，提前规避网络隔离能力缺失问题。

#### 📌面试/高频场景题

**Q：为什么NetworkPolicy配置后不生效？**

**答案**：大概率是CNI插件（如Flannel）不支持网络策略，生产必须使用Calico插件。

## 6. 集群安全加固与漏洞防护

### 6.1 控制面组件安全加固（APIServer/ETCD/Scheduler）

#### 概念定义

控制面组件（APIServer、ETCD、Scheduler、ControllerManager）是集群大脑，一旦被入侵集群彻底沦陷，必须全方位加固访问权限、端口暴露、通信加密。

#### 核心原理

控制面组件默认仅内网监听，禁止公网暴露，所有通信HTTPS加密，关闭匿名访问、放开最小权限。

#### 场景与价值

防止核心组件被外网扫描、匿名访问、恶意篡改集群资源。

#### 实操示例

核心加固项：关闭APIServer匿名访问、禁止未授权请求、ETCD加密存储、组件双向证书认证。

#### 验证与排错

通过kube-bench扫描控制面加固合规性，修复高危配置漏洞。

#### ⚠️避坑指南/💡最佳实践

绝对禁止APIServer公网直接暴露，必须通过内网、堡垒机访问。

#### 📌面试/高频场景题

**Q：APIServer核心安全加固方式？**

**答案**：关闭匿名访问、开启HTTPS加密、限制访问IP、严格RBAC权限、禁止公网暴露。

### 6.2 节点安全：kubelet 访问控制、主机权限收紧

#### 概念定义

Kubelet是节点核心组件，负责管理本机Pod，若权限放开、端口暴露，攻击者可直接操控节点所有容器。

#### 核心原理

kubelet默认10255端口无认证可读，10250端口支持读写，需收紧端口暴露、开启认证授权。

#### 场景与价值

防止节点被外网扫描、恶意读取容器信息、篡改Pod配置。

#### 实操示例

关闭kubelet匿名访问、开启证书认证、限制端口监听内网。

#### 验证与排错

扫描节点高危端口，关闭对外开放的高危端口。

#### ⚠️避坑指南/💡最佳实践

禁止kubelet端口对公网开放，所有节点端口仅内网通信。

#### 📌面试/高频场景题

**Q：kubelet安全风险与加固方案？**

**答案**：kubelet默认端口可匿名访问，容易泄露容器信息。加固方式：关闭匿名访问、开启证书认证、内网监听、收紧节点权限。

### 6.3 密钥与敏感配置安全：Secret 使用规范

#### 概念定义

Secret用于存储密码、密钥、token、证书等敏感数据，Secret仅Base64编码、非加密，配置不当极易泄露。

#### 核心原理

Base64可直接解码，因此Secret绝对不能等同于加密存储，必须配合权限管控、加密存储使用。

#### 场景与价值

防止数据库密码、接口密钥、证书泄露导致业务数据被盗。

#### 实操示例

Secret生产规范：禁止明文配置、最小权限访问、加密存储、定期轮换密钥。

#### 验证与排错

排查普通用户读取Secret权限，禁止低权限账号查看敏感密钥。

#### ⚠️避坑指南/💡最佳实践

禁止使用ConfigMap存储敏感数据，敏感数据统一使用Secret并加密。

#### 📌面试/高频场景题

**Q：Secret是加密存储吗？生产如何保障密钥安全？**

**答案**：Secret仅Base64编码，未加密。生产需配合RBAC最小权限、ETCD加密、外部密钥管理系统保障安全。

### 6.4 集群审计日志、操作追溯、安全监控

#### 概念定义

集群审计日志记录所有APIServer请求操作，包含账号、操作资源、操作时间、IP，是安全事故追溯、等保测评的核心依据。

#### 核心原理

所有集群操作必经APIServer，开启审计后完整记录增删改查操作，实现操作可追溯、事故可定位。

#### 场景与价值

出现权限泄露、资源篡改、数据丢失后，快速定位操作人与攻击链路。

#### 实操示例

配置apiserver审计参数，开启日志持久化，对接日志系统采集分析。

#### 验证与排错

操作资源后查看审计日志，校验日志记录完整性。

#### ⚠️避坑指南/💡最佳实践

生产集群必须永久开启审计日志，长期持久化存储。

#### 📌面试/高频场景题

**Q：集群审计日志作用？**

**答案**：完整记录所有集群操作行为，用于安全追溯、故障定位、安全合规与等保测评。

### 6.5 常见高危漏洞复盘与修复方案

#### 概念定义

K8s历年高危漏洞包含权限提升、容器逃逸、拒绝服务、越权访问等，漏洞暴露后可直接接管集群。

#### 核心原理

集群组件、容器运行时持续爆出漏洞，版本过低、配置不当是漏洞被利用的核心原因。

#### 场景与价值

规避已知高危漏洞，修复集群安全短板。

#### 实操示例

定期关注官方漏洞公告，迭代集群版本、修复组件漏洞、加固配置。

#### 验证与排错

通过安全扫描工具检测集群已知漏洞。

#### ⚠️避坑指南/💡最佳实践

集群版本不要长期停滞，定期小版本迭代修复漏洞。

#### 📌面试/高频场景题

**Q：如何应对K8s高危漏洞？**

**答案**：定期漏洞扫描、迭代集群小版本、加固安全配置、收紧权限与网络隔离。

### 6.6 开源安全检测工具：kube-bench / Trivy

#### 概念定义

**kube-bench**：集群合规检测工具，对照CIS标准扫描集群配置漏洞；**Trivy**：容器、镜像、文件漏洞扫描工具，检测高危漏洞与配置风险。

#### 核心原理

工具自动化对照官方安全标准，批量检测集群、镜像安全隐患，输出整改报告。

#### 场景与价值

替代人工巡检，标准化、自动化完成集群安全检测与合规测评。

#### 实操示例

```bash
# kube-bench集群安全检测
kube-bench
# trivy镜像漏洞扫描
trivy image 业务镜像名
```

#### 验证与排错

根据工具输出的高危、中危问题逐条修复加固。

#### ⚠️避坑指南/💡最佳实践

生产每周自动化执行安全扫描，形成常态化安全巡检机制。

#### 📌面试/高频场景题

**Q：kube-bench和Trivy各自作用？**

**答案**：kube-bench检测集群组件配置合规漏洞，Trivy扫描容器镜像高危漏洞，搭配实现集群全方位安全检测。
