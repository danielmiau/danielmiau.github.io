# 04-高频运维操作
## 本章概述
### 本章核心价值与学习目标
Kubernetes 的学习分为**理论认知**和**生产运维**两大板块，本章聚焦生产 90% 以上的日常运维场景，是从入门走向落地的关键章节。大部分企业K8s运维工作，无外乎资源管理、应用发布、容器调试、配置更新、节点维护、故障排查六大场景。

本章面向有Java开发基础的学习者，摒弃晦涩的底层源码，主打**实操落地 + 原理吃透 + 面试通关**。学习完本章后，你将具备独立完成企业K8s日常运维、线上故障快速排查、应用灰度回滚、集群维护的能力，同时覆盖面试高频运维考题，解决“看得懂概念，不会操作、不会排错、面试答不上”的问题。

### 前置知识依赖（核心资源对象）
学习本章必须掌握以下基础资源概念，本章所有操作均基于这些资源：
1. **Namespace**：命名空间，用于集群资源隔离，类比服务器的不同文件夹
2. **Pod**：K8s最小调度单元，承载容器运行，是所有应用的载体
3. **Deployment**：无状态应用控制器，负责Pod创建、管理、滚动更新
4. **Service**：实现Pod网络统一暴露与流量分发
5. **ConfigMap/Secret**：集群配置与敏感数据管理资源
6. **Node**：集群工作节点，承载所有Pod运行

### 本章知识地图：从基础操作到故障排查的运维全流程
本章遵循**由浅入深、从日常操作到故障应急**的生产运维逻辑：
`kubectl基础命令` → `资源增删改查` → `Pod精细化调试` → `应用发布与灰度回滚` → `配置热更新运维` → `节点停机维护` → `集群故障排查与备份恢复`

## 1. 集群运维基础命令全集
### 1.1 kubectl 基础配置与上下文管理
#### 概念定义
`kubectl` 是Kubernetes官方唯一的客户端命令行工具，相当于集群的**操作遥控器**，所有对集群的操作，全部通过该工具发送指令。上下文（context）可以简单理解为**集群登录账号**，用于在多套开发、测试、生产集群之间快速切换，无需重复配置认证信息。

#### 核心原理
K8s采用**C/S架构**，kube-apiserver是集群服务端，kubectl是客户端。所有kubectl命令本质是封装了HTTP请求，携带认证信息访问apiserver。
`kubeconfig` 配置文件统一存储**集群地址、用户认证密钥、默认命名空间**，上下文将三者绑定，让客户端可以精准、安全地对接不同集群，实现多集群隔离。

#### 场景与价值
企业生产普遍存在**多集群环境**（开发、测试、预发、生产），如果没有上下文管理，每次切换集群都需要修改配置文件，极易出错。上下文管理可以一键切换集群，搭配别名配置，大幅提升运维效率，降低人为操作事故。

#### 实操示例
##### 1. kubectl 安装（Linux CentOS7/8）
```bash
# 下载稳定版kubectl
curl -LO https://dl.k8s.io/v1.28.2/linux/amd64/kubectl
# 添加可执行权限
chmod +x kubectl
# 移动至全局环境目录
mv kubectl /usr/local/bin/
# 验证安装是否成功
kubectl version --client
```

##### 2. kubeconfig 文件核心解析
默认路径：`~/.kube/config`，文件分为三大核心模块：
- `clusters`：存储集群API地址、CA证书，用于连接集群
- `users`：存储客户端证书、token，用于集群身份认证
- `contexts`：关联集群与用户，绑定默认操作命名空间

##### 3. 多集群上下文切换
```bash
# 查看所有已配置集群上下文
kubectl config get-contexts
# 切换至生产集群
kubectl config use-context k8s-prod
# 为当前上下文设置默认命名空间
kubectl config set-context --current --namespace=prod-app
```

##### 4. 别名与自动补全
```bash
# 配置别名 k 替代 kubectl
echo "alias k=kubectl" >> ~/.bashrc
# 配置命令自动补全
kubectl completion bash > /etc/bash_completion.d/kubectl
# 生效所有配置
source ~/.bashrc
```

#### 验证与排错
1. 验证：执行 `k get nodes`，正常输出节点列表即为配置生效
2. 常见报错1：`permission denied`
   - 原因：kubeconfig文件权限过宽，集群拒绝认证
   - 解决：`chmod 600 ~/.kube/config`
3. 常见报错2：连接超时
   - 原因：集群地址错误、服务器防火墙拦截、网络不通

#### ⚠️避坑指南/💡最佳实践
1. 禁止手动编辑kubeconfig文件，容易造成格式错乱，优先使用config命令操作
2. 多集群上下文命名规范统一：`k8s-dev、k8s-test、k8s-prod`
3. 所有运维机器必须配置别名与自动补全，标准化运维操作

#### 📌面试/高频场景题
**Q：kubeconfig 三大核心组件是什么？各自作用？**
A：包含 clusters、users、contexts。
- clusters：定义集群连接地址与证书
- users：定义客户端用户身份与认证密钥
- contexts：绑定集群与用户，实现多集群快速切换，定义当前操作环境

**Q：为什么 kubeconfig 文件权限必须是600？**
A：该文件包含集群登录密钥，权限过大会导致其他用户读取窃取集群权限，造成集群安全风险。

### 1.2 常用资源查看命令
#### 概念定义
kubectl 查看三剑客：`get`、`describe`、`logs`、`events`，是运维最基础、最高频的命令。
- `get`：快速查询资源列表与基础状态
- `describe`：深度查询资源详情、调度信息、异常事件
- `logs`：查看容器应用日志
- `events`：查看集群所有资源变更、报错事件

#### 核心原理
apiserver 存储集群所有资源的Etcd数据，`get` 轻量化读取资源元数据；`describe` 聚合资源配置、调度策略、节点信息、事件日志；`events` 是集群的**操作日志**，记录所有资源的创建、更新、失败记录，是排错核心。

#### 场景与价值
日常集群巡检、资源核对、故障初筛全部依赖该组命令，通过格式化输出、筛选排序，可以在数百个资源中精准定位异常资源，极大提升排查效率。

#### 实操示例
```bash
# 1. 基础资源查看
k get pods              # 查看当前命名空间所有Pod
k get deploy,svc        # 批量查看部署和服务资源
k describe pod nginx-pod # 查看Pod详细配置与异常事件
k logs nginx-pod        # 查看容器标准输出日志

# 2. 格式化输出
k get pods -o wide      # 展示扩展信息：节点、IP、启动时间
k get deploy nginx -o yaml # 输出资源完整yaml配置
k get pods -o jsonpath='{.items[*].metadata.name}' # 仅输出Pod名称

# 3. 筛选与排序
k get pods --selector app=nginx # 按标签筛选资源
k get pods --field-selector status.phase=Running # 筛选运行中Pod
k get pods --sort-by=.metadata.creationTimestamp # 按创建时间正序排序

# 4. 查看集群事件
k get events --sort-by=.metadata.creationTimestamp
```

#### 验证与排错
1. 筛选无结果：核对标签名、字段名大小写（K8s严格区分大小写）
2. Pod运行异常但get状态正常：必须通过describe查看底层事件
3. 容器无日志输出：Java应用需确保日志输出至**标准控制台**，而非仅写入磁盘文件

#### ⚠️避坑指南/💡最佳实践
1. 日常巡检优先 `k get -o wide`，快速筛查异常资源
2. **所有集群故障，优先查 events 事件**，80%调度、启动故障可直接定位
3. 禁止单纯使用grep筛选资源，优先使用官方`--selector`参数，精准稳定

#### 📌面试/高频场景题
**Q：get 和 describe 的核心区别和适用场景？**
A：get 轻量化、速度快，适合批量查看、日常巡检；describe 聚合完整资源信息、事件、调度日志，适合**资源异常深度排错**。

### 1.3 集群信息与调试命令
#### 概念定义
集群调试命令用于查看集群整体健康状态、节点资源负载，同时提供容器远程调试、端口转发、集群代理能力，是**线上服务调试专用工具**，无需修改业务配置即可排查问题。

#### 核心原理
`cluster-info` 读取控制面组件状态；`top` 命令依赖metrics-server组件采集节点和Pod的CPU、内存使用率；`exec` 通过apiserver打通容器终端；`port-forward` 在宿主机与Pod端口建立隧道，实现内网服务本地访问。

#### 场景与价值
生产集群内网服务无法对外暴露，通过端口转发可本地调试接口；节点负载过高时，通过top命令快速定位资源占用过高的Pod，解决集群卡顿、服务超时问题。

#### 实操示例
```bash
# 1. 集群整体信息
k cluster-info                  # 查看控制面组件地址与状态
k api-resources                 # 查看集群所有支持的资源类型
k api-versions                  # 查看集群API版本

# 2. 节点资源监控
k top nodes                     # 查看所有节点CPU、内存占用
k top pods -n prod              # 查看指定命名空间Pod资源占用
k describe node node-01         # 查看节点详细配置、污点、资源限制

# 3. 调试命令
k exec -it nginx-pod /bin/sh    # 交互式进入容器终端
k port-forward svc/nginx 8080:80 # 本地8080端口映射集群服务80端口
k proxy                         # 开启集群本地代理，访问集群API
```

#### 验证与排错
1. top命令无数据：metrics-server组件未部署或异常
2. port-forward连接失败：Pod服务端口未启动、端口配置错误
3. exec进入容器失败：容器内无shell环境（极简镜像）

#### ⚠️避坑指南/💡最佳实践
1. 禁止生产环境长期开启port-forward，仅用于临时调试
2. 极简镜像无法exec进入，优先使用临时busybox调试容器
3. 定期通过top命令巡检节点资源，提前规避资源溢出故障

#### 📌面试/高频场景题
**Q：kubectl port-forward 的使用场景是什么？**
A：用于集群内网服务临时调试，无需配置Ingress、无需暴露公网端口，将集群内部Service/Pod端口映射到本地，适合线上问题临时排查、接口测试。

## 2. 资源创建/查看/编辑/删除
资源的**创建、查看、编辑、删除**是 Kubernetes 最核心、最基础的运维操作，相当于 K8s 世界里的“增删改查”。不管是 Java 应用部署、微服务发布，还是日常故障排查，90% 的操作都围绕这 4 个动作展开。

本小节会彻底讲清楚**命令式 vs 声明式**两种资源管理方式、资源状态怎么看、如何安全修改配置、如何正确清理资源，完全贴合生产规范与面试高频考点。

### 2.1 资源创建的两种方式
#### 概念定义
K8s 资源创建分为**命令式创建**和**声明式创建**两种：
- **命令式**：直接敲命令告诉 K8s “现在创建一个资源”，属于即时操作。
- **声明式**：写好 YAML 文件，告诉 K8s “我要这个最终状态”，由 K8s 自动达成目标。

可以理解为：
命令式 = 现场点菜；声明式 = 提前写好菜单交给厨师。

#### 核心原理
- **命令式**：直接向 apiserver 发送创建请求，执行一次就结束，不记录历史配置。
- **声明式**：`kubectl apply` 会对比当前集群状态与 YAML 定义，**增量更新、自动修复、可追溯**，是 K8s 设计的核心思想。

#### 场景与价值
- 命令式：适合**测试、调试、临时创建**，速度快。
- 声明式：适合**生产环境、正式业务、CI/CD**，可版本管理、可回滚、可审计。

#### 实操示例
```bash
# ------------------------------
# 1. 命令式创建（快速测试用）
# ------------------------------
# 创建 Nginx Pod
kubectl run nginx-pod --image=nginx:1.23

# 创建 Deployment
kubectl create deploy nginx-deploy --image=nginx:1.23 --replicas=2

# 暴露服务（Service）
kubectl expose deploy nginx-deploy --port=80 --target-port=80 --name=nginx-svc

# ------------------------------
# 2. 声明式创建（生产标准）
# ------------------------------
# 编写 YAML 文件（nginx.yaml）
cat > nginx.yaml << EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deploy
spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.23
        ports:
        - containerPort: 80
EOF

# 应用配置（创建或更新）
kubectl apply -f nginx.yaml
```

#### 验证与排错
- 验证：`kubectl get deploy,pods` 看到资源状态正常即为成功。
- 报错：镜像拉取失败 → 检查镜像地址/权限；YAML 报错 → 检查缩进。

#### ⚠️避坑指南 / 💡最佳实践
- **生产绝对禁止用命令式管理正式业务**，无法追溯配置、无法回滚。
- 所有业务必须使用 **YAML + Git** 管理，这是企业规范。
- `apply` 可重复执行，是安全的幂等操作。

#### 📌面试/高频场景题
**Q：命令式和声明式的核心区别？生产为什么必须用声明式？**
A：命令式是单次操作，无状态留存；声明式是期望状态管理，支持版本控制、增量更新、自动化发布。生产用声明式可追溯、可回滚、适合 CI/CD。

**Q：kubectl create 和 kubectl apply 有什么区别？**
A：create 只能创建，已存在就报错；apply 可创建+更新，是幂等操作，生产优先使用 apply。

### 2.2 资源查看与状态解析
#### 概念定义
`kubectl get` 用于**快速查看资源状态**，`kubectl describe` 用于**查看详细信息+排错**，是运维最常用的两个命令。

#### 核心原理
- `get`：从 etcd 读取资源基础信息，轻量、快速。
- `describe`：聚合资源配置、事件、调度信息、关联对象，用于故障定位。

#### 场景与价值
- 日常巡检用 `get`。
- 服务异常、起不来、调度失败用 `describe`。

#### 实操示例
```bash
# ------------------------------
# 1. 基础查看
# ------------------------------
kubectl get pods                         # 查看 Pod
kubectl get deploy                       # 查看 Deployment
kubectl get svc                          # 查看服务
kubectl get pods -n default              # 指定命名空间

# ------------------------------
# 2. 详细查看
# ------------------------------
kubectl get pods -o wide                 # 显示 IP、节点位置
kubectl get deploy nginx-deploy -o yaml   # 导出完整配置

# ------------------------------
# 3. 描述与排错（核心）
# ------------------------------
kubectl describe pod nginx-deploy-xxx    # 查看事件、错误、调度信息

# ------------------------------
# 4. 过滤筛选（生产必备）
# ------------------------------
kubectl get pods -l app=nginx             # 按标签筛选
kubectl get pods --field-selector status.phase=Running  # 按状态筛选
```

#### 资源状态解析（高频）
| 资源       | 异常状态         | 含义                          |
| ---------- | ---------------- | ----------------------------- |
| Pod        | Pending          | 调度失败（资源不足/端口冲突） |
| Pod        | ImagePullBackOff | 镜像拉取失败                  |
| Pod        | CrashLoopBackOff | 容器反复崩溃                  |
| Deployment | 0/2              | 副本未正常启动                |

#### 验证与排错
- Pod 异常 → 先 `describe` 看事件，再 `logs` 看日志。
- 筛选不到资源 → 检查标签、命名空间是否正确。

#### ⚠️避坑指南 / 💡最佳实践
- 排错固定顺序：`get` → `describe` → `logs`。
- 批量管理必须用**标签筛选**，不要手动点名。

#### 📌面试/高频场景题
**Q：Pod 状态 Pending 代表什么？怎么排查？**
A：代表 Pod 没有被调度到节点。排查：`kubectl describe pod` 看事件，通常是资源不足、污点容忍、存储/网络依赖不满足。

### 2.3 资源编辑与更新
#### 概念定义
资源编辑用于**修改运行中资源的配置**，包括在线编辑、补丁更新、强制替换。

#### 核心原理
- `edit`：直接打开编辑器修改配置，apiserver 实时生效。
- `patch`：只修改某一个字段，轻量、安全。
- `replace`：全量覆盖配置，风险高。

#### 场景与价值
- 临时调整参数用 `edit`。
- 自动化脚本修改用 `patch`。
- 生产尽量用 **YAML + apply**，最安全。

#### 实操示例
```bash
# ------------------------------
# 1. 在线编辑（临时调试）
# ------------------------------
kubectl edit deploy nginx-deploy

# ------------------------------
# 2. Patch 补丁更新（推荐）
# ------------------------------
# 修改副本数为 3
kubectl patch deploy nginx-deploy -p '{"spec":{"replicas":3}}'

# ------------------------------
# 3. Replace 强制替换
# ------------------------------
# 先导出 YAML → 修改 → 替换
kubectl get deploy nginx-deploy -o yaml > deploy.yaml
kubectl replace -f deploy.yaml
```

#### 验证与排错
- 修改后用 `kubectl get deploy` 查看副本数/配置是否生效。
- edit 保存报错 → YAML 格式错误，必须严格缩进。

#### ⚠️避坑指南 / 💡最佳实践
- **生产禁止频繁使用 edit**，容易造成配置混乱。
- 优先使用 **YAML 文件 + apply**，可追溯、可回滚。
- `replace` 会删除重建 Pod，可能造成业务中断。

#### 📌面试/高频场景题
**Q：kubectl patch 三种类型是什么？区别？**
A：json、merge、strategic。strategic 是 K8s 默认，支持数组合并，最常用；json 严格按路径覆盖；merge 简单合并。

### 2.4 资源删除与清理
#### 概念定义
删除用于**销毁不再使用的资源**，支持单个删除、批量删除、级联删除。

#### 核心原理
删除指令发送给 apiserver，资源进入 Terminating 状态，K8s 会优雅终止容器并清理依赖。

#### 场景与价值
- 测试环境清理临时资源。
- 业务下线、版本替换、故障重建。

#### 实操示例
```bash
# ------------------------------
# 1. 基础删除
# ------------------------------
kubectl delete pod nginx-deploy-xxx                 # 删除单个 Pod
kubectl delete -f nginx.yaml                        # 按 YAML 删除
kubectl delete deploy nginx-deploy                  # 删除控制器

# ------------------------------
# 2. 批量删除（生产常用）
# ------------------------------
kubectl delete pods -l app=nginx                    # 按标签删除
kubectl delete all --all -n test                   # 清空命名空间（谨慎）

# ------------------------------
# 3. 级联删除
# ------------------------------
# 默认级联删除 Pod
kubectl delete deploy nginx-deploy --cascade=true
```

#### 验证与排错
- 删除后 `get` 不到资源即为删除完成。
- 资源一直 Terminating →  finalizer 阻塞，强制删除：`--force --grace-period=0`。

#### ⚠️避坑指南 / 💡最佳实践
- **生产删除前必须确认命名空间、标签、名称**。
- 不要直接删 Pod，应该删除 Deployment 来统一管理。
- 批量删除一定要先 `get` 确认再执行。

#### 📌面试/高频场景题
**Q：什么是级联删除？前台删除 vs 后台删除区别？**
A：级联删除指删除父资源（如 Deployment）时自动清理子资源（Pod）。前台删除：等待子资源删完再删父；后台删除：直接删父，后台清理子。

### 核心要点速记
1. **生产必须用声明式 YAML + apply**，命令式只用于测试。
2. 排错固定三板斧：`get` → `describe` → `logs`。
3. 在线编辑 `edit` 只适合临时调试，不要用于生产业务。
4. 删除资源优先按**标签/YAML**删除，避免误删。
5. 所有操作尽量带上命名空间 `-n`，避免跨环境干扰。

### 生产最佳实践
- 资源配置一律使用 **YAML + Git 管理**。
- 发布更新一律使用 `kubectl apply -f`。
- 禁止手动在线修改关键服务。
- 删除前先查看，批量操作必筛选。

### 面试高频考点
- 命令式 vs 声明式
- create vs apply
- patch 用法与场景
- Pod 异常状态排查流程
- 级联删除机制

## 3. Pod运维：进入容器、日志查看、文件拷贝、调试
Pod 是 Kubernetes 最小调度单元，所有业务应用最终都运行在 Pod 容器内部。对于 Java 开发与运维人员来说，**90% 的线上应用故障、服务报错、接口异常、启动失败**，都需要通过 Pod 调试手段定位解决。

本小节作为高频核心运维技能，完整讲解**进入容器操作、日志分析、文件拷贝、Pod 全状态排错**四大核心能力。严格贴合生产落地场景，同时覆盖全部面试高频考点，帮助学习者掌握 K8s 应用故障排查标准流程，彻底解决容器应用看不见、查不到、调不通的问题。

### 3.1 进入容器与交互式操作
#### 概念定义
`kubectl exec` 是 Kubernetes 专属的容器远程操作命令，类比服务器的 `ssh` 命令。简单来说：**服务器用 SSH 登录系统，容器用 exec 进入容器**。

该命令支持两种操作模式：
1. **非交互式操作**：执行单次命令，执行完毕直接退出容器。
2. **交互式操作**：进入容器终端，持续手动输入命令调试。

同时支持**多容器 Pod 指定容器进入**，以及**临时调试容器**方案，专门解决极简镜像无终端、无命令工具的调试难题。

#### 核心原理
K8s 无法直接远程连接容器，整体链路如下：
1. 用户执行 `kubectl exec` 命令，请求发送至 **API Server**；
2. API Server 校验权限后，转发请求到 Pod 所在节点的 **kubelet**；
3. kubelet 调用容器运行时（containerd/docker），在容器内创建进程执行对应命令；
4. 建立双向数据流，将容器终端输出回传给本地客户端。

极简镜像（alpine、scratch）默认无 shell、无工具，无法直接 exec 调试，因此需要**独立临时调试容器**实现排错。

#### 场景与价值
1. 线上应急调试：进入容器核对配置文件、环境变量、目录结构；
2. 网络连通测试：容器内测试端口、DNS、服务连通性；
3. 极简镜像排错：解决基础镜像无终端无法调试的问题；
4. 多容器架构调试：精准进入 sidecar 辅助容器排查问题。

#### 实操示例
```bash
# 提前创建测试Pod
kubectl run java-demo --image=nginx:1.23

# ------------------------------
# 1. 非交互式操作：单次执行命令，执行即退出
# 适用：快速查询文件、环境变量，无需持续操作
# ------------------------------
# 查看容器根目录文件
kubectl exec java-demo -- ls /
# 查看容器环境变量（Java应用常用，核对配置环境变量是否注入成功）
kubectl exec java-demo -- env

# ------------------------------
# 2. 交互式操作：进入容器终端
# -i：保持标准输入打开，持续接收输入
# -t：分配伪终端，实现终端交互体验
# ------------------------------
kubectl exec -it java-demo -- /bin/bash

# ------------------------------
# 3. 多容器Pod 指定容器进入（-c 参数）
# 场景：Pod包含业务容器+日志sidecar容器
# ------------------------------
# 格式：kubectl exec -it Pod名称 -c 容器名 -- 终端
kubectl exec -it java-demo -c sidecar-log -- /bin/sh

# ------------------------------
# 4. 临时调试容器（生产万能调试方案）
# --rm：退出自动删除容器，不残留垃圾资源
# -it：交互式终端
# ------------------------------
kubectl run -it --rm debug-tool --image=busybox --restart=Never -- /bin/sh
```

#### 验证与排错
##### 验证方式
1. 非交互式：终端直接输出命令执行结果，无报错即为成功；
2. 交互式：终端切换为容器内终端，可自由执行 `ls`、`ping` 等命令；
3. 临时容器：成功进入 busybox 终端，退出后容器自动销毁。

##### 常见报错
1. **报错：container not found**
   - 原因：Pod 未就绪、容器崩溃退出、Pod名称输入错误
   - 排查：`kubectl get pods` 核对 Pod 运行状态
2. **报错：no such file or directory**
   - 原因：极简 alpine 镜像无 `/bin/bash`，仅支持 `/bin/sh`
   - 解决方案：将 `/bin/bash` 替换为 `/bin/sh`

#### ⚠️避坑指南 / 💡最佳实践
1. **禁止生产环境通过 exec 修改容器内文件**：容器文件系统是临时的，Pod 重启后所有修改全部丢失，极易造成配置不一致；
2. 极简镜像优先使用**临时 busybox 调试容器**，不要强行安装工具；
3. 多容器 Pod 调试必须加 `-c` 指定容器，否则默认进入第一个容器；
4. 所有线上调试操作尽量临时、可追溯，禁止长期驻留容器终端。

#### 📌面试/高频场景题
**Q：Java 应用镜像使用极简 alpine 构建，没有 bash、没有任何调试工具，如何排查容器问题？**
A：无法直接 exec 进入容器调试，解决方案有两种：
1. 使用 `kubectl run` 启动 busybox 临时调试容器，接入集群网络测试连通性、DNS、服务访问；
2. 使用 K8s 临时容器（ephemeral containers）共享目标 Pod 网络命名空间，精准调试。

**Q：kubectl exec -it 参数分别代表什么？缺一不可吗？**
A：`-i` 持续保留输入流，`-t` 分配伪终端；单独 `-i` 无终端界面，单独 `-t` 无法输入指令，**交互式操作必须两个参数同时使用**。

### 3.2 日志查看与分析
#### 概念定义
`kubectl logs` 是 K8s 专属日志查询命令，用于读取容器**标准输出(stdout)、标准错误(stderr)** 日志。类比 Java 后端查看服务器 `catalina.out`、项目日志文件。

容器化规范要求：**所有业务日志必须输出到控制台，禁止写入容器本地文件**，该命令是线上排查 Java 空指针、接口报错、启动异常的核心手段。

#### 核心原理
1. 容器运行时（containerd）持续采集容器控制台输出；
2. 日志文件默认存储在节点本地磁盘；
3. `kubectl logs` 调用 kubelet，读取节点本地容器日志并回显；
4. K8s 不会永久存储日志，节点重启、容器重建日志会丢失，因此生产需要搭配日志采集系统。

#### 场景与价值
1. 排查 Java 应用**启动失败、端口占用、配置错误、空指针异常**；
2. 实时监控线上接口请求、报错日志；
3. 追溯**已经崩溃退出**的容器报错日志；
4. 统计日志时间、筛选历史报错，定位故障时间点。

#### 实操示例
```bash
# ------------------------------
# 1. 基础日志查看：输出容器全部日志
# ------------------------------
kubectl logs java-demo

# ------------------------------
# 2. 实时流式日志（重点！生产高频）
# -f：持续监听日志输出，等同于 tail -f
# ------------------------------
kubectl logs -f java-demo

# ------------------------------
# 3. 查看已终止/崩溃容器日志（-p）
# 场景：容器 CrashLoopBackOff 反复重启，查看上一次崩溃日志
# ------------------------------
kubectl logs -p java-demo

# ------------------------------
# 4. 日志精准筛选参数
# --tail=100：只查看最新100行日志
# --since=10m：查看最近10分钟日志
# --timestamps：日志自带时间戳，精准定位故障时间
# ------------------------------
kubectl logs --tail=100 java-demo
kubectl logs --since=1h java-demo
kubectl logs --timestamps -f java-demo

# ------------------------------
# 5. 多容器Pod 指定容器日志
# ------------------------------
kubectl logs -f java-demo -c sidecar-log
```

#### 验证与排错
##### 验证方式
执行命令后正常输出业务日志、报错堆栈，即为生效。

##### 常见故障
1. **日志无输出**
   - 原因：Java 应用日志写入本地文件，未输出控制台；日志级别过高
   - 解决方案：修改日志框架（logback/log4j2），输出日志到 stdout
2. **节点磁盘爆满**
   - 原因：容器日志无轮转策略，日志文件无限增大
   - 解决方案：配置容器日志最大大小、最大文件数，自动清理过期日志
3. **-p 参数无日志**
   - 原因：容器正常退出，无异常日志，或旧容器日志已被清理

#### ⚠️避坑指南 / 💡最佳实践
1. **云原生规范：Java 应用必须控制台输出日志**，禁止日志落地容器本地文件；
2. 单机 `kubectl logs` 仅用于临时排错，**生产必须部署日志采集系统（ELK/Loki）**；
3. 必须配置容器日志轮转，防止日志打满节点磁盘导致集群瘫痪；
4. 容器重启后本地日志丢失，重要报错务必及时留存。

#### 📌面试/高频场景题
**Q：Pod 处于 CrashLoopBackOff 状态，如何查看崩溃原因？**
A：1. 使用 `kubectl logs` 查看当前容器日志；2. 使用 `kubectl logs -p` 查看上一次崩溃容器的日志；3. 结合 `kubectl describe pod` 查看容器退出码，定位启动异常、配置缺失、代码报错等问题。

**Q：为什么 Java 容器日志写入文件后，kubectl logs 查不到日志？**
A：kubectl logs 仅采集容器**标准输出流**，不会读取容器本地文件日志。云原生应用必须适配控制台输出日志。

### 3.3 容器文件拷贝与数据交换
#### 概念定义
`kubectl cp` 是 K8s 文件传输命令，用于**宿主机 ↔ 容器**双向文件拷贝。类比服务器的 `scp` 命令，专门用于应急场景下的文件上传、下载。

该命令仅用于**临时应急调试**，绝对不用于生产常规配置更新和数据持久化。

#### 核心原理
1. `kubectl cp` 底层依赖 `tar` 打包解压机制；
2. 拷贝文件时，先在源端打包为 tar 流，通过 API Server 传输，目标端解压；
3. 因此**容器内部必须安装 tar 工具**，否则拷贝失败；
4. 仅支持宿主机和容器互传，**原生不支持容器和容器直接互传**。

#### 场景与价值
1. 应急导出：容器内异常日志文件、缓存文件、配置文件，用于本地分析；
2. 临时调试：快速上传测试配置文件、脚本到容器；
3. 数据备份：临时导出容器内生成的业务数据。

#### 实操示例
```bash
# ------------------------------
# 1. 容器文件下载到宿主机
# 格式：kubectl cp Pod名称:容器文件路径 宿主机本地路径
# ------------------------------
kubectl cp java-demo:/etc/nginx/nginx.conf ./nginx-local.conf

# ------------------------------
# 2. 宿主机文件上传到容器
# 格式：kubectl cp 宿主机本地文件 Pod名称:容器目录
# ------------------------------
kubectl cp ./nginx-local.conf java-demo:/etc/nginx/

# ------------------------------
# 3. 多容器Pod 指定容器文件拷贝
# -c 指定目标容器
# ------------------------------
kubectl cp ./test.properties java-demo:/app/config/ -c java-business

# ------------------------------
# 4. 跨Pod文件拷贝（间接方案）
# 先下载到本地，再上传至目标Pod
# ------------------------------
kubectl cp pod-a:/app/log/error.log ./error.log
kubectl cp ./error.log pod-b:/app/log/
```

#### 验证与排错
##### 验证方式
1. 下载：宿主机目录可查看到对应文件；
2. 上传：进入容器，可查看到新增文件。

##### 常见报错
1. **tar: command not found**
   - 原因：极简容器无 tar 工具，cp 命令依赖 tar 打包，直接失败
   - 解决方案：使用 `kubectl exec + cat` 重定向实现文件传输
2. **文件权限不足**
   - 原因：容器内目录只读、权限不足
   - 解决方案：检查目录读写权限，优先挂载存储卷

#### ⚠️避坑指南 / 💡最佳实践
1. **严禁通过 kubectl cp 维护生产配置**：容器文件系统临时，Pod 重启配置全部丢失；
2. 所有正式配置统一使用 ConfigMap/Secret 挂载，不要手动上传文件；
3. 容器之间禁止直接文件传输，必须通过存储卷共享数据；
4. cp 仅作为应急调试手段，不可纳入常态化运维流程。

#### 📌面试/高频场景题
**Q：kubectl cp 的底层原理是什么？容器没有 tar 工具如何拷贝文件？**
A：底层基于 tar 打包流传输文件。若无 tar 工具，放弃 cp 命令，使用 `kubectl exec pod -- cat 容器文件 > 本地文件` 实现文件下载。

**Q：能否直接实现两个Pod之间文件互传？为什么？**
A：原生不支持。kubectl cp 仅支持宿主机与容器互传，跨Pod需要本地中转，生产推荐使用共享存储卷实现数据互通。

### 3.4 Pod 状态与问题排查
#### 概念定义
Pod 是 K8s 最小运行单元，所有业务故障都会体现在 Pod 状态上。本小节详解生产最常见的 **Pending、Running、CrashLoopBackOff、ImagePullBackOff** 四大核心状态，同时覆盖探针异常、资源不足、网络异常等高频故障。

帮助学习者实现：**看到Pod状态，直接定位故障根源**。

#### 核心原理
Pod 完整生命周期状态由 kubelet 持续监控，控制器根据状态自动调度、重启、驱逐Pod：
1. **Pending**：资源调度阶段，未完成调度、镜像拉取；
2. **Running**：调度完成，容器正常运行；
3. **ImagePullBackOff**：镜像拉取失败，持续重试；
4. **CrashLoopBackOff**：容器启动后立刻退出，kubelet 循环重启容器；
5. 探针异常：就绪探针/存活探针失败，导致流量摘除、容器重启。

#### 场景与价值
覆盖生产 95% 以上的应用故障场景，解决 Java 应用**启动失败、服务不可用、频繁重启、无法访问**等核心问题，是运维排查故障的标准流程。

#### 实操示例
```bash
# ------------------------------
# 1. 通用排错命令组合（生产标准）
# ------------------------------
# 查看Pod整体状态
kubectl get pods -o wide
# 查看Pod详细事件、调度、报错信息
kubectl describe pod java-demo
# 查看当前容器报错日志
kubectl logs java-demo
# 查看崩溃重启历史日志
kubectl logs -p java-demo

# ------------------------------
# 2. 探针故障排查
# ------------------------------
# 查看探针失败详细事件
kubectl describe pod java-demo | grep -i probe

# ------------------------------
# 3. 资源不足排查
# ------------------------------
# 查看节点资源占用
kubectl top nodes
# 查看Pod资源占用
kubectl top pods
```

#### 各类状态故障解析与排错
1. **Pending（调度失败）**
   - 原因：节点CPU/内存不足、节点污点、Pod亲和性限制、存储卷挂载失败
   - 排查：`kubectl describe` 查看 Events 调度失败日志

2. **ImagePullBackOff（镜像拉取失败）**
   - 原因：镜像地址错误、标签不存在、私有镜像无密钥、网络无法访问镜像仓库
   - 排查：核对镜像名称、版本，检查 imagePullSecret 配置

3. **CrashLoopBackOff（容器反复崩溃）**
   - 原因：Java 配置错误、端口占用、代码报错、环境变量缺失、权限不足
   - 排查：优先查看容器日志，获取 Java 异常堆栈

4. **Running但服务不可用**
   - 原因：就绪探针失败、应用未启动完成、网络策略拦截、Service 端点为空
   - 排查：检查探针配置、Endpoint 端点列表、网络策略

#### ⚠️避坑指南 / 💡最佳实践
1. **所有Java业务Pod必须配置存活探针+就绪探针**，自动剔除异常实例，保证服务高可用；
2. Pod 异常排查**固定顺序**：查状态 → 查事件(describe) → 查日志(logs)；
3. 不要盲目重启Pod，优先定位根因，避免掩盖故障日志；
4. 生产禁止Pod资源请求(requests)和限制(limits)为空，防止单Pod占满节点资源。

#### 📌面试/高频场景题
**Q：Pod 显示 Running，但是服务无法访问，可能是什么原因？**
A：1. 就绪探针失败，Pod 未加入 Service 端点，无流量接入；2. 应用内部启动失败，进程存活但服务未启动；3. 网络策略拦截流量；4. Service 标签选择器不匹配。

**Q：简述 CrashLoopBackOff 和 ImagePullBackOff 的区别和排查思路？**
A：ImagePullBackOff 是**启动前置故障**，镜像拉取失败；CrashLoopBackOff 是**启动后置故障**，镜像正常拉取，但应用启动后异常退出。前者排查镜像仓库，后者排查应用代码、配置、环境。

### 核心知识点汇总
1. **容器交互**：`kubectl exec` 实现容器登录，极简镜像使用 busybox 临时容器调试，是生产通用排错方案；
2. **日志运维**：`logs -f` 实时日志、`logs -p` 历史崩溃日志，是排查 Java 应用报错的核心手段，云原生应用必须控制台输出日志；
3. **文件拷贝**：`kubectl cp` 仅用于应急调试，底层依赖 tar，**禁止用于生产配置管理**；
4. **Pod故障排查**：熟练掌握四大Pod状态、探针异常、资源不足、网络故障的完整排查链路。

### 生产落地规范
1. 所有调试操作仅临时使用，业务配置统一通过 YAML、ConfigMap、Secret 管理；
2. 强制配置日志轮转，搭建集中日志采集平台，避免节点磁盘爆满；
3. 所有业务Pod配置资源限制、健康探针，实现故障自动自愈；
4. 严格遵循「状态→事件→日志」的标准排错流程。

### 面试高频考点
1. exec 交互式参数作用、极简镜像调试方案
2. logs `-p` 参数使用场景
3. kubectl cp 底层原理与局限性
4. 四大Pod异常状态根因与排查流程
5. Pod Running 但服务不可用的全部原因

## 4. 发布运维：滚动发布、灰度发布、暂停/恢复发布、版本回滚
在 Kubernetes 中，**应用发布与版本管理**是保障业务连续性的核心运维能力。Deployment 作为最常用的工作负载资源，提供了完善的发布控制能力，包括**滚动发布、灰度发布、发布暂停/恢复、一键版本回滚**四大核心能力。

本小节面向 Java 开发与运维人员，从配置原理、实操命令、流量机制、故障排查、生产规范全维度讲解，彻底掌握**无停机发布、无损回滚、安全灰度**的生产级发布流程，同时覆盖面试高频考点，实现发布零事故、故障秒回滚。

### 4.1 Deployment 发布策略与配置
#### 概念定义
Deployment 的 `strategy` 是定义**应用如何更新**的核心配置，决定了新版本替换旧版本的方式，K8s 提供两种策略：
- **RollingUpdate（滚动更新）**：逐步替换 Pod，生产环境默认、推荐策略
- **Recreate（重建更新）**：先删除全部旧 Pod，再创建新 Pod，会导致服务停机

#### 核心原理
1. Deployment 管理 ReplicaSet（版本控制器），每次发布都会生成新的 ReplicaSet；
2. **滚动更新**通过 `maxSurge` 和 `maxUnavailable` 控制发布速度，保证发布过程中始终有可用实例；
3. `maxSurge`：发布期间**最多超出**期望副本数的 Pod 数量
4. `maxUnavailable`：发布期间**最多不可用**的 Pod 数量
5. `kubectl rollout status` 实时监听发布流程，判断是否发布成功

#### 场景与价值
- 滚动更新：**生产环境核心业务**，零停机、无损发布、用户无感知
- 重建更新：测试环境、非核心服务、应用不支持多版本共存场景
- 精细化参数配置：避免发布瞬间流量波动、资源打满、服务雪崩

#### 实操示例
```yaml
# Java 应用 Deployment 完整发布策略配置
apiVersion: apps/v1
kind: Deployment
metadata:
  name: java-app
spec:
  replicas: 4  # 期望副本数
  selector:
    matchLabels:
      app: java-app
  # 核心：发布策略配置
  strategy:
    rollingUpdate:
      maxSurge: 25%        # 最多额外启动25%的Pod（4个→最多5个）
      maxUnavailable: 25%  # 最多不可用25%的Pod（最少保持3个可用）
    type: RollingUpdate    # 滚动更新策略（生产标配）
  template:
    metadata:
      labels:
        app: java-app
    spec:
      containers:
      - name: java-app
        image: java-app:v1
        ports:
        - containerPort: 8080
```
```bash
# 应用配置
kubectl apply -f java-deploy.yaml

# 监控发布状态（核心命令）
kubectl rollout status deployment java-app
```

#### 验证与排错
- 验证：命令行提示 `successfully rolled out` 代表发布完成
- 报错：发布卡住 → 检查镜像拉取、资源不足、健康检查失败
- 查看副本变化：`kubectl get pods -w` 实时监控新旧 Pod 切换

#### ⚠️避坑指南 / 💡最佳实践
- **生产环境严禁使用 Recreate 策略**，会直接导致服务中断
- `maxSurge` 和 `maxUnavailable` 推荐使用百分比，适配动态副本数
- 发布前必须配置健康探针，防止异常实例接管流量
- 发布过程必须使用 `rollout status` 监控，不可盲目执行

#### 📌面试/高频场景题
**Q：滚动更新中 maxSurge 和 maxUnavailable 同时配置会有什么效果？**
A：两个参数共同控制发布粒度，保证发布过程中服务可用性。例如25%/25%，4副本场景下，最少3个可用、最多5个运行，实现平滑无损发布。

**Q：RollingUpdate 和 Recreate 核心区别是什么？生产为什么必须用前者？**
A：Recreate 先删后建，服务停机；RollingUpdate 逐步替换，零停机。生产业务要求7×24小时可用，因此必须使用滚动更新。

### 4.2 滚动发布全流程
#### 概念定义
滚动发布是 K8s 标准的**无停机更新方式**，通过逐步启动新 Pod、逐步销毁旧 Pod，实现业务版本平滑切换。

#### 核心原理
1. Deployment 创建**新 ReplicaSet**，根据 `maxSurge` 启动新 Pod；
2. 新 Pod 通过就绪探针后，标记为可用，接入 Service 流量；
3. 逐步缩容**旧 ReplicaSet**，销毁旧 Pod，摘除流量；
4. 循环执行，直到新 ReplicaSet 达到期望副本数，发布完成；
5. 整个过程 Service 后端始终有可用实例，用户无感知。

#### 场景与价值
- Java 微服务、网关、核心应用**版本迭代**
- 配置更新、JVM 参数调整、镜像升级
- 高可用要求的生产环境，禁止任何业务中断

#### 实操示例
```bash
# 1. 发布Java应用v2版本（修改镜像标签）
kubectl set image deployment java-app java-app=java-app:v2

# 2. 实时监控发布过程
kubectl rollout status deployment java-app

# 3. 查看Pod变化（-w 实时监听）
kubectl get pods -w -l app=java-app

# 4. 查看当前ReplicaSet版本（新旧版本共存）
kubectl get rs
```

#### 验证与排错
- 验证：所有流量切换到新 Pod，旧 Pod 全部销毁
- 发布中断：新 Pod 启动失败、镜像拉取失败、健康检查不通过
- 恢复方式：立即执行**版本回滚**，快速恢复服务可用

#### ⚠️避坑指南 / 💡最佳实践
- 发布前先在测试环境验证镜像可用性
- 发布过程禁止手动删除 Pod、修改副本数
- 发布完成后验证业务接口、日志无异常
- 微服务架构建议分批发布，避免全线更新

#### 📌面试/高频场景题
**Q：滚动发布过程中，K8s 如何保证流量不丢失？**
A：1. 只有通过就绪探针的新 Pod 才会接入流量；2. 旧 Pod 会执行优雅终止，处理完现有请求再销毁；3. Service 自动更新后端端点，实现流量无损切换。

**Q：滚动发布时，如果新镜像启动失败，会发生什么？**
A：Deployment 会卡住发布，保留部分旧 Pod 提供服务，不会全部替换，保证业务不中断，可手动触发回滚恢复。

### 4.3 灰度/金丝雀发布
#### 概念定义
灰度发布（金丝雀发布）：**只让少量用户/流量访问新版本**，验证无问题后再全量发布，是生产环境最高安全级别的发布方式。

#### 核心原理
1. 基础灰度：通过**多 Deployment + 相同标签**实现新旧版本同时运行，按副本数分配流量；
2. 高级灰度：通过 Ingress-Nginx 配置流量权重，精准控制新版本流量比例；
3. 流量由 Service 统一负载，按 Pod 数量加权分发

#### 场景与价值
- 核心业务重大版本更新、架构重构、框架升级
- 新功能小流量验证，避免全量发布导致大规模故障
- 金融、支付、电商等高要求业务的标准发布流程

#### 实操示例
##### 方式1：基于副本数的简单灰度（生产常用）
```bash
# 1. 部署v1版本（正常服务）
kubectl create deploy java-app-v1 --image=java-app:v1

# 2. 部署v2版本（金丝雀，少量副本）
kubectl create deploy java-app-v2 --image=java-app:v2 --replicas=1

# 3. 两个Deployment绑定同一个Service（相同标签）
kubectl expose deploy java-app-v1 --port=8080 --name=java-app-svc
```

##### 方式2：基于 Nginx Ingress 的权重灰度
```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: java-app-ingress
  annotations:
    nginx.ingress.kubernetes.io/canary: "true"
    nginx.ingress.kubernetes.io/canary-weight: "10"  # 10%流量到v2
spec:
  rules:
  - host: demo.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: java-app-svc-v2
            port:
              number: 8080
```

#### 验证与排错
- 验证：压测/访问接口，10%流量命中 v2 版本
- 异常：立即删除金丝雀 Deployment/关闭 Ingress 灰度，恢复全量 v1 流量

#### ⚠️避坑指南 / 💡最佳实践
- 灰度时间不宜过长，验证完成后立即全量发布或回滚
- 灰度期间重点监控错误率、响应时间、JVM 监控
- 禁止同时运行超过2个版本，避免架构混乱
- 大型集群推荐使用 Argo Rollouts 专业灰度工具

#### 📌面试/高频场景题
**Q：K8s 原生支持金丝雀发布吗？如何实现？**
A：原生无直接控制器，但可通过两种方式实现：1. 多 Deployment 共享 Service，按副本数灰度；2. Ingress-Nginx 配置 canary 权重。企业级可使用 Argo Rollouts。

**Q：灰度发布和滚动发布的核心区别是什么？**
A：滚动发布是全量逐步替换；灰度发布是保留旧版本，仅小流量引入新版本，安全性更高，适合核心业务重大更新。

### 4.4 发布暂停、恢复与版本回滚
#### 概念定义
- **暂停发布**：中断正在进行的滚动更新，冻结当前状态
- **恢复发布**：继续执行中断的发布流程
- **版本回滚**：一键恢复到上一个/指定历史版本，是线上故障**最快恢复手段**

#### 核心原理
1. Deployment 会保留历史 ReplicaSet（默认保留10个），存储所有版本配置；
2. `pause` 暂停控制器协调，停止扩缩容 Pod；
3. `resume` 恢复控制器工作，继续发布；
4. `undo` 切换回历史 ReplicaSet，自动启动旧版本 Pod，销毁新版本 Pod

#### 场景与价值
- 发布过程发现异常，**立即暂停**，防止故障扩大
- 新版本验证失败，**一键回滚**，秒级恢复服务
- 发布中途需要调试、核对配置，使用暂停功能

#### 实操示例
```bash
# ------------------------------
# 1. 发布暂停/恢复
# ------------------------------
# 暂停发布
kubectl rollout pause deployment java-app

# 恢复发布
kubectl rollout resume deployment java-app

# ------------------------------
# 2. 查看版本历史（核心）
# ------------------------------
kubectl rollout history deployment java-app

# ------------------------------
# 3. 版本回滚（生产救命命令）
# ------------------------------
# 回滚到上一个版本
kubectl rollout undo deployment java-app

# 回滚到指定版本
kubectl rollout undo deployment java-app --to-revision=1
```

#### 验证与排错
- 验证：回滚后 Pod 全部切换为旧版本，业务恢复正常
- 回滚失败：历史 ReplicaSet 被清理、镜像已删除、配置丢失
- 解决方案：重新部署旧版本 YAML 文件

#### ⚠️避坑指南 / 💡最佳实践
- **生产必须保留历史版本**，不要清理 ReplicaSet
- 回滚后必须验证业务、日志、监控指标
- 发布完成无问题后，再清理历史版本
- 回滚是故障恢复手段，不是常规发布方式

#### 📌面试/高频场景题
**Q：kubectl rollout undo 回滚的原理是什么？依赖什么？**
A：原理是切换 Deployment 管理的 ReplicaSet，重新启用旧版本控制器。依赖 K8s 保留的历史 ReplicaSet，默认保留10个版本。

**Q：发布过程中为什么要使用 pause 暂停？有什么生产价值？**
A：发布时发现异常日志、报错、监控告警，立即暂停可防止更多异常实例启动，缩小故障范围，为排查问题争取时间，是发布防护的重要手段。

### 核心知识点汇总
1. **发布策略**：生产强制使用 `RollingUpdate`，`maxSurge/maxUnavailable` 控制发布粒度，保证零停机；
2. **滚动发布**：逐步替换新旧 Pod，Service 自动调度流量，用户无感知；
3. **灰度发布**：多 Deployment/Ingress 权重实现小流量验证，是核心业务最高安全发布方案；
4. **发布控制**：pause 暂停、resume 恢复、undo 回滚，构成完整的发布防护体系；
5. **回滚机制**：基于历史 ReplicaSet 实现，是线上故障最快恢复手段。

### 生产落地规范
1. 所有 Java 应用必须使用滚动更新策略，禁用重建策略；
2. 重大版本必须走灰度发布，禁止直接全量更新；
3. 发布必监控，异常必暂停，故障必回滚；
4. 保留历史版本，禁止手动清理 ReplicaSet；
5. 发布完成后必须进行业务验证、监控检查。

### 面试高频考点
1. RollingUpdate 与 Recreate 区别及生产选型
2. maxSurge / maxUnavailable 作用与流量保障机制
3. 灰度发布的实现方式与适用场景
4. 版本回滚原理与生产故障恢复流程
5. 滚动发布的流量无损切换原理

## 5. 配置运维：ConfigMap热更新、Secret更新策略
在传统 Java 开发中，我们习惯使用 `application.yml`、`properties` 存放配置；而在 **K8s 云原生架构** 中，绝对禁止将业务配置硬编码打包进镜像。K8s 提供了 **ConfigMap（明文配置）**、**Secret（密文配置）** 两类核心资源，专门实现**配置与镜像解耦**。

配置运维是生产高频操作，绝大多数 Java 微服务参数修改、开关变更、密钥更新都依赖 ConfigMap 和 Secret。本章重点讲解配置创建、注入方式、**热更新底层机制**、更新失效问题、安全规范，同时解决面试高频的「为什么挂载文件能热更新，环境变量不能更新」等核心问题，完全贴合生产落地标准。

### 5.1 ConfigMap 管理与使用
#### 概念定义
**ConfigMap** 是 K8s 用于存储**非敏感、明文配置**的资源，可以类比 Java 项目的配置文件。它专门存放应用配置、环境参数、启动参数，做到**镜像无配置、配置外部化**，无需重新构建镜像即可修改业务参数。

ConfigMap 支持三种创建形式：键值对、单个配置文件、整个目录；支持三种注入方式：环境变量、启动命令参数、容器文件挂载，适配所有 Java 微服务配置场景。

#### 核心原理
1. ConfigMap 资源存储在集群 etcd 中，全局独立管理；
2. Pod 通过关联 ConfigMap 对象，将集群配置映射到容器内部；
3. 不同 Pod/Deployment 可复用同一个 ConfigMap，实现**配置统一管理**；
4. K8s 设计理念：**镜像只负责程序，配置全部外置**，实现镜像一次构建、多环境复用。

#### 场景与价值
1. Java 微服务 `application.yml`、日志配置、线程池参数、功能开关外置；
2. 多副本 Pod 统一配置，无需逐个修改容器配置；
3. 开发、测试、生产多环境配置隔离，镜像完全一致；
4. 配置修改无需打包镜像、无需重构代码，极大提升迭代效率。

#### 实操示例
##### 1. 键值对方式创建 ConfigMap
```bash
# 直接命令行创建键值对配置（简单参数）
kubectl create configmap java-env-config \
--from-literal=spring.profiles.active=prod \
--from-literal=server.port=8080
```

##### 2. 通过配置文件创建 ConfigMap（Java项目最常用）
```bash
# 编写Java应用配置文件 application-prod.yml
cat > application-prod.yml << EOF
spring:
  profiles:
    active: prod
  datasource:
    url: jdbc:mysql://mysql:3306/test
    username: root
logging:
  level:
    root: INFO
EOF

# 通过配置文件创建ConfigMap
kubectl create configmap java-file-config --from-file=application-prod.yml
```

##### 3. 通过目录创建 ConfigMap
```bash
# 将当前目录所有配置文件统一封装为ConfigMap
kubectl create configmap java-dir-config --from-file=./
```

##### 4. Deployment 注入 ConfigMap 完整配置
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: java-config-demo
spec:
  replicas: 2
  selector:
    matchLabels:
      app: java-config-demo
  template:
    metadata:
      labels:
        app: java-config-demo
    spec:
      containers:
      - name: java-app
        image: nginx:1.23
        # 方式1：环境变量注入配置
        env:
        - name: SPRING_PROFILES_ACTIVE
          valueFrom:
            configMapKeyRef:
              name: java-env-config # 关联configmap名称
              key: spring.profiles.active
        # 方式2：挂载文件注入完整配置文件（Java项目主流）
        volumeMounts:
        - name: config-volume
          mountPath: /app/config # 容器内配置目录
      volumes:
      - name: config-volume
        configMap:
          name: java-file-config
```

#### 验证与排错
##### 验证方式
1. 查看 ConfigMap 内容：`kubectl get cm java-file-config -o yaml`
2. 进入容器查看挂载文件：`kubectl exec -it java-config-demo-xxx -- cat /app/config/application-prod.yml`
3. 查看环境变量：`kubectl exec java-config-demo-xxx -- env | grep SPRING`

##### 常见报错
1. **configmap not found**：Deployment 关联的 ConfigMap 名称/key 书写错误
2. **挂载配置为空**：配置文件编码异常、文件后缀不规范

#### ⚠️避坑指南 / 💡最佳实践
1. **简单开关、字符串参数使用环境变量注入**；完整配置文件、yml 配置**使用文件挂载**；
2. 禁止将密码、密钥、token 等敏感数据存入 ConfigMap；
3. 所有微服务配置统一外置，绝对禁止配置硬编码进镜像；
4. ConfigMap 建议按**服务名+环境**命名，例如 `user-service-prod-config`。

#### 📌面试/高频场景题
**Q：ConfigMap 有几种注入方式？分别有什么优缺点？**
A：一共三种：
1. **环境变量注入**：生效后永久固化，不会自动更新，适合固定参数；
2. **命令行参数注入**：启动生效，无法动态变更；
3. **文件挂载注入**：支持热更新，适合业务动态配置，是 Java 微服务首选。

**Q：为什么企业级微服务必须使用 ConfigMap，不能写死配置？**
A：实现配置与镜像解耦，一套镜像适配多环境，配置可动态修改、统一管理、版本可控，符合云原生十二要素规范。

### 5.2 ConfigMap 热更新机制与问题
#### 概念定义
**ConfigMap 热更新**：修改集群内 ConfigMap 资源后，无需重建 Pod，容器内挂载的配置文件自动同步更新的机制。

需要重点区分：**文件挂载支持热更新，环境变量绝对不支持热更新**，这是 K8s 原生机制，也是生产最容易踩坑的点。

#### 核心原理
1. kubelet 节点组件会**定时轮询监听**本地缓存的 ConfigMap 变更；
2. 当 ConfigMap 内容变更，kubelet 会同步更新节点内 Pod 的挂载文件；
3. 文件更新采用**软链接替换机制**，保证文件读写不中断；
4. 默认更新延迟：**10秒~60秒**，存在短暂延迟；
5. 环境变量在 Pod **启动时一次性注入**，存入容器内存，后续 ConfigMap 更新无法修改内存变量。

#### 场景与价值
1. Java 动态配置开关、日志级别动态调整，无需重启服务；
2. 临时调整超时时间、线程池参数，实现业务动态调优；
3. 无需重建 Pod，避免服务重启带来的流量波动。

#### 实操示例
```bash
# 1. 编辑已存在的ConfigMap，修改日志级别
kubectl edit cm java-file-config

# 2. 进入容器查看配置文件（等待10-30秒自动更新）
kubectl exec -it java-config-demo-xxx -- cat /app/config/application-prod.yml

# 3. 验证环境变量无法更新
# 修改configmap中spring.profiles.active值，查看容器环境变量，不会变化
kubectl exec java-config-demo-xxx -- env | grep SPRING
```

##### 生产标准热更新方案（解决延迟+不生效问题）
```yaml
# 通过注解触发滚动更新（主流生产方案）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: java-config-demo
  annotations:
    # 配置变更自动滚动重启Pod，彻底解决热更新延迟、应用不感知问题
    checksum/config: ${sha256sum application-prod.yml}
spec:
  # 省略重复配置
```

#### 验证与排错
1. **挂载文件更新延迟**：属于 K8s 原生机制，无法彻底消除
2. **应用不读取新配置**：容器文件更新成功，但 Java 程序不会主动加载文件，配置不生效
3. **环境变量永久不更新**：属于设计机制，无修复方案

#### ⚠️避坑指南 / 💡最佳实践
1. ❌ 致命坑：**ConfigMap 文件热更新成功，但 Java 应用不会自动加载配置**，仅文件变更，程序无感知；
2. ❌ 环境变量注入配置，修改 ConfigMap **永远不会生效**；
3. ✅ 最佳实践：生产不依赖原生热更新，通过**配置哈希注解+滚动重启**实现配置更新；
4. ✅ 动态参数优先使用 Nacos/Apollo 配置中心，K8s ConfigMap 仅用于基础静态配置。

#### 📌面试/高频场景题
**Q：ConfigMap 挂载文件可以热更新，为什么生产依旧建议重启 Pod？**
A：K8s 仅更新容器文件，**Java 应用程序不会主动重载配置文件**，配置不会生效；同时原生热更新存在10-60秒延迟，稳定性差。生产通过滚动重启实现无损更新，稳定性更高。

**Q：为什么环境变量方式注入的 ConfigMap 不支持热更新？**
A：环境变量是 Pod 启动时写入容器内存，属于静态数据。kubelet 无法修改运行中容器的内存数据，因此永久无法更新。

### 5.3 Secret 管理与更新
#### 概念定义
**Secret** 是 K8s 专门用于存储**敏感数据**的资源，比如数据库密码、JWT密钥、账号、SSL证书、token 等。

和 ConfigMap 对应记忆：**ConfigMap 存明文配置，Secret 存加密敏感配置**。

#### 核心原理
1. Secret 数据存储时采用 **base64 编码**（非加密）存储在 etcd；
2. 使用方式和 ConfigMap 完全一致：环境变量、文件挂载；
3. Secret 默认具有**自动过期缓存、权限隔离**能力；
4. 更新机制完全复用 ConfigMap 机制：文件挂载可热更新、环境变量不可更新。

#### 场景与价值
1. 存储 Java 应用数据库账号密码、Redis 密码；
2. 存储接口签名密钥、加密秘钥、登录 token；
3. 存储 Ingress HTTPS 证书；
4. 实现敏感配置与代码、镜像解耦，杜绝密码硬编码泄露。

#### 实操示例
##### 1. 通用秘钥 Secret 创建
```bash
# 创建数据库账号密码secret
kubectl create secret generic java-db-secret \
--from-literal=db-user=root \
--from-literal=db-password=123456
```

##### 2. Deployment 挂载 Secret
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: java-secret-demo
spec:
  replicas: 2
  selector:
    matchLabels:
      app: java-secret-demo
  template:
    metadata:
      labels:
        app: java-secret-demo
    spec:
      containers:
      - name: java-app
        image: nginx:1.23
        # 环境变量注入秘钥
        env:
        - name: DB_USER
          valueFrom:
            secretKeyRef:
              name: java-db-secret
              key: db-user
        # 文件挂载秘钥
        volumeMounts:
        - name: secret-volume
          mountPath: /app/secret
          readOnly: true # 秘钥目录只读，安全加固
      volumes:
      - name: secret-volume
        secret:
          secretName: java-db-secret
```

##### 3. Secret 更新操作
```bash
# 编辑更新秘钥
kubectl edit secret java-db-secret

# 强制滚动更新Pod，让秘钥生效
kubectl rollout restart deployment java-secret-demo
```

#### 验证与排错
1. 查看秘钥（解码查看明文）
```bash
# 查看secret编码内容
kubectl get secret java-db-secret -o yaml
# base64解码
echo "编码字符串" | base64 -d
```
2. 常见报错：**权限不足读取 Secret** → 需要配置对应 RBAC 权限
3. 秘钥更新不生效：环境变量注入，必须重启 Pod

#### ⚠️避坑指南 / 💡最佳实践
1. ⚠️ 重点：**Secret 只是 base64 编码，不是加密**，任何人可直接解码，不能等同于加密；
2. 生产必须开启 **etcd 加密存储**，防止秘钥泄露；
3. 配置严格的 RBAC 权限，禁止普通用户查看、编辑 Secret；
4. Secret 更新绝对不要依赖原生热更新，统一使用 `rollout restart` 滚动重启；
5. 高安全场景推荐使用 Vault 专业秘钥管理工具。

#### 📌面试/高频场景题
**Q：Secret 和 ConfigMap 的区别？**
A：
1. 用途：ConfigMap 存放明文普通配置，Secret 存放敏感秘钥；
2. 存储：ConfigMap 明文存储，Secret base64编码；
3. 权限：Secret 默认权限更严格，支持只读挂载；
4. 大小限制：Secret 最大1MB，ConfigMap 无严格限制。

**Q：K8s Secret 安全吗？为什么？**
A：不完全安全。Secret 仅做 base64 编码，可轻松解码，不属于加密。生产需要配合 etcd加密、RBAC权限控制、审计策略保证秘钥安全。

### 核心知识点汇总
1. **ConfigMap**：外置明文配置，支持键值对、文件、目录三种创建方式，支持环境变量、文件挂载、命令行三种注入方式；
2. **热更新机制**：仅**文件挂载**支持延迟热更新，环境变量完全不支持；且文件更新后 Java 应用不会自动加载；
3. **生产最优方案**：放弃原生热更新，使用配置哈希注解+滚动重启实现稳定配置更新；
4. **Secret**：专门存放敏感数据，底层base64编码，使用方式、更新机制和ConfigMap一致；
5. **安全规范**：Secret 非加密，必须配合集群加密、RBAC权限加固。

### 生产落地规范
1. 所有 Java 应用配置全部外置，禁止硬编码、禁止打包进镜像；
2. 普通业务配置使用 ConfigMap，账号密码密钥统一使用 Secret；
3. 所有配置更新统一采用**滚动重启**，不依赖原生不稳定热更新；
4. 严格管控 Secret 权限，禁止随意导出、查看秘钥；
5. 动态高频变更配置，接入 Apollo/Nacos 配置中心，弥补 K8s 原生配置短板。

### 面试高频考点
1. ConfigMap 三种创建方式、三种注入方式区别
2. ConfigMap 热更新底层原理、为什么环境变量不更新
3. 为什么生产不使用 K8s 原生热更新
4. ConfigMap 和 Secret 的核心区别
5. Secret 的安全性问题与生产加固方案

## 6. 节点运维：节点 cordon/uncordon/drain/驱逐/维护
### 本章概述
节点（Node）是 Kubernetes 集群的**工作载体**，所有 Pod 最终都运行在节点之上。在生产环境中，**节点内核升级、硬件维护、资源扩容、故障隔离**是高频运维场景。

K8s 提供了一整套节点维护机制：`cordon`（禁止调度）、`uncordon`（恢复调度）、`drain`（优雅驱逐 Pod），配合污点、标签、自动驱逐策略，实现**节点维护零业务中断**。

本章面向 Java 开发与运维人员，完整讲解节点状态、维护流程、故障处理、生产规范，是集群运维必须掌握的核心技能，同时覆盖面试高频考点。

### 6.1 节点状态与管理命令
#### 概念定义
节点是 K8s 集群中提供计算资源的服务器（虚拟机/物理机）。`kubectl get nodes` 用于查看节点基础状态，`kubectl describe node` 用于查看节点详细信息，是节点运维的入口命令。

节点核心状态：
- **Ready**：节点正常，可接收 Pod 调度
- **NotReady**：节点异常，kubelet 无法正常通信
- **SchedulingDisabled**：节点正常，但禁止新 Pod 调度

#### 核心原理
1. 节点状态由 **kubelet** 定时上报给 API Server，存储在 etcd；
2. 调度器（kube-scheduler）只向 **Ready + 可调度** 节点分配 Pod；
3. 节点标签用于分组调度，污点（Taint）用于排斥 Pod，实现节点隔离。

#### 场景与价值
- 日常巡检：快速判断集群节点健康状态
- 故障定位：节点 NotReady 时排查硬件、网络、kubelet 故障
- 分组管理：按环境/业务/配置给节点打标签，实现调度定向分发

#### 实操示例
```bash
# ------------------------------
# 1. 查看节点状态（最常用）
# ------------------------------
kubectl get nodes -o wide

# ------------------------------
# 2. 查看节点详细信息（排错核心）
# ------------------------------
kubectl describe node node-01

# ------------------------------
# 3. 节点标签管理（调度定向）
# ------------------------------
# 给节点打标签
kubectl label nodes node-01 env=prod
kubectl label nodes node-01 app=java-service

# 查看节点标签
kubectl get nodes --show-labels

# 删除节点标签
kubectl label nodes node-01 app-

# ------------------------------
# 4. 节点污点管理（隔离）
# ------------------------------
# 添加污点
kubectl taint nodes node-01 key=value:NoSchedule

# 删除污点
kubectl taint nodes node-01 key:NoSchedule-
```

#### 验证与排错
- 验证：节点状态为 Ready，无异常事件
- NotReady 排查：
  1. 节点网络不通、SSH 无法登录
  2. kubelet 服务崩溃：`systemctl status kubelet`
  3. 磁盘、内存、PID 资源不足

#### ⚠️避坑指南 / 💡最佳实践
- 生产节点必须按**环境/业务类型**打标签，便于定向调度
- 核心 Java 服务节点单独分组，避免混部影响性能
- 禁止随意给节点加污点，会导致 Pod 无法调度

#### 📌面试/高频场景题
**Q：节点 NotReady 会发生什么？Pod 会被驱逐吗？**
A：节点 NotReady 后，kubelet 无法上报心跳。默认 5 分钟后，节点上的 Pod 会被标记为 Evicting，并在其他可用节点重建。

**Q：SchedulingDisabled 和 NotReady 有什么区别？**
A：NotReady 是节点**故障**，无法运行任何 Pod；SchedulingDisabled 是节点**正常**，只是禁止新 Pod 调度，已有 Pod 继续运行。

### 6.2 节点维护操作全流程
#### 概念定义
节点维护是生产**最高频操作**：内核升级、磁盘扩容、硬件更换、系统重启。

标准流程：
**cordon → drain → 维护 → uncordon**

- **cordon**：禁止新 Pod 调度到节点
- **drain**：优雅驱逐节点上所有业务 Pod 到其他节点
- **uncordon**：恢复节点调度能力

#### 核心原理
1. `cordon` 给节点打上 `SchedulingDisabled` 标记
2. `drain` 触发 Pod 优雅终止，由控制器在其他节点重建
3. 驱逐遵循**最小可用策略**，保证服务不中断
4. DaemonSet  Pod 不会被主动驱逐，必须加参数忽略

#### 场景与价值
- 节点系统升级、内核更新、硬件维护
- 节点资源过载，临时下线维护
- 故障节点隔离，防止异常扩散

#### 实操示例
```bash
# ------------------------------
# 1. 标记节点不可调度（禁止新Pod）
# ------------------------------
kubectl cordon node-01

# ------------------------------
# 2. 优雅驱逐节点上所有Pod（生产标准命令）
# ------------------------------
kubectl drain node-01 \
  --ignore-daemonsets \     # 忽略DaemonSet（必须加）
  --delete-emptydir-data \  # 删除临时存储（必须加）
  --force                   # 强制驱逐无人管理的Pod

# ------------------------------
# 3. 执行节点维护操作（升级/重启/扩容）
# ------------------------------
# 示例：升级内核
yum update -y kernel

# ------------------------------
# 4. 维护完成，恢复节点调度
# ------------------------------
kubectl uncordon node-01
```

#### 验证与排错
- 验证：drain 后节点无业务 Pod，uncordon 后恢复调度
- 常见报错：
  - 无法驱逐 DaemonSet → 加 `--ignore-daemonsets`
  - 无法删除 emptyDir → 加 `--delete-emptydir-data`
  - 驱逐卡住 → 检查 Pod 优雅终止时间、PDB 策略

#### ⚠️避坑指南 / 💡最佳实践
- **生产节点维护必须严格遵循 cordon → drain → 维护 → uncordon**
- 禁止直接重启节点，会导致业务中断
- 维护前检查集群剩余资源，确保有足够节点接收迁移的 Pod
- 一次只维护一个节点，避免集群雪崩

#### 📌面试/高频场景题
**Q：kubectl drain 不加 --ignore-daemonsets 会发生什么？**
A：命令执行失败，无法驱逐节点。因为 DaemonSet 是节点强制运行的 Pod，不属于 Deployment 管理，必须显式忽略。

**Q：为什么 drain 时需要 --delete-emptydir-data？**
A：emptyDir 是节点临时存储，与节点生命周期绑定。驱逐时必须删除，否则 Pod 无法重建，命令会被阻塞。

### 6.3 节点驱逐与故障处理
#### 概念定义
节点故障驱逐：当节点**突然宕机、断网、NotReady**时，K8s 自动将节点上的 Pod 标记为驱逐状态，并在其他节点重建，保证服务高可用。

核心参数：`pod-eviction-timeout`（默认 5 分钟）

#### 核心原理
1. 节点心跳断开 → 进入 NotReady
2. 等待超时（默认 5min）→ 触发自动驱逐
3. 控制器在其他可用节点重建 Pod
4. 节点恢复后，旧 Pod 不会自动恢复，直接删除

#### 场景与价值
- 节点硬件故障、断电、断网自动容灾
- 系统内核崩溃、死机自动恢复服务
- 集群高可用保障核心机制

#### 实操示例
```bash
# ------------------------------
# 1. 查看故障节点状态
# ------------------------------
kubectl get nodes

# ------------------------------
# 2. 查看节点驱逐事件
# ------------------------------
kubectl describe node node-01 | grep -i evict

# ------------------------------
# 3. 强制清理故障节点上的Pod（节点无法恢复时）
# ------------------------------
kubectl delete node node-01
```

#### 验证与排错
- 故障节点：Pod 状态变为 `Terminating`/`Evicted`
- 自动重建：其他节点启动新 Pod，服务恢复
- 驱逐不触发：
  - 集群无剩余资源
  - PDB（Pod 中断预算）阻止驱逐
  - 驱逐超时未到

#### ⚠️避坑指南 / 💡最佳实践
- 生产必须配置 **PDB（PodDisruptionBudget）** 防止驱逐导致服务不可用
- 节点故障后不要等待，直接 cordon + drain 清理
- 自动驱逐超时 5 分钟，核心服务可调整为 1~2 分钟
- 节点恢复后，直接 uncordon，不要手动恢复旧 Pod

#### 📌面试/高频场景题
**Q：节点突然宕机，Pod 多久会被自动驱逐？**
A：默认 5 分钟（pod-eviction-timeout），等待超时后 K8s 认定节点不可恢复，开始在其他节点重建 Pod。

**Q：什么是 PDB？为什么节点维护必须配置 PDB？**
A：PDB 是 Pod 中断预算，限制同时不可用的 Pod 数量，防止驱逐时服务副本数过低导致不可用，是生产高可用必备配置。

### 核心知识点汇总
1. **节点状态**：Ready（正常）、NotReady（故障）、SchedulingDisabled（禁止调度）；
2. **维护流程**：cordon → drain → 维护 → uncordon，是生产标准零停机流程；
3. **核心命令**：
   - cordon：禁止调度
   - uncordon：恢复调度
   - drain：优雅驱逐 Pod
4. **驱逐机制**：节点故障 5 分钟后自动驱逐，保证服务高可用；
5. **污点与标签**：用于节点调度控制与隔离。

### 生产落地规范
1. 所有节点维护必须走标准流程，禁止直接重启节点；
2. drain 必须携带 `--ignore-daemonsets --delete-emptydir-data`；
3. 集群必须配置 PDB，防止批量驱逐导致服务中断；
4. 一次只维护一个节点，确保集群冗余资源充足；
5. 故障节点快速隔离，避免影响整体集群稳定性。

### 面试高频考点
1. cordon / drain / uncordon 区别与使用场景
2. drain 必须参数及作用
3. 节点 NotReady 后的 Pod 驱逐机制
4. 自动驱逐超时时间
5. PDB 的作用与生产价值

## 7. 集群故障基础排查流程
Kubernetes 集群运维工作中，**日常 90% 的工作都是故障排查与问题修复**。对于 Java 开发与运维人员而言，单纯会敲部署命令远远不够，必须具备系统化的故障排查思维。

集群故障层级分为三大层：**控制面故障、节点数据面故障、业务应用故障**。绝大多数新手排查问题杂乱无章、到处瞎查，效率极低。本节将统一标准化、生产通用的分层排查思路，覆盖集群组件故障、节点故障、应用故障、网络故障、集群备份恢复，同时搭配全套实操命令、排错流程、避坑方案与面试考点，帮你建立**结构化、自上而下、从集群到应用**的标准排错体系，完全适配生产落地与面试场景。

### 7.1 集群故障排查通用思路
#### 概念定义
集群故障排查通用思路是 K8s 生产标准排错规范：**自上而下分层排查**，优先排查集群底层组件，再排查节点，最后排查业务应用。类比 Java 排查思路：先看服务器状态、中间件状态，最后看代码与日志。

分层结构：
1. **控制面（顶层）**：集群核心组件，集群中枢，一旦异常全局故障
2. **数据面（中层）**：工作节点、kubelet、网络、资源状态
3. **应用面（底层）**：Pod、容器、业务代码、配置问题

#### 核心原理
K8s 是分层架构，**上层组件故障一定会导致下层业务全部异常**。
1. kube-apiserver 挂掉：所有 kubectl 命令失效，集群无法读写
2. etcd 异常：集群配置丢失、资源读写失败
3. 节点 kubelet 异常：当前节点所有 Pod 无法调度、重启、更新
4. 应用异常：仅单个业务故障，集群整体正常

遵循**从整体到局部、从底层到上层**的排查逻辑，可以避免盲目排查，快速定位故障层级，缩短故障恢复时间。

#### 场景与价值
1. 集群整体不可用、所有服务宕机，快速判断是否为控制面故障；
2. 单节点服务异常，区分节点故障和应用故障；
3. 业务个别服务报错，精准定位配置、代码、网络问题；
4. 统一团队排错规范，降低生产故障恢复时长。

#### 实操示例
##### 1. 分层排查标准命令集
```bash
# ------------------------------
# 1. 控制面排查（集群全局）
# ------------------------------
# 查看控制面组件状态
kubectl get cs
# 查看控制面 Pod 状态（kube-system命名空间）
kubectl get pods -n kube-system

# ------------------------------
# 2. 数据面节点排查
# ------------------------------
# 查看所有节点健康状态
kubectl get nodes
# 查看节点详细事件、资源、报错
kubectl describe node

# 查看节点kubelet日志（服务器终端执行）
journalctl -u kubelet -f

# ------------------------------
# 3. 应用面排查
# ------------------------------
# 查看Pod状态
kubectl get pods -o wide
# 查看Pod事件
kubectl describe pod
# 查看容器业务日志（Java应用排错核心）
kubectl logs -f pod-name
kubectl logs -p pod-name
```

##### 2. 网络排查工具使用
```bash
# 服务器网络状态排查
netstat -antlp

# 抓包排查集群通信异常
tcpdump -i any port 6443

# 临时调试容器，测试集群DNS、网络连通性
kubectl run -it --rm net-tools --image=busybox -- /bin/sh
```

#### 验证与排错
1. **全局所有服务异常**：优先检查**控制面组件**
2. **单节点所有服务异常**：优先检查**节点kubelet、资源、网络**
3. **单个服务异常**：仅排查**应用Pod、配置、代码、探针**
4. **命令全部卡住无响应**：大概率 apiserver/etcd 异常

#### ⚠️避坑指南 / 💡最佳实践
- ❌ 新手误区：服务报错直接看业务日志，忽略集群底层问题，导致排查效率极低；
- ✅ 标准规范：**先集群、后节点、最后应用**；先全局、后局部；
- ✅ 故障优先确认故障范围：是全局故障、单节点故障、单服务故障；
- ✅ 所有生产故障必须留存排查日志、复盘文档，避免重复踩坑。

#### 📌面试/高频场景题
**Q：K8s 服务访问异常，你的排查思路是什么？**
A：采用分层排查：
1. 检查集群控制面组件是否正常，确认集群整体可用；
2. 检查目标节点是否 Ready、资源是否耗尽、kubelet 是否正常；
3. 检查 Pod 运行状态、探针状态、启动日志，确认应用是否启动成功；
4. 检查 Service、Endpoint、网络策略，确认流量转发正常；
5. 最后查看业务日志定位代码/配置问题。

**Q：为什么不能直接优先查看业务日志排查故障？**
A：很多服务异常是集群底层故障导致（节点宕机、网络异常、组件挂掉），并非代码问题。优先查日志会误导排查方向，浪费排障时间。

### 7.2 常见故障场景排查
#### 概念定义
本节汇总生产环境**最高频、面试必考**的四大故障场景：控制面故障、节点故障、应用故障、网络故障。覆盖 99% 的集群日常异常，标准化每一类故障的现象、根因、排查流程、解决方案。

#### 核心原理
1. **控制面组件**：apiserver、scheduler、controller-manager、etcd 属于集群基石，组件异常集群整体瘫痪；
2. **节点组件**：kubelet 负责节点Pod生命周期管理，kube-proxy 负责 Service 流量转发；
3. **应用故障**：全部来源于镜像、配置、代码、资源、探针问题；
4. **网络故障**：跨节点通信、iptables 规则、网络策略、Service 规则异常。

#### 场景与价值
解决生产高频问题：集群不可用、节点离线、Pod反复重启、服务404/503、跨节点无法通信、接口超时等核心故障。

#### 实操示例
##### 1. 控制面故障排查
```bash
# 查看控制面组件健康状态
kubectl get cs

# 查看etcd集群状态（核心存储组件）
kubectl exec -it etcd-node-01 -n kube-system -- etcdctl endpoint health

# 查看apiserver日志
journalctl -u kube-apiserver -f
```

**常见问题**
- etcd 数据损坏：集群资源读写失败、kubectl命令报错
- scheduler 异常：所有新Pod无法调度、一直Pending
- controller-manager异常：Deployment无法更新、无法扩缩容

##### 2. 节点故障排查
```bash
# 查看节点状态与事件
kubectl describe node node-name

# 查看kubelet异常日志
journalctl -u kubelet --since=10m

# 查看节点资源占用
kubectl top nodes
```

**常见问题**
- 磁盘/内存爆满导致节点 NotReady
- kubelet 挂掉，节点离线
- 节点时间不一致，证书失效

##### 3. 应用故障排查
```bash
# 查看Pod详细报错事件
kubectl describe pod java-app

# 查看当前运行日志
kubectl logs java-app

# 查看崩溃重启历史日志
kubectl logs -p java-app
```

**常见问题**
- ImagePullBackOff：镜像拉取失败
- CrashLoopBackOff：Java应用启动崩溃
- 探针失败：服务启动成功但无法接入流量

##### 4. 网络故障排查
```bash
# 查看Service后端端点
kubectl get endpoints java-svc

# 查看kube-proxy日志
kubectl logs -n kube-system kube-proxy-xxx

# 测试集群内部DNS解析
nslookup java-svc.default.svc.cluster.local
```

#### 验证与排错
1. 控制面故障：组件状态 Unhealthy → 重启对应组件、修复 etcd
2. 节点故障：节点 NotReady → 重启 kubelet、清理磁盘、释放资源
3. 应用故障：Pod异常 → 核对镜像、配置、JVM参数、代码异常
4. 网络故障：Endpoint为空 → 核对Service标签选择器

#### ⚠️避坑指南 / 💡最佳实践
1. 禁止随意重启控制面组件，容易导致集群数据丢失；
2. 节点资源告警必须及时处理，磁盘打满会批量导致Pod崩溃；
3. Java应用必须配置资源限制、健康探针，避免单点故障扩散；
4. 禁止随意修改 kube-proxy 与网络策略，极易引发集群网络瘫痪。

#### 📌面试/高频场景题
**Q：kube-scheduler 异常会导致什么问题？如何排查？**
A：kube-scheduler 负责Pod调度，异常后**所有新创建、重建的Pod全部处于Pending状态**，已运行Pod不受影响。排查方式：查看 kube-system 下 scheduler 日志、组件健康状态。

**Q：Service 存在但是无法访问，有哪些核心原因？**
A：1. Endpoint 为空，标签选择器不匹配；2. 就绪探针失败，Pod未接入流量；3. kube-proxy异常，iptables规则未生成；4. 网络策略拦截流量。

### 7.3 集群备份与恢复基础
#### 概念定义
集群备份恢复是生产集群**容灾底线能力**。K8s 所有集群数据、资源配置全部存储在 **etcd** 中，etcd 就是集群的数据库。

集群备份分为两类：
1. **etcd 全量备份**：集群底层数据备份，可恢复整个集群
2. **资源文件备份**：YAML资源备份，用于业务资源恢复

#### 核心原理
1. etcd 存储集群所有对象：节点、Pod、配置、权限、网络规则；
2. etcd 快照可以完整保存集群所有状态，集群崩溃后可一键恢复；
3. 资源YAML备份用于日常误删资源、配置丢失恢复；
4. 生产容灾核心思想：**定时备份、定期演练、绝不裸奔**。

#### 场景与价值
1. 防止 etcd 数据损坏、集群彻底瘫痪；
2. 防止人为误删除集群资源、误改配置；
3. 集群版本升级、迁移前备份，保障回滚能力；
4. 满足企业生产容灾规范、等保要求。

#### 实操示例
##### 1. etcd 快照备份（生产必备）
```bash
# 设置etcd环境变量
export ETCDCTL_API=3

# etcd全量快照备份
# --cacert/--cert/--key：集群证书文件
etcdctl snapshot save /data/etcd-backup-$(date +%Y%m%d).snapshot \
--cacert=/etc/kubernetes/pki/etcd/ca.crt \
--cert=/etc/kubernetes/pki/etcd/server.crt \
--key=/etc/kubernetes/pki/etcd/server.key
```

##### 2. etcd 快照恢复
```bash
# 基于备份文件恢复集群数据
etcdctl snapshot restore /data/etcd-backup.snapshot \
--data-dir=/var/lib/etcd-restore
```

##### 3. 全局资源批量备份
```bash
# 备份所有命名空间资源
kubectl get all -A -o yaml > k8s-all-resources-backup.yaml

# 单独备份核心资源
kubectl get cm,secret,deploy,svc,ingress -A -o yaml > core-resource.yaml
```

#### 验证与排错
1. 备份验证：查看备份文件大小、文件是否生成成功；
2. 恢复失败：证书不匹配、目录权限不足、etcd服务未停止；
3. 资源恢复失败：命名空间不存在、资源冲突。

#### ⚠️避坑指南 / 💡最佳实践
1. 生产必须配置**定时etcd自动备份**（建议每小时备份、保留7天）；
2. **只备份不恢复=没有备份**，必须季度做恢复演练；
3. 禁止直接在故障集群恢复etcd，建议离线恢复；
4. 所有业务YAML必须Git托管，不依赖集群备份。

#### 📌面试/高频场景题
**Q：K8s集群彻底崩溃，如何紧急恢复？**
A：优先使用最近的 etcd 快照文件恢复集群底层数据，集群启动后，核对业务资源，缺失资源通过Git托管的YAML重新部署。

**Q：etcd备份的意义是什么？可以替代YAML备份吗？**
A：不可以替代。etcd备份是集群底层快照，用于集群整体瘫痪恢复；YAML是业务资源版本管理，用于日常变更、回滚、版本管控，二者互补。
