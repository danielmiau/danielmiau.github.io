# 1. 虚拟化技术演进

## 1.1 传统虚拟化技术（VMware/KVM）

### 概念定义

虚拟化技术是一种**资源隔离与复用技术**，通过软件模拟完整的硬件设备，在一台物理服务器上创建多个独立的虚拟服务器（虚拟机，VM）。主流虚拟化方案分为商用VMware、开源KVM，是容器技术诞生之前企业服务器部署的主流方案。

### 核心原理

传统虚拟机依赖**Hypervisor（虚拟机监控程序）**实现虚拟化。Hypervisor是介于物理硬件与操作系统之间的中间层，核心作用是：接管物理机的CPU、内存、磁盘、网卡等硬件资源，进行统一划分、调度、隔离，为每一台虚拟机分配独立硬件资源，并允许虚拟机安装独立完整的操作系统。

简单类比：物理机是一栋大楼，Hypervisor是大楼的装修隔断系统，虚拟机是独立房间，每个房间都需要单独装修（安装完整OS），拥有独立的水电资源（硬件资源），房间之间完全隔离。

### 场景与价值

在物理机裸部署时代，一台服务器只能跑一套业务，资源利用率极低（普遍低于10%），且环境无法复用、服务器故障影响极大。传统虚拟化技术解决了两大核心痛点：

1. **资源复用**：单台物理机拆分多台虚拟机，大幅提升服务器资源利用率

2. **环境隔离**：不同业务部署在不同虚拟机，操作系统独立，避免业务互相干扰

3. **环境一致性**：虚拟机镜像可复制、克隆，快速搭建统一的测试/生产环境

### 优势与局限性

**✅ 虚拟机优势**：

- 隔离性极强：硬件级、操作系统级完全隔离，安全性极高

- 兼容性极好：支持所有操作系统、各类老旧业务，无适配成本

**❌ 虚拟机局限性（核心痛点）**：

- 资源开销大：每台虚拟机需要独立安装完整OS（GB级别），占用大量CPU、内存、磁盘资源

- 启动速度慢：操作系统启动耗时分钟级，无法应对突发流量弹性扩容场景

- 镜像臃肿：虚拟机镜像大小通常数十GB，传输、迁移、部署效率极低

- 环境笨重：运维复杂，无法适配微服务、云原生高频发布的业务场景

### 避坑指南/最佳实践

- 虚拟机适合**老旧单体项目、需要完整操作系统权限、强安全隔离**的场景

- 微服务、高频迭代、需要弹性扩缩容的业务，不建议使用虚拟机部署

### 面试高频题

**问题**：传统虚拟机存在哪些弊端，为什么云原生场景逐渐淘汰虚拟机？

**参考答案**：传统虚拟机基于Hypervisor实现硬件虚拟化，每台虚拟机独立搭载完整操作系统，存在资源开销大、启动慢、镜像臃肿、迭代效率低的问题，无法适配微服务高频发布、秒级弹性扩容的云原生场景。因此云原生场景优先使用轻量化的容器技术，仅共享宿主机内核，无需独立操作系统，兼顾性能与隔离性。

## 1.2 容器技术的诞生与定位

### 概念定义

容器是一种**操作系统级虚拟化技术**，基于Linux内核的Namespace和Cgroups实现。它不需要模拟完整硬件、不需要独立操作系统，直接共享宿主机Linux内核，仅对应用的进程、资源、网络、文件系统进行隔离与限制，是轻量化的虚拟化方案。

### 核心原理

容器的核心本质：**一个被内核隔离、资源受限的普通Linux进程**。

虚拟机是「完整的计算机」，容器是「隔离的进程」。容器通过Namespace实现环境隔离，通过Cgroups实现资源限制，舍弃了独立操作系统的冗余开销，仅封装业务程序、依赖库、配置文件，实现极致轻量化。

### 场景与价值

容器完美解决了虚拟机的所有痛点，适配云原生微服务架构：

1. **秒级启动**：无需启动操作系统，容器启动耗时毫秒/秒级，支持突发流量弹性扩容

2. **极致轻量化**：镜像大小MB级别，传输、部署、迁移效率大幅提升

3. **环境一致性**：一次构建，到处运行，彻底解决「本地能跑、服务器报错」的环境问题

4. **资源利用率高**：共享内核，无冗余系统开销，服务器资源利用率可提升至60%以上

5. **适配微服务**：适合数量多、迭代快、生命周期短的微服务应用

### 容器与虚拟机核心对比

|对比维度|虚拟机(VM)|容器(Container)|
|---|---|---|
|虚拟化层级|硬件级虚拟化|操作系统级虚拟化|
|隔离粒度|强隔离（系统级）|弱隔离（进程级）|
|操作系统|独立完整OS|共享宿主机内核|
|启动速度|分钟级|秒级/毫秒级|
|镜像大小|GB级别|MB级别|
|资源开销|极高|极低|
|适用场景|老旧业务、强隔离需求|微服务、云原生、弹性伸缩|
### 技术发展脉络

容器技术演进分为三个核心阶段：

1. **LXC（2008）**：Linux原生容器工具，实现基础的进程隔离与资源限制，但是操作复杂、镜像打包繁琐，仅适合运维人员手动使用

2. **Docker（2013）**：封装LXC底层能力，标准化镜像格式、简化操作命令，让容器技术普及落地

3. **OCI标准（2016）**：制定容器运行时、镜像统一标准，打破Docker垄断，实现容器技术标准化、开源化，为K8s生态奠定基础

### 避坑指南/最佳实践

- 容器**隔离性弱于虚拟机**，核心金融、涉密业务需要额外搭配安全策略加固

- 容器依赖Linux内核，无法跨Windows系统运行，平台兼容性低于虚拟机

### 面试高频题

**问题**：容器和虚拟机的本质区别是什么？

**参考答案**：核心区别是虚拟化层级不同。虚拟机基于Hypervisor做硬件虚拟化，拥有独立操作系统，隔离性强、开销大；容器基于Linux内核Namespace+Cgroups做操作系统级虚拟化，共享宿主机内核，仅隔离进程资源，轻量化、启动快、资源利用率高，更适配云原生微服务架构。

# 2. Linux容器核心技术（容器的基石）

所有容器技术（Docker/K8s）底层只依赖Linux三大内核技术：**Namespace（隔离）、Cgroups（限流）、UnionFS（镜像分层）**，三者共同构成了容器的底层基石。

## 2.1 Namespace：容器的“隔离魔法”

### 概念定义

Namespace是Linux内核提供的**资源隔离机制**，作用是对系统全局资源进行包装，让不同Namespace下的进程拥有独立的资源视图，进程之间互相不可见，实现隔离效果。简单理解：Namespace就是给进程“画围墙”。

### 核心原理

Linux系统所有进程默认处于同一个全局Namespace，所有进程互相可见、共享网络、PID、文件系统。内核通过Namespace将系统资源拆分，每一组Namespace内的进程只能看到当前命名空间的资源，无法感知外部资源，从而实现容器隔离。

### 六大Namespace详解

|命名空间|隔离资源|作用说明|
|---|---|---|
|PID|进程ID|容器内进程PID从1开始，看不到宿主机进程|
|UTS|主机名、域名|容器可以独立设置主机名，不影响宿主机|
|IPC|进程间通信|隔离进程消息队列、共享内存，禁止跨容器通信|
|Mount|文件系统挂载点|容器拥有独立文件系统，看不到宿主机目录|
|Network|网络栈|容器拥有独立网卡、IP、端口、路由表|
|User|用户/用户组|容器内root用户不等于宿主机root，提升安全性|
### 实操示例：unshare命令体验Namespace隔离

通过Linux原生`unshare`命令手动创建Namespace，模拟容器隔离效果

```bash
# 1. 创建独立PID、UTS、Mount命名空间，开启新bash终端
unshare --pid --uts --mount --fork bash

# 2. 在新终端查看进程PID（当前终端PID为1，看不到宿主机进程）
ps -ef

# 3. 修改容器主机名，不会影响宿主机
hostname test-container

```

### 验证与排错

- 验证：执行`ps -ef`仅能看到当前进程，证明PID隔离生效

- 常见报错：普通用户执行unshare权限不足，解决方案：加上sudo提权执行

### 避坑指南/最佳实践

- Namespace**仅隔离不限制**：只能隔离视图，无法限制CPU/内存占用，所以必须搭配Cgroups使用

- Linux内核默认缺少时间命名空间，容器时间默认继承宿主机，需手动配置时区

### 面试高频题

**问题**：容器的隔离是怎么实现的？六大Namespace分别作用是什么？

**参考答案**：容器通过Linux内核Namespace实现资源视图隔离，包含六大命名空间，分别隔离PID进程、UTS主机名、IPC进程通信、Mount文件系统、Network网络栈、User用户权限。Namespace实现进程隔离，配合Cgroups资源限制，共同构成容器隔离能力。

## 2.2 Cgroups：容器的“资源管家”

### 概念定义

Cgroups（Control Groups，控制组）是Linux内核提供的**资源限制、统计、隔离机制**。如果说Namespace是给进程“画围墙”，那Cgroups就是给进程“限水电”，专门用于限制进程的CPU、内存、磁盘IO、网络带宽等资源使用上限。

### 核心原理

Cgroups将系统进程分组管理，可为不同的进程组单独配置资源配额，同时统计每组资源的使用量。当容器进程资源占用超过配额时，内核会触发限流、暂停进程或OOM杀死进程，避免单个容器耗尽宿主机资源，保障集群稳定性。

### 核心组件

- **subsystem（子系统）**：对应各类资源限制，包含cpu、memory、blkio、net_cls等

- **hierarchy（层级树）**：Cgroups支持父子层级，子组继承父组资源限制

- **cgroup目录**：挂载在`/sys/fs/cgroup`，所有资源限制配置文件均在此目录

### 实操示例：cgcreate限制进程资源

```bash
# 1. 安装cgroup工具
yum install libcgroup-tools -y

# 2. 创建自定义cgroup分组 test-cgroup
cgcreate -g cpu,memory:/test-cgroup

# 3. 限制该分组CPU使用率最大10%
echo 10000 > /sys/fs/cgroup/cpu/test-cgroup/cpu.cfs_quota_us
echo 100000 > /sys/fs/cgroup/cpu/test-cgroup/cpu.cfs_period_us

# 4. 限制内存最大128M
echo 134217728 > /sys/fs/cgroup/memory/test-cgroup/memory.limit_in_bytes

```

### 验证与排错

- 验证：将高负载进程加入该cgroup，进程CPU、内存将被严格限制

- 常见问题：内存限制失效，原因是未关闭swap，容器生产环境必须关闭swap

### 避坑指南/最佳实践

- Namespace负责**隔离视图**，Cgroups负责**限制资源**，二者缺一不可

- 生产环境所有容器必须配置CPU、内存限制，防止单容器雪崩影响整机

## 2.3 容器镜像与联合文件系统

### 概念定义

UnionFS（联合文件系统）是一种**分层、只读、联合挂载**的文件系统技术，是Docker镜像分层存储的底层核心。它可以将多个文件系统层叠加合并，对外展示为一个完整的文件系统。

### 核心原理

Docker镜像采用**分层设计**，分为多层只读层+一层读写层：

1. **只读层（镜像层）**：包含基础系统、依赖库、程序文件，所有容器共享只读层，实现镜像缓存复用

2. **读写层（容器层）**：容器启动后自动生成，仅当前容器独有，所有文件修改、新增、删除操作都只会作用于读写层

当容器删除时，读写层同步销毁，镜像只读层永久保留，极大节省磁盘空间。

### 容器镜像与容器实例区别

- **镜像**：静态只读的模板文件，包含程序运行所需环境、依赖、配置，不可修改

- **容器**：镜像运行后的动态实例，拥有独立读写层，是运行中的进程

- 一对多关系：**一个镜像可以启动无数个容器实例**

## 2.4 容器的生命周期

### 概念定义

容器生命周期指容器从创建、启动、运行、暂停、停止到销毁的完整状态流转，是理解容器重启策略、故障自愈的基础。

### 核心流程

镜像拉取 → 创建容器（Created） → 启动容器（Running） → 暂停/退出（Paused/Exited） → 删除销毁

### 进程视角理解容器

从Linux内核视角看：**容器就是一个普通的用户态进程**，只是被Namespace隔离、Cgroups限流。宿主机执行`ps -ef`可以查看到所有容器进程，这也是容器故障排查的底层依据。

# 3. Docker核心原理与实践

## 3.1 Docker架构与核心组件

### 概念定义

Docker是一套**容器工程化工具链**，封装了Linux内核原生的Namespace、Cgroups、UnionFS能力，屏蔽底层复杂的内核操作，提供标准化的镜像构建、容器运行、仓库管理能力，让容器技术可以落地商用。

### 核心组件与架构

- **Docker Client**：客户端，用户输入docker命令的交互入口

- **Docker Daemon**：后台守护进程，监听客户端请求，管理镜像、容器生命周期

- **containerd**：容器运行时，负责管理容器生命周期、镜像解压、文件挂载

- **runc**：最轻量容器运行时，严格遵循OCI标准，负责创建Namespace、Cgroups，启动容器进程

完整工作流程：用户输入命令 → Client → Daemon → containerd → runc → 内核创建容器

## 3.2 Docker镜像与仓库

### Dockerfile指令详解

Dockerfile是镜像构建的描述文件，每一行指令生成一个镜像层，核心常用指令：

- **FROM**：指定基础镜像，所有镜像必须基于已有镜像构建

- **RUN**：构建阶段执行命令，写入镜像只读层

- **COPY/ADD**：拷贝本地文件到镜像，ADD支持解压远程文件

- **CMD**：容器默认启动命令，可被运行命令覆盖

- **ENTRYPOINT**：容器入口命令，不可被覆盖，用于固定启动脚本

### 镜像构建最佳实践

- 减少镜像层数：合并RUN命令，避免多余镜像层

- 利用构建缓存：固定基础镜像，前置不变的依赖安装命令

- **多阶段构建**：分离构建环境和运行环境，极致瘦身镜像

## 3.3 Docker常用命令与实操

```bash
# 镜像管理
docker build -t demo:v1.0 .  # 构建镜像
docker pull nginx:latest     # 拉取镜像
docker push 仓库地址/demo:v1.0 # 推送镜像

# 容器管理
docker run -d --name demo -p 8080:80 nginx # 启动容器
docker exec -it demo bash   # 进入容器终端
docker logs -f demo         # 实时查看容器日志
docker stop demo            # 停止容器
docker rm demo              # 删除容器

# 数据卷挂载（持久化核心）
docker run -d -v /host/data:/container/data nginx

```

## 3.4 Docker与OCI标准

### 概念定义

OCI（Open Container Initiative）开放容器标准，是Linux基金会制定的容器统一规范，解决Docker私有标准的垄断问题，实现容器跨平台兼容。

### 两大核心标准

- **OCI Runtime Spec**：容器运行标准，规定容器如何创建、启动、销毁（runc实现）

- **OCI Image Spec**：镜像格式标准，统一所有容器镜像打包格式

K8s完全兼容OCI标准，不依赖Docker，这也是后续K8s舍弃Docker的核心原因。

# 4. 容器网络基础（K8s网络的前置）

## 4.1 容器网络模型与基础组件

容器网络核心解决三大问题：容器与容器通信、容器与宿主机通信、容器与外网通信。核心基础组件：

- **veth pair**：虚拟网卡对，相当于网线，连接宿主机网桥与容器网卡

- **网桥**：二层虚拟交换机，实现同主机容器互通

- **NAT**：地址转换，实现容器访问外网

- **路由表**：控制网络数据包转发规则

## 4.2 Docker默认网络模式详解

### bridge 桥接模式（默认）

所有容器连接docker0网桥，同主机容器互通，通过NAT访问外网，隔离性适中，是最常用模式。

### host 主机模式

容器直接复用宿主机网络栈，无独立IP和端口，网络性能最高，但是**完全丧失网络隔离**。

### none 无网络模式

容器关闭所有网络功能，无网卡、无IP，适用于纯本地计算、无需联网的业务。

### container 共享网络模式

当前容器复用指定容器的网络命名空间，共享IP和端口，常用于容器 sidecar 架构。

## 4.3 容器网络的核心痛点

- **跨主机容器无法直接通信**：Docker原生bridge仅支持单机容器互通

- **容器IP动态变化**：容器重启IP改变，无法固定通信地址

- **网络隔离能力弱**：原生网络无法精细化控制容器访问权限

以上痛点，全部由后续K8s的CNI网络插件、Service、NetworkPolicy解决。

# 5. 容器化思想与适配规范（工程落地关键）

## 5.1 12要素应用（Twelve-Factor App）

### 概念定义

12要素应用是云原生应用的**通用设计规范**，定义了适配容器、微服务、持续集成的应用开发标准，是所有容器化改造的理论依据。核心要素：代码单一仓库、依赖声明、配置环境化、无状态、日志标准化、进程前台运行等。

### 场景价值

解决传统项目依赖混乱、配置硬编码、日志杂乱、有状态无法扩容的问题，让项目可以完美适配容器编排、弹性伸缩、持续发布。

## 5.2 应用容器化改造的核心要点

1. **无状态化改造**：业务数据全部存入数据库、对象存储，禁止本地磁盘存储业务数据

2. **配置代码分离**：数据库地址、密钥、参数全部使用环境变量/配置中心，禁止硬编码

3. **日志标准化**：程序日志输出到标准输出stdout，而非本地日志文件，适配容器日志收集体系

4. **前台进程运行**：容器必须前台启动，后台进程会导致容器启动后立即退出

5. **健康检查适配**：暴露/health健康接口，供K8s探针检测服务可用性

## 5.3 容器化常见反模式与避坑

- ❌ 错误：把容器当虚拟机，频繁SSH登录容器改配置、调试代码

- ❌ 错误：镜像包含编译环境、调试工具，导致镜像臃肿、漏洞过多

- ❌ 错误：依赖容器本地磁盘存储持久化数据，容器删除数据丢失

## 5.4 容器安全基础

- **非root用户运行**：禁止容器以最高权限root运行，防止容器逃逸攻击

- **最小镜像原则**：优先使用alpine轻量镜像，减少系统漏洞

- **权限最小化**：通过capabilities禁用多余系统权限，缩小攻击面
