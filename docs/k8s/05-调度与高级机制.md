## 1. Scheduler 调度完整流程
### 1.1 调度器基础概念与架构
#### 概念定义
**kube-scheduler** 是K8s控制面核心组件，是集群内置的**全自动资源调度管家**。通俗来说：当用户创建Pending状态的Pod后，集群不会随机分配节点，由kube-scheduler根据集群规则、节点资源、调度策略，自动筛选最优节点，完成Pod与节点的绑定。

kube-scheduler仅负责**节点筛选与绑定**，不参与Pod创建、启动、销毁，Pod运行管理由节点kubelet组件负责。

#### 核心原理
kube-scheduler 采用**插件化架构**，所有调度规则均以插件形式实现，可插拔、可扩展。调度器持续监听API Server，实时获取集群内所有**未绑定节点、状态为Pending**的Pod。同时实时采集全集群节点的资源使用率、标签、污点、亲和规则等数据，通过标准化算法筛选最优节点，完成调度绑定。

该架构的设计目的：解耦调度逻辑与集群核心逻辑，兼顾通用性与扩展性，既能满足普通微服务调度，又支持自定义扩展特殊业务场景。

#### 场景与价值
1. 自动均衡集群节点CPU、内存资源，避免单节点负载过高引发服务卡顿；
2. 根据自定义调度规则，实现服务专属部署、打散部署、隔离部署；
3. 无需人工干预节点分配，实现集群自动化运维；
4. 是所有高级调度策略（亲和性、污点、优先级）的运行载体。

#### 实操示例
```bash
# 查看集群调度器健康状态
kubectl get cs
# 查看kube-scheduler运行Pod（控制面组件）
kubectl get pods -n kube-system | grep kube-scheduler
# 查看调度器日志，排查调度异常（生产排错核心命令）
kubectl logs -n kube-system kube-scheduler-master
```

#### 验证与排错
**验证方式**：创建测试Pod，Pod状态从Pending自动变为Running，且自动分配最优节点，代表调度器正常工作。
**常见报错**
1. 调度器组件异常：所有新建、扩容Pod全部持续Pending；
2. 节点资源耗尽：调度器无可用节点，调度失败；
3. 调度规则冲突：亲和性、污点规则互斥，无匹配节点。

#### ⚠️避坑指南 / 💡最佳实践
1. ❌ 禁止手动固定Pod节点，会丧失集群自愈、扩容、漂移能力；
2. ✅ 所有业务Pod交由调度器自动分配，保证集群灵活性；
3. ✅ 集群部署必须保证多个控制节点，实现调度器高可用；
4. ✅ 日常巡检需监控kube-scheduler组件运行状态。

#### 📌面试/高频场景题
**Q：kube-scheduler 宕机，对集群有什么影响？**
**答案**：已运行的Pod不受任何影响，业务正常运行；所有**新建、重建、扩容、迁移**的Pod无法完成调度，持续处于Pending状态，集群无法新增业务实例。

**Q：调度器和kubelet的分工是什么？**
**答案**：kube-scheduler负责为Pending Pod筛选、绑定最优节点；kubelet负责监听节点Pod，完成容器启动、健康检测、生命周期管理。

### 1.2 调度完整流程拆解（预选/优选/绑定）
#### 概念定义
K8s官方标准化调度分为**三大核心阶段：预选、优选、绑定**，所有Pod调度都会严格执行该流程。
1. **预选（Predicates）**：硬性过滤，剔除不符合规则的节点；
2. **优选（Priorities）**：软性打分，对合格节点排序择优；
3. **绑定（Bind）**：完成Pod与最优节点的绑定写入集群。

#### 核心原理
1. **预选阶段（硬规则）**
遍历集群所有节点，严格校验节点资源、污点、亲和性、端口占用、存储权限等硬性条件，**任意条件不满足直接淘汰节点，无协商空间**。核心作用是过滤掉所有无法运行Pod的无效节点。

2. **优选阶段（软规则）**
针对预选筛选后的合格节点，通过内置算法0-10分打分，打分维度包含节点资源利用率、Pod分布均衡度、调度偏好规则。分数越高，节点优先级越高。该阶段仅做择优，不会导致调度失败。

3. **绑定阶段**
调度器将最优节点信息提交至API Server，写入etcd存储，完成资源绑定。节点kubelet监听到绑定事件后，启动容器完成业务部署。

#### 场景与价值
精准解释生产所有调度现象：PodPending、Pod扎堆、节点分配不均、调度规则失效等问题，是调度排错的底层基础。

#### 实操示例
```bash
# 创建测试Pod触发调度流程
kubectl run schedule-demo --image=nginx:alpine
# 查看Pod完整调度事件，包含预选、优选、绑定全过程日志
kubectl describe pod schedule-demo
```

#### 验证与排错
1. 调度成功：事件中出现 `Successfully assigned` 绑定日志；
2. 预选失败：提示资源不足、污点不匹配、亲和规则不满足；
3. 优选均分：多节点分数一致，系统随机分配节点。

#### ⚠️避坑指南 / 💡最佳实践
1. 硬规则（预选）优先级最高，直接决定调度成败；
2. 软规则（优选）仅优化部署效果，不会导致Pod调度失败；
3. 切勿配置过多硬性调度规则，极易造成集群无可用节点。

#### 📌面试/高频场景题
**Q：简述K8s Pod完整调度流程？**
**答案**：1. kube-scheduler监听集群Pending状态Pod；2. 预选阶段过滤所有不满足硬性规则的节点；3. 优选阶段对剩余节点打分排序，筛选最优节点；4. 绑定节点信息写入集群；5. 对应节点kubelet启动Pod。

**Q：预选和优选的核心区别？**
**答案**：预选是硬性过滤，不满足条件直接调度失败；优选是软性打分择优，仅优化部署策略，不会导致调度失败。

### 1.3 调度失败场景与排查思路
#### 概念定义
调度失败指Pod持续处于**Pending状态**，调度器无法完成节点绑定，是生产最高频的集群问题。简单来说：Pod创建成功，但找不到可以运行的节点，无法启动业务。

#### 核心原理
所有调度失败仅分为两类底层原因：
1. **规则不匹配**：污点、亲和性、节点标签等硬性规则配置错误，无匹配节点；
2. **资源不足**：集群节点CPU、内存、存储资源耗尽，无法承载新增Pod。

#### 场景与价值
标准化Pod Pending故障排查流程，快速修复业务部署失败、扩容失败问题，降低生产故障时长。

#### 实操示例
```bash
# 核心排错命令：查看Pod详细报错事件
kubectl describe pod java-service-demo
# 查看全集群节点资源使用率
kubectl top nodes
# 查看节点标签、污点配置，校验调度规则
kubectl get nodes --show-labels
kubectl describe node worker-01
```

#### 验证与排错
##### 高频调度失败场景
1. `Insufficient cpu/memory`：节点CPU/内存资源不足；
2. `node selector does not match`：节点标签选择器不匹配；
3. `taint tolerance mismatch`：节点污点无对应Pod容忍规则；
4. `affinity rule failed`：Pod亲和性硬规则不满足。

#### ⚠️避坑指南 / 💡最佳实践
1. 调度排查**优先执行kubectl describe pod**，所有失败根因都会打印在事件中；
2. 排查顺序：先确认集群资源，再校验调度规则，最后检查调度组件；
3. 生产集群需预留10%缓冲资源，避免突发扩容调度失败。

#### 📌面试/高频场景题
**Q：Pod一直处于Pending状态，你的标准化排查思路是什么？**
**答案**：首先通过`kubectl describe pod`查看Pod事件定位报错类型；其次检查集群节点剩余资源，判断是否资源耗尽；接着校验标签、污点、亲和性调度规则；最后检查kube-scheduler组件运行状态。

## 2. 资源 QoS 机制（面试核心）
### 2.1 QoS 核心概念与设计目的
#### 概念定义
**QoS（Quality of Service，服务质量）** 是K8s内置的**Pod资源优先级与驱逐机制**。通俗类比：集群节点资源就像酒店房间资源，不同Pod是不同客户，QoS就是客户等级。当酒店房间资源不足，优先驱逐普通客户，保留VIP客户。

QoS无需手动声明，K8s会根据Pod的`requests`（请求资源）、`limits`（限制资源）**自动判定等级**，用于节点资源耗尽、OOM、磁盘爆满时，决定Pod的驱逐优先级。

#### 核心原理
K8s集群节点资源有限，当CPU、内存、磁盘资源耗尽触发压力阈值时，内核需要杀死部分Pod释放资源，避免节点宕机。QoS机制就是一套标准化的**Pod淘汰规则**，优先牺牲低优先级业务，最大程度保障核心Java微服务稳定运行。

#### 场景与价值
1. 生产集群资源过载时，保护订单、支付、用户等核心业务；
2. 自动驱逐日志采集、测试任务、离线脚本等低优先级Pod；
3. 解决生产高频问题：**测试服务OOM杀死核心业务服务**；
4. 统一集群资源隔离标准，实现业务分级容灾。

#### 实操示例
```bash
# 查看Pod的自动生成QoS等级
kubectl get pod demo-pod -o yaml | grep qosClass
```

#### 验证与排错
**验证**：不同资源配置的Pod自动生成对应QoS标签；
**常见问题**：核心业务被OOM驱逐，大概率是QoS等级配置过低。

#### ⚠️避坑指南 / 💡最佳实践
1. ❌ 禁止核心Java业务不配置资源规则，默认最低QoS等级；
2. ✅ 所有线上Pod必须配置requests/limits，规范QoS等级；
3. ✅ 严格区分核心、普通、离线业务的QoS层级。

#### 📌面试/高频场景题
**Q：QoS机制的作用是什么？**
**答案**：QoS是K8s的Pod服务质量分级机制，在节点资源耗尽、OOM、磁盘压力过高时，按照优先级驱逐低等级Pod，保障核心业务持续运行，实现集群分级容灾。

### 2.2 三类 QoS 等级判定规则与特性
#### 概念定义
K8s固定划分**三类QoS等级**，优先级从高到低：Guaranteed（最高）、Burstable（中等）、BestEffort（最低），**完全由资源配置自动判定，无手动配置字段**。

#### 核心原理
通过`requests`（最小预留资源）和`limits`（最大占用资源）的数值对比，区分Pod资源优先级，不同等级拥有不同的资源隔离、驱逐策略。

| QoS等级    | 判定规则                           | 优先级 | 驱逐顺序   |
| ---------- | ---------------------------------- | ------ | ---------- |
| Guaranteed | 所有容器 requests == limits        | 最高   | 最后被驱逐 |
| Burstable  | 配置requests，requests &lt; limits | 中等   | 中级驱逐   |
| BestEffort | 无任何requests、limits配置         | 最低   | 最先被驱逐 |

#### 场景与价值
适配企业分层业务架构，精准区分**核心交易服务、普通查询服务、离线测试服务**，适配全生产场景。

#### 实操示例
##### 1. Guaranteed 最高优先级Pod（核心业务）
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pod-guaranteed
spec:
  containers:
  - name: java-order-service
    image: openjdk:11-jre-slim
    # requests与limits完全相等，判定为Guaranteed
    resources:
      requests:
        cpu: "0.5"
        memory: "512Mi"
      limits:
        cpu: "0.5"
        memory: "512Mi"
```

##### 2. Burstable 中等优先级Pod（普通业务）
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pod-burstable
spec:
  containers:
  - name: java-query-service
    image: openjdk:11-jre-slim
    # 预留基础资源，上限弹性扩容
    resources:
      requests:
        cpu: "0.2"
        memory: "256Mi"
      limits:
        cpu: "0.5"
        memory: "512Mi"
```

##### 3. BestEffort 最低优先级Pod（测试/离线）
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pod-besteffort
spec:
  containers:
  - name: test-service
    image: nginx:alpine
    # 无任何资源配置，默认最低优先级
```

#### 验证与排错
```bash
# 批量查看三类Pod的QoS等级
kubectl get pod pod-guaranteed -o yaml | grep qosClass
kubectl get pod pod-burstable -o yaml | grep qosClass
kubectl get pod pod-besteffort -o yaml | grep qosClass
```

#### ⚠️避坑指南 / 💡最佳实践
1. 生产**绝对禁止核心业务使用BestEffort**，极易被OOM杀死；
2. Java微服务通用方案：普通业务使用Burstable；
3. 支付、订单核心业务强制使用Guaranteed。

#### 📌面试/高频场景题
**Q：简述三类QoS等级的判定规则和优先级？**
**答案**：Guaranteed：所有容器资源请求等于限制，优先级最高；Burstable：配置请求且请求小于限制，优先级中等；BestEffort：无资源配置，优先级最低。节点资源不足时，按BestEffort→Burstable→Guaranteed顺序驱逐。

### 2.3 QoS 在调度与 OOM 中的作用
#### 概念定义
QoS不仅管控OOM驱逐，同时参与**资源调度、节点资源预留、集群压力管控**，是集群资源稳定性的底层核心机制。

#### 核心原理
1. **调度层面**：Guaranteed Pod会严格预留资源，调度最保守，优先占用稳定资源；BestEffort无资源预留，仅使用集群闲置资源；
2. **OOM层面**：节点内存溢出时，系统优先杀死低等级、高内存占用的Pod；
3. **磁盘压力层面**：节点磁盘爆满时，同样遵循QoS分级驱逐规则。

#### 场景与价值
彻底解决集群混部场景下，低优先级测试服务抢占核心业务资源、导致线上故障的问题。

#### 实操示例
无需独立配置，集群资源压力过载时自动触发QoS驱逐机制。

#### 验证与排错
1. 节点内存爆满：优先销毁BestEffort、Burstable Pod；
2. 核心Guaranteed Pod几乎不会被驱逐；
3. 同等级Pod，**内存占用越高越优先驱逐**。

#### ⚠️避坑指南 / 💡最佳实践
1. 禁止核心业务与测试业务混部同一节点；
2. 离线日志、备份任务统一配置最低优先级；
3. OOM故障排查必须优先核对Pod QoS等级。

#### 📌面试/高频场景题
**Q：同QoS等级的Pod，OOM时优先杀死哪一个？**
**答案**：同等级下，系统优先驱逐**实际内存占用更高**的Pod，最大化释放节点资源。

### 2.4 QoS 生产配置规范
#### 概念定义
统一企业标准化QoS配置规范，平衡**集群稳定性与资源利用率**，适配所有Java微服务场景。

#### 核心原理
通过标准化资源配置，对业务分层分级，实现集群资源精细化管控，避免资源浪费与服务抖动。

#### 场景与价值
统一团队开发运维规范，杜绝资源配置混乱、集群不稳定问题。

#### 实操示例
1. **核心业务（支付/订单/用户）**：Guaranteed，requests=limits，严格资源隔离，零抖动；
2. **普通微服务（查询/列表）**：Burstable，预留基础资源，支持弹性扩容，平衡性能与利用率；
3. **离线任务（日志/测试/备份）**：BestEffort，占用闲置资源，优先被驱逐。

#### 验证与排错
批量扫描集群Pod，统一整改不规范的资源配置，规避线上风险。

#### ⚠️避坑指南 / 💡最佳实践
1. 禁止全集群Pod配置Guaranteed，会导致集群资源利用率极低、成本浪费；
2. 禁止核心服务无资源限制，极易单Pod打爆节点；
3. 所有资源配置纳入Git版本管理。

#### 📌面试/高频场景题
**Q：生产为什么不建议所有Pod使用Guaranteed等级？**
**答案**：Guaranteed会永久预留固定资源，即使Pod空闲，资源也无法被其他服务复用，会大幅降低集群资源利用率，增加服务器成本。

## 3. 标签与选择器
### 3.1 标签（Label）基础概念与规范
#### 概念定义
**标签（Label）** 是K8s所有资源通用的**自定义键值对标记**，可以打在Node、Pod、Service、Deployment等所有资源上。类比商品标签，用来标记资源的环境、业务、版本、优先级，**仅用于筛选匹配，不改变资源本身功能**。

#### 核心原理
标签是K8s**所有高级能力的底层基石**：服务关联、调度筛选、灰度发布、批量运维、亲和性调度，全部依赖标签匹配实现。

#### 场景与价值
1. 环境隔离：区分dev/test/prod环境资源；
2. 业务分组：区分Java订单、用户、商品微服务；
3. 版本管控：区分v1稳定版、v2灰度版；
4. 批量运维：精准筛选目标资源操作。

#### 实操示例
```bash
# 给Pod添加业务、环境标签
kubectl label pod java-order-demo biz=order env=prod
# 给节点打标签，用于定向调度
kubectl label nodes worker-01 server-type=java-biz
# 查看资源所有标签
kubectl get pods --show-labels
# 覆盖更新已有标签
kubectl label pod java-order-demo env=prod --overwrite
# 删除指定标签
kubectl label pod java-order-demo env-
```

#### 验证与排错
**验证**：资源列表展示自定义标签键值对；
**常见报错**：标签包含特殊字符、超长、格式非法。

#### ⚠️避坑指南 / 💡最佳实践
1. 企业统一标签规范：`env、biz、version、team`，禁止自定义杂乱标签；
2. **Label用于筛选调度，Annotation用于备注描述**，二者不可混用；
3. 禁止随意修改核心业务标签，会导致服务脱流。

#### 📌面试/高频场景题
**Q：Label 和 Annotation 的区别是什么？**
**答案**：Label是键值对标签，用于**资源筛选、调度匹配、服务绑定**；Annotation用于资源备注、存储描述信息，**无法用于筛选和调度**。

### 3.2 选择器（Selector）类型与用法
#### 概念定义
**标签选择器（Selector）** 是基于标签的筛选规则，通过自定义规则匹配带有指定标签的资源，分为**等值选择器**和**集合选择器**两大类，是Service、调度策略的核心匹配工具。

#### 核心原理
资源通过选择器规则遍历集群资源，匹配符合标签条件的Pod/节点，完成资源绑定、调度筛选、流量关联。

#### 场景与价值
实现Service绑定后端Pod、定向调度、灰度流量分组、批量资源管理。

#### 实操示例
##### 1. 等值选择器（精准匹配=、!=）
```yaml
apiVersion: v1
kind: Service
metadata:
  name: order-svc
spec:
  # 精准匹配prod环境、订单业务Pod
  selector:
    env: prod
    biz: order
  ports:
  - port: 8080 # Service对外端口
    targetPort: 8080 # Pod内部端口
```

##### 2. 集合选择器（In、NotIn、Exists）
```yaml
apiVersion: v1
kind: Service
metadata:
  name: multi-version-svc
spec:
  selector:
    matchExpressions:
    # 匹配v1、v2两个版本的Pod，用于灰度发布
    - key: version
      operator: In
      values: ["v1","v2"]
  ports:
  - port: 8080
```

#### 验证与排错
**验证**：`kubectl get endpoints` 查看Service成功匹配后端Pod；
**常见问题**：标签大小写不一致、键值不匹配，导致Endpoints为空，服务无法访问。

#### ⚠️避坑指南 / 💡最佳实践
1. 简单场景优先使用matchLabels等值匹配，简洁稳定；
2. 多版本、复杂筛选场景使用matchExpressions集合匹配；
3. Service选择器必须与Pod标签**完全一致**。

#### 📌面试/高频场景题
**Q：matchLabels 和 matchExpressions 的区别？**
**答案**：matchLabels 仅支持精准等值匹配，适用于简单筛选；matchExpressions 支持In、NotIn、Exists等复杂集合匹配，适配灰度、多版本复杂业务场景。

### 3.3 标签+选择器典型业务场景
#### 概念定义
标签+选择器是集群最基础的资源治理能力，几乎覆盖所有生产运维场景，是微服务集群治理的核心基础。

#### 核心原理
通过标签完成资源分组，通过选择器完成精准筛选，实现**资源隔离、流量分组、定向调度、批量运维**。

#### 场景与价值
1. **环境隔离**：区分测试、生产资源，避免环境互相干扰；
2. **灰度发布**：通过version标签区分新旧版本，实现流量灰度；
3. **专属节点调度**：指定Java服务仅运行在业务节点；
4. **批量运维**：批量删除、重启、查询指定业务资源。

#### 实操示例
```bash
# 批量查询所有生产环境订单服务Pod
kubectl get pods -l env=prod,biz=order
# 批量删除测试环境所有Pod
kubectl delete pods -l env=test
# 批量给指定业务节点添加调度权限
kubectl label nodes -l server-type=java-biz schedule=allow
```

#### 验证与排错
筛选结果精准匹配目标资源，无多筛、漏筛问题。

#### ⚠️避坑指南 / 💡最佳实践
1. 所有业务必须统一打标准标签，否则无法实现自动化运维；
2. 灰度发布必须依赖版本标签，是生产灰度的标准方案。

#### 📌面试/高频场景题
**Q：标签选择器在生产的核心用途有哪些？**
**答案**：用于Service绑定后端Pod、节点定向调度、灰度流量分组、批量资源运维、资源环境隔离。

### 3.4 标签管理实操与避坑
#### 概念定义
标准化集群标签的新增、修改、删除运维规范，避免标签混乱、匹配失效、服务脱流等线上问题。

#### 核心原理
标签是资源关联的唯一标识，标签变更会直接影响调度、流量绑定，必须标准化运维。

#### 场景与价值
解决集群标签杂乱、废弃标签残留、服务解绑、调度失效问题。

#### 实操示例
```bash
# 批量更新全量Pod环境标签
kubectl label pods --all env=prod --overwrite
# 批量清理废弃标签
kubectl label pods -l version=v0 version-
# 校验节点标签完整性
kubectl describe node worker-01 | grep Labels
```

#### 验证与排错
核对资源标签列表，确保无废弃标签、标签匹配一致。

#### ⚠️避坑指南 / 💡最佳实践
1. 禁止随意覆盖核心业务标签，会导致Service解绑、流量中断；
2. 业务迭代后及时清理废弃标签，避免干扰调度规则；
3. 所有标签变更必须走运维审批流程。

#### 📌面试/高频场景题
**Q：修改Pod的标签会产生什么影响？**
**答案**：会导致Service标签选择器匹配失效、流量脱流，同时影响亲和性、节点调度规则，触发Pod重新调度重建。

## 4. 亲和性与反亲和性
### 4.1 亲和性基础概念与三大分类
#### 概念定义
**亲和性**是K8s精细化调度规则，通俗理解：自定义Pod的部署偏好，控制Pod**喜欢在哪运行、喜欢和谁一起运行、不喜欢和谁一起运行**。

分为三大核心类型：
1. **节点亲和性（NodeAffinity）**：Pod筛选喜欢的节点；
2. **Pod亲和性（PodAffinity）**：Pod筛选喜欢共存的Pod；
3. **Pod反亲和性（PodAntiAffinity）**：Pod筛选需要远离的Pod。

#### 核心原理
亲和性基于标签选择器实现，在调度预选、优选阶段干预节点分配逻辑，弥补原生随机调度的短板，实现精细化、高可用调度。

#### 场景与价值
1. 异构节点：GPU、高配置机器专属运行算力服务；
2. 服务耦合：业务Pod与日志Sidecar同节点部署；
3. 高可用容灾：多副本打散部署，避免单点故障集群雪崩。

#### 实操示例
完整配置后续小节统一演示。

#### 验证与排错
通过`kubectl describe pod`查看亲和性匹配事件，校验调度结果。

#### ⚠️避坑指南 / 💡最佳实践
1. 禁止配置过于复杂的亲和规则，极易导致调度失败；
2. **所有生产多副本微服务必须配置Pod反亲和性**。

#### 📌面试/高频场景题
**Q：三类亲和性分别解决什么生产问题？**
**答案**：节点亲和性实现节点定向筛选；Pod亲和性实现耦合服务聚合部署；Pod反亲和性实现副本打散，保障服务高可用。

### 4.2 硬规则 & 软规则 核心区别
#### 概念定义
所有亲和性统一分为**硬规则、软规则**两类，决定调度规则的严格程度。
1. **硬规则（required）**：必须满足，不满足直接调度失败，Pod持续Pending；
2. **软规则（preferred）**：尽量满足，无匹配资源时，忽略规则正常调度。

#### 核心原理
硬规则运行在**调度预选阶段**，属于硬性过滤条件；软规则运行在**调度优选阶段**，仅做打分择优，不阻塞调度。

#### 场景与价值
精准控制调度严格度，兼顾**高可用稳定性**与**调度成功率**。

#### 实操示例
- 硬规则关键字：`requiredDuringSchedulingIgnoredDuringExecution`
- 软规则关键字：`preferredDuringSchedulingIgnoredDuringExecution`

#### 验证与排错
1. 硬规则不满足：Pod Pending调度失败；
2. 软规则不满足：Pod正常启动，仅未达到最优部署效果。

#### ⚠️避坑指南 / 💡最佳实践
1. 高可用、强制隔离场景使用硬规则；
2. 部署优化、偏好性场景使用软规则，避免调度失败。

#### 📌面试/高频场景题
**Q：亲和性硬规则和软规则的生产选型区别？**
**答案**：强制高可用、资源隔离需求使用硬规则；仅优化部署架构、无强制要求的场景使用软规则，保证调度成功率。

### 4.3 NodeAffinity 节点亲和性详解
#### 概念定义
**节点亲和性**：基于节点标签，控制Pod优先/必须部署在指定节点，实现**节点专属调度、异构资源隔离**，功能完全覆盖并超越简单的nodeSelector。

#### 核心原理
调度器通过节点标签选择器，匹配符合条件的节点，通过软硬规则实现强制筛选或偏好筛选。

#### 场景与价值
1. GPU算力、高内存节点专属AI、大数据服务；
2. 测试、生产节点环境完全隔离；
3. 核心业务专属高配节点部署。

#### 实操示例
##### 硬规则：必须部署在生产节点
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: java-node-affinity-demo
spec:
  replicas: 2
  selector:
    matchLabels:
      app: java-node-demo
  template:
    metadata:
      labels:
        app: java-node-demo
    spec:
      affinity:
        nodeAffinity:
          # 硬规则：必须匹配prod环境节点
          requiredDuringSchedulingIgnoredDuringExecution:
            nodeSelectorTerms:
            - matchExpressions:
              - key: env
                operator: In
                values: ["prod"]
      containers:
      - name: java-app
        image: openjdk:11-jre-slim
```

##### 软规则：优先部署高配节点
```yaml
spec:
  affinity:
    nodeAffinity:
      # 软规则：优先部署高配置节点
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 10
        preference:
          matchExpressions:
          - key: node-level
            operator: In
            values: ["high"]
```

#### 验证与排错
**验证**：Pod仅调度至匹配标签的节点；
**报错**：无匹配节点，Pod持续Pending。

#### ⚠️避坑指南 / 💡最佳实践
1. 资源专属隔离使用硬规则；
2. 部署优化、资源择优使用软规则。

#### 📌面试/高频场景题
**Q：nodeSelector 和节点亲和性的区别？**
**答案**：nodeSelector仅支持简单等值硬匹配，功能单一；节点亲和性支持软硬双规则、复杂匹配、权重打分，功能更强大、适配场景更广。

### 4.4 PodAffinity 与 PodAntiAffinity 详解
#### 概念定义
**Pod亲和性**：让关联依赖的Pod部署在同一拓扑域，实现服务聚合；
**Pod反亲和性**：让相同业务的Pod分散部署，避免扎堆单点，实现高可用。

#### 核心原理
基于**目标Pod标签**进行调度匹配，控制Pod之间的部署拓扑关系，实现服务聚合或打散。

#### 场景与价值
1. Pod亲和：业务Pod+监控/日志Sidecar同节点部署，降低网络延迟；
2. Pod反亲和：多副本微服务分散部署，节点宕机不会整体宕机。

#### 实操示例
##### 标准Pod反亲和性配置（生产高可用标配）
```yaml
spec:
  affinity:
    podAntiAffinity:
      # 硬规则：同服务副本禁止部署在同一节点
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values: ["java-ha-service"]
        topologyKey: kubernetes.io/hostname
```

#### 验证与排错
多副本Pod均匀分散在不同节点，无扎堆现象。

#### ⚠️避坑指南 / 💡最佳实践
**所有生产多副本微服务必须配置Pod反亲和性**，杜绝单点故障。

#### 📌面试/高频场景题
**Q：为什么生产多副本服务必须配置Pod反亲和？**
**答案**：若无反亲和性，多副本极易扎堆同一节点，该节点宕机后，服务所有实例全部下线，导致服务整体不可用，丧失高可用能力。

### 4.5 拓扑域（topologyKey）作用与配置
#### 概念定义
**topologyKey拓扑域**：定义Pod打散/聚合的维度，通俗理解：**按照什么范围拆分服务**，支持节点、机架、可用区多个维度。

常用拓扑键：
1. `kubernetes.io/hostname`：按**节点**维度打散；
2. `topology.kubernetes.io/zone`：按**可用区**维度打散；
3. `topology.kubernetes.io/region`：按**地域**维度打散。

#### 核心原理
调度器根据拓扑域标签，在指定维度范围内执行亲和/反亲和规则，实现多层级容灾。

#### 场景与价值
实现单节点、单机房、单地域多层级故障隔离，提升集群容灾能力。

#### 实操示例
```yaml
# 跨可用区容灾拓扑配置
topologyKey: topology.kubernetes.io/zone
```

#### 验证与排错
Pod均匀分布在集群多个可用区。

#### ⚠️避坑指南 / 💡最佳实践
1. 单机房集群使用hostname节点打散；
2. 多可用区生产集群必须使用zone可用区打散。

#### 📌面试/高频场景题
**Q：topologyKey的核心作用是什么？**
**答案**：定义Pod亲和/反亲和的生效范围，决定Pod在节点、机房、地域维度进行聚合或打散部署，是多层级容灾的核心。

### 4.6 高可用场景：Pod 反亲和性生产实践
#### 概念定义
生产标准化高可用方案：通过Pod硬反亲和性+可用区拓扑域，实现服务**跨节点、跨机房打散部署**，彻底杜绝集群单点故障。

#### 核心原理
强制相同业务副本无法共存于同一个拓扑域，保证单个节点/机房故障，不会影响整体服务可用性。

#### 场景与价值
解决生产服务扎堆部署、单点宕机全盘雪崩的致命问题。

#### 实操示例
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: java-ha-service
spec:
  replicas: 3
  selector:
    matchLabels:
      app: java-ha-service
  template:
    metadata:
      labels:
        app: java-ha-service
    spec:
      affinity:
        podAntiAffinity:
          # 硬规则强制副本打散
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchLabels:
                app: java-ha-service
            # 按节点维度打散
            topologyKey: kubernetes.io/hostname
      containers:
      - name: java-app
        image: openjdk:11-jre-slim
```

#### 验证与排错
3个副本分别运行在3台不同节点，实现节点级容灾。

#### ⚠️避坑指南 / 💡最佳实践
核心服务副本数必须≥2，搭配反亲和性，才算完整的高可用方案。

#### 📌面试/高频场景题
**Q：如何实现服务跨机房容灾调度？**
**答案**：修改topologyKey为可用区维度，配合Pod硬反亲和性，让副本分散在多个机房，实现机房级容灾。

## 5. 节点污点与 Pod 容忍
### 5.1 污点（Taint）& 容忍（Toleration）基础概念
#### 概念定义
**污点（Taint）**：打在**节点上**，作用是排斥Pod，限制Pod调度；
**容忍（Toleration）**：打在**Pod上**，作用是匹配节点污点，获得节点准入权限。

通俗类比：污点是给房间上锁，容忍是开门钥匙，**没有对应钥匙的Pod无法进入上锁的节点**。

标签是**Pod主动选节点**，污点是**节点筛选Pod**，二者双向互补。

#### 核心原理
节点添加污点后，会拒绝所有不具备对应容忍规则的Pod，实现节点隔离、专属资源、故障节点屏蔽。

#### 场景与价值
1. 控制节点隔离，禁止业务Pod占用集群核心资源；
2. GPU算力节点专属，仅算力服务可调度；
3. 故障节点、维护节点隔离，避免业务部署异常。

#### 实操示例
```bash
# 给节点添加自定义污点
kubectl taint nodes worker-gpu-01 gpu=device:NoSchedule
# 删除节点污点
kubectl taint nodes worker-gpu-01 gpu=device:NoSchedule-
```

#### 验证与排错
普通Pod无法调度至污点节点，配置容忍的Pod正常部署。

#### ⚠️避坑指南 / 💡最佳实践
禁止随意删除Master控制节点污点，会导致业务Pod抢占控制面资源，集群瘫痪。

#### 📌面试/高频场景题
**Q：污点和亲和性的核心区别？**
**答案**：亲和性是Pod主动筛选节点（被动匹配）；污点是节点筛选Pod（主动隔离），**污点优先级高于亲和性**。

### 5.2 污点三种 Effect 效果详解
#### 概念定义
节点污点包含三种Effect策略，严格程度递增，适配不同隔离场景。
1. **NoSchedule**：禁止新Pod调度，已有Pod不受影响；
2. **PreferNoSchedule**：尽量不调度，无资源时允许调度；
3. **NoExecute**：禁止新调度，**立刻驱逐节点已有Pod**。

#### 核心原理
三种策略对应轻度、中度、重度节点隔离，覆盖节点维护、故障隔离、专属资源场景。

#### 场景与价值
精细化管控节点准入规则，适配不同生产运维场景。

#### 实操示例
```bash
# 禁止新Pod调度（专属节点常用）
kubectl taint nodes worker-01 biz=lock:NoSchedule
# 尽量不调度（资源优化）
kubectl taint nodes worker-01 biz=lock:PreferNoSchedule
# 驱逐现有Pod，彻底隔离（节点维护专用）
kubectl taint nodes worker-01 biz=lock:NoExecute
```

#### 验证与排错
NoExecute风险极高，会直接清空节点业务，生产慎用。

#### ⚠️避坑指南 / 💡最佳实践
1. 资源专属隔离使用NoSchedule；
2. 节点故障下线、维护使用NoExecute；
3. 资源优化场景使用PreferNoSchedule。

#### 📌面试/高频场景题
**Q：三种污点Effect的区别和生产选型？**
**答案**：NoSchedule仅拦截新Pod，用于专属节点；PreferNoSchedule软性隔离，用于资源优化；NoExecute强制驱逐存量Pod，用于节点故障维护。

### 5.3 污点与容忍匹配规则
#### 概念定义
Pod的容忍规则必须**完全匹配节点污点的key、value、effect**，缺一不可，才能解除节点隔离限制。

#### 核心原理
污点与容忍是一一对应的钥匙锁关系，规则不完全匹配则准入失败。

#### 场景与价值
精准实现专属节点业务准入，保障隔离安全性。

#### 实操示例
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: gpu-app
spec:
  # 匹配GPU节点污点，获得准入权限
  tolerations:
  - key: "gpu"
    operator: "Equal"
    value: "device"
    effect: "NoSchedule"
  containers:
  - name: gpu-service
    image: nvidia/cuda:11.0-base
```

#### 验证与排错
容忍规则不全、字段错误，都会导致Pod调度失败。

#### ⚠️避坑指南 / 💡最佳实践
污点容忍必须**三字段完全匹配**，不可省略参数。

#### 📌面试/高频场景题
**Q：容忍规则不完整会出现什么问题？**
**答案**：Pod无法匹配节点污点，无法调度至目标节点，持续Pending。

### 5.4 控制节点隔离、GPU 专属节点实战
#### 概念定义
利用污点容忍机制，实现生产两大核心隔离场景：**控制节点安全隔离、GPU算力节点专属独占**。

#### 核心原理
控制节点自带默认污点，禁止业务Pod；GPU节点手动添加污点，仅算力服务配置容忍准入。

#### 场景与价值
保护集群控制面稳定，保障算力资源不被普通业务抢占。

#### 实操示例
```bash
# 查看master节点默认污点
kubectl describe node master-01 | grep Taints
# 给GPU节点添加专属污点
kubectl taint nodes gpu-worker-01 gpu=enable:NoSchedule
```
```yaml
# GPU业务专属容忍配置
spec:
  tolerations:
  - key: "gpu"
    value: "enable"
    effect: "NoSchedule"
```

#### 验证与排错
普通微服务无法调度GPU节点，仅AI、算力服务可准入。

#### ⚠️避坑指南 / 💡最佳实践
绝对不要删除master节点污点，会造成集群控制面崩溃。

#### 📌面试/高频场景题
**Q：Master节点为什么无法运行业务Pod？**
**答案**：Master控制节点自带NoSchedule污点，无对应容忍规则，实现控制面与业务面彻底隔离，保障集群稳定性。

### 5.5 节点维护模式污点配置
#### 概念定义
节点维护污点是生产节点下线维护的标准方案，通过NoExecute污点自动驱逐Pod，替代手动drain命令。

#### 核心原理
NoExecute污点会强制驱逐节点所有存量Pod，禁止新Pod调度，实现节点彻底下线。

#### 场景与价值
标准化节点内核升级、硬件维护、故障下线流程。

#### 实操示例
```bash
# 节点进入维护模式，自动驱逐所有Pod
kubectl taint nodes worker-01 maintenance=on:NoExecute
# 维护完成，删除污点恢复调度
kubectl taint nodes worker-01 maintenance=on:NoExecute-
```

#### 验证与排错
添加污点后，节点所有业务Pod自动重建迁移。

#### ⚠️避坑指南 / 💡最佳实践
维护完成必须及时清理污点，否则节点永久无法调度新业务。

#### 📌面试/高频场景题
**Q：kubectl drain 和 NoExecute污点的关系？**
**答案**：drain命令底层原理就是为节点临时添加NoExecute污点，实现Pod驱逐与节点隔离。

## 6. Pod 优先级与抢占机制
### 6.1 PriorityClass 核心概念
#### 概念定义
**PriorityClass** 是集群全局资源对象，用于定义Pod优先级等级。通俗来说：给集群业务划分高低等级，资源不足时，**高等级业务可以抢占低等级业务的资源**。

#### 核心原理
PriorityClass定义全局优先级数值（数值越高优先级越高），Pod引用后获得资源抢占、优先调度的权限。

#### 场景与价值
集群资源过载时，自动牺牲日志、测试、离线低优先级服务，保障订单、支付核心Java服务稳定运行。

#### 实操示例
```yaml
# 自定义核心业务高优先级类
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: core-business-priority
value: 1000 # 优先级数值越高越高
globalDefault: false
description: "订单、支付核心业务优先级"
```

#### 验证与排错
PriorityClass创建成功后，可全局供Pod引用。

#### ⚠️避坑指南 / 💡最佳实践
集群优先级统一规划，禁止随意创建大量优先级规则。

#### 📌面试/高频场景题
**Q：PriorityClass的作用是什么？**
**答案**：定义集群全局Pod优先级，实现资源不足时高优先级Pod抢占低优先级Pod资源，保障核心业务容灾能力。

### 6.2 Pod 优先级等级定义与引用
#### 概念定义
Pod本身无优先级配置，必须通过 `priorityClassName` 字段引用PriorityClass，获取优先级属性。

#### 核心原理
Pod创建时加载优先级参数，调度器在资源不足、调度排队时，优先保障高优先级Pod。

#### 场景与价值
分层管理集群所有业务优先级，区分核心、普通、离线任务。

#### 实操示例
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: java-order-core
spec:
  # 引用核心业务高优先级规则
  priorityClassName: core-business-priority
  containers:
  - name: order-service
    image: openjdk:11-jre-slim
```

#### 验证与排错
Pod详情中展示自定义优先级数值，代表引用成功。

#### ⚠️避坑指南 / 💡最佳实践
集群默认优先级为0，所有核心业务必须手动配置高优先级。

#### 📌面试/高频场景题
**Q：Pod默认优先级是多少？**
**答案**：集群所有Pod默认优先级为0，属于最低优先级，极易被抢占驱逐。

### 6.3 资源不足时抢占机制完整流程
#### 概念定义
**抢占机制**：当集群资源耗尽、无空闲资源时，高优先级Pod需要启动，调度器主动驱逐低优先级运行中的Pod，释放资源供高优先级Pod使用。

#### 核心原理
1. 高优先级Pod创建，处于Pending状态；
2. 调度器检测集群资源不足；
3. 筛选节点内运行的低优先级Pod；
4. 优雅驱逐低优先级Pod释放资源；
5. 高优先级Pod完成调度启动。

#### 场景与价值
作为集群**最后容灾兜底方案**，极端资源过载时保障核心业务存活。

#### 实操示例
抢占机制为集群内置能力，无需手动配置，资源不足自动触发。

#### 验证与排错
通过`kubectl describe pod`查看Pod抢占驱逐事件日志。

#### ⚠️避坑指南 / 💡最佳实践
抢占仅作为兜底容灾，**绝对不能作为日常资源管理方案**。

#### 📌面试/高频场景题
**Q：简述Pod资源抢占完整流程？**
**答案**：高优先级Pod调度失败Pending → 调度器筛选节点低优先级Pod → 优雅驱逐低优先级Pod释放资源 → 高优先级Pod成功启动。

### 6.4 抢占机制风险与生产限制
#### 概念定义
Pod抢占会主动销毁运行中的业务Pod，必然导致低优先级服务短暂中断，存在集群抖动风险，需要严格管控。

#### 核心原理
抢占机制会触发Pod重建、重启，低优先级离线任务、测试服务会频繁重启，影响服务稳定性。

#### 场景与价值
规范生产抢占行为，避免集群频繁抖动。

#### 实操示例
集群默认开启抢占机制，无手动开关。

#### 验证与排错
集群频繁出现Pod驱逐重启，代表集群资源不足，需要扩容节点。

#### ⚠️避坑指南 / 💡最佳实践
1. 禁止大量Pod配置高优先级，会导致集群频繁抢占、抖动；
2. 低优先级离线任务单独部署，避免影响线上业务。

#### 📌面试/高频场景题
**Q：Pod抢占会导致业务中断吗？**
**答案**：会，低优先级Pod会被强制驱逐重建，产生短暂业务中断，仅用于极端容灾场景。

### 6.5 核心业务高优先级落地配置
#### 概念定义
企业标准化三级优先级方案，覆盖**核心业务、普通业务、离线任务**，适配全生产场景。

#### 核心原理
通过分层优先级配置，构建完整的集群资源容灾体系，最大化保障线上稳定性。

#### 场景与价值
统一企业优先级规范，杜绝资源抢占混乱。

#### 实操示例
```yaml
# 1. 核心业务优先级（最高）
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: core-priority
value: 1000
globalDefault: false
description: "订单、支付、用户核心服务"
---
# 2. 普通业务优先级（中等）
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: normal-priority
value: 500
globalDefault: false
description: "查询、列表普通微服务"
---
# 3. 离线任务优先级（最低）
apiVersion: scheduling.k8s.io/v1
kind: PriorityClass
metadata:
  name: offline-priority
value: 100
globalDefault: false
description: "日志、备份、测试任务"
```

#### 验证与排错
资源不足时，仅离线、普通服务被驱逐，核心服务稳定运行。

#### ⚠️避坑指南 / 💡最佳实践
三级优先级严格区分，禁止业务越级配置优先级。

#### 📌面试/高频场景题
**Q：生产如何利用优先级机制保障服务稳定？**
**答案**：分层定义优先级，核心业务最高优先级，离线任务最低，集群资源过载时自动牺牲低优先级任务，保障线上核心业务零中断。

## 7. 自定义调度策略

默认的K8s原生调度器只能满足通用业务的调度需求：资源均衡、节点亲和、污点容忍等基础规则。但在企业复杂生产场景中，例如大数据离线任务、AI训练任务、高优先级核心交易业务、资源独占队列、任务抢占、优先级排队等场景，原生调度器能力完全不足。此时需要依靠**自定义调度策略**，实现个性化、业务定制化的Pod调度逻辑。

### 7.1 自定义调度三种实现方案

#### 概念定义

K8s 官方完整开放了调度扩展能力，目前行业内共有三种标准自定义调度实现方案，从简单到复杂、从轻量到重度定制分别为：**调度器扩展程序、Scheduler Framework 框架插件、独立自定义调度器**。三种方案适配不同业务场景，覆盖99%企业定制化调度需求。

#### 核心原理

K8s 调度整体是插件化设计，kube-scheduler 并非单体封闭程序。API Server 在创建未调度Pod后，会触发调度链路。开发者可以通过介入调度链路、扩展调度插件、甚至独立部署调度程序接管调度，实现脱离原生默认规则的自定义调度逻辑。三种方案底层均遵循 K8s 标准调度扩展协议，完全兼容原生集群。

#### 场景与价值

原生调度仅支持资源均衡、标签亲和、污点策略，无法实现：任务优先级抢占、资源队列隔离、资源超额复用、GPU资源独占、大数据任务插队、分时调度等高级能力。三种自定义方案可以按需扩展，适配微服务、大数据、AI算力、离线批量任务等复杂场景，极大提升集群资源利用率与业务稳定性。

#### 实操示例

三种方案能力对比选型表：

| 实现方案                                                   | 开发难度 | 侵入性   | 适用场景                             | 是否替换原生调度     |
| ---------------------------------------------------------- | -------- | -------- | ------------------------------------ | -------------------- |
| 调度器扩展程序                                             | 极低     | 无侵入   | 简单规则扩展（自定义过滤、打分）     | 否，附加扩展         |
| Scheduler Framework 插件                                   | 中等     | 低侵入   | 企业常规定制调度、权重打分、特殊过滤 | 否，原生调度加载插件 |
| 独立自定义调度器                                           | 极高     | 完全独立 | 大数据、AI算力、抢占式调度、队列调度 | 可选独立接管         |
| 指定Pod使用自定义调度器，通过 **schedulerName** 字段区分： |          |          |                                      |                      |

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: custom-sched-pod
spec:
  # 指定自定义调度器名称，必须与部署的调度器名称一致
  schedulerName: custom-scheduler
  containers:
  - name: nginx
    image: nginx:alpine

```

#### 验证与排错

```bash
# 查看Pod调度器名称
kubectl get pod custom-sched-pod -o yaml | grep schedulerName
# 查看调度事件，判断是否被自定义调度器接管
kubectl describe pod custom-sched-pod
# 排查自定义调度器日志
kubectl logs -n kube-system deployment/custom-scheduler

```

常见报错：schedulerName 与自定义调度器名称不匹配，导致 Pod 持续 Pending；自定义调度器权限不足，无法监听调度事件。

#### ⚠️避坑指南/💡最佳实践

- ❌ 禁止随意替换集群默认调度器，极易导致全局调度异常、集群瘫痪；

- ✅ 简单调度规则优先使用 Scheduler Framework 插件，开发成本最低、稳定性最高；

- ✅ 重度算力调度、队列调度场景使用独立开源调度组件，不重复造轮子。

#### 📌面试/高频场景题

**Q：K8s 自定义调度有哪几种方式，区别是什么？**

**解析与答案**：一共三种方式。1. 调度扩展程序：最轻量，仅扩展过滤和打分逻辑，依附原生调度器；2. Scheduler Framework 插件：基于官方框架开发插件，侵入性低、能力全面，企业最常用；3. 独立自定义调度器：完全独立部署，可单独接管Pod调度，适合大数据、AI复杂调度场景，开发成本最高。

### 7.2 Scheduler Framework 调度框架

#### 概念定义

**Scheduler Framework** 是 K8s 1.16 版本后正式稳定的官方调度插件化框架。可以通俗理解为：官方给原生调度器预留了大量“插槽”，开发者编写插件后插入插槽，**无需修改K8s源码、无需重启集群组件**，即可扩展自定义调度能力，是目前企业自定义调度的标准方案。

#### 核心原理

原生kube-scheduler调度流程分为两大核心阶段：预选（Filter）、优选（Score）。Scheduler Framework 将完整调度链路拆解为多个精细化扩展点，调度器执行过程中会自动回调开发者自定义插件逻辑。整体遵循**主调度流程不变，插件按需介入**的设计思想，保证集群稳定性的同时支持无限扩展。

调度完整链路：预处理 → 预选过滤 → 优选打分 → 绑定预留 → 后置绑定，每个环节均可插入自定义插件。

#### 场景与价值

解决了早期自定义调度开发复杂、兼容性差、需要维护独立调度程序的问题。适合绝大多数企业定制场景：例如自定义节点权重、禁止低配置节点调度核心服务、实现业务专属节点调度、自定义资源亲和策略等。

#### 实操示例

框架使用规范：所有自定义插件实现框架统一接口，编译为插件二进制，挂载至kube-scheduler，无需替换原生调度器。集群内Pod默认走原生调度，指定schedulerName的Pod走插件扩展调度。

#### 验证与排错

通过kube-scheduler日志查看插件加载状态，若插件加载失败，调度器会自动降级为原生默认策略，不会影响集群整体稳定性，容错性极高。

#### ⚠️避坑指南/💡最佳实践

Scheduler Framework 是官方主推标准，**企业自定义调度优先选用该方案**，不建议自研独立调度器，维护成本极高。

#### 📌面试/高频场景题

**Q：简述 K8s Scheduler Framework 的作用与优势？**

**答案**：Scheduler Framework 是官方插件化调度框架，将调度链路拆分为多个扩展点，支持开发者自定义插件扩展调度能力。优势为：无需修改集群源码、无全局侵入、兼容性强、容错性高、开发成本低，是企业定制调度的主流方案。

### 7.3 调度扩展点与插件化原理

#### 概念定义

**调度扩展点**指 Scheduler Framework 对外开放的、可以植入自定义逻辑的调度节点。每一个扩展点对应调度流程的一个阶段，开发者可以按需在对应阶段注入过滤、打分、抢占、绑定等自定义逻辑。

#### 核心原理

K8s调度是串行阶段执行机制，扩展点严格按照固定顺序执行：

1. **PreFilter**：预处理Pod信息，提前校验参数，不符合条件直接拦截；

2. **Filter**：节点预选，过滤掉不满足调度条件的节点；

3. **PostFilter**：预选失败后的兜底处理；

4. **Score**：节点优选打分，分数越高越优先调度；

5. **Reserve**：节点资源预留，防止调度冲突；

6. **Bind**：最终绑定节点；

7. **PostBind**：调度完成后置回调。

所有自定义插件只需实现对应扩展点接口，即可介入调度流程。

#### 场景与价值

不同扩展点适配不同业务需求：Filter 用于自定义调度黑名单、资源校验；Score 用于自定义权重调度；Reserve 用于资源独占、抢占调度。精细化的扩展点可以实现极其复杂的定制调度逻辑。

#### 实操示例

典型场景：自定义打分插件，让高配节点权重更高，优先调度核心业务

核心逻辑：在 Score 扩展点中，读取节点标签的机器配置等级，配置越高打分越高，实现高配节点优先承载核心服务。

#### 验证与排错

调度失败优先查看 PostFilter 阶段日志，确认是预选过滤拦截；调度不均衡、节点负载不均，排查 Score 打分逻辑异常。

#### ⚠️避坑指南/💡最佳实践

尽量只在 Filter、Score 扩展点自定义逻辑，Bind、Reserve 属于底层核心链路，自定义出错极易导致集群调度异常。

#### 📌面试/高频场景题

**Q：Scheduler Framework 核心扩展点有哪些？各自作用？**

**答案**：核心扩展点包含PreFilter预处理、Filter节点过滤、Score节点打分、Reserve资源预留、Bind节点绑定。Filter用于筛选可用节点，Score用于节点权重择优，是日常自定义调度最常用的两个扩展点。

### 7.4 自定义调度器部署与使用

#### 概念定义

独立自定义调度器是完全独立的 Deployment 应用，不依赖原生kube-scheduler，单独监听API Server的Pod资源，筛选带有指定 schedulerName 的Pod，独立完成调度决策与绑定。

#### 核心原理

API Server 支持多调度器并行工作。原生调度器只会处理 schedulerName 为 default-scheduler 的Pod，自定义调度器只会处理自身名称匹配的Pod，**多个调度器相互隔离、互不干扰**。

#### 场景与价值

适合完全独立的算力调度场景，例如AI训练、大数据批量任务、离线计算任务，避免离线任务影响线上核心微服务调度，实现调度链路物理隔离。

#### 实操示例

自定义调度器标准部署yaml：

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: custom-scheduler
  namespace: kube-system
spec:
  replicas: 1
  selector:
    matchLabels:
      app: custom-scheduler
  template:
    metadata:
      labels:
        app: custom-scheduler
    spec:
      serviceAccountName: custom-scheduler-sa # 绑定调度权限账号
      containers:
      - name: custom-scheduler
        image: custom-scheduler:v1.0 # 自定义调度器镜像
        command: ["/custom-scheduler"]

```

#### 验证与排错

```bash
# 检查调度器Pod运行状态
kubectl get deploy custom-scheduler -n kube-system
# 检查自定义调度Pod调度记录
kubectl events --field-selector involvedObject.name=custom-sched-pod

```

常见问题：权限不足无法监听Pod资源、schedulerName不匹配导致调度失效。

#### ⚠️避坑指南/💡最佳实践

独立自定义调度器必须配置专属RBAC权限，否则无法访问、修改Pod资源；生产必须配置高可用多副本，避免单点故障。

#### 📌面试/高频场景题

**Q：集群能否部署多个调度器？会不会冲突？**

**答案**：可以部署多个，不会冲突。不同调度器通过Pod的schedulerName字段区分接管对象，各司其职、相互隔离，原生调度器处理默认业务，自定义调度器处理定制化业务。

### 7.5 开源定制调度组件：Volcano、Yunikorn

#### 概念定义

**Volcano** 是云原生社区官方批量作业调度组件，专为大数据、AI、离线计算、高并发批量任务设计；**Yunikorn** 是Apache开源的通用资源队列调度组件，擅长资源分层、队列隔离、多租户资源管控。二者是企业生产最主流的成熟定制调度方案，无需自研。

#### 核心原理

原生调度器面向**常驻微服务**设计，追求负载均衡；Volcano/Yunikorn面向**批量任务、短时任务、算力任务**，支持任务排队、优先级抢占、资源队列、任务重试、资源超额分配，弥补原生调度短板。

#### 场景与价值

- **Volcano**：适用于Spark、Flink、AI训练、模型推理、批量离线任务；

- **Yunikorn**：适用于多租户集群、开发测试生产多环境、资源队列隔离、团队资源配额管控。

可以极大提升离线算力集群资源利用率，实现多团队资源隔离与公平抢占。

#### 实操示例

Volcano批量任务专属Pod配置，指定调度器：

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: volcano-job-pod
spec:
  # 指定volcano调度器接管
  schedulerName: volcano
  containers:
  - name: job
    image: busybox
    command: ["sleep","3600"]

```

#### 验证与排错

部署完成后，带有volcano调度器名称的Pod会被Volcano接管，支持优先级排队、任务抢占，适合批量算力任务。

#### ⚠️避坑指南/💡最佳实践

微服务常驻业务**禁止使用Volcano**，仅离线批量任务使用；线上核心业务依旧使用原生default-scheduler，保证稳定性。

#### 📌面试/高频场景题

**Q：Volcano 和原生调度器区别？适用什么场景？**

**答案**：原生调度器主打微服务负载均衡，不支持任务队列、抢占、批量调度；Volcano是批量作业专用调度器，支持任务优先级、排队、抢占重试，专门用于大数据、AI、离线批量算力任务。

## 本章总结

### 调度核心概念速查表

本章完整覆盖K8s全部调度核心能力，从基础调度流程、调度策略、亲和性、污点、资源抢占、Pod优先级、自定义调度全方位覆盖，核心概念汇总如下：

1. **调度核心流程**：预选过滤所有不符合条件节点 → 优选打分择优节点 → 绑定预留完成调度；

2. **节点调度策略**：污点Taints让节点拒绝特定Pod，容忍Toleration让Pod可以适配污点节点；

3. **亲和性**：节点亲和性筛选节点、Pod亲和性实现服务聚合、反亲和性实现服务打散高可用；

4. **优先级与抢占**：高优先级Pod可抢占低优先级Pod资源，保障核心业务优先调度；

5. **自定义调度**：分为扩展插件、框架插件、独立调度器三类，企业优先Scheduler Framework成熟方案；

6. **开源调度组件**：Volcano面向批量算力任务，Yunikorn面向多租户资源队列管控。

### 生产调度最佳实践

汇总企业生产统一调度规范，全部为落地标准：

1. 核心微服务配置**Pod反亲和性**，多副本打散至不同节点，杜绝单点故障；

2. 专属节点通过污点隔离，只允许指定业务调度，实现资源独占；

3. 核心业务配置高优先级，避免离线任务抢占资源；

4. 禁止自研调度器，常规定制用Scheduler Framework，算力场景用Volcano；

5. 多租户集群使用Yunikorn做资源队列隔离，防止团队资源抢占；

6. 所有业务规范配置requests/limits，配合QoS保障调度稳定性。

### 调度失败排查 Checklist

遇到Pod Pending调度失败，严格按照以下顺序排查：

1. 查看Pod事件，确认失败原因：资源不足、规则不匹配、污点不兼容；

2. 校验节点亲和、Pod亲和/反亲和规则是否过于严格；

3. 核对节点污点与Pod容忍配置是否匹配；

4. 检查集群剩余资源是否满足Pod requests请求；

5. 自定义调度场景检查schedulerName匹配、调度器运行状态与权限；

6. 高优先级场景排查是否存在资源抢占驱逐。
