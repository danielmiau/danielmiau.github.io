## 1. 镜像轻量化优化

### 1.1 容器镜像体积过大的影响与问题

#### 概念定义

容器镜像体积过大，指构建后的业务镜像包含大量冗余文件：编译依赖、系统工具、日志缓存、操作系统冗余组件、调试工具等无用文件，导致镜像GB级臃肿，不属于业务运行必需资源。

#### 核心原理

Docker镜像采用**分层存储机制**，每一条Dockerfile指令都会生成新镜像层。新手构建镜像通常采用“单一阶段构建”，直接在完整JDK系统镜像内编译、打包、运行，会将编译工具、maven依赖、系统库文件全部打包进最终镜像，造成镜像体积爆炸式增长。

#### 场景与价值

几乎所有传统Java项目容器化初期都会出现镜像过大问题，动辄500M-1G。臃肿镜像会带来四大生产致命问题：镜像推送/拉取速度极慢、业务发布耗时久、集群磁盘资源浪费、镜像漏洞数量多、容器启动缓慢。镜像轻量化可以大幅提升发布效率、节省集群存储资源、降低安全风险。

#### 实操示例

错误示例：传统单阶段Java镜像构建（臃肿镜像）

```dockerfile
# 基础完整JDK镜像，体积大、冗余多
FROM openjdk:8
# 拷贝本地jar包
COPY target/demo.jar /app.jar
# 启动命令
ENTRYPOINT ["java","-jar","/app.jar"]
```

该方式构建的镜像体积普遍在500MB以上，包含完整操作系统、系统工具、多余依赖。

#### 验证与排错

```bash
# 查看本地所有镜像体积
docker images
# 分析镜像分层冗余
docker history 镜像ID

```

排查特征：镜像层数多、每层体积大、包含大量系统冗余文件。

#### ⚠️避坑指南/💡最佳实践

- ❌ 禁止使用完整版JDK、CentOS、Ubuntu作为生产基础镜像；

- ❌ 禁止单阶段构建，将编译环境带入运行镜像；

- ✅ 生产全部使用多阶段构建，剥离编译依赖；

- ✅ 仅保留程序运行必需的最小依赖。

#### 📌面试/高频场景题

**Q：镜像体积过大会带来哪些生产问题？**

**答案**：1. 镜像推拉速度慢，发布耗时增加，故障回滚效率低；2. 占用大量集群磁盘存储，资源浪费严重；3. 镜像包含多余系统组件，漏洞数量多，安全风险高；4. 容器解压启动慢，影响业务启动速度。

### 1.2 多阶段构建（Multi-stage build）原理与实践

#### 概念定义

**多阶段构建**是Docker官方推出的镜像构建方案，简单理解：拆分“编译环境”和“运行环境”。第一阶段使用完整镜像编译打包项目，第二阶段使用极简镜像，只拷贝编译完成的jar包，丢弃所有编译工具、依赖缓存，最终产出极致轻量化镜像。

#### 核心原理

Docker多阶段构建支持一个Dockerfile多个FROM指令，每一个FROM代表一个构建阶段。前序阶段仅用于编译、打包，不会携带至最终镜像；最后一个阶段为最终运行镜像，只保留业务程序和运行必需依赖，从根源减少镜像体积。

#### 场景与价值

是生产环境Java、Go、Python项目**标准构建方式**，无需复杂脚本，零成本实现镜像瘦身，可将500M+镜像压缩至50M以内，压缩率超90%。

#### 实操示例

Java项目标准多阶段构建Dockerfile（生产可用）

```dockerfile
# 第一阶段：编译打包（构建阶段）
FROM maven:3.8-openjdk-8 AS builder
# 设置工作目录
WORKDIR /app
# 拷贝依赖配置，缓存maven依赖
COPY pom.xml .
# 预下载依赖，优化构建缓存
RUN mvn dependency:go-offline
# 拷贝项目源码
COPY src ./src
# 编译打包，跳过测试
RUN mvn clean package -DskipTests

# 第二阶段：运行阶段（最终镜像）
FROM openjdk:8-jre-alpine
# 设置工作目录
WORKDIR /app
# 从构建阶段拷贝打包完成的jar包，仅拷贝必需文件
COPY --from=builder /app/target/*.jar app.jar
# 启动命令
ENTRYPOINT ["java","-jar","/app.jar"]

```

#### 验证与排错

构建后对比镜像体积，多阶段构建体积大幅缩小。若镜像体积依旧偏大，排查是否拷贝了多余目录、是否未跳过测试、是否保留了缓存文件。

#### ⚠️避坑指南/💡最佳实践

- ✅ 构建阶段使用完整镜像，保证编译环境齐全；

- ✅ 运行阶段必须使用极简镜像，剥离所有编译依赖；

- ❌ 禁止在运行阶段安装多余工具、调试组件。

#### 📌面试/高频场景题

**Q：什么是Docker多阶段构建？解决了什么问题？**

**答案**：多阶段构建将镜像构建拆分为编译阶段和运行阶段，编译阶段使用完整环境打包程序，运行阶段仅拷贝可执行程序。解决了传统单阶段构建镜像臃肿、携带大量编译冗余依赖、安全漏洞多的问题，实现镜像轻量化。

### 1.3 基础镜像选型：alpine / distroless / scratch 对比

#### 概念定义

基础镜像决定了容器底层系统环境和体积大小，生产主流三类极简镜像：

- **alpine**：基于极简Linux系统，体积5M左右，自带基础命令，兼容性极强；

- **distroless**：谷歌官方极简镜像，无包管理器、无系统命令，只包含程序运行依赖；

- **scratch**：空镜像，纯裸镜像，无任何系统环境，体积0M。

#### 核心原理

镜像体积：scratch < distroless < alpine < 完整版系统镜像。安全性与体积成反比，镜像包含的组件越少，攻击面越小，漏洞越少，但兼容性越低。

#### 场景与价值

|镜像类型|优点|缺点|生产适用场景|
|---|---|---|---|
|alpine|体积小、命令齐全、兼容性高|存在少量系统漏洞|Java、中间件、绝大多数业务服务|
|distroless|极致安全、漏洞极少|无调试命令，排障困难|高安全要求业务、金融服务|
|scratch|体积最小、零攻击面|兼容性极差|Go静态编译二进制程序|
#### 实操示例

Java生产标准选型：openjdk:8-jre-alpine，Go服务优先使用scratch。

#### 验证与排错

alpine镜像出现时区、字符编码问题时，手动安装tzdata配置时区；distroless镜像无法exec进入容器调试，需提前内置排查工具。

#### ⚠️避坑指南/💡最佳实践

普通业务优先**alpine镜像**，平衡轻量化、安全性、可调试性；金融、核心支付服务使用distroless极致安全镜像。

#### 📌面试/高频场景题

**Q：alpine和distroless镜像区别与选型？**

**答案**：alpine自带基础系统命令，方便调试，兼容性好，适合绝大多数业务；distroless无多余系统组件，漏洞极少、安全性更高，但无法进入容器调试，适合高安全等级的核心业务系统。

### 1.4 镜像分层优化：减少层数、清理构建缓存

#### 概念定义

Docker镜像每一条RUN、COPY、ADD指令都会生成独立镜像层，层数过多会增加镜像体积、降低缓存复用率。镜像分层优化就是**合并指令、清理缓存、合理排序指令**，最大化利用镜像缓存、精简层数。

#### 核心原理

Docker镜像遵循缓存机制：指令不变、依赖不变，则复用上层镜像缓存。频繁变更的文件放在Dockerfile最下层，稳定依赖放在最上层，可以大幅提升构建速度。同时合并多条RUN指令，减少镜像层数。

#### 场景与价值

适用于所有业务镜像构建，可以大幅提升镜像构建速度、减少镜像体积、降低存储占用。

#### 实操示例

分层优化前后对比：

```dockerfile
# 不规范写法：多层RUN，层数冗余
RUN apk update
RUN apk add curl
RUN rm -rf /var/cache/apk/*

# 规范写法：合并指令，同步清理缓存
RUN apk update && \
    apk add curl && \
    rm -rf /var/cache/apk/*

```

#### 验证与排错

通过docker history查看镜像层数，层数越少、单层层体积越小，优化效果越好。构建时命中缓存则代表分层排序合理。

#### ⚠️避坑指南/💡最佳实践

- ✅ 稳定依赖（系统依赖、pom文件）前置；

- ✅ 频繁变更的业务代码、配置后置；

- ✅ 每一条RUN执行完成后必须清理缓存垃圾；

- ❌ 禁止拆分多条独立RUN指令。

#### 📌面试/高频场景题

**Q：如何优化Docker镜像层数与构建速度？**

**答案**：合并RUN指令减少镜像层数；稳定依赖前置、变动文件后置，最大化复用镜像缓存；构建完成后清理系统缓存和冗余文件；使用多阶段构建剥离编译依赖。

### 1.5 镜像安全与漏洞扫描：Trivy 实战

#### 概念定义

**Trivy** 是轻量级开源容器镜像漏洞扫描工具，专门用于检测镜像内系统依赖、程序依赖的高危漏洞，是企业生产标配镜像安全检测工具。

#### 核心原理

Trivy内置全球漏洞数据库，扫描镜像分层、系统包、程序依赖包，识别高危、中危、低危漏洞，并给出修复方案，提前规避镜像上线安全风险。

#### 场景与价值

生产禁止带高危漏洞镜像上线，通过Trivy自动化扫描，实现镜像安全准入，避免容器漏洞被攻击、容器逃逸等安全事故。

#### 实操示例

```bash
# 安装trivy
brew install aquasecurity/trivy/trivy
# 扫描本地镜像漏洞
trivy image 镜像名称:版本
# 只扫描高危漏洞
trivy image --severity HIGH,CRITICAL 镜像名称:版本

```

#### 验证与排错

扫描输出高危漏洞必须全部修复，中低危漏洞按需修复。漏洞大多来自基础镜像版本过旧，升级基础镜像即可解决。

#### ⚠️避坑指南/💡最佳实践

将Trivy扫描接入CI流水线，**高危漏洞镜像直接阻断构建上线**，实现自动化安全准入。

#### 📌面试/高频场景题

**Q：企业如何保障容器镜像安全？**

**答案**：选用极简基础镜像、多阶段构建瘦身、清理冗余依赖；使用Trivy工具自动化扫描漏洞；流水线拦截高危漏洞镜像；定期更新基础镜像版本，修复系统漏洞。

### 1.6 生产环境镜像优化 Checklist

生产镜像上线必须满足以下标准：

1. 全部采用**多阶段构建**，剥离编译环境；

2. 业务服务统一使用alpine极简基础镜像；

3. 合并Dockerfile指令，精简镜像层数；

4. 构建完成自动清理缓存、冗余文件；

5. 流水线接入Trivy漏洞扫描，阻断高危漏洞镜像；

6. 禁止内置调试工具、冗余系统组件。

## 2. 容器运行时深度安全优化

### 2.1 容器逃逸风险与攻击面分析

#### 概念定义

**容器逃逸**是容器最核心的安全风险：攻击者通过业务漏洞入侵容器后，突破容器隔离机制，获取宿主机权限、控制整个节点甚至集群，造成集群数据泄露、服务瘫痪。

#### 核心原理

K8s容器默认权限过大、文件系统可读写、拥有过多Linux权限、以root用户运行，一旦业务被入侵，攻击者可以利用高权限突破容器隔离，访问宿主机文件、进程、网络，实现容器逃逸。

#### 场景与价值

公网暴露的业务服务、用户可输入参数的接口极易被攻击，容器逃逸属于生产高危安全事故，容器运行时安全加固是生产集群**必做配置**。

#### 实操示例

不安全配置场景：容器默认root运行、文件系统可读写、权限全开。

#### 验证与排错

```bash
# 查看容器运行用户
kubectl exec pod-name -- id
# 查看容器文件系统读写权限
kubectl exec pod-name -- touch /test.txt

```

#### ⚠️避坑指南/💡最佳实践

生产容器**绝对禁止默认root权限运行**，必须全方位收缩权限，最小化攻击面。

#### 📌面试/高频场景题

**Q：什么是容器逃逸？如何防范？**

**答案**：容器逃逸是攻击者入侵容器后突破隔离，获取宿主机权限的攻击方式。通过非root运行、只读文件系统、收缩Linux权限、安全策略配置，最小化容器攻击面，防范容器逃逸。

### 2.2 安全上下文进阶配置：非root运行、只读根文件系统

#### 概念定义

**安全上下文（securityContext）** 是K8s专门用于容器权限管控的配置项，可以限制容器运行用户、文件系统权限、系统能力，是容器安全加固的核心配置。

#### 核心原理

通过安全上下文强制容器使用普通用户启动，禁止root权限；开启只读根文件系统，让容器无法写入、修改系统文件，即使被入侵，攻击者也无法篡改文件、植入木马，从根源防范攻击。

#### 场景与价值

所有生产业务容器**强制配置**，是成本最低、效果最好的安全加固手段。

#### 实操示例

生产标准安全上下文配置：

```yaml
securityContext:
  runAsUser: 1000          # 使用普通用户运行，禁止root
  runAsNonRoot: true       # 强制禁止root用户启动
  readOnlyRootFilesystem: true # 根文件系统只读
  allowPrivilegeEscalation: false # 禁止权限提升

```

#### 验证与排错

配置后进入容器，无法执行root权限命令、无法写入根目录文件，代表配置生效。若业务启动失败，多是日志、缓存目录需要写入，单独挂载emptyDir临时目录即可。

#### ⚠️避坑指南/💡最佳实践

只读根目录场景下，业务日志、临时文件需要单独挂载emptyDir目录，避免启动失败。

#### 📌面试/高频场景题

**Q：runAsNonRoot和readOnlyRootFilesystem作用？**

**答案**：runAsNonRoot强制容器非root运行，降低权限风险；readOnlyRootFilesystem锁定根文件系统只读，防止入侵后篡改系统文件、植入木马，加固容器安全。

### 2.3 Linux Capabilities 最小化配置

#### 概念定义

**Linux Capabilities** 是Linux内核细分权限机制，root用户拥有所有内核权限，Capabilities可以精细管控容器内核权限，按需赋予最小权限。

#### 核心原理

默认容器会携带大量冗余内核权限，存在逃逸风险。通过删除多余Capabilities，只保留业务必需的内核权限，实现权限最小化。

#### 场景与价值

适用于核心生产业务、对外暴露服务，精细化收缩内核权限，大幅降低逃逸概率。

#### 实操示例

```yaml
securityContext:
  # 删除所有多余内核权限
  capabilities:
    drop:
      - ALL

```

#### 验证与排错

删除所有权限后业务异常，则按需添加少量必需权限，遵循最小权限原则。

#### ⚠️避坑指南/💡最佳实践

生产默认**drop全部权限**，不使用的权限一律关闭。

### 2.4 AppArmor / SELinux 安全策略实践

AppArmor、SELinux是Linux内核安全模块，用于限制容器进程的系统调用、文件访问、网络调用，是容器底层内核级安全加固方案，用于兜底防护容器逃逸和恶意攻击。企业生产优先使用AppArmor，配置简单、性能损耗低。

### 2.5 容器运行时安全工具：Falco 简介

**Falco** 是云原生官方容器运行时安全监控工具，实时监控容器异常行为：文件篡改、权限提升、异常进程启动、容器逃逸尝试，实时告警，实现容器安全动态监控。

### 2.6 容器运行时安全加固最佳实践

容器安全加固六件套：非root运行、只读根文件系统、禁止权限提升、内核权限最小化、内核安全策略、运行时动态监控，全方位构建容器安全防线。

## 3. 集群资源治理

### 3.1 集群资源浪费的典型场景与问题

生产集群普遍存在大量资源浪费：测试环境Pod长期不删除、业务资源配置过大、闲置副本、资源无配额限制导致节点资源爆满、高低优先级业务抢占资源，导致集群资源利用率极低、稳定性差。资源治理核心目标：**杜绝浪费、合理分配、隔离抢占、提升利用率**。

### 3.2 Namespace 资源配额：ResourceQuota / LimitRange

#### 概念定义

**ResourceQuota**：命名空间级别总资源配额，限制该命名空间下所有Pod总CPU、内存、数量上限；**LimitRange**：单Pod资源限制，约束单个容器最大/最小资源，防止单个服务抢占大量资源。

#### 实操示例

```yaml
# 命名空间资源总配额
apiVersion: v1
kind: ResourceQuota
metadata:
  name: ns-quota
  namespace: test
spec:
  hard:
    pods: "50" # 最大Pod数量
    cpu: "20"  # 总CPU上限
    memory: 20Gi # 总内存上限

```

#### 📌面试/高频场景题

**Q：ResourceQuota和LimitRange区别？**

**答案**：ResourceQuota限制命名空间整体资源总量与资源对象数量；LimitRange限制单个容器资源上下限，二者配合实现命名空间资源全方位管控。

### 3.3 QoS 等级与资源调度优化

K8s QoS分为Guaranteed、Burstable、BestEffort三个等级，优先级依次降低。核心生产最佳实践：**核心业务配置Guaranteed保障稳定性，非核心业务使用Burstable，禁止使用BestEffort**，避免业务被驱逐抢占。

### 3.4 资源请求与限制（requests/limits）合理配置

**requests**是调度依据，保证业务最低资源；**limits**是资源上限，防止服务抢占节点资源。生产规范：核心业务requests=limits，保证QoS最高等级；普通业务requests略低于limits，平衡稳定性与资源利用率。

### 3.5 集群资源使用率监控与闲置资源清理

通过kubectl top、Prometheus监控集群资源使用率，定时清理废弃Deployment、PVC、ConfigMap、闲置Pod，释放集群资源，提升整体利用率。

### 3.6 资源治理自动化方案：KubeCost 等工具简介

KubeCost是开源集群资源成本监控工具，可自动统计各命名空间、各业务资源使用率、资源浪费量，精准定位浪费资源的服务，实现自动化资源治理与成本统计。

## 4. 节点负载压力优化

### 4.1 节点资源瓶颈分析：CPU/内存/磁盘IO/网络

节点四大瓶颈：CPU打满导致调度卡顿、内存溢出导致OOM、磁盘IO过高导致日志写入阻塞、网络带宽爆满导致服务超时。所有节点高负载问题，都源于这四类资源瓶颈。

### 4.2 kubelet 性能调优参数

kubelet默认参数适配小规模集群，大规模集群需要调优：调整节点状态上报频率、Pod最大并发数、镜像垃圾回收阈值、资源驱逐阈值，避免kubelet卡顿、节点NotReady。

### 4.3 节点资源隔离与Pod调度优化

通过污点、亲和性、资源配额，将核心业务与日志采集、监控组件、测试业务调度至不同节点，实现节点资源隔离，避免非核心业务抢占核心业务资源。

### 4.4 节点压力排查工具：top/htop/iotop/ss

top查看CPU内存负载、iotop排查磁盘IO瓶颈、ss统计网络连接数，全方位定位节点负载高的根因。

### 4.5 节点高负载场景优化方案

CPU瓶颈：优化业务代码、限制PodCPU上限、节点扩容；内存瓶颈：优化JVM参数、清理内存泄漏服务；IO瓶颈：日志归档、更换高性能存储；网络瓶颈：优化CNI插件、限制连接数。

### 4.6 节点维护与故障恢复流程

节点维护标准SOP：节点驱逐排空Pod → 锁定节点禁止调度 → 维护升级 → 检查节点状态 → 恢复调度，保障维护过程业务零中断。

## 5. HPA 弹性伸缩深度讲解

### 5.1 HPA 核心概念与工作原理

**HPA（Horizontal Pod Autoscaler）** 是K8s Pod水平自动扩缩容组件，核心作用是根据业务负载自动增减Pod副本数，低峰缩容节省资源、高峰扩容承载并发，无需人工干预，完美适配流量波动业务。

### 5.2 基于CPU/内存的HPA配置与实践

基础HPA基于容器CPU、内存使用率自动伸缩，是生产最通用的弹性方案。

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: java-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: java-demo
  minReplicas: 2  # 最小副本数
  maxReplicas: 10 # 最大副本数
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        averageUtilization: 70 # CPU使用率70%触发扩容

```

### 5.3 自定义指标与Prometheus Adapter 集成

原生HPA仅支持CPU、内存指标，无法适配QPS、连接数、队列长度等业务指标。通过Prometheus Adapter对接监控指标，实现**基于业务QPS、接口耗时的精准弹性伸缩**。

### 5.4 多指标HPA配置与策略优化

生产配置CPU+内存+QPS多维度指标伸缩，多个指标任意触发阈值即扩容，全部低于阈值才缩容，适配复杂高并发场景。

### 5.5 HPA 常见问题与排错（伸缩不及时/抖动）

常见问题：扩容滞后导致压垮服务、缩容过快导致抖动。解决方案：调整伸缩冷却时间、配置预热策略、多指标兜底。

### 5.6 生产环境HPA最佳实践

核心业务必须配置HPA；配置合理冷却时间避免抖动；多指标兜底伸缩；设置最大最小副本数保护业务稳定性。
