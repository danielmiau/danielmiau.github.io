# 03-核心资源对象

## 本章概述

### 本章核心价值与学习目标

前面章节我们掌握了 K8s 的底层架构、组件交互、设计理念。而 **Kubernetes 所有功能落地，全部依赖资源对象**。K8s 官方定义：**K8s 一切皆资源**，所有集群管理、应用部署、网络配置、存储挂载、权限管控，本质都是对资源对象的增删改查。

本章是整套教程**实操落地的核心章节**，也是面试占比最高、生产使用最频繁的章节。学完本章，你将具备独立编写生产级 YAML、根据业务场景选择合适控制器、排查资源异常、规范集群资源管理的能力，彻底摆脱只会敲命令、不懂原理、不会落地的问题。

**学习目标**：

- 吃透所有高频核心资源的原理、场景、配置、坑点

- 区分各类控制器差异，掌握无状态/有状态任务场景选型

- 掌握网络、配置、存储资源，实现完整微服务部署架构

- 熟练编写生产级 YAML，适配迭代发布、数据持久化、网络隔离需求

### 前置知识依赖

需要掌握上一章核心知识点：K8s 控制面/数据面组件、声明式 API、面向终态、控制器调谐循环、集群基础网络原理。

### 本章知识地图

本章将资源分为四大层级，层级自上而下依赖：

1. **基础资源**：Namespace、Pod、Label（集群最小原子单元）

2. **控制器资源**：Deployment/StatefulSet/DaemonSet/Job，用于管理Pod生命周期

3. **网络资源**：Service/Ingress/NetworkPolicy，负责流量接入与隔离

4. **配置存储资源**：ConfigMap/Secret/PV/PVC，实现配置解耦与数据持久化

# 1. 基础资源（K8s的原子单位）

## 1.1 Namespace：资源隔离与多租户

### 概念定义

Namespace（命名空间）是 K8s 提供的**逻辑资源隔离单元**。可以通俗理解为：集群是一栋大楼，Namespace 就是大楼里的独立房间。不同房间的资源互相隔离，实现集群多租户、多环境、多业务线的资源分组管理。

Namespace **仅隔离资源对象，不隔离网络与节点**，属于软隔离，并非安全级强隔离。

### 核心原理

K8s 集群默认所有资源全局共享，通过 Namespace 对资源做逻辑分组。所有资源都归属某一个命名空间，不同命名空间资源名称可以重复，不会冲突。控制面组件通过命名空间字段区分资源归属，实现分组管理。

### 场景与价值

- **环境隔离**：划分 dev/test/prod 环境，一套集群承载多套环境

- **业务隔离**：不同业务线使用独立命名空间，避免资源混淆

- **权限隔离**：配合 RBAC 实现不同团队仅能操作自己命名空间资源

- **资源配额**：针对命名空间配置资源上限，防止单业务占满集群资源

### 默认Namespace详解

- **default**：默认命名空间，不指定namespace的资源全部归属此处

- **kube-system**：集群核心组件资源专属命名空间，禁止用户部署业务

- **kube-public**：公共命名空间，资源全局可读，用于存放集群公共配置

### 实操示例

创建 namespace.yaml

```yaml
# 定义开发环境命名空间
apiVersion: v1
kind: Namespace
metadata:
  name: dev # 命名空间名称
  labels:
    env: dev # 环境标签，用于后续权限、配额筛选
```

```bash
# 创建命名空间
kubectl apply -f namespace.yaml

# 查看所有命名空间
kubectl get ns

# 临时切换默认命名空间
kubectl config set-context --current --namespace=dev

# 删除命名空间（会删除内部所有资源，生产慎用）
kubectl delete ns dev

```

### 验证与排错

- 验证：执行 `kubectl get pods -n dev` 查看对应环境资源

- 常见报错：删除命名空间一直 Terminating，通常是存在残留资源或准入控制器拦截，需清理残留资源

### 避坑指南/最佳实践

- ⚠️ 禁止所有业务部署在 default 命名空间，资源混乱无法管理

- ⚠️ 禁止删除 kube-system 命名空间，会直接导致集群瘫痪

- 💡 生产最佳实践：**按环境+业务线划分命名空间**，例如：dev-user、prod-order

### 面试高频题

**问题**：Namespace 能不能隔离网络？可以实现强安全隔离吗？

**参考答案**：Namespace 仅做资源逻辑隔离，无法隔离 Pod 网络通信，属于软隔离。如果需要网络安全隔离，必须配合 NetworkPolicy；如需强安全租户隔离，需要使用独立集群或虚拟集群方案。

## 1.2 Pod：K8s中最小的调度单元

### 概念定义

**Pod 是 K8s 最小、最基础的调度单元**，不可再拆分。一个 Pod 本质是一组**共享网络栈、共享存储卷、同节点部署**的多个容器集合。

通俗理解：容器是程序，Pod 是容纳程序的“运行包裹”，K8s 不直接管理容器，统一管理 Pod。

### 核心原理

同一个 Pod 内所有容器共享：Network Namespace（统一IP、端口、网卡）、存储卷、PID命名空间。多个容器部署在同一个 Pod，适合**辅助耦合场景**（日志采集、代理、监控），不适合独立微服务。

### 场景与价值

- 统一调度原子单元，标准化所有应用部署方式

- 支持 sidecar 架构，无侵入增强业务能力（日志、监控、网关代理）

- 通过重启策略、健康检查，实现基础自愈能力

### Pod生命周期阶段

- **Pending**：资源创建成功，未调度到节点、镜像拉取中

- **Running**：Pod 调度完成，所有容器启动成功

- **Succeeded**：一次性任务执行完成，正常退出

- **Failed**：容器异常退出、启动失败

### 实操示例：标准Pod配置

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: java-demo-pod # Pod名称
  namespace: dev # 归属命名空间
  labels:
    app: java-demo
    env: dev
spec:
  restartPolicy: Always # 重启策略
  containers:
  - name: java-demo # 容器名称
    image: nginx:1.21 # 镜像地址
    ports:
    - containerPort: 80 # 容器内部端口

```

```bash
# 创建pod
kubectl apply -f pod.yaml

# 查看pod详细状态，排查异常
kubectl describe pod java-demo-pod -n dev

```

### 重启策略详解

- **Always**：容器退出永远重启，适合常驻业务服务（默认）

- **OnFailure**：仅异常退出重启，正常退出不重启，适合任务类程序

- **Never**：无论是否异常，绝不重启，适合一次性离线任务

### 特殊Pod场景

**静态Pod（Static Pod）**：由节点 kubelet 直接管理，不经过 apiserver，常用于部署集群核心组件，仅本机可见。

**Init容器**：Pod 启动前置容器，必须执行成功完成后，业务容器才会启动，用于初始化配置、等待依赖服务就绪。

### 避坑指南

- ⚠️ 禁止将多个独立微服务部署在同一个 Pod，耦合度太高，无法单独扩容发布

- ⚠️ Pod 是短暂易变资源，随时可能重建，IP 不固定，禁止依赖Pod IP通信

### 面试高频题

**问题**：Pod 中可以运行多个容器吗？什么场景使用多容器 Pod？

**参考答案**：可以。同一Pod内多个容器共享网络与存储。仅适用于业务容器+辅助容器的强耦合场景，如日志采集、流量代理、监控探针。独立微服务必须单Pod单容器，保证服务解耦、独立迭代扩容。

## 1.3 Label与Selector：资源的标签与选择器

### 概念定义

**Label（标签）**：附着在 K8s 资源上的键值对元数据，相当于资源的“自定义属性”。

**Selector（选择器）**：通过标签筛选匹配指定资源，实现资源关联绑定。

### 核心原理

K8s 所有资源无固定强关联关系，全部通过 **Label+Selector 软关联**。例如 Service 通过标签匹配后端 Pod、调度器通过标签筛选节点，实现高度解耦。

### 场景与价值

- 资源分组筛选、环境区分、版本区分

- Service 绑定后端 Pod 实例

- 节点调度亲和、资源筛选

- 灰度发布、蓝绿发布筛选指定版本实例

### 常用Label规范

- env：环境标识 dev/test/prod

- app：业务应用名称

- version：版本号

- role：组件角色

### Selector分类

- **等式Selector**：key=value、key!=value，精准匹配

- **集合Selector**：key in (v1,v2)、key not in ()，范围匹配

### 实操示例

```bash
# 给pod打标签
kubectl label pod java-demo-pod env=test -n dev

# 查询指定标签资源
kubectl get pods -n dev -l env=dev

```

### 避坑指南

⚠️ Service、Deployment 依赖标签匹配 Pod，标签修改错误会导致服务失联、流量丢失、Pod 脱离管控。

# 2. 控制器资源（实现声明式运维的核心）

## 2.1 控制器模式核心原理

### 概念定义

控制器是 K8s 内置的**后台循环控制程序**，专门用于持续监控资源状态，保证集群实际状态无限趋近用户定义的期望状态，是 K8s 自愈、自动化运维的核心。

### 核心原理

所有控制器遵循统一逻辑：**监听资源变更 → 对比期望/实际状态 → 执行调谐修复**。用户无需手动管理 Pod，控制器全自动维护副本数量、运行状态、版本。

### 控制器分类

- 无状态控制器：Deployment

- 有状态控制器：StatefulSet

- 守护进程控制器：DaemonSet

- 任务类控制器：Job、CronJob

## 2.2 无状态应用控制器

### 2.2.1 Deployment：最常用的无状态控制器

#### 概念定义

Deployment 是生产**默认首选**的无状态应用控制器，专门用于管理微服务、Web服务等无状态常驻业务，依托 ReplicaSet 实现副本管理、滚动升级、版本回滚。

#### 核心原理

Deployment 不直接管理 Pod，而是管理 ReplicaSet。每次版本更新会生成新 ReplicaSet，逐步创建新 Pod、销毁旧 Pod，实现滚动发布，保证服务**零宕机更新**。

#### 实操示例

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: java-demo-deploy
  namespace: dev
spec:
  replicas: 3 # 期望副本数
  selector:
    matchLabels:
      app: java-demo # 匹配pod标签
  strategy:
    type: RollingUpdate # 滚动更新策略
    rollingUpdate:
      maxSurge: 1 # 最大超出副本数
      maxUnavailable: 0 # 最大不可用副本数
  template: # pod模板
    metadata:
      labels:
        app: java-demo
    spec:
      containers:
      - name: java-demo
        image: nginx:1.21
        ports:
        - containerPort: 80

```

#### 升级策略对比

- **RollingUpdate**：滚动更新，渐进替换Pod，零停机（生产默认）

- **Recreate**：先销毁所有旧Pod，再创建新Pod，会短暂宕机（仅测试使用）

#### 避坑指南

maxUnavailable 设置大于0会导致发布过程中部分服务不可用，生产核心服务建议配置为0，保证发布稳定性。

### 2.2.2 ReplicaSet：Pod副本控制器

ReplicaSet 核心作用是维持指定数量的 Pod 副本，保证副本不缺失。**生产禁止直接使用 ReplicaSet**，仅作为 Deployment 底层依赖组件，不支持版本更新与回滚。

## 2.3 有状态应用控制器：StatefulSet

### 概念定义

StatefulSet 是专门用于**有状态服务**的控制器，适用于数据库、Redis、MQ、分布式存储等需要固定网络标识、有序启动、数据持久化的业务。

### 核心特性

- 稳定唯一的主机名与网络标识

- 有序部署、有序扩容、有序删除

- 支持独立持久化存储，Pod重建数据不丢失

### 面试高频题

**问题**：Deployment 和 StatefulSet 区别？如何选型？

**参考答案**：Deployment 用于无状态服务，Pod 无固定标识、无序、存储共享；StatefulSet 用于有状态服务，Pod 有序、主机名固定、独立持久化存储。微服务统一用 Deployment，数据库、中间件集群用 StatefulSet。

## 2.4 守护进程控制器：DaemonSet

### 概念定义

DaemonSet 保证集群每一个符合条件的节点上**仅运行一个Pod**，节点新增自动部署、节点下线自动销毁。

### 适用场景

日志采集（fluent-bit）、监控代理（prometheus-node-exporter）、网络插件、安全审计组件。

## 2.5 任务类控制器

### 2.5.1 Job：一次性任务控制器

用于离线批处理、数据迁移、一次性脚本执行。Job 会自动重试失败任务，任务执行成功后 Pod 常驻完成状态，不会自动删除。

### 2.5.2 CronJob：定时任务控制器

基于 Cron 表达式周期性创建 Job，用于数据备份、报表统计、定时清理。支持三种并发策略：

- Forbid：禁止并发，上一次未完成则跳过本次

- Replace：替换上一次未完成任务

- Allow：允许任务并发执行

⚠️ 生产最大坑点：K8s CronJob **默认时区为UTC**，需要手动配置时区，否则定时时间偏差8小时。

# 3. 网络资源（服务发现与流量入口）

## 3.1 Service：Pod的统一访问入口

### 概念定义

Pod 动态变化、IP 不固定，无法直接对外提供稳定访问。**Service 是一组Pod的稳定统一入口**，提供固定 IP、四层负载均衡、自动服务发现。

### 四种Service类型

- **ClusterIP**：默认类型，仅集群内部可访问

- **NodePort**：暴露节点端口，外网可通过节点IP+端口访问

- **LoadBalancer**：对接云厂商LB，生产外网标准方案

- **ExternalName**：映射集群外域名，用于访问外部服务

### 无头服务（Headless Service）

ClusterIP 设置为 None，不提供负载均衡，仅提供 DNS 解析，专门配合 StatefulSet 实现有状态实例独立域名访问。

## 3.2 Ingress：集群外部流量入口

### 概念定义

Service 仅支持四层TCP/UDP负载，Ingress 是集群**七层HTTP/HTTPS网关**，支持域名路由、路径匹配、SSL证书、灰度、限流。

访问链路：**用户 → Ingress → Service → Pod**

### 生产实践

生产统一使用 Ingress-Nginx/APISIX 作为 Ingress Controller，统一管理所有业务域名与证书。

## 3.3 网络策略：Pod间访问控制

### 概念定义

NetworkPolicy 是集群内部的**Pod级防火墙**，基于标签、命名空间配置白名单规则，控制Pod入站、出站流量，实现微服务隔离、多租户安全隔离。

⚠️ 仅 Calico 等专业CNI插件支持，Flannel 不支持网络策略。

# 4. 配置与存储资源（解耦应用与环境）

## 4.1 ConfigMap：非敏感配置管理

### 概念定义

ConfigMap 用于存放**非敏感配置数据**（配置文件、参数、环境变量），实现配置与代码镜像解耦，无需重新构建镜像即可修改配置。

### 使用方式

- 环境变量注入：适合少量参数

- 文件挂载：适合完整配置文件

### 避坑指南

ConfigMap 更新后，**挂载的配置文件会热更新，但环境变量不会更新**，需要重启Pod生效。

## 4.2 Secret：敏感数据管理

### 概念定义

Secret 专门存放密码、密钥、Token、SSL证书等敏感数据，避免硬编码泄露。默认基于 base64 编码，配合集群加密插件可实现数据加密存储。

## 4.3 存储资源：Volume与PV/PVC体系

### 4.3.1 Volume：容器数据卷

容器文件系统随Pod销毁丢失，Volume 用于挂载外部存储，实现数据持久化。常用类型：

- EmptyDir：临时存储，Pod销毁数据丢失

- HostPath：挂载宿主机目录，单机存储

- ConfigMap/Secret：配置挂载

### 4.3.2 PV与PVC：持久化存储体系

**PV**：集群级别存储资源（磁盘）

**PVC**：用户存储申请（申领磁盘）

StorageClass 实现**动态存储供给**，无需管理员手动创建PV，生产标准持久化方案。

访问模式、回收策略、存储类型是面试与生产故障高频考点，可有效解决数据丢失、挂载失败问题。

# 本章总结

本章完整覆盖 K8s 全部**核心高频资源对象**，是日常运维、项目部署、面试答辩的绝对核心。

**核心知识点复盘**：

1. Namespace 做逻辑隔离，Pod 是最小调度单元，Label 实现资源软关联

2. 控制器选型：无状态 Deployment、有状态 StatefulSet、守护 DaemonSet、任务 Job/CronJob

3. 网络分层：Service 四层负载、Ingress 七层网关、NetworkPolicy 内部隔离

4. 配置存储：ConfigMap/Secret 解耦配置，PV/PVC 标准化持久化存储

**面试高频汇总**：控制器区别、发布策略、Service四种类型、存储回收策略、Pod重启策略。

**生产最佳实践汇总**：环境拆分命名空间、微服务统一Deployment、外网流量走Ingress、敏感数据用Secret、业务数据走PVC动态存储。
