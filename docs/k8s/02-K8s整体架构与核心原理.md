# 1. K8s诞生背景与优势

## 1.1 容器编排的演进历史

### 概念定义

**容器编排**：指自动化管理集群中大量容器的生命周期，包含容器创建、调度、发布、自愈、扩缩容、故障迁移、负载均衡的整套自动化机制。简单来说：Docker 负责“跑单个容器”，编排工具负责“管成千上万个容器”。

### 核心原理

微服务架构落地后，一个业务会拆分为数十上百个微服务，每个服务需要多实例部署。人工手动 Docker 启动、重启、更新容器完全无法支撑业务迭代，必须依赖自动化编排系统统一管理容器集群，这就是容器编排技术诞生的根本原因。

### 场景与价值

无编排场景下，微服务存在大量致命问题：服务挂掉无人重启、流量暴涨无法扩容、版本更新容易宕机、多节点容器通信混乱、运维成本极高。容器编排统一解决微服务集群运维自动化问题。

### 早期编排工具对比

|编排工具|特点|短板|
|---|---|---|
|Docker Swarm|简单轻量、和Docker深度集成、上手快|功能简单、生态弱、高可用差、不适合大规模集群|
|Apache Mesos|资源调度能力极强、适合超大规模集群|架构复杂、运维成本极高、上手难度大|
|Google Borg|谷歌内部生产级编排系统，稳定性极强|闭源、无法商用|
### K8s的诞生

Kubernetes（简称 K8s）由 Google 基于内部 Borg 系统开源重构，2015 年正式开源并捐赠给 CNCF 基金会。继承了 Borg 的生产稳定性，同时解决了 Swarm 功能弱、Mesos 太复杂的缺点，凭借**开源、中立、可扩展、功能齐全、生态强大**迅速成为容器编排事实标准。

### 避坑指南/最佳实践

- 不要使用 Docker Swarm、Mesos 做新项目容器编排，目前基本停止迭代，生态枯竭

- 所有云原生微服务项目，统一基于 K8s 作为底层编排平台

### 面试高频题

**问题**：为什么微服务必须使用 K8s？Docker 单机部署为什么不适合生产？

**参考答案**：Docker 仅能实现单机容器运行，缺少集群调度、自愈、扩缩容、滚动更新、服务发现能力。微服务实例多、迭代快、故障频繁，必须依赖 K8s 统一编排管理，实现运维自动化、服务高可用。

## 1.2 K8s解决的核心痛点

### 概念定义与原理

K8s 并非容器运行工具，而是**容器集群操作系统**，全方位解决传统部署、单机容器部署的所有生产痛点。

### 场景价值详解

- **容器编排与调度**：自动将容器调度到资源充足、负载较低的节点，无需人工干预，最大化集群资源利用率

- **服务发现与负载均衡**：解决容器 IP 不固定、服务重启后地址变化的问题，内置 Service 四层负载、Ingress 七层负载

- **自动扩缩容与故障自愈**：容器崩溃、节点宕机自动重启、迁移实例；流量波动自动增减副本，保障业务稳定

- **滚动升级与回滚**：版本更新不宕机，更新失败一键回滚，彻底解决发布事故

- **配置与秘钥管理**：统一管理配置文件、账号密码、证书，实现配置与代码解耦、配置热更新

### 避坑指南

很多初学者误区：K8s 是容器。正确认知：**Docker/containerd 是容器运行时，K8s 是管理容器的集群控制系统**。

## 1.3 K8s的核心优势与适用场景

### 优势对比

|部署方式|可用性|运维成本|弹性能力|迭代效率|
|---|---|---|---|---|
|物理机/虚拟机|低，依赖人工运维|极高|无弹性|慢|
|单机Docker|低，无自愈能力|高|弱|中|
|K8s集群|极高，自动化自愈|低，自动化运维|极强，HPA弹性伸缩|极高|
### 适用场景

- ✅ 适合：微服务架构、高频迭代业务、互联网高并发业务、多云混合部署、需要弹性伸缩的业务

- ❌ 不适合：简单单体程序、长期不迭代、流量极低、无容器化改造能力的老旧系统（引入 K8s 会过度增加架构复杂度）

# 2. K8s整体架构分层

K8s 采用经典的**控制面+数据面**分层架构，架构完全解耦、无状态、可横向扩展，是云原生最核心的架构设计。通俗类比：控制面是“大脑”，负责决策调度；数据面是“四肢”，负责真正执行业务、运行容器。

## 2.1 控制面（Control Plane）核心组件

### 概念定义

控制面部署在 Master 节点，不运行业务容器，只负责**集群管理、调度、决策、数据存储、权限校验**，是整个集群的指挥中心。

### 组件详解

#### kube-apiserver

**核心定位：集群唯一入口，所有组件的网关**

- 接收所有请求：kubectl、各组件、外部客户端所有读写请求

- 认证授权：校验用户权限、校验资源操作合法性

- 准入控制：通过内置/自定义 Webhook 拦截资源请求

- 无状态设计：支持多副本高可用部署

#### etcd

**核心定位：集群唯一数据库**

分布式键值数据库，集群所有资源对象（Pod、Deployment、Service、配置、状态）全部持久化存储在 etcd。仅 apiserver 可读写 etcd，其他组件全部通过 apiserver 交互，保证数据一致性。

#### kube-scheduler

**核心定位：集群调度决策者**

监听未调度的 Pod，根据节点资源、亲和策略、污点策略，筛选最优节点，将 Pod 绑定到目标节点。

#### kube-controller-manager

**核心定位：集群调谐控制器合集**

内置大量控制器：副本控制器、节点控制器、端点控制器、命名空间控制器等，持续循环对比期望状态与实际状态，自动修复集群资源。

#### cloud-controller-manager（可选）

对接公有云厂商资源，管理云负载均衡、云磁盘、弹性网卡，自建机房集群一般无需部署。

### 实操示例：查看控制面组件

```bash
# 查看master节点核心组件pod
kubectl get pods -n kube-system

```

### 避坑指南

etcd 极其重要，**一旦损坏集群数据全部丢失**，生产必须做定时备份、三副本高可用部署，禁止单节点 etcd 跑生产。

## 2.2 数据面（Node）核心组件

### 概念定义

数据面即 Worker 工作节点，专门负责运行用户业务容器，执行控制面下发的调度与管理指令。

### 组件详解

#### kubelet

每个工作节点必备的代理进程，是节点和控制面的通信桥梁。负责监听 apiserver 任务，管理本机 Pod 生命周期：创建、启动、停止、健康检查、状态上报。

#### kube-proxy

节点网络代理，负责在节点上生成 iptables/ipvs 规则，实现 Service 负载均衡、集群内部网络转发。

#### 容器运行时

遵循 CRI 标准，负责真正拉取镜像、创建容器、启停容器，主流为 containerd，已全面替代 Docker。

#### 节点插件

CNI：网络插件，解决跨节点 Pod 通信；CSI：存储插件，解决容器持久化存储挂载。

## 2.3 控制面与数据面的协作关系

### 核心原理

所有数据面组件**不会主动请求控制面**，全部采用 **List-Watch 监听机制**：kubelet、kube-proxy 持续监听 apiserver 资源变更，被动接收调度指令，实现控制面与数据面完全解耦。

### 场景价值

组件无状态、解耦设计，让集群支持横向扩容、滚动升级、节点随时上下线，不会影响集群整体稳定性。

# 3. K8s核心设计理念

## 3.1 声明式API（Declarative API）

### 概念定义

**命令式**：告诉系统“做什么动作”（例如：创建一个容器、停止容器）

**声明式**：告诉系统“最终想要什么状态”，系统自动计算如何达成状态

### 核心原理

K8s 所有资源均采用声明式定义。用户编写 YAML 定义资源期望状态，控制器持续对比**期望状态 vs 实际状态**，自动执行调谐逻辑，抹平状态差异。

### 对比场景

命令式缺点：操作不可逆、无法自愈、无法版本管理、人为操作容易出错；声明式优势：可版本管理、可自愈、可自动化、适合持续集成部署。

## 3.2 控制器模式与水平扩展

### 核心原理

控制器本质是**死循环调谐程序**，逻辑极其简单：监听资源变化 → 对比状态 → 修复差异。所有控制面组件无状态，支持多副本部署，实现水平扩容、高可用。

## 3.3 可插拔、可扩展的架构设计

### 概念定义

K8s 内核极度精简，所有扩展能力全部通过插件实现，遵循**内核极简，能力外扩**的设计思想。

### 四大标准插件

- CRI：容器运行时插件

- CNI：网络插件

- CSI：存储插件

- Device Plugin：硬件设备插件

同时支持 CRD 自定义资源、Webhook 准入扩展，极大丰富集群能力。

## 3.4 面向终态（Desired State）的设计思想

面向终态是 K8s 最核心、最灵魂的设计。无论集群发生什么故障（容器挂掉、节点宕机、配置丢失），控制器永远会持续不断地修复资源，让集群**无限趋近用户定义的期望状态**，实现天然自愈、高可用。

# 4. 请求完整链路（从kubectl到Pod运行）

本节完整拆解用户执行 kubectl 创建 Pod 的全链路，打通所有组件交互逻辑，是排查集群故障的核心基础。

## 4.1 客户端请求流程

1. 用户执行 `kubectl apply -f xxx.yaml`

2. kubectl 解析 yaml，封装 HTTP 请求发送给 apiserver

3. apiserver 完成认证、授权、准入控制

4. 校验通过后将资源数据写入 etcd

5. apiserver 触发事件通知所有监听组件

## 4.2 控制器调度流程

1. kube-controller-manager 监听到新增 Pod 资源

2. 若无绑定节点，交由 kube-scheduler 调度

3. **过滤阶段**：根据污点、亲和、资源限制筛选合格节点

4. **打分阶段**：根据节点负载、资源使用率打分

5. 选出最优节点，完成 Pod 节点绑定

## 4.3 节点执行流程

1. 目标节点 kubelet 监听 Pod 绑定事件

2. kubelet 调用 CRI 接口，通知 containerd

3. containerd 拉取镜像，调用 runc 创建容器

4. kube-proxy 监听 Service 变化，更新节点网络规则

5. kubelet 持续探测容器健康状态，上报集群

## 4.4 关键组件瓶颈与故障点

- apiserver：集群流量瓶颈，大规模集群需要多副本负载均衡

- etcd：单点故障源，必须高可用+定时备份

- kube-scheduler：调度策略不合理会导致节点负载不均

# 5. 集群网络模型

## 5.1 K8s网络模型的核心设计目标

K8s 统一标准化网络模型，强制满足四大通信能力：

- 任意节点之间可以互通

- 任意 Pod 之间（跨节点）可以直接互通，无需 NAT

- Pod 与 Service 稳定通信

- 支持外部流量安全访问集群内部服务

## 5.2 网络模型的实现基础

所有 Pod 拥有集群内唯一独立 IP。Service 提供稳定不变的 ClusterIP，屏蔽后端 Pod IP 变化。kube-proxy 三种工作模式：userspace（废弃）、iptables（默认稳定）、ipvs（高性能生产首选）。

### 实操：查看kube-proxy模式

```bash
# 查看kube-proxy配置
kubectl get configmap kube-proxy -n kube-system -o yaml

```

## 5.3 CNI网络插件基础

CNI 是容器网络标准接口，解耦 K8s 内核与网络实现。

|插件|优势|劣势|适用场景|
|---|---|---|---|
|Flannel|简单稳定、运维简单|无网络隔离、性能一般|测试、小型集群|
|Calico|支持网络策略、隔离性强、性能好|配置复杂|生产集群首选|
|Weave|部署简单|性能较差|极少使用|
## 5.4 集群网络的核心问题与演进

原生 K8s 网络仅解决四层通信，复杂的限流、熔断、灰度、链路追踪能力不足，因此衍生出 **Service Mesh 服务网格（Istio）**，作为 K8s 网络能力的上层扩展。

# 6. 集群生命周期与高可用原理

## 6.1 K8s集群的生命周期阶段

集群完整生命周期：初始化部署 → 日常运行调度 → 版本迭代升级 → 资源备份维护 → 集群下线销毁。生产集群必须全程保证稳定性，杜绝不可逆故障。

## 6.2 控制面高可用实现

- **etcd高可用**：基于 Raft 一致性算法，三副本部署，半数节点故障不影响集群

- **apiserver高可用**：无状态多副本，通过负载均衡统一入口

- **控制器/调度器高可用**：多副本部署，通过领导者选举，同一时间只有一个实例工作，其余待命

## 6.3 数据面高可用与故障自愈

- 节点故障：节点控制器检测节点失联，自动驱逐 Pod 至健康节点

- 容器故障：kubelet 通过就绪、存活探针检测异常，自动重启容器

- 副本异常：控制器自动扩缩副本，维持期望副本数

## 6.4 集群备份与恢复

### 实操：etcd备份命令

```bash
# etcd数据备份
ETCDCTL_API=3 etcdctl snapshot save /data/etcd-backup-$(date +%Y%m%d).db

```

生产最佳实践：每日自动备份 etcd、定期做灾难恢复演练，保证集群故障可恢复。
