# 06-Spring 事务管理机制

## 本章概述

事务是业务系统数据一致性的核心保障，几乎所有涉及数据库增删改的业务场景，都离不开事务的支撑。原生数据库事务功能基础、代码冗余、耦合度高，无法适配复杂的业务分层、嵌套调用、环境适配场景。

Spring 事务机制基于**AOP动态代理**实现，对原生JDBC事务做了顶层抽象与全面封装，提供了声明式事务、七大传播机制、多级隔离级别、统一回滚规则等核心能力。彻底解决了原生事务代码冗余、硬编码、灵活性差的问题，是Java后端开发的核心基础技术，也是面试高频重难点。

本章将从**基础概念、底层架构、核心特性、源码流程、失效场景、生产实践**全方位讲解Spring事务，兼顾原理理解、代码实操、面试背诵、生产避坑，帮助读者彻底吃透Spring事务核心，解决工作中99%的事务异常问题。

# 1. Spring 事务核心概念与核心价值

## 1.1 什么是事务、事务四大特性 ACID

### 1.1.1 事务核心定义

事务（Transaction）是数据库操作的**最小执行单元**，是一组不可分割的SQL操作集合。这一组SQL要么**全部执行成功并提交**，要么**全部执行失败并回滚**，绝对不会出现部分成功、部分失败的中间状态，以此保证业务数据的一致性。

典型场景：转账业务，A用户扣款、B用户加款，两条SQL必须绑定为一个事务。若扣款成功、加款失败，必须整体回滚，避免资金数据错乱。

### 1.1.2 事务四大特性 ACID（面试必背）

ACID是事务的四大核心特性，是所有关系型数据库事务的标准规范，Spring事务完全遵循该规范：

- **原子性（Atomicity）**：事务是一个不可分割的原子操作单元。所有SQL操作要么全部成功提交，要么全部失败回滚，不存在中间执行状态，是事务最核心的特性。

- **一致性（Consistency）**：事务执行前后，数据库的完整性约束（主键、唯一索引、外键、业务规则）不会被破坏。例如转账前后，总资金数额保持不变。

- **隔离性（Isolation）**：多个并发执行的事务之间相互隔离、互不干扰。数据库通过隔离级别控制事务并发可见性，解决脏读、不可重复读、幻读等并发问题。

- **持久性（Durability）**：事务一旦提交成功，数据会永久写入磁盘数据库，即使服务器宕机、重启，数据也不会丢失。

### 💡最佳实践

开发中所有涉及**数据变更、资金流转、库存扣减、订单创建**的复杂业务，必须开启事务，杜绝数据不一致问题。

## 1.2 数据库原生事务与 Spring 事务区别

很多开发者误以为Spring事务是全新的事务机制，实则不然：**Spring事务底层完全依赖数据库原生事务**，Spring仅做了上层封装、规则扩展、流程管控。二者是「底层实现」与「上层封装」的关系。

### 1.2.1 数据库原生事务（JDBC原生）

JDBC原生事务是数据库提供的最基础事务能力，核心API如下：

```java
// 关闭自动提交，开启事务
connection.setAutoCommit(false);
try {
    // 执行多条数据库操作SQL
    // ...
    // 全部执行成功，手动提交事务
    connection.commit();
} catch (Exception e) {
    // 出现异常，手动回滚事务
    connection.rollback();
} finally {
    // 释放资源
    connection.close();
}
```

**原生事务痛点**：

- 代码极度冗余，每个业务方法都需要重复写try-catch、commit、rollback模板代码

- 事务逻辑与业务逻辑强耦合，代码可读性、维护性极差

- 无事务传播机制、无隔离级别精细化控制

- 异常回滚规则死板，仅支持所有异常回滚，无法自定义

- 硬编码实现，无法统一管控、统一配置

### 1.2.2 Spring 事务

Spring事务基于AOP动态代理，对JDBC原生事务进行**全方位封装增强**，保留原生ACID特性的同时，解决了原生事务的所有痛点。

### 1.2.3 核心区别对比表

|对比维度|数据库原生事务|Spring 事务|
|---|---|---|
|代码耦合度|高，事务代码侵入业务|极低，无侵入式声明|
|编码方式|硬编码手动开启、提交、回滚|注解一键开启（@Transactional）|
|事务传播|不支持|支持7种传播机制|
|隔离级别|仅支持数据库默认级别|可精细化配置5种隔离级别|
|异常回滚|所有异常统一回滚，无法自定义|可指定回滚/不回滚异常|
|统一管控|无法统一配置，分散编码|全局统一配置、统一管控|
## 1.3 Spring 事务核心优势：统一抽象、无侵入、声明式事务

### 1.3.1 统一抽象架构

Spring 设计了一套**独立的事务顶层抽象接口**（PlatformTransactionManager、TransactionDefinition、TransactionStatus），彻底屏蔽不同持久层框架的差异。

无论项目使用MyBatis、JPA、Hibernate还是JDBC，上层事务使用方式完全一致，无需适配不同框架的事务API，实现了**一次编码、多框架适配**。

### 1.3.2 无侵入式增强

Spring事务基于AOP动态代理实现，遵循**开闭原则**。业务代码无需修改任何逻辑，仅需添加 **@Transactional** 注解，即可自动实现事务开启、提交、回滚、资源释放，完全解耦事务逻辑与业务逻辑。

### 1.3.3 丰富的扩展能力

Spring事务提供了原生事务不具备的超强扩展能力，适配复杂生产场景：

- 7种事务传播机制，完美适配嵌套业务调用

- 5种事务隔离级别，解决并发数据问题

- 支持事务超时、只读优化、自定义异常回滚

- 支持全局事务配置、统一异常管控

### 1.3.4 生产核心价值

极大提升开发效率、降低代码冗余、统一项目事务规范、减少人为事务BUG，是企业级项目标准事务解决方案。

## 1.4 编程式事务 vs 声明式事务（面试对比）

Spring 提供两种事务使用方式，分别适配简单场景和复杂业务场景，二者特性、适用场景、优缺点差异极大，是面试高频考点。

### 1.4.1 编程式事务

编程式事务是**手动编码控制事务**的方式，通过 Spring 提供的 TransactionTemplate 手动开启、提交、回滚事务，本质是对原生JDBC事务的轻度封装。

**代码示例**：

```java
@Autowired
private TransactionTemplate transactionTemplate;

public void transferMoney() {
    // 手动执行事务逻辑
    transactionTemplate.execute(status -> {
        try {
            // 业务数据库操作
            userMapper.deductMoney();
            userMapper.addMoney();
            return true; // 正常提交
        } catch (Exception e) {
            status.setRollbackOnly(); // 手动标记回滚
            return false;
        }
    });
}
```

**优缺点**：

- 优点：粒度极细，可精准控制代码片段事务，灵活度极高

- 缺点：代码冗余、侵入业务、维护成本高、无法统一管控

**适用场景**：极少数**局部小段代码**需要单独事务控制的特殊场景。

### 1.4.2 声明式事务

声明式事务是Spring主推的事务方案，基于**AOP注解**实现，通过 **@Transactional** 注解声明事务规则，无需手动编码，自动完成事务管控。

**代码示例**：

```java
@Service
public class UserService {

    // 一键开启事务，自动管控所有逻辑
    @Transactional(rollbackFor = Exception.class)
    public void transferMoney() {
        // 纯业务逻辑，无需事务代码
        userMapper.deductMoney();
        userMapper.addMoney();
    }
}
```

**优缺点**：

- 优点：无侵入、代码简洁、统一规范、支持所有高级特性（传播机制、隔离级别）

- 缺点：存在部分事务失效场景（后续章节详解）

**适用场景**：99%的企业级业务场景，是生产环境**唯一推荐**的事务使用方式。

### 1.4.3 面试标准对比总结

- **编程式事务**：手动编码、粒度细、冗余高、灵活、小众场景使用

- **声明式事务**：注解驱动、AOP实现、无侵入、功能强、生产主流

# 2. Spring 事务核心底层架构体系

## 2.1 事务核心三大顶层接口

Spring 事务的核心架构由**三大顶层接口**构成，所有事务功能、扩展、实现都基于这三个接口，是Spring事务的基石，也是理解底层源码的核心入口。三者分工明确、相互配合，支撑起整个Spring事务体系。

### 2.1.1 PlatformTransactionManager 事务管理器

**核心定位**：Spring事务的**核心执行器**，负责事务的真正开启、提交、回滚、资源管控，是事务的底层执行者。

这是Spring事务的顶层核心接口，所有事务操作最终都会调用该接口的实现类完成，屏蔽了不同数据库、不同持久层框架的差异。

**核心三大抽象方法**：

```java
// 根据事务定义，获取/创建事务，开启事务
TransactionStatus getTransaction(TransactionDefinition definition);

// 提交事务
void commit(TransactionStatus status);

// 回滚事务
void rollback(TransactionStatus status);
```

**核心作用**：

- 根据传播机制判断：复用旧事务、新建事务、挂起事务

- 执行事务提交、回滚核心操作

- 管理数据库连接资源、事务超时、锁资源释放

**💡面试考点**：Spring不直接管理事务，所有事务操作全部交由PlatformTransactionManager实现类执行。

### 2.1.2 TransactionDefinition 事务定义信息

**核心定位**：事务的**配置描述载体**，用来存储当前事务的所有配置属性，相当于事务的「配置文件」。

开发者在 @Transactional 注解中配置的所有参数，都会被解析封装为 TransactionDefinition 对象，供事务管理器使用。

**核心存储属性**：

- 事务传播机制（默认 REQUIRED）

- 事务隔离级别（默认数据库级别）

- 事务超时时间（默认-1，不超时）

- 是否为只读事务（默认false）

- 事务名称

**核心作用**：统一封装事务配置参数，让事务管理器根据配置参数，执行不同的事务逻辑，实现事务的精细化控制。

### 2.1.3 TransactionStatus 事务运行状态

**核心定位**：事务的**实时状态载体**，用来记录事务运行过程中的所有动态状态，相当于事务的「运行日志」。

事务开启后，Spring会生成该对象，全程记录事务的运行状态，用来判断后续是提交还是回滚。

**核心存储状态**：

- 是否是新创建的事务

- 事务是否已标记回滚

- 事务是否已完成（提交/回滚）

- 是否存在保存点（嵌套事务使用）

- 事务是否被挂起

**核心作用**：让事务管理器实时感知事务状态，根据状态动态执行提交、回滚、恢复、挂起逻辑。

## 2.2 Spring 事务架构整体执行流程

Spring事务整体架构遵循**「定义状态 → 管理执行 → 状态流转」**的核心逻辑，三大接口层层配合，完整执行流程如下：

1. **注解解析阶段**：开发者在方法上添加 @Transactional 注解，Spring启动时解析注解参数，封装为 **TransactionDefinition** 事务定义对象，记录事务配置规则。

2. **事务获取阶段**：方法被调用时，AOP拦截器触发事务逻辑，调用 **PlatformTransactionManager** 事务管理器，传入事务定义信息。

3. **事务创建阶段**：事务管理器根据传播机制、当前线程事务状态，判断复用旧事务、新建事务或挂起事务，生成 **TransactionStatus** 状态对象。

4. **业务执行阶段**：执行目标业务方法，执行业务SQL操作。

5. **事务收尾阶段**：业务执行无异常，事务管理器执行commit提交；出现异常，执行rollback回滚。

6. **资源释放阶段**：事务完成后，更新TransactionStatus状态，释放数据库连接、恢复挂起事务、清理资源。

**核心总结**：**TransactionDefinition（规则）+ TransactionStatus（状态）+ PlatformTransactionManager（执行）** = 完整Spring事务体系。

## 2.3 主流事务管理器实现类（DataSource、JPA、MyBatis）

PlatformTransactionManager 是顶层接口，Spring针对不同的持久层框架，提供了对应的**专属实现类**，自动适配不同数据库操作框架，无需开发者手动适配。

### 2.3.1 DataSourceTransactionManager（JDBC/MyBatis通用）

**适用场景**：使用原生JDBC、MyBatis、MyBatis-Plus的项目，是目前**生产环境最常用**的事务管理器。

**核心原理**：基于DataSource数据源，获取数据库连接，绑定事务，管控连接的提交、回滚、关闭操作。

**SpringBoot自动配置**：项目引入mybatis依赖后，SpringBoot会**自动装配**该事务管理器，无需手动配置。

### 2.3.2 JpaTransactionManager（JPA/Hibernate）

**适用场景**：使用Spring Data JPA、Hibernate的项目。

**核心原理**：适配JPA持久层框架，基于EntityManager管理事务，兼容JPA的缓存、持久化特性。

### 2.3.3 其他小众事务管理器

- HibernateTransactionManager：专属Hibernate框架

- JtaTransactionManager：支持分布式JTA事务，适配多数据源场景

### 2.3.4 生产最佳实践

95%的Java Web项目基于MyBatis开发，默认使用 **DataSourceTransactionManager**，SpringBoot自动装配生效，无需手动创建事务管理器Bean，开箱即用。

### 📌面试高频题

**问题**：Spring事务管理器需要手动配置吗？

**答案**：SpringBoot项目无需手动配置，引入对应持久层依赖后，框架会自动装配对应事务管理器；SSM传统项目需要手动注入DataSourceTransactionManager。

# 3. 事务传播机制

在实际业务开发中，业务逻辑往往不是单一方法，而是多层方法嵌套调用（例如A方法调用B方法、B方法调用C方法）。当多个带有事务的方法嵌套执行时，**父子方法的事务如何传递、是否共用事务、是否新建事务、异常是否互相影响**，全部由Spring的**事务传播机制**控制。

事务传播机制是Spring事务最核心、面试最高频的知识点，也是生产环境中事务失效、数据错乱的主要诱因。Spring一共定义了**七大事务传播机制**，每种机制对应不同的嵌套事务场景，开发者需要根据业务场景精准选型。

## 3.1 传播机制核心作用：控制父子事务关系

### 3.1.1 核心概念定义

事务传播机制，全称**事务传播行为（Propagation）**，是指当一个带有事务的方法被另一个方法调用时，**子方法如何继承、复用、创建、脱离父方法事务**的行为规则。

我们可以将嵌套调用的方法分为两类：**父方法（外部调用者）**和**子方法（内部被调用者）**。父方法是主动发起调用的外层方法，子方法是被嵌套调用的内层方法。传播机制的本质，就是定义**子方法面对父方法已有事务/无事务时的执行策略**。

### 3.1.2 底层设计原理

Spring事务基于AOP动态代理实现，当方法被调用时，Spring的事务拦截器会先检测当前线程是否存在**活跃事务**（即父方法开启的事务），再根据子方法配置的传播机制，执行对应的事务策略：复用当前事务、新建独立事务、放弃事务、强制事务等。

所有传播机制的判断逻辑，都围绕两个核心场景展开：

- **当前线程存在活跃事务（父方法有事务）**：子方法遵循对应规则融入父事务或独立运行

- **当前线程无活跃事务（父方法无事务）**：子方法遵循对应规则新建事务或无事务运行

### 3.1.3 生产价值与解决的问题

如果没有传播机制，嵌套方法的事务逻辑会完全混乱：外层方法回滚时内层方法是否回滚、内层异常是否导致外层事务失效、独立业务是否会被无关事务牵连等问题无法统一管控。

传播机制的核心价值：**精细化管控嵌套事务的隔离性和关联性**，让不同业务模块的事务互不干扰、关联业务数据一致性可控，完美适配复杂分层的业务架构。

### 3.1.4 面试高频考点

**问题**：什么是事务传播机制？核心作用是什么？

**参考答案**：事务传播机制是Spring定义的、用于规范嵌套方法调用时事务传递规则的机制。核心作用是控制父子方法嵌套调用时，子方法的事务创建、复用、隔离逻辑，解决多层业务嵌套场景下的事务一致性问题，是Spring事务适配复杂业务分层的核心设计。

## 3.2 七大传播机制逐条详解（场景+原理+案例）

Spring在`Propagation`枚举类中定义了七种传播行为，每种行为都有固定的执行逻辑和适配场景，下面逐条从**定义原理、执行规则、代码案例、生产场景、避坑要点**全方位解析。

### 3.2.1 REQUIRED（默认）

#### 1. 概念与核心原理

**REQUIRED**是Spring事务的**默认传播机制**，也是项目中使用最广泛的机制。

核心规则：**如果当前线程存在活跃事务，就加入当前事务，与父方法共用同一个事务；如果当前没有事务，就新建一个独立事务**。

简单来说：**有事务就共用，没事务就新建**。这是最贴合绝大多数业务的事务逻辑，保证关联业务操作在同一个事务中，实现数据一致性。

#### 2. 执行流程拆解

1. 父方法开启事务，调用子方法：子方法加入父事务，父子方法属于**同一个事务**，任意一方异常，整体全部回滚

2. 父方法无事务，调用子方法：子方法独立新建事务，自身操作遵循事务ACID特性

#### 3. 实操代码案例

场景：新增用户（父方法）+ 新增用户日志（子方法），两个操作必须同时成功或失败

```java
// 父方法：外层业务方法，默认REQUIRED
@Override
@Transactional
public void addUser() {
    // 1. 新增用户数据
    userMapper.insert(new User("张三", 20));
    // 2. 调用子方法新增日志
    logService.addUserLog();
    // 模拟异常，触发整体回滚
    int i = 1 / 0;
}

// 子方法：内层嵌套方法，默认REQUIRED
@Override
@Transactional(propagation = Propagation.REQUIRED)
public void addUserLog() {
    // 新增用户操作日志
    logMapper.insert(new UserLog("新增用户张三"));
}

```

**执行结果**：父方法抛出异常，父子方法共用同一个事务，用户数据和日志数据**全部回滚，无数据入库**，保证数据一致性。

#### 4. 生产适配场景

适用于**强关联业务场景**，多个操作必须同成功同失败，是90%的核心业务首选，例如：订单创建（生成订单+扣库存+扣余额）、用户注册（新增用户+初始化权限+生成账号）。

#### 5. 避坑指南

⚠️ 嵌套REQUIRED方法时，**子方法异常必须抛出到外层**，如果子方法内部try-catch捕获异常且不抛出，外层事务无法感知异常，会导致事务不回滚，出现数据不一致问题。

### 3.2.2 REQUIRES_NEW

#### 1. 概念与核心原理

**REQUIRES_NEW**是独立事务传播机制，核心规则：**无论当前线程是否存在活跃事务，都会新建一个全新的独立事务，暂停父方法的原有事务**。

执行逻辑：父方法有事务 → 挂起父事务 → 子方法新建独立事务执行 → 子方法执行完毕恢复父事务；父方法无事务 → 直接新建事务执行。

核心特点：**父子事务完全独立，互不影响**，子事务回滚不会导致父事务回滚，父事务回滚也不会影响已执行完成的子事务。

#### 2. 执行流程拆解

1. 父方法有事务：挂起父事务 → 子方法开启新事务执行 → 子事务提交/回滚 → 恢复父事务继续执行

2. 父方法无事务：子方法直接新建事务执行

#### 3. 实操代码案例

场景：核心订单业务（父事务）+ 日志记录（子事务），日志记录失败不影响订单创建成功

```java
// 父方法：核心订单业务
@Override
@Transactional
public void createOrder() {
    // 1. 创建订单核心数据
    orderMapper.insert(new Order("订单001", 99.9));
    try {
        // 2. 调用日志记录子方法（独立事务）
        logService.recordOrderLog();
    } catch (Exception e) {
        // 捕获日志异常，不影响订单业务
        log.error("日志记录失败", e);
    }
    // 父方法正常提交
}

// 子方法：日志记录，独立事务
@Override
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void recordOrderLog() {
    // 模拟日志记录异常
    int i = 1 / 0;
    logMapper.insert(new OrderLog("订单001创建成功"));
}

```

**执行结果**：子方法异常，独立事务回滚，日志数据不入库；父事务不受影响，订单数据**正常入库提交**。

#### 4. 生产适配场景

适用于**非核心附属业务**，附属操作失败不影响主业务，且附属操作需要独立事务保证自身一致性，例如：操作日志记录、消息推送、审计日志、业务埋点。

#### 5. 避坑指南

⚠️ REQUIRES_NEW 会频繁挂起、恢复事务，**性能损耗较高**，不可滥用；且必须通过外部类调用才会生效，同类方法内部调用会导致事务失效。

### 3.2.3 SUPPORTS

#### 1. 概念与核心原理

**SUPPORTS**是支持事务传播机制，核心规则：**被动适配事务，有事务就加入，无事务就以非事务方式执行**。

该机制不主动创建事务，完全依赖外层方法的事务环境，属于“随波逐流”型事务策略。

#### 2. 执行流程拆解

1. 父方法有事务：子方法加入父事务，共用同一个事务，异常互相影响

2. 父方法无事务：子方法无事务执行，操作自动提交，异常不会触发回滚

#### 3. 实操代码案例

```java
// 父方法：无事务
@Override
public void updateUser() {
    // 调用SUPPORTS子方法
    userService.updateUserName();
}

// 子方法：SUPPORTS传播机制
@Override
@Transactional(propagation = Propagation.SUPPORTS)
public void updateUserName() {
    // 无事务环境，修改直接提交
    userMapper.updateName(1L, "李四");
    // 此处异常，数据不会回滚
    int i = 1 / 0;
}

```

**执行结果**：父方法无事务，子方法非事务执行，修改数据**成功入库**，后续异常无法回滚。

#### 4. 生产适配场景

适用于**查询类方法、非核心更新方法**，方法既可以在事务中执行（适配业务嵌套），也可以单独无事务执行，通用性极强。例如：数据查询、基础数据更新。

#### 5. 避坑指南

⚠️ SUPPORTS在无事务环境下执行时，**异常不会触发回滚**，如果业务需要保证数据一致性，禁止使用该机制。

### 3.2.4 NOT_SUPPORTED

#### 1. 概念与核心原理

**NOT_SUPPORTED**是非事务传播机制，核心规则：**始终以非事务方式执行，如果外层存在事务，强制挂起父事务**。

无论父方法是否有事务，子方法都拒绝事务环境，全程无事务执行，操作执行后立即自动提交。

#### 2. 执行流程拆解

1. 父方法有事务：挂起父事务 → 子方法无事务执行 → 执行完成恢复父事务

2. 父方法无事务：子方法直接无事务执行

#### 3. 实操代码案例

```java
// 父方法：存在事务
@Override
@Transactional
public void batchOperate() {
    // 数据库更新操作
    userMapper.updateAge(1L, 25);
    // 调用NOT_SUPPORTED子方法
    logService.addTempLog();
    // 父方法抛出异常
    int i = 1 / 0;
}

// 子方法：强制非事务
@Override
@Transactional(propagation = Propagation.NOT_SUPPORTED)
public void addTempLog() {
    // 无事务执行，立即提交
    logMapper.insert(new UserLog("临时操作日志"));
}

```

**执行结果**：父方法事务回滚，用户年龄更新撤销；子方法无事务，日志数据**永久入库，不会回滚**。

#### 4. 生产适配场景

适用于**无需事务、允许数据独立提交的操作**，例如：临时日志记录、实时数据统计、缓存更新、非核心埋点数据。

#### 5. 避坑指南

⚠️ 该机制下所有操作**不支持回滚**，绝对不能用于核心业务数据的增删改操作，否则会出现数据脏数据。

### 3.2.5 MANDATORY

#### 1. 概念与核心原理

**MANDATORY**是强制事务传播机制，核心规则：**强制在事务环境中执行，如果外层无活跃事务，直接抛出异常**。

该机制不创建事务，只依赖外层事务，是强约束型事务策略，保证方法必须在事务中运行。

#### 2. 执行流程拆解

1. 父方法有事务：子方法加入父事务，共用事务执行

2. 父方法无事务：直接抛出`IllegalTransactionStateException`异常，方法执行失败

#### 3. 实操代码案例

```java
// 父方法：无事务
@Override
public void userOperate() {
    // 调用MANDATORY子方法，直接报错
    userService.saveUserInfo();
}

// 子方法：强制事务环境
@Override
@Transactional(propagation = Propagation.MANDATORY)
public void saveUserInfo() {
    userMapper.insert(new User("李四", 22));
}

```

**执行结果**：外层无事务，程序抛出异常：`No existing transaction found for transaction marked with propagation 'mandatory'`，方法执行失败。

#### 4. 生产适配场景

适用于**必须依赖事务、不允许单独执行的核心方法**，例如：数据库批量更新、资金变动、库存扣减等核心操作，避免无事务执行导致数据不一致。

#### 5. 避坑指南

⚠️ 该方法禁止单独调用，必须被事务方法嵌套调用，否则一定会报错，开发时需要严格控制调用链路。

### 3.2.6 NEVER

#### 1. 概念与核心原理

**NEVER**是禁止事务传播机制，核心规则：**强制在非事务环境执行，如果外层存在活跃事务，直接抛出异常**。

与MANDATORY完全相反，严格禁止事务环境，保证方法一定无事务执行。

#### 2. 执行流程拆解

1. 父方法有事务：直接抛出异常，执行失败

2. 父方法无事务：正常以非事务方式执行

#### 3. 实操代码案例

```java
// 父方法：存在事务
@Override
@Transactional
public void commonOperate() {
    // 调用NEVER子方法，直接报错
    commonService.saveTempData();
}

// 子方法：禁止事务环境
@Override
@Transactional(propagation = Propagation.NEVER)
public void saveTempData() {
    tempMapper.insert(new TempData("临时数据"));
}

```

**执行结果**：外层存在事务，程序抛出异常：`Existing transaction found for transaction marked with propagation 'never'`。

#### 4. 生产适配场景

适用于**绝对不能在事务中执行的操作**，例如：大数据量批量导入、文件读写、第三方接口调用、耗时统计操作，避免事务长时间占用数据库连接，导致连接池耗尽。

#### 5. 避坑指南

⚠️ 该方法禁止被事务方法嵌套调用，否则直接报错，适合独立执行的非核心耗时操作。

### 3.2.7 NESTED

#### 1. 概念与核心原理

**NESTED**是嵌套事务传播机制，是最特殊、面试最高频的传播机制。核心规则：**外层有事务时，子方法嵌套在父事务中，作为子事务（保存点事务）运行；外层无事务时，新建独立事务**。

核心特点：**父子事务不是完全独立，也不是完全统一**。子事务拥有独立的保存点，可以单独回滚，不影响父事务；但父事务整体回滚时，子事务必须跟随回滚。

#### 2. 核心原理与REQUIRES_NEW区别

很多开发者容易混淆NESTED和REQUIRES_NEW，核心区别：

- **REQUIRES_NEW**：完全独立事务，父子事务互不影响，父回滚子不回滚，子回滚父不回滚

- **NESTED**：嵌套子事务，基于父事务的保存点执行，子可单独回滚，父回滚则子必回滚

#### 3. 执行流程拆解

1. 父方法有事务：创建事务保存点，子方法基于保存点执行，子异常可单独回滚，父正常提交不受影响；父异常整体回滚

2. 父方法无事务：新建独立事务，与REQUIRED逻辑一致

#### 4. 实操代码案例

```java
// 父方法：主事务
@Override
@Transactional
public void orderFullOperate() {
    // 1. 主业务：创建订单
    orderMapper.insert(new Order("订单002", 199.9));
    try {
        // 2. 嵌套子事务：积分发放
        userService.addUserScore();
    } catch (Exception e) {
        log.error("积分发放失败", e);
    }
    // 父方法正常提交
}

// 子方法：NESTED嵌套事务
@Override
@Transactional(propagation = Propagation.NESTED)
public void addUserScore() {
    // 模拟积分发放异常
    int i = 1 / 0;
    scoreMapper.insert(new Score(1L, 100));
}

```

**执行结果**：子方法异常，基于保存点单独回滚，积分数据不入库；父事务正常提交，订单数据**成功入库**。如果父方法抛出异常，订单和积分操作全部回滚。

#### 5. 生产适配场景

适用于**主业务必须成功，附属子业务可失败回滚**的嵌套场景，例如：订单创建（主）+ 积分发放/优惠券发放（子），子业务失败不影响订单生效。

#### 6. 避坑指南

⚠️ NESTED事务**依赖数据库保存点机制**，仅支持MySQL、Oracle等主流数据库，部分小众数据库不支持；且同类方法内部调用会导致事务失效。

#### 7. 面试高频题

**问题**：NESTED和REQUIRES_NEW的核心区别是什么？

**参考答案**：1. 事务层级不同：REQUIRES_NEW是独立事务，完全脱离父事务；NESTED是嵌套子事务，依赖父事务保存点。2. 回滚规则不同：REQUIRES_NEW父子互不影响；NESTED子可单独回滚，父回滚子必回滚。3. 资源占用不同：REQUIRES_NEW会挂起父事务，性能更低；NESTED基于保存点，性能更高。

## 3.3 七大传播机制对比总结表

为方便快速记忆和选型，整理七大传播机制核心特性对比表，涵盖事务创建、父子影响、异常回滚、适配场景四大核心维度：

| 传播机制             | 外层有事务               | 外层无事务 | 父子事务关系                 | 核心适用场景                   |
| -------------------- | ------------------------ | ---------- | ---------------------------- | ------------------------------ |
| **REQUIRED**（默认） | 加入当前事务             | 新建事务   | 同一事务，互相影响           | 核心关联业务，必须同成功同失败 |
| **REQUIRES_NEW**     | 挂起父事务，新建独立事务 | 新建事务   | 完全独立，互不影响           | 独立附属业务，失败不影响主业务 |
| **SUPPORTS**         | 加入当前事务             | 无事务执行 | 有事务则关联，无事务则独立   | 查询、通用非核心更新方法       |
| **NOT_SUPPORTED**    | 挂起父事务，无事务执行   | 无事务执行 | 完全脱离事务                 | 临时日志、缓存更新、非核心埋点 |
| **MANDATORY**        | 加入当前事务             | 直接抛异常 | 强制依赖父事务               | 核心数据更新，禁止无事务执行   |
| **NEVER**            | 直接抛异常               | 无事务执行 | 强制脱离事务                 | 耗时操作、批量导入、第三方调用 |
| **NESTED**           | 嵌套子事务（保存点）     | 新建事务   | 子可单独回滚，父回滚子必回滚 | 主业务优先，子业务可降级失败   |

## 3.4 生产环境常用传播机制选型规范

结合千万级项目落地经验，总结生产环境**标准化选型规则**，杜绝滥用传播机制导致的bug和性能问题：

### 3.4.1 优先默认REQUIRED（90%业务）

所有核心业务的增删改操作，统一使用默认REQUIRED机制，保证关联业务数据一致性，无需手动指定propagation属性，简洁且安全。

### 3.4.2 附属日志/埋点选REQUIRES_NEW

操作日志、审计日志、业务埋点等非核心附属操作，使用REQUIRES_NEW独立事务，避免附属业务异常导致核心业务回滚，保证主业务稳定性。

### 3.4.3 查询方法统一使用SUPPORTS

所有数据库查询方法，配置SUPPORTS机制，既能适配事务嵌套场景，又能单独无事务执行，提升方法通用性，同时避免不必要的事务创建，优化性能。

### 3.4.4 耗时/批量操作选NEVER/NOT_SUPPORTED

大数据批量导入、文件解析、第三方接口调用、缓存刷新等耗时操作，使用NEVER或NOT_SUPPORTED，避免长时间占用数据库事务和连接池，防止数据库连接耗尽。

### 3.4.5 主次嵌套业务选NESTED

存在明确主次关系的嵌套业务（主业务必须成功，子业务可失败），使用NESTED嵌套事务，实现子业务独立回滚、主业务正常提交，兼顾一致性和业务容错性。

### 3.4.6 核心底层校验方法选MANDATORY

底层通用核心更新方法，配置MANDATORY机制，强制要求上层调用者必须开启事务，避免开发者疏忽导致无事务执行，引发数据脏数据问题。

# 4. 事务隔离级别

事务传播机制解决的是**多个事务之间的关联关系**，而事务隔离级别解决的是**多个并发事务之间的数据可见性问题**。数据库是多线程并发操作的，多个事务同时读写同一数据时，会产生脏读、不可重复读、幻读三大并发问题。Spring通过适配数据库的隔离级别，精细化控制事务并发读写规则，平衡数据一致性和并发性能。

## 4.1 数据库三大读问题：脏读、不可重复读、幻读

在讲解隔离级别之前，必须先掌握数据库并发事务的三大核心问题，所有隔离级别都是为了解决这三类问题而生。

### 4.1.1 脏读（Dirty Read）

#### 1. 概念定义

**脏读**：一个事务读取到了另一个事务**未提交的修改数据**，随后另一个事务回滚，导致当前读取的数据是无效、脏数据。

核心问题：**读取了未提交的无效数据**，是最严重的并发读问题。

#### 2. 场景演示

1. 事务A：修改用户余额为1000元，未提交

2. 事务B：读取用户余额，获取到未提交的1000元

3. 事务A：业务异常，执行回滚，余额恢复原值

4. 事务B：基于脏数据执行业务，导致逻辑错误

#### 3. 生产危害

资金计算、库存扣减等场景出现脏读，会直接导致资金错乱、库存超卖，造成业务损失。

### 4.1.2 不可重复读（Non-Repeatable Read）

#### 1. 概念定义

**不可重复读**：同一个事务内，**两次读取同一条数据**，结果不一致。原因是两次读取之间，其他事务修改并提交了该数据。

核心问题：**同一事务内，数据被其他事务修改更新**，聚焦于「数据修改」。

#### 2. 场景演示

1. 事务A：第一次读取用户余额为500元

2. 事务B：修改用户余额为800元，提交事务

3. 事务A：第二次读取同一用户余额，结果变为800元

4. 事务A：同一事务两次读取结果不一致，引发业务逻辑异常

#### 3. 生产危害

事务内多次校验同一数据、数据统计计算场景，会出现统计结果不一致、校验失效问题。

### 4.1.3 幻读（Phantom Read）

#### 1. 概念定义

**幻读**：同一个事务内，**两次查询批量数据**，结果的行数不一致。原因是其他事务新增或删除了数据并提交。

核心问题：**同一事务内，数据行数被其他事务增减**，聚焦于「数据新增/删除」。

#### 2. 场景演示

1. 事务A：查询所有未支付订单，共10条

2. 事务B：新增1条未支付订单，提交事务

3. 事务A：再次查询未支付订单，共11条

4. 事务A：感知到凭空多出数据，如同幻觉

#### 3. 生产危害

批量数据处理、批量统计、批量更新场景，会出现漏处理、统计数据不准的问题。

### 4.1.4 三大问题核心区别

- **脏读**：读未提交数据，最严重，必须杜绝

- **不可重复读**：读已提交的修改数据，单条数据变化

- **幻读**：读已提交的新增/删除数据，数据行数变化

## 4.2 Spring 五大隔离级别详解

Spring在`Isolation`枚举中定义了**五大事务隔离级别**，其中4种对应标准数据库隔离级别，1种为Spring专属默认级别，优先级从低到高、一致性从弱到强、并发性能从高到低依次递增。

### 4.2.1 DEFAULT（Spring默认）

#### 1. 核心原理

**DEFAULT**是Spring事务的默认隔离级别，核心规则：**不自定义隔离级别，直接使用底层数据库的默认隔离级别**。

#### 2. 适配规则

- MySQL默认隔离级别：**可重复读（REPEATABLE_READ）**

- Oracle默认隔离级别：**读已提交（READ_COMMITTED）**

#### 3. 生产场景

绝大多数项目无需手动修改隔离级别，直接使用DEFAULT，适配数据库原生特性，兼容性最好。

### 4.2.2 READ_UNCOMMITTED（读未提交）

#### 1. 核心原理

最低级别隔离级别，允许事务读取其他事务**未提交的修改数据**。

#### 2. 问题解决情况

**无法解决脏读、不可重复读、幻读**，所有并发问题都存在。

#### 3. 性能与场景

并发性能最高，但数据一致性最差，生产环境**绝对禁止使用**，仅用于测试场景。

### 4.2.3 READ_COMMITTED（读已提交）

#### 1. 核心原理

事务只能读取其他事务**已经提交的数据**，无法读取未提交的脏数据。

#### 2. 问题解决情况

**解决脏读**，无法解决不可重复读、幻读。

#### 3. 生产场景

Oracle数据库默认级别，适用于查询频繁、对数据一致性要求一般、追求高并发的业务，例如：资讯查询、商品列表展示。

### 4.2.4 REPEATABLE_READ（可重复读）

#### 1. 核心原理

MySQL默认隔离级别。同一个事务内，多次读取同一数据，结果保持一致。事务执行期间，其他事务无法修改当前事务读取过的数据。

#### 2. 问题解决情况

**解决脏读、不可重复读**，**无法彻底解决幻读**（MySQL通过间隙锁大幅降低幻读概率）。

#### 3. 生产场景

MySQL项目主流使用级别，兼顾一致性和并发性能，适用于绝大多数核心业务。

### 4.2.5 SERIALIZABLE（串行化）

#### 1. 核心原理

最高级别隔离级别，所有事务**串行执行**，禁止并发操作。事务执行时，锁定整张数据表，其他事务无法读写。

#### 2. 问题解决情况

**彻底解决脏读、不可重复读、幻读所有并发问题**。

#### 3. 性能与场景

并发性能最低，会产生大量锁等待、超时问题。仅适用于**对数据一致性绝对要求极高、并发量极低**的场景，例如：资金结算、核心账务对账。

## 4.3 隔离级别与数据库对应关系

整理Spring隔离级别、数据库隔离级别、并发问题解决能力、性能的完整对应表，方便快速查阅选型：

| Spring隔离级别   | 对应数据库级别        | 脏读       | 不可重复读 | 幻读       | 并发性能 |
| ---------------- | --------------------- | ---------- | ---------- | ---------- | -------- |
| DEFAULT（默认）  | 数据库默认级别        | 依赖数据库 | 依赖数据库 | 依赖数据库 | 最优     |
| READ_UNCOMMITTED | 读未提交              | 存在       | 存在       | 存在       | 最高     |
| READ_COMMITTED   | 读已提交              | 杜绝       | 存在       | 存在       | 较高     |
| REPEATABLE_READ  | 可重复读（MySQL默认） | 杜绝       | 杜绝       | 大概率杜绝 | 中等     |
| SERIALIZABLE     | 串行化                | 杜绝       | 杜绝       | 杜绝       | 最低     |

## 4.4 生产环境隔离级别最佳实践

### 4.4.1 MySQL项目统一使用DEFAULT

MySQL默认REPEATABLE_READ级别，已经解决99%的业务并发问题，兼顾一致性和性能，**所有普通业务无需手动修改隔离级别**。

### 4.4.2 高并发查询业务使用READ_COMMITTED

商品列表、资讯展示、用户信息查询等高频读场景，可手动指定READ_COMMITTED，减少事务锁等待，提升接口并发吞吐量，轻微的不可重复读不影响业务。

### 4.4.3 核心账务业务按需使用SERIALIZABLE

资金转账、余额结算、对账清算等绝对不能出现数据错乱的场景，可局部使用SERIALIZABLE级别，牺牲并发性能换取100%数据一致性，且需要做好接口限流，避免并发阻塞。

### 4.4.4 绝对禁止使用READ_UNCOMMITTED

该级别存在严重脏读问题，生产环境禁用，仅可用于本地测试调试。

### 4.4.5 避坑指南

⚠️ 隔离级别越高，锁粒度越大、并发性能越低，切勿盲目调高隔离级别；MySQL的REPEATABLE_READ通过MVCC机制，已经大幅优化了幻读问题，无需过度追求SERIALIZABLE级别。

# 5. 事务超时、只读、回滚规则
## 5.1 事务超时机制原理与失效场景
### 5.1.1 概念定义
**事务超时**：指Spring事务允许设置一个**最大执行时间**，单位为秒。若目标业务方法在事务内的执行耗时超过设定超时时间，Spring会**强制触发事务回滚**，并抛出`TransactionTimedOutException`超时异常。

通俗类比：就像线下办理业务设置**办理时限**，超过规定时间未完成流程，直接终止办理并撤销所有已操作步骤，避免事务长时间占用数据库连接、锁资源。

### 5.1.2 核心原理
1. **底层依赖**：Spring事务超时并非Spring自行计时，而是**委托给底层数据库事务管理器（DataSourceTransactionManager）**，最终由数据库连接（Connection）生效。
2. **计时起点**：从**事务正式开启、获取数据库连接**的时刻开始计时，包含业务SQL执行、业务逻辑运算、网络耗时等所有环节。
3. **触发逻辑**：当事务执行耗时超过配置的超时时间后，Spring拦截器会捕获超时信号，主动执行事务回滚，释放数据库锁与连接资源，避免长事务阻塞数据库、引发锁等待、死锁等线上问题。
4. **配置方式**
- 注解方式：`@Transactional(timeout = 10)` 表示事务最大执行时长10秒。
- 全局配置：可在事务管理器中统一设置默认超时时间，注解可覆盖全局配置。

### 5.1.3 生产场景与价值
- **解决长事务问题**：防止业务代码死循环、第三方接口阻塞导致事务长期不提交，占用数据库连接池资源。
- **减少锁竞争**：长事务会持有行锁/表锁，超时自动回滚可快速释放锁，提升数据库并发能力。
- **线上故障兜底**：突发业务卡顿、网络延迟时，自动终止事务，避免数据不一致。

### 5.1.4 实操示例
##### 注解配置事务超时
```java
import org.springframework.transaction.annotation.Transactional;
import org.springframework.stereotype.Service;

@Service
public class OrderService {

    /**
     * 设置事务超时时间：10秒
     * 超过10秒未执行完成，自动回滚并抛出超时异常
     */
    @Transactional(timeout = 10)
    public void createOrder() {
        // 1. 新增订单记录
        // 2. 扣减商品库存
        // 3. 新增支付记录
        // 若整体执行耗时 > 10s，事务强制回滚
    }
}
```

##### 事务管理器全局默认超时配置
```xml
<!-- 配置Spring事务管理器 -->
<bean id="transactionManager" class="org.springframework.jdbc.datasource.DataSourceTransactionManager">
    <property name="dataSource" ref="dataSource"/>
    <!-- 全局默认事务超时：15秒 -->
    <property name="defaultTimeout" value="15"/>
</bean>

<!-- 开启事务注解驱动 -->
<tx:annotation-driven transaction-manager="transactionManager"/>
```

### 5.1.5 超时机制常见失效场景 ⚠️避坑指南
1. **非事务内部耗时不计时**
   事务超时仅统计**事务开启后的执行时间**，方法调用前的初始化、参数校验等耗时不计入超时时间。
2. **只读事务超时不生效**
   配置`readOnly = true`的只读事务，数据库层面不会开启写事务，超时配置直接失效。
3. **本地调用导致事务失效**
   A类`a()`方法内部**本类直接调用**加了`@Transactional(timeout)`的`b()`方法，因无AOP代理，事务完全失效，超时配置也无效。
4. **嵌套事务传播行为影响**
   外层事务设置超时，内层嵌套事务**共享外层超时时间**，内层自身超时配置会被覆盖。
5. **数据库层面不支持超时**
   部分小众数据库或低版本驱动不支持事务超时属性，配置后无任何效果。

### 5.1.6 面试考点 📌
**面试题**：Spring事务timeout超时时间是从什么时候开始计时？包含方法全部耗时吗？
**参考答案**：
从**获取数据库连接、开启事务**时刻开始计时；不包含方法调用前的参数校验、对象初始化耗时，仅统计事务上下文内的业务执行、SQL执行、网络交互耗时；超时后自动回滚并抛出`TransactionTimedOutException`。

## 5.2 事务只读属性作用、底层优化原理
### 5.2.1 概念定义
**事务只读（readOnly）**：`@Transactional(readOnly = true)` 标识当前事务为**只读事务**，声明该业务方法**只会执行查询操作，不会新增、修改、删除数据**。

Spring和数据库会基于该标识做**双层优化**，提升查询性能、降低资源开销，是生产中查询类业务的最佳实践配置。

### 5.2.2 底层优化原理
#### 1. Spring框架层面优化
1. **事务标识标记**：Spring会将当前事务标记为只读，**不执行事务提交、回滚的冗余逻辑**，简化事务流程。
2. **无需事务日志**：只读事务不记录事务操作日志，减少框架底层IO开销。
3. **避免锁申请**：Spring不会为只读事务申请写锁资源，减少数据库锁竞争。

#### 2. 数据库JDBC层面优化
1. **关闭自动提交冗余校验**：JDBC连接识别到只读属性后，优化`autoCommit`逻辑，跳过写事务校验流程。
2. **数据库引擎优化**：MySQL InnoDB引擎收到只读事务请求后，**优先走快照读（MVCC）**，不上行级写锁，直接读取历史快照数据，极大提升查询并发。
3. **资源快速释放**：只读事务执行完毕后，数据库连接立即归还连接池，无需等待事务提交确认。

### 5.2.3 生产场景与价值
- **查询业务性能提升**：列表查询、详情查询、统计报表等纯读场景，配置只读事务可降低延迟、提升并发。
- **减少数据库锁冲突**：避免查询操作误加写锁，防止读写互斥阻塞。
- **代码语义规范化**：通过注解明确区分**读操作**和**写操作**，代码可读性更强，便于后期维护。
- **故障快速排查**：若只读事务中出现写操作，Spring会直接抛出异常，提前暴露代码错误。

### 5.2.4 实操示例
```java
@Service
public class UserQueryService {

    /**
     * 只读事务：纯查询业务
     * 底层触发Spring+MySQL双重优化，提升查询性能
     */
    @Transactional(readOnly = true)
    public User getUserInfo(Long userId) {
        // 仅执行数据库查询，无增删改操作
        return userMapper.selectById(userId);
    }
}
```

### 5.2.5 只读事务使用规范 ⚠️避坑指南
1. **只读事务中禁止增删改**
   若`readOnly=true`的方法内执行`insert/update/delete`，会直接抛出**只读连接无法执行写操作**异常，导致业务报错。
2. **写业务不要加只读属性**
   新增订单、扣库存等写操作误加`readOnly=true`，直接业务崩溃。
3. **只读事务无需配置超时**
   只读事务本身不占用写锁、执行速度快，配置`timeout`无效且无意义。
4. **嵌套事务只读属性传递**
   外层方法设置`readOnly=true`，内层被调用方法默认继承只读属性，无法执行写操作。

### 5.2.6 面试考点 📌
**面试题**：Spring事务readOnly=true底层做了哪些优化？为什么查询方法建议加只读事务？
**参考答案**：
1. Spring层面：标记只读事务，简化事务提交/回滚流程，减少日志记录开销；
2. JDBC层面：连接标记为只读，跳过写事务校验；
3. MySQL层面：InnoDB走MVCC快照读，不加行锁，提升查询并发；
4. 生产中查询方法加只读事务，既能规范代码语义，又能从框架、数据库双层优化性能，减少锁竞争。

## 5.3 @Transactional 回滚规则
### 5.3.1 概念定义
**事务回滚规则**：指Spring在**目标业务方法抛出异常时**，判定是否需要执行事务回滚的一套默认规则与自定义规则。

默认情况下，Spring仅对**运行时异常（RuntimeException）和Error**触发回滚；**受检异常（CheckedException）、自定义普通Exception**默认不会回滚，这也是生产中事务不回滚的高频坑点。

### 5.3.2 核心规则分类
| 异常类型                                 | 默认回滚行为 | 说明                                  |
| ---------------------------------------- | ------------ | ------------------------------------- |
| RuntimeException 及其子类                | **自动回滚** | 空指针、数组越界、业务运行异常等      |
| Error（OOM、栈溢出）                     | **自动回滚** | 系统级严重错误，必须回滚              |
| 受检异常 Exception（非RuntimeException） | **不回滚**   | IOException、SQLException等编译时异常 |
| 自定义继承Exception的异常                | **不回滚**   | 默认不会触发事务回滚                  |

### 5.3.3 底层原理
1. **异常捕获时机**：AOP事务拦截器`TransactionInterceptor`环绕目标方法，**捕获方法抛出的所有异常**。
2. **异常类型判定**：拦截器内部通过**异常类型匹配规则**，判断当前异常是否符合回滚条件。
3. **事务决策**：符合回滚规则 → 执行`rollback`；不符合 → 执行`commit`提交事务。
4. **设计初衷**：受检异常是开发者**主动捕获处理**的业务异常，Spring认为开发者已自行处理容错，无需强制回滚；运行时异常是未预期的系统异常，必须回滚保证数据一致。

### 5.3.4 生产场景问题
- 业务抛出自定义受检异常，事务**默认不回滚**，导致数据库部分数据提交、部分未提交，**数据不一致**。
- 开发者手动`try-catch`捕获异常后未重新抛出，事务拦截器无法感知异常，**不会回滚**。
- 多方法嵌套调用，内层异常被捕获，外层事务无法触发回滚。

### 5.3.5 面试考点 📌
**面试题**：Spring事务默认对什么异常回滚？对什么异常不回滚？
**参考答案**：
默认只对**RuntimeException运行时异常**和**Error系统错误**自动回滚；对**普通受检Exception、自定义非运行时异常**默认不回滚。若需要受检异常也回滚，需通过`rollbackFor`手动指定。

## 5.4 默认回滚规则、自定义回滚、指定异常不回滚
### 5.4.1 默认回滚规则详解
Spring事务默认回滚规则源码定义在`RuleBasedTransactionAttribute`中，核心逻辑：
1. 只要抛出**RuntimeException或Error**，立即回滚事务；
2. 抛出**CheckedException（直接继承Exception而非RuntimeException）**，正常提交事务，不回滚；
3. 方法正常执行无异常，统一执行事务提交。

**常见默认不回滚场景**
```java
// 自定义受检异常
public class BusinessException extends Exception {
    public BusinessException(String msg) {
        super(msg);
    }
}

@Service
class OrderService {
    @Transactional
    public void payOrder() throws BusinessException {
        // 扣减余额
        // 抛出受检异常
        throw new BusinessException("支付失败");
        // Spring默认不回滚，数据库扣余额操作会提交，造成数据错乱
    }
}
```

### 5.4.2 自定义回滚规则：指定异常触发回滚
通过`rollbackFor`属性，**指定任意异常（包括受检异常、自定义异常）触发事务回滚**，是生产最常用配置。

#### 1. 指定单个异常回滚
```java
// 只要抛出BusinessException，就触发回滚
@Transactional(rollbackFor = BusinessException.class)
public void payOrder() throws BusinessException {
    // 业务操作
    throw new BusinessException("支付失败");
}
```

#### 2. 指定多个异常回滚
```java
// 多个异常都触发回滚
@Transactional(rollbackFor = {BusinessException.class, IOException.class})
public void batchOperate() throws BusinessException, IOException {
    // 批量业务处理
}
```

#### 3. 全局所有Exception都回滚（通用写法）
```java
// 所有Exception及其子类，全部触发回滚，项目中最常用
@Transactional(rollbackFor = Exception.class)
public void commonBusiness() {
    // 任意业务异常都回滚
}
```

### 5.4.3 指定异常不回滚：noRollbackFor
通过`noRollbackFor`属性，**指定某些特定异常即使抛出，也不回滚事务，正常提交**。

适用场景：业务抛出提示性异常，无需回滚数据库数据，仅返回前端提示信息。

##### 实操示例
```java
/**
 * 抛出UserNotExistException时，不回滚事务
 * 其他异常仍遵循默认回滚规则
 */
@Transactional(rollbackFor = Exception.class, noRollbackFor = UserNotExistException.class)
public void login(Long userId) {
    if (userId == null) {
        throw new UserNotExistException("用户不存在");
    }
    // 正常业务逻辑
}
```

### 5.4.4 事务回滚失效经典场景 ⚠️避坑指南
1. **手动try-catch捕获异常，未重新抛出**
```java
@Transactional(rollbackFor = Exception.class)
public void test() {
    try {
        // 数据库操作
        int a = 1 / 0;
    } catch (Exception e) {
        // 直接捕获不抛出，AOP拦截器感知不到异常，事务不会回滚
        e.printStackTrace();
    }
}
```
**解决方案**：catch中手动重新抛出异常，或手动编程式回滚。

2. **本类内部方法调用**
无AOP代理，注解失效，异常无法触发回滚。

3. **传播行为为REQUIRES_NEW嵌套事务**
内层事务异常回滚不影响外层，外层异常也不会回滚已提交的内层事务。

### 5.4.5 最佳实践 💡
1. 项目统一规范：所有业务写方法**必加 @Transactional(rollbackFor = Exception.class)**，覆盖所有异常回滚；
2. 纯查询方法统一加 **@Transactional(readOnly = true)**；
3. 业务提示类异常，通过`noRollbackFor`指定不回滚；
4. 禁止在事务方法内无脑try-catch吃掉异常，如需捕获必须重新抛出。

# 6. 声明式事务底层 AOP 原理（核心）
## 6.1 Spring 事务本质：AOP 动态代理实现
### 6.1.1 核心本质
**Spring声明式事务的底层本质就是：AOP动态代理 + 环绕通知拦截**。

开发者添加的`@Transactional`注解本身**没有任何事务逻辑**，仅作为**标记注解**；Spring启动时会扫描带有该注解的类/方法，通过**JDK动态代理（接口代理）或CGLIB动态代理（类代理）** 生成代理对象，在代理对象中通过AOP环绕通知，**在目标方法执行前开启事务、执行后提交、异常时回滚**。

通俗类比：
目标业务方法是**核心办事流程**，AOP代理就是**中间代办员**，代办员在办事前先开事务、办事结束后提交、办事出错直接撤销，全程不改动原有业务代码。

### 6.1.2 代理方式选择规则
| 代理类型      | 使用场景                      | 特点                                           |
| ------------- | ----------------------------- | ---------------------------------------------- |
| JDK动态代理   | 目标类**实现了接口**          | 基于接口生成代理，只能拦截接口方法             |
| CGLIB动态代理 | 目标类**无接口** / 强制类代理 | 基于子类继承生成代理，可拦截普通方法、无接口类 |

Spring事务默认代理规则：
1. 目标类有接口 → **优先JDK动态代理**；
2. 目标类无接口 → **自动使用CGLIB代理**；
3. 可通过配置强制全程使用CGLIB代理：
```xml
<!-- 强制使用CGLIB事务代理 -->
<tx:annotation-driven proxy-target-class="true"/>
```

### 6.1.3 事务AOP核心角色
1. **标记注解**：`@Transactional`，用于标识需要事务增强的方法；
2. **切面类**：`TransactionAspectSupport` 事务切面基础类；
3. **拦截器**：`TransactionInterceptor`，AOP环绕通知核心入口；
4. **事务管理器**：`PlatformTransactionManager`，负责底层事务开启、提交、回滚；
5. **代理工厂**：AOP代理生成器，创建目标类的代理对象。

### 6.1.4 为什么事务会失效（代理角度）⚠️避坑
1. **本类方法内部调用**：`this.xxx()` 调用本类事务方法，绕过代理对象，直接调用原生方法，AOP拦截失效；
2. **方法权限非public**：private/protected/default方法，AOP无法拦截，事务注解失效；
3. **静态方法加注解**：静态方法不属于实例对象，代理无法增强，事务失效；
4. **final方法加注解**：CGLIB通过继承重写方法实现代理，final方法无法重写，事务失效。

### 6.1.5 面试考点 📌
**面试题**：Spring声明式事务底层实现原理是什么？
**参考答案**：
底层基于**AOP动态代理**实现，Spring扫描`@Transactional`标记的类和方法，通过JDK/CGLIB生成代理对象；由`TransactionInterceptor`环绕拦截目标方法，在方法执行前开启事务，正常执行完毕提交事务，抛出异常时回滚事务，实现无侵入式事务管控。

## 6.2 事务核心入口：TransactionInterceptor
### 6.2.1 角色定位
`TransactionInterceptor`是Spring事务**AOP拦截的核心入口**，实现了`MethodInterceptor`接口，属于AOP环绕通知。所有被`@Transactional`修饰的方法，执行时都会**优先进入该拦截器**，由它全权管控事务生命周期。

### 6.2.2 核心职责
1. 解析目标方法上的`@Transactional`注解属性（超时、只读、回滚规则、传播行为等）；
2. 获取配置的**事务管理器** `PlatformTransactionManager`；
3. 根据传播行为、隔离级别**创建/获取/挂起事务**；
4. 执行目标原始业务方法；
5. 捕获方法执行异常，根据回滚规则判断**提交或回滚**；
6. 事务执行完毕后**释放资源、恢复挂起事务**。

### 6.2.3 核心源码结构简述
核心入口方法 `invoke(MethodInvocation invocation)`，整体逻辑骨架：
1. 解析事务注解属性 `TransactionAttribute`；
2. 获取事务管理器 `TransactionManager`；
3. 开启事务并创建事务状态 `TransactionStatus`；
4. 调用`proceed()`执行目标业务方法；
5. try-catch捕获异常，按规则回滚/提交；
6. 最终关闭事务、释放数据库资源。

### 6.2.4 实操关联
开发者无需手动编写该拦截器，Spring事务注解驱动`<tx:annotation-driven>`会**自动注册TransactionInterceptor**，并自动匹配所有`@Transactional`方法完成拦截增强。

## 6.3 事务切面完整执行链路
### 6.3.1 完整执行时序流程
1. **容器启动阶段**
   - Spring扫描所有Bean，识别带有`@Transactional`注解的类和方法；
   - 加载事务切面、注册`TransactionInterceptor`拦截器；
   - 为目标Bean生成**AOP代理对象**，放入Spring容器。

2. **方法调用阶段**
   - 业务代码从Spring容器**注入代理对象**，调用事务方法；
   - 调用请求先进入 **TransactionInterceptor 拦截器**；
   - 拦截器解析注解配置、匹配事务规则；
   - 事务管理器**开启事务、获取数据库连接**；
   - 执行**目标原生业务方法**，执行SQL操作；
   - 方法正常结束 → 拦截器调用事务**提交**；
   - 方法抛出异常 → 拦截器按回滚规则**事务回滚**；
   - 释放数据库连接、恢复挂起事务、释放锁资源。

### 6.3.2 链路流程图解（文字版）
```
业务调用代理对象方法 
        ↓
TransactionInterceptor 拦截器入口
        ↓
解析@Transactional注解属性
        ↓
事务管理器创建/获取事务
        ↓
执行目标业务方法
        ↓
无异常 → 事务提交
有异常 → 规则匹配 → 事务回滚
        ↓
释放数据库连接、事务资源
```

### 6.3.3 关键节点说明
1. **代理对象是关键**：必须从Spring容器注入代理Bean，才能走AOP拦截链路；直接new对象、本类this调用都不走代理；
2. **拦截器是中枢**：所有事务逻辑都封装在TransactionInterceptor中，业务代码无感知；
3. **事务管理器是底层执行者**：拦截器只做流程调度，真正操作commit/rollback的是TransactionManager。

## 6.4 事务代理对象创建时机（和AOP代理统一时机）
### 6.4.1 代理创建时机
Spring事务代理对象的创建时机，**和普通AOP代理完全一致**：
> 在**Spring Bean初始化完成后、放入容器前**，通过AOP代理工厂**提前生成代理对象**，原生对象被包装，容器中对外暴露的始终是代理对象。

### 6.4.2 完整Bean生命周期代理时机
1. Bean实例化（构造方法创建对象）；
2. 属性填充（依赖注入）；
3. 初始化方法执行（InitializingBean、init-method）；
4. **AOP代理增强**：检测到`@Transactional`或AOP切面，生成代理对象；
5. 代理对象放入Spring单例容器，供后续依赖注入使用。

### 6.4.3 为什么不调用时才创建代理？
1. **性能损耗**：调用时动态创建代理会增加实时耗时；
2. **单例一致性**：Spring Bean默认单例，容器启动时一次性创建代理，保证全局使用同一个代理对象；
3. **依赖注入要求**：其他Bean依赖注入时，需要提前拿到代理对象，而非原生对象。

### 6.4.4 特殊场景：懒加载Bean
添加`@Lazy`懒加载的事务Bean，**代理创建时机延迟到第一次调用时**，而非容器启动阶段，其余代理逻辑完全一致。

### 6.4.5 面试考点 📌
**面试题**：Spring事务代理对象是什么时候创建的？
**参考答案**：
默认在**Spring容器启动、Bean初始化完成后**立即创建AOP代理对象；懒加载Bean则在**第一次方法调用时**创建；事务代理和普通AOP代理创建时机、底层机制完全统一。

# 7. 事务完整执行源码流程
## 7.1 事务拦截器 invoke() 整体流程
### 7.1.1 核心入口方法
`TransactionInterceptor#invoke()` 是整个Spring事务的**总入口**，所有事务方法都会进入该方法，完成全流程管控。

### 7.1.2 七大核心执行步骤
1. **步骤1：获取目标方法与类信息**
   从`MethodInvocation`中获取当前调用的目标方法、目标类、方法注解信息。

2. **步骤2：解析事务属性**
   通过`TransactionAttributeSource`解析方法上的`@Transactional`注解，提取**传播行为、隔离级别、超时时间、只读、回滚规则**等配置，封装为`TransactionAttribute`对象。

3. **步骤3：获取事务管理器**
   根据配置匹配对应的`PlatformTransactionManager`（常用`DataSourceTransactionManager`）。

4. **步骤4：开启事务，创建事务状态**
   调用事务管理器`getTransaction()`方法，根据传播行为判断：新建事务、加入当前事务、挂起外层事务，返回`TransactionStatus`事务状态对象。

5. **步骤5：执行目标业务方法**
   调用`invocation.proceed()`执行原生业务方法，执行内部SQL逻辑。

6. **步骤6：异常捕获与事务决策**
   环绕try-catch捕获业务方法所有异常，匹配**回滚规则**：
   - 符合回滚条件 → 调用`rollback()`；
   - 不符合回滚条件 → 调用`commit()`。

7. **步骤7：事务收尾与资源释放**
   无论是否异常，最终执行事务资源清理：归还数据库连接、恢复被挂起的外层事务、清空事务上下文。

### 7.1.3 源码逻辑伪代码
```java
public Object invoke(MethodInvocation invocation) throws Throwable {
    // 1. 解析事务注解属性
    TransactionAttribute attr = transactionAttributeSource.getTransactionAttribute(...);
    // 2. 获取事务管理器
    PlatformTransactionManager tm = getTransactionManager();
    // 3. 获取/创建事务状态
    TransactionStatus status = tm.getTransaction(attr);
    Object result;
    try {
        // 4. 执行目标业务方法
        result = invocation.proceed();
    } catch (Throwable ex) {
        // 5. 异常：按规则回滚
        txManager.rollback(status);
        throw ex;
    }
    // 6. 无异常：提交事务
    txManager.commit(status);
    return result;
}
```

## 7.2 事务获取、创建、挂起、恢复机制
### 7.2.1 核心接口与对象
- `PlatformTransactionManager`：事务管理器顶层接口，定义`getTransaction()`、`commit()`、`rollback()`三大核心方法；
- `TransactionStatus`：事务状态对象，记录事务是否新建、是否被挂起、是否已完成等状态；
- `TransactionDefinition`：事务定义，包含传播行为、隔离级别、超时、只读等配置。

### 7.2.2 事务获取与创建逻辑
核心方法：`DataSourceTransactionManager#getTransaction()`
1. **判断当前线程是否已有事务**
   从**当前线程本地变量（ThreadLocal）** 获取已有事务资源（数据库连接、事务上下文）。
2. **根据传播行为分支处理**
   - REQUIRED：有则加入当前事务，无则新建事务；
   - REQUIRES_NEW：**强制新建独立事务**，挂起外层已有事务；
   - SUPPORTS：有则加入，无则以非事务方式执行；
   - 其他传播行为按对应规则处理。
3. **新建事务流程**
   从数据源获取数据库连接 → 设置`autoCommit=false` → 绑定到当前ThreadLocal → 标记为新事务。

### 7.2.3 事务挂起机制
**适用场景**：REQUIRES_NEW、NESTED等传播行为，内层需要独立事务时。
1. 把**外层事务的数据库连接、事务状态**从当前线程移除，保存到临时容器；
2. 新建内层独立事务，绑定到当前线程；
3. 内层事务执行完毕后，**恢复外层挂起的事务**，继续执行外层逻辑。

### 7.2.4 事务恢复机制
1. 内层独立事务提交/回滚完成后；
2. 清空当前线程内层事务资源；
3. 将之前挂起的**外层事务重新绑定到当前ThreadLocal**；
4. 外层事务继续后续执行流程。

### 7.2.5 生产价值
事务挂起与恢复是**嵌套事务、多事务隔离执行**的底层支撑，保证不同传播行为下事务互不干扰、数据一致性可控。

## 7.3 目标方法执行、异常捕获、提交/回滚判断
### 7.3.1 目标方法执行
拦截器调用`invocation.proceed()`，通过AOP链式调用，最终执行**原生业务实现类的目标方法**，执行内部的Mapper/DAO数据库操作，所有SQL都在**当前事务连接**中执行。

整个方法执行期间，数据库连接**不会归还连接池**，一直持有直到事务提交/回滚完成。

### 7.3.2 异常捕获范围
TransactionInterceptor的try-catch**捕获目标方法抛出的所有Throwable**：
- 包括RuntimeException、Error、普通Exception、自定义业务异常；
- 捕获后不直接吞掉异常，而是**先处理事务，再向上抛出异常**，保证业务异常能被上层感知。

### 7.3.3 提交/回滚判断逻辑
1. **无任何异常**
   直接执行事务管理器`commit()`，提交所有SQL操作，释放数据库锁。

2. **有异常**
   拦截器匹配`@Transactional`的**回滚规则**：
   - 命中rollbackFor指定异常 / 默认RuntimeException/Error → 执行`rollback()`；
   - 命中noRollbackFor指定异常 → 不回滚，执行`commit()`；
   - 普通受检异常无配置 → 默认不回滚，正常提交。

### 7.3.4 回滚底层动作
1. 调用JDBC Connection的`rollback()`方法；
2. 撤销当前事务内所有SQL增删改操作；
3. 释放数据库行锁/表锁；
4. 标记事务状态为已回滚。

## 7.4 事务资源释放、连接归还流程
### 7.4.1 事务资源包含内容
1. 数据库Connection连接对象；
2. 事务状态TransactionStatus；
3. 线程本地变量ThreadLocal中绑定的事务上下文；
4. 数据库锁资源、事务日志资源。

### 7.4.2 资源释放完整流程
1. **事务提交/回滚完成后**
   进入资源清理阶段，无论成功还是失败都必须执行。

2. **解除线程绑定**
   把当前线程ThreadLocal中绑定的**事务连接、事务上下文**清空，避免线程复用导致事务上下文错乱。

3. **恢复连接自动提交**
   将数据库Connection的`autoCommit`恢复为默认true，避免连接池复用连接时状态异常。

4. **归还数据库连接池**
   将Connection放回Druid/HikariCP连接池，供其他业务请求复用，**避免连接泄露**。

5. **恢复挂起的外层事务**
   若当前是嵌套事务，内层执行完毕后，恢复之前挂起的外层事务上下文。

6. **清空事务状态**
   销毁当前TransactionStatus对象，标记事务流程结束。

### 7.4.3 资源释放的重要性
1. **防止连接泄露**：不及时归还连接会导致连接池耗尽，线上业务数据库请求阻塞；
2. **避免线程上下文污染**：ThreadLocal不清理会导致线程复用后事务错乱、数据脏写；
3. **释放数据库锁**：及时释放行锁，提升数据库并发能力，减少锁等待超时。

### 7.4.4 面试考点 📌
**面试题**：Spring事务执行完毕后，数据库连接什么时候归还连接池？
**参考答案**：
事务**提交或回滚完成后**，Spring会自动恢复连接autoCommit状态、清空线程事务上下文，**立即将数据库连接归还连接池**；并非方法执行完就归还，而是事务整个生命周期结束后才归还。

# 8. Spring 事务全部失效场景

Spring事务的核心实现依赖**Spring AOP动态代理**，所有事务的开启、提交、回滚操作，都是由代理对象完成的。原生对象的方法调用不会触发AOP增强，这是绝大多数事务失效场景的底层根本原因。除此之外，异常规则不匹配、传播机制误用、多线程隔离、异常被捕获等场景，也会直接导致事务失效。本节全面覆盖生产100%会遇到的7大失效场景，逐一拆解原理、实操复现、解决方案、避坑指南及面试考点。

## 8.1 本类内部调用导致事务失效

### 8.1.1 概念与失效原理

**本类内部调用事务失效**：指同一个类中，一个无事务的普通方法，直接调用本类中添加了`@Transactional`注解的事务方法，最终导致事务完全不生效，异常无法触发回滚的现象。

底层核心原理：Spring事务基于**AOP动态代理**实现。Spring容器启动时，会为添加事务方法的类生成代理对象，所有事务增强逻辑（开启事务、切面拦截、提交、回滚）都封装在代理对象中。而本类内部方法调用，是通过**原生this对象**调用方法，而非代理对象调用，AOP无法完成拦截，事务增强逻辑完全不会执行，最终事务失效。

生活化类比：代理对象是“带审批权限的管理员”，只有通过管理员执行操作，才会记录流程、容错回滚；直接自己（原生对象）执行操作，没有任何流程管控，事务自然失效。

### 8.1.2 生产问题场景

开发中常出现代码复用场景，开发者为了便捷，直接在本类普通方法中调用本类事务方法，未感知到代理失效问题。最终出现：代码执行无报错，但数据库数据不一致，报错后数据不会回滚，引发线上数据脏数据问题。

### 8.1.3 失效代码示例（可直接复现）

```java
@Service
public class UserService {

    // 普通无事务方法，内部调用本类事务方法
    public void updateUserInfo() {
        // 原生this调用，非代理对象调用
        this.addUser();
        // 手动抛出异常，测试事务是否回滚
        int i = 1 / 0;
    }

    // 声明式事务方法
    @Transactional(rollbackFor = Exception.class)
    public void addUser() {
        // 模拟数据库新增用户操作
        userMapper.insert(new User());
    }
}
```

执行结果：程序抛出算数异常，但**数据库用户数据成功插入，事务未回滚**，事务完全失效。

### 8.1.4 解决方案与最佳实践

核心解决思路：**必须通过Spring代理对象调用事务方法**，绕过原生this调用。提供三种生产常用方案：

1. **自身注入代理对象（最简单常用）**：在当前类中自动注入自身，通过注入的代理对象调用事务方法。

2. **拆分业务类**：将事务方法抽离到独立Service，跨类调用天然走代理，无失效问题。

3. **手动获取AOP代理对象**：通过`AopContext.currentProxy()`获取当前代理对象调用方法。

修复代码示例（自身注入方案）：

```java
@Service
public class UserService {

    // 注入当前类的代理对象
    @Autowired
    private UserService userService;

    public void updateUserInfo() {
        // 通过代理对象调用事务方法，触发AOP增强
        userService.addUser();
        int i = 1 / 0;
    }

    @Transactional(rollbackFor = Exception.class)
    public void addUser() {
        userMapper.insert(new User());
    }
}
```

### 8.1.5 ⚠️避坑指南

- 不要存在侥幸心理，所有本类`this.xxx()`调用事务方法，100%事务失效，无例外。

- 拆分类是最稳定的生产方案，避免循环依赖、代理获取异常等问题。

- `AopContext.currentProxy()`使用时，必须开启注解`@EnableAspectJAutoProxy(exposeProxy = true)`，否则报错。

### 8.1.6 📌面试高频题

**问题**：为什么本类内部调用事务方法会导致事务失效？如何解决？

**参考答案**：Spring事务基于AOP动态代理实现，仅代理对象的方法调用会触发事务拦截增强。本类内部通过this调用方法，使用的是原生对象，AOP无法拦截，事务开启、回滚逻辑不会执行，导致事务失效。解决方案：1. 注入自身代理对象调用；2. 抽离事务方法到独立类跨类调用；3. 开启暴露代理对象，通过工具类获取代理对象调用。

## 8.2 方法非 public 事务失效

### 8.2.1 原理与失效原因

Spring AOP动态代理（JDK动态代理、CGLIB代理）**仅支持拦截public修饰的方法**，private、protected、default权限的方法无法被代理拦截。

从底层源码角度：Spring事务拦截器`TransactionInterceptor`在执行前置校验时，会通过`Modifier.isPublic()`判断方法权限，非public方法直接跳过事务增强逻辑，不会创建事务、不会触发回滚，最终事务完全失效。

很多新手误区：只要加了`@Transactional`注解就有事务，忽略方法权限修饰符的限制，导致测试、生产环境数据不一致。

### 8.2.2 失效代码示例

```java
@Service
public class OrderService {

    // 非public方法（private），添加事务注解无效
    @Transactional(rollbackFor = Exception.class)
    private void createOrder() {
        // 模拟订单新增
        orderMapper.insert(new Order());
        // 抛出异常
        throw new RuntimeException("订单创建失败");
    }

    // 对外公开调用入口
    public void buildOrder() {
        createOrder();
    }
}
```

执行结果：程序抛出异常，但订单数据成功入库，**事务未回滚，完全失效**。

### 8.2.3 权限适配规则详解

- **public**：可被AOP拦截，事务正常生效（唯一有效权限）。

- **private**：无法被代理拦截，事务失效。

- **protected**：JDK代理不支持、CGLIB代理无法拦截，事务失效。

- **default（包访问权限）**：无代理拦截，事务失效。

### 8.2.4 💡最佳实践

1. 所有需要事务控制的方法，**必须强制使用public修饰**，代码规范层面统一约束。

2. 禁止将事务方法定义为私有方法，业务拆分时优先保证事务方法公开可代理。

3. 代码审查重点校验：事务方法权限修饰符，提前规避线上bug。

### 8.2.5 📌面试高频题

**问题**：private方法上加@Transactional注解，事务会生效吗？为什么？

**参考答案**：不会生效。Spring事务基于AOP动态代理实现，AOP仅能拦截public公开方法，private私有方法无法被代理对象拦截，Spring事务拦截器不会对其执行事务增强逻辑，因此事务完全失效。

## 8.3 异常类型不匹配导致不回滚

### 8.3.1 核心规则与原理

Spring事务**默认仅对RuntimeException（运行时异常）和Error（系统错误）触发回滚**，对编译时异常（CheckedException，如Exception、IOException、SQLException）默认不回滚。

底层设计原理：Spring默认规则认为，编译时异常是开发者主动捕获、预判的异常，属于业务可控异常，无需事务回滚；而运行时异常是不可预判的系统异常，属于程序bug，需要回滚数据保证一致性。

关键误区：很多开发者直接使用`@Transactional`默认配置，抛出普通Exception后，发现事务不回滚，导致数据脏写。

### 8.3.2 失效场景代码示例

```java
@Service
public class PayService {

    // 默认事务配置，仅回滚运行时异常
    @Transactional
    public void payMoney() throws Exception {
        // 模拟扣款操作
        userAccountMapper.updateBalance();
        // 主动抛出编译时异常
        throw new Exception("支付异常，需要回滚");
    }
}
```

执行结果：抛出Exception编译时异常，**事务未回滚，扣款数据持久化入库**，业务数据异常。

### 8.3.3 自定义回滚异常规则

通过`rollbackFor`属性指定需要回滚的异常类型，覆盖Spring默认规则，生产通用配置：

```java
// 所有Exception及其子类异常，全部触发事务回滚
@Transactional(rollbackFor = Exception.class)
public void payMoney() throws Exception {
    userAccountMapper.updateBalance();
    throw new Exception("支付异常，需要回滚");
}
```

拓展配置：`noRollbackFor`指定**不回滚**的异常类型，用于业务自定义可控异常。

### 8.3.4 ⚠️避坑指南

- 禁止使用`@Transactional`默认配置，生产环境**必须统一配置rollbackFor = Exception.class**。

- 区分异常类型：编译时异常必须手动指定回滚规则，运行时异常默认回滚。

- 自定义业务异常需继承`RuntimeException`，可默认触发回滚，简化配置。

### 8.3.5 📌面试高频题

**问题**：@Transactional默认对什么异常回滚？抛出Exception为什么不回滚？如何解决？

**参考答案**：默认仅回滚RuntimeException和Error，不回滚编译时Exception。因为Spring默认认为编译时异常是业务可控异常，无需回滚。解决方式：通过`rollbackFor = Exception.class`配置，指定所有异常均触发事务回滚。

## 8.4 try-catch 吞异常导致事务不回滚

### 8.4.1 失效核心原理

Spring事务回滚的触发条件：**事务方法执行过程中，抛出的异常被Spring AOP拦截器捕获**。

如果开发者在事务方法内部通过try-catch手动捕获并处理了所有异常，异常不会向上抛出，Spring事务拦截器无法感知异常发生，会判定方法正常执行完成，最终**自动提交事务，不会回滚**。这是生产中最高频、最难排查的事务失效场景之一。

### 8.4.2 失效代码示例

```java
@Service
public class OrderService {

    @Transactional(rollbackFor = Exception.class)
    public void createOrder() {
        try {
            // 新增订单数据
            orderMapper.insert(new Order());
            // 模拟业务异常
            int i = 1 / 0;
        } catch (Exception e) {
            // 手动吞掉异常，无向上抛出
            log.error("订单创建失败", e);
        }
    }
}
```

执行结果：代码报错被日志打印，但异常被捕获，Spring未感知异常，**事务正常提交，订单数据入库，事务失效**。

### 8.4.3 生产解决方案

根据业务场景提供两种最优解决方案：

#### 方案1：捕获异常后手动回滚（精准可控）

通过`TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()`手动触发事务回滚，适合需要自定义异常处理逻辑的场景。

```java
@Transactional(rollbackFor = Exception.class)
public void createOrder() {
    try {
        orderMapper.insert(new Order());
        int i = 1 / 0;
    } catch (Exception e) {
        log.error("订单创建失败", e);
        // 手动告知Spring事务需要回滚
        TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
    }
}
```

#### 方案2：捕获后重新抛出异常（简单通用）

捕获异常后封装业务异常并向上抛出，让Spring拦截器感知异常，自动回滚，适合绝大多数通用业务场景。

```java
@Transactional(rollbackFor = Exception.class)
public void createOrder() {
    try {
        orderMapper.insert(new Order());
        int i = 1 / 0;
    } catch (Exception e) {
        log.error("订单创建失败", e);
        // 重新抛出异常，触发事务回滚
        throw new BusinessException("订单创建失败，请重试");
    }
}
```

### 8.4.4 ⚠️避坑指南

- 事务方法内try-catch捕获异常后，不抛异常、不手动回滚，100%事务不回滚。

- 禁止为了消除控制台报错，无脑捕获所有异常，会掩盖数据一致性问题。

- 手动回滚工具类仅能在事务方法内使用，非事务方法调用会报错。

### 8.4.5 📌面试高频题

**问题**：事务方法中try-catch捕获异常，事务还会回滚吗？为什么？如何解决？

**参考答案**：不会回滚。因为Spring事务回滚依赖AOP拦截器捕获方法抛出的异常，try-catch吞掉异常后，异常无法向上传递，Spring感知不到异常，判定方法执行成功并提交事务。解决方案：1. 捕获异常后重新抛出业务异常；2. 手动调用事务工具类触发回滚。

## 8.5 传播机制使用不当导致失效

### 8.5.1 事务传播机制核心概念

事务传播机制定义了**多个事务方法嵌套调用时，事务如何传递、创建、挂起、合并**的规则。Spring默认7种传播机制，其中`SUPPORTS`、`NOT_SUPPORTED`、`NEVER`、`REQUIRED`误用是事务失效的核心原因。

核心失效逻辑：子方法传播级别配置不当，会导致子方法脱离父事务、无事务执行，最终异常无法回滚。

### 8.5.2 高频失效场景拆解

#### 场景1：REQUIRED嵌套调用，子方法异常被捕获导致父事务回滚（伪失效）

`REQUIRED`（默认级别）：存在事务则加入当前事务，无事务则新建事务。嵌套调用时父子方法属于**同一个事务**，子方法异常被捕获后，事务标记为回滚，父方法后续执行会报事务已回滚异常，整体事务失效。

#### 场景2：NOT_SUPPORTED 强制无事务，彻底失效

`NOT_SUPPORTED`：始终以非事务方式执行，若当前存在事务，会**挂起原有事务**，方法执行完毕后恢复。该级别下的方法，所有数据库操作均为自动提交，异常绝对不会回滚，是高频失效场景。

```java
@Service
public class UserService {

    @Transactional(rollbackFor = Exception.class)
    public void updateUser() {
        userMapper.update();
        // 调用无事务传播级别的方法
        addUser();
    }

    // 强制非事务执行
    @Transactional(propagation = Propagation.NOT_SUPPORTED)
    public void addUser() {
        userMapper.insert(new User());
        int i = 1 / 0;
    }
}
```

执行结果：addUser方法无事务，异常后新增数据不回滚，updateUser事务被挂起，整体数据不一致。

#### 场景3：SUPPORTS 无事务环境下失效

`SUPPORTS`：有事务则加入，无事务则非事务执行。如果外层方法无事务，内层`SUPPORTS`方法**无事务生效**，异常不回滚。

### 8.5.3 💡最佳实践

- 通用业务增删改，统一使用默认`REQUIRED`传播级别。

- 查询类方法可使用`SUPPORTS`，避免不必要的事务创建。

- 绝对禁止业务修改方法使用`NOT_SUPPORTED`、`NEVER`级别。

- 嵌套事务场景，优先使用`REQUIRES_NEW`、`NESTED`，精准控制事务隔离。

### 8.5.4 📌面试高频题

**问题**：哪些事务传播机制会导致事务失效？核心原因是什么？

**参考答案**：NOT_SUPPORTED、NEVER、SUPPORTS（无事务环境）会导致事务失效。NOT_SUPPORTED强制非事务执行、挂起原有事务；NEVER强制要求无事务，存在事务直接报错；SUPPORTS在无外层事务时，自身无事务，异常无法回滚。

## 8.6 多线程事务失效原理

### 8.6.1 底层核心原理

Spring事务的底层存储依赖**ThreadLocal**，事务上下文（事务连接、事务状态、回滚标记）是**线程私有**的。

主线程开启事务后，事务信息存储在主线程的ThreadLocal中。子线程是全新的线程，无法获取主线程的事务上下文、数据库连接、事务状态，因此**子线程无法加入主线程事务，也无法共享事务**。

最终现象：主线程、子线程事务**相互独立**，子线程异常不会回滚主线程事务，主线程异常也无法回滚子线程已提交的数据，多线程事务完全失效，无法保证原子性。

### 8.6.2 失效代码示例

```java
@Service
public class BatchService {

    @Transactional(rollbackFor = Exception.class)
    public void batchSave() {
        // 主线程保存数据
        mainMapper.insert();

        // 开启子线程执行批量操作
        new Thread(() -> {
            childMapper.insert();
            // 子线程抛出异常
            throw new RuntimeException("子线程执行失败");
        }).start();
    }
}
```

执行结果：子线程异常，主线程事务正常提交，主线程数据入库，子线程数据根据自身逻辑提交/回滚，**整体事务原子性完全失效**。

### 8.6.3 生产解决方案

1. **避免多线程嵌套事务**：核心事务操作统一在主线程执行，子线程仅做非核心异步任务（日志、通知）。

2. **分布式事务方案**：多线程跨线程数据一致性，使用Seata、TCC、本地消息表等分布式事务方案。

3. **线程池事务上下文传递**：自定义线程池，传递ThreadLocal事务上下文（仅适用于特殊场景，不推荐常规使用）。

### 8.6.4 ⚠️避坑指南

- 绝对不要期望多线程共享同一个Spring事务，ThreadLocal线程隔离特性决定无法实现。

- 异步线程（@Async）默认事务独立，无法和主线程事务联动。

- 批量多线程处理数据，必须做好最终一致性补偿，不要依赖本地事务。

### 8.6.5 📌面试高频题

**问题**：为什么多线程环境下Spring事务无法保证原子性？

**参考答案**：Spring事务上下文基于ThreadLocal存储，属于线程私有资源，子线程无法获取主线程的事务连接和状态。多线程执行数据库操作时，各自开启独立事务，线程间事务互不感知、互不影响，无法实现统一提交、统一回滚，事务原子性失效。

## 8.7 嵌套事务、挂起事务失效场景

### 8.7.1 核心概念铺垫

**嵌套事务**：基于父事务的保存点执行的子事务，依赖父事务存在，子事务独立回滚不影响父事务，父事务回滚会连带子事务回滚。

**挂起事务**：当前线程存在已有事务时，新事务将原有事务挂起，独立创建全新事务执行，执行完成后恢复原有事务。

二者均为嵌套场景高频使用机制，但配置不当、场景误用会直接导致事务失效、数据错乱。

### 8.7.2 REQUIRES_NEW 挂起事务失效场景

`REQUIRES_NEW`：每次新建独立事务，挂起原有事务。失效核心场景：**本类内部调用、异常被外层捕获**。

失效场景1：本类内部调用REQUIRES_NEW方法，代理失效，无法挂起事务，子事务合并到父事务。

失效场景2：子事务REQUIRES_NEW执行成功，父事务抛出异常回滚，开发者期望子事务独立提交，但因嵌套调用异常传递问题，导致数据错乱。

### 8.7.3 NESTED 嵌套事务失效场景

`NESTED`：基于父事务保存点的嵌套事务，失效核心场景：

1. **无外层事务时使用NESTED**：无法创建保存点，降级为普通事务，失去嵌套特性。

2. **数据库不支持保存点**：MyISAM引擎不支持事务保存点，NESTED完全失效，仅InnoDB支持。

3. **外层事务手动回滚**：父事务回滚会清空所有子事务保存点，子事务无法独立提交。

### 8.7.4 嵌套事务通用避坑点

- 嵌套事务必须**跨类调用**，本类调用代理失效，嵌套机制完全不生效。

- REQUIRES_NEW独立事务，子事务提交后，父事务回滚不会影响子事务数据。

- NESTED嵌套事务依赖保存点，父事务全局回滚，所有子事务全部回滚。

# 9. 嵌套事务、事务挂起、事务恢复底层原理

事务挂起、恢复、嵌套保存点是Spring高阶事务核心原理，也是面试压轴考点、生产复杂业务（多级嵌套事务、独立子事务）的核心支撑。本章从源码层面拆解`REQUIRES_NEW`事务挂起恢复流程、`NESTED`保存点机制，深度对比两大高频嵌套传播机制的核心差异，彻底解决嵌套事务认知误区。

## 9.1 REQUIRES_NEW 事务挂起原理

### 9.1.1 定义与核心特性

`REQUIRES_NEW` 传播机制：**无论当前是否存在事务，都会创建一个全新的独立事务**。如果当前线程已有正在运行的事务，会**挂起原有事务**，执行新事务逻辑，新事务执行完成（提交/回滚）后，**恢复原有挂起的事务**。

核心特性：新事务与原事务**完全独立、互不影响**，拥有独立的数据库连接、独立的事务状态、独立的提交/回滚权限。

### 9.1.2 底层执行流程（源码级拆解）

Spring事务核心处理器`TransactionInterceptor`执行嵌套`REQUIRES_NEW`事务时，完整流程分为5步：

1. **获取当前事务上下文**：从ThreadLocal中获取外层已存在的事务信息（连接、状态、隔离级别）。

2. **挂起原有事务**：调用`suspend()`方法，将原有事务上下文从ThreadLocal中移除，封装为挂起事务对象保存，此时外层事务暂停执行，数据库连接被释放挂起。

3. **创建全新独立事务**：获取新的数据库连接，开启全新事务，初始化独立事务状态，执行当前方法业务逻辑。

4. **新事务收尾**：当前方法执行完毕，根据执行结果完成独立提交或回滚，新事务彻底结束。

5. **恢复原有事务**：调用`resume()`方法，将之前挂起的外层事务上下文重新放回ThreadLocal，外层事务继续执行。

### 9.1.3 事务挂起核心源码逻辑

Spring通过`TransactionSynchronizationManager`管理事务上下文，挂起核心逻辑：

```java
// 挂起当前事务核心方法
protected final TransactionStatus suspend(Object transaction) {
    // 1. 获取当前线程事务资源
    TransactionSynchronizationManager.unbindResource(this.dataSource);
    // 2. 清空当前线程事务上下文
    TransactionSynchronizationManager.clear();
    // 3. 返回被挂起的事务状态，用于后续恢复
    return new SuspendedResourcesHolder(...);
}
```

### 9.1.4 生产场景与价值

适用场景：主业务事务回滚，但**子业务日志、记录、告警数据需要永久留存**的场景。例如：订单创建失败（主事务回滚），需要留存订单失败日志（子事务独立提交，不随主事务回滚）。

核心价值：实现**局部业务数据持久化、全局业务数据回滚**的差异化事务控制，解决单一事务无法灵活拆分的问题。

## 9.2 NESTED 嵌套事务保存点原理

### 9.2.1 定义与核心特性

`NESTED` 嵌套事务：必须依赖**外层已有事务**才能生效，不会创建全新独立事务，而是在原有事务中**创建数据库保存点（Savepoint）**，子事务基于保存点执行。

核心特性：子事务是父事务的子集，共用同一个数据库连接、同一个事务，无事务挂起操作，仅通过保存点实现局部回滚。

### 9.2.2 保存点执行底层流程

1. **校验外层事务**：检测当前线程是否存在有效事务，无事务则降级为普通事务。

2. **创建事务保存点**：在父事务执行链路中，创建唯一标记的数据库Savepoint保存点，记录当前事务执行状态。

3. **执行子事务逻辑**：执行嵌套方法的业务代码，所有数据库操作归属父事务。

4. **子事务异常处理**：子事务抛出异常，仅**回滚到当前保存点**，不影响保存点之前的父事务数据。

5. **事务收尾**：子事务执行成功，释放保存点；父事务最终统一提交/全局回滚。

### 9.2.3 保存点核心原理

数据库InnoDB引擎原生支持Savepoint保存点机制，Spring NESTED事务是对数据库保存点的封装：

- 保存点可以记录事务执行的某个节点状态。

- 支持局部回滚：仅回滚到指定保存点，无需回滚整个事务。

- 所有嵌套子事务共用父事务连接，无独立事务资源。

### 9.2.4 生产适用场景

适用于**多级业务嵌套，子业务失败不影响主业务**的场景。例如：批量处理用户数据，单条用户处理失败仅回滚当前用户操作，不影响批量其他数据，主事务最终可正常提交。

## 9.3 REQUIRES_NEW 与 NESTED 核心区别（面试高频）

`REQUIRES_NEW` 和 `NESTED` 是Spring嵌套事务最易混淆的两个传播机制，也是面试必问考点，本节从底层原理、事务独立性、回滚规则、资源占用、适用场景5个维度全面对比，同时总结核心面试答题要点。

### 9.3.1 全方位对比表格

| 对比维度           | REQUIRES_NEW                   | NESTED                                 |
| ------------------ | ------------------------------ | -------------------------------------- |
| **事务本质**       | 全新独立事务                   | 父事务的子事务（保存点机制）           |
| **事务资源**       | 独立数据库连接、独立事务状态   | 共用父事务连接、共用事务               |
| **事务挂起**       | 会挂起原有事务，执行完成后恢复 | 无事务挂起，延续当前事务               |
| **子事务回滚**     | 仅自身回滚，完全不影响父事务   | 仅回滚到当前保存点，父事务前置数据保留 |
| **父事务回滚**     | 不影响已提交的子事务数据       | 全局回滚，所有子事务操作全部回滚       |
| **无外层事务场景** | 直接新建独立事务，正常生效     | 降级为普通事务，无保存点特性           |
| **底层依赖**       | Spring事务挂起/恢复机制        | 数据库Savepoint保存点机制              |

### 9.3.2 核心逻辑案例对比

#### 案例1：子事务异常，父事务正常执行

REQUIRES_NEW：子事务独立回滚，父事务不受影响，继续执行最终提交。

NESTED：子事务回滚到保存点，父事务前置数据保留，父事务可正常提交。

#### 案例2：子事务正常，父事务异常回滚

REQUIRES_NEW：子事务已独立提交，数据永久入库，父事务回滚不影响。

NESTED：父事务全局回滚，子事务所有操作全部回滚，数据无留存。

### 9.3.3 💡生产最佳实践

- 需要**子事务数据独立持久化**（日志、记录），选择REQUIRES_NEW。

- 需要**局部回滚、全局统一一致性**（批量操作），选择NESTED。

- NESTED必须保证数据库为InnoDB引擎，否则失效。

- 二者均需要跨类调用，本类调用代理失效，嵌套机制不生效。

### 9.3.4 📌面试终极答题模板

**问题**：REQUIRES_NEW和NESTED的区别？

**参考答案**：

1. **事务独立性不同**：REQUIRES_NEW是全新独立事务，拥有独立数据库连接；NESTED是父事务的子事务，共用事务连接。

2. **底层机制不同**：REQUIRES_NEW基于事务挂起与恢复机制；NESTED基于数据库保存点机制。

3. **回滚规则不同**：子事务异常，二者均不影响父事务前置逻辑；父事务异常，REQUIRES_NEW已提交的子事务不回滚，NESTED所有子事务全部随父事务回滚。

4. **无外层事务表现不同**：REQUIRES_NEW正常新建事务；NESTED降级为普通事务。

# 10. 分布式事务基础铺垫（进阶拓展）

分布式事务是微服务架构下必须掌握的核心知识点，所有微服务之间的跨库、跨服务数据操作，都离不开分布式事务的支撑。本节从本地事务短板切入，逐步拆解分布式事务的核心问题，厘清Spring事务与分布式事务的从属关系，为后续分布式事务解决方案学习奠定基础。

## 10.1 本地事务局限性

本地事务也叫**数据库事务**，是Spring事务管理的基础底层依赖，本质是单一数据库、单一服务内的事务机制，遵循ACID四大特性，由数据库原生支持，Spring仅做统一封装和声明式管理。在单体架构、单数据库场景下，本地事务稳定高效、无额外开销，但在分布式场景中存在致命的局限性，无法保障数据一致性。

### 10.1.1 本地事务核心适用场景

想要清晰认知局限性，首先要明确本地事务的有效边界。本地事务仅适用于**单一应用、单一数据库、单次数据库连接**的业务场景。例如单体商城的下单操作：在同一个MySQL数据库中，完成订单新增、库存扣减、余额扣除，通过Spring的`@Transactional`注解即可实现要么全部成功、要么全部回滚，数据一致性可以得到完美保障。

其核心优势是**性能高、无中间件依赖、稳定性强、实现简单**，是单体应用的最优事务方案，无任何额外运维和开发成本。

### 10.1.2 分布式场景下的核心局限性

随着业务迭代，系统会进行服务拆分、数据库分库分表，此时业务操作会跨越多个服务、多个数据库，本地事务的边界被彻底打破，核心局限性主要体现在以下四点：

**1. 仅支持单数据库连接，无法跨库事务**

本地事务的事务上下文绑定的是单个数据库连接，一个事务只能操作一个数据库。如果业务需要同时操作订单库、库存库、用户库三个独立数据库，本地事务只能分别在三个库执行独立事务，任意一个库操作失败，已执行成功的库无法回滚，直接导致数据不一致。

**2. 仅支持单服务内事务，无法跨服务事务**

微服务架构下，业务流程往往需要多个服务协同完成。比如用户下单流程：订单服务创建订单、库存服务扣减库存、支付服务完成扣款。三个服务是独立部署、独立运行的，各自拥有独立的Spring上下文和数据库连接。本地事务只能管控当前服务的数据库操作，无法远程控制其他服务的事务提交与回滚。若支付服务扣款失败，订单服务和库存服务已提交的数据无法回滚，出现超卖、无效订单等线上问题。

**3. 事务边界受限，无法适配分库分表场景**

当单表数据量过大，需要进行分库分表（如按用户ID哈希分表），同一条业务数据会分散在不同的分片库、分片表中。此时一次业务操作跨越多个分片，本地单一数据库事务无法覆盖所有分片数据，导致分片之间数据一致性无法保障。

**4. 无网络容错机制，分布式网络异常会导致数据错乱**

本地事务基于本地内存和数据库连接，无网络延迟、断连、超时等问题。但分布式场景下，服务调用依赖网络，存在网络抖动、超时、重试、节点宕机等问题。本地事务无法感知远程服务状态，无法处理网络异常导致的部分成功、部分失败问题。

### 10.1.3 生活化类比总结

可以把**本地事务**理解为「单人办公室办公」：一个人在同一个房间内完成所有工作，做错了可以直接作废重来，全程可控、无差错。而**分布式业务场景**是「多人跨部门协同办公」：多个人员在不同房间、不同部门完成同一项工作，单人无法控制其他人的工作结果，一旦有人出错，整体工作就会出现漏洞，这就是本地事务的核心短板。

### 💡最佳实践 & ⚠️避坑指南

1. 只要是**单服务、单库**业务，优先使用本地事务，无需引入分布式事务，避免过度设计导致性能损耗和复杂度提升；

2. 一旦业务涉及**跨库、跨服务、跨分片**操作，必须摒弃本地事务思维，使用分布式事务方案，否则100%会出现数据不一致问题。

## 10.2 分布式事务常见问题

分布式事务的核心本质是：**在跨网络、跨进程、跨数据库的分布式系统中，保障多个独立事务的最终数据一致性**。由于分布式系统存在网络不可靠、服务节点宕机、调用超时、时钟不一致等固有问题，衍生出一系列本地事务不存在的核心问题，也是分布式事务需要解决的核心痛点。

### 10.2.1 网络不可靠导致的一致性问题

本地事务的执行环境是绝对可靠的本地环境，而分布式系统的所有服务调用、数据交互都依赖网络，网络具有**延迟、抖动、丢包、超时、断连**等不可控特性，这是分布式事务所有问题的根源。

典型场景：下单业务中，订单服务成功创建订单并提交事务，随后调用库存服务扣减库存。此时网络突然中断，库存服务未收到请求或执行失败。此时订单数据已持久化，库存数据未变更，出现**订单存在、库存未扣减**的数据不一致问题。

更复杂的场景：服务请求成功执行，但响应报文网络丢失，调用方判定为超时失败，触发重试机制，导致**接口幂等失效、数据重复提交**。

### 10.2.2 服务节点故障导致的事务中断

分布式系统由多个独立服务节点组成，节点随时可能因为重启、宕机、OOM、服务器故障等原因中断服务，导致整体业务流程执行中断，出现部分事务提交、部分事务未执行的情况。

典型场景：转账业务，A服务扣减用户余额成功后，服务突然宕机，后续B服务增加目标用户余额的逻辑未执行，最终出现**资金凭空消失**的严重数据错误，这是生产环境高危故障之一。

### 10.2.3 超时与重试引发的幂等性问题

微服务架构中，为了提升系统容错性，所有远程调用都会配置超时时间和重试机制。但超时属于**结果未知状态**：无法判定是对方未执行，还是执行成功后响应丢失。

此时自动重试会导致同一笔业务多次执行，若无幂等控制，会出现重复下单、重复扣款、重复扣库存等问题。而本地事务中不存在超时重试场景，无需考虑此类问题。

### 10.2.4 数据状态不一致的典型场景汇总

为了直观理解，汇总生产中最常见的三类分布式事务数据不一致问题：

| 问题类型           | 触发原因                         | 业务现象                                 |
| ------------------ | -------------------------------- | ---------------------------------------- |
| 部分成功、部分失败 | 跨服务执行中途网络中断、服务宕机 | 部分服务事务提交，部分未执行，数据对半错 |
| 重复数据产生       | 调用超时重试、接口无幂等         | 重复下单、重复扣款、流水重复             |
| 数据状态滞后不一致 | 异步调用、消息队列消费延迟       | 主数据已更新，附属数据长期未同步         |

### 10.2.5 分布式事务的核心矛盾

分布式系统遵循**CAP理论**：一致性、可用性、分区容错性三者无法同时满足。分布式事务的核心矛盾就是：**在保证系统高可用、支持分区容错的前提下，最大限度保障数据一致性**。

本地事务可以实现强一致性，但分布式事务无法做到绝对强一致，只能根据业务场景选择**最终一致性**或**柔性事务**，这也是分布式事务方案复杂度远高于本地事务的核心原因。

### 📌面试考点

**问题：分布式事务产生的根本原因是什么？**

**参考答案：**

1. 根本原因：分布式系统的**网络不可靠性**和**服务节点独立性**；

2. 直接原因：业务操作跨越多个服务、多个数据库，本地事务的单连接、单进程事务机制无法覆盖；

3. 底层理论：CAP理论限制，分布式系统必须保证分区容错性，无法同时满足强一致性和高可用性，进而产生数据一致性问题。

## 10.3 Spring 事务与分布式事务关系

很多开发者存在认知误区：认为Spring事务只能做本地事务，和分布式事务无关，这是完全错误的。Spring事务是一套**通用的事务管理框架**，既支持本地事务，也可以整合各类分布式事务方案，是分布式事务落地的基础载体。本节将彻底厘清二者的从属关系、区别与联动逻辑。

### 10.3.1 核心定义区分

**1. Spring事务**

Spring事务是**框架层面的事务管理规范**，提供统一的事务编程模型、声明式事务注解、事务传播机制、隔离级别管理等能力。其核心作用是**统一事务操作入口，屏蔽底层事务实现差异**。

底层既可以对接数据库本地事务（JDBC事务），也可以对接分布式事务（XA事务、Seata事务等），属于**上层工具框架**。

**2. 分布式事务**

分布式事务是**业务场景下的事务类型**，是针对跨库、跨服务数据一致性问题的解决方案，属于**底层业务问题场景**。其实现需要依赖各类中间件（Seata、RocketMQ）、协议（XA、TCC、SAGA），最终落地仍需要依托Spring事务的编程模型。

### 10.3.2 二者层级关系

用层级逻辑可以清晰概括：**Spring事务是上层封装，分布式事务是底层实现场景**

1. 基础层：数据库本地事务、各类分布式事务协议/中间件（底层实现）；

2. 封装层：Spring事务框架，统一封装所有事务的开启、提交、回滚、传播逻辑；

3. 应用层：开发者通过`@Transactional`注解，即可使用本地事务或分布式事务，无需感知底层实现差异。

### 10.3.3 核心区别与关联

通过表格直观对比Spring本地事务与Spring分布式事务的差异：

| 对比维度   | Spring本地事务         | Spring分布式事务                         |
| ---------- | ---------------------- | ---------------------------------------- |
| 适用场景   | 单服务、单数据库       | 跨服务、跨库、分库分表场景               |
| 底层依赖   | JDBC数据库原生事务     | Seata、XA、消息队列等分布式方案          |
| 一致性级别 | 强一致性               | 最终一致性（绝大多数场景）               |
| 性能开销   | 极低，无额外中间件开销 | 较高，需要网络交互、日志记录             |
| 使用方式   | @Transactional直接使用 | 整合分布式中间件后，仍使用@Transactional |

### 10.3.4 核心联动逻辑（重点）

1. **分布式事务离不开Spring事务**：市面上所有主流分布式事务框架（Seata），均完全适配Spring事务体系，沿用Spring的事务传播机制、注解规范、回滚规则，开发者无需学习全新的事务编程方式；

2. **Spring事务拓展了分布式能力**：Spring并未局限于本地JDBC事务，通过事务抽象接口，支持拓展各类分布式事务实现，是Java项目分布式事务落地的标准载体；

3. **事务传播机制依然生效**：分布式场景中，Spring的REQUIRED、REQUIRES_NEW等传播机制，依然可以控制跨方法、跨服务的事务嵌套逻辑。

### ⚠️避坑指南

1. 不要认为加了`@Transactional`就可以解决分布式事务问题：原生Spring注解仅支持本地事务，必须整合Seata等中间件后，才能实现分布式事务管控；

2. 分布式事务场景中，依然需要遵循Spring事务的使用规范，否则会出现分布式事务失效、回滚失败等问题。

### 📌面试考点

**问题：Spring的@Transactional注解能实现分布式事务吗？为什么？**

**参考答案：**

原生的`@Transactional`注解**无法实现分布式事务**，仅能实现单库本地事务。因为原生Spring事务的事务上下文仅绑定本地JDBC连接，无法跨服务、跨库传递事务状态。

但Spring事务支持拓展，整合Seata等分布式事务框架后，`@Transactional`可以用于分布式事务场景，框架会通过全局事务ID传递、分支事务注册、全局回滚/提交机制，实现分布式数据一致性。

# 11. 生产环境事务最佳实践与避坑总结

Spring事务的基础用法简单，但在生产环境中，**90%的事务故障都不是框架bug，而是使用不规范导致的**。包括事务失效、大事务拖垮系统、事务不回滚、数据脏读、性能卡顿等线上问题，均有固定的成因和解决方案。本章从注解规范、大事务优化、失效排查、面试高频问题四个维度，总结生产可落地的事务最佳实践，彻底解决线上事务隐患。

## 11.1 事务注解规范使用姿势

`@Transactional`注解是Spring事务的核心入口，绝大多数事务问题都源于注解使用不规范、参数配置错误、使用场景错误。本节整理生产环境**强制统一的注解使用规范**，覆盖注解位置、参数配置、回滚规则、访问权限等所有细节。

### 11.1.1 注解标准使用位置

生产环境**强制规范**：事务注解统一加在**Service层公共业务方法上**，禁止加在Controller层、Mapper层、私有方法上。

**原因解析**：

1. Controller层：仅负责参数接收、请求转发，不承载业务逻辑，加事务会导致事务范围过大，包含参数校验、日志打印等非数据库操作，降低系统性能；

2. Mapper层：仅负责单条SQL执行，业务层通常是多SQL组合操作，Mapper加事务无法覆盖完整业务流程；

3. 私有方法：Spring事务基于AOP动态代理实现，私有方法无法被代理拦截，直接导致**事务彻底失效**，是高频坑点。

### 11.1.2 事务回滚规则规范（核心）

Spring默认事务回滚规则：**仅对RuntimeException（运行时异常）和Error回滚，对CheckedException（受检异常）不回滚**。这是生产中事务不回滚的核心原因之一。

**生产强制规范配置**：手动指定回滚异常，覆盖所有业务异常场景，杜绝漏回滚。

```java
import org.springframework.transaction.annotation.Transactional;

/**
 * 生产标准事务注解配置
 * rollbackFor = Exception.class：所有异常均触发回滚
 * noRollbackFor：指定无需回滚的业务异常（如业务提示异常）
 */
@Transactional(rollbackFor = Exception.class)
public void businessMethod() {
    // 核心业务数据库操作
}

```

**原理说明**：业务开发中会大量自定义受检异常，如果不手动指定rollbackFor，抛出此类异常时，Spring不会触发事务回滚，导致数据库数据已提交，程序抛出异常，出现数据不一致。

### 11.1.3 事务传播机制规范选型

生产环境99%的业务场景，仅需使用两种传播机制，禁止随意使用冷门传播机制导致事务嵌套混乱：

1. **REQUIRED（默认）**：适用于95%的普通业务，有事务则加入，无事务则新建事务；

2. **REQUIRES_NEW**：适用于独立日志记录、消息记录、流水记录场景，需要新开独立事务，不受主事务回滚影响；

3. 禁止滥用NESTED、SUPPORTS、NOT_SUPPORTED等传播机制，极易导致事务嵌套失效、数据错乱。

### 11.1.4 超时时间与隔离级别规范

**1. 事务超时时间**

生产必须配置事务超时时间，默认超时时间为-1（永不超时），会导致卡死的长事务一直占用数据库连接，拖垮整个数据库。标准配置为`timeout = 10`（10秒超时），根据业务复杂度微调。

**2. 隔离级别**

默认使用数据库默认隔离级别（MySQL默认可重复读），**禁止随意修改隔离级别**。仅在核心金融、对账场景，需要解决幻读问题时，手动指定为READ_COMMITTED或SERIALIZABLE。

### 11.1.5 完整生产标准注解模板

```java
import org.springframework.transaction.annotation.Transactional;

/**
 * 生产环境标准事务方法
 * 1. 所有异常触发回滚，杜绝漏回滚
 * 2. 配置超时时间，防止长事务阻塞
 * 3. 使用默认传播机制，适配绝大多数业务
 * 4. 仅作用于public公共方法
 */
@Transactional(rollbackFor = Exception.class, timeout = 10)
public void createOrder(OrderDTO orderDTO) {
    // 1. 校验订单参数
    // 2. 新增订单数据
    // 3. 扣减库存
    // 4. 生成支付流水
}

```

### ⚠️避坑指南汇总

1. 禁止在**同一个类内调用带事务的方法**：A类无事务方法调用本类有事务方法，AOP代理失效，事务完全失效；

2. 禁止事务方法内部捕获异常不抛出：try-catch捕获所有异常且不向上抛出，Spring无法感知异常，不会触发回滚；

3. 禁止事务方法使用final、static修饰：无法被AOP动态代理，事务失效。

## 11.2 大事务、长事务优化方案

大事务、长事务是生产环境**性能杀手TOP1**。所谓长事务，指事务执行时间过长；大事务指事务内包含过多数据库操作、非数据库操作。长事务会占用数据库连接、锁资源不释放，导致数据库连接池耗尽、行锁升级为表锁、接口超时、系统雪崩等严重线上故障。本节提供可直接落地的优化方案，从根源解决事务性能问题。

### 11.2.1 大事务/长事务的核心危害

1. **占用数据库连接池**：数据库连接数量有限，长事务持续占用连接，会导致正常业务无法获取连接，接口大面积超时；

2. **锁等待、锁升级**：事务未提交，行锁不会释放，其他请求阻塞等待，并发量大时直接引发死锁；长时间持有行锁，MySQL会自动升级为表锁，锁住全表数据；

3. **事务日志堆积**：长事务会导致undo日志、redo日志持续堆积，占用磁盘空间，降低数据库读写性能；

4. **数据脏读风险增加**：事务执行时间越长，并发修改的概率越高，数据一致性风险越大。

### 11.2.2 大事务产生的核心原因

1. 事务范围过大：`@Transactional`方法内包含大量**非数据库操作**，如远程调用、文件解析、日志打印、循环遍历、参数校验等耗时操作；

2. 批量操作未拆分：一次性批量新增、修改上万条数据，单事务执行时间过长；

3. 事务嵌套冗余：多层事务嵌套，无效延长事务生命周期；

4. 业务逻辑臃肿：单个事务方法承载过多无关业务逻辑。

### 11.2.3 生产落地优化方案（全维度）

**1. 拆分事务范围，剥离非DB操作（核心最优解）**

严格遵循：**事务方法内只保留数据库CRUD操作**，所有耗时的非数据库操作全部提取到事务外执行，最大限度缩短事务执行时间。

反例（错误写法，长事务根源）：

```java
@Transactional(rollbackFor = Exception.class)
public void createOrder() {
    // 耗时的远程调用（非DB操作，严重拖长事务）
    userFeign.getUserInfo();
    // 耗时的文件解析
    parseFile();
    // 核心DB操作（仅少量代码）
    orderMapper.insert();
    stockMapper.update();
}

```

正例（标准优化写法）：

```java
// 非事务方法，执行所有非DB耗时操作
public void createOrder() {
    // 远程调用、文件解析提前执行
    userFeign.getUserInfo();
    parseFile();
    // 仅将DB操作放入事务，最小化事务范围
    doSaveOrder();
}

// 纯数据库事务方法，执行极快
@Transactional(rollbackFor = Exception.class)
public void doSaveOrder() {
    orderMapper.insert();
    stockMapper.update();
}

```

**2. 大批量操作拆分，分批次提交事务**

上万条数据批量处理时，禁止单事务一次性执行，采用**分批处理、分批提交**的方式，控制单事务数据量（建议单事务100-500条），避免事务过大。

**3. 禁止事务内远程调用**

远程Feign、RPC调用网络延迟不可控，一旦网络抖动，会无限拉长事务时间。所有远程调用必须放在事务外执行，必要时可先查询缓存、预校验，减少事务内耗时。

**4. 关闭不必要的事务嵌套**

统一事务传播机制，内层方法无需重复加事务注解，避免无效事务嵌套，缩短事务生命周期。

**5. 配置合理的事务超时时间**

根据业务场景设置超时时间，超时自动回滚，避免卡死事务长期占用资源，常规业务10秒，批量任务30秒。

### 11.2.4 大事务排查定位方式

1. **日志排查**：开启Spring事务日志，监控事务开启、提交、回滚时间，定位执行过长的事务；

2. **数据库慢日志**：通过MySQL slow_log查询执行时间过长的事务SQL；

3. **连接池监控**：监控数据库连接池活跃连接、空闲连接，排查长期占用连接的事务。

### 💡最佳实践

事务优化的核心原则：**事务范围最小化、事务时间最短化**。能不加事务就不加事务，能缩小事务范围就缩小范围，绝不滥用大事务。

## 11.3 事务失效排查万能思路

事务失效是面试和生产最高频问题，很多开发者遇到事务失效只能盲目排查，无固定思路。本节总结**万能排查流程**，覆盖100%的事务失效场景，按照步骤排查即可快速定位问题，同时梳理所有失效成因与解决方案。

### 11.3.1 事务失效核心本质

Spring事务基于**AOP动态代理**实现，所有事务失效的根本原因只有两个：

1. **AOP代理未生效**：目标方法未被代理拦截，事务逻辑完全没执行；

2. **代理生效但回滚规则不满足**：代理拦截成功，但未触发事务回滚，导致数据提交。

### 11.3.2 万能排查步骤（按优先级排序）

**第一步：检查方法访问权限**

确认事务方法是**public公共方法**，private、protected、default方法无法被AOP代理，事务必然失效。

**第二步：检查是否存在本类自调用**

当前类的无事务方法调用本类有事务方法，不会走代理对象，事务失效。解决方案：通过Spring上下文获取当前Bean，通过代理对象调用方法。

```java
// 解决本类自调用事务失效方案
@Autowired
private ApplicationContext context;

public void noTxMethod() {
    // 从上下文获取代理Bean，而非直接this调用
    OrderService orderService = context.getBean(OrderService.class);
    orderService.txMethod();
}

```

**第三步：检查异常是否被捕获且未抛出**

事务方法内部try-catch捕获所有异常，且没有手动抛出异常，Spring无法感知错误，不会回滚。

解决方案：捕获异常后手动抛出，或手动触发事务回滚：

```java
try {
    // 业务操作
} catch (Exception e) {
    // 手动触发事务回滚
    TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
    throw new RuntimeException("业务执行失败", e);
}

```

**第四步：检查回滚异常配置是否正确**

未配置`rollbackFor = Exception.class`，抛出受检异常时事务不回滚，补充统一回滚配置即可。

**第五步：检查方法是否被重写/修饰**

被final、static修饰的方法无法被动态代理重写，事务失效，禁止事务方法使用此类修饰符。

**第六步：检查传播机制是否导致失效**

使用NOT_SUPPORTED、NEVER等传播机制，会强制关闭事务，导致事务失效。

**第七步：检查数据库引擎是否支持事务**

MySQL MyISAM引擎不支持事务，只有InnoDB支持，表引擎错误会导致事务完全失效。

### 11.3.3 高频事务失效场景汇总表

| 失效场景               | 失效原因                   | 解决方案                          |
| ---------------------- | -------------------------- | --------------------------------- |
| 私有方法加事务         | AOP无法拦截私有方法        | 改为public方法                    |
| 本类自调用事务方法     | 直接this调用，未走代理对象 | 通过上下文获取代理Bean调用        |
| 异常被try-catch未抛出  | Spring无法感知异常         | 手动抛异常或主动回滚              |
| 受检异常不回滚         | 默认仅回滚运行时异常       | 配置rollbackFor = Exception.class |
| final/static方法加事务 | 无法被动态代理重写         | 去除修饰符                        |
| MyISAM引擎表事务失效   | 引擎不支持事务             | 改为InnoDB引擎                    |

## 11.4 面试高频问答汇总

本节汇总本章及Spring事务进阶**企业高频面试题**，包含问题、核心答案、答题思路，适配校招、社招、面试复试场景，所有答案贴合生产实际，拒绝纸上谈兵。

### Q1：为什么本类中调用加了@Transactional的方法，事务会失效？

**答题答案：**

1. Spring事务基于**JDK/CGLIB动态代理**实现，只有通过Spring容器的**代理对象**调用方法，才会触发AOP切面的事务增强逻辑；

2. 本类内部通过this.方法名()调用，是**原生对象直接调用**，没有经过代理对象，AOP切面无法拦截，事务开启、提交、回滚逻辑都不会执行，最终导致事务失效；

3. 解决方案：通过Spring上下文获取当前Bean的代理对象，再调用事务方法，或拆分方法到不同类中。

### Q2：事务方法中try-catch捕获异常后，事务为什么不回滚？如何解决？

**答题答案：**

1. 核心原理：Spring事务的回滚触发机制是**捕获方法抛出的异常**，如果开发者手动try-catch捕获所有异常且不向外抛出，Spring的事务切面无法感知到业务异常，会判定业务执行正常，自动提交事务；

2. 解决方案两种：

① 捕获异常后，手动向上抛出运行时异常，触发自动回滚；

② 捕获异常后，通过`TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()`手动标记事务回滚。

### Q3：长事务/大事务会引发哪些线上问题？生产如何优化？

**答题答案：**

**引发问题：**

1. 长期占用数据库连接池，导致连接耗尽，正常业务接口超时雪崩；

2. 长期持有行锁，引发锁等待、死锁，甚至行锁升级为表锁，阻塞全表读写；

3. 数据库事务日志堆积，影响数据库性能，增加数据恢复风险。

**优化方案：**

1. 最小化事务范围，剥离事务内所有非数据库耗时操作（远程调用、文件解析、日志打印）；

2. 批量数据拆分处理，分批次提交事务，避免单事务数据量过大；

3. 事务内禁止远程RPC/Feign调用，规避网络延迟导致的事务拉长；

4. 配置合理的事务超时时间，自动释放卡死事务。

### Q4：原生Spring @Transactional能否解决分布式事务？为什么？

**答题答案：**

原生注解**无法解决分布式事务**。

1. 原生Spring事务的事务上下文仅绑定**本地单个JDBC数据库连接**，只能管控单库、单服务的事务；

2. 分布式场景跨服务、跨库操作，多个独立数据库连接无法通过本地事务统一管控，无法传递事务状态、实现统一回滚和提交；

3. 但Spring事务具备良好的拓展性，整合Seata等分布式事务中间件后，可沿用@Transactional注解实现分布式事务管控。

### Q5：生产中如何快速排查事务失效问题？

**答题答案：**

按照优先级快速排查：

1. 检查方法是否为public，排除私有方法事务失效；

2. 检查是否本类自调用，未走代理对象；

3. 检查异常是否被try-catch拦截未抛出；

4. 检查rollbackFor配置，是否覆盖所有异常；

5. 检查方法是否被final/static修饰；

6. 检查表引擎是否为InnoDB，确认支持事务。

# 本章总结

本章作为Spring事务管理的进阶核心内容，聚焦生产落地与面试核心，完成了从本地事务到分布式事务的进阶拓展，同时梳理了生产全维度事务规范与故障解决方案，核心知识点总结如下：

1. **本地事务边界清晰**：本地事务仅适用于单服务、单库场景，跨库、跨服务、分库分表场景下存在致命局限性，无法保障分布式数据一致性；

2. **分布式事务核心痛点**：分布式事务问题根源是网络不可靠、服务节点独立，衍生出部分成功、重复提交、数据不一致等问题，受CAP理论限制，多采用最终一致性方案；

3. **Spring与分布式事务关系**：Spring事务是上层通用框架，原生仅支持本地事务，可拓展整合分布式中间件，沿用统一的注解和传播机制；

4. **生产事务规范**：事务注解统一作用于Service层public方法，配置全局异常回滚、合理超时时间，杜绝不规范用法；

5. **大事务优化核心**：遵循事务最小化原则，剥离非数据库耗时操作、拆分批量事务、禁止事务内远程调用，从根源解决长事务性能问题；

6. **事务失效万能排查**：所有失效问题均围绕AOP代理失效、回滚规则不满足两大核心，固定排查流程可快速定位所有线上问题；

7. **面试核心**：掌握事务失效成因、分布式事务基础、大事务优化方案，覆盖Spring事务进阶90%以上面试考点。
