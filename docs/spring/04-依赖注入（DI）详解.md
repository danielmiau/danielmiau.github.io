# 04-依赖注入（DI）详解

## 本章概述

在上一章《Bean生命周期与扩展点》中，我们了解了Bean从创建到销毁的完整流程，而**依赖注入（DI）**是整个Bean生命周期中最核心、最关键的环节，也是Spring IoC思想的**唯一落地手段**。

如果说IoC（控制反转）是Spring的**设计思想**（把对象控制权交给容器），那么DI（依赖注入）就是**具体实现手段**（容器自动给对象赋值、装配依赖）。没有依赖注入，IoC就是空谈，Spring所有的解耦、AOP、事务、自动配置特性都无法生效。

本章将从底层原理、三种注入方式、自动装配规则、主流注解对比、循环依赖核心难点、源码流程、生产坑点、面试真题全方位拆解DI。解决开发者日常最大的困惑：为什么官方不推荐字段注入？循环依赖到底怎么产生、如何解决？@Autowired和@Resource有什么区别？单例依赖多例为什么会失效？

读完本章，你将彻底吃透Spring依赖注入的底层逻辑，解决99%的Bean注入报错、循环依赖异常、作用域注入失效问题，同时拿下Java后端面试Spring DI全部高频考点。

## 1. 依赖注入核心基础概念

### 1.1 什么是DI依赖注入？和IoC的关系与区别

#### 1. 什么是DI

**DI（Dependency Injection，依赖注入）**：指Spring IoC容器主动将Bean所需要的依赖对象，自动赋值、注入到当前Bean中，无需开发者手动new对象、手动赋值。

通俗类比：以前我们需要什么工具自己去造（new 对象）；现在Spring是管家，提前把你需要的所有工具准备好，直接送到你手里（注入依赖）。

#### 2. DI与IoC的核心关系

很多开发者容易混淆两者，这里给出**标准答案**：

- **IoC 是思想/理论**：**控制权反转**，将对象的创建、管理、销毁权限，从开发者代码交给Spring容器。

- **DI 是实现/手段**：**依赖自动装配**，是IoC思想唯一的落地实现方式。

**核心结论**：IoC是目的，DI是手段；没有DI，IoC无法落地，二者是**思想与实现**的绑定关系。

### 1.2 传统New对象耦合问题 vs DI解耦优势

#### 1. 传统开发方式（硬编码new）

在没有Spring DI时，Java开发需要手动创建依赖对象，代码如下：

```java
// 传统硬编码方式，高度耦合
public class OrderService {
    // 主动new依赖对象，代码写死
    private UserService userService = new UserService();
}
```

**存在致命问题**：

- **高耦合**：OrderService强依赖UserService，修改、替换底层类需要改上层代码

- **难以扩展**：如需切换UserService不同实现，需要修改大量业务代码

- **无法统一管理**：对象频繁创建销毁，资源浪费、无法做单例复用

- **单元测试困难**：无法动态替换Mock对象，测试耦合严重

#### 2. DI注入方式（Spring推荐）

```java
// Spring DI方式，完全解耦
@Service
public class OrderService {
    // 无需手动new，容器自动注入
    @Autowired
    private UserService userService;
}

```

**核心优势**：

-**彻底解耦**：业务层无需关心对象创建，只关心业务逻辑

- **可拔插扩展**：替换实现类无需改动业务代码

- **统一资源管理**：容器统一单例复用、资源释放

- **便于测试**：可动态注入Mock对象，解耦单元测试

### 1.3 依赖注入核心设计思想与底层价值

#### 1. 核心设计思想

DI的核心思想可以总结为八个字：**被动依赖，外部装配**。

开发者的Bean只负责**声明需要什么依赖**，不负责**如何获取依赖**，所有依赖的查找、创建、赋值全部由容器全权负责。

#### 2. 底层设计价值

- **单一职责**：业务Bean只处理业务，容器处理对象管理，职责分离

- **依赖倒置**：上层业务不依赖下层实现，只依赖抽象接口，符合设计原则

- **高可拓展性**：为AOP、事务、动态代理、自动配置提供底层支撑

- **统一规范**：所有组件依赖管理统一标准，项目结构更整洁

### 1.4 依赖注入在框架中的整体运行定位

结合上一章Bean生命周期，DI的执行位置是**承上启下的核心节点**：

**Bean实例化（造空对象） → 依赖注入（填充属性） → 初始化阶段（执行业务初始化）**

如果没有DI属性填充，Bean只是一个空属性的空白对象，无法执行业务逻辑。所有的初始化方法、业务调用、代理增强，**全部依赖DI注入完成**。

从框架整体架构来看：DI是Spring生态的**地基能力**，SpringBoot自动配置、SpringCloud微服务组件、中间件整合，全部建立在依赖注入的基础之上。

## 2. 依赖注入三大实现方式

### 2.1 构造器注入（Constructor Injection）

#### 2.1.1 原理与执行时机

**原理**：通过类的构造方法传入依赖对象，在**对象实例化的同时**完成依赖赋值。

**执行时机**：三大注入方式中**最早执行**，在Bean反射创建对象的一瞬间同步完成注入，不存在空对象阶段。

底层逻辑：Spring解析BeanDefinition中的构造器参数 → 递归创建依赖Bean → 通过构造器反射实例化当前Bean并完成赋值。

#### 2.1.2 写法规范、省略@Autowired条件

**Spring4.3+ 核心特性**：当类中**仅有一个有参构造器**时，可以**省略 @Autowired 注解**，容器自动识别并完成构造器注入。

```java
@Service
public class OrderService {

    private final UserService userService;

    // 仅有一个有参构造，无需@Autowired
    public OrderService(UserService userService) {
        this.userService = userService;
    }
}

```

**强制规范**：构造器注入的依赖变量建议使用 **final** 修饰，保证依赖不可修改，线程安全。

#### 2.1.3 适用场景、强制依赖最佳实践

**适用场景**：项目中**必须的、不可缺失的核心依赖**（Service、Dao、核心工具类）。

**最佳实践**：所有业务核心强制依赖，**优先使用构造器注入**。如果缺少该依赖，Bean无法创建，项目启动直接报错，提前暴露问题，避免运行时空指针。

#### 2.1.4 优缺点与生产使用建议

**优点**：

1. 保证依赖**不可为空**，Bean初始化必然拥有完整依赖，杜绝空指针

2. 天然线程安全，final修饰变量不可修改

3. 完全遵循单一职责、依赖倒置原则

4. 单元测试友好，可手动传参构造对象，无需Spring容器

**缺点**：

1. 不适合可选依赖、动态变更的依赖

2. 大量依赖会导致构造器参数臃肿

**生产建议**：官方首选推荐，核心业务Bean统一使用构造器注入。

### 2.2 Setter方法注入（Setter Injection）

#### 2.2.1 原理与执行时机

**原理**：通过Java Bean规范的setXxx()方法完成依赖赋值。

**执行时机**：**构造器实例化完成之后、初始化阶段之前**执行。先创建空对象，后通过set方法填充属性。

Spring底层通过解析setter方法，自动匹配对应类型的Bean完成注入。

#### 2.2.2 适用可选依赖、动态变更依赖场景

**核心场景**：

1. **可选依赖**：该依赖可有可无，缺失不影响核心业务运行

2. **动态变更依赖**：运行期间需要动态修改、替换的依赖对象

3. 第三方组件、配置类的可选参数注入

```java
@Service
public class PayService {

    private LogService logService;

    // 可选依赖，可动态注入、动态修改
    public void setLogService(LogService logService) {
        this.logService = logService;
    }
}

```

#### 2.2.3 优缺点与使用边界

**优点**：

1. 支持动态修改依赖，灵活性极高

2. 适配可选依赖，不强制依赖加载

**缺点**：

1. 无法保证依赖非空，运行时可能出现空指针

2. 依赖可变，存在线程安全风险

**使用边界**：只用于可选、动态依赖，**禁止用于核心强制依赖**。

### 2.3 字段注入（Field Injection / @Autowired）

#### 2.3.1 底层实现原理：后置处理器+反射

字段注入就是我们日常最常用的 `@Autowired` 直接标注在字段上的注入方式。

**底层原理**：Spring内置的 **AutowiredAnnotationBeanPostProcessor** 后置处理器，在Bean实例化完成后，通过**Java反射机制**直接对字段进行赋值，绕过setter和构造器。

```java
@Service
public class UserService {
    // 字段注入，简洁高效
    @Autowired
    private RedisService redisService;
}

```

#### 2.3.2 开发便捷性与原生弊端

**优势**：代码极简、无需编写构造器和setter，开发效率最高。

**原生弊端**：

1. 依赖隐藏在类内部，无法直观看到类的所有依赖

2. 强耦合Spring框架，脱离容器无法运行

3. 极易出现循环依赖问题

#### 2.3.3 单元测试痛点、循环依赖隐患

**单元测试痛点**：字段注入基于Spring后置处理器，脱离Spring容器无法自动赋值，手动单元测试需要反射赋值，极其繁琐。

**循环依赖隐患**：字段注入是**Spring唯一推荐允许的循环依赖场景**，开发者容易滥用，导致项目隐性循环依赖，架构混乱。

#### 2.3.4 生产环境禁用/慎用原因剖析

**⚠️避坑指南**：阿里开发手册、Spring官方均**不推荐字段注入**。

核心原因：

1. 隐藏依赖，代码可读性差，无法快速判断类依赖复杂度

2. 容易写出非法循环依赖代码，埋下架构隐患

3. 不利于单元测试、代码解耦、组件复用

### 2.4 三种注入方式执行时机、优缺点、场景对比表

|注入方式|执行时机|空指针风险|循环依赖支持|适用场景|官方推荐度|
|---|---|---|---|---|---|
|构造器注入|实例化同步执行|无|不支持|核心强制依赖|⭐⭐⭐⭐⭐|
|Setter注入|实例化后初始化前|有|支持|可选、动态依赖|⭐⭐⭐⭐|
|字段注入|实例化后初始化前|有|支持|快速开发、临时场景|⭐⭐|
### 2.5 官方推荐注入规范与阿里开发手册约束

#### 1. Spring官方规范

1. 优先使用**构造器注入**保证依赖不可缺失

2. 可选依赖使用**Setter注入**

3. **禁止滥用字段注入**

#### 2. 阿里Java开发手册约束

- 禁止使用字段注入，必须使用构造器或Setter注入

- 强制依赖使用构造器注入，可选依赖使用Setter注入

- 依赖变量统一使用final修饰，保证线程安全

## 3. 依赖注入自动装配规则

### 3.1 ByType 按类型注入原理与匹配流程

**ByType（按类型装配）**是Spring **默认的自动装配规则**。

#### 1. 核心原理

Spring根据**目标字段的类型**，去IoC容器中匹配该类型的Bean，找到唯一Bean直接注入。

#### 2. 匹配流程

1. 获取当前字段的Class类型

2. 遍历容器中所有Bean，匹配类型一致的Bean

3. 匹配唯一：直接注入

4. 匹配多个：抛出 **NoUniqueBeanDefinitionException** 冲突异常

5. 匹配不到：抛出找不到Bean异常

#### 3. 优势

无需关心Bean名称，只关心类型，解耦性更强，是Spring首选装配方式。

### 3.2 ByName 按名称注入原理与Bean名称规则

**ByName（按名称装配）**：根据字段名称，匹配容器中同名的Bean。

#### 1. 核心原理

Spring优先获取字段名，去容器中查找**Bean名称完全一致**的对象，找到则注入。

#### 2. Bean名称默认规则

- 类名首字母小写：UserService → userService

- @Bean注解默认方法名作为Bean名称

- 可通过注解value属性手动指定Bean名称

#### 3. 适用场景

解决同接口多实现、类型冲突场景，精准匹配指定名称Bean。

### 3.3 @Qualifier 精准限定Bean、解决同接口多实现冲突

当一个接口存在多个实现类，ByType匹配会出现冲突，此时通过 **@Qualifier**精准指定Bean名称，解决冲突。

#### 实操示例

```java
// 支付接口
public interface PayService {}

// 两个实现类
@Service
public class AliPayService implements PayService {}
@Service
public class WechatPayService implements PayService {}

// 精准注入指定Bean
@Service
public class OrderBizService {
    @Autowired
    @Qualifier("aliPayService")
    private PayService payService;
}

```

**核心作用**：在ByType基础上，二次通过名称精准过滤，彻底解决多实现冲突。

### 3.4 @Primary 优先候选Bean、默认首选注入

**@Primary** 用于设置**默认优先Bean**，当多实现冲突时，优先注入被该注解标记的Bean，无需每次都用@Qualifier。

```java
@Service
@Primary // 优先注入当前Bean
public class AliPayService implements PayService {}

```

**适用场景**：项目存在默认实现，其他实现为拓展实现。

### 3.5 泛型依赖注入原理与实战应用

Spring支持**带泛型的依赖注入**，泛型会作为类型匹配的一部分，精准区分不同泛型实现的Bean，避免冲突。

#### 实操示例

```java
// 泛型父类
public abstract class BaseHandler<T> {}

// 不同泛型实现
@Service
public class UserHandler extends BaseHandler<User> {}
@Service
public class OrderHandler extends BaseHandler<Order> {}

// 精准按泛型类型注入
@Service
public class HandlerService {
    @Autowired
    private BaseHandler<User> userBaseHandler;
}

```

**核心价值**：泛型成为类型区分维度，完美适配通用模板、处理器架构。

### 3.6 集合/数组/Map批量注入同类型Bean

#### 3.6.1 List/Set/数组批量注入

当需要一次性获取某个接口**所有实现类Bean**时，直接使用集合注入，Spring自动将所有同类型Bean封装入集合。

```java
@Service
public class PayContext {
    // 自动注入所有PayService实现类
    @Autowired
    private List<PayService> payServiceList;
}

```

#### 3.6.2 Map注入Key-Value映射规则

Map注入中，**key为Bean名称，value为对应Bean实例**，可通过key精准匹配对应实现。

```java
@Autowired
private Map<String, PayService> payServiceMap;

```

#### 3.6.3 批量注入落地：策略模式消除if-else

批量注入是生产中**消除大量if-else**的核心方案，经典落地策略模式：

1. 定义统一业务接口

2. 编写多个业务实现类并交给Spring管理

3. 通过Map批量注入所有实现

4. 根据业务标识key动态获取对应Bean，彻底消除分支判断

## 4. 注解式依赖注入全解

### 4.1 @Autowired 底层原理、生效范围、依赖 required 属性

#### 1. 底层原理

由Spring提供的专属注解，依托 **AutowiredAnnotationBeanPostProcessor** 后置处理器完成解析、匹配、反射注入。

#### 2. required 属性

- **required=true（默认）**：必须找到Bean，找不到直接报错

- **required=false**：找不到Bean不报错，属性为null

```java
@Autowired(required = false)
private TestService testService;

```

#### 3. 生效范围

可标注在：字段、构造器、setter方法、参数上。

### 4.2 @Resource JSR标准注解、默认匹配规则与切换

**@Resource** 是Java原生JSR250标准注解，不属于Spring，通用性更强。

#### 1. 默认匹配规则

- **默认先ByName**：优先根据字段名称匹配Bean

- 名称匹配失败，自动降级为**ByType**匹配

#### 2. 指定名称注入

```java
@Resource(name = "aliPayService")
private PayService payService;

```

### 4.3 @Inject 注解使用与和@Autowired差异

@Inject是JSR330标准注解，功能和@Autowired几乎一致，**默认ByType注入**，无required属性，不常用。

**核心差异**：@Inject无是否必填配置，@Autowired支持required属性，灵活性更高。

### 4.4 @Value 普通值、配置文件、SpEL表达式注入

@Value专门用于**非对象依赖注入**，支持三种注入方式：

```java
// 1. 直接常量注入
@Value("123")
private Integer num;

// 2. 配置文件取值
@Value("${server.port}")
private Integer port;

// 3. SpEL表达式注入
@Value("#{1+2}")
private Integer result;

```

### 4.5 注解注入常见报错：找不到Bean、多个Bean匹配冲突

#### 1. 找不到Bean报错

**原因**：依赖类未交给Spring管理、扫描路径缺失、注解未添加

**解决**：添加@Component/@Service注解、配置扫描包路径

#### 2. 多Bean冲突报错

**原因**：接口多实现，ByType匹配不唯一

**解决**：@Qualifier指定名称 / @Primary设置默认Bean

## 5. 循环依赖深度剖析（DI核心难点）

### 5.1 什么是循环依赖？三种常见场景复现

**循环依赖**：两个或多个Bean相互依赖对方，形成闭环依赖关系。

#### 三种场景

1. 双向依赖：A依赖B，B依赖A（最常见）

2. 环形依赖：A→B→C→A

3. 自依赖：A依赖自身

#### 场景复现

```java
@Service
public class AService {
    @Autowired
    private BService bService;
}

@Service
public class BService {
    @Autowired
    private AService aService;
}

```

### 5.2 Spring能解决/不能解决的循环依赖边界

**✅ 可以解决**：单例Bean + 字段注入 / Setter注入 的循环依赖

**❌ 无法解决**：

1. 构造器注入循环依赖

2. 多例Bean循环依赖

3. 代理Bean复杂循环依赖

### 5.3 三级缓存结构与每一级职责详解

Spring通过三级缓存解决循环依赖，三级缓存全部是ConcurrentHashMap：

#### 一级缓存 singletonObjects

存放**完全初始化完成、可直接使用**的单例Bean

#### 二级缓存 earlySingletonObjects

存放**已实例化、未完成初始化**的半成品Bean（普通无代理Bean）

#### 三级缓存 singletonFactories

存放**Bean工厂对象**，用于生成原始Bean或代理Bean，解决AOP循环依赖

### 5.4 三级缓存解决单例Setter/字段循环依赖完整流程

以A、B双向依赖为例：

1. 容器创建A，实例化生成空对象

2. 将A的工厂对象存入三级缓存

3. A开始注入依赖B，触发B创建

4. B注入依赖A，一二级缓存无A，从三级缓存获取A半成品

5. B完成所有初始化，存入一级缓存

6. A获取完整B，完成注入与初始化，存入一级缓存

7. 清除二三级缓存临时数据

### 5.5 为什么二级缓存不够、必须三级缓存？

**核心原因：AOP代理**

二级缓存只能存放原始Bean，无法动态生成代理Bean。三级缓存的ObjectFactory工厂，可以**动态返回原始Bean或代理Bean**，保证循环依赖中注入的Bean和最终容器Bean一致，解决代理场景循环依赖。

### 5.6 构造器注入为什么无法解决循环依赖？

构造器注入在**实例化阶段就需要依赖对象**，此时Bean还未创建完成，没有半成品、无法存入缓存，没有任何提前暴露引用的机会，直接卡死循环，无法解决。

### 5.7 多例Bean为什么不支持循环依赖？

多例Bean**不进入三级缓存**，每次创建都是全新对象，无缓存复用机制，无法通过半成品Bean完成依赖注入，必然报错。

### 5.8 循环依赖常见异常排查与生产解决方案

**常见异常**：BeanCurrentlyInCreationException

**排查思路**：

1. 检查是否存在构造器双向依赖

2. 检查多例Bean循环依赖

3. 梳理代码架构，解除不必要的双向依赖

**生产最优解**：重构代码，避免双向循环依赖，从架构层面根治问题。

## 6. 依赖注入与Bean生命周期、作用域关联

### 6.1 依赖注入在生命周期中的准确执行位置

**精准顺序**：构造器实例化 → **依赖注入** → 后置处理器前置增强 → 初始化方法执行

所有初始化方法中，依赖已经100%完成赋值，不会出现空指针。

### 6.2 singleton单例Bean注入特点

1. 容器启动一次性注入完成，全局复用

2. 依赖全局唯一，不会重复注入

3. 完整支持三级缓存、循环依赖解决

### 6.3 prototype多例Bean注入失效经典坑

**经典坑点**：单例Bean注入多例Bean，**多例彻底失效**。

原因：单例Bean仅启动时初始化一次，多例Bean只会被注入一次，后续永远复用旧对象，无法实现多例效果。

**解决方案**：通过ApplicationContext动态获取多例Bean。

### 6.4 request/session作用域Bean注入单例解决方案

Web作用域Bean生命周期短，注入单例会失效，Spring通过**代理占位Bean**解决，底层动态代理切换真实对象。

### 6.5 懒加载@Lazy对依赖注入的影响与原理

被@Lazy标记的依赖，不会在容器启动时注入，而是在**第一次调用时**才完成创建和注入，可解决部分循环依赖、优化启动速度。

## 7. 高级依赖注入拓展
### 7.1 原型Bean依赖单例、单例依赖原型相互影响
#### 7.1.1 基础概念回顾
- **单例Bean（Singleton）**：Spring IoC 容器中**全局只创建1个**，全程复用。
- **原型Bean（Prototype）**：每次 `getBean()` / 依赖注入都会**新建一个全新对象**。

#### 7.1.2 场景一：原型Bean 依赖 单例Bean
#### 原理
每次创建新的原型Bean时，去容器找依赖的单例Bean，**永远拿到同一个全局单例**。
#### 执行特点
1. 原型Bean每次都是新对象
2. 依赖的单例Bean全局唯一、不会重复创建
3. **完全无任何问题，业务中最常用、完全正常**
#### 通俗总结
多例找单例：**新人用老员工，完全没问题**。

---

#### 7.1.3 场景二：单例Bean 依赖 原型Bean（重点坑点）
#### 底层原理
单例Bean**只创建一次**，创建时就一次性把依赖的原型Bean注入完成；
因为单例生命周期全局只初始化一次，**后续永远不会再重新注入**。

#### 致命现象
你明明给 Bean 标了 `@Scope("prototype")`，但是**注入到单例Bean里永久变成同一个对象**，原型失效。

#### 为什么失效？
1. 单例Bean：容器启动只实例化、只注入**一次**
2. 注入时赋值了一个原型对象
3. 后续一直复用这个引用，**不会主动刷新、不会重新创建**

#### 怎么解决？
1. 主动从 `BeanFactory` / `ApplicationContext` 每次手动获取
2. 用 `ObjectProvider` 延迟获取
3. 用 `ApplicationContextAware` 上下文主动拉取
4. 拆分业务设计，避免单例直接依赖可变原型

#### 面试标准结论
- **原型依赖单例**：完全正常，互不影响
- **单例依赖原型**：原型失效，只会固定第一次注入的那个对象

---

### 7.2 FactoryBean参与依赖注入的特殊规则
#### 7.2.1 什么是 FactoryBean
`FactoryBean` 是 Spring 提供的**工厂Bean接口**，用来**复杂对象、第三方对象、无法直接new的对象**自定义创建。
- 普通Bean：直接由Spring反射new出来
- FactoryBean：由自己自定义逻辑生成对象

#### 7.2.2 核心规则（面试必考）
1. 当你 `@Autowired` 注入 **FactoryBean 名称**时：
   默认注入的是 **`getObject()` 方法返回的产品Bean**，**不是工厂本身**。
2. 如果**非要注入 FactoryBean 工厂本身**：
   需要在 BeanName 前面加 **`&` 前缀**。

#### 7.2.3 示例演示
```java
public class MyFactoryBean implements FactoryBean<User> {
    @Override
    public User getObject() throws Exception {
        // 自定义复杂创建逻辑
        return new User();
    }
    @Override
    public Class<?> getObjectType() {
        return User.class;
    }
}
```
- 注入 `myFactoryBean` → 拿到 **User 对象**
- 注入 `&myFactoryBean` → 拿到 **MyFactoryBean 工厂本身**

#### 7.2.4 底层意义
Spring 用 FactoryBean 屏蔽复杂对象创建，比如 MyBatis 的 `SqlSessionFactoryBean`、Redis、中间件集成全靠它。

---

### 7.3 自定义Bean工厂与手动依赖注入
#### 7.3.1 适用场景
有些对象无法靠自动注入：
- 动态参数创建
- 运行时动态生成
- 规则动态变化
需要**手动从容器拿Bean、手动控制创建时机**。

#### 7.3.2 核心API
1. `BeanFactory`：底层工厂，基础获取Bean
2. `ApplicationContext`：继承BeanFactory，多了扩展功能
3. `BeanFactoryAware`：注入原生工厂

#### 7.3.3 手动获取Bean示例
```java
// 1. 通过名称获取
User user = (User) beanFactory.getBean("user");

// 2. 通过类型获取
User user = beanFactory.getBean(User.class);

// 3. 带构造参数动态创建原型Bean
User user = beanFactory.getBean("user", 1001L, "张三");
```

#### 7.3.4 手动依赖注入意义
- 脱离自动DI限制，**运行时动态控制Bean**
- 适配动态配置、动态业务、动态策略模式
- 框架底层大量使用：SpringBoot、MyBatis、中间件

---

### 7.4 条件注入@Conditional配合DI使用
#### 7.4.1 @Conditional 作用
**满足指定条件，才将Bean注册到容器；不满足直接跳过不创建**。
只有注册成功，后续 `@Autowired` 依赖注入**才能成功**。

#### 7.4.2 内置常用条件注解
- `@ConditionalOnClass`：类存在才注入
- `@ConditionalOnMissingBean`：容器没有该Bean才注册
- `@ConditionalOnProperty`：配置文件满足条件才生效
- `@ConditionalOnWebApplication`：Web环境才生效

#### 7.4.3 核心使用场景
1. **环境隔离**：开发/测试/生产 不同Bean实现
2. **自动装配**：SpringBoot Starter 核心原理
3. **兜底默认实现**：用户没自定义Bean，框架给默认Bean
4. **按需加载**：依赖引入才启用对应功能Bean

#### 7.4.4 配合DI底层逻辑
1. 容器启动扫描配置类
2. 解析 `@Conditional` 判断条件
3. 条件通过 → 注册Bean到IoC
4. 后续 `populateBean` 依赖注入才能找到并赋值
5. 条件不通过 → 不注册 → 注入直接报错

---

### 7.5 ImportSelector动态注册Bean后的自动装配
#### 7.5.1 什么是 ImportSelector
`ImportSelector` 是 Spring **动态批量注册Bean** 的核心接口，可以**代码逻辑动态返回全类名**，由Spring自动帮你注册为Bean。

#### 7.5.2 核心特点
- 不是写死 `@Bean`，而是**运行时动态选择要注册的类**
- 通过 `@Import(XXXSelector.class)` 导入
- 动态注册进去的Bean **和普通Bean完全无区别**

#### 7.5.3 依赖注入规则
1. ImportSelector 动态注册的类，会被Spring解析为BeanDefinition
2. 进入IoC容器生命周期：实例化 → 注入 → 初始化
3. 支持 `@Autowired`、构造器注入、Setter注入**全部正常生效**
4. 可以被AOP切面拦截、可以被事务管理

#### 7.5.4 实际应用价值
**SpringBoot Starter 底层核心就是靠 ImportSelector**：

- 引入starter依赖 → 自动通过Selector导入配置类
- 自动注册批量Bean
- 开发者无需手动 `@Bean`，开箱即用

#### 7.5.5 面试总结
ImportSelector动态注册的Bean，**完全融入Spring容器生命周期，支持所有依赖注入、AOP、事务、三级缓存，和手动@Bean毫无区别**。

---

## 8. 依赖注入底层源码核心流程
### 8.1 populateBean 属性填充入口
#### 8.1.1 核心定位
`populateBean()` 是 **AbstractAutowireCapableBeanFactory** 中的方法，是 **Spring 依赖注入、属性填充的唯一底层入口**。

所有注入方式全部走这个方法：
- `@Autowired`
- `@Resource`
- 构造器注入
- Setter注入
- 自动装配byName/byType

#### 8.1.2 执行时机
Bean生命周期顺序：
1. 实例化（构造方法new）
2. **进入 populateBean() 做属性填充、依赖注入**
3. 执行初始化方法（@PostConstruct、InitializingBean）
4. 生成代理对象（AOP）

#### 8.1.3 核心职责
1. 解析当前Bean所有需要注入的属性
2. 按 byName / byType / 注解规则 寻找依赖Bean
3. 处理循环依赖、三级缓存
4. 反射给字段/方法完成赋值

---

### 8.2 BeanDefinition解析依赖元数据
#### 8.2.1 BeanDefinition 是什么
BeanDefinition 是 **Bean的元数据描述对象**，Spring不会直接拿着Class去解析，而是先把Class解析成BeanDefinition。

里面存着：
- 类全限定名
- 构造参数信息
- 字段依赖信息
- Setter方法依赖
- 作用域、懒加载、依赖关系

#### 8.2.2 解析时机
容器启动**刷新阶段**，扫描所有 `@Component`、`@Service`、配置类，提前预解析成 BeanDefinition，**提前把所有依赖关系记录好**，为后续DI做准备。

#### 8.2.3 作用
后续 `populateBean` 不需要再扫描Class注解，直接从BeanDefinition读取**已经解析好的依赖元数据**，提升容器启动效率。

---

### 8.3 依赖查找、缓存匹配、递归创建依赖Bean
#### 8.3.1 第一步：先从三级缓存依次查找
Spring 先按顺序找依赖Bean：
1. **一级缓存 singletonObjects**：完整成品Bean
2. **二级缓存 earlySingletonObjects**：原始半成品Bean
3. **三级缓存 singletonFactories**：Bean工厂对象，可生成代理Bean

#### 8.3.2 第二步：缓存找不到 → 递归创建
如果三级缓存都没找到依赖Bean：
1. 触发依赖Bean的创建流程
2. 走同样的：实例化 → 放入三级缓存 → 填充属性 → 初始化
3. **递归解决多层依赖、循环依赖**

#### 8.3.3 第三步：反射完成属性赋值
找到依赖Bean后：
- 字段注入：反射 `Field.set()` 赋值
- Setter注入：反射调用方法传参赋值

#### 8.3.4 和循环依赖的关系
整个**三级缓存、递归创建**，就是在这一步支撑起来的，
DI底层能解决循环依赖，全靠：**先缓存、再赋值、递归创建**。

---

### 8.4 AutowiredAnnotationBeanPostProcessor 解析流程
#### 8.4.1 核心定位
`AutowiredAnnotationBeanPostProcessor` 是 **处理 @Autowired、@Value 注解的专属后置处理器**。

没有它，`@Autowired` 完全失效，Spring 不会自动注入。

#### 8.4.2 核心职责
1. 在Bean实例化后，扫描当前类所有字段、方法
2. 识别标有 `@Autowired` / `@Value` 的成员
3. 解析注入规则：byType、byName、是否必须
4. 交给 `populateBean` 完成依赖查找与赋值
5. 处理 required = true/false 缺失Bean是否报错

#### 8.4.3 执行流程
1. Bean 实例化完成
2. 进入 AutowiredAnnotationBeanPostProcessor
3. 扫描类上所有注解依赖
4. 封装成注入元数据
5. 交由底层工厂完成查找、递归创建、反射赋值

#### 8.4.4 面试核心总结
@Autowired 不是 Spring 原生自带语法，
全靠 **AutowiredAnnotationBeanPostProcessor** 后置处理器**扫描、解析、驱动完成注入**。

---

## 9. 本章高频面试题 & 易错点总结

### 9.1 三种依赖注入方式面试问答、官方推荐顺序

**面试满分答案**：Spring有构造器、Setter、字段三种注入方式。官方优先推荐构造器注入用于强制依赖，保证依赖非空、线程安全、解耦性好；Setter注入用于可选动态依赖；字段注入仅用于快速开发，生产不推荐，存在依赖隐藏、测试困难、循环依赖隐患。

### 9.2 @Autowired 和 @Resource 区别高频题

1. 来源不同：@Autowired是Spring注解，@Resource是Java原生JSR标准

2. 匹配规则不同：@Autowired默认ByType，@Resource默认ByName降级ByType

3. 属性不同：@Autowired支持required必填配置，@Resource无

### 9.3 循环依赖面试满分答题模板

Spring仅支持单例Bean的字段、Setter循环依赖，通过三级缓存解决。一级缓存存完整Bean，二级缓存存普通半成品Bean，三级缓存存Bean工厂支持代理Bean生成。通过提前暴露半成品Bean引用，完成双向依赖赋值。构造器注入、多例Bean无法解决循环依赖。

### 9.4 多例注入失效、同接口多Bean注入冲突易错点

1. 单例依赖多例必然失效，仅初始化注入一次

2. 同接口多实现默认ByType冲突，需用@Qualifier或@Primary解决

### 9.5 依赖注入相关报错快速排查思路

1. 找不到Bean：检查注解、扫描路径、Bean注册

2. 多Bean冲突：使用限定注解指定唯一Bean

3. 循环依赖报错：检查构造器依赖、多例依赖、双向依赖

## 本章总结

本章系统性、全方位讲解了Spring DI依赖注入的**所有核心知识点、底层原理、生产坑点、面试考点**。DI作为IoC的落地核心，是Spring整个生态的基石，所有高级特性都依赖依赖注入实现。

核心知识点复盘：

1. IoC是思想，DI是手段，二者密不可分，是Spring解耦的核心。

2. 三种注入方式各有适用场景，生产严格遵循「构造器优先、Setter为辅、字段慎用」的规范。

3. 自动装配默认ByType，配合@Qualifier、@Primary解决多实现冲突，集合批量注入可优雅消除if-else。

4. 三级缓存是解决单例循环依赖的核心，明确区分可解决/不可解决的边界条件。

5. 作用域直接影响注入效果，单例依赖多例失效是生产高频坑点。

6. 底层通过populateBean完成属性填充，后置处理器解析注解实现自动注入。
