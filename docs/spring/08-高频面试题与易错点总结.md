# 1.Spring 基础核心面试题 & 易错点

## 1.1 Spring IoC 容器核心面试题

### 1.1.1 IoC 核心概念与底层原理

**概念定义**：IoC 全称 Inversion of Control，即**控制反转**，是Spring框架的核心设计思想。简单来说，就是**将对象的创建、管理、依赖关系维护的控制权，从开发者代码中转移到Spring容器**。传统开发中，开发者主动 new 对象、维护对象依赖，是「主动控制」；Spring IoC模式下，开发者只需要声明依赖，由容器全权负责对象创建和依赖注入，控制权发生反转，因此叫控制反转。

**核心原理**：IoC的底层核心逻辑可以拆解为三个核心步骤，也是面试必背流程。首先是**资源加载与解析**，Spring读取XML配置、注解配置、JavaConfig配置，扫描项目中的Bean标识，收集所有需要托管的类信息；其次是**Bean定义注册**，将解析后的类信息封装为BeanDefinition对象，存储在注册中心，此时仅保存元数据，未创建对象；最后是**容器实例化与依赖注入**，容器启动后根据BeanDefinition，通过反射创建Bean实例，并自动完成依赖对象的注入、初始化、销毁管理。

**场景与价值**：IoC彻底解决了传统Java开发的耦合度过高问题。传统开发中，类与类之间硬编码依赖，修改一个底层类，所有上层调用代码都需要改动，代码扩展性极差。而IoC容器统一管理所有Bean，实现了**组件解耦、统一管理、可复用、易扩展**，是Spring AOP、事务、Spring Boot自动配置等所有高级特性的基础。

**生活化类比**：可以把IoC容器比作「工厂总管」，开发者只需要告诉工厂需要哪些零件（Bean）、零件之间的装配关系（依赖），不需要自己手动生产、组装零件。所有对象的生产、维护、销毁都由总管负责，开发者只需要直接使用成品即可。

### 1.1.2 IoC 和 DI 的区别与联系

这是面试**超高频基础题**，90%的初级面试都会问到，也是新手最容易混淆的知识点。

**核心定义**：IoC是一种**设计思想、架构模式**（控制反转），描述的是控制权转移的整体理念；DI 全称 Dependency Injection，即**依赖注入**，是IoC思想的**具体实现方式**。简单理解：**IoC是目的，DI是手段**。

**底层逻辑**：单纯的控制反转只是抽象概念，无法落地。Spring通过依赖注入的方式，完成控制权的反转——容器将对象需要的依赖资源，自动注入到目标Bean中，无需开发者主动获取依赖。没有DI，IoC思想就无法落地执行。

### 1.1.3 高频面试真题与解析

**📌面试题1：为什么要使用IoC？不用IoC会有什么问题？**

**参考答案**：使用IoC的核心价值是**解耦、统一管理、便于扩展和维护**。不使用IoC时，开发者手动new对象，存在三大核心问题：第一，代码耦合度极高，依赖硬编码，修改依赖类需要改动多处代码；第二，对象生命周期混乱，开发者自行创建、销毁对象，容易出现内存泄漏、对象重复创建问题；第三，无法实现统一的扩展管控，难以实现AOP增强、事务管理、统一配置等高级特性。

**📌面试题2：Spring IoC容器的顶层核心接口有哪些？各自作用？**

**参考答案**：核心顶层接口分为两大核心：

1. **BeanFactory**：Spring IoC容器的**最顶层根接口**，定义了Bean容器的最基础能力，包括获取Bean、判断Bean是否存在、获取Bean类型等，采用**延迟加载**策略，启动速度快，适合轻量级场景。

2. **ApplicationContext**：BeanFactory的**子接口**，是实际开发中使用的容器接口，在BeanFactory基础上扩展了大量高级能力：支持国际化、事件发布、资源加载、Bean自动扫描、容器刷新等，Spring项目默认使用的容器实现类都是ApplicationContext的实现类。

### 1.1.4 避坑指南与最佳实践

⚠️ **避坑1：混淆IoC和DI概念**

错误认知：认为IoC就是依赖注入，二者是同一个东西。

正确认知：IoC是设计思想（控制权反转），DI是实现方式（依赖注入），二者是**思想与实现**的关系，不可等同。面试回答时必须明确区分，否则会直接扣分。

⚠️ **避坑2：认为BeanFactory是日常使用的容器**

错误认知：开发中使用的Spring容器是BeanFactory。

正确认知：BeanFactory是顶层规范接口，功能极简；实际开发中使用的 `ClassPathXmlApplicationContext`、`AnnotationConfigApplicationContext` 都是ApplicationContext的实现类，具备完整的容器能力。

## 1.2 IoC 初始化流程高频问答

IoC容器初始化流程是Spring面试**中级必考题**，面试官重点考察开发者是否了解Spring底层启动逻辑，能否排查容器启动异常，本节完整拆解核心流程、高频问题、易错点。

### 1.2.1 IoC 容器完整初始化核心流程

Spring IoC容器初始化整体分为**五大核心阶段**，流程固定，面试需完整口述：

**阶段一：容器资源加载与环境准备**

创建ApplicationContext容器对象，初始化容器环境、配置参数、系统变量、资源加载器，为后续解析配置做准备。如果是注解开发，会初始化注解扫描器；如果是XML开发，会初始化XML资源解析器。

**阶段二：配置解析与 BeanDefinition 注册**

容器扫描指定路径下的配置类、XML文件、注解标识的类，解析每一个需要托管的Bean，将类的权限定名、属性、依赖、作用域、初始化方法、销毁方法等信息，封装为**BeanDefinition**对象，统一注册到BeanDefinition注册表中。**此阶段仅保存元数据，不创建任何Bean实例**。

**阶段三：BeanFactory 后置处理**

容器执行所有 `BeanFactoryPostProcessor`（Bean工厂后置处理器），对已注册的BeanDefinition进行修改、增强、替换，比如修改Bean属性、动态注册Bean、扫描自定义注解等，是Spring扩展Bean定义的核心入口。

**阶段四：单例 Bean 实例化与初始化**

容器刷新完成后，触发**非懒加载的单例Bean**的实例化流程，通过反射创建Bean对象，完成依赖注入、属性填充、前置处理、初始化方法执行，最终生成完整的可用Bean实例，存入单例池。

**阶段五：容器刷新完成**

执行容器刷新完成的后置方法，发布容器刷新完成事件，启动完成，对外提供Bean获取、调用能力。

### 1.2.2 高频面试真题与深度解析

**📌面试题1：Spring IoC容器启动时，什么时候创建Bean？**

**参考答案**：分为两种情况：1. **单例非懒加载Bean**：在容器**刷新阶段（refresh方法）**统一创建、初始化，容器启动完成后就已存在于单例池；2.**多例Bean、懒加载单例Bean**：容器启动时不创建，**第一次被获取、调用时**才会实例化。

**📌面试题2：BeanDefinition和Bean实例的区别？**

**参考答案**：BeanDefinition是**Bean的元数据模板**，仅存储类的配置信息，占用内存极小，容器启动阶段优先注册；Bean实例是**根据模板创建的真实对象**，占用堆内存，是程序实际调用的对象。简单类比：BeanDefinition是「图纸」，Bean实例是「根据图纸造出来的实物」。

**📌面试题3：BeanFactoryPostProcessor 和 BeanPostProcessor 区别？**

**参考答案**：二者核心区别是**执行时机、作用对象不同**：

1. BeanFactoryPostProcessor：作用于**BeanDefinition**，在Bean实例创建之前执行，用于修改Bean的定义信息；

2. BeanPostProcessor：作用于**Bean实例**，在Bean创建完成、属性填充之后、初始化方法前后执行，用于增强Bean实例（如AOP代理）。

### 1.2.3 易错点与生产排错指南

⚠️ **易错点1：认为容器启动会创建所有Bean**

错误认知：所有Bean都在容器启动时创建完成。

正确逻辑：只有**单例、非懒加载**Bean启动时创建，多例、懒加载Bean、作用域为request/session的Bean，均在使用时创建。这也是Spring启动速度优化的核心原理。

⚠️ **易错点2：混淆两个后置处理器的执行顺序**

执行顺序固定：**BeanFactory后置处理器 → Bean实例创建 → Bean后置处理器**，顺序不可逆，面试高频考点，也是排查Bean初始化异常的核心依据。

**💡最佳实践**：生产中自定义扩展Spring容器能力时，修改Bean定义用BeanFactoryPostProcessor，增强Bean实例功能用BeanPostProcessor，不要混用，避免出现Bean创建异常。

## 1.3 Bean 生命周期面试必问

Bean生命周期是Spring**面试TOP3必考知识点**，几乎所有Java后端面试都会问到完整流程，同时也是生产中Bean初始化报错、属性为空、代理失效的核心排查依据，必须完整掌握。

### 1.3.1 Bean 完整生命周期全流程拆解

Spring Bean的生命周期从**实例化**到**销毁**共7个核心步骤，流程顺序固定，无特殊情况不会变动，下面结合原理和执行时机详细说明：

**1. 实例化（创建对象）**：Spring根据BeanDefinition信息，通过**反射机制**调用类的无参构造方法，创建Bean的空对象。此时对象已存在，但所有属性都是默认空值，未赋值。

**2. 属性填充（依赖注入）**：容器解析当前Bean的所有依赖，自动完成属性赋值、依赖注入（构造器注入、setter注入、字段注入），此时Bean的业务属性已赋值完成。

**3. Bean后置处理器前置处理（postProcessBeforeInitialization）**：执行所有BeanPostProcessor的前置增强方法，对刚完成属性填充的Bean进行预处理，Spring AOP的代理预处理、属性校验都在此阶段执行。

**4. 初始化方法执行**：分为三层执行顺序，优先级从高到低：
① 执行 `@PostConstruct` 注解标注的方法
② 执行 InitializingBean 接口的 afterPropertiesSet 方法
③ 执行 XML/JavaConfig 中指定的 init-method 初始化方法

**5. Bean后置处理器后置处理（postProcessAfterInitialization）**：执行后置处理器的后置增强方法，完成Bean的最终增强，**AOP代理对象的最终生成在此阶段完成**。

**6. 就绪可用阶段**：Bean初始化完成，存入Spring单例池，正式对外提供服务，响应业务调用。

**7. 销毁阶段**：容器关闭时，执行Bean的销毁方法，执行顺序：
① 执行 `@PreDestroy` 注解方法
② 执行 DisposableBean 接口的 destroy 方法
③ 执行配置的 destroy-method 销毁方法

### 1.3.2 生命周期核心面试真题

**📌面试题1：简述Spring Bean的完整生命周期，说出关键执行顺序**

**参考答案（精简面试版）**：Bean生命周期分为七步：实例化（反射创建对象）→ 属性填充（依赖注入）→ 后置处理器前置处理 → 初始化方法执行（注解>接口>配置方法）→ 后置处理器后置处理 → 业务使用 → 容器关闭销毁。核心关键点：**先赋值、后初始化、最后增强**。

**📌面试题2：@PostConstruct 和 init-method 的区别和执行顺序？**

**参考答案**：执行顺序：**@PostConstruct 优先于 init-method**。区别：@PostConstruct是JSR250标准注解，通用性更强；init-method是Spring原生配置方式，灵活性更高，可配置无侵入的初始化方法。

**📌面试题3：Bean是在哪个阶段完成AOP代理的？**

**参考答案**：在**BeanPostProcessor后置处理阶段**完成代理。初始化方法执行时，Bean还是原生对象，初始化完成后，后置处理器生成代理对象，替换原生Bean存入容器。

### 1.3.3 生产易错点与避坑指南

⚠️ **避坑1：初始化方法中调用依赖属性出现空指针**

错误场景：开发者在构造方法中直接使用依赖属性，导致空指针。

原因：构造方法执行在**实例化阶段**，此时还未进行属性填充，所有依赖都为null。

解决方案：需要初始化执行的业务逻辑，统一放到 **@PostConstruct** 初始化方法中执行，此时属性已全部注入完成。

⚠️ **避坑2：混淆AOP代理时机导致业务失效**

错误认知：初始化方法中的逻辑会被AOP代理增强。

正确逻辑：初始化方法执行时，Bean还是原生对象，尚未生成代理，因此**初始化方法内部的调用不会触发AOP切面**，这是生产中AOP失效的高频原因。

## 1.4 依赖注入 DI 高频考点

依赖注入是Spring实现IoC的核心手段，也是日常开发使用最多的特性，面试重点考察**注入方式区别、优先级、优缺点、生产使用规范**，同时包含大量开发易错点。

### 1.4.1 三种依赖注入方式详解

Spring支持三种核心依赖注入方式：**构造器注入、Setter注入、字段注入**，三者原理、优先级、适用场景完全不同。

**1. 字段注入（@Autowired）**

通过 `@Autowired`、`@Resource` 注解直接标注在类属性上，Spring自动根据类型或名称注入依赖。优点：代码简洁、开发高效；缺点：耦合度高、违背单一职责、容易导致循环依赖、不利于单元测试。

**2. Setter注入**

通过Java Bean的setXxx方法完成依赖注入，可配合@Autowired使用。核心特点：**适合可选依赖、动态修改依赖**，对象创建后可动态替换依赖资源。

**3. 构造器注入**

通过类的构造方法传入依赖参数，Spring 4.3+版本支持单构造器自动注入，无需加注解。核心特点：**强制依赖、不可变、线程安全、避免循环依赖**，是Spring官方推荐的首选注入方式。

### 1.4.2 注入优先级与核心区别

**Spring官方注入优先级**：**构造器注入 > Setter注入 > 字段注入**

**核心区别对比（生产重点）**：

1. **必填依赖**：必须使用构造器注入，保证对象初始化完成后，所有必填依赖已就绪，避免空指针；

2. **可选依赖**：使用Setter注入，允许依赖为空，支持动态修改；

3.**字段注入**：仅用于简单业务场景，禁止在底层工具类、核心服务类中使用。

### 1.4.3 高频面试真题解析

**📌面试题1：Spring推荐使用哪种依赖注入方式？为什么？**

**参考答案**：Spring官方推荐**构造器注入**。核心原因：① 保证对象不可变，初始化完成后依赖不会被修改，线程安全；② 强制依赖注入，避免遗漏依赖导致空指针；③ 有效减少循环依赖问题；④ 无注解侵入，更符合Java面向对象规范，便于单元测试。

**📌面试题2：@Autowired 和 @Resource 的区别？**

**参考答案**：1. 来源不同：@Autowired是Spring注解，@Resource是JSR250标准注解；2. 注入规则不同：@Autowired**默认按类型注入**，@Resource**默认按名称注入**；3. 优先级不同：@Resource找不到名称Bean时会降级按类型匹配，@Autowired可配合@Qualifier指定名称；4. 必填性：@Autowired默认必填，可设置required=false，@Resource默认非必填。

### 1.4.4 生产易错点与最佳实践

⚠️ **避坑1：滥用字段注入导致循环依赖**

错误场景：两个Service互相字段注入，启动报错循环依赖。

解决方案：核心服务统一使用构造器注入，Spring会提前检测循环依赖，同时架构上避免双向强依赖。

⚠️ **避坑2：@Autowired注入静态变量失效**

核心原理：Spring依赖注入仅针对**实例属性**，静态属性属于类，不属于Bean实例，无法被注入，这是新手高频报错点。

**💡最佳实践**：1. 所有核心业务服务、底层组件，统一使用构造器注入；2. 临时可选依赖使用Setter注入；3. 简单Controller层可酌情使用字段注入；4. 禁止注入静态变量。

## 1.5 循环依赖面试深度问答

循环依赖是Spring面试**深度必考难点**，面试官重点考察底层原理、支持场景、不支持场景、解决方案，同时也是生产启动报错的高频问题，本节做深度拆解。

### 1.5.1 循环依赖概念与场景分类

**概念定义**：循环依赖指两个或多个Spring Bean之间互相依赖，形成闭环依赖关系。例如：A类依赖B类，B类又依赖A类，容器初始化时无法确定先创建哪个Bean，形成循环阻塞。

**三种循环依赖场景**：

1. **字段注入循环依赖**：A、B均使用字段注入，Spring**支持解决**；

2. **Setter注入循环依赖**：A、B均使用Setter注入，Spring**支持解决**；

3. **构造器注入循环依赖**：A、B均使用构造器注入，Spring**无法解决，直接报错**。

### 1.5.2 Spring 能解决/不能解决循环依赖的底层原理

**核心前提**：Spring解决循环依赖的核心是**三级缓存+提前暴露半成品Bean**，依赖于「Bean实例化和属性填充分离」的机制。

**为什么字段/Setter注入可以解决？**

字段和Setter注入的流程：**先实例化（创建空对象）→ 提前暴露半成品Bean → 再填充属性**。当A创建空对象后，提前存入三级缓存，此时初始化B，B依赖A时，从缓存中获取半成品A，完成注入，后续A再完成自身属性填充，成功解决循环依赖。

**为什么构造器注入无法解决？**

构造器注入的依赖必须在**实例化阶段**全部传入，A创建时必须先创建B，B创建时必须先创建A，无半成品对象可以提前暴露，形成死循环，Spring无法处理，直接抛出循环依赖异常。

### 1.5.3 高频面试真题与解析

**📌面试题1：Spring默认能解决哪种循环依赖？为什么构造器循环依赖无法解决？**

**参考答案**：Spring仅支持**单例Bean的字段、Setter注入循环依赖**，不支持构造器循环依赖、多例Bean循环依赖。构造器注入需要在对象创建前完成依赖注入，无提前暴露的半成品Bean，无法打破循环闭环，因此无法解决。

**📌面试题2：多例Bean为什么无法解决循环依赖？**

**参考答案**：三级缓存仅缓存**单例Bean**，多例Bean每次获取都会创建新对象，不会存入缓存，无法提前暴露半成品对象，因此多例Bean循环依赖直接报错。

### 1.5.4 生产循环依赖解决方案与避坑

**💡生产解决方案**：

1. **代码层面优化**：重构代码架构，消除双向强依赖，将公共逻辑抽离为工具类、公共Service，从根源解决循环依赖；

2. **注入方式调整**：临时解决方案，将其中一个Bean改为Setter注入，绕过构造器循环依赖；

3. **使用@Lazy注解**：延迟加载依赖Bean，打破初始化闭环。

⚠️ **避坑指南**：不要依赖Spring的三级缓存自动解决循环依赖，**循环依赖本质是代码架构问题**，自动解决只是兜底方案，生产中必须重构消除，否则会导致Bean初始化不稳定、代理失效等隐性问题。

## 1.6 基础模块常见易错点汇总

本节汇总Spring IoC、DI、生命周期、循环依赖基础模块中，**生产最高频、面试最易答错**的易错点，统一梳理纠错，规避认知偏差。

### 1.6.1 IoC容器易错点

1. **错误认知**：ApplicationContext和BeanFactory是两个独立容器
**正确认知**：ApplicationContext是BeanFactory的子接口，是对BeanFactory的功能扩展，底层核心容器能力均来自BeanFactory。

2. **错误认知**：IoC容器启动会初始化所有Bean
**正确认知**：仅单例非懒加载Bean启动初始化，多例、懒加载Bean使用时才初始化。

3. **错误认知**：BeanDefinition包含Bean的实例对象
**正确认知**：BeanDefinition仅存储配置元数据，无实例对象，实例由容器后续反射创建。

### 1.6.2 依赖注入易错点

1. **错误认知**：@Autowired和@Resource可以随意替换
**正确认知**：默认注入规则不同，多Bean同类型时，@Autowired会报找不到Bean异常，@Resource可通过名称精准匹配。

2. **错误认知**：静态属性可以通过Spring注解注入
**正确认知**：Spring仅支持实例属性注入，静态属性注入永久失效，需通过静态工具类手动赋值。

3. **错误认知**：字段注入比构造器注入更优
**正确认知**：字段注入仅代码简洁，安全性、可维护性、规范性均不如构造器注入，不推荐核心场景使用。

### 1.6.3 生命周期与循环依赖易错点

1. **错误认知**：初始化方法内的逻辑会被AOP增强
**正确认知**：初始化方法执行时Bean尚未代理，内部调用无AOP效果。

2. **错误认知**：Spring可以解决所有循环依赖
**正确认知**：仅支持单例、字段/Setter注入循环依赖，构造器、多例循环依赖无解。

3. **错误认知**：循环依赖是Spring的bug
**正确认知**：循环依赖是代码设计缺陷，Spring的兜底方案只是临时兼容，不能根治问题。

# 2. Spring Bean 相关面试题 & 易错点

## 2.1 Bean 作用域面试详解（singleton/prototype 等）

### 一、Spring 核心五大作用域（面试必背）

Spring 官方定义5种Bean作用域，Spring Boot环境下默认仅启用**singleton、prototype**两种，其余三种仅在Web环境生效：

- **singleton（单例，默认）**：Spring IoC容器中仅存在一个Bean实例，Bean在容器初始化时（非懒加载）创建，全局共享。所有请求、线程共用同一个Bean实例。

- **prototype（多例）**：每次从容器中获取Bean（getBean、依赖注入）时，都会创建一个全新的实例，容器仅负责创建，不管理多例Bean的生命周期。

- **request**：Web环境专属，每个HTTP请求创建一个独立Bean实例，请求结束后Bean销毁。

- **session**：Web环境专属，每个用户会话创建一个独立Bean实例，会话过期后Bean销毁。

- **application**：Web环境专属，作用域等同于ServletContext，整个Web应用全局唯一，随应用启动创建、关闭销毁。

### 二、高频面试题

**1. singleton 是线程安全的吗？**

不是。Spring单例仅保证**容器内唯一实例**，不保证线程安全。若Bean包含成员变量（可变状态），多线程并发操作会出现线程安全问题。无状态Bean（仅方法、无成员变量）是线程安全的，有状态Bean非线程安全。

**2. singleton 和 prototype 核心区别？**

- 创建时机：单例容器启动创建（非懒加载），多例每次获取时创建；

- 实例数量：单例全局唯一，多例每次获取新实例；

- 生命周期：单例由Spring容器全程管理，多例容器仅创建、不销毁；

- 性能：单例仅初始化一次，性能高；多例频繁创建销毁，开销更大。

### 三、易错点汇总

- 误区1：认为singleton是Java单例（全局唯一）。正确：Spring单例是**IoC容器内单例**，多个容器可存在多个实例。

- 误区2：多例Bean会被Spring管理生命周期。正确：prototype Bean的初始化方法会执行，但销毁方法（@PreDestroy）不会执行。

- 误区3：懒加载单例Bean是多例。正确：@Lazy仅延迟创建时机（首次使用时创建），依然是单例。

## 2.2 懒加载、FactoryBean、BeanFactory 区别

### 一、@Lazy 懒加载详解

默认情况下，Spring单例Bean在**容器启动时预初始化**；添加`@Lazy`注解后，Bean会延迟至**首次被使用/注入时**初始化。

**核心作用**：缩短项目启动时间、解决循环依赖、延迟初始化耗时Bean。

**易错点**：@Lazy仅针对**单例Bean**生效，prototype本身就是延迟创建，添加@Lazy无意义。

### 二、BeanFactory 与 FactoryBean 核心区别（面试高频对比）

| 对比维度 | BeanFactory                                    | FactoryBean                                       |
| -------- | ---------------------------------------------- | ------------------------------------------------- |
| 角色定位 | **Bean工厂顶层接口**，Spring容器顶级接口       | **Bean生产工具接口**，用于自定义复杂Bean创建逻辑  |
| 核心功能 | 负责Bean的创建、获取、管理、依赖注入           | 用户自定义Bean的创建过程，封装复杂对象初始化      |
| 常见实现 | ApplicationContext、DefaultListableBeanFactory | MyBatis的SqlSessionFactoryBean、自定义FactoryBean |
| 获取对象 | getBean() 获取普通Bean实例                     | 默认获取**工厂生产的对象**，加&前缀获取工厂本身   |

### 三、高频面试题 & 易错点

**1. FactoryBean 的使用场景？**

用于创建**初始化逻辑复杂、非简单Java类**的Bean，例如：MyBatis SqlSessionFactory、Redis客户端、第三方工具类实例，屏蔽复杂的对象创建细节。

**2. 如何获取 FactoryBean 本身对象？**

在Bean名称前加&前缀，如：`getBean("&myFactoryBean")`，默认获取的是FactoryBean生产的目标对象。

**易错坑点**：混淆两个接口职责，BeanFactory是**容器**，FactoryBean是**Bean创建工具**，二者完全不是同一层级。

## 2.3 三级缓存原理与易错理解

### 一、三级缓存核心结构（Spring解决单例Bean循环依赖核心）

Spring通过三级缓存解决**setter/字段注入**的单例Bean循环依赖，构造器循环依赖无法解决。

- **一级缓存（singletonObjects）**：存放**完全初始化完成**的单例Bean，成品Bean缓存。

- **二级缓存（earlySingletonObjects）**：存放**已实例化、未初始化完成**的Bean（半成品Bean），无代理对象。

- **三级缓存（singletonFactories）**：存放**Bean工厂对象（Lambda表达式）**，用于提前生成Bean代理对象，解决循环依赖中的代理问题。

### 二、循环依赖解决流程

1. ABean创建：实例化后，将自身工厂对象放入三级缓存；

2. ABean需要注入BBean，开始创建BBean；

3. BBean需要注入ABean，从三级缓存获取ABean工厂，生成半成品ABean；

4. BBean完成初始化，ABean完成后续初始化，最终移入一级缓存。

### 三、高频易错点（面试重灾区）

- **误区1：三级缓存能解决所有循环依赖**。正确：仅支持**单例、setter/字段注入**，**构造器注入循环依赖直接报错**，因为实例化前就需要依赖对象，无机会存入三级缓存。

- **误区2：只有三级缓存就能解决循环依赖**。正确：二级缓存用于保证**单例唯一性**，避免多次创建半成品Bean，三级缓存核心解决**代理Bean循环依赖**。

- **误区3：多例Bean存在循环依赖问题**。正确：多例Bean每次创建新实例，无缓存机制，不会存在循环依赖缓存问题，但会无限创建实例导致栈溢出。

- **核心考点**：如果Bean需要AOP代理，三级缓存的工厂会提前生成代理对象，保证循环依赖注入的是代理Bean而非原生Bean。

## 2.4 Bean 配置方式对比与易错坑点

### 一、Spring Bean 四种主流配置方式

#### 1. XML 配置方式（传统方式）

通过`<bean>`标签定义Bean，手动配置id、class、属性注入，适配老项目，灵活性高但配置繁琐。

#### 2. 注解扫描方式（主流）

通过`@Component、@Service、@Controller、@Repository`声明Bean，配合`@ComponentScan`自动扫描注册，开发高效，SpringBoot默认方式。

#### 3. Java Config 配置方式

通过`@Configuration`配置类 + `@Bean`方法手动创建Bean，适合**第三方类、复杂初始化Bean**配置。

#### 4. 自动装配条件配置

配合`@Conditional`系列注解，根据环境、配置、Bean存在与否动态注册Bean。

### 二、四种配置方式对比

- **XML配置**：优点：解耦、无需改代码；缺点：配置冗余、维护成本高，新项目基本不用。

- **组件注解**：优点：简洁、自动化；缺点：无法配置第三方类，只能用于自定义类。

- **@Bean配置**：优点：可控性强、适配第三方Bean、支持复杂初始化；缺点：需要手动编写创建逻辑。

- **条件化配置**：优点：动态适配多环境；缺点：配置复杂，易出现Bean注册冲突。

### 三、高频易错坑点

- **坑点1：@Configuration 和 @Component 混淆**。`@Configuration`是**全配置类**，会被CGLIB代理，保证@Bean方法单例；`@Component`修饰的配置类，@Bean方法会多次创建实例，破坏单例。

- **坑点2：重复Bean注册冲突**。同一类通过多种方式注册Bean，会出现Bean覆盖、注入异常，默认**后注册的Bean覆盖先注册的**。

- **坑点3：@Bean 方法参数注入失效**。@Bean方法的参数会自动从容器获取Bean，无需额外@Autowired，手动new对象会导致无法被Spring管理。

- **坑点4：静态@Bean失效**。静态方法上的@Bean无法依赖注入，不会触发Spring完整生命周期。

## 2.5 泛型依赖注入、条件化 Bean 面试题

### 一、泛型依赖注入（Spring4+ 核心特性）

Spring支持**带泛型的父类/接口依赖注入**，可通过泛型区分不同子类Bean，实现精准注入，常用于通用DAO、通用Service封装。

**核心原理**：Spring会将**泛型类型**作为Bean的限定标识，同接口不同泛型的Bean视为不同Bean，可精准匹配注入。

### 二、高频面试题 & 易错点

**1. 泛型依赖注入的优势？**

无需手动创建多个接口，通过泛型实现接口复用，代码高度通用，避免大量重复接口定义，适配分层通用架构。

**易错点**：泛型擦除会导致运行时泛型丢失，但Spring通过**BeanDefinition存储泛型元数据**，可在容器启动阶段识别泛型，不影响注入。

### 三、条件化 Bean（@Conditional 系列注解）

#### 1. 核心注解

- **@ConditionalOnClass**：类路径存在指定类时注册Bean；

- **@ConditionalOnMissingClass**：类路径不存在指定类时注册Bean；

- **@ConditionalOnBean**：容器存在指定Bean时注册；

- **@ConditionalOnMissingBean**：容器无指定Bean时注册（SpringBoot自动装配核心）；

- **@ConditionalOnProperty**：配置文件满足条件时注册Bean。

#### 2. 面试核心考点

**SpringBoot自动装配原理核心**：通过`@ConditionalOnMissingBean`实现**用户自定义Bean优先，框架默认Bean兜底**。

#### 3. 易错坑点

- 条件注解仅对**当前配置类的Bean**生效，无法控制已注册的Bean；

- `@ConditionalOnMissingBean`基于**Bean类型/名称**匹配，容易出现匹配失误导致Bean未注册；

- 条件化Bean初始化优先级低于普通Bean，需注意依赖注入顺序问题。

# 3. Spring AOP 高频面试题 & 易错点

## 3.1 AOP 核心概念与专业术语面试

### 一、AOP 核心定义

AOP（面向切面编程），是**基于动态代理的代码增强技术**，在不修改原有业务代码的前提下，对方法进行拦截、增强，实现日志、事务、权限、限流等通用功能解耦。

### 二、核心专业术语（面试必问）

- **连接点（JoinPoint）**：可以被切面拦截的**所有方法**，是可增强的点位（Spring仅支持方法级连接点）。

- **切点（Pointcut）**：**筛选后的连接点**，即真正被切面拦截的方法，通过切点表达式匹配。

- **通知（Advice）**：切面的**具体增强逻辑**，分为前置、后置、返回、异常、环绕通知。

- **切面（Aspect）**：**切点 + 通知的结合体**，封装所有增强逻辑的类。

- **目标对象（Target）**：被AOP代理增强的原始业务Bean对象。

- **代理对象（Proxy）**：AOP生成的代理对象，对外提供访问，拦截目标方法执行。

- **织入（Weaving）**：将切面增强逻辑植入目标方法的过程，Spring为**运行时织入**。

### 三、高频面试题

**问：连接点和切点的区别？**

连接点是所有可拦截的方法（范围大），切点是通过表达式筛选后、真正被拦截的方法（范围小），切点是连接点的子集。

## 3.2 静态代理、JDK 动态代理、CGLIB 对比面试

### 一、三种代理方式核心对比

| 代理方式      | 实现原理                                   | 适用场景       | 优缺点                                         |
| ------------- | ------------------------------------------ | -------------- | ---------------------------------------------- |
| 静态代理      | 手动编写代理类，实现目标接口，包装目标对象 | 少量固定类代理 | 代码冗余、扩展性差，每新增类需新增代理类       |
| JDK动态代理   | 基于**接口**动态生成代理类                 | 目标类实现接口 | 无需手动写代理类，性能较好；**无接口无法使用** |
| CGLIB动态代理 | 基于**继承目标类**，生成子类代理           | 无接口的普通类 | 无需接口；无法代理final类、final方法           |

### 二、Spring AOP 代理规则（核心考点）

- Spring 5.2+ **默认优先使用CGLIB代理**，无论是否实现接口；

- 目标类**有接口**：可使用JDK/CGLIB；

- 目标类**无接口**：只能使用CGLIB；

- **final类、final方法、静态方法**：无法被CGLIB代理，AOP失效。

### 三、易错点

- 误区：JDK代理可以代理普通类。正确：JDK动态代理**必须基于接口**，无接口直接报错；

- 误区：CGLIB可以代理所有类。正确：final类、private方法、静态方法无法被CGLIB增强；

- 核心区别：JDK代理是**接口代理**，CGLIB是**类继承代理**。

## 3.3 AOP 五种通知执行顺序面试必问

### 一、五种通知类型

- **@Before**：前置通知，目标方法执行前执行；

- **@AfterReturning**：返回通知，目标方法**正常执行结束、无异常**时执行；

- **@AfterThrowing**：异常通知，目标方法**抛出异常**时执行；

- **@After**：后置通知，目标方法执行完毕（无论正常/异常）都会执行；

- **@Around**：环绕通知，包裹目标方法，可手动控制方法执行、前后增强、异常捕获。

### 二、标准执行顺序（无异常/有异常）

**1. 方法正常执行无异常**

环绕前置逻辑 → 前置通知@Before → 目标方法执行 → 环绕后置逻辑 → 返回通知@AfterReturning → 最终后置@After

**2. 方法执行抛出异常**

环绕前置逻辑 → 前置通知@Before → 目标方法报错 → 异常通知@AfterThrowing → 最终后置@After → 环绕异常处理

### 三、面试易错点

- **@AfterReturning 和 @After 区别**：@After一定执行，@AfterReturning仅无异常执行；

- 环绕通知优先级**高于所有普通通知**，是最优先、最后结束的通知；

- 如果环绕通知不执行`proceed()`，**目标方法和所有普通通知都不会执行**。

## 3.4 切点表达式写法与易错配置

### 一、常用切点表达式（面试常用）

#### 1. execution 表达式（最常用）

语法：`execution(返回值 包名.类名.方法名(参数))`

示例：拦截所有service层所有方法：`execution(* com.xxx.service.*.*(..))`

#### 2. 其他常用表达式

- **within**：按包/类范围拦截；

- **annotation**：拦截带有指定注解的方法；

- **args**：按方法参数类型拦截。

### 二、通配符含义

- *****：匹配任意返回值、任意类名、任意方法名、单个任意参数；

- **..**：匹配任意多层包、任意个数任意类型参数；

- **+**：匹配当前类及所有子类。

### 三、易错配置坑点

- 坑点1：表达式包名写错、层级不对，导致拦截不到目标方法；

- 坑点2：混淆`*`和`..`，参数匹配错误导致拦截失效；

- 坑点3：切点范围过大，拦截到系统方法、非业务方法，引发异常；

- 坑点4：注解切点仅匹配**方法注解**，类注解需单独配置。

## 3.5 AOP 失效经典场景与易错点

### 一、八大AOP失效经典场景（面试必考）

- **1. 同类内部调用**：A类的a方法调用本类b方法，b方法的切面失效。原因：内部调用不走代理对象，直接调用原生对象方法。

- **2. 静态方法/私有方法**：无法被动态代理增强，切面失效。

- **3. final 方法/类**：CGLIB无法继承重写final方法，无法代理。

- **4. 未被Spring管理的Bean**：普通new对象的类，不在容器中，无代理增强。

- **5. 切面类未加@Aspect、@Component**：切面未注册到容器，不生效。

- **6. 多切面优先级冲突**：未设置order优先级，切面执行顺序混乱，看似失效。

- **7. 环绕通知未执行proceed()**：目标方法无法执行，切面逻辑异常。

- **8. 事务失效同源问题**：@Transactional基于AOP，同类内部调用导致事务失效。

### 二、同类内部调用解决方案

1. 手动从容器获取代理对象调用；2. 通过AopContext获取当前代理对象；3. 拆分方法到不同类。

## 3.6 AOP 底层原理与面试深挖

### 一、AOP 核心底层原理

Spring AOP 底层基于**动态代理机制 + Bean后置处理器**实现：

1. Spring容器启动，扫描所有@Aspect切面类，解析切点表达式；

2. 目标Bean初始化完成后，通过**AnnotationAwareAspectJAutoProxyCreator（Bean后置处理器）**判断是否需要代理；

3. 若需要增强，自动生成JDK/CGLIB代理对象，替换原生Bean放入容器；

4. 调用方法时，通过代理对象拦截，执行切面通知逻辑。

### 二、AOP 与 AspectJ 区别（深挖考点）

- **Spring AOP**：**运行时动态代理**，纯Java实现，无需额外编译，仅支持方法级切面，轻量易用；

- **AspectJ**：**编译期织入**，支持字段、构造器、代码块等全方位切面，功能强大，需特殊编译插件。

### 三、高频深挖面试题

**1. Spring AOP 是静态代理还是动态代理？**

纯动态代理，运行时生成代理类，无预编译过程，属于运行时织入。

**2. AOP 和 拦截器、过滤器的区别？**

- Filter（过滤器）：Servlet层级，拦截请求，Web专属，先执行；

- Interceptor（拦截器）：Spring MVC层级，拦截Controller方法；

- AOP：Spring容器层级，拦截任意Bean方法，粒度更细、范围更广。

### 四、底层易错点

- Spring AOP **不支持字段级拦截**，仅支持方法级，区别于原生AspectJ；

- 代理对象仅在**容器中生效**，原生对象始终无增强逻辑；

- Bean后置处理器是AOP实现的核心，无后置处理器则无法生成代理。

# 4.Spring 事务 Transaction 面试题 & 易错点

## 4.1 事务四大特性与隔离级别面试

### 4.1.1 事务四大特性（ACID）核心详解

事务是数据库操作的最小执行单元，是一组不可分割的SQL操作集合，要么全部执行成功，要么全部执行失败回滚。所有关系型数据库事务都遵循**ACID四大核心特性**，这是事务的底层基础，也是面试必背核心考点。

**1. 原子性（Atomicity）**：事务是一个整体，不可拆分。事务内的所有数据库操作，要么全部成功提交，要么全部失败回滚，不会出现部分执行成功、部分失败的情况。

生活化类比：转账业务中，A给B转100元，A扣款、B加款两个操作必须同时成功或同时失败，不可能出现A钱扣了、B没到账的中间状态。Spring事务通过数据库原生的事务机制保证原子性，Spring框架仅做事务的统一管理与封装。

**2. 一致性（Consistency）**：事务执行前后，数据库的完整性约束（主键、唯一索引、外键、数据业务规则）不会被破坏，数据始终处于合法合规的一致状态。

核心价值：原子性是执行层面的一致，一致性是**数据业务层面的一致**。例如转账前后，A和B的总金额保持不变，不会出现凭空多出或消失资金的情况，这就是业务一致性。

**3. 隔离性（Isolation）**：多个事务并发执行时，事务之间相互隔离、互不干扰。数据库通过不同的隔离级别，控制并发事务的可见性，解决并发读写带来的数据问题。

核心意义：隔离性是为了平衡**并发性能**和**数据安全性**，隔离级别越高，数据一致性越好，但并发性能越低；隔离级别越低，并发性能越高，但会出现各类数据脏读问题。

**4. 持久性（Durability）**：事务一旦提交成功，对数据库的数据修改是永久生效的，即使服务器宕机、断电、重启，修改后的数据也不会丢失。数据库通过redo日志保证持久性，提交后的事务日志会持久化到磁盘，故障后可恢复数据。

### 4.1.2 并发事务三大问题

在事务并发执行场景下，低隔离级别会产生三类数据一致性问题，严重影响业务数据准确性，是理解隔离级别的前提：

**1. 脏读**：一个事务读取到了另一个事务**未提交**的修改数据。若另一个事务最终回滚，当前读取到的数据就是无效脏数据。

案例：事务A修改用户余额但未提交，事务B读取到修改后的余额，此时事务A回滚，事务B读取的就是脏数据。

**2. 不可重复读**：同一个事务内，多次读取同一条数据，结果不一致。原因是期间被其他事务**提交修改**了数据。

案例：事务A第一次查询用户余额为100元，事务B修改余额为200元并提交，事务A再次查询，余额变为200元，同一事务两次查询结果不同。

**3. 幻读**：同一个事务内，多次执行范围查询，结果条数不一致。原因是期间被其他事务**新增/删除**了数据。

案例：事务A查询所有余额大于100元的用户，得到10条数据，事务B新增一条符合条件的用户并提交，事务A再次查询，得到11条数据，出现“幻觉数据”。

### 4.1.3 四种隔离级别详解与对比

Spring完全兼容数据库的四种事务隔离级别，可通过`@Transactional(isolation = Isolation.XXX)`指定，级别从低到高排序，对应解决的并发问题、性能开销各不相同。

| 隔离级别                     | 脏读 | 不可重复读 | 幻读                               | 并发性能 | 适用场景                                     |
| ---------------------------- | ---- | ---------- | ---------------------------------- | -------- | -------------------------------------------- |
| 读未提交（READ UNCOMMITTED） | 存在 | 存在       | 存在                               | 最高     | 几乎不用，无数据一致性保障                   |
| 读已提交（READ COMMITTED）   | 解决 | 存在       | 存在                               | 较高     | Spring、MySQL默认级别，绝大多数业务场景      |
| 可重复读（REPEATABLE READ）  | 解决 | 解决       | 存在（InnoDB通过间隙锁大部分解决） | 中等     | 金融、订单、对账等对数据一致性要求较高场景   |
| 串行化（SERIALIZABLE）       | 解决 | 解决       | 解决                               | 最低     | 核心交易、账务结算、库存扣减等极致一致性场景 |

### 4.1.4 高频面试题与答案解析

**📌面试题1：MySQL和Spring默认的事务隔离级别是什么？**

答：MySQL InnoDB引擎默认隔离级别为**可重复读（REPEATABLE READ）**；Spring默认的事务隔离级别为**读已提交（READ COMMITTED）**。Spring不会强制修改数据库级别隔离，若注解未指定隔离级别，会沿用数据库默认配置。

**📌面试题2：可重复读级别为什么能解决大部分幻读问题？**

答：MySQL InnoDB引擎在可重复读级别下，通过**间隙锁+临键锁**锁定查询范围，阻止其他事务在该范围内新增、删除数据，从而规避幻读。但对于纯快照查询的场景，仍存在极小概率幻读，严格杜绝幻读需使用串行化级别。

### 4.1.5 避坑指南与最佳实践

⚠️避坑1：不要随意使用串行化级别，该级别会将并发事务变为串行执行，极大降低系统吞吐量，容易引发接口超时、数据库锁等待问题。

⚠️避坑2：误以为可重复读完全杜绝幻读，导致对账、库存业务出现数据偏差。

💡最佳实践：90%的普通业务使用默认读已提交级别；金融、订单、对账业务使用可重复读；核心资金交易、秒杀库存场景按需使用串行化。

## 4.2 传播机制 7 种类型面试详解

### 4.2.1 传播机制核心概念

Spring事务传播机制是指：**当一个带有事务的方法被另一个方法调用时，事务如何传递、嵌套、创建或挂起的规则**。简单来说，就是解决「嵌套方法调用时，事务是否共用、是否新建、是否挂起」的核心问题，是Spring事务最核心、面试频率最高的知识点。

数据库本身仅支持单层事务，**事务传播机制是Spring独有的特性**，不属于数据库原生功能。Spring通过AOP动态代理实现7种传播行为，适配不同的业务嵌套场景。

核心场景：业务分层调用（Controller→Service→Service）、Service内部方法互调、嵌套事务调用，几乎所有事务失效问题都和传播机制使用不当相关。

### 4.2.2 7种传播机制完整详解（含场景+案例）

Spring定义了7种事务传播行为，在`Propagation`枚举中定义，核心分为**支持当前事务**、**不支持当前事务**、**嵌套事务**三大类。

#### 1. REQUIRED（默认传播机制）

规则：**如果当前存在事务，就加入当前事务；如果没有事务，就新建一个事务**。是Spring事务的默认传播行为，也是生产使用最多的机制。

场景：绝大多数普通业务方法，方法执行必须在事务中，保证操作原子性。

特点：所有嵌套调用的方法共用同一个事务，任意一个方法异常，整体全部回滚，事务一致性最强。

#### 2. SUPPORTS

规则：**有事务就加入，无事务就以非事务方式执行**。

场景：查询类方法，无需强制开启事务，有事务时参与事务，无事务时直接查询，提升查询性能。

避坑：若外层无事务，当前方法出现异常不会回滚，不可用于增删改业务。

#### 3. MANDATORY

规则：**必须在已有事务中执行，无事务则直接抛出异常**。

场景：核心业务子方法，必须依赖外层事务执行，禁止独立执行，保证业务流程完整性。

#### 4. REQUIRES_NEW

规则：**无论当前是否有事务，都会新建一个独立事务，同时挂起外层事务**。内层事务与外层事务完全独立，互不影响。

核心特性：内层事务提交/回滚，不会影响外层事务；外层事务回滚，也不会影响已提交的内层事务。

场景：日志记录、消息投递、异步记录等需要独立落库、不受主业务事务回滚影响的操作。

#### 5. NOT_SUPPORTED

规则：**始终以非事务方式执行，若当前存在事务，直接挂起外层事务**。

场景：大批量数据查询、文件导出、统计汇总等无需事务的操作，避免事务占用数据库连接，提升性能。

#### 6. NEVER

规则：**必须在非事务环境下执行，若检测到当前有事务，直接抛出异常**。

场景：绝对禁止事务的独立查询、缓存刷新操作，避免事务锁影响业务。

#### 7. NESTED

规则：**有当前事务则嵌套执行子事务（保存点机制），无事务则新建事务**。子事务依赖主事务，主事务回滚则子事务必回滚，子事务可独立回滚不影响主事务。

核心区别：与REQUIRES_NEW的不同是，嵌套事务属于主事务的子事务，并非完全独立；REQUIRES_NEW是完全独立的两个事务。

场景：主业务流程不可回滚，子步骤失败可单独回滚重试的业务场景。

### 4.2.3 7种传播机制对比总结表

| 传播机制      | 外层有事务               | 外层无事务 | 事务独立性 | 常用场景                 |
| ------------- | ------------------------ | ---------- | ---------- | ------------------------ |
| REQUIRED      | 加入当前事务             | 新建事务   | 共用事务   | 所有常规增删改业务       |
| SUPPORTS      | 加入当前事务             | 无事务执行 | 依赖外层   | 普通查询业务             |
| MANDATORY     | 加入当前事务             | 抛异常     | 依赖外层   | 核心子业务方法           |
| REQUIRES_NEW  | 新建独立事务，挂起外层   | 新建事务   | 完全独立   | 日志、消息、独立记录     |
| NOT_SUPPORTED | 挂起外层事务，无事务执行 | 无事务执行 | 无事务     | 大数据查询、导出         |
| NEVER         | 抛异常                   | 无事务执行 | 无事务     | 禁止事务的独立操作       |
| NESTED        | 嵌套子事务（保存点）     | 新建事务   | 半独立     | 主流程不变、子步骤可重试 |

### 4.2.4 高频面试题解析

**📌面试题1：REQUIRES_NEW和NESTED的核心区别是什么？**

答：1. 事务层级不同：REQUIRES_NEW是两个完全独立的事务，互不干扰；NESTED是主事务下的子事务，依赖主事务存在。2. 回滚影响不同：REQUIRES_NEW内层事务提交后，外层回滚不影响内层；NESTED主事务回滚，子事务必须全部回滚，子事务回滚不影响主事务。3. 底层实现不同：REQUIRES_NEW是新建数据库事务；NESTED基于数据库保存点实现。

**📌面试题2：为什么日志记录方法要用REQUIRES_NEW？**

答：主业务事务失败回滚时，需要保留操作日志用于排查问题。使用REQUIRES_NEW可让日志事务独立提交，不受主事务回滚影响，保证日志永久落库。

## 4.3 @Transactional 注解底层原理

### 4.3.1 注解核心作用与使用方式

`@Transactional`是Spring提供的声明式事务注解，无需手动编写`begin、commit、rollback`事务代码，通过注解即可快速实现事务管理，是目前企业开发的主流事务使用方式。

注解可作用在**类上、方法上**：作用在类上时，当前类所有公共方法都继承事务配置；作用在方法上时，仅当前方法生效，方法级注解优先级高于类级注解。

基础使用示例：

```java
@Service
// 类级事务：所有方法默认开启事务
@Transactional(propagation = Propagation.REQUIRED, rollbackFor = Exception.class)
public class UserService {

    // 方法级事务：优先级高于类配置
    @Transactional(isolation = Isolation.REPEATABLE_READ)
    public void transferMoney() {
        // 转账业务逻辑，自动开启、提交、回滚事务
    }
}
```

### 4.3.2 底层核心原理：AOP动态代理

**@Transactional事务的核心底层原理是Spring AOP动态代理 + 事务拦截器**，属于典型的声明式AOP增强，无侵入业务代码。整体执行流程分为5个核心步骤：

**步骤1：Bean初始化扫描**：Spring启动时，通过后置处理器`TransactionAnnotationParser`扫描所有类和方法，识别带有`@Transactional`注解的Bean，标记为需要事务增强的对象。

**步骤2：创建代理对象**：Spring通过JDK动态代理（接口类）或CGLIB代理（无接口类），为目标Bean生成代理对象，后续业务调用均通过代理对象执行，而非原生对象。

**步骤3：事务拦截器拦截请求**：代理对象调用方法时，触发`TransactionInterceptor`事务拦截器，在目标方法执行前进行事务预处理。

**步骤4：事务开启与执行**：拦截器根据当前事务状态、传播机制规则，判断是否新建事务、加入事务或挂起事务，获取数据库连接、关闭自动提交、开启事务，随后执行目标业务方法。

**步骤5：事务提交/回滚**：方法执行无异常，代理自动提交事务；方法抛出指定异常，代理自动触发事务回滚，最终释放数据库连接。

### 4.3.3 核心执行链路源码简化梳理

完整核心链路：`业务调用方法 → AOP代理拦截 → TransactionInterceptor.invoke() → TransactionAspectSupport.invokeWithinTransaction() → 事务创建/传播判断 → 目标方法执行 → 异常捕获 → 事务提交/回滚`

关键核心类说明：

- **TransactionInterceptor**：事务核心拦截器，负责拦截方法、触发事务逻辑

- **PlatformTransactionManager**：事务管理器顶层接口，负责事务的提交、回滚、获取连接

- **TransactionDefinition**：事务定义接口，封装传播机制、隔离级别、超时时间、只读属性

### 4.3.4 面试高频考点与避坑

**📌面试题1：@Transactional为什么只能作用于public方法？**

答：Spring AOP动态代理仅能拦截**公共public方法**，private、protected、default方法无法被代理拦截，事务增强逻辑不会生效。若注解加在非public方法上，事务完全失效，不会报错但无事务效果。

**📌面试题2：Spring事务是基于静态代理还是动态代理？**

答：基于**运行时动态代理**，默认优先JDK动态代理（实现接口的类），无接口则使用CGLIB动态代理，在Bean初始化阶段生成代理对象。

⚠️避坑指南：很多开发者误以为注解加了就一定有事务，忽略代理生效条件，导致事务静默失效，这是生产最高频的bug来源。

## 4.4 事务失效 10 大经典场景（高频易错）

事务失效是面试必考、生产最高频的问题，99%的事务异常都源于以下10种场景。所有失效场景的核心本质：**Spring AOP代理未生效、事务规则不匹配、异常捕获逻辑覆盖事务**。本节逐一拆解场景、原因、复现方式、解决方案。

### 4.4.1 场景1：注解作用在非public方法

**失效原因**：Spring AOP动态代理仅拦截public公共方法，private、protected、包访问权限方法无法被代理，事务拦截逻辑不会执行，事务直接失效。

**错误代码**：

```java
// 非public方法，事务失效
@Transactional
void updateUser() {
    // 增删改操作，异常不会回滚
}
```

**解决方案**：所有需要事务的方法统一改为public修饰。

### 4.4.2 场景2：类内部方法自调用

**失效原因**：内部方法调用（this.方法名()）是通过**原生对象**调用，而非Spring代理对象，AOP拦截器无法触发，事务逻辑不执行。

**错误场景**：A方法（无事务）调用本类B方法（有@Transactional），B方法事务失效。

**解决方案**：1. 拆分方法到不同类；2. 通过Spring上下文获取代理对象调用；3. 使用`AopContext.currentProxy()`获取当前代理对象。

### 4.4.3 场景3：未捕获的受检异常（默认不回滚）

**失效原因**：Spring事务**默认仅对RuntimeException和Error回滚**，对Exception受检异常（IOException、SQLException等）不触发回滚。

**解决方案**：手动指定回滚异常：`@Transactional(rollbackFor = Exception.class)`

### 4.4.4 场景4：手动try-catch捕获所有异常

**失效原因**：业务代码手动try-catch捕获异常，未向外抛出，事务拦截器检测不到异常，默认认为方法执行正常，自动提交事务，不会回滚。

**解决方案**：捕获异常后手动抛出，或手动触发事务回滚。

```java
// 正确写法：捕获后抛出异常
try {
    // 业务操作
} catch (Exception e) {
    throw new RuntimeException(e);
}
```

### 4.4.5 场景5：传播机制配置错误

**失效场景**：子方法配置`Propagation.SUPPORTS`、`NOT_SUPPORTED`、`NEVER`，外层无事务时，子方法无事务，异常不回滚。

**解决方案**：增删改业务统一使用默认的REQUIRED传播机制。

### 4.4.6 场景6：数据库引擎不支持事务

**失效原因**：MySQL MyISAM引擎**不支持事务、锁机制**，仅InnoDB引擎支持事务。即使添加注解，事务也完全失效。

**解决方案**：数据表统一使用InnoDB引擎。

### 4.4.7 场景7：方法被final/static修饰

**失效原因**：CGLIB代理通过重写方法实现增强，final方法无法重写；static方法属于类而非对象，无法被代理拦截，事务失效。

**解决方案**：事务方法禁止添加final、static修饰符。

### 4.4.8 场景8：多线程异步调用事务方法

**失效原因**：Spring事务基于**ThreadLocal**存储事务上下文，多线程场景下，子线程无法获取主线程的事务连接，无法加入当前事务，事务失效。

**解决方案**：异步方法独立开启事务，不与主线程事务共用。

### 4.4.9 场景9：只读事务配置导致写入失败

**失效原因**：配置`@Transactional(readOnly = true)`只读事务，数据库连接被设置为只读，执行增删改操作会直接报错，事务失效。

**解决方案**：查询方法开启只读事务，增删改方法关闭只读配置。

### 4.4.10 场景10：事务超时时间过短

**失效原因**：事务默认超时时间为30秒，若业务执行耗时超过超时时间，Spring自动强制回滚事务，导致业务执行失败。

**解决方案**：长耗时业务适当调整超时时间，或拆分事务。

### 4.4.11 面试总结题

**📌面试题：请说出至少5种Spring事务失效的场景及解决方案**

答：整合高频场景：非public方法、内部自调用、受检异常未指定回滚、try-catch吞异常、MyISAM引擎、final/static方法、多线程调用，对应解决方案如上，是面试必背答题模板。

## 4.5 事务回滚规则、受检/非受检异常坑点

### 4.5.1 Spring默认回滚规则详解

Spring事务的默认回滚规则是面试和生产最容易踩坑的点，很多开发者默认认为「所有异常都会触发事务回滚」，实际完全相反。

**默认规则**：

- **触发回滚**：运行时异常（RuntimeException）、系统错误（Error），如空指针、数组越界、Assert断言异常。

- **不触发回滚**：所有受检异常（Exception直接子类），如IOException、SQLException、自定义受检异常。

核心原理：Spring事务拦截器捕获异常后，会判断异常类型，仅对**非受检异常**执行回滚，受检异常默认提交事务。

### 4.5.2 受检异常与非受检异常核心区别

| 异常类型                       | 定义                             | 默认事务行为 | 常见案例                                            |
| ------------------------------ | -------------------------------- | ------------ | --------------------------------------------------- |
| 非受检异常（RuntimeException） | 程序运行时异常，无需手动捕获声明 | 自动回滚     | NullPointerException、IndexOutOfBoundsException     |
| 受检异常（Exception）          | 编译期异常，必须捕获或抛出       | 默认不回滚   | IOException、FileNotFoundException、自定义Exception |

### 4.5.3 高频坑点与解决方案

⚠️坑点1：自定义受检异常抛出后，事务不回滚，导致数据脏数据。

解决方案：全局指定回滚所有异常，统一业务规则：

```java
// 所有异常均触发回滚，生产通用配置
@Transactional(rollbackFor = Exception.class)
```

⚠️坑点2：手动捕获异常后打印日志，未抛出，事务正常提交。

解决方案：捕获异常后必须重新抛出，或手动回滚事务：

```java
// 手动触发事务回滚
TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
```

### 4.5.4 面试高频题

**📌面试题：为什么Spring默认只对运行时异常回滚？**

答：设计理念：受检异常是开发者可预知、可处理的异常，属于业务可控异常，Spring认为开发者可以自行处理，无需强制回滚；运行时异常是程序未知bug、不可控异常，属于系统异常，必须回滚保证数据一致性。

## 4.6 嵌套事务、挂起事务面试深挖

### 4.6.1 嵌套事务核心原理（NESTED）

嵌套事务是Spring高级事务特性，基于数据库**保存点（Savepoint）**实现，区别于独立事务。当外层存在事务时，内层NESTED事务不会新建独立事务，而是在主事务中创建一个保存点，形成子事务。

**核心特性**：

1. 子事务异常回滚，仅回滚到当前保存点，**不影响外层主事务**，主事务可继续执行并正常提交。

2. 外层主事务异常回滚，**所有嵌套子事务必须全部回滚**。

3. 外层无事务时，NESTED等同于REQUIRED，新建独立事务。

### 4.6.2 挂起事务原理（REQUIRES_NEW）

挂起事务是REQUIRES_NEW传播机制的核心特性。当外层存在事务时，调用REQUIRES_NEW方法，Spring会**暂停外层事务（挂起）**，新建一个完全独立的内层事务，内层事务执行完成后，再恢复外层事务。

**核心特性**：两个事务完全独立，拥有独立的数据库连接、独立的提交/回滚状态，互不干扰。内层事务提交后，外层回滚不会影响内层数据。

### 4.6.3 嵌套事务与挂起事务深度对比

| 对比维度   | NESTED嵌套事务     | REQUIRES_NEW挂起事务 |
| ---------- | ------------------ | -------------------- |
| 底层实现   | 数据库保存点       | 新建独立数据库事务   |
| 事务独立性 | 半独立，依赖主事务 | 完全独立，互不依赖   |
| 主事务回滚 | 子事务强制回滚     | 不影响已提交的子事务 |
| 子事务回滚 | 不影响主事务       | 不影响主事务         |
| 资源占用   | 低，共用连接       | 高，新建连接         |

### 4.6.4 生产使用场景与最佳实践

💡NESTED适用场景：主流程核心操作不可回滚，子步骤操作失败可单独回滚重试，比如订单创建（主事务）+ 积分发放（子嵌套事务），积分发放失败不影响订单创建。

💡REQUIRES_NEW适用场景：需要独立落库的日志记录、消息投递、操作记录，主业务失败回滚，日志依然保留。

### 4.6.5 面试深挖题

**📌面试题：嵌套事务和独立事务的最大区别？生产中如何选择？**

答：最大区别是事务依赖关系和回滚联动性。嵌套事务依附主事务，主事务失败整体回滚，适合子步骤可重试场景；独立事务完全隔离，适合日志、审计等需要永久落库的场景，生产中优先使用REQUIRED常规事务，特殊场景按需使用两种高级事务。

## 4.7 分布式事务基础面试问答

### 4.7.1 分布式事务产生背景

单体架构中，所有数据库操作在同一个库、同一个连接内，Spring本地事务即可保证ACID。微服务架构下，一次业务操作需要调用多个服务、操作多个数据库，**本地事务无法跨库、跨服务保证数据一致性**，由此产生分布式事务问题。

核心痛点：跨服务操作时，部分服务执行成功、部分服务执行失败，本地事务无法统一回滚，导致数据不一致。

### 4.7.2 分布式事务核心理论：CAP与BASE

**CAP定理**：分布式系统无法同时满足一致性（C）、可用性（A）、分区容错性（P），只能三选二。互联网微服务系统默认保证P（分区容错），因此只能在C和A之间取舍。

**BASE理论**：是分布式事务的落地理论，核心是**基本可用、软状态、最终一致性**。放弃实时强一致性，保证系统高可用，允许数据短暂不一致，最终通过补偿、重试达到数据一致，是目前互联网分布式事务的主流方案。

### 4.7.3 四大经典分布式事务方案（面试基础）

#### 1. 2PC（两阶段提交）

分为准备阶段、提交阶段，协调者统一管理所有参与者事务。优点是强一致性，缺点是阻塞严重、性能差，不适合高并发互联网场景，多用于传统金融系统。

#### 2. TCC（补偿事务）

手动编码实现Try（资源校验、预留）、Confirm（确认提交）、Cancel（回滚补偿）三个接口，无锁、高性能，适合核心交易场景，缺点是代码侵入性强、开发成本高。

#### 3. SAGA模式

长事务拆分多个本地子事务，每个子事务对应一个补偿操作，失败后反向执行补偿逻辑，适合长流程业务，最终一致性。

#### 4. 本地消息表/可靠消息最终一致性

基于本地事务+消息队列实现，业务操作和消息落库在同一个本地事务，定时任务补偿重试，是互联网使用最广的低成本方案。

### 4.7.4 高频面试问答

**📌面试题1：Spring本地事务和分布式事务的区别？**

答：本地事务基于单数据库连接，由Spring AOP实现，保证单库ACID；分布式事务跨库、跨服务，无统一事务上下文，需要额外的分布式方案保证最终一致性，无法使用原生@Transactional实现。

**📌面试题2：为什么微服务不能用本地事务？**

答：本地事务的ThreadLocal上下文仅当前服务有效，无法传递到远程服务，跨服务调用的数据库连接相互独立，无法统一提交和回滚，因此无法保证数据一致性。

# 5.Spring 资源加载 & 国际化面试题 & 易错点

## 5.1 Resource 资源加载体系面试

### 5.1.1 核心概念定义

**Spring Resource资源加载体系**是Spring框架自研的一套统一资源访问抽象体系，用于替代Java原生的URL、File、IO流等零散的资源读取方式。简单来说，就是Spring为了统一读取**本地文件、类路径文件、网络资源、系统资源、Jar包内资源**等不同来源的资源，定义的一套标准化接口和实现类。

Java原生IO存在明显短板：原生URL协议仅支持http、https、file等少数协议，无法读取Jar包内资源、类路径资源，且不同资源的读取方式不统一，代码冗余且兼容性差。Spring Resource体系完美解决了这一问题，实现了**资源读取统一化、协议拓展化、场景全覆盖**。

### 5.1.2 核心原理与核心组件

Spring资源加载体系的核心是**Resource接口**和**ResourceLoader资源加载器**，两大核心组件各司其职，构成完整的资源加载链路。

**1. Resource接口（资源模型）**

作为所有资源的顶层抽象，定义了资源读取的统一规范，包含获取输入流、获取资源路径、判断资源是否存在、获取文件大小等核心方法，核心实现类如下：

- **FileSystemResource**：读取本地文件系统资源（绝对路径/相对路径文件）

- **ClassPathResource**：读取类路径下资源（resources目录、classpath路径文件），是开发最常用的资源类型

- **UrlResource**：读取网络URL资源（http/https协议资源）

- **JarFileResource**：读取Jar包内部嵌套资源，解决打包后资源读取失效问题

**2. ResourceLoader接口（资源加载器）**

核心作用：根据**资源路径前缀协议**，自动匹配对应的Resource实现类，完成资源加载。Spring容器顶层接口ApplicationContext直接继承了ResourceLoader，因此**所有Spring容器本身就是资源加载器**，可以直接加载资源。

核心协议规则（Spring自动识别）：

- classpath: ：加载类路径资源，对应ClassPathResource

- file: ：加载本地文件系统资源，对应FileSystemResource

- http/https: ：加载网络资源，对应UrlResource

- 无协议前缀：默认优先从类路径加载资源

### 5.1.3 场景与生产价值

1. 统一资源读取代码，无需针对不同资源编写不同IO逻辑，大幅简化代码；

2. 支持Jar包内资源读取，解决SpringBoot打包后resources目录资源读取失败的经典问题；

3. 为Spring国际化配置、配置文件加载、静态资源映射、自定义配置加载提供底层支撑；

4. 支持动态资源加载，适配开发、测试、生产多环境资源路径差异。

### 5.1.4 实操示例（可直接运行）

```java
import org.springframework.core.io.Resource;
import org.springframework.core.io.ResourceLoader;
import org.springframework.stereotype.Component;
import javax.annotation.Resource;
import java.io.BufferedReader;
import java.io.IOException;
import java.io.InputStreamReader;

/**
 * Spring Resource资源加载实操示例
 * 容器自动注入ResourceLoader，无需手动创建
 */
@Component
public class ResourceLoadDemo {

    // 注入Spring容器内置的资源加载器
    @Resource
    private ResourceLoader resourceLoader;

    public void loadResource() {
        try {
            // 1.加载类路径资源（resources目录下的test.txt文件）
            Resource classPathResource = resourceLoader.getResource("classpath:test.txt");
            readResource(classPathResource);

            // 2.加载本地文件资源（绝对路径）
            Resource fileResource = resourceLoader.getResource("file:D:/test.txt");
            readResource(fileResource);

            // 3.加载网络资源
            Resource urlResource = resourceLoader.getResource("https://www.baidu.com");
            readResource(urlResource);
        } catch (IOException e) {
            e.printStackTrace();
        }
    }

    /**
     * 统一资源读取方法
     * @param resource 任意类型Spring资源
     */
    private void readResource(Resource resource) throws IOException {
        // 判断资源是否存在
        if (resource.exists()) {
            // 读取资源输入流
            BufferedReader reader = new BufferedReader(new InputStreamReader(resource.getInputStream()));
            String line;
            while ((line = reader.readLine()) != null) {
                System.out.println("资源内容：" + line);
            }
            reader.close();
        }
    }
}

```

### 5.1.5 避坑指南

⚠️ **坑点1：打包后ClassPathResource读取资源失败**

问题原因：开发环境资源是文件形式，打包后资源嵌套在Jar包中，Java原生File无法读取Jar内嵌套文件。

解决方案：必须使用**getResource().getInputStream()**流读取，绝对不能使用getFile()方法获取文件路径读取。

⚠️ **坑点2：无协议前缀资源加载优先级混乱**

问题原因：无协议前缀时，不同容器加载规则不同，FileSystemXmlApplicationContext默认读取相对路径文件，ClassPathXmlApplicationContext默认读取类路径。

解决方案：生产环境必须**显式添加协议前缀**，避免模糊匹配导致资源加载异常。

### 5.1.6 面试高频题

📌 **面试题1：Spring Resource体系和Java原生IO的区别？**

答：1. 统一性：Spring Resource统一各类资源读取API，原生IO不同资源读取方式零散；2. 兼容性：Spring支持Jar包内资源读取，原生IO不支持；3. 拓展性：Spring支持自定义资源协议拓展，原生IO协议固定；4. 便捷性：Spring结合Spring容器自动管理资源，无需手动关闭流、处理路径差异。

📌 **面试题2：Spring中ResourceLoader的核心作用？ApplicationContext和ResourceLoader的关系？**

答：ResourceLoader核心是根据路径协议自动匹配资源类型，完成资源加载。ApplicationContext直接继承ResourceLoader，所以Spring容器本身具备资源加载能力，可直接调用getResource()方法加载资源。

## 5.2 国际化 I18N 核心组件面试

### 5.2.1 核心概念定义

**国际化I18N（Internationalization）**是指网站/系统可以根据用户的地区、语言设置，自动展示对应语言的文字、提示信息、页面内容，无需修改代码重启服务。Spring框架提供了一套完整的国际化解决方案，开箱即用，无需依赖第三方组件。

Spring国际化的核心设计思想：**将所有展示文案抽离到独立的语言配置文件中，通过动态切换语言文件，实现页面、接口提示信息的语言切换**，彻底硬编码文字带来的多语言适配难题。

### 5.2.2 核心组件与底层原理

Spring国际化体系由**四大核心组件**构成，各司其职，形成完整的国际化解析链路，是面试核心考点：

**1. MessageSource（核心顶层接口）**

国际化信息源核心接口，负责读取多语言配置文件、解析文案、替换占位符。Spring容器默认自动注册**ReloadableResourceBundleMessageSource**实现类，支持动态刷新语言配置，无需重启服务。

核心功能：读取basename匹配的语言文件、解析带占位符的文案、默认文案兜底、多语言优先级匹配。

**2. LocaleResolver（区域解析器）**

负责**识别用户当前的语言环境（Locale）**，也就是判断用户需要展示中文、英文还是其他语言，是语言切换的核心枢纽。Spring提供多种默认实现，适配不同业务场景。

**3. LocaleChangeInterceptor（语言切换拦截器）**

拦截用户请求，根据请求参数动态修改用户的语言环境，实现**动态语言切换**。默认识别请求参数locale，可自定义参数名。

**4. 多语言配置文件**

资源文件后缀规范：basename_语言代码_国家代码.properties，例如：messages_zh_CN.properties（简体中文）、messages_en_US.properties（美式英文），默认文件为messages.properties（兜底文案）。

### 5.2.3 完整执行流程

1. 系统启动：Spring自动加载MessageSource，读取classpath下的多语言配置文件，缓存所有文案；

2. 用户请求：LocaleResolver解析当前请求的语言环境（默认根据请求头Accept-Language）；

3. 动态切换：若携带locale请求参数，LocaleChangeInterceptor拦截请求并修改当前Locale；

4. 文案渲染：系统根据最终Locale匹配对应的语言文件，读取并解析文案，返回给用户。

第二步：SpringBoot国际化配置类

```java

import org.springframework.context.annotation.Configuration;
import org.springframework.context.support.ReloadableResourceBundleMessageSource;
import org.springframework.web.servlet.i18n.LocaleChangeInterceptor;
import org.springframework.web.servlet.i18n.SessionLocaleResolver;
import java.util.Locale;

@Configuration
public class I18nConfig {

    /**
     * 配置国际化信息源
     */
    @Bean
    public ReloadableResourceBundleMessageSource messageSource() {
        ReloadableResourceBundleMessageSource messageSource = new ReloadableResourceBundleMessageSource();
        // 指定语言文件前缀（无需写后缀.properties）
        messageSource.setBasename("classpath:messages");
        // 设置编码，解决中文乱码
        messageSource.setDefaultEncoding("UTF-8");
        // 开启配置文件动态刷新（生产可关闭）
        messageSource.setCacheSeconds(0);
        return messageSource;
    }

    /**
     * 配置区域解析器（session级别）
     */
    @Bean
    public SessionLocaleResolver localeResolver() {
        SessionLocaleResolver resolver = new SessionLocaleResolver();
        // 设置系统默认语言为中文
        resolver.setDefaultLocale(Locale.CHINA);
        return resolver;
    }

    /**
     * 配置语言切换拦截器
     */
    @Bean
    public LocaleChangeInterceptor localeChangeInterceptor() {
        LocaleChangeInterceptor interceptor = new LocaleChangeInterceptor();
        // 指定切换语言的请求参数名
        interceptor.setParamName("locale");
        return interceptor;
    }
}

```

### 5.2.6 面试高频题

📌 **面试题：Spring国际化的核心组件及各自作用？**

答：1. MessageSource：读取、解析多语言配置文件，提供文案查询能力；2. LocaleResolver：解析用户当前语言环境；3. LocaleChangeInterceptor：拦截请求，动态切换语言；4. 多语言资源文件：存储不同语言的静态文案，实现文案与代码解耦。

## 5.3 LocaleResolver 四种实现对比面试

### 5.3.1 核心概念

**LocaleResolver（区域解析器）**是SpringMVC用于识别用户语言环境的核心组件，整个国际化语言的匹配、切换、持久化全部依赖该组件。Spring官方提供了**四种核心实现类**，适配不同的业务场景，也是面试高频对比考点，极易混淆。

### 5.3.2 四种实现类详细对比

| 实现类                         | 解析规则                                                    | 语言持久化方式                                   | 适用场景                                                   | 优缺点                                                       |
| ------------------------------ | ----------------------------------------------------------- | ------------------------------------------------ | ---------------------------------------------------------- | ------------------------------------------------------------ |
| **AcceptHeaderLocaleResolver** | 读取HTTP请求头的Accept-Language字段，根据浏览器默认语言解析 | 不持久化，每次请求重新解析                       | 简单展示型系统、无需手动切换语言的官网                     | 优点：无需手动配置，自适应浏览器语言；缺点：不支持手动切换语言，无法记忆用户选择 |
| **SessionLocaleResolver**      | 优先读取Session中存储的Locale，无则使用默认语言             | 存储在用户Session中，会话有效期间持久            | 后台管理系统、单用户会话系统、中小型项目                   | 优点：支持手动切换语言，会话内记忆用户选择；缺点：会话失效后语言重置，集群环境需共享Session |
| **CookieLocaleResolver**       | 优先读取Cookie中存储的Locale，无则使用默认语言              | 存储在客户端Cookie中，永久有效（可配置过期时间） | 前后端分离项目、大型分布式系统、需要长期记忆用户语言的场景 | 优点：跨会话、跨设备持久化，适配集群环境；缺点：依赖客户端Cookie，用户清空Cookie后失效 |
| **FixedLocaleResolver**        | 固定全局唯一语言，不随用户请求、操作改变                    | 无持久化，全局固定                               | 仅单语言的国际化项目、测试环境                             | 优点：配置简单，性能最高；缺点：**不支持任何语言动态切换**，生产极少使用 |

### 5.3.3 核心原理与场景选型

1. **默认生效规则**：SpringMVC默认使用**AcceptHeaderLocaleResolver**，如果手动注册了其他LocaleResolver，默认实现会被覆盖；

2. **动态切换前提**：只有Session、Cookie解析器支持动态语言切换，AcceptHeader、Fixed解析器**无法配合LocaleChangeInterceptor实现语言切换**；

3. **分布式场景最优解**：分布式、微服务项目优先使用CookieLocaleResolver，避免Session共享问题；单体项目优先使用SessionLocaleResolver，配置简单。

### 5.3.4 实操配置（Cookie解析器示例）

```java
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.servlet.i18n.CookieLocaleResolver;
import java.util.Locale;

@Configuration
public class CookieI18nConfig {

    @Bean
    public CookieLocaleResolver localeResolver() {
        CookieLocaleResolver resolver = new CookieLocaleResolver();
        // 默认语言
        resolver.setDefaultLocale(Locale.CHINA);
        // Cookie名称
        resolver.setCookieName("system-locale");
        // Cookie有效期（7天）
        resolver.setCookieMaxAge(60 * 60 * 24 * 7);
        return resolver;
    }
}

```

### 5.3.5 面试高频题

📌 **面试题1：四种LocaleResolver的区别及生产选型？**

答：核心区别在于**语言存储方式和是否支持动态切换**。AcceptHeader基于请求头、无持久化、不可切换；Session基于会话、会话持久、可切换、适配单体项目；Cookie基于客户端缓存、跨会话持久、可切换、适配分布式项目；Fixed固定语言、不可切换。生产单体项目选SessionLocaleResolver，分布式项目选CookieLocaleResolver。

📌 **面试题2：为什么配置了语言切换参数不生效？**

答：大概率是使用了默认的AcceptHeaderLocaleResolver，该解析器不支持动态修改Locale，必须手动注册Session或Cookie解析器才能实现语言切换。

## 5.4 中文乱码、语言切换失效易错坑点

### 5.4.1 坑点一：国际化中文乱码终极解决方案

**问题现象**：配置文件中文显示正常，项目运行后国际化文案中文乱码，英文正常显示。

**根本原因**：

1. Spring默认资源文件编码为ISO-8859-1，不支持中文；

2. IDEA默认properties文件编码不是UTF-8，文件保存时中文被转义；

3. 未配置MessageSource默认编码。

**完整解决步骤**：

1. IDEA设置：Settings-File Encodings，将Properties文件编码设置为UTF-8，开启透明native-to-ascii转换；

2. 代码层面强制指定UTF-8编码：messageSource.setDefaultEncoding("UTF-8")；

3. 全局Web编码过滤器配置，强制请求响应UTF-8编码；

4. 重新创建语言配置文件，避免旧编码缓存。

### 5.4.2 坑点二：语言切换参数不生效

**问题现象**：请求携带?locale=en_US参数，页面语言无变化，始终为默认语言。

**核心原因与排查步骤**：

1. 未注册可切换的LocaleResolver：使用默认AcceptHeader解析器，无法动态修改语言，必须手动注册Session/Cookie解析器；

2. 未将拦截器注册到Spring容器：仅创建LocaleChangeInterceptor实例，未添加到Web拦截器链，请求无法被拦截；

3. 请求参数名不匹配：自定义了paramName，前端传参和后端配置不一致；

4. 配置文件basename错误，文案匹配失败，默认展示兜底语言。

**修复方案**：实现WebMvcConfigurer接口，将拦截器添加到拦截器链：

```java
import org.springframework.web.servlet.config.annotation.InterceptorRegistry;
import org.springframework.web.servlet.config.annotation.WebMvcConfigurer;
import org.springframework.web.servlet.i18n.LocaleChangeInterceptor;
import org.springframework.context.annotation.Configuration;

@Configuration
public class WebConfig implements WebMvcConfigurer {

    private final LocaleChangeInterceptor localeChangeInterceptor;

    public WebConfig(LocaleChangeInterceptor localeChangeInterceptor) {
        this.localeChangeInterceptor = localeChangeInterceptor;
    }

    // 注册国际化拦截器
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(localeChangeInterceptor).addPathPatterns("/**");
    }
}

```

### 5.4.3 坑点三：打包后国际化文案失效

**问题现象**：开发环境正常，Jar包部署后提示文案找不到、切换失效。

**原因**：SpringBoot打包后，properties文件被编译到Jar包内，默认的资源读取方式无法读取嵌套资源。

**解决方案**：使用**ReloadableResourceBundleMessageSource**替代原生MessageSource，支持Jar包内资源读取，同时开启动态刷新。

### 5.4.4 坑点四：多语言文件优先级混乱

**问题现象**：存在多个语言文件，文案覆盖异常、兜底文案不生效。

**优先级规则**：精准匹配文件 > 通用语言文件 > 默认文件，例如：zh_CN.properties > zh.properties > messages.properties。

**避坑**：相同key不要在多个文件重复定义，避免优先级覆盖导致文案异常。

## 5.5 生产级国际化最佳实践面试题

### 5.5.1 生产落地最佳实践

💡 **最佳实践1：统一文件规范**

所有多语言文件统一放置在resources根目录，统一basename为messages，文件命名严格遵循 前缀_语言_国家.properties，禁止自定义混乱命名，便于统一维护。

💡 **最佳实践2：分布式项目优先Cookie持久化**

微服务、集群环境绝对禁止使用SessionLocaleResolver，避免会话共享问题，统一使用CookieLocaleResolver，配置7天有效期，提升用户体验。

💡 **最佳实践3：开启文案动态刷新**

开发环境开启动态刷新（cacheSeconds=0），修改语言文件无需重启服务；生产环境关闭刷新，设置缓存时间，提升接口性能。

💡 **最佳实践4：统一全局异常国际化**

将所有自定义异常、参数校验提示、业务提示全部抽离到国际化文件，禁止代码硬编码文字，实现全站统一国际化。

💡 **最佳实践5：配置默认文案兜底**

所有国际化key必须在默认messages.properties文件配置兜底文案，避免匹配不到语言文件时出现空文案、系统报错。

### 5.5.2 高频面试真题解析

📌 **面试题1：分布式微服务项目国际化如何落地？有哪些坑？**

答：1. 技术选型：统一使用CookieLocaleResolver实现语言持久化，规避Session集群共享问题；2. 资源统一管理：将多语言文件统一放到公共依赖模块，所有微服务统一引用，避免重复配置；3. 全局统一解析：通过RequestContextHolder获取当前请求的Locale，统一工具类解析国际化文案；4. 避坑点：禁止各服务单独配置语言文件、禁止使用Session存储语言、必须配置UTF-8编码防止乱码。

📌 **面试题2：国际化文案占位符如何实现动态参数替换？生产如何规范使用？**

答：Spring MessageSource原生支持占位符替换，通过getMessage的Object[]参数传递动态参数。生产规范：1. 占位符按顺序编号{0}{1}，禁止无序；2. 动态参数数量必须和文案占位符数量一致；3. 复杂文案拆分多个key，避免单条文案参数过多难以维护。

📌 **面试题3：如何实现无侵入的全局国际化？**

答：1. 自定义全局响应处理器，统一拦截所有返回文案，自动替换国际化内容；2. 全局异常处理器中统一解析异常文案；3. 参数校验提示通过ValidationMessages.properties统一国际化；4. 基于ThreadLocal存储当前Locale，全局可获取，无侵入解析文案。

# 6.Spring Web & SpringMVC 高频面试题

## 6.1 SpringMVC 整体执行流程面试必问

### 6.1.1 核心概念定义

SpringMVC是Spring框架原生的**MVC分层Web框架**，基于Servlet规范实现，用于快速开发Java Web接口、处理HTTP请求。其核心设计思想是**中心化调度、组件解耦**，通过统一的核心调度器分发所有请求，配合映射器、适配器、解析器等组件，完成请求接收、处理、响应的全流程。

相比于原生Servlet，SpringMVC规避了原生Servlet重复编写注册、参数解析、视图跳转、异常处理的冗余代码，通过自动化组件完成Web开发的通用逻辑，让开发者只需要聚焦业务代码实现。

### 6.1.2 完整执行核心原理（九大核心步骤）

SpringMVC的执行流程是面试**必考原题**，底层围绕核心中枢**DispatcherServlet（前端控制器）**展开，完整执行流程分为9个闭环步骤，每一步各司其职、完全解耦：

**步骤1：请求接收**：客户端（浏览器、Postman、前端项目）发起HTTP/HTTPS请求，请求到达服务器后，被SpringMVC的核心调度器 **DispatcherServlet** 拦截接收，所有Web请求统一进入该入口。

**步骤2：处理器映射（HandlerMapping）**：DispatcherServlet接收请求后，不会直接处理业务，而是调用**处理器映射器**。映射器根据请求的URL路径、请求方式，匹配Spring容器中注册的Controller处理器（Handler），找到对应的处理方法，并返回包含处理器信息和拦截器链的**HandlerExecutionChain**执行链。

**步骤3：处理器适配（HandlerAdapter）**：由于SpringMVC支持多种处理器（普通Controller、注解Controller、静态资源处理器等），不同处理器执行规则不同。因此需要**处理器适配器**根据匹配到的处理器类型，适配对应的执行规则，统一调用标准。

**步骤4：拦截器前置处理**：适配器执行处理器前，会依次执行拦截器链的 `preHandle()` 方法。若任意拦截器返回false，直接终止请求，返回响应；全部返回true才继续执行后续逻辑。

**步骤5：执行目标处理器方法**：适配器调用目标Controller的业务方法，同时完成**参数绑定、类型转换、数据格式化、参数校验**，执行开发者编写的业务逻辑代码，最终返回ModelAndView对象（包含视图名称、数据模型）。

**步骤6：拦截器后置处理**：业务方法执行完成后，执行拦截器链的 `postHandle()` 方法，此时可以对ModelAndView数据进行二次修改、视图调整。

**步骤7：视图解析（ViewResolver）**：DispatcherServlet接收ModelAndView后，调用**视图解析器**，将逻辑视图名（如index、success）解析为物理视图资源（如/WEB-INF/index.jsp、HTML页面），同时填充Model中的数据。

**步骤8：视图渲染**：系统加载物理视图，将Model数据渲染到视图页面，生成最终的响应内容。

**步骤9：最终拦截处理与响应返回**：执行拦截器的 `afterCompletion()` 方法（无论业务是否报错都会执行，用于资源释放、日志记录），最终将渲染完成的响应数据返回给客户端。

### 6.1.3 场景与核心价值

生产中所有SpringBoot/SpringMVC接口请求都遵循该流程，其核心价值在于**组件化解耦**：将请求分发、路径匹配、方法执行、视图解析、拦截控制等通用逻辑拆分独立组件，开发者可单独扩展任意组件（如自定义拦截器、自定义视图解析器），无需修改核心源码，符合开闭原则。

### 6.1.4 避坑指南

- **误区1**：认为Controller直接接收请求。**真相**：所有请求必须经过DispatcherServlet统一调度，Controller仅负责业务处理，不直接对接请求。

- **误区2**：拦截器postHandle在视图渲染后执行。**真相**：postHandle在业务方法执行完、视图渲染**之前**执行，afterCompletion才是视图渲染完成后执行。

- **误区3**：参数绑定在Controller方法内执行。**真相**：参数绑定、类型转换由HandlerAdapter在调用Controller方法前自动完成。

### 6.1.5 面试高频真题

**Q1：简述SpringMVC完整执行流程，核心组件有哪些？**

**参考答案**：核心流程为请求被DispatcherServlet拦截→HandlerMapping匹配处理器→HandlerAdapter适配执行→拦截器前置处理→执行Controller业务方法→拦截器后置处理→ViewResolver解析视图→视图渲染→最终拦截处理并返回响应。核心四大组件：DispatcherServlet（核心调度）、HandlerMapping（路径匹配）、HandlerAdapter（方法适配）、ViewResolver（视图解析）。

**Q2：preHandle、postHandle、afterCompletion的执行顺序和区别？**

**参考答案**：执行顺序：preHandle → 业务方法 → postHandle → 视图渲染 → afterCompletion。preHandle可拦截请求、终止流程；postHandle可修改模型数据和视图；afterCompletion用于资源释放、异常日志记录，全局执行。

## 6.2 DispatcherServlet 初始化流程

### 6.2.1 核心概念定义

DispatcherServlet是SpringMVC的**核心入口、前端控制器**，继承自原生HttpServlet，是整个Web请求的统一调度中枢。所有HTTP请求都必须经过它分发处理，同时它负责SpringMVC所有核心组件的初始化、加载、注册工作，是SpringMVC框架的基石。

区别于普通Bean，DispatcherServlet是**Web容器级组件**，由Tomcat等Web容器创建和管理，而非Spring容器直接创建，其初始化时机早于普通业务Bean。

### 6.2.2 底层初始化核心流程

DispatcherServlet的初始化分为**Web容器初始化、Spring上下文初始化、MVC核心组件初始化**三大阶段，核心执行步骤如下：

**阶段1：Web容器加载触发初始化**：Tomcat启动时，读取web.xml配置或SpringBoot自动配置，检测到DispatcherServlet注册信息，调用其 `init()` 方法，启动初始化流程。

**阶段2：创建Web专属Spring上下文**：DispatcherServlet会创建一个**独立的WebApplicationContext（Web Spring容器）**，该容器继承父容器（Root容器）的所有Bean，同时加载Web专属的Bean（Controller、拦截器、视图解析器等），实现Web Bean与业务Bean的隔离。

**阶段3：加载SpringMVC核心配置**：读取mvc配置文件、注解配置、自动配置类，初始化MVC环境，开启注解驱动、静态资源映射、消息转换器等基础能力。

**阶段4：初始化九大核心MVC组件**（核心考点）：DispatcherServlet会自动探测、初始化SpringMVC的所有内置核心组件，若无自定义实现，则使用框架默认实现：

- 处理器映射器（HandlerMapping）

- 处理器适配器（HandlerAdapter）

- 视图解析器（ViewResolver）

- 全局异常处理器（HandlerExceptionResolver）

- 文件上传解析器（MultipartResolver）

- 本地化解析器（LocaleResolver）

- 主题解析器（ThemeResolver）

- 请求头解析器（RequestToViewNameTranslator）

- Flash映射管理器（FlashMapManager）

**阶段5：完成初始化，就绪待命**：所有组件初始化完成后，DispatcherServlet注册到Web容器，开始监听HTTP请求，等待客户端请求接入。

### 6.2.3 SpringBoot环境自动初始化机制

传统SSM项目需要手动在web.xml中配置DispatcherServlet，而SpringBoot通过**自动配置类DispatcherServletAutoConfiguration**实现自动初始化：

框架自动创建DispatcherServlet实例、设置默认路径为`/`、注册到Servlet容器，同时自动装配所有MVC核心组件，无需开发者手动配置，大幅简化开发。

### 6.2.4 生产场景价值

理解初始化流程，可解决生产中常见的**启动报错、组件不生效、Bean加载异常**问题。例如：自定义的拦截器、视图解析器不生效，大概率是DispatcherServlet初始化优先级问题；接口404异常，多是核心映射器初始化失败。

### 6.2.5 避坑指南

- **易错点1**：混淆Spring父容器与Web子容器。**避坑**：Root父容器加载Service、Repository等业务Bean，Web子容器加载Controller、Web组件，子容器可访问父容器Bean，父容器无法访问子容器Bean，不可跨容器依赖。

- **易错点2**：自定义MVC组件不生效。**避坑**：自定义组件需交给Spring容器管理（添加@Bean/@Component），否则DispatcherServlet不会加载自定义实现，仍使用默认组件。

- **易错点3**：DispatcherServlet路径配置错误。**避坑**：配置`/*`会拦截所有请求（包含静态资源），配置`/`仅拦截业务请求，放行静态资源，生产统一使用`/`。

### 6.2.6 面试高频真题

**Q1：DispatcherServlet的初始化时机和核心作用？**

**参考答案**：初始化时机：Web容器（Tomcat）启动阶段，早于请求处理、晚于Spring父容器初始化。核心作用：作为SpringMVC唯一入口，统一拦截所有HTTP请求，调度所有MVC核心组件，完成请求全流程处理，是整个MVC框架的调度中枢。

**Q2：SpringMVC九大核心组件有哪些？必须掌握的核心组件是哪几个？**

**参考答案**：九大组件包含映射器、适配器、视图解析器、异常处理器等。面试核心重点：HandlerMapping、HandlerAdapter、ViewResolver、HandlerExceptionResolver，这四个是日常开发和面试最高频组件。

## 6.3 处理器映射器、适配器、视图解析器面试

### 6.3.1 三大组件核心定义与分工

处理器映射器、处理器适配器、视图解析器是SpringMVC**三大核心支柱组件**，三者各司其职、层层配合，完成请求匹配、方法执行、响应渲染的核心能力，是面试深挖的重点。

**1. 处理器映射器（HandlerMapping）**：核心职责是**根据请求URL匹配处理器**。简单来说就是「找方法」，接收请求后，遍历容器中所有Controller的请求路径，匹配出对应的处理方法，同时封装拦截器链，返回执行链路。

**2. 处理器适配器（HandlerAdapter）**：核心职责是**适配并执行处理器方法**。简单来说就是「跑方法」，由于SpringMVC支持多种处理器类型，适配器统一适配不同处理器的执行规则，同时完成参数绑定、返回值处理，是真正调用业务方法的组件。

**3. 视图解析器（ViewResolver）**：核心职责是**解析视图、渲染数据**。简单来说就是「拼页面」，将Controller返回的逻辑视图名，解析为真实的视图资源，并且把Model中的数据填充到视图中，生成最终响应。

### 6.3.2 核心底层原理

#### 6.3.2.1 HandlerMapping 底层原理

SpringMVC默认提供多个映射器，主流为**RequestMappingHandlerMapping**，专门处理@RequestMapping注解的接口。其底层会在容器初始化时，扫描所有带@RestController/@Controller的类，解析所有@RequestMapping、@GetMapping、@PostMapping注解，将「URL路径-方法映射关系」缓存到Map集合中。请求到来时，直接根据URL匹配缓存映射关系，快速定位目标方法，保证请求处理效率。

#### 6.3.2.2 HandlerAdapter 底层原理

默认核心适配器为**RequestMappingHandlerAdapter**，专门适配注解式Controller方法。其核心能力包含三大块：一是参数解析（解析请求体、请求参数、路径变量）；二是类型转换与格式化；三是返回值解析（将方法返回值转为JSON、视图对象等）。所有Controller方法的调用、参数封装、返回值处理，全部由该适配器完成。

#### 6.3.2.3 ViewResolver 底层原理

传统SpringMVC默认使用**InternalResourceViewResolver**，通过配置前缀、后缀拼接物理视图路径。例如前缀`/WEB-INF/pages/`、后缀`.jsp`，Controller返回`index`，即可解析为`/WEB-INF/pages/index.jsp`。SpringBoot前后端分离项目中，默认使用**ContentNegotiatingViewResolver**，支持自适应JSON、页面等多种响应格式。

### 6.3.3 三大组件协作流程

客户端请求→DispatcherServlet→HandlerMapping匹配目标方法→返回执行链→HandlerAdapter接收执行链→解析参数、执行Controller方法→返回ModelAndView→ViewResolver解析视图、填充数据→渲染响应，三者形成完整闭环，缺一不可。

### 6.3.4 实操配置示例

SpringBoot自定义视图解析器配置（生产常用）：

```java
/**
 * 自定义SpringMVC视图解析器配置
 */
@Configuration
public class WebViewConfig implements WebMvcConfigurer {

    /**
     * 配置内部资源视图解析器
     * 用于传统页面跳转场景
     */
    @Bean
    public ViewResolver internalResourceViewResolver() {
        InternalResourceViewResolver resolver = new InternalResourceViewResolver();
        // 视图文件前缀：页面存放路径
        resolver.setPrefix("/WEB-INF/html/");
        // 视图文件后缀：文件格式
        resolver.setSuffix(".html");
        // 设置优先级，高于默认解析器
        resolver.setOrder(Ordered.HIGHEST_PRECEDENCE);
        return resolver;
    }
}

```

### 6.3.5 易错点与避坑指南

- **易错点1**：接口404，认为是路径写错。**根因**：RequestMappingHandlerMapping初始化失败，URL映射关系未缓存。**避坑**：检查Controller是否添加注解、是否被Spring扫描到。

- **易错点2**：参数接收不到、报错类型不匹配。**根因**：HandlerAdapter参数解析器失效。**避坑**：避免自定义Web配置覆盖默认适配器，如需扩展需保留默认配置。

- **易错点3**：视图跳转失效。**根因**：多视图解析器优先级冲突。**避坑**：自定义解析器手动设置优先级，保证自定义规则优先执行。

### 6.3.6 面试高频真题

**Q1：HandlerMapping和HandlerAdapter的区别与联系？**

**参考答案**：区别：HandlerMapping负责**路径匹配、查找目标方法**，不执行业务逻辑；HandlerAdapter负责**适配、调用目标方法**，处理参数和返回值。联系：二者配套使用，Mapping找到方法后，由Adapter执行方法，是前后衔接的两个核心组件。

**Q2：SpringBoot前后端分离项目，视图解析器还需要吗？**

**参考答案**：不需要手动配置。前后端分离项目返回JSON数据，无需页面渲染，框架默认的内容协商视图解析器会自动适配JSON响应，无需传统视图解析器，可直接废弃视图跳转逻辑。

## 6.4 参数绑定、类型转换器、格式化易错点

### 6.4.1 核心概念定义

**参数绑定**：SpringMVC自动将HTTP请求中的参数（URL参数、表单参数、JSON请求体），封装绑定到Controller方法的形参或实体类属性中的过程，是接口开发最基础的能力。

**类型转换器**：负责**不同数据类型的自动转换**，例如String转Integer、String转Date、自定义字符串转枚举、字符串转自定义对象，解决请求参数类型与Java类型不匹配的问题。

**数据格式化**：是类型转换的细分场景，专门针对**日期、数字、货币**等格式进行规范化转换，例如将字符串`2026-05-09`格式化为Date对象，将`1000`格式化为保留两位小数的数字。

### 6.4.2 核心执行原理

参数绑定的完整执行链路：客户端传递参数 → HandlerAdapter接收参数 → 参数解析器解析参数 → 类型转换器统一转换类型 → 格式化器规范化数据 → 绑定到方法形参 → 完成业务调用。

SpringMVC内置**ConversionService**转换服务核心接口，统一管理所有转换器（Converter）和格式化器（Formatter）。Converter负责跨类型转换（如String→Date），Formatter负责同类型格式化（如日期格式化、数字精度处理），二者配合完成参数预处理。

### 6.4.3 常见参数绑定场景与规则

- **普通参数绑定**：URL拼接参数、表单参数，直接绑定到基本数据类型形参，参数名与形参名一致即可自动绑定。

- **实体类参数绑定**：请求参数名与实体类属性名一致，自动封装为实体对象，支持层级嵌套对象绑定。

- **JSON参数绑定**：请求体为JSON格式，通过@RequestBody注解，由Jackson序列化工具解析为Java对象。

- **路径变量绑定**：通过@PathVariable绑定URL路径中的动态参数。

### 6.4.4 自定义转换器实操示例

生产中默认转换器无法满足自定义场景（如自定义日期格式、字符串转枚举），需要自定义转换器，以下是通用自定义日期转换器实现：

```java
/**
 * 自定义String转Date类型转换器
 * 解决默认日期格式单一、自定义格式无法解析问题
 */
@Component
public class StringToDateConverter implements Converter<String, Date> {

    // 支持多种常用日期格式
    private static final List<String> DATE_PATTERNS = Arrays.asList(
            "yyyy-MM-dd",
            "yyyy/MM/dd",
            "yyyy-MM-dd HH:mm:ss"
    );

    @Override
    public Date convert(String source) {
        // 空参数直接返回null
        if (StringUtils.isEmpty(source)) {
            return null;
        }
        // 遍历所有日期格式，匹配转换
        for (String pattern : DATE_PATTERNS) {
            try {
                SimpleDateFormat sdf = new SimpleDateFormat(pattern);
                sdf.setLenient(false);
                return sdf.parse(source);
            } catch (ParseException e) {
                continue;
            }
        }
        throw new IllegalArgumentException("日期格式错误，支持格式：yyyy-MM-dd、yyyy/MM/dd、yyyy-MM-dd HH:mm:ss");
    }
}

```

注册自定义转换器到Spring容器：

```java
@Configuration
public class ConvertConfig implements WebMvcConfigurer {

    @Autowired
    private StringToDateConverter stringToDateConverter;

    @Override
    public void addFormatters(FormatterRegistry registry) {
        // 注册自定义转换器
        registry.addConverter(stringToDateConverter);
    }
}

```

### 6.4.5 高频易错点汇总（生产90%报错来源）

- **易错点1：日期转换报错**。默认转换器仅支持固定日期格式，前端传自定义格式日期会报类型转换异常。**解决方案**：自定义日期转换器或使用@DateTimeFormat注解指定格式。

- **易错点2：@RequestBody与普通参数混用**。JSON请求体必须用@RequestBody接收，普通URL参数不能用该注解，混用会导致参数绑定失败。

- **易错点3：基本类型参数空值报错**：使用int、long等基本类型接收参数，前端不传参数会报空指针、类型不匹配。**避坑**：统一使用包装类型Integer、Long。

- **易错点4：参数名大小写不匹配**：前端参数名与实体类属性名大小写不一致，导致绑定为空。**避坑**：统一前后端命名规范，或通过@JsonProperty指定映射名称。

- **易错点5：转换器注册失效**：自定义转换器未注册到FormatterRegistry，导致自定义规则不生效。

### 6.4.6 面试高频真题

**Q1：Converter和Formatter的区别？**

**参考答案**：1. 作用不同：Converter用于**跨类型转换**（如String→Date、String→Integer）；Formatter用于**同类型格式化**（日期格式化、数字精度格式化）。2. 场景不同：Converter侧重类型转换，Formatter侧重数据规范化展示。3. 优先级：自定义Converter优先级高于默认Formatter。

**Q2：@DateTimeFormat和@JsonFormat的区别？**

**参考答案**：@DateTimeFormat用于**普通参数、表单参数**的日期转换；@JsonFormat用于**JSON请求体参数**的日期序列化与反序列化，前后端分离项目优先使用@JsonFormat。

## 6.5 全局异常处理器面试与实战易错

### 6.5.1 核心概念定义

全局异常处理器是SpringMVC提供的**统一异常处理机制**，核心作用是统一捕获程序运行中的所有异常（业务异常、参数异常、系统异常、未知异常），统一封装返回结果，避免前端接收原生报错堆栈信息，提升接口规范性和用户体验。

在没有全局异常处理时，接口报错会直接返回冗长的Java异常堆栈，暴露服务器底层信息，存在安全风险，且前端无法统一解析错误信息。全局异常处理器彻底解决该问题，是生产项目**必备组件**。

### 6.5.2 底层核心原理

SpringMVC的异常处理核心接口为**HandlerExceptionResolver**，DispatcherServlet在执行流程中，若Controller业务方法抛出异常，会终止后续流程，回调全局异常处理器的解析方法，匹配对应异常处理规则，返回统一响应结果。

SpringBoot主流实现方式为**@RestControllerAdvice + @ExceptionHandler**组合注解：

- @RestControllerAdvice：标识全局异常增强类，拦截所有@RestController接口的异常

- @ExceptionHandler：指定拦截的异常类型，精准匹配处理不同异常

执行优先级：精准异常处理 > 父类异常处理 > 全局未知异常兜底。

### 6.5.3 生产级完整实操示例

包含业务异常、参数校验异常、空指针异常、未知异常的全局统一处理，可直接用于生产：

```java
/**
 * 全局统一异常处理器
 * 统一处理所有接口异常，规范返回结果
 */
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    /**
     * 1. 自定义业务异常处理（精准匹配）
     */
    @ExceptionHandler(BusinessException.class)
    public ResultVO<Void> businessExceptionHandler(BusinessException e) {
        // 打印业务异常日志
        log.error("业务异常：{}", e.getMessage());
        // 返回自定义错误码和错误信息
        return ResultVO.error(e.getCode(), e.getMessage());
    }

    /**
     * 2. 参数校验异常处理
     */
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResultVO<Void> validExceptionHandler(MethodArgumentNotValidException e) {
        BindingResult bindingResult = e.getBindingResult();
        String message = bindingResult.getFieldError().getDefaultMessage();
        log.error("参数校验失败：{}", message);
        return ResultVO.error(400, "参数错误：" + message);
    }

    /**
     * 3. 空指针异常处理
     */
    @ExceptionHandler(NullPointerException.class)
    public ResultVO<Void> nullPointExceptionHandler() {
        log.error("系统空指针异常");
        return ResultVO.error(500, "系统数据异常，请稍后重试");
    }

    /**
     * 4. 全局兜底异常（处理所有未知异常）
     */
    @ExceptionHandler(Exception.class)
    public ResultVO<Void> globalExceptionHandler(Exception e) {
        log.error("系统未知异常：", e);
        return ResultVO.error(500, "服务器繁忙，请稍后重试");
    }
}

```

### 6.5.4 生产易错点与避坑指南

- **易错点1：异常兜底失效**：未配置Exception全局兜底，部分未知异常直接抛出堆栈信息。**避坑**：必须添加Exception通用处理方法，作为最后兜底。

- **易错点2：异常日志打印不规范**：只打印异常信息，不打印堆栈，线上无法定位报错位置。**避坑**：未知异常必须打印完整异常堆栈。

- **易错点3：异常优先级混乱**：父类异常写在子类异常前面，导致精准异常处理失效。**避坑**：精准异常在前，通用异常在后。

- **易错点4：拦截静态资源异常**：全局异常拦截范围过大，拦截静态资源报错。**避坑**：通过注解指定扫描包范围，仅拦截业务接口。

- **易错点5：事务失效**：全局异常捕获后，事务无法回滚。**避坑**：如需事务回滚，需手动抛出异常或手动回滚事务。

### 6.5.5 面试高频真题

**Q1：@ControllerAdvice的底层原理和适用场景？**

**参考答案**：底层基于Spring AOP动态代理实现，对所有Controller接口进行增强，拦截执行过程中抛出的异常。适用场景：统一异常处理、统一数据返回、全局参数预处理、接口日志统一打印。

**Q2：全局异常捕获后，Spring事务会回滚吗？**

**参考答案**：**不会自动回滚**。Spring事务回滚的条件是方法抛出未被捕获的异常，全局异常处理器捕获异常后，异常不会向上抛出，事务判定为正常执行，不会回滚。需要手动开启事务回滚。

## 6.6 拦截器与过滤器区别面试深挖

### 6.6.1 核心概念定义

**过滤器（Filter）**：是**Servlet规范原生组件**，不属于Spring框架，运行在Web容器层面（Tomcat）。主要用于对所有Web请求进行过滤拦截，是请求进入Spring容器**之前**的第一道关卡。

**拦截器（Interceptor）**：是**SpringMVC框架专属组件**，运行在Spring容器层面。主要用于对Controller接口请求进行拦截处理，是请求进入DispatcherServlet**之后**、执行Controller方法**之前**的处理组件。

### 6.6.2 底层执行流程与顺序

完整请求执行顺序（核心面试考点）：**客户端请求 → Filter过滤器 → DispatcherServlet → Interceptor拦截器 → Controller业务方法 → 拦截器后置处理 → 过滤器后置处理 → 响应返回**

过滤器是Web容器级别的拦截，拦截所有请求（静态资源、动态接口、错误请求）；拦截器是SpringMVC级别的拦截，**只拦截Controller接口请求**，不拦截静态资源。

### 6.6.3 核心区别对比（面试必背表格）

| 对比维度 | 过滤器（Filter）                | 拦截器（Interceptor）                    |
| -------- | ------------------------------- | ---------------------------------------- |
| 所属规范 | Servlet原生规范，和Spring无关   | SpringMVC框架专属机制                    |
| 运行层级 | Web容器层级（Tomcat）           | Spring容器层级                           |
| 拦截范围 | 拦截所有请求（接口、静态资源）  | 仅拦截Controller接口请求                 |
| 执行时机 | SpringMVC流程执行前、响应后     | DispatcherServlet之后，Controller前后    |
| 依赖注入 | 无法使用Spring DI，不能注入Bean | 属于Spring组件，可直接注入Bean           |
| 功能能力 | 全局编码、跨域、请求过滤、限流  | 权限校验、登录拦截、日志记录、数据预处理 |

### 6.6.4 适用场景分工（生产最佳实践）

#### 过滤器适用场景

处理**全局通用、与业务无关**的Web底层逻辑：统一编码设置、跨域处理、请求限流、黑名单拦截、静态资源过滤、请求参数全局预处理。

#### 拦截器适用场景

处理**业务相关、接口级别**的逻辑：用户登录状态校验、Token校验、接口权限拦截、操作日志记录、接口参数校验、业务数据预处理。

### 6.6.5 高频易错点

- **易错点1**：在Filter中注入Spring Bean，出现空指针。**根因**：Filter由Tomcat创建，早于Spring容器初始化，无法依赖注入。**避坑**：业务相关逻辑放拦截器，过滤器只做底层处理。

- **易错点2**：拦截器拦截静态资源导致页面无法访问。**避坑**：配置拦截器放行静态资源路径（/static/**、/css/**）。

- **易错点3**：执行顺序混淆，导致权限拦截失效。**避坑**：过滤器优先执行，拦截器后置执行，核心权限校验建议放在拦截器中。

### 6.6.6 面试高频真题

**Q1：为什么拦截器可以注入Bean，过滤器不行？**

**参考答案**：过滤器是Servlet规范组件，由Tomcat容器实例化和管理，初始化时机早于Spring容器，无法感知Spring的Bean工厂，因此不能依赖注入。拦截器是SpringMVC组件，由Spring容器管理，属于Spring Bean，可直接使用@Autowired注入业务Bean。

**Q2：跨域配置用过滤器还是拦截器？为什么？**

**参考答案**：优先用过滤器。跨域属于Web全局底层配置，需要在请求进入SpringMVC流程前处理，过滤器执行时机更早，可拦截所有请求完成跨域响应头设置；拦截器执行时机靠后，部分场景无法处理预检请求，跨域配置不彻底。

# 7.Spring 注解核心面试题 & 易错点

## 7.1 核心注解 @Component/@Bean/@Configuration 区别

### 7.1.1 核心概念定义

这三个注解都是Spring中用于**将对象交由Spring容器管理**的核心注解，但适用场景、作用对象、底层逻辑完全不同，是面试必问、开发必懂的基础核心知识点。

- **@Component**：通用组件注解，作用于**类**上，是Spring最基础的Bean注册注解。包含衍生注解@Controller、@Service、@Repository，分别对应控制层、业务层、数据访问层，本质都是@Component的派生注解。

- **@Bean**：方法级别的Bean注册注解，作用于**方法**上，专门用于将方法返回值注册为Spring Bean，多用于第三方类、无法添加@Component注解的类的注册场景。

- **@Configuration**：配置类注解，作用于**类**上，用于标记当前类是Spring配置类，核心作用是承载@Bean方法，同时保证配置类内部的Bean创建**单例、无重复**。

### 7.1.2 核心底层原理

1. @Component原理：Spring启动时通过**包扫描机制**，扫描被该注解标记的类，自动通过反射创建类实例，纳入IOC容器管理，Bean的创建权限由Spring全权掌控。

2. @Bean原理：开发者手动定义方法、手动指定对象创建逻辑，Spring仅负责**接管方法返回的对象**，注册到容器中。对象的创建逻辑由开发者控制，Spring只做托管。

3. @Configuration原理：Spring 3.0+推出的配置注解，核心底层是**Full模式配置类**。被该注解标记的类会被Spring动态生成CGLIB代理类，拦截类中所有@Bean方法的调用，确保无论方法被调用多少次，都只会返回容器中的单例Bean，避免多实例重复创建问题。

💡 核心区别关键点：**不加@Configuration的配置类是Lite模式，加了是Full模式**，这是三者最核心的底层差异。

### 7.1.3 适用场景与生产价值

| 注解           | 适用场景                                                     | 生产核心价值                                                 |
| -------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| @Component     | 自定义业务类、通用工具类，开发者可修改源码的类               | 简化开发，自动扫描注册Bean，无需手动创建对象，适配业务层常规开发 |
| @Bean          | 第三方框架类、原生JDK类、无法修改源码的类（如RedisTemplate、RestTemplate） | 实现外部资源的Bean托管，灵活配置对象参数，自定义对象创建逻辑 |
| @Configuration | 集中配置多个@Bean、全局资源配置、框架整合配置                | 保证Bean单例、统一管理配置、解决Lite模式多实例bug，规范项目配置结构 |

### 7.1.4 实操代码示例

```java
// 1. @Component 使用示例（自定义业务组件）
@Component
public class UserUtil {
    // 自定义工具类，Spring自动扫描创建Bean
    public String getUserName() {
        return "test-user";
    }
}

// 2. @Configuration + @Bean 标准用法（生产推荐）
@Configuration // 标记为Full模式配置类，开启CGLIB代理
public class WebConfig {

    /**
     * 注册RestTemplate第三方Bean
     * 开发者自定义创建逻辑，Spring托管返回对象
     */
    @Bean
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }

    /**
     * 测试配置类单例特性
     */
    @Bean
    public UserUtil userUtil() {
        // 调用上方Bean方法，Full模式下不会重复创建
        return new UserUtil();
    }
}

```

### 7.1.5 ⚠️ 避坑指南（高频易错）

**坑1：配置类不加@Configuration，只加@Component**

后果：配置类变为Lite模式，无CGLIB代理。类中@Bean方法相互调用时，会**重复创建多个Bean实例**，破坏单例特性，导致资源浪费、事务失效、配置不一致等问题。

**坑2：滥用@Bean注册自定义类**

自定义业务类可直接用@Component，无需用@Bean，过度使用会导致代码冗余、配置类臃肿，违背Spring自动化开发思想。

**坑3：混淆注解作用对象**

@Component、@Configuration只能作用于类；@Bean只能作用于方法，混用会直接启动报错。

### 7.1.6 📌 面试高频题

**问题1：@Component和@Bean的核心区别是什么？**

参考答案：1. 作用对象不同：@Component作用于类，@Bean作用于方法；2. 创建逻辑不同：@Component由Spring自动创建实例，@Bean由开发者手动定义创建逻辑；3. 适用场景不同：@Component用于自定义类，@Bean用于第三方类；4. 扫描方式不同：@Component依赖包扫描，@Bean依赖配置类加载。

**问题2：@Configuration的Full模式和Lite模式区别？**

参考答案：Full模式（加@Configuration）：生成CGLIB代理，拦截@Bean方法调用，保证单例；Lite模式（不加@Configuration，仅@Component）：无代理，方法调用会重复创建实例，不保证单例。生产环境配置类必须使用Full模式。

## 7.2 @Autowired、@Resource、@Inject 对比面试

### 7.2.1 核心概念定义

三者均为Spring中**依赖注入注解**，用于自动装配Bean对象，消除new创建对象的硬编码，但来源、装配规则、特性差异极大，是面试最高频的对比考题，也是开发中最容易注入失败的易错点。

- **@Autowired**：Spring原生注解，核心特性是**默认按类型装配**，是项目中最常用的注入注解。

- **@Resource**：JSR-250官方规范注解（Java原生），核心特性是**默认按名称装配**，找不到名称再按类型装配。

- **@Inject**：JSR-330官方规范注解，功能和@Autowired几乎一致，默认按类型装配，极少在国内项目使用。

### 7.2.2 核心原理与装配规则

**1. @Autowired 装配流程**

第一步：根据**目标类型**去IOC容器中匹配Bean；第二步：如果容器中存在多个同类型Bean，抛出NoUniqueBeanDefinitionException唯一Bean冲突异常；第三步：可配合@Qualifier指定Bean名称，精准匹配注入；第四步：默认要求Bean必须存在，可设置required=false允许为空。

**2. @Resource 装配流程**

第一步：优先根据**name属性/变量名**匹配容器中的Bean名称；第二步：名称匹配失败后，降级为按类型匹配；第三步：无required属性，默认必须注入成功，否则报错。

**3. @Inject 装配流程**

完全对标@Autowired，默认按类型装配，不支持required属性，必须配合@Named注解实现名称匹配，无特殊优势，Spring项目基本不用。

### 7.2.3 核心差异对比表

| 对比维度     | @Autowired             | @Resource                    | @Inject            |
| ------------ | ---------------------- | ---------------------------- | ------------------ |
| 来源         | Spring原生             | JSR-250 Java标准             | JSR-330 Java标准   |
| 默认装配规则 | 按类型（byType）       | 按名称（byName），失败按类型 | 按类型（byType）   |
| 核心属性     | required、@Qualifier   | name                         | 无，配合@Named使用 |
| 是否允许空值 | 支持（required=false） | 不支持                       | 不支持             |
| 项目使用率   | 95%（主流）            | 少量老项目                   | 极少使用           |

### 7.2.4 实操示例与冲突解决

```java
// 定义接口与两个实现类（制造多Bean冲突场景）
public interface UserService {}

@Service
public class UserServiceImpl implements UserService {}

@Service
public class UserServiceTestImpl implements UserService {}

// 1. @Autowired 多类型冲突报错
@Component
public class UserController {
    // 报错：容器中有两个UserService类型Bean，无法唯一匹配
    @Autowired
    private UserService userService;

    // 正确写法：配合@Qualifier指定Bean名称
    @Autowired
    @Qualifier("userServiceImpl")
    private UserService userService2;
}

// 2. @Resource 按名称自动匹配（无报错）
@Component
public class UserController2 {
    // 优先根据变量名userServiceImpl匹配Bean，直接注入成功
    @Resource
    private UserService userServiceImpl;
}

```

### 7.2.5 ⚠️ 高频易错点

**坑1：@Autowired多实现类不指定名称，直接启动报错**

原因：byType装配遇到多个同类型Bean，无法区分。解决方案：添加@Qualifier指定Bean名称、给实现类添加@Primary优先注入、使用@Resource按名称注入。

**坑2：误以为@Resource也是默认按类型注入**

新手极易混淆，导致注入失败，变量名和Bean名不一致时，@Resource不会自动降级类型匹配，或匹配失败报错。

**坑3：@Autowired(required=false)滥用**

过度设置required=false会掩盖Bean注册失败的问题，导致运行时空指针异常，生产中仅用于可选依赖。

### 7.2.6 📌 面试真题解析

**问题1：@Autowired和@Resource的区别？工作中怎么选择？**

参考答案：核心区别为装配规则、来源、属性支持不同。选择原则：日常开发优先用@Autowired，灵活性更高；存在多实现类且不想用@Qualifier时，用@Resource按变量名注入；追求Java标准、无框架依赖时可选@Resource。

**问题2：@Autowired如何解决Bean多实例冲突问题？**

参考答案：三种方案：1. 使用@Qualifier指定具体Bean名称；2. 在优先使用的实现类上添加@Primary；3. 使用@Resource注解按名称精准注入。

## 7.3 @Conditional 条件注解面试考点

### 7.3.1 概念定义

@Conditional是Spring4.0推出的**条件装配核心注解**，核心作用是：**满足指定条件时，才将Bean注册到Spring容器，不满足则不注册**。简单来说，就是实现Bean的「按需加载、条件化创建」。

SpringBoot中的自动配置核心底层就是@Conditional系列注解，所有的自动配置类、默认Bean的加载，全部依赖该注解实现条件装配，是SpringBoot自动化的核心基石。

### 7.3.2 核心原理

1. 底层核心接口：**Condition接口**，包含唯一方法matches()，返回布尔值。

2. 执行流程：Spring容器扫描Bean时，检测类/方法上的@Conditional注解 → 加载对应的Condition实现类 → 执行matches()方法判断条件是否成立 → 成立则注册Bean，不成立则跳过。

3. 执行时机：**Bean定义加载阶段**（早于Bean实例化阶段），仅控制Bean是否注册，不影响已注册Bean的使用。

4. 衍生注解：Spring基于@Conditional封装了大量常用条件注解，简化开发，如@ConditionalOnClass、@ConditionalOnMissingBean、@ConditionalOnProperty等。

### 7.3.3 常用衍生注解与生产场景

| 条件注解                     | 生效条件                     | 生产应用场景                                                 |
| ---------------------------- | ---------------------------- | ------------------------------------------------------------ |
| @ConditionalOnClass          | classpath下存在指定类        | 依赖jar包存在时才加载对应配置，如引入redis依赖才加载Redis配置 |
| @ConditionalOnMissingBean    | 容器中无指定Bean             | 实现默认兜底配置，用户自定义Bean优先，无自定义则使用系统默认Bean |
| @ConditionalOnProperty       | 配置文件存在指定配置且匹配值 | 通过yml/properties配置开关，控制功能是否启用                 |
| @ConditionalOnWebApplication | 当前是Web环境                | 仅在web项目中加载web相关配置，非web项目不加载                |

### 7.3.4 实操代码示例（自定义条件+内置条件）

```java
// 1. 自定义条件：判断当前系统是否为Windows
public class WindowsCondition implements Condition {
    @Override
    public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
        // 获取系统环境变量
        String osName = context.getEnvironment().getProperty("os.name");
        // 判断系统名称是否包含Windows
        return osName.toLowerCase().contains("windows");
    }
}

// 2. 基于自定义条件装配Bean
@Configuration
public class ConditionalConfig {

    // 仅Windows系统注册该Bean
    @Bean
    @Conditional(WindowsCondition.class)
    public String windowsOsBean() {
        return "Windows系统专属Bean";
    }

    // 容器中无RestTemplate时，才注册默认Bean（SpringBoot经典兜底写法）
    @Bean
    @ConditionalOnMissingBean
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }

    // 配置文件开启custom.config.enable=true时才注册
    @Bean
    @ConditionalOnProperty(prefix = "custom.config", name = "enable", havingValue = "true")
    public String customConfigBean() {
        return "自定义配置开启Bean";
    }
}

```

### 7.3.5 ⚠️ 避坑指南

**坑1：混淆@Conditional生效阶段**

条件判断仅在Bean定义阶段生效，无法控制Bean实例化后的逻辑，不要用于业务动态逻辑判断，仅用于容器Bean装配控制。

**坑2：@ConditionalOnMissingBean使用误区**

该注解基于**Bean类型**判断，不是Bean名称，同类型任意名称Bean存在，就不会加载默认Bean。

**坑3：配置条件注解不生效**

原因：配置类未被扫描、条件类未加载、配置文件前缀/名称写错，是生产中自动配置失效的常见原因。

### 7.3.6 📌 面试核心考点

**问题1：@Conditional注解的作用和底层原理？SpringBoot自动配置如何依赖它？**

参考答案：@Conditional用于条件化注册Bean，底层通过Condition接口的matches方法判断条件。SpringBoot自动配置核心依托各类Conditional衍生注解，实现「有依赖则加载、无自定义则用默认、环境匹配则生效」的自动化配置逻辑，避免手动配置大量Bean。

**问题2：@ConditionalOnMissingBean的优先级是什么？**

参考答案：用户自定义Bean优先级高于框架默认Bean。当用户手动注册同类型Bean后，@ConditionalOnMissingBean修饰的默认Bean不会加载，实现自定义覆盖默认配置的效果。

## 7.4 常用注解使用误区与易错写法

### 7.4.1 类注解常见误区

**误区1：@Controller/@Service/@Repository混用**

很多开发者认为四个注解完全通用，随意替换。实际生产规范：@Controller仅用于控制层、接收请求；@Service仅用于业务逻辑层；@Repository用于数据持久层；@Component用于通用工具类、非分层组件。

后果：混用会导致项目分层混乱，AOP切面、事务注解、异常处理等基于分层的功能失效。

**误区2：配置类只写@Bean，不加@Configuration**

前文核心易错点，Lite模式下Bean方法相互调用会创建多实例，导致事务失效、连接池重复创建、资源占用过高。生产中**所有包含@Bean的类必须加@Configuration**。

**误区3：注解标记的类，未被包扫描覆盖**

自定义组件放在启动类同级目录以外，未配置scanBasePackages，导致注解失效、Bean无法注入，出现空指针异常。

### 7.4.2 注入注解高频易错写法

**易错1：字段注入滥用（@Autowired直接注入字段）**

新手常用写法：直接在私有字段上加@Autowired注入。看似简洁，实则存在严重问题：循环依赖无法检测、单元测试难以构造、违反单一职责、对象耦合严重。

✅ 最佳实践：**统一使用构造器注入**，Spring4.3+单构造器可省略@Autowired，自动注入。

```java
// 错误写法：字段注入
@Service
public class UserService {
    @Autowired
    private UserMapper userMapper;
}

// 正确写法：构造器注入
@Service
public class UserService {
    private final UserMapper userMapper;

    // 单构造器自动注入，无需注解
    public UserService(UserMapper userMapper) {
        this.userMapper = userMapper;
    }
}

```

**易错2：静态变量无法被注解注入**

大量新手误区：给静态字段加@Autowired，期望注入Bean。Spring仅能注入**实例变量**，静态变量属于类，不属于Bean实例，永远无法注入成功，始终为null。

✅ 解决方案：通过非静态setter方法注入，或使用工具类静态方法调用实例Bean。

### 7.4.3 事务注解@Transactional 致命误区

这是生产环境**最高频、最隐蔽、导致数据事故最多**的注解易错点。

**误区1：非public方法加@Transactional**

Spring事务基于AOP动态代理，仅拦截public方法，private/protected/default方法事务完全失效，无报错但事务不生效。

**误区2：类内部方法调用，事务失效**

AOP代理仅拦截外部调用，类内部this调用带事务的方法，不会经过代理，事务完全失效。

**误区3：捕获异常后事务不回滚**

@Transactional默认仅捕获RuntimeException和Error回滚，捕获Exception、自定义检查异常不会回滚，导致数据不一致。

### 7.4.4 综合避坑最佳实践

- 所有分层注解严格对应层级，不混用、不滥用@Component替代分层注解

- 统一使用构造器注入，杜绝字段注入，规避循环依赖和测试问题

- 事务方法必须为public，内部调用需通过代理对象调用，指定回滚异常

- 配置类必须添加@Configuration，保证Bean单例特性

- 静态资源禁止使用依赖注入，通过工具类特殊处理

### 7.4.5 📌 面试真题

**问题1：为什么不推荐字段注入？构造器注入的优势？**

参考答案：字段注入缺点：隐藏依赖、循环依赖无法检测、单元测试困难、耦合度高。构造器注入优势：强制依赖不可为空、启动即校验依赖完整性、方便单元测试、符合Spring设计规范、避免循环依赖隐患。

**问题2：说说@Transactional事务失效的常见场景？**

参考答案：1. 方法非public；2. 类内部this自调用；3. 捕获异常未抛出；4. 异常类型非运行时异常；5. 数据库引擎不支持事务；6. 传播机制配置不当。

## 7.5 组合注解、元注解底层原理面试

### 7.5.1 核心概念定义

**元注解**：用于**修饰其他注解的注解**，是注解的注解，是所有Spring注解的底层基础。Java原生提供5个元注解，定义了注解的生命周期、作用范围、继承特性等核心属性。

**组合注解**：基于元注解和基础注解，**自定义封装的复合注解**，一个组合注解可以整合多个基础注解的功能，简化代码、统一规范、减少冗余代码。

日常使用的@Service、@Controller、@Configuration全部都是**Spring内置的组合注解**，底层均基于元注解封装。

### 7.5.2 五大原生元注解详解

Java JDK内置5个核心元注解，所有自定义注解、Spring注解都必须依托它们实现：

- **@Target**：指定注解的**作用位置**（类、方法、字段、参数等），限制注解使用范围。如ElementType.TYPE表示仅能作用于类、接口。

- **@Retention**：指定注解的**生命周期**，核心三个级别：SOURCE（源码阶段，编译丢弃）、CLASS（编译阶段，运行丢弃）、RUNTIME（运行阶段，可反射获取）。Spring所有核心注解均为RUNTIME级别。

- **@Documented**：标记注解，生成Java文档时保留该注解信息。

- **@Inherited**：标记注解，子类可继承父类的该注解。

- **@Repeatable**：Java8+新增，支持同一位置重复添加该注解。

### 7.5.3 组合注解底层原理

1. 底层核心机制：**注解派生与属性覆盖**。组合注解通过元注解修饰基础注解，实现功能继承。

2. 经典案例拆解：@Service注解底层源码

@Service本质是组合注解，底层被@Component修饰，继承了@Component的所有功能，同时限定了作用层级，是Spring典型的注解复用设计。

3. 执行逻辑：Spring扫描组合注解时，会**递归解析其底层所有元注解、基础注解**，将所有底层注解的功能全部生效，实现一键多注解效果。

### 7.5.4 自定义组合注解实操示例

生产场景：项目中接口统一开启日志、统一权限校验、统一响应格式，可自定义组合注解整合多个功能注解，简化代码。

```java
// 自定义组合注解：整合Controller、日志记录、权限校验
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Controller // 继承控制器功能
@RequestMapping("/api") // 统一接口前缀
public @interface ApiController {
    // 自定义属性，可覆盖底层注解属性
    String value() default "";
}

// 使用自定义组合注解（一行替代多个注解）
@ApiController
public class UserApiController {
    // 接口自动拥有/api前缀、控制器特性、统一规范
}

```

### 7.5.5 ⚠️ 易错点与避坑指南

**坑1：Retention级别设置错误，导致注解失效**

自定义Spring功能注解必须设置**@Retention(RetentionPolicy.RUNTIME)**，否则运行时无法通过反射获取注解信息，功能完全失效。

**坑2：Target范围设置过宽/过窄**

作用位置配置错误会导致注解无法使用或滥用，如类注解不能作用于方法，编译直接报错。

**坑3：误解@Inherited继承特性**

@Inherited仅作用于**类继承**，接口实现、方法重写均无法继承注解，是高频认知误区。

### 7.5.6 📌 面试高频考点

**问题1：什么是元注解？Spring常用注解的底层是什么？**

参考答案：元注解是修饰注解的注解，JDK共5个。Spring的@Service、@Controller、@Configuration都是组合注解，底层依托@Component和五大元注解实现，是注解复用的核心设计。

**问题2：组合注解的优势是什么？生产中如何应用？**

参考答案：优势：简化代码、统一项目规范、减少注解冗余、便于统一维护。生产应用：自定义接口统一注解、自定义缓存注解、自定义权限注解，整合多个基础功能，提升开发效率。

## 8.1 Spring 容器刷新 refresh() 十二大步面试

### 8.1.1 核心概念定义

**refresh()**是Spring IoC容器的**核心入口方法**，也是Spring容器启动、初始化、加载所有资源、创建Bean、完成上下文初始化的核心方法，隶属于`AbstractApplicationContext`抽象类。整个Spring IoC容器的生命周期，几乎全部封装在refresh()方法的执行流程中。

简单来说：Spring容器从**空容器**变成**完整可用、包含所有Bean、可对外提供服务**的容器，全过程都是由refresh()方法的十二大步流程驱动完成。如果把Spring容器比作一座工厂，refresh()就是工厂的**全套开工流程**，从清理场地、加载图纸、注册设备、生产零件、质检零件到最终开工运行，全部流程一一对应。

该方法是**面试必问核心考点**，面试官重点考察：流程顺序、每一步核心作用、易错环节、Spring各类功能的底层触发时机。

### 8.1.2 底层核心原理与十二大步完整流程

refresh()方法被Spring定义为**模板方法**，固定执行十二步核心流程，顺序严格固定、不可颠倒，每一步负责独立的容器初始化职责，整体遵循「环境准备→配置加载→工厂初始化→Bean处理→容器启动→后置收尾」的逻辑。以下是源码级十二大步完整拆解，附带每一步核心作用与底层逻辑：

#### 第一步：prepareRefresh() 容器刷新前置准备

核心职责：完成容器刷新前的基础环境准备工作。底层会设置容器的**刷新状态**、初始化当前容器的启动时间、激活容器的活跃状态、初始化环境变量和属性配置源、校验必备环境参数是否合法。同时会初始化**早期事件监听器**，用于接收容器刷新过程中提前发布的事件。

设计目的：保证容器刷新前环境干净、参数合法、状态正常，避免后续流程因环境异常导致初始化失败。

#### 第二步：obtainFreshBeanFactory() 获取刷新后的Bean工厂

核心职责：刷新并获取Spring底层唯一的**BeanFactory工厂**（默认是DefaultListableBeanFactory）。底层会判断BeanFactory是否已存在，若存在则销毁所有Bean、清空工厂缓存、重置工厂状态；若不存在则新建BeanFactory。同时会**加载所有配置文件、注解、包扫描的Bean定义信息**，将所有Bean的配置信息解析为`BeanDefinition`（Bean定义），存入工厂的Bean定义注册表。

关键认知：这一步**只加载Bean定义，不创建Bean实例**，仅仅是把Bean的配置信息解析存储，相当于把工厂的生产图纸全部录入系统。

#### 第三步：prepareBeanFactory(beanFactory) 预处理Bean工厂

核心职责：对初始化完成的BeanFactory进行**属性填充和功能增强**。底层会设置Bean工厂的类加载器、表达式解析器、属性编辑器；注册Spring内置的**环境Bean、系统属性Bean**；添加Bean后置处理器`ApplicationContextAwareProcessor`，用于后续处理Aware接口的依赖注入；同时设置容器的依赖注入忽略规则、自动装配规则。

设计目的：完善Bean工厂的基础能力，为后续Bean的创建、依赖注入、属性赋值提供底层支撑。

#### 第四步：postProcessBeanFactory(beanFactory) Bean工厂后置处理

核心职责：这是**空模板扩展方法**，由子类容器实现（如AnnotationConfigApplicationContext）。Spring预留该方法用于子类在Bean工厂初始化完成后、Bean创建之前，对BeanFactory进行自定义修改、扩展配置、新增Bean定义、修改Bean属性等操作。

场景价值：是Spring扩展机制的早期入口，常用于自定义Bean工厂配置、动态注册Bean、修改已有Bean定义。

#### 第五步：invokeBeanFactoryPostProcessors(beanFactory) 执行工厂后置处理器

核心职责：执行所有实现了`BeanFactoryPostProcessor`（工厂后置处理器）和`BeanDefinitionRegistryPostProcessor`（Bean定义注册后置处理器）的组件。这是**修改Bean定义的唯一时机**，可以动态新增、删除、修改、替换项目中所有的BeanDefinition。

核心场景：Spring的`ConfigurationClassPostProcessor`就是在此步骤执行，完成注解配置解析、包扫描、@Bean注解解析、循环依赖预处理等核心操作。

#### 第六步：registerBeanPostProcessors(beanFactory) 注册Bean后置处理器

核心职责：扫描项目中所有实现`BeanPostProcessor`的后置处理器，统一注册到Bean工厂中，并且**按照优先级排序**（优先级：PriorityOrdered > Ordered > 普通）。

关键区别：此步骤只是**注册处理器**，不会执行处理器逻辑；Bean后置处理器的执行时机是**每个Bean创建前后**。常见的后置处理器包括：AutowiredAnnotationBeanPostProcessor（处理@Autowired）、CommonAnnotationBeanPostProcessor（处理@Resource）。

#### 第七步：initMessageSource() 初始化国际化资源

核心职责：初始化Spring容器的**国际化消息资源组件**MessageSource，用于项目的多语言、国际化文案解析。若项目中自定义了MessageSource则使用自定义Bean，否则使用Spring默认的空实现。

#### 第八步：initApplicationEventMulticaster() 初始化事件广播器

核心职责：初始化Spring事件机制的核心组件**ApplicationEventMulticaster（事件广播器）**，负责统一发布、分发所有Spring容器事件。自定义事件、内置事件（容器刷新、容器关闭）全部由该组件广播。若无自定义实现，Spring默认使用SimpleApplicationEventMulticaster。

#### 第九步：onRefresh() 容器刷新扩展钩子

核心职责：又是**空模板扩展方法**，子类容器可重写该方法，在容器核心初始化完成后、Bean完全就绪前执行自定义逻辑。典型场景：SpringBoot中内嵌Tomcat容器的启动，就是重写该方法实现的。

#### 第十步：registerListeners() 注册事件监听器

核心职责：扫描项目中所有实现`ApplicationListener`的监听器Bean，以及通过@EventListener注解定义的监听方法，统一注册到事件广播器中，完成事件监听的绑定配置。

#### 第十一步：finishBeanFactoryInitialization(beanFactory) 初始化所有单例Bean

**整个refresh流程最核心、最耗时的步骤**。核心职责：完成所有**非懒加载、单例Bean**的实例化、属性填充、依赖注入、初始化方法执行、后置处理器增强。

底层流程：遍历所有BeanDefinition → 判断单例、非懒加载 → 调用无参构造创建实例 → 解决循环依赖 → 依赖注入 → 执行初始化方法（@PostConstruct、init-method）→ 执行Bean后置处理器 → 完成Bean就绪。我们日常使用的绝大部分Bean，都在这一步完成创建。

#### 第十二步：finishRefresh() 完成容器刷新

核心职责：容器收尾工作，标记容器**刷新完成、启动成功**；初始化容器生命周期处理器；发布**ContextRefreshedEvent容器刷新完成事件**；启动所有实现Lifecycle接口的组件；最终容器进入可用状态，可对外提供服务。

### 8.1.3 生产场景价值

1. **问题定位依据**：生产中Bean创建失败、循环依赖报错、注解失效、事件监听不生效等问题，均可根据refresh()流程定位报错环节，快速排查问题根源。

2. **自定义扩展落地**：开发者可根据十二大步的扩展钩子（空模板方法、后置处理器），在容器不同阶段植入自定义逻辑，实现动态注册Bean、修改配置、自定义生命周期逻辑。

3. **性能优化**：了解Bean创建时机，可合理使用懒加载、多例Bean，减少容器启动耗时，优化项目启动速度。

### 8.1.4 ⚠️避坑指南（高频易错点）

1. **混淆Bean定义加载和Bean实例创建**：很多新手误以为包扫描时就创建了Bean实例，实际包扫描仅加载BeanDefinition，单例Bean实例在**第十一步**才会创建。

2. **后置处理器执行时机混淆**：工厂后置处理器在**第五步执行**（修改Bean定义），Bean后置处理器在**Bean创建时执行**（修改Bean实例），顺序不可颠倒。

3. **刷新事件监听失效**：自定义监听器若未生效，大概率是监听器注册时机晚于容器刷新事件发布时机。

4. **循环依赖报错**：仅单例Bean、Setter/字段注入可解决循环依赖，构造器注入的循环依赖在refresh流程中会直接报错，无法解决。

### 8.1.5 📌高频面试真题

**真题1：请详细说说Spring refresh()方法的十二大步流程，以及核心步骤的作用？**

参考答案：首先说明refresh()是Spring容器初始化的核心模板方法，固定十二步执行流程，核心流程可分为四大阶段：1. 环境准备（1-2步）：准备容器环境、加载Bean定义；2. 工厂增强（3-6步）：完善Bean工厂、执行工厂后置处理器、注册Bean后置处理器；3. 组件初始化（7-10步）：初始化国际化、事件广播器、注册监听器；4. Bean创建与收尾（11-12步）：实例化所有单例Bean、完成容器刷新。其中**第五步执行工厂后置处理器、第六步注册Bean后置处理器、第十一步初始化单例Bean**是核心核心步骤，支撑Spring的注解解析、Bean创建、依赖注入核心能力。

**真题2：refresh()方法中，BeanDefinition和Bean实例分别在哪个步骤完成加载和创建？**

参考答案：BeanDefinition（Bean定义）在**第二步obtainFreshBeanFactory**中完成加载解析；Bean实例（单例非懒加载）在**第十一步finishBeanFactoryInitialization**中完成实例化、依赖注入、初始化。

## 8.2 Bean 后置处理器 BeanPostProcessor 原理

### 8.2.1 核心概念定义

**BeanPostProcessor（Bean后置处理器）**是Spring提供的**Bean实例级别扩展接口**，是Spring AOP、注解增强、Bean属性修改的底层核心支撑。通俗理解：它是Spring为每一个Bean准备的**质检和增强工具**，在Bean实例创建完成、属性赋值结束后，对Bean进行自定义修改、增强、包装。

该接口包含两个核心方法，对应Bean初始化的两个后置节点：

- **postProcessBeforeInitialization**：Bean初始化方法执行**之前**执行

- **postProcessAfterInitialization**：Bean初始化方法执行**之后**执行

核心特性：**作用于所有Bean实例**，全局生效，优先级可自定义，是Spring底层功能增强的核心入口。

### 8.2.2 底层核心原理与执行流程

想要彻底理解BeanPostProcessor，必须掌握其**注册时机、执行时机、完整链路**，结合refresh()容器刷新流程，完整底层逻辑如下：

#### 1. 注册时机

在Spring容器刷新的**第六步registerBeanPostProcessors**中，Spring会扫描项目中所有实现BeanPostProcessor的Bean，统一注册到Bean工厂的后置处理器列表中，并根据优先级排序，优先级高的优先执行。

#### 2. 执行时机（核心）

每一个单例Bean在第十一步初始化时，完整执行链路为：**Bean实例创建（构造器）→ 依赖注入（属性赋值）→ postProcessBeforeInitialization（前置后置处理器）→ 初始化方法（@PostConstruct/init-method）→ postProcessAfterInitialization（后置后置处理器）→ Bean就绪**。

简单总结：**Bean创建、赋值完成后，初始化前后，各执行一次后置处理器逻辑**。

#### 3. 底层核心作用

Spring几乎所有的**Bean增强功能**都基于此接口实现：

- AutowiredAnnotationBeanPostProcessor：处理@Autowired、@Value注解依赖注入

- CommonAnnotationBeanPostProcessor：处理@Resource、@PostConstruct、@PreDestroy注解

- AopAnnotationBeanPostProcessor：完成AOP动态代理创建，生成代理Bean

### 8.2.3 自定义BeanPostProcessor实操示例

下面提供可直接运行的自定义后置处理器示例，实现对所有Bean初始化前后的自定义增强，附带完整注释：

```java
// 自定义Bean后置处理器，实现全局Bean增强
@Component
public class CustomBeanPostProcessor implements BeanPostProcessor {

    /**
     * Bean初始化方法执行之前执行
     * @param bean 当前正在处理的Bean实例
     * @param beanName 当前Bean的名称
     * @return 处理后的Bean实例
     */
    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) throws BeansException {
        // 对所有Bean初始化前进行自定义处理，比如打印日志、修改属性
        System.out.println("Bean初始化前置处理：" + beanName);
        return bean;
    }

    /**
     * Bean初始化方法执行之后执行
     * @param bean 当前正在处理的Bean实例
     * @param beanName 当前Bean的名称
     * @return 处理后的Bean实例
     */
    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) throws BeansException {
        // 初始化后增强，比如AOP代理、动态修改Bean逻辑
        System.out.println("Bean初始化后置处理：" + beanName);
        return bean;
    }
}

```

启动项目后，所有Spring管理的Bean在初始化时，都会自动执行该处理器的两个方法，实现全局统一增强。

### 8.2.4 生产场景价值

1. **全局Bean统一增强**：无需逐个修改业务Bean，通过后置处理器统一为所有Bean添加日志、权限、参数校验、属性填充逻辑。

2. **自定义注解实现**：开发者可基于该接口自定义业务注解，实现类似@Autowired的自动注入、自动赋值能力。

3. **AOP底层支撑**：Spring AOP的代理对象创建、切面植入，全部依赖BeanPostProcessor实现，是AOP的底层核心。

### 8.2.5 ⚠️高频易错点与避坑指南

1. **后置处理器不会处理自身**：自定义的BeanPostProcessor本身是Bean，但不会被自身的处理器逻辑增强，仅处理普通业务Bean。

2. **多处理器执行顺序混乱**：多个BeanPostProcessor共存时，必须实现Ordered接口或@Order注解指定优先级，否则执行顺序随机，会导致增强逻辑错乱。

3. **原型Bean同样生效**：BeanPostProcessor不仅作用于单例Bean，**每次获取原型Bean都会执行一次后置处理器**，频繁获取原型Bean会导致性能损耗。

4. **返回null导致Bean失效**：后置处理器方法必须返回Bean实例，若返回null，Spring会丢弃当前Bean，导致项目启动报错。

### 8.2.6 📌高频面试真题

**真题1：BeanPostProcessor的执行时机和核心作用是什么？和初始化方法的执行顺序？**

参考答案：执行时机为每个Bean实例创建、依赖注入完成后，初始化方法执行前后。完整执行顺序：构造器创建Bean → 依赖注入 → Before后置处理器 → @PostConstruct初始化方法 → After后置处理器 → Bean就绪。核心作用是对Bean实例进行自定义增强，是Spring注解解析、AOP代理、依赖注入的底层实现。

**真题2：BeanPostProcessor能否作用于多例Bean？为什么？**

参考答案：可以作用于多例Bean。因为BeanPostProcessor的执行时机是**每次Bean初始化时**，单例Bean仅容器启动时初始化一次，所以仅执行一次；多例Bean每次从容器获取都会初始化，因此每次获取都会执行后置处理器逻辑。

## 8.3 工厂后置处理器 BeanFactoryPostProcessor

### 8.3.1 核心概念定义

**BeanFactoryPostProcessor（Bean工厂后置处理器）**是Spring提供的**工厂级别扩展接口**，作用于整个Bean工厂，区别于BeanPostProcessor的Bean实例级别增强。通俗理解：BeanPostProcessor是「修改每一个生产好的零件」，而BeanFactoryPostProcessor是「修改工厂的生产图纸、生产规则」。

该接口仅有一个核心方法`postProcessBeanFactory`，执行时机在**所有Bean定义加载完成、Bean实例创建之前**，核心能力是**修改、新增、删除、替换BeanDefinition（Bean定义）**，不会影响已创建的Bean实例（此时暂无Bean实例）。

### 8.3.2 核心原理与和BeanPostProcessor的核心区别

结合refresh()容器流程，工厂后置处理器在**第五步执行**，早于Bean后置处理器的注册和执行，是Spring最早的扩展入口之一。为了清晰区分两大后置处理器，这里用表格对比核心差异（面试高频考点）：

| 对比维度 | BeanFactoryPostProcessor（工厂后置处理器）             | BeanPostProcessor（Bean后置处理器）  |
| -------- | ------------------------------------------------------ | ------------------------------------ |
| 作用级别 | Bean工厂级别，全局作用于所有Bean定义                   | Bean实例级别，作用于单个Bean对象     |
| 执行时机 | 容器刷新第五步，Bean定义加载完成，**Bean实例创建之前** | Bean初始化阶段，**Bean实例创建之后** |
| 操作对象 | BeanDefinition（Bean定义信息）                         | Bean实例（Java对象）                 |
| 核心能力 | 动态注册Bean、修改Bean属性、删除Bean定义               | 增强Bean实例、代理包装、属性二次修改 |
| 执行次数 | 整个容器生命周期**仅执行一次**                         | 每个Bean初始化时执行一次             |

### 8.3.3 子接口：BeanDefinitionRegistryPostProcessor

BeanFactoryPostProcessor有一个核心子接口**BeanDefinitionRegistryPostProcessor**，是开发中更常用的扩展接口，在父接口基础上新增了`postProcessBeanDefinitionRegistry`方法，执行时机**早于父接口方法**。

核心价值：专门用于**动态注册自定义Bean定义**，很多框架的自动配置、动态Bean注册都是基于该接口实现，比如MyBatis的Mapper接口动态注册、SpringBoot自动配置类加载。

### 8.3.4 自定义工厂后置处理器实操示例

以下示例实现自定义BeanFactoryPostProcessor，完成修改已有Bean定义属性、打印工厂信息的功能，可直接运行：

```java
// 自定义工厂后置处理器，作用于Bean工厂级别
@Component
public class CustomBeanFactoryPostProcessor implements BeanFactoryPostProcessor {

    @Override
    public void postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory) throws BeansException {
        // 1. 获取容器中所有Bean的名称
        String[] beanDefinitionNames = beanFactory.getBeanDefinitionNames();
        System.out.println("容器加载的所有Bean定义：" + Arrays.toString(beanDefinitionNames));

        // 2. 获取指定Bean的定义，修改其属性
        if (beanFactory.containsBeanDefinition("userService")) {
            BeanDefinition beanDefinition = beanFactory.getBeanDefinition("userService");
            // 修改Bean的作用域为多例
            beanDefinition.setScope(BeanDefinition.SCOPE_PROTOTYPE);
            System.out.println("已修改userService为多例Bean");
        }
    }
}

```

项目启动后，会在Bean创建之前修改userService的作用域，后续获取的userService均为多例实例，完美体现工厂后置处理器修改Bean定义的能力。

### 8.3.5 生产场景价值

1. **动态注册Bean**：无需通过@Component、@Bean注解，通过代码动态注册第三方类为Spring Bean，常用于集成第三方框架、动态扩展功能。

2. **统一修改Bean配置**：全局修改项目中指定Bean的作用域、初始化方法、依赖属性，无需逐个修改配置类。

3. **框架底层自动配置**：SpringBoot、MyBatis、Dubbo等框架的自动配置核心，全部基于该接口实现，实现零配置、自动加载组件。

### 8.3.6 ⚠️高频易错点与避坑指南

1. **无法修改Bean实例**：工厂后置处理器执行时，所有Bean均未创建，只能修改Bean定义，不能操作Bean实例，新手极易混淆。

2. **执行顺序问题**：多个工厂后置处理器共存时，同样需要指定优先级，优先级低的处理器无法修改优先级高的处理器注册的Bean定义。

3. **禁止在该阶段获取Bean实例**：若在postProcessBeanFactory方法中调用getBean()获取Bean，会导致Bean提前初始化，破坏Spring正常的Bean生命周期，引发循环依赖、属性缺失等诡异问题。

### 8.3.7 📌高频面试真题

**真题1：BeanFactoryPostProcessor和BeanPostProcessor的区别？底层执行顺序？**

参考答案：1. 执行顺序：BeanFactoryPostProcessor先执行（容器刷新第五步），BeanPostProcessor后执行（Bean初始化阶段）；2. 作用对象：前者操作Bean定义，后者操作Bean实例；3. 执行次数：前者全局一次，后者每个Bean一次；4. 核心场景：前者用于动态注册、修改Bean配置，后者用于Bean实例增强、AOP代理。

**真题2：为什么工厂后置处理器中不能调用getBean()？**

参考答案：因为此时Spring尚未完成所有Bean定义的加载，也未开始统一初始化Bean。提前调用getBean()会强制Bean提前实例化，跳过Spring的后置处理器增强、依赖注入校验、循环依赖处理等核心流程，导致Bean属性缺失、代理失效、循环依赖报错等生产问题。

## 8.4 事件发布与监听机制面试

### 8.4.1 核心概念定义

Spring事件机制是Spring内置的**观察者模式实现**，核心作用是**解耦业务代码**，实现业务逻辑的异步/同步解耦执行。通俗理解：事件发布者负责「发布消息」，事件监听者负责「接收消息、执行逻辑」，两者完全解耦，无需直接依赖调用。

整个事件机制包含三大核心组件，缺一不可：

- **事件（ApplicationEvent）**：自定义/内置事件载体，用于存储事件数据、标识事件类型

- **事件发布器（ApplicationEventPublisher）**：负责发布事件，触发事件广播

- **事件监听器（ApplicationListener/@EventListener）**：监听指定事件，执行自定义业务逻辑

### 8.4.2 底层核心原理与执行流程

结合Spring容器刷新流程，事件机制的完整初始化和执行链路如下：

#### 1. 组件初始化时机

容器刷新第八步：初始化事件广播器ApplicationEventMulticaster；第十步：注册所有事件监听器，完成监听绑定。容器启动完成后，即可正常发布和监听事件。

#### 2. 完整执行链路

**业务代码发布事件 → 事件广播器接收事件 → 匹配所有对应类型的监听器 → 同步/异步执行监听逻辑**

Spring默认是**同步执行**，即发布事件的线程会阻塞，等待所有监听器逻辑执行完成后，才继续执行后续业务代码；可通过配置开启异步监听。

#### 3. Spring内置核心事件

Spring内置多个容器生命周期事件，面试高频考察：

- ContextRefreshedEvent：容器刷新完成事件（项目启动完成）

- ContextStartedEvent：容器启动事件

- ContextStoppedEvent：容器停止事件

- ContextClosedEvent：容器关闭事件

### 8.4.3 事件机制完整实操示例

下面实现「自定义事件+事件发布+事件监听」完整可运行案例，实现业务解耦：

#### 1. 自定义业务事件

```java
// 自定义用户注册事件，继承Spring基础事件ApplicationEvent
public class UserRegisterEvent extends ApplicationEvent {
    // 存储事件传递的业务数据
    private String username;

    // 构造方法：source为事件发布源
    public UserRegisterEvent(Object source, String username) {
        super(source);
        this.username = username;
    }

    // getter/setter
    public String getUsername() {
        return username;
    }
}

```

#### 2. 事件监听（两种实现方式）

```java
// 方式1：注解式监听（推荐、简洁）
@Component
public class UserEventListener {

    // 监听用户注册事件，触发后续业务
    @EventListener(UserRegisterEvent.class)
    public void handleRegisterEvent(UserRegisterEvent event) {
        System.out.println("监听到用户注册事件，用户名：" + event.getUsername());
        // 可执行解耦业务：发送短信、发送邮件、记录日志、积分发放等
    }
}

// 方式2：接口式监听（适合复杂监听逻辑）
@Component
public class UserRegisterListener implements ApplicationListener<UserRegisterEvent> {
    @Override
    public void onApplicationEvent(UserRegisterEvent event) {
        System.out.println("接口监听用户注册事件：" + event.getUsername());
    }
}

```

#### 3. 事件发布

```java
@Service
public class UserService {

    // 注入事件发布器
    @Autowired
    private ApplicationEventPublisher eventPublisher;

    public void register(String username) {
        // 1. 核心注册业务逻辑
        System.out.println("用户完成注册：" + username);

        // 2. 发布自定义事件，解耦后续业务
        eventPublisher.publishEvent(new UserRegisterEvent(this, username));
    }
}

```

调用register方法后，会自动触发所有监听器的执行，注册核心业务和短信、邮件等附属业务完全解耦。

### 8.4.4 同步与异步监听配置

Spring默认同步监听，高并发场景下会阻塞主线程，影响接口响应速度。开启异步监听只需两步：

1. 启动类添加`@EnableAsync`开启异步支持；

2. 监听方法添加`@Async`注解。

异步效果：事件发布后，主线程立即返回，监听逻辑由异步线程执行，不阻塞主业务。

### 8.4.5 生产场景价值

1. **业务解耦**：核心业务与附属业务（通知、日志、积分、统计）解耦，符合开闭原则，新增业务无需修改核心代码。

2. **提升接口性能**：异步监听可剥离非核心耗时逻辑，大幅提升接口响应速度。

3. **框架扩展**：SpringBoot、SpringCloud大量基于事件机制实现容器生命周期扩展、配置刷新、服务注册等能力。

### 8.4.6 ⚠️高频易错点与避坑指南

1. **同步监听阻塞主线程**：新手容易忽略默认同步机制，导致非核心业务耗时过长，拖慢主接口响应。

2. **异步监听事务失效**：异步监听方法会脱离主线程事务，若监听方法需要事务，需单独配置事务，否则数据不一致。

3. **监听器Bean未注册**：监听器必须交给Spring管理（添加@Component），否则无法被事件广播器识别，监听失效。

4. **事件传递数据空指针**：自定义事件必须重写带参构造，否则无法传递业务数据，导致监听获取数据为空。

### 8.4.7 📌高频面试真题

**真题1：Spring事件机制的原理？默认是同步还是异步？如何开启异步？**

参考答案：Spring事件机制基于观察者模式实现，由事件、事件发布器、监听器三大组件组成。业务发布事件后，由事件广播器匹配对应监听器执行逻辑。Spring**默认同步执行**，发布事件线程阻塞等待监听逻辑执行完毕。开启异步需要启动类添加@EnableAsync，监听方法添加@Async注解，实现异步解耦。

**真题2：Spring事件监听异步场景下，会有什么事务问题？如何解决？**

参考答案：主线程事务和异步监听线程事务相互独立，主线程事务未提交时，异步线程可能已执行监听逻辑，导致查询不到最新数据、数据不一致。解决方式：保证监听逻辑无强事务依赖，或手动控制事务提交时机，避免跨线程事务问题。

## 8.5 Spring 扩展接口全家桶面试汇总

### 8.5.1 核心概念与整体梳理

Spring的核心优势除了IoC、AOP之外，极强的**可扩展性**是其成为主流框架的核心原因。Spring提供了一套完整的扩展接口体系，覆盖**容器生命周期、Bean生命周期、属性注入、事件监听、工厂配置、资源加载**全场景。

本节汇总面试、生产中**所有高频Spring扩展接口**，统一梳理执行顺序、核心作用、使用场景、易错点，是Spring面试的终极汇总考点。所有扩展接口均由Spring容器自动触发，无需手动调用。

### 8.5.2 Spring全套高频扩展接口详解

#### 1. Bean生命周期扩展接口（最常用）

- **InitializingBean**：Bean初始化接口，afterPropertiesSet()方法在Bean属性赋值完成后、自定义init-method之前执行，用于Bean初始化自定义逻辑。

- **DisposableBean**：Bean销毁接口，destroy()方法在容器关闭、Bean销毁时执行，用于资源释放、连接关闭等收尾逻辑。

- **FactoryBean**：Bean工厂接口，用于自定义复杂Bean的创建逻辑，MyBatis的Mapper代理Bean就是基于该接口实现，返回的Bean是getObject()方法的返回值。

#### 2. Aware系列感知接口（面试高频）

Aware接口的核心作用：**让普通Bean感知Spring容器资源**，普通Bean默认无法获取容器、工厂、环境等内置资源，实现对应Aware接口即可自动注入。

- **BeanNameAware**：感知当前Bean的名称，回调setBeanName()获取beanName

- **BeanFactoryAware**：感知Bean工厂，回调setBeanFactory()获取BeanFactory，可动态获取容器Bean

- **ApplicationContextAware**：感知Spring上下文，回调setApplicationContext()获取上下文，可发布事件、获取环境变量

- **EnvironmentAware**：感知环境变量，获取配置文件参数

#### 3. 后置处理器扩展接口（核心底层）

前文已详细讲解，此处汇总核心定位：

- **BeanFactoryPostProcessor**：工厂级别扩展，修改Bean定义，容器启动仅执行一次

- **BeanDefinitionRegistryPostProcessor**：动态注册Bean定义，优先级高于父接口

- **BeanPostProcessor**：Bean实例级别扩展，增强Bean、实现AOP、注解解析

#### 4. 容器生命周期扩展接口

- **Lifecycle**：容器启停生命周期接口，自定义组件启动、停止逻辑

- **ApplicationListener**：事件监听接口，实现事件解耦

### 8.5.3 完整执行顺序（面试必考终极顺序）

为方便记忆，整理**单个Bean从创建到销毁的完整扩展接口执行顺序**，全网通用标准答案：

**容器启动阶段：**

BeanDefinitionRegistryPostProcessor → BeanFactoryPostProcessor → 注册BeanPostProcessor

**单个Bean初始化阶段：**

构造器创建实例 → Aware接口回调（BeanName→BeanFactory→ApplicationContext）→ BeanPostProcessor前置处理 → InitializingBean → @PostConstruct → 自定义init-method → BeanPostProcessor后置处理 → Bean就绪可用

**容器关闭阶段：**

DisposableBean → 自定义destroy-method → Bean销毁

### 8.5.4 各接口生产最佳实践

1. **优先使用注解，少用接口**：初始化优先用@PostConstruct，销毁优先用@PreDestroy，比实现接口更简洁、低侵入。

2. **FactoryBean用于复杂Bean**：第三方复杂对象、代理对象创建统一使用FactoryBean，解耦Bean创建逻辑。

3. **Aware接口谨慎使用**：Aware接口会让Bean强依赖Spring容器，违背面向对象设计思想，非必要不使用。

4. **后置处理器统一排序**：多个后置处理器共存时，必须指定优先级，避免执行顺序错乱。

### 8.5.5 ⚠️全局高频易错汇总

1. **@PostConstruct优先级高于InitializingBean**：很多开发者顺序记忆错误，注解初始化优先于接口初始化。

2. **FactoryBean和BeanFactory混淆**：BeanFactory是Spring核心工厂（生产Bean），FactoryBean是自定义Bean的工厂（生产特殊Bean），两者完全不同。

3. **Aware接口执行时机在依赖注入之后**：所有Aware回调均在属性赋值完成后执行，可安全使用容器资源。

4. **原型Bean不执行销毁方法**：Spring仅管理单例Bean的销毁生命周期，多例Bean的销毁方法不会被触发，需手动管理资源。

### 8.5.6 📌高频面试真题汇总

**真题1：请说出Spring Bean的完整生命周期，以及各扩展接口的执行顺序？**

参考答案：1. 容器加载Bean定义；2. 构造器实例化Bean；3. 依赖注入属性赋值；4. 执行各类Aware感知接口；5. 执行BeanPostProcessor前置处理；6. 执行@PostConstruct、InitializingBean、自定义初始化方法；7. 执行BeanPostProcessor后置处理，Bean就绪；8. 容器关闭时执行销毁方法。

**真题2：FactoryBean和BeanFactory的区别？**

参考答案：1. BeanFactory是Spring IoC容器的顶层接口，是所有Bean的生产工厂，负责Bean的创建、存储、获取；2. FactoryBean是自定义Bean的创建接口，用于创建复杂、代理类Bean，其返回的Bean是业务使用的Bean，而FactoryBean本身是容器的工具Bean；3. 作用层级完全不同，一个是容器核心，一个是Bean扩展工具。

# 9.Spring 生产环境踩坑 & 综合易错点

Spring项目的绝大多数线上故障、功能异常、BUG问题，并非代码逻辑错误，而是**对Spring底层机制不熟悉、配置不规范、场景适配错误**导致的隐性问题。本章节从开发环境、生产环境、配置混合使用、高并发场景、版本升级五个核心维度，全方位拆解高频坑点，搭配原理分析、报错场景、解决方案和最佳实践，帮助开发者彻底规避常见问题。

## 9.1 开发环境常见配置易错点

本地开发环境是问题滋生的源头，很多线上隐患都是开发阶段不规范配置、隐性错误积累导致的。开发环境的易错点主要集中在配置文件、注解使用、Bean注入、环境适配、日志配置等基础场景，看似简单却高频出错，且极易被忽略。

### 9.1.1 配置文件优先级与覆盖问题

**概念定义**：Spring Boot/Spring 拥有固定的配置文件加载优先级，不同路径、不同后缀、不同环境的配置文件会存在覆盖、互补逻辑，开发者如果不熟悉优先级规则，会出现配置不生效、参数错乱、环境配置混用的问题。

**核心原理**：Spring 配置加载遵循「**就近优先、高优先级覆盖低优先级**」原则，核心加载顺序（由低到高，后者覆盖前者）：resources/application.yml → resources/application.properties → resources/config/application.yml → 外部配置文件。同时，多环境配置（dev/test/prod）会叠加主配置，同名参数直接覆盖，不同参数互补。

**常见错误场景**：很多开发者习惯在主配置文件写通用参数，在开发环境配置文件修改参数，但因为不清楚优先级，导致开发配置未生效；或者同时使用yml和properties两种配置文件，出现参数冲突、配置丢失问题。

**避坑指南**⚠️

- 禁止项目中**同时混用yml和properties**两种配置格式，统一使用yml格式，避免解析冲突、优先级错乱。

- 主配置文件(application.yml)仅存放通用配置，环境专属配置（端口、数据库地址、日志级别、开关参数）统一放在application-dev.yml、application-prod.yml中。

- 启动项目时必须指定有效环境，避免默认加载主配置导致开发/生产参数混用。

**实操示例**

```yaml
# 主配置文件 application.yml（通用配置）
spring:
  application:
    name: spring-demo # 项目名称，全环境通用
  datasource:
    driver-class-name: com.mysql.cj.jdbc.Driver # 通用驱动配置

# 开发环境配置 application-dev.yml
server:
  port: 8080 # 开发环境端口
spring:
  datasource:
    url: jdbc:mysql://localhost:3306/dev_db # 开发库地址
    username: root
    password: 123456

# 启动指定环境，避免配置错乱
# IDEA启动参数：--spring.profiles.active=dev
```

**验证与排错**：通过项目启动日志「The following profiles are active: dev」确认环境生效；通过@Value注解打印配置参数，验证配置是否加载成功；配置不生效时优先检查文件名、路径、环境激活参数。

### 9.1.2 注解使用不规范导致Bean注入失败

**概念定义**：Spring 通过扫描注解实现Bean的自动注册，常见注解包括@Controller、@Service、@Repository、@Component、@Configuration，不同注解适用场景不同，误用会导致Bean无法注册、注入失败。

**核心原理**：Spring启动时会根据启动类的@ComponentScan注解扫描指定包下的标注类，只有被对应注解标记的类，才会被纳入Spring IOC容器管理，未扫描、注解误用、包路径不对，都会导致Bean不存在，抛出NoSuchBeanDefinitionException异常。

**高频易错场景**

- 工具类、普通业务辅助类未添加@Component注解，直接@Autowired注入，报错找不到Bean。

- 配置类未加@Configuration注解，导致@Bean注解失效，自定义Bean无法注册。

- 启动类扫描包范围过小，子包中的Service、Mapper无法被扫描到。

- 接口上加@Service注解，实现类未加注解，导致注入失败。

**💡最佳实践**

- 业务逻辑层统一使用@Service，控制器使用@Controller，持久层使用@Repository，通用工具类使用@Component，配置类固定使用@Configuration。

- 启动类默认扫描当前包及子包，项目分包规范，避免跨包无法扫描；如需扫描外部包，手动配置@ComponentScan指定路径。

- 接口不添加实例化注解，仅在实现类添加注解。

### 9.1.3 @Autowired与@Resource注入混淆易错

这是开发阶段最高频的易错点之一，两种注入注解功能相似，但底层匹配规则、依赖来源完全不同，混用会导致开发环境正常、线上报错的隐性问题。

**核心区别与易错点**：@Autowired是Spring专属注解，默认**按类型byType**注入；@Resource是JSR250标准注解，默认**按名称byName**注入。

**常见报错场景**：当一个接口存在多个实现类时，使用@Autowired会因为类型匹配不唯一，抛出UnsatisfiedDependencyException；而开发者误以为两者通用，直接使用@Autowired导致注入失败。

**解决方案**

- 单一实现类场景：优先使用@Autowired，简洁高效。

- 多实现类场景：@Autowired配合@Qualifier指定Bean名称，或使用@Resource(name="beanName")精准注入。

**📌面试考点**：面试官高频提问@Autowired和@Resource的区别、多实现类注入报错原因及解决方案。

## 9.2 线上生产环境经典坑点汇总

线上生产环境的坑点核心特点：**开发环境正常，线上环境异常**，问题隐蔽性强、影响范围广、排查难度大，多由环境差异、资源限制、参数默认值、安全配置缺失、日志问题导致。本小节汇总生产环境90%以上的经典高频坑点，附带落地解决方案。

### 9.2.1 生产环境热部署依赖未剔除

**问题现象**：本地开发使用spring-boot-devtools热部署依赖，实现代码实时生效，提升开发效率，但打包上线后，出现**项目启动缓慢、随机重启、接口偶发超时、内存泄漏**等问题。

**核心原理**：devtools是开发专属工具，核心机制是监听文件变动、自动重启上下文，生产环境无文件变动需求，该依赖会持续占用线程、监听资源，导致项目不稳定，同时会改变Spring类加载机制，引发隐性BUG。

**避坑方案**：打包插件中配置**生产环境自动排除devtools依赖**，禁止打包进入生产环境。

```xml
<!-- Maven打包配置，排除热部署依赖 -->
<plugin>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-maven-plugin</artifactId>
    <configuration>
        &lt;excludeDevtools&gt;true&lt;/excludeDevtools&gt; <!-- 生产剔除热部署 -->
    </configuration>
</plugin>
```

### 9.2.2 生产环境日志配置不当导致磁盘爆满

**问题现象**：开发环境日志输出随意，未配置日志分割、过期清理、日志级别，上线后日志文件无限增大，短时间内占满服务器磁盘，导致项目宕机、服务器卡死。

**核心易错点**：很多开发者仅配置控制台日志，未配置文件日志滚动策略；生产环境开启DEBUG日志级别，日志输出量暴增；未设置日志保留天数，历史日志不自动清理。

**💡生产最佳实践**

- 生产环境统一设置日志级别为INFO/WARN，关闭DEBUG详细日志。

- 配置日志文件按大小、日期双分割，单文件最大100MB，保留7天日志。

- 关闭控制台日志输出，仅保留文件日志，减少IO消耗。

### 9.2.3 上下文路径、端口冲突线上启动失败

**问题场景**：本地端口固定8080，无冲突，项目正常启动；线上服务器部署多个Spring项目，未修改端口和上下文路径，导致端口占用、项目启动失败。

**解决方案**：生产环境通过配置文件动态指定端口和上下文路径，禁止硬编码固定端口。

```yaml
# 生产环境配置
server:
  port: 8090 # 自定义线上端口，避免冲突
  servlet:
    context-path: /api # 统一上下文路径，适配反向代理
```

### 9.2.4 静态资源拦截、跨域配置线上失效

**经典坑点**：开发环境前端后端同源，无跨域问题，线上前后端分离部署后，出现跨域请求失败、静态资源404。核心原因是自定义WebMvcConfigurer配置类未正确重写跨域、资源映射方法，或拦截器拦截了静态资源。

**解决方案**：统一配置跨域全局拦截，放行静态资源路径，适配线上前后端分离场景。

## 9.3 配置类、XML 混合使用易错点

Spring 支持XML传统配置和Java Config注解配置两种方式，在老旧项目迭代、微服务改造、项目迁移场景中，普遍存在**XML+配置类混合使用**的情况，也是高频报错场景。两种配置方式的加载顺序、Bean注册规则不同，极易出现Bean覆盖、重复注册、配置失效、循环依赖等问题。

### 9.3.1 混合配置加载顺序错乱导致配置失效

**核心原理**：Spring容器启动时，**优先加载XML配置，后加载Java Config配置类**，后加载的Bean会覆盖先加载的同名Bean。如果开发者不清楚加载顺序，会出现自定义配置覆盖原生配置、参数不生效的问题。

**易错场景**：老旧项目使用XML配置数据源、事务，新代码使用@Configuration配置同类Bean，导致XML配置被覆盖，数据库连接异常、事务失效。

**避坑指南**⚠️

- 项目尽量**统一配置方式**，新项目全部使用Java Config注解配置，老旧项目逐步迁移，减少混合配置。

- 混合场景下，禁止XML和配置类注册**同名Bean**，避免覆盖冲突。

- 通过@ImportResource注解精准加载需要保留的XML配置，不全局加载所有XML文件。

```java
// 精准加载指定XML配置文件，避免全局加载冲突
@Configuration
@ImportResource("classpath:spring-datasource.xml")
public class SpringConfig {
    // 自定义Java配置Bean，与XML Bean名称区分
}
```

### 9.3.2 Bean重复注册引发单例冲突

**问题现象**：XML中注册了UserService Bean，Java配置类中又通过@Bean注册同名UserService，启动不报异常，但项目运行逻辑错乱，数据不一致。

**原理剖析**：Spring IOC容器允许同名Bean后注册覆盖先注册，默认无报错，属于**隐性致命BUG**。开发阶段难以发现，线上运行后出现业务逻辑异常，排查难度极高。

**解决方案**

- 规范Bean命名，XML和注解配置的Bean名称严格区分。

- 开启Spring Bean重复注册校验，生产环境强制报错，提前发现问题。

- 逐步淘汰XML配置，统一使用注解配置，彻底规避混合冲突。

### 9.3.3 事务配置混合导致事务失效

这是混合配置最致命的生产坑点。XML全局事务拦截规则和注解@Transactional事务规则冲突，会出现**部分方法事务生效、部分失效**，数据回滚异常，引发生产数据问题。

**最佳实践**：全局事务统一使用**注解式事务**，废弃XML事务配置，保证事务规则统一、优先级一致。

## 9.4 高并发下 Spring Bean 与事务坑点

Spring默认Bean为**单例模式**，这是高并发场景下绝大多数问题的根源。单例Bean的线程不安全、事务传播机制误用、事务超时、锁机制缺失，会导致高并发下数据错乱、事务失效、死锁、接口超时等线上重大故障，是面试和生产落地的核心重难点。

### 9.4.1 单例Bean线程不安全坑点

**核心原理**：Spring Bean默认scope为singleton（单例），整个容器中仅存在一个Bean实例，所有请求线程共享同一个Bean。如果Bean中存在**成员变量**，多线程并发修改会出现线程安全问题，导致数据覆盖、参数错乱。

**高频错误场景**：在Service、Controller中定义成员变量存储临时数据，高并发下数据相互覆盖，业务逻辑异常。

**避坑指南**⚠️

- **绝对禁止**在Controller、Service中定义可修改的成员变量。

- 临时数据、请求参数统一存放在方法局部变量中，线程隔离。

- 如需成员变量，使用ThreadLocal实现线程隔离，或修改Bean作用域为prototype（多例）。

**💡最佳实践**：绝大多数业务场景下，保持Bean单例，仅通过局部变量处理业务数据，保证性能和线程安全。

### 9.4.2 高并发下事务传播机制误用导致数据异常

Spring事务传播机制是高并发场景的核心易错点，不同传播级别适配不同并发场景，误用会导致**事务嵌套失效、部分提交、部分回滚**。

**高频坑点场景**

- 内层方法使用REQUIRES_NEW（新开事务），外层事务回滚，内层已提交数据无法回滚，导致数据不一致。

- 嵌套方法默认REQUIRED级别，内外事务共用一个事务，内层异常被捕获后，外层无法感知，导致事务不回滚。

**📌面试核心考点**：事务传播机制7种级别区别、嵌套事务失效场景、高并发事务数据一致性解决方案。

### 9.4.3 事务超时、锁等待超时高并发坑点

**问题现象**：高并发秒杀、批量更新场景下，大量请求同时操作同一行数据，引发数据库行锁竞争，事务超时、接口报错、请求堆积。

**解决方案**

- 合理设置事务超时时间，避免事务长时间占用锁资源。

- 大事务拆分小事务，减少锁持有时间，提升并发能力。

- 高并发热点数据使用缓存兜底，减少数据库直接操作。

## 9.5 版本升级兼容易错与避坑方案

Spring、Spring Boot版本迭代较快，从Spring Boot 1.x升级到2.x、3.x，以及Spring框架版本升级，存在大量API废弃、配置变更、规则修改问题，版本升级是项目迭代中的高频难点，极易出现启动失败、功能失效、兼容报错等问题。本小节汇总主流版本升级的核心兼容坑点和落地避坑方案。

### 9.5.1 Spring Boot 1.x 升级 2.x 核心兼容坑点

**1. 配置参数废弃**：1.x中的部分日志、数据源、Tomcat配置参数在2.x中彻底废弃，直接沿用旧配置会导致参数不生效、启动警告。

**2. 数据源配置变更**：Spring Boot2.x默认移除自动数据源配置，需要手动指定驱动类、数据源参数，否则启动报错。

**3. 拦截器、跨域配置适配变更**：2.x中WebMvc配置规则优化，旧的拦截器配置方式会导致静态资源拦截、跨域失效。

**4. 日期格式化默认规则变更**：1.x默认支持自动日期转换，2.x严格校验日期格式，旧项目日期参数会出现解析异常。

### 9.5.2 Spring Boot 2.x 升级 3.x 重大兼容问题

**核心变更**：Spring Boot3.x基于Spring6，彻底废弃Javax注解，全面迁移至Jakarta注解，这是最致命的兼容问题。旧项目中所有javax.servlet、javax.annotation相关注解全部失效，直接启动报错。

**避坑方案**⚠️

- 批量替换依赖：将所有javax包依赖替换为jakarta对应依赖。

- 升级JDK版本：Spring Boot3.x强制要求JDK17及以上，低版本JDK无法运行。

- 废弃过时API：移除所有3.x已废弃的方法、配置、注解，避免隐性BUG。

### 9.5.3 版本升级通用最佳实践

- **分步升级**：禁止跨大版本直接升级，1.x→2.x→3.x分步迭代，逐个解决兼容问题。

- **环境隔离**：测试环境完成全量功能回归测试后，再上线生产环境。

- **依赖统一管理**：通过parent统一Spring Boot版本，避免依赖版本混乱导致兼容冲突。

## 10.1 基础选择题真题

本部分为Spring基础高频单选题，考察基础概念、核心机制、常用注解的基础用法，属于面试必考题，难度偏基础，重点排查基础认知误区。

**真题1：Spring容器默认的Bean作用域是？（）**

A. prototype  B. singleton  C. request  D. session

**标准答案**：B

**解析**：Spring Bean默认作用域为**singleton单例**，整个IOC容器仅存在一个实例。prototype为多例，每次获取创建新对象；request、session为web专属作用域。

**真题2：以下哪个注解不属于Spring原生注解？（）**

A. @Autowired  B. @Resource  C. @Component  D. @Configuration

**标准答案**：B

**解析**：@Resource是JSR250Java标准注解，非Spring原生注解；其余均为Spring专属核心注解。@Autowired为Spring专属，默认byType注入。

**真题3：Spring事务默认的传播级别是？（）**

A. REQUIRED  B. REQUIRES_NEW  C. SUPPORTS  D. MANDATORY

**标准答案**：A

**解析**：REQUIRED是Spring事务默认传播级别，规则：当前有事务则加入，无事务则新建事务，是最常用的事务级别。

**真题4：以下哪种情况不会导致Spring事务失效？（）**

A. 方法私有修饰  B. 内部类调用  C. 同类方法普通调用  D. 外部类调用public事务方法

**标准答案**：D

**解析**：Spring事务基于AOP动态代理实现，私有方法、同类自调用、内部类调用无法触发代理，事务失效；外部类调用public方法可正常触发代理，事务生效。

**真题1：以下关于Spring单例Bean说法正确的是？（多选）（）**

A. Spring单例Bean是线程安全的

B. 单例Bean存在成员变量会引发线程安全问题

**标准答案**：BC

**解析**：A错误，单例Bean本身非线程安全，线程安全取决于是否存在共享可变成员变量；B正确，多线程共享成员变量会数据错乱；C正确，单例仅初始化一次，减少对象创建销毁开销，性能更优；D错误，高并发无需强制多例，通过局部变量、ThreadLocal即可保证线程安全。

**真题2：以下属于Spring事务失效场景的有？（多选）（）**

C. 同类中非事务方法调用本类事务方法

**标准答案**：ABC

**解析**：ABC均为经典事务失效场景；D错误，Spring事务默认捕获RuntimeException并回滚，不会失效。

**真题3：关于@Autowired和@Resource区别说法正确的是？（多选）（）**

## 10.3 简答论述高频真题（含标准答案）

简答论述题是面试核心得分点，重点考察底层原理、问题排查、场景解决方案，以下为互联网大厂高频原题，答案为面试标准满分话术。

**真题1：请简述Spring IOC的核心原理和执行流程？**

**满分标准答案**：

Spring IOC（控制反转）是Spring的核心思想，核心是**将对象的创建、管理、依赖注入权限交给Spring容器，摆脱开发者手动new对象的硬编码模式，实现组件解耦**。

核心执行流程分为5步：

1. **资源加载**：容器启动，加载配置文件、注解配置，扫描指定包下的所有组件类；

2. **Bean定义解析**：解析扫描到的类，生成BeanDefinition，存储Bean的类信息、作用域、依赖关系、属性等元数据；

3. **Bean实例化**：容器根据BeanDefinition，通过反射创建Bean实例，单例Bean在容器启动时提前实例化，多例在获取时实例化；

4. **属性填充（依赖注入）**：自动解析Bean的依赖关系，通过byType/byName方式完成依赖注入；

5. **初始化与销毁**：执行Bean的初始化方法（afterPropertiesSet、init-method），容器关闭时执行销毁方法。

**真题2：Spring事务失效的常见场景有哪些？至少说出5种**

**满分标准答案**：

Spring事务基于AOP动态代理实现，所有失效场景本质都是**未触发AOP代理增强**或**事务规则不匹配**，常见场景如下：

1. 事务方法被private、static、final修饰，无法被代理拦截；

2. 同类中非事务方法调用本类事务方法，绕过代理直接调用原方法；

3. 事务方法内部捕获异常且未抛出，容器无法感知异常，不会回滚；

4. 抛出受检异常（Exception），默认事务不回滚（仅回滚RuntimeException）；

5. 数据库引擎不支持事务（如MyISAM），底层无事务能力；

6. 事务传播机制配置错误，导致嵌套事务无法回滚。

**真题3：Spring单例Bean为什么线程不安全？如何解决？**

**满分标准答案**：

1. 不安全原因：Spring单例Bean全局唯一，所有请求线程共享同一个Bean实例，若Bean中存在**可变成员变量**，多线程并发修改会出现数据覆盖、错乱，引发线程安全问题。

2. 解决方案：

- 核心方案：不定义可变成员变量，所有临时数据使用方法局部变量，实现线程隔离；

- 辅助方案：使用ThreadLocal存储线程专属变量，隔离多线程数据；

- 兜底方案：将Bean作用域修改为prototype多例，每个请求创建独立Bean实例。

## 10.4 手撕代码题：循环依赖、AOP、事务场景题

手撕代码题是中高级面试核心考点，重点考察开发者对Spring底层机制的落地能力，本小节提供3道高频手撕场景题，附带完整代码、问题分析、解决方案。

### 10.4.1 场景题1：Spring Bean循环依赖解决方案手撕

**场景描述**：A服务依赖B服务，B服务依赖A服务，形成循环依赖，启动报错Bean创建失败，请实现代码并解决问题。

**问题分析**：Spring单例Bean setter注入可解决循环依赖，构造器注入无法解决。核心原理是三级缓存提前暴露半成品Bean。

**错误代码（构造器注入，循环依赖报错）**

```java
@Service
public class AService {
    // 构造器注入，循环依赖无法解决
    private final BService bService;
    public AService(BService bService) {
        this.bService = bService;
    }
}

@Service
public class BService {
    private final AService aService;
    public BService(AService aService) {
        this.aService = aService;
    }
}
```

**正确解决方案（setter注入/字段注入，解决循环依赖）**

```java
@Service
public class AService {
    @Autowired
    private BService bService; // 字段注入，支持三级缓存解决循环依赖
}

@Service
public class BService {
    @Autowired
    private AService aService;
}
```

**核心总结**：Spring默认支持**单例Bean、非构造器注入**的循环依赖，无法解决构造器注入、多例Bean的循环依赖。

### 10.4.2 场景题2：手写AOP切面实现日志统一记录

**场景需求**：通过Spring AOP实现所有Service层方法的统一日志记录，记录方法名、入参、出参、执行耗时。

**完整实现代码**

```java
// 1. 自定义切面类
@Aspect
@Component
public class ServiceLogAspect {

    // 定义切入点：匹配所有Service层方法
    @Pointcut("execution(* com.example.demo.service.*.*(..))")
    public void servicePointCut(){}

    // 环绕通知：记录方法执行日志、耗时
    @Around("servicePointCut()")
    public Object around(ProceedingJoinPoint joinPoint) throws Throwable {
        // 获取方法名
        String methodName = joinPoint.getSignature().getName();
        // 获取入参
        Object[] args = joinPoint.getArgs();
        long startTime = System.currentTimeMillis();
        Object result = null;
        try {
            // 执行目标方法
            result = joinPoint.proceed();
            // 记录执行日志
            System.out.println("方法【"+methodName+"】执行成功，入参："+ Arrays.toString(args)
                    +"，执行耗时："+(System.currentTimeMillis()-startTime)+"ms");
        } catch (Throwable e) {
            System.err.println("方法【"+methodName+"】执行异常，异常信息："+e.getMessage());
            throw e;
        }
        return result;
    }
}
```

### 10.4.3 场景题3：事务失效场景修复手撕代码

**场景问题**：同类方法自调用导致事务失效，数据异常不回滚，修复该问题。

**错误代码（事务失效）**

```java
@Service
public class UserService {

    public void updateUser() {
        // 同类自调用，未走代理，事务失效
        doTrans();
    }

    @Transactional(rollbackFor = Exception.class)
    public void doTrans() throws Exception {
        // 数据库更新操作
        // 手动抛出异常，测试回滚
        throw new Exception("业务异常");
    }
}
```

**修复方案（获取代理类调用，触发事务）**

```java
@Service
public class UserService {

    // 注入自身代理类
    @Autowired
    private UserService userService;

    public void updateUser() throws Exception {
        // 通过代理类调用事务方法，触发AOP增强，事务生效
        userService.doTrans();
    }

    @Transactional(rollbackFor = Exception.class)
    public void doTrans() throws Exception {
        // 数据库更新操作
        throw new Exception("业务异常");
    }
}
```
