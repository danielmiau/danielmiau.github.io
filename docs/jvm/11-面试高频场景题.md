## 2. 基础认知与整体架构

### 2.1 什么是JVM？它的核心作用是什么？

**概念定义**：JVM（Java Virtual Machine，Java 虚拟机）是一个**虚构的、基于栈的字节码执行虚拟机**，是运行 Java 程序的专属运行环境，不属于操作系统，是介于代码和系统之间的中间层。

**核心原理**：Java 源代码编译为跨平台的 class 字节码文件，JVM 负责解析字节码、翻译为操作系统可识别的指令，屏蔽 Windows、Linux、Mac 系统差异，实现跨平台运行。同时提供内存管理、垃圾回收、安全校验、代码编译等核心能力。

**场景与价值**：解决不同操作系统的兼容性问题；自动管理内存，减少手动内存释放的代码成本；统一 Java 程序运行标准，保证所有平台运行行为一致。

**💡最佳实践**：JVM 并非只支持 Java 语言，基于 class 字节码的语言（Kotlin、Groovy、Scala）都可以运行在 JVM 之上。

### 2.2 请描述JVM的整体架构与各模块职责

标准 HotSpot JVM 分为五大核心模块：

1. **类加载子系统**：负责读取 class 字节码文件，加载、校验、解析、初始化类信息，将静态数据载入内存。

2. **运行时数据区**：内存核心区域，包含堆、栈、程序计数器、方法区、本地方法栈，存储程序运行期间所有数据。

3. **执行引擎**：包含解释器 + JIT 编译器，解释器实时解析字节码，编译器热点代码优化编译，混合执行兼顾启动速度与运行效率。

4. **本地方法接口**：对接操作系统本地 C/C++ 方法，实现文件操作、网络、系统调用等 Java 无法实现的底层能力。

5. **垃圾回收模块**：自动识别垃圾对象、回收空闲内存、整理内存碎片，避免内存泄漏与溢出。

### 2.3 JDK、JRE、JVM三者的区别与联系

**概念定义**：

- **JVM**：Java 虚拟机，仅负责运行字节码，是 Java 程序的运行容器。

- **JRE**（Java 运行环境）：包含 JVM + Java 核心类库，仅用于运行 Java 程序，无法开发编译代码。

- **JDK**（Java 开发工具包）：包含 JRE + 编译工具、调试工具、监控工具，面向开发者，支持开发、编译、运行、调试。

**包含关系**：JDK ⊃ JRE ⊃ JVM

### 2.4 常见面试场景题

**Q1：为什么Java被称为“平台无关的语言”？**
**参考答案**：Java 源代码编译后生成统一格式的 class 字节码文件，不绑定任何操作系统。不同平台安装对应版本 JVM，JVM 负责将字节码翻译为本地系统指令，实现「一次编译，到处运行」，因此具备平台无关性。

**Q2：说一下你对跨平台原理的理解**
**参考答案**：跨平台的核心本质是**字节码中间层隔离**。Java 不直接编译为系统机器码，而是编译为中间字节码，由各平台 JVM 适配系统差异，屏蔽底层系统区别，最终实现跨平台运行。

## 3. 运行时数据区与内存模型

### 3.1 JVM运行时数据区包含哪些部分？各自的作用？

**概念定义**：运行时数据区是 JVM 在程序运行时，在内存中开辟的独立内存空间，用于存储类信息、对象、变量、线程栈数据。

共五大区域：

- **程序计数器**：记录当前线程执行的字节码行号，线程私有，无内存溢出。

- **虚拟机栈**：存储方法栈帧、局部变量、操作数栈，线程私有，方法调用即入栈，执行结束即出栈。

- **本地方法栈**：作用和虚拟机栈一致，专门用于执行 native 本地方法。

- **堆内存**：所有对象实例、数组的存储区域，线程共享，是 GC 主战场。

- **方法区/元空间**：存储类元信息、常量、静态变量、即时编译代码，线程共享。

### 3.2 程序计数器为什么是JVM中唯一没有OOM的区域？

**核心原理**：程序计数器仅存储**当前线程执行的字节码地址**，占用内存极小且固定，不会随业务流量、对象创建增长。JVM 规范明确规定该区域无需垃圾回收、不会产生内存溢出，是整个 JVM 唯一绝对安全的内存区域。

### 3.3 虚拟机栈中会出现哪些异常？`StackOverflowError`和`OutOfMemoryError`分别在什么场景出现？

**StackOverflowError**：单个线程方法调用层级过深（无限递归、多层嵌套调用），栈帧数量超过栈容量，栈内存耗尽。

**OutOfMemoryError**：系统创建海量线程，每个线程独占独立栈内存，操作系统无法分配新内存，导致栈内存溢出。

### 3.4 堆内存的分代模型是什么？为什么要分代？

**概念定义**：JVM 将堆内存分为新生代、老年代，JDK8 之前包含永久代，JDK8 之后永久代移除。新生代分为 Eden、Survivor0、Survivor1。

**核心原理**：基于**弱分代假说**：绝大多数对象朝生夕灭，存活时间极短；少量对象长期存活。分代后针对不同生命周期对象使用不同回收算法，大幅提升 GC 效率、降低 STW 停顿。

### 3.5 方法区/元空间的区别？永久代为什么会被元空间替代？

**区别**：JDK7 永久代位于堆内存中，大小受限、容易溢出；JDK8 元空间使用**本地直接内存**，不在堆内，默认无上限。

**替换原因**：永久代大小难以预估，极易出现 Metaspace OOM；堆内存储类元信息浪费堆内存；本地内存更灵活、扩容成本更低。

### 3.6 直接内存是什么？为什么会出现直接内存溢出？

**概念定义**：直接内存是操作系统本地内存，不属于 JVM 堆，由 NIO/Netty 的 DirectBuffer 操作。

**溢出原因**：手动分配直接内存、未主动释放、框架资源泄漏，导致本地内存耗尽，触发 Direct buffer memory OOM。

### 3.7 常见面试场景题

**Q1：请说一下对象创建后在内存中的布局**
参考答案：对象分为对象头、实例数据、对齐填充。对象头存储标记字、类指针；实例数据存储成员变量；对齐填充保证对象内存地址 8 字节对齐。

**Q2：说一下你对逃逸分析的理解，它有什么作用？**
参考答案：逃逸分析是 JIT 编译优化手段，分析对象是否逃出当前方法。未逃逸对象可实现**栈上分配、标量替换、锁消除**，减少堆对象创建，大幅降低 GC 压力。

**Q3：为什么要将字符串常量池放到堆中？**
参考答案：JDK7 之前常量池在永久代，永久代空间小、极易溢出；移至堆中后空间更大、可被 GC 回收，减少内存溢出概率，提升内存利用率。

## 4. Java对象模型与内存布局

### 4.1 对象在内存中的布局由哪几部分组成？

64 位 JVM 下，Java 对象内存布局分为三部分：**对象头（Header）、实例数据（Instance Data）、对齐填充（Padding）**。全部存储在堆内存中。

### 4.2 什么是对象头？Mark Word中包含哪些信息？

**概念定义**：对象头是对象的元数据信息，分为 Mark Word 和 Klass Pointer。

**Mark Word 核心信息**：哈希码、GC 分代年龄、锁状态标志、偏向锁标识、线程持有锁信息，是 JVM 实现锁机制、GC 分代的核心依据。

### 4.3 什么是实例数据和对齐填充？为什么需要对齐填充？

**实例数据**：存储对象所有成员变量、基本数据类型、引用类型数据。

**对齐填充**：不属于业务数据，仅用于内存对齐。JVM 要求所有对象占用字节数必须是 8 的整数倍，不足则自动补位，提升 CPU 内存读取效率。

### 4.4 如何计算一个Java对象的大小？

64 位开启压缩指针（默认开启）：对象头 12 字节 + 实例数据 + 对齐填充；
64 位关闭压缩指针：对象头 16 字节 + 实例数据 + 对齐填充。

### 4.5 常见面试场景题

**Q1：new一个对象的过程是怎样的？**
参考答案：检查类是否加载 → 分配堆内存 → 对象属性默认初始化 → 对象头赋值 → 构造方法初始化 → 返回对象引用。

**Q2：为什么使用new Object()会占用16字节（64位JVM）？**
参考答案：开启压缩指针后对象头12字节，无实例数据，补齐4字节填充，合计16字节。

**Q3：说一下你对-XX:+UseCompressedOops的理解**
参考答案：压缩对象指针，将64位指针压缩为32位，节省堆内存占用，降低内存开销，JDK8 默认开启。

## 5. 类加载机制与双亲委派

### 5.1 什么是类加载？它的生命周期包含哪些阶段？

**概念定义**：类加载是 JVM 将 class 字节码文件读取到内存，转换为可直接使用的 java.lang.Class 对象的全过程。

**完整生命周期**：加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载。前五个阶段是固定加载流程。

### 5.2 加载、验证、准备、解析、初始化阶段分别做了什么？

- **加载**：读取字节码，生成 Class 对象。

- **验证**：校验字节码合法性、安全性，防止恶意代码。

- **准备**：为静态变量分配内存、赋默认初始值。

- **解析**：将符号引用替换为直接内存引用。

- **初始化**：执行静态代码块、静态变量赋值，是类加载最后一步。

### 5.3 什么是双亲委派模型？它的工作流程是什么？

**概念定义**：双亲委派是 JVM 默认的类加载规则：子类加载器收到加载请求，优先委派给父加载器，顶层父加载器无法加载，再由子类加载器加载。

**流程**：应用类加载器 → 扩展类加载器 → 启动类加载器，自上而下委派，自下而上加载。

### 5.4 为什么要使用双亲委派模型？

两大核心作用：**沙箱安全机制**，防止自定义核心类替换系统类；**保证类唯一性**，同一个类不会被多次重复加载。

### 5.5 如何打破双亲委派模型？举个例子

**核心原理**：重写类加载器 loadClass 方法，改变委派逻辑。典型场景：JDBC SPI 机制、Tomcat 自定义类加载器。

### 5.6 什么是自定义类加载器？它的应用场景是什么？

自定义类加载器是开发者继承 ClassLoader 实现的加载器，用于实现**热部署、代码加密、模块隔离、动态加载**，广泛用于中间件、框架、容器环境。

### 5.7 常见面试场景题

**Q1：说一下Tomcat的类加载机制，它是如何打破双亲委派的？**
参考答案：Tomcat 自定义 WebappClassLoader，优先自己加载 web 应用 jar 包类，再委派父加载器，实现应用类隔离、热加载，打破默认双亲委派。

**Q2：什么是类加载器的命名空间？不同加载器加载的类能否强转？**
参考答案：每个类加载器拥有独立命名空间，不同加载器加载的同名类相互隔离，无法强制类型转换。

**Q3：说一下你对SPI机制的理解，它和双亲委派有什么关系？**
参考答案：SPI 通过线程上下文类加载器打破双亲委派，实现接口定义在核心包、实现类在业务包的动态加载能力。

## 6. 垃圾回收核心原理

### 6.1 如何判断对象是否为垃圾？引用计数法和可达性分析的区别？

**引用计数法**：对象被引用计数+1，失效-1，计数为0判定为垃圾，存在**循环引用无法回收**的致命缺陷。

**可达性分析**：以 GC Roots 为起点遍历引用链，无法遍历到达的对象判定为垃圾，是 HotSpot 唯一使用的垃圾判定算法，解决循环引用问题。

### 6.2 什么是GC Roots？常见的GC Roots有哪些？

**概念定义**：GC Roots 是 JVM 认定的存活根节点对象，作为可达性分析的起点。

**常见GC Roots**：虚拟机栈局部变量、本地方法栈变量、静态变量、常量、锁对象、JVM 内部对象。

### 6.3 强引用、软引用、弱引用、虚引用的区别与应用场景？

**强引用**：默认引用，永不回收，直至程序结束，用于普通业务对象。

**软引用**：内存不足时回收，用于缓存场景。

**弱引用**：下次 GC 必回收，用于 ThreadLocal、缓存临时数据。

**虚引用**：无引用能力，仅用于监控对象回收状态，用于堆外内存释放。

### 6.4 对象的finalize()方法是什么？它的执行流程与局限性？

finalize 是对象被回收前的收尾方法，仅执行一次。存在执行时机不确定、性能差、无法保证资源释放的缺陷，**生产绝对禁止使用**。

### 6.5 标记-清除、复制、标记-整理三种算法的优缺点与适用场景？

- **标记清除**：速度快、产生内存碎片，适用于老年代（CMS）。

- **复制算法**：无碎片、效率高、占用双倍内存，适用于新生代。

- **标记整理**：无碎片、速度慢，适用于老年代（G1/ZGC）。

### 6.6 什么是分代回收？新生代和老年代分别采用什么算法？

分代回收依据对象存活周期差异化回收。新生代对象短命，使用复制算法；老年代对象长寿、数量少，使用标记清除/标记整理算法。

### 6.7 跨代引用问题是如何解决的？卡表和写屏障的作用？

**概念定义**：老年代对象引用新生代对象，Minor GC 需要扫描整个老年代，效率极低。

**卡表**：记录跨代引用的缓存表，无需全量扫描老年代。

**写屏障**：对象引用变更时动态更新卡表数据，保证数据准确性。

### 6.8 常见面试场景题

**Q1：为什么新生代要分为Eden和两个Survivor区？比例为什么是8:1:1？**
参考答案：解决复制算法内存浪费问题，保证短命对象快速回收，8:1:1 是 JVM 统计的最优对象存活比例，平衡内存利用率和回收效率。

**Q2：对象什么时候会进入老年代？**
参考答案：年龄达到阈值、大对象直接晋升、动态年龄判定、MinorGC 存活对象过多。

**Q3：什么是三色标记和并发标记的对象消失问题？**
参考答案：三色标记用于并发 GC 标记存活对象，并发阶段业务线程修改引用，会出现**对象消失**，通过读写屏障、增量更新、原始快照解决。

## 7. 主流垃圾收集器详解

### 7.1 请列举HotSpot中的常见垃圾收集器，并说明它们的分代模型？

经典分代收集器：Serial、ParNew、Parallel Scavenge、CMS；
区域化收集器：G1；
低延迟收集器：ZGC、Shenandoah。

### 7.2 Serial、Serial Old、ParNew、Parallel Scavenge、Parallel Old的特点与适用场景？

Serial 单线程收集器，适用于客户端、低流量服务；Parallel 系列主打**高吞吐量**，适用于离线批处理、数据分析任务。

### 7.3 CMS收集器的执行流程是什么？它的优缺点与淘汰原因？

**四大流程**：初始标记、并发标记、重新标记、并发清除。

**优点**：并发回收、低停顿；**缺点**：内存碎片、并发失败、CPU 占用高、无法处理超大堆。

**淘汰原因**：G1 性价比更高，CMS 碎片化严重、故障多、维护成本高。

### 7.4 G1收集器的核心设计理念是什么？它的执行流程与关键特性？

G1 将堆内存划分为多个大小相等的 Region 分区，面向**低延迟、可控停顿**，支持 Mixed GC，兼顾吞吐量和延迟，替代 CMS 成为 JDK9+ 默认收集器。

### 7.5 ZGC和Shenandoah收集器的核心技术与优势？

ZGC 基于染色指针、读屏障，实现**亚毫秒级停顿**，支持 TB 级大堆，是目前互联网低延迟微服务首选收集器。

### 7.6 不同业务场景下，如何选择合适的垃圾收集器？

- 低延迟交易服务：ZGC > G1

- 离线批处理任务：Parallel GC

- 老旧兼容项目：CMS

### 7.7 常见面试场景题

**Q1：CMS和G1的区别是什么？为什么G1能替代CMS？**
参考答案：CMS 基于传统分代、内存碎片严重、无停顿控制；G1 分区回收、可自定义最大停顿、内存整理、稳定性更强，全面替代 CMS。

**Q2：G1的Mixed GC和Full GC的区别？**
参考答案：Mixed GC 回收全部新生代+部分收益最高的老年代分区；Full GC 全堆回收，长时间 STW，属于故障级 GC。

**Q3：低延迟和高吞吐业务分别用什么收集器？**
参考答案：低延迟 ZGC/G1，高吞吐 Parallel GC。

## 8. 执行引擎与JIT编译

### 8.1 解释器和编译器的区别？为什么HotSpot要采用混合模式？

**解释器**：逐行解析字节码，启动快、执行慢；**编译器**：编译为机器码，启动慢、执行快。混合模式兼顾**快速启动+高效运行**。

### 8.2 什么是分层编译？它的层级划分与触发条件？

分层编译将编译分为多层，根据方法调用次数、循环次数动态触发编译优化，低频方法解释执行，高频方法深度编译优化。

### 8.3 JIT编译的核心优化技术有哪些？

包含逃逸分析、方法内联、循环展开、常量折叠、死代码消除、锁消除等核心优化手段。

### 8.4 什么是逃逸分析？它能带来哪些优化？

逃逸分析判断对象是否逃出当前方法，未逃逸对象可实现栈上分配、标量替换、锁消除，极大减少堆对象数量，降低 GC 压力。

### 8.5 方法内联的作用是什么？它的限制条件？

方法内联将短小方法直接嵌入调用处，减少方法调用栈开销，提升执行效率；仅针对短小、高频调用方法生效。

### 8.6 常见面试场景题

**Q1：为什么循环次数多的代码会被JIT优化？**
参考答案：循环代码属于热点代码，执行频次极高，JIT 编译优化收益最大，能够显著提升运行效率。

**Q2：什么是逆优化（Deoptimization）？什么时候会触发？**
参考答案：编译后的代码失效、假设条件不成立时，退回解释执行，保证代码正确性。

**Q3：说一下你对GraalVM和AOT编译的理解**
参考答案：AOT 提前编译，启动速度极快，适合云原生、微服务冷启动优化，弥补 JIT 启动慢的缺陷。

## 9. 参数配置、调优与故障排查

### 9.1 常用的JVM内存参数有哪些？`-Xms`、`-Xmx`、`-Xmn`、`-Xss`的作用？

**-Xms**：初始堆内存；**-Xmx**：最大堆内存；**-Xmn**：新生代大小；**-Xss**：单个线程栈大小。生产环境通常设置 Xms=Xmx，避免堆内存动态扩容损耗性能。

### 9.2 如何配置GC日志？常用的日志参数有哪些？

生产标准 GC 日志参数包含：打印详细 GC、打印时间戳、日志轮转、持久化输出，方便故障回溯分析。

```shell
# 生产标准GC日志配置
java -XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:gc.log -XX:+UseGCLogFileRotation -jar app.jar

```

### 9.3 线上出现OOM时，如何快速定位问题？

第一步区分 OOM 类型；第二步保留现场导出 dump 文件；第三步 MAT 分析泄漏对象与引用链；第四步定位代码问题；最后修复灰度验证。

### 9.4 频繁Full GC的常见原因与排查思路？

常见原因：内存泄漏、大对象分配、元空间不足、内存碎片、手动调用 System.gc()。排查 GC 日志、堆快照、业务代码，逐一排除根因。

### 9.5 高CPU占用的排查流程是什么？

top 定位进程 → top -H 定位线程 → jstack 导出栈信息 → 定位死循环、锁竞争、频繁 GC 问题。

### 9.6 常见面试场景题

**Q1：说一下你处理过的一次线上JVM故障，你的排查流程是什么？**
参考答案：按照止血恢复、保留现场、工具排查、根因分析、灰度修复、复盘预防的标准化流程回答，结合真实 OOM/FullGC 案例。

**Q2：如何给一个高并发业务的JVM做参数配置？**
参考答案：固定堆大小、使用 ZGC/G1 低延迟收集器、配置 GC 日志、限制元空间、开启容器适配参数，保证低抖动、低停顿。

**Q3：如何使用jstack和MAT分析堆Dump？**
参考答案：jstack 排查线程阻塞、死锁；MAT 解析堆快照，定位内存泄漏对象、引用链、大对象。

## 10. 综合场景题与手写题

### 10.1 手写：描述一次对象从创建到被回收的完整生命周期

类加载校验 → 内存分配 → 默认初始化 → 对象头赋值 → 构造初始化 → 进入新生代存活 → 多次 GC 存活晋升老年代 → 失去引用变为垃圾 → GC 标记回收 → 内存释放，完成完整生命周期。

### 10.2 场景题

**Q1：电商大促期间频繁超时，怀疑JVM问题如何排查？**
参考答案：查看监控确认 P99 超时、GC 停顿、CPU 使用率；分析 GC 日志查看 STW 时长；排查线程阻塞、锁竞争；定位大对象、内存抖动问题，优化 JVM 参数与业务代码。

**Q2：应用启动快，运行越久越慢，原因是什么？**
参考答案：大概率存在**内存泄漏**，对象持续累积无法回收，可用内存越来越少，GC 越来越频繁，导致服务逐渐卡顿。

**Q3：微服务频繁Full GC，老年代使用率高如何解决？**
参考答案：dump 堆快照排查泄漏对象；优化批量查询产生的大对象；调整晋升阈值；切换低延迟收集器；修复代码内存泄漏问题。

### 10.3 对比题

**Q1：CMS vs G1 vs ZGC 三者的核心区别**
参考答案：CMS 并发标记、碎片化严重、停顿不稳定；G1 分区回收、可控停顿、兼顾均衡；ZGC 亚毫秒级停顿、适配超大堆、极致低延迟。

**Q2：引用计数法 vs 可达性分析**
参考答案：引用计数简单高效、存在循环引用漏洞；可达性分析精准稳定，是商用 JVM 标准算法，牺牲少量性能换取稳定性。
