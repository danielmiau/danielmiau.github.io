# 05-垃圾回收核心原理

## 本章概述

在前四章中，我们系统学习了 JVM 运行时数据区、Java 对象内存模型、类加载机制，掌握了「对象如何创建、内存如何存储、类如何加载」的底层逻辑。而本章将解决 JVM 最核心、面试与生产调优最高频的知识点：**垃圾回收（GC，Garbage Collection）**。

Java 相比于 C/C++ 最大的优势就是**自动化内存管理**，开发者无需手动申请和释放内存，由 JVM 自动识别垃圾对象、回收内存、整理内存空间。但自动化不代表零成本，线上服务的 **接口卡顿、吞吐量下降、STW 停顿、OOM 内存溢出、频繁 GC** 等绝大部分性能问题，根源全部来自垃圾回收机制。

本章将从零讲解垃圾回收完整底层逻辑：垃圾判定算法、对象引用体系、对象死亡流程、三大经典回收算法、分代回收思想、卡表、写屏障、安全点、三色标记等核心底层技术，全覆盖**原理、实操、生产调优、故障排查、面试考点**，是后续学习各类垃圾收集器、线上 JVM 调优的核心前置基础。

### 1.2 垃圾回收的本质：自动内存管理

**概念定义**：垃圾回收的本质就是 **JVM 自动识别堆内存中无效、不再被使用的对象，释放其占用的内存空间，避免内存冗余与内存泄漏，实现自动化内存复用**。

**核心原理**：程序运行过程中会不断创建对象，方法执行结束、引用失效后，大量对象变成无效垃圾。如果不及时回收，堆内存会持续膨胀，最终导致内存溢出。GC 就是 JVM 内置的「内存清洁工」，持续清理无效对象，保证内存可用。

**生活化类比**：堆内存相当于房间，对象相当于房间内的物品，程序运行不断添置物品，GC 相当于保洁人员，定期清理废弃无用物品，保证房间空间充足。

### 1.3 垃圾回收的三大核心问题

所有垃圾回收机制，本质都是解决三个核心问题，也是本章全部内容的核心主线：

1. **怎么找垃圾**：如何精准判定对象是否失效（引用计数、可达性分析）

2. **怎么清垃圾**：如何回收无效对象（标记清除、复制、标记整理算法）

3. **怎么高效清垃圾**：如何减少停顿、提升吞吐量、降低内存碎片（分代、卡表、三色标记、安全点）

### 1.4 本章内容与后续章节的关联

本章是 JVM 调优体系的基石，承接前文对象模型、内存布局，开启后续高阶内容：

## 2. 垃圾回收的前置认知

### 2.1 为什么需要GC？——手动内存管理的痛点

**概念定义**：在 C/C++ 等原生语言中，内存需要开发者手动申请 `malloc`、手动释放 `free`，属于手动内存管理；而 Java 通过 GC 实现全自动内存管理。

**核心原理**：手动内存管理存在两大致命痛点：一是开发者容易遗忘释放内存，导致**内存泄漏**；二是容易重复释放、错误释放内存，导致**程序崩溃、野指针异常**。GC 的诞生就是为了解决人工操作的不稳定性。

**场景与价值**：Java GC 屏蔽了底层内存操作，降低开发门槛，让开发者专注业务逻辑，同时极大降低线上内存故障概率，保证服务稳定性。

### 2.2 JVM GC的优势与局限性

**优势**：自动化内存回收、无需人工干预、稳定性高、极大减少内存泄漏与程序崩溃问题。

**局限性**：GC 并不是免费的，垃圾回收过程需要占用 CPU 资源，且会触发 **STW（Stop-The-World，全局停顿）**，暂停所有用户线程，导致接口短暂卡顿，这也是 Java 服务需要 JVM 调优的根本原因。

**⚠️避坑指南**：很多开发者误以为 GC 是完全智能化、无需调优，实际上默认的 GC 策略仅适配通用场景，高并发、大流量业务必须手动调优，否则极易出现频繁 GC、接口超时、OOM 问题。

### 2.3 GC的作用范围：堆内存与方法区

**概念定义**：JVM GC **主要作用于堆内存**，次要作用于方法区（元空间），虚拟机栈、本地方法栈、程序计数器属于线程私有内存，线程结束自动释放，不会参与 GC。

**核心原理**：堆内存存放所有实例对象，是内存占用最大、垃圾最多的区域，是 GC 核心回收区域；方法区存放类元数据、静态常量，仅在类卸载时回收垃圾，回收频率极低。

### 2.4 GC性能目标：吞吐量、暂停时间、内存占用的“不可能三角”

**概念定义**：JVM 垃圾回收有三大核心性能指标，三者互相制约，无法同时最优，被称为 GC 不可能三角：

- **吞吐量**：用户线程运行时间 / 总运行时间，越高越好，代表业务执行效率高

- **暂停时间**：GC 停顿 STW 时长，越短越好，代表接口卡顿越少

- **内存占用**：JVM 运行常驻内存，越小越好，节省服务器资源

**核心原理**：想要低暂停时间，需要频繁 GC，会降低吞吐量；想要高吞吐量，需要减少 GC 次数，会增大单次停顿时间；内存压缩优化会消耗 CPU，影响吞吐量。三者只能取舍平衡。

**💡最佳实践**：互联网高并发业务优先保证**低暂停时间**，离线批量任务优先保证**高吞吐量**。

**📌面试高频题**：Q：简述 JVM GC 不可能三角？
参考答案：GC 的吞吐量、暂停时间、内存占用三者相互制约，无法同时最优。生产调优需根据业务场景取舍，在线高并发服务优先降低停顿时间，离线任务优先提升吞吐量。

## 3. 第一步：如何判定对象为“垃圾”？

### 3.1 引用计数算法

#### 3.1.1 核心原理：对象引用计数器

**概念定义**：引用计数算法是最简单的垃圾判定算法：每个对象头部维护一个**引用计数器**，记录当前对象被引用的次数。引用次数大于0，对象存活；引用次数为0，判定为垃圾对象，可被回收。

**核心流程**：对象被引用，计数器+1；引用失效、置为 null，计数器-1；计数器为0，标记为垃圾。

#### 3.1.2 优缺点与致命缺陷（循环引用）

**优点**：判定简单、实时性高、无需全局扫描、无 STW。

**缺点**：需要额外空间维护计数器、频繁增减计数器消耗 CPU、存在**循环引用致命缺陷**。

**循环引用场景**：对象 A 引用对象 B，对象 B 引用对象 A，无外部引用。两个对象计数器均为1，永远无法归零，导致垃圾对象常驻内存，造成**内存泄漏**。

#### 3.1.3 应用场景与JVM未采用的原因

Java HotSpot JVM **未使用引用计数算法**，核心原因就是无法解决循环引用内存泄漏问题。目前仅 Python、Objective-C 等语言使用该算法。

**实操示例：循环引用代码**

```java
public class ReferenceCountTest {
    // 定义内部类
    static class Node{
        Node next;
    }
    public static void main(String[] args) {
        Node a = new Node();
        Node b = new Node();
        // 互相引用，形成循环引用
        a.next = b;
        b.next = a;
        // 切断外部引用
        a = null;
        b = null;
        // 此时a、b对象互相引用，引用计数不为0，无法被回收
        System.gc();
    }
}

```

**验证与排错**：上述代码使用引用计数算法会内存泄漏，JVM 使用可达性分析可正常回收。

### 3.2 可达性分析算法（JVM主流实现）

#### 3.2.1 核心原理：从GC Roots出发的引用链追踪

**概念定义**：可达性分析是 HotSpot 官方默认的垃圾判定算法。核心逻辑：JVM 预设一系列**GC Roots（垃圾回收根节点）**，从根节点向下遍历所有引用链，**能够遍历到达的对象即为存活对象，无法遍历到达的对象即为垃圾对象**。

**生活化类比**：GC Roots 相当于大楼地基，存活对象是和地基相连的房间，没有任何链路连接地基的房间就是废弃垃圾，需要清理。

#### 3.2.2 什么是GC Roots？——JVM中的根节点集合

GC Roots 是绝对存活的对象集合，包含以下五类：

1. 虚拟机栈中局部变量引用的对象

2. 本地方法栈 JNI 引用的对象

3. 方法区中静态属性、常量引用的对象

4. 活跃线程对象

5. JVM 内部系统对象

#### 3.2.3 可达性分析的执行时机与性能影响

可达性分析需要全局扫描堆内存，遍历所有引用链，资源消耗高。因此 JVM 不会实时扫描，仅在**内存不足、触发GC**时执行。且为了保证引用链状态稳定，可达性分析期间需要暂停所有用户线程，触发 STW。

#### 3.2.4 根节点枚举的优化：安全点（Safepoint）机制

**概念定义**：安全点是用户线程执行过程中可以暂停、响应 GC 的特定位置。GC 不会随时随地暂停线程，只有线程运行到安全点时，才会挂起线程，开始根节点枚举与可达性分析。

**价值**：避免频繁线程暂停，减少不必要的 STW，大幅优化 GC 性能。

### 3.3 引用类型与对象的回收状态

Java 将对象引用分为**强、软、弱、虚**四种，不同引用类型决定对象的回收优先级，适配不同业务缓存场景。

#### 3.3.1 强引用：默认引用，GC永不回收

日常代码普通对象引用就是强引用，只要 GC Roots 可达，永远不会被回收，是导致内存泄漏最主要的原因。

```java
// 强引用，默认引用类型
Object obj = new Object();

```

#### 3.3.2 软引用（SoftReference）：内存不足时回收

内存充足时不回收，内存溢出风险时优先回收，适合用作**业务缓存**，兼顾缓存效果与内存安全。

```java
import java.lang.ref.SoftReference;
public class SoftRefTest {
    public static void main(String[] args) {
        // 软引用包裹对象
        SoftReference<Object> softRef = new SoftReference<>(new Object());
    }
}

```

#### 3.3.3 弱引用（WeakReference）：下一次GC必回收

无论内存是否充足，只要发生 GC，弱引用对象一定会被回收，典型应用：**ThreadLocal、WeakHashMap**，用于解决缓存内存泄漏。

#### 3.3.4 虚引用（PhantomReference）：仅用于跟踪回收

最弱的引用，无法通过引用获取对象，仅用于监听对象回收时机，用于**资源释放、内存监控**。

#### 3.3.5 引用队列（ReferenceQueue）的作用与使用场景

软引用、弱引用、虚引用可以绑定引用队列，对象被回收后，引用对象会进入队列，开发者可以监听队列，做资源收尾、日志记录、监控统计。

**💡最佳实践**：业务缓存优先使用软引用，临时缓存、防内存泄漏场景优先使用弱引用。

### 3.4 对象的“死亡”过程

#### 3.4.1 第一次标记：可达性分析判定为不可达

GC 触发后，通过可达性分析扫描堆内存，将 GC Roots 不可达的对象第一次标记为垃圾对象。

#### 3.4.2 第二次标记：`finalize()`方法的执行逻辑

第一次标记后，若对象重写了 `finalize()` 方法且未执行过，对象会被放入 F-Queue 队列，由低优先级线程执行 finalize 方法，开发者可以在该方法中重新建立引用，拯救对象。

#### 3.4.3 `finalize()`方法的争议与淘汰

`finalize()` 执行优先级极低、执行时机不确定、极易导致对象滞留内存、引发内存泄漏，**JDK9 已彻底废弃该方法**，生产环境禁止使用。

#### 3.4.4 对象被回收的最终判定条件

经过两次标记，且 finalize 方法未成功拯救对象，对象才会被判定为彻底死亡，GC 执行回收。

**📌面试高频题**：Q：Java 对象的回收过程是什么？对象一定会被一次回收吗？
参考答案：对象需要经过两次标记，第一次可达性分析判定不可达，第二次执行 finalize 方法尝试自救，自救失败后才会被回收，不会直接回收。

## 4. 经典垃圾回收算法详解

### 4.1 标记-清除算法（Mark-Sweep）

#### 4.1.1 执行流程：标记 → 清除

第一阶段标记：从 GC Roots 遍历所有对象，标记存活对象；第二阶段清除：遍历堆内存，回收所有未被标记的垃圾对象内存。

#### 4.1.2 优缺点分析

**优点**：算法逻辑简单、无需移动对象、实现成本低。

**缺点**：回收后产生大量**内存碎片**；两次全量扫描堆内存，执行效率低。

#### 4.1.3 适用场景：老年代CMS收集器

适合老年代：老年代对象存活率高、移动对象成本极高，无需频繁复制整理，因此 CMS 收集器底层采用标记清除算法。

### 4.2 复制算法（Copying）

#### 4.2.1 核心原理：分块复制存活对象

将内存划分为两块对等区域，一块区域使用，一块区域空闲。GC 时将当前区域存活对象全部复制到空闲区域，复制完成后清空当前区域所有内存，彻底清除垃圾。

#### 4.2.2 执行流程：标记 → 复制 → 清空原区域

标记存活对象、将存活对象有序复制到备用内存、清空原内存区域，保证内存绝对规整。

#### 4.2.3 优缺点分析

**优点**：无内存碎片、内存规整、回收效率极高。

**缺点**：需要预留一半空闲内存，**内存利用率仅有50%**，内存浪费严重。

#### 4.2.4 优化实现：HotSpot的Appel式回收（Eden + 两个Survivor）

HotSpot 未采用1:1内存划分，而是采用 8:1:1 的新生代内存模型：1块 Eden 区、2块 Survivor 区。每次使用 Eden + 一块 Survivor，另一块空闲，将内存利用率从50%提升至90%，极致优化复制算法缺陷。

#### 4.2.5 适用场景：新生代

新生代对象存活率极低、创建销毁频繁，复制少量存活对象成本极低，完美适配复制算法。

### 4.3 标记-整理算法（Mark-Compact）

#### 4.3.1 核心原理：标记存活对象 → 整理压缩

结合前两种算法优势，先标记存活对象，再将所有存活对象向内存一端移动、紧凑排列，最后直接清理边界外所有垃圾内存。

#### 4.3.2 执行流程：标记 → 整理 → 清除

标记存活对象、移动对象压缩内存、清除末端垃圾区域。

#### 4.3.3 优缺点分析

**优点**：内存无碎片、内存利用率100%。

**缺点**：需要大量移动对象、更新对象引用地址，CPU 开销大、速度最慢。

#### 4.3.4 适用场景：老年代Serial Old

适用于老年代高存活、碎片化严重场景，Serial Old 收集器底层使用标记整理算法。

### 4.4 三大核心算法对比与选型分析

|算法|优点|缺点|适用场景|
|---|---|---|---|
|标记-清除|实现简单、无需移动对象|产生大量内存碎片、效率低|老年代 CMS 收集器|
|复制算法|无碎片、回收效率极高|内存利用率低、需要预留空闲空间|新生代所有收集器|
|标记-整理|无碎片、内存利用率100%|对象移动成本高、速度最慢|老年代 Serial Old|
**📌面试高频题**：Q：新生代为什么用复制算法，老年代为什么不用？
参考答案：新生代对象存活率极低，复制少量存活对象成本低、速度快；老年代对象存活率极高，复制成本极高，因此老年代使用标记清除/标记整理算法。

## 5. 分代回收思想：为什么要分代？

### 5.1 分代假说：弱分代与强分代假说

**概念定义**：分代假说是 JVM 分代回收的理论基础，包含两条核心规律：

- **弱分代假说**：绝大多数对象都是朝生夕灭、短期存活

- **强分代假说**：存活越久的对象，越难被回收

**核心原理**：基于对象生命周期差异，对不同存活周期的对象采用不同回收算法，实现整体性能最优。

### 5.2 堆内存的分代模型：新生代 + 老年代

HotSpot 将堆内存划分为两大区域：

- **新生代**：存放新生、短期存活对象，采用复制算法，高频快速 GC

- **老年代**：存放长期存活、大对象，采用标记清除/整理算法，低频 GC

### 5.3 分代回收的优势：针对性选择算法，提升整体效率

如果全局使用单一算法，无法适配所有对象：新生代对象多、存活少适合复制算法；老年代对象存活多适合标记类算法。分代回收实现**因地制宜、差异化回收**，最大化 GC 整体吞吐量。

### 5.4 跨代引用问题与解决方案：卡表（Card Table）

**概念定义**：跨代引用指**老年代对象引用新生代对象**。新生代 GC 时需要扫描整个老年代查找引用，成本极高。JVM 通过**卡表**解决跨代引用扫描开销问题。

**核心原理**：将堆内存划分为多个 512 字节的卡片，卡表记录每一张卡片是否存在跨代引用。新生代 GC 仅扫描标记存在跨代引用的卡片，无需扫描整个老年代，极大减少扫描范围。

### 5.5 卡表的维护与优化：写屏障（Write Barrier）

当发生跨代引用赋值时，JVM 通过**写屏障**拦截赋值操作，实时更新卡表标记，保证卡表数据准确，是卡表机制的底层支撑。

**⚠️避坑指南**：写屏障会带来微小的 CPU 开销，是 GC 调优中不可避免的性能损耗，是并发 GC 的基础开销。

**📌面试高频题**：Q：什么是卡表？解决了什么问题？
参考答案：卡表是 JVM 维护的内存标记表，用于记录老年代到新生代的跨代引用，避免新生代 GC 扫描整个老年代，大幅降低 GC 扫描开销。

## 6. 垃圾回收的关键技术与优化

### 6.1 安全点（Safepoint）与安全区域（Safe Region）

#### 6.1.1 安全点的作用：GC暂停的触发时机

**概念定义**：安全点是线程代码执行中可以暂停、响应 GC 的固定位置。GC 不会随意中断线程，只有线程执行到安全点，JVM 才会挂起线程，触发 STW。

**核心价值**：保证线程执行状态稳定，避免代码执行一半中断导致数据异常，同时减少 GC 停顿次数。

#### 6.1.2 安全点的选择原则与实现方式

安全点优先设置在**循环末尾、方法返回、异常抛出**位置，保证不会长时间阻塞 GC，兼顾性能与稳定性。

#### 6.1.3 安全区域：解决线程长时间阻塞的问题

**概念定义**：安全区域是一段代码区间，线程进入该区域后不会修改引用数据，全程处于安全状态。用于解决线程休眠、阻塞、等待场景下无法到达安全点的问题。

### 6.2 写屏障（Write Barrier）

#### 6.2.1 写屏障的核心作用：拦截对象引用修改

写屏障是 JVM 内置的指令拦截机制，在对象引用赋值前后植入底层指令，拦截所有引用修改操作。

#### 6.2.2 写屏障与卡表的配合机制

发生跨代引用赋值时，写屏障触发，自动更新卡表标记，维护跨代引用状态，保证 GC 扫描精准。

#### 6.2.3 写屏障的性能开销与优化

写屏障属于额外指令，会轻微消耗 CPU，是并发 GC、卡表机制必须承担的性能损耗，属于空间换稳定、性能换精准的设计。

### 6.3 并发标记与三色标记算法

#### 6.3.1 三色标记的核心思想：白/灰/黑三色标记

三色标记是并发 GC 核心算法，通过三种颜色区分对象扫描状态：

- **白色**：未扫描，默认是垃圾对象

- **灰色**：已扫描自身，未扫描子引用，存活待定

- **黑色**：自身和所有子引用全部扫描完成，确定存活

#### 6.3.2 并发标记的执行流程

初始所有对象白色，从 GC Roots 开始遍历，根节点置黑，遍历到的对象置灰，逐层扫描，最终黑色存活、白色回收。

#### 6.3.3 并发标记的“对象消失”问题与解决方案

并发标记阶段用户线程与 GC 线程并发执行，可能出现**黑色对象引用白色对象、白色对象失去所有外部引用**，导致存活对象被误回收，即对象消失问题。

#### 6.3.4 增量更新 vs 原始快照（SATB）

- **增量更新**：拦截黑色对象新增白色引用，重新标记，CMS 收集器使用

- **SATB原始快照**：标记开始时快照所有引用，保证存活对象不丢失，G1、ZGC 使用

**📌面试高频题**：Q：并发GC的对象消失问题如何解决？两种方案的区别？
参考答案：通过增量更新和 SATB 快照解决。增量更新拦截新引用，修复漏标；SATB 基于扫描初始快照，保证对象不丢失，准确性更高，现代收集器普遍使用 SATB。
