# 06-主流垃圾收集器详解

## 本章概述

上一章我们系统掌握了垃圾回收底层核心原理，包含垃圾判定算法、三大基础回收算法、分代思想、卡表、写屏障、三色标记、STW、安全点等底层基础。所有的理论算法最终都会落地为具体的**垃圾收集器**，垃圾收集器就是 JVM 垃圾回收算法的具体工程实现。

JVM 并没有统一的垃圾回收实现，HotSpot 虚拟机提供了十余种垃圾收集器，从最古老的单线程收集器，到多线程吞吐量收集器、低延迟并发收集器，再到现代化的 ZGC、Shenandoah 亚毫秒级收集器。不同收集器基于不同算法、拥有完全不同的性能特征、适配不同的业务场景。

线上 JVM 调优的本质，**80% 的工作都是收集器选型 + 参数调优**。很多项目出现频繁 GC、接口抖动、OOM、吞吐量不足、延迟过高等问题，根本原因就是收集器选型错误、参数配置不合理。本章将从零讲解所有主流垃圾收集器的底层原理、执行流程、参数配置、优缺点、生产选型、故障排查，覆盖面试高频考点与生产落地实战。

## 1. 本章导读与知识地图

### 1.1 本章学习目标与核心考点

**学习目标**：掌握 HotSpot 全系列垃圾收集器的底层原理、执行流程、参数配置、优缺点；能够根据业务场景精准选型收集器；独立完成线上 GC 调优、GC 故障排查；彻底打通 GC 理论与生产落地。

**核心考点**：各收集器算法对应关系、CMS 四大阶段与缺陷、G1 Region 分区模型、混合 GC、ZGC 染色指针、收集器选型对比、各收集器 STW 时长、CMS 淘汰原因、G1 与 ZGC 区别。本章是 JVM 面试、调优实战**最高频、最重要**的章节。

### 1.2 垃圾收集器的评价维度：吞吐量、暂停时间、内存占用

**概念定义**：承接上一章 GC 不可能三角，所有垃圾收集器的性能全部由三大维度衡量，也是选型的唯一标准：

- **吞吐量**：用户业务线程执行时间 / 系统总运行时间。代表业务处理能力，越高越好，适合离线批量任务。

- **暂停时间（STW）**：GC 全局停顿时长，暂停期间所有用户线程停止运行。代表接口响应速度，越短越好，适合线上高并发业务。

- **内存占用**：GC 自身算法、数据结构占用的额外内存。占用越低，服务器资源利用率越高。

**核心原理**：三大维度互相制约，没有全能收集器。所有收集器的迭代升级，本质都是在不断平衡三者的取舍关系。

### 1.3 收集器与垃圾回收算法的对应关系

垃圾回收算法是理论，收集器是落地实现，一一对应关系如下：

- 复制算法：Serial、ParNew、Parallel Scavenge（新生代）

- 标记-清除算法：CMS（老年代）

- 标记-整理算法：Serial Old、Parallel Old（老年代）

- 复合算法：G1、ZGC、Shenandoah（整合多种算法，基于分区模型）

### 1.4 本章内容与上一章（GC核心原理）的关联

上一章是**底层理论基础**，本章是**工程落地实战**。上一章讲解的可达性分析、三色标记、写屏障、卡表、STW、分代假说，全部是本章所有收集器的底层依赖，没有理论基础就无法理解收集器的设计优劣与性能差异。

### 1.5 HotSpot收集器的发展脉络与分类体系

HotSpot 收集器迭代分为四个阶段，完整代表 JVM 垃圾回收的演进方向：

1. **单线程阶段**：Serial / Serial Old，最原始、简单、单线程回收

2. **吞吐量优先阶段**：Parallel 系列，多线程回收，主打高吞吐量

3. **低延迟阶段**：CMS、G1，主打缩短 STW，适配互联网在线业务

4. **极致低延迟阶段**：ZGC、Shenandoah，亚毫秒级停顿，适配超大堆、高并发低延迟核心业务

**📌面试高频题**：Q：JVM 垃圾收集器的演进趋势是什么？
参考答案：从单线程到多线程、从高停顿到低停顿、从分代固定回收到动态分区回收、从追求吞吐量到**极致平衡延迟与吞吐量**。

## 2. 单线程时代的经典收集器

### 2.1 Serial 收集器（新生代）

#### 2.1.1 核心原理：复制算法 + 单线程回收

**概念定义**：Serial 收集器是 JDK 初代、最基础的新生代垃圾收集器，是所有收集器的鼻祖。它是**单线程独占式收集器**，回收垃圾时仅有一个 GC 线程工作，同时暂停所有用户业务线程。

**核心原理**：新生代采用标准**复制算法**，依托 Eden + Survivor 分区模型，将存活对象复制到空闲 Survivor 区，清空原区域内存。全程单线程执行，无需多线程竞争、无需复杂并发逻辑，算法极简。

#### 2.1.2 执行流程：Stop-The-World（STW）机制

Serial 是典型的独占式 GC，完整执行流程：

1. 新生代内存占满，触发 Minor GC

2. **触发STW**，暂停所有 Java 用户线程

3. 单线程扫描新生代，标记存活对象

4. 复制存活对象至备用 Survivor 分区

5. 清空当前 Eden、Survivor 垃圾内存

6. STW 结束，恢复所有用户线程执行业务

#### 2.1.3 优缺点分析与适用场景

**优点**：算法简单、无线程竞争、内存消耗极小、无额外复杂数据结构开销、单线程执行效率高（针对小内存场景）。

**缺点**：全程 STW、单线程回收速度极慢、大内存场景停顿时间极长，完全无法适配高并发服务。

**适用场景**：客户端程序、桌面应用、嵌入式设备、内存极小的单机程序，是 JDK1.3 之前默认收集器。

#### 2.1.4 启动参数与日志分析

**实操示例：启动参数开启 Serial 收集器**

```shell
# 开启Serial新生代收集器 + Serial Old老年代收集器
# -XX:+UseSerialGC：启用单线程收集器组合
java -XX:+UseSerialGC -jar application.jar

```

**参数解释**：`-XX:+UseSerialGC` 全局开启经典单线程收集器组合，新生代 Serial、老年代 Serial Old。

**验证与排错**：启动项目后查看 GC 日志，日志包含 `DefNew` 标识，代表 Serial 新生代收集器生效。

**⚠️避坑指南**：生产服务**绝对禁止使用 Serial 收集器**，高并发场景会导致频繁长时间卡顿、接口超时，仅用于本地测试、客户端应用。

### 2.2 Serial Old 收集器（老年代）

#### 2.2.1 核心原理：标记-整理算法 + 单线程回收

**概念定义**：Serial Old 是 Serial 对应的老年代收集器，同样是单线程独占式收集器，专门负责老年代垃圾回收。

**核心原理**：老年代对象存活率极高，不适合复制算法，因此 Serial Old 采用**标记-整理算法**。先标记所有存活对象，再将存活对象统一向内存一端压缩整理，清除末端垃圾内存，保证老年代内存绝对规整、无内存碎片。

#### 2.2.2 Serial + Serial Old 组合的应用场景

该组合是最古老、最稳定的 GC 组合，整体开销极低、无并发异常、无内存碎片风险。仅适用于**低并发、小内存、客户端、嵌入式**场景，不适用于服务端线上业务。

#### 2.2.3 作为CMS失败时的备用收集器

**核心特性**：在 JDK 中，Serial Old 不仅仅是独立收集器，同时是 **CMS 并发失败、GC 异常时的兜底备用收集器**。当 CMS 并发回收失败、出现 Concurrent Mode Failure 时，JVM 会自动降级为 Serial Old 单线程 Full GC。

**⚠️避坑指南**：线上 CMS 降级 Serial Old 会触发长时间 STW，导致服务雪崩，是生产严重故障的高频诱因。

**📌面试高频题**：Q：Serial Old 收集器除了独立使用还有什么作用？
参考答案：作为 CMS 并发失败的兜底备用收集器，当 CMS 并发回收失败，自动降级为 Serial Old 单线程 Full GC。

## 3. 多线程时代的吞吐量优先收集器

### 3.1 ParNew 收集器（新生代）

#### 3.1.1 核心原理：Serial 的多线程版本

**概念定义**：ParNew 收集器是 Serial 收集器的**多线程升级版**，算法、内存模型、回收逻辑完全和 Serial 一致，唯一区别是将单线程回收改为多线程并行回收。

**核心原理**：基于复制算法，GC 时开启多条 GC 线程同时扫描、复制对象，缩短 GC 整体耗时，降低 STW 停顿时间，多核 CPU 下性能提升显著。

#### 3.1.2 与CMS收集器的适配性

ParNew 是**唯一可以和 CMS 老年代收集器配合使用**的新生代收集器。在 G1 诞生之前，「ParNew + CMS」是互联网项目最主流的低延迟 GC 组合。其他新生代收集器无法适配 CMS。

#### 3.1.3 执行流程与参数配置

**实操示例：ParNew 启动参数**

```shell
# 启用ParNew新生代 + CMS老年代组合
# -XX:+UseParNewGC：开启ParNew收集器
# -XX:+UseConcMarkSweepGC：开启CMS收集器
java -XX:+UseParNewGC -XX:+UseConcMarkSweepGC -jar application.jar

```

#### 3.1.4 适用场景与局限性

**适用场景**：JDK8 及以下、需要低延迟、高并发的互联网服务。

**局限性**：仅优化新生代，老年代依赖 CMS；多核 CPU 优势明显，单核 CPU 下多线程切换开销反而比 Serial 更慢。

### 3.2 Parallel Scavenge 收集器（新生代）

#### 3.2.1 核心目标：可控的吞吐量优先

**概念定义**：Parallel Scavenge 是**吞吐量专用收集器**，核心设计目标不是最小化停顿时间，而是最大化程序吞吐量，优先保证业务执行效率。

**核心原理**：同样基于复制算法、多线程回收，但是拥有独特的**自适应调优策略**，可以自动调整堆内存大小、新生代比例、GC 阈值，适配批量计算业务。

#### 3.2.2 关键参数：`-XX:MaxGCPauseMillis` 与 `-XX:GCTimeRatio`

**实操示例：吞吐量调优参数**

```shell
# Parallel收集器吞吐量优先配置
# -XX:MaxGCPauseMillis：最大GC停顿时间（毫秒）
# -XX:GCTimeRatio：GC时间占比，数值越大吞吐量越高
java -XX:+UseParallelGC -XX:MaxGCPauseMillis=200 -XX:GCTimeRatio=99 -jar application.jar

```

**参数详解**：

- `MaxGCPauseMillis`：限制单次 GC 最大停顿时间，收集器会自动适配内存大小满足该阈值

- `GCTimeRatio`：计算公式 1/(1+N)，99 代表 GC 占用时间不超过 1%

#### 3.2.3 自适应调节策略：`-XX:+UseAdaptiveSizePolicy`

**概念定义**：自适应策略是 Parallel 系列最大特色，开启后 JVM 自动动态调整新生代大小、Survivor 比例、晋升阈值，无需开发者手动配置，自动适配业务流量波动。

#### 3.2.4 与ParNew的核心区别

ParNew 主打**低停顿、适配CMS**；Parallel Scavenge 主打**高吞吐量、自适应调优**，二者侧重点完全不同。

### 3.3 Parallel Old 收集器（老年代）

#### 3.3.1 核心原理：标记-整理算法 + 多线程回收

Parallel Old 是 Parallel Scavenge 的老年代配套收集器，多线程并行执行**标记-整理算法**，内存无碎片、吞吐量极高。

#### 3.3.2 Parallel Scavenge + Parallel Old 组合

JDK8 默认收集器组合，整体性能稳定、吞吐量高、无需复杂调优，适合绝大多数后台服务。

#### 3.3.3 适用场景：后台计算型应用

离线计算、数据分析、批量任务、日志处理、大数据任务，这类业务对接口延迟不敏感，对吞吐量、任务处理速度敏感。

**📌面试高频题**：Q：ParNew 和 Parallel 收集器的区别？
参考答案：ParNew 偏向低延迟，适配 CMS；Parallel 偏向高吞吐量，自带自适应调优，是 JDK8 默认收集器。

## 4. 并发标记清除（CMS）收集器（老年代）

### 4.1 CMS的设计目标：低延迟、低暂停时间

**概念定义**：CMS（Concurrent Mark Sweep）是 Java 第一款真正意义上的**并发低延迟收集器**，打破了传统独占 GC 的模式，实现 GC 线程与用户业务线程并发执行，极大缩短 STW 全局停顿时间。

**核心原理**：将完整 GC 流程拆分，仅在初始标记、重新标记阶段短暂停顿用户线程，其余阶段全部并发执行，大幅降低服务卡顿，适配互联网高并发在线业务。

### 4.2 CMS的执行流程详解

#### 4.2.1 初始标记（Initial Mark）

**特点：短暂STW**。仅扫描 GC Roots 直接关联的老年代对象，标记速度极快，停顿时间极短。

#### 4.2.2 并发标记（Concurrent Mark）

**特点：无STW，并发执行**。GC 线程与用户线程同时运行，从初始标记对象遍历全部引用链，完成所有老年代对象标记。

#### 4.2.3 重新标记（Remark）

**特点：短暂STW**。修复并发标记阶段用户线程修改引用导致的标记偏差，修正漏标、错标对象，保证标记准确性。

#### 4.2.4 并发清除（Concurrent Sweep）

**特点：无STW，并发执行**。清除所有标记为垃圾的对象，不移动存活对象，因此会产生内存碎片。

#### 4.2.5 并发重置（Concurrent Reset）

重置 CMS 标记状态、数据结构，等待下一次 GC 触发。

### 4.3 CMS的核心缺陷

#### 4.3.1 内存碎片化问题

CMS 基于标记-清除算法，**不整理内存**，长期运行会产生大量内存碎片，导致大对象无法分配，触发频繁 Full GC。

#### 4.3.2 并发失败与浮动垃圾

**浮动垃圾**：并发清除阶段用户线程持续创建对象，新增垃圾无法被本次 GC 回收，只能等待下次 GC。

**并发失败**：若并发回收期间内存占满，会触发 Concurrent Mode Failure，自动降级为 Serial Old 单线程 Full GC，服务严重卡顿。

#### 4.3.3 对CPU资源的占用

并发阶段持续占用 CPU 线程资源，高并发场景会轻微降低业务吞吐量。

### 4.4 CMS的关键参数调优

#### 4.4.1 `-XX:CMSInitiatingOccupancyFraction`

设置老年代 GC 触发阈值，默认 68，代表老年代占用率达到 68% 触发 CMS。内存大的场景可适当调低，提前 GC 避免溢出。

#### 4.4.2 `-XX:+UseCMSCompactAtFullCollection`

开启 Full GC 后内存整理，解决 CMS 内存碎片问题。

#### 4.4.3 `-XX:CMSFullGCsBeforeCompaction`

设置多少次 Full GC 后执行一次内存压缩整理，平衡性能与碎片问题。

**实操完整配置**

```shell
# CMS生产经典调优参数
java -XX:+UseParNewGC -XX:+UseConcMarkSweepGC \
-XX:CMSInitiatingOccupancyFraction=70 \
-XX:+UseCMSCompactAtFullCollection \
-XX:CMSFullGCsBeforeCompaction=3 \
-jar application.jar

```

### 4.5 CMS的淘汰与替代方案

CMS 在 JDK9 被标记废弃、JDK14 彻底移除。核心原因：内存碎片严重、并发失败风险高、调优复杂、无法适配大堆内存。官方替代方案为 **G1、ZGC**。

**📌面试高频题**：Q：CMS 为什么被淘汰？
参考答案：存在内存碎片、浮动垃圾、并发降级风险、调优成本高、不适合大内存堆，性能稳定性不如 G1/ZGC。

## 5. G1 收集器（Garbage-First）

### 5.1 G1的设计理念：面向服务端、平衡吞吐量与低延迟

**概念定义**：G1 是一款**面向服务端、可预测停顿、平衡延迟与吞吐量**的通用型垃圾收集器，是 JDK9 之后的默认收集器，用来全面替代 CMS。

**核心原理**：摒弃传统连续分代模型，将堆内存拆分为大量独立小分区，优先回收垃圾最多的分区，最大化回收效率、最小化停顿时间。

### 5.2 G1的核心创新：基于Region的堆内存模型

#### 5.2.1 分区（Region）的设计与作用

G1 将整个堆内存划分为多个大小统一的**Region（分区）**，默认 1MB~32MB，每个分区独立管理、独立回收，不再是整块新生代/老年代统一回收。

#### 5.2.2 分代的逻辑概念（新生代/老年代/大对象Humongous）

G1 物理上不分代，仅逻辑区分：新生代 Region、老年代 Region、Humongous 大对象 Region（占用连续多个分区），专门存储超大对象，避免大对象直接进入老年代引发 Full GC。

#### 5.2.3 记忆集（Remembered Set）与卡表的优化

G1 基于**记忆集**优化传统卡表，精准记录跨分区引用，GC 时仅扫描关联分区，彻底解决全堆扫描开销，大幅提升并发性能。

### 5.3 G1的回收流程详解

#### 5.3.1 初始标记（Initial Marking）

短暂 STW，标记 GC Roots 直达对象，速度极快。

#### 5.3.2 并发标记（Concurrent Marking）

GC 与业务线程并发，遍历所有引用链，标记全堆存活对象。

#### 5.3.3 最终标记（Final Marking）

短暂 STW，修正并发标记误差，处理剩余引用变化。

#### 5.3.4 筛选回收（Live Data Counting and Evacuation）

根据预设停顿时间，筛选垃圾最多、收益最高的分区回收，复制存活对象，压缩内存，无内存碎片。

### 5.4 G1的关键技术细节

#### 5.4.1 混合回收（Mixed GC）

G1 最核心特性，一次 GC 同时回收部分新生代 + 部分老年代分区，替代传统独立的 Minor GC、Full GC，减少 GC 次数。

#### 5.4.2 预测模型与停顿时间控制

G1 内置停顿时间预测模型，开发者指定最大允许停顿时间，G1 自动计算每次回收多少分区，精准控制 STW 时长。

#### 5.4.3 大对象（Humongous Object）的处理

超大对象直接分配在专门的 Humongous 分区，单独回收，避免老年代内存碎片化与频繁 Full GC。

### 5.5 G1的关键参数配置

**实操完整配置**

```shell
# G1生产通用调优参数
java -XX:+UseG1GC \
-XX:MaxGCPauseMillis=100 \
-XX:G1HeapRegionSize=4m \
-XX:InitiatingHeapOccupancyPercent=45 \
-jar application.jar

```

**参数详解**：

- `-XX:+UseG1GC`：开启 G1 收集器

- `MaxGCPauseMillis`：目标最大停顿时间 100ms

- `G1HeapRegionSize`：设置分区大小4MB

- `InitiatingHeapOccupancyPercent`：堆占用率45%触发GC

### 5.6 G1的优缺点与适用场景

**优点**：可控停顿时间、无内存碎片、兼顾吞吐量与延迟、适配中大型堆内存、稳定性远超 CMS。

**缺点**：小内存场景吞吐量略低于 Parallel 收集器、并发阶段存在轻微 CPU 开销。

**适用场景**：绝大多数互联网服务、中大型堆内存、对延迟和吞吐量均有要求的通用业务。

**📌面试高频题**：Q：G1 和 CMS 的核心区别？
参考答案：CMS 基于传统分代，存在内存碎片、无法精准控制停顿；G1 基于分区模型、支持混合 GC、精准预测停顿时间、无内存碎片，稳定性更强。

## 6. 新一代低延迟收集器：ZGC 与 Shenandoah

### 6.1 ZGC 收集器（Z Garbage Collector）

#### 6.1.1 设计目标：亚毫秒级停顿、TB级堆支持

**概念定义**：ZGC 是 JDK11 推出的革命性低延迟垃圾收集器，核心目标：**无论堆内存多大（GB~TB级），GC 停顿时间控制在 10ms 以内**，几乎无感 GC。

#### 6.1.2 核心技术：染色指针（Colored Pointers）

ZGC 将标记状态、引用状态直接存储在对象指针高位，无需遍历堆内存标记，无需复杂扫描，极大降低标记开销，是 ZGC 低延迟的核心基石。

#### 6.1.3 读屏障（Load Barrier）与并发转移

ZGC 基于**读屏障**替代传统写屏障，实现对象并发转移、并发压缩，所有耗时操作全部并发执行，仅保留极短的初始化 STW。

#### 6.1.4 ZGC的执行流程

几乎全程并发，仅初始标记、最终标记极短暂停顿，对象标记、转移、压缩、清理全部与业务线程并发执行。

#### 6.1.5 ZGC的发展历程（JDK11 → JDK17 → JDK21）

- JDK11：实验性 ZGC，仅支持 Linux

- JDK17：正式转正，性能大幅优化

- JDK21：功能完备、稳定性极高，可大规模生产落地

#### 6.1.6 适用场景与局限性

**适用场景**：金融、支付、交易、核心链路、超大堆、极致低延迟业务。

**局限性**：CPU 开销略高、JDK 版本要求高、极致追求吞吐量场景不如 Parallel。

### 6.2 Shenandoah 收集器

#### 6.2.1 设计目标：与ZGC类似的低延迟

Shenandoah 由 RedHat 开源，设计目标与 ZGC 一致，主打亚毫秒级低停顿，弱化分代、全程并发回收。

#### 6.2.2 核心技术：连接矩阵（ Brooks Pointer ）

不使用染色指针，通过 Brooks 指针转发机制实现对象并发转移，解决对象移动后的引用寻址问题。

#### 6.2.3 与ZGC的核心区别

ZGC 基于染色指针，内存开销更低、性能更好；Shenandoah 兼容性更好，支持更低 JDK 版本。

#### 6.2.4 适用场景与局限性

适用于需要低延迟、无法升级高版本 JDK 的服务，整体生态与稳定性弱于 ZGC。

**📌面试高频题**：Q：ZGC 为什么能做到亚毫秒级停顿？
参考答案：基于染色指针、读屏障、全程并发标记转移，极大缩短 STW，几乎所有耗时操作全部并发执行。

## 7. 收集器选型对比与实战建议

### 7.1 主流收集器核心指标对比表

|收集器|分代模型|算法|停顿时间|吞吐量|适用场景|
|---|---|---|---|---|---|
|Serial/Serial Old|分代|复制/标记-整理|高|低|客户端/嵌入式|
|ParNew/CMS|分代|复制/标记-清除|低|中|JDK8及以下低延迟互联网应用（已淘汰）|
|Parallel Scavenge/Old|分代|复制/标记-整理|中|高|后台计算/批处理/离线任务|
|G1|分区+逻辑分代|标记-复制|低|中高|服务端通用场景、JDK9+ 默认|
|ZGC/Shenandoah|不分代|并发转移|极低(亚毫秒)|中|低延迟、TB级大堆、核心交易服务|
### 7.2 如何根据业务场景选择收集器

#### 7.2.1 低延迟优先（如互联网服务）

JDK8 使用 ParNew+CMS；JDK11+ 优先 G1；核心交易、高敏感延迟业务优先 **ZGC**。

#### 7.2.2 高吞吐量优先（如离线计算）

全部使用 Parallel 系列收集器，最大化任务处理速度，无需关注短暂延迟。

#### 7.2.3 大内存堆场景（如TB级堆）

G1 适合数 GB~数十 GB 堆；**ZGC 适配百 GB~TB 级超大堆**。

#### 7.2.4 JDK版本对收集器的支持差异

- JDK8：默认 Parallel，支持 CMS、G1

- JDK11：默认 G1，实验性 ZGC

- JDK17+：ZGC 稳定可用，全面替代 G1/CMS

**💡最佳实践**：新项目 JDK17+ 直接使用 ZGC；老项目 JDK8 优先 G1，不建议继续使用 CMS。

## 8. 本章总结与高频考点梳理

### 8.1 核心知识回顾：收集器分类、算法、执行流程

垃圾收集器迭代经历单线程独占、多线程吞吐量、并发低延迟、极致低延迟四个阶段。Serial 极简稳定、Parallel 主打吞吐量、CMS 初代低延迟但缺陷众多、G1 通用均衡、ZGC/Shenandoah 实现亚毫秒级极致延迟。不同收集器对应不同回收算法、不同内存模型，适配差异化业务场景。

### 8.2 高频面试考点清单

CMS 五大阶段、CMS 缺陷与淘汰原因、G1 分区模型与混合 GC、ZGC 染色指针、各收集器 STW 差异、收集器选型标准、ParNew 与 Parallel 区别。

### 8.3 收集器选型的核心原则

离线批处理选吞吐量、在线业务选低延迟、超大堆选 ZGC、通用场景选 G1、老旧项目兼容 G1 过渡。

### 8.4 本章内容在调优实战中的指导意义

所有线上 GC 调优，第一步永远是**正确选择收集器**，参数调优只是辅助。收集器选型错误，后续所有参数调优都无法解决底层性能缺陷。
