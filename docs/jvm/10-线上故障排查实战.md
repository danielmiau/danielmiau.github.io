# 10-线上故障排查实战

## 本章概述

前面九章我们系统性学习了 JVM 底层原理、内存模型、垃圾回收机制、JVM 启动参数、性能监控工具、性能调优方法论。所有理论、工具、调优方案，最终目的只有一个：**解决线上真实 Java 服务故障，保障生产稳定性**。

线上环境和测试环境完全不同：流量不可预测、环境复杂、依赖繁多、故障突发、影响用户、容错率极低。测试环境可以慢慢调试分析，而线上故障要求工程师**快速止血、精准定位、彻底根治**。绝大多数初级开发只会写业务代码，不会排查线上故障，而**线上故障排查能力是高级开发、架构师的核心分水岭**。

本章作为 JVM 系列收尾实战章节，将聚焦企业生产高频故障，包含 OOM 内存溢出、频繁 GC/Full GC、CPU 飙高、线程阻塞死锁、服务卡顿超时五大核心故障场景。统一标准化排查流程、手把手工具实操、复盘真实企业案例、总结排查误区与预防预案，帮助读者构建**标准化、体系化、可落地**的线上故障排查思维，同时全覆盖面试故障类真题，兼顾面试通关与生产落地。

**核心考点**：四类 OOM 溢出区别与排查方案、频繁 Full GC 根因、CPU 100% 排查链路、线程死锁与阻塞定位、服务 STW 卡顿原因、线上故障排查原则、内存泄漏完整排查流程、故障恢复与复盘规范。本章是**面试实操问答、简历项目亮点、工作核心能力**的必考章节。

### 1.2 线上故障的特点与排查难点（环境不可控、影响范围广、时间压力大）

**概念定义**：线上故障指生产环境 Java 服务出现异常，导致接口超时、报错、服务不可用、用户体验下降的各类问题。

**核心难点拆解**：

- **环境不可控**：生产流量随机、用户行为不可控、服务器资源、中间件、网络环境复杂，无法复现测试环境稳定场景。

- **影响范围广**：单实例故障可能扩散至整个集群，影响全站用户、产生资损。

- **时间压力大**：线上故障分秒必争，超时越久资损越高，对排查速度、精准度要求极高。

- **现场易丢失**：重启服务、扩容、回滚操作会直接销毁故障现场，导致无法定位根因。

**生活化类比**：测试环境修车相当于「修理厂静态检修」，环境可控、时间充足；线上故障修车相当于「高速路上车辆抛锚」，车流密集、风险极高、必须快速修复。

### 1.3 线上故障排查的通用方法论

所有线上故障统一遵循标准化闭环方法论：**故障感知 → 快速止血 → 现场保留 → 分层排查 → 根因定位 → 灰度修复 → 监控验证 → 复盘预防**。杜绝盲目操作，保证每一次故障可追溯、可修复、可预防。

### 1.4 本章内容与调优、工具章节的关联

本章是前序所有章节的**最终落地收口**：依托 JVM 原理判断故障底层逻辑，依托监控工具采集故障数据，依托调优方案完成故障修复与性能优化。没有原理和工具基础，故障排查只能靠猜；只学原理工具不练实战，无法应对真实生产复杂场景。

**📌面试考点**：Q：线上故障和测试环境 bug 最大的区别是什么？
参考答案：线上故障具备**不可控、高影响、高压力、现场易丢失**的特点，需要优先恢复服务、保留现场，再定位根因；测试环境无用户影响，可长时间调试复现。

## 2. 线上故障排查的前置流程与原则

### 2.1 故障分级与响应流程（P0-P4故障定义）

**概念定义**：企业生产会将故障分级，不同级别对应不同响应速度、处理标准、复盘要求，避免小故障过度消耗人力、大故障响应不及时。

**企业通用故障分级标准**：

- **P0 致命故障**：全站服务不可用、核心交易中断、产生直接资损，响应时间 0 分钟，立即全员介入。

- **P1 严重故障**：部分核心功能不可用、大量用户报错，5 分钟内响应，15 分钟内止血。

- **P2 一般故障**：非核心功能异常、少量用户报错，30 分钟内响应修复。

- **P3 轻微故障**：后台任务、管理端异常，不影响用户，工作日修复。

- **P4 优化类问题**：性能瑕疵、日志不规范、无报错无资损，迭代优化。

**场景与价值**：规范团队故障处理节奏，保障核心故障优先处理，统一团队运维标准。

### 2.2 故障排查的核心原则：先恢复、后定位，先整体、后局部

**概念定义**：线上故障第一优先级是**恢复用户服务、降低资损**，而非定位 bug。很多初级工程师容易陷入「纠结根因、忘记止血」的误区，导致故障持续扩大。

**核心原理**：线上故障时间成本极高，重启、扩容、熔断、降级、切流量可以快速恢复服务；待服务稳定、用户恢复正常后，再通过日志、快照复盘定位底层根因。

**💡最佳实践**：标准处理顺序：**止血恢复 > 保留现场 > 根因排查 > 优化复盘**。

### 2.3 排查前的准备工作：日志、监控、权限、备份

故障排查前必须完成四项准备工作，否则极易丢失关键证据：

1. **采集监控数据**：导出 CPU、内存、GC、QPS、报错率监控曲线，确认故障起止时间。

2. **备份日志文件**：拷贝应用日志、GC 日志、系统日志，避免日志轮转覆盖丢失。

3. **校验服务器权限**：确保拥有进程查看、文件导出、命令执行权限。

4. **备份故障现场**：导出线程栈、堆快照，禁止直接重启进程。

### 2.4 故障复盘的重要性：事后分析与预防

**概念定义**：故障复盘是故障解决后的总结动作，用于定位人为、代码、流程、架构问题，避免同类故障重复发生。

**核心价值**：线上故障可以允许发生一次，但绝对不允许重复发生。排查解决故障只能解决当下问题，复盘才能根治问题。

**复盘核心维度**：故障现象、故障时长、影响范围、直接原因、根本原因、排查短板、优化方案、预防预案。

**📌面试考点**：Q：线上故障第一处理原则是什么？为什么？
参考答案：先恢复、后定位。线上故障核心损失是用户流失与资金损耗，优先通过重启、降级、扩容恢复服务，止血止损，稳定后再排查根因。

## 3. 线上高频故障场景实战

### 3.1 OOM（内存溢出）故障排查

OOM 是线上最高发、最经典、面试必考的故障，本质是**内存资源耗尽，JVM 无法分配新内存**，直接导致进程崩溃、服务重启。OOM 分为四大类型，每种类型根因、场景、排查方式完全不同。

#### 3.1.1 常见OOM场景与现象

**1. java.lang.OutOfMemoryError: Java heap space 堆溢出**

**概念定义**：Java 堆内存耗尽，无法创建新对象实例。

**现象**：服务频繁重启、GC 频率极高、老年代内存占满、接口大量报错。

**根因**：内存泄漏、批量查询超大数据、无限创建对象、静态集合持续累积对象。

**2. java.lang.OutOfMemoryError: Metaspace 元空间溢出**

**概念定义**：JDK8+ 元空间内存耗尽，无法加载新的类、方法、注解、代理类数据。

**现象**：服务运行一段时间后崩溃、热部署频繁报错、动态代理异常。

**根因**：热部署框架、动态代理、反射生成大量临时类、元空间未配置上限。

**3. java.lang.OutOfMemoryError: Direct buffer memory 直接内存溢出**

**概念定义**：堆外直接内存耗尽，NIO/Netty 缓冲区无法分配内存。

**现象**：网络请求超时、网关服务崩溃、Netty 报错缓冲区分配失败。

**根因**：DirectBuffer 对象未释放、Netty 资源泄漏、未限制直接内存上限。

**4. java.lang.StackOverflowError 栈溢出**

**概念定义**：单个线程栈内存耗尽，方法栈帧过多。

**现象**：单接口固定报错、线程崩溃、无全局服务宕机。

**根因**：方法无限递归、复杂嵌套调用、线程栈设置过小。

#### 3.1.2 排查流程

**标准化OOM排查五步流程**

1. **确认OOM类型与错误日志**：根据报错关键字区分堆/元空间/直接内存/栈溢出。

2. **保留故障现场**：立即导出 heapdump、线程栈、GC 日志，禁止重启服务。

3. **分析Heap Dump**：使用 MAT 工具查看大对象、存活对象、引用链。

4. **定位泄漏源头**：区分是代码泄漏、参数配置不足、第三方框架问题。

5. **修复验证**：优化代码、调整 JVM 参数、压测验证，上线修复。

#### 3.1.3 典型案例

- **ThreadLocal内存泄漏**：线程池线程复用，ThreadLocal 存储对象未调用 remove()，对象持续累积无法回收。最佳实践：使用 finally 强制清除 ThreadLocal 数据。

- **静态集合/缓存泄漏**：static Map/List 全局集合，程序生命周期内永不销毁，持续新增对象导致堆溢出。

- **第三方库泄漏**：老旧版本 Netty、Redis 客户端存在已知内存泄漏 bug，升级依赖版本即可修复。

- **批量数据溢出**：数据库全表查询、批量导出未分页，一次性加载百万级数据到内存。

**⚠️避坑指南**：遇到 OOM 绝对不要第一时间重启服务，重启会清空堆内存，丢失所有故障现场，导致无法定位根因，故障反复复现。

### 3.2 频繁GC/Full GC故障排查

#### 3.2.1 故障现象：接口超时、CPU飙高、系统卡顿

**概念定义**：频繁 GC 分为频繁 Young GC 和频繁 Full GC，是线上最隐蔽、最容易被忽视的故障。不同于直接宕机，该故障不会导致服务崩溃，但会持续造成**接口抖动、P99 超时、CPU 居高不下**。

**核心原理**：JVM 不断消耗资源执行垃圾回收，用户线程频繁被 STW 暂停，业务线程无法正常执行，导致整体服务吞吐量下降、延迟升高。

#### 3.2.2 排查流程

1. **查看GC日志**：统计 GC 次数、单次停顿、GC 类型，区分是年轻代还是老年代 GC。

2. **分析内存趋势**：通过监控查看 Eden、Old 区内存上涨速度，判断对象生产速率。

3. **定位根因**：排查大对象、内存泄漏、元空间不足、手动 GC、内存碎片。

4. **参数校验**：核对堆内存比例、晋升阈值、收集器参数是否适配业务。

#### 3.2.3 典型案例

- **大对象导致Full GC**：业务批量生成超大对象，超过 Survivor 区域大小，直接晋升老年代，快速占满老年代触发 Full GC。

- **元空间类加载过多**：动态生成大量代理类，元空间占满，JVM 触发 Full GC 尝试回收类数据。

- **CMS内存碎片并发失败**：CMS 标记清除算法产生大量内存碎片，老年代剩余空间充足但无法分配连续大对象，导致并发失败、退化为 Full GC。

- **高峰期Young GC频繁**：接口高并发，每秒创建大量短命对象，Eden 区快速占满，频繁触发 Minor GC，造成轻微接口抖动。

### 3.3 高CPU占用故障排查

#### 3.3.1 故障现象：CPU使用率持续100%，系统响应缓慢

**概念定义**：Java 进程 CPU 持续打满，CPU 资源耗尽，操作系统无法分配时间片给业务线程，所有接口响应变慢、超时率飙升。

**核心原理**：CPU 高占用只有两类原因：**业务线程死循环计算**、**JVM GC 持续回收**。

#### 3.3.2 排查流程

**企业标准六步排查法**

1. `top`：定位服务器 CPU 最高的 Java 进程 PID。

2. `top -H -p pid`：查看进程内所有线程，定位 CPU 最高线程 TID。

3. `printf "%x\n" TID`：将线程 ID 转为十六进制。

4. `jstack pid > thread.log`：导出线程快照。

5. 检索十六进制线程 ID，定位对应代码行。

6. 分析代码逻辑，修复死循环、锁竞争、GC 问题。

**实操示例**

```shell
# 1. 查看高cpu进程
top
# 2. 查看进程内高cpu线程
top -H -p 10086
# 3. 线程id转16进制
printf "%x\n" 10090
# 4. 导出线程栈
jstack 10086 > cpu-thread.log

```

#### 3.3.3 典型案例

- **业务代码死循环**：while 循环无休眠、集合遍历死循环、递归无终止条件。

- **频繁Full GC飙高CPU**：GC 线程持续占用 CPU 资源，业务线程无法执行。

- **锁竞争自旋**：大量线程竞争同一把锁，CAS 自旋消耗大量 CPU。

- **正则回溯CPU爆满**：复杂正则表达式匹配特殊字符串，出现灾难性回溯，占用满 CPU。

### 3.4 线程阻塞/死锁故障排查

#### 3.4.1 故障现象：接口超时、线程数飙升、服务不可用

**概念定义**：线程阻塞是线程获取不到资源、锁、连接，处于等待状态；线程死锁是两个及以上线程互相持有对方需要的锁，互相无限等待，导致线程永久阻塞。

**影响**：阻塞线程不断累积，线程池耗尽，新请求无法处理，服务彻底不可用。

#### 3.4.2 排查流程

1. `jstack` 导出线程快照。

2. 筛选 BLOCKED、WAITING、TIMED_WAITING 状态线程。

3. 查看线程等待资源、持有的锁，梳理等待链路。

4. 定位死锁、连接耗尽、接口挂起等根因。

#### 3.4.3 典型案例

- **数据库连接池耗尽**：未释放数据库连接、慢查询占用连接，线程获取连接阻塞。

- **第三方接口超时无超时时间**：调用外部接口未配置超时，线程永久挂起。

- **同步锁死锁**：两个线程互相持有对方锁资源，形成环形等待。

- **线程池队列堆积**：任务执行过慢，新任务持续堆积，服务响应超时。

### 3.5 服务卡顿/响应超时故障排查

#### 3.5.1 故障现象：偶发/持续接口超时，TP99/TP999指标异常

服务无宕机、无 OOM、CPU 正常，但用户接口偶尔超时、页面加载缓慢，属于**最隐蔽、最难排查**的软性故障，绝大多数是 GC STW、资源等待导致。

#### 3.5.2 排查流程

1. 查看 GC 日志，确认是否存在长时间 STW 停顿。

2. 排查网络、数据库、Redis、MQ 等依赖组件耗时。

3. 分析线程栈，定位长时间阻塞的业务线程。

4. 校验 JVM 安全点、内存碎片、大对象分配问题。

#### 3.5.3 典型案例

- **CMS/G1并发标记卡顿**：并发标记阶段耗时过长，产生短暂 STW。

- **大对象分配停顿**：超大对象分配触发 GC，导致瞬时卡顿。

- **安全点等待**：JVM 全局安全点等待所有线程就绪，造成全局停顿。

- **数据库慢查询**：SQL 耗时极高，导致业务线程阻塞、接口超时。

**📌面试考点**：Q：服务偶发卡顿无报错，如何排查？
参考答案：优先排查 GC 日志确认 STW 停顿，其次排查中间件、数据库慢查询、网络波动，最后通过线程栈定位阻塞线程。

## 4. 线上故障排查的工具实战

### 4.1 紧急排查必备工具：`jps`/`jstat`/`jstack`/`jmap`/`jcmd`

服务器无图形化界面，线上紧急故障**只能使用 JDK 自带命令**，轻量化、无需安装、即时生效，是工程师必备技能。

**工具分工与最佳实践**：

- **jps**：快速获取 Java 进程 PID，所有排查的前置命令。

- **jstat**：实时监控 GC 频率、内存变化，快速判断是否为 GC 故障。

- **jstack**：排查线程阻塞、死锁、高 CPU 线程。

- **jmap**：查看堆内存占用、生成快照，排查内存溢出。

- **jcmd**：全能工具，官方推荐替代所有老旧命令，性能更高、STW 更短。

**⚠️避坑指南**：线上大堆进程慎用 `jmap -dump`，会触发长时间 STW，建议低峰期执行或使用 jcmd 替代。

### 4.2 阿里Arthas线上诊断实战

Arthas 是阿里开源线上诊断工具，**无侵入、无需重启、功能全面**，是目前互联网公司线上排查首选工具。

**实操部署命令**

```shell
# 在线安装启动arthas
curl -O https://arthas.aliyun.com/arthas-boot.jar
java -jar arthas-boot.jar

```

**核心命令实战**

- **dashboard**：全局仪表盘，实时展示 CPU、内存、GC、线程状态，快速感知服务健康度。

- **thread**：查看线程状态、定位死锁、展示 TOP 高 CPU 线程。

- **heapdump**：生成线上堆快照，用于离线分析内存泄漏。

- **trace**：追踪方法调用链路与耗时，精准定位慢接口、慢方法。

- **watch**：实时监控方法入参、出参、异常，无需改代码重启。

### 4.3 堆Dump分析工具MAT实战

MAT（Memory Analyzer Tool）是内存泄漏分析专业工具，用于解析 hprof 堆快照，精准定位泄漏对象、引用链、大对象。

**实战流程**：导入堆快照 → 查看 Leak Suspects 泄漏报告 → 通过 Dominator Tree 查看对象占用排行 → 查看 GC Roots 引用链 → 定位无法回收的代码位置。

**💡最佳实践**：线上只生成 dump 文件，离线通过 MAT 分析，减少线上服务压力。

### 4.4 GC日志分析工具GCEasy实战

GCEasy 是在线 GC 日志分析工具，无需本地安装，上传 GC 日志即可自动生成可视化报告，统计 GC 频率、停顿时间、内存晋升、异常 GC 点位。

**实战价值**：替代人工逐行解析繁琐日志，快速识别 GC 瓶颈、内存碎片、并发失败、晋升异常问题。

## 5. 线上故障排查的避坑指南与最佳实践

### 5.1 排查过程中的常见误区

- **盲目重启/回滚**：故障第一时间重启，丢失现场，无法根治问题，故障反复复现。

- **只看表象不看根因**：解决表面报错，忽略底层内存、代码缺陷，遗留隐患。

- **过度依赖经验**：凭经验判断故障，忽略监控、日志、数据证据，导致误判。

### 5.2 故障现场的保留技巧

**最佳现场保留顺序**：先导出日志、线程栈、监控数据 → 生成堆快照 → 再执行重启恢复。

**高危命令避坑**：线上高峰期禁止执行 jmap 全量 dump、禁止频繁 jstack，容易加重服务卡顿。

### 5.3 线上故障的预防与预案

- **完善监控告警**：配置 GC 次数、停顿时间、内存使用率、线程池、CPU、报错率全方位告警。

- **压测验证**：上线前压测，提前暴露内存泄漏、性能瓶颈。

- **降级熔断预案**：高并发场景提前降级，避免服务雪崩。

### 5.4 典型故障的复盘模板与改进措施

标准复盘模板包含：故障基本信息、现象描述、影响范围、处理过程、直接原因、根本原因、短板分析、短期修复方案、长期优化方案、预防机制。所有线上故障必须完成复盘文档归档。
