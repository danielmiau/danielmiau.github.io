# 08-JVM常用参数配置

## 本章概述

在前七章的学习中，我们已经系统掌握了 JVM 内存模型、类加载机制、运行时数据区、垃圾回收算法、主流垃圾收集器、执行引擎与 JIT 编译等**底层核心原理**。所有的 JVM 底层机制，最终全部依靠**参数配置**落地到生产环境。

理论原理决定“为什么这么设计”，JVM 参数决定“程序实际怎么运行”。绝大多数线上 Java 性能问题：接口卡顿、频繁 GC、OOM 内存溢出、线程栈溢出、元空间溢出、服务启动缓慢，本质都是**参数配置不合理、场景与参数不匹配**导致。

本章作为 JVM 调优的**落地核心章节**，将系统性讲解 JVM 全品类常用参数，包含内存参数、GC 收集器参数、执行引擎参数、日志调试参数、OOM 故障处理参数。同时提供生产通用配置模板、不同业务场景专属配置、参数验证排错方案、生产避坑指南与高频面试题，彻底打通「底层原理 → 参数配置 → 线上调优 → 故障排查」的完整闭环。

## 1. 本章导读与知识地图

### 1.1 本章学习目标与核心考点

**学习目标**：熟练区分 JVM 三类参数格式；掌握堆、栈、元空间、直接内存全量配置参数；精通主流收集器专属调优参数；能够独立配置 GC 日志、OOM 故障兜底参数；根据低延迟、高吞吐、大内存、批处理等不同业务场景编写生产级 JVM 启动参数；排查参数配置导致的各类线上故障。

**核心考点**：JVM 参数三类分类区别、堆内存核心参数、新生代比例与晋升参数、永久代与元空间差异、各收集器启动参数、CMS/G1 专属调优参数、GC 日志参数、OOM dump 参数、不同业务场景参数选型原则。本章是面试 JVM 调优、生产落地**必考、最实用**的章节。

### 1.2 JVM参数配置的核心价值：调优与故障排查的基础

**概念定义**：JVM 启动参数是开发者与虚拟机交互的唯一入口，用来修改 JVM 默认运行行为、内存大小、垃圾回收策略、编译策略、日志输出、故障兜底机制。

**核心原理**：HotSpot JVM 拥有大量默认参数，适配通用场景，但无法满足差异化业务。互联网高并发、批处理离线任务、超大堆、低延迟交易服务，必须通过自定义参数，修改虚拟机底层运行策略，适配业务特性。

**场景与价值**：合理的参数配置可以**降低 GC 频率、减少 STW 停顿、杜绝 OOM、提升服务吞吐量、降低接口延迟、方便线上故障定位**，是所有 JVM 性能调优的基础。

### 1.3 JVM参数的分类与格式说明

JVM 参数严格分为三类，权限、稳定性、兼容性各不相同，生产配置必须严格区分，避免参数失效或版本兼容问题，后续章节会详细拆解：标准参数、非标准参数、不稳定高级参数。同时参数分为布尔开关型、键值赋值型两种格式，适配不同配置场景。

### 1.4 本章内容与前后章节的关联（内存模型、GC、调优实战）

承接前文：基于内存模型章节配置堆栈内存，基于 GC 章节配置收集器策略，基于执行引擎章节配置 JIT 编译参数。
开启后文：本章是后续 JVM 实战调优、线上故障排查、GC 问题定位的前置基础，所有调优动作全部基于本章参数完成。

**📌面试高频题**：Q：JVM 参数配置的意义是什么？
参考答案：JVM 默认参数仅适配通用场景，无法满足差异化业务。通过自定义参数，可以定制内存大小、GC 策略、编译策略、日志策略，适配业务低延迟/高吞吐需求，规避内存溢出、GC 卡顿等线上故障。

## 2. JVM参数基础与分类

### 2.1 标准参数（-开头）

**概念定义**：以单横线 **-** 开头的参数，属于 JVM 标准规范参数，**所有 JDK 版本完全兼容、永不废弃**，稳定性最高。

**核心原理**：属于 Java 官方规范定义的基础参数，功能简单通用，主要用于基础运行配置、环境配置、版本查看。

**场景与价值**：适用于所有环境，无版本兼容风险，多用于日志输出、类路径配置、运行模式配置。

**实操示例**

```shell
# 查看java版本（标准参数）
java -version
# 开启基础gc日志输出
java -verbose:gc -jar application.jar

```

**避坑指南**：标准参数功能有限，无法实现精细化调优，生产调优几乎不依赖标准参数，仅用于基础配置。

### 2.2 非标准参数（-X开头）

**概念定义**：以 **-X** 开头的参数，属于 HotSpot 虚拟机扩展参数，不属于 Java 官方标准规范，但是**通用性极强，大部分 JDK 版本兼容**。

**核心原理**：由 HotSpot 厂商自定义扩展，用于内存配置、运行模式配置，是日常开发、测试、生产最常用的基础调优参数。

**实操示例**

```shell
# 设置最大堆内存、初始堆内存、线程栈大小（非标准参数）
java -Xms2g -Xmx2g -Xss256k -jar application.jar

```

**避坑指南**：少数 -X 参数在大版本迭代中会微调，跨 JDK 升级时需要核对参数兼容性。

### 2.3 不稳定参数（-XX开头）

**概念定义**：以 **-XX** 开头的高级参数，是 JVM 底层精细化调优参数，**版本兼容性最差、最不稳定**，JDK 升级极易废弃、变更。

**核心原理**：直接修改虚拟机底层运行逻辑、GC 策略、编译优化、内存回收规则，是生产性能调优、故障优化的核心参数。

**场景与价值**：精细化调优、定制 GC 规则、开启编译优化、配置 OOM 兜底策略、解决疑难线上 GC 故障。

### 2.4 布尔型、键值型参数的使用方式

**布尔开关型参数**：用于开启/关闭某个功能

- `-XX:+参数名`：开启对应功能

- `-XX:-参数名`：关闭对应功能

**键值赋值型参数**：用于设置数值、路径、比例

- `-XX:参数名=数值`：自定义参数值

**实操示例**

```shell
# 布尔型：开启OOM堆转储
java -XX:+HeapDumpOnOutOfMemoryError -jar app.jar
# 键值型：设置CMS触发阈值70%
java -XX:CMSInitiatingOccupancyFraction=70 -jar app.jar

```

### 2.5 查看与验证JVM参数的命令

#### 2.5.1 `java -XX:+PrintFlagsFinal`

查看当前 JDK 所有参数默认值、生效值，可用于核对参数默认配置。

```shell
# 输出所有JVM参数详情
java -XX:+PrintFlagsFinal -version

```

#### 2.5.2 `jinfo` 命令的使用

线上运行中进程动态查看、修改 JVM 参数，无需重启服务。

```shell
# 查看进程所有参数（pid为Java进程id）
jinfo -flags pid
# 动态开启GC日志打印
jinfo -flag +PrintGCDetails pid

```

#### 2.5.3 运行时参数的动态修改

部分非稳定参数支持运行时动态变更，适用于线上临时调优、故障应急，无需重启服务，极大降低故障影响范围。

**⚠️避坑指南**：并非所有参数支持动态修改，内存上限、收集器类型等核心参数**不支持动态修改**，必须重启服务生效。

**📌面试高频题**：Q：简述 JVM 三类参数区别？
参考答案：- 标准参数稳定兼容；-X 非标准参数常用、兼容性较好；-XX 高级参数精细化调优、兼容性差、版本迭代易变更。

## 3. 运行时数据区参数配置

### 3.1 堆内存配置

堆内存是 Java 最大的内存区域，存放所有对象实例、数组，是 GC 主要回收区域，也是 OOM 最高发区域，参数配置直接决定服务稳定性与性能。

#### 3.1.1 `-Xms`：初始堆内存大小

**概念定义**：JVM 启动初始申请的堆内存大小。

**核心原理**：JVM 启动时向操作系统申请对应内存，若初始内存过小，服务运行过程中需要频繁扩容，带来性能损耗。

**最佳实践**：生产环境**Xms与Xmx设置一致**，避免堆内存动态扩容收缩，减少 GC 压力。

#### 3.1.2 `-Xmx`：最大堆内存大小

**概念定义**：JVM 堆内存上限，达到该阈值会触发 GC，耗尽则抛出 OOM。

**避坑指南**：最大堆内存不能超过服务器物理内存的 70%，预留内存给系统、直接内存、元空间、线程栈，避免整机内存溢出。

#### 3.1.3 `-Xmn`：新生代内存大小

统一设置新生代整体内存，简化配置，适合通用场景。

#### 3.1.4 `-XX:NewRatio`：新生代/老年代比例

默认值 2，代表老年代:新生代 = 2:1，堆内存 1/3 为新生代，2/3 为老年代。适合对象存活率高的业务。

#### 3.1.5 `-XX:SurvivorRatio`：Eden/Survivor比例

默认 8，代表 Eden:S0:S1 = 8:1:1，控制新生代分区占比，影响对象晋升频率。

#### 3.1.6 `-XX:MaxTenuringThreshold`：对象晋升老年代年龄

对象经过多少次 Minor GC 后晋升老年代，默认 15。数值越大对象越晚晋升，老年代压力越小；数值越小越容易进入老年代。

#### 3.1.7 `-XX:PretenureSizeThreshold`：大对象直接进入老年代阈值

超过该大小的对象不经过新生代，直接分配到老年代，避免大对象在新生代反复复制。

**堆内存完整实操配置**

```shell
# 标准生产堆内存配置
# -Xms2g：初始堆2G
# -Xmx2g：最大堆2G
# -Xmn800m：新生代800M
# -XX:MaxTenuringThreshold=10：10次GC晋升老年代
# -XX:PretenureSizeThreshold=2097152：大于2M对象直接进老年代
java -Xms2g -Xmx2g -Xmn800m -XX:MaxTenuringThreshold=10 -XX:PretenureSizeThreshold=2097152 -jar app.jar

```

### 3.2 方法区/元空间配置

#### 3.2.1 `-XX:PermSize` / `-XX:MaxPermSize`（JDK7及以前永久代）

JDK7 及之前使用永久代存储类信息、常量、方法数据，内存固定，极易溢出，JDK8 彻底废弃。

#### 3.2.2 `-XX:MetaspaceSize`：元空间初始大小（JDK8+）

元空间初始阈值，达到阈值触发元空间 GC，回收无效类、方法数据。

#### 3.2.3 `-XX:MaxMetaspaceSize`：元空间最大大小（JDK8+）

限制元空间上限，防止动态加载类、热部署导致元空间无限膨胀，引发整机内存溢出。

#### 3.2.4 `-XX:CompressedClassSpaceSize`：压缩类空间大小

开启指针压缩后，专门存储压缩类信息的独立内存空间，默认 1G。

**元空间实操配置**

```shell
# JDK8+元空间生产配置
java -XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=256m -XX:CompressedClassSpaceSize=512m -jar app.jar

```

### 3.3 栈内存配置

#### 3.3.1 `-Xss`：线程栈大小

控制单个线程栈内存大小，每个线程独立占用栈空间，存储栈帧、局部变量、方法调用信息。

#### 3.3.2 栈大小对线程数量与栈溢出的影响

栈越大，单个线程内存占用越高，服务器支持的最大线程数越少；栈过小，方法嵌套过深会抛出 **StackOverflowError**。生产通用配置 256k。

```shell
# 设置线程栈大小256k
java -Xss256k -jar app.jar

```

### 3.4 直接内存配置

#### 3.4.1 `-XX:MaxDirectMemorySize`：直接内存最大值

限制 NIO 直接缓冲区占用的堆外内存上限，防止 Netty、NIO 框架堆外内存泄露。

#### 3.4.2 直接内存溢出的场景与排查

Netty 组件未释放缓冲区、NIO 资源未关闭，会导致直接内存泄露，报错 `Direct buffer memory`。可通过该参数限制内存，配合资源关闭代码解决溢出。

**📌面试高频题**：Q：为什么生产环境 Xms 和 Xmx 需要设置一致？
参考答案：避免 JVM 运行时频繁扩容和收缩堆内存，减少 GC 触发次数，提升服务稳定性。

## 4. 垃圾回收器参数配置

### 4.1 收集器选择参数

收集器选型是 GC 调优**第一步、最重要**的步骤，参数直接决定垃圾回收算法、STW 时长、吞吐量、适配场景。

#### 4.1.1 `-XX:+UseSerialGC`：Serial + Serial Old组合

单线程收集器组合，仅适用于客户端、测试、嵌入式场景，生产服务**禁止使用**。

#### 4.1.2 `-XX:+UseParNewGC`：ParNew + Serial Old组合

新生代多线程、老年代单线程，仅用于老旧 JDK8 项目搭配 CMS 使用。

#### 4.1.3 `-XX:+UseParallelGC`：Parallel Scavenge + Parallel Old组合

JDK8 默认收集器，主打**高吞吐量**，适配离线批处理、数据分析、后台任务。

#### 4.1.4 `-XX:+UseConcMarkSweepGC`：ParNew + CMS组合

初代低延迟收集器，JDK9 废弃，老旧互联网项目遗留使用。

#### 4.1.5 `-XX:+UseG1GC`：G1收集器

JDK9+ 默认收集器，通用型收集器，平衡延迟与吞吐量，适配绝大多数线上业务。

#### 4.1.6 `-XX:+UseZGC`：ZGC收集器（JDK11+）

新一代极致低延迟收集器，亚毫秒级停顿，适配核心交易、高并发、超大堆业务。

### 4.2 CMS收集器参数

#### 4.2.1 `-XX:CMSInitiatingOccupancyFraction`：CMS触发阈值

设置老年代占用百分比阈值，达到阈值触发 CMS 并发 GC，默认68。高并发场景建议调低，提前 GC 避免并发失败。

#### 4.2.2 `-XX:+UseCMSCompactAtFullCollection`：Full GC后整理内存

CMS 标记清除算法会产生内存碎片，开启该参数，Full GC 后自动压缩整理内存。

#### 4.2.3 `-XX:CMSFullGCsBeforeCompaction`：整理前Full GC次数

每执行指定次数 Full GC 后，执行一次内存压缩，平衡性能与碎片问题。

#### 4.2.4 `-XX:+CMSClassUnloadingEnabled`：CMS回收方法区

开启后 CMS 可以回收无效类、元空间数据，防止方法区内存溢出。

### 4.3 G1收集器参数

#### 4.3.1 `-XX:MaxGCPauseMillis`：目标最大暂停时间

G1 核心参数，自定义单次 GC 最大允许停顿时间，G1 自动适配回收分区数量，精准控制 STW。

#### 4.3.2 `-XX:G1HeapRegionSize`：Region大小

设置 G1 分区大小，1MB~32MB，适配不同堆内存大小。

#### 4.3.3 `-XX:InitiatingHeapOccupancyPercent`：触发并发标记的堆占比

整体堆内存占用达到阈值，触发 G1 并发标记，提前回收垃圾。

#### 4.3.4 `-XX:G1NewSizePercent` / `-XX:G1MaxNewSizePercent`：新生代占比范围

控制 G1 逻辑新生代最小、最大堆占比，动态适配业务流量。

**G1完整实操配置**

```shell
# G1生产通用低延迟配置
java -XX:+UseG1GC -XX:MaxGCPauseMillis=100 -XX:G1HeapRegionSize=4m -XX:InitiatingHeapOccupancyPercent=45 -jar app.jar

```

**⚠️避坑指南**：CMS 已废弃，新项目禁止使用；G1 适合中小堆，16G+大堆优先使用 ZGC。

## 5. JIT编译与执行引擎参数

本章参数对应上一章执行引擎知识点，用于控制代码编译模式、JIT 优化策略，直接影响代码执行速度与峰值性能。

### 5.1 `-Xint`：纯解释模式

关闭所有 JIT 编译，所有代码逐条解释执行，启动最快、性能最差，仅用于代码调试。

### 5.2 `-Xcomp`：纯编译模式

程序启动直接编译所有代码，无解释执行，启动慢、峰值性能极高。

### 5.3 `-Xmixed`：混合模式（默认）

JVM 默认模式，冷代码解释执行、热点代码 JIT 编译，平衡启动速度与运行性能。

### 5.4 `-XX:TieredStopAtLevel`：分层编译层级控制

自定义分层编译最高层级，限制编译深度，适配短生命周期微服务。

### 5.5 `-XX:+PrintCompilation`：打印编译日志

输出方法 JIT 编译日志，用于排查代码编译失败、优化失效问题。

### 5.6 `-XX:+DoEscapeAnalysis`：开启逃逸分析

默认开启，开启栈上分配、标量替换、锁消除优化，大幅减少临时对象 GC 压力。生产**禁止关闭**。

### 5.7 `-XX:+EliminateLocks`：锁消除优化

开启无竞争场景锁消除，减少 synchronized 锁开销，提升同步代码执行效率。

**执行引擎实操配置**

```shell
# 开启全部编译优化、打印编译日志
java -Xmixed -XX:+DoEscapeAnalysis -XX:+EliminateLocks -XX:+PrintCompilation -jar app.jar

```

**📌面试高频题**：Q：生产环境是否需要关闭逃逸分析？
参考答案：不需要，逃逸分析可以实现栈上分配、锁消除，极大减少 GC 次数，是重要性能优化，生产必须开启。

## 6. 日志与调试参数配置

GC 日志、错误日志是线上故障排查的唯一依据，没有日志则无法定位 GC 卡顿、OOM、内存泄露问题，是生产**必配参数**。

### 6.1 GC日志参数

#### 6.1.1 `-verbose:gc` / `-XX:+PrintGC`：打印基础GC日志

输出简单 GC 前后内存占用、GC 耗时，信息简单，仅适用于测试环境。

#### 6.1.2 `-XX:+PrintGCDetails`：打印GC详细日志

输出内存分区变化、GC 类型、停顿时间、回收内存大小，生产必备。

#### 6.1.3 `-XX:+PrintGCTimeStamps` / `-XX:+PrintGCDateStamps`：打印时间戳

PrintGCTimeStamps 输出程序启动相对时间；PrintGCDateStamps 输出系统绝对时间，生产优先使用绝对时间，方便定位故障时间点。

#### 6.1.4 `-Xloggc:gc.log`：GC日志输出文件

将 GC 日志输出到指定文件，替代控制台输出，方便日志持久化与分析。

#### 6.1.5 `-XX:+UseGCLogFileRotation`：GC日志轮转

开启日志分割轮转，避免单一日志文件过大，占用磁盘空间。

#### 6.1.6 `-XX:+PrintHeapAtGC`：GC前后打印堆快照

每次 GC 前后输出完整堆内存快照，适合排查内存泄露、异常 GC 问题。

### 6.2 错误日志与OOM处理参数

#### 6.2.1 `-XX:+HeapDumpOnOutOfMemoryError`：OOM时生成堆Dump

OOM 溢出时自动生成堆快照文件，是排查内存泄露**最核心参数**。

#### 6.2.2 `-XX:HeapDumpPath`：堆Dump文件路径

自定义 dump 文件存储路径，避免默认路径磁盘空间不足导致 dump 失败。

#### 6.2.3 `-XX:OnOutOfMemoryError`：OOM时执行的脚本

OOM 故障触发时，自动执行自定义脚本，实现告警、重启、备份兜底。

#### 6.2.4 `-XX:+ExitOnOutOfMemoryError`：OOM时直接退出

OOM 立即退出进程，避免异常进程占用资源、持续报错。

**生产日志完整配置**

```shell
# 生产必备日志&OOM兜底参数
java -XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:./gc.log -XX:+UseGCLogFileRotation -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=./dump.hprof -jar app.jar

```

**⚠️避坑指南**：线上服务禁止关闭 GC 日志，一旦发生内存故障，无日志将无法定位根因。

## 7. 常用组合配置示例

### 7.1 互联网低延迟服务推荐配置（G1/ZGC）

适用于网关、接口服务、用户端业务，优先保证低延迟、接口稳定。

```shell
# JDK8 G1低延迟通用配置
java -Xms4g -Xmx4g -Xss256k -XX:MetaspaceSize=256m -XX:MaxMetaspaceSize=256m -XX:+UseG1GC -XX:MaxGCPauseMillis=100 -XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:./gc.log -XX:+HeapDumpOnOutOfMemoryError -jar api.jar

```

### 7.2 高吞吐量批处理服务推荐配置（Parallel收集器）

适用于数据分析、定时任务、离线计算，优先保证吞吐量，可接受轻微延迟。

```shell
java -Xms8g -Xmx8g -XX:+UseParallelGC -XX:GCTimeRatio=99 -jar batch.jar

```

### 7.3 开发/测试环境通用配置模板

内存较小、方便调试、日志齐全，兼顾启动速度与问题排查。

```shell
java -Xms1g -Xmx1g -Xss256k -XX:+PrintGCDetails -Xloggc:gc.log -jar dev.jar

```

### 7.4 大内存应用（16G+）配置示例

大内存服务优先使用 ZGC，保证亚毫秒级停顿，避免超大堆 GC 卡顿。

```shell
java -Xms16g -Xmx16g -XX:+UseZGC -XX:MetaspaceSize=512m -jar bigheap.jar

```

### 7.5 参数配置避坑指南：常见错误与解决方案

- **错误**：Xmx 设置过大，超出服务器物理内存 → **后果**：整机内存溢出、OOM、服务被杀死 → **方案**：堆内存不超过物理内存70%

- **错误**：元空间不设置上限 → **后果**：动态类加载导致内存泄露 → **方案**：固定 Metaspace 大小

- **错误**：不配置 OOM dump 参数 → **后果**：内存溢出无法定位 → **方案**：所有生产服务标配 dump 参数

- **错误**：混用废弃 CMS 参数 → **后果**：JDK14+ 启动报错 → **方案**：新项目统一使用 G1/ZGC
