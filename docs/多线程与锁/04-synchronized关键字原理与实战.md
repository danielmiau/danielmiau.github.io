# 1. `synchronized` 关键字的作用与使用场景

synchronized是Java原生内置的**隐式互斥锁**，由JVM底层实现，无需手动加锁、解锁，可一站式解决并发编程中**原子性、可见性、有序性**三大核心问题，是保障线程安全的基础核心方案。

## 1.1 `synchronized` 关键字的核心作用

### 1.1.1 保证原子性：确保代码块/方法执行不可分割

**原子性定义**：被synchronized修饰的代码，在同一时刻仅允许**一个线程执行**，操作全程不可被中断、不可被穿插，要么全部执行成功，要么不执行。

**底层原理**：synchronized通过排他锁机制，将多线程并发执行的临界区代码，转换为单线程串行执行，彻底解决i++、读取-修改-写入等复合操作的并发数据覆盖问题，杜绝竞态条件引发的数据不一致。

**核心价值**：完美解决多线程共享变量的复合操作安全问题，是普通业务场景保障原子性的首选方案。

### 1.1.2 保证可见性：通过锁释放刷新主内存，解决数据可见性问题

基于JMMJava内存模型规范，synchronized拥有强制内存同步的机制，彻底解决线程本地内存与主内存的数据延迟问题：

- **加锁时**：线程会清空当前工作内存的共享变量缓存，强制从主内存加载最新数据；

- **解锁时**：线程会强制将工作内存中修改后的共享变量数据，刷新到主内存，对其他线程实时可见。

相较于volatile仅能保证可见性，synchronized在保障可见性的同时，兼顾原子性，能力更加全面。

### 1.1.3 保证有序性：通过内存屏障禁止指令重排

为提升执行效率，JVM编译器、CPU会对无依赖的指令进行重排序优化，多线程环境下会引发逻辑错乱。

synchronized会在**锁获取、锁释放位置插入内存屏障**，禁止临界区内的指令与外部指令重排序，保证代码执行顺序与开发者编写顺序一致，彻底解决指令重排序导致的有序性问题，杜绝DCL单例半空对象逃逸等偶现bug。

## 1.2 `synchronized` 的适用场景

### 1.2.1 多线程共享资源读写场景

这是synchronized最通用的场景。多线程并发操作**共享可变资源**（成员变量、静态变量、全局缓存、文件资源、数据库数据）时，通过加锁保证同一时间仅有一个线程修改资源，避免数据竞争。

**典型业务**：用户信息更新、配置修改、共享缓存刷新、公共资源读写。

### 1.2.2 计数器、累加器等原子操作场景

针对请求计数、访问量统计、订单流水自增、库存扣减、积分增减等**非原子复合操作**，通过synchronized加锁，保证读取-修改-写入整套操作的原子性，避免数据丢失、统计不准、超卖等问题。

### 1.2.3 单例模式、懒加载等线程安全场景

经典DCL双重检查锁单例模式中，依靠synchronized锁定实例创建代码块，防止多线程并发创建多个实例，保障单例唯一性。配合volatile禁止指令重排，彻底解决懒加载单例的线程安全问题。

### 1.2.4 生产者-消费者等线程协作场景

生产者消费者模型中，通过synchronized配合wait()、notify()方法，实现线程间的等待与唤醒，控制生产、消费节奏，解决队列空、队列满的线程协作同步问题，保证业务有序执行。

# 2. `synchronized` 的三种使用方式（修饰实例方法/静态方法/代码块）

synchronized共有三种语法形式，核心差异在于**锁对象不同、锁粒度不同、影响范围不同**，直接决定并发性能与线程阻塞范围，是开发选型和面试高频考点。

## 2.1 修饰实例方法

### 2.1.1 锁对象：当前实例对象（this）

当synchronized修饰普通实例方法时，默认锁对象为**当前类的实例对象this**。同一个实例的所有synchronized实例方法，共用同一把锁；不同实例之间锁相互独立，互不阻塞。

### 2.1.2 适用场景：实例级别的方法同步

适用于**单个实例内部共享资源**的同步场景，仅限制当前实例的多线程并发访问，不同实例可并行执行，并发度相对较高。常用于对象维度的业务数据更新、实例资源操作。

### 2.1.3 代码示例与注意事项

**完整可运行代码示例**：

```java
/**
 * synchronized修饰实例方法
 * 锁对象：当前实例this
 * 特点：同一实例多线程互斥，不同实例并行
 */
public class SyncInstanceMethodDemo {
    // 实例共享变量
    private int count = 0;

    /**
     * 修饰实例方法，默认锁this
     */
    public synchronized void increment() {
        count++;
        // 打印线程名称与计数结果，观察执行顺序
        System.out.println(Thread.currentThread().getName() + " 执行累加，当前值：" + count);
    }

    public static void main(String[] args) {
        // 创建单个实例
        SyncInstanceMethodDemo demo = new SyncInstanceMethodDemo();

        // 启动3个线程，操作同一个实例
        for (int i = 0; i < 3; i++) {
            new Thread(demo::increment, "线程" + (i + 1)).start();
        }
    }
}

```

**核心注意事项**：

- 同一个实例下，多个synchronized实例方法**互斥阻塞**，一个线程占用锁，其他线程全部等待；

- 不同实例调用方法**不会阻塞**，锁相互独立；

- 锁粒度为整个方法，若方法内存在大量非临界逻辑，会造成不必要的性能阻塞。

## 2.2 修饰静态方法

### 2.2.1 锁对象：当前类的Class对象

当synchronized修饰静态方法时，锁对象为**当前类的Class对象（全局唯一）**，不属于任何实例，是类级别的全局锁。整个项目中所有该类的实例，全部共用同一把Class锁。

### 2.2.2 适用场景：静态变量、静态方法的类级别同步

适用于**全局静态资源**同步场景，所有实例共享的静态变量修改、全局计数器、类级别的统一操作，需要全局互斥的业务场景。

### 2.2.3 代码示例与注意事项

**完整可运行代码示例**：

```java
/**
 * synchronized修饰静态方法
 * 锁对象：当前类Class对象（全局唯一）
 * 特点：所有实例共用一把锁，全局互斥
 */
public class SyncStaticMethodDemo {
    // 全局静态共享变量
    private static int count = 0;

    /**
     * 修饰静态方法，锁当前类Class对象
     */
    public static synchronized void increment() {
        count++;
        System.out.println(Thread.currentThread().getName() + " 执行静态累加，当前值：" + count);
    }

    public static void main(String[] args) {
        // 创建两个不同实例
        SyncStaticMethodDemo demo1 = new SyncStaticMethodDemo();
        SyncStaticMethodDemo demo2 = new SyncStaticMethodDemo();

        // 多个实例的线程依然会互斥阻塞
        new Thread(SyncStaticMethodDemo::increment, "线程1").start();
        new Thread(SyncStaticMethodDemo::increment, "线程2").start();
        new Thread(SyncStaticMethodDemo::increment, "线程3").start();
    }
}

```

**核心注意事项**：

- 静态锁与实例锁**互不冲突**，静态方法加锁、实例方法加锁，不会互相阻塞；

- 锁范围是全局类级别，所有实例共享，并发阻塞范围大，**性能最差**；

- 仅用于静态资源同步，禁止滥用在普通实例业务场景。

## 2.3 修饰代码块

### 2.3.1 锁对象：自定义锁对象（任意Object实例）

synchronized修饰代码块时，支持**自定义任意Object对象作为锁**，锁粒度由开发者自主控制，是三种用法中最灵活、性能最优的方式。

### 2.3.2 适用场景：缩小锁范围，提升并发性能

适用于方法内**仅有部分代码为临界区**的场景，通过缩小锁范围，只对核心共享资源操作代码加锁，非核心逻辑无需阻塞，最大限度提升并发执行效率，是生产环境**首选使用方式**。

### 2.3.3 代码示例与锁对象选型建议

**完整可运行代码示例**：

```java
/**
 * synchronized修饰代码块
 * 锁对象：自定义私有锁
 * 特点：锁粒度最小、性能最优、灵活可控
 */
public class SyncCodeBlockDemo {
    // 自定义私有锁对象（推荐）
    private final Object lock = new Object();
    private int count = 0;

    public void increment() {
        // 仅对核心临界代码加锁，缩小锁范围
        synchronized (lock) {
            count++;
            System.out.println(Thread.currentThread().getName() + " 代码块累加，当前值：" + count);
        }
        // 此处非临界逻辑，多线程可并行执行，无需阻塞
        doOtherBusiness();
    }

    /**
     * 非临界业务逻辑，无需加锁
     */
    private void doOtherBusiness() {
        try {
            Thread.sleep(10);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }

    public static void main(String[] args) {
        SyncCodeBlockDemo demo = new SyncCodeBlockDemo();
        for (int i = 0; i < 3; i++) {
            new Thread(demo::increment, "线程" + (i + 1)).start();
        }
    }
}

```

**锁对象选型最佳实践**：

- **优先使用私有final锁对象**：`private final Object lock = new Object()`，避免锁对象被篡改、被外部类共用，保证锁唯一、安全；

- 尽量不使用this作为锁：容易被外部类竞争锁，造成无意阻塞；

- 尽量不使用字符串常量作为锁：字符串常量池复用，极易造成锁冲突、阻塞异常。

## 2.4 三种使用方式对比与锁对象本质

### 2.4.1 锁对象的本质差异（实例对象/Class对象/自定义对象）

三种用法的核心区别本质只有一点：**锁的竞争对象不同**：

- **修饰实例方法**：锁 = 当前实例this，对象级别锁，单实例互斥、多实例并行；

- **修饰静态方法**：锁 = 类Class对象，类级别锁，全局所有实例互斥；

- **修饰代码块**：锁 = 自定义Object对象，锁粒度完全自定义，灵活度最高。

### 2.4.2 不同锁对象的锁范围与影响范围对比

|使用方式|锁对象|锁粒度|阻塞范围|并发性能|
|---|---|---|---|---|
|实例方法锁|实例this|整个实例方法|同实例所有同步方法互斥|中等|
|静态方法锁|Class对象|整个静态方法|全局所有实例互斥|最差|
|代码块锁|自定义Object|自定义临界代码段|仅锁定指定代码逻辑|最优|
### 2.4.3 生产环境锁对象选型最佳实践

1. **优先使用代码块锁**：精准缩小锁范围，只锁住核心临界区，最大化提升并发性能，是生产环境标准规范；

2. **私有final锁优先**：统一使用private final Object作为锁对象，杜绝锁复用、篡改、外部竞争问题，安全稳定；

3. **实例方法锁简化场景使用**：方法全部逻辑都是临界区、代码简洁的简单场景，可直接使用实例方法锁，简化代码；

4. **静态锁谨慎使用**：仅全局静态资源同步场景使用，避免全局大锁导致并发阻塞、性能降级；

5. **禁止锁对象混用**：实例锁与静态锁互不阻塞，严禁混用导致线程安全失效。

# 3. 对象头结构与Mark Word详解

synchronized 的锁状态、锁升级、线程竞争信息，全部**存储在 Java 对象头中**。想要读懂锁原理，必须先掌握 Java 对象的内存布局与 Mark Word 结构，这是所有锁机制的底层载体。

## 3.1 Java对象内存结构总览

### 3.1.1 对象头（Object Header）、实例数据、对齐填充

在 64 位 JVM 中，一个 Java 对象在堆内存中的布局分为**三部分**，缺一不可：

1. **对象头（Object Header）**：占用 12 字节（默认开启指针压缩），是锁机制、GC、对象元信息的核心存储区域，包含 Mark Word 和 Klass Pointer 两部分；

2. **实例数据（Instance Data）**：对象真实的成员变量数据，存储对象的属性值，大小由成员变量类型与数量决定；

3. **对齐填充（Padding）**：JVM 要求对象内存大小必须是**8 字节的整数倍**，不足部分通过填充字节补全，无任何业务含义，仅做内存对齐。

**核心结论**：**所有 synchronized 锁状态、GC 分代年龄、哈希码、偏向线程ID 全部存在对象头中**，与实例数据无关。

### 3.1.2 64位JVM下对象内存布局详解

日常开发默认使用 64 位 JDK8，默认开启**指针压缩（UseCompressedOops）**，内存布局固定：

- Mark Word：8 Byte

- Klass Pointer：4 Byte（开启指针压缩）/ 8 Byte（关闭指针压缩）

- 实例数据：自定义大小

- 对齐填充：按需补全至 8 的整数倍

对象头是锁的核心载体，JVM 对对象加锁，本质就是**修改对象头 Mark Word 中的锁标志位**。

## 3.2 Mark Word（标记字）结构与作用

### 3.2.1 Mark Word的核心作用：存储对象运行时状态信息

**Mark Word** 是对象头中最核心的区域，占用 8 字节 64 位空间，是一个**动态复用的状态寄存器**。

核心作用：存储对象在运行期间的所有关键状态，包含：哈希码、GC 分代年龄、锁状态标志、偏向线程ID、偏向时间戳等。

**最核心设计亮点**：Mark Word 并非固定结构，会根据**对象锁状态动态切换存储内容**，极致节省内存空间。

### 3.2.2 Mark Word的位结构详解（锁标志位、偏向锁标识、GC年龄等）

64位 Mark Word 核心位分区（重点记忆面试核心字段）：

- **2bit：锁标志位（Lock Flag）**：区分无锁/轻量级锁/重量级锁状态；

- **1bit：偏向锁标识（Biased Flag）**：0=非偏向锁，1=偏向锁；

- **4bit：GC分代年龄**：对象存活年龄，最大15，超过则进入老年代；

- **31bit：哈希码/线程ID/时间戳**：根据锁状态动态复用；

- 剩余空位：预留扩展位。

**关键区分规则**：通过「偏向锁标识 + 锁标志位」组合，唯一确定当前对象锁状态，是锁升级的判断依据。

### 3.2.3 Mark Word的复用设计：不同锁状态下存储信息的变化

Mark Word 采用**分时复用机制**，不同锁状态下存储的数据完全不同，是 JVM 极致优化内存的核心体现：

1. **无锁状态**：存储对象哈希码、GC年龄；

2. **偏向锁状态**：存储偏向线程ID、偏向时间戳、GC年龄；

3. **轻量级锁状态**：存储栈中锁记录指针；

4. **重量级锁状态**：存储 Monitor 监视器指针。

**面试高频结论**：Mark Word 不会同时存储所有信息，不同锁状态只保留当前需要的数据，废弃无用数据，实现内存高效复用。

## 3.3 Klass Pointer（类型指针）详解

### 3.3.1 指向类元数据的指针，标识对象所属类

Klass Pointer 是对象头的第二组成部分，本质是一个指针，指向该对象对应的**方法区类元数据（Klass 类）**。

核心作用：JVM 通过该指针识别对象的真实类型，用于 instanceof 判断、类型转换、方法调用、多态绑定等核心逻辑。

### 3.3.2 指针压缩对Klass Pointer的影响

64位 JVM 原生指针占 8 字节，所有对象都存储 8 字节指针会造成极大内存浪费，因此 JDK1.6 之后**默认开启指针压缩（-XX:+UseCompressedOops）**。

- 开启压缩：Klass Pointer 占用 4 字节，对象头总大小 12 字节；

- 关闭压缩：Klass Pointer 占用 8 字节，对象头总大小 16 字节；

**生产默认配置**：JDK8 默认开启指针压缩，大幅降低堆内存占用，提升内存利用率。

# 4. 锁升级过程：偏向锁→轻量级锁→重量级锁

JDK1.6 之前，synchronized 是纯粹的重量级锁，依赖操作系统互斥量，性能极差。JDK1.6 之后 JVM 引入**锁自适应升级机制**，根据并发竞争激烈程度，自动从无锁→偏向锁→轻量级锁→重量级锁逐步升级，在低并发场景近乎无开销，彻底优化了 synchronized 性能。

## 4.1 锁升级的核心设计思想

### 4.1.1 无锁、偏向锁、轻量级锁、重量级锁的演进逻辑

锁升级遵循**由轻到重、按需升级**的演进逻辑，完全贴合实际业务并发场景：

- **无锁**：初始状态，无任何线程竞争；

- **偏向锁**：单线程反复获取锁，无竞争，极致轻量化；

- **轻量级锁**：多线程交替执行，竞争轻微，CAS自旋无阻塞；

- **重量级锁**：多线程激烈竞争，CAS自旋失败，线程阻塞等待。

### 4.1.2 锁升级的核心目标：在不同并发场景下平衡性能与安全

传统固定重量级锁的问题：无论竞争是否激烈，都会触发操作系统线程阻塞、上下文切换，性能损耗极大。

锁升级机制核心目标：

1. 无竞争场景：**零开销**，避免无用同步消耗；

2. 轻微竞争场景：**用户态自旋**，避免内核态阻塞；

3. 激烈竞争场景：降级为重量级锁，**保证线程安全**，牺牲性能换稳定性。

## 4.2 偏向锁（Biased Locking）阶段

### 4.2.1 偏向锁的适用场景：无竞争的单线程访问场景

偏向锁是 JVM 针对**单线程重复获取同一把锁**的极致优化，适用于：整个生命周期内，锁资源始终被**同一个线程反复获取**，无其他线程竞争的场景。

业务中 80% 的锁都是无竞争的，偏向锁可以消除大量无意义的同步开销。

### 4.2.2 偏向锁的核心原理：Mark Word存储线程ID，无额外同步开销

当对象第一次被线程获取锁时，JVM 将该线程的 ID 记录在对象头 Mark Word 中，同时标记为偏向锁状态。

后续该线程再次获取锁时，只需对比 Mark Word 中的线程ID，一致则**直接放行，无任何CAS、无任何同步操作**，开销几乎为0。

只有当**其他线程尝试竞争锁**时，偏向锁才会撤销，进入锁升级流程。

### 4.2.3 偏向锁的开启、关闭配置参数（-XX:+UseBiasedLocking）

- **开启参数**：-XX:+UseBiasedLocking（JDK8 默认开启）

- **关闭参数**：-XX:-UseBiasedLocking

**生产特殊场景**：高并发、锁竞争频繁的系统，可手动关闭偏向锁，避免频繁撤销偏向锁带来的性能损耗，直接进入轻量级锁状态。

## 4.3 轻量级锁（Lightweight Locking）阶段

### 4.3.1 轻量级锁的适用场景：多线程交替访问、低竞争场景

当偏向锁被撤销、出现多线程竞争，但线程**交替执行、同一时刻只有一个线程竞争**，无并发抢占时，锁升级为轻量级锁。

典型场景：多个线程错开访问同步代码块，竞争不激烈，短时冲突。

### 4.3.2 轻量级锁的核心原理：栈中锁记录（Lock Record）与Mark Word交换

线程获取轻量级锁时，会在当前线程栈帧中创建一个**Lock Record（锁记录）**空间，用于存储对象当前的 Mark Word 副本。

随后通过 CAS 操作，将对象头的 Mark Word 替换为锁记录指针：

1. CAS 成功：线程获取锁，执行临界区代码；

2. CAS 失败：说明有线程竞争，开始自旋重试。

### 4.3.3 轻量级锁的CAS自旋尝试机制

轻量级锁全程处于**用户态**，不进入内核态、不阻塞线程。竞争失败的线程会循环执行 CAS 自旋，尝试获取锁。

自旋次数有限，当自旋次数耗尽、仍无法获取锁时，说明竞争激烈，**直接升级为重量级锁**。

**优点**：避免线程阻塞与上下文切换，低竞争性能极高；

**缺点**：自旋会占用 CPU，高竞争下 CPU 损耗严重。

## 4.4 重量级锁（Heavyweight Locking）阶段

### 4.4.1 重量级锁的适用场景：多线程竞争激烈场景

当大量线程并发竞争同一锁、轻量级锁 CAS 自旋频繁失败，锁升级为重量级锁。适用于高并发、锁抢占激烈、线程大量阻塞的业务场景。

### 4.4.2 重量级锁的核心原理：依赖操作系统的Monitor实现线程阻塞与唤醒

重量级锁底层依赖操作系统的**Monitor（监视器锁）** 实现，属于**内核态锁**。

锁升级为重量级锁后，对象头 Mark Word 存储 Monitor 指针，所有竞争失败的线程不再自旋，直接进入内核态阻塞。

Monitor 内部维护等待队列，统一管理阻塞、唤醒逻辑。

### 4.4.3 线程等待队列与唤醒机制

Monitor 包含两个核心队列：

- **EntryList（就绪队列）**：所有竞争锁失败、等待获取锁的线程；

- **WaitSet（等待队列）**：调用 wait() 方法主动放弃锁、进入等待的线程。

当持有锁的线程释放锁后，JVM 从队列中唤醒一个线程抢占锁，完成线程调度。

**缺点**：线程阻塞、唤醒需要用户态与内核态切换，上下文切换开销大，性能最低，但能保证高并发下线程安全。

## 4.5 锁升级的不可逆特性

### 4.5.1 锁只能升级、不能降级的设计原因

**核心结论：synchronized 锁一旦升级，永远无法降级**。

设计原因：

1. 锁降级需要额外的状态检测、状态回滚逻辑，会引入极大的性能开销，得不偿失；

2. JVM 设计理念：优先保证高并发稳定性，牺牲极端场景的微小性能；

3. 一旦出现激烈竞争，说明当前业务存在并发冲突，无需回退为轻量锁，避免再次频繁升级。

**面试高频考点**：JDK1.6+ 锁**只升不降**，偏向锁→轻量级锁→重量级锁不可逆。

### 4.5.2 锁升级过程的性能变化曲线

锁层级越高，安全性越强，性能越低：

- **偏向锁**：性能极高，近乎无锁开销，适合无竞争；

- **轻量级锁**：性能中等，CAS自旋开销，适合低竞争；

- **重量级锁**：性能最低，线程阻塞切换开销大，适合高竞争保稳定。

**生产调优启示**：开发中尽量缩小锁范围、降低锁竞争，让锁维持在偏向锁或轻量级锁状态，避免升级为重量级锁，保证并发性能。

# 5. 偏向锁的开启、关闭与撤销

偏向锁是 JDK1.6 为优化**无竞争单线程同步场景**推出的轻量化锁，是锁升级体系的首个层级。相比于轻量级锁、重量级锁，偏向锁几乎零开销，但存在专属的延迟开启、主动失效、批量撤销特性，是面试高频难点，也是生产调优的重要依据。

## 5.1 偏向锁的开启与延迟

### 5.1.1 JVM默认开启偏向锁的配置（JDK 1.6+）

从 **JDK1.6 开始，JVM 默认开启偏向锁**，作为 synchronized 的默认初始锁状态，目的是优化项目中占比极高的**单线程无竞争同步场景**。

在 JDK1.5 及之前版本，偏向锁默认关闭，synchronized 直接使用轻量级锁或重量级锁，性能开销较大。JDK1.6 之后官方将其默认开启，大幅提升了低并发场景下 synchronized 的执行效率。

**核心设计初衷**：绝大多数锁在整个生命周期内，只会被同一个线程反复获取，几乎不存在竞争，偏向锁可以消除 CAS 同步开销，实现近乎无锁的性能。

### 5.1.2 偏向锁的延迟开启机制（默认4秒延迟）

这是极易被忽略的面试高频考点：**JVM 启动后，偏向锁不会立即生效，默认延迟 4 秒开启**。

**延迟机制原因**：JVM 启动过程中，大量基础类、系统线程会频繁初始化、竞争锁，如果启动初期就开启偏向锁，会触发大量偏向锁撤销操作，带来不必要的性能损耗。延迟开启可以避开系统启动高峰期，减少无效锁撤销。

**现象说明**：项目启动前 4 秒，所有对象默认是**无锁状态**，4 秒后新创建的对象才会默认开启偏向锁；启动初期创建的对象，大概率不会开启偏向锁。

### 5.1.3 偏向锁开启的JVM参数配置

JVM 提供专属参数控制偏向锁的开启、关闭、延迟时间，生产环境可按需调优：

- **开启偏向锁（默认）**：`-XX:+UseBiasedLocking`

- **关闭偏向锁**：`-XX:-UseBiasedLocking`

- **修改偏向锁延迟时间**：`-XX:BiasedLockingStartupDelay=0`（设置为0，启动立即开启偏向锁）

**生产小知识**：高并发秒杀、高频接口场景，通常会关闭偏向锁，避免频繁撤销带来的性能抖动；普通业务、低并发场景保留默认配置即可。

## 5.2 偏向锁的关闭场景

### 5.2.1 多线程竞争场景下偏向锁的撤销触发条件

偏向锁是**单线程专属优化**，一旦出现多线程竞争，偏向锁会立即触发**锁撤销**，升级为轻量级锁。

**完整撤销流程**：

1. 当前对象已偏向线程A，Mark Word 记录线程A的 ID；

2. 线程B尝试竞争该锁，发现锁已偏向其他线程；

3. JVM 触发安全点（STW），暂停所有用户线程；

4. 检测原持有线程是否存活、是否仍持有锁；

5. 撤销偏向锁，清空线程ID，锁升级为轻量级锁；

6. 恢复线程运行，进入 CAS 自旋竞争逻辑。

**关键特性**：偏向锁撤销需要短暂 STW，频繁竞争会导致频繁撤销，反而降低性能，这也是高并发场景关闭偏向锁的核心原因。

### 5.2.2 调用`hashCode()`/`wait()`/`notify()`方法导致偏向锁失效

除了多线程竞争，调用特定方法会**直接销毁偏向锁**，使对象永久丢失偏向能力：

- **调用 hashCode()**：偏向锁的 Mark Word 会存储偏向线程ID，无法存储哈希码。一旦调用 hashCode()，JVM 必须清空偏向信息，将对象恢复为无锁状态，偏向锁彻底失效；

- **调用 wait()/notify()**：wait、notify 依赖 Monitor 监视器机制，仅重量级锁支持，调用后会直接撤销偏向锁，锁升级为重量级锁。

**面试结论**：只要对象调用过 hashCode()、wait()、notify()，**永远不会再进入偏向锁状态**。

### 5.2.3 批量重偏向（Bulk Rebiasing）与批量撤销（Bulk Revocation）机制

为解决大量对象频繁触发偏向锁撤销导致的性能雪崩，JVM 引入了**类级别批量优化机制**，针对同一个类的所有对象统一优化。

**1. 批量重偏向（Bulk Rebiasing）**

触发条件：1 秒内，同一个类的对象**偏向锁撤销次数达到 20 次**。

核心逻辑：JVM 判定该类对象存在线程切换，但竞争不激烈，允许后续新线程直接重偏向，无需每次都执行完整撤销流程，大幅降低开销。

**2. 批量撤销（Bulk Revocation）**

触发条件：1 秒内，同一个类的对象**偏向锁撤销次数达到 40 次**。

核心逻辑：JVM 判定该类对象竞争激烈、线程频繁切换，偏向锁已无优化价值，直接**永久取消该类所有对象的偏向锁能力**，后续该类对象默认直接进入轻量级锁状态。

**核心面试总结**：20 次触发批量重偏向，40 次触发批量撤销，且批量撤销是**类级别永久失效**。

## 5.3 偏向锁的生产环境使用建议

### 5.3.1 低并发场景下偏向锁的性能优势

在后台管理系统、离线任务、普通业务接口等**低并发、单线程复用锁**的场景，偏向锁优势极大：

- 无需 CAS 自旋、无需线程阻塞、无上下文切换；

- 锁获取、释放开销几乎为零，性能接近无锁；

- 减少大量无效同步操作，提升系统吞吐量。

此类场景**推荐保留默认开启偏向锁**，无需手动修改 JVM 参数。

### 5.3.2 高并发场景下关闭偏向锁的优化建议

在秒杀、抢购、高频网关接口、消息队列消费等高并发场景，线程切换频繁、锁竞争激烈，偏向锁弊大于利：

- 频繁触发偏向锁撤销、STW 停顿，造成接口性能抖动；

- 极易触发批量撤销机制，锁直接固化为轻量级/重量级锁；

- 撤销开销远大于偏向锁带来的性能收益。

**生产优化方案**：启动参数添加 `-XX:-UseBiasedLocking` 关闭偏向锁，让锁直接从无锁→轻量级锁流转，避免频繁撤销的性能损耗。

# 6. 轻量级锁的CAS自旋原理

轻量级锁是JDK1.6优化synchronized性能的核心设计，其全程**无内核阻塞、无操作系统互斥量参与**，完全依赖CAS自旋机制实现锁竞争。掌握CAS原理是彻底理解轻量级锁运行机制、性能优劣及锁升级触发逻辑的关键。

## 6.1 CAS（Compare-And-Swap）的核心原理

### 6.1.1 CAS的操作流程：预期值-修改值对比交换

**CAS（比较并交换）**是一种硬件级别的无锁原子操作，由CPU指令保证原子性，无需线程阻塞，是所有无锁并发机制的底层核心。

CAS操作包含三个核心参数：**内存地址V、预期旧值A、目标新值B**，执行流程固定且原子不可分割：

1. 线程读取内存地址V中存储的当前数据，记录为旧值；

2. 比较当前内存值V与预期旧值A是否一致；

3. 一致：说明期间无其他线程修改数据，直接将内存值更新为新值B，操作成功；

4. 不一致：说明数据已被其他线程修改，放弃本次更新，返回失败并可重试操作。

**核心本质**：先对比、后修改，乐观锁思想，默认认为并发冲突极少发生，通过重试机制保证最终数据一致性。

### 6.1.2 CAS在轻量级锁中的应用：尝试替换Mark Word

轻量级锁的加锁、竞争逻辑完全基于CAS实现，核心操作就是**通过CAS替换对象头的Mark Word**，具体流程如下：

1. 线程竞争锁时，先在当前线程栈帧创建Lock Record（锁记录），拷贝对象原Mark Word作为预期旧值；

2. 通过CAS指令，尝试将对象头Mark Word替换为当前Lock Record的指针地址；

3. CAS成功：当前线程抢占锁成功，对象标记为轻量级锁状态；

4. CAS失败：说明已有线程持有锁或存在竞争，进入自旋重试逻辑。

整个过程全程处于**用户态**，无需切换内核态、无需线程阻塞，这也是轻量级锁低开销的核心原因。

### 6.1.3 CAS的ABA问题与轻量级锁的规避方式

**ABA问题定义**：CAS仅对比数值是否一致，无法感知数据的变更过程。若数据从A→B→A，CAS会判定数据未变更，直接执行更新，可能引发隐性并发问题。

**轻量级锁场景下的ABA问题规避**：

普通变量CAS极易出现ABA问题，但**轻量级锁的CAS操作不会出现ABA问题**，底层JVM做了专属规避：

- 轻量级锁竞争时，Mark Word一旦被替换为Lock Record指针，**锁标志位会同步修改**，对象锁状态永久变更；

- 即使后续锁释放、Mark Word恢复原值，锁状态的变更记录可被JVM识别，不会出现无感知的状态回环；

- 同时轻量级锁的竞争周期极短，数据反复变更的场景几乎不存在。

**面试结论**：轻量级锁CAS无需版本号机制，JVM底层状态标记天然规避ABA问题。

## 6.2 轻量级锁的自旋过程

### 6.2.1 线程竞争时的CAS自旋尝试

当多个线程交替竞争锁、CAS首次抢占失败时，线程不会直接阻塞，而是触发**自适应CAS自旋**机制：

1. 竞争失败的线程，循环执行CAS尝试替换Mark Word；

2. 每次自旋都会检测锁是否被释放、Mark Word是否恢复无锁状态；

3. 一旦锁释放，当前线程立即抢占锁，完成加锁操作；

4. 全程线程处于运行状态，无阻塞、无挂起。

自旋的核心目的：**短时间内乐观等待锁释放，避免重量级锁的线程阻塞与上下文切换开销**。

### 6.2.2 自旋次数与CPU资源消耗的平衡

CAS自旋是**CPU密集型操作**，线程空循环重试会持续占用CPU资源，因此JVM采用**自适应自旋策略**平衡性能与资源消耗：

- JDK1.6之前：固定自旋次数，次数耗尽直接升级重量级锁；

- JDK1.6及之后：**自适应自旋**，JVM根据历史锁竞争情况动态调整自旋次数；

- 若当前锁经常无竞争、自旋大概率成功，则增加自旋次数；

- 若当前锁竞争激烈、自旋频繁失败，则减少甚至放弃自旋，直接升级锁。

**生产核心特性**：低竞争场景自旋成功，性能远超重量级锁；高竞争场景快速终止自旋，避免CPU空转耗尽资源。

### 6.2.3 自旋失败后升级为重量级锁的触发条件

满足以下任一条件，轻量级锁自旋终止，**强制升级为重量级锁**：

1. 线程自适应自旋次数耗尽，仍未抢占到锁；

2. 同一时刻出现**两个及以上线程并发竞争锁**，交替访问场景变为激烈竞争场景；

3. 偏向锁撤销后，多次CAS重试均失败。

锁升级后，JVM创建Monitor监视器，所有竞争失败的线程进入内核态阻塞，加入等待队列，彻底终止用户态自旋，牺牲性能保证线程安全。

# 7. 锁粗化、锁消除优化

除了锁升级机制外，JVM在编译期还提供两大**自动锁优化策略**：锁粗化、锁消除。这两类优化无需开发者手动编码，JVM即时编译器自动触发，可有效减少无效锁竞争、频繁加解锁的性能损耗，是synchronized高性能的重要保障。

## 7.1 锁粗化（Lock Coarsening）

### 7.1.1 锁粗化的定义：合并相邻的同步代码块，减少锁的获取与释放次数

**锁粗化**是JVM编译期优化机制：当检测到**连续多次对同一把锁进行加锁、解锁操作**时，JVM会自动将多个零散的同步代码块，合并为一个大的同步代码块，只执行一次加锁、一次解锁。

核心目的：**消除频繁加解锁的冗余开销**，锁的获取与释放存在系统开销，频繁操作会严重损耗性能，粗化后可大幅提升执行效率。

### 7.1.2 锁粗化的触发场景：循环内重复加锁/解锁

最典型触发场景：**循环体内嵌套同步代码块**，示例伪代码：

```java
// 优化前：循环内频繁加锁、解锁
for (int i = 0; i < 1000; i++) {
    synchronized (lock) {
        // 简单业务逻辑
    }
}

```

若不优化，循环1000次会执行1000次加锁、1000次解锁，产生大量冗余开销。

**JVM自动优化后等效代码**：

```java
// 锁粗化后：仅加锁一次、解锁一次
synchronized (lock) {
    for (int i = 0; i < 1000; i++) {
        // 简单业务逻辑
    }
}

```

除循环场景外，连续相邻的多个同步代码块、无隔离的同锁同步逻辑，均会触发锁粗化。

### 7.1.3 锁粗化的性能优化效果

- 大幅减少**锁竞争、锁初始化、锁释放**的系统开销；

- 避免频繁的用户态状态切换，提升代码整体执行效率；

- 对循环密集型同步代码，性能提升效果极其明显；

- 完全自动优化，无需开发者修改任何代码。

**避坑要点**：锁粗化不会无限扩大锁范围，仅合并连续无竞争隔离的同锁代码，不会造成锁粒度过大引发的并发阻塞问题。

## 7.2 锁消除（Lock Elimination）

### 7.2.1 锁消除的定义：JVM逃逸分析后消除不存在竞争的锁

**锁消除**是JVM深度编译优化：JVM通过**逃逸分析**判断，若某个锁对象仅为方法局部对象、不会逃逸出方法、不存在多线程竞争，JVM会直接**消除该锁的所有同步逻辑**，代码完全按照无锁方式执行。

核心本质：**无竞争的锁就是无效锁，直接剔除，零开销运行**。

### 7.2.2 锁消除的触发场景：局部对象锁、无逃逸的锁对象

最典型场景：方法内部创建的局部锁对象、局部容器，仅当前线程使用，无多线程共享。

**代码示例（会触发锁消除）**：

```java
public void test() {
    // 局部对象，仅当前线程使用，不会发生对象逃逸
    StringBuilder sb = new StringBuilder();
    for (int i = 0; i < 100; i++) {
        // StringBuffer 方法自带synchronized锁
        sb.append(i);
    }
}

```

StringBuffer所有方法均被synchronized修饰，但sb是局部对象，仅单线程访问，无任何并发竞争。JVM通过逃逸分析识别后，直接消除所有锁逻辑，**等价于无锁执行**，性能与StringBuilder一致。

### 7.2.3 JVM逃逸分析（Escape Analysis）对锁消除的支持

锁消除的前置依赖是**逃逸分析**，JDK1.6默认开启，JDK1.7、1.8持续优化：

- **无逃逸**：对象仅在当前方法、当前线程使用，不传递到外部、不共享，触发锁消除；

- **方法逃逸**：对象传递到方法外部，可能被其他方法使用，不消除锁；

- **线程逃逸**：对象传递到其他线程，存在并发竞争，不消除锁。

**面试高频结论**：日常代码中大量看似加锁的方法，实际都会被JVM锁消除优化，这也是同步容器在单线程下性能几乎无损耗的原因。

# 8. `synchronized` 锁的可重入性

可重入性是synchronized的核心特性之一，是避免线程自身嵌套调用死锁的关键，也是面试高频考点。彻底掌握可重入性的底层实现、代码场景与常见误区，可规避生产环境嵌套同步调用的隐性bug。

## 8.1 可重入锁的定义与意义

### 8.1.1 可重入锁的定义：同一线程可重复获取同一把锁

**可重入锁**：同一个线程，在已经持有某一把锁的前提下，**可以再次重复获取同一把锁**，不会被自身锁阻塞、不会触发死锁，这种锁特性即为可重入性。

synchronized是典型的**隐式可重入锁**，ReentrantLock是显式可重入锁，二者均支持锁重入。

### 8.1.2 可重入锁的核心意义：避免线程自身死锁

若无重入性，线程持有锁后，嵌套调用其他同步方法/代码块时，会因为自身已占用锁、无法再次获取锁，导致**自身阻塞、永久死锁**。

可重入性的核心价值：**支持嵌套同步调用，保证代码逻辑正常执行，杜绝线程自死锁**，极大提升了同步代码的灵活性与兼容性。

## 8.2`synchronized` 可重入性的底层实现

### 8.2.1 Mark Word中线程ID与重入计数的维护

synchronized可重入性依托**对象头Mark Word + Monitor监视器**共同实现：

1. 线程首次获取锁：Mark Word记录当前线程ID，Monitor的计数器recorder+1，计数为1；

2. 同一线程再次获取同一锁：JVM对比Mark Word中的线程ID与当前线程一致，直接放行，**无需竞争**，计数器再次+1；

3. 不同线程竞争锁：线程ID不匹配，触发锁竞争、自旋或阻塞。

**核心机制**：通过线程ID匹配判断是否为持有锁的线程，通过计数器记录重入次数。

### 8.2.2 重入次数的自增与释放次数的匹配

synchronized锁的释放遵循**次数匹配原则**：

- 每重入一次锁，计数器**自增1**；

- 每退出一次同步代码块/方法，计数器**自减1**；

- **只有计数器归0时，锁才会真正释放**，其他线程才可竞争锁。

该机制保证了多层嵌套重入时，锁不会提前释放，彻底保证嵌套代码的原子性与安全性。

## 8.3 可重入锁的代码示例与常见误区

### 8.3.1 嵌套调用同步方法的可重入场景

**完整可运行代码示例（锁重入成功）**：

```java
/**
 * synchronized可重入性演示
 * 同一线程嵌套调用同步方法，不会死锁
 */
public class SyncReentrantDemo {

    /**
     * 第一层同步方法
     */
    public synchronized void method1() {
        System.out.println("进入第一层同步方法，锁重入次数+1");
        // 嵌套调用第二层同步方法，同一锁对象，触发锁重入
        method2();
    }

    /**
     * 第二层同步方法，同一锁对象this
     */
    public synchronized void method2() {
        System.out.println("进入第二层同步方法，锁重入次数+1");
    }

    public static void main(String[] args) {
        SyncReentrantDemo demo = new SyncReentrantDemo();
        // 单线程调用，嵌套重入成功，无死锁
        demo.method1();
    }
}

```

**运行结果**：代码正常执行，无阻塞、无死锁，证明synchronized支持锁重入。

### 8.3.2 锁对象不一致导致的重入失败问题

**高频误区**：锁重入的前提是**同一锁对象**，锁对象不同，不具备重入性，依然会触发阻塞死锁。

**错误代码示例（重入失败、死锁）**：

```java
/**
 * 锁对象不一致，重入失败，触发死锁
 */
public class SyncReentrantErrorDemo {
    // 两个不同的锁对象
    private final Object lock1 = new Object();
    private final Object lock2 = new Object();

    public void method1() {
        synchronized (lock1) {
            System.out.println("持有lock1锁，尝试获取lock2锁");
            // 锁对象不同，无法重入，多线程交叉调用会死锁
            method2();
        }
    }

    public void method2() {
        synchronized (lock2) {
            System.out.println("持有lock2锁");
        }
    }
}

```

**核心避坑结论**：

- 可重入性仅限**同一锁对象**有效；

- 不同锁对象之间不存在重入关系，嵌套调用极易引发死锁；

- 生产环境嵌套同步代码，必须保证锁对象统一。

# 9.`synchronized` 与 `Lock` 的对比

synchronized 是 JVM 原生隐式锁，Lock（ReentrantLock）是 JUC 提供的显式锁，二者是 Java 并发编程最核心的两套锁体系。掌握二者差异、性能区别与选型场景，是面试必考题，也是生产环境锁优化的核心依据。

## 9.1 核心特性对比

### 9.1.1 锁的获取与释放方式：自动vs手动

**synchronized（隐式锁）**

- 锁的获取与释放由 JVM 自动完成，无需人工干预；

- 进入同步代码块自动加锁，代码执行完毕/异常退出自动解锁；

- **绝对不会出现锁泄露**，安全性高，无需手动处理异常解锁。

**Lock（显式锁）**

- 锁的获取与释放需要开发者手动编码控制；

- 通过 `lock()` 加锁，`unlock()` 解锁；

- **必须在 finally 中解锁**，否则出现异常会导致锁永久不释放，引发线程死锁。

### 9.1.2 锁的可中断性：不可中断vs可中断

**synchronized**：**不可中断锁**。线程阻塞等待锁时，无法被中断，只能一直阻塞直到获取锁或出现异常，无法响应外部中断信号。竞争激烈时容易出现线程堆积、接口超时。

**Lock**：**可中断锁**。支持 `lockInterruptibly()` 方法，线程等待锁的过程中可以响应中断，主动放弃锁竞争，避免永久阻塞。

### 9.1.3 锁的超时机制：无vs支持tryLock超时

**synchronized**：无超时机制。线程一旦竞争失败，会无限阻塞等待，直到获取锁，无法规避死锁、长时间阻塞问题。

**Lock**：支持超时机制。通过 `tryLock(long time, TimeUnit unit)` 方法，指定等待锁的超时时间，超时未获取锁直接返回失败，有效避免线程永久阻塞，提升系统容错能力。

### 9.1.4 公平锁与非公平锁：仅非公平vs支持两种模式

**synchronized**：**只支持非公平锁**。线程竞争锁随机抢占，不遵循先来先服务，吞吐量高，但可能出现线程饥饿问题。

**Lock（ReentrantLock）**：同时支持两种模式：

- 非公平锁（默认）：性能高，适合大多数业务场景；

- 公平锁：先来先服务，解决线程饥饿问题，牺牲部分吞吐量换公平性。

## 9.2 性能对比

### 9.2.1 低竞争场景下的性能差异

低并发、低竞争场景下，**synchronized 性能略优于 Lock**。

原因：JDK1.6 后 synchronized 拥有完善的锁升级优化，偏向锁、轻量级锁几乎无开销；而 Lock 基于 AQS 实现，存在固定的代码调用、状态更新开销，低竞争下优势不明显。

### 9.2.2 高竞争场景下的性能差异

高并发、锁竞争激烈场景下，**Lock 性能优于 synchronized**。

原因：Lock 基于 AQS 队列实现精准线程调度，支持可中断、超时、公平锁，线程阻塞可控；而 synchronized 升级为重量级锁后，依赖操作系统 Monitor，线程批量阻塞、唤醒，上下文切换开销大，且无法灵活控制。

### 9.2.3 锁升级机制对`synchronized`性能的影响

synchronized 最大性能优势来自**自适应锁升级机制**：无竞争偏向锁、轻微竞争轻量级锁、激烈竞争重量级锁，自适应适配不同场景。

但存在短板：锁升级**只升不降**，一旦升级为重量级锁，即使后续竞争降低，也无法回退轻量级锁，长期高并发后性能会固化下降；而 Lock 无锁升级机制，状态稳定，高并发下性能更平稳。

## 9.3 适用场景与选型建议

### 9.3.1 优先使用`synchronized`的场景：简单同步、低竞争场景

生产环境中，以下场景**优先使用 synchronized**：

- 同步逻辑简单、代码量少、低并发低竞争；

- 不需要可中断、超时、公平锁等高级特性；

- 追求代码简洁、自动解锁、零锁泄露风险；

- 普通接口数据更新、简单计数器、单例模式等场景。

**优势**：代码简洁、JVM 优化极致、安全性高、无需手动维护锁状态。

### 9.3.2 优先使用`Lock`的场景：高竞争、需灵活控制锁的场景

以下场景**必须使用 ReentrantLock**：

- 高并发、锁竞争激烈的核心接口；

- 需要锁超时、可中断、公平锁特性；

- 需要条件变量 Condition 精准唤醒线程（生产者消费者复杂场景）；

- 需要统计锁等待次数、持有次数等监控信息。

### 9.3.3 生产环境锁选型最佳实践

1. **默认优先 synchronized**：绝大多数普通业务场景，优先使用原生锁，代码简洁、稳定、无泄露风险；

2. **有高级特性需求必用 Lock**：需要超时、中断、公平锁、精准唤醒，直接选用 ReentrantLock；

3. **超高并发场景组合优化**：核心热点接口关闭偏向锁，使用 Lock 保证并发稳定性；普通接口保留 synchronized；

4. **杜绝过度使用 Lock**：简单同步场景强行使用 Lock，会增加代码复杂度和锁泄露风险，得不偿失。
