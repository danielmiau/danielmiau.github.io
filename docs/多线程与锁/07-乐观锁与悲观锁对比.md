# 1. 悲观锁的概念与实现（synchronized/ReentrantLock）

悲观锁是传统并发安全的核心实现方案，也是Java并发编程中最基础的锁设计思想。其核心逻辑是预判并发冲突，通过提前加锁、阻塞线程的方式保证数据强一致性，synchronized、ReentrantLock等常用锁均属于悲观锁范畴。

## 1.1 悲观锁的核心思想

### 1.1.1 悲观锁的定义：默认多线程一定会发生竞争冲突

**悲观锁**，顾名思义是一种**悲观的并发设计思想**：线程在操作共享资源前，默认认为多线程并发场景下**一定会发生数据竞争、数据覆盖冲突**。

基于该预判，悲观锁会采取“提前防御”的策略，不允许线程无锁操作共享数据，从源头杜绝并发安全问题。该思想优先保障**数据一致性**，牺牲部分并发性能，是强一致场景的首选锁方案。

### 1.1.2 悲观锁的核心逻辑：提前加锁，阻塞其他线程访问共享资源

悲观锁的完整执行逻辑可以概括为：**先加锁、后操作、释放锁、再放行**，具体流程如下：

1. 线程访问共享资源前，优先尝试获取锁资源；

2. 加锁成功后，独占共享资源的读写权限，执行业务操作；

3. 其他未获取锁的线程全部阻塞排队，无法访问共享资源；

4. 持有锁的线程执行完毕，主动释放锁资源；

5. 阻塞队列中的线程依次竞争锁，串行执行后续操作。

核心特点：**同一时间仅一个线程操作共享资源，所有并发操作串行执行，彻底杜绝并发冲突**。

### 1.1.3 悲观锁的底层实现：依赖操作系统的互斥锁/线程阻塞

Java中所有悲观锁的底层实现，最终都依赖**操作系统内核态的互斥锁（Mutex）**实现线程阻塞与唤醒。

当线程抢锁失败时，JVM会将用户态的线程挂起，交由操作系统调度器阻塞，线程放弃CPU执行权，进入WAITING/BLOCKED状态；当锁释放后，操作系统重新唤醒阻塞线程，触发线程上下文切换，让线程重新竞争锁资源。

正因为涉及**用户态与内核态切换、线程上下文切换**，悲观锁在高并发竞争场景下会产生较大的性能开销。

## 1.2 Java中悲观锁的典型实现

### 1.2.1 synchronized关键字：基于对象监视器的悲观锁实现

**synchronized** 是Java原生隐式悲观锁，是JVM层面内置的锁机制，无需手动加锁解锁，自动生效。

底层依托**Java对象监视器（ObjectMonitor）**实现悲观阻塞逻辑，JDK1.6之前为重量级锁，直接依赖操作系统互斥锁，性能极低；JDK1.6之后引入**锁升级机制**（偏向锁→轻量级锁→重量级锁），无竞争场景下无需阻塞，大幅优化性能。

核心特性：自动加锁、自动解锁、不可中断、默认非公平、可重入，适合简单同步场景。

```java

/**
 * synchronized悲观锁示例
 * 保证count自增操作原子性，多线程串行执行
 */
public class SynchronizedPessimisticLock {
    private static int count = 0;

    // 静态同步方法，锁当前类对象
    public static synchronized void increment() {
        // 同一时间仅一个线程执行，无并发冲突
        count++;
    }

    public static void main(String[] args) throws InterruptedException {
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 10000; i++) {
                increment();
            }
        });

        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 10000; i++) {
                increment();
            }
        });

        t1.start();
        t2.start();
        t1.join();
        t2.join();
        // 最终结果固定20000，数据安全无错乱
        System.out.println("最终count值：" + count);
    }
}

```

### 1.2.2 ReentrantLock：基于AQS的显式悲观锁实现

**ReentrantLock** 是JDK1.5推出的显式悲观锁，底层完全基于AQS独占模式实现，属于代码层面的悲观锁，需要手动加锁、手动解锁。

其悲观思想与synchronized完全一致：抢锁失败线程阻塞排队，串行执行任务；同时扩展了公平锁、超时抢锁、可中断抢锁、多条件精准唤醒等高级特性，是复杂并发场景的首选悲观锁方案。

```java

import java.util.concurrent.locks.ReentrantLock;

/**
 * ReentrantLock显式悲观锁示例
 */
public class ReentrantLockPessimisticDemo {
    private static final ReentrantLock LOCK = new ReentrantLock();
    private static int count = 0;

    public static void increment() {
        // 手动加锁，悲观阻塞
        LOCK.lock();
        try {
            count++;
        } finally {
            // 手动释放锁，必须finally保证执行
            LOCK.unlock();
        }
    }

    public static void main(String[] args) throws InterruptedException {
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 10000; i++) increment();
        });
        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 10000; i++) increment();
        });
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        System.out.println("最终count值：" + count);
    }
}

```

### 1.2.3 两种实现的底层对比：锁升级vs AQS队列

synchronized与ReentrantLock同为悲观锁，核心思想一致，但底层实现、机制、能力差异极大，核心对比如下：

|对比维度|synchronized|ReentrantLock|
|---|---|---|
|实现层级|JVM原生底层实现，隐式锁|JDK代码层AQS实现，显式锁|
|核心机制|锁升级机制：偏向锁→轻量级锁→重量级锁|AQS双向同步队列，线程排队阻塞|
|锁模式|仅支持非公平锁|支持公平/非公平锁切换|
|功能拓展|功能固定，无拓展能力|支持超时、可中断、精准唤醒、读写分离|
|可控性|自动加解锁，无需手动控制|手动加解锁，精细化控制|
## 1.3 悲观锁的优缺点与适用场景

### 1.3.1 优点：实现简单、线程安全、无ABA问题

- **数据强一致性**：独占锁机制，串行执行，彻底杜绝并发数据覆盖、错乱问题；

- **无ABA并发问题**：全程独占资源，其他线程无法修改数据，规避CAS经典ABA漏洞；

- **实现简单稳定**：synchronized无需手动管控，底层JVM优化成熟，稳定性极高；

- **无需自旋空转**：抢锁失败直接阻塞，不会消耗CPU空转。

### 1.3.2 缺点：高竞争场景下线程阻塞、上下文切换开销大

- **并发性能差**：所有操作串行执行，无法利用多核并发能力，吞吐量低；

- **线程阻塞开销**：高并发竞争下大量线程阻塞、唤醒，频繁触发线程上下文切换；

- **用户态内核态切换**：重量级锁依赖操作系统互斥锁，切换开销极大；

- **存在线程饥饿风险**：非公平锁模式下，部分线程长期无法抢占锁。

### 1.3.3 适用场景：写多读少、并发冲突概率高的场景

悲观锁优先保证数据一致性，牺牲并发性能，适配以下场景：

1. **写多读少场景**：数据修改频繁、查询极少，并发冲突概率极高；

2. **数据强一致场景**：资金交易、订单结算、库存扣减等不允许数据错乱的核心业务；

3. **长耗时同步场景**：同步业务执行时间长，CAS自旋会耗尽CPU资源，必须阻塞；

4. **高冲突并发场景**：线程竞争激烈，无锁重试效率远低于阻塞排队。

# 2. 乐观锁的概念与实现（CAS）

乐观锁是现代高并发优化的核心思想，彻底颠覆悲观锁的阻塞模式，采用**无锁、自旋重试**的机制实现线程安全，极大提升并发吞吐量。Java中Atomic原子类、数据库版本号、CAS操作均为乐观锁的经典落地，是读多写少高并发场景的最优解。

## 2.1 乐观锁的核心思想

### 2.1.1 乐观锁的定义：默认多线程竞争冲突概率低

**乐观锁** 是一种乐观的并发设计思想：默认多线程并发操作共享资源时，**大概率不会发生数据竞争冲突**。

因此乐观锁不会提前加锁阻塞线程，所有线程均可无锁并发操作资源，仅在数据提交更新时校验数据是否被篡改，冲突概率极低、性能极高。该思想优先保障**并发性能**，容忍极小概率的冲突重试。

### 2.1.2 乐观锁的核心逻辑：不加锁，操作前检查数据是否被修改

乐观锁核心流程可概括为：**无锁执行、校验版本、成功更新、失败重试**，具体步骤：

1. 线程无锁读取共享数据，记录当前数据版本/值；

2. 线程执行业务计算、数据修改逻辑；

3. 提交更新前，校验当前内存数据版本是否等于读取时的版本；

4. **版本一致（无冲突）**：直接更新数据，操作成功；

5. **版本不一致（有冲突）**：说明数据被其他线程修改，放弃本次更新，重试操作。

### 2.1.3 乐观锁的底层实现：无锁机制+版本校验

乐观锁全程**不依赖操作系统互斥锁、无线程阻塞、无上下文切换**，底层基于两大核心机制实现：

- **版本号校验机制**：通过版本号、时间戳标记数据状态，判断数据是否被篡改（数据库乐观锁核心）；

- **CAS硬件级原子指令**：比较并交换，内存值、预期值不一致则重试（Java乐观锁核心）。

所有线程并行执行，仅冲突时自旋重试，最大化利用CPU并发能力。

## 2.2 Java中乐观锁的典型实现

### 2.2.1 CAS（Compare-And-Swap）：硬件级无锁原子操作

**CAS（比较并交换）** 是Java乐观锁的底层核心，是**CPU硬件层面支持的原子指令**，无需加锁即可保证多线程操作原子性。

CAS包含三个核心参数：**内存值V、预期旧值A、更新新值B**，执行逻辑：

1. 判断当前内存值V是否等于预期旧值A；

2. 相等：说明数据未被修改，原子性将V更新为B，返回成功；

3. 不相等：说明数据已被其他线程修改，放弃更新，返回失败并自旋重试。

CAS 是无锁操作，全程用户态执行，无内核切换、无线程阻塞，并发性能远超悲观锁。

### 2.2.2 数据库版本号机制：业务层乐观锁实现

数据库场景无CAS指令，通过**版本号字段**实现乐观锁，是业务开发高频用法，适配数据库高并发更新场景。

**实现方案**：数据表新增version版本字段，数据每次更新版本号+1；更新时校验版本号一致才执行更新。

```sql
-- 数据表包含version版本字段
UPDATE goods_stock 
SET stock = stock - 1, version = version + 1 
WHERE id = 1 AND version = 旧版本号;

```

若更新行数为0，说明数据已被其他线程修改，本次更新冲突，业务层重试或降级处理。

### 2.2.3 Atomic原子类：基于CAS的Java层乐观锁实现

JUC **Atomic系列原子类** 是CAS乐观锁的Java落地封装，包含AtomicInteger、AtomicLong、AtomicBoolean等，全程基于CAS无锁实现原子操作，替代悲观锁完成数值安全更新。

```java

import java.util.concurrent.atomic.AtomicInteger;

/**
 * AtomicInteger乐观锁CAS示例
 * 无锁、无阻塞，保证数值自增原子性
 */
public class AtomicCasDemo {
    // 基于CAS的原子整型，默认初始值0
    private static final AtomicInteger COUNT = new AtomicInteger(0);

    public static void increment() {
        // 底层循环CAS自旋重试
        COUNT.getAndIncrement();
    }

    public static void main(String[] args) throws InterruptedException {
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 10000; i++) increment();
        });
        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 10000; i++) increment();
        });
        t1.start();
        t2.start();
        t1.join();
        t2.join();
        // 结果固定20000，无并发错乱，性能优于悲观锁
        System.out.println("最终count值：" + COUNT.get());
    }
}

```

## 2.3 乐观锁的优缺点与适用场景

### 2.3.1 优点：无阻塞、无上下文切换、高并发性能好

- **并发性能极高**：无线程阻塞、无队列排队，多线程并行执行，最大化利用多核CPU；

- **无上下文切换开销**：全程用户态CAS自旋，无需内核态切换，系统开销极低；

- **线程无挂起**：冲突仅自旋重试，不放弃CPU执行权，响应速度快；

- **吞吐量远超悲观锁**：读多写少场景下性能提升数十倍。

### 2.3.2 缺点：存在ABA问题、自旋开销、只能保证单个变量原子性

- **经典ABA问题**：线程读取数据为A，其他线程先改为B、再改回A，当前线程CAS校验通过，无法感知数据被修改过，引发数据隐患；

- **自旋空转消耗CPU**：高并发冲突场景下，CAS持续重试自旋，会耗尽CPU资源；

- **仅支持单个变量原子性**：无法保证多变量、复合操作的原子性；

- **不适合长耗时操作**：业务耗时越长，冲突概率越高，自旋重试开销越大。

### 2.3.3 适用场景：读多写少、并发冲突概率低的场景

乐观锁优先保障并发性能，适配冲突概率低的业务场景：

1. **读多写少高并发场景**：接口查询、缓存读取、静态数据更新；

2. **并发冲突概率低场景**：极少出现多线程同时修改同一数据的业务；

3. **短耗时原子操作**：单纯数值增减、状态修改等简单原子操作；

4. **高吞吐量优先场景**：允许极小概率重试，优先保证系统并发能力。

# 3. CAS（Compare-And-Swap）的工作原理

CAS 是整个Java乐观锁体系的**底层核心基石**，区别于悲观锁的阻塞机制，CAS依靠硬件指令实现无锁原子操作，是Atomic原子类、AQS自旋抢锁、ConcurrentHashMap无锁并发的核心实现原理，是高并发高性能编程的关键技术。

## 3.1 CAS的核心定义与操作流程

### 3.1.1 CAS的定义：比较并交换的原子操作

**CAS（Compare And Swap，比较并交换）** 是一种**硬件级的无锁原子操作**，是乐观锁最核心的实现方式。

其核心设计思想：**不加锁、不阻塞线程，通过“内存值比对”的方式判断数据是否被篡改，无冲突则更新数据，有冲突则重试或放弃**。CAS 全程在用户态执行，无需线程阻塞、无需上下文切换，相较于悲观锁，极大提升了并发执行性能。

CAS 所有的比较、判断、赋值操作，**整体具备不可分割的原子性**，多线程环境下不会出现指令交错问题。

### 3.1.2 CAS的三个关键参数：内存地址V、预期值A、新值B

每一次 CAS 操作都依赖三个核心参数，三者缺一不可，是CAS实现原子更新的核心要素：

- **内存地址V**：共享变量在主内存中的真实存储地址，CAS 直接操作主内存数据，不操作线程工作内存副本；

- **预期旧值A**：线程读取到的、当前预期的变量旧值，是数据是否被篡改的判断依据；

- **更新新值B**：线程计算完成后，需要写入主内存的最新变量值。

简单概括：CAS 就是**基于内存地址，比对旧值，合法则替换新值**的原子逻辑。

### 3.1.3 CAS的执行流程：V==A → 更新为B；否则重试/失败

CAS 拥有固定的原子执行流程，全程无中断、无指令交错，具体步骤如下：

1. **读取数据**：线程从主内存地址V中读取共享变量的值，记录为预期旧值A；

2. **业务计算**：线程基于旧值A完成业务逻辑计算，得到需要更新的新值B；

3. **原子比对**：再次读取主内存地址V的最新值，与预期旧值A做全量比对；

4. **分支处理**：
            

   - 若 **V == A**：证明数据未被其他线程修改，原子性将内存值V更新为新值B，操作成功结束；

   - 若 **V != A**：证明数据已被其他线程篡改，本次更新冲突，直接返回失败或进入自旋重试逻辑。

**核心特点**：整个比对+交换的过程由CPU硬件保证原子性，软件层面无法被多线程打断。

## 3.2 CAS的底层实现原理

### 3.2.1 CPU指令级支持：cmpxchg指令

很多开发者误以为CAS是Java代码实现的原子逻辑，实际上：**CAS 的原子性并非JDK保证，也非虚拟机保证，而是CPU硬件指令保证**。

Java 中所有 CAS 操作，最终都会调用 CPU 底层的 **cmpxchg（比较交换）指令**，该指令是CPU硬件原生支持的原子指令，具备执行不可分割的特性。

普通的读取、赋值、比较都是单条简单指令，多线程下会出现指令交错；而 cmpxchg 是复合原子指令，硬件层面锁定执行过程，确保“比较+交换”两步操作全程原子，彻底杜绝并发指令错乱问题。

### 3.2.2 总线锁与缓存一致性协议：保证多CPU下的原子性

在多核CPU环境下，多个核心可能同时执行cmpxchg指令，仍存在并发竞争风险，CPU通过两套机制保障多核CAS原子性：

1. **总线锁（早期方案）**：当一个CPU核心执行cmpxchg指令时，锁定系统总线，其他CPU核心无法访问主内存，保证当前操作独占内存资源。缺点是总线阻塞，性能损耗较大；

2. **缓存一致性协议（现代CPU主流方案）**：基于CPU缓存行机制，通过MESI协议保证多核缓存数据一致性。仅锁定当前操作的缓存行，不阻塞全局总线，在保证原子性的同时，大幅提升并发性能。

**核心结论**：单核CPU依靠指令本身保证原子性，多核CPU依靠缓存一致性协议+硬件指令双重保障CAS原子性。

### 3.2.3 Java中Unsafe类的native方法调用

Java 无法直接操作硬件指令，JDK 通过 **sun.misc.Unsafe** 工具类作为底层入口，调用本地native方法，间接执行CPU cmpxchg指令。

Unsafe 是JDK底层不安全工具类，具备直接操作内存、调用硬件指令、操作对象字段内存地址的能力，所有原子类、AQS的CAS方法最终都会下沉到Unsafe的native方法。

核心源码示例（Unsafe核心CAS方法）：

```java
// 基于对象、内存偏移量、旧值、新值实现CAS原子更新
public final native boolean compareAndSwapInt(Object o, long offset, int expected, int x);

```

该方法由C++底层实现，最终汇编生成cmpxchg硬件指令，是Java CAS的最底层入口。

## 3.3 CAS在Java中的应用场景

### 3.3.1 Atomic原子类的核心实现

JUC 下所有 **Atomic系列原子类**（AtomicInteger、AtomicLong、AtomicBoolean、AtomicReference等）的底层全部基于CAS实现，完全摒弃悲观锁阻塞机制，实现无锁原子操作。

原子类中的自增、更新、赋值操作，底层都是循环CAS重试逻辑，冲突则自旋重试，无冲突则快速更新，是轻量级并发安全的首选方案。

### 3.3.2 AQS框架中的CAS自旋

AQS 作为JUC锁体系的核心框架，其**抢锁、入队、状态更新**全部依赖CAS自旋：

- 线程尝试抢占独占锁时，通过CAS修改state状态；

- 线程入队、更新队列节点状态依赖CAS无锁更新；

- 失败线程不会立即阻塞，而是先CAS自旋尝试多次抢锁，提升并发性能。

可以说：**没有CAS，就没有AQS的高性能无锁自旋机制**。

### 3.3.3 并发容器（如ConcurrentHashMap）的无锁优化

JDK1.8及以上的 **ConcurrentHashMap** 大量使用CAS做无锁优化：

- 数组初始化、节点插入、表头替换操作，优先使用CAS无锁更新；

- 仅当哈希冲突严重时，才启用synchronized悲观锁；

- 通过CAS+分段锁的组合，实现高并发下的极致性能。

除此之外，ConcurrentLinkedQueue等无锁并发队列，全程基于CAS实现线程安全。

# 4. CAS的三大问题：ABA问题、自旋开销、只能保证单个变量原子性

CAS 虽然性能远超悲观锁，但并非完美无缺，其无锁、值比对的设计天然存在三大缺陷，也是面试高频考点和生产环境主要避坑点，所有基于CAS的框架都需要针对性解决这三类问题。

## 4.1 ABA问题

### 4.1.1 ABA问题的定义：数据被修改两次后恢复原值，导致CAS误判

**ABA问题** 是CAS最经典的并发漏洞，具体场景：

1. 线程1读取内存值为A，准备执行CAS更新；

2. 线程2抢占CPU，将内存值A修改为B，随后又修改回A；

3. 线程1恢复执行CAS比对，发现内存值仍然是A，判定数据未被修改，执行更新操作；

4. **漏洞核心**：数据实际经历了两次修改，状态已发生流转，但CAS仅比对数值，无法感知中间修改过程，造成**误判放行**。

### 4.1.2 ABA问题的底层成因：CAS仅校验值，不校验修改过程

CAS 的核心比对逻辑仅关注**数据最终值**，完全忽略数据的**修改次数、修改轨迹、状态流转过程**。只要最终原值一致，无论中间经历多少次修改，CAS 都会判定数据安全，这是ABA问题的根本成因。

简单总结：**CAS只看结果、不看过程，导致隐蔽的数据篡改无法被识别**。

### 4.1.3 ABA问题的危害：业务逻辑异常、数据状态不一致

多数简单数值场景下ABA问题无明显影响，但在**状态流转、资金交易、链表结构、库存锁定**等复杂业务中，会引发严重隐患：

- **链表结构错乱**：无锁队列中节点被删除又恢复，导致CAS替换节点异常，引发链表循环、数据丢失；

- **业务状态错乱**：订单状态、支付状态被反复修改，CAS误判导致状态覆盖、业务流程异常；

- **数据一致性问题**：库存、金额等核心数据经历多次流转后，最终值一致但中间流程异常，引发资金隐患。

## 4.2 自旋开销问题

### 4.2.1 自旋开销的成因：CAS失败后不断重试，持续占用CPU

CAS 为了避免线程阻塞，冲突后不会挂起线程，而是采用**循环自旋重试**的机制：不断读取内存值、比对、尝试更新，直至操作成功。

该机制在低冲突场景下性能极佳，但在高并发竞争场景下，大量线程持续空转重试，**全程占用CPU资源，无释放、无休眠**，造成严重的CPU资源消耗。

### 4.2.2 高竞争场景下的自旋风暴问题

**自旋风暴** 是CAS高并发下的典型生产问题：当大量线程同时竞争同一个共享变量时，CAS更新成功率极低，所有线程持续无限自旋重试，CPU利用率瞬间拉满，导致服务卡顿、接口超时、线程无法执行正常业务逻辑。

典型场景：秒杀瞬时并发、热点数据更新、全局计数器修改，极易触发CAS自旋风暴。

### 4.2.3 自旋次数与CPU资源消耗的平衡

CAS 自旋存在明显的性能两极分化：

- 低冲突：少量自旋即可成功，性能远优于悲观锁阻塞；

- 高冲突：无限自旋耗尽CPU，性能远低于悲观锁串行阻塞。

因此生产中不会使用无限CAS自旋，通常会设置**有限自旋次数**：超过阈值则放弃自旋、降级为悲观锁或返回失败，平衡重试收益与CPU开销。

## 4.3 只能保证单个变量原子性的问题

### 4.3.1 复合操作的非原子性：多变量更新无法通过CAS原子化

CAS 的原子性仅针对**单个变量、单次赋值操作**生效，无法保证多个变量联动更新、复合业务操作的原子性。

如果业务需要同时更新A、B、C三个变量，三次独立CAS操作之间会出现指令交错，无法保证整体事务原子性，会出现**部分更新成功、部分失败**的数据不一致问题。

### 4.3.2 典型场景：多个变量的关联更新操作

生产中大量场景需要多变量原子联动更新，CAS无法单独实现：

- 订单场景：同时更新订单状态、支付金额、更新时间三个字段；

- 库存场景：同时更新可用库存、锁定库存、已售库存；

- 统计场景：同时更新访问次数、今日统计、累计统计。

### 4.3.3 多变量场景下的局限性

CAS 仅能保证单一变量的无锁安全，面对多变量复合操作存在天然短板，此时有两种解决方案：

1. 将多个变量封装为一个对象，使用对象维度的CAS更新（AtomicReference）；

2. 复杂事务场景直接使用悲观锁（synchronized/ReentrantLock）保证整体原子性。

**核心选型结论**：简单单变量更新用CAS乐观锁，复杂多变量、事务性更新用悲观锁。

# 5. ABA问题的解决：版本号机制、AtomicStampedReference

在CAS三大问题中，自旋开销、单变量限制可通过业务选型规避，而ABA问题属于隐蔽性极强、难以排查的底层BUG，必须通过技术手段彻底解决。业界主流解决方案分为两大类：**版本号校验机制** 和 **标记位校验机制**。

## 5.1 版本号机制

### 5.1.1 版本号机制的核心思想：每次修改更新版本号，记录修改次数

版本号机制是解决ABA问题最通用、最直观的方案，核心思路：**放弃单纯的值比对，采用「原值+版本号」双重比对机制**。

核心规则：

- 为共享数据新增一个独立的**版本号字段**，初始值为0；

- **每一次数据修改，无论值是否变化，版本号自增1**；

- CAS更新时，同时校验「数据值+版本号」，只有两者完全匹配才允许更新。

此时即使数据值被修改后恢复原值，版本号已经发生变化，CAS可以精准识别数据被篡改的过程，彻底杜绝ABA误判。

### 5.1.2 版本号+数据的组合校验方式

组合校验完整流程：

1. 线程读取数据值A + 版本号V1；

2. 其他线程修改数据为B、版本号+1，再改回A、版本号+1（此时版本号为V3）；

3. 当前线程执行CAS，比对数据值A一致，但版本号V1≠V3，判定数据已被修改，更新失败；

4. 彻底规避ABA误判问题。

### 5.1.3 数据库乐观锁的版本号实现示例

版本号机制在数据库乐观锁中应用最广泛，是生产标准写法，完整落地示例：

**数据表结构（包含版本号字段）**

```sql
CREATE TABLE `goods` (
  `id` BIGINT PRIMARY KEY COMMENT '商品ID',
  `stock` INT DEFAULT 0 COMMENT '库存数量',
  `version` INT DEFAULT 0 COMMENT '乐观锁版本号'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

```

**业务更新SQL（双重校验+版本自增）**

```sql
-- 仅当库存和版本号都匹配时，才更新库存并自增版本号
UPDATE goods 
SET stock = stock - 1, version = version + 1 
WHERE id = 1 AND stock = 100 AND version = 0;

```

该方案彻底解决数据库更新的ABA问题，保证库存、订单等核心数据的一致性。

## 5.2 AtomicStampedReference的实现

### 5.2.1 AtomicStampedReference：带版本戳的引用原子类

**AtomicStampedReference** 是JDK1.5提供的、**专门解决CAS ABA问题的引用类型原子类**，核心是为引用类型数据增加一个**时间戳/版本戳（stamp）**，实现版本号机制的Java代码落地。

普通AtomicReference仅校验数据引用值，而AtomicStampedReference同时校验**引用对象+版本戳**，彻底杜绝ABA漏洞。

### 5.2.2 类结构解析：value+stamp双字段存储

AtomicStampedReference 内部封装了一个静态内部类Pair，存储双核心字段：

- **value**：存储真实的共享数据引用；

- **stamp**：存储数据版本戳，对应版本号机制的版本字段。

每次数据修改，必须同步更新stamp版本值，保证数据和版本一一对应。

### 5.2.3 compareAndSet()方法：同时校验引用值与版本戳

AtomicStampedReference 的核心CAS方法 **compareAndSet**，实现双重校验逻辑，完整可运行代码示例：

```java
import java.util.concurrent.atomic.AtomicStampedReference;

/**
 * AtomicStampedReference 解决ABA问题演示
 */
public class AtomicStampedDemo {
    public static void main(String[] args) {
        // 初始化数据：初始值100，初始版本戳0
        AtomicStampedReference<Integer> atomic = new AtomicStampedReference<>(100, 0);

        // 线程1：读取值和版本号
        int oldValue = atomic.getReference();
        int oldStamp = atomic.getStamp();

        // 线程2：模拟ABA操作 100→200→100，版本号连续更新
        new Thread(() -> {
            // 第一次修改
            atomic.compareAndSet(100, 200, atomic.getStamp(), atomic.getStamp() + 1);
            // 第二次改回原值
            atomic.compareAndSet(200, 100, atomic.getStamp(), atomic.getStamp() + 1);
        }).start();

        // 主线程尝试CAS更新：值相同，但版本戳已变更，更新失败，彻底解决ABA
        boolean result = atomic.compareAndSet(oldValue, 300, oldStamp, oldStamp + 1);
        System.out.println("CAS更新结果：" + result); // 输出false，更新失败
    }
}

```

**核心结论**：即使数据值恢复原样，版本戳已经改变，CAS会精准拦截，彻底解决ABA问题。

## 5.3 其他ABA问题解决方案

### 5.3.1 使用AtomicMarkableReference：标记修改状态

**AtomicMarkableReference** 是JDK提供的轻量级ABA解决方案，不同于版本号的数字递增，该类通过一个**布尔标记位（mark）**记录数据是否被修改过。

核心特点：

- 仅记录「是否被修改」，不记录修改次数；

- 适合**只关心数据是否被动过，不关心修改次数**的场景；

- 开销比AtomicStampedReference更低，性能更优。

### 5.3.2 使用AtomicReference配合自定义版本号

如果需要自定义版本规则、灵活控制版本逻辑，可以手动封装实体类，结合AtomicReference实现自定义版本号机制，适配复杂业务场景。

实现思路：新建业务实体，包含「业务数据+自定义版本号」两个字段，通过AtomicReference对整个实体做CAS更新，手动控制版本递增逻辑，灵活性更高。

### 5.3.3 业务层避免ABA问题的设计思路

除了技术层面的解决方案，业务层设计也可从源头规避ABA问题，是生产最优实践：

- **避免数据回滚场景**：业务设计上禁止数据修改后恢复原值的流转逻辑；

- **使用唯一标识替代原值比对**：基于时间戳、唯一ID做校验，而非业务数值；

- **短事务优先**：缩短CAS操作的业务耗时，减少中间被篡改的概率；

- **高敏感业务强制版本校验**：资金、订单、库存场景，一律启用版本号机制。

# 6. Atomic原子类体系：AtomicInteger/AtomicLong/AtomicReference

Atomic原子类是JDK1.5随JUC包推出的**无锁线程安全工具类**，全程基于CAS乐观锁实现，摒弃了synchronized、ReentrantLock的阻塞机制，在轻量级并发场景下性能远超悲观锁，是Java无锁并发编程的核心落地组件。

## 6.1 Atomic原子类体系概述

### 6.1.1 Atomic原子类的核心作用：基于CAS实现无锁原子操作

Atomic原子类的核心定位：**在不使用悲观锁、不阻塞线程的前提下，保证基础变量、引用对象、数组字段的读写、增减、替换操作的原子性**。

其底层完全依赖Unsafe类的CAS硬件指令，通过循环自旋重试机制解决并发冲突，全程用户态执行、无线程阻塞、无上下文切换，专门用于解决简单变量的并发安全问题，替代传统的同步代码块，实现高性能线程安全。

**核心价值**：以极小的CPU开销，解决轻量级并发安全问题，最大化提升并发吞吐量。

### 6.1.2 Atomic体系的分类：基础类型、引用类型、数组类型、字段更新器

JUC Atomic体系工具类分工明确，覆盖Java所有基础数据类型与对象场景，完整分为四大类，适配不同业务需求：

1. **基础原子类**：AtomicInteger、AtomicLong、AtomicBoolean，用于基本数据类型的原子操作；

2. **引用原子类**：AtomicReference、AtomicStampedReference、AtomicMarkableReference，用于对象引用的原子更新，解决ABA问题；

3. **数组原子类**：AtomicIntegerArray、AtomicLongArray、AtomicReferenceArray，用于数组内部元素的独立原子更新；

4. **字段更新器**：AtomicIntegerFieldUpdater、AtomicLongFieldUpdater、AtomicReferenceFieldUpdater，用于对象指定字段的无锁原子更新，节省内存；

5. **高并发累加器**：LongAdder、DoubleAdder、LongAccumulator，JDK1.8优化，超高并发场景替代基础原子类，解决CAS自旋风暴问题。

### 6.1.3 Atomic原子类与synchronized的性能对比

Atomic（乐观无锁）与synchronized（悲观锁）是基础变量并发安全的两大方案，核心性能与场景对比如下：

| 对比维度   | Atomic原子类（乐观锁）   | synchronized（悲观锁）             |
| ---------- | ------------------------ | ---------------------------------- |
| 实现机制   | CAS无锁自旋，用户态执行  | 对象监视器锁，涉及内核态切换       |
| 线程状态   | 不阻塞，仅自旋空转       | 竞争失败线程阻塞挂起               |
| 低竞争性能 | 极优，无切换开销         | 较好，JDK锁升级优化后开销低        |
| 高竞争性能 | 差，自旋风暴耗尽CPU      | 稳定，串行执行无无限空转           |
| 适用场景   | 轻量级、单变量、读多写少 | 复杂业务、多变量、高竞争、事务场景 |

## 6.2 基础原子类

### 6.2.1 AtomicInteger：int类型的原子操作

**AtomicInteger** 是专门针对int基本数据类型的原子工具类，可保证int变量的自增、自减、累加、赋值、比对更新操作的原子性，彻底解决多线程i++线程不安全问题。

底层通过volatile修饰value变量保证可见性，结合CAS自旋保证原子性，无锁无阻塞。

```java
import java.util.concurrent.atomic.AtomicInteger;

/**
 * AtomicInteger 基础使用示例
 * 替代普通int，实现线程安全自增
 */
public class AtomicIntegerDemo {
    // 原子整型变量，初始值0
    private static final AtomicInteger COUNT = new AtomicInteger(0);

    public static void main(String[] args) throws InterruptedException {
        // 开启10个线程，每个线程自增1000次
        for (int i = 0; i < 10; i++) {
            new Thread(() -> {
                for (int j = 0; j < 1000; j++) {
                    // 原子自增，线程安全
                    COUNT.getAndIncrement();
                }
            }).start();
        }
        Thread.sleep(2000);
        // 最终结果固定10000，无并发错乱
        System.out.println("最终结果：" + COUNT.get());
    }
}

```

### 6.2.2 AtomicLong：long类型的原子操作

**AtomicLong** 作用与AtomicInteger完全一致，专门用于long长整型变量的原子操作，适用于主键自增、大数值计数器、时间戳统计等场景。

在64位JVM下long读写本身原子，但并发累加、复合操作仍存在线程安全问题，必须使用AtomicLong保证并发安全。

### 6.2.3 AtomicBoolean：boolean类型的原子操作

**AtomicBoolean** 用于布尔值的原子更新，核心场景：**多线程下控制开关、状态标识、一次性初始化**。

典型业务场景：全局任务启动开关、系统初始化标识、防止重复执行定时任务。

```java
import java.util.concurrent.atomic.AtomicBoolean;

public class AtomicBooleanDemo {
    // 全局任务开关
    private static final AtomicBoolean START_FLAG = new AtomicBoolean(false);

    public static void main(String[] args) {
        // 多线程尝试启动任务，保证任务仅执行一次
        for (int i = 0; i < 5; i++) {
            new Thread(() -> {
                // CAS原子更新：false改为true，仅一个线程成功
                if (START_FLAG.compareAndSet(false, true)) {
                    System.out.println("任务启动成功，当前线程：" + Thread.currentThread().getName());
                } else {
                    System.out.println("任务已启动，无需重复执行");
                }
            }).start();
        }
    }
}

```

### 6.2.4 核心方法：getAndIncrement()/compareAndSet()/addAndGet()

所有基础原子类通用核心方法，生产高频使用，全部为原子操作：

- **getAndIncrement()**：先获取当前值，再自增1，等价于i++；

- **incrementAndGet()**：先自增1，再获取最新值，等价于++i；

- **getAndAdd(int delta)**：先取值，再累加指定数值；

- **addAndGet(int delta)**：先累加，再取值；

- **compareAndSet(int expect, int update)**：核心CAS方法，预期值匹配则原子更新；

- **get()**：获取当前最新值；**set()**：直接赋值。

## 6.3 引用原子类

### 6.3.1 AtomicReference：引用类型的原子操作

**AtomicReference** 是对象引用的原子操作类，解决**自定义对象、引用类型**的并发安全更新问题，弥补基础原子类仅支持基本类型的短板。

核心作用：保证多线程下对象引用替换、属性更新的原子性，适用于实体类、配置类、全局对象的并发更新场景。

```java
import java.util.concurrent.atomic.AtomicReference;

// 自定义用户实体
class User {
    private String name;
    private Integer age;
    public User(String name, Integer age) {
        this.name = name;
        this.age = age;
    }
    // 省略getter/setter
    public String getName() { return name; }
    public Integer getAge() { return age; }
}

public class AtomicReferenceDemo {
    private static final AtomicReference<User> USER_REF = new AtomicReference<>(new User("张三", 18));

    public static void main(String[] args) {
        // 原子替换对象引用
        USER_REF.compareAndSet(USER_REF.get(), new User("李四", 20));
        System.out.println(USER_REF.get().getName());
    }
}

```

### 6.3.2 AtomicStampedReference：带版本戳的引用原子类

普通AtomicReference存在**ABA问题**，无法识别对象中间修改过程，而**AtomicStampedReference** 通过「引用值+版本戳」双重校验，彻底解决引用类型ABA漏洞，是高敏感业务的首选。

核心特点：每次修改必须同步更新版本号，即使对象引用恢复原值，版本号不可逆，CAS可精准拦截篡改。

### 6.3.3 AtomicMarkableReference：带修改标记的引用原子类

**AtomicMarkableReference** 是轻量级防ABA方案，维护一个 **boolean 标记位** 替代版本号，仅记录数据「是否被修改过」，不记录修改次数。

适用场景：只需判断数据是否被动过、无需追溯修改次数的业务，相比AtomicStampedReference开销更低、性能更优。

## 6.4 其他Atomic工具类

### 6.4.1 数组原子类：AtomicIntegerArray/AtomicLongArray

普通数组元素读写存在线程安全问题，数组原子类专门用于**数组内部元素的独立原子更新**，无需锁定整个数组，粒度更细、并发性能更高。

核心优势：支持数组单个元素独立CAS更新，多线程可同时操作不同数组元素，互不阻塞，并发度极高。

### 6.4.2 字段更新器：AtomicIntegerFieldUpdater/AtomicReferenceFieldUpdater

字段更新器是**轻量化、节省内存**的原子方案，无需创建原子类对象，直接对普通实体类的指定字段做CAS原子更新。

使用前提：字段必须被 **volatile 修饰**、非static、非final，保证可见性与可更新性。

适用场景：大量对象字段并发更新，规避创建大量原子类对象造成的内存冗余。

### 6.4.3 累加器：LongAdder/DoubleAdder（高并发优化）

**LongAdder** 是JDK1.8推出的高并发累加器，是AtomicLong的升级版，彻底解决高并发场景下CAS自旋风暴、CPU空转严重的问题，是生产高并发计数器的最优解。

**核心优化原理：分段累加**

- AtomicLong：所有线程竞争同一个value，高冲突自旋严重；

- LongAdder：将数据分散到多个Cell分段数组，不同线程竞争不同分段，分散竞争压力；

- 最终结果 = 基础值base + 所有分段Cell值累加，大幅降低CAS冲突概率。

**生产选型结论**：低并发计数器用AtomicLong，高并发秒杀、统计计数**必须用LongAdder**。

# 7. 乐观锁与悲观锁的适用场景对比

掌握两类锁的原理与缺陷后，核心落地能力就是**精准锁选型**。没有绝对最优的锁，只有最适配业务场景的锁，本节通过场景、性能、实践三个维度，统一锁选型标准，解决生产锁滥用问题。

## 7.1 适用场景对比

### 7.1.1 悲观锁适用场景：写多读少、高竞争、业务复杂场景

悲观锁牺牲并发性能、保障数据强一致性，适配以下四类核心场景：

1. **写多读少场景**：数据修改频率远高于查询，并发冲突概率极高，CAS重试开销大于阻塞开销；

2. **高并发竞争场景**：热点数据更新、瞬时高并发写入，乐观锁自旋风暴严重，悲观锁串行更稳定；

3. **复杂业务事务场景**：多变量联动更新、长耗时业务、数据库事务操作，CAS无法保证复合操作原子性；

4. **强一致性核心业务**：资金交易、订单结算、库存扣减、账户变动，不允许任何数据错乱与重试覆盖。

### 7.1.2 乐观锁适用场景：读多写少、低竞争、轻量级原子操作场景

乐观锁牺牲部分容错性、极致提升并发性能，适配以下场景：

1. **读多写少场景**：绝大多数请求为查询，少量更新，冲突概率极低，无锁性能优势极大；

2. **轻量级单变量操作**：计数器、状态位、统计数值等简单原子更新，无复杂业务逻辑；

3. **高吞吐量优先场景**：接口限流统计、访问量计数、缓存更新，优先保证系统吞吐；

4. **短耗时业务**：操作执行极快，CAS自旋次数少、开销极低。

### 7.1.3 混合锁场景：分段锁、读写锁的结合应用

生产绝大多数业务并非纯读或纯写，需要**乐观锁+悲观锁混合使用**，兼顾性能与一致性：

- **分段锁思想**：如ConcurrentHashMap，分段CAS无锁 + 局部synchronized悲观锁，低冲突无锁执行、高冲突局部阻塞；

- **读写锁思想（ReentrantReadWriteLock）**：读操作共享无锁（乐观并发）、写操作独占阻塞（悲观锁），适配读写交织场景；

- **分级重试机制**：低冲突CAS自旋重试，超过阈值降级为悲观锁阻塞，平衡性能与稳定性。

## 7.2 性能对比

### 7.2.1 低竞争场景：悲观锁vs乐观锁的性能差异

**低竞争、低冲突场景下，乐观锁性能远优于悲观锁**。

原因分析：低冲突时CAS几乎一次成功，无自旋、无阻塞、无上下文切换；而悲观锁即使经过锁升级优化，仍存在少量锁校验、对象头标记开销，性能略低于无锁CAS。此时Atomic、CAS乐观锁是最优选择。

### 7.2.2 高竞争场景：悲观锁vs乐观锁的性能差异

**高竞争、高冲突场景下，悲观锁性能稳定性远超乐观锁**。

原因分析：大量线程竞争同一资源时，乐观锁CAS持续自旋重试，CPU占用率飙升，触发自旋风暴，业务执行效率急剧下降；而悲观锁让多余线程阻塞排队，串行有序执行，牺牲并发度换取执行稳定性，整体吞吐量更平稳。

### 7.2.3 自旋次数、上下文切换对性能的影响

两类锁的性能瓶颈核心来源于两个指标：

- **乐观锁瓶颈：自旋次数**：冲突概率越高，自旋重试次数越多，CPU空转开销越大，性能断崖式下跌；

- **悲观锁瓶颈：上下文切换**：线程越多，阻塞唤醒越频繁，用户态内核态切换开销越大，性能缓慢下降。

**核心结论**：自旋开销是指数级增长，切换开销是线性增长，因此高冲突下悲观锁稳定性更强。

## 7.3 锁选型最佳实践

### 7.3.1 优先考虑业务场景：并发冲突概率、读写比例

锁选型第一优先级：**根据业务读写比例与冲突概率判定**：

1. 读多写少、冲突极低 → 优先 **CAS乐观锁/Atomic原子类**；

2. 写多读少、冲突极高 → 优先 **悲观锁（synchronized/ReentrantLock）**；

3. 读写均衡 → 优先 **读写锁、分段混合锁**。

### 7.3.2 性能与复杂度的平衡：简单场景用synchronized，高并发用Atomic

生产极简选型准则，兼顾开发效率与性能：

- **简单同步场景**：代码量少、并发不高，直接使用synchronized，开发简单、无需手动管控、稳定性最高；

- **高并发简单计数**：优先LongAdder、Atomic系列无锁工具；

- **复杂并发场景**：需要超时、可中断、公平锁、多条件唤醒，使用ReentrantLock；

- **数据库并发更新**：读多写少用版本号乐观锁，写多冲突高用事务悲观锁。

### 7.3.3 避免滥用：乐观锁不适合复杂业务场景

生产最大避坑点：**禁止盲目追求性能滥用乐观锁**，以下场景坚决不用乐观锁：

- 多变量复合更新、事务性业务操作；

- 长耗时业务逻辑，自旋重试概率极大；

- 核心金融、交易、库存强一致场景，无法容忍重试覆盖；

- 超高并发热点更新场景，极易触发自旋风暴。
