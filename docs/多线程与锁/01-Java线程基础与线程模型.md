# 1. 线程与进程的核心概念

## 1.1 进程的定义与核心特性

**进程**是操作系统进行**资源分配和调度的最小单位**，是程序的一次动态执行过程。简单来说，我们在电脑上打开的每一个软件、每一个运行的程序，对应操作系统中的一个或多个进程，例如运行的IDEA、浏览器、Java程序等，本质都是独立的进程。

进程具备以下**核心特性**，也是操作系统设计进程的核心逻辑：

- **资源独立性**：每个进程都拥有独立的内存空间（堆内存、方法区、栈内存隔离）、CPU时间片、文件句柄、网络资源等系统资源，进程之间无法直接共享内存数据，完全相互隔离。

- **动态性**：进程不是静态的程序文件，程序是存储在磁盘的静态代码，进程是程序加载到内存后，动态运行的实体，拥有完整的生命周期（创建、就绪、运行、阻塞、销毁）。

- **调度独立性**：操作系统以进程为单位分配CPU资源，不同进程的执行由操作系统内核调度、切换，进程切换成本极高。

- **容错隔离性**：单个进程崩溃，不会影响其他进程的正常运行，例如浏览器崩溃不会导致桌面软件、开发工具闪退，隔离性极强。

## 1.2 线程的定义与核心特性

**线程**是进程内部的**执行单元**，是操作系统进行**CPU调度的最小单位**，也被称为“轻量级进程”。一个进程启动后，默认会创建一个主线程，进程内所有的线程共享该进程的全部系统资源，多个线程并行执行不同的任务，实现进程内的并发处理。

线程具备以下**核心特性**，也是多线程编程的核心依据：

- **资源共享性**：同一进程下的所有线程共享进程的堆内存、方法区、全局变量、文件资源等，仅拥有独立的**栈内存**（存储局部变量、方法调用栈、程序计数器），这也是线程间可以快速通信、数据共享的底层原因。

- **轻量级低开销**：线程的创建、销毁、切换成本远低于进程，因为无需重新分配系统资源，仅需维护线程私有栈内存和调度信息，资源损耗极小。

- **依附性**：线程不能独立存在，必须依附于进程，进程销毁后，内部所有线程都会随之终止。

- **高耦合容错性**：同一进程内，若一个非守护线程出现异常崩溃，若无异常捕获机制，大概率会导致整个进程终止，所有线程全部失效，容错性远低于进程。

## 1.3 进程与线程的核心区别

进程和线程是操作系统并发的两大核心载体，二者的核心差异集中在资源、调度、开销、隔离性等维度，是面试高频考点，同时也是生产环境技术选型的核心依据，具体区别如下表所示：

|对比维度|进程|线程|
|---|---|---|
|定义本质|操作系统资源分配的最小单位|操作系统CPU调度的最小单位|
|资源隔离|完全隔离，独立内存、独立资源|共享进程资源，仅栈内存私有|
|创建/切换开销|极高，需要申请、回收系统资源|极低，仅维护线程私有数据|
|通信方式|复杂，需管道、socket、共享内存等跨进程通信方式|简单，直接共享进程全局变量、堆数据|
|容错性|高，进程崩溃互不影响|低，单线程异常可能导致整个进程宕机|
|依附关系|独立存在，无依附主体|依附进程存在，无法独立运行|
## 1.4 多进程与多线程的适用场景对比

基于进程和线程的特性差异，在服务开发、程序设计中，多进程和多线程拥有完全不同的适用场景，合理选型可以最大化利用系统资源，提升程序并发效率。

### 1.4.1 多进程适用场景

- **高隔离、高容错需求场景**：例如服务器核心服务、独立插件系统，单个服务故障不影响整体系统运行，保证服务稳定性。

- **资源独立占用场景**：例如大数据分布式计算、多用户独立服务，需要独占内存、CPU资源，避免任务相互干扰。

- **跨语言混合开发场景**：不同语言编写的程序无法共享线程资源，只能通过多进程方式独立运行，通过跨进程通信交互数据。

- **CPU密集型重度任务**：多进程可以充分利用多核CPU，规避单进程线程调度的局限性，最大化CPU利用率。

### 1.4.2 多线程适用场景

- **IO密集型场景（核心主流场景）**：例如接口请求、数据库查询、文件读写、网络请求等，IO操作会导致线程阻塞，多线程可以利用阻塞时间切换执行其他任务，大幅提升并发吞吐量，是Java后端服务的核心并发方案。

- **快速通信、数据共享场景**：业务需要频繁共享数据、实时交互，多线程无需复杂通信机制，执行效率更高。

- **低开销、高并发轻量任务**：例如秒杀接口、高频消息推送，需要大量并发执行任务，线程创建切换开销低，支持高并发场景。

# 2. Java线程的生命周期与状态流转

Java线程拥有完整的生命周期，从线程创建到最终销毁，会经历不同的状态变化。JDK1.5及以上版本定义了**6种标准线程状态**，状态之间根据特定触发条件自动流转。掌握线程状态与流转规则，是排查线程死锁、阻塞、卡死等生产问题，以及应对并发面试题的核心基础。

## 2.1 Java线程的5种状态详解

日常开发与面试中，常简化为5种核心运行状态，结合JDK源码标准6种状态整合讲解，覆盖所有实际运行场景。

### 2.1.1 NEW（新建状态）

**状态定义**：线程对象已经通过new关键字创建完成，但尚未调用`start()`启动方法，此时线程仅存在于JVM内存中，是一个普通的Java对象，操作系统尚未创建真实线程，无CPU调度权限。

**核心特性**：

- 仅初始化线程对象的属性、栈内存，无任何执行逻辑；

- 该状态为瞬时初始状态，无法直接执行任务，必须调用`start()`方法触发状态流转；

- 一个线程只能处于一次NEW状态，启动后无法回退。

**代码示例**：

```java
public class ThreadStateDemo {
    public static void main(String[] args) {
        // 仅创建线程对象，未启动，处于NEW状态
        Thread thread = new Thread(() -> System.out.println("线程执行"));
        // 打印线程状态：NEW
        System.out.println("当前线程状态：" + thread.getState());
    }
}

```

### 2.1.2 RUNNABLE（可运行状态）

**状态定义**：调用线程的`start()`方法后，线程进入可运行状态，包含操作系统层面的「就绪状态」和「运行状态」，是Java独有的合并状态。

**核心细分与特性**：

- **就绪状态**：线程已启动，获取到除CPU外的所有资源，等待操作系统分配CPU时间片，随时可以执行；

- **运行状态**：线程成功抢占CPU时间片，正在执行run()方法的业务逻辑；

- Java中无法区分就绪和运行状态，统一标记为RUNNABLE；

- 该状态是线程活跃状态，无阻塞、无等待，持续参与CPU调度竞争。

**代码示例**：

```java
public class ThreadStateDemo {
    public static void main(String[] args) throws InterruptedException {
        Thread thread = new Thread(() -> {
            // 线程持续执行，处于RUNNABLE状态
            while (true) {}
        });
        thread.start(); // 启动线程
        Thread.sleep(100);
        // 打印线程状态：RUNNABLE
        System.out.println("当前线程状态：" + thread.getState());
    }
}

```

### 2.1.3 BLOCKED（阻塞状态）

**状态定义**：线程正在等待获取**synchronized同步锁**，属于被动阻塞状态。当多个线程竞争同一把对象锁时，未获取到锁的线程会进入BLOCKED阻塞队列，等待锁释放后重新竞争CPU资源。

**核心特性**：

- 仅针对**synchronized悲观锁**生效，Lock锁不会触发该状态；

- 线程暂停执行，不占用CPU时间片，不参与CPU调度；

- 当持有锁的线程执行完毕、释放锁后，阻塞线程会自动转为RUNNABLE状态，重新竞争锁。

### 2.1.4 WAITING（无限等待状态）

**状态定义**：线程主动放弃CPU执行权和锁资源，进入**永久无限等待状态**，需要其他线程主动唤醒，否则会一直等待，不会主动恢复执行。

**触发方式**：调用无参的`Object.wait()`、`Thread.join()`方法。

**核心特性**：

- 主动阻塞，释放锁资源，避免线程死占资源；

- 无超时时间，必须依赖其他线程调用`notify()`或`notifyAll()`唤醒；

- 唤醒后不会直接运行，而是转为RUNNABLE状态，重新竞争锁和CPU资源。

### 2.1.5 TIMED_WAITING（限时等待状态）

**状态定义**：线程主动进入**有超时时间的等待状态**，等待指定时间后自动唤醒，无需其他线程干预，是开发中最常用的等待状态。

**触发方式**：调用`Thread.sleep(long)`、带参`Object.wait(long)`、`Thread.join(long)`、`LockSupport.parkNanos()`等方法。

**核心特性**：

- 限时阻塞，时间结束自动唤醒，容错性更高；

- `sleep`方法不会释放锁资源，`wait(long)`会释放锁资源；

- 超时后自动流转为RUNNABLE状态，继续执行后续逻辑。

**代码示例**：

```java
public class ThreadStateDemo {
    public static void main(String[] args) throws InterruptedException {
        Thread thread = new Thread(() -> {
            try {
                // 限时等待2秒，进入TIMED_WAITING状态
                Thread.sleep(2000);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        });
        thread.start();
        Thread.sleep(100);
        // 打印线程状态：TIMED_WAITING
        System.out.println("当前线程状态：" + thread.getState());
    }
}

```

### 2.1.6 TERMINATED（终止状态）

**状态定义**：线程执行完毕、正常退出，或执行过程中抛出异常终止，线程生命周期彻底结束，是线程的最终状态。

**核心特性**：

- 线程执行完run()方法所有逻辑，或被中断终止；

- 线程资源等待JVM垃圾回收，无法再次启动（重复调用start()会抛出异常）；

- 该状态为最终状态，无后续状态流转。

## 2.2 线程状态流转条件与触发场景

Java线程的六种状态并非孤立存在，会根据代码执行、锁竞争、等待唤醒等操作自动流转，完整的流转逻辑是排查线程卡死、阻塞、死锁问题的核心依据，完整流转规则如下：

1. **NEW → RUNNABLE**：新建线程对象后，首次调用`start()`方法，线程初始化完成，进入可运行状态。

2. **RUNNABLE → BLOCKED**：线程执行代码时，遇到`synchronized`同步代码块/方法，锁已被其他线程占用，线程竞争锁失败，进入阻塞状态。

3. **BLOCKED → RUNNABLE**：持有synchronized锁的线程执行完毕、释放锁资源，阻塞队列中的线程成功竞争到锁，恢复可运行状态。

4. **RUNNABLE → WAITING**：线程调用无参`wait()`、`join()`方法，主动放弃锁和CPU资源，进入无限等待。

5. **RUNNABLE → TIMED_WAITING**：线程调用`sleep()`、带参`wait()`、`join()`等限时等待方法，进入限时阻塞。

6. **WAITING/TIMED_WAITING → RUNNABLE**：WAITING线程被其他线程唤醒（notify/notifyAll）、TIMED_WAITING线程等待时间结束，重新参与CPU调度。

7. **所有状态 → TERMINATED**：线程正常执行完毕、被中断终止、异常退出，最终进入终止状态。

**核心避坑点（面试高频）**：**线程从阻塞/等待状态唤醒后，不会直接运行，必须先转为RUNNABLE状态，重新竞争CPU和锁资源**，这是绝大多数并发初学者的认知误区。

## 2.3 源码视角：`Thread.State`枚举解析

JDK通过`Thread.State`枚举类定义了Java线程的6种标准状态，是线程状态的官方定义，所有线程状态的获取、判断均基于该枚举实现，以下为JDK1.8源码完整解析：

```java
public enum State {
    /**
     * 新建状态：线程对象创建，未调用start()
     */
    NEW,

    /**
     * 可运行状态：包含就绪、运行两种场景
     * 正在执行或等待CPU时间片调度
     */
    RUNNABLE,

    /**
     * 阻塞状态：等待获取synchronized同步锁
     * 仅针对内置锁生效
     */
    BLOCKED,

    /**
     * 无限等待状态：无超时，需主动唤醒
     * 触发方法：Object.wait()、Thread.join()
     */
    WAITING,

    /**
     * 限时等待状态：有超时时间，自动唤醒
     * 触发方法：sleep、带参wait、带参join等
     */
    TIMED_WAITING,

    /**
     * 终止状态：线程执行结束，生命周期终止
     */
    TERMINATED;
}
```

**源码核心总结**：

- Java官方严格定义**6种线程状态**，不存在所谓的“运行状态”，运行和就绪统一合并为RUNNABLE；

- BLOCKED状态仅针对synchronized锁，AQS、Lock锁的阻塞不会进入该状态，这是锁机制的核心底层区别；

- 所有线程状态均可通过`thread.getState()`方法获取，可用于生产环境线程堆栈排查、问题定位。

# 3. Java创建线程的4种方式

在Java中，线程创建本质只有两类核心逻辑：**继承Thread类**、**实现Runnable/Callable接口**。我们常说的四种创建方式，是结合实际使用形态划分的：继承Thread、实现Runnable、实现Callable+FutureTask、线程池创建线程。其中前三种为原生手动创建方式，线程池为工程化规范创建方式，也是生产环境唯一推荐使用的方式。

## 3.1 方式一：继承`Thread`类

### 3.1.1 实现方式与代码示例

实现原理：Java中`Thread`类是线程的核心类，继承该类后，重写**run()方法**定义线程执行逻辑，通过调用实例的`start()`方法启动线程，由JVM调用操作系统内核完成线程创建与调度。

核心注意：**start()方法是真正创建并启动线程，run()只是普通业务方法，直接调用run()不会开启新线程，仅在当前主线程执行**。

完整可运行代码示例：

```java
/**
 * 方式一：继承Thread类创建线程
 */
public class ThreadExtendDemo extends Thread {

    // 重写run方法：定义线程执行的业务逻辑
    @Override
    public void run() {
        // 获取当前执行线程名称
        System.out.println("线程执行中，线程名称：" + Thread.currentThread().getName());
    }

    public static void main(String[] args) {
        // 1. 创建线程对象（NEW状态）
        ThreadExtendDemo thread = new ThreadExtendDemo();
        // 2. 调用start()启动线程（转为RUNNABLE状态，JVM调度执行）
        thread.start();
    }
}

```

### 3.1.2 优缺点分析与使用限制

**优点**：

- 实现简单、代码直观，入门门槛极低，适合新手理解线程执行逻辑；

- 内部可直接调用Thread类的所有方法，无需额外获取线程对象。

**缺点（核心痛点）**：

- **违背Java单继承机制**：类一旦继承Thread类，无法再继承其他父类，扩展性极差；

- 线程和业务逻辑耦合严重：线程执行逻辑绑定在Thread子类中，无法复用任务逻辑；

- 每次执行任务必须创建新线程对象，线程创建销毁开销大，性能低下。

**使用限制**：仅适用于简单测试、Demo演示，**严禁在生产环境使用**。

## 3.2 方式二：实现`Runnable`接口

### 3.2.1 实现方式与代码示例

实现原理：`Runnable`是函数式接口，仅包含一个`run()`抽象方法，用于定义线程任务逻辑。通过将Runnable任务交给Thread类包装，实现线程创建与任务逻辑分离，是最基础的解耦式线程创建方式。

完整可运行代码示例（包含传统写法+Lambda简写）：

```java
/**
 * 方式二：实现Runnable接口创建线程
 */
public class RunnableDemo implements Runnable {

    // 实现run方法：定义线程任务逻辑
    @Override
    public void run() {
        System.out.println("Runnable线程执行，线程名：" + Thread.currentThread().getName());
    }

    public static void main(String[] args) {
        // 1. 传统写法
        RunnableDemo runnableTask = new RunnableDemo();
        Thread thread1 = new Thread(runnableTask);
        thread1.start();

        // 2. Lambda简写（JDK8+，函数式接口支持）
        new Thread(() -> {
            System.out.println("Lambda线程执行，线程名：" + Thread.currentThread().getName());
        }).start();
    }
}
```

### 3.2.2 优缺点分析与适用场景

**优点**：

- 规避了单继承限制：实现接口后依然可以继承其他类、实现其他接口，代码扩展性强；

- **任务与线程解耦**：任务逻辑独立封装，可被多个线程复用，资源利用率更高；

- 代码简洁，支持JDK8 Lambda表达式简化开发。

**缺点**：

- 线程无返回值：run()方法无返回值，无法获取线程执行结果；

- 无法抛出受检异常：run()方法签名无throws声明，内部受检异常必须手动try-catch；

- 依然需要手动创建线程，频繁创建销毁依然存在性能损耗。

**适用场景**：无需获取线程返回结果、简单异步任务、基础并发场景测试。

## 3.3 方式三：实现`Callable`接口（带返回值）

### 3.3.1 `Callable`与`Runnable`的区别

`Callable`是JDK1.5新增的线程任务接口，专门解决Runnable无返回值、无法抛出受检异常的痛点，二者核心区别如下表：

| 对比维度   | Runnable                       | Callable                         |
| ---------- | ------------------------------ | -------------------------------- |
| 方法签名   | void run()                     | V call() throws Exception        |
| 返回值     | 无返回值                       | 支持泛型返回值                   |
| 异常处理   | 无法抛出受检异常，必须内部捕获 | 可抛出所有异常，支持上层捕获处理 |
| 函数式接口 | 是，支持Lambda                 | 是，支持Lambda                   |
| 使用方式   | 直接交给Thread执行             | 必须配合FutureTask包装使用       |

### 3.3.2 `FutureTask`的使用与执行流程

Thread类不支持直接接收Callable任务，必须通过**FutureTask**包装Callable对象。FutureTask是Runnable接口的实现类，同时持有Callable任务引用，具备获取线程执行结果、取消任务、判断任务状态的能力。

**核心执行流程**：

1. 自定义类实现Callable接口，重写call()方法，定义带返回值的任务逻辑；

2. 使用FutureTask包装Callable任务，承接任务执行结果；

3. 将FutureTask对象交给Thread类，创建并启动线程；

4. 线程执行完毕后，通过FutureTask.get()方法获取返回结果。

**核心特性**：`get()`方法是**阻塞方法**，如果线程未执行完毕，主线程会一直阻塞等待结果返回。

### 3.3.3 代码示例：获取线程执行结果

```java
import java.util.concurrent.Callable;
import java.util.concurrent.ExecutionException;
import java.util.concurrent.FutureTask;

/**
 * 方式三：实现Callable接口，支持返回值
 */
public class CallableDemo implements Callable<Integer> {

    // 重写call方法：带返回值、可抛异常
    @Override
    public Integer call() throws Exception {
        System.out.println("Callable线程执行，开始计算任务");
        // 模拟耗时业务计算
        Thread.sleep(1000);
        return 100 + 200;
    }

    public static void main(String[] args) {
        // 1. 创建Callable任务对象
        CallableDemo callableTask = new CallableDemo();
        // 2. FutureTask包装任务，用于接收返回值
        FutureTask<Integer> futureTask = new FutureTask<>(callableTask);
        // 3. 交给Thread执行
        new Thread(futureTask).start();

        try {
            System.out.println("主线程继续执行其他业务...");
            // 4. 获取线程执行结果（阻塞方法，任务完成后才会返回）
            Integer result = futureTask.get();
            System.out.println("线程执行结果：" + result);
        } catch (InterruptedException | ExecutionException e) {
            // 处理线程中断、执行异常
            e.printStackTrace();
        }
    }
}

```

**生产避坑点**：`futureTask.get()`无限阻塞会导致主线程卡死，生产环境建议使用带超时的`get(long timeout, TimeUnit unit)`方法，防止线程永久阻塞。

## 3.4 方式四：使用线程池`ExecutorService`

### 3.4.1 线程池创建线程的优势

前面三种手动创建线程的方式，都存在一个致命问题：**频繁创建、销毁线程，会产生大量系统开销，导致服务吞吐量下降、GC频繁、CPU飙升**。线程池是JDK提供的线程复用管理工具，也是**生产环境唯一推荐的线程创建方式**，核心优势如下：

- **线程复用**：提前初始化线程，任务执行完毕后线程不销毁，循环复用，避免频繁创建销毁的性能损耗；

- **资源可控**：可自定义核心线程数、最大线程数，限制并发线程数量，防止无限创建线程导致系统OOM、CPU打满；

- **任务管理**：内置任务队列、拒绝策略，支持批量处理异步任务，并发调度更稳定；

- **统一管控**：支持线程统一关闭、监控、异常捕获，便于线上问题排查与运维。

### 3.4.2 提交`Runnable`/`Callable`任务的示例

线程池核心通过`ExecutorService`接口实现，常用创建方式分为：工具类创建、手动ThreadPoolExecutor创建（生产推荐）。下面提供两种任务类型的完整执行示例：

```java
import java.util.concurrent.*;

/**
 * 方式四：线程池创建线程（生产推荐）
 */
public class ThreadPoolDemo {
    public static void main(String[] args) {
        // 1. 创建线程池（生产禁止使用Executors工具类，此处仅演示）
        ExecutorService threadPool = Executors.newFixedThreadPool(3);

        // 2. 提交Runnable无返回值任务
        for (int i = 0; i < 3; i++) {
            threadPool.execute(() -> {
                System.out.println("Runnable任务执行，线程：" + Thread.currentThread().getName());
            });
        }

        // 3. 提交Callable有返回值任务
        for (int i = 0; i < 2; i++) {
            Future<Integer> future = threadPool.submit(() -> {
                Thread.sleep(500);
                return i * 10;
            });
            try {
                System.out.println("Callable任务结果：" + future.get());
            } catch (Exception e) {
                e.printStackTrace();
            }
        }

        // 4. 关闭线程池
        threadPool.shutdown();
    }
}

```

### 3.4.3 生产环境推荐实践

**核心禁忌（面试高频+生产红线）**：**禁止使用Executors工具类创建线程池**！

- newFixedThreadPool、newSingleThreadExecutor：队列无界，任务堆积会导致OOM；

- newCachedThreadPool、newScheduledThreadPool：最大线程数无界，会创建大量线程导致CPU耗尽。

**生产标准实践**：手动通过`ThreadPoolExecutor`构造方法创建线程池，自定义核心参数、队列、拒绝策略，适配业务场景。

生产可用极简模板：

```java
import java.util.concurrent.*;

/**
 * 生产环境标准线程池创建方式
 */
public class ProductThreadPool {
    public static void main(String[] args) {
        // 手动创建线程池，所有参数可控
        ThreadPoolExecutor threadPool = new ThreadPoolExecutor(
                5,                          // 核心线程数
                10,                         // 最大线程数
                10L,                        // 空闲线程存活时间
                TimeUnit.SECONDS,           // 时间单位
                new ArrayBlockingQueue<>(100), // 有界任务队列
                Executors.defaultThreadFactory(), // 线程工厂
                new ThreadPoolExecutor.AbortPolicy() // 拒绝策略
        );

        // 执行业务任务
        threadPool.execute(() -> System.out.println("生产线程池任务执行"));

        // 优雅关闭
        threadPool.shutdown();
    }
}
```

## 3.5 四种创建方式对比与选型建议

结合原理、性能、扩展性、生产适用性，对四种线程创建方式做全面汇总对比，同时给出明确的业务选型方案：

| 创建方式              | 返回值         | 异常处理 | 线程复用     | 扩展性           | 生产适用性                       |
| --------------------- | -------------- | -------- | ------------ | ---------------- | -------------------------------- |
| 继承Thread类          | 无             | 受限     | 不支持       | 差（单继承限制） | 禁止使用                         |
| 实现Runnable接口      | 无             | 受限     | 不支持       | 良好             | 简单临时任务可用，不推荐大量使用 |
| 实现Callable接口      | 有泛型返回值   | 完善     | 不支持       | 良好             | 需要返回值的临时任务可用         |
| 线程池ExecutorService | 支持有无返回值 | 完善     | 支持线程复用 | 极强             | **全场景唯一推荐**               |
| **最终选型建议**：    |                |          |              |                  |                                  |

- 日常开发、业务异步、批量任务、高并发场景：**强制使用手动创建的ThreadPoolExecutor线程池**；

- 临时测试、简单Demo：可使用Runnable/Lambda快速创建线程；

- 需要获取异步任务执行结果：线程池提交Callable任务；

- **彻底摒弃继承Thread类创建线程的方式**。

# 4. 线程优先级与守护线程

## 4.1 线程优先级详解

Java线程优先级是JVM提供的线程调度辅助参数，用于**提示操作系统优先调度高优先级线程**。优先级本质是一种调度“建议”，而非强制规则，不能绝对保证线程执行顺序，是并发编程极易踩坑的基础知识点。

### 4.1.1 优先级取值范围与默认值

Java通过整型数值定义线程优先级，取值范围固定为**1~10**，JDK内置三个静态常量统一规范优先级等级，避免硬编码混乱：

- **MIN_PRIORITY = 1**：最低优先级

- **NORM_PRIORITY = 5**：默认优先级（所有线程默认值）

- **MAX_PRIORITY = 10**：最高优先级

默认情况下，主线程（main线程）优先级为5，自定义线程继承创建者线程的优先级，默认同样为5。

### 4.1.2 优先级对线程调度的影响

很多初学者会误以为优先级越高，线程一定越先执行，这是**严重认知误区**。线程优先级的核心作用与真实调度逻辑如下：

Java线程调度采用**抢占式调度模型**，优先级仅作为操作系统CPU时间片分配的参考依据：

- 高优先级线程：获取CPU时间片的**概率更高**，被调度执行的机会更多；

- 低优先级线程：获取CPU时间片的概率更低，大概率处于等待状态；

- **不保证绝对顺序**：在CPU空闲、线程竞争不激烈的场景下，低优先级线程依然可能先执行。

**底层核心结论**：Java线程优先级是**概率性调度建议**，而非强制性规则，无法用于控制线程执行顺序，生产环境禁止依赖优先级做业务逻辑控制。

### 4.1.3 `setPriority()`方法使用与注意事项

Java提供`setPriority(int newPriority)`方法用于动态修改线程优先级，搭配`getPriority()`获取当前优先级，完整可运行示例如下：

```java
public class ThreadPriorityDemo {
    public static void main(String[] args) {
        Thread thread1 = new Thread(() -> System.out.println("低优先级线程执行"), "thread-1");
        Thread thread2 = new Thread(() -> System.out.println("高优先级线程执行"), "thread-2");

        // 设置优先级
        thread1.setPriority(Thread.MIN_PRIORITY);
        thread2.setPriority(Thread.MAX_PRIORITY);

        // 打印优先级
        System.out.println("thread1优先级：" + thread1.getPriority());
        System.out.println("thread2优先级：" + thread2.getPriority());

        thread1.start();
        thread2.start();
    }
}

```

**核心注意事项（生产避坑）**：

- 优先级取值必须在1~10之间，超出范围会直接抛出`IllegalArgumentException`非法参数异常；

- **优先级依赖操作系统适配**：部分操作系统不支持10级优先级细分，会出现优先级映射失效问题，跨平台兼容性差；

- 优先级不可靠：高并发场景下，优先级失效极其常见，绝对不能用于业务时序控制；

- 优先级设置必须在`start()`启动线程之前，启动后设置依然生效，但不推荐动态修改运行中线程优先级。

## 4.2 守护线程（Daemon Thread）概念与特性

Java线程分为两大类：**用户线程（普通线程）**和**守护线程（后台线程）**，二者的核心区别在于JVM退出机制，守护线程是支撑Java后台辅助任务的核心线程类型。

### 4.2.1 守护线程的定义与作用

**守护线程**是一种**后台辅助线程**，专门用于为用户线程提供服务、支撑系统后台任务，不承载核心业务逻辑。

核心特性：**守护线程依附于用户线程存在，当所有用户线程执行完毕退出后，JVM会直接退出，同时终止所有正在运行的守护线程**。

简单理解：用户线程是“干活的主线任务”，守护线程是“打杂的后台辅助任务”，主线任务结束，辅助任务无需继续执行，直接终止。

### 4.2.2 JVM退出条件与守护线程的关系

这是面试**超高频考点**，也是守护线程的核心底层原理：

**JVM虚拟机退出的唯一条件：所有用户线程执行完毕，无存活用户线程**。

详细规则：

- 只要存在任意一个**存活的用户线程**，JVM就不会退出，会持续运行；

- 如果仅剩守护线程在运行，无任何用户线程，JVM会立即终止，守护线程被强制中断；

- 守护线程的终止是**无感知强制终止**，不会抛出异常、不会执行finally收尾代码。

### 4.2.3 常见守护线程示例（GC线程）

JVM底层大量使用守护线程，最典型、最常考的守护线程就是**GC垃圾回收线程**。

运行逻辑：

- 当程序存在用户线程运行业务逻辑时，GC线程持续在后台运行，回收垃圾对象、释放内存；

- 当所有用户线程执行完毕，程序业务结束，JVM无需继续回收垃圾，直接退出，GC守护线程随之终止。

除此之外，JVM的定时监控线程、内存检测线程、信号监听线程均为守护线程。

## 4.3 守护线程的创建与使用场景

### 4.3.1 `setDaemon(true)`方法设置守护线程

Java通过`setDaemon(boolean on)`方法设置线程类型，默认值为false（用户线程），设置为true即为守护线程。

**关键语法规则**：**必须在调用start()启动线程之前设置守护线程属性，否则抛出IllegalThreadStateException异常**。

完整可运行代码示例：

```java
public class DaemonThreadDemo {
    public static void main(String[] args) {
        Thread daemonThread = new Thread(() -> {
            // 无限循环模拟后台持续任务
            while (true) {
                try {
                    Thread.sleep(1000);
                    System.out.println("守护线程后台执行中...");
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            }
        });

        // 重点：启动前设置为守护线程
        daemonThread.setDaemon(true);
        daemonThread.start();

        // 主线程（用户线程）休眠3秒后退出
        try {
            Thread.sleep(3000);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        System.out.println("主线程执行完毕，即将退出");
    }
}

```

**运行结果说明**：主线程执行3秒后退出，所有用户线程结束，JVM直接关闭，守护线程的无限循环会被强制终止，不会持续运行。

### 4.3.2 守护线程与用户线程的区别

为方便记忆与面试答题，整理核心对比表格：

| 对比维度     | 用户线程（普通线程）                   | 守护线程（后台线程）                 |
| ------------ | -------------------------------------- | ------------------------------------ |
| 默认类型     | 所有自定义线程、主线程默认都是用户线程 | 需手动setDaemon(true)设置            |
| JVM退出影响  | 存在存活用户线程，JVM不会退出          | 不影响JVM退出，无用户线程则随JVM终止 |
| 执行任务类型 | 核心业务任务，必须执行完成             | 后台辅助任务、监控任务、日志任务     |
| 终止方式     | 正常执行完毕/异常终止                  | 可被JVM强制终止，无收尾逻辑          |
| 优先级       | 默认优先级5                            | 通常优先级较低，后台静默运行         |

### 4.3.3 守护线程的使用注意事项

守护线程在生产环境使用存在严格限制，核心避坑要点如下：

- **禁止承载核心业务**：守护线程随时可能被JVM强制终止，无法保证任务执行完成，绝对不能处理订单、支付、数据入库等核心业务；

- **finally代码块不保证执行**：线程被强制终止时，不会执行finally收尾逻辑，资源释放、事务提交等操作会失效；

- **设置时机强制要求**：必须在start()之前设置守护线程属性，运行中修改直接报错；

- **适用场景严格受限**：仅用于日志打印、心跳检测、状态监控、缓存清理等可中断、无状态的后台辅助任务。

# 5. 线程上下文切换

线程上下文切换是Java并发性能调优的**核心底层知识点**，高并发场景下大部分性能损耗都来自上下文切换，掌握其原理、开销与优化手段，是进阶高级开发、解决线上CPU偏高、吞吐量偏低问题的关键。

## 5.1 上下文切换的定义与触发场景

### 5.1.1 定义

CPU在多个线程之间轮流执行，当**CPU从一个正在执行的线程，切换到另一个线程执行**时，需要保存当前线程的运行状态（上下文），加载新线程的运行状态，这个**保存+恢复线程状态**的过程，就是**线程上下文切换**。

线程上下文包含：程序计数器、寄存器值、栈指针、线程状态等线程运行的现场数据。

### 5.1.2 常见触发场景

Java中触发上下文切换的核心场景分为四类：

- **时间片耗尽**：操作系统分配给线程的CPU时间片用完，线程让出CPU，触发切换；

- **线程阻塞**：线程执行wait、sleep、锁竞争、IO阻塞等操作，主动放弃CPU，触发切换；

- **线程唤醒**：阻塞线程被唤醒，抢占CPU资源，触发线程切换；

- **高并发多线程竞争**：大量线程同时竞争CPU资源，频繁抢占、让出，导致密集上下文切换。

## 5.2 上下文切换的开销与性能影响

### 5.2.1 核心开销

上下文切换**不执行业务逻辑**，属于纯性能损耗操作，主要开销包含：

- **寄存器保存与恢复开销**：保存当前线程寄存器数据，加载新线程寄存器数据，消耗CPU算力；

- **缓存失效开销**：CPU高速缓存数据失效，新线程需要重新加载缓存，缓存命中率下降；

- **内核态用户态切换开销**：线程调度需要操作系统内核介入，用户态与内核态切换存在性能损耗；

- **线程队列调度开销**：操作系统重新排序线程就绪队列，消耗系统资源。

### 5.2.2 生产性能影响

大量频繁的上下文切换会导致典型线上问题：

- **CPU使用率虚高**：CPU大量算力消耗在切换调度上，业务处理算力不足；

- **系统吞吐量下降**：线程频繁切换，有效业务执行时间被压缩；

- **接口响应变慢、卡顿**：高并发场景下切换耗时累积，导致RT升高。

**核心结论**：**并不是线程越多，并发性能越高**，线程数量超过CPU核心数后，会引发大量上下文切换，性能不升反降。

## 5.3 减少上下文切换的优化方式

结合生产调优经验，整理可落地的上下文切换优化方案，适配高并发系统调优：

- **合理设置线程池线程数**：CPU密集型任务线程数设置为CPU核心数±1，避免大量线程竞争CPU，从根源减少切换；IO密集型任务适度加大线程数，平衡阻塞与切换开销；

- **减少无效锁竞争**：缩小锁粒度、避免大锁、杜绝循环内加锁，减少线程阻塞与唤醒次数，降低切换频率；

- **使用无锁并发方案**：优先使用CAS、ThreadLocal、不可变对象等无锁技术，替代重量级同步锁，避免阻塞切换；

- **批量处理任务**：将零散小任务合并为批量任务，减少线程频繁启停、任务切换次数；

- **避免频繁调用阻塞方法**：减少无意义的sleep、wait、短时IO阻塞，减少主动切换场景；

- **使用协程/轻量级任务**：高并发场景可使用虚拟线程、协程，减少操作系统内核级线程切换开销。

# 6. 面试高频题与易错点

## 6.1 进程与线程的区别？

**面试标准满分回答**：

进程是操作系统**资源分配的最小单位**，线程是操作系统**CPU调度的最小单位**。线程依附于进程存在，一个进程可包含多个线程，线程共享进程资源，是进程内的独立执行单元。二者核心区别如下：

**1. 资源隔离层面**：进程拥有独立内存空间、系统资源，进程间完全隔离；同一进程下的线程共享堆内存、方法区、全局资源，仅私有栈内存与程序计数器。

**2. 开销成本层面**：进程创建、销毁、切换开销极大，需要操作系统重新分配回收资源；线程属于轻量级进程，创建切换开销极低。

**3. 通信方式层面**：进程间通信复杂，需要管道、Socket、共享内存等跨进程通信方式；线程间可直接共享数据，通信简单高效。

**4. 容错性层面**：进程相互独立，单个进程崩溃不影响其他进程；一个线程异常崩溃，大概率导致整个进程退出。

**5. 依附关系**：进程可独立运行，线程必须依托进程运行，进程销毁后线程全部终止。

**易错点避坑**：

- 不要混淆核心定义：资源分配看**进程**，CPU调度看**线程**；

- 并非线程越多性能越好，线程过多会触发大量上下文切换，导致性能下降。

## 6.2 Java线程有几种状态？它们之间如何转换？

**面试标准满分回答**：

JDK1.8 源码定义 **6种标准线程状态**，全部定义在 `Thread.State` 枚举中，分别是：NEW、RUNNABLE、BLOCKED、WAITING、TIMED_WAITING、TERMINATED。

**各状态核心释义**：

- **NEW**：线程对象已创建，未调用 `start()`，无操作系统真实线程；

- **RUNNABLE**：包含就绪、运行两种状态，线程已启动，等待/正在占用CPU执行；

- **BLOCKED**：线程竞争 **synchronized 内置锁** 失败，处于阻塞等待锁状态；

- **WAITING**：无限等待，调用无参 `wait()`、`join()`，需其他线程主动唤醒；

- **TIMED_WAITING**：限时等待，调用 `sleep()`、带参 wait/join，超时自动唤醒；

- **TERMINATED**：线程执行完毕或异常终止，生命周期结束。

**核心流转链路（面试必背）**：

1. NEW → RUNNABLE：调用 `start()` 启动线程；

2. RUNNABLE → BLOCKED：竞争 synchronized 锁失败；

3. BLOCKED → RUNNABLE：获取到 synchronized 锁；

4. RUNNABLE → WAITING：执行 `wait()` / `join()`；

5. RUNNABLE → TIMED_WAITING：执行 `sleep()` / 限时等待方法；

6. WAITING/TIMED_WAITING → RUNNABLE：被唤醒或超时结束；

7. 所有状态最终可流转为 TERMINATED：线程执行结束/异常终止。

**高频易错点**：

- Java 没有独立的“运行状态”，就绪+运行统一合并为 **RUNNABLE**；

- **BLOCKED 仅针对 synchronized 锁**，Lock 锁阻塞不会进入该状态；

- 线程唤醒后不会直接运行，必须转为 RUNNABLE 重新竞争CPU与锁。

## 6.3 创建线程的方式有几种？各自的优缺点是什么？

**面试标准满分回答**：

从实际开发使用形态，Java线程创建分为**四种常用方式**；从底层原理本质，仅分为「继承Thread类」和「实现Runnable/Callable接口」两种。四种方式优缺点与适用场景如下：

**1. 继承 Thread 类**

优点：实现简单、上手直观；缺点：Java单继承限制，扩展性差，任务与线程耦合严重，无法复用任务逻辑，频繁创建开销大。场景：仅用于测试Demo，生产环境禁止使用。

**2. 实现 Runnable 接口**

优点：解耦线程与任务，规避单继承限制，代码简洁可复用；缺点：无返回值、无法抛出受检异常，需要手动捕获异常。场景：无需返回值的简单异步任务。

**3. 实现 Callable 接口 + FutureTask**

优点：支持泛型返回值、可抛出受检异常，能获取线程执行结果；缺点：使用相对繁琐，`get()` 方法存在阻塞风险。场景：需要获取异步任务执行结果的业务场景。

**4. 线程池 ExecutorService**

优点：实现线程复用，减少线程创建销毁开销，可控并发数量、防止OOM与CPU打满，统一管理线程；缺点：需要手动配置参数，使用不当易出现任务堆积、拒绝策略异常。场景：**生产环境所有并发场景唯一推荐方式**。

**面试总结话术**：生产环境绝对摒弃前三种手动创建方式，统一使用**ThreadPoolExecutor手动创建线程池**，禁止使用Executors工具类。

## 6.4 `start()`和`run()`方法的区别？为什么不能直接调用`run()`？

**面试标准满分回答**：

1. **核心职责不同**

`start()`：是线程启动的核心方法，由JVM底层调用，会向操作系统申请创建真实线程，分配栈内存资源，将线程纳入调度队列，真正开启异步执行。

`run()`：只是一个普通的成员方法，仅用于封装线程业务逻辑，无任何线程创建、调度逻辑。

2. **执行线程不同**

调用 `start()`：开启新子线程，异步执行任务；直接调用 `run()`：在**当前主线程**同步执行，不会开启新线程。

3. **执行次数限制不同**

`start()`只能调用一次，重复调用会抛出 `IllegalThreadStateException`；`run()` 可反复调用，无次数限制。

**核心结论**：直接调用 run() 只是普通方法调用，**失去了多线程异步并发的意义**，无法实现多线程效果。

## 6.5 守护线程和用户线程的区别？使用场景？

**面试标准满分回答**：

Java线程分为用户线程（普通线程）和守护线程（后台线程），核心区别在于**对JVM退出的影响**：

**1. 核心特性区别**

用户线程是核心业务线程，只要存在存活的用户线程，JVM就不会退出；守护线程是后台辅助线程，依附用户线程运行，当所有用户线程执行完毕，JVM直接退出，守护线程会被**强制无感知终止**。

**2. 创建方式**

所有线程默认是用户线程；守护线程需要在 start() 前手动设置 `setDaemon(true)`。

**3. 执行保障**

用户线程任务必须执行完毕，有完整的异常捕获与收尾逻辑；守护线程随时可能终止，无法保证任务执行完成，finally 代码块不一定执行。

**使用场景**：

**用户线程**：所有核心业务场景，如接口处理、数据计算、订单业务、文件解析等必须完成的任务。

**守护线程**：后台辅助、可中断、无状态任务，如JVM的GC线程、心跳检测、日志定时打印、缓存清理、状态监控。

**生产避坑**：守护线程绝对不能处理核心业务，会出现任务丢失、数据不完整问题。

## 6.6 线程优先级是否能保证线程执行顺序？为什么？

**面试标准满分回答**：

**不能保证绝对执行顺序**。

Java线程优先级取值为1~10，默认优先级为5，优先级的本质是**给操作系统的调度建议**，而非强制规则。高优先级线程只是获取CPU时间片的**概率更高**，并非一定优先执行。

核心原因三点：

1. JVM线程调度是**抢占式概率调度**，优先级仅为参考，不强制执行；

2. 部分操作系统不支持10级优先级细分，存在优先级映射失效问题，跨平台兼容性差；

3. CPU空闲、线程竞争较少时，低优先级线程依然可以抢占CPU优先执行。

**生产结论**：业务代码中**禁止使用线程优先级控制执行时序**，如需控制顺序，使用锁、等待唤醒、线程池队列、CountDownLatch 等工具。

## 6.7 上下文切换是什么？如何减少上下文切换？

**面试标准满分回答**：

**上下文切换定义**：CPU在多个线程间切换执行时，需要保存当前线程的寄存器、程序计数器、栈状态等上下文数据，加载下一个线程的上下文数据，这个**保存现场+恢复现场**的过程就是线程上下文切换。

上下文切换是**纯性能损耗操作**，不执行业务逻辑，频繁切换会导致CPU虚高、吞吐量下降、接口RT升高。

**常见触发场景**：CPU时间片耗尽、线程锁阻塞、IO阻塞、线程唤醒、大量线程并发竞争。

**减少上下文切换的落地方案**：

1. **合理配置线程池线程数**：CPU密集型任务线程数≈CPU核心数，避免线程过多引发竞争切换；IO密集型合理扩容，平衡阻塞与切换开销。

2. **优化锁竞争**：缩小锁粒度、避免大锁、减少循环内加锁，降低线程阻塞唤醒频率。

3. **使用无锁并发技术**：优先使用CAS、ThreadLocal、不可变对象，减少重量级锁阻塞。

4. **任务批量处理**：合并零散小任务，减少线程频繁调度次数。

5. **减少无效阻塞**：避免频繁sleep、短时IO、无意义等待，减少主动切换。
