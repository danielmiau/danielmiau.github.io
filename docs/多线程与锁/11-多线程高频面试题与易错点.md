# 1. 线程安全、锁机制、AQS、线程池高频面试题

## 1.1 线程安全相关高频题

### 1.1.1 什么是线程安全？如何判断一个类是否线程安全？

**1. 线程安全定义**

多线程环境下，多个线程同时访问、修改共享资源，在**无需额外加锁、无需同步控制**的前提下，程序运行结果与单线程串行执行结果完全一致，不会出现数据脏读、数据覆盖、数据错乱等问题，即为**线程安全**。

简单理解：**多线程并发操作共享数据，结果依旧正确**。

**2. 如何判断类是否线程安全**

- **无共享变量**：类中无成员变量，仅存在局部变量，所有数据线程私有，绝对线程安全；

- **共享变量只读**：共享变量被final修饰，仅初始化赋值、后续不可修改，无写操作，线程安全；

- **共享变量读写受控**：存在读写操作，但通过synchronized、Lock、CAS等机制保证原子性、可见性、有序性，线程安全；

- **存在并发写且无同步控制**：多线程同时修改共享变量，无任何同步手段，线程不安全。

**常见线程安全类**：String、Integer、Long、StringBuffer、ConcurrentHashMap

**常见线程不安全类**：ArrayList、HashMap、StringBuilder、SimpleDateFormat

### 1.1.2 线程安全问题的三大根源：可见性、原子性、有序性分别如何解决？

Java并发线程安全问题，本质由CPU缓存、指令重排、线程切换三大硬件机制导致，核心包含三大特性缺失，对应解决方案明确：

**1. 原子性：操作不可分割，要么全部执行成功，要么全部不执行**

问题根源：线程切换导致复合操作（i++、先查后改）被拆分，出现数据覆盖。

解决方案：**synchronized、ReentrantLock、CAS**，保证复合操作原子执行。

**2. 可见性：一个线程修改共享变量，其他线程能立即感知最新值**

问题根源：CPU多级缓存，每个线程缓存变量副本，变量修改仅更新本地缓存，未及时刷回主内存。

解决方案：**volatile（轻量）、synchronized、Lock**，强制刷新主内存、失效缓存。

**3. 有序性：程序执行顺序与代码编写顺序一致**

问题根源：JVM和CPU为优化性能，对无依赖指令进行**指令重排序**，多线程下破坏业务逻辑。

解决方案：**volatile（禁止重排）、synchronized、Lock（保证串行执行）**。

**总结**：volatile 只能解决可见性、有序性，**无法保证原子性**；synchronized 和 Lock 可同时解决三大特性问题。

### 1.1.3 竞态条件与数据竞争的区别是什么？

二者是并发高频易混淆概念，是线程不安全的两大核心场景，区别如下：

**1. 数据竞争（Data Race）**

定义：多个线程**同时读写同一个共享变量**，且至少有一个是写操作，同时无同步控制。

核心特征：针对**单一变量**的并发读写冲突，是语法层面的并发问题。

**2. 竞态条件（Race Condition）**

定义：多线程执行时序不可控，导致**业务逻辑执行结果出错**，不一定是单一变量读写冲突。

核心特征：针对**业务逻辑**，由时序、步骤错乱导致，常见于先查后改、判断执行的复合逻辑。

**核心区别**

- 数据竞争一定导致线程不安全；

- 存在竞态条件不一定有数据竞争，但一定会业务结果错误；

- 数据竞争是**底层原因**，竞态条件是**上层结果**。

### 1.1.4 ArrayList/HashMap为什么是线程不安全的？并发场景下如何解决？

**1. ArrayList 线程不安全原因**

- **扩容并发异常**：多线程同时触发扩容，会导致数组下标越界、数组覆盖；

- **尾插数据覆盖**：add方法无同步锁，多线程同时尾插，会出现数据覆盖、元素丢失；

- **迭代并发修改异常**：遍历同时修改集合，触发ConcurrentModificationException。

**2. HashMap 线程不安全原因（JDK1.8）**

- **数据覆盖丢失**：put方法无锁，多线程同时put相同hash位置数据，发生覆盖；

- **扩容数据错乱**：多线程同时扩容，导致链表、红黑树结构错乱，数据丢失；

- **死循环风险**：JDK1.7存在链表头尾倒置死循环，1.8已修复，但仍有数据丢失问题。

**3. 并发场景解决方案**

- 简单场景：使用 **Collections.synchronizedList / synchronizedMap**（全局锁，性能差）；

- 生产推荐：使用 **CopyOnWriteArrayList、ConcurrentHashMap**（分段锁、写时复制，高性能线程安全）。

## 1.2 锁机制相关高频题

### 1.2.1 `synchronized`的底层实现原理是什么？锁升级过程是怎样的？

**1. 底层实现原理**

synchronized 是 **JVM内置锁**，底层基于 **对象头Mark Word** 实现锁机制，无需手动编码解锁。

同步代码块基于 **monitorenter、monitorexit** 字节码指令实现；同步方法通过方法标识位判断锁状态。

每个Java对象都自带一个Monitor监视器，同一时刻仅一个线程能获取Monitor锁。

**2. 锁升级全过程（JDK1.6之后，无锁→偏向锁→轻量级锁→重量级锁）**

- **无锁**：对象初始状态，无线程竞争，Mark Word无锁标记；

- **偏向锁（单线程优化）**：只有一个线程反复获取锁，无竞争，锁直接偏向该线程，无CAS开销，性能极高；

- **轻量级锁（轻微竞争）**：多线程交替竞争锁，无同时抢占，基于CAS自旋抢锁，无阻塞、无内核态切换；

- **重量级锁（激烈竞争）**：多线程同时抢占锁，CAS自旋失败，线程进入阻塞队列，依赖操作系统内核阻塞，性能开销大。

**核心特性**：锁只能单向升级，**无法降级**，目的是最大化提升并发性能。

### 1.2.2 偏向锁、轻量级锁、重量级锁的适用场景与优缺点对比？

|锁类型|适用场景|优点|缺点|
|---|---|---|---|
|偏向锁|单线程重复获取锁、无竞争场景|无CAS开销、无阻塞、性能最优|出现多线程竞争会立即撤销，有轻微撤销开销|
|轻量级锁|多线程交替抢锁、竞争不激烈|用户态CAS自旋，无内核阻塞，响应快|自旋消耗CPU，竞争激烈时CPU空转严重|
|重量级锁|多线程同时抢占、高并发激烈竞争|线程阻塞休眠，不消耗CPU资源|内核态阻塞、上下文切换频繁，性能最差|
### 1.2.3 `synchronized`修饰实例方法、静态方法、代码块的锁对象分别是什么？

- **修饰实例方法**：锁对象为 **当前实例对象 this**，不同实例互不影响，多实例可并行执行；

- **修饰静态方法**：锁对象为 **当前类的Class对象**，属于全局锁，所有实例共用一把锁，并发互斥；

- **修饰代码块**：锁对象为 **手动指定的任意对象**（this、自定义对象、Class对象），粒度最细、灵活性最高。

**面试易错点**：静态方法锁Class对象，实例方法锁this对象，**两把锁互不冲突**，可以并发执行。

### 1.2.4 可重入锁的定义是什么？`synchronized`和`ReentrantLock`都是可重入锁吗？

**1. 可重入锁定义**

同一个线程，**在外层方法已经获取锁的前提下**，再次尝试获取同一把锁，不会被阻塞、不会死锁，可以直接获取成功，这就是可重入特性。核心作用：避免同一线程递归调用锁方法导致自我死锁。

**2. 两者判定**

- **synchronized**：是可重入锁，JVM底层自动维护锁计数器，获取锁+1，释放锁-1，计数器为0才真正释放；

- **ReentrantLock**：顾名思义（Reentrant），本身就是专门的可重入锁，手动维护重入次数。

**结论**：两者**都是可重入锁**，均支持递归重入，避免自我死锁。

## 1.3 AQS相关高频题

### 1.3.1 AQS的核心组件与实现原理是什么？

**AQS（AbstractQueuedSynchronizer）**：Java并发包的**核心底层基石**，所有Lock、锁工具类的底层实现依赖，是一个模板抽象类。

**三大核心组件**

1. **state 同步状态变量（核心）**：int类型变量，volatile修饰，用于标记锁状态。0代表无锁、大于0代表持有锁，可重入锁每重入一次+1；

2. **CLH双向阻塞队列**：存储抢锁失败的阻塞线程，线程抢锁失败后封装为Node节点，进入队列排队；

3. **线程等待唤醒机制**：基于LockSupport的park/unpark实现线程阻塞与唤醒，替代老旧的wait/notify。

**核心原理**

通过CAS操作修改state变量实现抢锁，抢锁成功执行业务逻辑；抢锁失败进入CLH队列阻塞，等待持有锁的线程释放锁后，唤醒队列头部线程继续抢锁，全程模板方法统一管控。

### 1.3.2 AQS的独占模式与共享模式分别如何实现？

AQS定义两种资源抢占模式，适配不同锁场景：

**1. 独占模式（Exclusive）**

同一时刻**仅允许一个线程获取锁**，线程独占资源。

核心方法：tryAcquire、tryRelease

适用组件：ReentrantLock、synchronized

**2. 共享模式（Shared）**

同一时刻**允许多个线程同时获取锁**，共享资源，读多写少场景专用。

核心方法：tryAcquireShared、tryReleaseShared

适用组件：ReentrantReadWriteLock、CountDownLatch、Semaphore

**核心区别**：独占模式释放锁仅唤醒一个后继线程；共享模式释放锁会**传播唤醒**，批量唤醒多个排队线程。

### 1.3.3 `ReentrantLock`和`ReentrantReadWriteLock`如何基于AQS实现？

**1. ReentrantLock 实现原理**

基于AQS**独占模式**实现，内部定义两个内部类：公平同步器、非公平同步器。

- state=0：无锁状态；

- state>0：已有线程持有锁，当前线程是持有者则重入+1，否则阻塞排队；

- 支持可重入、可中断、超时抢锁、公平/非公平切换。

**2. ReentrantReadWriteLock 实现原理**

基于AQS**共享+独占混合模式**实现，将32位int类型的state变量拆分为高低16位：

- 高16位：读锁计数（共享模式，多线程可同时获取）；

- 低16位：写锁计数（独占模式，仅单线程可获取）。

核心特性：**读读共享、读写互斥、写写互斥**，完美适配读多写少场景。

### 1.3.4 `Condition`条件队列的实现原理是什么？与`wait()/notify()`有什么区别？

**1. Condition 原理**

Condition 是 Lock 锁的条件等待队列，基于AQS实现，每个Condition拥有**独立的单向等待队列**。线程调用await()释放锁、进入条件队列等待；调用signal()唤醒指定队列线程，重新进入主CLH队列抢锁。

**2. 与 wait/notify 的核心区别（面试重点）**

- **所属对象不同**：wait/notify是Object原生方法，绑定对象监视器；Condition是Lock配套的条件队列，Java层实现；

- **队列数量不同**：一个锁仅对应一个wait队列；**一个Lock可对应多个Condition队列**，精准唤醒；

- **唤醒精度不同**：notify()随机唤醒一个、notifyAll()唤醒全部，无法精准控制；Condition可唤醒指定条件队列线程，无无效唤醒；

- **功能拓展**：Condition支持超时等待、可中断等待，功能更强大。

## 1.4 线程池相关高频题

### 1.4.1 线程池的核心参数有哪些？各参数的作用是什么？

ThreadPoolExecutor 七大核心参数，面试必背：

1. **corePoolSize 核心线程数**：线程池常驻线程数量，空闲不回收，承担常态并发任务；

2. **maximumPoolSize 最大线程数**：线程池最大资源上限，控制最高并发量；

3. **keepAliveTime 空闲存活时间**：非核心线程空闲超时时间，超时自动回收，释放资源；

4. **TimeUnit 时间单位**：空闲时间单位；

5. **BlockingQueue 任务队列**：存储等待执行的任务，用于流量削峰；

6. **ThreadFactory 线程工厂**：统一创建线程，自定义线程名、优先级、守护线程；

7. **RejectedExecutionHandler 拒绝策略**：线程池满载后，新任务的兜底处理策略。

### 1.4.2 线程池的工作流程是怎样的？任务提交后如何流转？

任务提交execute()后，严格按照以下顺序流转，**不可逆、不跳过**：

1. 判断当前工作线程数是否小于**核心线程数**，是则直接新建核心线程执行任务；

2. 核心线程已满，判断**任务队列是否未满**，是则任务入队排队；

3. 队列已满，判断当前线程数是否小于**最大线程数**，是则新建非核心线程执行任务；

4. 最大线程数打满、队列已满，触发**拒绝策略**，执行兜底逻辑。

**核心口诀**：先核心、再队列、后扩容、最后拒绝

### 1.4.3 线程池的四种拒绝策略分别是什么？适用场景？

- **AbortPolicy（默认抛异常）**：直接抛出RejectedExecutionException，适合非核心任务、需要快速发现过载问题的场景；

- **CallerRunsPolicy（调用者执行）**：由提交任务的主线程执行任务，不丢数据、不抛异常，适合核心业务、不允许任务丢失场景；

- **DiscardPolicy（静默丢弃）**：直接丢弃任务，无日志无异常，适合日志、埋点等可丢失非核心任务；

- **DiscardOldestPolicy（丢弃最旧任务）**：丢弃队列头部最久未执行任务，插入新任务，适合实时性高、旧数据无效的场景。

**生产结论**：默认策略不推荐，核心业务自定义拒绝策略+MQ兜底。

### 1.4.4 `Executors`创建线程池有什么风险？为什么不推荐使用？

阿里开发规范强制禁止使用Executors，核心风险如下：

- **newFixedThreadPool / newSingleThreadExecutor**：使用**无界LinkedBlockingQueue**，任务无限堆积，导致OOM内存溢出；

- **newCachedThreadPool**：最大线程数为Integer.MAX_VALUE，可无限创建线程，导致线程泛滥、耗尽系统资源、OOM；

- **newScheduledThreadPool**：同样存在无限线程风险，资源不可控。

**核心原因**：Executors封装的线程池**资源无上限、无过载保护**，生产完全不可控。生产必须手动自定义ThreadPoolExecutor，使用有界队列、限制最大线程数。

### 1.4.5 线程池的五种状态分别是什么？状态流转过程？

线程池内部通过ctl变量维护五种运行状态，优先级从高到低：

1. **RUNNING（运行中）**：正常接收新任务、处理排队任务；

2. **SHUTDOWN（关闭）**：不接收新任务，继续处理已有队列任务；

3. **STOP（停止）**：不接收新任务、不处理队列任务，中断正在执行的任务；

4. **TIDYING（整理）**：所有任务终止、线程数为0，准备收尾；

5. **TERMINATED（终止）**：线程池彻底关闭，资源释放完成。

**核心流转**：RUNNING → SHUTDOWN/STOP → TIDYING → TERMINATED

# 2. `synchronized` 与 `Lock` 的区别

## 2.1 核心特性对比

### 2.1.1 实现方式：JVM内置关键字 vs Java层接口实现

**synchronized**：是Java**原生关键字**，由JVM底层C++代码实现，属于虚拟机层面的锁，无需依赖额外类库，底层依靠对象头、Monitor监视器实现锁机制。

**Lock（ReentrantLock）**：是**Java语言层面的接口**，属于JDK并发包工具，基于AQS纯Java代码实现锁逻辑，完全由代码控制锁的获取与释放，不受JVM底层硬编码限制。

### 2.1.2 锁的获取与释放：自动vs手动

**synchronized**：**自动加锁、自动释放**。进入同步代码块自动获取锁，代码执行完毕、异常抛出时JVM自动释放锁，无需手动控制，不会出现锁泄漏。

**Lock**：**手动加锁、手动释放**。必须主动调用lock()加锁，执行完毕必须在finally块中主动调用unlock()释放锁，编码不当容易出现**锁未释放、线程死锁**问题。

### 2.1.3 可中断性：不可中断vs支持可中断获取锁

**synchronized**：**不可中断**。线程阻塞等待锁时，无法被中断，会一直阻塞直到获取锁或线程死亡，高并发下容易出现大量线程堆积阻塞。

**Lock**：**支持可中断抢锁**。调用 lockInterruptibly() 方法，线程等待锁的过程中可响应中断，主动退出抢锁阻塞状态，避免死锁堆积。

### 2.1.4 超时获取：不支持vs支持`tryLock(long timeout, TimeUnit unit)`

**synchronized**：无超时机制，抢锁失败永久阻塞，无法主动退出。

**Lock**：支持超时抢锁，通过 tryLock(timeout) 指定最大等待时间，超时未获取锁直接返回false，放弃抢锁，完美解决永久阻塞问题，适配超时业务场景。

### 2.1.5 公平锁支持：仅非公平vs支持公平/非公平两种模式

**synchronized**：**只支持非公平锁**，线程随机抢占锁，不遵循先来先服务，吞吐量高，但可能出现线程饥饿。

**Lock**：**同时支持公平锁、非公平锁**。构造方法传true为公平锁（先来先服务，无线程饥饿），传false为非公平锁（默认，吞吐量更高）。

### 2.1.6 条件队列支持：单个vs支持多个`Condition`条件队列

**synchronized**：仅支持**唯一的等待队列**，依靠wait/notify实现等待唤醒，只能批量唤醒或随机唤醒，精准度差。

**Lock**：支持**多个Condition条件队列**，可根据业务场景拆分不同等待队列，精准唤醒指定线程，避免无效唤醒，并发效率更高。

## 2.2 性能对比

### 2.2.1 低竞争场景：`synchronized`锁升级后的性能优势

在低竞争、单线程重复抢锁、轻微交替抢锁场景下，synchronized 经过**偏向锁、轻量级锁升级**后，无CAS自旋开销、无内核态切换，JVM底层极致优化，代码简洁、开销极低，性能**优于Lock**。

### 2.2.2 高竞争场景：`Lock`的CAS自旋与`AQS`队列管理性能优势

高并发激烈竞争场景下，synchronized 直接升级为重量级锁，线程进入内核态阻塞，上下文切换频繁，性能急剧下降。

Lock 基于AQS的CAS自旋+队列排队机制，用户态自旋抢锁，减少内核阻塞切换开销，队列调度更精细，**高竞争下性能远优于synchronized**。

### 2.2.3 不同JDK版本下两者的性能变化

- **JDK1.6之前**：synchronized 是重量级锁，性能极差，Lock全面碾压synchronized；

- **JDK1.6及之后**：JVM引入锁升级、偏向锁、轻量级锁优化，synchronized 性能大幅提升，低竞争下性能持平甚至超越Lock；

- **当前版本**：无绝对性能差距，性能取舍主要看**竞争激烈程度**，而非锁本身。

## 2.3 适用场景与选型建议

### 2.3.1 优先使用`synchronized`的场景：简单同步、低竞争场景

业务逻辑简单、代码量少、并发竞争不激烈、无需特殊锁特性（超时、可中断、公平锁、多条件队列），优先使用 synchronized。

优势：代码简洁、自动释放、无锁泄漏风险、低竞争性能最优、JVM持续优化。

### 2.3.2 优先使用`Lock`的场景：高竞争、需灵活控制锁的场景

满足以下任意场景，优先使用 ReentrantLock：

- 高并发、多线程激烈竞争场景；

- 需要**超时抢锁、可中断抢锁**；

- 需要**公平锁**机制，解决线程饥饿；

- 需要多条件队列、精准线程唤醒；

- 需要读写分离锁（ReentrantReadWriteLock）。

### 2.3.3 生产环境锁选型最佳实践

1. **简单同步优先synchronized**：减少编码复杂度，避免锁泄漏，稳定性更高；

2. **复杂并发优先Lock**：高竞争、灵活控锁、读写分离场景统一使用Lock体系；

3. **禁止滥用锁**：缩小锁粒度，只锁住共享变量操作逻辑，避免大代码块加锁；

4. **Lock必须finally解锁**：生产强制规范，杜绝锁泄漏导致的死锁问题。

# 3. `volatile` 的作用与使用场景

## 3.1 `volatile`的两大核心作用

### 3.1.1 保证可见性：强制读写主内存，禁用本地缓存

在JVM内存模型中，每个线程拥有独立的**工作内存（本地缓存）**，共享变量会从主内存拷贝到线程本地缓存，线程读写变量默认操作本地缓存，不会实时同步主内存。这就会导致一个线程修改变量后，其他线程无法及时感知最新值，引发**可见性问题**。

**volatile保证可见性的核心原理**：

- 被volatile修饰的变量，**写操作强制刷新回主内存**，修改完成立即同步主内存；

- 被volatile修饰的变量，**读操作强制从主内存读取**，清空本地缓存，直接加载主内存最新数据；

- 彻底杜绝线程本地缓存导致的数据不一致问题，实现多线程变量实时可见。

**代码演示（可见性问题复现与volatile解决）**

```java
/**
 * volatile 可见性演示
 * 不加volatile：子线程死循环，无法感知flag变化
 * 加volatile：子线程立即感知主内存最新值，结束循环
 */
public class VolatileVisibleDemo {
    // 不加volatile，存在可见性问题
    private static volatile boolean flag = true;

    public static void main(String[] args) throws InterruptedException {
        // 子线程循环读取flag
        new Thread(() -> {
            while (flag) {
                // 空循环，无IO操作，JVM会优化缓存flag，不会重读主内存
            }
            System.out.println("子线程感知flag变更，循环结束");
        }).start();

        // 主线程修改flag
        Thread.sleep(1000);
        flag = false;
        System.out.println("主线程修改flag为false");
    }
}

```

**结论**：不加volatile时，子线程会一直读取本地缓存的true，程序死循环；添加volatile后，强制读写主内存，变量实时可见，程序正常结束。

### 3.1.2 禁止指令重排：通过内存屏障实现有序性

为提升执行效率，JVM编译器和CPU会对**无依赖关系的指令**进行**指令重排序**，单线程下重排序不影响执行结果，但多线程环境下会导致业务逻辑错乱、执行结果异常。

**volatile禁止重排核心原理**：

volatile通过插入**内存屏障（Memory Barrier）**，锁住指令执行顺序，禁止上下指令乱序执行：

- 写屏障：volatile写之前的指令，禁止重排到写之后；

- 读屏障：volatile读之后的指令，禁止重排到读之前；

- 彻底保证volatile变量的读写顺序与代码编写顺序一致，解决多线程有序性问题。

### 3.1.3 `volatile`不保证原子性的原因与解决方案

**核心结论**：volatile **只能保证可见性、有序性，完全不保证原子性**。

**1. 不保证原子性的根本原因**

原子性要求操作**不可分割、要么全部成功、要么全部失败**。而日常的 `i++`、自增、复合赋值操作，本质是**三步复合操作**：

1. 从主内存读取变量值；

2. 线程本地计算修改；

3. 重新写入主内存。

volatile仅能保证每一步读写主内存实时生效，但**无法阻止多线程交替执行三步操作**，线程切换会导致数据覆盖、结果错乱，因此无法保证原子性。

**2. 代码复现原子性问题**

```java
/**
 * volatile 不保证原子性演示
 * 20个线程各累加1000次，最终结果小于20000
 */
public class VolatileAtomicDemo {
    private static volatile int num = 0;

    public static void add() {
        num++;
    }

    public static void main(String[] args) throws InterruptedException {
        Thread[] threads = new Thread[20];
        for (int i = 0; i < 20; i++) {
            threads[i] = new Thread(() -> {
                for (int j = 0; j < 1000; j++) {
                    add();
                }
            });
            threads[i].start();
        }
        // 等待所有线程执行完毕
        for (Thread thread : threads) {
            thread.join();
        }
        // 最终结果一定小于20000
        System.out.println("最终累加结果：" + num);
    }
}

```

**3. 原子性问题生产解决方案**

- 简单场景：使用 **synchronized** 加锁，保证复合操作原子执行；

- 高性能场景：使用 **JUC原子类（AtomicInteger、AtomicLong）**，基于CAS实现无锁原子操作；

- 复杂复合操作：使用 **ReentrantLock** 精准控制锁粒度。

## 3.2 `volatile`的使用场景

volatile轻量化、无锁开销、性能极高，核心适用场景为：**无需保证原子性，仅需要保证可见性、有序性**的场景。

### 3.2.1 状态标记场景：线程间状态通知

这是volatile最常用的生产场景，定义全局状态变量，一个线程修改状态，其他线程实时感知、终止任务，无复合操作、无需原子性。

常见场景：服务启停标记、任务暂停/恢复标记、开关控制。

```java
// 全局任务停止标记
private static volatile boolean STOP_TASK = false;

```

### 3.2.2 DCL双重检查锁定单例模式：防止指令重排

DCL（双重检查锁）单例是volatile**禁止指令重排**的经典落地场景。

**问题根源**：单例对象初始化分为三步：分配内存→初始化对象→赋值引用。指令重排可能导致顺序变为：分配内存→赋值引用→初始化对象，多线程下会出现**拿到未初始化完成的空对象**，引发空指针异常。

**volatile解决方案**：禁止对象初始化指令重排，保证初始化顺序正常。

```java
/**
 * 标准DCL双重检查锁单例模式（生产可用）
 * volatile 禁止指令重排，解决线程不安全问题
 */
public class Singleton {
    // volatile 禁止实例化指令重排
    private static volatile Singleton instance;

    // 私有构造
    private Singleton() {}

    public static Singleton getInstance() {
        // 第一次校验：无锁快速判断，提升性能
        if (instance == null) {
            synchronized (Singleton.class) {
                // 第二次校验：防止多线程穿透第一次判断
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}

```

### 3.2.3 禁止指令重排场景：避免并发逻辑错乱

多线程下存在依赖顺序的初始化逻辑、配置加载逻辑、资源创建逻辑，使用volatile修饰核心变量，禁止指令重排，避免因代码执行顺序错乱导致的业务异常、资源初始化失败等问题。

### 3.2.4 单写多读场景：写线程修改、多线程读取

严格满足 **单线程写、多线程读** 的场景，无并发写竞争，无需保证原子性，仅需要保证读线程实时感知最新数据，volatile是最优方案，性能远超锁机制。

常见场景：全局配置参数、动态开关、系统常量更新。

## 3.3 高频面试延伸问题

### 3.3.1 `volatile`如何通过内存屏障禁止指令重排？

JVM基于Java内存模型规范，为volatile变量的读写操作**强制插入四类内存屏障**，从语法层面禁止指令重排：

1. **LoadLoad屏障**：volatile读前插入，禁止前面读、后面读重排；

2. **LoadStore屏障**：volatile读后插入，禁止前面读、后面写重排；

3. **StoreStore屏障**：volatile写前插入，禁止前面写、后面写重排；

4. **StoreLoad屏障**：volatile写后插入，禁止前面写、后面读重排（最重的屏障）。

通过内存屏障锁定指令执行顺序，彻底杜绝volatile变量相关的指令重排序问题，保证多线程有序性。

### 3.3.2 `volatile`修饰的变量在多线程下是否线程安全？

**分场景判定，不能一概而论**：

- **线程安全场景**：仅读操作、单写多读、状态标记赋值（简单赋值，无复合运算），仅需可见性、有序性，volatile可保证安全；

- **线程不安全场景**：存在并发写、复合操作（i++、先查后改、累加计算），无法保证原子性，线程不安全。

**面试标准答案**：volatile不保证原子性，仅保证可见性和有序性，因此**无法完全保证多线程安全**，仅适用于无并发写、无复合操作的简单场景。

### 3.3.3 `volatile`和`synchronized`的区别是什么？

| 对比维度 | volatile                         | synchronized                     |
| -------- | -------------------------------- | -------------------------------- |
| 保障特性 | 可见性、有序性，**不保证原子性** | 原子性、可见性、有序性全部保证   |
| 实现层级 | JVM内存屏障、内存读写规则        | JVM内置锁、Monitor监视器         |
| 锁粒度   | 无锁、轻量级机制                 | 重量级锁、排他锁                 |
| 性能开销 | 极低，无阻塞、无上下文切换       | 较高，竞争激烈会阻塞线程         |
| 阻塞特性 | 不阻塞线程                       | 竞争失败线程会阻塞等待           |
| 适用场景 | 单写多读、状态标记、禁止重排     | 并发写、复合操作、需要原子性场景 |

# 4. CAS的原理与问题

## 4.1 CAS的工作原理

### 4.1.1 CAS的定义与操作流程：比较并交换

**CAS（Compare And Swap，比较并交换）**，是一种**无锁乐观并发机制**，不通过加锁、阻塞的方式实现线程安全，全程用户态自旋，性能极高。

**CAS核心三参数**：内存地址V、预期旧值A、目标新值B

**完整执行流程**：

1. 线程读取内存地址V中的当前值，作为预期旧值A；

2. 计算得到需要更新的目标新值B；

3. 再次判断内存V的真实值是否等于预期旧值A：
            

   - 相等：说明期间无其他线程修改，**直接更新为新值B**，操作成功；

   - 不相等：说明数据已被修改，放弃更新，**重新读取、重试自旋**，直到成功。

**核心优势**：无锁、无线程阻塞、无内核切换，高并发下性能远超重量级锁。

### 4.1.2 CAS的底层实现：CPU指令级支持（`cmpxchg`）

CAS的原子性并非JVM保证，而是**CPU硬件指令级保证**，是真正的硬件原子操作，不会被线程调度打断。

**底层核心指令：cmpxchg**

- 多核CPU下，执行cmpxchg指令时，CPU会**锁定缓存行**，禁止其他CPU核心修改当前数据；

- 单核CPU下，直接保证指令不可分割；

- 从硬件层面实现“比较+交换”两步操作的原子性，无并发安全问题。

这也是CAS无需加锁就能保证原子性的根本原因。

### 4.1.3 CAS在Java中的应用：`Unsafe`类与`Atomic`原子类

Java层面无法直接调用CPU指令，通过 **sun.misc.Unsafe** 本地类调用native方法，底层映射CPU cmpxchg指令，实现CAS操作。

**1. Unsafe核心作用**

提供底层内存操作、CAS操作、线程挂起唤醒等能力，是JUC所有无锁工具类的底层基石，普通业务代码禁止直接使用。

**2. Atomic原子类（CAS落地应用）**

JUC封装的原子工具类，全部基于CAS实现无锁原子操作，生产高频使用：

- 基础类型：AtomicInteger、AtomicLong、AtomicBoolean；

- 引用类型：AtomicReference、AtomicStampedReference；

- 数组类型：AtomicIntegerArray、AtomicLongArray。

**代码示例：AtomicInteger CAS自增**

```java
import java.util.concurrent.atomic.AtomicInteger;

/**
 * CAS原子类解决自增原子性问题
 * 无锁实现线程安全，性能优于synchronized
 */
public class CasAtomicDemo {
    // 基于CAS的原子整型
    private static final AtomicInteger num = new AtomicInteger(0);

    public static void main(String[] args) throws InterruptedException {
        Thread[] threads = new Thread[20];
        for (int i = 0; i < 20; i++) {
            threads[i] = new Thread(() -> {
                for (int j = 0; j < 1000; j++) {
                    // CAS自旋自增，原子操作
                    num.getAndIncrement();
                }
            });
            threads[i].start();
        }
        for (Thread thread : threads) {
            thread.join();
        }
        // 结果永远为20000，线程安全
        System.out.println("CAS原子累加结果：" + num.get());
    }
}

```

## 4.2 CAS的三大问题

### 4.2.1 ABA问题：定义、危害与解决方案（版本号/`AtomicStampedReference`）

**1. ABA问题定义**

CAS仅对比数值是否一致，无法识别数据是否被修改过。线程A读取值为A，线程B将A修改为B，又改回A，线程A再次对比发现还是A，认为数据未被修改，直接更新，**忽略了中间的修改过程**。

**2. 核心危害**

基础数值场景无明显危害，但在**对象引用、栈结构、队列数据**场景下，会导致数据错乱、引用覆盖、业务异常。

**3. 生产解决方案**

新增**版本号/时间戳**机制，不仅对比数值，还对比版本号，数据每修改一次版本号+1，彻底杜绝ABA问题。

Java专属工具类：**AtomicStampedReference**（带版本号的引用原子类）

```java
import java.util.concurrent.atomic.AtomicStampedReference;

/**
 * AtomicStampedReference 解决CAS的ABA问题
 * 维护值+版本号双重校验
 */
public class ABADemo {
    public static void main(String[] args) {
        // 初始值100，初始版本号0
        AtomicStampedReference<Integer> ref = new AtomicStampedReference<>(100, 0);

        new Thread(() -> {
            // 获取当前版本号
            int stamp = ref.getStamp();
            try {
                Thread.sleep(1000);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            // 尝试更新：值100、版本不变，更新失败，规避ABA问题
            boolean success = ref.compareAndSet(100, 200, stamp, stamp + 1);
            System.out.println("线程A更新结果：" + success);
        }).start();

        new Thread(() -> {
            // 模拟ABA操作：100->200->100，版本号两次递增
            ref.compareAndSet(100, 200, ref.getStamp(), ref.getStamp() + 1);
            ref.compareAndSet(200, 100, ref.getStamp(), ref.getStamp() + 1);
            System.out.println("线程B完成ABA修改");
        }).start();
    }
}

```

### 4.2.2 自旋开销问题：高竞争场景下CPU资源消耗

**问题本质**：CAS是**循环自旋重试**机制，高并发激烈竞争场景下，线程会无限循环重试CAS操作，一直占用CPU资源做空转，导致**CPU飙升、性能急剧下降**。

**适用边界**：低竞争、冲突少的场景CAS性能远超锁；高竞争、冲突频繁场景，CAS自旋开销大于线程阻塞开销，得不偿失。

**解决方案**

- 限制自旋次数，超时失败后降级为锁；

- 高竞争场景放弃CAS，使用ReentrantLock重量级锁；

- 使用JDK8新增的Adder累加器，分段CAS降低竞争粒度。

### 4.2.3 只能保证单个变量原子性：复合操作的局限性

CAS仅支持**单个变量、单次比较交换**的原子操作，无法保证多个变量、复合业务操作的原子性。

例如：同时修改两个变量、先查询后修改、事务性复合操作，CAS无法保证整体原子性。

**解决方案**：复杂复合原子操作，必须使用 **Lock或synchronized锁机制**。

## 4.3 CAS与乐观锁的关系

### 4.3.1 CAS是乐观锁的一种实现方式

**乐观锁核心思想**：默认线程之间无冲突，不加锁，操作提交时校验数据是否被修改，无冲突则更新，冲突则重试/放弃。

**悲观锁核心思想**：默认线程一定有冲突，操作前先加锁，阻塞其他线程，保证安全。

**核心关系**：**CAS是Java并发领域乐观锁的经典实现**，完全遵循乐观锁无锁、冲突重试的核心逻辑。数据库版本号更新、时间戳更新，也属于乐观锁实现。

### 4.3.2 乐观锁与悲观锁的适用场景对比

| 锁类型        | 代表实现                                | 适用场景                       | 优缺点                          |
| ------------- | --------------------------------------- | ------------------------------ | ------------------------------- |
| 乐观锁（CAS） | Atomic原子类、数据库版本号              | 低并发、冲突少、读多写少场景   | 无阻塞高性能，高冲突CPU空转严重 |
| 悲观锁        | synchronized、ReentrantLock、数据库行锁 | 高并发、写多读少、冲突频繁场景 | 稳定无空转，存在线程阻塞开销    |

### 4.3.3 高并发场景下的性能差异

- **低并发无冲突**：CAS乐观锁性能碾压悲观锁，无阻塞、无内核切换；

- **中并发少量冲突**：CAS性能依旧优于锁机制，少量自旋开销可忽略；

- **高并发大量冲突**：CAS无限自旋导致CPU满载，悲观锁线程阻塞休眠、释放CPU资源，性能反超CAS。

**生产最佳实践**：读多写少用CAS乐观锁，写多冲突多用悲观锁。

# 5. 线程池的参数调优与拒绝策略

## 5.1 线程池参数调优

### 5.1.1 核心线程数设置：计算密集型vsIO密集型场景

**核心概念**：核心线程数（corePoolSize）是线程池常驻线程数量，线程池初始化后会维持该数量的线程存活，即使空闲也不会被回收，是处理常态任务的核心工作线程。核心线程数设置不合理会导致：线程过少、CPU打不满、吞吐量低；线程过多、线程上下文切换频繁、性能下降。

**1. 计算密集型任务**

任务特征：大量CPU运算、无阻塞、无IO等待、CPU持续高速运转。

**配置公式**：`核心线程数 = CPU核心数 + 1`

**原理说明**：计算密集型任务瓶颈在CPU，线程数等于CPU核心数可最大化利用CPU算力，多余1个线程用于兜底处理偶尔的线程停顿，避免CPU空闲。如果线程数超过CPU核心数，会引发大量线程上下文切换，严重降低运算效率。

**2. IO密集型任务**

任务特征：数据库查询、接口调用、文件读写、网络请求，线程大量时间处于阻塞等待状态，CPU空闲时间多。

**配置公式**：`核心线程数 = CPU核心数 * 2` 或 `CPU核心数 / (1 - 阻塞系数)`

**原理说明**：IO阻塞时线程释放CPU资源，可通过增加线程数压榨CPU空闲算力，提升整体吞吐量。阻塞系数一般取值0.8~0.9。

**生产调优补充**：线上禁止写死线程数，可通过 `Runtime.getRuntime().availableProcessors()` 获取当前机器CPU核心数，动态适配环境。

### 5.1.2 队列选择：有界队列vs无界队列的风险与选型

任务队列是线程池流量削峰的核心组件，队列选型是**线程池OOM、线上雪崩**的关键影响点，生产环境必须严格区分。

**1. 无界队列（LinkedBlockingQueue 无参构造）**

特性：队列容量无上限，可无限存放任务。

风险：任务堆积时队列无限扩容，占用大量堆内存，直接引发**OOM内存溢出**；且队列永远不会满，导致**最大线程数失效**，永远只会创建核心线程，高并发峰值无法扩容提升并发能力。

**2. 有界队列（ArrayBlockingQueue、带容量的LinkedBlockingQueue）**

特性：队列容量固定，任务量超过队列容量后，触发线程扩容，打满后触发拒绝策略。

优势：内存可控、任务堆积有上限、不会OOM；可正常触发线程扩容与拒绝策略，实现过载保护。

**生产选型结论**

- **线上强制使用有界队列**，杜绝无界队列OOM风险；

- 队列容量根据业务峰值预估设置，不宜过大（堆积过多任务超时失效）、不宜过小（频繁触发拒绝策略）；

- 常规业务推荐 ArrayBlockingQueue，有界、线程安全、性能稳定。

### 5.1.3 最大线程数设置：峰值并发量与系统资源平衡

最大线程数（maximumPoolSize）是线程池的并发上限，用于应对**瞬时流量峰值**，队列打满后，线程池会新建非核心线程直至达到最大线程数。

**设置原则**

1. 不能无限大：线程过多会占用大量栈内存、操作系统线程句柄，导致系统资源耗尽、服务卡顿、宕机；

2. 不能过小：无法应对流量峰值，频繁触发拒绝策略，丢任务、报错；

3. 核心逻辑：根据业务**峰值QPS、单任务耗时、机器配置**综合估算。

**简易估算公式**

`最大线程数 = 峰值QPS * 单任务最大耗时(秒) * 冗余系数(1.2~1.5)`

**生产最佳实践**：预留系统资源余量，避免线程占满CPU导致其他核心业务线程无法执行，非核心业务线程池最大线程数需严格限流。

### 5.1.4 `keepAliveTime`设置：非核心线程回收策略

**参数作用**：非核心线程的空闲存活时间，当非核心线程空闲时间超过该值，会被自动回收，释放系统线程资源。核心线程默认永久存活、不会回收。

**场景化配置**

- **流量平稳业务**：keepAliveTime 设置较短（10~30秒），快速回收闲置线程，节省资源；

- **流量波动大、突发峰值多业务**：设置较长（60秒以上），保留线程预热，避免频繁创建销毁线程损耗性能；

- **JDK1.8+优化**：可通过 `allowCoreThreadTimeOut(true)` 开启核心线程超时回收，极致节约闲置资源。

**避坑点**：时间设置过短会导致线程频繁创建销毁，增加系统开销；过长会导致大量闲置线程占用资源。

## 5.2 拒绝策略详解

### 5.2.1 四大内置拒绝策略：`AbortPolicy`/`CallerRunsPolicy`/`DiscardPolicy`/`DiscardOldestPolicy`

线程池拒绝策略触发条件：**核心线程已满 + 队列已满 + 最大线程数打满**，此时新任务触发拒绝策略，JDK内置四种实现：

**1. AbortPolicy（默认策略）**

直接抛出 **RejectedExecutionException** 运行时异常，终止任务提交。

**2. CallerRunsPolicy（调用者执行策略）**

不抛异常、不丢弃任务，将任务回退给**提交任务的主线程**执行，通过主线程限流保护线程池。

**3. DiscardPolicy（静默丢弃策略）**

直接丢弃当前新任务，无日志、无异常、无任何提示。

**4. DiscardOldestPolicy（丢弃最旧任务策略）**

丢弃队列头部**等待最久的未执行任务**，腾出空间放入当前新任务。

### 5.2.2 各拒绝策略的适用场景与优缺点

| 拒绝策略            | 优点                             | 缺点                                   | 适用场景                                       |
| ------------------- | -------------------------------- | -------------------------------------- | ---------------------------------------------- |
| AbortPolicy         | 快速失败、问题暴露及时、可控性强 | 直接抛异常，影响用户体验，需要异常处理 | 核心业务、不允许静默失败、需要监控告警的场景   |
| CallerRunsPolicy    | 不丢数据、无异常、天然限流       | 阻塞主线程、吞吐下降、响应变慢         | 核心业务、不允许任务丢失、对一致性要求高的场景 |
| DiscardPolicy       | 无异常、无阻塞、性能最高         | 静默丢数据，问题无法感知               | 非核心日志、埋点、统计类可丢弃任务             |
| DiscardOldestPolicy | 优先执行新任务，适配实时场景     | 丢失旧任务，业务可能错乱               | 实时性高、旧数据无效、最新数据优先的场景       |

### 5.2.3 自定义拒绝策略的实现与生产环境降级方案

**生产痛点**：JDK内置策略无法满足复杂业务需求，要么抛异常、要么丢数据，无日志、无兜底、无监控，线上问题难以排查。企业级开发必须**自定义拒绝策略**，实现日志打印、告警、降级、MQ兜底。

**自定义拒绝策略完整代码示例（可直接生产使用）**

```java
import java.util.concurrent.RejectedExecutionHandler;
import java.util.concurrent.ThreadPoolExecutor;

/**
 * 生产级自定义拒绝策略
 * 核心能力：打印日志、告警标记、降级兜底、不粗暴抛异常
 */
public class CustomRejectPolicy implements RejectedExecutionHandler {

    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        // 1. 打印详细报错日志，记录任务、线程池状态，方便排查问题
        System.err.println("线程池任务拒绝！触发过载保护，当前队列任务数：" + executor.getQueue().size());

        // 2. 自定义降级逻辑：可接入监控告警、推送钉钉/短信预警
        // 3. 高级兜底：将任务投递至延迟MQ，后续重试执行，避免数据丢失

        // 可选：根据业务场景选择兜底策略
        // 核心业务：存入MQ重试
        // 非核心业务：记录日志后丢弃
    }
}

```

**生产完整降级方案**

1. 任务拒绝后打印完整日志，记录时间、任务信息、线程池状态；

2. 接入监控系统，触发线程池过载告警；

3. 核心任务投递至RocketMQ/Kafka做重试兜底；

4. 非核心任务记录日志后优雅丢弃，保证主流程稳定。

## 5.3 高频面试延伸问题

### 5.3.1 线程池为什么推荐使用有界队列？

1. **杜绝OOM风险**：无界队列可无限堆积任务，高并发下内存持续增长，最终引发内存溢出，有界队列容量固定，内存可控；

2. **保证线程池完整能力**：无界队列永远不会满，导致最大线程数参数失效，无法扩容提升并发能力，有界队列可正常触发线程扩容；

3. **具备过载保护能力**：队列打满后触发拒绝策略，主动限流、降级，避免服务被流量打垮；

4. **符合生产稳定性规范**：阿里开发手册、大厂规范均强制线程池使用有界队列。

### 5.3.2 线程池满了之后，任务如何处理？

严格遵循线程池执行四大流程，顺序不可逆：

1. 优先创建核心线程执行任务；

2. 核心线程已满，任务进入**有界队列**排队；

3. 队列已满，创建非核心线程执行任务，直至达到最大线程数；

4. **线程数、队列全部打满**，触发拒绝策略，执行降级、报错、丢弃逻辑。

### 5.3.3 如何避免线程池OOM问题？

1. **强制使用有界队列**，禁止一切无界队列创建线程池；

2. **合理限制最大线程数**，避免无限创建线程耗尽系统资源；

3. **自定义拒绝策略兜底**，避免任务无限堆积；

4. **配置合理超时回收**，闲置线程及时释放资源；

5. **禁止使用Executors工具类**创建线程池，全部手动new ThreadPoolExecutor；

6. 线上增加线程池队列堆积监控、线程数监控，提前预警。

# 6. 死锁的产生条件与解决方法

## 6.1 死锁的定义与产生条件

### 6.1.1 死锁的定义：多个线程互相持有对方需要的资源，导致永久阻塞

**死锁**是多线程并发中的严重故障，指两个或多个线程互相持有对方需要的锁资源，同时都不释放自己已持有的锁，互相等待、僵持不下，导致所有线程**永久阻塞、无法执行、无法释放资源**，最终服务卡死、接口超时、线程池耗尽。

死锁属于**不可逆恶性BUG**，一旦发生不会自动解除，只能重启服务恢复，生产环境必须提前预防。

### 6.1.2 死锁产生的四大必要条件：互斥、持有并等待、不可剥夺、循环等待

死锁产生**必须同时满足四个条件**，缺一不可，破坏任意一个条件即可破除死锁：

**1. 互斥条件**

锁资源是排他性的，同一时刻只能被一个线程持有，其他线程必须等待。synchronized、Lock均满足互斥，该条件无法人为破除。

**2. 持有并等待条件**

一个线程已经持有A锁，在不释放A锁的前提下，继续请求获取B锁，等待B锁释放。

**3. 不可剥夺条件**

线程持有锁后，不会被其他线程强制抢占、剥夺资源，只能自己主动释放锁。

**4. 循环等待条件**

多个线程形成闭环等待：线程1持有A锁等B锁，线程2持有B锁等A锁，形成循环依赖等待。

### 6.1.3 死锁产生的典型场景示例

最经典的**双锁嵌套循环等待死锁**代码示例，可直接复现死锁现象：

```java
/**
 * 经典死锁场景演示
 * 满足四大死锁条件，程序永久阻塞
 */
public class DeadLockDemo {
    // 定义两把独立锁资源
    private static final Object LOCK_A = new Object();
    private static final Object LOCK_B = new Object();

    public static void main(String[] args) {
        // 线程1：先拿A锁，再请求B锁
        new Thread(() -> {
            synchronized (LOCK_A) {
                System.out.println("线程1 获取到LOCK_A，等待LOCK_B");
                try {
                    Thread.sleep(1000);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
                synchronized (LOCK_B) {
                    System.out.println("线程1 同时获取两把锁，执行完毕");
                }
            }
        }).start();

        // 线程2：先拿B锁，再请求A锁
        new Thread(() -> {
            synchronized (LOCK_B) {
                System.out.println("线程2 获取到LOCK_B，等待LOCK_A");
                try {
                    Thread.sleep(1000);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
                synchronized (LOCK_A) {
                    System.out.println("线程2 同时获取两把锁，执行完毕");
                }
            }
        }).start();
    }
}

```

**现象说明**：线程1持有A等B，线程2持有B等A，四个条件同时满足，程序卡死、无日志输出、线程永久阻塞。

## 6.2 死锁的解决方法

死锁无法自动解除，解决核心思路：**破坏四大必要条件中的任意一个**，日常开发主要破坏「持有并等待、不可剥夺、循环等待」三个可人为干预条件。

### 6.2.1 破坏持有并等待条件：一次性申请所有资源

**核心思路**：禁止线程持有部分资源再等待其他资源，线程必须**一次性申请所有需要的锁**，全部获取成功才执行业务，任意一个锁获取失败则全部释放，重新申请。

**优缺点**：简单有效、彻底杜绝持有并等待；缺点是锁粒度变大、并发吞吐量降低，仅适用于锁资源少的简单场景。

### 6.2.2 破坏不可剥夺条件：超时等待、中断线程

**核心思路**：线程持有锁后，等待其他锁时**不永久阻塞**，设置超时时间或可中断等待，超时/中断后主动释放已持有的锁，破除不可剥夺条件。

**技术实现**

synchronized不支持超时与中断，生产使用**ReentrantLock.tryLock(timeout)** 超时抢锁、**lockInterruptibly()** 可中断抢锁，超时主动释放资源，避免永久等待。

### 6.2.3 破坏循环等待条件：按统一顺序申请锁

**核心思路（生产最常用方案）**：所有线程必须按照**全局统一的固定顺序**申请锁资源，例如：永远先申请编号小的锁、再申请编号大的锁，彻底杜绝循环依赖等待。

**改造上述死锁案例**：所有线程统一先拿LOCK_A、再拿LOCK_B，彻底消除循环等待，不会产生死锁。

**优势**：不降低并发性能、改造简单、适配绝大多数嵌套锁场景，是企业级首选方案。

### 6.2.4 死锁检测与恢复：JVM工具（`jstack`）排查死锁

线上出现死锁后，可通过JDK自带工具快速定位死锁线程、锁资源、代码位置：

**排查步骤**

1. `jps -l` 查看Java进程PID；

2. `jstack PID` 打印线程堆栈信息；

3. 检索 **Found one Java-level deadlock** 关键字，定位死锁线程、锁对象、阻塞代码行。

**恢复方案**：线上死锁无法自动恢复，只能**重启服务**临时恢复，根据jstack日志修复代码逻辑彻底根治。

## 6.3 生产环境死锁预防实践

### 6.3.1 锁的使用规范：统一锁顺序、减少锁持有时间

**统一锁顺序**：项目内部制定锁资源排序规则，所有嵌套锁严格按照固定顺序获取；

**缩小锁粒度**：只锁住需要同步的核心代码，禁止大代码块、大事务加锁；

**减少锁持有时间**：锁内只做同步操作，禁止锁内IO、网络请求、耗时查询，快速释放锁资源。

### 6.3.2 避免嵌套锁、循环依赖锁

**生产红线规范**：尽量杜绝多层嵌套锁，嵌套锁是死锁最高发场景；

业务设计上避免锁循环依赖，禁止互相持有对方锁资源的业务逻辑；

复杂并发场景优先使用细粒度锁、读写锁，而非多层嵌套独占锁。

### 6.3.3 锁超时设置与监控告警

1. 生产优先使用ReentrantLock，通过tryLock超时机制，避免永久阻塞等待；

2. 接入线程监控工具，监控线程阻塞、死锁状态；

3. 定时线程检测死锁状态，触发钉钉、短信告警，及时发现线上故障；

4. 代码Review强制校验嵌套锁顺序、锁依赖问题，提前规避风险。

# 7. 线程间通信的方式

## 7.1 线程间通信的核心方式

多线程执行时默认相互独立、互不感知，线程间通信的本质是**实现多线程数据共享、执行时序协同、任务协作调度**。Java提供了多种通信机制，从基础的共享变量到JUC高级工具类，适配不同复杂度的并发场景。

### 7.1.1 共享变量通信：`volatile`修饰变量实现状态通知

**核心原理**：利用volatile的**内存可见性**特性，多个线程共享同一个被volatile修饰的全局变量，一个线程修改变量值并立即刷新至主内存，其他线程实时读取主内存最新值，实现线程间状态传递与通信。

**核心特点**：无锁、轻量化、性能极高、实现简单，仅适用于**状态标记、单写多读**场景，无法实现复杂的线程等待、精准唤醒、时序控制。

**完整实操代码**

```java
/**
 * volatile共享变量实现线程通信
 * 场景：主线程修改状态，子线程感知并终止任务
 */
public class VolatileCommunicateDemo {
    // 共享状态变量，volatile保证可见性
    private static volatile boolean running = true;

    public static void main(String[] args) throws InterruptedException {
        // 工作线程：循环执行任务，监听状态变量
        new Thread(() -> {
            while (running) {
                // 执行业务逻辑
            }
            System.out.println("子线程感知状态变更，任务终止");
        }).start();

        // 主线程修改共享变量，通知子线程
        Thread.sleep(2000);
        running = false;
        System.out.println("主线程修改共享状态，发送停止通知");
    }
}

```

**生产适用场景**：服务启停标记、任务开关控制、全局配置动态更新等简单状态通知场景。

### 7.1.2 `wait()/notify()`等待-通知机制：基于对象锁的线程协作

**核心定位**：JDK原生基于**对象监视器锁**的线程等待唤醒机制，是经典的生产者-消费者模型底层实现，用于解决多线程**执行时序依赖**问题。

**核心方法说明**

- **wait()**：释放当前对象锁，线程进入WAITING等待池，等待被其他线程唤醒；

- **notify()**：随机唤醒等待池中一个等待线程；

- **notifyAll()**：唤醒等待池中所有等待线程。

**核心执行流程**

1. 线程获取synchronized对象锁；

2. 条件不满足时调用wait()，**主动释放锁**，进入阻塞等待；

3. 其他线程获取锁，修改条件后调用notify/notifyAll()；

4. 等待线程被唤醒，重新竞争锁，继续执行业务逻辑。

**生产者消费者极简示例**

```java
/**
 * wait/notify 实现生产者消费者通信
 */
public class WaitNotifyDemo {
    // 共享数据标记
    private static boolean hasData = false;
    private static final Object LOCK = new Object();

    // 生产者线程
    public static void producer() {
        synchronized (LOCK) {
            try {
                // 已有数据则等待消费
                while (hasData) {
                    LOCK.wait();
                }
                System.out.println("生产者：生产数据完成");
                hasData = true;
                // 唤醒消费者
                LOCK.notify();
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }
    }

    // 消费者线程
    public static void consumer() {
        synchronized (LOCK) {
            try {
                // 无数据则等待生产
                while (!hasData) {
                    LOCK.wait();
                }
                System.out.println("消费者：消费数据完成");
                hasData = false;
                // 唤醒生产者
                LOCK.notify();
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
        }
    }

    public static void main(String[] args) {
        new Thread(WaitNotifyDemo::producer).start();
        new Thread(WaitNotifyDemo::consumer).start();
    }
}

```

### 7.1.3 `Lock`与`Condition`：多条件队列实现精准通知

**核心优势**：相较于原生的wait/notify，**ReentrantLock + Condition** 支持**多条件等待队列**，可以实现精准唤醒、分组唤醒，解决原生机制只能随机唤醒、效率低的问题。

**核心原理**：一个Lock锁可以绑定多个Condition条件队列，不同线程监听不同条件，唤醒时只唤醒对应条件的等待线程，实现精准通信。

**核心方法**

- await()：等效于wait()，释放锁并等待；

- signal()：等效于notify()，精准唤醒当前条件下一个线程；

- signalAll()：唤醒当前条件所有线程。

**精准唤醒代码示例**

```java
import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.ReentrantLock;

/**
 * Lock+Condition 实现精准线程通信
 * 区分生产者、消费者独立条件队列，精准唤醒
 */
public class ConditionCommunicateDemo {
    private static final ReentrantLock LOCK = new ReentrantLock();
    // 生产者条件队列
    private static final Condition PRODUCER_COND = LOCK.newCondition();
    // 消费者条件队列
    private static final Condition CONSUMER_COND = LOCK.newCondition();
    private static boolean hasData = false;

    public static void producer() {
        LOCK.lock();
        try {
            while (hasData) {
                // 生产者等待
                PRODUCER_COND.await();
            }
            System.out.println("Condition生产者：生产数据完成");
            hasData = true;
            // 精准唤醒消费者
            CONSUMER_COND.signal();
        } catch (InterruptedException e) {
            e.printStackTrace();
        } finally {
            LOCK.unlock();
        }
    }

    public static void consumer() {
        LOCK.lock();
        try {
            while (!hasData) {
                // 消费者等待
                CONSUMER_COND.await();
            }
            System.out.println("Condition消费者：消费数据完成");
            hasData = false;
            // 精准唤醒生产者
            PRODUCER_COND.signal();
        } catch (InterruptedException e) {
            e.printStackTrace();
        } finally {
            LOCK.unlock();
        }
    }

    public static void main(String[] args) {
        new Thread(ConditionCommunicateDemo::producer).start();
        new Thread(ConditionCommunicateDemo::consumer).start();
    }
}

```

### 7.1.4 `CountDownLatch`/`CyclicBarrier`：线程同步工具类

JUC并发包提供的线程同步工具，专门用于**多线程协同工作、等待批量线程执行完毕**，是生产高频使用的通信同步方案。

**1. CountDownLatch：倒计时闭锁（一次性同步）**

核心作用：允许一个或多个线程等待其他一批线程执行完成后，再继续执行。计数器递减，归零后唤醒等待线程，**一次性使用、不可重置**。

适用场景：主线程等待多个子线程任务全部执行完毕后汇总结果。

**极简示例**

```java
import java.util.concurrent.CountDownLatch;

public class CountDownLatchDemo {
    public static void main(String[] args) throws InterruptedException {
        // 计数器初始值3
        CountDownLatch latch = new CountDownLatch(3);

        for (int i = 1; i <= 3; i++) {
            new Thread(() -> {
                System.out.println(Thread.currentThread().getName() + " 执行完毕");
                // 计数器-1
                latch.countDown();
            }, "线程" + i).start();
        }
        // 主线程阻塞等待计数器归零
        latch.await();
        System.out.println("所有子线程执行完毕，主线程汇总结果");
    }
}

```

**2. CyclicBarrier：循环栅栏（可重复同步）**

核心作用：让一组线程互相等待，所有线程到达栅栏节点后，再统一同时执行。计数器可**循环重置、重复使用**。

适用场景：多线程分批次执行、分批等待、批量任务同步。

### 7.1.5 `Semaphore`：信号量实现资源控制与通信

**核心原理**：Semaphore（信号量）通过维护一个**许可计数器**，控制同时访问共享资源的线程数量，实现限流、资源抢占、线程间同步通信。

**核心能力**

- acquire()：获取许可，无许可则阻塞等待；

- release()：释放许可，唤醒等待线程。

适用场景：接口限流、资源池控制（连接池、线程池）、并发数控制，通过许可的抢占与释放实现线程间接通信。

**限流示例**

```java
import java.util.concurrent.Semaphore;

/**
 * Semaphore实现并发限流：同时最多2个线程执行任务
 */
public class SemaphoreDemo {
    // 设置许可数为2，最大并发2
    private static final Semaphore SEMAPHORE = new Semaphore(2);

    public static void main(String[] args) {
        // 开启5个线程竞争资源
        for (int i = 1; i <= 5; i++) {
            new Thread(() -> {
                try {
                    // 获取许可，无许可则阻塞
                    SEMAPHORE.acquire();
                    System.out.println(Thread.currentThread().getName() + " 获取资源，执行任务");
                    Thread.sleep(2000);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                } finally {
                    // 释放许可
                    SEMAPHORE.release();
                    System.out.println(Thread.currentThread().getName() + " 释放资源");
                }
            }, "线程" + i).start();
        }
    }
}

```

## 7.2 各通信方式的适用场景对比

### 7.2.1 简单状态通知：优先共享变量+`volatile`

针对**无复杂等待、无需唤醒、仅状态变更通知**的极简场景，优先使用volatile共享变量通信。优势是无锁开销、性能极高、代码简洁，仅需保证变量可见性，适合启停开关、全局状态、动态配置更新等场景。

### 7.2.2 线程等待-唤醒：优先`wait()/notify()`或`Condition`

针对**线程间存在执行依赖、需要阻塞等待、条件满足后唤醒**的场景（典型生产者消费者模型）：

- 简单单条件等待唤醒：使用原生 wait/notify；

- 多条件、分组唤醒、精准唤醒：优先 Condition + ReentrantLock，性能与可控性更强。

### 7.2.3 多线程同步控制：优先并发工具类

针对**批量线程协同、任务汇总、并发限流、分阶段执行**场景，放弃底层wait/notify手写逻辑，优先使用JUC工具类：

- 批量线程执行完毕后汇总：CountDownLatch；

- 多线程分批同步、循环等待：CyclicBarrier；

- 并发限流、资源抢占控制：Semaphore。

优势：封装完善、无需手动处理锁与唤醒逻辑、代码简洁、不易出错。

### 7.2.4 生产环境选型建议

1. **极简状态通信**：volatile 共享变量，追求极致性能；

2. **基础生产者消费者**：简单场景用 wait/notify，复杂多条件场景用 Condition；

3. **批量任务同步**：优先JUC工具类，杜绝手写等待唤醒逻辑；

4. **并发限流、资源控制**：固定使用 Semaphore；

5. **禁止过度设计**：简单场景不滥用复杂工具类，复杂场景不手写底层通信逻辑。

## 7.3 高频面试延伸问题

### 7.3.1 `wait()/notify()`为什么必须在同步代码块中调用？

**核心原因（面试标准答案）**：

1. **防止线程丢失唤醒（核心原因）**：如果不在同步代码块执行，会存在「判断条件不满足->还未执行wait->其他线程执行notify」的时间差，导致唤醒信号提前丢失，后续线程永久阻塞，引发程序卡死；

2. **锁的归属要求**：wait()方法的核心逻辑是**释放当前对象锁**，只有获取锁的线程才能释放锁，因此必须在同步代码块/同步方法内执行；

3. **JDK语法强制校验**：脱离同步锁直接调用会直接抛出 **IllegalMonitorStateException** 异常。

### 7.3.2 生产者-消费者模型有几种实现方式？优缺点？

**三种主流实现方式**

1. **synchronized + wait/notify**

   - 优点：JDK原生、无需依赖工具类、实现简单；

   - 缺点：随机唤醒、无法精准通知、并发效率低、容易线程饿死。

2. **ReentrantLock + Condition**
   - 优点：多条件队列、精准唤醒、并发性能高、可控性强；

   - 缺点：代码稍复杂，需要手动加锁解锁。

3. **阻塞队列 BlockingQueue**

   - 优点：JUC封装、开箱即用、线程安全、无需手动控制等待唤醒；

   - 缺点：灵活性较低，适配固定的生产消费场景。

### 7.3.3 `Condition`与`wait()/notify()`的区别是什么？

| 对比维度   | wait/notify                | Condition                     |
| ---------- | -------------------------- | ----------------------------- |
| 所属锁机制 | 基于synchronized内置锁     | 基于Lock显式锁                |
| 等待队列   | 单个锁对应唯一等待队列     | 一个锁可绑定多个Condition队列 |
| 唤醒精度   | 随机唤醒，无法精准控制     | 精准唤醒指定条件的线程        |
| 性能       | 高并发下无效唤醒多，性能差 | 无无效唤醒，并发性能更高      |
| 灵活性     | 低，功能固定               | 高，支持超时等待、可中断等待  |

# 8. 常见易错点：`wait()` 与 `sleep()` 的区别、`notify()` 与 `notifyAll()` 的选择、虚假唤醒问题

## 8.1 `wait()` 与 `sleep()` 的区别

wait()与sleep()是面试**最高频对比题**，二者均能让线程进入阻塞等待状态，但底层原理、锁行为、调用约束完全不同，是开发极易混淆的易错点。

### 8.1.1 所属类不同：`Object`类vs`Thread`类

- **wait()**：属于 **Object类** 的方法，所有对象都拥有该方法，是锁级别的协作方法；

- **sleep()**：属于 **Thread类** 的静态方法，是线程自身的休眠方法。

本质区别：wait是**线程间通信协作**方法，sleep是**线程自身延时**方法。

### 8.1.2 锁释放情况：释放锁vs不释放锁

**核心最关键区别**

- **wait()**：调用后**主动释放当前持有的锁**，其他线程可以正常竞争锁执行任务，不会阻塞其他线程；

- **sleep()**：调用后**持有锁不释放**，线程休眠期间依旧占用锁资源，其他线程无法获取锁，会造成锁阻塞。

### 8.1.3 调用场景：必须在同步代码块中vs任意场景

- **wait()**：强制要求在**同步代码块/同步方法**中调用，否则直接抛出监视器状态异常；

- **sleep()**：无任何场景限制，任意位置均可调用。

### 8.1.4 唤醒方式：需被`notify()`/`notifyAll()`唤醒或超时vs超时自动唤醒

- **wait()**：无参wait()永久阻塞，必须被其他线程notify/notifyAll唤醒；有参wait(time)超时后自动唤醒；

- **sleep()**：固定时间休眠，**超时自动唤醒**，无需其他线程干预，也无法被其他线程唤醒（仅可中断）。

### 8.1.5 异常处理：不抛异常vs必须处理`InterruptedException`

- **wait()**：抛出检查异常，需要捕获或抛出；

- **sleep()**：同样抛出InterruptedException检查异常，必须手动捕获或抛出；

**补充区别总结**：wait()是为了**多线程协作**，主动让步资源；sleep()是为了**线程延时执行**，独占资源不释放。

## 8.2 `notify()` 与 `notifyAll()` 的选择

### 8.2.1 两者的核心区别：唤醒单个线程vs唤醒所有等待线程

- **notify()**：随机唤醒对象等待池中的**任意一个**等待线程，其余线程继续等待；

- **notifyAll()**：唤醒对象等待池中的**所有**等待线程，被唤醒线程重新竞争锁资源。

### 8.2.2 生产环境推荐使用`notifyAll()`的原因：避免线程饿死

**生产强制规范**：业务代码中**优先使用notifyAll()**，尽量避免使用notify()。

**核心原因**：notify()随机唤醒单个线程，可能导致部分线程永久无法被唤醒，产生**线程饿死问题**。例如多生产者多消费者场景，可能一直唤醒同类线程，导致另一类线程永久阻塞，业务卡死。

notifyAll()虽然会产生短暂的锁竞争消耗，但能保证所有线程都有执行机会，**牺牲极小性能、换取业务绝对安全**，符合生产稳定性优先原则。

### 8.2.3 仅使用`notify()`的风险与适用场景

**notify()风险汇总**

1. 线程饿死：部分线程长期得不到唤醒，永久阻塞；

2. 业务卡死：多条件等待场景下，唤醒错误类型线程，导致所有线程继续等待；

3. 问题隐蔽：线上偶现卡死，难以复现、难以排查。

**唯一适用场景**：严格的**一对一等待唤醒**场景，一个等待线程、一个唤醒线程，无多线程竞争，可安全使用notify()。

## 8.3 虚假唤醒问题

### 8.3.1 虚假唤醒的定义：线程被意外唤醒但条件未满足

**虚假唤醒**指：处于WAITING状态的线程，**没有被notify/notifyAll唤醒**，被系统自动唤醒，此时线程等待的业务条件**并未满足**，如果直接执行业务逻辑，会导致程序异常、数据错乱。

### 8.3.2 虚假唤醒的底层原因：操作系统调度机制

虚假唤醒是**操作系统层面的机制**，并非JavaBUG。为了提升CPU调度效率，操作系统会偶尔主动唤醒处于阻塞等待状态的线程，JVM无法阻止该行为，属于正常的系统调度现象。

官方文档明确说明：Java线程的wait()方法存在极小概率的虚假唤醒，开发者必须自行处理。

### 8.3.3 虚假唤醒的危害与解决方案：使用`while`循环二次校验条件

**核心危害**：如果使用if判断条件，线程被虚假唤醒后，不会二次校验条件，直接向下执行代码，由于业务条件不满足，直接引发空数据、数据覆盖、数组越界、业务异常等问题。

**唯一解决方案**：**wait()必须配合while循环做条件循环校验**

原理：线程被唤醒后，会**重新循环判断条件**，如果是虚假唤醒、条件不满足，会再次调用wait()进入等待状态，彻底杜绝虚假唤醒导致的异常。

**错误写法（if，存在虚假唤醒风险）**

```java
// 错误写法！if只判断一次，虚假唤醒直接执行，存在BUG
if (!hasData) {
    lock.wait();
}

```

**正确写法（while，生产强制规范）**

```java
// 正确写法！循环校验条件，杜绝虚假唤醒
while (!hasData) {
    lock.wait();
}

```

### 8.3.4 生产环境强制规范：`wait()`必须配合`while`循环使用

**企业级开发硬性规范**：所有使用 wait()、await() 的代码，**禁止使用if判断条件**，必须通过while循环包裹，实现唤醒后二次条件校验。该规范可以100%解决虚假唤醒问题，是所有并发开发的基础红线。
