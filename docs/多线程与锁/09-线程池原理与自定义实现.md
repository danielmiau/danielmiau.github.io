# 1. 线程池的核心作用：复用线程、控制并发数、管理线程生命周期

在Java原始线程使用模式中，业务需要异步任务时手动`new Thread()`创建线程，任务执行完毕后线程自动销毁。这种即兴创建、即时销毁的模式，在高并发、大量短任务场景下存在严重性能损耗与系统风险，而**线程池**就是为解决原生线程滥用问题而生的统一线程管理框架。

## 1.1 线程池的核心设计思想

### 1.1.1 线程池的定义：预先创建线程、复用线程执行任务的并发框架

**线程池**是JDK提供的一套**线程复用与并发管控的容器框架**，核心设计逻辑为：程序启动或任务初始化时，预先创建固定数量的线程并常驻内存，统一维护、调度、复用这批线程处理业务任务，任务执行完成后线程不销毁，回归线程池等待下一次任务，实现线程的循环复用。

线程池将**线程创建、任务调度、线程销毁**的逻辑统一封装，开发者只需提交业务任务，无需手动管理线程，彻底解耦线程与业务任务，是工业级并发编程的标准实现方式。

### 1.1.2 线程池解决的核心问题：频繁创建/销毁线程的开销

Java中线程属于**重量级资源**，线程的创建和销毁需要操作系统参与，涉及内核态与用户态切换，存在极高的系统开销：

- **内存开销**：每个Java线程默认拥有独立栈内存（默认1M左右），大量线程创建会占用大量堆外内存，容易导致OOM内存溢出；

- **CPU开销**：线程创建、销毁、上下文切换会消耗大量CPU资源，频繁操作会导致CPU使用率飙升、系统吞吐量下降；

- **系统调度开销**：操作系统可调度的线程数有限，大量瞬时线程会导致内核调度压力过大，出现任务卡顿、响应超时。

原生`new Thread()`模式，每一个短任务都需要创建、销毁一次线程，任务量越大，无效系统开销越高。**线程池的核心初衷就是消灭这种无效开销**，通过线程复用，让一个线程循环处理多个任务，避免频繁创建销毁线程。

### 1.1.3 线程池的核心优势：性能提升、资源控制、统一管理

相比于手动创建线程，线程池具备三大核心不可替代的优势，也是生产环境强制使用线程池的核心原因：

1. **性能大幅提升**：规避线程频繁创建销毁的开销，线程常驻复用，任务响应速度更快，系统吞吐量显著提升；

2. **系统资源可控**：限制最大并发线程数，杜绝无限创建线程导致的内存溢出、CPU打满、系统雪崩问题；

3. **统一运维管理**：线程池统一管理线程生命周期，支持任务排队、超时控制、拒绝策略、监控统计，便于线上问题排查、任务管控与系统调优。

## 1.2 线程池的三大核心作用

### 1.2.1 复用线程：减少线程创建与销毁的系统开销

**线程复用**是线程池最核心、最基础的作用。手动创建线程模式为：`创建线程→执行任务→销毁线程`，单次任务伴随一次完整的线程生命周期，短任务场景下，线程创建销毁的开销甚至大于任务执行本身的开销。

线程池模式为：`初始化创建线程→循环执行多个任务→长期常驻→空闲回收`。线程创建一次，可反复处理成千上万个业务任务，彻底消灭了频繁创建销毁线程的无效损耗，极大提升系统并发处理效率，这也是高并发场景性能优化的核心手段。

### 1.2.2 控制并发数：限制同时执行的线程数量，避免系统过载

系统硬件资源（CPU、内存）是有限的，支持的最大并发线程数存在上限。若无限创建线程，会出现大量线程抢占CPU资源、频繁上下文切换、内存占用过高的问题，最终导致系统过载、服务卡顿、雪崩宕机。

线程池支持**自定义最大并发线程数**，严格限制同一时间正在执行的任务线程数量，超出并发上限的任务会进入队列排队，不会新建线程抢占资源，实现**流量削峰、资源保护、并发限流**的效果，保障高并发场景下系统的稳定性。

### 1.2.3 管理线程生命周期：统一管理线程的创建、执行、销毁

原生手动创建的线程，生命周期完全不可控，开发者无法统一管理线程状态、空闲回收、异常终止，极易出现线程泄漏、僵尸线程、后台线程堆积等线上问题。

线程池实现了**线程全生命周期自动化管理**：

- 任务少时：自动维持核心线程常驻，快速响应新任务；

- 任务多时：自动扩容非核心线程，提升处理效率；

- 任务空闲时：自动回收超时闲置线程，释放系统资源；

- 任务溢出时：通过拒绝策略兜底，避免任务堆积。

所有线程的创建、运行、空闲、销毁全部由线程池统一调度，极大降低了多线程开发的运维成本。

## 1.3 线程池的适用场景

### 1.3.1 大量短任务并发执行场景（如接口请求处理）

这是线程池**最适配、最常用**的场景。短任务指执行耗时极短（毫秒级）的任务，例如HTTP接口请求处理、参数校验、数据封装、简单查询、消息消费等。

这类场景任务量大、瞬时并发高、单任务耗时短，若手动创建线程，绝大部分系统资源都会消耗在线程创建销毁上，业务处理效率极低。使用线程池复用线程，可最大化提升吞吐量，是微服务接口、网关服务的标准实现方案。

### 1.3.2 异步任务、后台任务处理场景

业务中无需同步阻塞等待的后台任务，全部适合线程池处理：日志异步打印、操作记录异步存储、消息推送、邮件短信发送、数据统计、定时任务兜底处理等。

通过线程池实现业务主线程与后台任务解耦，主线程无需等待耗时后台任务执行完成，大幅提升接口响应速度，优化用户体验，同时统一管控后台异步任务，避免异步线程失控堆积。

### 1.3.3 高并发服务、微服务接口限流场景

高并发秒杀、活动引流、高频查询等场景，瞬时流量极大，极易打垮后端服务与数据库。线程池通过**限制最大并发线程数**，实现单机接口的底层限流，控制同时处理的请求数量，配合队列缓冲瞬时流量，避免流量峰值冲击导致的服务过载，是高并发服务稳定性保障的基础手段。

同时，线程池的任务队列、拒绝策略可实现流量削峰、过载保护，防止大量请求堆积导致服务宕机，适配生产高可用要求。

# 2. `ThreadPoolExecutor` 核心参数：`corePoolSize`/`maximumPoolSize`/`keepAliveTime`/`unit`/`workQueue`/`handler`

ThreadPoolExecutor 是 JDK 线程池的**真正底层实现类**，所有 Executors 工具类的线程池，底层都是对该类的封装。其完整构造方法包含七大参数，每一个参数都决定线程池的运行特性，生产环境**禁止使用 Executors 快捷创建**，必须手动通过该构造方法自定义线程池。

标准构造方法如下：

```java

                          int maximumPoolSize,
                          long keepAliveTime,
                          TimeUnit unit,
                          BlockingQueue<Runnable> workQueue,
                          ThreadFactory threadFactory,
                          RejectedExecutionHandler handler) {

}
```

## 2.1 核心线程数相关参数

### 2.1.1 `corePoolSize`：核心线程数（常驻线程数）

**定义**：核心线程数，是线程池**常驻、不会被回收**的最小线程数量，也是线程池的基础工作线程数量。

**核心特性**：

- 线程池初始化后，默认不会提前创建核心线程，在提交任务时才按需创建（可通过 `prestartAllCoreThreads()` 预创建）；

- 无论线程池是否空闲、是否超时，**核心线程默认永远不会被回收**；

- 线程池正常工作时，优先使用核心线程处理任务，保证低并发场景下的响应速度。

**生产意义**：核心线程数用于保障日常平稳流量的任务处理能力，避免频繁创建销毁线程，保证服务稳态性能。

### 2.1.2 `maximumPoolSize`：最大线程数（核心+非核心线程总数）

**定义**：线程池允许创建的**最大线程总数**，包含核心线程与非核心线程，是线程池并发能力的上限。

**核心特性**：

- 最大线程数 = 核心线程数 + 非核心线程数；

- 仅当**核心线程全部繁忙、任务队列已满**时，线程池才会创建非核心线程处理新任务；

- 线程池绝对不会创建超过该数值的线程，用于限制系统最大并发，保护服务器资源。

**硬性约束**：maximumPoolSize 必须大于 corePoolSize，否则直接抛出非法参数异常。

### 2.1.3 核心线程与非核心线程的区别与管理机制

很多面试场景会重点区分两者，核心差异集中在**生命周期、回收机制、优先级**：

| 对比维度 | 核心线程               | 非核心线程                 |
| -------- | ---------------------- | -------------------------- |
| 生命周期 | 常驻线程，默认永久存活 | 临时线程，属于流量扩容线程 |
| 回收机制 | 不会被空闲回收         | 空闲超时后自动被销毁       |
| 创建时机 | 任务少时优先创建       | 队列满、核心线程忙时才创建 |
| 作用     | 保障稳态流量处理       | 峰值流量临时扩容           |

**补充机制**：可通过 `allowCoreThreadTimeOut(true)` 开启核心线程超时回收，适用于低频任务场景，进一步节省资源。

## 2.2 线程存活时间参数

### 2.2.1 `keepAliveTime`：非核心线程的空闲存活时间

**定义**：非核心线程在空闲状态下的最大存活时长，是临时线程的回收阈值。

**核心逻辑**：当非核心线程执行完任务后，会进入空闲等待状态，如果空闲等待时间超过 keepAliveTime，线程池判定当前流量回落、资源富余，自动回收该非核心线程，释放内存与CPU资源。

**生产适配**：流量波动大的服务，可适当调大该值，避免流量反复波动导致线程频繁创建销毁。

### 2.2.2 `unit`：时间单位（TimeUnit）

用于指定 keepAliveTime 的时间单位，枚举类型 **TimeUnit**，常用单位：纳秒、微秒、毫秒、秒、分钟、小时。

**生产标准写法**：统一使用 `TimeUnit.SECONDS` 或 `TimeUnit.MILLISECONDS`，可读性高、不易出错。

### 2.2.3 空闲线程回收机制：超过存活时间的非核心线程会被回收

线程池的资源动态收缩机制完整流程：

1. 峰值流量过后，任务量减少，大量非核心线程进入空闲状态；

2. 线程池定时检测空闲线程的等待时长；

3. 空闲时间超过 keepAliveTime 的非核心线程被销毁回收；

4. 最终线程池收缩至核心线程数规模，维持稳态最小资源占用。

**关键结论**：线程池可以**自动扩容应对峰值、自动收缩节省资源**，这是手动创建线程无法实现的优势。

## 2.3 任务队列 `workQueue`

workQueue 是**阻塞任务队列**，用于存储所有等待执行的任务。当核心线程全部繁忙时，新提交的任务会进入队列排队，队列满后才会创建非核心线程，队列+线程共同实现流量削峰。JDK 提供四类高频队列，特性完全不同，直接决定线程池行为。

### 2.3.1 `ArrayBlockingQueue`：有界阻塞队列

**底层结构**：基于数组实现的**固定长度有界队列**，必须初始化队列容量。

**核心特性**：

- 队列容量固定，不会无限扩容；

- 遵循先进先出规则；

- 队列满后继续提交任务，会触发非核心线程创建，线程数打满后触发拒绝策略。

**适用场景**：生产高并发核心服务，**推荐优先使用**，可限制任务堆积量，避免 OOM。

### 2.3.2 `LinkedBlockingQueue`：无界/有界阻塞队列

**底层结构**：基于链表实现的队列，可指定容量，不指定则为**无界队列**。

**核心特性**：

- 无界模式下队列可无限存放任务，不会触发队列满的状态；

- **致命特点**：无界队列永远不会创建非核心线程，线程池最大并发永远等于核心线程数；

- 任务无限堆积，高并发下极易内存溢出。

**避坑指南**：Executors.newFixedThreadPool()、newSingleThreadExecutor() 默认使用无界 LinkedBlockingQueue，生产禁止使用。

### 2.3.3 `SynchronousQueue`：同步移交队列

**底层特性**：**不存储任务的队列**，队列容量永远为0。

**执行逻辑**：提交任务后，必须立刻找到线程执行，没有空闲线程则直接新建线程，不会排队。

**核心特点**：最大线程数必须设置很大，否则极易触发拒绝策略。

**适用场景**：瞬时高并发、任务不允许排队、需要立刻执行的场景，对应 Executors.newCachedThreadPool()。

### 2.3.4 `PriorityBlockingQueue`：优先级阻塞队列

**底层特性**：无界优先级队列，可自定义任务排序规则，优先级高的任务优先执行。

**适用场景**：任务有优先级差异的业务，例如VIP用户请求优先、紧急任务优先处理。

**缺点**：会打乱任务执行顺序，存在低优先级任务饥饿问题。

### 2.3.5 队列类型对线程池行为的影响

队列是线程池行为的“开关”，不同队列直接改变线程池的扩容逻辑：

1. **无界队列**：永远不扩容非核心线程、永远不触发拒绝策略，任务无限堆积，高危；

2. **有界队列**：正常走核心线程→队列排队→扩容非核心线程→拒绝策略完整流程，生产可控；

3. **同步队列**：无排队逻辑，直接创建线程执行，并发极高、资源消耗极大。

## 2.4 拒绝策略 `handler`

### 2.4.1 拒绝策略的触发条件：线程池与队列均满时触发

拒绝策略不是随时触发，仅在**线程池满载 + 任务队列已满**的双重条件下触发：

1. 所有核心线程、非核心线程全部繁忙（线程数 == maximumPoolSize）；

2. 任务队列已经写满，无法继续排队；

3. 新任务继续提交，触发拒绝策略，执行过载兜底逻辑。

### 2.4.2 `AbortPolicy`：直接抛出异常（默认策略）

**默认策略**，线程池满载后直接抛出**RejectedExecutionException** 运行时异常。

**优缺点**：异常明显、便于发现流量峰值问题，但会直接导致接口报错、用户体验差。

**适用场景**：后台任务、非核心业务，需要快速暴露问题的场景。

### 2.4.3 `CallerRunsPolicy`：调用者线程执行任务

线程池无法处理任务时，**退回给提交任务的主线程执行**，不抛异常、不丢弃任务。

**核心价值**：无损任务、天然限流，主线程被占用，无法继续提交新任务，实现流量自我抑制。

**适用场景**：核心业务、不允许丢任务的场景，生产最常用兜底策略。

### 2.4.4 `DiscardPolicy`：直接丢弃任务

线程池满载后，**静默丢弃新任务，无异常、无日志、无提示**。

**风险**：业务无声丢失数据，线上问题极难排查。

**适用场景**：非核心、可丢弃的日志、统计、监控任务。

### 2.4.5 `DiscardOldestPolicy`：丢弃队列中最旧的任务

线程池满载时，丢弃队列头部排队最久的旧任务，腾出位置，放入当前新任务。

**风险**：丢弃旧业务任务，业务顺序错乱、数据丢失。

**适用场景**：时效性极强的任务，旧任务无保留价值（如实时热点统计、实时刷新数据）。

### 2.4.6 自定义拒绝策略的实现方式

JDK自带策略无法满足业务日志、告警、重试需求，生产环境**推荐自定义拒绝策略**，实现日志记录、告警推送、任务重试、降级兜底。

```java
import java.util.concurrent.RejectedExecutionHandler;
import java.util.concurrent.ThreadPoolExecutor;

/**
 * 生产级自定义拒绝策略
 * 功能：打印日志、告警、降级处理，不粗暴抛异常
 */
public class CustomRejectHandler implements RejectedExecutionHandler {
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        // 1. 打印详细日志，方便问题排查
        System.err.println("线程池满载，任务被拒绝，当前核心线程数：" + executor.getCorePoolSize()
                + "，活跃线程数：" + executor.getActiveCount()
                + "，队列剩余容量：" + executor.getQueue().remainingCapacity());
        
        // 2. 可拓展：发送钉钉/邮件告警
        // 3. 可拓展：任务降级、存入MQ重试、本地缓存重试
        throw new RuntimeException("系统繁忙，请稍后重试");
    }
}

```

## 2.5 其他辅助参数

### 2.5.1 `ThreadFactory`：线程工厂，自定义线程创建方式

线程工厂用于统一创建线程，可自定义**线程名称、线程优先级、是否守护线程、线程组**。

**生产必做配置**：自定义线程名称，线上故障排查时，可通过线程名称快速定位业务线程，默认线程池线程名称无意义，无法排查问题。

```java
import java.util.concurrent.ThreadFactory;
import java.util.concurrent.atomic.AtomicInteger;

/**
 * 自定义线程工厂：统一线程命名
 */
public class CustomThreadFactory implements ThreadFactory {
    private final AtomicInteger count = new AtomicInteger(1);
    @Override
    public Thread newThread(Runnable r) {
        Thread thread = new Thread(r, "business-thread-" + count.getAndIncrement());
        // 设置非守护线程，保证任务正常执行完毕
        thread.setDaemon(false);
        return thread;
    }
}

```

### 2.5.2 `RejectedExecutionHandler`：拒绝策略接口

所有拒绝策略的顶级接口，仅有一个方法 `rejectedExecution`，所有内置策略、自定义策略都实现该接口。用户可通过实现该接口，完全自定义过载兜底逻辑，是线程池拓展的核心入口。

### 2.5.3 核心参数配置的最佳实践

结合面试与生产调优，总结通用标准配置规范：

1. **队列必须有界**：禁止使用无界队列，防止任务无限堆积OOM；

2. **线程名称自定义**：方便线上堆栈排查问题；

3. **拒绝策略自定义**：默认策略体验差，自定义实现日志+告警+降级；

4. **CPU密集型任务**：核心线程数 = CPU核心数+1，减少上下文切换；

5. **IO密集型任务**：核心线程数可适当加大（20~50），线程多等待IO不占用CPU；

6. **空闲时间统一配置**：默认5~10秒，快速回收闲置资源；

7. **禁止使用Executors工具类**：所有线程池手动new ThreadPoolExecutor构建。

# 3. 线程池的核心工作流程

线程池的所有任务处理，都遵循一套**固定的四级优先级流转机制**。理解该流程，是掌握线程池参数配置、排查线程池异常、做线程池调优的核心前提。所有参数（核心线程数、最大线程数、队列、拒绝策略）的作用，全部体现在这套执行流程中。

## 3.1 线程池工作流程概述

### 3.1.1 任务提交后的处理步骤

当业务代码通过 `execute()` 或 `submit()` 向线程池提交任务时，线程池不会随意创建线程执行任务，而是严格按照**核心线程 → 任务队列 → 非核心线程 → 拒绝策略**的优先级逐级处理，完整步骤如下：

1. 优先判断当前**核心线程数**是否已满，未满则新建核心线程执行任务；

2. 核心线程已满、全部繁忙时，判断**任务队列**是否已满，队列未满则任务入队排队；

3. 任务队列已满，无法继续排队时，判断**最大线程数**是否已满，未满则新建非核心线程执行任务；

4. 线程数已达到最大值、队列也已满，新任务触发**拒绝策略**执行过载兜底逻辑。

该流程保证了线程池在低并发、中并发、高并发、过载四种场景下，都能有序处理任务，实现稳态运行、峰值扩容、过载保护。

### 3.1.2 核心线程、队列、非核心线程、拒绝策略的流转关系

线程池采用**先核心、再排队、最后扩容、过载拒绝**的保守设计，核心流转关系可总结为：

- **低并发场景**：仅核心线程工作，无排队、无扩容，资源消耗最低；

- **中并发场景**：核心线程满载，任务进入队列排队，实现流量削峰，不扩容线程；

- **高并发峰值场景**：队列打满，临时扩容非核心线程，提升瞬时处理能力；

- **过载雪崩场景**：线程、队列全部打满，触发拒绝策略，保护服务不宕机。

核心设计思想：**尽量少创建线程、优先队列缓冲，仅峰值流量临时扩容**，兼顾性能与资源稳定性。

## 3.2 详细工作流程拆解

### 3.2.1 步骤1：判断核心线程数是否已满

任务提交的第一优先级：创建核心线程执行任务。

线程池会首先判断当前**正在工作的线程数**是否小于 **corePoolSize（核心线程数）**：

- 若是（核心线程未满）：直接新建核心线程执行当前任务，无需排队；

- 若否（核心线程已满，全部繁忙）：进入下一步队列判断逻辑。

**关键细节**：线程池默认不会预创建核心线程，只有任务提交时才会逐步创建，实现懒加载，节省启动资源。

### 3.2.2 步骤2：判断任务队列是否已满

核心线程满载后，不会直接扩容线程，而是优先将任务存入阻塞队列，这是线程池**削峰的核心**。

线程池判断 **workQueue** 是否可以继续存入任务：

- 队列未满：新任务进入队列排队，等待核心线程空闲后消费；

- 队列已满：无法缓冲任务，进入下一步线程扩容逻辑。

**面试高频坑点**：如果使用**无界队列LinkedBlockingQueue**，队列永远不会满，线程池永远不会创建非核心线程，最大并发数永远等于核心线程数。

### 3.2.3 步骤3：判断最大线程数是否已满

核心线程繁忙、队列已满，流量达到峰值，线程池开始扩容非核心线程。

线程池判断当前**总线程数**是否小于 **maximumPoolSize（最大线程数）**：

- 总线程数未达上限：新建**非核心线程**直接执行当前任务；

- 总线程数已达上限：无法继续扩容，进入拒绝策略逻辑。

**核心特点**：非核心线程仅处理队列满后的突发流量，流量回落之后会被自动回收，不长期占用资源。

### 3.2.4 步骤4：触发拒绝策略

当且仅当 **核心线程满载 + 队列已满 + 总线程数达到最大值** 三个条件同时满足时，线程池判定系统已过载，新提交任务触发拒绝策略。

此时线程池会根据预设的拒绝策略，执行兜底逻辑：抛异常、调用者执行、丢弃任务、丢弃旧任务，避免无限堆积任务导致OOM、服务雪崩。

## 3.3 任务执行与线程复用机制

### 3.3.1 线程从队列中获取任务的方式（take/poll）

线程池线程复用的核心，依赖阻塞队列的两个阻塞方法 **take()** 和 **poll()**，分别对应核心线程与非核心线程的取任务逻辑：

- **核心线程 take() 获取任务**：无任务时**永久阻塞**，不释放线程资源，实现常驻效果；

- **非核心线程 poll() 获取任务**：带超时时间获取，空闲超时未获取到任务，直接返回null，触发线程回收。

JDK通过两个方法的差异化调用，完美实现了**核心线程常驻、非核心线程可回收**的特性。

### 3.3.2 线程复用的底层实现：循环获取任务执行

很多开发者误以为“一个线程只执行一个任务”，而线程池的核心本质是：**一个线程循环不断从队列获取任务执行，实现永久复用**。

底层伪代码逻辑：

```java
// 线程池工作线程死循环逻辑
while (true) {
    // 从队列获取任务
    Runnable task = getTask();
    if (task != null) {
        // 执行任务
        task.run();
    } else {
        // 获取任务为空，线程退出（非核心线程超时场景）
        break;
    }
}

```

线程创建后不会执行完任务就销毁，而是进入死循环，持续消费队列任务，这就是**线程复用的底层原理**，彻底消灭了线程频繁创建销毁的开销。

### 3.3.3 空闲线程的回收流程

流量回落、任务减少后，线程池自动收缩资源，空闲非核心线程回收完整流程：

1. 峰值流量结束，队列任务消费完毕，非核心线程执行完任务后再次尝试 poll() 获取任务；

2. 在 keepAliveTime 时间内未获取到新任务，poll() 返回null；

3. 线程退出循环任务逻辑，线程终止；

4. 最终线程池仅保留核心线程常驻，完成资源收缩。

**拓展**：开启 `allowCoreThreadTimeOut(true)` 后，核心线程也会走超时回收逻辑，适用于低频任务场景。

# 4. 4种拒绝策略：`AbortPolicy`/`CallerRunsPolicy`/`DiscardPolicy`/`DiscardOldestPolicy`

拒绝策略是线程池**过载保护的最后一道防线**，当线程池线程、队列全部满载，无法处理新任务时，由拒绝策略决定任务的兜底处理逻辑。JDK原生提供四种内置拒绝策略，每种策略的行为、场景、风险完全不同，同时支持自定义拓展，是生产线程池配置的核心要点。

## 4.1 `AbortPolicy`（默认策略）

### 4.1.1 策略行为：直接抛出RejectedExecutionException异常

AbortPolicy 是线程池**默认拒绝策略**，也是最严格的策略。当线程池触发拒绝逻辑时，直接抛出运行时异常 **RejectedExecutionException**，中断任务提交，不做任何兜底处理。

### 4.1.2 适用场景：对任务丢失零容忍、必须处理异常的场景

适用于**后台定时任务、数据统计任务、非核心业务任务**，这类任务不允许静默丢失，需要通过异常快速发现线程池满载、流量过载问题，便于及时告警、扩容优化。

### 4.1.3 优缺点分析

**优点**：

- 问题暴露及时，不会隐藏线上流量峰值、线程池配置不足的问题；

- 有异常堆栈信息，便于日志排查问题。

**缺点**：

- 直接抛异常，导致前端/接口报错，用户体验差；

- 核心业务场景会直接导致业务失败，无兜底能力。

## 4.2 `CallerRunsPolicy`

### 4.2.1 策略行为：由提交任务的调用者线程执行任务

当线程池满载无法处理任务时，**不会抛异常、不会丢任务**，而是将任务回退给提交任务的主线程（调用者线程）自行执行。

### 4.2.2 适用场景：不允许任务丢失、且调用者线程可承受阻塞的场景

生产**最推荐的核心业务策略**，适用于订单处理、支付流程、业务数据同步等**不允许任务丢失**的核心场景，通过调用者执行实现无损兜底。

### 4.2.3 优缺点分析：会降低调用者线程性能

**优点**：

- 零任务丢失，保证核心业务数据完整性；

- 天然限流：主线程被任务阻塞，无法继续快速提交新任务，自动抑制流量。

**缺点**：

- 调用者线程被阻塞，接口响应变慢，吞吐量下降；

- 高并发下会导致大量主线程阻塞，服务吞吐量降级。

## 4.3 `DiscardPolicy`

### 4.3.1 策略行为：直接丢弃任务，不抛出异常

最佛系的拒绝策略，线程池过载时，**静默丢弃新提交的任务，无异常、无日志、无任何提示**，程序无报错、无感知。

### 4.3.2 适用场景：允许任务丢失、非关键任务场景

适用于日志打印、监控上报、实时埋点、临时统计等**可丢失、不影响核心业务**的轻量化任务。

### 4.3.3 优缺点分析：无异常但可能导致数据丢失

**优点**：服务无报错、无阻塞、吞吐量稳定，不会因为过载导致程序异常；

**缺点**：任务无声丢失，线上问题完全无法排查，长期使用会导致数据统计缺失、日志不全。

### 4.4 `DiscardOldestPolicy`

### 4.4.1 策略行为：丢弃队列中最早的任务，执行当前新任务

线程池满载时，主动丢弃**队列头部排队最久、最老旧的任务**，腾出队列空间，将当前最新的任务入队执行。

### 4.4.2 适用场景：允许丢弃旧任务、优先处理新任务的场景

适用于时效性极强的场景，旧任务无保留价值，例如：实时大屏数据刷新、实时热点统计、定时刷新缓存、设备状态上报等场景。

### 4.4.3 优缺点分析：可能导致旧任务丢失

**优点**：保证最新任务一定被执行，适配实时性业务，避免旧无效任务占用资源；

**缺点**：主动丢弃旧业务任务，若业务不兼容会导致数据错乱、流程缺失，通用性极差。

## 4.5 自定义拒绝策略

JDK四种内置拒绝策略均存在明显短板：默认策略抛异常、调用者策略降吞吐、丢弃策略丢数据、旧任务策略不通用。**生产环境一律推荐自定义拒绝策略**，实现日志记录、告警通知、任务降级、重试兜底。

### 4.5.1 实现RejectedExecutionHandler接口

所有拒绝策略的顶层接口为 **RejectedExecutionHandler**，仅需实现核心方法 `rejectedExecution(Runnable r, ThreadPoolExecutor executor)` 即可自定义兜底逻辑。

### 4.5.2 自定义策略的常见场景：任务日志记录、告警通知

生产自定义拒绝策略统一增强能力：

- 打印详细报错日志：记录线程池状态、队列容量、任务信息，便于问题排查；

- 触发告警推送：钉钉/企业微信告警，及时发现流量过载、线程池配置不足；

- 自定义降级逻辑：任务存入本地缓存、MQ重试、兜底执行；

- 友好异常返回：给前端返回“系统繁忙，请稍后重试”，替代原生异常堆栈。

### 4.5.3 自定义拒绝策略的代码示例

以下为**可直接生产落地的自定义拒绝策略完整代码**，包含日志打印、状态监控、友好提示，兼顾排查与用户体验：

```java
import java.util.concurrent.RejectedExecutionHandler;
import java.util.concurrent.ThreadPoolExecutor;

/**
 * 生产级自定义线程池拒绝策略
 * 核心能力：日志记录、状态监控、友好兜底、便于告警排查
 */
public class CustomRejectedHandler implements RejectedExecutionHandler {

    @Override
    public void rejectedExecution(Runnable runnable, ThreadPoolExecutor executor) {
        // 1. 打印详细监控日志，记录线程池满载状态
        System.err.println("【线程池过载拒绝】任务提交失败，线程池当前状态："
                + "核心线程数：" + executor.getCorePoolSize()
                + "，最大线程数：" + executor.getMaximumPoolSize()
                + "，活跃线程数：" + executor.getActiveCount()
                + "，队列已存任务数：" + executor.getQueue().size());

        // 2. 可拓展：接入告警工具，推送钉钉/邮件告警

        // 3. 抛出友好业务异常，替代原生粗暴异常
        throw new RuntimeException("系统当前访问繁忙，请稍后再试！");
    }
}

```

# 5. 线程池的5种状态：`RUNNING`/`SHUTDOWN`/`STOP`/`TIDYING`/`TERMINATED`

ThreadPoolExecutor 内部通过一个 **原子整型变量 ctl** 同时维护「线程池状态」和「工作线程数量」，高位存储线程池状态，低位存储线程数。线程池所有的任务提交、任务执行、关闭逻辑，都会先判断当前状态，再执行对应逻辑。线程池一共包含五种固定状态，状态单向流转、不可逆，覆盖线程池从启动到彻底销毁的完整生命周期。

## 5.1 线程池状态概述

### 5.1.1 状态的作用：控制线程池的生命周期

线程池状态是线程池的**全局行为控制器**，决定了线程池在当前阶段能否接收新任务、能否执行排队任务、能否中断工作线程、能否释放资源。所有线程池操作方法（提交任务、关闭线程池、查询状态）都依赖状态判断，核心作用如下：

- **管控任务准入**：不同状态下，线程池对新任务的接收规则不同；

- **管控任务执行**：控制正在执行、队列排队的任务是否继续执行、是否被中断；

- **管控资源回收**：控制工作线程销毁、资源释放、钩子方法执行；

- **保证流转有序**：状态单向流转，避免线程池关闭过程中出现任务错乱、资源泄漏等问题。

### 5.1.2 状态流转的触发条件

线程池状态不会自动变更，全部由**主动调用API方法**或**任务执行完毕、资源回收完成**触发，核心触发条件分为两类：

1. **人工主动触发**：调用 shutdown()、shutdownNow() 等关闭方法，主动修改线程池状态；

2. **内部自动触发**：队列任务消费完毕、工作线程全部退出、钩子方法执行完成，自动向下一状态流转。

核心特性：**状态只能由活跃状态向关闭状态单向流转，无法逆向恢复**，线程池一旦进入关闭流程，无法重新恢复运行状态。

## 5.2 各状态详解

### 5.2.1 `RUNNING`：运行状态，接受新任务、处理队列任务

**状态含义**：线程池正常运行的默认状态，线程池初始化完成后默认处于 RUNNING 状态。

**核心行为**：

- 允许接收外部提交的**新任务**；

- 持续消费队列中排队的任务；

- 正常进行线程复用、线程扩容、空闲线程回收等常规逻辑。

**场景**：项目正常运行、线程池正常工作的所有场景。

### 5.2.2 `SHUTDOWN`：关闭状态，不接受新任务，处理队列中已有任务

**状态含义**：平滑关闭中间状态，由调用 **shutdown()** 方法触发。

**核心行为**：

- **拒绝接收新任务**：此时提交新任务会触发拒绝策略；

- **不中断正在执行的任务**：正在运行的任务正常执行完毕；

- **继续处理队列任务**：会消费完阻塞队列中所有已排队的任务；

- 属于**优雅停机状态**，最大限度保证任务不丢失。

### 5.2.3 `STOP`：停止状态，不接受新任务，中断正在执行的任务

**状态含义**：强制关闭中间状态，由调用 **shutdownNow()** 方法触发。

**核心行为**：

- 拒绝接收所有新任务；

- **强制中断正在执行的任务**：通过线程中断标识终止运行中任务；

- **放弃队列剩余任务**：不再处理队列中未执行的任务，直接清空；

- 属于**暴力停机状态**，会导致未完成任务、排队任务丢失。

### 5.2.4 `TIDYING`：整理状态，所有任务已终止，等待钩子方法执行

**状态含义**：过渡终态，属于线程池彻底关闭前的最后整理阶段。

**触发时机**：当线程池处于 SHUTDOWN/STOP 状态，且**所有工作线程全部退出、队列任务全部处理/清空完毕**，自动进入 TIDYING 状态。

**核心行为**：

- 无任何任务在执行、无任务排队；

- 自动执行空钩子方法 **terminated()**，用户可重写该方法实现自定义收尾逻辑（资源释放、日志记录、回调通知）；

### 5.2.5 `TERMINATED`：终止状态，线程池完全关闭

**状态含义**：线程池**最终终止状态**，代表线程池彻底销毁、资源完全释放。

**触发时机**：TIDYING 状态下，钩子方法 terminated() 执行完毕后，自动流转为 TERMINATED。

**核心行为**：

- 所有线程已销毁、所有任务已结束、所有资源已释放；

- 线程池彻底失效，无法再处理任何任务，彻底完成生命周期。

## 5.3 状态流转过程

线程池状态为**单向不可逆流转**，不会出现反向恢复，以下为所有合法流转链路，也是面试核心考点。

### 5.3.1 `RUNNING` → `SHUTDOWN`：调用shutdown()方法

业务主动调用 `shutdown()` 平滑关闭方法，线程池从运行态进入平滑关闭态。

流转后特性：不接新任务、执行完运行中任务、消费完队列任务，是生产环境**推荐的优雅关闭方式**。

### 5.3.2 `RUNNING`/`SHUTDOWN` → `STOP`：调用shutdownNow()方法

无论线程池处于正常运行态还是平滑关闭态，只要主动调用 `shutdownNow()` 强制关闭方法，会立即流转为 STOP 状态。

流转后特性：强制中断运行任务、丢弃队列任务，适用于紧急停机、服务宕机场景。

### 5.3.3 `SHUTDOWN`/`STOP` → `TIDYING`：所有任务处理完毕

两种前置状态均可进入整理状态：

- SHUTDOWN 状态：队列任务全部消费、所有线程空闲退出 → 进入 TIDYING；

- STOP 状态：线程全部中断退出、队列任务清空 → 进入 TIDYING。

该流转为**内部自动触发**，无需人工干预。

### 5.3.4 `TIDYING` → `TERMINATED`：钩子方法terminated()执行完毕

TIDYING 状态下执行完收尾钩子方法后，线程池立即进入最终终止状态，生命周期彻底结束。用户可重写 terminated() 方法，在线程池彻底关闭前执行自定义收尾逻辑，例如关闭连接池、打印关闭日志、发送关闭通知等。

## 5.4 状态相关方法

### 5.4.1 `shutdown()`：平滑关闭线程池

**方法作用**：优雅关闭线程池，无任务丢失风险，生产环境优先使用。

**执行逻辑**：

1. 将线程池状态从 RUNNING 修改为 SHUTDOWN；

2. 停止接收新任务，新任务触发拒绝策略；

3. 已提交、队列中排队的任务继续执行；

4. 正在运行的任务正常执行完毕，不强制中断；

5. 所有任务执行完成后，线程自动回收，最终线程池终止。

**优缺点**：任务零丢失、业务安全，但关闭耗时较长，需要等待队列任务执行完毕。

### 5.4.2 `shutdownNow()`：强制关闭线程池

**方法作用**：暴力强制关闭线程池，快速停机，但会丢失未完成任务。

**执行逻辑**：

1. 将线程池状态立刻修改为 STOP；

2. 拒绝所有新任务提交；

3. 遍历所有工作线程，发起线程中断请求；

4. 清空任务队列，返回队列中未执行的任务集合；

5. 快速终止线程池，无需等待任务执行完毕。

**优缺点**：关闭速度极快，但会导致运行中任务中断、排队任务丢失，仅适用于紧急停机场景。

### 5.4.3 `isShutdown()`/`isTerminated()`/`awaitTermination()`

**1. isShutdown()**

判断线程池是否处于**非运行状态**，只要线程池进入 SHUTDOWN、STOP、TIDYING、TERMINATED 任意状态，均返回 true，仅 RUNNING 状态返回 false。常用于判断线程池是否已经发起关闭操作。

**2. isTerminated()**

精准判断线程池是否**完全终止**，仅线程池处于 TERMINATED 最终状态时返回 true，其余所有状态均返回 false。可用于校验线程池是否彻底关闭、资源是否完全释放。

**3. awaitTermination(long timeout, TimeUnit unit)**

**生产核心方法**：阻塞等待线程池关闭完成，设置超时时间。

执行逻辑：调用后阻塞当前线程，直到线程池彻底终止 或 等待超时，返回布尔值标识是否正常终止。

**生产最佳实践**：服务停机时，搭配 shutdown() + awaitTermination() 实现优雅停机，保证任务执行完毕再关闭服务，避免任务丢失。

**优雅关闭完整代码示例（可直接生产使用）**

```java
import java.util.concurrent.ThreadPoolExecutor;
import java.util.concurrent.TimeUnit;

/**
 * 线程池优雅关闭工具示例
 * 先平滑关闭，超时后强制关闭，最大限度保证任务不丢失
 */
public class ThreadPoolShutdownDemo {

    public static void shutdownGracefully(ThreadPoolExecutor threadPool) {
        // 1. 发起平滑关闭，不再接收新任务
        threadPool.shutdown();
        try {
            // 2. 阻塞等待30秒，等待已有任务执行完毕
            if (!threadPool.awaitTermination(30, TimeUnit.SECONDS)) {
                // 3. 超时未关闭，强制停机，防止线程卡死
                threadPool.shutdownNow();
                // 再次等待收尾
                if (!threadPool.awaitTermination(10, TimeUnit.SECONDS)) {
                    System.err.println("线程池强制关闭失败，存在卡死线程");
                }
            }
        } catch (InterruptedException e) {
            // 异常时强制关闭
            threadPool.shutdownNow();
            Thread.currentThread().interrupt();
        }
    }
}

```

# 6. `Executors` 工具类创建线程池的风险（`FixedThreadPool`/`CachedThreadPool`/`ScheduledThreadPool`/`SingleThreadExecutor`）

## 6.1 `Executors` 工具类概述

### 6.1.1 `Executors` 工具类的作用：简化线程池创建

**Executors** 是 JDK 提供的线程池工具类，核心设计目的是**简化线程池创建流程**，通过静态方法一键生成常用类型线程池，屏蔽底层 ThreadPoolExecutor 复杂的多参数构造方法，降低新手使用门槛。

该工具类封装了日常开发中最常用的四种线程池，开发者无需手动配置核心线程数、最大线程数、队列、拒绝策略等参数，开箱即用。但这种“简化封装”是以**牺牲可控性、隐藏线上风险**为代价，因此生产环境被严格禁用。

### 6.1.2 工具类创建的线程池底层实现

所有 Executors 创建的线程池，**底层全部基于 ThreadPoolExecutor 构造方法封装**，没有任何特殊实现，只是 JDK 帮我们预设了参数、队列、拒绝策略。

核心底层共性问题：Executors 为了适配通用场景、简化使用，大量使用**无界队列**和**无上限线程数**配置，完全舍弃了资源限制与过载保护机制，这也是所有线上风险的根源。

## 6.2 常见线程池及其风险

### 6.2.1 `FixedThreadPool`：定长线程池，使用无界队列导致OOM风险

**创建方式**：

```java
// 创建固定线程数的线程池
ExecutorService pool = Executors.newFixedThreadPool(5);

```

**底层核心参数**：

- 核心线程数 & 最大线程数：自定义固定值

- 空闲存活时间：0（无意义，无扩容线程）

- **任务队列：无界 LinkedBlockingQueue**

- 拒绝策略：默认 AbortPolicy

**核心风险**：**任务无限堆积，引发 OOM 内存溢出**

由于队列是无界队列，容量无限，永远不会触发队列已满的条件。结合线程池工作流程：队列永远不会满，线程池**永远不会创建非核心线程**，且**永远不会触发拒绝策略**。高并发瞬时大量任务涌入时，所有任务全部堆积在队列中，无限占用堆内存，最终导致内存溢出、服务宕机。

### 6.2.2 `CachedThreadPool`：缓存线程池，最大线程数无限制，可能创建大量线程

**创建方式**：

```java
// 创建可缓存线程池
ExecutorService pool = Executors.newCachedThreadPool();

```

**底层核心参数**：

- 核心线程数：0

- **最大线程数：Integer.MAX_VALUE（无限大）**

- 空闲存活时间：60秒

- 任务队列：SynchronousQueue（容量为0，不存储任务）

**核心风险**：**无限创建线程，耗尽服务器资源**

SynchronousQueue 不存储任务，每来一个任务必须有线程执行，无空闲线程则直接新建线程。由于最大线程数无上限，高并发下会瞬间创建成千上万条线程，导致：CPU 上下文切换爆满、线程栈内存溢出、操作系统线程资源耗尽、服务彻底雪崩。

### 6.2.3 `ScheduledThreadPool`：定时线程池，无界队列+核心线程数固定，任务堆积风险

**创建方式**：

```java
// 创建定时任务线程池
ScheduledExecutorService pool = Executors.newScheduledThreadPool(3);

```

**底层核心参数**：

- 核心线程数：自定义固定值

- 最大线程数：Integer.MAX_VALUE

- 任务队列：无界延时队列 DelayedWorkQueue

**核心风险**：**定时任务堆积、任务延迟严重、OOM风险**

队列无界，支持无限存储定时任务。若定时任务执行耗时过长、任务消费速度小于生产速度，会导致大量定时任务堆积在队列中，任务执行严重延迟，长期运行引发内存溢出，同时无限扩容线程，耗尽系统资源。

### 6.2.4 `SingleThreadExecutor`：单线程线程池，无界队列导致任务堆积OOM

**创建方式**：

```java
// 创建单线程线程池
ExecutorService pool = Executors.newSingleThreadExecutor();

```

**底层核心参数**：

- 核心线程数、最大线程数：1

- 任务队列：无界 LinkedBlockingQueue

**核心风险**：**单线程消费、无界队列堆积，高并发必OOM**

全程只有1个线程消费任务，处理速度极慢，无界队列可无限存储任务。高并发场景下任务堆积速度远超消费速度，队列任务无限膨胀，快速占满堆内存，引发OOM。同时单线程故障后无备用线程，任务全部卡死，可用性极低。

## 6.3 阿里Java开发规范的线程池创建要求

### 6.3.1 禁止使用Executors创建线程池的原因

结合上述四种线程池风险，阿里Java开发手册明确规定：**禁止使用 Executors 创建线程池，所有线程池必须手动通过 ThreadPoolExecutor 构造**，核心原因总结为两点：

1. **资源不可控**：Executors 封装的线程池要么队列无界、要么线程数无界，完全丧失资源限流能力，高并发极易引发OOM、线程溢出、服务雪崩；

2. **问题难排查**：默认线程无自定义名称、无完善拒绝策略、无日志告警，线上任务丢失、线程卡死、任务堆积问题无法快速定位。

### 6.3.2 生产环境推荐使用ThreadPoolExecutor自定义创建

生产唯一标准方案：**手动 new ThreadPoolExecutor 自定义线程池**，自主控制七大核心参数，实现资源可控、过载保护、问题可排查。

自定义核心优势：

- 强制使用**有界队列**，限制任务堆积上限，杜绝OOM；

- 限制**最大线程数**，保护服务器CPU、线程资源；

- 自定义线程工厂，统一线程命名，便于线上堆栈排查；

- 自定义拒绝策略，实现日志、告警、降级兜底，提升可用性。

### 6.3.3 线程池参数配置的最佳实践

结合生产调优经验，总结通用可落地的配置规则：

1. **队列必选有界**：优先使用 ArrayBlockingQueue，根据业务峰值设置合理容量，禁止无界队列；

2. **任务类型区分配置**

   - CPU密集型（计算多、IO少）：核心线程数 = CPU核心数 + 1，减少上下文切换；

   - IO密集型（接口、数据库、网络请求）：核心线程数 20~50，利用IO等待时间提升吞吐量；

3. **最大线程数合理扩容**：根据业务峰值设置，不盲目过大，防止线程泛滥；

4. **空闲时间统一配置**：5~10秒，快速回收峰值过后的闲置线程；

5. **拒绝策略自定义**：核心业务使用自定义告警拒绝策略，非核心业务可使用CallerRunsPolicy；

6. **必须自定义线程工厂**：业务线程命名，保障线上问题可追踪。

# 7. 自定义线程池的实现

前文我们掌握了原生线程池的参数、流程、风险，本节将**从零手写简易版线程池**，复刻 JDK ThreadPoolExecutor 核心逻辑，彻底吃透线程池底层原理，实现从“会用”到“懂底层、能落地”的跨越。

## 7.1 自定义线程池的核心设计

### 7.1.1 自定义线程池的核心组件：线程集合、任务队列、拒绝策略

参照原生线程池架构，自定义线程池包含三大核心组件，缺一不可：

1. **工作线程集合**：存储所有常驻工作线程，循环消费任务；

2. **阻塞任务队列**：存储等待执行的任务，实现任务排队、流量削峰；

3. **拒绝策略器**：队列和线程全部满载时的过载兜底组件。

### 7.1.2 核心参数定义：核心线程数、最大线程数、存活时间

复刻原生线程池核心参数，保证逻辑一致：

- corePoolSize：核心常驻线程数；

- maximumPoolSize：最大总线程数；

- keepAliveTime：非核心线程空闲存活时间；

- unit：时间单位。

### 7.1.3 核心工作流程实现

完全复刻原生四级执行流程：

1. 任务提交，判断核心线程数是否未满，未满则新建核心线程执行任务；

2. 核心线程已满，判断队列是否未满，未满则任务入队；

3. 队列已满，判断总线程数是否小于最大值，是则新建非核心线程；

4. 线程、队列全部满载，触发拒绝策略。

## 7.2 自定义线程池的实现步骤

### 7.2.1 定义线程池接口与核心参数

首先定义线程池统一接口，规范任务提交、关闭、拒绝策略方法，同时定义全局核心参数。

### 7.2.2 实现任务提交方法（execute()）

核心核心方法，严格按照四级流转逻辑实现，是整个线程池的核心调度入口。

### 7.2.3 实现工作线程类，循环从队列获取任务执行

自定义内部工作线程类，通过 while 死循环持续从阻塞队列获取任务，实现**线程复用**核心逻辑。

### 7.2.4 实现线程池状态管理与线程回收机制

定义运行、关闭两种基础状态，通过空闲超时判断，实现非核心线程自动回收。

### 7.2.5 实现拒绝策略接口与自定义拒绝策略

自定义拒绝策略顶层接口，实现默认抛出异常策略，可自由拓展多种兜底策略。

## 7.3 自定义线程池的完整代码示例

### 7.3.1 基础版自定义线程池实现

以下为**可直接运行的完整手写线程池代码**，复刻原生核心逻辑：

```java
import java.util.ArrayList;
import java.util.concurrent.ArrayBlockingQueue;
import java.util.concurrent.BlockingQueue;
import java.util.concurrent.TimeUnit;

/**
 * 自定义拒绝策略接口
 */
@FunctionalInterface
interface MyRejectHandler {
    void reject(Runnable task, MyThreadPool pool);
}

/**
 * 自定义线程池核心实现类
 * 复刻ThreadPoolExecutor核心逻辑：线程复用、队列排队、扩容、拒绝策略
 */
public class MyThreadPool {
    // 核心参数
    private final int corePoolSize;
    private final int maximumPoolSize;
    private final long keepAliveTime;
    private final TimeUnit unit;

    // 任务阻塞队列
    private final BlockingQueue<Runnable> taskQueue;

    // 工作线程集合
    private final ArrayList<Worker> workers = new ArrayList<>();

    // 拒绝策略
    private MyRejectHandler rejectHandler;

    // 线程池运行状态
    private volatile boolean isRunning = true;

    /**
     * 构造方法初始化线程池
     */
    public MyThreadPool(int corePoolSize, int maximumPoolSize, long keepAliveTime, TimeUnit unit,
                        int queueCapacity, MyRejectHandler rejectHandler) {
        this.corePoolSize = corePoolSize;
        this.maximumPoolSize = maximumPoolSize;
        this.keepAliveTime = keepAliveTime;
        this.unit = unit;
        this.taskQueue = new ArrayBlockingQueue<>(queueCapacity);
        this.rejectHandler = rejectHandler;
    }

    /**
     * 任务提交核心方法，复刻原生四级流程
     */
    public void execute(Runnable task) {
        if (!isRunning) {
            throw new RuntimeException("线程池已关闭，无法提交任务");
        }

        // 1. 核心线程数未满，新建核心线程执行任务
        if (workers.size() < corePoolSize) {
            createNewWorker(task);
            return;
        }

        // 2. 核心线程已满，队列未满，任务入队
        boolean offer = taskQueue.offer(task);
        if (offer) {
            return;
        }

        // 3. 队列已满，总线程数小于最大值，新建非核心线程
        if (workers.size() < maximumPoolSize) {
            createNewWorker(task);
            return;
        }

        // 4. 全部满载，触发拒绝策略
        rejectHandler.reject(task, this);
    }

    /**
     * 创建新工作线程并启动
     */
    private void createNewWorker(Runnable firstTask) {
        Worker worker = new Worker(firstTask);
        workers.add(worker);
        worker.start();
    }

    /**
     * 自定义工作线程类：实现线程复用、任务循环消费、空闲回收
     */
    private class Worker extends Thread {
        private Runnable firstTask;

        public Worker(Runnable firstTask) {
            this.firstTask = firstTask;
        }

        @Override
        public void run() {
            // 优先执行首次任务
            if (firstTask != null) {
                firstTask.run();
                firstTask = null;
            }

            // 循环从队列获取任务，实现线程复用
            while (isRunning) {
                Runnable task = null;
                try {
                    // 非核心线程：超时获取，超时无任务则退出回收
                    if (workers.size() > corePoolSize) {
                        task = taskQueue.poll(keepAliveTime, unit);
                    } else {
                        // 核心线程：永久阻塞常驻
                        task = taskQueue.take();
                    }
                } catch (InterruptedException e) {
                    break;
                }

                // 超时无任务，线程回收
                if (task == null) {
                    break;
                }

                // 执行队列任务
                task.run();
            }

            // 线程退出，从工作集合移除
            workers.remove(this);
        }
    }

    // 获取队列，供拒绝策略使用
    public BlockingQueue<Runnable> getTaskQueue() {
        return taskQueue;
    }
}

/**
 * 默认拒绝策略：抛出异常（对标AbortPolicy）
 */
class DefaultRejectHandler implements MyRejectHandler {
    @Override
    public void reject(Runnable task, MyThreadPool pool) {
        throw new RuntimeException("自定义线程池满载，任务拒绝，队列已满");
    }
}
```
