# 1. 线程安全的定义与判定标准

## 1.1 线程安全的核心定义

**线程安全**是指：在**多线程并发场景**下，多个线程同时访问、修改**共享资源**时，程序依然可以保持**正确的数据结果、稳定的业务逻辑、符合预期的执行状态**，不会出现数据脏读、数据覆盖、逻辑错乱、结果不一致等异常问题。

简单通俗理解：**多线程随便并发操作，程序结果都不会出错，就是线程安全；只要并发运行会出错，就是线程不安全**。

线程安全是并发编程的核心目标，几乎所有锁机制、JUC工具类、并发优化方案，本质都是为了解决线程不安全问题。

## 1.2 线程安全的判定标准：多线程执行结果与单线程结果一致

业界通用、面试标准的线程安全判定准则：

**在多线程并发无序执行的情况下，最终执行结果、程序状态，与单线程串行顺序执行的结果完全一致，则判定为线程安全；反之则为线程不安全。**

详细拆解判定维度：

- **数据一致性**：多线程读写共享变量，最终数据结果和单线程运算结果相同，无丢失、无覆盖、无脏数据；

- **逻辑正确性**：并发执行不会出现代码逻辑错乱、指令执行异常、程序状态混乱；

- **结果可复现**：无论并发多少次、线程调度顺序如何变化，最终结果始终符合预期，不会偶现bug。

**核心认知**：单线程运行永远不会出现线程安全问题，所有线程安全问题**只存在于多线程共享资源场景**。

## 1.3 线程安全问题的本质：多线程对共享资源的无序访问

所有线程不安全问题，归根结底只有一个核心本质：**多线程环境下，多个线程同时、无序、无保护地读写同一个共享资源**。

拆解三大必要产生条件（缺一不可）：

1. **多线程环境**：存在多个线程并发执行；

2. **共享资源**：多个线程操作同一个共享变量、对象、文件、数据库数据等资源；

3. **读写并发冲突**：至少有一个线程执行**写操作**，多线程同时读不会产生安全问题。

底层核心逻辑：单线程操作资源是有序串行执行；多线程无锁保护时，CPU抢占式调度、指令乱序执行、内存数据不同步，导致共享资源被无序覆盖、篡改，最终引发数据不一致。

**极简总结**：**共享+可变+并发写 = 线程不安全**；无共享、或只读、或串行执行，都是线程安全。

# 2. 可见性、原子性、有序性三大问题详解

Java并发编程中，所有线程安全问题，全部源于底层三大特性缺失：**可见性缺失、原子性缺失、有序性缺失**。这三大特性是JMM内存模型定义的并发基础特性，也是后续所有锁、关键字、并发工具的解决目标。

## 2.1 可见性问题

### 2.1.1 可见性的定义与JMM内存模型基础

**可见性定义**：一个线程对共享变量的修改，能够**实时被其他线程感知、读取到最新值**，这个特性就是可见性。

**JMM内存模型基础**：JMM（Java内存模型）规定，所有共享变量都存储在**主内存**中；每个线程拥有独立的**本地内存（工作内存）**。

线程读写共享变量的固定流程：

1. 线程读取变量：从主内存拷贝副本到本地内存，线程读取本地内存数据；

2. 线程修改变量：修改本地内存中的变量副本，并非直接修改主内存；

3. 刷新主内存：在特定时机，本地内存的最新值会刷新回主内存。

可见性问题的核心根源，就是**本地内存与主内存的数据不同步**。

### 2.1.2 可见性问题产生的原因：本地内存与主内存数据延迟

为了提升CPU执行效率，JMM采用了缓存优化机制，线程不会每次都读写低速的主内存，而是优先操作高速的本地工作内存。这种优化带来了性能提升，也带来了可见性隐患。

**可见性问题完整产生链路**：

1. 线程A、线程B同时加载主内存的共享变量到各自本地内存；

2. 线程A修改本地内存的变量值，但**未及时刷新回主内存**；

3. 线程B一直读取自己本地内存的旧副本数据，无法感知线程A的修改；

4. 最终出现：变量已被修改，但其他线程读取到旧值，数据不一致。

**核心结论**：普通变量没有强制刷新主内存的机制，多线程下必然存在**数据可见性延迟**。

### 2.1.3 可见性问题的典型场景与危害

**经典代码复现场景（循环等待退出失败）**：

```java
/**
 * 可见性问题复现案例
 * 现象：主线程修改flag为true，子线程永远感知不到，无限死循环
 */
public class VisibilityDemo {
    // 普通共享变量，无可见性保证
    private static boolean flag = false;

    public static void main(String[] args) throws InterruptedException {
        // 子线程：循环检测flag状态
        new Thread(() -> {
            // 死循环读取flag
            while (!flag) {
                // 空循环
            }
            System.out.println("子线程感知到flag变更，退出循环");
        }).start();

        // 主线程休眠2秒，保证子线程先启动
        Thread.sleep(2000);
        // 主线程修改共享变量
        flag = true;
        System.out.println("主线程已将flag修改为true");
    }
}

```

**运行现象**：主线程修改flag为true并打印日志，但子线程**永远不会退出循环**。

**原因解析**：子线程将flag变量缓存到本地工作内存，一直读取本地旧值false；主线程修改后刷新主内存，但子线程不会重新从主内存加载最新值，产生可见性问题。

**生产环境危害**：

- 程序死循环、线程卡死、后台任务无法终止；

- 配置热更新失效，新配置无法被业务线程感知；

- 状态位更新不及时，导致业务逻辑阻塞、超时、数据错乱。

## 2.2 原子性问题

### 2.2.1 原子性的定义：操作不可分割、不可中断

**原子性**是指：一个或一组操作，**要么全部执行成功，要么全部不执行，执行过程不可被其他线程中断、不可分割**。

原子性是数据库事务、并发编程的核心基础特性。在多线程并发场景中，只有具备原子性的操作，才能保证并发修改数据不被覆盖、不丢失。

**Java原生原子操作范围**：简单的赋值、读取操作是原子操作；**复合操作全部是非原子操作**。

### 2.2.2 非原子操作的典型场景（如i++）

绝大多数初学者踩坑的非原子操作：**i++、i--、i = i + 1** 这类自增运算，看似一行代码，底层拆解为**三步独立操作**，完全不具备原子性。

**i++底层三步指令**：

1. **读**：从主内存读取i的当前值到线程本地内存；

2. **改**：本地内存中执行i+1运算；

3. **写**：将运算后的新值刷新回主内存。

这三步操作可以被线程调度打断、穿插执行，多线程并发时必然出现数据覆盖。

**原子性问题完整复现代码**：

```java
/**
 * 原子性问题复现案例
 * 预期结果：10个线程各累加1000次，最终结果=10000
 * 实际结果：永远小于10000，出现数据丢失
 */
public class AtomicityDemo {
    // 共享计数器
    private static int count = 0;

    // 累加方法（非原子操作）
    public static void add() {
        count++;
    }

    public static void main(String[] args) throws InterruptedException {
        Thread[] threads = new Thread[10];
        // 创建10个线程
        for (int i = 0; i < 10; i++) {
            threads[i] = new Thread(() -> {
                // 每个线程累加1000次
                for (int j = 0; j < 1000; j++) {
                    add();
                }
            });
            threads[i].start();
        }

        // 等待所有线程执行完毕
        for (Thread thread : threads) {
            thread.join();
        }
        // 打印最终结果
        System.out.println("最终计数器结果：" + count);
    }
}

```

### 2.2.3 原子性问题导致的数据不一致风险

**出错原理解析**：

两个线程同时读取到同一个count旧值，同时执行+1运算，同时写入新值。原本两次累加应该+2，最终只+1，出现**数据覆盖、数据丢失**。并发量越大，丢失数据越多，误差越大。

**生产环境核心危害**：

- 计数器统计不准、订单数量统计错误、流量统计偏差；

- 库存超卖、数量扣减异常，引发业务事故；

- 积分、金额计算错乱，出现资金、数据不一致问题。

**核心总结**：所有**读取-修改-写入**的复合操作，都不具备原子性，多线程并发必然线程不安全。

## 2.3 有序性问题

### 2.3.1 有序性的定义：程序执行顺序与编写顺序一致

**有序性**是指：**代码的最终执行顺序，和开发者编写代码的顺序保持一致**，不会出现代码乱序执行。

单线程环境下，JVM、CPU会保证程序最终执行结果和代码顺序一致；但在多线程环境下，为了提升执行效率，JVM和CPU会对指令进行**重排序优化**，破坏代码有序性，引发并发逻辑错乱。

### 2.3.2 指令重排序的底层原因（JVM/CPU优化）

指令重排序是**编译器、CPU为了提升执行效率**，在保证**单线程执行结果不变**的前提下，打乱代码指令的执行顺序，并行执行无依赖的指令，提升吞吐量。

重排序分为三类：

1. **编译器重排序**：Javac编译器编译代码时优化指令顺序；

2. **CPU指令重排序**：CPU执行指令时，乱序执行、并行执行无依赖指令；

3. **内存重排序**：缓存、读写缓冲区导致的内存操作顺序打乱。

**重排序优化原则**：单线程下绝对安全、结果不变；**多线程共享变量场景下，会破坏业务逻辑**。

### 2.3.3 指令重排序导致的并发逻辑错乱问题

有序性问题是三大并发问题中**最隐蔽、最难复现、最难排查**的问题，属于偶现性bug，常规测试很难发现。

**经典场景：双重检查锁DCL单例失效（有序性问题经典案例）**

普通单例模式在指令重排序下，会出现**半初始化对象逃逸**问题：

对象创建正常顺序：1.分配内存空间 → 2.初始化对象成员变量 → 3.将引用指向内存地址

**重排序后乱序执行**：1.分配内存空间 → 3.引用指向内存地址 → 2.初始化对象

**并发危害**：其他线程检测到引用不为null，直接获取**未初始化完成的半空对象**，调用方法时报空指针、数据异常。

**生产环境危害**：

- 单例对象空指针异常、成员变量未初始化；

- 状态赋值乱序，导致业务判断逻辑颠倒；

- 偶现诡异bug，测试环境无法复现，线上随机触发。

# 3. 竞态条件与数据竞争的概念

在Java并发编程中，绝大多数线程安全bug，归根结底都由**数据竞争**和**竞态条件**引发。二者是面试高频混淆考点，也是生产环境并发问题排查的核心切入点：数据竞争是底层资源访问冲突，竞态条件是上层业务逻辑时序异常，二者相互关联但概念完全不同。

## 3.1 竞态条件（Race Condition）

### 3.1.1 竞态条件的定义：多线程执行时序依赖导致的结果异常

**竞态条件**是指：多线程并发执行业务逻辑时，**程序最终执行结果依赖于线程的调度时序、执行顺序**，不同的线程执行顺序会产生不同的业务结果，导致程序出现非预期的异常数据或逻辑错乱。

通俗理解：代码逻辑本身没有问题，**但多线程抢占式执行的随机性，打乱了业务执行时序**，导致偶现bug。这类问题最大的特点是**偶发、难复现、测试环境不易暴露**，仅在高并发线程切换时机巧合时触发。

核心本质：**业务操作不具备原子性，多线程穿插执行破坏了业务时序逻辑**。

### 3.1.2 竞态条件的核心场景：检查-执行、读取-修改-写入

生产环境中，竞态条件**99%只出现在两类复合操作场景**，也是排查并发bug的核心重点：

#### 1. 检查-执行 操作（Check-Then-Act）

先判断状态，再执行对应业务，两步操作不具备原子性，中间可被其他线程插入执行。

典型业务场景：单例判空创建、库存扣减判断、幂等状态校验。

问题链路：线程A判断资源空闲 → 未执行后续操作 → 线程B抢占CPU同样判断空闲并执行 → 最终出现重复执行、超量执行问题。

#### 2. 读取-修改-写入 操作（Read-Modify-Write）

先读取数据、再计算修改、最后写回数据，三步复合操作无法被原子执行。

典型场景：计数器累加、积分增减、库存扣减、状态更新。

问题链路：多线程同时读取同一个旧值，分别计算后覆盖写回，导致数据丢失、统计不准。

**竞态条件完整复现代码（库存超卖场景）**：

```java
/**
 * 竞态条件复现：库存超卖问题
 * 核心问题：检查库存、扣减库存两步操作非原子，触发竞态条件
 */
public class RaceConditionDemo {
    // 初始库存10
    private static int stock = 10;

    public static void main(String[] args) {
        // 20个线程并发抢购
        for (int i = 0; i < 20; i++) {
            new Thread(() -> {
                // 【检查-执行】经典竞态场景
                if (stock > 0) {
                    // 模拟业务耗时，放大线程穿插执行概率
                    try {
                        Thread.sleep(20);
                    } catch (InterruptedException e) {
                        e.printStackTrace();
                    }
                    stock--;
                    System.out.println(Thread.currentThread().getName() + " 抢购成功，剩余库存：" + stock);
                }
            }).start();
        }
    }
}

```

**运行结果**：初始库存10，最终剩余库存会出现**负数**，发生超卖，严重违背业务逻辑。

### 3.1.3 竞态条件与线程安全的关系

- **核心关系**：**存在竞态条件，一定线程不安全**；线程不安全的核心表现就是各类竞态条件引发的结果异常。

- 竞态条件是线程不安全的**上层现象**，是业务逻辑层面的并发问题；

- 消除竞态条件的核心方案：**保证复合操作的原子性、统一线程执行时序**，通过加锁、原子类、同步机制让多线程操作串行化。

特殊说明：单线程环境绝对不存在竞态条件，该问题**仅存在于多线程并发场景**。

## 3.2 数据竞争（Data Race）

### 3.2.1 数据竞争的定义：无同步机制下多线程同时读写共享变量

**数据竞争**是JMM定义的底层并发异常，官方标准定义：在**没有同步锁、volatile等任何同步机制保护**的前提下，多个线程同时访问同一个共享可变变量，且**至少有一个线程是写操作**，就会产生数据竞争。

数据竞争的成立三大必要条件：

1. 存在**共享可变变量**（成员变量、静态变量）；

2. **多线程并发访问**；

3. **至少一个线程执行写操作**（多线程只读不会产生数据竞争）。

### 3.2.2 数据竞争的危害：可见性、原子性问题的直接诱因

**所有的数据竞争，都会直接触发前文的三大并发问题**，是线程不安全的底层根源：

- **触发可见性问题**：无同步机制，线程本地内存与主内存数据不实时同步，不同线程读取到过期旧数据；

- **触发原子性问题**：多线程写操作互相覆盖，复合操作被中断，数据丢失、统计异常；

- **触发有序性问题**：无内存屏障保护，JVM、CPU指令重排序打乱执行逻辑，引发偶现逻辑bug。

**核心结论**：**数据竞争是因，线程不安全是果**。解决线程安全问题，本质就是消除共享变量的数据竞争。

### 3.2.3 竞态条件与数据竞争的区别与联系

这是面试高频对比考点，也是开发极易混淆的两个概念，下面通过**核心区别+关联关系+总结**彻底厘清：

| 对比维度       | 数据竞争（Data Race）                | 竞态条件（Race Condition）         |
| -------------- | ------------------------------------ | ---------------------------------- |
| 问题层级       | 底层内存访问问题（JVM层级）          | 上层业务时序问题（代码逻辑层级）   |
| 产生原因       | 无同步保护，多线程并发读写共享变量   | 复合操作非原子，执行时序不可控     |
| 必然出错？     | 一定存在数据风险                     | 不一定出错，随机偶现，依赖调度时机 |
| 解决思路       | 加同步、加volatile、消除共享可变变量 | 保证操作原子性、串行化执行         |
| **二者联系**： |                                      |                                    |

1. **数据竞争是竞态条件的前提**：绝大多数竞态条件，都是由数据竞争引发；

2. 存在数据竞争**大概率**出现竞态条件，最终导致线程安全问题；

3. 解决数据竞争，即可从根源杜绝绝大部分竞态条件问题。

# 4. 线程安全问题的常见场景

结合生产实战经验，汇总开发中**100%高频触发线程不安全的5类场景**，所有线上并发数据错乱、偶现bug均出自以下场景，开发中需重点规避。

## 4.1 成员变量/静态变量共享场景

**场景描述**：Spring Bean、自定义工具类中定义**成员变量、静态变量**，由于Spring Bean默认单例，多线程并发请求会共享同一个成员变量实例，产生数据竞争。

**问题根源**：单例对象的成员变量属于共享可变资源，多线程读写无同步保护。

**生产危害**：接口请求数据串参、用户数据错乱、状态值被覆盖、不同请求互相干扰。

**避坑方案**：

- 尽量避免在单例Bean中定义可变成员变量；

- 使用ThreadLocal存储线程私有变量，实现线程隔离；

- 必要时加锁或改为多例Bean（不推荐，损耗性能）。

## 4.2 计数器、累加器场景（如订单号、请求计数）

**场景描述**：接口请求次数统计、订单流水号自增、商品访问量统计、积分增减、库存扣减等需要**自增/自减**的业务场景。

**问题根源**：i++、i--属于**读取-修改-写入**非原子复合操作，多线程并发触发原子性缺失、竞态条件，数据丢失。

**生产危害**：统计数据不准、订单号重复、库存超卖、积分扣减异常。

**解决方案**：优先使用JUC原子类（AtomicInteger、AtomicLong），高并发场景使用LongAdder。

## 4.3 缓存数据更新场景

**场景描述**：本地缓存、Redis缓存的**查询后更新**场景，例如：读取缓存值 → 业务计算 → 重新写入缓存。

**问题根源**：查询和写入是两步非原子操作，多线程并发更新触发**检查-执行竞态条件**，后写入的数据覆盖先写入的数据。

**生产危害**：缓存数据更新丢失、缓存脏数据、业务状态更新不生效。

**解决方案**：加分布式锁、本地锁，或使用CAS乐观锁更新，保证更新操作原子性。

## 4.4 单例模式双重检查锁定（DCL）场景

**场景描述**：经典的双重检查锁单例模式，未加volatile修饰的写法。

**问题根源**：指令重排序导致有序性缺失，出现**半初始化对象逃逸**。线程可能获取到未初始化完成的单例对象，引发空指针、成员变量默认值异常。

**生产危害**：偶现空指针异常、单例对象属性错乱，线上极难复现排查。

**标准解决方案**：单例变量添加**volatile**关键字，禁止指令重排序，保证有序性。

## 4.5 容器并发读写场景（如List、Map）

**场景描述**：多线程并发读写普通ArrayList、HashMap、HashSet等非线程安全容器。

**问题根源**：原生集合类读写操作无同步机制，多线程并发写触发数据竞争、数组覆盖、链表结构错乱。

**生产危害**：

- ArrayList：并发抛出**ArrayIndexOutOfBoundsException**数组越界、数据丢失；

- HashMap：并发扩容死循环、数据覆盖、CPU飙升。

**解决方案**：并发场景使用线程安全容器：CopyOnWriteArrayList、ConcurrentHashMap、CopyOnWriteSet。

# 5. 线程安全的解决方案对比（synchronized/volatile/Lock/原子类）

Java解决线程安全问题的核心思路，就是针对性修复**可见性、原子性、有序性**三大问题。目前主流分为四大解决方案，各自底层机制、能力范围、性能、适用场景完全不同，面试高频考察区分度，也是生产选型的核心依据。

## 5.1 synchronized关键字

### 5.1.1 原理：通过锁机制保证原子性、可见性、有序性

**synchronized**是Java内置的**隐式重量级锁**，是JVM层面原生支持的同步机制，能够同时完美解决并发三大特性问题：

- **保证原子性**：同一时间只允许一个线程获取锁执行代码块，多线程并发操作变为串行执行，彻底杜绝操作被穿插、中断的问题；

- **保证可见性**：锁释放前会强制将线程本地内存的数据刷新到主内存，锁获取时会清空本地缓存、从主内存加载最新数据，线程间数据实时同步；

- **保证有序性**：通过内存屏障机制，禁止临界区内的指令重排序，保证代码执行顺序与编写顺序一致。

JDK1.6之后JVM对synchronized做了大量优化，引入**偏向锁、轻量级锁、重量级锁、锁膨胀、锁消除、锁粗化**机制，大幅降低锁开销，性能大幅提升，不再是传统认知中的低效锁。

### 5.1.2 适用场景：方法级、代码块级同步

synchronized支持两种粒度的加锁方式，适配不同业务场景：

##### 1. 修饰实例方法

锁对象为**当前实例对象this**，不同实例之间互不影响，适用于对象级别的并发同步。

##### 2. 修饰静态方法

锁对象为**当前类的Class对象**，全局唯一，所有实例共享同一把锁，适用于静态变量、全局资源同步。

##### 3. 修饰代码块

可自定义锁对象，粒度可控，支持精细化锁控制，**生产环境推荐优先使用**，缩小锁范围、提升并发性能。

**核心适用场景**：普通业务并发同步、复合操作原子性保障、临界资源争抢场景，是日常开发最通用的同步方案。

### 5.1.3 优缺点：自动释放锁、使用简单，但存在性能损耗

**优点**：

- 语法简洁、JVM原生支持，无需手动维护锁逻辑；

- **自动释放锁**：代码块执行完毕或线程异常终止时，JVM自动解锁，杜绝死锁风险；

- 一把锁同时保障原子性、可见性、有序性，能力全面；

- JDK1.6后优化充分，低竞争场景性能极高。

**缺点**：

- 锁机制不灵活，不支持超时锁、可中断锁、公平锁等高级特性；

- 高竞争场景下锁膨胀为重量级锁后，存在线程阻塞、上下文切换开销，性能下降；

- 无法精细化控制锁的获取与释放时机。

## 5.2 volatile关键字

### 5.2.1 原理：内存屏障保证可见性与禁止指令重排

**volatile**是JVM提供的**轻量级并发修饰符**，底层依靠**内存屏障**实现并发特性，核心能力非常单一：

- **保证可见性**：写volatile变量时，强制刷新本地内存到主内存；读volatile变量时，强制从主内存读取最新值，彻底解决缓存不一致问题；

- **保证有序性**：通过插入读写内存屏障，禁止volatile变量前后的指令重排序，避免乱序执行；

- **不保证原子性**：volatile仅针对单条读写指令生效，无法保护复合操作，不能解决原子性问题。

volatile是Java中**开销最小**的并发同步机制，无锁竞争、无线程阻塞、无上下文切换。

### 5.2.2 适用场景：状态标记、双重检查锁定（DCL）

volatile仅适用于两类特定场景，使用范围极其受限，不可乱用：

##### 1. 线程启停状态标记位

用于多线程感知状态变更，例如全局开关、任务终止标记，仅做**单次赋值、单次读取**操作。

##### 2. 双重检查锁（DCL）单例模式

禁止对象初始化过程的指令重排序，解决半初始化对象逃逸问题，是DCL单例的**必备修饰符**。

##### 3. 读写分离的简单变量

多线程只读、单线程写的场景，保证数据实时可见。

### 5.2.3 优缺点：轻量级、性能高，但不保证原子性

**优点**：

- 极致轻量，无锁开销、无线程阻塞、性能远超锁机制；

- 有效解决多线程数据可见性和指令重排序问题。

**缺点**：

- **核心短板：不保证原子性**，无法处理i++、数据累加等复合操作；

- 功能单一，无法解决线程安全的核心数据覆盖问题；

- 使用场景受限，绝大多数业务并发场景无法单独使用。

## 5.3 Lock接口（ReentrantLock）

### 5.3.1 原理：AQS框架实现的显式锁机制

**Lock**是JDK1.5引入的**显式锁接口**，核心实现类为ReentrantLock（可重入锁），底层基于**AQS（抽象队列同步器）**实现。

核心原理：通过AQS的**state状态值**记录锁持有状态，通过CLH双向队列存储阻塞线程，由Java代码层面实现锁竞争、等待、唤醒逻辑，属于**代码级锁**（synchronized是JVM级锁）。

ReentrantLock同样可以同时保障**原子性、可见性、有序性**，能力全覆盖，且扩展性更强。

### 5.3.2 适用场景：灵活控制锁的获取与释放、可中断锁、超时锁

相较于synchronized，Lock锁主打**灵活性与高级特性**，适配复杂并发场景：

- **可中断锁**：支持lockInterruptibly()，线程阻塞时可被中断，避免死锁僵持；

- **超时锁**：支持tryLock(time)，指定时间获取不到锁直接放弃，避免无限阻塞；

- **公平锁**：可配置为公平锁，保证线程先来先执行，避免线程饥饿；

- **锁绑定多个条件**：通过Condition精准唤醒指定线程，替代传统wait/notify；

- **高竞争复杂并发场景**：需要精细化控制锁逻辑的业务。

### 5.3.3 优缺点：功能更灵活，但需手动释放锁，使用复杂度高

**优点**：

- 功能强大，支持公平锁、超时锁、可中断锁、精准唤醒等高级特性；

- 锁粒度、锁时机完全可控，灵活性远超隐式锁；

- 高并发高竞争场景下性能优于synchronized。

**缺点**：

- **必须手动释放锁**，必须在finally代码块中解锁，否则100%出现死锁；

- 代码冗余、使用复杂，需要手动处理异常、锁释放逻辑；

- 低竞争场景下，性能略低于优化后的synchronized。

**标准代码模板（生产必用）**

```java
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantLock;

public class ReentrantLockDemo {
    // 创建可重入锁，true为公平锁，false为非公平锁（默认）
    private static final Lock lock = new ReentrantLock(true);
    private static int count = 0;

    public static void add() {
        // 手动加锁
        lock.lock();
        try {
            // 临界区业务逻辑
            count++;
        } finally {
            // 必须在finally中手动释放锁，防止异常导致死锁
            lock.unlock();
        }
    }
}

```

## 5.4 Atomic原子类

### 5.4.1 原理：CAS无锁机制实现原子操作

JUC原子类（AtomicInteger、AtomicLong等）底层基于**CAS（比较并交换）无锁机制**实现，不使用线程阻塞加锁，通过硬件指令级别的原子操作保障数据安全。

**CAS核心原理**：包含三个参数（内存地址V、旧预期值A、新值B）

1. 获取内存中当前变量值V；

2. 比较V是否等于线程预期旧值A；

3. 相等说明无其他线程修改，直接更新为新值B；不相等则重试，直至更新成功。

原子类**仅保证原子性与可见性**，无锁开销，是轻量级线程安全方案。

### 5.4.2 适用场景：简单的计数器、状态更新场景

专门适配**单个变量的简单读写、自增、状态更新**场景：

- 接口请求计数器、访问量统计；

- 订单序号、流水号自增；

- 布尔、数字类型的状态标记更新；

- 简单的数值累加、扣减场景。

**实操代码示例**

```java
import java.util.concurrent.atomic.AtomicInteger;

public class AtomicDemo {
    // 原子整型计数器，默认初始值0
    private static final AtomicInteger count = new AtomicInteger(0);

    public static void main(String[] args) throws InterruptedException {
        Thread[] threads = new Thread[10];
        for (int i = 0; i < 10; i++) {
            threads[i] = new Thread(() -> {
                for (int j = 0; j < 1000; j++) {
                    // 原子自增，线程安全
                    count.getAndIncrement();
                }
            });
            threads[i].start();
        }
        for (Thread thread : threads) {
            thread.join();
        }
        // 最终结果永远为10000，无数据丢失
        System.out.println("最终计数：" + count.get());
    }
}

```

### 5.4.3 优缺点：无锁、性能高，但仅支持单个变量的原子操作

**优点**：

- 无锁、无阻塞、无上下文切换，高并发性能极佳；

- 使用简单，无需手动加锁解锁；

- 完美解决数值自增、单变量更新的原子性问题。

**缺点**：

- **仅支持单个变量原子操作**，无法实现多变量、复合业务逻辑的原子性；

- 高并发竞争下CAS会频繁重试，消耗CPU资源；

- 存在ABA问题（可通过AtomicStampedReference解决）。

## 5.5 四大方案对比与选型建议

结合原理、性能、场景，整理生产环境**标准选型对照表**，覆盖面试答题与开发落地：

| 解决方案      | 保证特性                     | 底层机制              | 性能                 | 适用场景                                            |
| ------------- | ---------------------------- | --------------------- | -------------------- | --------------------------------------------------- |
| synchronized  | 原子性、可见性、有序性       | JVM隐式锁，锁膨胀机制 | 中等，低竞争性能优异 | 普通业务同步、复合操作、代码块/方法同步（通用首选） |
| volatile      | 可见性、有序性，**无原子性** | 内存屏障              | 极高（无锁）         | 状态标记、DCL单例、读写分离场景                     |
| ReentrantLock | 原子性、可见性、有序性       | AQS显式锁             | 高竞争下性能最优     | 高并发、需要超时/可中断/公平锁、精准线程唤醒        |
| Atomic原子类  | 原子性、可见性               | CAS无锁重试           | 极高                 | 单变量数值更新、计数器、状态修改                    |

**生产最终选型口诀**：

- 简单状态标记、只需可见性：用 **volatile**
- 单变量计数、数值自增：用 **Atomic原子类**
- 普通业务同步、追求简洁稳定：用 **synchronized**
- 高并发、需要高级锁特性：用 **ReentrantLock**

# 6. 线程安全与非线程安全的容器对比（ArrayList/Vector/HashMap/ConcurrentHashMap）

Java集合容器是并发场景高频出错点，原生容器大多非线程安全，并发读写会出现数据丢失、异常、死循环等问题。本节全面对比List、Map系列容器的线程安全特性、底层优劣与生产选型方案。

## 6.1 List容器对比

### 6.1.1 ArrayList：非线程安全，并发修改抛出ConcurrentModificationException

**核心特性**：ArrayList是基于动态数组实现的非线程安全容器，读写操作无任何同步机制。

**并发问题**：

- 多线程并发添加元素，会出现**数组覆盖、数据丢失**；

- 一边遍历、一边修改，直接抛出 **ConcurrentModificationException（并发修改异常）**；

- 并发扩容时会出现数组越界、元素丢失。

**适用场景**：单线程业务、局部变量、只读场景，**禁止用于多线程并发读写**。

### 6.1.2 Vector：线程安全，所有方法加synchronized锁，性能低下

**核心特性**：Vector是JDK早期提供的线程安全List，所有增删改查方法都被 **synchronized** 修饰。

**优点**：绝对线程安全，并发读写不会出现数据异常。

**致命缺点**：

- 锁粒度极大，所有方法串行执行，并发度为1，**性能极其低下**；

- 即使读操作也加锁，读多写少场景性能浪费严重；

- API老旧、扩展性差，**生产环境已彻底淘汰**。

### 6.1.3 并发List推荐方案：Collections.synchronizedList、CopyOnWriteArrayList

#### 1. Collections.synchronizedList

通过装饰器模式，给普通ArrayList所有方法加synchronized锁，实现线程安全。

特点：读写都加锁、性能一般，遍历依然需要手动加锁，适合**读写均衡、低并发**场景。

#### 2. CopyOnWriteArrayList（生产首选）

**核心原理**：写时复制，读无锁、写加锁。写操作会复制出新数组，修改新数组后替换原数组，读操作直接访问原数组，无锁并发。

**优点**：读操作完全并发、性能极高，不会抛出并发修改异常；

**缺点**：写操作需要复制数组，内存开销大、写性能偏低，存在数据弱一致性（读不到最新写入数据）；

**适用场景**：**读多写少、高并发查询**场景（生产环境并发List首选）。

## 6.2 Map容器对比

### 6.2.1 HashMap：非线程安全，并发场景可能出现死循环、数据丢失

**核心特性**：HashMap是线程不安全的键值对容器，JDK1.7数组+链表、JDK1.8数组+链表+红黑树。

**并发严重问题**：

- JDK1.7并发扩容会出现**链表环形死循环**，导致CPU 100%飙升；

- 多线程并发put会出现**数据覆盖、key丢失**；

- 并发读写大概率出现数据错乱、null值异常。

**强制规范**：**绝对禁止在多线程环境使用HashMap**。

### 6.2.2 Hashtable：线程安全，所有方法加synchronized锁，性能低下

Hashtable是早期线程安全Map，与Vector设计一致，所有方法添加全局synchronized锁。

**缺陷**：

- 全局锁粒度太粗，所有线程竞争同一把锁，并发度极低；

- 不允许key、value为null，使用限制多；

- 性能极差，完全无法适配高并发场景，**已彻底废弃**。

### 6.2.3 ConcurrentHashMap：线程安全，分段锁/无锁CAS实现，高并发性能优异

**ConcurrentHashMap**是目前**生产环境唯一推荐的高并发Map容器**，JDK不同版本优化差异极大：

#### JDK1.7：分段锁机制

将整个Map分为16个Segment分段，每个分段独立加锁，不同分段互不影响，大幅提升并发度。

#### JDK1.8：CAS+Synchronized细粒度锁

废弃分段锁，采用数组+链表+红黑树结构，**无锁CAS自旋**实现写入，冲突后对单个链表头节点加synchronized锁，锁粒度极致细化，并发性能拉满。

**核心优势**：

- 锁粒度极小，支持超高并发读写；

- 兼顾读写性能，无全局锁阻塞问题；

- 解决HashMap所有并发bug，线程绝对安全。

**小缺陷**：存在数据弱一致性，高并发下读取可能短暂获取旧数据（不影响业务）。

## 6.3 容器线程安全选型最佳实践

结合生产实战与阿里开发规范，整理**容器最终选型标准**，可直接落地使用：

1. **单线程场景**：一律使用 ArrayList、HashMap，性能最优；

2. **并发List、读多写少**：首选 **CopyOnWriteArrayList**；

3. **并发List、读写均衡/写多场景**：使用 Collections.synchronizedList 或自定义锁封装ArrayList；

4. **并发Map所有场景**：**唯一选择 ConcurrentHashMap**；

5. **永久禁用**：Vector、Hashtable，老旧低效容器，禁止出现在项目中。
