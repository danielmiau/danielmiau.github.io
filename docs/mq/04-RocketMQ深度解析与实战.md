本章是RocketMQ整套体系的**地基核心高权重章节**，也是面试、生产落地必考内容。RocketMQ作为阿里开源的面向互联网业务的高可靠、低延迟消息中间件，广泛应用于异步解耦、流量削峰、事务最终一致性、数据分发等场景。本章聚焦RocketMQ底层核心概念、基础架构组件、完整消息流转链路、集群元数据管理以及主流部署架构模式，从零拆解NameServer、Broker、Topic、Message Queue等核心要素的作用与底层逻辑，梳理消息从生产、存储到消费的全流程原理，同时对比各类部署架构的优缺点与适用场景。学完本章，可以彻底搞懂RocketMQ“架构如何跑、数据如何存、路由如何管、集群如何搭”，为后续消息类型、高可用、性能调优、事务消息等进阶内容打下坚实基础，具备独立识别、搭建基础RocketMQ集群的能力。

# 1. RocketMQ 核心概念：Broker、Topic、Message Queue、NameServer

RocketMQ的整体设计简洁、分层清晰，相较于Kafka更加贴合业务开发。想要掌握RocketMQ进阶特性，必须先吃透基础核心概念，区分**逻辑概念**与**物理存储概念**，规避开发中常见的认知误区。

## 1.1 核心组件详解

RocketMQ核心组件分为四大核心角色：NameServer、Broker、Topic、Message Queue，搭配Offset消费位点机制，共同完成消息的路由、存储、分发与消费。

### 1.1.1 NameServer：集群元数据管理中心，提供路由信息服务

**概念定义**：NameServer是RocketMQ的**轻量级注册中心与路由中心**，无状态、不持久化业务数据，仅维护集群元数据与路由关系，是整个集群的“导航中心”。

**核心特性**：

- **无状态、高可用**：NameServer之间互不通信、无需选举，支持单机或多节点集群部署，任意节点宕机不影响集群整体可用。

- **仅维护路由数据**：只存储Broker节点信息、Topic与Broker的映射关系，不存储消息数据、不处理消息收发请求，性能压力极低。

- **服务发现与路由分发**：接收Broker注册心跳，为生产者、消费者提供Topic路由查询服务。

**生产价值**：替代传统Zookeeper的繁重协调能力，轻量化架构降低集群运维成本，是RocketMQ架构简洁、高稳定的核心原因。

**面试重点**：NameServer不主动推送路由，由客户端定时拉取路由信息；无主从选举，集群容错性极强。

### 1.1.2 Broker：消息存储与转发节点，分为主节点（Master）和从节点（Slave）

**概念定义**：Broker是RocketMQ的**核心服务节点**，承担消息接收、存储、转发、查询、消费请求处理的全部工作，是集群中最核心、压力最大的组件。

**主从节点机制**：

- **Master主节点**：负责处理所有读写请求，接收生产者消息写入、响应消费者拉取请求，同时同步数据至从节点。

- **Slave从节点**：不处理写入请求，仅同步主节点数据，分担读请求压力，实现数据备份与高可用故障切换。

**Broker核心存储文件**：

- **CommitLog**：物理消息存储文件，所有Topic的消息统一顺序追加写入，是真正存储消息实体的文件。

- **ConsumeQueue**：消息消费索引文件，每个Topic每个队列对应一个消费队列，记录消息在CommitLog中的偏移，提升消费效率。

- **IndexFile**：消息索引文件，支持按消息Key、时间范围快速检索消息。

**避坑指南**：Broker是有状态节点，存储真实消息数据，宕机可能导致读写中断、数据丢失，生产必须配置主从架构。

### 1.1.3 Topic：消息的逻辑分类，生产者向 Topic 发送消息，消费者从 Topic 消费

**概念定义**：Topic是**逻辑消息主题**，用于对业务消息进行分类隔离，同一类业务、同一用途的消息统一发送至同一个Topic。

**核心特性**：

- 纯逻辑概念，不存储真实消息数据，仅作为消息分类标识。

- 一个Topic可以分布在多个Broker节点上，支持横向扩容。

- 支持多生产者发送、多消费者组订阅，业务隔离性极强。

**生产最佳实践**：严格按业务模块拆分Topic，订单、支付、库存、日志拆分独立Topic，禁止不同业务混用Topic，避免消息互相干扰、故障扩散。

### 1.1.4 Message Queue：Topic 的物理分片，每个 Queue 是一个有序的消息队列

**概念定义**：Message Queue（简称MQ队列）是Topic的**物理分片单元**，是RocketMQ消息读写、并发消费的最小粒度。

**核心原理**：

- 一个Topic默认包含多个Message Queue，队列数量决定Topic的**并发吞吐上限**。

- 单个Queue内部消息严格有序，不同Queue之间消息无序。

- 生产者发送消息时，通过负载均衡算法选择Queue写入；消费者按Queue维度分配消费，实现并发处理。

**面试高频考点**：RocketMQ的消息有序性，是**单队列有序、多队列无序**；需要全局有序必须设置Topic仅一个队列。

### 1.1.5 Offset：消息在 Queue 中的位置偏移量，消费者通过 Offset 记录消费进度

**概念定义**：Offset是消息在对应Message Queue中的**递增位置标识**，本质是消息的唯一位置下标，用于标记消费者的消费进度。

**两种Offset核心区分**：

- **Broker端Offset（最大偏移量）**：记录当前队列最新消息的位置，代表消息生产进度。

- **消费者Offset（消费位点）**：记录消费者当前消费到的位置，由消费者组提交、Broker持久化存储。

**核心作用**：通过两个Offset的差值，可精准计算**消息堆积量**；消费者重启后根据Offset继续消费，实现消息不丢失、不重复（配合消费模式）。

**生产避坑**：手动修改Offset需谨慎，位点前移会导致重复消费，位点后移会导致消息丢失。

## 1.2 消息模型与流转

RocketMQ采用「生产推送、消费拉取」的经典模型，整体流转链路清晰、解耦彻底，下面拆解消息从发送、存储到消费的完整全流程。

### 1.2.1 生产者发送流程：发送消息到 Topic 的 Message Queue

生产者发送消息的完整步骤，所有发送方式（同步、异步、单向）均遵循该底层流程：

1. **获取路由信息**：生产者启动后定时从NameServer拉取目标Topic的路由数据，获取Topic对应的Broker节点、队列列表。

2. **负载均衡选队列**：根据轮询、随机、Hash等策略，从可用队列中选择一个Message Queue。

3. **构建消息请求**：封装消息主题、消息体、标签、Key、队列信息，构建发送请求。

4. **发送至Broker**：通过Netty网络请求，将消息发送至对应Broker主节点。

5. **接收响应结果**：等待Broker写入成功响应，返回发送结果，异步模式通过回调处理结果。

**核心优势**：客户端自主选队列、自主路由，减轻Broker计算压力，大幅提升集群吞吐能力。

### 1.2.2 消息存储流程：消息追加写入 Broker 的 CommitLog 文件

Broker接收消息后，执行高性能落地存储流程，是RocketMQ高吞吐的核心底层：

1. **接收消息校验**：Broker校验消息权限、大小、格式合法性，过滤非法消息。

2. **CommitLog顺序追加**：所有Topic消息**统一顺序追加**写入CommitLog物理文件，磁盘顺序写速度极致高效。

3. **更新索引文件**：写入成功后，异步更新ConsumeQueue消费索引、IndexFile检索索引。

4. **主从数据同步**：Master节点将新消息同步至Slave节点，完成数据冗余备份。

5. **返回写入成功**：同步完成（根据配置）后返回生产者写入成功。

**原理亮点**：采用**顺序写、随机读**机制，规避磁盘随机IO的性能瓶颈，支撑百万级高吞吐。

### 1.2.3 消费者消费流程：按 Offset 拉取消息并处理

RocketMQ消费者采用**主动拉取模式**，相较于推送模式更可控、更稳定，适合海量消息消费：

1. **拉取路由信息**：消费者从NameServer获取订阅Topic的路由与队列信息。

2. **队列分配**：消费者组内通过负载均衡策略，将Topic的队列分配给各个消费者实例。

3. **主动拉取消息**：消费者根据自身存储的Offset，向Broker拉取对应位置的批量消息。

4. **业务逻辑处理**：本地执行消费业务逻辑，完成消息处理。

5. **提交消费Offset**：消费成功后，向Broker提交最新消费位点，更新消费进度。

**核心优势**：消费者自主控制拉取速度、批量大小，不会被Broker推送压垮，适配高低峰不同流量场景。

## 1.3 集群元数据管理

RocketMQ的集群元数据全部由NameServer统一管理，实现服务自动发现、路由动态更新、集群无感扩容，是集群高可用的基础。

### 1.3.1 Broker 注册：Broker 启动后向所有 NameServer 注册

Broker节点启动后，会自动遍历配置的所有NameServer节点，主动发起注册请求：

- 注册内容包含：Broker节点地址、主从角色、端口信息、所属集群、承载的Topic列表。

- Broker会**定时发送心跳包**（默认10s），维持注册状态。

- 若Broker宕机、心跳中断，NameServer超时后自动剔除该节点路由信息。

**高可用保障**：多NameServer部署时，Broker向所有节点注册，单个NameServer宕机不影响集群注册服务。

### 1.3.2 路由信息更新：NameServer 维护 Topic 与 Broker 的路由关系

NameServer内存中维护全局路由表，核心数据结构为：**Topic → Broker节点 → Message Queue列表**。

**路由更新场景**：

- 新Broker节点注册上线，自动更新路由表。

- Broker宕机下线，超时清理失效路由。

- Topic新增、删除、队列扩容，实时更新路由映射关系。

**核心特性**：NameServer仅维护路由，不参与消息流转，路由更新轻量化、无锁竞争，性能极高。

### 1.3.3 客户端路由查询：生产者/消费者从 NameServer 获取 Topic 路由信息

所有客户端（生产者、消费者）不会硬编码Broker地址，全部通过NameServer动态获取路由：

- 客户端启动时，主动拉取一次全量路由信息，缓存至本地。

- 客户端**定时拉取更新**路由（默认30s），感知集群节点上下线、队列变更。

- 路由查询失败时，使用本地缓存路由兜底，保证业务不中断。

**面试考点**：客户端路由为**主动拉取**，非服务端推送，大幅降低NameServer压力，是RocketMQ高并发设计的精髓。

# 2. RocketMQ 架构模型：生产者、消费者、Broker、NameServer

本节系统性梳理RocketMQ整体架构分层、各组件职责、完整消息流转链路以及主流部署架构，帮助建立全局架构认知，区分不同部署模式的生产适配场景。

## 2.1 架构组件角色与职责

RocketMQ整体架构分为四大核心角色，各司其职、完全解耦，形成完整的消息生产、存储、消费闭环。

### 2.1.1 生产者（Producer）：发送消息到 Broker，支持同步/异步发送

**角色定位**：消息数据的入口，负责业务消息的组装、发送，是消息链路的起点。

**核心能力**：

- **多种发送模式**：支持同步发送、异步回调发送、单向发送，适配不同业务场景。

- **负载均衡**：客户端自主选择Broker与队列，实现消息均匀分发。

- **容错重试**：发送失败自动重试，支持指定重试次数，保障消息可靠投递。

- **消息类型支持**：普通消息、有序消息、延时消息、事务消息统一适配。

**场景适配**：核心业务用同步发送保证可靠，高吞吐日志业务用异步发送提升性能。

### 2.1.2 消费者（Consumer）：从 Broker 拉取消息并处理，支持集群/广播消费

**角色定位**：消息数据的出口，负责拉取Broker存储的消息，执行业务消费逻辑。

**核心能力**：

- **两种消费模式**：集群消费（默认，负载均衡消费）、广播消费（所有实例全量消费）。

- **主动拉取机制**：自主控制消费节奏，避免消费过载、消息堆积雪崩。

- **位点持久化**：自动/手动提交Offset，保证消费进度不丢失。

- **重试机制**：消费异常自动进入重试队列，规避消息丢失。

**生产规范**：业务消费优先使用集群消费，需要全实例同步场景（配置更新、全局通知）使用广播消费。

### 2.1.3 Broker：存储消息、处理生产者/消费者请求，提供主从复制

**角色定位**：集群核心服务节点，承担所有消息存储、读写请求处理、数据备份、重试队列管理等核心工作。

**核心职责汇总**：

- 接收生产者消息写入请求，落地持久化存储。

- 响应消费者消息拉取、Offset提交请求。

- 主从节点数据同步，实现数据冗余备份。

- 管理重试队列、死信队列、延时队列。

- 持久化消费者Offset、集群权限、Topic配置。

**架构优势**：Broker职责高度集中，架构简单，无复杂选举逻辑，稳定性远超同类MQ。

### 2.1.4 NameServer：提供路由信息、服务发现，实现集群元数据管理

**角色定位**：集群的服务注册中心与路由调度中心，轻量化、无状态、高稳定。

**核心职责汇总**：

- 接收所有Broker节点的注册与心跳，维护集群存活节点列表。

- 维护Topic与Broker、队列的路由映射关系。

- 为生产者、消费者提供路由查询服务。

- 自动清理宕机节点的失效路由信息。

**架构亮点**：彻底去中心化，无中心节点瓶颈，集群扩容、运维成本极低。

## 2.2 消息流转流程

本节串联完整的「生产-存储-消费」全链路，拆解每一步底层逻辑，帮助彻底理解RocketMQ数据流转本质。

### 2.2.1 生产者发送流程：获取路由信息→选择 Broker→发送消息

完整标准化发送链路：

1. **路由预加载**：生产者初始化时，连接NameServer，拉取目标Topic的全部路由信息并本地缓存。

2. **筛选可用节点**：过滤宕机、不可用的Broker节点，保留健康节点与队列。

3. **负载均衡选队列**：默认轮询策略，均匀分发消息至不同队列与Broker，避免热点。

4. **Netty网络发送**：通过高性能Netty客户端，将消息发送至目标Master Broker。

5. **结果响应与重试**：接收Broker写入结果，失败则自动重试指定次数，保障可靠投递。

### 2.2.2 Broker 存储流程：写入 CommitLog→更新 ConsumeQueue/IndexFile

Broker接收消息后的三层存储写入流程，是RocketMQ高吞吐、可检索的核心：

1. **CommitLog物理写入**：消息主体、属性、时间戳等信息，**顺序追加**写入CommitLog文件，保证写入性能最大化。

2. **异步构建消费索引**：写入成功后，异步生成ConsumeQueue索引，记录消息在CommitLog中的偏移、长度、Tag哈希，供消费者快速拉取。

3. **异步构建检索索引**：根据消息Key、时间戳构建IndexFile索引，支持后台消息检索、问题排查。

4. **主从同步**：Master同步数据至Slave节点，完成数据备份。

5. **返回成功**：同步完成后响应生产者发送成功。

**性能优化原理**：物理文件顺序写+索引异步更新，兼顾写入速度与查询效率。

### 2.2.3 消费者消费流程：获取路由信息→拉取消息→提交 Offset

消费者标准消费链路，保证消费稳定、进度可追溯：

1. **订阅Topic并拉取路由**：消费者订阅指定Topic，从NameServer获取最新路由与队列信息。

2. **队列负载分配**：消费者组内均衡分配队列，一个队列同一时间仅被一个消费者消费，保证有序性。

3. **批量拉取消息**：消费者根据本地Offset，批量拉取未消费消息，减少网络IO次数。

4. **执行业务消费**：逐条或批量处理消息，完成业务逻辑。

5. **提交消费位点**：消费成功后提交Offset至Broker，更新消费进度；失败则触发重试机制。

## 2.3 部署架构模式

RocketMQ支持多种部署架构，从测试单机架构到生产高可用集群架构逐级递进，不同架构的可用性、可靠性、性能、运维成本差异极大，需根据业务场景选型。

### 2.3.1 单主模式：单个 Broker 节点，无备份，适合测试环境

**架构说明**：仅部署一台NameServer+单台Master Broker，无Slave从节点，无数据备份。

**优势**：部署简单、资源占用少、启动快速，零运维成本。

**劣势**：无高可用、无数据冗余，Broker宕机直接导致业务中断、数据丢失，不具备生产能力。

**适用场景**：本地开发、单元测试、功能调试、学习演练，**禁止用于生产环境**。

### 2.3.2 主从模式：一主一从/一主多从，实现数据备份与故障切换

**架构说明**：单Master节点+多Slave节点，Master负责读写，Slave同步数据、分担读请求。

**核心能力**：

- 数据实时同步备份，避免单节点数据丢失。

- 读请求负载均衡，提升集群消费吞吐量。

- 支持手动故障切换，Master宕机后可手动提升Slave为Master恢复服务。

**优缺点**：可靠性优于单主模式，部署简单；缺点是存在**单写瓶颈**，无法横向扩容写入能力，无自动故障切换。

**适用场景**：中小型非核心业务、流量较低的生产业务。

### 2.3.3 集群模式：多主多从集群部署，实现高可用与水平扩展

**架构说明**：部署多组主从节点，多台Master同时提供读写服务，所有节点共同组成集群，数据分散存储。

**核心优势**：

- **读写能力横向扩容**：多Master并行写入，彻底解决单写瓶颈。

- **集群高可用**：单组主从故障不影响整体集群可用。

- **负载均衡极致**：消息分散在多组节点，无热点节点。

**适用场景**：中大型互联网业务、流量波动大、需要持续扩容的生产场景。

### 2.3.4 双主双从模式：高可用生产环境推荐架构

**架构说明**：两组独立的一主一从架构组成集群，是**中小企业生产标准架构**。

**架构优势（生产黄金标准）**：

- **极致高可用**：任意一台Master、Slave宕机，集群不中断业务。

- **读写均衡**：双Master分担写入压力，双Slave分担读取压力。

- **数据零丢失**：每组节点均有备份，主从同步保障数据可靠。

- **运维成本适中**：相较于多主多从，部署简单、易维护，性价比最高。

**生产最佳实践**：绝大多数互联网公司的RocketMQ生产集群默认采用**双主双从+双NameServer**架构，兼顾高可用、高性能、低运维成本。

# 3. RocketMQ 消息类型：普通消息、顺序消息、事务消息、延迟消息

RocketMQ针对互联网复杂业务场景，设计了四种核心消息类型，分别适配通用业务、有序流转、分布式事务、定时延迟场景。不同消息类型的底层实现、发送规则、可靠性机制完全不同，业务开发中必须根据场景精准选型，避免出现消息错乱、数据不一致、业务异常等问题。

## 3.1 普通消息

普通消息是RocketMQ**最基础、最高吞吐、使用最广泛**的消息类型，无顺序、无事务、无延迟约束，核心优势是简单高效、性能极强，是绝大多数通用异步业务的首选。

### 3.1.1 普通消息的定义：无特殊顺序/事务要求的业务消息

普通消息指不强制要求消息消费顺序、无需保证消息与本地事务原子性、无需延迟投递的基础业务消息。该类型消息仅负责实现业务**异步解耦、流量削峰、数据分发**核心能力，底层采用随机队列分发、并行消费机制，最大化集群吞吐性能。

普通消息无额外业务约束，容错性高、开发成本低，是MQ最核心的基础能力，所有复杂消息类型均基于普通消息底层架构拓展实现。

### 3.1.2 发送方式：同步发送、异步发送、单向发送

RocketMQ普通消息提供三种发送方式，适配不同业务性能与可靠性诉求，以下附带**可直接运行的完整Java实操代码**，包含依赖、核心逻辑与注释。

**第一步：引入Maven依赖**

```xml
<!-- RocketMQ客户端依赖 -->
<dependency>
    <groupId>org.apache.rocketmq</groupId>
    <artifactId>rocketmq-client</artifactId>
    <version>4.9.5</version>
</dependency>
```

**第二步：三种发送方式完整代码示例**

```java
import org.apache.rocketmq.client.producer.DefaultMQProducer;
import org.apache.rocketmq.client.producer.SendCallback;
import org.apache.rocketmq.client.producer.SendResult;
import org.apache.rocketmq.common.message.Message;
import org.apache.rocketmq.remoting.exception.RemotingException;

/**
 * 普通消息三种发送方式实操示例
 */
public class CommonMessageProducer {
    public static void main(String[] args) throws Exception {
        // 1. 创建生产者实例，指定生产者组名
        DefaultMQProducer producer = new DefaultMQProducer("common-message-producer-group");
        // 指定NameServer集群地址
        producer.setNamesrvAddr("127.0.0.1:9876");
        // 启动生产者
        producer.start();

        // 构建消息对象：参数(主题,标签,消息内容)
        Message message = new Message("common_topic", "test_tag", "普通消息测试内容".getBytes());

        // ==================== 1. 同步发送 ====================
        // 特点：发送后同步阻塞等待Broker响应，可靠性最高
        SendResult syncResult = producer.send(message);
        System.out.println("同步发送结果：" + syncResult.getSendStatus());

        // ==================== 2. 异步发送 ====================
        // 特点：发送后立即返回，通过回调接收结果，吞吐高
        producer.send(message, new SendCallback() {
            @Override
            public void onSuccess(SendResult sendResult) {
                System.out.println("异步发送成功：" + sendResult.getSendStatus());
            }
            @Override
            public void onException(Throwable e) {
                System.err.println("异步发送失败：" + e.getMessage());
            }
        });

        // ==================== 3. 单向发送 ====================
        // 特点：只发送不等待响应、无回调，性能极致最高，不保证可靠
        producer.sendOneway(message);

        // 休眠保证异步回调执行
        Thread.sleep(1000);
        // 关闭生产者
        producer.shutdown();
    }
}
```

**三种发送方式核心对比与选型**

- **同步发送**：可靠最高、吞吐最低、阻塞等待响应，适用于支付、订单等核心可靠业务

- **异步发送**：吞吐高、可靠性中等、非阻塞，适用于高并发业务通知、数据同步

- **单向发送**：吞吐极致最高、无可靠性保障，适用于日志上报、监控埋点等非核心数据

### 3.1.3 适用场景：通用业务消息、通知推送、日志上报

普通消息无顺序与事务约束，主打**高性能、高并发、低成本**，核心适用场景如下：

- **通用业务异步解耦**：用户注册成功发送欢迎短信、积分发放、会员等级更新

- **消息通知推送**：系统公告、业务消息推送、站内信通知

- **海量日志上报**：操作日志、系统日志、监控日志收集与落地

- **流量削峰**：秒杀、活动报名等突发流量缓冲，削峰填谷保护后端服务

### 3.1.4 普通消息的可靠性保障：重试机制、消息持久化

普通消息虽为基础消息类型，RocketMQ提供完整可靠性机制，杜绝消息丢失：

- **消息持久化机制**：所有消息发送成功后，立即落地CommitLog磁盘文件，而非内存缓存，宕机不丢失

- **生产者重试机制**：同步/异步发送失败时，客户端自动重试（默认3次），可自定义重试次数

- **消费重试机制**：业务消费异常时，消息自动进入重试队列，多次重试失败后转入死信队列

- **主从副本备份**：消息同步至从节点，单节点故障可通过副本恢复数据

**避坑指南**：单向发送无重试机制，非核心业务才可使用，核心业务禁止使用单向发送。

## 3.2 顺序消息

普通消息默认多队列并行收发，消息无序。顺序消息是RocketMQ为**有状态、强流程顺序业务**设计的特殊消息类型，严格保证消息按照生产顺序完成消费。

### 3.2.1 顺序消息的定义：保证消息按生产顺序消费（全局有序/局部有序）

顺序消息核心定义：生产者按照业务流程顺序发送消息，消费者严格按照**发送顺序**处理消息，不会出现乱序、插队现象。

RocketMQ顺序消息分为两类，是面试高频考点：

- **局部有序（业务有序）**：同一业务维度（同一订单、同一用户）的消息有序，不同业务维度消息无序

- **全局有序**：整个Topic所有消息严格按照生产顺序消费，全局无乱序

**核心底层原理**：**单队列有序，多队列无序**。单个Message Queue内的消息严格FIFO有序，多个队列并行收发必然乱序，所有有序方案均基于该原理实现。

### 3.2.2 局部有序实现：同一业务 ID 的消息发送到同一个 Message Queue

局部有序是生产**最常用、性价比最高**的有序方案，兼顾有序性与并发性能。

**实现原理**：自定义消息队列选择器，将**同一业务ID**（订单ID、用户ID）的所有消息，固定发送到同一个Message Queue，保证该业务的消息单队列有序消费。不同业务ID的消息分发至不同队列，实现并行处理。

**局部有序完整代码示例**

```java
import org.apache.rocketmq.client.producer.DefaultMQProducer;
import org.apache.rocketmq.client.producer.MessageQueueSelector;
import org.apache.rocketmq.common.message.Message;
import org.apache.rocketmq.common.message.MessageQueue;
import java.util.List;

/**
 * 局部有序消息生产者：同一订单消息进入同一队列
 */
public class OrderMessageProducer {
    public static void main(String[] args) throws Exception {
        DefaultMQProducer producer = new DefaultMQProducer("order-producer-group");
        producer.setNamesrvAddr("127.0.0.1:9876");
        producer.start();

        // 模拟同一个订单的3条状态消息：创建、支付、发货
        String orderId = "ORDER_20260515001";
        String[] orderMessages = {"订单创建", "订单支付成功", "订单发货"};

        for (String msg : orderMessages) {
            Message message = new Message("order_topic", "order_tag", orderId, msg.getBytes());
            // 自定义队列选择器：根据orderId哈希固定队列
            producer.send(message, new MessageQueueSelector() {
                @Override
                public MessageQueue select(List<MessageQueue> mqList, Message msg, Object arg) {
                    // arg为业务ID：orderId
                    String businessId = (String) arg;
                    // 哈希取模，固定队列
                    int index = businessId.hashCode() % mqList.size();
                    return mqList.get(Math.abs(index));
                }
            }, orderId);
        }
        producer.shutdown();
    }
}
```

**消费端注意事项**：消费端需使用**顺序消费监听器**，单线程消费单个队列，禁止并发消费，否则仍会乱序。

### 3.2.3 全局有序实现：Topic 只设置一个 Message Queue，单线程消费

**实现原理**：将目标Topic的队列数量设置为**1**，所有消息全部进入唯一队列，配合消费端单线程串行消费，实现全局所有消息严格有序。

**配置方式**：创建Topic时，指定队列数量num=1，集群所有消息统一排队。

**核心特点**：全局绝对有序，但彻底丧失并发能力，所有消息串行处理。

### 3.2.4 顺序消息的适用场景：订单状态流转、数据同步

顺序消息适用于**有强流程依赖、状态不可逆**的业务场景：

- **订单状态流转**：订单创建 → 支付 → 发货 → 完成，必须顺序执行，禁止乱序

- **用户状态变更**：注册 → 实名认证 → 开通权限，状态流程不可逆

- **数据库数据同步**：增量binlog同步、数据新增/修改/删除顺序同步

- **流程审批业务**：多级审批流程流转，必须按顺序推进

### 3.2.5 顺序消息的性能影响：降低并发能力、吞吐量下降

顺序消息是**以性能换有序**，存在明显性能损耗，生产必须重点关注：

- **局部有序性能损耗**：单个业务维度串行消费，热点业务ID会出现消费瓶颈，无法扩容并发

- **全局有序性能损耗**：整个Topic单队列单线程消费，吞吐量极低，仅适配低流量有序业务

- **故障影响放大**：单条消息消费失败，会阻塞整个队列后续所有消息消费，直至故障恢复

**生产最佳实践**：优先使用局部有序，禁止滥用全局有序；有序业务尽量拆分Topic，隔离流量避免影响核心业务。

## 3.3 事务消息

事务消息是RocketMQ解决**分布式最终一致性**的核心能力，彻底解决「本地事务执行」与「消息发送」无法原子性的行业痛点，是面试与生产核心高频知识点。

### 3.3.1 事务消息的核心需求：消息发送与本地事务原子性

在分布式业务中，普遍存在「执行本地数据库事务 + 发送MQ消息」的场景，传统普通消息存在两大问题：

- 先发消息、后执行事务：事务失败回滚，但消息已发送，导致下游脏数据

- 先执行事务、后发消息：事务成功，但消息发送失败，导致下游业务未执行，数据不一致

**事务消息核心价值**：保证**本地数据库事务执行**与**MQ消息发送**具备最终原子性，要么同时成功，要么同时失败，实现分布式事务最终一致。

### 3.3.2 半消息机制：生产者发送半消息→执行本地事务→提交/回滚

半消息（Half Message）是事务消息的核心底层机制，区别于普通消息：

- **半消息定义**：生产者发送的临时消息，Broker接收后**不会投递到消费者**，对下游消费者不可见

- **执行流程**：生产者发送半消息成功 → 执行本地数据库事务 → 根据事务结果向Broker发送提交/回滚指令

- **提交**：Broker将半消息转为正式消息，投递至消费者

- **回滚**：Broker删除半消息，下游永远接收不到该消息

### 3.3.3 事务消息流程：半消息发送→事务执行→事务状态回查→消息投递

RocketMQ事务消息完整闭环流程（四大步骤，面试必背）：

1. **发送半消息**：生产者向Broker发送半事务消息，Broker持久化后响应成功，消息暂不投递

2. **执行本地事务**：生产者收到半消息响应后，立即执行本地数据库事务（新增订单、扣库存等）

3. **提交事务状态**：
            

   - 事务成功：发送Commit指令，Broker投递消息给消费者

   - 事务失败：发送Rollback指令，Broker删除半消息

   - 状态未知：网络超时、服务重启，Broker触发事务回查

4. **事务状态回查**：Broker长时间未收到事务指令，主动回调生产者接口，查询本地事务状态，根据状态完成提交/回滚，保证最终一致

**核心特性**：通过回查机制解决超时、重启、网络异常问题，实现分布式事务最终一致性，无需锁、无需强一致，性能远高于XA事务。

### 3.3.4 事务消息的适用场景：订单创建+扣减库存、支付+通知

事务消息主打**高并发、最终一致性**分布式业务场景：

- **订单业务**：创建订单（本地事务）+ 发送订单消息（下游扣库存、生成物流单）

- **支付业务**：支付扣款成功（本地事务）+ 发送支付通知、积分发放消息

- **库存同步**：本地库存扣减 + 远程库存数据同步消息

- **营销活动**：用户中奖记录入库 + 发放奖品、优惠券消息

### 3.3.5 事务消息的可靠性保障：事务状态回查、超时事务处理

RocketMQ通过双重机制保障事务消息零数据不一致：

- **事务状态回查机制**：默认1分钟开始回查，多次重试回查，直至获取事务最终状态

- **超时事务处理**：长期无法确认状态的事务消息，自动标记为异常，人工兜底处理

- **消息持久化**：半消息全程落地磁盘，服务重启不丢失事务状态

**避坑指南**：事务回查接口必须**幂等**，避免多次回查导致本地事务重复执行；本地事务必须可查询状态，否则回查失效。

## 3.4 延迟消息

延迟消息是RocketMQ实现**轻量级定时任务**的核心能力，无需依赖Quartz、XXL-Job等定时框架，即可实现消息延迟投递，适配各类超时、延时业务场景。

### 3.4.1 延迟消息的定义：消息发送后延迟指定时间再投递

延迟消息指生产者发送消息后，Broker不会立即将消息投递到消费者，而是等待指定时长后，再将消息分发至消费者进行业务处理，核心用于实现延时触发的业务逻辑。

延迟消息本质是**时间驱动的特殊消息**，投递时机由服务端控制，而非客户端定时触发。

### 3.4.2 延迟级别：支持 18 个预设延迟级别（1s~2h）

RocketMQ**不支持自定义任意延迟时间**，仅支持系统预设的18个延迟级别，生产常用级别如下：

```java
// 延迟级别对应时间
1s、5s、10s、30s、
1m、2m、3m、4m、5m、6m、7m、8m、9m、10m、20m、30m、
1h、2h
```

**代码设置方式**：通过消息对象设置延迟级别，示例如下：

```java
// 设置延迟级别为5：对应5分钟延迟
message.setDelayTimeLevel(5);
```

### 3.4.3 延迟消息实现：消息写入延迟队列，到期后转发到目标 Topic

延迟消息底层实现原理：

1. 生产者发送带延迟级别的消息，Broker接收后**不存入目标Topic**

2. 消息统一存入系统内置的**延迟队列（SCHEDULE_TOPIC_XXXX）**

3. Broker后台定时线程轮询延迟队列，判断消息是否到期

4. 消息到期后，从延迟队列移除，转发至业务目标Topic

5. 消费者正常消费到期消息，完成延时业务逻辑

**核心优势**：服务端统一调度，客户端无压力，支持海量延时消息调度，性能远超定时任务框架。

### 3.4.4 延迟消息的适用场景：订单超时关闭、定时任务、重试延迟

延迟消息主打**短时延时、高并发定时**场景：

- **订单超时关闭**：用户下单后15分钟未支付，自动关闭订单、释放库存

- **业务延时重试**：接口调用失败、业务处理异常，延时重试处理

- **定时通知推送**：活动开始前提醒、到期续费提醒

- **异步延时校验**：用户收货后7天自动确认收货、结算佣金

### 3.4.5 延迟消息的局限性：不支持自定义延迟时间，仅支持预设级别

生产使用必须熟知延迟消息的核心局限性，避免业务踩坑：

- **无自定义时间**：仅支持18个预设级别，无法设置13分钟、2.5小时等自定义时间

- **高精度误差**：秒级延迟存在少量时间误差，无法用于高精度定时业务

- **不支持大范围时长**：最大仅支持2小时延迟，超长定时任务需使用专业定时框架

**解决方案**：如需自定义时间，可通过「多级延迟消息叠加」或「业务层时间校验兜底」实现。

# 4. RocketMQ 高可用：主从复制、多副本、刷盘策略

高可用是RocketMQ支撑生产集群7*24小时稳定运行的核心保障，核心解决两大问题：**服务故障不中断、数据故障不丢失**。本章从主从复制、多副本机制、刷盘策略三个底层核心，拆解RocketMQ高可用原理、性能权衡与生产落地架构。

## 4.1 主从复制机制

RocketMQ通过Master-Slave主从架构实现集群高可用，区分读写节点、实现数据备份与故障容错，是生产集群的基础架构。

### 4.1.1 主从架构：Master 节点可读写，Slave 节点只读，主从同步

标准主从架构角色分工明确，职责隔离：

- **Master主节点**：全权处理**读写所有请求**，负责消息写入、消费响应、Offset管理、事务处理，是集群核心写入节点

- **Slave从节点**：**不处理写入请求**，仅同步Master数据，分担读请求压力，作为数据备份节点

核心运行机制：Master实时产生数据，Slave主动拉取同步数据，最终实现主从数据一致。

### 4.1.2 同步复制 vs 异步复制：同步复制等待 Slave 写入成功，异步复制不等待

主从数据同步分为两种模式，是生产可靠性与性能调优的核心参数，二者取舍为面试高频考点：

**异步复制（ASYNC_MASTER_SLAVE）**

- 核心逻辑：Master写入本地磁盘成功后，立即响应生产者成功，后台异步同步数据至Slave

- 优势：无等待耗时，性能极高，吞吐量大

- 劣势：Master写入成功、Slave未同步时，Master宕机，存在少量数据丢失风险

- 适用场景：日志、监控、非核心高吞吐业务

**同步复制（SYNC_MASTER_SLAVE）**

- 核心逻辑：Master写入本地成功后，**必须等待Slave节点写入成功**，才响应生产者

- 优势：主从数据实时一致，Master宕机无数据丢失，可靠性最高

- 劣势：多一次网络与磁盘IO等待，吞吐量下降、延迟升高

- 适用场景：支付、订单、金融等核心零丢失业务

### 4.1.3 主从切换：Master 节点故障时，Slave 节点可提升为新 Master

RocketMQ支持手动主从切换，部分高版本支持自动切换，保障服务不中断：

- **故障场景**：Master节点宕机、磁盘损坏、网络中断，无法提供读写服务

- **切换逻辑**：运维手动将同步完成的Slave节点提升为新Master，接管读写请求

- **恢复逻辑**：旧Master修复重启后，自动降级为Slave，加入集群同步数据

**生产注意**：开源版RocketMQ默认**不支持自动主从切换**，需手动运维，商业版支持自动故障转移。

### 4.1.4 主从复制的性能与可靠性权衡：同步复制可靠性高但性能低

生产选型核心权衡原则：

- **优先异步复制**：90%通用业务，追求高吞吐、低延迟，可容忍极小概率数据丢失

- **强制同步复制**：金融、交易、支付核心业务，优先保障数据零丢失，牺牲部分性能

## 4.2 多副本机制

多副本机制是RocketMQ数据可靠的核心兜底方案，通过多节点冗余存储，彻底解决单节点磁盘故障、数据丢失问题。

### 4.2.1 副本的作用：数据备份、故障恢复、读写分离

副本的三大核心生产价值：

- **数据冗余备份**：单节点磁盘损坏、数据丢失，可通过副本恢复完整数据

- **故障快速恢复**：主节点故障，副本节点可快速接管服务，无需人工恢复数据

- **读写分离扩容**：读请求分散至多个副本节点，分担读取压力，提升消费吞吐

### 4.2.2 副本配置：设置 Topic 的副本数量，实现多副本存储

创建Topic时可指定**副本数量**，控制数据冗余份数：

- 副本数量=1：无备份，仅主节点存储，测试环境使用

- 副本数量=2：一主一从双副本，生产最低标配

- 副本数量=3：一主多从三副本，金融级高可靠架构

**配置命令示例**

```shell
# 创建topic，指定队列数8，副本数2
sh mqadmin updateTopic -n 127.0.0.1:9876 -t test_topic -c DefaultCluster -q 8 -r 2
```

### 4.2.3 副本同步：Master 节点消息同步到所有 Slave 副本

Master节点写入消息后，会根据集群复制模式，将数据同步至**所有副本Slave节点**，保证全集群副本数据一致。多副本同步遵循最终一致性，同步完成前副本数据略滞后于主节点。

### 4.2.4 多副本对高可用的保障：单副本故障不影响服务可用

多副本机制的核心高可用能力：

- 集群中**任意单个副本节点故障**，集群读写服务完全不受影响

- 故障节点重启后，自动同步缺失数据，补齐副本，恢复冗余能力

- 多副本配合主从切换，实现服务高可用+数据零丢失双重保障

**生产最佳实践**：生产环境副本数**至少2**，核心业务建议3副本，禁止单副本运行。

## 4.3 刷盘策略

刷盘策略决定消息从内存PageCache落地到磁盘的时机，是**单机数据可靠性与性能的核心平衡点**，直接影响消息是否丢失、集群吞吐高低。

### 4.3.1 同步刷盘（SYNC_FLUSH）：消息写入后立即刷盘，可靠性高但性能低

**核心原理**：消息写入系统PageCache内存后，**立即强制同步刷入物理磁盘**，刷盘完成后才响应生产者发送成功。

- **优点**：消息100%落地磁盘，服务宕机、断电无数据丢失，可靠性顶级

- **缺点**：频繁磁盘IO，吞吐大幅下降，延迟升高，性能损耗极大

- **适用场景**：支付、结算、金融核心业务，零数据丢失要求

### 4.3.2 异步刷盘（ASYNC_FLUSH）：消息写入 PageCache 后返回，后台异步刷盘，性能高但存在数据丢失风险

**核心原理**：消息写入内存PageCache后，**立即响应成功**，由后台异步线程批量将内存数据刷入磁盘。

- **优点**：规避频繁磁盘IO，吞吐极高、延迟极低，适配高并发场景

- **缺点**：内存数据未刷盘时，机器断电、系统崩溃，未落地消息永久丢失

- **适用场景**：日志、通知、统计数据等可容忍少量数据丢失的业务

### 4.3.3 刷盘策略的适用场景：金融级业务优先同步刷盘，高吞吐业务优先异步刷盘

生产标准化选型规范：

- **同步刷盘+同步主从复制**：金融、交易、支付核心业务，极致可靠

- **异步刷盘+异步主从复制**：日志、监控、推送通知，极致性能

- **异步刷盘+同步主从复制**：绝大多数普通业务，兼顾性能与可靠性（生产最常用）

### 4.3.4 刷盘参数配置：`flushDiskType`、`flushInterval`、`flushCommitLogLeastPages`

Broker核心刷盘配置参数（conf/broker.conf），可直接生产落地：

```properties
# 刷盘类型：SYNC_FLUSH同步刷盘、ASYNC_FLUSH异步刷盘
flushDiskType=ASYNC_FLUSH
# 异步刷盘最大间隔时间（毫秒）
flushInterval=500
# 异步刷盘最小落盘页数，达到阈值强制刷盘
flushCommitLogLeastPages=4
# 同步刷盘等待超时时间
syncFlushTimeout=5000
```

**参数作用**：通过时间阈值+页数阈值双重控制，平衡异步刷盘的性能与数据安全。

## 4.4 高可用架构实践

结合前文主从复制、多副本、刷盘机制，整合生产可直接落地的高可用架构方案，实现稳定性与性能的最优平衡。

### 4.4.1 生产环境推荐架构：多主多从集群 + 同步/异步刷盘

**中小企业标准生产架构**：双主双从 + 双NameServer + 异步刷盘 + 同步主从复制

- 双主节点分担写入压力，解决单主写入瓶颈

- 双从节点提供数据备份与读负载分担

- 异步刷盘保证高吞吐，同步主从复制兜底数据安全

- 双NameServer保证路由服务高可用

**金融级架构**：多主多从三副本 + 同步刷盘 + 同步主从复制，极致数据可靠

### 4.4.2 故障演练：Master 节点故障、网络分区场景验证

生产上线前必须完成的高可用故障演练，验证集群容错能力：

- **Master节点宕机演练**：手动停止主节点进程，验证从节点数据完整性、服务是否中断

- **网络分区演练**：断开节点网络，验证集群路由剔除、故障自愈能力

- **磁盘IO打满演练**：模拟磁盘瓶颈，验证集群是否卡顿、消息是否丢失

### 4.4.3 高可用与性能的权衡：副本数量、刷盘策略、复制模式的组合配置

生产核心权衡公式（落地必看）：

- **高吞吐优先**：异步刷盘 + 异步主从复制 + 2副本 + 多主部署

- **高可靠优先**：同步刷盘 + 同步主从复制 + 3副本 + 主从热备

- **均衡最优（通用）**：异步刷盘 + 同步主从复制 + 2副本 + 双主双从

# 5. RocketMQ 消费者模式：集群消费、广播消费、负载均衡

RocketMQ消费者核心分为两大消费模式：集群消费、广播消费，配合客户端负载均衡机制实现消息的分布式消费。绝大多数线上消费异常、堆积、重复消费、扩容不生效问题，本质都源于对消费模式与重平衡机制理解不到位。

## 5.1 集群消费模式

集群消费是RocketMQ**默认、最常用、生产首选**的消费模式，主打分布式负载均衡、高并发消费、水平扩容，适配绝大多数业务消息场景。

### 5.1.1 集群消费的定义：同一个消费者组内的消费者共同消费 Topic 消息

**集群消费核心定义**：同一消费者组（ConsumerGroup）下的多个消费者实例，**共同分担消费同一个Topic的消息**，同一条消息只会被组内**一个实例消费一次**，天然实现消息负载均衡与分布式并行消费。

核心约束：**不同消费者组相互独立**，不同组可以重复消费同一个Topic的所有消息，同组唯一消费。

### 5.1.2 消费规则：一个 Message Queue 只能被组内一个消费者消费

集群消费的底层核心规则（面试高频必考）：

- 一个Topic包含多个Message Queue，队列是消费负载的**最小分配单元**，不可拆分

- 在同一个消费者组中，**一个Message Queue同一时间只能被一个消费者实例消费**

- 一个消费者实例可以同时消费多个Message Queue

- 该规则直接决定消费并发上限：**消费最大并发数 = Topic队列数量**，消费者实例数超过队列数后，多余实例会空闲无法消费

**生产避坑**：想要提升消费并发，优先扩容Topic队列数，单纯加消费者节点无法提升吞吐。

### 5.1.3 负载均衡：Message Queue 均匀分配给组内消费者

集群消费模式下，消费者启动后会自动触发**客户端负载均衡（Rebalance）**，将当前Topic的所有Message Queue均匀分配给组内所有在线消费者实例。

分配特点：

- 分配均匀、尽量平均，避免单实例消费压力过大

- 实例数量变化、队列数量变化时，自动重新分配队列归属

- 负载均衡完全由**客户端自主完成**，Broker不参与分配计算，性能极高

### 5.1.4 适用场景：业务消息处理、高并发消费场景

集群消费是互联网业务标准消费模式，适配所有需要**高并发、分布式处理、可扩容**的业务场景：

- 订单、支付、库存等核心业务消息异步处理

- 海量日志、埋点数据异步消费落地

- 流量削峰场景的大批量消息消费

- 需要支持服务水平扩容的所有MQ消费业务

### 5.1.5 Offset 提交：集群消费模式下 Offset 存储在 Broker

集群消费模式的核心特性：**消费位点Offset持久化在Broker服务端**，而非本地机器。

核心优势与原理：

- 消费者实例重启、扩容、下线后，Broker保存的消费进度不丢失

- 新上线消费者实例通过重平衡获取队列，从Broker拉取最新Offset继续消费，保证进度连续

- 同组所有消费者共享服务端Offset，保证全局消费进度统一

**实操代码（集群消费配置）**

```java
import org.apache.rocketmq.client.consumer.DefaultMQPushConsumer;
import org.apache.rocketmq.client.consumer.listener.ConsumeConcurrentlyContext;
import org.apache.rocketmq.client.consumer.listener.ConsumeConcurrentlyStatus;
import org.apache.rocketmq.client.consumer.listener.MessageListenerConcurrently;
import org.apache.rocketmq.common.message.MessageExt;
import java.util.List;

/**
 * 集群消费模式消费者（默认模式）
 * 特点：同组负载均衡、消息唯一消费、Offset存在Broker
 */
public class ClusterConsumer {
    public static void main(String[] args) throws Exception {
        // 1. 创建消费者，指定消费组
        DefaultMQPushConsumer consumer = new DefaultMQPushConsumer("business-consumer-group");
        // 指定NameServer地址
        consumer.setNamesrvAddr("127.0.0.1:9876");
        // 2. 设置集群消费模式（默认即为集群消费，可省略）
        consumer.setMessageModel(org.apache.rocketmq.common.protocol.heartbeat.MessageModel.CLUSTERING);
        // 订阅主题
        consumer.subscribe("common_topic", "*");

        // 3. 注册并发消费监听器
        consumer.registerMessageListener((List<MessageExt> msgs, ConsumeConcurrentlyContext context) -> {
            try {
                // 遍历处理消息
                for (MessageExt msg : msgs) {
                    System.out.println("集群消费消息：" + new String(msg.getBody()));
                }
                // 消费成功，返回消费成功状态，自动提交Offset
                return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
            } catch (Exception e) {
                e.printStackTrace();
                // 消费失败，触发重试机制
                return ConsumeConcurrentlyStatus.RECONSUME_LATER;
            }
        });

        // 启动消费者
        consumer.start();
        System.out.println("集群消费者启动成功");
    }
}
```

## 5.2 广播消费模式

广播消费是小众但不可或缺的消费模式，主打**全实例全量消费**，适用于需要所有服务节点同步感知消息的场景，与集群消费完全互补。

### 5.2.1 广播消费的定义：每个消费者接收 Topic 的所有消息

**广播消费核心定义**：同一消费者组内的**所有消费者实例**，都会独立消费Topic的**每一条消息**，消息会被组内所有节点全量消费，不存在负载均衡、消息分担机制。

简单理解：集群消费是「抢着消费，一人干活」，广播消费是「全员消费，人人干活」。

### 5.2.2 消费规则：每个消费者消费 Topic 的所有 Message Queue

广播消费打破集群消费的队列独占规则，核心规则如下：

- 关闭队列负载均衡机制，组内每个消费者实例都会订阅Topic的**全部Message Queue**

- 所有实例独立拉取、独立消费，互不干扰

- 不存在队列分配、不存在空闲实例，所有节点均会接收全量消息

### 5.2.3 适用场景：配置同步、日志采集、事件通知

广播消费仅适用于**全局同步、节点感知**类场景，禁止用于业务数据处理：

- **配置动态同步**：全局配置更新、黑白名单更新，所有服务节点实时加载最新配置

- **系统事件通知**：服务启停通知、系统公告、全局开关变更

- **全节点日志采集**：特殊运维指令、日志采集触发

### 5.2.4 Offset 提交：广播消费模式下 Offset 存储在本地

广播消费与集群消费的核心区别：**广播消费的Offset持久化在消费者本地磁盘，不提交到Broker**。

原理与影响：

- 每个节点独立维护自己的消费进度，节点之间进度互不影响

- 节点重启、重装服务后，本地Offset丢失，会重新消费历史消息

- Broker不维护广播消费进度，无法统一管控消费位点

**生产避坑**：广播消费天然支持重复消费，业务必须做好**幂等性**处理。

### 5.2.5 广播消费的局限性：不支持水平扩展，消费速度受限于单个消费者

广播消费存在明显短板，生产需严格控制使用范围：

- **无法水平扩容**：增加消费者节点不会提升消费速度，只会增加重复消费次数

- **单节点性能瓶颈**：全量消息由每个节点完整消费，消费速度取决于单节点性能

- **无重试兜底**：广播消费失败默认不进入重试队列，需业务自行处理异常

**实操代码（广播消费配置）**

```java
import org.apache.rocketmq.client.consumer.DefaultMQPushConsumer;
import org.apache.rocketmq.client.consumer.listener.ConsumeConcurrentlyContext;
import org.apache.rocketmq.client.consumer.listener.ConsumeConcurrentlyStatus;
import org.apache.rocketmq.client.consumer.listener.MessageListenerConcurrently;
import org.apache.rocketmq.common.protocol.heartbeat.MessageModel;
import org.apache.rocketmq.common.message.MessageExt;
import java.util.List;

/**
 * 广播消费模式消费者
 * 特点：全节点消费、Offset本地存储、不支持扩容
 */
public class BroadcastConsumer {
    public static void main(String[] args) throws Exception {
        DefaultMQPushConsumer consumer = new DefaultMQPushConsumer("config-sync-group");
        consumer.setNamesrvAddr("127.0.0.1:9876");
        // 开启广播消费模式
        consumer.setMessageModel(MessageModel.BROADCASTING);
        consumer.subscribe("config_sync_topic", "*");

        consumer.registerMessageListener((List<MessageExt> msgs, ConsumeConcurrentlyContext context) -> {
            for (MessageExt msg : msgs) {
                System.out.println("广播消费-配置更新消息：" + new String(msg.getBody()));
            }
            return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
        });

        consumer.start();
        System.out.println("广播消费者启动成功");
    }
}
```

## 5.3 负载均衡策略

负载均衡重平衡（Rebalance）是集群消费的核心底层机制，负责队列分配、节点扩容缩容适配，同时也是**消息重复、消费抖动、堆积**的主要诱因，必须掌握底层流程与优化方案。

### 5.3.1 负载均衡的触发条件：消费者组变化、Broker 节点变化

客户端会在以下场景**自动触发Rebalance重平衡**：

- 消费者组节点变化：新增消费者、下线消费者、节点重启

- Topic队列变化：队列扩容、队列缩容

- 集群Broker节点上下线、路由信息变更

- 客户端定时路由刷新、心跳超时重连

### 5.3.2 负载均衡算法：平均分配、轮询分配

RocketMQ客户端默认提供两种队列分配算法，核心以**平均分配**为主：

- **平均分配算法（默认）**：将队列数量均匀平分给所有消费者，余数队列依次分配给靠前实例，保证压力最均衡

- **轮询分配算法**：按顺序轮流分配队列，适用于节点性能不均的特殊场景

算法核心目标：最大化均分队列，避免单实例负载过高。

### 5.3.3 负载均衡流程：停止消费→分配 Message Queue→恢复消费

一次完整的Rebalance执行流程：

1. **暂停消费**：消费者停止拉取新消息，完成当前批次消息消费

2. **获取最新路由**：从NameServer拉取最新Topic队列列表与节点信息

3. **重新分配队列**：通过负载均衡算法，重新计算当前实例需要消费的队列

4. **切换消费队列**：释放旧队列，绑定新队列

5. **恢复消费**：从新队列的最新Offset继续拉取消息

**核心问题**：重平衡过程中会短暂停止消费，频繁重平衡会导致**消费抖动、消息堆积、重复消费**。

### 5.3.4 减少 Rebalance 影响的优化：合理设置会话超时、避免频繁启停消费者

生产优化方案（解决重平衡抖动核心方案）：

- **禁止频繁启停服务**：避免大批量节点同时重启，分批灰度发布

- **调优心跳超时时间**：适当延长客户端心跳检测周期，避免误判节点下线

- **扩容队列优先于扩容节点**：提前规划队列数量，避免上线节点触发重平衡

- **关闭无效重平衡**：稳定运行期减少路由刷新频率

## 5.4 消费者配置与优化

消费者参数直接决定消费吞吐量、稳定性、异常容错能力，合理的参数配置是线上服务稳定的关键。

### 5.4.1 消费模式选择：集群消费 vs 广播消费的适用场景

生产标准化选型：

- **业务数据消费**：订单、支付、库存、数据同步 → 强制使用**集群消费**

- **全局配置、事件通知**：配置更新、系统公告、开关同步 → 使用**广播消费**

- **绝对禁止混用**：同一Topic禁止不同实例使用不同消费模式，会导致消费异常

### 5.4.2 消费者线程数：与 Message Queue 数量匹配，最大化消费并发

核心优化公式：**消费者总线程数 ≤ Topic队列总数**

- 单实例消费线程默认20，可通过 `setConsumeThreadMax` 修改

- 队列数量不足时，调大线程数无法提升并发，只会造成线程空闲

- 最佳实践：队列数 = 总消费线程数，资源利用率最高

### 5.4.3 消息拉取配置：`pullBatchSize`、`pullInterval`，减少拉取次数

核心拉取参数优化：

- **pullBatchSize**：单次拉取最大消息数，默认32，高吞吐场景可调整为100~500，减少网络IO次数

- **pullInterval**：两次拉取间隔，默认0（无间隔），低流量场景可适当增加，降低空轮询压力

```java
// 单次最多拉取100条消息
consumer.setPullBatchSize(100);
// 拉取间隔0，极致消费
consumer.setPullInterval(0);
```

### 5.4.4 消费重试机制：消费失败重试、死信队列处理

集群消费内置完善的重试机制，广播消费无重试：

- **重试规则**：消费返回RECONSUME_LATER时，消息进入当前消费者组重试队列

- **重试次数**：默认16次，逐级延时重试

- **死信队列**：超过最大重试次数仍失败，消息转入死信队列，停止自动重试，等待人工排查

**生产最佳实践**：必须监听死信队列消息，配置告警，及时处理异常数据。

# 6. RocketMQ 生产环境配置：参数调优、集群部署、监控告警

本章聚焦RocketMQ**生产落地全流程**，覆盖核心参数调优、标准化集群部署、监控告警搭建、企业级最佳实践，所有配置与流程均经过线上验证，是搭建稳定生产集群的标准规范，也是面试中「线上优化、集群运维」核心考点。

## 6.1 核心参数调优

RocketMQ的性能、可靠性、稳定性完全由Broker核心参数控制，不同业务需要差异化调优，默认参数无法适配线上高并发场景。

### 6.1.1 Broker 配置：刷盘策略、主从复制模式、消息存储路径

生产核心基础配置（broker.conf）：

```properties
# 1. 刷盘策略：高吞吐用异步，金融核心用同步
flushDiskType=ASYNC_FLUSH
# 2. 主从复制模式：同步复制保障数据可靠
brokerRole=SYNC_MASTER
# 3. 消息存储路径（分开挂载独立磁盘，避免系统盘IO争抢）
storePathRootDir=/data/rocketmq/store
storePathCommitLog=/data/rocketmq/store/commitlog
storePathConsumeQueue=/data/rocketmq/store/consumequeue
storePathIndex=/data/rocketmq/store/index
```

**优化说明**：存储路径必须独立磁盘挂载，防止日志、系统IO挤占MQ读写IO，引发消息堆积。

### 6.1.2 存储参数：`mapedFileSizeCommitLog`、`mapedFileSizeConsumeQueue`

文件映射大小参数，决定磁盘文件分片大小，影响IO效率：

```properties
# CommitLog单个文件大小，默认1G，生产保持默认即可
mapedFileSizeCommitLog=1073741824
# ConsumeQueue单个文件大小，默认600W条索引
mapedFileSizeConsumeQueue=6000000
```

**优化说明**：大流量集群无需修改，小流量集群可适当缩小，减少磁盘空间占用。

### 6.1.3 网络参数：`listenPort`、`sendMessageThreadPoolNums`、`pullMessageThreadPoolNums`

网络线程池参数，直接决定集群吞吐上限：

```properties
# 服务监听端口
listenPort=10911
# 消息发送处理线程数，高吞吐调大至64
sendMessageThreadPoolNums=64
# 消息拉取处理线程数，消费密集场景调大
pullMessageThreadPoolNums=64
```

**避坑**：线程数过小会导致请求排队、消息堆积，过大会造成线程竞争、CPU飙升。

### 6.1.4 内存参数：`transientStorePoolEnable`（异步刷盘内存池）

生产**高频优化参数**，异步刷盘场景必备：

```properties
# 开启堆外内存池，提升异步刷盘性能，减少GC
transientStorePoolEnable=true
# 内存池缓存页数
transientStorePoolSize=8
```

**作用**：开启后消息优先写入堆外内存，避免JVM GC影响消息写入性能，大幅提升高并发稳定性。

### 6.1.5 消息参数：`maxMessageSize`、`defaultTopicQueueNums`

```properties
# 最大消息大小，默认4M，大消息场景可适当调整
maxMessageSize=4194304
# 默认Topic队列数量，生产建议8~16
defaultTopicQueueNums=16
```

**最佳实践**：普通业务队列数16，高吞吐业务32，避免后期频繁扩容队列。

## 6.2 集群部署实践

标准化集群部署是线上稳定的基础，错误的部署架构会导致集群无法扩容、单点故障、数据丢失等严重问题。

### 6.2.1 NameServer 部署：至少 2 个节点，实现高可用

NameServer无状态、无主从，部署规范：

- 生产必须部署**2个及以上节点**，防止单NameServer单点故障

- 节点部署在不同服务器，避免机器故障集群不可用

- 所有Broker、客户端配置全部NameServer地址，实现路由容错

### 6.2.2 Broker 集群部署：多主多从架构，建议 2~4 个 Master 节点

生产标准架构：**双主双从**为中小企业标配，大型集群采用4主4从

- Master节点负责读写，分散写入压力，解决单主瓶颈

- Slave节点负责数据备份与读分流，提升消费吞吐

- 所有主从节点跨机器部署，避免单机故障影响整体集群

### 6.2.3 集群扩容：新增 Broker 节点、Topic 分区扩容

集群在线扩容方案（**无需停机、业务无感知**）：

- **节点扩容**：新增主从节点，加入集群，NameServer自动感知路由

- **队列扩容**：通过控制台/命令行增加Topic队列数，客户端自动重平衡适配

**扩容原则**：队列扩容优先于节点扩容，队列是并发瓶颈的根本。

### 6.2.4 集群部署注意事项：节点网络规划、端口开放、时间同步

部署必查事项，规避线上隐形故障：

- **网络规划**：集群节点内网互通，禁止跨公网部署，保证低延迟

- **端口开放**：9876(NameServer)、10911(Broker)、10912(同步端口)全部放行

- **时间同步**：所有节点开启NTP时间同步，避免时间差导致消息异常、日志错乱

- **磁盘规划**：存储目录独立磁盘，禁止使用系统盘

## 6.3 监控告警配置

监控是线上运维的眼睛，无监控的集群等同于裸奔，必须实现指标监控、异常告警、故障闭环。

### 6.3.1 核心监控指标：Broker 节点状态、消息生产/消费速率、堆积量

生产核心监控指标（必须监控）：

- **节点状态**：Broker在线状态、主从同步状态、节点心跳状态

- **流量指标**：生产TPS、消费TPS、消息丢弃率、异常率

- **堆积指标**：Topic总堆积、单队列堆积、堆积增长速率

- **资源指标**：CPU、内存、磁盘使用率、磁盘IO负载

### 6.3.2 监控工具：RocketMQ Dashboard、Prometheus + Grafana、RocketMQ-Exporter

生产监控组合方案：

- **RocketMQ Dashboard**：官方可视化控制台，用于日常 Topic 管理、消息查询、集群状态查看

- **RocketMQ-Exporter**：采集集群指标，对接Prometheus

- **Prometheus+Grafana**：持久化指标、可视化大屏、历史趋势分析，生产核心监控方案

### 6.3.3 告警配置：消息堆积告警、节点故障告警、副本同步异常告警

生产必备告警规则：

- **节点故障告警**：Broker/NameServer节点离线立即告警

- **消息堆积告警**：单Topic堆积超1000条、堆积持续增长告警

- **主从同步异常**：主从数据同步延迟、同步中断告警

- **生产异常告警**：消息发送失败率突增告警

### 6.3.4 告警处理流程：故障定位、问题排查、优化验证

标准化故障闭环流程：

1. **告警接收**：确认告警范围、影响业务

2. **快速定位**：查看监控指标，区分节点故障、消费异常、流量突增

3. **紧急修复**：重启服务、扩容节点、清理异常消息、临时限流

4. **根因排查**：定位代码bug、参数不合理、资源瓶颈

5. **优化验证**：上线优化方案，观察监控恢复稳定

## 6.4 生产环境最佳实践

汇总企业级RocketMQ生产落地规范，规避90%以上线上问题。

### 6.4.1 Topic 设计：按业务模块划分 Topic，避免过大 Topic

Topic设计规范：

- **按业务模块拆分**：订单Topic、支付Topic、库存Topic严格隔离

- **禁止大杂烩Topic**：不同优先级、不同业务流量禁止混用Topic

- **分区预扩容**：新建Topic直接配置16/32队列，预留扩容空间

### 6.4.2 消息设计：消息体大小控制、序列化方式选择

- **消息体大小**：单条消息控制在4K以内，超大消息拆分处理，避免IO阻塞

- **序列化选型**：优先Protobuf，其次JSON，拒绝笨重序列化方案

- **消息幂等字段**：每条消息必须携带唯一业务ID，用于消费幂等

### 6.4.3 安全配置：ACL 权限控制、IP 白名单、SSL/TLS 加密

- **ACL权限控制**：开启账号密码认证，区分读写权限，防止非法访问

- **IP白名单**：限制业务服务器IP访问集群，杜绝外网非法连接

- **传输加密**：核心业务开启SSL/TLS，防止消息数据窃听篡改

### 6.4.4 备份与恢复：元数据备份、CommitLog 备份、故障恢复流程

- **元数据备份**：定期备份Topic配置、消费者组配置、ACL权限

- **日志备份**：开启CommitLog定时归档、异地备份

- **故障预案**：制定节点宕机、数据丢失、消息堆积的标准化恢复流程
