# 1. 限流算法：计数器、滑动窗口、令牌桶、漏桶

限流是高并发防护的**第一道防线**，核心作用是限制单位时间内的请求访问量，防止突发流量打垮服务、数据库、缓存等资源。所有限流框架（Sentinel、Guava、Redis限流）底层均基于四大基础算法实现：固定窗口计数器、滑动窗口计数器、令牌桶、漏桶。不同算法的流量整形能力、精度、突发流量适配性差异极大，选型错误会导致限流失效、误杀正常流量等生产问题。

## 1.1 计数器算法

### 1.1.1 计数器算法原理：单位时间内请求数累加，超过阈值则限流

固定窗口计数器是最简单、最基础的限流算法，核心逻辑通俗易懂、实现成本极低，是入门级限流方案。

**核心原理**：人为划定固定的时间窗口（如1秒、1分钟），在当前时间窗口内维护一个请求计数器，每接收一次合法请求，计数器数值自增1；请求进入时先判断计数器数值，若**超过预设阈值**，直接拒绝请求（限流）；当时间窗口结束，计数器清零重置，进入下一个统计周期。

**执行流程**：

1. 定义规则：1秒内最多允许100次请求；

2. 开启1秒固定时间窗口，初始计数器=0；

3. 每来一个请求，计数器+1；

4. 计数器>100，直接返回限流提示；

5. 1秒时间结束，计数器清零，重新统计。

**核心优势**：逻辑简单、代码极简、性能极高、无复杂计算，适合简单流量拦截场景。

### 1.1.2 固定窗口计数器的问题：临界值流量突增（边界问题）

固定窗口计数器存在**致命边界漏洞**，无法应对临界时间点的流量冲击，是生产环境禁止直接使用的核心原因。

**漏洞场景还原**：假设限流规则为「1秒最多100请求」

- 第0.9秒：当前窗口剩余额度充足，瞬间涌入100个请求，打满当前窗口阈值；

- 第1.0秒：时间窗口清零重置，计数器恢复为0；

- 第1.1秒：新窗口再次瞬间涌入100个请求；

**最终结果**：在**0.9s~1.1s这0.2秒的极短时间内**，系统接收了200次请求，远超1秒100的限流阈值，导致限流失效，突发流量直接冲击系统，极易引发服务卡顿、雪崩。

该问题的本质：**固定窗口时间切换的临界点，会出现双倍流量穿透漏洞**，无法精准控制瞬时流量。

### 1.1.3 计数器算法的实现方式与适用场景

**简单Java代码实现（本地固定窗口计数器）**

```java
/**
 * 固定窗口计数器限流实现
 * 规则：1秒内最多允许100次请求
 */
public class FixedWindowRateLimit {
    // 限流阈值：每秒最大请求数
    private static final int LIMIT = 100;
    // 时间窗口大小：1000ms = 1秒
    private static final long WINDOW_TIME = 1000;
    // 当前窗口请求计数器
    private int count = 0;
    // 当前窗口开始时间
    private long windowStartTime = System.currentTimeMillis();

    /**
     * 请求限流判断
     * @return true=放行，false=限流
     */
    public synchronized boolean tryAcquire() {
        long now = System.currentTimeMillis();
        // 判断是否超出当前时间窗口，超出则重置窗口与计数器
        if (now - windowStartTime > WINDOW_TIME) {
            count = 0;
            windowStartTime = now;
        }
        // 未超过阈值则放行，计数器自增
        if (count < LIMIT) {
            count++;
            return true;
        }
        // 超过阈值，限流拦截
        return false;
    }
}

```

**适用场景**：仅适用于**流量平稳、无突发峰值、对限流精度要求极低**的后台管理系统、内部工具接口，严禁用于对外高并发接口、秒杀、大促场景。

**生产避坑**：分布式场景下本地计数器失效，需结合Redis实现分布式固定窗口限流，但仍无法解决临界流量漏洞。

### 1.1.4 优化思路：滑动窗口替代固定窗口

为解决固定窗口的临界流量漏洞，业界标准优化方案是**滑动窗口算法**。核心优化思想：将一个大的固定时间窗口，拆分为多个细小的时间格子，通过滚动替换格子的方式，实现流量的平滑统计，消除时间窗口切换的临界漏洞，大幅提升限流精度。

## 1.2 滑动窗口算法

### 1.2.1 滑动窗口原理：将时间片细分为多个格子，滚动统计请求数

**滑动窗口（Rolling Window）**是固定窗口的迭代优化算法，核心是**细分时间粒度、滚动统计流量**。

**核心原理**：

1. 将原本1秒的固定大窗口，均匀拆分为N个细小时间格子（例如拆分为10格，每格100ms）；

2. 始终统计「最近1秒时间范围内」所有格子的总请求数，而非固定的自然秒；

3. 时间滚动前进，每经过一个小格子时间，丢弃最老旧的格子数据，新增最新的格子数据，实现窗口滑动；

4. 通过总请求数与阈值对比，判断是否限流。

**通俗理解**：固定窗口是「整段清零」，滑动窗口是「淘汰老旧、新增最新」，始终保持统计最新时间段的真实流量。

### 1.2.2 滑动窗口解决固定窗口的边界问题

沿用上述案例：限流规则1秒100请求，1秒拆分为10个100ms小窗口。

临界场景：0.9s打满前9个窗口（90个请求），1.0s后进入新时间格子。此时滑动窗口统计的是**最近1秒内的所有请求**，前9个窗口的流量仍在统计范围内，新格子涌入请求会累计计算，总流量超过100直接触发限流，**彻底杜绝双倍流量穿透问题**。

**核心价值**：流量统计更贴合真实时间维度，无窗口切换漏洞，限流精度大幅提升。

### 1.2.3 窗口粒度与限流精度的权衡

滑动窗口的**精度与性能完全取决于格子拆分数量**，生产环境需要做精准权衡：

- **拆分格子越多（粒度越细）**：限流精度越高，流量统计越精准，但内存占用、计算开销越大，性能略有损耗；

- **拆分格子越少（粒度越粗）**：计算性能更高，但精度下降，仍会存在少量临界流量漏洞。

**生产最佳实践**：1秒时间窗口默认拆分10~20格，兼顾精度与性能，是互联网通用配置。

### 1.2.4 滑动窗口算法的实现与性能优化

**核心实现思路**：通过数组/队列存储每个小时间格子的请求计数，定时淘汰过期格子、叠加最新计数。分布式场景下，通常基于Redis ZSet实现高精度滑动窗口限流。

**性能优化方案**：

1. 避免频繁创建对象，复用格子计数对象；

2. 采用定时批量清理过期数据，而非每次请求遍历清理；

3. 高并发场景使用Redis Lua脚本原子执行，保证分布式限流一致性。

**适用场景**：对外普通接口、中台业务接口、需要精准限流的常规高并发场景。

## 1.3 令牌桶算法（Token Bucket）

### 1.3.1 令牌桶原理：按固定速率生成令牌，请求需获取令牌才能执行

令牌桶算法是**互联网最主流的限流算法**，Guava RateLimiter、Sentinel默认限流算法，兼顾平稳限流与突发流量适配。

**核心原理**：

1. 系统维护一个固定容量的令牌桶，桶内存储虚拟令牌；

2. 系统以**固定匀速**向桶内生成、投放令牌，例如每秒生成100个令牌；

3. 令牌桶存满后不再新增，避免资源溢出；

4. 每一个客户端请求，必须先从桶内获取一个令牌，获取成功则放行执行；

5. 桶内无可用令牌时，直接限流拦截请求。

核心逻辑：**令牌匀速生成，请求抢占令牌执行**。

### 1.3.2 令牌桶的核心特性：允许短时间突发流量

令牌桶区别于其他算法的**最大核心优势**：支持合法的短时间突发流量。

**突发流量原理**：

- 当系统流量低谷、空闲时，令牌会持续生成并存储在桶内，直至桶满；

- 当瞬间突发流量到来时，可一次性消耗桶内预存的所有令牌，瞬间承接大量请求；

- 桶内令牌耗尽后，后续请求将按照令牌生成的匀速节奏平稳放行。

**业务价值**：完美适配互联网业务**平时流量平稳、偶尔突发峰值**的流量特征，既保护系统不被持续高流量打垮，又不拦截合法的突发业务流量。

### 1.3.3 令牌桶算法的实现：Guava RateLimiter、Redis实现

生产环境两种主流实现：**本地Guava RateLimiter（单机）**、**Redis Lua（分布式）**

**1. Guava RateLimiter 单机令牌桶实现（可直接运行）**

第一步：引入Maven依赖

```xml
<!-- Guava工具类，内置令牌桶限流 -->
<dependency>
    <groupId>com.google.guava</groupId>
    <artifactId>guava</artifactId>
    <version>31.1-jre</version>
</dependency>

```

第二步：核心限流代码

```java
import com.google.common.util.concurrent.RateLimiter;

/**
 * Guava令牌桶限流示例
 * 每秒生成100个令牌，支持突发流量
 */
public class TokenBucketRateLimit {
    // 创建令牌桶：每秒投放100个令牌
    private static final RateLimiter RATE_LIMITER = RateLimiter.create(100.0);

    public static void main(String[] args) {
        // 模拟200次并发请求
        for (int i = 0; i < 200; i++) {
            new Thread(() -> {
                // tryAcquire：非阻塞获取令牌，获取成功返回true，失败false
                if (RATE_LIMITER.tryAcquire()) {
                    System.out.println(Thread.currentThread().getName() + "：请求放行，执行业务逻辑");
                } else {
                    System.out.println(Thread.currentThread().getName() + "：请求限流，拒绝访问");
                }
            }).start();
        }
    }
}

```

**2. Redis+Lua 分布式令牌桶实现**

适用于集群部署、分布式服务场景，保证全局限流统一，避免单机限流误差。通过Lua脚本原子性实现令牌生成、扣除、判断逻辑，解决并发竞争问题。

### 1.3.4 适用场景：允许短时间突发流量的业务场景

令牌桶是**生产环境首选限流算法**，适配绝大多数互联网业务：

- 电商商品查询、列表接口、详情接口；

- 用户登录、个人中心、普通查询类接口；

- 秒杀、大促预热等存在合法突发流量的场景；

- 所有需要兼顾平稳限流与突发流量兼容的高并发业务。

## 1.4 漏桶算法（Leaky Bucket）

### 1.4.1 漏桶原理：请求进入桶内，按固定速率流出，超出桶容量则限流

漏桶算法的核心思想是**流量整形、强制匀速**，与令牌桶算法逻辑完全相反。

**核心原理**：

1. 系统维护一个固定容量的漏桶，客户端所有请求统一进入漏桶排队；

2. 漏桶以**固定匀速**从底部流出请求、执行业务逻辑；

3. 请求进入速度 > 流出速度时，桶内请求堆积；

4. 桶内请求堆满容量后，新进入的请求直接限流丢弃。

通俗理解：像水桶漏水一样，**无论进水多快，出水速度恒定**，强制抹平所有流量波动。

### 1.4.2 漏桶的核心特性：强制平滑流量，不允许突发

漏桶算法最核心的特性：**绝对不允许任何突发流量**，强制将所有请求转换为匀速流量。

无论客户端瞬时请求量多大，最终服务端处理请求的速率完全固定，流量曲线绝对平滑，彻底杜绝流量峰值冲击，但会扼杀合法的突发业务流量。

### 1.4.3 漏桶与令牌桶的核心区别：突发流量支持 vs 强制平滑

|对比维度|令牌桶算法|漏桶算法|
|---|---|---|
|流量特性|支持短时间突发流量|强制流量匀速，禁止突发|
|核心目标|流量控制，兼容合法峰值|流量整形，绝对平滑流量|
|资源利用|资源利用率高，空闲预存令牌|资源利用率低，严格匀速处理|
|业务适配性|适配绝大多数互联网业务|仅适配匀速异步任务场景|
### 1.4.4 适用场景：必须限制请求速率、不允许突发的场景

漏桶算法适用场景极少，仅用于**绝对禁止流量突发、必须匀速处理**的业务：

- 第三方接口调用限流（防止频繁调用被封禁）；

- 数据库批量写入、异步数据同步任务；

- 消息消费匀速处理、防止消费突发压垮下游；

- 风控、审核类匀速处理业务。

## 1.5 四大限流算法对比与选型

### 1.5.1 精度、性能、突发支持的对比分析

|算法|限流精度|性能开销|支持突发流量|优缺点总结|
|---|---|---|---|---|
|固定窗口计数器|低（存在临界漏洞）|极低|不支持|实现简单、性能极高，存在流量穿透漏洞，生产极少使用|
|滑动窗口计数器|高|低|不支持|解决临界漏洞，精度优秀，流量完全匀速，无突发能力|
|令牌桶|高|低|**支持**|兼顾精度与突发流量，适配绝大多数业务，生产首选|
|漏桶|极高|低|不支持|强制流量平滑，扼杀突发流量，仅用于特殊匀速场景|
### 1.5.2 不同业务场景下的算法选型建议

1. **对外高并发接口、秒杀、商品查询**：优先选择**令牌桶算法**，兼容合法突发流量，保护系统同时保证用户体验；

2. **普通后台接口、内部系统接口**：选择**滑动窗口算法**，精度足够、性能优异；

3. **异步任务、数据同步、第三方调用**：选择**漏桶算法**，强制匀速执行，避免打爆下游资源；

4. **极简内部工具、低流量场景**：可使用固定窗口计数器，简化开发。

### 1.5.3 生产环境中Guava RateLimiter的令牌桶实现实践

**生产最佳实践总结**：

- **单机限流首选Guava RateLimiter**：无需部署中间件、性能极致、开箱即用，适合单实例服务、网关单机限流；

- **分布式限流禁用Guava**：Guava是本地内存限流，集群部署时多实例限流阈值不统一，导致限流失效，需替换为Redis+Lua令牌桶；

- **阈值配置规范**：限流阈值需根据服务单机QPS、集群扩容数量合理配置，预留20%冗余，避免误杀正常流量；

- **异常处理规范**：限流拦截后统一返回友好提示，记录限流日志，便于流量监控与风控分析。

# 2. 限流实现方式：接口限流、服务限流、分布式限流

前文讲解的四大限流算法是理论基础，本章聚焦**工程落地实现**。在微服务架构中，限流按照管控范围可划分为接口限流、服务限流、分布式限流三类，分别适配不同流量层级、不同部署架构的防护场景。单机限流仅适用于单实例服务，集群部署必须依赖分布式限流保证全局流量可控，三者层层递进，构成系统完整的限流防护体系。

## 2.1 接口限流

接口限流是**粒度最细**的限流方式，针对系统单个API接口单独配置流量规则，精准保护核心接口、高频接口，避免单一接口流量异常拖垮整个服务，是日常开发最常用的限流手段。

### 2.1.1 接口限流的定义：针对单个接口设置限流规则

接口限流指以**单个HTTP/API接口**为维度，独立配置QPS阈值、并发数、访问频次的限流策略。不同接口可以配置完全不同的限流规则，核心接口宽松限流、非核心接口严格限流，实现流量精细化管控。

**核心价值**：精准隔离接口流量，秒杀、查询、支付等高频核心接口可单独防护，避免非核心接口流量异常影响核心业务，实现**流量精细化隔离**。

### 2.1.2 接口限流的实现位置：网关层、应用层

接口限流在工程中有两个标准实现位置，各有优劣，生产中常组合使用：

- **网关层限流（前置限流）**：实现于Spring Cloud Gateway、Nginx层，请求未进入业务服务即拦截，性能损耗极低，拦截范围覆盖所有接口，属于**第一道防护**；

- **应用层限流（后置限流）**：实现于业务服务内部，通过拦截器、过滤器、注解实现，可精准控制单个接口、适配复杂业务限流规则，属于**第二道精细化防护**。

**最佳实践**：网关层做通用粗粒度限流，应用层做业务细粒度限流，双层防护，杜绝流量穿透。

### 2.1.3 接口限流的维度：IP限流、用户限流、请求参数限流

接口限流支持多维度组合管控，解决不同场景的恶意访问、刷接口、流量倾斜问题：

1. **IP限流**：限制单个IP单位时间内的接口访问次数，主要防护**单IP恶意刷接口、爬虫攻击、CC攻击**，是基础防护手段；

2. **用户限流**：基于登录用户ID限流，限制单个用户的接口访问频次，防止单个用户高频请求占用大量服务资源，保障公平性；

3. **请求参数限流**：基于接口入参维度限流，例如限制单个订单ID、单个商品ID的查询频次，防止热点参数打爆缓存、数据库。

**生产场景**：商品详情接口常用「IP+用户+商品ID」三维度组合限流，彻底杜绝热点流量与恶意访问。

### 2.1.4 网关层限流：Nginx、Spring Cloud Gateway的限流实现

**1. Nginx限流（接入层限流）**

Nginx基于漏桶算法实现全局限流，作用于最外层接入层，拦截所有入口流量，性能极强，不占用业务服务资源。核心配置为`limit_req_zone`，支持IP维度限流。

**Nginx核心限流配置示例**

```nginx
# 定义限流区域：基于IP限流，内存存储，速率100请求/秒
limit_req_zone $binary_remote_addr zone=ip_limit:10m rate=100r/s;

server {
    listen 80;
    server_name localhost;
    
    # 应用限流规则，burst=20允许20个请求排队，nodelay不延迟放行
    location /api/ {
        limit_req zone=ip_limit burst=20 nodelay;
        proxy_pass http://localhost:8080;
    }
}

```

**参数说明**：rate为限流速率，burst为突发排队请求数，nodelay用于快速放行合法突发流量。

**2. Spring Cloud Gateway限流（微服务网关限流）**

Spring Cloud Gateway基于Redis+令牌桶实现分布式网关限流，支持接口级、IP级、用户级动态限流，适配微服务集群架构，是SpringCloud体系标准网关限流方案。可结合Sentinel网关限流组件快速实现，无需手动编码。

### 2.1.5 应用层限流：拦截器/过滤器+Guava RateLimiter实现

应用层限流依托Spring拦截器、过滤器实现，结合Guava令牌桶算法，可精准实现**单个接口、自定义维度**的限流逻辑，开发灵活、适配业务复杂场景。

**完整可运行实操代码**

1. 自定义限流注解（标记需要限流的接口）

```java
import java.lang.annotation.*;

/**
 * 接口限流注解
 */
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface ApiRateLimit {
    // 每秒最大请求数
    double limit() default 100.0;
}

```

2. 自定义拦截器实现限流逻辑

```java
import com.google.common.util.concurrent.RateLimiter;
import org.springframework.stereotype.Component;
import org.springframework.web.method.HandlerMethod;
import org.springframework.web.servlet.HandlerInterceptor;

import javax.servlet.http.HttpServletRequest;
import javax.servlet.http.HttpServletResponse;
import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;

/**
 * 接口限流拦截器
 */
@Component
public class RateLimitInterceptor implements HandlerInterceptor {

    // 缓存每个接口的限流器，key=接口方法名，value=对应令牌桶
    private final Map<String, RateLimiter> limitMap = new ConcurrentHashMap<>();

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception {
        // 仅拦截注解标记的接口方法
        if (handler instanceof HandlerMethod) {
            HandlerMethod handlerMethod = (HandlerMethod) handler;
            ApiRateLimit rateLimit = handlerMethod.getMethodAnnotation(ApiRateLimit.class);
            if (rateLimit != null) {
                String methodName = handlerMethod.getMethod().getName();
                // 初始化对应接口的限流器
                RateLimiter rateLimiter = limitMap.computeIfAbsent(methodName, 
                        k -> RateLimiter.create(rateLimit.limit()));
                // 尝试获取令牌，获取失败则限流
                if (!rateLimiter.tryAcquire()) {
                    response.setContentType("text/html;charset=UTF-8");
                    response.getWriter().write("系统繁忙，请稍后再试！");
                    return false;
                }
            }
        }
        return true;
    }
}

```

3. 接口使用示例

```java
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class TestController {

    // 该接口每秒最多允许50次请求
    @ApiRateLimit(limit = 50.0)
    @GetMapping("/api/test")
    public String test() {
        return "接口请求成功";
    }
}

```

**避坑指南**：该方案为单机限流，集群部署时多实例会导致总QPS超标，集群环境必须搭配分布式限流使用。

## 2.2 服务限流

服务限流是**粗粒度、全局级**的限流方式，管控范围为整个微服务实例，不区分具体接口，用于保护整个服务的最大承载流量，防止服务被整体打挂。

### 2.2.1 服务限流的定义：针对整个服务设置限流规则

服务限流指以**单个微服务**为整体，设置全局最大QPS、最大并发数，无论请求访问服务下的任意接口，都会统一纳入全局流量统计。当服务整体流量达到阈值时，统一拦截新请求，保障服务现有请求正常执行，避免服务过载崩溃。

**适用场景**：服务整体流量突增、大促全量流量冲击、服务资源瓶颈固定的场景。

### 2.2.2 服务限流的维度：服务QPS、服务并发数

服务限流核心两大管控维度，生产中组合使用：

- **服务QPS限流**：限制服务每秒最大处理请求总量，控制请求流入速率，适配流量突发场景；

- **服务并发数限流**：限制服务同时正在处理的请求数量，保护线程池、CPU、内存、数据库连接等核心资源，防止资源耗尽。

**核心区别**：QPS控制请求速率，并发数控制资源占用，并发数限流优先级更高，能直接防止服务资源耗尽。

### 2.2.3 服务限流的实现：Sentinel、Hystrix的服务级限流

**1. Sentinel服务限流（生产首选）**

Sentinel原生支持**服务全局限流**，分为QPS限流、线程数限流两种模式，配置简单、动态生效、可视化管控，是目前微服务服务限流的主流方案。支持实时监控流量、动态调整阈值，无需重启服务。

**2. Hystrix服务限流**

Hystrix通过线程池隔离实现服务级限流，通过配置服务线程池最大容量，限制服务最大并发请求数，超出请求直接拒绝。缺点是不支持动态阈值调整，灵活性较差，目前逐步被Sentinel替代。

### 2.2.4 服务限流的优先级：核心服务优先保障、非核心服务优先限流

生产环境多服务共存场景，必须遵循**分级限流优先级策略**，保障核心业务可用性：

1. **核心服务（订单、支付、库存）**：放宽限流阈值，优先保障流量通行，最大限度保障核心交易；

2. **非核心服务（日志、统计、通知）**：严格限流、优先降级，流量高峰直接拦截，释放服务器资源；

3. **流量分配原则**：系统总流量过载时，优先牺牲非核心业务，保全核心业务可用性。

## 2.3 分布式限流

单机限流仅能管控单实例流量，微服务集群部署模式下，多实例单机限流叠加会导致**全局流量失控**，分布式限流是集群架构下的必备方案，实现全节点流量统一管控。

### 2.3.1 分布式限流的定义：跨节点统一控制请求速率

分布式限流是针对**服务集群**的全局限流方案，统一统计整个集群所有实例的请求总量，而非单实例独立统计，保证整个服务集群的QPS、并发数严格控制在预设阈值内，彻底解决集群流量倾斜、限流失效问题。

**核心解决问题**：3实例服务单机限流100QPS，集群总QPS可达300，超出预期阈值，分布式限流可将集群总QPS严格限制为100。

### 2.3.2 分布式限流的实现方式：Redis计数器/令牌桶、Sentinel集群限流

生产主流两种分布式限流方案：

- **Redis实现（通用方案）**：基于Redis计数器、令牌桶+Lua脚本实现，无框架依赖，适配所有微服务框架，通用性极强；

- **Sentinel集群限流（框架方案）**：Sentinel原生集群限流模式，通过Token Server统一分发流量，配置简单、自带监控，SpringCloud体系首选。

### 2.3.3 Redis分布式限流的实现：Lua脚本原子操作

分布式限流的核心难点是**并发竞争与数据一致性**，通过Redis Lua脚本可保证限流判断、计数增减、过期设置全程原子执行，杜绝并发超量问题。

**完整Lua脚本（分布式令牌桶限流）**

```lua
-- 分布式限流Lua脚本
-- KEYS[1]: 限流key
-- ARGV[1]: 限流阈值
-- ARGV[2]: 时间窗口，单位秒

local key = KEYS[1]
local limit = tonumber(ARGV[1])
local windowTime = tonumber(ARGV[2])

-- 获取当前计数
local current = tonumber(redis.call('get', key) or "0")

-- 超过阈值，限流返回0
if current >= limit then
    return 0
end

-- 未超过阈值，计数+1，设置过期时间
redis.call("INCR", key)
redis.call("EXPIRE", key, windowTime)
-- 放行返回1
return 1

```

**Java调用示例**

```java
import org.springframework.data.redis.core.RedisTemplate;
import org.springframework.data.redis.core.script.DefaultRedisScript;
import org.springframework.stereotype.Component;

import javax.annotation.Resource;
import java.util.Collections;

@Component
public class RedisRateLimit {

    @Resource
    private RedisTemplate<String, Object> redisTemplate;

    // 加载Lua脚本
    private static final DefaultRedisScript<Long> SCRIPT;
    static {
        SCRIPT = new DefaultRedisScript<>();
        SCRIPT.setScriptText("Lua脚本内容");
        SCRIPT.setResultType(Long.class);
    }

    /**
     * 分布式限流判断
     * @param key 限流key
     * @param limit 阈值
     * @param windowTime 时间窗口
     * @return true=放行，false=限流
     */
    public boolean tryLimit(String key, int limit, int windowTime) {
        Long result = redisTemplate.execute(SCRIPT, Collections.singletonList(key), limit, windowTime);
        return result != null && result == 1;
    }
}

```

### 2.3.4 分布式限流的一致性问题：时钟同步、Redis主从延迟

分布式限流生产两大核心一致性问题及解决方案：

- **时钟同步问题**：多服务器本地时间不一致，导致时间窗口统计偏差。解决方案：统一基于Redis服务时间统计，不使用本地服务器时间；

- **Redis主从延迟问题**：主从架构下数据同步延迟，导致计数统计不准。解决方案：限流key强制读主库，或使用Redis Cluster集群保证数据一致性。

### 2.3.5 分布式限流的性能优化：本地缓存+Redis的二级限流

纯Redis分布式限流每次请求都需要访问Redis，高并发下存在网络IO性能瓶颈，生产最优优化方案为**本地限流+Redis分布式二级限流**：

1. **一级本地限流**：基于Guava令牌桶做单机快速限流，拦截绝大多数超限请求，无需访问Redis；

2. **二级分布式限流**：本地放行的请求，再经过Redis全局限流校验；

**优化价值**：大幅减少Redis请求量，提升限流性能，同时保证全局流量可控，兼顾性能与准确性。

## 2.4 限流实现的生产实践

### 2.4.1 网关层限流 vs 应用层限流的选型对比

| 对比维度   | 网关层限流           | 应用层限流             |
| ---------- | -------------------- | ---------------------- |
| 拦截时机   | 请求进入服务前拦截   | 请求进入服务后拦截     |
| 性能损耗   | 极低，不占用业务资源 | 较低，占用少量业务线程 |
| 限流粒度   | 服务、接口粗粒度     | 接口、用户、参数细粒度 |
| 业务适配性 | 通用限流，无业务逻辑 | 支持复杂业务定制限流   |
| 生产选型   | 全局第一道防护       | 精细化业务防护         |

### 2.4.2 不同限流维度的组合使用（IP+用户+接口）

单一限流维度存在防护漏洞，生产高并发核心接口必须**多维度组合限流**：

- **IP限流**：防爬虫、防恶意IP批量请求；

- **用户限流**：防单用户薅流量、高频刷接口；

- **接口限流**：防单接口流量过载；

- **参数限流**：防热点参数打爆底层资源。

多维度层层拦截，实现全方位流量防护，无防护死角。

### 2.4.3 限流策略的动态调整：配置中心集成

静态限流阈值无法适配流量动态变化，生产必须实现**动态限流**：

结合Nacos/Apollo配置中心，将限流阈值、限流维度、黑白名单配置放入配置中心，支持**实时动态修改、无需重启服务**，适配大促流量扩容、日常流量波动场景。同时配合Sentinel控制台，实现可视化动态管控、实时流量监控。

# 3. 熔断降级策略：服务雪崩、降级逻辑、熔断器状态流转

限流是防护**流量突增**，熔断降级是防护**服务故障传导**。在微服务调用链路中，下游服务超时、宕机、异常会逐级向上传导，引发服务雪崩。熔断降级是微服务高可用的核心保障，通过故障隔离、快速失败、兜底降级，阻断故障传播，保证系统整体可用。

## 3.1 服务雪崩问题

### 3.1.1 服务雪崩的定义：一个服务故障引发连锁反应，导致整个系统瘫痪

**服务雪崩**是分布式系统最严重的故障场景：微服务链路存在强依赖关系，当下游某一个服务出现故障（超时、宕机、报错），上游服务大量请求阻塞、线程池耗尽，进而导致上游服务瘫痪，故障逐级向上传导，最终**整个调用链路全部不可用，系统整体瘫痪**。

### 3.1.2 雪崩的成因：依赖故障、超时等待、重试风暴

服务雪崩三大核心成因，生产故障90%源于此：

1. **下游依赖故障**：下游服务宕机、数据库挂掉、接口报错，无法正常响应请求；

2. **超时等待阻塞**：上游无超时控制，大量请求阻塞等待下游响应，占用全部业务线程；

3. **重试风暴放大故障**：客户端失败自动重试机制，导致故障后请求量翻倍，进一步压垮故障服务，形成恶性循环。

### 3.1.3 雪崩的危害：系统整体不可用、数据不一致

- **服务整体瘫痪**：单个下游故障拖垮整条业务链路，核心业务无法访问；

- **资源永久占用**：线程池全部阻塞、连接池耗尽，服务无法处理新请求；

- **数据不一致**：大量超时、重试请求导致重复提交、事务中断，引发脏数据；

- **故障恢复缓慢**：雪崩形成后，即使下游恢复，堆积的海量请求会再次冲垮服务。

### 3.1.4 雪崩防护的核心思路：熔断、降级、隔离

业界标准雪崩防护三板斧，缺一不可：

- **隔离**：线程池隔离、信号量隔离，不同服务、不同接口资源互不占用，避免单点故障扩散；

- **熔断**：下游故障达到阈值，自动切断调用链路，停止调用故障服务，快速失败；

- **降级**：调用失败时返回预设兜底结果，保障业务可用，提升用户体验。

## 3.2 降级逻辑设计

### 3.2.1 降级的定义：当依赖故障或系统压力过大时，返回预设的降级结果

降级是系统**主动牺牲非核心能力、保全核心能力**的容错策略。当系统压力过载、下游服务故障、超时率过高时，主动放弃部分非核心业务逻辑，不执行正常业务流程，直接返回预设兜底数据、友好提示或缓存数据，保证核心业务正常运行。

### 3.2.2 降级的核心原则：核心业务优先、快速失败

1. **核心业务不降级，非核心业务优先降级**：订单、支付、登录等核心业务禁止降级；日志、推荐、弹窗、统计等非核心业务优先降级；

2. **快速失败原则**：故障场景不阻塞、不重试、不等待，快速返回结果，释放线程资源；

3. **用户体验优先**：降级提示友好，不出现500、异常堆栈，保证业务流程可走完。

### 3.2.3 降级策略的分类：返回默认值、调用备用接口、本地缓存兜底

生产三类标准降级方案：

- **默认值降级**：直接返回空数据、默认状态、友好提示，适用于非核心查询接口；

- **备用接口降级**：主接口故障时，调用备用兜底接口，适用于重要查询业务；

- **本地缓存降级**：读取本地缓存、Redis缓存历史数据兜底，保证页面正常展示，是最常用的降级方案。

### 3.2.4 降级逻辑的实现：静态降级vs动态降级

- **静态降级**：代码硬编码降级逻辑，固定场景触发兜底，简单稳定，灵活性差；

- **动态降级**：基于Sentinel、配置中心动态开关，可实时开启、关闭降级策略，根据流量、异常率自动触发降级，适配生产动态场景，是生产首选。

### 3.2.5 降级方案的用户体验设计：友好提示、重试引导

降级不是报错，需保证用户体验：

- 禁止返回系统异常、错误堆栈、404/500状态码；

- 使用友好文案：「系统繁忙，稍后重试」「暂无更多数据」；

- 可增加低频率重试引导，避免用户频繁刷新加重系统压力。

## 3.3 熔断器状态流转

熔断器是实现熔断机制的核心组件，通过统计接口异常率、超时率，自动切换熔断状态，实现故障自动阻断、自动恢复，是微服务容错的核心核心。

### 3.3.1 熔断器的三大状态：关闭（Closed）、打开（Open）、半开（Half-Open）

熔断器存在三种固定状态，循环流转：

1. **关闭状态（Closed）**：正常状态，所有请求正常调用下游服务，持续统计异常、超时请求比例；

2. **打开状态（Open）**：故障状态，异常率/超时率超过阈值，熔断开启，**直接拦截所有请求，不调用下游服务**，快速返回降级结果；

3. **半开状态（Half-Open）**：恢复探测状态，熔断等待时间结束后，放行少量探测请求，测试下游服务是否恢复。探测成功则关闭熔断，探测失败则重新打开熔断。

### 3.3.2 状态流转触发条件：失败率阈值、恢复时间窗口

- **关闭→打开**：指定统计时间窗口内，接口失败率、超时率超过预设阈值，触发熔断开启；

- **打开→半开**：熔断持续时间（休眠时间）结束，自动进入半开探测状态；

- **半开→关闭**：探测请求全部成功，判定下游服务恢复，关闭熔断，恢复正常调用；

- **半开→打开**：探测请求存在失败，判定服务未恢复，重新开启熔断。

### 3.3.3 状态流转的核心逻辑：故障统计→熔断开启→恢复探测→恢复关闭

**完整闭环流转流程**：

1. 服务正常运行，熔断器处于关闭状态，正常调用并统计异常数据；

2. 下游故障，异常请求持续增多，失败率突破阈值，熔断器打开，阻断所有请求，执行降级兜底；

3. 进入熔断休眠期，等待服务自动恢复；

4. 休眠结束，进入半开状态，放行少量探测请求；

5. 探测成功则恢复正常，探测失败继续熔断，循环往复。

### 3.3.4 熔断器的实现：Hystrix、Sentinel的熔断器模型

- **Hystrix熔断器**：基于滑动窗口统计异常率，支持线程池隔离，状态流转经典，但不支持动态配置，现已停止更新；

- **Sentinel熔断器（生产首选）**：基于慢调用比例、异常比例、异常数三种规则触发熔断，支持动态阈值、实时监控、灵活配置，适配云原生微服务架构。

### 3.3.5 状态流转的配置参数：失败阈值、恢复时间、探测请求数

生产核心配置参数（大厂通用标准）：

- **异常比例阈值**：默认50%，1秒内异常请求占比超过50%触发熔断；
- **熔断休眠时间**：默认5秒，熔断开启后等待5秒进入半开探测；
- **最小请求数**：默认10次，统计周期内请求数低于阈值不触发熔断，避免误判；
- **半开探测请求数**：默认少量流量探测，避免大量请求打垮恢复中的服务。

# 4. 服务雪崩防护：超时控制、舱壁模式、降级预案

限流、熔断、降级是**被动流量防护**，而超时控制、舱壁模式是**主动故障隔离**，三者结合构成完整的服务雪崩防护体系。微服务雪崩的根本原因是：故障线程不释放、资源被永久占用、故障持续扩散。本章讲解的三种手段，核心目的就是**快速释放资源、彻底隔离故障、预设兜底方案**，从根源杜绝服务级联故障。

## 4.1 超时控制

超时控制是雪崩防护的**第一道基础屏障**，所有微服务远程调用必须配置超时时间，无超时配置的远程调用是生产环境最大的稳定性隐患，也是绝大多数线程阻塞、服务卡死问题的根源。

### 4.1.1 超时控制的定义：设置依赖调用的最大等待时间，超时则终止请求

超时控制指在**远程服务调用、第三方接口请求、数据库查询**等所有阻塞式调用中，预先设置最大允许等待时长。当调用耗时超过预设阈值时，系统主动终止当前请求链路，抛出超时异常，不再无限阻塞等待，强制释放当前业务线程资源。

核心本质：**不允许业务线程无限等待，卡死资源**。

### 4.1.2 超时控制的作用：避免请求长时间阻塞占用资源

在微服务调用链路中，若下游服务卡顿、数据库慢查询、网络抖动，无超时控制会导致：

- 大量业务线程持续阻塞，无法回收，线程池快速耗尽；

- 服务无可用线程处理新请求，接口大面积超时、报错；

- 上游调用方持续重试，形成重试风暴，逐级打爆整条链路服务，最终触发雪崩。

**超时控制核心价值**：快速释放阻塞线程，止损故障链路，保证服务资源可复用，从根源阻断故障扩散。

### 4.1.3 超时时间的设置原则：业务可接受的最大延迟、依赖服务的P99响应时间

超时时间绝对不能凭感觉设置，生产环境遵循两大核心原则，兼顾用户体验与系统稳定性：

1. **基于P99响应时间设置（核心原则）**：统计依赖服务近7天P99响应耗时，超时时间设置为 P99耗时 × 1.5~2倍，覆盖99%的正常业务响应，避免正常流量误超时；

2. **基于业务可接受延迟设置**：前端用户可感知的最大等待时长一般为1~3秒，核心接口超时不超过1.5s，非核心接口不超过3s；

3. **禁止过长超时**：超时时间禁止设置10s、30s等超大值，失去资源释放意义；

4. **禁止过短超时**：避免正常峰值流量下大量误超时，引发业务故障。

**生产通用标准**：Dubbo接口默认1s，HTTP接口默认1.5s，第三方接口默认3s。

### 4.1.4 超时控制的实现：HTTP客户端超时配置、Dubbo超时配置

所有远程调用必须强制配置超时，以下为生产最常用的完整可落地配置：

**1. Dubbo超时配置（服务调用核心）**

```yaml
# dubbo 全局超时配置
dubbo:
  consumer:
    # 全局调用超时时间：1000ms
    timeout: 1000
    # 失败重试次数：核心业务禁止重试，非核心1次重试
    retries: 0

```

支持接口级单独配置，优先级高于全局配置，适配不同接口耗时差异。

**2. HTTP客户端超时配置（RestTemplate/OkHttp）**

```java
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.http.client.ClientHttpRequestFactory;
import org.springframework.http.client.OkHttp3ClientHttpRequestFactory;
import org.springframework.web.client.RestTemplate;

import java.util.concurrent.TimeUnit;

/**
 * RestTemplate超时配置
 * 连接超时、读取超时、写入超时全部手动指定
 */
@Configuration
public class RestTemplateConfig {

    @Bean
    public RestTemplate restTemplate() {
        OkHttp3ClientHttpRequestFactory factory = new OkHttp3ClientHttpRequestFactory();
        // 连接超时：500ms
        factory.setConnectTimeout(500);
        // 读取超时：1500ms
        factory.setReadTimeout(1500);
        // 写入超时：1500ms
        factory.setWriteTimeout(1500);
        return new RestTemplate(factory);
    }
}

```

### 4.1.5 超时控制的异常处理：超时重试、降级返回

超时不是终点，必须配套完整的异常处理机制，否则会大面积报错：

- **瞬时超时（网络抖动）**：非核心业务配置1次重试，禁止多次重试避免风暴；核心业务**禁止重试**，防止流量放大；

- **持续超时（服务故障）**：触发熔断机制，停止调用下游服务；

- **最终兜底**：所有超时请求统一走降级逻辑，返回友好兜底数据，不抛出异常堆栈。

**生产避坑**：超时时间 + 重试总耗时，必须小于接口最大可容忍耗时，避免链路超时叠加。

## 4.2 舱壁模式（Bulkhead Pattern）

舱壁模式是**故障隔离的核心设计模式**，源自船舶舱壁设计：船舶分割为多个独立舱室，单个舱室进水不影响整船沉没。对应到服务架构，就是不同业务、不同依赖占用独立资源，实现故障物理隔离。

### 4.2.1 舱壁模式的定义：为不同依赖/业务分配独立的资源池（线程池）

舱壁模式核心定义：将服务内部的调用资源进行**物理隔离**，为不同的下游依赖服务、不同优先级的业务接口分配独立的线程池或信号量资源。

不同业务的资源相互独立、互不占用，A依赖故障阻塞线程，只会耗尽A专属线程池，不会影响B、C业务的正常执行，彻底实现故障隔离。

### 4.2.2 舱壁模式的核心价值：隔离故障，避免单个依赖故障拖垮整个服务

无舱壁隔离的致命问题：服务所有接口共用一个全局线程池，任意一个下游依赖故障阻塞线程，会耗尽整个服务的所有线程，导致**服务所有接口全部不可用**。

**舱壁模式核心价值**：

- 单点故障局部化，故障只影响对应业务，不扩散全局；

- 核心业务线程资源独立，永远优先保障核心业务可用；

- 彻底解决「一业务挂、全服务挂」的雪崩问题。

### 4.2.3 舱壁模式的实现：线程池隔离、信号量隔离

生产环境舱壁模式有两种标准实现，分别适配不同业务场景：

**1. 线程池隔离（物理隔离）**

为每个下游服务、每个核心业务单独创建独立线程池，资源完全独立。调用下游A服务使用A线程池，调用下游B服务使用B线程池，互不干扰。

特点：隔离级别最高、稳定性最强、支持超时、支持异步回调，但资源开销较大。

**2. 信号量隔离（逻辑隔离）**

不创建独立线程池，基于Semaphore信号量控制最大并发数，限制某类业务的最大同时请求数，共用主线程池。

特点：开销极低、性能更高、实现简单，但**不支持超时、不支持异步**，隔离力度弱于线程池。

### 4.2.4 线程池隔离vs信号量隔离的对比与选型

| 对比维度 | 线程池隔离                         | 信号量隔离                       |
| -------- | ---------------------------------- | -------------------------------- |
| 隔离级别 | 物理级强隔离                       | 逻辑级弱隔离                     |
| 资源开销 | 较高，需维护多组线程池             | 极低，无额外线程开销             |
| 超时支持 | 支持线程超时终止                   | 不支持超时                       |
| 异步能力 | 支持异步调用                       | 仅支持同步调用                   |
| 适用场景 | 核心业务、慢接口、不稳定第三方依赖 | 快接口、高并发查询、稳定内部依赖 |

### 4.2.5 舱壁模式的生产实践：Hystrix线程池隔离、Sentinel资源隔离

**1. Hystrix 舱壁实现**

Hystrix原生基于**线程池隔离**实现舱壁模式，可为不同服务分组配置独立线程池大小，是经典的舱壁落地方案，隔离性极强，但线程切换开销较大。

**2. Sentinel 舱壁实现（生产首选）**

Sentinel提供**线程数隔离**（信号量隔离），直接限制资源最大并发数，无需手动维护线程池，轻量化、高性能、动态可调，适配云原生高并发场景，目前主流微服务全部采用Sentinel实现舱壁隔离。

**核心配置逻辑**：为订单、支付、通知、日志等不同业务资源设置不同最大并发数，限制资源上限，避免资源耗尽。

## 4.3 降级预案设计

限流、熔断、隔离是「技术防护」，降级预案是「业务兜底」。生产故障无法百分百避免，必须提前预设完整的故障处理方案，实现故障有预案、降级不慌乱、用户无感知。

### 4.3.1 降级预案的定义：预先设计的故障场景处理方案

降级预案是针对系统流量突增、下游服务故障、数据库压力过载、节点宕机等各类异常场景，**提前预设的标准化业务处理方案与恢复流程**。

其核心思想是：故障发生时不依赖人工临时处理，自动按照预设预案执行降级、限流、关停非核心业务，最大限度保障系统核心可用性。

### 4.3.2 降级预案的分级：一级降级（核心功能保留）、二级降级（非核心功能关闭）

生产环境统一采用**两级降级规范**，精准权衡可用性与功能完整性：

**1. 一级降级（轻度降级）**

系统压力较小、局部依赖异常时触发，**100%保留核心业务**，仅对非核心逻辑做兜底处理：

- 商品详情缓存兜底、忽略推荐、弹窗、热榜等附属数据；

- 订单流程正常执行，关闭订单日志、积分发放、消息推送；

- 保留支付、下单、查询核心链路。

**2. 二级降级（重度降级）**

系统压力极大、核心依赖故障、集群过载时触发，**彻底关闭所有非核心功能**，仅保留基础核心能力：

- 关闭所有活动、推荐、搜索、评论、分享功能；

- 仅保留用户登录、下单、支付、订单查询核心链路；

- 所有附属业务统一返回兜底数据，最大限度释放资源。

### 4.3.3 降级预案的触发条件：流量突增、依赖故障、系统压力超标

降级预案支持**自动触发+人工触发**双模式，触发条件标准化：

- **流量维度**：接口QPS、并发数持续超过阈值，流量突增3倍以上；

- **依赖维度**：下游服务失败率、超时率持续高于50%，熔断器持续打开；

- **系统维度**：CPU、内存、线程池、数据库连接池使用率持续超标；

- **人工维度**：大促峰值、故障排查期间，人工手动开启降级预案。

### 4.3.4 降级预案的内容：降级逻辑、恢复流程、监控告警

一份标准生产降级预案必须包含四大要素，缺一不可：

1. **降级逻辑**：明确哪些接口降级、降级返回规则、关闭哪些功能、兜底方案是什么；

2. **触发条件**：明确阈值、触发场景、自动/人工触发方式；

3. **恢复流程**：自动恢复时间、人工恢复操作步骤、恢复校验标准；

4. **监控告警**：降级触发后的告警方式、告警接收人、故障排查优先级。

### 4.3.5 降级预案的演练：压测、故障注入演练

预案不演练等于无预案，生产环境必须定期做降级演练，避免预案失效：

- **压测演练**：通过压测模拟大促峰值流量，验证限流降级规则是否生效、核心业务是否可用；

- **故障注入演练**：手动模拟下游超时、宕机、报错，验证熔断触发、降级兜底是否正常执行；

- **常态化演练**：月度常规演练、大促前全量演练，保证预案100%可用。

# 5. 限流降级的生产环境配置与监控

限流熔断的难点不在于代码实现，而在于**生产合理配置、精准监控、动态运维**。错误的阈值配置、缺失的监控告警、无迭代优化，会导致防护失效或误杀正常流量，本节统一规范生产级配置标准、监控指标、告警体系与运维最佳实践。

## 5.1 限流降级配置规范

所有限流、熔断、降级配置必须遵循统一生产规范，拒绝随意配置、硬编码配置，保证可监控、可动态调整、可灰度回滚。

### 5.1.1 限流配置：阈值设置、限流算法、限流维度

**1. 限流算法选型规范**

- 通用业务接口：优先**令牌桶算法**，支持合法突发流量；

- 匀速异步任务、第三方调用：使用**漏桶算法**，强制流量平滑；

- 精准风控场景：使用**滑动窗口**，杜绝临界流量漏洞。

**2. 阈值设置规范**

- 基于单机压测最大QPS，预留20%~30%冗余，不打满机器性能；

- 集群阈值 = 单机阈值 × 实例数，分布式限流统一按集群总阈值配置；

- 核心接口阈值宽松、非核心接口阈值严格，差异化配置。

**3. 限流维度规范**

对外接口：必须配置「IP限流 + 用户限流 + 接口QPS限流」多维度组合防护；对内接口：以服务QPS、并发限流为主，简化配置。

### 5.1.2 熔断配置：失败率阈值、恢复时间、探测请求数

生产通用熔断标准配置（大厂通用）：

- **统计窗口时长**：1秒，实时统计瞬时故障；

- **最小请求数**：10次，避免少量报错误触发熔断；

- **失败率阈值**：50%，1秒内超半数请求失败即熔断；

- **熔断休眠恢复时间**：5秒，熔断后5秒进入半开探测；

- **半开探测请求数**：少量流量探测，避免大量请求冲击恢复中服务。

### 5.1.3 降级配置：降级策略、降级触发条件、降级恢复条件

- **降级策略**：查询类接口优先缓存兜底，业务类接口优先关闭非核心逻辑；

- **触发条件**：熔断打开、超时率过高、系统CPU/线程池过载；

- **恢复条件**：连续多次探测请求成功、系统压力回落、下游服务恢复正常。

### 5.1.4 配置的动态调整：配置中心（Nacos/Apollo）集成

生产**禁止硬编码限流熔断阈值**，所有规则统一托管配置中心：

- 基于Nacos/Apollo存储限流阈值、熔断参数、降级开关；

- 支持动态修改、实时生效，无需重启服务；

- 大促前可临时调高阈值，低谷期收紧阈值，适配流量动态变化。

### 5.1.5 配置的灰度发布与回滚机制

限流规则属于核心防护规则，修改存在业务风险，必须支持灰度与回滚：

- **灰度发布**：先更新单实例规则，验证无误后全量推送；

- **版本记录**：所有配置修改留痕，记录修改人、修改时间、修改内容；

- **一键回滚**：配置异常、误杀流量时，支持秒级回滚至上一稳定版本。

## 5.2 限流降级监控指标

无监控的防护等于裸奔，生产环境必须全覆盖监控指标，实现故障提前发现、问题精准定位。

### 5.2.1 限流监控指标：限流请求数、限流率、被限流的接口/用户

核心限流监控指标清单：

- **总请求QPS**：接口实时总流量；

- **限流请求数**：单位时间内被拦截的请求总量；

- **限流率**：限流数/总请求数，核心预警指标；

- **限流维度明细**：被限流的IP、用户ID、接口路径，用于排查恶意访问、热点流量。

### 5.2.2 熔断监控指标：熔断器状态、失败率、恢复探测结果

- **熔断器状态**：实时展示关闭/打开/半开三种状态；

- **接口失败率、超时率**：熔断触发的核心依据；

- **熔断触发次数**：单位时间内熔断触发频次；

- **探测请求成功率**：判断服务恢复情况。

### 5.2.3 降级监控指标：降级请求数、降级成功率、降级触发场景

- **降级请求数**：单位时间内兜底降级的请求总量；

- **降级成功率**：降级兜底逻辑执行成功比例，排查降级bug；

- **降级触发场景**：区分流量过载、下游故障、系统压力触发。

### 5.2.4 系统整体指标：服务可用性、响应时间、错误率

通过整体指标判断防护体系是否生效：

- **服务可用率**：核心业务可用性是否维持99.99%；

- **P95/P99响应时间**：限流熔断后接口延迟是否平稳；

- **全局错误率**：是否无大面积500、超时异常。

## 5.3 监控告警配置

监控用于发现问题，告警用于及时处理问题，生产必须配置分级告警机制，区分故障严重程度。

### 5.3.1 限流告警：限流率过高、核心接口限流触发

**告警规则**：

- 核心接口限流率持续超过10%触发告警，说明流量异常或阈值过小；

- 瞬时大量限流触发，预警恶意刷量、爬虫攻击。

### 5.3.2 熔断告警：熔断器打开、长时间处于打开状态

**告警规则**：

- 熔断器由关闭变为打开，立即告警，代表下游服务出现故障；

- 熔断器持续打开超过30秒，升级告警，代表服务长时间未恢复。

### 5.3.3 降级告警：降级请求数突增、核心业务降级触发

- 非核心业务降级：普通告警，仅做通知；

- 核心业务（订单、支付）触发降级：紧急告警，立即排查处理。

### 5.3.4 告警分级：一级告警（核心业务故障）、二级告警（非核心业务降级）

- **一级紧急告警**：核心接口限流、核心业务降级、熔断器长时间打开，电话+短信+企业微信多级通知，必须立即处理；

- **二级普通告警**：非核心业务降级、少量限流、短时熔断，仅消息通知，定时巡检处理。

## 5.4 限流降级的运维实践

限流熔断不是配置完就结束，需要持续运维、验证、迭代优化，适配业务流量迭代变化。

### 5.4.1 限流降级的压测验证：模拟高流量、依赖故障场景

上线前必须全量压测验证：

- 模拟超阈值流量，验证限流精准拦截、无误杀、无漏放；

- 模拟下游故障，验证熔断自动触发、降级兜底正常返回；

- 压测后观察CPU、线程池、响应时间，确认防护有效。

### 5.4.2 故障演练：主动触发限流/熔断，验证防护效果

遵循「混沌工程」思想，主动制造故障验证防护能力：

- 手动调低阈值触发限流，验证监控、告警、拦截逻辑；

- 手动模拟下游超时报错，验证熔断状态流转、降级兜底；

- 验证故障场景下核心业务可用性，确保防护体系有效。

### 5.4.3 限流降级规则的迭代优化：基于监控数据调整阈值

流量随业务迭代持续变化，限流规则不能一成不变，需定期迭代：

- 基于近7天P99流量、峰值流量动态调整限流阈值；

- 根据报错率、超时率优化熔断阈值、超时时间；

- 根据降级触发频次，优化降级预案，减少不必要降级；

- 大促前专项优化规则，适配峰值流量。
