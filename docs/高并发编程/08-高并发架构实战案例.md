# 1. 秒杀系统设计：流量削峰、库存扣减、防超卖、防刷

秒杀系统是高并发标杆场景，核心矛盾是**有限库存 + 无限瞬时流量**。普通业务流量平缓可控，而秒杀活动会在几秒内涌入数万甚至数十万QPS的请求，极易造成服务阻塞、数据库锁竞争、超卖、系统雪崩。整套秒杀架构的设计核心思想：**分层拦截、逐级削峰、异步解耦、缓存兜底、风控前置、最终一致**。

## 1.1 秒杀系统核心挑战

正式架构设计前，必须明确秒杀场景的四大核心痛点，所有技术方案均围绕以下问题针对性解决。

### 1.1.1 流量突增：短时间内大量请求涌入

秒杀活动开启瞬间，用户集中点击抢购，会产生**瞬时流量洪峰**，流量是日常流量的几十倍甚至上百倍。若不做拦截，海量请求会直接穿透到应用、数据库，导致CPU打满、线程堆积、数据库连接耗尽，引发整体服务不可用。

**核心问题**：流量峰值远超系统承载上限，同步处理无法应对瞬时并发。

**解决思路**：不硬碰硬，通过多级拦截、排队、异步削峰，将瞬时洪峰流量抹平为平稳流量。

### 1.1.2 库存控制：防止超卖、保证库存一致性

秒杀商品库存有限且数值极小（常见几十、几百、几千库存），海量并发请求同时执行库存扣减，会出现**并发超卖、库存负数、库存不一致**问题。数据库普通减库存SQL无原子防护，多线程并发下会出现覆盖更新，导致实际售出数量大于真实库存，引发资损故障。

**核心问题**：高并发下库存扣减不具备原子性，读写分离、普通事务无法解决竞争问题。

**解决思路**：缓存预扣减 + 原子操作 + 分布式锁/乐观锁兜底，保障库存精准可控。

### 1.1.3 防刷防作弊：恶意请求、脚本攻击

真实业务中，**90%以上秒杀请求为脚本、爬虫、作弊请求**。黑产通过脚本批量请求、多IP刷取、设备伪造、高频重试等方式抢占库存，导致普通用户无法抢购，同时无效流量进一步放大系统压力。

**核心问题**：无效恶意流量挤占正常流量，恶意竞争导致活动公平性失效、系统压力倍增。

**解决思路**：风控前置，多层校验拦截恶意请求，只放行正常用户流量。

### 1.1.4 系统稳定性：避免秒杀拖垮整个平台

秒杀属于**非核心高风险业务**，一旦流量失控，会占用服务器CPU、内存、数据库连接、网络带宽等全局资源，导致用户中心、订单中心、支付中心等核心业务卡顿雪崩，造成全站故障。

**核心问题**：单一高并发业务引发全局服务降级、雪崩效应。

**解决思路**：服务隔离、流量隔离、熔断降级、快速失败，保障秒杀故障不扩散。

## 1.2 流量削峰方案

流量削峰是秒杀架构的第一道防线，核心思想是**能前置拦截绝不后置处理，能拒绝绝不排队，能异步绝不同步**，通过多层架构逐级消化流量，避免流量直接打穿数据层。

### 1.2.1 前端限流：按钮置灰、验证码、排队页面

前端限流是**最低成本、最高收益**的第一道拦截，拦截大量无效重复请求，减轻后端压力。

**核心实现方案**

- **按钮置灰防重复点击**：用户点击秒杀按钮后，立即置灰锁定，禁止重复提交，防止用户手动频繁点击产生重复请求；

- **动态验证码/滑块验证**：秒杀开启瞬间触发验证码校验，拦截机器脚本无脑请求，增加作弊成本；

- **排队页面**：流量超阈值时跳转排队页面，请求排队等待，限流不降权，提升用户体验；

- **静态资源CDN**：秒杀页面静态化，托管至CDN，避免大量页面请求打到业务服务器。

**优缺点**：性能零损耗、拦截量大，但可被脚本绕过，仅能拦截普通用户误操作与浅层爬虫。

### 1.2.2 网关层限流：Nginx限流、IP限制

网关层是后端流量的统一入口，适合做**全局流量限流、IP限流、频次限流**，拦截批量爬虫、高频恶意IP。

**核心限流策略**

- **Nginx漏桶限流**：限制单IP每秒请求次数，平滑流量，避免瞬时脉冲流量；

- **黑名单拦截**：运维配置恶意IP黑名单，直接403拦截；

- **网关全局QPS限流**：设置秒杀接口最大总QPS，超过阈值直接返回“活动繁忙，请稍后重试”，快速失败。

**生产价值**：在请求未进入应用服务前拦截大量恶意流量，保护后端服务稳定。

### 1.2.3 应用层削峰：消息队列异步排队

经过前端、网关拦截后的剩余流量，依然可能超出应用同步处理能力，此时通过**消息队列实现异步削峰**，是秒杀架构最核心的削峰手段。

**核心原理**

同步模式下，请求需要全程阻塞等待库存扣减、订单创建、数据库落库，吞吐量极低；异步模式下，服务接收请求后，快速校验合法性，将秒杀请求投递至MQ队列，立即返回排队成功，消费者匀速消费队列数据，完成扣库存与下单。

**核心优势**：将瞬时数万QPS的洪峰，抹平为每秒几百QPS的平稳流量，彻底解决流量冲击问题。

**代码落地思路**

```java
/**
 * 秒杀请求入口 - 异步削峰实现
 * 快速校验 + 投递MQ，快速响应，不阻塞用户
 */
@RestController
@RequestMapping("/seckill")
public class SeckillController {

    @Autowired
    private RabbitTemplate rabbitTemplate;

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    @PostMapping("/do")
    public Result seckill(Long goodsId, Long userId) {
        // 1. 快速前置校验：活动是否开启、库存是否充足
        Boolean hasStock = redisTemplate.hasKey("seckill:stock:" + goodsId);
        if (!hasStock) {
            return Result.fail("秒杀库存不足或活动未开启");
        }
        // 2. 校验用户是否已秒杀（一人一单限制）
        String userRecord = redisTemplate.opsForValue().get("seckill:user:" + goodsId + ":" + userId);
        if (Objects.nonNull(userRecord)) {
            return Result.fail("您已参与过本次秒杀，请勿重复抢购");
        }
        // 3. 合法请求投递MQ，异步处理下单
        SeckillMessage message = new SeckillMessage();
        message.setGoodsId(goodsId);
        message.setUserId(userId);
        rabbitTemplate.convertAndSend("seckill_exchange", "seckill.routing", message);
        // 4. 立即返回排队成功，削峰核心
        return Result.success("秒杀排队成功，正在处理中...");
    }
}

```

### 1.2.4 多级削峰架构：前端→网关→应用→消息队列

生产标准秒杀**四级削峰架构**，层层过滤、层层减压，是高并发秒杀的标准架构范式：

1. **前端层削峰**：防重复点击、验证码、CDN静态化、排队页，拦截普通无效请求；

2. **Nginx网关层削峰**：IP频次限流、全局QPS限流、黑名单拦截，拦截爬虫恶意流量；

3. **应用层前置削峰**：Redis库存预校验、用户重复抢购校验，拦截无意义请求；

4. **MQ异步削峰**：剩余合法请求入队排队，异步匀速消费，彻底抹平流量洪峰。

**架构价值**：没有任何单一层面扛住全部流量，通过分层分担压力，实现系统稳如泰山。

## 1.3 库存扣减与防超卖

库存超卖是秒杀最致命的生产故障，会直接造成公司资金损失。本节讲解四种生产级防超卖方案，从缓存预扣到数据库兜底，实现**绝对无超卖、库存精准一致**。

### 1.3.1 库存预扣减：Redis预减库存，异步落库

数据库读写性能极低，无法支撑秒杀瞬时并发，因此秒杀核心库存操作**全部转移至Redis执行**。

**核心流程**

1. 秒杀活动预热：将商品库存数量提前写入Redis缓存；

2. 用户抢购请求：先在Redis执行**原子递减库存**；

3. 库存扣减成功：允许进入下单流程；

4. 库存不足：直接返回秒杀结束；

5. 异步通过MQ将Redis库存变动同步至MySQL，实现最终一致。

**原子扣减代码（Lua脚本，保证原子性）**

```lua
-- 秒杀库存原子扣减Lua脚本
-- key：商品库存key
-- return：1成功 0库存不足
local key = KEYS[1]
-- 判断库存是否存在且大于0
local stock = tonumber(redis.call('get', key) or 0)
if stock > 0 then
    -- 库存-1
    redis.call('decr', key)
    return 1
end
return 0

```

**优势**：Redis单线程模型，Lua脚本保证扣减原子性，性能百万级，彻底解决并发扣减性能瓶颈。

### 1.3.2 乐观锁控制：数据库版本号/条件更新防超卖

Redis预扣库存是前置拦截，数据库层需要兜底防超卖，**数据库乐观锁**是最常用的兜底方案，无锁、高性能。

**方案一：条件更新乐观锁（生产首选）**

```sql
-- 库存扣减SQL，仅库存大于0时才扣减，天然防超卖
UPDATE seckill_goods 
SET stock = stock - 1 
WHERE id = #{goodsId} AND stock > 0;

```

原理：并发场景下，只有查询条件匹配成功的请求才能扣减成功，库存为0后所有请求直接失败，不会出现负数超卖。

**方案二：版本号乐观锁**

```sql
UPDATE seckill_goods 
SET stock = stock - 1, version = version + 1 
WHERE id = #{goodsId} AND version = #{oldVersion} AND stock > 0;

```

**优缺点**：条件更新更简洁高效，版本号乐观锁适合需要精准控制数据版本的场景。

### 1.3.3 分布式锁控制：Redis分布式锁保证扣减原子性

针对复杂库存业务、多步骤扣减场景，可通过**Redis分布式锁**保证单个商品同一时间只有一个请求操作库存，彻底杜绝并发竞争。

**核心逻辑**

以商品ID为锁Key，抢到锁的请求执行库存扣减与下单，未抢到锁的请求直接排队或失败，串行化处理库存竞争逻辑。

**适用场景**：多级库存、组合商品库存、复杂扣减逻辑，简单秒杀优先使用Lua原子扣减，无需过度加锁。

### 1.3.4 库存扣减流程设计：预扣减→订单创建→支付确认→最终扣减

生产标准**四段式库存生命周期**，解决秒杀未付款导致库存冻结、库存浪费问题：

1. **Redis预扣减**：抢购成功，缓存库存-1，锁定库存资源；

2. **创建秒杀订单**：生成待支付订单，标记库存占用；

3. **支付确认**：用户完成支付，数据库正式扣减库存，完成交易；

4. **超时释放**：订单超时未支付，自动回滚Redis库存、取消订单，释放库存资源。

**核心价值**：兼顾并发性能与库存利用率，避免秒杀抢到不付款导致库存冻结、活动亏损。

## 1.4 防刷防作弊方案

秒杀防刷的核心思想：**前置风控、多层校验、差异化拦截**，区分正常用户与黑产脚本，在不影响普通用户体验的前提下，拦截恶意流量。

### 1.4.1 用户校验：登录态校验、用户等级限制

**基础校验规则**

- **强制登录**：未登录用户直接拦截，禁止参与秒杀，拦截匿名脚本流量；

- **一人一单限制**：基于用户ID做Redis幂等限制，同一用户仅限抢购一次，防止多刷；

- **用户等级过滤**：新注册用户、低活跃度用户限制参与，拦截批量注册小号作弊。

### 1.4.2 IP限制：单IP请求频率限制

黑产脚本通常单IP高频批量请求，通过IP频次限流可有效拦截：

- 限制**单IP每秒最大请求数**，超过阈值直接拦截；

- 短时高频IP加入临时黑名单，冷却后自动解除；

- 针对代理IP、跳板机IP做特征拦截。

**实现方式**：Redis + 计数器，记录IP访问频次，轻量高效。

### 1.4.3 设备指纹：客户端设备信息校验

单纯IP限制可通过换IP绕过，**设备指纹**是更精准的客户端标识，对应真实设备，作弊成本极高。

**实现原理**：客户端采集设备ID、设备型号、系统版本、屏幕参数等信息，加密生成唯一设备指纹，请求携带指纹上报服务端，服务端校验设备合法性、设备请求频次，拦截批量模拟器、脚本设备。

### 1.4.4 行为分析：异常请求模式识别、风控系统拦截

高级黑产可模拟正常账号、正常IP，需要通过**行为特征**识别作弊请求：

- **请求速度异常**：秒杀开启瞬间毫秒级请求，远超正常用户操作速度，判定为脚本；

- **请求链路异常**：跳过首页、验证码、前置接口，直接请求秒杀接口，判定为爬虫；

- **风控引擎拦截**：对接平台风控系统，基于大数据模型识别恶意账号、设备、IP，精准拦截作弊流量。

## 1.5 秒杀系统整体架构

整合前文限流、削峰、防超卖、防刷能力，形成完整的生产级秒杀分层架构，具备高并发、高可用、高安全、最终一致的特性。

### 1.5.1 架构分层：前端层、网关层、应用层、数据层

**四层标准秒杀架构**

1. **前端静态层**：页面静态化、CDN加速、按钮防重、验证码、排队页，拦截浅层流量；

2. **网关限流层**：Nginx/网关全局限流、IP黑名单、频次拦截、流量监控；

3. **业务应用层**：参数校验、用户风控、Redis库存预扣、幂等判断、MQ异步削峰；

4. **数据持久层**：数据库乐观锁兜底、订单落库、库存异步同步、超时补偿回滚。

### 1.5.2 核心流程：用户请求→限流→排队→扣减库存→创建订单

完整秒杀执行链路（生产标准流程）：

1. 用户访问秒杀页面，CDN返回静态资源，前端拦截重复点击；

2. 用户发起抢购请求，网关层校验IP频次、黑名单，拦截恶意流量；

3. 应用层校验登录态、用户资格、是否重复抢购；

4. Redis Lua原子脚本预扣库存，库存不足直接返回失败；

5. 抢购成功请求进入MQ队列，异步排队处理；

6. 消费者消费消息，创建待支付订单，数据库乐观锁扣减库存；

7. 用户支付完成，订单状态变更，库存正式生效；

8. 超时未支付，自动回滚Redis库存、关闭订单。

### 1.5.3 数据一致性保障：最终一致性、异步补偿

秒杀高并发场景不适合强一致性，生产采用**最终一致性方案**，性能与一致性兼顾：

- 缓存库存与数据库库存允许短暂不一致，通过异步MQ同步数据；

- 定时任务巡检库存差异，自动补偿修正数据；

- 订单超时、支付异常场景统一回滚兜底；

- 核心兜底：数据库条件更新兜底防超卖，保证绝对数据正确。

### 1.5.4 性能优化：热点key缓存、静态资源CDN

秒杀专属性能优化手段，支撑十万级并发：

- **热点Key缓存优化**：秒杀商品为超级热点Key，本地缓存+Redis分布式缓存多级缓存，避免缓存穿透、缓存击穿；

- **页面全静态化+CDN**：秒杀页面无动态渲染，全部静态资源托管CDN，降低源站压力；

- **接口轻量化**：秒杀前置校验全部轻量化，无复杂业务逻辑，快速失败、快速响应；

- **服务隔离部署**：秒杀服务独立集群、独立数据库，避免影响核心业务。

# 2. 订单系统高并发优化：异步处理、状态机、幂等性设计

订单系统是电商平台的核心链路，几乎所有营销、交易、支付、物流业务都围绕订单展开。相比于秒杀瞬时洪峰流量，订单系统属于**持续性高并发+复杂状态流转+跨服务一致性**场景，优化重点不再只是削峰，而是解决响应速度、状态可控、请求幂等、数据一致四大核心问题。

## 2.1 订单系统核心挑战

在高并发场景下，传统同步订单架构会暴露大量性能与数据问题，本节梳理生产环境订单系统四大核心痛点，所有后续优化方案均针对性解决对应问题。

### 2.1.1 高并发下单：大量用户同时创建订单

大促、秒杀、活动场景下，数万用户同时发起下单请求，传统同步架构需要依次完成订单入库、库存扣减、日志记录、消息推送等一系列同步操作，单次下单耗时高、线程阻塞严重，极易导致：接口响应超时、服务线程堆积、下单失败率飙升、数据库连接耗尽等问题，无法支撑大促流量。

### 2.1.2 订单状态流转：创建、支付、发货、完成、关闭

订单并非静态数据，而是存在**完整的生命周期流转**，包含待支付、已支付、已发货、已完成、已关闭等多个状态，每个状态都有固定的前置条件、触发动作和后置逻辑。传统硬编码状态判断方式，极易出现状态跳转混乱、非法状态流转、重复状态更新等问题，导致订单数据异常、资损、用户纠纷。

### 2.1.3 幂等性保障：重复请求、重复支付

网络重试、前端重复点击、网关重试、支付回调重试等场景，会产生大量重复请求。若无幂等设计，会出现**重复创建订单、重复扣减库存、重复支付、重复发货、重复退款**等严重生产故障，直接引发资金与商品资损。幂等性是订单系统必须保障的基础能力。

### 2.1.4 数据一致性：订单、库存、支付状态同步

下单流程涉及订单服务、库存服务、支付服务、物流服务多服务联动，分布式场景下网络超时、服务宕机、重试异常频发，极易出现：订单创建成功但库存未扣、支付成功订单未更新、订单关闭库存未回滚等**数据状态不一致**问题，是分布式订单系统的核心难点。

## 2.2 异步处理方案

订单高并发优化的核心手段之一就是**全链路异步化**，核心思想：剥离非核心阻塞逻辑，仅保留核心下单校验与单号生成逻辑同步执行，将耗时、非强实时的操作异步处理，大幅缩短下单响应时长，提升系统并发吞吐量。

### 2.2.1 异步订单创建：同步创建订单号，异步落库

**核心原理**：订单号是用户下单成功的唯一标识，生成规则简单、速度极快，同步执行；而订单数据入库、关联数据写入、日志记录等耗时操作，通过MQ异步完成。

**执行流程**

1. 用户发起下单请求，服务端完成参数校验、库存预校验、用户权限校验；

2. **同步生成唯一订单号**，立即返回用户下单成功，展示待支付页面；

3. 将订单基础信息封装消息，投递至消息队列；

4. 消费者异步消费消息，完成订单数据库落库、关联数据初始化。

**生产避坑**：需做好订单幂等与空订单兜底，防止异步落库失败导致用户有单号、无订单数据的异常。

### 2.2.2 异步库存扣减：订单创建后异步扣减库存

传统同步扣减库存需要锁数据库、执行SQL，IO耗时极高，严重拖慢下单速度。高并发架构中采用**预扣+异步落库**方案。

**核心流程**

1. 下单阶段：Redis原子预扣库存，锁定库存资源，防止超卖；

2. 同步返回下单成功，无需等待数据库库存更新；

3. 异步MQ消费，执行数据库真实库存扣减、库存流水记录；

4. 订单超时未支付，异步回滚Redis与数据库库存。

**价值**：彻底消除数据库库存锁竞争带来的耗时，支撑超高并发下单。

### 2.2.3 异步通知推送：订单状态变更后异步发送通知

订单状态变更（创建、支付、发货、完成）后，需要推送短信、APP推送、小程序模板消息、站内信等通知，这类业务**非核心、强耗时、允许延迟**，必须全部异步化。

**实现方案**：订单状态更新成功后，发送状态变更消息至MQ，专门的消息通知服务消费消息，统一处理各类推送逻辑，即使推送失败也不影响主订单流程，可通过重试机制兜底。

### 2.2.4 异步化对下单响应速度的提升

**同步架构耗时拆解**：参数校验(10ms) + 订单入库(20ms) + 库存扣减(30ms) + 通知推送(50ms) = **110ms+**

**异步架构耗时拆解**：参数校验 + 单号生成 = **10ms以内**

**核心提升**：下单响应速度提升10倍以上，单机吞吐量提升5~10倍，彻底解决大促下单排队、超时问题，用户体验与系统承载力双重提升。

## 2.3 订单状态机设计

订单状态机是保证订单流程**规范、可控、无错乱**的核心设计，通过标准化状态与流转规则，替代散乱的if/else硬编码，解决状态跳转混乱、非法状态变更、异常数据问题，是生产级订单系统的标准设计范式。

### 2.3.1 订单状态定义：待支付、已支付、已发货、已完成、已关闭

生产通用订单标准状态枚举，覆盖完整订单生命周期：

- **待支付(0)**：订单创建成功，未完成付款，库存处于锁定状态；

- **已支付(1)**：用户付款成功，资金已确认，等待商家发货；

- **已发货(2)**：商家已发货，等待用户确认收货；

- **已完成(3)**：用户确认收货，订单交易正常结束；

- **已关闭(4)**：订单超时未支付、主动取消、售后关闭，交易终止，库存释放。

### 2.3.2 状态流转规则：状态变更的前置条件与触发动作

所有订单状态**单向、有序流转**，禁止跨状态、逆状态跳转，标准流转规则如下：

1. 待支付 → 已支付：前置条件为订单未过期、未取消；触发动作：更新支付时间、扣减真实库存、生成支付流水；

2. 已支付 → 已发货：前置条件为支付状态正常；触发动作：生成物流单、推送发货通知；

3. 已发货 → 已完成：前置条件为物流签收；触发动作：结算佣金、更新订单完结时间；

4. 待支付 → 已关闭：前置条件为超时未支付/用户取消；触发动作：释放库存、关闭订单、清空支付链路。

### 2.3.3 状态机实现：数据库状态字段+业务逻辑控制

中小型项目主流轻量化实现方案：**数据库状态字段 + 枚举状态机 + 前置校验逻辑**，无需引入复杂框架，稳定高效。

**核心代码示例（状态枚举与流转校验）**

```java
/**
 * 订单状态枚举 - 状态机核心定义
 */
public enum OrderStatusEnum {
    WAIT_PAY(0, "待支付"),
    PAID(1, "已支付"),
    DELIVERED(2, "已发货"),
    FINISHED(3, "已完成"),
    CLOSED(4, "已关闭");

    private final Integer code;
    private final String desc;

    // 合法状态流转映射：key=当前状态，value=可跳转的状态集合
    public static final Map<Integer, Set<Integer>> STATUS_TRANSFER_MAP;

    static {
        STATUS_TRANSFER_MAP = new HashMap<>();
        // 待支付可跳转：已支付、已关闭
        STATUS_TRANSFER_MAP.put(0, Sets.newHashSet(1, 4));
        // 已支付可跳转：已发货、已关闭（售后关闭）
        STATUS_TRANSFER_MAP.put(1, Sets.newHashSet(2, 4));
        // 已发货可跳转：已完成、已关闭
        STATUS_TRANSFER_MAP.put(2, Sets.newHashSet(3, 4));
        // 已完成、已关闭不可再流转
        STATUS_TRANSFER_MAP.put(3, Sets.newHashSet());
        STATUS_TRANSFER_MAP.put(4, Sets.newHashSet());
    }

    // 校验状态是否可以流转
    public static boolean canTransfer(Integer currentStatus, Integer targetStatus) {
        return STATUS_TRANSFER_MAP.getOrDefault(currentStatus, Sets.newHashSet()).contains(targetStatus);
    }
}

```

业务更新订单前，优先调用`canTransfer`方法校验，拦截所有非法状态跳转。

### 2.3.4 状态机异常处理：状态流转失败、数据不一致

生产异常兜底方案：

- **非法状态拦截**：所有状态变更必须经过状态机校验，拦截跨状态、逆状态非法更新；

- **状态幂等更新**：更新SQL携带当前状态条件，仅匹配成功才更新，防止重复流转；

- **定时巡检补偿**：定时任务扫描异常状态订单，修复状态不一致数据；

- **异常日志记录**：所有状态流转失败记录详细日志，便于问题回溯排查。

## 2.4 幂等性设计

**幂等性**是分布式系统的基石，订单、支付、库存场景必须100%保证幂等，无论请求重试多少次，最终业务结果保持一致，无重复数据、无重复执行业务。

### 2.4.1 幂等性的定义：同一请求多次执行结果一致

幂等性指：**一个接口/操作，执行一次与执行N次，对系统产生的影响完全一致**，不会产生副作用。

**订单场景非幂等风险**：前端重复点击、网关重试、MQ重试、支付回调重试，导致重复下单、重复扣库存、重复入账、重复发货，引发资损故障。

### 2.4.2 幂等性实现方式：订单号/请求号作为幂等键

订单系统通用幂等方案：**全局唯一幂等键**，优先使用订单号、外部请求号、支付流水号作为幂等标识。

**实现逻辑**

1. 客户端/网关每次请求携带唯一幂等ID（订单号/请求流水号）；

2. 服务端处理前，先查询Redis/数据库是否已处理过该幂等ID；

3. 已处理：直接返回上次成功结果，不重复执行业务；

4. 未处理：执行业务逻辑，执行成功后缓存幂等标识。

### 2.4.3 数据库幂等控制：唯一索引、条件更新

数据库层是幂等的最后一道兜底防线，两种核心方案：

**1. 唯一索引防重复下单**

对订单号、业务唯一请求号建立数据库唯一索引，重复插入相同数据直接报错，杜绝重复订单。

**2. 条件更新防重复状态变更**

```sql
-- 仅待支付状态的订单可更新为已支付，防止重复更新支付状态
UPDATE order_info 
SET status = 1, pay_time = NOW() 
WHERE order_no = #{orderNo} AND status = 0;
```

通过状态条件限制，保证同一状态变更仅能执行一次，完美实现数据库层幂等。

### 2.4.4 支付场景幂等性：防止重复支付、重复回调

支付是资金核心场景，幂等要求最高，需双重防护：

- **下单层幂等**：一个订单仅允许创建一次，避免生成多笔相同订单；

- **支付回调幂等**：以支付流水号为幂等键，回调成功后缓存标记，重复回调直接忽略；

- **状态校验兜底**：已支付订单拒绝二次支付处理，防止重复入账；

- **对账补偿**：日终对账，修正支付与订单状态不一致数据。

## 2.5 订单系统高并发架构

针对千万级订单数据、大促高并发场景，需从存储、索引、缓存、定时任务全方位架构优化，解决数据量大、查询慢、热点集中、超时关闭不及时等问题。

### 2.5.1 订单分库分表：按用户ID/订单ID分片

订单数据属于典型海量数据，单表数据量极易突破千万，必须做**水平分库分表**。

**分片规则选型**

- **用户ID哈希分片（主流）**：同一用户所有订单落在同一分片，用户查询订单列表无需跨分片，查询性能极高；

- **订单ID分片**：数据分布极度均匀，无热点分片，但用户查询需要跨分片聚合；

- **时间范围分片**：按月份分表，适合订单归档、历史数据查询场景。

**生产最佳实践**：采用**用户ID哈希分片 + 时间归档**，兼顾查询性能与存储压力。

### 2.5.2 订单索引优化：订单号、用户ID、状态索引

订单表查询场景固定，精准建立索引，避免全表扫描：

- **唯一索引：order_no**，精准查询订单详情、防重复订单；

- **联合索引：user_id + status + create_time**，适配用户分页查询、按状态筛选订单高频场景；

- **普通索引：pay_time、close_time**，适配对账、超时订单扫描、归档统计场景。

**避坑**：禁止建立过多冗余索引，避免影响订单写入性能。

### 2.5.3 订单缓存优化：热点订单缓存、订单列表缓存

大促场景下订单查询QPS极高，通过多级缓存减轻数据库压力：

- **热点订单缓存**：用户近期待支付、已支付订单为热点数据，缓存订单详情，减少DB查询；

- **订单列表缓存**：缓存用户首页订单列表，提升页面加载速度；

- **缓存更新策略**：订单状态变更同步更新缓存，保证缓存一致性；

- **过期策略**：短期订单缓存时效短，历史归档订单不缓存，节省内存资源。

### 2.5.4 订单超时处理：定时任务关闭超时订单

待支付订单占用库存资源，必须自动超时关闭、释放库存，是订单系统必备兜底能力。

**生产实现方案（最优）：MQ延迟队列**

用户下单成功后，发送延迟30分钟的延迟消息，时间到达后触发校验：

- 订单仍为待支付：自动关闭订单、回滚库存、更新订单状态；

- 订单已支付/已关闭：直接忽略消息，不做处理。

**兜底方案**：定时任务分钟级扫描超时未关闭订单，防止MQ消息丢失导致库存死锁。

**优势**：相比于传统定时任务全表扫描，延迟队列精准触发、性能极高、无数据库压力。

本章完整落地了高并发订单系统的全套优化方案，是电商核心业务架构设计的标准模板。首先剖析了高并发下单、状态流转、幂等防护、分布式数据一致性四大核心业务痛点；其次通过全链路异步化方案，解决了同步下单响应慢、吞吐量低的问题，大幅提升系统并发承载力；接着通过标准化订单状态机设计，彻底解决订单状态错乱、非法流转、异常数据问题，让订单生命周期完全可控；然后详解了全场景幂等性设计方案，从业务层、缓存层、数据库层三重兜底，杜绝重复下单、重复支付、资损故障；最后从分库分表、索引优化、缓存优化、超时兜底四个维度，完成订单系统高并发架构落地优化。本章方案可直接复用至所有交易类业务系统，同时承接前文秒杀高并发设计思想，进一步完善了高并发业务落地体系，为后续消息推送、热点防护等实战场景铺垫基础。

# 3. 消息推送系统设计：推送通道选择、在线用户管理、推送限流

消息推送系统不同于秒杀、订单的交易型高并发，属于**广播式高并发场景**。秒杀、订单是单点用户交易，而推送系统常面临全量用户、百万级用户批量推送场景，核心难点不在于单请求耗时，而在于**海量并发IO、状态实时维护、通道适配、防骚扰、成功率保障**，是中大型互联网项目的必备基础服务。

## 3.1 消息推送系统核心挑战

批量推送、实时触达、多渠道分发场景下，传统同步推送架构会暴露大量问题，本节拆解生产环境推送系统四大核心挑战，为后续优化方案做铺垫。

### 3.1.1 高并发推送：大量用户同时推送消息

运营活动、系统通知、版本更新场景下，需要向数十万、数百万在线用户批量推送消息，瞬间产生海量推送请求。若采用同步循环推送模式，会出现服务线程被大量IO阻塞、推送耗时极长、任务堆积、超时失败等问题，无法支撑大规模批量推送。同时海量推送请求会瞬间打满第三方推送通道接口，引发限流、封禁风险。

### 3.1.2 多通道推送：APP推送、短信、邮件、站内信

业务消息类型繁多，不同消息需要适配不同推送通道：系统通知依赖APP推送、验证码依赖短信、对账通知依赖邮件、动态消息依赖站内信。多通道场景存在通道优先级混乱、重复推送、通道适配错误、推送成功率参差不齐等问题，若无统一的通道调度策略，会出现用户收不到消息、多渠道重复骚扰、资源浪费等线上问题。

### 3.1.3 在线用户管理：实时用户在线状态维护

精准推送的前提是精准识别用户在线状态：在线用户实时推送APP消息，离线用户缓存消息、上线补发。用户在线状态是动态变化的，用户随时上下线、多端切换，如何**低损耗、高实时、高精度**维护全平台用户在线状态，同时支撑百万级用户状态查询，是推送系统的核心技术难点。

### 3.1.4 推送限流防骚扰：避免用户被频繁推送

营销活动、日常通知消息频繁，若不做限流控制，会导致单用户短时间内收到数十条推送，造成严重用户骚扰，引发用户投诉、关闭推送权限、流失等问题。同时无差别全量推送会造成大量无效推送、浪费通道成本与服务器资源，必须建立完善的限流、过滤、防骚扰机制。

## 3.2 推送通道选择策略

推送通道是消息触达用户的载体，主流包含APP推送、短信、邮件、站内信四大通道。生产架构中必须建立**标准化通道优先级、适配规则、限流规则、监控体系**，实现消息精准、高效、低成本触达。

### 3.2.1 通道优先级：优先APP推送，失败则短信兜底

基于触达实时性、成本、用户体验、到达率，生产通用**通道优先级策略**，从上至下优先级递减：

**APP推送（最高优先级）→ 站内信 → 短信兜底 → 邮件兜底**

- **APP推送**：免费、实时性最强、用户体验好，作为首选推送通道；

- **站内信**：用于用户动态、系统消息留存，无推送打扰，适合非紧急消息；

- **短信**：收费、有骚扰性，仅作为APP推送失败后的紧急兜底通道；

- **邮件**：实时性差，适合对账、公告、归档类非紧急消息。

**核心执行逻辑**：优先调用高优先级通道推送，推送失败（设备离线、推送接口报错、权限关闭）后，自动降级使用次优先级通道兜底，保障消息必达。

### 3.2.2 通道适配：不同消息类型适配不同通道（通知/营销/验证码）

不同业务消息特性不同，必须精准适配推送通道，兼顾体验、成本、实时性：

| 消息类型                | 业务特性                 | 适配通道         | 适配原因                                        |
| ----------------------- | ------------------------ | ---------------- | ----------------------------------------------- |
| 验证码/登录通知（紧急） | 高实时、高可靠、需必达   | APP推送+短信兜底 | 保障用户即使关闭APP推送，也能通过短信接收验证码 |
| 系统通知（订单/支付）   | 中等实时、重要、低骚扰   | APP推送+站内信   | 实时推送提醒，站内信留存记录，无需短信成本      |
| 营销活动消息            | 非紧急、高频、易骚扰     | 仅APP推送+站内信 | 禁止短信推送，避免用户投诉，降低营销成本        |
| 对账/公告/归档消息      | 低实时、需留存、数据量大 | 邮件+站内信      | 满足归档留存需求，无需实时推送                  |

### 3.2.3 通道限流：各通道推送频率限制

第三方推送通道（短信、厂商推送）均有**QPS频次限制**，无节制推送会触发通道限流、接口报错、账号封禁，必须针对各通道独立配置限流规则。

**生产通用限流策略**

- **全局通道限流**：限制短信、APP推送单秒最大请求量，平滑批量推送流量，避免瞬时打爆第三方接口；

- **单账号限流**：第三方平台维度限制单日推送总量，提前阈值预警；

- **分时段限流**：夜间低活跃时段降低推送频次，减少无效推送与骚扰。

**技术实现**：基于Redis令牌桶算法实现通道级QPS限流，拦截超限请求，加入等待队列异步重试。

### 3.2.4 通道可用性监控：推送成功率、延迟监控

多通道推送架构下，必须建立监控体系，快速感知通道故障、推送异常：

- **成功率监控**：监控各通道推送成功数、失败数、成功率，成功率低于阈值触发告警；

- **延迟监控**：统计消息从发起推送至用户接收的平均延迟，识别通道卡顿问题；

- **异常监控**：统计第三方接口报错、超时、限流频次，快速识别通道故障；

- **成本监控**：统计短信、付费通道消耗成本，避免恶意消耗与资源浪费。

**故障兜底**：某通道成功率骤降时，自动熔断降级，切换至备用通道，保证消息可达。

## 3.3 在线用户管理方案

在线用户管理是精准推送的核心基础，核心目标：**精准识别用户在线状态、低性能消耗、支持百万级并发查询、实现离线消息补发**，彻底解决无效推送、离线推送失败、状态错乱问题。

### 3.3.1 在线状态维护：客户端心跳+服务端定时清理

生产标准在线状态维护方案：**心跳机制 + 过期淘汰机制**，兼顾实时性与准确性。

**客户端心跳**：APP/客户端每30s主动向服务端上报心跳请求，携带用户ID、设备ID、登录终端，更新在线过期时间；

**服务端过期清理**：服务端设定过期阈值（默认90s），超过阈值未上报心跳，判定用户离线；同时通过定时任务批量清理过期在线数据，避免脏数据堆积。

**优势**：无需用户主动触发上下线接口，适配APP闪退、网络异常、后台杀进程等异常离线场景，状态判断更精准。

### 3.3.2 在线用户存储：Redis存储在线用户ID与连接信息

用户在线状态属于高频读写数据，数据库性能无法支撑，生产统一采用**Redis高性能存储**。

**存储结构设计**

- Key：`push:online:userId`

- Value：设备ID、终端类型、最近心跳时间、推送Token

- 过期时间：90s，自动过期淘汰离线用户

**核心优势**：Redis读写性能百万级，支撑全平台用户状态查询、更新，自带过期机制，无需手动清理离线数据，性能极高。

**核心代码（心跳更新）**

```java
/**
 * 用户心跳上报接口
 * 更新用户在线状态，续期过期时间
 */
@PostMapping("/heartbeat")
public Result heartbeat(@RequestParam Long userId, @RequestParam String deviceId) {
    // 封装用户在线信息
    UserOnlineDTO onlineDTO = new UserOnlineDTO();
    onlineDTO.setUserId(userId);
    onlineDTO.setDeviceId(deviceId);
    onlineDTO.setLastHeartTime(System.currentTimeMillis());
    // 写入Redis，90秒过期
    String key = "push:online:" + userId;
    redisTemplate.opsForValue().set(key, JSON.toJSONString(onlineDTO), 90, TimeUnit.SECONDS);
    return Result.success();
}

```

### 3.3.3 多端在线管理：同一用户多端在线状态同步

现如今用户普遍手机、平板、电脑多端登录，需要支持**多端同时在线、多端同步推送**能力。

**解决方案**

修改存储结构，由单用户单Key存储，改为**用户维度Hash结构**，一个用户可存储多个设备在线信息：

- Key：`push:online:userId`

- HashKey：设备ID

- HashValue：设备在线信息

推送时遍历用户所有在线设备，实现多端同步推送；单设备下线仅删除对应设备数据，不影响其他终端在线状态。

### 3.3.4 离线消息处理：用户上线后推送离线消息

用户离线期间产生的消息不能丢失，需实现**离线消息缓存+上线补发**机制，保障消息完整性。

**完整流程**

1. 推送前校验用户在线状态，用户离线则不实时推送；

2. 将消息存入Redis离线消息队列，设置最大缓存时长（7天）；

3. 用户重新上线、上报第一次心跳时，触发离线消息批量补发；

4. 补发成功后清理离线消息，避免重复推送。

**避坑要点**：离线消息需区分消息类型，营销类消息超时自动丢弃，通知类消息强制补发。

## 3.4 推送限流与防骚扰

推送系统最大的产品风险是**用户骚扰**，技术层面必须通过多层限流、过滤、优先级控制，实现“该推的不漏推、不该推的不滥推”。

### 3.4.1 用户维度限流：单用户单位时间推送次数限制

核心限流规则：**限制单个用户单位时间内接收消息最大数量**，杜绝高频骚扰。

**生产配置规范**

- 普通通知消息：单用户每分钟最多3条；

- 营销活动消息：单用户每日最多5条；

- 紧急验证码、告警消息：不限制频次，保证必达。

**实现方式**：Redis计数器，以「用户ID+消息类型」为Key，统计单位时间推送次数，超限直接拦截。

### 3.4.2 业务维度限流：营销类消息推送频率限制

营销消息属于高骚扰、低紧急消息，需单独做严格限流策略：

- **时段限流**：夜间23:00-次日8:00禁止推送营销消息；

- **日限额限流**：每个用户每日营销消息推送上限固定，超额全部丢弃；

- **活动限流**：同一活动仅推送一次，避免活动反复弹窗推送。

### 3.4.3 黑名单机制：用户屏蔽的推送类型过滤

完善的用户自主屏蔽机制，是防骚扰的核心兜底：

- **全局黑名单**：用户关闭所有推送，所有消息全部拦截；

- **分类黑名单**：用户可单独屏蔽营销、通知、活动类消息，精准过滤；

- **临时黑名单**：用户短时免打扰时段，自动屏蔽所有推送。

**执行优先级**：黑名单过滤 > 限流过滤 > 正常推送，优先拦截用户屏蔽消息。

### 3.4.4 推送频率控制：消息优先级+批量推送

面对海量推送任务，通过**消息优先级分级**保障核心消息优先触达，非核心消息批量削峰：

- **高优先级**：验证码、支付通知、系统告警，实时推送、不限流、优先占用通道；

- **中优先级**：订单、物流通知，正常频次推送；

- **低优先级**：营销、活动、资讯消息，批量聚合推送、错峰推送。

**批量优化**：多条低优先级消息合并为一条推送，减少用户弹窗打扰次数，同时降低接口请求量。

## 3.5 消息推送系统架构

整合上述通道策略、状态管理、限流防骚扰能力，形成一套生产级高可用、高并发、可监控、可兜底的消息推送架构，支撑百万级用户批量推送场景。

### 3.5.1 推送服务架构：推送中心+通道适配器+在线管理

标准三层推送架构，解耦业务、通道、状态管理，扩展性极强：

1. **推送调度中心（核心层）**：接收业务推送请求、参数校验、限流过滤、黑名单过滤、消息优先级排序、分发推送任务；

2. **通道适配器（适配层）**：统一封装APP、短信、邮件、站内信推送接口，屏蔽各第三方通道差异，自动实现通道降级、兜底；

3. **在线用户管理（基础层）**：维护用户在线状态、处理心跳上报、离线消息缓存与补发、多端状态同步。

**架构优势**：新增推送通道、新增消息类型无需改动核心逻辑，符合开闭原则。

### 3.5.2 异步推送流程：消息队列接收请求→异步推送→结果回调

高并发批量推送**禁止同步执行**，全部采用MQ异步架构，彻底解决阻塞、超时、堆积问题。

**完整异步推送链路**

1. 业务系统发起推送请求，投递至推送专属MQ队列；

2. 推送服务消费者匀速消费任务，避免瞬时流量冲击；

3. 执行限流、黑名单、在线状态校验，过滤无效推送；

4. 通道适配器选择对应通道执行推送；

5. 推送成功/失败结果入库，回调业务方接口同步结果。

**核心价值**：支持百万级推送任务削峰处理，服务稳定无压力。

### 3.5.3 推送失败重试：失败消息重推+死信队列

网络抖动、通道临时故障会导致推送失败，必须建立重试兜底机制，保障消息可达：

- **分级重试**：临时失败消息重试3次，间隔1s、3s、10s梯度重试；

- **通道重试降级**：当前通道重试失败，自动切换兜底通道重试；

- **死信队列兜底**：多次重试失败的消息进入死信队列，人工排查修复后可批量补发，避免消息丢失。

### 3.5.4 推送数据统计：推送成功率、到达率、点击率

生产推送系统必须完善数据统计体系，用于运营分析与故障排查：

- **推送总量指标**：总推送数、有效推送数、拦截数、离线缓存数；

- **质量指标**：推送成功率、消息到达率、用户点击率；

- **异常指标**：各通道失败率、重试次数、死信消息数量；

- **运营指标**：营销消息转化率、用户退订率、投诉率。

通过数据大盘实时监控推送质量，及时发现通道故障、用户体验问题。

本章完整落地了高并发消息推送系统的全套架构设计与生产方案，是广播型高并发业务的典型实战模板。首先梳理了推送系统高并发推送、多通道适配、在线状态维护、用户防骚扰四大核心痛点；其次建立了标准化的多通道优先级与适配策略，实现不同消息类型精准触达、低成本、高可用；接着讲解了基于Redis+心跳机制的在线用户管理方案，解决多端在线同步、离线消息补发难题；同时通过用户维度、业务维度、黑名单、消息优先级多层防骚扰限流机制，平衡推送触达与用户体验；最后搭建了由调度中心、通道适配、状态管理组成的三层架构，结合MQ异步推送、失败重试、数据统计体系，实现整套推送系统高并发、高可用、可监控、可兜底的生产能力。本章设计思想可直接复用至APP推送、站内通知、营销触达、系统告警等所有消息推送场景，进一步完善了高并发业务场景的架构设计体系。

# 4. 热点数据处理：热点key发现、缓存预热、本地缓存兜底

高并发流量并非均匀分布，互联网流量遵循**二八定律**：20%的热点数据承载80%以上的请求流量，秒杀商品、爆款活动、首页配置、热搜内容均为典型热点数据。普通数据的并发问题是「流量大」，热点数据的并发问题是「流量极度集中」，若不做专项防护，再高可用的集群架构也会出现单点瓶颈，引发全线故障。

## 4.1 热点数据问题与危害

热点数据是高并发系统稳定性的最大隐形杀手之一，多数大促雪崩、Redis节点宕机、数据库突发压力问题，根源均为热点数据未做防护。本节详细拆解热点数据的定义、核心危害与识别方式。

### 4.1.1 热点数据的定义：访问量极高的少数数据

**热点数据（热点Key）**：指在短时间内被海量请求集中访问、读写QPS远超普通数据的少量数据，是流量高度聚合的业务数据。

行业通用判定标准：单Key每秒访问量**远超集群平均QPS**，单Key承载单机/集群30%以上流量，即可判定为热点Key。

**典型热点数据场景**

- 秒杀活动、爆款商品详情数据；

- 平台首页配置、全局活动开关、系统常量配置；

- 热搜资讯、热门视频、全民话题数据；

- 统一支付配置、公共规则配置等全局缓存数据。

### 4.1.2 热点数据的危害：缓存击穿、数据库压力、Redis分片热点

热点数据流量高度集中，会引发连锁式线上故障，危害远大于普通流量洪峰：

**1. 缓存击穿风险**

热点Key过期瞬间，海量并发请求同时穿透缓存、直达数据库，瞬间打满数据库连接池、CPU，导致数据库雪崩，进而引发整个服务瘫痪。

**2. Redis分片热点与单点瓶颈**

Redis集群通过Key哈希分片，同一热点Key只会落在**单一节点**，海量流量集中打在单个Redis节点，导致该节点CPU、内存、网络打满，集群负载严重不均，出现单点宕机，进而引发集群雪崩。

**3. 应用服务性能瓶颈**

海量请求集中查询同一数据，会导致应用服务大量线程阻塞在缓存查询逻辑，线程池耗尽，无法处理其他正常请求。

**4. 数据一致性故障**

热点数据更新频率高、并发读写激烈，极易出现缓存更新不一致、脏数据覆盖等问题。

### 4.1.3 热点数据的识别方式：监控分析、业务预判、流量分析

处理热点数据的前提是**精准识别热点数据**，生产环境分为被动识别与主动预判两种方式：

**1. 业务预判（主动识别，最高效）**

结合业务场景提前预判热点数据，所有营销活动、秒杀活动、首页主推资源，天然属于热点数据，可提前纳入防护范围，是生产首选方案。

**2. 监控分析（被动识别，兜底发现）**

通过Redis监控、应用监控、链路监控，识别QPS异常偏高的Key与接口，精准定位突发热点数据（如突发热搜、爆火商品）。

**3. 流量日志分析**

定时清洗应用访问日志，统计单Key、单资源的访问频次，批量挖掘隐藏热点数据，提前防护。

## 4.2 热点key发现方案

热点Key发现是热点防护的前置能力，生产环境采用「业务预判+技术监控+自动告警」的组合方案，实现**事前预判、事中监控、事后复盘**，无遗漏覆盖所有热点数据。

### 4.2.1 Redis监控：Key的访问频率监控

Redis自带监控能力，可精准统计Key访问频次，是识别热点Key的核心技术手段。

**核心监控手段**

- **INFO stats 统计**：查看Redis整体key_hits、key_misses命中率，判断是否存在集中访问热点；

- **Redis慢查询日志**：高频访问的热点Key会大量出现在短耗时请求中，结合频次统计热点；

- **第三方监控平台**：基于Prometheus+Grafana监控单Key QPS、单节点流量，可视化识别热点分片与热点Key。

**生产优势**：精准、实时、无侵入，不影响业务性能。

### 4.2.2 应用层埋点：请求日志分析热点key

在应用层对所有Redis查询请求做统一埋点，记录Key、访问时间、访问次数，通过日志聚合分析热点数据。

**实现思路**

1. 基于Redis操作切面统一埋点，无需侵入业务代码；

2. 异步采集所有Key的访问日志，推送至日志系统；

3. 定时聚合统计单位时间内各Key的访问次数，排序筛选Top热点Key；

4. 对超过阈值的Key标记为热点数据，触发预警。

**适用场景**：挖掘业务未知的隐性热点Key，解决纯Redis监控无法精准定位业务Key的问题。

### 4.2.3 业务预判：活动商品、热点事件相关数据

90%的线上热点故障均可通过业务提前预判规避，是成本最低、最有效的热点发现方式。

**预判规则**

- 所有秒杀、限时折扣、爆款活动商品，强制标记为热点数据；

- 首页轮播、弹窗活动、全局配置，默认按热点数据防护；

- 平台主推内容、运营重点资源，提前纳入热点防护清单。

**生产价值**：提前防护，避免热点数据上线后引发突发故障，是大促保障的核心手段。

### 4.2.4 热点key自动发现：监控告警+人工确认

生产成熟方案：**自动化发现+告警+人工确认入库**，形成闭环防护。

**完整流程**

1. 监控系统实时采集Key访问频次，超过预设阈值自动判定为临时热点Key；

2. 触发钉钉/短信告警，推送热点Key、访问QPS、所属业务、所在分片信息；

3. 运维/开发人工确认是否为长期热点数据；

4. 确认后自动加入热点防护白名单，开启预热、本地缓存、分片打散策略。

## 4.3 缓存预热策略

缓存预热是解决**热点Key冷启动、缓存过期大批量失效、缓存击穿**的核心方案，核心思想：在流量进入前，主动将热点数据加载至缓存，避免流量落地时缓存无数据。

### 4.3.1 预热触发时机：活动前、缓存重启后

生产固定三大预热时机，覆盖所有缓存冷启动场景：

- **大促/活动开启前预热**：秒杀、活动上线前5~10分钟，批量预热所有活动热点商品数据；

- **缓存重启/故障恢复后预热**：Redis重启、集群切换、数据清空后，主动预热核心热点数据，避免冷启动击穿；

- **定时预热刷新**：长期热点数据定时刷新缓存，保证缓存常驻、数据新鲜。

### 4.3.2 预热实现方式：批量加载、异步线程池、定时任务

生产三种主流预热实现方案，适配不同业务场景：

**1. 接口手动批量预热**

提供后台管理接口，支持手动传入热点Key列表，一键批量加载数据库数据写入缓存，适用于临时活动预热。

**2. 异步线程池预热（推荐）**

使用独立线程池异步执行预热任务，不占用业务线程，避免预热影响线上流量。

**3. 定时任务自动预热**

通过XXL-Job、Quartz定时任务，定时刷新热点缓存，适用于长期常驻热点数据。

**核心预热代码示例**

```java
/**
 * 热点数据缓存预热工具类
 * 独立线程池异步预热，不阻塞主线程、不影响业务
 */
@Component
public class HotDataPreheat {

    @Autowired
    private RedisTemplate<String, Object> redisTemplate;

    @Autowired
    private GoodsMapper goodsMapper;

    // 自定义预热独立线程池
    private static final Executor PREHEAT_EXECUTOR = new ThreadPoolExecutor(
            5, 10, 60L, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(100),
            new ThreadFactoryBuilder().setNameFormat("preheat-thread-%d").build()
    );

    /**
     * 批量预热商品热点数据
     */
    public void preheatGoodsData(List<Long> goodsIdList) {
        // 异步执行预热任务
        PREHEAT_EXECUTOR.execute(() -> {
            for (Long goodsId : goodsIdList) {
                // 从数据库查询最新数据
                Goods goods = goodsMapper.selectById(goodsId);
                if (Objects.nonNull(goods)) {
                    // 写入Redis缓存，设置过期时间
                    redisTemplate.opsForValue().set("goods:hot:" + goodsId, goods, 1, TimeUnit.HOURS);
                }
            }
        });
    }
}

```

### 4.3.3 预热数据更新：定时刷新热点数据缓存

热点数据并非一成不变，需要定时刷新保证缓存与数据库数据一致性。

**更新策略**

- **短期活动热点**：活动期间高频定时刷新（5分钟一次），保证价格、库存数据实时性；

- **长期静态热点**（首页配置、系统参数）：低频刷新（30分钟~1小时一次），减少DB查询压力；

- **数据更新主动刷新**：热点数据业务更新后，主动触发缓存更新，结合定时兜底，实现最终一致。

### 4.3.4 预热过程性能控制：分批加载、避免影响业务

大批量数据预热若一次性全量加载，会瞬间打满数据库查询压力，引发性能问题，必须做流量控制。

**生产性能控制方案**

- **分批分页加载**：批量预热拆分小批次执行，每次预热20~50条数据，避免一次性大量DB查询；

- **预热限速**：批次之间休眠短暂时间，平滑预热流量；

- **错峰预热**：避开业务高峰期，优先在低峰期完成缓存预热；

- **独立数据源**：预热查询走从库，不占用主库资源，杜绝影响业务写入。

## 4.4 本地缓存兜底方案

缓存预热只能解决缓存无数据问题，无法解决Redis节点故障、热点流量集中问题。**多级缓存（Redis+本地缓存）** 是热点数据最核心的兜底防护方案，可彻底隔绝热点流量对Redis的冲击。

### 4.4.1 多级缓存架构：Redis+本地缓存二级防护

**标准二级缓存架构**：本地缓存（L1） + Redis分布式缓存（L2）

**查询流程**

1. 请求优先查询应用本地缓存，命中直接返回，**不经过Redis**；

2. 本地缓存未命中，查询Redis分布式缓存；

3. Redis未命中，查询数据库，同时回写Redis和本地缓存。

**核心价值**：热点数据99%的流量会被本地缓存拦截，极大减少Redis访问量，彻底解决Redis热点分片压力，同时提升接口响应速度。

### 4.4.2 本地缓存选型：Caffeine、Guava Cache

生产主流本地缓存对比与选型：

| 缓存框架    | 性能             | 淘汰策略                         | 适用场景                         |
| ----------- | ---------------- | -------------------------------- | -------------------------------- |
| Caffeine    | 极高，命中率最优 | Window TinyLFU，业界最优淘汰算法 | **生产首选**，高并发热点数据缓存 |
| Guava Cache | 较高             | LRU最近最少使用                  | 老项目兼容、普通缓存场景         |

**最佳实践**：新项目统一使用**Caffeine**，性能远超Guava，更适配热点数据高并发场景。

**Caffeine核心依赖**

```xml
<!-- Caffeine本地缓存依赖 -->
<dependency>
    <groupId>com.github.benmanes.caffeine</groupId>
    <artifactId>caffeine</artifactId>
    <version>2.9.3</version>
</dependency>

```

### 4.4.3 本地缓存更新：定时刷新、事件通知更新

本地缓存存在多实例数据不一致问题，需要配套的更新机制保障数据新鲜度：

**1. 定时刷新（通用方案）**

热点数据本地缓存设置较短过期时间（1~5分钟），定时自动刷新，牺牲短暂一致性换取高性能。

**2. 事件通知更新（高一致性方案）**

数据更新后，发送MQ消息通知所有服务实例，各实例主动清空/更新本地缓存，实现实时一致。

### 4.4.4 本地缓存一致性：最终一致性、短暂不一致容忍

**核心设计思想**：热点高并发场景，优先保障**性能与稳定性**，容忍极短时间的数据不一致，保证最终一致性。

**生产适配规则**

- 营销、活动、商品展示类热点数据：容忍1~5分钟短暂不一致，优先保性能；

- 支付、库存、资金类核心数据：不使用本地缓存，仅用Redis，保证强一致。

**避坑指南**：本地缓存无过期淘汰、无更新机制会导致永久脏数据，必须配置过期时间+定时刷新兜底。

## 4.5 热点数据分片优化

多级缓存可以拦截绝大部分流量，但无法彻底解决极端热点Key单点问题。热点分片打散是**根治Redis单点热点瓶颈**的终极方案，将集中流量均匀分散到集群多个节点。

### 4.5.1 热点key分片：将一个热点key拆分为多个key

**核心原理**：将单一热点Key `goods:1001`，拆分为多个分片Key，让流量分散到不同Redis节点。

**分片实现方案**

1. 预设固定分片数量（如16分片、32分片）；

2. 将原Key拼接随机分片后缀，生成`goods:1001_0` ~ `goods:1001_15`；

3. 查询时随机选取一个分片Key访问；

4. 更新时批量更新所有分片Key，保证数据一致。

**效果**：原本10万QPS的单点流量，均匀分散到16个节点，单节点压力降至原来的1/16，彻底解决单点瓶颈。

### 4.5.2 Redis集群分片优化：热点key迁移、增加副本

**临时应急方案**，适用于突发热点、来不及代码改造的场景：

- **热点Key迁移**：将热点Key所在分片迁移至负载较低的空闲节点，避免节点过载；

- **增加从节点副本**：热点分片增加多个从节点，读请求分摊至从节点，主节点只处理写请求，分担读压力；

- **临时扩容分片**：紧急扩容Redis集群节点，分散热点流量。

### 4.5.3 热点key的流量分散：请求负载均衡到多个节点

结合分片Key实现流量负载均衡：

- 应用层通过随机算法、哈希算法均匀选择分片Key；

- 让热点流量均匀打散至集群所有节点，无流量倾斜；

- 配合本地缓存，实现双层流量打散，极致提升系统并发能力。

### 4.5.4 热点数据处理的综合防护方案

整合本章所有知识点，形成生产级**热点数据四层防护体系**（由浅入深、层层兜底）：

1. **第一层：事前预判**：业务提前识别热点数据，纳入防护清单；

2. **第二层：缓存预热**：流量到来前预热缓存，避免冷启动击穿；

3. **第三层：二级缓存兜底**：Caffeine本地缓存拦截99%热点流量，保护Redis；

4. **第四层：热点分片打散**：拆分热点Key，彻底解决Redis单点瓶颈。

该方案是互联网大厂大促热点防护的标准架构，可100%解决所有热点数据引发的线上故障。

## 本章总结

本章系统落地了高并发架构中热点数据防护的全套生产方案，是保障高并发系统稳定性的核心章节。首先明确了热点数据的定义与流量倾斜特性，深度剖析了缓存击穿、Redis单点热点、数据库雪崩等核心线上危害；其次讲解了业务预判、Redis监控、应用埋点三种热点Key发现手段，实现热点数据全量识别；接着通过缓存预热机制解决缓存冷启动、过期击穿问题，并配套分批预热、错峰执行等性能控制方案；重点落地了Caffeine+Redis二级缓存架构，通过本地缓存拦截绝大部分热点流量，同时明确了高并发场景下最终一致性的取舍原则；最后通过热点Key分片、集群优化、流量负载均衡彻底根治Redis单点瓶颈，整合出四层热点综合防护体系。本章方案可完美适配秒杀、爆款商品、热搜资讯等所有热点场景，补齐了高并发系统稳定性优化的核心能力，为后续高并发架构整体复盘与实战落地提供完整理论与实操支撑。

# 5. 高并发系统的容灾与多活设计

高并发系统的架构设计分为两大维度：**性能优化**与**可用性保障**。前文讲解的异步化、缓存、热点防护均属于性能优化，解决系统扛得住高并发流量的问题；而容灾与多活设计属于可用性保障，解决系统故障不宕机、业务不中断的问题，是高并发架构落地不可或缺的核心能力。

## 5.1 容灾设计核心思想

容灾是所有线上系统的基础能力，核心思想为**冗余备份、故障隔离、快速恢复**，通过物理资源冗余与架构分层兜底，避免单点、单机房故障导致业务全线不可用。

### 5.1.1 容灾的定义：系统故障时保证服务可用的能力

**容灾全称灾难恢复**，指系统遭遇突发故障（服务器宕机、磁盘损坏、机房断电、光缆中断、硬件故障、程序BUG）时，通过预设的备份机制、切换策略、恢复方案，保证核心业务**不中断、数据不丢失、服务可快速恢复**的架构能力。

容灾的核心本质：**用冗余成本换取业务可用性与数据安全性**，通过多份资源、多节点、多机房部署，消除单点故障，是高并发高可用系统的设计底线。

**常见容灾场景**：单服务器宕机、单机房断电、数据库主库故障、Redis集群节点宕机、网络分区故障等。

### 5.1.2 容灾等级：数据备份、冷备、温备、热备

行业根据**资源冗余程度、故障恢复速度、数据实时性**，将容灾划分为四个等级，适配不同重要级别的业务场景，实现成本与可用性的平衡：

- **1. 数据备份（最低等级）**：无实时冗余服务，仅定时备份数据，服务故障后需重新部署、恢复数据。恢复速度极慢，仅适用于后台非核心业务、离线任务系统，无法支撑线上高并发业务。

- **2. 冷备容灾**：备用节点长期处于关闭/休眠状态，不承接流量，仅主节点故障后手动启动备用节点恢复服务。资源成本低，但恢复耗时极长，人工介入成本高，不适用于核心交易业务。

- **3. 温备容灾**：备用节点长期开机、服务正常运行，但不承接线上流量，数据实时同步主节点。主节点故障后，可通过手动/半自动切换流量恢复服务，恢复速度较快，适用于一般核心业务。

- **4. 热备容灾（最高等级）**：主备节点、多节点同时运行、实时同步数据、共同承接流量，故障节点自动被剔除，流量实时切换至正常节点，**无感知故障恢复**，是电商交易、支付、订单等高并发核心业务的标准容灾方案。

### 5.1.3 容灾的核心目标：RTO（恢复时间目标）、RPO（恢复点目标）

RTO与RPO是衡量容灾能力的**两大核心量化指标**，所有容灾方案的设计、落地、验收均围绕这两个指标展开，是生产环境容灾定级的唯一标准。

**1. RTO（Recovery Time Objective 恢复时间目标）**

定义：系统发生故障后，业务从中断到**完全恢复正常可用**的最大耗时。核心代表**服务恢复速度**。

生产标准：核心交易业务RTO≤30s，一般业务RTO≤5min，离线业务RTO≤30min。

**2. RPO（Recovery Point Objective 恢复点目标）**

定义：系统故障恢复后，**允许丢失的最大数据量、最远丢失时间**。核心代表**数据安全程度**。

生产标准：核心资金、交易业务RPO=0（零数据丢失），普通业务允许秒级、分钟级数据丢失。

**核心总结**：RTO越小，服务恢复越快；RPO越小，数据越安全，容灾能力越强，对应的架构成本与复杂度越高。

## 5.2 多活架构设计

多活架构是**最高级别的热备容灾方案**，也是大厂高并发系统的标准架构，彻底解决单机房整体故障的致命问题，实现机房级容灾，保障大促、峰值流量下的业务永续。

### 5.2.1 多活的定义：多个机房同时提供服务

**多活架构**指部署在**多个物理独立机房/区域**的服务集群，同时处于正常运行状态，同时承接线上用户流量，机房之间互为容灾备份。任意一个机房整体故障，其余机房可无缝承接全部流量，业务无中断、用户无感知。

多活与传统容灾的核心区别：传统容灾是**一主一备、备机闲置**，多活是**多主并行、全员干活**，资源利用率更高、故障切换更快、并发承载能力更强。

### 5.2.2 同城双活：同一城市两个机房互为备份

**同城双活**是中小型互联网公司主流落地的多活方案，指在**同一城市、两个物理隔离的机房**部署全套业务集群，双机房同时承接流量、实时同步数据、互相容灾。

**核心特性**

- 机房距离近、网络延迟极低（毫秒级），数据同步延迟小，数据一致性极易保障；

- 部署成本低、架构复杂度低、运维难度小；

- 可抵御单机房断电、网络中断、设备故障等大部分机房级故障。

**局限性**：无法抵御城市级灾难（地震、洪水、城市大面积停电），仅能满足基础容灾需求。

**适用场景**：中小型平台、非国家级核心业务、预算有限的高并发系统。

### 5.2.3 异地多活：跨城市多个机房同时服务

**异地多活**是大型大厂、金融、支付、电商平台的标准架构，指在**不同城市、不同地域**部署多个独立机房集群，多地域同时承接流量、异地数据同步、跨地域容灾。

**核心特性**

- 彻底规避城市级灾难，容灾等级最高，业务可用性无限接近100%；

- 可就近接入用户流量，降低跨地域网络延迟，提升用户访问速度；

- 支持超大流量拆分，多地域分担全网并发压力，突破单机房性能上限。

**局限性**：跨城网络延迟高，数据同步存在延迟，架构复杂度极高、运维成本高、需要解决跨机房数据一致性问题。

**适用场景**：头部电商、支付系统、金融核心系统、国家级高可用要求业务。

### 5.2.4 单元化架构：按用户ID/业务单元划分多活节点

**单元化架构**是异地多活的进阶落地形态，是超高并发系统的终极多活方案，核心思想是**业务分片、单元隔离、自治闭环**。

**设计原理**

将全网用户按照**用户ID哈希、手机号、区域**等维度划分为多个独立业务单元，每个单元对应一个机房集群，每个单元内部包含完整的网关、业务服务、缓存、数据库，实现单元内业务自治。

**核心优势**

- **故障隔离**：单个单元故障仅影响分片用户，不会全局雪崩；

- **流量均分**：海量用户流量均匀分散至多机房，完美支撑超高并发；

- **切换简单**：单元故障可快速将分片流量迁移至其他正常单元；

- **扩展性极强**：流量增长可直接新增单元扩容，无架构瓶颈。

**典型落地**：阿里、字节、拼多多大促高并发系统均采用单元化多活架构。

## 5.3 容灾与多活关键技术

多活架构的落地依赖四大核心技术：数据同步、流量切换、故障检测、故障自愈，是实现多机房无缝容灾、无感知切换的技术基石。

### 5.3.1 数据同步：主从复制、双写同步、异步同步

多机房部署的核心难点是**跨机房数据同步**，不同业务场景适配不同同步方案，平衡一致性与性能。

**1. 主从异步复制**

单机房主写入、多机房从节点异步同步数据，架构简单、性能高、无写入延迟，存在短暂数据延迟，适合非核心、高并发查询业务。

**2. 双写同步（半同步）**

业务同时向双机房写入数据，等待本地写入成功、异地写入异步确认，兼顾一致性与性能，是同城双活主流方案。

**3. 跨机房强同步**

必须等待多机房数据全部写入成功才返回结果，零数据丢失、一致性最高，但跨城延迟高、性能差，仅适用于支付、资金等核心金融数据。

### 5.3.2 流量切换：DNS切换、负载均衡切换、灰度切换

故障发生后，需要快速、精准、安全地完成流量切换，是容灾落地的关键动作。

- **DNS全局切换**：修改域名解析规则，将全网流量从故障机房切换至正常机房，适用于机房级整体故障，切换粒度粗、覆盖范围广；

- **四层/七层负载均衡切换**：通过Nginx、SLB后台配置，剔除故障节点/机房，精准切换应用层流量，切换速度快、粒度细；

- **灰度流量切换**：故障恢复、版本上线时，逐步放量流量，避免瞬间流量冲击导致正常机房过载雪崩，保障切换过程平稳。

### 5.3.3 故障检测：心跳检测、服务健康检查

故障自愈的前提是**精准、实时的故障检测**，生产采用双层检测机制，避免误判、漏判。

- **心跳检测**：服务、节点、机房定时上报心跳，连续多次超时未上报，判定节点/机房故障，底层基础检测；

- **业务健康检查**：主动调用业务健康接口，检测接口成功率、响应延迟、错误率，规避服务存活但业务异常的假活问题；

- **指标阈值检测**：监控QPS、错误率、CPU、内存、网络指标，超过阈值触发故障预警与判定。

### 5.3.4 故障自动恢复：自动切换、故障自愈

成熟多活架构必须实现**全自动故障自愈**，无需人工介入，保障RTO最小化。

**自愈流程**

1. 监控系统精准识别故障节点/机房；

2. 自动更新负载均衡、DNS配置，剔除故障资源；

3. 流量自动调度至正常机房节点，业务恢复；

4. 故障节点恢复后，自动重新接入集群，同步补齐数据；

5. 流量灰度回切，完成自愈闭环。

## 5.4 容灾与多活的业务适配

多活架构不是简单的机器扩容、机房部署，需要业务层配合改造，解决数据一致性、有状态服务、重复请求等问题，同时通过演练保障容灾能力真实有效。

### 5.4.1 数据一致性保障：最终一致性、跨机房事务

跨机房网络延迟不可消除，强一致性成本极高，多活架构行业通用设计原则：**核心业务局部一致，非核心业务最终一致**。

- **交易/支付核心场景**：采用分布式事务、半同步复制、本地消息表，保障资金数据零丢失、强一致性；

- **资讯/营销/展示场景**：容忍秒级、分钟级数据延迟，采用异步数据同步，保证最终一致性，换取高性能；

- **跨机房冲突解决**：通过版本号、时间戳、唯一键、幂等机制解决多机房数据覆盖、冲突问题。

### 5.4.2 业务适配改造：无状态化、数据分片、幂等性设计

传统单体、有状态业务无法适配多活流量切换，必须完成三大核心改造：

- **服务无状态化**：所有业务服务不存储本地会话、不缓存本地数据，会话、状态全部存入Redis、数据库，保证任意机房节点可无缝承接流量；

- **数据分片隔离**：按照单元化规则分片数据，避免跨机房大量数据交互，降低同步复杂度；

- **全局幂等性**：所有接口、写入操作支持幂等，避免流量重复切换、重试导致重复下单、重复支付、重复数据问题。

### 5.4.3 容灾演练：定期故障注入、切换演练

**未演练的容灾都是伪容灾**，线上环境必须定期开展容灾演练，提前暴露架构隐患。

**生产主流演练方式**

- **故障注入演练**：主动模拟节点宕机、机房断网、数据库故障、Redis宕机，验证系统自愈能力；

- **全量流量切换演练**：定期将单机房流量全部切换至备用机房，验证多活承载能力；

- **压测演练**：故障切换后对目标机房压测，验证流量切换后无性能瓶颈。

### 5.4.4 容灾成本与收益平衡：容灾等级与业务重要性匹配

容灾等级越高，机房、服务器、运维成本越高，生产严禁一刀切设计，必须**分级容灾、按需适配**。

- **P0核心业务（支付、订单、秒杀）**：异地多活、热备容灾，RTO=0、RPO=0，最高优先级资源保障；

- **P1重要业务（商品、首页、资讯）**：同城双活，快速恢复，容忍极小数据延迟；

- **P2普通业务（日志、统计、后台配置）**：冷备、定时备份，降低容灾成本；

- **P3离线业务**：仅数据备份，无需实时容灾。
