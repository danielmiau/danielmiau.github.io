# 05-数据访问与事务管理

## 本章概述

数据访问是后端项目的核心基石，所有业务系统的核心数据读写、持久化、一致性保障都依赖数据层能力，而事务管理是保障数据安全、杜绝数据错乱、解决并发数据问题的关键核心。绝大多数线上生产故障、数据BUG、业务异常，都源于**数据源配置不当、连接池参数不合理、数据层操作不规范、事务失效、事务粒度失控**等问题。

通过本章学习，你将彻底掌握Spring Boot数据层底层机制，能够独立完成生产级数据源配置、连接池参数调优、数据异常排查，规避90%以上的数据层线上坑点，同时吃透数据访问相关面试核心考点。

# 1. Spring Boot 数据访问整体认知

Spring Boot 彻底简化了传统Spring的数据访问配置流程，通过自动配置机制，实现数据源、连接池、持久层框架的开箱即用。不同于原生Spring需要手动配置DataSource、SqlSession、连接池等繁琐代码，Spring Boot仅需简单配置即可完成数据层搭建，同时提供多套数据访问方案适配不同业务场景。本节将带你建立完整的数据访问体系认知，明确不同方案的选型逻辑与底层机制。

## 1.1 数据访问体系介绍

### 1.1.1 Spring Boot 数据访问主流方案对比（JdbcTemplate / JPA / MyBatis）

Spring Boot 生态提供了三套主流的数据持久化方案，分别是 **JdbcTemplate、Spring Data JPA、MyBatis**，三套方案底层均基于JDBC规范实现，核心目的都是完成Java程序与数据库的数据交互，但在封装程度、使用难度、灵活性、性能、适配场景上差异极大。下面通过通俗讲解+核心对比，全方位区分三套方案。

**JdbcTemplate**：Spring 原生提供的极简JDBC封装框架，是三套方案中封装程度最低、最贴近原生JDBC的实现。原生JDBC存在代码冗余、资源频繁创建销毁、重复代码多的问题，JdbcTemplate对其做了轻度封装，统一管理数据库连接、自动关闭资源、简化CRUD代码，无需开发者手动获取连接、处理异常、释放资源。但缺点明显，依然需要手动编写SQL语句、手动封装结果集，开发效率较低。

**Spring Data JPA**：基于JPA规范的高度封装持久层框架，底层封装了Hibernate，是全自动ORM框架。核心特点是**几乎零SQL开发**，开发者只需定义实体类和Mapper接口，通过方法名规则、注解即可自动生成SQL语句，自动完成数据库CRUD操作，同时支持分页、排序、条件查询等通用能力。优点是开发效率极高，缺点是SQL可控性差、复杂查询适配困难、底层黑盒多、调优难度大。

**MyBatis**：半自动化ORM持久层框架，介于JdbcTemplate和JPA之间，是平衡**开发效率与SQL可控性**的最优方案。开发者可自定义SQL语句，将SQL与Java代码解耦，支持XML、注解两种编写方式，同时自动完成参数映射、结果集封装，灵活适配简单CRUD、复杂联表查询、动态SQL场景。

|对比维度|JdbcTemplate|Spring Data JPA|MyBatis|
|---|---|---|---|
|封装程度|低（贴近原生JDBC）|高（全自动ORM）|中（半自动化ORM）|
|SQL可控性|完全可控，手动写SQL|不可控，框架自动生成|完全可控，自定义SQL|
|开发效率|低，代码冗余|极高，零基础SQL|中等，兼顾效率与灵活|
|复杂查询适配|适配，编写繁琐|不适配，动态SQL困难|完美适配复杂联表、动态SQL|
|性能调优|易调优，无多余封装损耗|难调优，黑盒机制多|易调优，SQL完全自主可控|
### 1.1.2 企业选型标准：为什么国内项目首选 MyBatis

在国内企业级开发、互联网项目、微服务架构中，**MyBatis 是绝对主流选型**，几乎覆盖90%以上的后端项目，核心原因贴合生产落地需求，而非技术噱头，具体选型逻辑如下：

第一，适配国内业务场景。企业项目大多存在复杂业务查询、多表联查、动态条件查询、数据统计场景，JPA自动生成的SQL语句死板、无法适配复杂场景，且容易出现低效SQL、全表扫描问题；而MyBatis支持自定义动态SQL、关联查询、批量操作，完美适配复杂业务。

第二，性能可控、便于调优。互联网项目对数据库性能要求极高，JPA封装过重，开发者无法感知底层SQL执行逻辑，容易产生隐形性能问题，线上故障排查难度大；MyBatis所有SQL由开发者手动编写，可自主优化索引、优化SQL逻辑，线上慢查询排查、优化极其方便。

第三，学习成本适中、团队适配性强。JdbcTemplate代码冗余、开发效率太低，不适合快速迭代的互联网项目；JPA学习门槛低但底层原理复杂，团队新人容易写出垃圾代码；MyBatis上手简单、逻辑清晰、代码规范统一，适合团队协作开发。

第四，生态完善、适配性强。MyBatis拥有丰富的生态插件，如分页插件、逻辑删除插件、乐观锁插件，同时完美适配Spring Boot、Spring Cloud生态，适配单体、微服务所有架构场景。

💡**最佳实践**：小型简单后台、CRUD快速迭代项目可选用JPA；企业级、互联网、高并发、复杂业务项目，统一使用MyBatis。

### 1.1.3 数据访问自动配置核心机制简述

Spring Boot 数据访问的核心优势源于**自动配置机制**，彻底告别原生Spring的繁琐手动配置。其核心底层逻辑是：基于条件注解，根据项目引入的依赖、配置文件参数，自动装配数据源、连接池、持久层核心组件。

核心执行流程：项目启动时，Spring Boot的 **DataSourceAutoConfiguration** 数据源自动配置类生效，检测项目中是否引入数据库驱动、连接池依赖；若依赖存在，且配置文件中配置了数据库连接参数，则自动创建DataSource数据源、初始化连接池；同时，引入MyBatis/JPA依赖后，对应自动配置类会自动完成Mapper扫描、SqlSession创建、参数绑定等核心操作，实现开箱即用。

📌**面试考点**：Spring Boot为什么无需手动配置数据源？核心依靠什么机制实现数据层自动化初始化？

**参考答案**：依靠Spring Boot自动配置机制，通过DataSourceAutoConfiguration自动配置类，结合条件注解按需装配数据源、连接池组件，开发者仅需配置数据库连接参数即可完成数据层初始化。

## 1.2 数据源核心概念

### 1.2.1 数据源、连接、连接池的关系

很多开发者容易混淆数据源、数据库连接、连接池三个概念，三者是**包含与被包含、管理与被管理**的关系，这里用生活化类比通俗解读：

**数据库连接（Connection）**：类比「水管」，是Java程序与MySQL数据库建立的专属通信通道，程序所有的增删改查操作，都需要通过这条通道传输数据。原生JDBC模式下，每次操作数据库都需要新建水管、用完销毁，频繁创建销毁会产生极大的性能损耗。

**数据源（DataSource）**：类比「供水总开关」，是数据库连接的工厂，负责统一创建、管理、获取数据库连接，是所有连接的统一入口，Java官方定义的标准接口，所有连接池都实现了该接口。

**连接池**：类比「蓄水池」，是数据源的具体实现，提前初始化一批数据库连接存入池中，程序需要操作数据库时，从池中直接获取连接，使用完毕后归还池中，无需频繁创建销毁连接，大幅提升数据库读写性能。

三者层级关系：**连接池 包含 多个数据库连接，数据源统一管理连接池**，Spring Boot项目中，开发者操作数据库，本质是从数据源对应的连接池中获取连接，完成数据交互。

### 1.2.2 Spring Boot 默认数据源 HikariCP 优势

在Spring Boot 2.x及以上版本中，**HikariCP 是官方默认的连接池实现**，完全替代了传统的C3P0、Druid，成为Spring Boot原生首选连接池，其核心优势聚焦「高性能、低开销、轻量稳定」。

第一，极致性能。HikariCP 优化了大量冗余代码，底层采用优化的FastPath机制，减少锁竞争、减少线程阻塞，数据库连接的获取、归还速度远超其他连接池，高并发场景下性能优势极其明显。

第二，极低的资源占用。相较于Druid、C3P0，HikariCP代码精简、无冗余依赖、无多余监控逻辑，内存占用极小，不会给项目带来额外性能负担。

第三，稳定性极强。作为Spring官方默认组件，经过海量生产项目验证，BUG极少、兼容性极强，适配所有Spring Boot版本，无版本冲突问题。

第四，配置极简。默认参数已经做了最优适配，开发者无需大量自定义配置，开箱即用，仅需根据业务并发量微调核心参数即可。

### 1.2.3 生产环境数据源核心诉求：高性能、稳定、可控

开发环境的数据源仅需实现「能连接、能读写」即可，但生产环境面向海量用户、高并发请求、7*24小时不间断运行，对数据源有严格的三大核心诉求，也是生产调优的核心目标：

**高性能**：高并发场景下，能够快速响应数据库读写请求，无连接等待、无请求堆积，支撑业务峰值流量。

**高稳定**：长期运行无连接泄露、无连接耗尽、无超时断开问题，不会因为连接池异常导致项目宕机、接口超时。

**可管控**：支持参数调优、状态监控、异常告警，能够快速排查连接超时、连接耗尽、SQL卡顿等线上问题。

# 2. 数据源与 HikariCP 连接池配置

数据源和连接池是数据访问的基础核心，所有数据库操作都依赖其正常工作。本节深度拆解Spring Boot数据源自动配置原理、数据库核心配置、HikariCP连接池参数调优、线上报错排查方案，全部为生产级实操内容，可直接落地使用，同时覆盖高频面试考点。

## 2.1 数据源自动配置原理

### 2.1.1 DataSourceAutoConfiguration 自动配置机制

Spring Boot 实现数据源自动配置的核心类是 **DataSourceAutoConfiguration**，该类是Spring Boot自动装配体系的核心组件，依托条件注解实现**按需装配、自动初始化**，彻底省略手动创建DataSource、连接池的代码。

核心条件注解机制：该配置类上标注了大量条件注解，仅在满足以下条件时才会生效：项目引入数据库驱动依赖、引入HikariCP连接池依赖、配置文件中存在数据库连接参数、容器中不存在开发者自定义的DataSource Bean。

核心执行流程：项目启动→扫描自动配置类→条件校验通过→自动读取application.yml中的数据库配置→创建HikariDataSource数据源实例→初始化连接池→将数据源Bean注入IOC容器，供全局持久层框架使用。

⚠️**避坑指南**：若开发者手动创建了自定义DataSource Bean，会覆盖Spring Boot默认的自动配置数据源，可能导致连接池参数失效、默认配置不生效问题，自定义数据源时需完整配置所有参数。

### 2.1.2 配置文件数据库四件套配置详解

Spring Boot 数据库基础配置核心为**数据库四件套**，分别是数据库地址、用户名、密码、驱动类，是数据源初始化的必备参数，缺一不可。下面提供生产标准配置，并逐行详解参数作用。

```yaml
# 生产级数据库基础配置（适配MySQL8.x）
spring:
  datasource:
    # 1.数据库连接地址：指定数据库IP、端口、库名、编码、时区、SSL配置
    url: jdbc:mysql://127.0.0.1:3306/test_db?useUnicode=true&characterEncoding=utf-8&serverTimezone=Asia/Shanghai&useSSL=false&allowMultiQueries=true
    # 2.数据库登录用户名
    username: root
    # 3.数据库登录密码
    password: 123456
    # 4.数据库驱动类全限定名
    driver-class-name: com.mysql.cj.jdbc.Driver
```

**参数详解**：

1. url参数：核心连接地址，末尾拼接的参数为生产必备配置，useUnicode和characterEncoding保证中文不乱码；serverTimezone指定时区，解决时区报错、时间偏差问题；useSSL关闭SSL加密，本地和内网生产环境无需开启；allowMultiQueries允许批量执行SQL，适配批量操作场景。

2. username/password：数据库登录凭证，生产环境需配置复杂密码，禁止明文密码可结合配置加密使用。

3. driver-class-name：驱动类路径，区分MySQL5.x和8.x版本，版本不可混用，否则启动报错。

### 2.1.3 驱动类适配（5.x / 8.x MySQL 驱动差异）

MySQL 5.x和8.x版本的驱动包、驱动类路径存在**重大差异**，版本不匹配会直接导致项目启动失败、数据源初始化异常，是新手高频踩坑点。

**MySQL 5.x 版本配置**：

驱动依赖：mysql:mysql-connector-java（无需指定runtime，通用依赖）

驱动类：`com.mysql.jdbc.Driver`

**MySQL 8.x 版本配置**：

驱动依赖：com.mysql:mysql-connector-j（必须指定runtime作用域）

驱动类：`com.mysql.cj.jdbc.Driver`

**核心差异与避坑**：8.x驱动包新增cj包层级，废弃了旧的驱动类；若8.x数据库使用5.x驱动类，会抛出类找不到异常；同时8.x必须指定时区参数，否则启动报错。

## 2.2 HikariCP 连接池核心参数详解

HikariCP 默认参数适配大多数普通项目，但高并发、大数据量生产场景必须手动调优核心参数，否则会出现连接耗尽、接口超时、请求堆积等问题。本节详解生产核心参数、调优逻辑、场景适配方案。

### 2.2.1 核心参数：最大连接数、最小空闲、超时时间、空闲存活时间

HikariCP 四大核心参数是生产调优的重中之重，每个参数都直接影响连接池性能和稳定性，下面附完整可落地配置与参数详解。

```yaml
# HikariCP 生产级核心参数配置
spring:
  datasource:
    hikari:
      # 最大连接数：连接池允许的最大连接数量
      maximum-pool-size: 10
      # 最小空闲连接数：连接池常驻的空闲连接数量
      minimum-idle: 5
      # 连接超时时间：空闲连接最大存活时间
      idle-timeout: 600000
      # 连接最长生命周期：池中连接的最大使用时长
      max-lifetime: 1800000
      # 获取连接超时时间：等待连接的最大超时时长
      connection-timeout: 30000
```

**参数逐行解析**：

1. maximum-pool-size（最大连接数）：连接池可创建的最大数据库连接数量，是限制数据库并发压力的核心参数，防止连接无限创建压垮数据库。

2. minimum-idle（最小空闲连接）：系统常驻的空闲连接，项目启动后提前初始化，应对突发流量，避免请求到来时新建连接产生延迟。

3. idle-timeout（空闲超时）：空闲连接超过该时长未被使用，自动释放销毁，节省数据库连接资源。

4. max-lifetime（最大生命周期）：所有连接的最大存活时间，无论是否空闲，到期自动销毁重建，避免连接长期存活产生失效、断开问题。

5. connection-timeout（获取连接超时）：程序从连接池获取连接的最大等待时间，超时则抛出连接耗尽异常。

### 2.2.2 参数适配业务场景（高并发/低并发场景调优）

连接池参数没有固定万能值，必须根据**业务并发量、数据库配置、服务器性能**动态调优，错误的参数配置会直接导致性能瓶颈或资源浪费。

**低并发后台管理系统（常规项目）**

场景特点：访问量小、并发低、无峰值流量，无需大量连接资源。

调优方案：maximum-pool-size=8~10，minimum-idle=3~5，默认超时参数即可，满足日常业务需求，避免资源闲置。

**高并发互联网项目（秒杀、订单、支付）**

场景特点：瞬时流量大、并发高、请求密集，容易出现连接等待。

调优方案：maximum-pool-size=15~20，minimum-idle=8~10，适当缩短max-lifetime，保证连接活性；同时严格控制SQL执行时长，避免长事务占用连接。

⚠️**核心避坑**：最大连接数并非越大越好，MySQL单库默认最大连接数为151，若项目连接池最大连接数超过数据库上限，会出现连接失败、数据库拒绝连接问题。

### 2.2.3 HikariCP 优于 Druid、C3P0 的核心原因

市面上主流连接池包含 C3P0、Druid、HikariCP，Spring Boot 官方舍弃前两者、默认选用HikariCP，核心原因聚焦**性能、稳定性、轻量化**三大维度：

1. 性能碾压：HikariCP 采用无锁设计、优化的并发控制机制，减少线程竞争，连接获取和归还速度远超Druid和C3P0，高并发场景性能优势显著。

2. 轻量化无冗余：C3P0代码臃肿、资源占用高、性能落后，基本已被淘汰；Druid内置大量监控、防SQL注入、日志统计功能，冗余过多，增加项目性能开销；HikariCP专注核心连接管理能力，无多余功能，极致轻量化。

3. 稳定性更高：Druid存在部分版本内存泄漏、连接异常问题；C3P0长期不更新、BUG无人修复；HikariCP由Spring官方维护，迭代稳定、BUG极少，适配所有生产场景。

💡**最佳实践**：常规业务项目、微服务项目优先使用默认HikariCP；需要SQL监控、防注入、日志统计的特殊项目，可按需引入Druid。

## 2.3 数据源常见报错与排查

数据源连接异常是线上最高频的故障类型，多数问题源于参数配置不当、连接池耗尽、连接泄露，本节汇总两大核心故障场景，提供完整原因分析和落地排查方案。

### 2.3.1 连接超时、连接耗尽问题

**问题现象**：项目运行一段时间后，接口报错、请求超时，控制台抛出 `Timeout waiting for idle connection` 连接耗尽异常，无法操作数据库。

**核心原因**

1. 连接池最大连接数设置过小，高并发场景下连接被瞬间占满，无空闲连接可用；

2. 业务存在长事务、慢SQL，长期占用数据库连接不释放，导致连接池耗尽；

3. 数据库服务器性能瓶颈、网络延迟过高，导致连接响应缓慢。

**排查与解决方案**

1. 查看项目日志，定位慢SQL、长事务，优化SQL语句、拆分大事务，减少连接占用时间；

2. 根据业务并发量适当调大最大连接数，适配峰值流量；

3. 检查数据库CPU、内存、IO使用率，排查数据库服务器性能瓶颈；

4. 调整连接超时参数，及时回收无效连接。

### 2.3.2 数据库连接泄露原因与定位方案

**连接泄露定义**：程序获取数据库连接后，因为代码异常、逻辑漏洞，未将连接归还连接池，导致连接永久占用，逐渐耗尽连接池所有资源，最终项目无法访问数据库。

**核心泄露场景**

1. 手动获取连接后，异常场景下未执行close归还操作；

2. 事务嵌套、异常捕获不规范，导致连接无法正常释放；

3. 第三方框架、自定义工具类连接管理异常。

**定位与解决方法**

1. 开启HikariCP连接泄露检测，配置泄露检测时长，自动打印泄露堆栈信息；

```yaml
# 开启连接泄露检测，超时2秒未归还则打印日志
spring.datasource.hikari.leak-detection-threshold: 2000
```

2. 统一代码规范，所有手动获取连接的场景，通过try-finally确保连接必然归还；

3. 排查项目中自定义数据操作工具类，修复连接释放漏洞。

📌**面试考点**：什么是数据库连接泄露？会导致什么问题？如何排查和解决？

**参考答案**：连接泄露是程序获取数据库连接后未正常归还连接池，导致连接永久占用；会引发连接耗尽、接口超时、项目宕机；可通过HikariCP泄露检测参数定位异常代码，通过代码规范确保连接强制归还。

# 3. MyBatis 极速集成与生产配置

## 3.1 MyBatis 自动配置原理

### 3.1.1 官方 starter 依赖引入

在Spring Boot项目中，MyBatis的极速集成核心依赖官方提供的自动化启动器，无需手动配置大量基础Bean，可快速实现ORM框架整合。目前企业主流使用`mybatis-spring-boot-starter`官方适配依赖，该依赖已整合MyBatis核心包、Spring适配包、JDBC适配依赖，同时兼容Spring Boot自动装配机制。

核心Maven依赖配置如下（适配Spring Boot 2.x/3.x通用版本）：

```xml
<!-- MyBatis Spring Boot 官方启动器 -->
<dependency>
    <groupId>org.mybatis</groupId>
    <artifactId>mybatis-spring-boot-starter</artifactId>
    <version>2.3.2</version>
</dependency>
<!-- 数据库驱动（以MySQL为例） -->
<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-j</artifactId>
    <scope>runtime</scope>
</dependency>
```

该启动器的核心优势：自动适配Spring Boot版本、自动注入SqlSessionFactory、SqlSessionTemplate等核心Bean、默认完成Mapper扫描适配，规避手动开发的冗余代码，同时保留配置自定义扩展能力。

### 3.1.2 MyBatisAutoConfiguration 自动装配流程

MyBatis的自动装配核心由`MyBatisAutoConfiguration`自动配置类实现，依托Spring Boot SPI机制（META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports）实现自动加载，完整装配流程分为5个核心步骤：

**1. 前置条件校验**：自动配置类生效需满足核心条件，项目中存在`DataSource`数据源Bean、存在MyBatis核心依赖包，且未被用户手动覆盖核心Bean，保证自动配置优先级合理。

**2. 自动创建 SqlSessionFactory**：根据项目中配置的数据源，自动构建`SqlSessionFactoryBean`，加载全局配置文件、映射文件，初始化MyBatis核心配置环境，包括事务管理、连接池适配、参数映射规则等。

**3. 注入 SqlSessionTemplate**：自动生成线程安全的SqlSessionTemplate，替代原生非线程安全的DefaultSqlSession，适配Spring事务管理，保证多线程环境下数据库操作安全。

**4. 开启 Mapper 动态扫描**：配合`@MapperScan`注解或自动扫描规则，扫描指定包下的Mapper接口，通过动态代理机制生成Mapper接口的代理实现类，交由Spring容器管理。

**5. 配置属性绑定**：将`application.yml/application.properties`中mybatis开头的配置，自动绑定到`MybatisProperties`配置实体类，完成全局参数初始化。

## 3.2 核心配置详解

### 3.2.1 Mapper 扫描、映射文件路径配置

Mapper扫描与映射文件路径是MyBatis整合的基础配置，直接决定项目能否正常加载Mapper接口与XML映射文件，生产环境需统一规范配置路径，避免扫描失效、映射文件找不到等问题。

**1. Mapper 接口扫描配置**：通过`@MapperScan`注解指定Mapper接口所在包路径，通常配置在Spring Boot启动类上，支持多包扫描、通配符扫描。核心作用是告诉Spring容器扫描指定包下的所有Mapper接口，生成代理Bean。

```java
@SpringBootApplication
@MapperScan("com.xxx.project.mapper") // 统一Mapper扫描路径
public class MybatisApplication {
    public static void main(String[] args) {
        SpringApplication.run(MybatisApplication.class, args);
    }
}
```

**2. XML映射文件路径配置**：默认情况下，MyBatis会加载resources目录下的映射文件，生产环境通常将XML文件统一存放，通过配置文件指定路径，规范项目结构。YAML配置如下：

```yaml
mybatis:
  mapper-locations: classpath:mapper/*.xml  # 单个目录映射文件
  # 多目录扫描配置
  # mapper-locations: classpath:mapper/**/*.xml,classpath:mapper2/**/*.xml
```

企业规范：统一将Mapper接口放在`com.xxx.mapper`包，XML文件放在`resources/mapper`目录，文件名与Mapper接口名保持一致，便于维护。

### 3.2.2 驼峰命名、日志打印、分页默认配置

该部分为生产高频通用配置，可解决数据库下划线字段与Java实体类驼峰属性映射、SQL日志排查、基础分页适配等核心问题。

**1. 驼峰命名自动转换**：数据库字段通常采用下划线命名（user_name、user_age），Java实体类采用驼峰命名（userName、userAge），开启驼峰转换后，无需手动配置字段映射，自动匹配对应属性。

**2. SQL日志打印**：生产开发、测试环境开启SQL日志，可打印完整执行SQL、参数、执行耗时，极大提升问题排查效率；生产环境可按需关闭，减少日志输出。

**3. 通用默认配置完整YAML示例**：

```yaml
mybatis:
  configuration:
    map-underscore-to-camel-case: true  # 开启驼峰命名自动转换
    log-impl: org.apache.ibatis.logging.stdout.StdOutImpl  # 控制台打印SQL日志
    default-page-size: 10  # 默认分页条数
    default-limit: 100  # 最大分页条数限制，防止超大分页
```

### 3.2.3 主键自增、返回值封装配置

1. **主键自增配置**：针对MySQL自增主键场景，配置后新增数据可自动回显主键ID，无需手动查询，适配新增后获取主键的业务场景。

全局开启主键自增回显配置：

```yaml
mybatis:
  configuration:
    use-generated-keys: true  # 开启自增主键回显
```

2. **返回值封装配置**：MyBatis支持对查询结果进行统一封装，生产中常用全局开启空值适配、结果集别名适配，同时统一返回值类型规范。开启`call-setters-on-nulls`配置，可保证查询结果为空时，依然调用实体类setter方法，避免前端接收字段缺失问题。

```yaml
mybatis:
  configuration:
    call-setters-on-nulls: true  # 空值字段调用setter赋值
    return-column-label: true  # 使用列别名作为返回字段名
```

## 3.3 MyBatis 生产级最佳实践

### 3.3.1 Mapper 接口与 XML 文件规范

生产环境必须统一Mapper层开发规范，降低协作成本、减少BUG、便于后期维护，核心规范如下：

**1. Mapper接口规范**：接口命名采用`表名+Mapper`格式，如UserMapper、OrderMapper；方法命名遵循业务语义，精准体现操作逻辑，如selectUserById、listUserByCondition、insertUser、updateUserById，禁止使用模糊命名；接口方法参数尽量封装DTO/实体类，避免多参数混乱；返回值统一规范，单条数据返回实体、多条返回List、分页返回Page对象。

**2. XML映射文件规范**：XML文件名必须与对应Mapper接口名完全一致；namespace必须精准对应Mapper接口全类名，否则会出现绑定失败；标签书写规范，select、insert、update、delete标签各司其职，禁止混用；所有SQL语句必须换行对齐，参数占位统一使用`#{}`，禁止使用`${}`防止SQL注入；新增、修改语句必须指定字段，禁止使用INSERT INTO 表 VALUES()、UPDATE 表 SET * 等不规范写法。

### 3.3.2 SQL 编写规范、避免全表查询

SQL编写规范是生产性能优化、数据安全的核心，也是线上问题高发点，核心约束如下：

**1. 基础编写规范**：禁止使用`select *`查询所有字段，按需查询业务所需字段，减少网络传输、内存占用；WHERE条件必须精准，禁止省略条件导致批量更新、批量删除；关联查询必须添加关联条件，避免笛卡尔积；排序、分组场景必须建立对应索引，提升查询效率。

**2. 严格规避全表查询**：生产环境绝对禁止无WHERE条件的SELECT、UPDATE、DELETE操作；批量查询必须添加分页限制，即使业务看似数据量少，也需兜底分页；针对大表查询，必须添加索引字段筛选条件，控制扫描行数；可通过MyBatis拦截器、代码检查插件强制校验全表操作SQL，提前拦截违规代码。

**3. 拓展规范**：复杂SQL拆分多条简单SQL，提升可读性和执行效率；避免子查询嵌套过深；优先使用联表查询替代循环查询，减少数据库连接次数。

### 3.3.3 分页插件 PageHelper 集成与使用规范

PageHelper是国内企业主流的MyBatis分页插件，可快速实现物理分页，无需手动编写分页SQL，生产集成简单、性能稳定。

**1. 依赖引入**

```xml
<!-- PageHelper 分页插件 -->
<dependency>
    <groupId>com.github.pagehelper</groupId>
    <artifactId>pagehelper-spring-boot-starter</artifactId>
    <version>1.4.6</version>
</dependency>
```

**2. 生产级配置**

```yaml
pagehelper:
  helper-dialect: mysql  # 指定数据库方言
  reasonable: true  # 合理化分页，页码小于1查第一页，大于总页数查最后一页
  support-methods-arguments: true  # 支持通过方法参数传递分页参数
  page-size-zero: false  # 禁止pageSize=0时查询全部数据，规避全表查询风险
```

**3. 标准使用规范**：分页代码必须紧跟查询语句，保证ThreadLocal参数生效；禁止跨方法分页；分页查询仅作用于紧邻的第一条SQL；统一返回PageInfo分页对象，包含总条数、总页数、当前页、数据列表等通用参数，适配前端分页展示。

```java
// 标准分页代码
PageHelper.startPage(pageNum, pageSize);
// 紧邻的查询SQL
List<User> userList = userMapper.listUserByCondition(condition);
// 封装分页结果
PageInfo<User> pageInfo = new PageInfo<>(userList);
```

# 4. 多数据源配置实战（生产高频）

## 4.1 多数据源场景与使用背景

### 4.1.1 业务分库、主从分离、多业务库场景

在中大型企业项目中，单数据源无法满足业务发展需求，多数据源是生产高频场景，核心应用场景分为三类：

**1. 业务分库场景**：随着业务拆分，系统拆分为用户库、订单库、商品库、日志库等独立业务库，不同业务模块操作不同数据库，实现业务解耦、数据隔离，避免单库数据量过大导致性能下降。

**2. 主从分离场景**：为优化数据库读写性能，搭建数据库主从架构，主库负责新增、修改、删除等写操作，从库负责查询读操作，实现读写分离，分担主库压力，提升系统并发能力，是高并发项目的标配架构。

**3. 多业务库兼容场景**：部分老旧系统迭代、跨系统数据对接场景中，项目需要同时对接多个独立数据库，实现多源数据读写、数据同步、数据聚合等业务需求。

### 4.1.2 多数据源核心难点：Bean 冲突、事务隔离

多数据源配置相较于单数据源，存在两大核心生产难点，也是线上BUG高发问题：

**1. Bean冲突问题**：Spring Boot默认自动配置只会创建一个全局DataSource、SqlSessionFactory、TransactionManager Bean，多数据源场景下，多个数据源Bean会出现重复注册、覆盖冲突问题，导致数据源切换失效、数据库连接混乱。核心原因是默认自动配置为单例Bean，无法适配多实例数据源场景。

**2. 事务隔离问题**：单数据源事务可通过`@Transactional`轻松实现回滚，但多数据源跨库操作时，Spring默认事务管理器仅支持单库事务，跨库操作无法实现原子性，出现部分库成功、部分库失败的脏数据问题，分布式事务不一致是核心痛点。

## 4.2 多数据源手动配置实现

### 4.2.1 关闭默认数据源自动配置

多数据源手动配置的第一步是关闭Spring Boot默认的数据源自动配置和MyBatis自动配置，避免默认Bean与手动创建的多数据源Bean冲突。

在启动类上通过`exclude`排除自动配置类：

```java
@SpringBootApplication(exclude = {
        DataSourceAutoConfiguration.class, // 关闭默认数据源自动配置
        MybatisAutoConfiguration.class // 关闭MyBatis默认自动配置
})
public class MultiDataSourceApplication {
    public static void main(String[] args) {
        SpringApplication.run(MultiDataSourceApplication.class, args);
    }
}
```

### 4.2.2 手动创建主库/从库数据源、SqlSession、MapperScan

关闭自动配置后，需手动编写配置类，分别创建主库、从库的数据源、SqlSessionFactory、SqlSessionTemplate，并通过分包扫描的方式隔离不同数据源的Mapper，实现数据源精准绑定。

**1. 多数据源配置参数（YAML）**

```yaml
# 主从多数据源配置
spring:
  datasource:
    master:
      url: jdbc:mysql://127.0.0.1:3306/db_master?useUnicode=true&characterEncoding=utf-8&serverTimezone=GMT%2B8
      username: root
      password: 123456
      driver-class-name: com.mysql.cj.jdbc.Driver
    slave:
      url: jdbc:mysql://127.0.0.1:3306/db_slave?useUnicode=true&characterEncoding=utf-8&serverTimezone=GMT%2B8
      username: root
      password: 123456
      driver-class-name: com.mysql.cj.jdbc.Driver
```

**2. 主库配置类（MasterDataSourceConfig）**：绑定主库数据源、创建主库SqlSession、扫描主库Mapper包

```java
@Configuration
@MapperScan(basePackages = "com.xxx.mapper.master", sqlSessionFactoryRef = "masterSqlSessionFactory")
public class MasterDataSourceConfig {

    @Bean("masterDataSource")
    @Primary // 优先主库数据源，解决默认数据源冲突
    @ConfigurationProperties(prefix = "spring.datasource.master")
    public DataSource masterDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean("masterSqlSessionFactory")
    public SqlSessionFactory masterSqlSessionFactory(@Qualifier("masterDataSource") DataSource dataSource) throws Exception {
        SqlSessionFactoryBean factoryBean = new SqlSessionFactoryBean();
        factoryBean.setDataSource(dataSource);
        // 设置主库映射文件路径
        factoryBean.setMapperLocations(new PathMatchingResourcePatternResolver().getResources("classpath:mapper/master/*.xml"));
        return factoryBean.getObject();
    }

    @Bean("masterSqlSessionTemplate")
    public SqlSessionTemplate masterSqlSessionTemplate(@Qualifier("masterSqlSessionFactory") SqlSessionFactory sqlSessionFactory) {
        return new SqlSessionTemplate(sqlSessionFactory);
    }
}
```

**3. 从库配置类（SlaveDataSourceConfig）**：同理实现从库数据源配置，分包扫描从库Mapper

### 4.2.3 多数据源业务调用方式

通过分包隔离后，业务调用无需手动切换数据源，Spring会根据Mapper所在包自动绑定对应数据源，实现精准调用：

1. 主库Mapper存放于`com.xxx.mapper.master`，所有新增、修改、删除写操作自动走主库数据源；

2. 从库Mapper存放于`com.xxx.mapper.slave`，所有查询读操作自动走从库数据源；

3. 业务层直接注入对应Mapper即可完成读写分离调用，代码无侵入、适配性强。

```java
@Service
public class UserService {

    // 注入主库Mapper，执行写操作
    @Autowired
    private UserMasterMapper userMasterMapper;

    // 注入从库Mapper，执行读操作
    @Autowired
    private UserSlaveMapper userSlaveMapper;

    // 写操作-主库
    public void addUser(User user) {
        userMasterMapper.insert(user);
    }

    // 读操作-从库
    public User getUserById(Long id) {
        return userSlaveMapper.selectById(id);
    }
}
```

## 4.3 多数据源常见坑点

### 4.3.1 数据源切换不生效

数据源切换不生效是多数据源最常见问题，核心原因及解决方案如下：

**1. 配置类未指定SqlSessionFactory**：多个数据源配置类未通过`sqlSessionFactoryRef`指定对应SqlSession，导致Mapper统一绑定默认数据源，切换失效。解决方案：每个数据源的`@MapperScan`必须精准绑定对应SqlSessionFactory Bean名称。

**2. Mapper包路径混淆**：主从库Mapper包路径交叉、未严格分包，导致扫描错乱。解决方案：严格区分主从Mapper包路径、XML文件路径，一一对应。

**3. 未添加@Primary注解**：多数据源场景下无默认主数据源，导致Bean注入冲突。解决方案：主库数据源配置添加`@Primary`，优先加载主库Bean。

**4. 缓存导致数据源复用**：MyBatis缓存、Spring容器Bean缓存导致数据源未刷新，切换失效。解决方案：重启项目刷新Bean，关闭测试环境二级缓存。

### 4.3.2 多数据源事务无法回滚问题说明

Spring默认的声明式事务`@Transactional`仅支持**单数据源事务**，无法支持跨多数据源事务。当业务方法中同时操作主库、从库（或多个业务库），若其中一个库操作失败，其他库已执行的操作**无法自动回滚**，会造成数据不一致。

**问题根源**：Spring事务管理器基于单个DataSource实现，每个数据源对应独立的连接和事务，跨库操作属于多个独立本地事务，无法实现原子性。

**生产解决方案**：

1. **业务规避**：尽量将跨库操作拆分，单个方法只操作一个数据源，从业务层面规避分布式事务问题；

2. **手动事务控制**：通过编程式事务，分别手动开启、提交、回滚多个数据源事务；

3. **引入分布式事务框架**：核心业务场景引入Seata、Atomikos等分布式事务框架，实现跨库事务最终一致性，满足生产高可用要求。

# 5. Spring 事务核心机制（面试+生产核心）

Spring事务是Spring框架最核心的功能之一，也是企业项目开发中保障数据一致性的核心方案。相较于原生JDBC手动编写事务开启、提交、回滚代码，Spring通过**声明式事务**极大简化了事务开发，基于AOP动态代理实现事务的无侵入增强。同时Spring对数据库原生事务进行了封装、扩展，提供了独立的事务传播机制、隔离级别配置、超时控制、回滚规则等高级特性，适配复杂的企业级业务场景。

绝大多数线上数据异常问题（订单扣款成功但订单状态未更新、库存超卖、批量数据部分成功部分失败），根源都是事务使用不当、隔离级别适配错误、传播机制选择错误。本章将从基础到底层原理，再到实战场景，全方位拆解Spring事务核心机制。

## 5.1 事务核心基础认知

想要精通Spring事务，必须先掌握数据库原生事务的基础特性。Spring事务本质是对数据库事务的封装和增强，所有Spring事务的底层能力，都依赖于数据库原生事务的支持。只有理解基础事务特性、隔离级别，才能精准适配业务场景，避免底层原理盲区导致的线上bug。

### 5.1.1 事务四大特性 ACID 通俗解读

事务（Transaction）是数据库操作的最小执行单元，由一组SQL语句组成，这组语句要么**全部执行成功**，要么**全部执行失败回滚**，不会出现中间状态。事务的四大特性简称**ACID**，是所有关系型数据库事务的核心规范，也是Spring事务实现的底层基础。下面结合生活化场景+业务场景通俗解读四大特性，同时标注生产价值与面试考点。

**1. 原子性（Atomicity）**

通俗定义：**事务是一个不可分割的原子单元，整体要么全部成功，要么全部失败回滚，不存在部分执行成功的情况**。

核心原理：数据库通过undo log（回滚日志）实现原子性。事务执行前会记录所有操作的前置数据快照，若事务执行过程中出现异常、断电、程序报错等问题，数据库会根据undo log回滚所有已执行的SQL，保证数据恢复到事务执行前的状态。

生产场景：最典型的转账业务，A给B转账100元，包含两个核心操作：A账户余额-100、B账户余额+100。如果扣款成功、入账失败，没有原子性就会出现资金丢失的严重问题。事务原子性会保证两个操作同时成功或同时失败。

**2. 一致性（Consistency）**

通俗定义：**事务执行前后，数据库的整体数据完整性、约束规则保持一致，不会出现非法数据状态**。

核心原理：一致性是事务的最终目标，原子性、隔离性、持久性都是为了保障一致性。数据库会通过主键唯一、外键约束、字段非空、数据校验规则等机制，杜绝事务执行后产生脏数据、非法数据。

生产场景：数据库总资金守恒规则，系统所有用户账户总金额是固定值，转账事务执行前后，总金额不会发生变化。如果事务执行后总金额异常，就破坏了事务一致性。

**3. 隔离性（Isolation）**

通俗定义：**多个事务并发执行时，事务之间相互隔离、互不干扰，每个事务都感知不到其他并发事务的执行状态**。

核心原理：数据库通过锁机制、MVCC（多版本并发控制）实现事务隔离性。不同的隔离级别对应不同的隔离强度，隔离性越高，并发性能越低，数据安全性越高。

生产场景：多个用户同时下单扣库存，事务隔离性可以避免多个事务同时读取、修改同一库存数据，导致库存超卖、数据错乱问题。

**4. 持久性（Durability）**

通俗定义：**事务一旦提交成功，数据的修改会永久写入数据库，即使服务器断电、重启、崩溃，数据也不会丢失**。

核心原理：数据库通过redo log（重做日志）实现持久性。事务提交前，所有操作日志都会持久化到磁盘，数据库重启后会根据redo log恢复已提交的事务数据。

生产场景：订单支付成功后，事务提交，即使服务器立刻宕机，重启后订单支付状态、用户扣款数据依然存在，不会丢失。

**最佳实践**：四大特性中，**原子性、隔离性、持久性是手段，一致性是最终目的**。日常事务bug，90%都是隔离性适配不当、原子性失效导致的一致性破坏。

**面试考点**：面试官常问「ACID中哪个是核心？」「原子性和一致性的区别？」

参考答案：一致性是核心目标，其余三者是保障一致性的技术手段；原子性侧重**操作不可分割**，一致性侧重**数据状态合法**。

### 5.1.2 数据库隔离级别详解与场景适配

事务隔离性是为了解决**并发事务数据干扰问题**。数据库并发执行多个事务时，会产生三类读异常问题，为了解决这些问题，数据库定义了四种隔离级别，隔离级别从低到高，安全性递增，并发性能递减。Spring事务的隔离级别**完全基于数据库原生隔离级别**，仅做封装配置。

#### 一、并发事务三大读问题

1. **脏读**：一个事务读取到了另一个事务**未提交**的修改数据，随后另一个事务回滚，导致读取的数据是无效脏数据。是最严重的并发数据问题。

2. **不可重复读**：同一个事务内，两次读取同一数据，结果不一致。原因是中间有其他事务提交了修改并更新数据。侧重**数据修改**。

3. **幻读**：同一个事务内，两次查询批量数据，结果条数不一致。原因是中间有其他事务提交了新增/删除数据。侧重**数据新增、删除**。

#### 二、四种数据库隔离级别（MySQL InnoDB）

MySQL InnoDB引擎支持四种标准隔离级别，默认隔离级别为**可重复读（REPEATABLE READ）**，也是生产最常用的级别。

**1. 读未提交（READ UNCOMMITTED）**

特性：所有事务可以读取其他事务未提交的数据，无任何隔离效果。

问题：存在**脏读、不可重复读、幻读**所有并发问题。

生产场景：几乎不用，仅用于数据日志、临时统计等对数据一致性无要求的场景。

**2. 读已提交（READ COMMITTED）**

特性：事务只能读取其他事务**已提交**的数据，实时感知其他事务的数据修改。

解决问题：彻底解决**脏读**问题。

遗留问题：存在不可重复读、幻读问题。

底层原理：基于MVCC的最新快照读，每次查询都会读取最新已提交数据。

生产场景：互联网高并发查询业务，如商品列表、资讯查询，牺牲部分一致性换取高并发性能。

**3. 可重复读（REPEATABLE READ）【MySQL默认】**

特性：同一个事务内，多次读取同一数据，结果始终一致，不受其他事务修改影响。

解决问题：解决脏读、不可重复读问题。

遗留问题：存在幻读问题（InnoDB通过间隙锁极大缓解幻读，但未彻底解决）。

底层原理：事务开启时生成数据快照，整个事务周期内复用该快照数据。

生产场景：绝大多数金融、订单、库存核心业务，是**生产首选隔离级别**，平衡了安全性和并发性能。

**4. 串行化（SERIALIZABLE）**

特性：最高隔离级别，所有事务串行执行，完全杜绝并发干扰。

解决问题：彻底解决脏读、不可重复读、幻读所有问题。

缺点：并发性能极低，所有读写操作加表锁，容易产生阻塞、超时。

生产场景：极低并发、极高数据一致性场景，如资金结算、账务对账、核心配置修改。

#### 三、隔离级别对比总结

| 隔离级别                                                     | 脏读   | 不可重复读 | 幻读     | 并发性能 | 生产使用率 |
| ------------------------------------------------------------ | ------ | ---------- | -------- | -------- | ---------- |
| 读未提交                                                     | 存在   | 存在       | 存在     | 最高     | 极低       |
| 读已提交                                                     | 不存在 | 存在       | 存在     | 较高     | 中等       |
| 可重复读（默认）                                             | 不存在 | 不存在     | 极少出现 | 中等     | 极高       |
| 串行化                                                       | 不存在 | 不存在     | 不存在   | 最低     | 极低       |
| **避坑指南**：不要随意修改默认隔离级别！很多开发者为了解决幻读问题强行改为串行化，会导致系统并发吞吐量断崖式下跌，引发大量接口超时、阻塞问题。绝大部分业务无需彻底杜绝幻读，可通过业务锁、唯一索引规避。 |        |            |          |          |            |

### 5.1.3 Spring 事务 vs 数据库事务的关系

很多开发者容易混淆Spring事务和数据库事务，误以为Spring事务是独立于数据库的全新事务机制，这是核心认知误区。本小节彻底厘清两者的关系，理解Spring事务的本质定位。

**1. 核心本质：Spring事务是数据库事务的「高级封装与增强工具」**

数据库事务是**底层基础**，是MySQL等数据库原生支持的事务能力，包含ACID特性、隔离级别、锁机制、日志机制等，是事务生效的根本保障，没有数据库事务，Spring事务完全无法生效。

Spring事务是**上层应用封装**，本身不创造事务能力，只是对JDBC原生事务代码的封装、简化、扩展。原生JDBC事务需要手动编写setAutoCommit(false)、commit()、rollback()代码，繁琐且冗余，Spring通过AOP机制，实现事务的自动开启、提交、回滚、异常捕获，大幅提升开发效率。

**2. 两者能力差异与补充**

数据库事务：仅提供基础事务能力（开启、提交、回滚、隔离级别），无法适配复杂的业务嵌套场景，没有事务传播机制、精细化回滚规则、超时控制等能力。

Spring事务：在数据库事务基础上，扩展了**7种事务传播机制、精细化异常回滚配置、事务超时时间、只读事务、事务嵌套控制**等高级特性，适配复杂的分层业务场景（Service多层嵌套调用）。

**3. 执行层级关系**

业务代码 → Spring事务拦截器（AOP） → 接管JDBC连接 → 调用数据库原生事务 → 数据库执行SQL并保障事务特性

简单来说：**Spring管事务规则，数据库管事务落地**。Spring定义事务如何传播、何时回滚、超时多久，数据库负责真正执行事务的原子性、持久性、隔离性。

**面试高频题**：Spring事务和数据库事务的区别？

参考答案：1. 层级不同：数据库事务是底层原生能力，Spring事务是上层封装增强；2. 能力不同：数据库事务提供基础ACID，Spring事务提供传播机制、精细化回滚、超时控制等高级能力；3. 依赖关系：Spring事务完全依赖数据库事务，无法独立生效；4. 作用范围：数据库事务针对单连接SQL执行，Spring事务适配Java分层业务嵌套场景。

## 5.2 Spring 事务核心原理

掌握基础事务认知后，需要深入Spring事务底层原理，理解事务如何自动生效、为何会出现事务失效问题。Spring事务的核心底层是**AOP动态代理机制**，无需手动编码即可实现事务增强，这也是声明式事务的核心优势。本小节完整拆解底层原理、执行链路、注解机制。

### 5.2.1 基于 AOP 动态代理实现事务

Spring声明式事务（@Transactional）的核心实现原理是**Spring AOP + 动态代理**，这是事务自动生效的核心本质。如果不理解AOP代理，就无法解决事务失效、嵌套事务异常等线上问题。

**1. 核心设计思想**

Spring事务遵循**无侵入增强**原则，开发者只需在类或方法上添加@Transactional注解，无需修改业务代码，Spring会自动通过动态代理生成目标类的代理对象，在方法执行前后进行事务增强：方法执行前开启事务、方法正常执行完毕提交事务、方法抛出异常回滚事务。

**2. 动态代理两种实现方式**

Spring事务的代理模式分为两种，框架自动适配：

JDK动态代理：目标类**实现接口**时使用，基于接口生成代理对象，性能较高。

CGLIB动态代理：目标类**无接口**时使用，基于子类继承生成代理对象，覆盖原有方法实现增强。

**3. AOP事务核心组件**

TransactionInterceptor：事务拦截器，AOP的核心切面，负责拦截目标方法，执行事务的开启、提交、回滚逻辑。

PlatformTransactionManager：事务管理器，真正对接数据库的核心接口，负责获取数据库连接、控制事务状态。

TransactionDefinition：事务定义信息，存储隔离级别、传播机制、超时时间、只读状态等配置。

**避坑指南（高频失效场景）**：Spring事务基于代理生效，因此**类内部方法自调用、非public方法、静态方法、final方法**都会导致代理失效，事务完全不生效，这是生产环境最常见的事务bug！

### 5.2.2 事务执行完整链路

为了清晰理解事务执行全过程，我们以「带@Transactional注解的Service方法调用」为例，拆解**完整执行链路**，从请求进入到事务结束，每一步底层逻辑清晰透明。

**步骤1：获取代理对象**

Spring容器启动时，扫描所有带@Transactional注解的类和方法，自动为目标Bean生成代理对象。开发者调用Service方法时，实际调用的是**代理对象的方法**，而非原生对象。

**步骤2：事务拦截器前置拦截**

TransactionInterceptor拦截目标方法执行，读取方法/类上的@Transactional注解配置，解析事务传播机制、隔离级别、超时时间、回滚规则等参数，生成TransactionDefinition事务定义对象。

**步骤3：获取数据库连接、开启事务**

事务管理器PlatformTransactionManager根据配置，从数据库连接池获取有效连接，关闭连接自动提交（setAutoCommit(false)），正式开启Spring事务，绑定当前线程数据库连接。

**步骤4：执行目标业务方法**

代理对象放行，执行开发者编写的业务代码、SQL操作，所有数据库操作共用同一个事务连接。

**步骤5：事务后置处理（核心分支）**

分支1：**方法正常执行完毕，无异常** → 事务管理器执行commit提交事务，释放数据库连接，事务结束。

分支2：**方法抛出指定异常** → 事务管理器执行rollback回滚事务，撤销所有SQL操作，释放连接。

分支3：**抛出未指定回滚的异常** → 默认不回滚，直接提交事务（Spring默认只对RuntimeException和Error回滚）。

**步骤6：资源释放**

无论事务提交还是回滚，最终都会释放数据库连接，归还至连接池，避免连接泄露。

**最佳实践**：Spring事务是**线程级**事务，同一个线程内的所有嵌套方法调用，默认共用同一个事务连接，跨线程事务完全隔离，互不影响。

### 5.2.3 声明式事务 @Transactional 底层机制

@Transactional是Spring开发中使用最多的**声明式事务注解**，无需手动编码事务逻辑，一行注解即可实现事务控制。本小节详解注解属性、底层执行逻辑、默认规则与自定义配置。

**1. 注解核心属性与生产用法**

@Transactional注解包含多个核心属性，精准控制事务行为，以下是生产常用配置及注释说明：

```java
// 完整生产级@Transactional配置示例
@Transactional(
    // 事务传播机制：默认REQUIRED，最常用
    propagation = Propagation.REQUIRED,
    // 事务隔离级别：默认跟随数据库，核心业务可指定可重复读
    isolation = Isolation.REPEATABLE_READ,
    // 事务超时时间：单位秒，超时自动回滚，避免长事务阻塞
    timeout = 30,
    // 是否只读事务：查询业务开启，优化数据库性能
    readOnly = false,
    // 指定触发回滚的异常：默认只回滚运行时异常，手动扩展检查型异常
    rollbackFor = Exception.class,
    // 指定不触发回滚的异常
    noRollbackFor = BusinessException.class
)
public void orderCreate() {
    // 业务代码
}
    
```

**2. 底层生效规则**

注解生效范围：**优先读取方法上的注解配置，方法无注解则继承类上的注解配置**，方法配置优先级高于类配置。

默认回滚规则：Spring默认**只对RuntimeException（运行时异常）和Error（系统错误）触发回滚**，对Exception检查型异常（如IOException）默认不回滚，这是高频坑点！

默认传播规则：默认REQUIRED传播机制，有事务则加入，无事务则新建事务。

**3. 声明式事务优势与劣势**

优势：代码无侵入、开发高效、统一规范、易于维护，99%的业务场景首选。

劣势：基于AOP代理，存在事务失效场景；精细化控制能力弱于编程式事务，极端复杂事务场景需手动编程式事务实现。

**避坑指南**：生产中必须手动指定rollbackFor = Exception.class！否则抛出文件异常、校验异常等检查型异常时，事务不会回滚，导致数据部分更新，产生脏数据。

## 5.3 事务传播机制详解（面试必考）

事务传播机制是Spring事务**最核心、面试频率最高、生产最容易出错**的知识点。传播机制用于定义：**当一个带有事务的方法调用另一个带事务的方法时，内外事务如何协作、传递、嵌套**。

数据库原生事务没有传播机制，这是Spring独有的高级特性，专门适配Java分层开发（Controller→Service→Service嵌套调用）的业务场景。Spring共定义7种事务传播机制，每种机制的事务创建、嵌套、回滚规则完全不同。

### 5.3.1 7种传播机制通俗解读

我们统一定义术语：**外层方法**：调用方方法；**内层方法**：被调用的嵌套方法。下面用通俗语言+核心规则，全覆盖7种传播机制，标注核心特性与区别。

**1. REQUIRED（默认传播机制）**

通俗解读：**有事务就加入，无事务就新建**。

核心规则：如果外层方法存在事务，内层方法加入当前事务，两者共用同一个事务；如果外层无事务，内层新建独立事务。

事务关系：内外事务合并为一个整体，任意位置异常，整体全部回滚。

**2. SUPPORTS**

通俗解读：**有事务就跟随，无事务就非事务执行**。

核心规则：外层有事务则加入共用事务；外层无事务，内层直接以非事务方式运行。

特点：被动适配事务，不主动创建事务。

**3. MANDATORY**

通俗解读：**强制依赖外层事务，无事务直接报错**。

核心规则：必须运行在已有事务中，外层无事务则直接抛出异常，拒绝执行。

特点：强依赖父事务，适用于必须在事务中执行的核心数据操作。

**4. REQUIRES_NEW**

通俗解读：**无论外层有无事务，强制新建独立事务**。

核心规则：外层有事务，**暂停外层事务**，新建内层独立事务；外层无事务，直接新建事务。内外事务完全独立，互不干扰。

特点：最高独立性，内层事务回滚不会影响外层，外层回滚也不会撤销内层已提交数据。

**5. NOT_SUPPORTED**

通俗解读：**强制非事务执行，挂起所有外层事务**。

核心规则：无论外层是否存在事务，内层始终以非事务方式运行，外层事务会被挂起，内层执行完毕后恢复。

特点：内层操作不支持事务，执行失败不会回滚数据。

**6. NEVER**

通俗解读：**强制无事务环境，有事务直接报错**。

核心规则：必须在非事务环境执行，检测到外层存在事务则直接抛出异常。

特点：与MANDATORY完全相反，杜绝事务环境。

**7. NESTED（嵌套事务）**

通俗解读：**外层有事务则嵌套执行，无事务则新建事务**。

核心规则：外层存在事务时，内层作为外层事务的**子事务**，存在独立保存点；外层无事务时，等同于REQUIRED新建事务。

关键区别：子事务可以独立回滚，不会影响外层事务，但外层事务回滚会带动所有子事务回滚。

#### 5.3.2 各类传播机制适用业务场景

理解传播机制定义后，核心是**业务场景精准适配**，避免乱用导致事务数据异常。下面结合生产真实业务，匹配每种传播机制的适用场景，给出最佳实践。

**1. REQUIRED（默认）—— 90%常规业务首选**

适用场景：单流程连贯业务，所有操作必须同时成功或失败。如订单创建（创建订单、扣库存、生成流水）、用户注册（新增用户、初始化权限、生成账户）。

优势：保证整体业务原子性，逻辑简单、安全稳定。

**2. SUPPORTS —— 查询类业务专属**

适用场景：纯数据查询接口、统计查询、列表查询。有事务时跟随事务，无事务时快速查询，提升性能。

禁忌：不能用于数据新增、修改、删除业务，无事务时数据异常无法回滚。

**3. MANDATORY —— 核心数据校验、附属操作**

适用场景：必须依附主事务的附属操作，如订单创建后的日志记录、数据校验、状态同步。保证所有核心操作必须在事务内执行，杜绝脏数据。

**4. REQUIRES_NEW —— 独立日志、异步记录、核心兜底操作**

适用场景：需要独立提交、不受主业务失败影响的操作。例如：订单失败记录错误日志、用户操作日志记录、业务流水留存。即使主事务回滚，日志数据依然可以保存，方便问题排查。

**5. NOT_SUPPORTED —— 非核心统计、缓存更新**

适用场景：无需事务保障的非核心操作，如缓存刷新、数据统计更新、日志打印。非事务执行，减少事务阻塞，提升接口响应速度。

**6. NEVER —— 绝对独立的非事务操作**

适用场景：禁止事务嵌套的操作，如批量数据导入、数据初始化、第三方接口调用，避免事务嵌套导致的锁超时问题。

**7. NESTED —— 局部可回滚的复杂业务**

适用场景：大事务中存在局部可独立回滚的子业务。例如：批量下单业务，主事务管控整体订单，单个商品扣减失败可单独回滚，不影响其他商品下单逻辑。

### 5.3.3 嵌套事务、内外事务协作规则

嵌套事务是面试和生产的重难点，核心区分**REQUIRES_NEW（独立事务）**和**NESTED（嵌套事务）**的协作规则，这是90%开发者的知识盲区。本小节通过场景案例、异常规则、对比差异彻底讲透。

#### 一、REQUIRED 内外事务协作（默认）

1. 内外事务合并为**同一个事务**，无独立边界。

2. 内层异常：内层抛异常 → 整体事务标记回滚 → 外层必然回滚。

3. 外层异常：外层抛异常 → 整体事务回滚 → 内层所有操作回滚。

4. 特殊坑点：内层异常被try-catch捕获时，事务依然会被标记回滚，外层无法正常提交，会报「事务已标记回滚，无法提交」异常。

#### 二、REQUIRES_NEW 内外事务协作（独立事务）

1. 内外是**两个完全独立的事务**，外层事务挂起，内层事务执行完毕后恢复外层。

2. 内层异常：内层回滚，**不影响外层事务**，外层可以正常提交。

3. 外层异常：外层回滚，**不影响内层已提交的事务数据**。

4. 核心价值：实现核心业务与日志、兜底操作的事务隔离。

#### 三、NESTED 内外事务协作（嵌套事务）

1. 外层有事务时，内层是**子事务**，基于外层事务的保存点执行，共用一个数据库连接。

2. 内层异常：可独立回滚子事务，**不影响外层主事务继续执行**。

3. 外层异常：外层主事务回滚，**带动所有内层子事务全部回滚**。

4. 外层无事务：等同于REQUIRED，新建独立事务。

#### 四、核心对比（面试必考）

问题：REQUIRES_NEW 和 NESTED 的核心区别？

1. 事务独立性：REQUIRES_NEW是两个独立事务；NESTED是父子嵌套事务，从属主事务。

2. 外层回滚影响：REQUIRES_NEW内层已提交数据不受影响；NESTED内层必然跟随外层回滚。

3. 连接资源：REQUIRES_NEW占用新的数据库连接；NESTED复用外层连接。

4. 适用场景：独立日志用REQUIRES_NEW，局部子业务回滚用NESTED。

**生产避坑**：禁止在大事务中滥用REQUIRES_NEW，频繁创建新事务、挂起恢复事务，会极大增加数据库锁竞争，导致接口超时、并发性能下降。

# 6. 事务失效场景与生产避坑（本章重点）

在Spring Boot项目开发中，**Spring声明式事务**是数据一致性保障的核心手段，基于AOP动态代理机制实现事务的自动开启、提交与回滚，极大简化了事务编码成本。但在实际生产环境中，大量事务异常、数据不一致问题并非框架Bug，而是开发者对事务底层原理、生效条件、传播机制不熟悉导致的人为失效。

本章重点拆解生产中**最常见的八大事务失效场景**，逐一分析失效原理、复现案例、解决方案，同时针对性讲解大事务优化方案、数据访问层高频坑点，帮助开发者彻底规避生产事务隐患，同时覆盖面试高频考点，实现「落地实操+面试通关」双重目标。

## 6.1 经典事务失效八大场景

Spring事务的核心生效前提：**基于Spring AOP动态代理拦截目标方法，只有代理对象调用的public方法，且满足异常传播、数据库支持、传播机制匹配等条件，事务才会生效**。所有事务失效场景，本质都是破坏了AOP代理执行流程或事务触发条件。下面逐一详解八大经典失效场景。

### 6.1.1 方法非 public 修饰

**概念定义**：被`@Transactional`注解标记的方法，如果是private、protected、default（包访问权限）修饰，Spring事务会完全失效，无法开启事务、也无法触发回滚。

**核心原理**：Spring事务基于JDK动态代理或CGLIB动态代理实现。JDK动态代理仅支持拦截**public方法**；CGLIB虽然可以代理非public方法，但Spring源码中做了强制校验，`TransactionInterceptor`事务拦截器只会对public方法进行事务增强。非public方法不会被代理拦截，事务注解直接失效，等同于普通方法执行。

**场景与价值**：新手开发常为了封装代码，将事务方法定义为private，导致异常时数据不回滚，引发数据库脏数据。该场景是入门最常见的事务失效问题，100%可复现。

**实操错误示例**

```java
@Service
public class UserService {

    // 错误：非public修饰，事务完全失效
    @Transactional
    private void updateUser() {
        // 执行数据库更新操作
        userMapper.updateById(new User());
        // 手动抛出异常，测试回滚
        int i = 1 / 0;
    }
}
```

**验证与排错**：执行上述代码，抛出算术异常后，数据库数据依然更新成功，无任何回滚。排查方式：查看项目日志，无`Creating new transaction`事务开启日志，直接判定为非public方法导致失效。

**⚠️避坑指南**：所有需要事务控制的方法，**必须强制使用public修饰**，禁止使用private、protected、default权限。

**📌面试考点**：提问：为什么非public方法事务会失效？答：Spring事务基于AOP代理实现，事务拦截器仅拦截public方法，非public方法无法被代理增强，事务注解不生效，与代理机制底层逻辑强绑定。

### 6.1.2 同类方法自调用导致无代理

**概念定义**：在同一个Service类中，一个普通方法直接调用本类中被`@Transactional`修饰的事务方法，会导致事务失效，该现象称为**自调用失效**。

**核心原理**：Spring AOP代理的核心是**通过代理对象执行方法**，只有代理对象调用方法才会触发事务拦截、增强逻辑。而同类方法自调用时，是**当前类的原生对象（this）**调用方法，并非Spring容器生成的代理对象，因此不会触发AOP事务增强，事务直接失效。

**场景与价值**：生产中常出现一个业务方法需要调用本类事务方法的场景，开发者直接内部调用，导致事务不生效，异常数据无法回滚，是中高级开发高频踩坑点。

**实操错误示例**

```java
@Service
public class OrderService {

    // 普通方法，内部调用本类事务方法
    public void createOrder() {
        // 自调用：this原生对象调用，无代理增强
        saveOrder();
    }

    // 事务方法
    @Transactional
    public void saveOrder() {
        orderMapper.insert(new Order());
        // 抛出异常，测试回滚
        int i = 1 / 0;
    }
}
```

**验证与排错**：调用`createOrder()`方法，程序报错但数据库数据插入成功，无事务回滚。排查要点：确认事务方法是否被本类普通方法直接调用，无代理介入。

**💡最佳实践**：1、通过**Spring上下文获取当前类代理对象**调用事务方法；2、将事务方法抽离到独立Service，通过注入Bean调用；3、使用`AopContext.currentProxy()`获取代理对象。

**📌面试考点**：提问：同类方法自调用事务失效的根本原因？答：Spring事务依赖AOP代理对象增强，自调用使用this原生对象，未经过代理拦截，事务逻辑无法执行，导致失效。

### 6.1.3 异常被 try-catch 捕获未抛出

**概念定义**：事务方法内部通过try-catch捕获了所有异常，未将异常向上抛出，Spring无法感知异常，不会触发事务回滚，导致数据不一致。

**核心原理**：Spring事务回滚的触发机制是：**事务拦截器捕获方法抛出的异常，才会执行回滚逻辑**。如果开发者手动try-catch捕获异常且不抛出，异常会被消化，代理层无法检测到异常，默认认为方法正常执行，自动提交事务。

**场景与价值**：生产中为了避免程序崩溃，开发者习惯全局try-catch，但未处理事务回滚逻辑，导致业务异常时数据库数据错误提交，是线上数据异常的高频原因。

**实操错误示例**

```java
@Service
public class GoodsService {

    @Transactional
    public void updateGoodsStock() {
        try {
            goodsMapper.updateStock();
            // 抛出异常
            int i = 1 / 0;
        } catch (Exception e) {
            // 仅打印日志，未抛出异常，事务无法感知
            log.error("更新库存异常", e);
        }
    }
}
```

**验证与排错**：方法执行报错日志，但数据库库存更新成功，事务未回滚。排查核心：事务方法内异常被本地捕获，无异常向外抛出。

**💡最佳实践**：1、捕获异常后**手动抛出运行时异常**，触发自动回滚；2、捕获异常后手动调用`TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()`强制回滚事务。

**正确示例（手动回滚）**

```java
@Transactional
public void updateGoodsStock() {
    try {
        goodsMapper.updateStock();
        int i = 1 / 0;
    } catch (Exception e) {
        log.error("更新库存异常", e);
        // 手动强制回滚事务
        TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
    }
}
```

**⚠️避坑指南**：事务方法中禁止无脑try-catch吞异常，必须保证异常可被代理层感知，或手动触发回滚。

### 6.1.4 抛出非运行时异常不回滚

**概念定义**：Spring事务**默认仅对RuntimeException（运行时异常）和Error触发回滚**，如果方法抛出CheckedException（受检异常、编译时异常），事务不会回滚。

**核心原理**：Spring事务默认回滚规则源码定义：`rollbackFor = RuntimeException.class、Error.class`。受检异常（如IOException、SQLException、自定义普通Exception）不属于默认回滚范围，Spring认为这类异常是业务可控异常，默认提交事务。

**场景与价值**：开发者自定义受检异常、抛出IO异常时，误以为事务会自动回滚，最终导致异常报错但数据提交，引发数据一致性问题。

**实操错误示例**

```java
@Service
public class FileService {

    // 默认配置，不支持受检异常回滚
    @Transactional
    public void fileOperate() throws Exception {
        fileMapper.insertFileInfo();
        // 抛出受检异常，事务不回滚
        throw new Exception("文件操作失败");
    }
}
```

**验证与排错**：抛出Exception后，程序报错但数据库数据插入成功。排查方式：检查抛出异常类型是否为受检异常，核对事务注解回滚规则。

**💡最佳实践**：手动指定`rollbackFor = Exception.class`，让所有异常（运行时+受检）都触发回滚，适配绝大多数业务场景。

**正确示例**

```java
// 所有异常均触发回滚
@Transactional(rollbackFor = Exception.class)
public void fileOperate() throws Exception {
    fileMapper.insertFileInfo();
    throw new Exception("文件操作失败");
}
```

**📌面试考点**：提问：Spring事务默认对什么异常回滚？如何让受检异常触发回滚？答：默认仅回滚运行时异常和错误；通过`rollbackFor = Exception.class`全局配置所有异常回滚。

### 6.1.5 数据库引擎不支持事务

**概念定义**：MySQL数据库中，MyISAM引擎**不支持事务、不支持ACID特性**，即使代码配置了事务注解，也完全无法生效，异常不会触发回滚。

**核心原理**：事务是数据库引擎层面的能力，Spring事务是**基于数据库原生事务的上层封装**。如果底层数据库引擎不支持事务，Spring所有的事务增强、回滚逻辑都无底层支撑，相当于无源之水。MyISAM引擎设计初衷是高性能查询，舍弃了事务、锁、崩溃恢复能力；InnoDB引擎完整支持事务、行锁、MVCC，是事务的基础。

**场景与价值**：老旧项目数据库表使用MyISAM引擎，迁移、改造时未检查引擎类型，导致事务失效，数据无法回滚，是底层环境级坑点。

**验证与排错**：执行事务异常代码，数据不回滚。通过SQL查询表引擎：`show create table 表名;`，若引擎为MyISAM，即可定位问题。

**💡最佳实践**：1、所有业务表**强制使用InnoDB引擎**；2、新建数据库默认配置InnoDB，统一数据库参数；3、老旧项目批量检查表引擎，统一迁移改造。

**⚠️避坑指南**：事务失效优先排查底层环境，先确认数据库引擎、数据库版本是否支持事务，再排查代码问题。

### 6.1.6 传播机制配置错误导致失效

**概念定义**：Spring事务传播机制定义了**当前事务方法被另一个事务方法调用时，事务如何传递、创建、挂起**，传播属性配置错误会导致事务不生效、不回滚、事务隔离失效等问题。

**核心原理**：Spring内置7种事务传播机制，不同机制的事务创建、复用、挂起逻辑完全不同。高频失效场景：**REQUIRES_NEW、NOT_SUPPORTED、SUPPORTS**配置不当，导致子方法脱离父事务控制。

**高频失效场景详解**：

1. **NOT_SUPPORTED**：始终以非事务方式执行，若当前存在事务则挂起，方法内操作无事务，异常不回滚；
2. **SUPPORTS**：有事务则加入，无事务则非事务执行，无外层事务时，自身事务失效；
3. **REQUIRES_NEW**：新建独立事务，挂起外层事务，子事务异常不影响父事务，父事务异常不回滚子事务已提交数据。

**实操错误示例（SUPPORTS失效）**

```java
@Service
public class PayService {

    // 无事务的普通方法
    public void pay() {
        // 调用传播机制为SUPPORTS的方法，无外层事务，子方法非事务执行
        savePayLog();
    }

    // 无外层事务时，事务失效
    @Transactional(propagation = Propagation.SUPPORTS)
    public void savePayLog() {
        payLogMapper.insert(new PayLog());
        int i = 1 / 0;
    }
}
```

**💡最佳实践**：1、核心业务事务统一使用默认**REQUIRED**机制；2、仅特殊业务（日志记录、异步统计）使用非事务传播机制；3、嵌套事务必须明确传播属性，避免层级混乱。

**📌面试考点**：提问：常用事务传播机制及失效场景？答：默认REQUIRED（有则加入、无则新建）；SUPPORTS无外层事务则失效，NOT_SUPPORTED强制非事务，REQUIRES_NEW独立事务互不影响。

### 6.1.7 事务方法嵌套导致部分失效

**概念定义**：多个事务方法嵌套调用时，由于传播机制、异常抛出时机问题，出现**部分方法回滚、部分方法提交**的事务部分失效问题，无法保证整体一致性。

**核心原理**：默认REQUIRED传播机制下，嵌套方法会加入外层事务，整体为同一个事务。存在两个核心问题：1、内层异常被捕获，外层无感知，外层提交导致内层数据无法回滚；2、内层REQUIRES_NEW独立事务，提交后外层事务回滚无法撤销内层数据。

**场景与价值**：复杂业务分层开发中，多层事务嵌套是常态，极易出现部分数据脏写，是生产高风险问题。

**实操错误示例（嵌套部分失效）**

```java
@Service
public class BusinessService {

    @Transactional
    public void mainBusiness() {
        // 外层数据库操作
        orderMapper.insert(new Order());
        try {
            // 内层事务方法
            subBusiness();
        } catch (Exception e) {
            log.error("子业务异常", e);
            // 仅捕获不抛出，外层事务正常提交
        }
    }

    @Transactional
    public void subBusiness() {
        goodsMapper.updateStock();
        int i = 1 / 0;
    }
}
```

**失效结果**：内层方法异常触发回滚（库存更新撤销），但外层捕获异常后继续提交，订单数据成功插入，出现**数据不一致**。

**💡最佳实践**：1、嵌套事务禁止内层异常被外层无脑捕获；2、需要独立事务的嵌套场景明确指定REQUIRES_NEW；3、统一全局异常处理，保证嵌套事务异常可传递。

### 6.1.8 多线程事务失效问题

**概念定义**：在事务方法中开启新线程执行数据库操作，**新线程的事务完全失效**，主线程事务无法控制子线程数据一致性。

**核心原理**：Spring事务底层基于**ThreadLocal**存储事务上下文、连接、状态信息，事务资源与当前线程绑定。多线程场景下，子线程拥有独立的ThreadLocal空间，无法获取主线程的事务上下文，因此子线程操作会开启独立的新事务，或直接无事务执行，主线程回滚无法影响子线程。

**场景与价值**：高并发业务中，开发者为提升效率，在事务内异步多线程处理批量数据，导致部分线程数据无法回滚，引发严重数据不一致问题。

**实操错误示例**

```java
@Service
public class BatchService {

    @Transactional(rollbackFor = Exception.class)
    public void batchSave() {
        // 主线程数据库操作
        userMapper.insert(new User());

        // 子线程执行数据库操作，事务失效
        new Thread(() -> {
            orderMapper.insert(new Order());
            int i = 1 / 0;
        }).start();
    }
}
```

**失效结果**：子线程抛出异常，子线程数据无事务回滚，主线程事务正常提交，最终用户数据插入成功、订单数据也插入成功，完全违背事务一致性。

**💡最佳实践**：1、**禁止在事务内开启异步线程执行数据库写操作**；2、批量操作优先使用同步事务，保证线程上下文统一；3、异步场景拆分事务，实现最终一致性。

**📌面试考点**：提问：为什么多线程会导致事务失效？答：Spring事务基于ThreadLocal绑定线程上下文，子线程无法继承主线程事务资源，事务隔离失效，无法实现统一回滚。

## 6.2 事务失效解决方案与修复代码

针对上述八大事务失效场景，本节汇总**统一解决方案、标准修复代码、通用规范**，可直接落地到生产项目，彻底规避事务失效问题。所有修复方案均遵循「简单高效、无侵入、生产可用」原则。

### 1、非public方法失效修复

统一规范：所有事务方法强制使用public修饰，禁止其他权限修饰符。

```java
// 修复后：public修饰，事务正常生效
@Transactional(rollbackFor = Exception.class)
public void updateUser() {
    userMapper.updateById(new User());
    int i = 1 / 0;
}
```

### 2、同类自调用失效修复（三种生产方案）

**方案一：通过AopContext获取代理对象（推荐，无侵入）**

第一步：启动类开启代理暴露注解`@EnableAspectJAutoProxy(exposeProxy = true)`

```java
@SpringBootApplication
@EnableAspectJAutoProxy(exposeProxy = true) // 暴露代理对象
public class TransactionApplication {
}
```

第二步：业务类通过代理对象调用自身事务方法

```java
@Service
public class OrderService {

    public void createOrder() {
        // 获取当前类代理对象，通过代理调用事务方法
        OrderService proxy = (OrderService) AopContext.currentProxy();
        proxy.saveOrder();
    }

    @Transactional(rollbackFor = Exception.class)
    public void saveOrder() {
        orderMapper.insert(new Order());
        int i = 1 / 0;
    }
}
```

**方案二：拆分Service（大型项目最佳实践）**：将被调用的事务方法抽离为独立Service，通过@Autowired注入调用，彻底避免自调用问题。

### 3、异常被捕获未回滚修复

两种标准修复方式，适配不同业务场景：

**方式一：捕获后重新抛出异常（推荐）**

```java
@Transactional(rollbackFor = Exception.class)
public void updateGoodsStock() {
    try {
        goodsMapper.updateStock();
        int i = 1 / 0;
    } catch (Exception e) {
        log.error("更新库存异常", e);
        // 重新抛出异常，触发事务回滚
        throw new RuntimeException("库存更新失败", e);
    }
}
```

**方式二：手动强制回滚**

```java
@Transactional(rollbackFor = Exception.class)
public void updateGoodsStock() {
    try {
        goodsMapper.updateStock();
        int i = 1 / 0;
    } catch (Exception e) {
        log.error("更新库存异常", e);
        // 手动标记事务回滚
        TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
    }
}
```

### 4、受检异常不回滚修复

全局统一配置事务注解，覆盖所有异常场景，项目所有事务方法统一使用该配置。

```java
// 核心配置：所有异常均触发回滚
@Transactional(rollbackFor = Exception.class)
public void fileOperate() throws Exception {
    fileMapper.insertFileInfo();
    throw new Exception("文件操作失败");
}
```

#### 5、数据库引擎不支持事务修复

1、SQL修改表引擎为InnoDB：`alter table 表名 engine = innodb;`

2、全局数据库配置，新建表默认InnoDB，my.cnf配置：

```properties
default-storage-engine=InnoDB
```

#### 6、传播机制、嵌套事务失效修复

1、通用业务统一使用默认REQUIRED传播机制，不随意修改；2、嵌套事务禁止内层异常吞噬，保证异常向上传递；3、独立事务场景显式指定REQUIRES_NEW。

#### 7、多线程事务失效修复

核心修复规则：**事务内禁止异步写库**。所有数据库增删改操作统一同步执行，批量任务拆分事务，异步仅用于非核心日志、统计、通知场景。如需分布式事务，引入Seata等框架实现跨线程、跨服务事务一致性。

## 6.3 生产大事务问题与优化方案

大事务是生产环境数据库性能瓶颈、超时、锁等待、并发卡顿的核心诱因之一。所谓**大事务**，指事务执行时间过长、单次事务操作数据量过大、占用数据库连接和锁资源过久的事务。大事务不会直接导致事务失效，但会引发一系列线上性能、并发、数据风险，是生产优化的重点场景。

### 6.3.1 大事务引发的锁等待、超时、并发卡顿

**核心原理**：InnoDB事务遵循**锁等待机制**，事务开启后，会对操作的数据加行锁/表锁，直到事务提交或回滚后才释放锁。事务执行时间越长，锁占用时间越久，其他并发请求会持续阻塞等待锁，引发连锁问题。

**生产核心问题拆解**

1、**锁等待超时**：大事务长期占用行锁，后续并发请求超时，抛出`Lock wait timeout exceeded`异常，业务报错。

2、**并发卡顿、吞吐量下降**：大量请求阻塞锁等待，数据库连接池被占满，新请求无可用连接，系统接口响应缓慢、吞吐量暴跌。

3、**死锁概率大幅提升**：长事务多资源占用、执行链路复杂，多线程竞争锁的时序混乱，极易触发数据库死锁。

4、**数据库日志堆积**：长事务导致undo log、redo log无法及时清理，磁盘占用升高，数据库性能持续下降。

5、**主从延迟增大**：大事务产生大量binlog日志，从库同步耗时增加，主从数据延迟过高，引发读写分离数据不一致。

**大事务常见诱因**

1、事务内包含大量非DB操作：文件IO、网络请求、第三方接口调用、循环业务逻辑；2、单次事务批量处理上万条数据；3、事务嵌套过多，执行链路冗长；4、事务范围过大，将无关查询、空逻辑纳入事务；5、异常重试机制导致事务持续占用资源。

### 6.3.2 大事务拆分最佳实践

大事务优化的核心思想：**缩小事务范围、缩短锁持有时间、拆分批量操作、剥离非数据库逻辑**。以下为可直接落地的生产最佳实践。

**1、剥离事务内非数据库操作**

所有网络请求、文件读写、日志打印、消息发送、第三方API调用，**全部移出事务方法**。仅将数据库增删改逻辑保留在事务内，最大限度缩短事务执行时长。

**错误示例**

```java
@Transactional
public void createOrder() {
    // 非DB操作，耗时极高，拖慢事务
    sendSms();
    callThirdApi();
    // 核心DB操作
    orderMapper.insert();
}
```

**正确示例**

```java
public void createOrder() {
    // 非DB操作移出事务
    sendSms();
    callThirdApi();
    // 仅核心DB操作开启事务
    saveOrder();
}

@Transactional(rollbackFor = Exception.class)
public void saveOrder() {
    orderMapper.insert();
}
```

**2、批量数据分批拆分事务**

禁止单次事务批量处理千条、万条数据，采用**分批提交**策略，每批次处理少量数据，短事务快速提交，释放锁资源。

**分批批量处理示例**

```java
// 批量拆分，每500条一个事务
public void batchUpdate(List<User> userList) {
    int batchSize = 500;
    int total = userList.size();
    for (int i = 0; i < total; i += batchSize) {
        int end = Math.min(i + batchSize, total);
        List<User> batchList = userList.subList(i, end);
        // 分批执行事务
        updateBatch(batchList);
    }
}

@Transactional(rollbackFor = Exception.class)
public void updateBatch(List<User> batchList) {
    userMapper.updateBatch(batchList);
}
```

**3、最小化事务范围**

遵循**事务最小粒度原则**，仅对需要数据一致性的增删改操作开启事务，查询操作、校验逻辑、参数封装逻辑全部移出事务。禁止将整个业务方法全部包裹事务。

**4、异步化非核心业务**

订单日志、操作记录、数据统计、消息通知等非核心一致性业务，通过异步线程、消息队列异步处理，不占用主事务流程，大幅缩短主事务耗时。

**5、关闭不必要的事务嵌套**

梳理业务嵌套事务，无独立一致性需求的子事务，取消`@Transactional`注解，统一加入外层事务，减少事务创建、挂起、切换开销，简化事务链路。

**6、大事务兜底方案：最终一致性**

超大型批量业务、跨服务业务，放弃强一致性事务，采用**本地消息表、可靠消息队列、定时补偿**机制实现最终一致性，彻底规避长事务风险。

**💡生产最佳实践总结**：短事务优先、分批执行、非DB逻辑剥离、异步解耦、最小锁范围，是解决大事务线上问题的核心方案。

# 7. 数据访问层高频踩坑总结

数据访问层是Spring Boot项目与数据库交互的核心层级，包含连接池配置、MyBatis映射、多数据源切换、并发锁控制等核心能力。大部分线上数据库异常、接口报错、数据异常、并发卡顿，均来自数据访问层配置不当和编码不规范。本节汇总生产四大高频坑点，结合原理、报错场景、修复方案、最佳实践，全面规避数据访问层问题。

## 7.1 连接池参数配置不当坑点

Spring Boot默认使用HikariCP数据库连接池，连接池参数直接决定数据库连接的复用、超时、并发承载能力，参数配置不合理会直接导致**连接耗尽、接口超时、数据库宕机、并发报错**等线上严重问题。

**核心坑点1：最大连接数配置过大/过小**

1、最大连接数过小：并发场景下连接池无可用连接，请求阻塞，接口超时、吞吐量暴跌；2、最大连接数过大：超出数据库最大连接上限，数据库连接被占满，新连接创建失败，数据库拒绝服务。

**原理**：MySQL默认最大连接数`max_connections`默认151，HikariCP最大连接数需**小于数据库最大连接数**，同时适配业务并发量。

**核心坑点2：空闲连接超时、生命周期配置不合理**

1、连接生命周期大于数据库超时时间：数据库主动断开空闲连接，连接池留存无效连接，调用时报错`Communications link failure`；2、空闲超时过短：连接频繁销毁创建，浪费资源、降低性能。

**核心坑点3：未配置超时参数**

未配置连接获取超时、语句执行超时，锁等待、慢查询场景下，请求无限阻塞，占满线程和连接资源，引发服务雪崩。

**生产标准最优配置（可直接落地）**

```yaml
spring:
  datasource:
    hikari:
      # 最大连接数：生产推荐10-50，不超过MySQL max_connections 80%
      maximum-pool-size: 30
      # 最小空闲连接：保证基础并发能力
      minimum-idle: 10
      # 连接最大生命周期：小于数据库wait_timeout（默认8小时），推荐30分钟
      max-lifetime: 1800000
      # 空闲连接超时
      idle-timeout: 600000
      # 获取连接超时时间：2秒，快速失败，避免阻塞
      connection-timeout: 2000
      # 语句执行超时，防止慢查询卡死
      statement-timeout: 5000
```

**⚠️避坑指南**：1、禁止盲目调大最大连接数，适配数据库配置；2、连接生命周期必须小于数据库空闲超时；3、必须配置超时参数，实现快速失败兜底；4、测试环境与生产环境参数区分配置。

## 7.2 MyBatis 映射失败、参数绑定错误

MyBatis作为主流持久层框架，映射异常、参数绑定错误是开发、测试、上线阶段最高频的报错，问题虽小但极易引发功能故障，汇总四大高频坑点及解决方案。

**坑点1：实体类属性名与数据库字段名映射失败**

场景：数据库下划线命名（user_name），实体类驼峰命名（userName），未开启驼峰自动转换，导致查询字段为null。

**解决方案**：全局开启驼峰命名自动转换

```yaml
mybatis:
  configuration:
    map-underscore-to-camel-case: true
```

**坑点2：单参数未指定@Param注解导致绑定失败**

场景：Mapper方法单个参数，XML中使用参数名取值，未加`@Param`，MyBatis无法绑定参数，报错`parameter not found`。

**错误示例**

```java
// 错误：单参数无@Param，XML无法识别参数名
User selectUser(String username);
```

**正确示例**

```java
User selectUser(@Param("username") String username);
```

**坑点3：动态SQL空判断错误**

高频错误：`if test="status != ''"`判断数值类型、Integer类型参数，数值0会被判定为空，导致动态SQL失效。

**正确规则**：字符串判断 `test="str != null and str != ''"`；数值类型判断 `test="num != null"`，禁止判空字符串。

**坑点4：返回值映射不匹配**

场景：数据库查询无数据返回null，Mapper方法返回基本类型（int、boolean），触发空指针异常；多字段查询未对应实体属性，导致数据丢失。

**💡最佳实践**：1、统一开启驼峰转换，全局统一映射规则；2、所有Mapper接口参数强制添加@Param注解；3、严格区分字符串和数值的动态SQL判断规则；4、返回值统一使用包装类，避免基本类型空指针；5、复杂查询使用ResultMap手动映射，保证字段精准匹配。

## 7.3 多数据源切换异常坑点

生产项目中多数据源（主从分离、多库拆分）场景普遍，数据源切换异常会导致**查错库、写错库、切换失效、连接报错**等严重问题，汇总核心坑点。

**坑点1：事务内数据源切换失效**

**核心原理**：Spring事务开启后，会绑定当前数据源连接，整个事务周期内**数据源不可切换**。同一事务内先操作主库，再切换从库查询，会依然走主库，切换失效。

**解决方案**：数据源切换操作**必须在事务开启前执行**，事务内禁止切换数据源；多库操作拆分不同事务。

**坑点2：数据源未指定默认值，启动报错**

多数据源配置时，未通过`@Primary`指定默认主数据源，Spring无法自动注入默认数据源，项目启动失败。

**坑点3：动态数据源切换线程上下文未清空**

基于ThreadLocal实现的动态数据源切换，请求结束后未清空上下文，线程复用后导致**数据源错乱、串库**，是线上隐形高危问题。

**解决方案**：通过拦截器、AOP在请求结束后强制清空数据源上下文，保证线程复用干净。

**坑点4：主从数据源读写配置颠倒**

误将写操作指向从库，导致数据写入失败；读操作指向主库，压垮主库性能，违背读写分离设计初衷。

**💡最佳实践**：1、事务内禁止切换数据源，多库操作拆分事务；2、多数据源必须指定@Primary默认主库；3、请求结束清空数据源线程上下文；4、AOP切面统一控制读写路由，读从库、写主库；5、禁止手动硬编码切换数据源，统一切面管控。

## 7.4 高并发下数据库锁竞争、超时问题

高并发秒杀、下单、库存扣减、订单状态更新场景中，数据库锁竞争激烈，极易出现**锁超时、死锁、并发超卖、接口卡顿**问题，是生产高并发场景的核心难点。

**核心坑点1：默认行锁升级为表锁**

**原理**：InnoDB行锁基于索引生效，**无索引、索引失效、模糊查询**时，行锁会升级为表锁，锁定整张数据表，所有并发请求阻塞，彻底拖垮并发性能。

**避坑方案**：所有更新、删除操作**必须基于唯一索引、主键索引**，禁止无索引条件更新数据。

**核心坑点2：锁等待超时未处理**

高并发竞争场景下，多条SQL同时竞争同一行数据，部分请求锁等待超时，抛出异常，业务未做兜底处理，导致用户操作失败。

**解决方案**：捕获锁超时异常，做重试机制、降级兜底、排队处理。

**核心坑点3：更新顺序不一致导致死锁**

**原理**：多线程多表更新时，加锁顺序不一致，线程A锁A表等B表，线程B锁B表等A表，循环等待触发死锁。

**解决方案**：全局统一数据库更新、加锁顺序，固定先更新A表再更新B表，彻底规避死锁。

**核心坑点4：悲观锁滥用导致并发瓶颈**

大量场景无脑使用`select ... for update`悲观锁，长期占用行锁，并发吞吐量极低，无法支撑高并发业务。

**优化方案**：高并发场景优先使用**乐观锁（版本号机制）**，无锁竞争，高性能；核心强一致性场景使用悲观锁，且缩短锁持有时间。

**💡高并发锁优化最佳实践**

1、所有写操作必须走索引，避免行锁升级表锁；2、统一SQL加锁、更新顺序，杜绝死锁；3、高并发优先乐观锁，减少悲观锁使用；4、缩短事务时长，快速释放锁资源；5、锁超时异常增加重试、降级兜底；6、热点数据拆分、库存分片，分散锁竞争压力。
