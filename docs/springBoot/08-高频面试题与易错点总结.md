# 1. SpringBoot 基础核心
## 1.1 高频面试真题

### Q1：请结合源码，描述 `SpringApplication.run()` 的核心执行流程，重点说下「上下文刷新」阶段做了什么？
**标准答案：**
`SpringApplication.run()` 的执行流程可以分为两大阶段：**初始化阶段**和**运行阶段**，其中核心逻辑在 `run()` 方法中。

1.  **初始化阶段（`new SpringApplication()`）**
    -   推断应用类型：判断是 `NONE`（非Web）、`SERVLET`（传统Web）还是 `REACTIVE`（响应式Web）。
    -   加载初始化器（`ApplicationContextInitializer`）：从 `META-INF/spring.factories` 加载，用于对上下文做预处理。
    -   加载监听器（`ApplicationListener`）：同样从 `spring.factories` 加载，用于监听应用启动过程中的事件。
    -   推断主启动类：找到包含 `main` 方法的类。

2.  **运行阶段（`run()` 方法核心流程）**
    1.  **启动计时&设置系统属性**：创建 `StopWatch` 计时，配置 `java.awt.headless` 属性。
    2.  **发布 `ApplicationStartingEvent` 事件**：通知所有监听器应用即将启动。
    3.  **准备环境（`prepareEnvironment`）**：
        -   根据应用类型创建 `Environment`（如 `StandardServletEnvironment`）。
        -   加载配置源（`application.properties/yml`、系统环境变量、JVM参数等）。
        -   配置 Profile，激活指定环境配置。
    4.  **打印 Banner**：控制台输出 SpringBoot Logo（可自定义）。
    5.  **创建 `ApplicationContext` 上下文**：根据应用类型创建对应的上下文（如 `AnnotationConfigServletWebServerApplicationContext`）。
    6.  **准备上下文（`prepareContext`）**：
        -   将 `Environment` 注入上下文。
        -   执行所有 `ApplicationContextInitializer` 的 `initialize` 方法。
        -   加载主启动类作为配置源。
    7.  **刷新上下文（`refreshContext`）—— 核心阶段**：
        这一步是调用 Spring 原生的 `refresh()` 方法，完成所有核心初始化：
        -   **BeanFactory 准备**：设置 `BeanFactory`，加载 Bean 定义。
        -   **BeanFactory 后置处理**：执行 `BeanFactoryPostProcessor`，其中最重要的是 `ConfigurationClassPostProcessor`，处理 `@Configuration` 类，解析 `@Bean`、`@ComponentScan` 等注解，自动配置也在这里被加载。
        -   **注册 BeanPostProcessor**：注册所有 `BeanPostProcessor`，为后续 Bean 初始化做准备。
        -   **初始化消息源、事件多播器**：处理国际化和事件发布。
        -   **注册监听器 Bean**：注册实现了 `ApplicationListener` 接口的 Bean。
        -   **刷新 Web 服务器（仅Web应用）**：创建并启动内嵌的 Tomcat/Jetty 服务器。
        -   **实例化所有单例 Bean**：调用 `finishBeanFactoryInitialization`，实例化所有非懒加载的单例 Bean，包括自动配置创建的 Bean。
        -   **完成刷新**：发布 `ContextRefreshedEvent` 事件，上下文刷新完成。
    8.  **调用 `Runners`**：执行所有 `ApplicationRunner` 和 `CommandLineRunner` 的 `run` 方法。
    9.  **发布 `ApplicationStartedEvent`/`ApplicationReadyEvent` 事件**：通知监听器应用已启动/准备就绪。
    10. **返回上下文**：返回 `ConfigurableApplicationContext`。

### Q2：SpringBoot 如何解决 Spring 框架的「循环依赖」问题？它的处理逻辑和 Spring 原生有什么区别？
**标准答案：**
首先，SpringBoot 没有改变 Spring 解决循环依赖的核心机制（三级缓存），但它的自动配置和默认行为，使得循环依赖问题的表现形式和解决场景发生了变化。

1.  **Spring 原生的循环依赖解决方案**
    -   **核心机制：三级缓存**
        -   `singletonObjects`：一级缓存，存放完全初始化好的单例 Bean。
        -   `earlySingletonObjects`：二级缓存，存放早期暴露的 Bean（已实例化但未初始化完成）。
        -   `singletonFactories`：三级缓存，存放 Bean 工厂对象，用于生成早期引用。
    -   **适用场景**：仅支持 **Setter/字段注入** 的循环依赖。如果是 **构造器注入** 的循环依赖，Spring 会直接抛出 `BeanCurrentlyInCreationException`，无法解决。

2.  **SpringBoot 对循环依赖的处理与区别**
    -   **默认开启「构造器注入循环依赖」支持**：
        -   SpringBoot 2.6+ 之前，默认允许构造器注入的循环依赖。它通过 `allowCircularReferences` 配置，在底层通过特殊处理（比如延迟初始化）来尝试解决，但本质上是一种「妥协方案」，可能导致 Bean 初始化顺序不可控。
        -   SpringBoot 2.6+ 之后，默认禁用了构造器注入的循环依赖（`spring.main.allow-circular-references` 默认为 `false`），和 Spring 原生保持一致，鼓励开发者通过重构代码来解决问题。
    -   **自动配置类的循环依赖问题**：
        -   SpringBoot 的自动配置类之间可能存在依赖关系，比如 `A` 配置依赖 `B` 配置生成的 Bean。这种情况下，Spring 的三级缓存依然生效，但如果是构造器注入，同样会触发异常。
    -   **@Lazy 注解的广泛使用**：
        -   SpringBoot 中解决循环依赖最常用的方案之一是 `@Lazy` 注解。当一个 Bean 被 `@Lazy` 修饰时，Spring 不会立即实例化它，而是注入一个代理对象，打破了循环依赖的死锁。这是 SpringBoot 项目中最常见的「临时解决」方案。
    -   **总结区别**：
        | 对比项                 | Spring 原生          | SpringBoot                                           |
        | :--------------------- | :------------------- | :--------------------------------------------------- |
        | 构造器注入循环依赖     | 直接报错，不支持     | 2.6前默认尝试兼容，2.6+ 默认不支持                   |
        | 解决循环依赖的推荐方式 | 依赖三级缓存         | 更推荐 `@Lazy` 或重构，避免循环依赖                  |
        | 常见问题场景           | 业务代码中的循环依赖 | 自动配置类之间、业务Bean与自动配置Bean之间的循环依赖 |

### Q3：`@SpringBootApplication` 底层包含哪些注解？各自作用是什么？
**标准答案：**
`@SpringBootApplication` 是一个组合注解，它主要包含以下三个核心注解（以 SpringBoot 2.x/3.x 为例）：

```java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Inherited
@SpringBootConfiguration
@EnableAutoConfiguration
@ComponentScan(excludeFilters = { ... })
public @interface SpringBootApplication {
    // ...
}
```

1.  **`@SpringBootConfiguration`**
    -   本质上就是 `@Configuration` 的派生注解，作用完全一样。
    -   标记该类为配置类，Spring 会将其作为配置源，扫描其中的 `@Bean` 注解，生成 Bean 定义。
    -   它的特殊之处在于，它会被 SpringBoot 识别为「主配置类」，项目启动时会自动加载它。

2.  **`@EnableAutoConfiguration`**
    -   这是 SpringBoot 自动配置的核心注解。
    -   它通过 `AutoConfigurationImportSelector` 类，从 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`（3.x）或 `spring.factories`（2.x）中加载所有自动配置类。
    -   这些自动配置类会根据条件注解（如 `@ConditionalOnClass`），自动为项目配置所需的 Bean，省去了手动配置的麻烦。

3.  **`@ComponentScan`**
    -   作用是自动扫描并加载指定包下的所有组件（`@Component`、`@Service`、`@Repository`、`@Controller` 等）。
    -   默认扫描范围：**主启动类所在的包及其所有子包**。
    -   如果你的主启动类在 `com.example` 包下，那么 `com.example` 及其子包下的所有组件都会被自动扫描。
    -   它也支持通过 `basePackages`、`basePackageClasses` 等属性手动指定扫描范围，同时也内置了一些排除过滤器，用于排除自动配置类本身。

## 1.2 本章核心易错点&踩坑总结

### 误区1：混淆 `SpringApplication` 初始化和上下文刷新的阶段，启动报错无法定位是在哪个环节失败
**问题描述**：项目启动失败，只知道报错了，不知道是在 `prepareEnvironment` 阶段（加载配置）、`refreshContext` 阶段（Bean初始化）还是 `runRunners` 阶段（执行初始化器）失败的。
**正确做法**：
-   看日志里的报错堆栈，找到报错的类和方法，对应到启动流程的具体阶段。
-   比如 `IllegalArgumentException` 且堆栈指向 `PropertySources`，大概率是配置文件加载问题（`prepareEnvironment` 阶段）。
-   比如 `NoSuchBeanDefinitionException`，大概率是 Bean 初始化问题（`refreshContext` 阶段）。

### 误区2：循环依赖问题只知道三级缓存，不知道 SpringBoot 对「构造器注入循环依赖」的特殊处理
**问题描述**：项目中两个 Bean 使用构造器注入，形成循环依赖，在 SpringBoot 2.6+ 版本中直接报错，误以为是三级缓存失效了。
**正确认知**：
-   Spring 的三级缓存只能解决 setter/字段注入的循环依赖，对构造器注入的循环依赖无能为力。
-   SpringBoot 2.6+ 默认关闭了 `allowCircularReferences`，构造器注入的循环依赖会直接报错。
-   解决方案：
    1.  重构代码，消除循环依赖（推荐）。
    2.  改用 setter/字段注入。
    3.  在其中一个 Bean 上添加 `@Lazy` 注解。
    4.  不推荐：全局配置 `spring.main.allow-circular-references=true`。

### 误区3：自定义 Starter 时依赖 scope 配置错误，导致项目引入后出现依赖冲突
**问题描述**：自定义 Starter 时，将依赖的核心包（如 SpringBoot 自动配置包）的 scope 设为了 `provided` 或 `test`，导致其他项目引入后找不到相关类。
**正确做法**：
-   Starter 中的依赖，核心业务依赖应该使用 `compile`（默认） scope，这样依赖会被传递到引入 Starter 的项目中。
-   仅对测试相关、编译时注解（如 `lombok`）使用 `provided` scope。
-   同时，Starter 自身的依赖最好设置 `optional=true`，避免强制传递，让使用者自己决定是否引入。

### 误区4：`ApplicationRunner` 执行顺序仅依赖 `@Order`，忽略了 SpringBoot 内置初始化器的优先级影响
**问题描述**：自定义了多个 `ApplicationRunner`，加了 `@Order(1)`、`@Order(2)`，但执行顺序和预期不符。
**正确认知**：
-   `@Order` 注解的数值越小，优先级越高（越先执行）。
-   但 SpringBoot 内置的一些 `ApplicationRunner`（比如和健康检查、日志相关的）也有自己的 `@Order` 值，它们的优先级可能比你自定义的更高。
-   同时，`CommandLineRunner` 和 `ApplicationRunner` 之间，`ApplicationRunner` 会先执行。
-   排查时，可以通过日志打印执行顺序，确认是否有内置的 Runner 影响了你的执行顺序。

# 2. SpringBoot 自动配置原理
## 2.1 高频面试真题
### Q1：请结合源码，详细描述 SpringBoot 自动配置的完整触发流程，重点说下 `AutoConfigurationImportSelector` 的核心作用
**标准答案（分层回答，适配不同面试深度）：**

​	1. **触发入口：`@EnableAutoConfiguration`** 它是自动配置的开关，核心逻辑由导入的 `AutoConfigurationImportSelector` 实现。这个类是一个 `DeferredImportSelector`，会在所有 `@Configuration` 类处理完成后再执行，保证自动配置的优先级低于用户自定义配置。

2.  **加载自动配置候选类（3.x vs 2.x 差异）**
    -   SpringBoot 2.x：从 `META-INF/spring.factories` 文件中读取 `org.springframework.boot.autoconfigure.EnableAutoConfiguration` 对应的配置类列表。
    -   SpringBoot 3.x：改为读取 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件，解决了旧版 SPI 加载效率低、无法排序的问题。

3.  **过滤与条件匹配**
    加载到所有候选配置类后，会执行两步过滤：
    -   **排除过滤**：根据 `@EnableAutoConfiguration` 注解的 `exclude`/`excludeName` 属性，移除用户明确排除的配置类。
    -   **条件注解过滤**：通过 `@Conditional` 系列注解（如 `@ConditionalOnClass`、`@ConditionalOnMissingBean`）进行匹配，只有满足所有条件的配置类才会被保留。

4.  **排序与加载**
    过滤后的配置类，会根据 `@AutoConfigureOrder`、`@AutoConfigureBefore`/`@AutoConfigureAfter` 注解进行排序，保证依赖关系的加载顺序，最终被 Spring 容器加载为普通配置类。

### Q2：请详细说明 `@Conditional` 系列注解的底层匹配原理，并举例说明 `@ConditionalOnClass` 和 `@ConditionalOnMissingBean` 的使用场景与区别

**标准答案：**

1. **底层原理：`Condition` 接口与 `ConditionEvaluator`**
     	所有 `@Conditional` 系列注解的核心，都是实现了 `Condition` 接口的匹配类（如 `OnClassCondition`、`OnBeanCondition`）。Spring 在解析配置类时，会通过 `ConditionEvaluator` 调用 `matches()` 方法，判断当前环境是否满足条件。

- ​	`ConditionContext`：提供上下文信息，如 `BeanDefinitionRegistry`、`ConfigurableListableBeanFactory`、`Environment`、`ResourceLoader`、`ClassLoader`。
- `AnnotatedTypeMetadata`：提供当前注解的元数据信息，如注解的属性值。

2.  **`@ConditionalOnClass` 与 `@ConditionalOnMissingBean` 对比**
    
    | 注解                        | 匹配依据                           | 使用场景                                                     | 注意事项                                              |
    | :-------------------------- | :--------------------------------- | :----------------------------------------------------------- | :---------------------------------------------------- |
    | `@ConditionalOnClass`       | 类路径中是否存在指定类             | 第三方依赖集成（如 `RedisTemplate` 存在时，才加载 Redis 配置） | 只检查类是否存在，不检查 Bean 是否被实例化            |
    | `@ConditionalOnMissingBean` | 容器中是否存在指定类型/名称的 Bean | 提供默认实现，允许用户自定义覆盖（如用户未定义 `RedisTemplate` 时，自动配置一个默认的） | 仅匹配已注册的 Bean，自动配置类之间的 Bean 也会被检查 |

### Q3：SpringBoot 3.x 中自动配置加载方式从 `spring.factories` 改为 `AutoConfiguration.imports`，这个改动解决了什么问题？它的加载顺序是如何控制的？

**标准答案：**

**旧版 `spring.factories` 的问题**

- 加载效率低：启动时需要遍历所有 jar 包的 `spring.factories` 文件，解析键值对，启动性能差。
- 无法直接排序：旧版只能通过 `@Order` 间接控制，自动配置类之间的依赖关系无法明确表达，容易出现加载顺序问题。
- 扩展性差：SPI 机制的扩展性有限，无法支持更复杂的条件过滤。

2.  **新版 `AutoConfiguration.imports` 的改进**
    -   直接读取：文件中直接列出自动配置类的全限定名，无需键值对解析，加载效率大幅提升。
    -   支持排序：文件中的顺序就是基础加载顺序，同时保留 `@AutoConfigureOrder`/`@AutoConfigureBefore`/`@AutoConfigureAfter` 注解进行微调，解决了依赖加载问题。
    -   更清晰的职责：专门用于自动配置类的加载，和其他 SPI 场景分离，更易维护。

### Q4：如何排除/禁用某个自动配置类？有哪些方式？它们的优先级和适用场景有什么区别？

**标准答案：**
共有4种主流方式，优先级从高到低如下：

1. **`@EnableAutoConfiguration(exclude = XxxAutoConfiguration.class)`**

- 方式：在主启动类的 `@SpringBootApplication` 注解中，通过 `exclude`/`excludeName` 属性排除。
- 优先级：最高，直接在自动配置加载阶段就排除，不会进入后续的条件匹配。
- 适用场景：项目全局永久排除某个自动配置类。

2.  **配置文件 `spring.autoconfigure.exclude`**
    -   方式：在 `application.yml` 中配置 `spring.autoconfigure.exclude: com.example.XxxAutoConfiguration`。
    -   优先级：次之，和注解排除效果一致，但更灵活，支持多环境配置。
    -   适用场景：根据不同环境动态排除自动配置类。

3.  **自定义 Bean 覆盖自动配置 Bean**
    -   方式：自己定义一个和自动配置中类型相同的 Bean，利用 `@ConditionalOnMissingBean` 的规则，让自动配置类不生效。
    -   优先级：再次之，仅当自动配置类依赖 `@ConditionalOnMissingBean` 时有效。
    -   适用场景：不是完全禁用，而是替换自动配置的默认实现。

4.  **修改条件注解匹配条件**
    -   方式：通过配置 `spring.xxx.enabled=false` 等方式，让 `@ConditionalOnProperty` 条件不匹配，从而禁用配置类。
    -   优先级：最低，依赖配置类自身的条件注解。
    -   适用场景：配置类提供了开关属性的场景。

### Q5：请描述自定义 Starter 的完整实现思路，重点说明如何保证它的「可扩展性」和「可覆盖性」

**标准答案：**

1. **模块拆分**：分为 `xxx-spring-boot-autoconfigure`（自动配置核心）和 `xxx-spring-boot-starter`（依赖聚合）两个模块。

- `starter` 模块只依赖 `autoconfigure` 和相关的第三方依赖，不写任何业务代码，仅作为依赖聚合器。
- `autoconfigure` 模块包含所有自动配置类、条件注解、配置绑定类。

2.  **配置绑定与条件匹配**
    -   使用 `@ConfigurationProperties` 绑定用户配置，提供默认值。
    -   配置类上使用 `@ConditionalOnClass`、`@ConditionalOnProperty` 控制自动配置的开启条件。
    -   关键 Bean 上使用 `@ConditionalOnMissingBean`，允许用户自定义 Bean 覆盖默认实现。

3.  **注册自动配置类**
    -   SpringBoot 2.x：在 `autoconfigure` 模块的 `META-INF/spring.factories` 中注册自动配置类。
    -   SpringBoot 3.x：在 `autoconfigure` 模块的 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件中写入自动配置类的全限定名。

4.  **保证可扩展性**
    -   提供 `xxx-spring-boot-autoconfigure` 模块的 `optional` 依赖，用户可以按需引入，避免依赖冲突。
    -   配置类中提供 `@Configuration` 标记的配置项，允许用户通过 `@Bean` 自定义扩展。
    -   配置文件提供 `spring.xxx.enabled` 开关，支持一键禁用 Starter。

## 2.2 本章核心易错点&踩坑总结
### 误区1：认为引入依赖就一定会自动配置，忽略条件注解匹配规则
- **问题本质**：自动配置不是“引入即生效”，而是**条件匹配后才生效**。
- **典型场景**：引入了 `spring-boot-starter-data-redis`，但项目中没有 `RedisTemplate` 类（依赖缺失），或者配置了 `spring.redis.enabled=false`，导致自动配置类不生效。
- **排查方式**：开启 `--debug` 模式，查看自动配置报告（`Positive matches`/`Negative matches`），确认是哪个条件未匹配。

### 误区2：自定义 Bean 和自动配置 Bean 冲突，不清楚优先覆盖规则
- **问题本质**：用户自定义的 Bean 优先级 **高于** 自动配置类中定义的 Bean，前提是自动配置的 Bean 上使用了 `@ConditionalOnMissingBean`。
- **典型场景**：自动配置类中定义了 `@Bean @ConditionalOnMissingBean public RedisTemplate redisTemplate()`，用户自己定义了一个 `RedisTemplate` Bean，自动配置的 Bean 就不会被加载。
- **避坑点**：如果自动配置类的 Bean 没有加 `@ConditionalOnMissingBean`，就会出现 `NoUniqueBeanDefinitionException` 异常。

### 误区3：使用旧版 `spring.factories` 适配 SpringBoot 3.x 导致配置失效
- **问题本质**：SpringBoot 3.x 已经移除了对 `spring.factories` 中 `EnableAutoConfiguration` 条目的支持，改用 `AutoConfiguration.imports` 文件。
- **典型场景**：升级到 3.x 后，自定义 Starter 或第三方 Starter 仍使用 `spring.factories` 注册自动配置类，导致配置类无法被加载。
- **解决方案**：将自动配置类的全限定名写入 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件中。

### 误区4：不理解自动配置加载顺序，导致自定义配置不生效
- **问题本质**：自动配置类的加载顺序会影响 Bean 的创建顺序，如果自定义配置依赖自动配置的 Bean，而自动配置类加载顺序靠后，就会出现 `NoSuchBeanDefinitionException`。
- **典型场景**：自定义配置类中依赖了自动配置类创建的 Bean，但自动配置类加载顺序晚于自定义配置类，导致自定义配置类初始化时 Bean 还未创建。
- **解决方案**：
    -   使用 `@AutoConfigureBefore`/`@AutoConfigureAfter` 调整自动配置类的加载顺序。
    -   自定义配置类使用 `@DependsOn` 注解指定依赖的 Bean。
    -   优先使用 `@ConditionalOnBean` 条件注解，保证依赖的 Bean 存在时才加载配置类。

# 3. 配置文件与多环境
## 3.1 高频面试真题

### Q1：请描述 SpringBoot 配置文件的完整加载优先级（含外部配置/环境变量/命令行参数），并说明为什么外部配置能覆盖内部配置？
**标准答案：**
SpringBoot 的配置加载遵循「**高优先级覆盖低优先级**」的原则，完整的优先级顺序（从高到低）如下：

1.  **命令行参数**：`java -jar app.jar --spring.profiles.active=prod --server.port=8081`
    -   最高优先级，直接通过命令行传递的参数，会覆盖所有其他配置源。
2.  **`SPRING_APPLICATION_JSON` 环境变量**：通过环境变量传递的 JSON 格式配置。
3.  **Servlet 上下文初始化参数**：Web 应用中的 `ServletContext` 初始化参数。
4.  **Servlet 初始化参数**：`ServletConfig` 初始化参数。
5.  **`JNDI` 属性**：`java:comp/env` 中的 JNDI 配置。
6.  **Java 系统属性**：`System.getProperties()`，如 `-Dspring.profiles.active=prod`。
7.  **操作系统环境变量**：系统级别的环境变量。
8.  **`@PropertySource` 注解**：在代码中通过 `@PropertySource` 加载的配置文件。
9.  **配置文件（外部 > 内部）**：
    -   外部配置文件：当前目录下的 `config/` 子目录、当前目录下的 `application.properties/yml`。
    -   内部配置文件：classpath 下的 `config/` 子目录、classpath 根目录下的 `application.properties/yml`。
10. **默认配置（SpringBoot 自动配置内置的默认值）**：如 `server.port=8080`。

**为什么外部配置能覆盖内部配置？**
SpringBoot 配置加载的核心机制是 `PropertySources`，它是一个有序的配置源集合。高优先级的配置源会被放在集合的前面，当配置键重复时，Spring 会直接返回第一个匹配的配置值，从而实现高优先级覆盖低优先级。
外部配置（如命令行参数、外部配置文件）会被添加到 `PropertySources` 的头部，而内部配置（如 classpath 下的配置文件）在尾部，因此外部配置会覆盖内部配置。

### Q2：`@ConfigurationProperties` 配置绑定的底层实现原理是什么？它和 `@Value` 在 Spring 容器初始化阶段的执行时机有什么区别？
**标准答案：**

#### 1. `@ConfigurationProperties` 底层实现原理
它的核心是由 `ConfigurationPropertiesBindingPostProcessor` 实现的，这是一个 `BeanPostProcessor`，负责在 Bean 初始化时，将配置文件中的值绑定到标记了 `@ConfigurationProperties` 的 Bean 上。

1.  **解析注解**：扫描所有标记了 `@ConfigurationProperties` 的类，获取 `prefix` 属性，作为配置文件中的前缀。
2.  **创建绑定器**：使用 `Binder` 类，结合 `Environment`（包含所有配置源）和 `ConversionService`（类型转换器），将配置文件中的键值对，按照前缀匹配到目标 Bean 的字段上。
3.  **类型转换**：自动将配置文件中的字符串值，转换为目标字段的类型（如 `String` → `int`、`List`、`Map`、自定义对象），支持嵌套对象绑定。
4.  **校验（可选）**：如果配合 `@Validated` 使用，会对绑定后的值进行 JSR-303 校验（如 `@NotNull`、`@Min`）。

#### 2. 与 `@Value` 的执行时机区别
| 对比项         | `@ConfigurationProperties`                                   | `@Value`                                                     |
| :------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **底层实现**   | `ConfigurationPropertiesBindingPostProcessor`（`BeanPostProcessor`） | `AutowiredAnnotationBeanPostProcessor`（`BeanPostProcessor`）的一部分 |
| **执行时机**   | 在 `BeanPostProcessor` 阶段执行，**晚于 Bean 的属性注入阶段** | 同样在 `BeanPostProcessor` 阶段执行，但**早于 `@ConfigurationProperties`** |
| **配置源支持** | 支持所有配置源（配置文件、环境变量、命令行参数）             | 支持所有配置源，但语法上依赖 `${}` 占位符                    |
| **绑定能力**   | 支持批量绑定、嵌套对象、List/Map、自动类型转换               | 仅支持单个字段绑定，需要手动处理类型转换                     |
| **适用场景**   | 批量配置绑定（如 Redis、数据库连接配置）                     | 单个配置项注入（如开关、简单字符串）                         |

## 3.2 本章核心易错点&踩坑总结

### 误区1：配置加载优先级理解错误，导致外部配置覆盖内部配置时排查困难
- **问题描述**：明明在 `application.yml` 里配置了 `server.port=8080`，但项目启动后端口却是 8081，不知道被哪里的配置覆盖了。
- **正确排查方法**：
  1.  开启 `--debug` 模式，查看 `Environment` 中的配置源顺序。
  2.  检查是否有外部配置文件（如当前目录下的 `application.yml`）、命令行参数、环境变量覆盖了配置。
  3.  使用 `Spring Boot Actuator` 的 `/env` 端点查看所有配置源的优先级。

### 误区2：`@ConfigurationProperties` 绑定失败，忽略了字段名、前缀、类型转换器的问题
- **问题描述**：配置文件中的值无法绑定到 Bean 字段上，没有报错，但字段始终是默认值或 `null`。
- **常见原因与解决：**
  1.  **前缀不匹配**：`@ConfigurationProperties(prefix = "redis")` 必须和配置文件中的 `redis.host` 完全对应。
  2.  **字段名大小写问题**：配置文件中的 `redis.max-active` 会自动映射到 `redis.maxActive`（驼峰命名），但如果字段名不规范（如 `RedisMaxActive`）会绑定失败。
  3.  **类型转换失败**：配置文件中写了非数字字符串，但字段类型是 `int`，且没有自定义转换器，会导致绑定失败。
  4.  **忘记加 `@EnableConfigurationProperties`**：在 SpringBoot 2.2+ 中，标记了 `@ConfigurationProperties` 的类会被自动扫描，但如果配置类不在主启动类的扫描包下，需要手动添加 `@EnableConfigurationProperties(XxxProperties.class)`。

### 误区3：多环境 Profile 激活方式错误，导致生产环境加载了测试配置
- **问题描述**：生产环境启动时，加载了测试环境的配置，导致连接到测试数据库。
- **常见原因与解决：**
  1.  **`spring.profiles.active` 配置位置错误**：该配置项不能写在 `application-test.yml` 中，只能写在 `application.yml` 或外部配置中，否则会被忽略。
  2.  **配置文件名错误**：多环境配置文件必须以 `application-{profile}.yml` 命名，否则不会被识别为 Profile 配置。
  3.  **Profile 激活方式冲突**：同时使用了命令行参数 `--spring.profiles.active=prod` 和配置文件中的 `spring.profiles.active=test`，由于命令行参数优先级更高，会以命令行为准。
  4.  **未正确使用 `@Profile` 注解**：标记了 `@Profile("dev")` 的配置类，在生产环境被错误激活，导致加载了测试 Bean。

# 4. Web开发与MVC增强 - 完整解答版
## 4.1 高频面试真题（附标准答案）

### Q1：SpringBoot 自定义 MVC 配置有哪两种方式？为什么推荐实现 `WebMvcConfigurer` 接口，而不是加 `@EnableWebMvc`？请结合自动配置的源码逻辑说明原因。
**标准答案：**
#### 1. 两种自定义 MVC 配置方式
- **方式一：实现 `WebMvcConfigurer` 接口（推荐）**
  仅重写需要修改的方法（如 `addInterceptors`、`addCorsMappings`、`configureMessageConverters` 等），不添加 `@EnableWebMvc` 注解。
- **方式二：添加 `@EnableWebMvc` 注解 / 继承 `WebMvcConfigurationSupport` 类（不推荐）**
  直接启用 SpringMVC 的完全手动配置模式，需要重写所有需要的配置方法。

#### 2. 底层源码逻辑与核心差异
SpringBoot 的 MVC 自动配置由 `WebMvcAutoConfiguration` 类实现，它的生效条件决定了两种方式的本质区别：
```java
@Configuration(proxyBeanMethods = false)
@ConditionalOnWebApplication(type = ConditionalOnWebApplication.Type.SERVLET)
@ConditionalOnClass({ Servlet.class, DispatcherServlet.class, WebMvcConfigurer.class })
@ConditionalOnMissingBean(WebMvcConfigurationSupport.class)
@AutoConfigureOrder(Ordered.HIGHEST_PRECEDENCE + 10)
@AutoConfigureAfter({ DispatcherServletAutoConfiguration.class, TaskExecutionAutoConfiguration.class,
    ValidationAutoConfiguration.class })
public class WebMvcAutoConfiguration {
    // ... 自动配置静态资源、消息转换器、视图解析器等
}
```
- `@ConditionalOnMissingBean(WebMvcConfigurationSupport.class)` 是关键：**只有当容器中没有 `WebMvcConfigurationSupport` 类型的 Bean 时，`WebMvcAutoConfiguration` 才会生效**。
  - 实现 `WebMvcConfigurer` 接口时，不会创建 `WebMvcConfigurationSupport` 类型的 Bean，`WebMvcAutoConfiguration` 会正常生效，SpringBoot 提供的默认 MVC 配置（如静态资源映射、默认消息转换器、日期格式化、视图解析器）都会保留，你的配置仅作为增强补充。
  - 添加 `@EnableWebMvc` 注解时，会导入 `DelegatingWebMvcConfiguration`（`WebMvcConfigurationSupport` 的子类），导致 `WebMvcAutoConfiguration` 失效，所有 MVC 配置都需要手动重写，否则会丢失大量默认行为（如静态资源无法访问、JSON 序列化异常等）。

#### 3. 结论
绝大多数场景下，使用 `WebMvcConfigurer` 接口做增强是最佳实践，既能自定义配置，又能保留 SpringBoot 的默认 MVC 能力；只有需要完全接管所有 MVC 配置的特殊场景，才使用 `@EnableWebMvc`。

### Q2：SpringBoot 中跨域问题的三种主流解决方案（`@CrossOrigin`/全局配置/过滤器），它们的底层实现和适用场景有什么区别？为什么有时配置了跨域还是会报预检请求（OPTIONS）错误？
**标准答案：**
#### 1. 三种跨域解决方案对比
| 方案                                           | 底层实现                                                     | 适用场景                                             | 优缺点                                                       |
| :--------------------------------------------- | :----------------------------------------------------------- | :--------------------------------------------------- | :----------------------------------------------------------- |
| `@CrossOrigin` 注解                            | SpringMVC 在 `RequestMappingHandlerMapping` 中处理请求时，检查方法/类上的 `@CrossOrigin` 注解，动态添加跨域响应头 | 仅少数接口需要跨域的场景                             | 优点：灵活度高，可按接口单独配置；缺点：每个接口都要加注解，维护成本高 |
| 全局配置（`WebMvcConfigurer#addCorsMappings`） | SpringMVC 在 `DispatcherServlet` 处理请求前，通过 `CorsProcessor` 统一添加跨域响应头 | 全应用统一跨域规则                                   | 优点：统一配置，维护成本低；缺点：只能配置规则，无法处理 Filter 链中的请求 |
| 自定义 `CorsFilter` 过滤器                     | 在 Servlet Filter 链的最前端，直接给响应添加跨域头，无需依赖 SpringMVC | 全应用跨域，包括静态资源、WebSocket、Filter 中的请求 | 优点：覆盖所有请求，优先级最高；缺点：配置不当容易出现预检请求问题 |

#### 2. 预检请求（OPTIONS）报错的核心原因
浏览器在发送复杂请求（如带自定义头、PUT/DELETE 方法、带 Cookie 的请求）前，会先发送 OPTIONS 预检请求，验证服务器是否允许跨域。报错的常见原因：
- 未允许 OPTIONS 方法：配置中 `allowedMethods` 未包含 `OPTIONS`，服务器直接返回 405 或 403。
- 未配置允许的请求头：`allowedHeaders` 未包含请求中的自定义头（如 `Authorization`）。
- 未开启允许凭证：请求带 Cookie 时，服务器未配置 `allowCredentials=true`，且 `allowedOrigins` 不能用 `*`。
- Filter 链顺序问题：跨域 Filter 不是第一个 Filter，后续 Filter 抛出异常，导致跨域头未被写入响应。

### Q3：`@RestControllerAdvice` 全局异常处理的底层原理是什么？哪些场景下它会失效？（比如 Filter 中抛出的异常、异步线程中的异常、404/405 等非业务异常）
**标准答案：**
#### 1. 底层实现原理
`@RestControllerAdvice` 本质上是通过 SpringMVC 的 `ExceptionHandlerExceptionResolver` 实现的：
1.  启动时扫描所有标记了 `@RestControllerAdvice` 的类，收集其中 `@ExceptionHandler` 注解的方法，建立「异常类型 → 处理方法」的映射关系。
2.  请求处理过程中，`DispatcherServlet` 捕获 Controller 层抛出的异常，遍历所有 `HandlerExceptionResolver`。
3.  `ExceptionHandlerExceptionResolver` 匹配到对应的 `@ExceptionHandler` 方法，执行方法并返回响应，最终写入 HTTP 响应。

#### 2. 全局异常处理失效的场景
- **Filter 链中抛出的异常**：Filter 在进入 SpringMVC 上下文之前执行，异常不会被 `DispatcherServlet` 捕获，因此 `@RestControllerAdvice` 无法处理，只能通过自定义 `ErrorPageFilter` 或容器层面的错误页面处理。
- **异步线程中的异常**：Controller 方法返回 `Callable`/`CompletableFuture` 时，业务逻辑在子线程执行，子线程抛出的异常不会被主线程的 `DispatcherServlet` 捕获，需要通过 `AsyncUncaughtExceptionHandler` 单独处理。
- **非 Controller 层的异常**：如 404（找不到 HandlerMapping）、405（请求方法不支持）、参数绑定异常（如 `MethodArgumentNotValidException`），部分异常由 SpringMVC 内置的 `DefaultHandlerExceptionResolver` 处理，若未配置对应 `@ExceptionHandler`，会返回默认错误响应。
- **响应已提交后的异常**：如响应流已写入部分数据后抛出异常，无法再修改响应头和状态码，全局异常处理也会失效。

### Q4：SpringBoot 内嵌 Tomcat 如何做性能优化？请从线程池、连接数、IO模型三个维度说明配置参数和调优思路。
**标准答案：**
内嵌 Tomcat 的性能优化，核心是根据业务场景调整线程模型、连接池和 IO 参数，避免高并发下出现请求堆积、连接超时。

#### 1. 线程池配置（核心调优项）
Tomcat 的线程池负责处理 HTTP 请求，默认配置在高并发下容易成为瓶颈：
```yaml
server:
  tomcat:
    threads:
      max: 400          # 最大工作线程数，建议设置为 CPU核心数*4~CPU核心数*8，根据业务IO密集程度调整
      min-spare: 100    # 最小空闲线程数，避免频繁创建销毁线程
```
- `max-threads`：建议根据业务类型调整：IO密集型接口可设置较高（如 200-500），CPU密集型接口建议较低（如 50-200），避免上下文切换开销。
- `min-spare-threads`：保持一定数量的空闲线程，减少请求等待时间。

#### 2. 连接数与队列配置
控制同时处理的连接数，避免请求堆积：
```yaml
server:
  tomcat:
    max-connections: 8192   # 最大同时连接数，超过后请求会被放入等待队列
    accept-count: 1000      # 等待队列长度，队列满后新请求会被拒绝（返回 503）
    connection-timeout: 20s  # 连接超时时间，超时后关闭连接，释放资源
```
- `max-connections`：Tomcat 能同时处理的最大连接数，建议根据服务器性能设置，一般不超过 10000。
- `accept-count`：当线程池已满时，新请求的等待队列长度，不宜设置过大，否则会导致请求排队时间过长。

#### 3. IO 模型优化
Tomcat 默认使用 NIO 模型，高并发场景下可进一步优化：
- **开启 NIO2/AIO（Tomcat 8+ 支持）**：
  ```yaml
  server:
    tomcat:
      protocol: org.apache.coyote.http11.Http11NioProtocol  # NIO 模型（默认）
      # 高并发下可尝试 NIO2，性能略优于 NIO
      # protocol: org.apache.coyote.http11.Http11Nio2Protocol
  ```
- **开启 APR 模式（需要安装 APR 库）**：基于操作系统原生的 IO 模型，性能最高，但配置复杂，生产环境较少使用。
- **开启 HTTP/2（Tomcat 9+ 支持）**：减少请求头阻塞，提升多资源请求性能，需要配置 HTTPS。

## 4.2 本章核心易错点&踩坑总结

### 误区1：自定义 MVC 配置时误用 `@EnableWebMvc`，导致 SpringBoot 自动配置的静态资源映射、消息转换器等默认配置全部失效
- **问题本质**：`@EnableWebMvc` 会让 `WebMvcAutoConfiguration` 失效，丢失所有默认 MVC 配置。
- **典型场景**：加了 `@EnableWebMvc` 后，静态资源无法访问、JSON 日期格式化失效、跨域配置不生效。
- **解决方式**：优先使用 `WebMvcConfigurer` 接口，不要随意添加 `@EnableWebMvc`。

### 误区2：全局异常处理器无法捕获 Filter 链中抛出的异常，误以为 `@RestControllerAdvice` 能处理所有请求异常
- **问题本质**：Filter 在进入 SpringMVC 上下文之前执行，异常不会被 `DispatcherServlet` 捕获。
- **典型场景**：认证 Filter 中抛出未登录异常，前端收到 403 状态码但没有统一的错误响应体。
- **解决方式**：通过 `@ControllerAdvice` 无法处理，需要在 Filter 中直接写响应，或配置 `server.error.path` 统一处理。

### 误区3：跨域配置只设置了 `Access-Control-Allow-Origin`，未配置 `Allow-Methods`/`Allow-Headers`/`Allow-Credentials`，导致预检请求报错
- **问题本质**：浏览器预检请求需要完整的跨域响应头验证，仅配置 Origin 无法通过校验。
- **典型场景**：POST/DELETE 等复杂请求报错，GET 请求正常，原因是预检请求 OPTIONS 被服务器拒绝。
- **解决方式**：跨域配置必须同时设置 `allowedOrigins`、`allowedMethods`、`allowedHeaders`，带 Cookie 的请求还需开启 `allowCredentials=true`，且不能用 `*` 作为 Origin。

### 误区4：忽略内嵌 Tomcat 的线程池配置，高并发场景下出现连接超时、请求堆积，只知道优化业务代码却没优化容器参数
- **问题本质**：Tomcat 默认线程池配置（max-threads=200）在高并发下容易成为瓶颈，导致请求排队、超时。
- **典型场景**：接口 QPS 上升后，响应时间变长，部分请求超时，业务代码无明显瓶颈。
- **解决方式**：根据服务器 CPU 核心数和业务类型调整 `max-threads`、`max-connections`、`accept-count`，同时监控线程池使用情况，避免线程耗尽。

# 5. 数据访问与事务管理 - 完整解答版
## 5.1 高频面试真题

### Q1：SpringBoot 配置多数据源时，如何解决跨数据源的事务一致性问题？常见的实现方案有哪些？
**标准答案：**
跨数据源的事务一致性问题，本质上是**分布式事务**问题，因为每个数据源都有自己独立的本地事务，单库事务无法跨数据源保证原子性。SpringBoot 中常见的解决方案按实现复杂度和适用场景，可分为以下几类：

#### 1. 方案一：`JTA` 分布式事务（强一致性）
- **原理**：基于 `XA` 协议的两阶段提交（2PC），由事务管理器（如 `Atomikos`、`Bitronix`）协调多个数据源的事务提交/回滚，保证所有数据源的事务同时成功或失败。
- **实现方式**：
  1.  引入 `spring-boot-starter-jta-atomikos` 依赖。
  2.  配置多个 `XADataSource` 数据源，注册到 Atomikos 事务管理器。
  3.  业务方法使用 `@Transactional` 注解，由 JTA 事务管理器统一管理跨数据源事务。
- **优缺点**：
  - ✅ 优点：强一致性，对业务代码侵入小，无需额外改造。
  - ❌ 缺点：性能较差（两阶段提交的阻塞问题），部分数据库（如 MySQL）对 XA 协议支持不完善，生产环境较少使用。

#### 2. 方案二：`Seata` 分布式事务（主流方案）
- **原理**：阿里开源的分布式事务框架，支持 AT（自动事务）、TCC、SAGA 等模式，其中 AT 模式对业务代码零侵入，是多数据源场景的首选。
- **实现方式（AT 模式）**：
  1.  部署 Seata Server 服务端。
  2.  引入 `seata-spring-boot-starter` 依赖，配置 Seata 注册中心、配置中心。
  3.  业务库中创建 `undo_log` 表，用于回滚日志记录。
  4.  业务方法添加 `@GlobalTransactional` 注解，即可实现跨数据源事务的全局控制。
- **优缺点**：
  - ✅ 优点：性能优于 JTA，对业务代码零侵入，支持多种数据库，社区活跃。
  - ❌ 缺点：需要额外部署 Seata Server，依赖第三方组件，存在网络故障时的极端一致性风险。

#### 3. 方案三：业务层改造（柔性事务，无强一致性依赖）
- **原理**：通过业务设计规避跨数据源事务，将跨库操作转为单库操作，或通过最终一致性方案实现。
- **常见实现方式**：
  1.  **合并数据源**：将多个数据源的表迁移到同一个数据库，从根源上消除跨库事务问题（推荐优先考虑）。
  2.  **异步消息队列**：跨库操作改为同步写主库 + 异步写从库，通过 MQ 实现最终一致性，失败时通过重试/补偿机制保证数据最终一致。
  3.  **TCC 模式**：手动实现 Try-Confirm-Cancel 三个阶段，通过业务代码控制事务的提交和回滚，性能高但对业务侵入性大。
- **优缺点**：
  - ✅ 优点：性能最高，无第三方依赖，可根据业务场景灵活实现。
  - ❌ 缺点：业务改造量大，需要处理重试、幂等、补偿逻辑，开发成本高。

#### 4. 方案对比与选型建议
| 方案     | 一致性   | 性能 | 业务侵入性 | 适用场景                               |
| :------- | :------- | :--- | :--------- | :------------------------------------- |
| JTA      | 强一致   | 低   | 低         | 简单多数据源场景，性能要求不高         |
| Seata AT | 最终一致 | 中   | 低         | 大多数分布式业务场景，性能与一致性平衡 |
| 业务改造 | 最终一致 | 高   | 高         | 高并发、高性能场景，可接受最终一致性   |

## 5.2 本章核心易错点&踩坑总结

### 误区1：受检异常不回滚
- **问题描述**：业务方法抛出 `IOException` 等受检异常，但事务没有回滚，导致数据不一致。
- **原因**：`@Transactional` 注解默认仅回滚 `RuntimeException`（运行时异常）和 `Error`，受检异常（Checked Exception）默认不会触发回滚。
- **解决方案**：
  1.  在 `@Transactional` 注解中显式配置 `rollbackFor` 属性：
     ```java
     @Transactional(rollbackFor = Exception.class)
     public void bizMethod() throws IOException {
         // 业务逻辑
     }
     ```
  2.  业务方法中统一抛出 `RuntimeException` 或自定义运行时异常。

### 误区2：多数据源下单库事务无效
- **问题描述**：业务方法中操作多个数据源，使用 `@Transactional` 注解，但其中一个数据源操作失败时，其他数据源的操作已经提交，数据不一致。
- **原因**：`@Transactional` 仅能控制单个数据源的本地事务，跨多个数据源的操作属于分布式事务，单库事务管理器无法感知其他数据源的事务状态，无法实现跨库回滚。
- **解决方案**：
  1.  优先考虑业务层面优化，如合并数据源、使用 MQ 异步化跨库操作。
  2.  必须跨库时，使用 Seata 或 JTA 实现分布式事务。

### 误区3：同类自调用导致事务失效
- **问题描述**：同一个类中，非事务方法调用 `@Transactional` 方法，事务没有生效。
- **原因**：Spring 事务基于 AOP 代理实现，只有外部调用经过代理对象时，事务增强逻辑才会生效。同类自调用时，方法直接通过 `this` 调用，绕过了代理对象，事务增强逻辑不会被执行。
- **解决方案**：
  1.  将事务方法抽取到单独的类中，通过 Spring 注入调用。
  2.  使用 `AopContext.currentProxy()` 获取当前代理对象，通过代理对象调用事务方法：
     ```java
     public void nonTxMethod() {
         ((ThisClass) AopContext.currentProxy()).txMethod();
     }
     
     @Transactional
     public void txMethod() {
         // 业务逻辑
     }
     ```

# 6. 日志、监控与项目部署 - 完整解答版
## 6.1 高频面试真题

### Q1：SpringBoot 项目中，如何排查日志依赖冲突？如果需要将默认的 Logback 切换为 Log4j2，具体要做哪些配置？
**标准答案：**

#### 一、日志依赖冲突排查方法
SpringBoot 默认使用 SLF4J + Logback 的日志组合，依赖冲突的核心表现是：控制台出现 `SLF4J: Class path contains multiple SLF4J bindings` 警告，或日志配置文件不生效、日志输出异常。排查步骤如下：

1.  **定位冲突依赖**
    使用 Maven 命令打印依赖树，定位冲突的日志实现包：
    ```bash
    mvn dependency:tree -Dincludes=ch.qos.logback:logback-classic,org.apache.logging.log4j:log4j-slf4j-impl
    ```
    找到同时引入了多个 SLF4J 绑定实现（如 `logback-classic` 和 `log4j-slf4j-impl`）的依赖。

2.  **排除冲突依赖**
    在冲突依赖中，排除掉不需要的日志实现，例如：
    ```xml
    <dependency>
        <groupId>com.example</groupId>
        <artifactId>conflict-lib</artifactId>
        <exclusions>
            <exclusion>
                <groupId>ch.qos.logback</groupId>
                <artifactId>logback-classic</artifactId>
            </exclusion>
        </exclusions>
    </dependency>
    ```

3.  **检查依赖传递**
    SpringBoot 自动配置中，`spring-boot-starter-logging` 默认引入了 Logback，若要切换日志框架，必须先排除该依赖。

#### 二、从 Logback 切换为 Log4j2 的配置步骤
1.  **排除默认 Logback 依赖**
    在 `spring-boot-starter-web` 中排除默认的 `spring-boot-starter-logging`：
    ```xml
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-web</artifactId>
        <exclusions>
            <exclusion>
                <groupId>org.springframework.boot</groupId>
                <artifactId>spring-boot-starter-logging</artifactId>
            </exclusion>
        </exclusions>
    </dependency>
    ```

2.  **引入 Log4j2 依赖**
    引入 `spring-boot-starter-log4j2`：
    ```xml
    <dependency>
        <groupId>org.springframework.boot</groupId>
        <artifactId>spring-boot-starter-log4j2</artifactId>
    </dependency>
    ```

3.  **添加 Log4j2 配置文件**
    在 `src/main/resources` 下创建 `log4j2.xml` 或 `log4j2-spring.xml` 配置文件（推荐 `log4j2-spring.xml`，支持 Spring 环境变量占位符），配置日志级别、输出格式、滚动策略等。

4.  **验证配置生效**
    启动项目，控制台无 SLF4J 冲突警告，日志按配置文件输出，说明切换成功。

### Q2：SpringBoot Actuator 有哪些常用端点？如何通过配置控制端点的暴露、安全和权限？生产环境中哪些端点建议开启，哪些建议关闭？
**标准答案：**

#### 一、常用 Actuator 端点
| 端点        | 作用                                    | 风险等级                   |
| :---------- | :-------------------------------------- | :------------------------- |
| `/health`   | 健康检查，返回应用的健康状态（UP/DOWN） | 低                         |
| `/info`     | 返回应用的基本信息（版本、环境等）      | 低                         |
| `/metrics`  | 应用运行指标（内存、CPU、请求数等）     | 中                         |
| `/loggers`  | 查看/修改日志级别                       | 中                         |
| `/env`      | 查看所有环境变量和配置属性              | 高（可能泄露敏感信息）     |
| `/heapdump` | 下载堆转储文件                          | 高（可能泄露内存敏感数据） |
| `/shutdown` | 关闭应用                                | 极高（可直接停止服务）     |

#### 二、端点暴露、安全与权限控制配置
1.  **控制端点暴露**
    -   `management.endpoints.web.exposure.include`：指定暴露的端点（如 `health,info`），`*` 表示暴露所有。
    -   `management.endpoints.web.exposure.exclude`：指定不暴露的端点（如 `shutdown,env`）。
    ```yaml
    management:
      endpoints:
        web:
          exposure:
            include: health,info,metrics
            exclude: shutdown,env,heapdump
    ```

2.  **开启端点安全认证**
    引入 `spring-boot-starter-security` 依赖，配置 Actuator 端点的访问权限：
    ```yaml
    management:
      endpoints:
        web:
          base-path: /actuator  # 自定义端点路径，避免默认路径被扫描
      security:
        enabled: true
    ```
    同时在 Security 配置中，限制只有管理员角色才能访问敏感端点：
    ```java
    @Configuration
    public class ActuatorSecurityConfig {
        @Bean
        public SecurityFilterChain actuatorSecurityFilterChain(HttpSecurity http) throws Exception {
            http
                .requestMatcher(new AntPathRequestMatcher("/actuator/**"))
                .authorizeRequests()
                .requestMatchers("/actuator/health", "/actuator/info").permitAll()
                .requestMatchers("/actuator/**").hasRole("ADMIN")
                .and().httpBasic();
            return http.build();
        }
    }
    ```

3.  **生产环境配置建议**
    -   **必开启**：`/health`（健康检查）、`/info`（基本信息）。
    -   **按需开启**：`/metrics`（监控指标）、`/loggers`（动态调整日志级别），需配置安全认证。
    -   **必须关闭**：`/shutdown`、`/env`、`/heapdump`，或仅在内部网络中通过 VPN/堡垒机访问。

### Q3：SpringBoot 项目 Jar 包部署到线上后，启动失败或运行异常，你会按什么步骤排查？请从日志、配置、端口、依赖四个维度说明。
**标准答案：**

#### 一、第一步：查看启动日志，定位错误根源
1.  **查看控制台/日志文件**：优先查看启动时的控制台输出或 `nohup.out` 文件，找到 `ERROR` 级别的日志，定位异常栈信息。
2.  **开启调试日志**：若日志信息不足，可临时修改 `application.yml` 中的日志级别为 `debug`，查看启动过程的详细日志，重点关注：
    -   自动配置类加载失败（`AutoConfiguration` 相关日志）。
    -   Bean 初始化失败（`NoSuchBeanDefinitionException`、`UnsatisfiedDependencyException`）。
    -   数据库连接失败（`CommunicationsException`）。

#### 二、第二步：检查配置文件，确认环境差异
1.  **配置文件加载**：检查 `application.yml`/`application.properties` 是否在 Jar 包中，或外部配置文件路径是否正确。
2.  **多环境 Profile**：确认 `spring.profiles.active` 配置正确，生产环境是否加载了测试/开发环境的配置。

2.  **网络连通性**：检查服务器防火墙是否开放了应用端口，或安全组策略是否限制了访问；测试数据库、Redis 等中间件的网络连通性（如 `telnet ip port`）。

#### 四、第四步：排查依赖冲突与 Jar 包问题
1.  **Jar 包完整性**：检查 Jar 包是否上传完整，可通过 `md5sum` 校验文件哈希值，或重新上传 Jar 包。
2.  **依赖冲突排查**：启动失败若出现 `NoClassDefFoundError` 或 `ClassNotFoundException`，说明依赖缺失或冲突，可通过 `jar tf app.jar` 查看 Jar 包中是否包含相关类，或使用 `mvn dependency:tree` 排查依赖冲突。
3.  **JDK 版本兼容**：确认线上服务器的 JDK 版本与项目编译版本一致，SpringBoot 3.x 要求 JDK 17+，若使用低版本 JDK 会启动失败。

## 7.1 高频面试真题

### Q1：`RedisTemplate` 和 `StringRedisTemplate` 底层区别是什么？`RedisTemplate` 默认序列化为什么会导致乱码？如何自定义序列化方案解决？
**标准答案：**
#### 1. 底层区别
`StringRedisTemplate` 是 `RedisTemplate` 的子类，二者核心差异在序列化器配置：
- `StringRedisTemplate`：默认使用 `StringRedisSerializer` 序列化 Key 和 Value，只能存储字符串类型数据，直接与 Redis 命令交互，无乱码问题，适合纯字符串场景。
- `RedisTemplate`：默认使用 `JdkSerializationRedisSerializer` 序列化 Key 和 Value，支持任意 Java 对象的序列化存储，但会产生乱码、性能差、可读性低等问题。

#### 2. 乱码产生原因
`JdkSerializationRedisSerializer` 会将对象序列化为二进制字节数组，并在数据中添加 JDK 序列化协议头、类信息等额外数据，存入 Redis 后，Key/Value 会出现 `\xAC\xED` 开头的乱码前缀，导致无法直接通过 Redis 客户端查看和管理数据。

#### 3. 自定义序列化解决方案
推荐自定义 `RedisTemplate`，配置 `StringRedisSerializer` 序列化 Key，`Jackson2JsonRedisSerializer` 序列化 Value：
```java
@Configuration
public class RedisConfig {
    @Bean
    public RedisTemplate<String, Object> redisTemplate(RedisConnectionFactory factory) {
        RedisTemplate<String, Object> template = new RedisTemplate<>();
        template.setConnectionFactory(factory);

        // Key 使用 String 序列化
        StringRedisSerializer stringSerializer = new StringRedisSerializer();
        template.setKeySerializer(stringSerializer);
        template.setHashKeySerializer(stringSerializer);

        // Value 使用 JSON 序列化
        Jackson2JsonRedisSerializer<Object> jsonSerializer = new Jackson2JsonRedisSerializer<>(Object.class);
        ObjectMapper mapper = new ObjectMapper();
        mapper.setVisibility(PropertyAccessor.ALL, JsonAutoDetect.Visibility.ANY);
        mapper.activateDefaultTyping(LaissezFaireSubTypeValidator.instance, ObjectMapper.DefaultTyping.NON_FINAL);
        jsonSerializer.setObjectMapper(mapper);
        template.setValueSerializer(jsonSerializer);
        template.setHashValueSerializer(jsonSerializer);

        template.afterPropertiesSet();
        return template;
    }
}
```
该方案解决了乱码问题，同时支持对象序列化，且数据在 Redis 中以 JSON 格式存储，可读性高。

### Q2：RabbitMQ 的消息确认机制（生产者确认/消费者确认）是什么？如何配置 `confirm` 和 `return` 回调？死信队列有哪些典型使用场景？
**标准答案：**
#### 1. 消息确认机制
RabbitMQ 的消息确认分为生产者确认和消费者确认两部分，保证消息不丢失：
- **生产者确认（Publisher Confirms）**：生产者发送消息后，Broker 会返回确认或否定回执，告知生产者消息是否成功到达交换机。
- **消费者确认（Consumer Acknowledgements）**：消费者处理消息后，手动/自动发送 ACK/NACK，告知 Broker 消息是否处理成功，失败的消息会被重新投递或进入死信队列。

#### 2. `confirm` 和 `return` 回调配置
- **Confirm 回调**：用于确认消息是否成功到达交换机：
  ```yaml
  spring:
    rabbitmq:
      publisher-confirm-type: correlated  # 开启 confirm 模式
  ```
  ```java
  rabbitTemplate.setConfirmCallback((correlationData, ack, cause) -> {
      if (ack) {
          // 消息成功到达交换机
      } else {
          // 消息未到达交换机，可进行重试或补偿
      }
  });
  ```
- **Return 回调**：用于处理消息到达交换机但无法路由到队列的情况：
  ```yaml
  spring:
    rabbitmq:
      publisher-returns: true
  ```
  ```java
  rabbitTemplate.setReturnsCallback(returned -> {
      // 消息路由失败，可记录日志或重发
  });
  ```

#### 3. 死信队列典型使用场景
死信队列（DLX）用于存储无法被正常消费的消息，典型场景：
- **消息消费失败重试**：消息被消费者 NACK 且不重入队列，或重试次数达到上限时，进入死信队列，后续人工处理。
- **延迟消息实现**：通过设置消息过期时间，消息超时后进入死信队列，由死信队列的消费者处理，实现延迟任务。
- **消息过滤与归档**：将无效消息、过期消息存入死信队列，不影响主业务队列，便于后续排查和分析。

### Q3：如何解决 MQ 消息重复消费、消息丢失问题？幂等性处理有哪些常见实现方案？
**标准答案：**
#### 1. 消息重复消费解决方案（核心是幂等性）
重复消费的根源是 MQ 的至少一次投递语义，需通过幂等性保证多次消费结果一致。常见实现方案：
- **基于消息 ID/业务唯一键**：为每个消息设置唯一 ID（如订单号、请求 ID），消费前查询数据库/Redis 中是否存在该 ID，存在则直接返回，不存在则执行业务逻辑并记录。
- **基于状态机控制**：业务数据设计状态字段（如订单状态），消费时判断状态，仅处理未处理/待处理状态的消息，避免重复更新。
- **基于 Redis 分布式锁**：消费时尝试获取锁，成功则处理，失败则放弃，利用锁的互斥性保证同一消息只被处理一次。

#### 2. 消息丢失解决方案
- **生产者端**：开启 `confirm` 和 `return` 回调，消息未到达交换机/队列时进行重试或补偿；使用事务发送，保证消息成功发送后再提交事务。
- **Broker 端**：开启消息持久化（交换机、队列、消息均设置持久化），集群模式下使用镜像队列，保证 Broker 宕机后消息不丢失。
- **消费者端**：使用手动 ACK 模式，业务处理成功后再发送 ACK，避免自动 ACK 导致消息丢失；消费失败时使用 NACK 并重入队列，或进入死信队列。

### Q4：SpringBoot 集成 Elasticsearch 时，如何解决版本兼容问题？`RestHighLevelClient` 的连接配置与优化思路是什么？
**标准答案：**
#### 1. 版本兼容问题解决
SpringBoot 与 Elasticsearch 客户端的版本必须严格匹配，否则会出现方法废弃、序列化失败、连接异常等问题：
- **版本对应关系**：SpringBoot 3.x 推荐使用 Elasticsearch 7.17+ 或 8.x 客户端，SpringBoot 2.x 推荐使用 7.x 客户端，禁止跨大版本使用（如 6.x 客户端连接 7.x 集群）。
- **依赖配置**：排除 SpringBoot 自动配置的低版本客户端，手动指定与集群版本一致的 `elasticsearch-rest-high-level-client` 依赖：
  ```xml
  <dependency>
      <groupId>org.elasticsearch.client</groupId>
      <artifactId>elasticsearch-rest-high-level-client</artifactId>
      <version>7.17.10</version>
  </dependency>
  ```

#### 2. `RestHighLevelClient` 连接配置与优化
- **基础配置**：配置集群节点、超时时间、认证信息：
  ```java
  @Bean
  public RestHighLevelClient restHighLevelClient() {
      RestClientBuilder builder = RestClient.builder(
          new HttpHost("es-node1", 9200, "http"),
          new HttpHost("es-node2", 9200, "http")
      );
      // 配置超时时间
      builder.setRequestConfigCallback(requestConfigBuilder -> 
          requestConfigBuilder.setConnectTimeout(5000)
                              .setSocketTimeout(30000)
      );
      // 配置认证（如需）
      builder.setHttpClientConfigCallback(httpClientBuilder -> 
          httpClientBuilder.setDefaultCredentialsProvider(credentialsProvider)
      );
      return new RestHighLevelClient(builder);
  }
  ```
- **优化思路**：
  - 配置连接池参数（最大连接数、路由连接数），避免高并发下连接耗尽。
  - 启用 HTTP/2 协议，提升请求性能。
  - 客户端单例模式，避免频繁创建销毁连接。

### Q5：Redis 主从复制、哨兵、集群模式分别解决了什么问题？SpringBoot 新的 `RedisClient`（Lettuce）如何配置集群连接？
**标准答案：**
#### 1. 三种模式的核心作用
- **主从复制**：实现数据备份和读写分离，主节点负责写，从节点负责读，解决单点故障和读性能瓶颈问题，但主节点故障时无法自动切换。
- **哨兵模式**：在主从复制基础上，实现主节点故障自动转移，哨兵集群监控主从节点状态，主节点宕机时自动将从节点提升为主节点，保证服务高可用。
- **集群模式**：实现数据分片和分布式存储，通过哈希槽将数据分布到多个节点，解决单节点内存、并发和存储瓶颈问题，同时支持高可用和水平扩展。

#### 2. Lettuce 集群连接配置
SpringBoot 2.x+ 默认使用 Lettuce 客户端，配置 Redis 集群连接：
```yaml
spring:
  redis:
    cluster:
      nodes: node1:6379,node2:6379,node3:6379  # 集群节点列表
      max-redirects: 3  # 集群重定向次数
    lettuce:
      pool:
        max-active: 16  # 最大连接数
        max-idle: 8    # 最大空闲连接数
        min-idle: 4    # 最小空闲连接数
```
配置后，`RedisTemplate` 会自动适配集群模式，自动处理槽位重定向和节点故障。

### Q6：高并发场景的 Redis 缓存如何做？如何避免缓存雪崩？缓存击穿的解决方案是什么？（追问：如何保证缓存和数据库的一致性？）
**标准答案：**
#### 1. 高并发 Redis 缓存设计要点
- **热点 Key 处理**：对热点 Key 进行本地缓存（如 Caffeine）或分片存储，避免单个 Key 压力过大。
- **缓存过期策略**：设置合理的过期时间，避免 Key 永不过期导致脏数据，同时避免大量 Key 同时过期引发雪崩。
- **数据预热**：系统启动时提前加载热点数据到缓存，避免冷启动时大量请求打到数据库。
- **降级与限流**：缓存故障时快速降级，直接访问数据库或返回默认值，避免服务雪崩。

#### 2. 缓存雪崩解决方案
缓存雪崩是指大量缓存 Key 同时过期，导致所有请求直接打到数据库，压垮数据库。解决方案：
- **过期时间打散**：给缓存 Key 的过期时间加上随机偏移量，避免大量 Key 同时过期。
- **多级缓存**：本地缓存 + Redis 缓存，Redis 缓存过期时，本地缓存可作为兜底。
- **永不过期策略**：热点 Key 设置逻辑永不过期，后台异步更新缓存，避免过期。
- **限流降级**：数据库层面限流，缓存雪崩时限制请求量，避免数据库压力过大。

#### 3. 缓存击穿解决方案
缓存击穿是指热点 Key 过期的瞬间，大量请求打到数据库。解决方案：
- **互斥锁**：缓存过期时，使用 Redis 分布式锁控制只有一个请求去更新缓存，其他请求等待。
- **永不过期**：热点 Key 设置逻辑永不过期，后台异步更新缓存。
- **提前续期**：缓存快过期时，提前异步更新缓存，避免过期瞬间大量请求涌入。

#### 4. 缓存与数据库一致性保证
- **更新策略**：先更新数据库，再删除缓存（推荐），或先更新数据库，再更新缓存，避免并发更新导致的数据不一致。
- **重试机制**：缓存更新失败时，通过消息队列重试更新，或通过定时任务兜底更新。
- **读写分离**：读请求走缓存，写请求直接操作数据库，保证缓存最终一致性。

## 7.2 本章核心易错点&踩坑总结

### 误区1：Redis 默认 JDK 序列化导致乱码与性能问题
- **问题描述**：使用 `RedisTemplate` 时，未自定义序列化器，默认的 `JdkSerializationRedisSerializer` 导致 Key/Value 乱码、可读性差，且序列化后数据体积大，占用 Redis 内存多。
- **避坑方法**：自定义 `RedisTemplate`，使用 `StringRedisSerializer` 序列化 Key，`Jackson2JsonRedisSerializer` 序列化 Value，或直接使用 `StringRedisTemplate` 手动序列化对象。

### 误区2：未配置缓存过期时间与空值缓存，引发缓存三大问题
- **问题描述**：缓存 Key 永不过期导致脏数据，热点 Key 同时过期引发雪崩，不存在的 Key 未做空值缓存导致穿透。
- **避坑方法**：给所有缓存 Key 设置合理的过期时间并打散，不存在的数据设置空值缓存（短期过期），热点 Key 使用永不过期策略。

### 误区3：MQ 消费无幂等性处理，导致重复消费数据异常
- **问题描述**：MQ 消息重复投递，消费者未做幂等校验，导致数据重复插入、更新异常。
- **避坑方法**：基于消息 ID/业务唯一键、状态机、Redis 分布式锁实现幂等性，保证同一消息只被处理一次。

### 误区4：忽略中间件与 SpringBoot 版本适配，出现连接失败/方法废弃问题
- **问题描述**：SpringBoot 升级后，中间件客户端版本未同步升级，导致方法废弃、连接池失效、序列化失败等问题。
- **避坑方法**：严格按照 SpringBoot 官方文档的版本对应关系配置中间件客户端，避免跨大版本使用，升级时同步测试兼容性。
