# 1. ElasticSearch整体认知

## 1.1 ES介绍与技术定位

### 1.1.1 什么是全文检索搜索引擎

想要理解ES，首先要搞懂**全文检索**的核心概念，这是区别于传统数据库精准查询的核心关键。

传统的MySQL、Oracle等关系型数据库，仅支持**精准匹配**或简单模糊查询（like %关键词%），这类查询存在极大局限性：一是模糊查询无法利用索引，海量数据下查询效率极低；二是无法实现分词匹配、语义模糊检索、相关性权重排序等功能。

**全文检索搜索引擎**是一种专门针对文本内容进行**分词、索引、匹配、排序**的检索工具，核心逻辑是：提前将海量文本数据按照分词规则拆分关键词，构建专属索引库，用户检索时，通过匹配关键词而非完整文本，快速筛选出相关性最高的数据并排序返回。

简单来说，精准查询是“找完全一样的内容”，全文检索是“找语义相关、关键词匹配的内容”。而ElasticSearch就是一款基于Lucene内核开发的、开源分布式全文检索搜索引擎，同时具备数据存储、聚合分析的能力，是目前行业主流的检索中间件。

### 1.1.2 ES适用业务场景

ES的核心能力聚焦**海量文本检索、模糊匹配、相关性排序、实时聚合统计**，所有适配该能力的业务场景均可使用，主流适用场景分为以下几类：

**1. 全文文本检索场景**：核心业务需要根据文本关键词模糊搜索，如电商商品搜索、公众号文章检索、APP站内内容搜索、知识库文档查询等，支持分词匹配、拼音搜索、高亮显示、权重排序等高级检索功能。

**2. 海量日志分析场景**：服务器日志、业务操作日志、用户行为日志数据量大、格式不固定，MySQL无法高效存储和检索。ES可实时采集、存储日志，支持按关键词、时间、维度筛选日志，搭配Kibana实现日志可视化分析、异常日志快速排查。

**3. 实时数据聚合统计场景**：针对海量业务数据，需要快速实现分组统计、求和、平均值、 TOP排名等聚合分析，如实时订单统计、用户访问量统计、商品销量排行等，查询速度远快于MySQL。

**4. 数据中台检索场景**：企业海量结构化、非结构化数据的统一检索入口，支持多维度、多条件组合检索，适配大数据量、高并发的检索请求。

### 1.1.3 ES不适合使用场景

ES并非万能中间件，存在明确的场景局限性，盲目使用会导致业务复杂度提升、性能变差，以下场景**不建议使用ES**：

**1. 高频更新、高频写入的交易核心场景**：ES是近实时检索组件，数据写入后存在短暂刷新延迟，无法做到MySQL的实时一致性。同时高频更新、删除操作会产生大量版本日志，引发索引膨胀、性能下降，不适合订单支付、资金交易等强实时、高更新的核心业务。

**2. 强事务、强一致性业务场景**：ES不支持关系型数据库的ACID事务、行锁、表锁机制，无法保证数据强一致性，不适合需要事务回滚、数据强一致的业务（如库存扣减、转账交易）。

**3. 少量数据简单查询场景**：业务数据量小（万级及以下）、仅需简单精准查询，使用MySQL完全足够。引入ES会增加部署、运维、开发成本，属于过度设计。

**4. 纯结构化数据的复杂关联查询**：ES不支持多表关联JOIN查询，复杂的多表联查、嵌套关联统计，使用MySQL更简洁高效，ES实现成本极高。

### 1.1.4 ES与MySQL使用场景取舍

实际开发中，ES和MySQL通常是**互补搭配**的关系，而非替代关系，掌握二者取舍规则是业务落地的关键，也是高频面试考点，具体取舍规则如下：

|对比维度|MySQL|ElasticSearch|场景取舍结论|
|---|---|---|---|
|查询方式|精准查询、简单模糊查询、多表关联查询|全文分词检索、模糊匹配、相关性排序、多维聚合|文本检索、排序、聚合用ES；精准、关联查询用MySQL|
|数据一致性|支持ACID强事务、强一致性|最终一致性，近实时，无事务支持|交易、库存等强一致场景用MySQL|
|数据量级|适合千万级及以下数据，海量数据查询性能骤降|支持亿级、十亿级海量数据，检索性能稳定|海量数据检索、统计优先用ES|
|写入更新|实时写入、更新、删除，性能稳定|批量写入优，高频更新删除性能差|高频更新业务主库用MySQL，检索用ES同步数据|
|运维成本|低，成熟稳定，运维简单|中高，集群部署、分片优化、内存调优成本高|小数据简单业务无需引入ES|

**企业通用最佳实践**：MySQL作为业务主库，负责存储核心业务数据、处理事务与精准CRUD；ES作为检索辅助库，通过同步机制（Canal、MQ）同步MySQL数据，专门负责全文检索、海量数据聚合统计。

## 1.2 ES核心优势与特点

### 1.2.1 近实时检索特性

ES最核心的特性之一就是**近实时（NRT）**，官方定义的数据延迟为1秒左右，区别于离线检索工具的小时级、分钟级延迟。

核心原理：ES数据写入后，不会立即写入磁盘持久化，而是先写入内存缓冲区和translog日志，默认1秒执行一次refresh刷新，将内存数据生成可检索的索引快照，此时数据即可被查询到。后续再通过flush操作将数据持久化到磁盘。

该特性让ES既保证了**极高的写入性能**，又满足了业务**实时检索**的需求，适配线上动态更新的商品、文章、日志数据检索场景。相较于传统离线搜索引擎，无需等待批量构建索引，体验大幅提升。

### 1.2.2 分布式横向扩展能力

ES原生支持**分布式集群架构**，无需额外改造即可实现横向扩容，完美适配业务数据量持续增长的场景，这是单体数据库无法比拟的优势。

核心特点：ES集群由多个节点组成，数据会自动拆分分片存储在不同节点，集群支持动态扩容、缩容。当数据量增长、检索并发升高时，只需新增服务器节点，集群会自动重新分配分片、均衡数据，无需停机、无需手动迁移数据，业务无感知。

同时集群具备**容错机制**，通过副本分片实现数据冗余，单个节点故障不会导致数据丢失、服务宕机，保证检索服务的高可用，适配生产环境高可用要求。

### 1.2.3 海量数据存储与快速查询

基于**倒排索引**的底层数据结构，是ES海量数据快速查询的核心支撑。传统MySQL正向索引在海量数据下，模糊查询会全表扫描，性能极差；而ES提前对文本分词、构建倒排索引，检索时直接通过关键词定位数据，查询时间复杂度极低。

在亿级数据场景下，MySQL的模糊查询、聚合查询可能耗时数秒甚至数十秒，而ES可实现**毫秒级响应**。同时ES支持冷热数据分离、分片存储，可高效承载TB、PB级海量数据存储，且查询性能不会随数据量增长出现断崖式下跌。

### 1.2.4 丰富聚合统计能力

ES不仅是搜索引擎，还是一款**轻量级大数据分析引擎**，内置强大的聚合（Aggs）查询能力，支持各类复杂的数据统计分析。

核心聚合能力包括：基础聚合（求和、平均值、最大值、最小值、计数）、分组聚合（按时间、维度分组统计）、区间聚合、TOP排行聚合、嵌套聚合等。无需编写复杂SQL，通过简洁的DSL语句即可实现海量数据的实时统计。

该特性让ES可以替代部分大数据组件的实时统计场景，无需借助Hadoop、Spark，即可快速完成业务数据的实时分析，常搭配Kibana实现数据可视化大屏、日志统计报表。

## 1.3 行业主流使用场景

### 1.3.1 商品电商搜索

电商是ES最经典、最广泛的应用场景，淘宝、京东、拼多多等主流电商平台的商品检索均基于ES实现。

业务需求：电商商品数量动辄千万、上亿级别，需要支持用户输入任意关键词模糊搜索，同时实现分词匹配（如“无线蓝牙耳机”可匹配“耳机”“蓝牙”）、拼音搜索、错别字纠错、商品相关性排序、价格筛选、销量排序、检索结果高亮等功能。

落地方案：MySQL存储商品核心交易数据，通过同步机制将商品名称、简介、分类、品牌等文本数据同步至ES，用户搜索请求直接访问ES，快速返回匹配度最高的商品数据，保证高并发、高响应速度。

### 1.3.2 日志收集分析系统

日志分析是后端开发中ES的核心落地场景，几乎所有互联网企业的日志系统均采用**ELK架构（ElasticSearch+Logstash+Kibana）**。

业务需求：服务器运行日志、接口请求日志、异常日志、用户行为日志数据量大、更新频繁、格式零散，传统日志文件排查效率极低，无法快速定位线上问题。

落地方案：Logstash/Flink采集各类日志数据，清洗过滤后实时写入ES，ES对日志数据建立索引，支持按时间、接口、异常关键词、用户ID等多维度检索，开发者可快速筛选异常日志、统计报错率、分析接口性能，同时通过Kibana实现日志可视化展示、实时监控告警。

### 1.3.3 站内文章内容检索

自媒体平台、博客系统、知识库、新闻网站等内容类平台，核心需求是站内全文内容检索，是ES的典型适配场景。

业务需求：平台存在海量文章、文档、资讯内容，用户需要通过标题、正文关键词检索相关内容，支持模糊匹配、内容高亮、热度排序、时间筛选等功能，MySQL完全无法满足海量文本检索需求。

落地方案：文章发布后，自动将标题、作者、正文、标签等数据同步至ES，利用ES分词能力实现全文检索，同时根据文章热度、发布时间、匹配权重进行排序，提升用户检索体验。

### 1.3.4 大数据实时统计分析

针对海量业务数据的**实时多维统计**场景，ES凭借强大的聚合能力，成为轻量化大数据分析工具。

常见业务场景：网站用户访问量实时统计、小程序UV/PV统计、商品实时销量排行、各区域订单量统计、接口调用频次统计、异常请求占比分析等。

落地优势：相较于传统的离线大数据分析框架，ES无需等待批量计算，数据写入后即可实时聚合统计，响应速度快、开发成本低，适合业务实时数据监控、中小型数据统计分析场景，可快速搭建数据可视化大屏，满足运营、运维的实时数据查看需求。

# 2. ES核心架构与基础概念

## 2.1 ES基础架构组成

### 2.1.1 节点作用与节点类型

节点是ES集群的最小组成单元，一台部署了ES服务的服务器就是一个节点。ES节点支持**角色拆分**，不同节点承担不同职责，合理划分节点角色是生产环境集群优化的核心，共分为4种核心节点类型：

**1. 主节点（Master Node）**

核心职责：负责集群全局管理，包括集群节点管理、索引创建/删除、分片分配、集群状态同步、节点故障检测等全局操作，不负责数据检索、写入的具体业务请求。

特性：主节点不存储业务数据，压力小，稳定性要求高。集群中通过选举机制选出唯一主节点，其他备用主节点作为容错节点，主节点故障后自动重新选举，保证集群可用。

**2. 数据节点（Data Node）**

核心职责：集群核心业务节点，负责**数据存储、数据写入、检索查询、聚合计算**，所有业务数据的读写操作、分片存储均由数据节点完成。

特性：占用服务器内存、磁盘、CPU资源较高，是集群压力最大的节点，生产环境需重点优化硬件配置，承担核心业务流量。

**3. 客户端节点（Client Node/协调节点）**

核心职责：请求调度与负载均衡，接收客户端的读写请求，将请求分发到对应的数据节点，汇总多个节点的执行结果并返回给客户端。

特性：不存储数据、不参与主节点选举，仅负责请求转发、结果聚合、负载均衡，可有效分担数据节点的请求压力，提升集群并发能力。

**4. 预处理节点（Ingest Node）**

核心职责：负责数据写入前的预处理工作，包括数据清洗、字段替换、格式转换、文本拆分、数据过滤等，无需额外开发代码，通过ES内置管道即可实现数据预处理。

适用场景：日志数据清洗、脏数据过滤、文本格式统一等数据预处理场景。

### 2.1.2 集群工作机制

ES集群是多节点协同工作的分布式架构，核心工作机制包含**集群选举、分片分配、故障容错、数据同步**四大核心逻辑，是理解ES高可用、高扩展的关键。

**1. 集群选举机制**

ES集群中所有具备主节点资格的节点会参与选举，遵循**少数服从多数**原则，选举出唯一的活跃主节点管理集群。为避免脑裂问题，生产环境通常将最小主节点选举数设置为集群半数以上节点。主节点负责统一集群状态，所有集群变更操作均由主节点统筹调度。

**2. 分片分配机制**

索引创建时，主节点会根据集群节点数量、硬件资源、分片规则，将索引的主分片、副本分片均匀分配到不同的数据节点。核心原则：**主分片和对应的副本分片绝对不会存储在同一个节点**，避免单节点故障导致数据丢失。

**3. 故障容错机制**

集群实时监控所有节点状态，若某个数据节点故障下线，主节点会立即检测到节点异常，自动将该节点上的主分片、副本分片重新分配到其他正常节点，通过副本分片恢复数据完整性，同时集群自动重启分片同步，故障恢复后再重新均衡数据，全程无需人工干预，保证服务高可用。

**4. 集群扩容机制**

新增节点加入集群后，主节点会自动识别新节点，重新均衡所有索引的分片分布，将部分分片迁移至新节点，实现负载均衡，全程业务无感知，支持横向无限扩容。

### 2.1.3 客户端访问交互流程

客户端（Java代码、Kibana、Postman）访问ES集群的完整交互流程，是开发调试、问题排查的基础，标准流程分为6步：

1. **请求接入**：客户端发起检索、写入、更新等DSL请求，优先发送至协调节点（无协调节点则随机发送至集群任意节点）。

2. **路由解析**：协调节点根据索引分片路由规则，解析出当前请求对应的**主分片/副本分片**所在的数据节点地址。

3. **请求分发**：协调节点将请求转发至目标数据节点，写请求默认转发至主分片节点，读请求可转发至主分片或副本分片节点，实现读请求负载均衡。

4. **节点执行**：数据节点接收请求，执行数据写入、检索、聚合等操作，完成本地数据处理。

5. **结果汇总**：若请求涉及多个分片、多个节点，协调节点会接收所有节点的执行结果，进行合并、排序、聚合、去重等二次处理。

6. **结果返回**：协调节点将处理后的最终结果封装，返回给客户端，完成一次完整请求交互。

**核心优化点**：读请求支持主副分片分流，大幅提升集群查询并发能力；写请求仅操作主分片，成功后同步至副本分片，保证数据一致性。

## 2.2 核心基础名词解释

### 2.2.1 索引、类型、文档、字段

ES的核心数据概念可与MySQL数据库做**一一对应**，方便开发者快速理解记忆，同时明确新旧版本差异（面试高频考点）。

**1. 索引（Index）**

对应MySQL的**数据库**，是存储一类相同业务数据的集合，是ES数据存储的顶层单元。例如：商品数据创建goods索引、日志数据创建log索引、文章数据创建article索引。所有数据的读写、检索操作都基于索引完成。

**2. 类型（Type）**

对应MySQL的**数据表**，用于对索引内的数据做细分分类。**重点版本差异**：ES7.0版本之后彻底废弃Type概念，一个索引只能对应一种数据类型，废弃多Type设计，目的是简化架构、避免类型冲突，生产环境新版本均不再使用Type。

**3. 文档（Document）**

对应MySQL的**数据行（一条记录）**，是ES最小的数据存储单元，格式为JSON格式。一条文档就是一条完整的业务数据，例如一条商品数据、一篇文章、一条日志。每个文档都有唯一的_id作为主键，用于唯一标识数据。

**4. 字段（Field）**

对应MySQL的**数据表字段**，是文档中的单个属性，例如商品的名称、价格、分类、创建时间都属于字段。ES字段支持多种数据类型（文本、数值、日期、布尔、数组等），且可灵活拓展，无需提前严格定义表结构。

### 2.2.2 分片、主分片、副本分片基础认知

分片是ES实现**分布式存储、海量数据拆分、高可用**的核心机制，单个索引的数据量过大时，ES会将数据拆分存储在多个分片上，分散至不同节点。

**1. 主分片（Primary Shard）**

核心定义：索引数据的核心分片，负责数据的**写入、更新、删除**操作，是数据的基准分片。

核心特性：索引创建时**主分片数量固定，无法修改**（修改会导致数据重组、索引失效）。主分片数量决定了索引的最大写入并发能力，合理的主分片数量是集群性能优化的关键。数据写入仅能写入主分片，写入成功后同步至副本分片。

**2. 副本分片（Replica Shard）**

核心定义：主分片的备份分片，是主分片的完整数据副本，用于数据容错和读请求负载均衡。

核心特性：副本分片数量**支持动态修改**，可根据集群压力随时调整。副本分片不参与写请求，仅承担读请求、数据备份容错。且副本分片与对应主分片不在同一节点，单节点故障时，副本分片可升级为主分片，保证数据不丢失、服务不中断。

**3. 分片核心公式**：集群总分片数 = 主分片数 + 主分片数 × 副本分片数

**生产避坑要点**：主分片数量不宜过多（过多导致分片分散、查询聚合效率低），不宜过少（过少无法扩容、并发不足），通常根据数据量级设置，亿级数据常用3-5个主分片。

### 2.2.3 倒排索引核心原理

**倒排索引是ES实现高速全文检索的底层核心**，也是面试必问核心考点，区别于MySQL的正向索引，彻底解决了海量文本检索慢的问题。

**1. 正向索引（MySQL默认）**

存储逻辑：以**数据行为核心**，一行数据存储完整的字段信息。检索逻辑：用户输入关键词，需要逐行遍历所有数据，匹配对应字段内容，海量数据下遍历效率极低，这也是MySQL模糊查询慢的根本原因。

**2. 倒排索引（ES核心）**

存储逻辑：以**关键词（词条）为核心**，先对所有文本数据分词拆分，建立「关键词-数据文档」的映射关系，核心包含两部分：

- 词条词典：存储所有拆分后的唯一关键词，自带索引，可快速定位关键词；

- 倒排记录表：存储每个关键词对应的所有文档ID、出现次数、出现位置、权重等信息。

**3. 检索执行流程**

① 用户输入检索关键词，ES对关键词进行相同规则的分词处理；

② 在词条词典中快速匹配对应的关键词；

③ 通过倒排记录表获取该关键词对应的所有文档ID；

④ 根据文档ID读取完整文档数据，结合相关性权重排序后返回结果。

**4. 核心优势**

无需遍历全量数据，直接通过关键词映射定位数据，查询效率提升数百倍；同时可记录关键词权重、位置，实现**相关性排序、高亮显示、精准匹配**，完美适配全文检索场景。

# 3. 分词器原理与常用分词

## 3.1 分词器工作流程

### 3.1.1 字符过滤、分词、词条过滤

分词器（Analyzer）是ES实现**全文检索**的核心组件，所有文本写入、文本查询都会经过分词器处理。ES分词并非一步完成，而是固定分为**字符过滤、分词切割、词条过滤**三阶段流水线处理，每个阶段各司其职，最终生成可用于索引和检索的标准词条。

**1. 字符过滤器（Character Filters）**

属于预处理阶段，作用是对原始文本进行清洗、修正，去除无效字符、替换特殊符号，不改变文本核心语义。该阶段可以处理三类常见问题：

- 去除特殊字符：去除HTML标签、空格、换行符、特殊转义字符等无效内容；

- 字符替换：将全角字符转为半角、统一大小写、替换敏感符号；

- 文本规整：修正文本格式，保证后续分词准确性。

示例：原始文本为`<h1>Java开发 实战教程！</h1>`，经过字符过滤后，清洗为`Java开发 实战教程`。

**2. 分词器核心切割（Tokenizer）**

整个分词流程的核心阶段，负责将**完整字符串按照指定规则切割为独立词条（Term）**。不同分词器的核心差异就体现在切割规则上。

- 英文文本：按空格、标点符号切割，拆分出独立单词；

- 中文文本：原生分词器按单字切割，中文分词器按词语、语义切割；

- 数字、符号：按规则独立拆分。

**3. 词条过滤器（Token Filters）**

分词完成后的后置处理阶段，对切割后的词条进行二次优化、过滤、标准化，最终生成可入库的索引词条。常用处理逻辑：

- 小写转换：统一英文单词为小写，避免大小写检索不匹配；

- 停用词过滤：去除无意义虚词（的、了、a、the），减少索引体积；

- 同义词替换：配置同义词库，实现近义词检索；

- 词条去重、长度过滤：过滤过短、无效词条。

**整体流程总结**：原始脏文本 → 字符清洗过滤 → 规则切割分词 → 词条标准化优化 → 最终索引词条，整套流程也是ES倒排索引构建的前置核心步骤。

### 3.1.2 标准分词器特点与缺陷

**标准分词器（Standard Analyzer）**是ES的**默认内置分词器**，无需手动配置，系统所有文本字段默认使用该分词器处理，是面试高频考点。

**1. 核心特点**

- 适配英文场景：基于Unicode文本分割算法，自动按空格、标点、符号切割英文单词；

- 自动小写转换：所有英文字母统一转为小写，检索不区分大小写；

- 轻量无冗余：无复杂过滤逻辑，分词速度快，性能损耗低；

- 开箱即用：无需任何配置，ES默认启用，适合纯英文项目。

**2. 致命缺陷（中文场景不适用）**

标准分词器**不支持中文语义分词**，对中文文本采用**单字切割**方式，会导致严重的业务问题：

- 分词结果碎片化：例如`Java开发教程`会被拆分为`J、a、v、a、开、发、教、程`单个字符；

- 索引体积爆炸：单字分词会生成海量无效词条，大幅占用磁盘和内存资源；

- 检索精度异常：无法实现词语匹配，搜索“开发”无法匹配“Java开发教程”；

- 相关性排序失效：碎片化词条导致检索权重混乱。

**生产结论**：纯英文项目可使用默认标准分词器，**所有中文业务项目必须替换为IK中文分词器**。

## 3.2 中文分词器使用

### 3.2.1 IK分词器安装配置

**IK Analyzer**是目前ES生态中**最主流、最稳定的开源中文分词器**，专门适配中文语义分词，支持词语拆分、自定义词库、停用词配置，完全满足国内业务场景。

**1. 版本适配原则（生产必看）**

IK分词器版本**必须与ES版本完全一致**，例如ES8.13.0必须搭配IK8.13.0，版本不匹配会直接导致ES启动失败、分词失效。

**2. 安装步骤（Linux生产环境）**

1）进入ES插件目录

```shell
# 进入ES插件安装目录
cd /usr/local/elasticsearch/plugins/
# 创建ik分词器专属目录
mkdir ik-analyzer
cd ik-analyzer
```

2）下载对应版本IK分词器（以8.13.0为例）

```shell
# 下载官方安装包
wget https://github.com/medcl/elasticsearch-analysis-ik/releases/download/v8.13.0/elasticsearch-analysis-ik-8.13.0.zip
# 解压安装包
unzip elasticsearch-analysis-ik-8.13.0.zip
# 删除压缩包，清理冗余文件
rm -rf elasticsearch-analysis-ik-8.13.0.zip
```

3）重启ES服务生效

```shell
# 停止ES进程
pkill -f elasticsearch
# 后台启动ES
nohup /usr/local/elasticsearch/bin/elasticsearch 
```

**3. 验证安装是否成功**

通过Kibana DevTools执行分词测试，无报错且正常分词即为安装成功：

```json
  "analyzer": "ik_max_word",
  "text": "Java后端开发实战教程"
```

### 3.2.2 细粒度/粗粒度分词区别

IK分词器提供**两种核心分词模式**，适配不同的业务检索场景，是业务开发和面试高频考点。

**1. ik_max_word（细粒度分词/最大分词）**

核心逻辑：对文本进行**穷尽式拆分**，拆分出所有可能的词语组合，分词粒度最细、词条数量最多。

示例：`程序员开发教程` 拆分结果：程序员、程序、开发、教程

**适用场景**：**索引创建阶段**。写入数据时使用细粒度分词，生成最全的词条索引，保证用户任何相关关键词都能检索到数据，提升检索覆盖率。

**2. ik_smart（粗粒度分词/智能分词）**

核心逻辑：智能语义拆分，只拆分出**最贴合语义的核心词语**，无冗余词条，分词粒度粗、词条数量少。

示例：`程序员开发教程` 拆分结果：程序员、开发、教程

**适用场景**：**用户检索阶段**。用户输入关键词检索时使用粗粒度分词，避免检索词条冗余，提升查询效率和精准度。

**生产最佳实践（必记）**

索引字段分词规则统一配置：**索引写入用ik_max_word，查询检索用ik_smart**，兼顾检索全覆盖与查询高性能。

### 3.2.3 自定义词库、停用词配置

原生IK分词器词库为通用词库，无法识别行业专属词汇（如微服务、中间件、AI大模型），同时包含大量无意义通用词汇，因此生产环境必须配置**自定义扩展词库**和**停用词库**。

**1. 配置文件位置**

IK分词器配置文件路径：`ES安装目录/plugins/ik-analyzer/config/IKAnalyzer.cfg.xml`

**2. 自定义扩展词库（让分词器识别新词）**

场景：原生词库无法识别`微服务`、`云原生`、`豆包AI`等行业新词，需要手动添加。

操作步骤：

1）在config目录新建文件：`custom_ext.dic`，一行一个自定义词汇；

2）修改`IKAnalyzer.cfg.xml`配置，加载自定义词库：

```xml
<!-- 自定义扩展词库 -->
<entry key="ext_dict">custom_ext.dic</entry>
```

**3. 自定义停用词库（过滤无意义词汇）**

场景：过滤`的、了、啊、就是、非常`等无检索意义的词汇，减少索引体积，提升检索效率。

操作步骤：

1）在config目录新建文件：`custom_stop.dic`，一行一个停用词汇；

2）修改配置文件加载停用词库：

```xml
<!-- 自定义停用词库 -->
<entry key="ext_stopwords">custom_stop.dic</entry>
```

**4. 热更新特性**

IK分词器支持**词库热更新**，修改自定义词库文件后无需重启ES，等待1分钟左右自动加载生效，不影响业务运行。

**生产避坑点**

1. 自定义词库文件必须为**UTF-8无BOM编码**，否则分词失效；

2. 词汇不要重复、不要过长，避免分词混乱；

3. 行业专属词汇优先加入扩展词库，保证业务检索精准性。

# 4. ES环境快速搭建

## 4.1 单机ES部署

### 4.1.1 版本选择与环境依赖

**1. 版本选择原则（生产选型）**

- 企业稳定首选：**7.x版本**（生态成熟、Bug少、兼容性强，目前主流生产版本）；

- 新技术学习首选：**8.x版本**（支持新特性、性能优化，默认开启安全认证）；

- 禁止使用6.x及以下老旧版本，停止维护、存在安全漏洞。

**2. 核心环境依赖**

- JDK环境：ES7.x内置JDK11，ES8.x内置JDK17，**无需手动安装JDK**，避免版本冲突；

- 系统环境：Linux CentOS7+/Ubuntu18+，Windows仅适合本地测试；

- 硬件要求：单机测试最低2G内存，生产环境建议8G以上内存。

**3. 系统前置配置（必须配置，否则启动报错）**

ES不支持root用户启动，且需要修改系统资源限制，提前执行以下配置：

```shell
# 1. 创建普通用户es，禁止root启动
useradd es
# 授权目录权限
chown -R es:es /usr/local/elasticsearch

# 2. 修改系统最大文件打开数
echo "* soft nofile 65535" >> /etc/security/limits.conf
echo "* hard nofile 65535" >> /etc/security/limits.conf

# 3. 修改虚拟内存配置
echo "vm.max_map_count=262144" &gt;&gt; /etc/sysctl.conf
sysctl -p
```

### 4.1.2 配置文件核心修改

ES核心配置文件共2个，分别控制**服务器参数**和**JVM内存参数**，是部署核心重点。

**1. elasticsearch.yml（服务核心配置）**

路径：`ES安装目录/config/elasticsearch.yml`，修改核心配置（注释原默认配置）：

```yaml
# 集群名称，单机可自定义
cluster.name: es-cluster
# 节点名称
node.name: node-1
# 绑定所有IP，允许外网访问
network.host: 0.0.0.0
# 对外服务端口
http.port: 9200
# 集群通信端口
transport.tcp.port: 9300
# 单机环境禁止集群自动发现
discovery.type: single-node
# 开启跨域，方便Kibana和前端访问
http.cors.enabled: true
http.cors.allow-origin: "*"
```

**2. jvm.options（内存配置）**

路径：`ES安装目录/config/jvm.options`，根据服务器内存调整：

- 测试机器2G内存：-Xms512m -Xmx512m

- 生产机器8G内存：-Xms4g -Xmx4g

**核心规则**：最大最小内存必须一致，防止内存抖动；ES堆内存**不超过物理内存的50%**，不超过31G。

### 4.1.3 启动关闭与后台运行

**1. 切换普通用户启动（必须）**

```shell
su es
# 前台启动（适合测试，可查看日志）

# 后台启动（生产使用）
```

**2. 查看索引结构**

```json
GET /article_index/_mapping
```

**3. 删除索引**

```json
DELETE /article_index
```

**生产避坑**：删除索引为高危操作，会清空所有数据，生产环境禁止随意执行。
