# 1. ES线上性能调优

ES性能调优是生产环境稳定运行的核心，主要分为**JVM内存调优、写入性能调优、查询性能调优**三大板块。其中内存调优是基础，直接决定集群是否出现OOM、卡顿、GC频繁等致命问题；读写调优针对业务场景解决吞吐量低、响应慢的问题，所有调优方案均经过生产验证，可直接落地复用。

## 1.1 JVM内存调优

ES基于Java虚拟机运行，JVM内存参数不合理是线上**内存溢出、集群卡顿、节点掉线、查询超时**的首要原因。ES官方对JVM内存有严格规范，禁止随意配置，本节详解生产标准配置、优化比例、溢出排查方案。

### 1.1.1 堆内存设置规范

**1. 核心官方硬性规范（面试必考、生产必守）**

- **最大堆内存不超过31GB**：JVM压缩指针在内存大于31GB时失效，内存占用激增、性能大幅下降；

- **最小堆等于最大堆（Xms=Xmx）**：避免运行时动态扩容缩容，减少GC压力，稳定内存占用；

- **堆内存不超过物理内存50%**：ES需要充足系统内存用于系统缓存、页缓存，剩余内存留给操作系统缓存磁盘数据。

**2. 生产分级配置标准**

- 8G物理内存服务器：JVM堆内存设置4G；

- 16G物理内存服务器：JVM堆内存设置8G；

- 32G物理内存服务器：JVM堆内存设置16G；

- 64G物理内存服务器：JVM堆内存设置31GB（上限）。

**3. 配置文件实操（jvm.options）**

```properties
# 生产标准配置：16G堆内存，固定大小
-Xms16g
-Xmx16g
# 开启GC日志，方便故障排查
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
# 使用G1垃圾收集器（ES默认推荐）
-XX:+UseG1GC
```

**4. 核心原理说明**

ES的索引元数据、分片信息、查询缓存、聚合计算数据全部存在堆内存，堆过小会导致频繁GC、内存溢出；堆过大会挤占系统缓存，磁盘读取命中率下降，检索性能暴跌。

### 1.1.2 新生代老年代比例优化

**1. 内存区域特性**

ES属于**频繁创建短期临时对象**的应用，查询、聚合、分片计算会产生大量瞬时对象，新生代回收频率极高，合理的新老生代比例可以大幅减少Full GC。

**2. 生产最优比例**

默认新老生代比例1:2，生产ES推荐调整为**新生代占堆内存1/3**，提升新生代空间，减少Minor GC频率，避免短期对象提前进入老年代触发Full GC。

**3. 优化配置**

```properties
# 设置新生代比例，优化短期对象回收
-XX:NewRatio=2
# 避免内存溢出时退出，保留堆快照用于排查
-XX:+HeapDumpOnOutOfMemoryError
```

**4. 优化价值**

有效解决ES运行一段时间后老年代快速填满、频繁Full GC、集群周期性卡顿的线上问题。

### 1.1.3 内存溢出常见原因

**1. 高频OOM原因汇总（生产全覆盖）**

- JVM堆内存设置过小，无法承载集群查询、聚合压力；

- 深分页、超大聚合查询、无限制聚合导致堆内存瞬时暴涨；

- 字段数据过大、大文本字段未拆分，加载大量数据至内存；

- GC配置不合理，频繁GC导致内存碎片、内存泄漏；

- 集群分片过多，元数据占用堆内存过高。

**2. OOM排查流程**

1. 查看堆快照文件，定位大对象、常驻内存对象；

2. 监控GC日志，判断是Minor GC频繁还是Full GC频繁；

3. 排查近期是否上线复杂聚合、超大分页、批量查询语句；

4. 核对分片数量、单分片数据量是否超标。

## 1.2 写入性能优化

ES单条写入性能极差，海量数据写入场景极易出现吞吐量低、写入超时、磁盘IO打满问题。写入优化核心思路：**减少IO次数、降低刷新频率、合并小文件、减少磁盘刷盘压力**。

### 1.2.1 批量批量Bulk写入优化

**1. 单条写入痛点**

单条文档写入会触发一次完整的网络请求、内存刷新、磁盘操作，IO开销极大，海量数据场景完全无法满足性能要求。

**2. Bulk批量写入原理**

将多条增删改请求合并为一次网络请求，批量提交至ES，大幅减少网络交互次数、线程切换开销，写入吞吐量可提升5~10倍。

**3. 生产最佳批量参数**

- 单次Bulk条数：**1000~3000条**；

- 单次Bulk数据大小：**5MB~15MB**；

- 设置超时时间，避免大批量阻塞线程池。

**4. Java客户端核心实操代码（可直接复用）**

```java
/**
 * ES Bulk批量写入优化示例
 * 生产标准：1000条一批次批量提交
 */
public void bulkInsert() {
    // 构建批量请求
    BulkRequest bulkRequest = new BulkRequest();
    // 设置超时时间
    bulkRequest.timeout(TimeValue.timeValueSeconds(10));

    // 循环添加1000条文档
    for (int i = 0; i < 1000; i++) {
        IndexRequest request = new IndexRequest("goods_index");
        request.source(JSON.toJSONString(entityList.get(i)), XContentType.JSON);
        bulkRequest.add(request);
    }

    // 批量执行
    BulkResponse response = restHighLevelClient.bulk(bulkRequest, RequestOptions.DEFAULT);
    // 批量失败排查
    if (response.hasFailures()) {
        log.error("批量写入失败：{}", response.buildFailureMessage());
    }
}
```

**5. 避坑指南**

批量条数不宜过大，超过5000条容易导致请求超时、内存瞬间飙升，引发集群压力雪崩。

### 1.2.2 刷新间隔、事务日志调优

**1. 核心参数原理**

- **refresh_interval（刷新间隔）**：ES默认1秒刷新一次，将内存数据刷新到文件缓存，实现数据可搜索。频繁刷新会极大损耗写入性能；

- **translog（事务日志）**：类似MySQL redo log，保证数据持久化不丢失，默认每次写入都刷盘，性能极低。

**2. 海量数据初始化优化方案**

数据全量初始化、大批量同步场景，临时调优参数，写入完成后恢复默认：

```json
# 海量写入临时优化配置
PUT /goods_index/_settings
{
  "refresh_interval": "30s",    // 刷新间隔从1s改为30s，减少刷盘次数
  "translog.durability": "async", // 事务日志异步刷盘，提升写入速度
  "number_of_replicas": 0       // 关闭副本，避免副本同步消耗性能
}
```

**3. 生产常态配置**

日常业务写入场景，refresh_interval保持默认1s，保证数据实时可见；translog保持同步持久化，保证数据安全。

### 1.2.3 段合并后台优化

**1. 段合并机制**

ES写入数据会生成大量小分片段文件，后台线程会自动合并小段为大段，段合并会消耗大量CPU、磁盘IO，是写入卡顿的重要原因。

**2. 优化策略**

- 业务低峰期自动执行段合并，高峰期限制合并线程数；

- 海量写入期间，临时延迟段合并，写入完成后统一合并；

- 避免频繁更新删除数据，减少碎片段产生。

### 1.3 查询检索性能优化

查询优化核心目标：**减少不必要计算、减少磁盘读取、减少内存占用、精准命中数据**，解决线上检索超时、聚合卡顿、首页加载慢等问题。

### 1.3.1 合理使用filter减少算分

**1. query与filter核心区别（面试高频）**

- **query**：会进行相关性打分、排序计算，消耗CPU，适用于模糊搜索、关键词检索；

- **filter**：仅做精准匹配、不进行算分、支持缓存，性能远高于query，适用于等值筛选、范围筛选。

**2. 优化原则**

**能使用filter绝不使用query**，所有精准条件、时间范围、状态筛选全部放入filter，仅关键词检索使用query。

**3. 优化示例**

```json
# 高性能检索：query做关键词匹配，filter做精准筛选
GET /goods_index/_search
{
  "query": {
    "bool": {
      "must": [
        {"match": {"goods_name": "Java教程"}}
      ],
      "filter": [
        {"term": {"status": "上架"}},
        {"range": {"price": {"gte": 100}}}
      ]
    }
  }
}
```

### 1.3.2 索引冷热分离设计

**1. 冷热分离原理**

业务数据存在明显冷热差异：近期热门数据（热数据）频繁查询，历史归档数据（冷数据）极少访问。冷热分离将不同热度数据拆分存储，针对性优化配置，提升查询性能、节省集群资源。

**2. 生产落地方案**

- **热数据**：近期日志、新商品、新订单，存储在高性能SSD节点，开启副本、开启缓存，保证查询速度；

- **冷数据**：历史归档数据，存储在普通磁盘节点，关闭副本、降低刷新频率，节省资源；

- 按天/按月拆分索引，实现冷热数据自动拆分。

### 1.3.3 字段索引关闭优化

**1. 核心优化思想**

索引越多，写入压力越大、磁盘占用越高、检索越慢。**不参与查询、筛选、排序、聚合的字段，一律关闭索引**。

**2. 字段索引配置实操**

```json
# 无用字段关闭索引，仅存储不检索
PUT /goods_index
{
  "mappings": {
    "properties": {
      "goods_detail": {
        "type": "text",
        "index": false  // 关闭索引，仅用于存储展示，不参与检索
      },
      "create_user": {
        "type": "keyword",
        "index": true
      }
    }
  }
}
```

**3. 最佳实践**

- 大文本详情、备注、冗余字段全部关闭索引；

- 仅业务检索、筛选、排序字段开启索引；

- text字段按需开启，避免全字段分词索引。

# 2. ES常见线上故障排查

本节汇总生产环境最高频的ES线上故障，提供**标准化排查流程、故障原因、解决方案、预防机制**，覆盖集群红黄异常、分片分配失败、读写卡顿、磁盘爆满、数据堆积等核心问题，实现故障快速定位、快速修复。

## 2.1 集群状态异常排查

ES集群状态分为Green（健康）、Yellow（亚健康）、Red（异常），生产出现非Green状态必须立即排查，避免故障扩大。

### 2.1.1 红索引、黄索引故障原因

**1. 黄色索引（Yellow）原因**

**主分片全部正常，部分副本分片未分配、未同步**，集群可正常读写，无数据丢失，但丧失容灾能力。

常见诱因：节点资源不足、分片数量超限、节点下线、网络波动、磁盘水位过高。

**2. 红色索引（Red）原因**

**存在主分片未分配、丢失、损坏**，对应索引部分或全部不可读写，数据丢失、业务异常，属于严重故障。

常见诱因：主分片所在节点宕机、磁盘损坏、索引文件损坏、分片强制删除、集群扩容缩容异常。

### 2.1.2 分片无法分配故障处理

**1. 排查命令（生产必备）**

```json
# 查看未分配分片详情，定位故障原因
GET /_cat/shards?v&h=index,shard,prirep,state,node,unassigned.reason
```

**2. 常见未分配原因与解决方案**

- **磁盘使用率过高**：ES默认磁盘使用率超85%停止分片分配，清理磁盘、扩容磁盘；

- **分片数超过节点数**：副本节点不足，减少副本数或新增集群节点；

- **分片分配规则限制**：自定义路由规则导致无法分配，重置索引分配策略；

- **节点属性不匹配**：冷热分离节点属性不一致，导致分片无法迁移。

**3. 紧急修复方案**

临时降低副本数，让集群先恢复健康状态，业务恢复后再排查根因、扩容节点。

## 2.2 查询慢、写入慢排查思路

**1. 标准化排查流程**

1. **查看集群监控**：确认CPU、磁盘IO、内存、网络是否打满；

2. **定位慢日志**：开启ES慢查询日志、慢写入日志，捕获耗时异常的DSL；

3. **分析DSL语句**：排查是否存在深分页、无边界聚合、大量text排序、模糊匹配；

4. **检查分片状态**：是否存在分片不均衡、单节点压力过大；

5. **检查JVM状态**：是否频繁GC、内存溢出、线程池阻塞。

**2. 高频慢查询诱因**

- 使用match_all无条件全量查询；

- 超大from深分页查询；

- 未使用filter，全量数据算分；

- 无索引字段检索、排序；

- 单次聚合维度过多、数据量过大。

## 2.3 磁盘爆满告警处理方案

**1. 磁盘爆满核心危害**

磁盘使用率超过阈值，ES会自动触发**只读保护模式**，所有写入请求失败，业务新增数据完全中断。

**2. 紧急处理步骤**

1. 清理无用日志、临时文件、过期备份，快速释放磁盘空间；

2. 删除过期无用索引、测试索引、临时索引；

3. 执行磁盘碎片清理、段合并清理冗余文件；

4. 磁盘空间释放后，解除只读保护；

5. 长期解决：扩容磁盘、冷热分离、定时清理过期数据。

**3. 解除只读保护命令**

```json
# 磁盘空间释放后，解除集群只读冻结
PUT /_all/_settings
{
  "index.blocks.read_only_allow_delete": null
}
```

## 2.4 索引数据冷热清理策略

**1. 生产通用清理方案**

针对日志、监控、流水类时序数据，采用**按天拆分索引+定时删除过期索引**策略，避免单索引数据无限膨胀。

**2. 最佳实践规范**

- 热数据：保留近7~30天数据，正常读写查询；

- 温数据：迁移至低成本磁盘，降低副本数，节省资源；

- 冷数据：超过保留周期，定时删除索引，彻底释放空间；

- 禁止单索引永久存储海量时序数据。

**3. 自动化方案**

通过定时任务、ES生命周期管理ILM，实现索引自动创建、自动迁移、自动删除，无需人工干预。

# 3. ES业务实战避坑总结

ES大部分线上故障、性能瓶颈、业务异常，并非复杂架构问题，而是开发阶段的**基础使用不规范**导致。本节汇总生产最高频的四大实战大坑，拆解问题本质、危害、解决方案与落地规范，适配日常CRUD开发与项目迭代。

## 3.1 text与keyword使用大坑

**1. 两种字段类型核心本质**

**keyword**：精准匹配类型，整体存储原始文本，**不分词**，支持精准查询、模糊查询、排序、聚合；

**text**：全文检索类型，写入数据后**自动分词拆分**，仅支持分词模糊检索，不支持精准匹配、排序、聚合。

### 3.1.1 线上高频踩坑点

- **坑1：状态、分类、编号字段误用text类型**：例如商品状态、订单类型、用户标签存储为text，导致term精准查询查不到数据、分组聚合结果错乱，是线上最高频bug；

- **坑2：搜索字段误用keyword类型**：商品名称、文章标题存储为keyword，无法分词模糊搜索，只能完整匹配，业务功能失效；

- **坑3：全字段默认text**：新手开发直接自动创建索引，所有字符串字段默认text，导致大量筛选、排序、聚合功能异常。

### 3.1.2 生产标准使用规范

- **必用keyword**：状态码、分类ID、手机号、邮箱、订单号、标签、枚举值、需要筛选/排序/聚合的固定文本字段；

- **必用text**：商品名称、文章内容、简介、描述、需要模糊分词搜索的长文本字段；

- **最优方案（复合字段）**：核心业务文本字段采用 **text+keyword双字段** 设计，兼顾搜索与筛选排序。

### 3.1.3 标准映射示例

```json
PUT /goods_index
{
  "mappings": {
    "properties": {
      // 商品名称：text分词搜索 + keyword精准筛选排序
      "goods_name": {
        "type": "text",
        "fields": {
          "keyword": {
            "type": "keyword",
            "ignore_above": 256
          }
        }
      },
      // 商品状态：固定枚举，仅用keyword
      "status": {
        "type": "keyword"
      },
      // 商品详情：仅展示不检索，关闭索引
      "detail": {
        "type": "text",
        "index": false
      }
    }
  }
}
```

## 3.2 深分页线上业务禁忌

**1. 深分页定义**

ES中 **from + size** 分页，当from值过大（通常超过1000）即为深分页，属于线上**严格禁止**的操作。

### 3.2.1 深分页底层致命问题

ES分页底层逻辑：查询第N页数据时，会**加载前所有页全部数据、全局排序、丢弃前置数据、返回当前页**。

例如 from=10000、size=10，需要加载10010条数据到内存，排序后丢弃前10000条，仅返回10条，会造成**堆内存暴涨、CPU打满、查询超时、集群卡顿**，并发场景直接拖垮整个集群。

### 3.2.2 线上业务禁忌规范

- 禁止前端传入无限分页参数，后台必须做参数拦截，限制from最大值；

- ES官方默认限制最大分页深度为1000，超出直接报错；

- 所有列表业务，禁止使用深分页实现翻页。

### 3.2.3 生产替代方案

- **常规列表**：限制最大分页页数，仅支持前100页，超出提示无更多数据；

- **大数据量翻页**：使用 **scroll滚动分页**（离线导出、批量同步）；

- **实时业务翻页**：使用 **search_after 游标分页**，无深度限制、性能极高。

## 3.3 大索引拆分时间拆分设计

**1. 单大索引致命问题**

日志、订单、流水、监控等时序数据，若全部写入同一个索引，会导致索引数据量无限膨胀，出现分片超大、查询变慢、备份缓慢、扩容困难、故障恢复耗时极长等问题。

### 3.3.1 时间拆分核心方案

生产标准方案：**按时间维度拆分索引**，根据业务数据量选择按天/按周/按月拆分索引。

- 数据量极大（每日千万级）：**按天拆分**，如 log-2026-05-16；

- 数据量中等（每日百万级）：**按周拆分**；

- 数据量较小：**按月拆分**。

### 3.3.2 时间拆分核心优势

- **查询性能提升**：检索指定时间段数据，仅查询对应索引，无需扫描全量数据；

- **运维成本降低**：过期数据直接删除对应索引，无需批量删除文档，无碎片压力；

- **集群负载均衡**：单个索引数据量可控，分片大小稳定，避免超大分片；

- **容灾能力更强**：单索引故障不影响全量业务数据。

### 3.3.3 通配符查询适配

多索引场景可通过通配符批量查询，无需逐个索引适配，业务无感知：

```json
# 批量查询所有5月日志索引
GET /log-2026-05-*/_search
```

## 3.4 ES与MySQL数据同步方案

ES不承担业务数据存储能力，仅用作检索加速，所有核心数据最终落地MySQL，因此**MySQL与ES的数据实时同步**是ES业务落地的核心环节，必须保证数据一致性、实时性、可靠性。

行业主流同步分为三类：业务双写、Logstash定时同步、Canal实时监听，三种方案适配不同业务场景，后续章节详细拆解，本节先明确选型核心原则：

- **高实时、强一致业务**：优先 Canal 监听binlog同步；

- **低实时、日志归档、全量初始化**：优先 Logstash 同步；

- **简单小型项目、快速迭代**：临时使用业务双写，规避复杂架构。

# 4. ES高频面试题汇总

本节汇总互联网面试**必考、高频、高阶**ES面试题，覆盖基础原理、检索机制、集群高可用、性能优化、数据同步五大核心板块，全部提供**标准化满分答题模板**，可直接背诵面试。

## 4.1 基础原理面试题

### 4.1.1 倒排索引原理详解

**满分答题模板：**

倒排索引是ES实现高速全文检索的核心数据结构，与数据库正排索引（文档→字段）相反，核心结构是**词条→文档**的映射关系。

1. 构建流程：文档写入ES后，分词器对文本字段分词，拆分出独立词条；记录每个词条对应的文档ID、出现位置、频次等信息，生成倒排索引表持久化落地。

2. 检索流程：用户检索关键词时，先对关键词分词，匹配倒排索引中的词条，快速命中对应文档ID，无需全表扫描，检索效率远超MySQL模糊查询。

3. 核心优势：支持海量数据分词检索、模糊匹配、相关性算分，是搜索引擎的底层核心原理。

### 4.1.2 分片与副本作用与规划

**满分答题模板：**

1. 主分片（Primary Shard）：是数据读写的核心分片，索引创建时指定，数量**永久不可修改**。核心作用是拆分海量数据，实现分布式水平扩容，突破单机存储上限，所有写请求必须在主分片执行。

2. 副本分片（Replica Shard）：是主分片的完整数据备份，数量可动态调整。核心作用是实现**高可用容灾**，主节点故障时副本自动升主；同时分担读请求压力，提升查询并发能力。

3. 规划原则：单分片数据量控制在10-30GB；主分片数根据预估数据量计算；生产核心索引副本数至少1，保证容灾能力。

## 4.2 检索相关面试题

### 4.2.1 match与term区别

**满分答题模板：**

1. term：精准匹配查询，**不分词**，完全匹配字段原始内容，适用于keyword字段精准筛选、状态匹配、ID匹配，无相关性算分，自带缓存性能高。

2. match：全文检索查询，**对检索关键词分词**，模糊匹配text字段，会做相关性打分排序，适用于商品名称、文章内容等全文搜索场景。

3. 核心避坑：text字段无法使用term精准查询，keyword字段无法使用match分词模糊查询，混用会导致查询无结果。

### 4.2.2 分页三种方式优缺点

**满分答题模板：**

ES主流三种分页方案：from+size、scroll、search_after

1. from+size分页：优点是使用简单、适配常规列表；缺点是存在深分页问题，from过大会内存溢出、性能极差，仅支持浅分页。

2. scroll滚动分页：优点是支持海量数据分页、性能稳定；缺点是基于快照实现，不支持实时数据，游标占用资源，适合离线导出、数据同步，不适合线上用户翻页。

3. search_after游标分页：优点是无分页深度限制、性能极高、支持实时数据；缺点是需要携带上一页最后一条数据游标，不支持跳页查询，是线上大数据量翻页最优方案。

## 4.3 集群高可用面试题

### 4.3.1 脑裂问题如何解决

**满分答题模板：**

1. 脑裂成因：集群网络抖动导致集群分裂为两个子集群，无合理投票机制时，两个子集群各自选举主节点，出现双主节点异常，导致数据错乱、集群失控。

2. 核心解决方案：第一，主节点采用**奇数节点部署（3/5个）**；第二，配置**最小主节点投票数=总主节点数/2+1**，保证分裂后的子集群无法满足选举条件，从根源杜绝双主产生。

3. 辅助优化：优化集群内网环境，调优心跳超时时间，避免临时网络抖动误判节点下线。

### 4.3.2 主分片为什么不能修改

**满分答题模板：**

1. ES文档分片路由公式：shard = hash(routing) % 主分片数，文档的归属分片由主分片数量固定计算得出。

2. 若主分片数量修改，所有文档的取模结果会全部变更，原有数据分片归属彻底错乱，导致所有数据无法检索、数据丢失。

3. 因此ES限制索引创建后主分片不可修改，如需扩容只能新建索引、迁移数据。

## 4.4 性能优化面试题

### 4.4.1 如何提升ES写入速度

**满分答题模板：**

1. 批量写入：使用Bulk批量API，单次1000-3000条，减少网络IO次数；

2. 临时调优：海量初始化时关闭副本、延长refresh刷新间隔、开启translog异步刷盘；

3. 架构优化：大索引按时间拆分，避免单索引数据臃肿；

4. 资源优化：优化JVM内存配比，避免频繁GC，提升节点吞吐；

5. 规避碎片：减少频繁更新删除操作，降低段合并开销。

### 4.4.2 如何优化ES查询速度

**满分答题模板：**

1. DSL优化：精准筛选使用filter不算分，减少CPU消耗；避免match_all、深分页、无边界聚合；

2. 索引优化：无用字段关闭索引，文本字段合理拆分text/keyword；大索引时间拆分；

3. 架构优化：冷热数据分离，热数据使用高性能磁盘、增加副本分担读压力；

4. 分页优化：深分页替换为search_after游标分页；

5. 缓存优化：合理利用filter缓存、分片缓存，提升重复查询命中率。

## 4.5 数据同步场景面试题

### 4.5.1 MySQL如何同步ES

**满分答题模板：**

行业主流三种同步方案：

1. 业务双写：代码层同时写入MySQL和ES，实现简单，实时性高；

2. Logstash同步：定时抓取MySQL数据同步ES，无需改代码，适合全量初始化、低实时场景；

3. Canal同步：监听MySQL binlog日志，实时解析增量数据同步ES，高实时、高可用、解耦业务，是生产主流方案。

### 4.5.2 实时同步与延时同步选型

**满分答题模板：**

1. 实时同步（Canal）：适用于商品搜索、订单检索、用户查询等**对数据一致性、实时性要求高**的核心业务，秒级同步，无数据延迟。

2. 延时同步（Logstash定时同步）：适用于日志统计、报表分析、数据归档、后台统计等**允许分钟级延迟**的非核心业务，架构简单、维护成本低。

# 5. 主流数据同步方案

MySQL与ES数据同步是ES项目落地的核心难点，不同同步方案的实时性、一致性、复杂度、适用场景差异极大。本节深度拆解三种主流同步方案的原理、实操、优缺点与生产选型，覆盖99%业务场景。

## 5.1 Logstash同步方案

### 5.1.1 方案原理

Logstash是ES官方数据同步工具，通过JDBC驱动定时查询MySQL数据表，增量/全量拉取数据，转换格式后同步至ES索引。

### 5.1.2 核心配置示例

```ruby
# Logstash同步MySQL配置
input {
  jdbc {
    # MySQL数据库连接
    jdbc_connection_string => "jdbc:mysql://127.0.0.1:3306/goods_db"
    jdbc_user => "root"
    jdbc_password => "123456"
    jdbc_driver_library => "/usr/local/mysql-connector-java.jar"
    jdbc_driver_class => "com.mysql.cj.jdbc.Driver"
    # 定时同步，每分钟执行一次
    schedule => "* * * * *"
    # 增量同步标识，根据更新时间筛选数据
    tracking_column => "update_time"
    use_column_value => true
  }
}
filter {
  json {
    source => "message"
    remove_field => ["message"]
  }
}
output {
  # 同步至ES指定索引
  elasticsearch {
    hosts => ["127.0.0.1:9200"]
    index => "goods_index"
    document_id => "%{id}"
  }
}
```

### 5.1.3 优缺点总结

**优点**：零代码侵入、无需修改业务代码、配置简单、支持全量+增量同步、适合数据初始化。

**缺点**：定时轮询存在延迟、频繁查询MySQL造成数据库压力、无删除同步、数据一致性差、不适合核心实时业务。

## 5.2 Canal监听binlog实时同步

### 5.2.1 方案原理

Canal是阿里开源的MySQL增量数据同步组件，模拟MySQL从节点，监听主节点**binlog日志**，实时解析增删改操作，推送增量数据至MQ，消费端消费数据同步至ES。

是目前互联网企业**生产主流、最优同步方案**。

### 5.2.2 核心流程

MySQL增删改操作→记录binlog日志→Canal实时监听解析→推送至RocketMQ/Kafka→Java消费端消费数据→调用ES API同步数据。

### 5.2.3 优缺点总结

**优点**：秒级实时同步、无数据库轮询压力、精准监听增删改、数据一致性高、解耦业务代码、支持高并发、稳定性极强。

**缺点**：架构组件多、部署维护成本略高、需要额外维护MQ和Canal服务。

## 5.3 业务代码双写同步优缺点

### 5.3.1 方案原理

在业务代码中，执行MySQL数据库增删改的同时，同步调用ES客户端API，完成ES索引数据的新增、修改、删除。

### 5.3.2 实操代码示例

```java
/**
 * 业务双写同步示例
 * 新增商品：同时写入MySQL和ES
 */
@Transactional(rollbackFor = Exception.class)
public void addGoods(Goods goods) {
    // 1. 写入MySQL数据库
    goodsMapper.insert(goods);
    // 2. 同步写入ES索引
    IndexRequest request = new IndexRequest("goods_index");
    request.id(goods.getId().toString());
    request.source(JSON.toJSONString(goods), XContentType.JSON);
    restHighLevelClient.index(request, RequestOptions.DEFAULT);
}
```

### 5.3.3 优缺点总结

**优点**：实时性最高、架构最简单、无中间组件依赖、开发快速。

**缺点**：

- 代码侵入性极强，业务代码耦合ES操作；

- 存在**数据不一致风险**：MySQL成功、ES超时失败，导致数据不一致；

- 事务无法保证双写一致性，无可靠兜底机制；

- 高并发场景拖慢业务接口响应速度。

### 5.3.4 适用场景

小型项目、快速迭代项目、低并发非核心业务，**中大型生产项目禁止单独使用**，需配合重试机制、日志兜底使用。
