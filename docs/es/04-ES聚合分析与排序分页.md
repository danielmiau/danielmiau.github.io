# 1. ES聚合查询基础

ES聚合（Aggregation）是专门用于**大数据统计分析、分组计算、数据报表生成**的核心能力，对标MySQL的group by、sum、count、max、min、avg等聚合函数。相比于数据库，ES聚合支持海量数据实时统计、毫秒级响应，是大数据量报表统计的最优方案。ES所有聚合能力可划分为**桶聚合（Bucket）**与**指标聚合（Metric）**两大类，所有复杂聚合场景均基于这两类聚合嵌套实现。

## 1.1 聚合查询两大分类

ES官方将聚合查询严格分为桶聚合、指标聚合两类，二者分工明确、配合使用：**桶聚合负责分组、指标聚合负责计算**，是所有聚合场景的底层核心逻辑。

### 1.1.1 桶聚合Bucket分组聚合

**1. 概念定义**

桶聚合（Bucket Aggregation）核心作用是**对数据进行分组**，类似于MySQL的 `group by`。ES会根据指定字段、规则，将符合条件的文档划分到不同的“桶”中，每一个桶代表一个分组，桶内存储当前分组的所有文档数据。

**2. 核心特性**

- 无数值计算能力，仅负责数据分组归类；

- 支持单层分组、多层嵌套分组；

- 一个桶聚合结果可嵌套多个指标聚合完成统计；

- 支持精准分组、区间分组、时间周期分组等多种分组规则。

**3. 常见桶聚合类型**

- term：精准词条分组（适配keyword、数值字段）

- range：数值区间分组

- date_histogram：时间周期直方图分组

- terms：多词条批量分组

**4. 适用场景**

按商品状态分组、按价格区间分组、按天/月/年统计订单数据、按分类统计商品数量等所有需要**数据分组**的业务场景。

### 1.1.2 指标聚合Metric数值统计

**1. 概念定义**

指标聚合（Metric Aggregation）核心作用是**对分组后的数据集做数值统计计算**，对标MySQL的sum、avg、max、min、count函数。指标聚合不能单独实现分组，必须基于桶聚合的分组结果进行计算，也可直接对全局数据统计。

**2. 核心特性**

- 专注数值计算，无分组能力；

- 支持单值统计（sum、avg、max、min）、多值统计、去重统计；

- 可嵌套在任意桶聚合内部，实现分组后二次计算。

**3. 常见指标聚合类型**

- sum：求和统计

- avg：平均值统计

- max/min：最大值、最小值统计

- count：文档数量统计

- cardinality：去重数量统计

**4. 核心协作逻辑（面试必背）**

**Bucket分组 + Metric计算 = 完整聚合统计**：先通过桶聚合把数据分成不同组别，再通过指标聚合对每个组别做数值统计，最终实现分组统计报表。

## 1.2 简单指标聚合

简单指标聚合无需依赖桶分组，可直接对索引全局数据、或筛选后的局部数据进行数值统计，是最基础、最高频的统计语法，适配单维度数据统计场景。

### 1.2.1 求和、平均值、最大值、最小值

**1. 语法说明**

sum、avg、max、min为基础单值指标聚合，分别实现数值字段的总和、平均值、最大值、最小值统计，仅支持**数值类型字段（integer、long、double、float）**，text/keyword文本字段无法使用。

**2. 业务场景**

统计商品总销售额、商品均价、最高价格、最低价格、订单总金额等。

**3. 完整实操示例**

需求：统计所有上架商品的总销售额、均价、最高价、最低价

```json
# 基础数值指标聚合统计
GET /goods_index/_search
{
  # 只做统计，不需要返回原始文档，提升性能
  "size": 0,
  "query": {
    # 过滤统计范围：仅上架商品
    "term": {
      "status": "上架"
    }
  },
  "aggs": {
    # 自定义聚合名称：总销售额
    "total_sales_price": {
      "sum": {
        "field": "price"
      }
    },
    # 商品均价
    "avg_price": {
      "avg": {
        "field": "price"
      }
    },
    # 最高价格
    "max_price": {
      "max": {
        "field": "price"
      }
    },
    # 最低价格
    "min_price": {
      "min": {
        "field": "price"
      }
    }
  }
}
```

**4. 关键参数说明**

- **size:0**：核心优化参数，关闭原始文档返回，仅返回聚合统计结果，大幅提升聚合查询性能；

- aggs：聚合查询根节点，所有统计逻辑均写在该节点内；

- 自定义聚合名：可随意命名，用于接收对应统计结果。

**5. 生产避坑**

统计字段必须为数值类型，若字段为text分词类型，会直接报错；若存在空值字段，ES会自动忽略空值数据，不参与统计。

### 1.2.2 去重数量统计cardinality

**1. 概念定义**

cardinality是ES专属**去重计数聚合**，对标MySQL的 `count(distinct 字段)`，用于统计字段不重复的唯一值数量。

**2. 业务场景**

统计店铺商品种类数、下单用户数、商品标签数量、独立访问用户数等去重统计场景。

**3. 实操示例**

需求：统计当前上架商品的独立分类数量

```json
# cardinality 去重统计
GET /goods_index/_search
{
  "size": 0,
  "query": {
    "term": {
      "status": "上架"
    }
  },
  "aggs": {
    # 统计不重复的分类数量
    "distinct_category_count": {
      "cardinality": {
        "field": "category_id",
        # 精度阈值，数值越大越精准，默认100
        "precision_threshold": 100
      }
    }
  }
}
```

**4. 核心精度说明（面试高频）**

cardinality基于**HyperLogLog**算法实现，属于**近似去重统计**，非绝对精准；通过precision_threshold控制精度，阈值范围内100%精准，超出阈值存在微小误差，适配海量数据近似统计场景。

**5. 生产最佳实践**

精准去重场景（数据量小）可直接使用；海量数据大屏统计可接受微小误差，优先使用该语法，性能远高于精准去重。

### 1.2.3 文档总数统计

**1. 概念定义**

通过count聚合实现符合条件的**文档总数量统计**，对标MySQL的count(*)，统计过滤条件后的总数据量。

**2. 两种实现方式**

方式一：直接读取查询结果的hits.total.value，无需额外聚合；方式二：通过value_count聚合实现字段有效数据统计。

**3. 实操示例：全局文档数统计**

```json
# 统计上架商品总数量
GET /goods_index/_search
{
  "size": 0,
  "query": {
    "term": {
      "status": "上架"
    }
  }
}
```

直接通过返回结果中 `hits.total.value` 获取总数。

**4. 实操示例：有效字段文档统计**

需求：统计有价格数据的商品数量（排除价格为空的脏数据）

```json
# value_count 字段有效数据统计
GET /goods_index/_search
{
  "size": 0,
  "aggs": {
    "price_valid_count": {
      "value_count": {
        "field": "price"
      }
    }
  }
}
```

**5. 核心区别**

hits.total：统计所有命中文档总数；value_count：统计**字段非空**的文档数量，可用于清洗脏数据统计。

# 2. 分组桶聚合实战

桶聚合是ES聚合分析的核心难点与重点，生产90%的报表统计、分组统计场景均依赖桶聚合+指标聚合嵌套实现。本节详解三大核心分组场景、多层嵌套聚合、聚合结果排序过滤，全部配套可落地实操案例与业务场景。

## 2.1 常用分组聚合

### 2.1.1 term字段精准分组

**1. 概念定义**

term分组聚合是最常用的精准分组语法，根据**词条精准匹配**进行数据分组，等价于MySQL `group by 字段`，仅适配keyword、数值、日期等结构化字段，**不支持text分词字段**。

**2. 业务场景**

按商品状态分组统计、按分类分组统计、按用户性别分组统计等固定维度精准分组场景。

**3. 完整实操示例**

需求：根据商品分类分组，统计每个分类的商品数量、平均价格、总销售额

```json
# term精准分组 + 指标聚合嵌套
GET /goods_index/_search
{
  "size": 0,
  "aggs": {
    # 第一层：桶聚合 - 按分类精准分组
    "category_group": {
      "terms": {
        "field": "category_id",
        "size": 100 # 展示最多100个分组结果
      },
      # 第二层：指标聚合 - 对每个分组统计
      "aggs": {
        # 分组内商品数量
        "goods_count": {
          "count": {
            "field": "id"
          }
        },
        # 分组内平均价格
        "group_avg_price": {
          "avg": {
            "field": "price"
          }
        },
        # 分组内总销售额
        "group_total_price": {
          "sum": {
            "field": "price"
          }
        }
      }
    }
  }
}
```

**4. 生产避坑（高频Bug）**

**禁止对text字段进行term分组**！text字段会被分词拆分，分组结果会出现乱码、单字分组，完全不符合业务预期；文本分组必须使用对应keyword子字段。

### 2.1.2 range区间范围分组

**1. 概念定义**

range区间分组聚合，支持自定义**数值区间范围**进行数据分组，适配无法固定分组值的场景，可自由划定区间维度实现统计。

**2. 业务场景**

商品价格区间统计（0-100、100-500、500+）、销量区间分组、年龄区间分组、分数区间分组。

**3. 实操示例**

需求：按价格区间分组，统计不同价格区间的商品数量与均价

```json
# range区间分组聚合
GET /goods_index/_search
{
  "size": 0,
  "aggs": {
    "price_range_group": {
      "range": {
        "field": "price",
        # 自定义区间规则
        "ranges": [
          {"to": 100},        # 0 ~ 100
          {"from": 100, "to": 500}, # 100 ~ 500
          {"from": 500}       # 500以上
        ]
      },
      # 分组内二次统计
      "aggs": {
        "range_count": {"value_count": {"field": "id"}},
        "range_avg_price": {"avg": {"field": "price"}}
      }
    }
  }
}
```

**4. 参数说明**

- to：小于该数值；

- from：大于等于该数值；

- 区间左闭右开，符合常规业务认知。

### 2.1.3 date_histogram时间日期直方图

**1. 概念定义**

date_histogram是ES专属**时间维度分组聚合**，支持按天、周、月、季度、年等固定时间周期自动分组，是时间维度报表统计的核心语法。

**2. 核心时间间隔参数**

minute（分钟）、hour（小时）、day（天）、week（周）、month（月）、year（年）

**3. 业务场景**

每日订单量统计、每月销售额统计、年度数据报表、小时级流量统计。

**4. 实操示例**

需求：按天统计近30天每日上架商品数量

```json
# 时间日期直方图分组统计
GET /goods_index/_search
{
  "size": 0,
  "query": {
    "range": {
      "create_time": {
        "gte": "now-30d"
      }
    }
  },
  "aggs": {
    "day_time_group": {
      "date_histogram": {
        "field": "create_time",
        "calendar_interval": "day", // 按天分组
        "format": "yyyy-MM-dd",     // 时间格式化
        "min_doc_count": 0          // 无数据也展示分组，补零
      },
      "aggs": {
        "daily_goods_count": {
          "value_count": {
            "field": "id"
          }
        }
      }
    }
  }
}
```

**5. 生产核心优化参数**

**min_doc_count:0**：默认无数据的时间分组会被隐藏，开启后自动补零，保证时间报表连续无空缺，是数据大屏必备配置。

## 2.2 多层嵌套聚合

简单单层聚合仅能满足基础统计，企业级复杂报表需要**多层嵌套聚合**：先大范围分组，再二次细分分组、二次统计，同时支持聚合结果排序过滤，适配复杂多维数据分析场景。

### 2.2.1 先分组再二次统计

**1. 嵌套聚合原理**

ES聚合支持无限层级嵌套，核心逻辑：**外层大维度分组 → 内层小维度分组/统计**，实现多维交叉统计。

**2. 业务场景**

先按商品分类分组，再统计每个分类下不同价格区间的商品数量分布。

**3. 完整实操示例**

```json
# 多层嵌套聚合：分类分组 + 价格区间二次分组
GET /goods_index/_search
{
  "size": 0,
  "aggs": {
    # 第一层：按商品分类分组
    "category_group": {
      "terms": {
        "field": "category_id"
      },
      # 第二层：每个分类下，按价格区间二次分组
      "aggs": {
        "price_range_group": {
          "range": {
            "field": "price",
            "ranges": [
              {"to": 100},
              {"from": 100, "to": 500},
              {"from": 500}
            ]
          },
          # 第三层：最终数值统计
          "aggs": {
            "final_count": {"value_count": {"field": "id"}}
          }
        }
      }
    }
  }
}
```

**4. 场景价值**

实现多维交叉统计，替代数据库复杂多表分组查询，海量数据下性能远超MySQL，适配大数据报表分析。

### 2.2.2 聚合结果排序过滤

**1. 聚合排序原理**

默认聚合结果按文档数量降序排序，ES支持自定义聚合排序规则，可根据求和、平均值、最值等统计结果进行升序/降序排列。

**2. 业务场景**

按分类分组后，根据分类总销售额倒序排序，展示销量最高的分类TOP10。

**3. 实操示例**

```json
# 聚合结果自定义排序+过滤
GET /goods_index/_search
{
  "size": 0,
  "aggs": {
    "category_group": {
      "terms": {
        "field": "category_id",
        "size": 10, # 只展示TOP10分组
        # 自定义排序：根据总销售额降序
        "order": {
          "group_total_price": "desc"
        }
      },
      "aggs": {
        "group_total_price": {"sum": {"field": "price"}},
        "group_avg_price": {"avg": {"field": "price"}}
      }
    }
  }
}
```

**4. 聚合结果过滤**

通过桶过滤参数，可筛选满足条件的分组，例如只展示商品数量大于10的分类分组，过滤无效小数据分组。

# 3. 结果排序与高亮显示

ES默认根据检索相关性分数进行降序排序，仅能满足基础搜索需求。真实业务中，需要结合业务字段实现多维度组合排序，同时对搜索关键词进行高亮标红展示，提升用户搜索体验。本节详解排序优先级规则、多字段组合排序、多字段高亮配置，完全对标电商、内容平台搜索落地场景。

## 3.1 多字段组合排序

ES支持**相关性打分 + 自定义业务字段**混合排序，支持单字段、多字段叠加升降序规则，排序优先级可自定义，完美解决「搜索相关性优先、热门数据置顶、价格时间辅助排序」的业务需求。

### 3.1.1 普通字段升降序排序

**1. 概念定义**

普通字段排序指基于**数值、日期、关键词**等结构化字段进行升序（asc）或降序（desc）排序，完全忽略检索打分，仅按照业务字段规则对结果重排。支持单个字段排序与多个字段优先级叠加排序。

**2. 排序优先级规则**

多字段排序遵循**前置优先、依次兜底**原则：优先按照第一个字段排序，第一个字段值相同时，再按照第二个字段排序，以此类推。

**3. 适用场景**

电商按销量降序、价格升序、最新上架时间排序；榜单数据固定排序、无检索打分的列表页排序场景。

**4. 完整实操示例**

需求：商品搜索结果优先按销量降序，销量一致时按价格升序，价格一致时按创建时间降序

```json
# 多字段组合升降序排序
GET /goods_index/_search
{
  "query": {
    "match": {
      "goods_name": "Java教程"
    }
  },
  "sort": [
    {"sales": "desc"},        // 第一优先级：销量降序
    {"price": "asc"},         // 第二优先级：价格升序
    {"create_time": "desc"}   // 第三优先级：最新创建时间优先
  ]
}
```

**5. 生产避坑指南**

- text分词字段**禁止直接排序**，会直接报错，排序字段必须为keyword、数值、日期类型；

- 排序字段建议建立索引，海量数据下无索引排序会触发全量排序，性能极差；

- 空值字段默认会排在最前面，可通过配置参数调整空值排序位置。

### 3.1.2 检索打分优先级排序

**1. 概念定义**

检索打分排序是**搜索业务专属排序规则**，优先以ES相关性打分（_score）为第一排序依据，保证关键词匹配度最高的结果置顶，再叠加业务字段做辅助兜底排序，兼顾搜索精准度与业务规则。

**2. 核心业务价值**

纯业务字段排序会丢失搜索相关性，导致关键词匹配度低的热门商品置顶，用户体验极差；打分优先排序能保证**搜索精准度优先，业务规则辅助优化**，是主流搜索平台的标准排序方案。

**3. 实操示例**

需求：优先按关键词匹配分数降序，分数一致时按商品销量降序

```json
# 检索打分优先 + 业务字段兜底排序
GET /goods_index/_search
{
  "query": {
    "match": {
      "goods_name": "ES聚合查询"
    }
  },
  "sort": [
    {"_score": "desc"},  // 第一优先级：相关性打分，匹配度越高越靠前
    {"sales": "desc"}    // 第二优先级：同分情况下，高销量商品优先
  ]
}
```

**4. 高频面试考点**

问：加入sort自定义排序后，ES默认打分排序是否失效？答：完全失效，sort字段优先级高于默认打分，未写入sort的字段规则不会生效，如需保留相关性，必须手动添加 `_score: desc`。

## 3.2 搜索结果高亮

搜索高亮是前端展示必备功能，用于将用户搜索的关键词在标题、简介中特殊标红展示，直观提升用户搜索体验。ES内置highlight高亮组件，支持自定义标签、多字段高亮、高亮片段截取等能力。

### 3.2.1 关键字高亮标签设置

**1. 核心原理**

ES检索完成后，会自动解析查询关键词，在目标字段中匹配关键词并包裹自定义HTML标签，前端直接解析标签即可实现标红、加粗高亮效果，**不修改原始索引数据**，仅返回临时高亮片段。

**2. 核心参数说明**

- pre_tags：自定义高亮前缀标签，默认<em>，生产常用<span style="color:red">；

- post_tags：自定义高亮后缀标签，默认</em>；

- fields：指定需要高亮的字段，仅text分词字段支持高亮。

**3. 完整实操示例**

需求：搜索商品标题，关键词标红高亮展示

```json
# 单字段关键词高亮配置
GET /goods_index/_search
{
  "query": {
    "match": {
      "goods_name": "ES聚合"
    }
  },  
  "highlight": {
    "pre_tags": ["<span style='color:red'>"],  // 自定义前置标签
    "post_tags": ["</span>"],                     // 自定义后置标签
    "fields": {
      "goods_name": {} // 指定标题字段开启高亮
    }
  }
}
```

**4. 生产避坑**

高亮仅支持**text分词字段**，keyword字段无分词逻辑，无法生成高亮片段；原始数据不会被修改，高亮数据单独存在于返回结果的highlight节点中。

### 3.2.2 多字段同时高亮配置

**1. 场景说明**

业务中常需要同时对标题、简介、详情多个字段的搜索关键词做高亮展示，ES支持批量配置多字段高亮，统一标签样式，简化前端适配逻辑。

**2. 实操示例**

需求：同时对商品标题、商品简介字段的关键词标红高亮

```json
# 多字段同时高亮
GET /goods_index/_search
{
  "query": {
    "multi_match": {
      "query": "ES分页",
      "fields": ["goods_name","goods_desc"]
    }
  },
  "highlight": {
    "pre_tags": ["<span style='color:red'>"],
    "post_tags": ["</span>"],
    "fields": {
      "goods_name": {},
      "goods_desc": {}
    }
  }
}
```

**3. 最佳实践**

多字段高亮统一使用一套标签样式，保证前端展示一致性；长文本字段可配置高亮片段长度，避免返回内容过长影响渲染。

# 4. ES分页查询解决方案

分页是列表查询必备能力，ES分页分为普通分页、滚动分页、游标分页三类。普通分页简单易用，但存在**深分页性能雪崩**问题，海量数据场景必须使用专用分页方案。本节彻底拆解三种分页原理、优缺点、适用场景，解决生产深分页超时、OOM、数据错乱问题。

## 4.1 普通from+size分页

from+size是ES最基础、最常用的分页方式，语法简单、适配常规列表分页，是前端通用分页的首选方案。

### 4.1.1 基础分页用法

**1. 参数定义**

- **from**：起始偏移量，即跳过前面多少条数据，默认0；

- **size**：每页展示数据条数，默认10。

计算公式：**from = (页码 - 1) * 每页条数**

**2. 实操示例**

需求：查询第2页数据，每页10条

```json
# from+size 基础分页查询
GET /goods_index/_search
{
  "query": {
    "match_all": {}
  },
  "from": 10,  // 跳过前10条
  "size": 10   // 每页展示10条
}
```

### 4.1.2 深分页性能缺陷

**1. 深分页定义**

当from偏移量过大（通常超过1000），即用户访问页码过大时，即为深分页场景。

**2. 底层缺陷原理（面试必问）**

ES是分布式架构，数据分散在多个分片。执行from+size分页时，**会在所有分片查询前from+size条数据，汇总后再全局排序、截取对应分页数据**。页码越深，需要查询、排序、汇总的数据量越大，内存开销、CPU开销指数级增长。

**3. 核心问题**

- 深分页会触发海量数据全局排序，性能极差；

- 极易导致集群CPU飙升、查询超时、服务OOM；

- ES官方默认限制最大查询深度为10000，超出直接报错。

**4. 适用场景限制**

仅适用于**浅分页场景（页码≤100）**，禁止用于海量数据、无限下拉、超大页码分页。

## 4.2 海量数据深度分页方案

针对from+size深分页缺陷，ES提供两种专业级海量数据分页方案：scroll滚动分页、search_after游标分页，分别适配不同业务场景，彻底解决深分页性能问题。

### 4.2.1 scroll滚动分页原理与使用

**1. 核心原理**

scroll滚动分页是**快照式分页**，首次查询时ES会生成当前索引数据的快照，记录数据排序位置，后续分页直接基于快照读取数据，无需重复全局排序，彻底解决深分页排序性能问题。

**2. 核心特性**

- 基于数据快照，分页过程中**无法感知新增、修改、删除数据**；

- 适合一次性全量数据导出、数据同步、批量处理场景；

- 不适合用户实时翻页、无限下拉场景。

**3. 实操示例**

```json
# 第一步：初始化滚动查询，获取scroll_id
# 初始化滚动分页，快照保留1分钟
GET /goods_index/_search?scroll=1m
{
  "query": {"match_all": {}},
  "size": 50
}
# 第二步：通过scroll_id持续获取下一页数据
# 滚动获取下一页
GET /_search/scroll
{
  "scroll": "1m",
  "scroll_id": "返回的scroll_id值"
}
```

**4. 生产避坑**

scroll快照会占用集群内存，使用完毕需手动清除快照资源，避免内存泄漏；快照有效期不宜设置过长，按需配置。

### 4.2.2 search_after高效分页实现

**1. 核心原理**

search_after是ES**生产最优海量分页方案**，属于游标分页思想。通过上一页最后一条数据的排序字段值作为游标，下一次查询直接从游标位置向后读取数据，**无偏移量、无全局排序**，性能极高。

**2. 核心优势**

- 无深分页性能问题，支持无限下拉分页；

- 实时感知数据新增、修改、删除，数据最新；

- 无快照内存开销，集群性能损耗极低。

**3. 强制使用条件**

必须设置**唯一有序排序字段**（如id、时间戳+id），保证排序唯一性，避免数据重复、遗漏。

**4. 实操示例**

```json
# 第一步：首次分页查询，获取第一页数据及最后一条游标值
# 首次查询
GET /goods_index/_search
{
  "query": {"match_all": {}},
  "size": 10,
  "sort": [
    {"create_time": "desc"},
    {"id": "desc"} // 唯一兜底排序，防止数据排序重复
  ]
}
# 第二步：根据上一页最后一条sort值，search_after查询下一页
# 游标分页查询下一页
GET /goods_index/_search
{
  "query": {"match_all": {}},
  "size": 10,
  "search_after": [1712345678000, 10086], // 上一页最后一条sort数组值
  "sort": [
    {"create_time": "desc"},
    {"id": "desc"}
  ]
}
```

**5. 业务适配场景**

APP无限下拉加载、首页流式列表、海量数据实时分页展示，是目前企业主流分页方案。

### 4.2.3 三种分页场景选型对比

**分页方案生产选型对照表（面试+实战必背）**

| 分页方案             | 优点                                         | 缺点                                     | 适用场景                                      |
| -------------------- | -------------------------------------------- | ---------------------------------------- | --------------------------------------------- |
| from+size普通分页    | 语法简单、支持任意页码跳转、无需游标         | 深分页性能极差、有10000条数限制          | 后台管理系统、浅分页场景、页码跳转式列表      |
| scroll滚动分页       | 批量查询性能高、无偏移量缺陷                 | 数据快照不实时、占用内存、不支持跳转页码 | 全量数据导出、数据同步、批量数据迁移          |
| search_after游标分页 | 性能极致、数据实时、支持无限分页、无内存开销 | 不支持页码跳转、必须依赖唯一排序字段     | 用户端无限下拉、APP流式列表、海量数据实时分页 |

# 5. 聚合查询业务实战

结合前文桶聚合、指标聚合核心知识点，落地两大企业高频实战场景：电商商品统计分析、日志小时级流量统计，完整复现生产报表开发全流程，打通「语法学习→业务落地」的闭环。

## 5.1 电商商品统计分析

**1. 业务需求**

统计各商品分类下的商品总数、总销售额、平均售价、价格区间分布，实现电商后台数据看板核心统计功能。

**2. 实现思路**

外层通过term按商品分类分组，内层嵌套指标聚合统计数量、销售额、均价，再嵌套range实现价格区间二次分布统计。

**3. 完整实操DSL**

```json
# 电商商品多维统计分析
GET /goods_index/_search
{
  "size": 0,
  "query": {
    "term": {
      "status": "上架" // 仅统计上架有效商品
    }
  },
  "aggs": {
    # 一级分组：按商品分类分组
    "category_group": {
      "terms": {
        "field": "category_id"
      },
      "aggs": {
        # 基础指标统计
        "goods_total": {"value_count": {"field": "id"}},
        "total_sales_price": {"sum": {"field": "price"}},
        "avg_goods_price": {"avg": {"field": "price"}},
        "max_goods_price": {"max": {"field": "price"}},
        # 二级分组：价格区间分布统计
        "price_range_group": {
          "range": {
            "field": "price",
            "ranges": [
              {"to": 100},
              {"from": 100, "to": 500},
              {"from": 500}
            ]
          }
        }
      }
    }
  }
}
```

**4. 业务落地说明**

该语句可直接用于电商数据大屏、后台统计报表、品类销售分析，替代MySQL多组分组查询，海量数据下性能提升10倍以上。

## 5.2 日志按小时流量统计

**1. 业务需求**

针对系统操作日志，统计近24小时每小时的请求流量、访问IP数、接口请求量，实现日志监控、流量峰值预警功能。

**2. 实现思路**

通过date_histogram按小时时间直方图分组，结合count统计总请求量、cardinality去重统计独立IP数。

**3. 完整实操DSL**

```json
# 日志小时级流量统计
GET /log_index/_search
{
  "size": 0,
  "query": {
    "range": {
      "log_time": {
        "gte": "now-24h"
      }
    }
  },
  "aggs": {
    # 按小时分组
    "hour_flow_group": {
      "date_histogram": {
        "field": "log_time",
        "calendar_interval": "hour",
        "format": "yyyy-MM-dd HH",
        "min_doc_count": 0
      },
      "aggs": {
        "total_request": {"value_count": {"field": "log_id"}},
        "distinct_ip_count": {"cardinality": {"field": "ip"}}
      }
    }
  }
}
```

**4. 生产价值**

可直接用于运维监控、系统流量统计、异常峰值排查，自动补零保证24小时数据连续，适配监控大屏展示需求。
