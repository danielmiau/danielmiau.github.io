# 1. ES集群完整架构

ES天然支持分布式集群架构，单机仅适用于测试环境，生产环境必须基于多节点集群部署，实现**负载均衡、故障转移、高可用、海量数据存储**。ES集群由多个节点组成，不同节点承担不同职责，同时内置自主选举、自动容错、自动分片迁移机制，是其高可用的核心基础。

## 1.1 集群节点角色划分

ES节点支持多角色混合部署，也支持专用角色独立部署。默认情况下，单个节点同时具备主节点、数据节点、协调节点能力，适配测试环境；生产环境为保证集群稳定性，通常采用**专用节点职责分离**部署模式。

### 1.1.1 主节点、数据节点、协调节点

**1. 主节点（Master Node）**

**核心概念**：主节点是ES集群的「管理者节点」，整个集群有且仅有一个活跃主节点，负责统一调度集群所有元数据操作，不参与具体数据读写，压力极小。

**核心职责**

- 集群节点管理：监听节点上下线、剔除故障节点、接纳新节点；

- 索引元数据管理：索引创建、删除、映射修改、分片分配调整；

- 集群状态同步：统一维护集群全局状态，同步至所有节点；

- 分片调度管理：故障后分片重新分配、集群扩容后分片均衡迁移。

**配置标识**：`node.master: true`，默认开启。

**2. 数据节点（Data Node）**

**核心概念**：数据节点是ES集群的「存储与计算节点」，承担所有数据的持久化存储、检索计算、聚合统计、读写IO请求，是集群压力最大的节点。

**核心职责**

- 数据存储：存储主分片、副本分片的真实文档数据；

- 读写执行：处理分片级别的增删改查、聚合排序、检索请求；

- 数据同步：主副本分片数据实时同步、副本数据兜底；

- 磁盘管理：数据落地、段合并、磁盘清理、分片迁移。

**配置标识**：`node.data: true`，默认开启。

**3. 协调节点（Coordinate Node）**

**核心概念**：协调节点又称客户端节点，是ES集群的「请求入口节点」，不存储数据、不参与主节点选举，仅负责请求转发、结果聚合、负载均衡。

**核心职责**

- 请求接收：接收客户端所有HTTP、TCP读写请求；

- 路由转发：根据分片路由规则，将请求分发至对应数据节点；

- 结果聚合：汇总多个分片的查询结果、排序去重、统一返回；

- 负载均衡：规避故障节点，分发请求至健康节点。

**配置标识**：无独立配置，所有节点默认具备协调节点能力，可手动关闭master和data角色，打造纯协调节点。

**三节点核心区别总结（面试高频）**

主节点管**集群规则与元数据**、数据节点管**数据存储与计算**、协调节点管**请求入口与结果聚合**。

### 1.1.2 专用节点职责分离部署

**1. 混合部署问题（生产避坑）**

默认混合角色节点（同时是master、data、coordinate）在小集群可用，但生产大集群存在严重风险：数据节点承担海量读写压力、CPU和磁盘负载极高，可能导致主节点选举超时、集群状态更新延迟，引发**集群脑裂、分片异常、集群不可用**等严重故障。

**2. 生产专用节点部署方案（最佳实践）**

生产环境严格采用**角色分离部署**，分为三类专用节点，互不干扰，提升集群稳定性：

- **专用主节点（3台）**：仅开启master角色，关闭data数据存储，配置高配CPU、低配磁盘，专注集群管理；

- **专用数据节点（N台）**：仅开启data角色，关闭master选举，配置高配磁盘、大内存，专注数据存储与计算；

- **专用协调节点（2台+）**：关闭master和data角色，配置高配CPU，专注请求转发与结果聚合，作为业务访问入口。

**3. 专用节点配置文件示例**

专用主节点配置（elasticsearch.yml）：

```yaml
# 专用主节点：开启master，关闭数据存储
node.master: true
node.data: false
node.ingest: false
```

专用数据节点配置：

```yaml
# 专用数据节点：关闭master，开启数据存储
node.master: false
node.data: true
node.ingest: true
```

专用协调节点配置：

```yaml
# 专用协调节点：关闭所有核心角色，仅做请求转发
node.master: false
node.data: false
node.ingest: false
```

**4. 部署价值**

彻底隔离集群管理压力、数据计算压力、请求接入压力，避免业务读写压力拖垮集群管理，保证集群7*24小时高可用。

## 1.2 集群选举机制

ES集群通过自主选举机制实现主节点自动切换、故障自动转移，无需人工干预，是集群高可用的核心底层保障。理解选举规则与最小主节点配置，是排查集群脑裂、主节点频繁切换故障的关键。

### 1.2.1 主节点选举规则

**1. 选举前置条件**

只有标记为 **master-eligible**（可参选主节点）的节点，才有资格参与集群主节点选举，普通数据节点、协调节点无选举权和被选举权。

**2. 核心选举规则**

- 集群初始化或主节点故障下线后，所有可参选节点自动发起选举；

- **票数过半原则**：获得超过半数参选节点投票的节点，成为新的活跃主节点；

- 主节点一旦选举成功，集群锁定状态，其他节点自动放弃竞选，不再抢占；

- 旧主节点恢复上线后，自动降级为普通从节点，服从新主节点管理。

**3. 选举核心特性**

ES选举机制保证集群**同一时刻只有一个主节点**，从根本上避免多主节点冲突问题，为集群一致性提供保障。

### 1.2.2 最小主节点配置

**1. 核心概念**

`discovery.zen.minimum_master_nodes` 是ES集群**防脑裂核心配置**，用于指定「形成有效集群所需的最小主节点数量」，是生产集群必配参数。

**2. 脑裂问题原理（面试必考）**

当集群网络抖动、节点心跳超时，集群会被分割为两个独立的子集群，两个子集群各自选举出主节点，出现**双主节点**状态，即为集群脑裂。脑裂会导致索引数据错乱、分片冲突、数据丢失，是ES最严重的生产故障之一。

**3. 配置计算公式（通用公式）**

**最小主节点数 = 总主节点参选数量 / 2 + 1**

示例：3个主节点候选节点，最小主节点配置为 3/2+1=2；必须至少2个节点存活，才能组成有效集群。

**4. 配置实操与避坑**

```yaml
# elasticsearch.yml 核心防脑裂配置
# 3主节点集群标准配置
discovery.zen.minimum_master_nodes: 2
# 心跳超时配置，避免网络轻微抖动触发重选
discovery.zen.ping_timeout: 3s
```

**5. 最佳实践**

- 生产集群主节点数量必须为**奇数（3、5）**，杜绝偶数节点，最大程度规避脑裂；

- 必须严格配置最小主节点数，默认配置不适合生产；

- ES7.0+版本自动适配该规则，无需手动配置，但原理必须掌握。

# 2. 主分片与副本分片机制

分片是ES分布式存储的最小单元，也是ES实现**海量数据存储、水平扩容、高可用容灾**的核心基石。所有索引数据都会被拆分为主分片和副本分片分散存储在集群节点中，彻底理解分片机制是ES架构调优、容量规划、故障排查的核心前提。

## 2.1 分片核心作用

### 2.1.1 主分片水平扩容存储

**1. 主分片概念**

主分片（Primary Shard）是索引数据的**核心读写分片**，索引创建时指定主分片数量，数据会按照路由规则均匀拆分存储在各个主分片中。

**2. 核心作用：水平扩容突破单机限制**

单台服务器磁盘、内存、IO存在上限，无法承载TB、PB级海量数据。ES通过多主分片拆分数据，将海量数据分散存储在多台数据节点，**突破单机存储上限，实现分布式水平扩容**。

**3. 读写特性**

- 所有新增、修改、删除写请求，必须先在主分片执行成功，再同步至副本分片；

- 读请求可在主分片、副本分片并行执行，分担查询压力；

- 主分片故障后，集群自动将对应副本分片提升为主分片，保证业务不中断。

### 2.1.2 副本分片高可用容灾

**1. 副本分片概念**

副本分片（Replica Shard）是主分片的**完整数据备份**，每个主分片都可以配置若干个副本分片，副本数据与主分片数据实时同步、完全一致。

**2. 核心作用：双层高可用容灾**

- **节点故障容灾**：主分片所在节点宕机、磁盘损坏，副本分片自动升级为主分片，数据不丢失、业务不中断；

- **读写负载分担**：查询请求可分流至副本分片，分担主分片查询压力，提升集群查询吞吐量；

- **数据冗余备份**：避免单节点故障导致索引数据丢失，实现数据冗余存储。

**3. 核心部署规则（硬性规则）**

**主分片与对应的副本分片绝对不会部署在同一台节点上**，规避单节点故障导致主副本同时丢失的风险。

## 2.2 分片分配规则

分片分配规则决定了索引数据的拆分逻辑、部署位置、读写路由，是ES架构设计的核心，也是面试高频重难点，包含主分片不可修改特性、副本动态调整、路由计算公式三大核心知识点。

### 2.2.1 主分片数量不可修改原因

**1. 核心结论**

**索引创建后，主分片数量绝对无法修改**，仅副本分片数量支持动态调整，这是ES的硬性机制。

**2. 底层原理（深度拆解）**

ES文档的存储分片由**哈希路由公式**固定计算，文档写入后，归属的主分片位置永久固定。如果允许修改主分片数量，所有文档的哈希路由结果会全部错乱，导致所有数据无法查询、数据彻底失效。

**3. 生产影响与解决方案**

- 影响：索引创建前必须预估好数据量，规划主分片数，创建后无法扩容主分片；

- 解决方案：主分片规划过小导致容量不足时，只能新建索引、重新迁移数据；

- 优化方案：生产优先使用**索引模板**，统一规范分片配置，避免手动创建索引失误。

### 2.2.2 副本分片动态调整

**1. 核心特性**

副本分片数量支持**动态在线修改**，无需停机、无需迁移数据，修改后集群自动完成副本分片的创建、同步、迁移，不影响业务读写。

**2. 动态调整DSL实操**

修改指定索引副本数为1：

```json
# 动态修改索引副本分片数量
PUT /goods_index/_settings
{
  "number_of_replicas": 1
}
```

**3. 生产动态调优场景**

- 集群节点扩容后，增加副本数，提升容灾能力与查询并发；

- 集群节点缩容、维护升级时，临时降低副本数，避免分片未分配报错；

- 海量数据初始化导入时，临时设置副本数为0，提升写入速度，导入完成后再恢复副本。

### 2.2.3 分片路由计算公式

**1. 核心路由公式（面试必背）**

ES根据文档_id或自定义路由字段，通过哈希算法计算归属主分片，公式如下：

**shard = hash(routing) % number_of_primary_shards**

- routing：默认是文档_id，也可以自定义指定路由字段；

- hash：哈希散列算法，保证字符均匀分布；

- number_of_primary_shards：索引主分片总数。

**2. 公式核心意义**

所有文档通过该公式均匀分布在各个主分片，保证数据均衡存储；同时解释了主分片不可修改的核心原因——主分片数变更，取模结果全部变更，数据分片归属彻底错乱。

**3. 自定义路由场景**

生产可通过自定义routing字段，将相同维度数据（如同店铺、同用户）固定存储在同一个分片，提升聚合查询、关联查询性能。

## 2.3 分片合理规划设计

分片数量规划不合理是生产ES集群性能卡顿、分片不均衡、资源浪费、故障频发的核心原因。本节讲解生产分片容量上限与最佳实践，可直接用于线上集群架构规划。

### 2.3.1 单分片存储数据上限

**1. 官方与生产通用阈值**

**单个分片数据量建议控制在 10GB ~ 30GB 之间**，绝对禁止单个分片超过50GB。

**2. 分片过大的严重危害**

- 分片过大导致分片迁移、故障恢复、副本同步速度极慢；

- 段合并耗时激增，CPU、磁盘IO持续打满；

- 检索、聚合查询性能大幅下降，超时率飙升；

- 节点故障后，分片恢复时间过长，集群长时间处于不健康状态。

**3. 分片过小问题**

分片数量过多、单分片数据过小，会产生大量空分片、小分片，占用集群元数据内存，导致集群状态臃肿、查询路由效率降低，浪费集群资源。

### 2.3.2 索引分片数最佳实践

**1. 分片数量计算公式（生产落地）**

预估单索引总数据量 ÷ 20GB = 推荐主分片数量，向上取整。

示例：预估索引数据量100GB，100/20=5，推荐设置5个主分片。

**2. 副本分片通用规范**

- 测试环境：副本数0，节省资源；

- 生产小型集群：副本数1，兼顾容灾与资源；

- 核心重要数据：副本数2，实现多重容灾备份。

**3. 终极最佳实践总结**

- 主分片数提前预估、宁多勿少，创建后不可修改；

- 单分片严格控制在10-30GB，杜绝超大分片；

- 副本数动态可调，根据集群节点数、业务重要度灵活配置；

- 海量日志、时序数据优先使用**滚动索引**，按天拆分索引，避免单索引数据过大。

# 3. 集群脑裂问题

集群脑裂是ES生产环境中**最危险、破坏性最强**的集群故障之一，一旦发生会导致集群双主运行、数据错乱、索引损坏、读写异常，且故障隐蔽、极难排查。所有生产ES集群必须提前做防脑裂配置，本节完整拆解脑裂成因、危害、根治方案与生产优化手段。

## 3.1 脑裂产生原因

ES集群默认依靠节点间心跳通信维持集群统一状态，当集群网络不稳定、节点心跳超时，完整集群会被分裂为两个独立的子集群，各自选举主节点，形成**双主节点**的异常状态，该现象即为集群脑裂。

### 3.1.1 网络延迟引发集群分裂

**1. 底层原理**

ES集群节点之间默认每1秒发送一次心跳探测，默认超时时间为3秒。当集群出现**网络抖动、网卡卡顿、机房网络延迟、防火墙限流**等问题时，部分节点无法收到其他节点的心跳响应，会误判对方节点宕机。

此时原本完整的集群会被分割为两个互不通信的子集群，两个子集群各自独立运行，不再同步状态数据，为脑裂发生提供前置条件。

**2. 生产常见触发场景**

- 服务器网卡流量打满、网络丢包、延迟飙升；

- 虚拟机资源抢占、宿主机卡顿导致节点响应超时；

- 机房割接、网络策略变更、防火墙拦截心跳端口；

- 跨机房部署集群，公网/专线网络不稳定。

**3. 故障特征**

集群无明显宕机节点，但集群状态间歇性异常、分片莫名失衡、读写时好时坏，日志出现大量集群状态同步失败异常。

### 3.1.2 双主节点集群异常

**1. 双主形成过程**

网络分裂形成两个子集群后，若未配置合理的最小主节点投票数，两个子集群均满足选举条件，会各自选举出一个新的主节点。此时整个物理集群存在**两个活跃Master主节点**，正式触发脑裂故障。

**2. 脑裂核心危害（生产致命问题）**

- **数据错乱丢失**：两个主节点各自维护索引元数据，分别接收读写请求，导致分片数据不一致、索引结构混乱，最终数据永久丢失；

- **集群彻底失控**：双主节点互相覆盖集群配置，分片频繁重新分配、迁移，集群持续处于yellow/red状态；

- **业务大面积异常**：客户端请求随机路由至两个子集群，部分请求成功、部分失败，业务数据不一致；

- **故障不可逆**：脑裂持续时间越长，数据错乱越严重，多数情况下只能重建索引、迁移数据。

**3. 核心诱因总结**

脑裂本质两大原因：**网络不稳定导致集群分裂 + 最小主节点配置不合理导致双主选举成功**。

## 3.2 脑裂解决方案

脑裂问题**无法事后修复，只能事前预防**，生产环境通过「核心参数配置+网络环境优化」双重方案彻底根治脑裂故障，是集群上线的必备配置。

### 3.2.1 最小主节点参数优化

**1. 核心原理**

通过配置 **discovery.zen.minimum_master_nodes** 最小主节点数，限制集群有效成立条件：只有存活主候选节点数量大于等于该值时，才能选举主节点、形成有效集群。从根源杜绝分裂后的子集群单独选举主节点。

**2. 标准计算公式（生产必背）**

**最小主节点数 = 总主候选节点数 / 2 + 1**

该公式保证：集群分裂后，两个子集群的节点数均不满足投票条件，无法选举主节点，集群处于无主冻结状态，避免双主冲突。

**3. 生产标准配置示例**

生产集群统一采用**3个主候选节点**，计算公式：3/2+1=2，配置如下：

```yaml
# elasticsearch.yml 防脑裂核心配置
# 最小主节点投票数，3主节点标准配置
discovery.zen.minimum_master_nodes: 2
# 延长心跳超时，规避临时网络抖动
discovery.zen.ping_timeout: 5s
# 重试次数优化
discovery.zen.ping_retries: 3
```

**4. ES7+版本说明**

ES7.0及以上版本**自动管理最小主节点配置**，无需手动配置，但底层防脑裂原理完全一致，面试与故障排查必须掌握。

**5. 最佳实践**

- 主候选节点必须部署**奇数节点（3/5）**，严禁偶数节点，彻底规避投票平局；

- 所有生产集群必须开启该配置，无例外；

- 集群扩容缩容调整主节点数量后，必须同步更新该参数。

### 3.2.2 集群网络环境优化

**1. 网络问题优化方案**

参数配置只能规避脑裂结果，无法解决网络分裂根源，生产需同步优化网络环境：

- **集群内网隔离**：ES集群节点仅通过内网通信，禁止跨公网部署，降低网络延迟与丢包率；

- 服务器资源优化：保证ES节点服务器CPU、内存、磁盘IO充足，避免服务器卡顿导致心跳超时；

- 网络阈值调优：合理调大心跳超时时间，规避瞬时网络抖动误判节点下线；

- 关闭不必要防火墙拦截：集群内网端口9300（集群通信端口）永久放行。

**2. 脑裂故障应急排查步骤**

1. 查看集群节点状态，确认节点是否全部在线、是否存在节点失联；

2. 查看集群日志，检索master节点切换、集群分裂相关日志；

3. 检查服务器网络延迟、丢包率、网卡流量；

4. 重启异常子集群节点，统一集群状态，重新选举唯一主节点；

5. 核对最小主节点配置是否规范，修复配置漏洞。

# 4. 集群读写负载均衡

ES集群通过合理的读写路由分发机制，实现请求负载均衡，避免单节点压力过大。掌握读写分片路由规则、主副分片负载分担策略，是优化ES集群吞吐量、提升并发能力的核心，也是生产调优高频考点。

## 4.1 写入请求分片路由

**1. 写入路由核心流程**

所有写入请求（增/删/改）必须固定路由至**主分片**执行，副本分片不参与写入操作，具体流程如下：

1. 客户端请求发送至协调节点；

2. 协调节点通过 `hash(routing) % 主分片数` 计算文档归属主分片；

3. 将请求转发至对应主分片所在数据节点；

4. 主分片执行写入操作，写入本地磁盘；

5. 主分片同步数据至所有副本分片，同步成功后返回客户端写入成功。

**2. 写入强制主分片执行的原因**

保证数据一致性！若写入分散至主副分片，会导致多副本数据不一致、版本冲突，ES通过**主分片单点写入、副本同步**机制保证数据强一致性。

**3. 自定义路由写入场景**

生产可指定routing参数，将同一业务维度数据（如同一店铺、同一用户）写入同一分片，减少跨分片聚合查询开销，大幅提升统计性能。

## 4.2 查询请求主副分片负载分担

**1. 查询负载均衡机制**

查询请求与写入不同，**主分片、副本分片均可处理读请求**，ES默认开启读请求负载均衡，自动将查询请求分散至主、副分片执行，分担单节点查询压力，提升集群查询并发吞吐量。

**2. 核心执行逻辑**

1. 协调节点接收查询请求；

2. 根据分片分布情况，随机选择主分片或任意一个健康副本分片处理请求；

3. 汇总所有分片查询结果，排序聚合后返回客户端。

**3. 生产价值**

副本分片不仅用于容灾备份，还可分担读压力，集群副本数越多，读并发承载能力越强，非常适合读多写少的业务场景（搜索、报表查询）。

**4. 一致性避坑**

主副本数据同步存在极短延迟，瞬时查询可能出现数据短暂不一致，若业务需要**强一致性查询**，可强制查询主分片数据。

## 4.3 集群读写性能优化策略

结合读写路由机制，总结生产可直接落地的集群性能优化方案，适配高并发读写场景。

**1. 写入性能优化**

- 海量数据初始化阶段，临时将副本数设置为0，关闭副本同步，大幅提升写入速度，初始化完成后恢复副本；

- 批量写入采用Bulk批量API，减少网络IO次数；

- 合理控制单分片数据量，避免超大分片导致写入卡顿。

**2. 查询性能优化**

- 适当增加副本分片数量，提升读负载分担能力；

- 热点数据索引拆分分片，避免单分片查询压力集中；

- 专用协调节点承接所有查询请求，分担数据节点压力。

**3. 读写分离架构落地**

生产高并发场景可实现物理读写分离：写入请求固定路由主节点集群，查询请求分流至副本节点，彻底隔离读写压力。

# 5. 集群扩容与数据迁移

ES集群支持**不停机横向扩容**，无需停机、不影响业务，是分布式架构的核心优势。本节详解生产标准扩容流程、分片均衡原理、扩容安全规范，解决集群容量不足、性能瓶颈问题。

## 5.1 横向新增节点流程

**1. 扩容前置准备**

新节点服务器配置、系统环境、JDK版本、ES版本、配置文件与旧集群保持一致，保证节点兼容性。

**2. 标准扩容步骤（生产实操）**

1. 修改新节点elasticsearch.yml配置，配置集群名称、集群通信地址，保证与原集群一致；

2. 根据节点角色配置master、data权限，匹配集群角色分离架构；

3. 启动新ES节点，节点自动探测集群，自动加入原有集群；

4. 主节点感知新节点上线，触发分片自动均衡迁移；

5. 等待集群分片迁移完成，集群状态变为green，扩容结束。

## 5.2 分片自动重新均衡原理

**1. 均衡触发时机**

集群新增节点、下线节点、修改副本数后，主节点自动检测集群分片分布不均，触发**分片重均衡机制**。

**2. 核心均衡逻辑**

- 主节点统计所有数据节点的分片数量、磁盘使用率；

- 自动将密集节点的分片迁移至空闲新节点；

- 严格遵循主副本分片不同节点部署规则；

- 迁移过程增量同步数据，不影响业务正常读写。

**3. 均衡核心价值**

自动抹平节点数据压力，避免单节点分片过多、磁盘爆满、性能瓶颈，保证集群整体负载均匀。

## 5.3 生产环境安全扩容注意事项

**1. 扩容避坑核心规范**

- **禁止高峰期扩容**：分片迁移会占用CPU、磁盘IO、网络带宽，高峰期扩容会导致业务响应变慢；

- 单批次扩容节点不宜过多：逐台上线，避免一次性大量分片迁移引发集群压力雪崩；

- 扩容期间禁止重启节点：防止分片迁移中断导致集群异常；

- 提前预留磁盘空间：新节点磁盘容量需高于集群平均使用量，避免均衡后磁盘告警。

**2. 扩容后验证标准**

集群状态为green、所有分片分配正常、节点分片数量均匀、无未分配分片、业务读写正常。

# 6. ES高可用容灾架构

高可用容灾是生产集群稳定运行的终极保障，ES通过节点故障自动切换、数据备份恢复、合理架构部署，实现节点级故障、数据丢失故障的容灾兜底，本节讲解企业级容灾方案与最佳部署架构。

## 6.1 单机故障自动切换

**1. 故障感知机制**

集群通过心跳机制实时监听节点状态，当某数据节点宕机、进程退出、网络中断，主节点会在超时后标记该节点为故障下线状态。

**2. 自动容灾切换流程**

1. 故障节点下线，其上所有主分片、副本分片全部丢失；

2. 集群检测到主分片丢失，立即将对应健康的**副本分片提升为主分片**，恢复读写服务；

3. 集群自动在其他健康节点生成新的副本分片，补齐副本冗余；

4. 分片重新均衡，集群恢复green健康状态，全程无需人工干预。

**3. 容灾核心能力**

只要存在至少一份完整副本数据，集群即可实现**零停机、零数据丢失**的单机故障自动恢复。

## 6.2 索引数据备份与恢复

分片容灾仅能解决节点故障，无法解决**误删索引、误改数据、磁盘损坏、集群整体故障**问题，生产必须配置定时快照备份，实现数据终极兜底。

**1. 快照备份核心原理**

ES snapshot快照机制，基于增量备份原理，首次全量备份，后续仅备份增量数据，备份速度快、占用空间小，支持单索引、多索引、全集群备份恢复。

**2. 核心实操命令**

注册备份仓库：

```json
# 注册本地备份仓库
PUT /_snapshot/es_backup
{
  "type": "fs",
  "settings": {
    "location": "/data/es/backup",
    "compress": true
  }
}
```

创建索引快照备份：

```json
# 对商品索引创建快照备份
PUT /_snapshot/es_backup/goods_backup_2026
{
  "indices": "goods_index",
  "ignore_unavailable": true,
  "include_global_state": false
}
```

快照恢复数据：

```json
# 从快照恢复索引数据
POST /_snapshot/es_backup/goods_backup_2026/_restore
{
  "indices": "goods_index",
  "rename_pattern": "goods_index",
  "rename_replacement": "goods_index_restore"
}
```

**3. 生产最佳实践**

- 核心业务索引每日定时增量备份；

- 备份文件与集群数据磁盘物理隔离；

- 定期测试恢复流程，保证备份文件可用；

- 保留近7天备份快照，实现数据误操作兜底恢复。

## 6.3 生产集群部署最佳架构

结合前文所有知识点，总结企业生产**高可用、高并发、高稳定**的标准ES集群部署架构，为线上落地提供标准模板。

**1. 角色分离部署架构**

- **3台专用主节点**：仅负责集群管理，无数据存储，保证集群管控稳定，防脑裂；

- **N台专用数据节点**：负责数据存储、读写计算，按需横向扩容；

- **2台以上专用协调节点**：承接所有业务请求，负载均衡，保护数据节点。

**2. 分片配置规范**

- 单分片数据量严格控制在10-30GB；

- 核心索引副本数≥1，重要业务副本数=2；

- 日志时序数据采用按天滚动索引，避免单索引过大。

**3. 容灾兜底规范**

- 开启防脑裂核心配置，奇数主节点部署；

- 定时快照备份+异地备份；

- 集群监控告警：节点状态、分片状态、磁盘使用率、JVM内存、查询超时告警。

**4. 架构优势**

彻底实现**节点故障自动容灾、数据丢失可恢复、集群压力均衡、不停机扩容、高并发承载**，满足企业生产7*24小时稳定运行要求。
