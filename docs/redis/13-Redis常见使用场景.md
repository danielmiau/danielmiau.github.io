# 二、Redis核心常用场景详解（重点）

本节按“高频程度+业务优先级”排序，拆解Redis最常用的8个场景，每个场景均衔接前序章节知识点，给出完整的实现方案与实操细节，同时标注面试高频考点，兼顾实战与备考。

## 2.1 缓存场景（最核心、最常用）

核心背景：在高并发业务中，数据库（如MySQL）的查询性能有限，频繁查询相同数据会导致数据库压力过大、接口响应缓慢。Redis作为高性能内存数据库，可将高频访问的数据缓存到内存中，减少数据库查询次数，提升接口响应速度，这是Redis最核心的使用场景，适配所有高并发业务（如电商、社交、资讯）。

### 2.1.1 核心实现方案（衔接第六章数据结构）

缓存场景主要使用String、Hash两种数据结构，根据缓存数据的类型选择：

- 单值缓存（如用户信息、商品详情、配置参数）：使用String数据结构，key为唯一标识（如user:id:1001），value为序列化后的字符串（JSON/XML格式）。

- 多字段缓存（如用户信息包含姓名、年龄、手机号等）：使用Hash数据结构，key为唯一标识，field为字段名，value为字段值，无需序列化整个对象，便于单独修改某个字段。

缓存核心流程（闭环）：

1. 客户端请求数据时，先查询Redis缓存，若缓存存在（缓存命中），直接返回数据，无需访问数据库。

2. 若缓存不存在（缓存未命中），客户端访问数据库，查询到数据后，将数据存入Redis缓存，设置合理的过期时间，便于后续复用。

3. 当数据库中的数据发生修改（如更新、删除）时，同步更新或删除Redis缓存，避免缓存与数据库数据不一致（缓存脏数据）。

### 2.1.2 实操命令（实战重点）

```bash
# 1. 单值缓存（String）：存储用户信息（JSON格式）
SET user:id:1001 '{"id":1001,"name":"张三","age":25,"phone":"13800138000"}' EX 3600  # 过期时间1小时

# 2. 多字段缓存（Hash）：存储用户信息
HSET user:id:1002 name "李四" age 28 phone "13900139000"
HGETALL user:id:1002  # 查询所有字段
HGET user:id:1002 name  # 单独查询某个字段
HSET user:id:1002 age 29  # 单独更新某个字段

# 3. 缓存更新/删除
SET user:id:1001 '{"id":1001,"name":"张三","age":26,"phone":"13800138000"}' EX 3600  # 更新缓存
DEL user:id:1001  # 删除缓存

# 4. 缓存命中查询（查看缓存是否存在）
EXISTS user:id:1001  # 存在返回1，不存在返回0
```

### 2.1.3 注意事项（生产避坑重点）

- 缓存过期时间：必须设置合理的过期时间（根据业务数据更新频率调整，如高频更新数据设10-60分钟，低频更新设1-24小时），避免缓存长期占用内存，同时减少缓存脏数据概率（衔接第九章内存管理）。

- 缓存一致性：数据更新时，需同步更新/删除缓存，推荐“先更新数据库，再删除缓存”（避免并发场景下的缓存脏数据），复杂场景可引入消息队列保证一致性。

- 缓存穿透：避免查询不存在的数据（如查询id=-1的用户），导致缓存未命中、频繁访问数据库，解决方案：缓存空值（如SET user:id:-1 "" EX 60）、布隆过滤器拦截。

- 缓存击穿：热点数据（如爆款商品）过期时，大量请求同时访问数据库，解决方案：热点数据设置永不过期、加互斥锁、使用Redis集群分担压力（衔接第八章集群）。

- 缓存雪崩：大量缓存同时过期，导致所有请求访问数据库，解决方案：缓存过期时间加随机值（如EX 3600±60）、分批次设置过期时间、开启Redis持久化（衔接第九章）。

### 2.1.4 面试考点

- 考点1：Redis缓存的核心优势是什么？
答：1. 高性能：内存操作，响应时间毫秒级；2. 高并发：支持每秒数万次请求，可通过集群进一步提升并发能力；3. 减轻数据库压力：减少数据库查询次数，保护数据库稳定性。

- 考点2：缓存穿透、击穿、雪崩的区别及解决方案？
答：（区别）穿透是查询不存在的数据，击穿是热点数据过期，雪崩是大量缓存同时过期；（解决方案）见本节注意事项，重点掌握布隆过滤器、互斥锁、过期时间随机化、持久化配置。

- 考点3：缓存与数据库一致性如何保证？
答：核心是“数据更新时同步维护缓存”，推荐方案：先更新数据库，再删除缓存；复杂场景可引入消息队列，确保数据库更新成功后，再删除缓存，避免并发脏数据。

## 2.2 分布式锁场景（高并发必用）

核心背景：分布式系统中，多台服务器（多进程/多线程）同时操作同一资源（如秒杀商品库存、订单创建），会出现并发安全问题（如超卖、重复下单）。Redis可实现分布式锁，保证同一时间只有一个进程/线程操作资源，解决分布式并发安全问题，适配秒杀、订单、库存等场景。

### 2.2.1 核心实现方案（衔接第六章String数据结构）

分布式锁的核心原理：利用Redis的SETNX命令（SET if Not Exists），当key不存在时，设置key-value，获取锁；当key存在时，获取锁失败，等待或放弃。同时需设置过期时间，避免锁未释放导致死锁。

核心优化：Redis6.0+支持SET命令的扩展参数，可一次性实现“SETNX+过期时间”，避免原子性问题（旧版本需用SETNX+EXPIRE组合，存在原子性漏洞）。

### 2.2.2 实操命令（实战重点，Redis6.0+）

```bash
# 1. 获取分布式锁（key为锁标识，value为唯一标识，避免误释放他人的锁）
# NX：key不存在时才设置，PX：过期时间（毫秒），这里设30000毫秒（30秒）
SET lock:seckill:1001 1001_redis PX 30000 NX

# 2. 释放锁（需判断value是否为自己的唯一标识，避免误释放，推荐用Lua脚本保证原子性）
EVAL "if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('del', KEYS[1]) else return 0 end" 1 lock:seckill:1001 1001_redis

# 3. 锁续约（若业务执行时间超过锁过期时间，需续约，避免锁提前释放）
# 定时执行（如每10秒），判断锁是否存在，存在则重新设置过期时间
EVAL "if redis.call('get', KEYS[1]) == ARGV[1] then return redis.call('pexpire', KEYS[1], 30000) else return 0 end" 1 lock:seckill:1001 1001_redis
```

补充：生产环境中，可使用Redis客户端（如Java的Redisson）封装分布式锁，自动实现锁获取、释放、续约，简化开发，同时支持公平锁、可重入锁等高级特性。

### 2.2.3 注意事项（生产避坑重点）

- 锁的唯一标识：value必须是唯一的（如服务器IP+进程ID），避免误释放其他进程的锁（如进程A获取锁后崩溃，进程B获取锁，进程A恢复后误释放进程B的锁）。

- 过期时间设置：过期时间需大于业务执行时间（如业务执行需10秒，过期时间设30秒），同时通过锁续约解决业务执行时间过长的问题。

- 原子性保证：获取锁需用“SET NX PX”一次性完成，释放锁需用Lua脚本（判断value+删除key），避免原子性漏洞。

- 高可用保障：分布式锁场景需使用Redis集群（主从+哨兵），避免主节点宕机后锁丢失，确保锁的可用性（衔接第八章集群）。

### 2.2.4 面试考点

- 考点1：Redis实现分布式锁的核心原理是什么？
答：利用SETNX命令的原子性，key不存在时设置key-value（获取锁），key存在时获取失败；结合过期时间避免死锁，通过唯一value避免误释放锁。

- 考点2：Redis6.0之前实现分布式锁的漏洞是什么？如何解决？
答：漏洞：旧版本用SETNX+EXPIRE组合，两步操作非原子性，若SETNX成功后Redis宕机，EXPIRE未执行，导致锁死锁；解决：Redis6.0+用SET NX PX一次性完成，或用Lua脚本保证原子性。

- 考点3：分布式锁的死锁如何避免？
答：1. 设置合理的过期时间，确保锁能自动释放；2. 实现锁续约机制，避免业务执行时间过长导致锁提前释放；3. 业务执行失败时，主动释放锁；4. 结合Redis集群，避免主节点宕机导致锁丢失。

## 2.3 计数器场景（高频统计）

核心背景：生产环境中，需频繁统计各类计数（如商品浏览量、文章阅读量、接口访问量、点赞数），这类场景要求计数实时更新、支持高并发，Redis的String数据结构支持原子自增/自减命令，完美适配该场景，性能远超数据库计数。

### 2.3.1 核心实现方案（衔接第六章String数据结构）

计数器场景主要使用String数据结构，利用INCR、INCRBY、DECR、DECRBY等原子命令，实现计数的实时更新，无需加锁，支持高并发。同时可结合过期时间，实现临时计数（如每日访问量）。

核心优势：原子操作，避免并发计数错误；内存操作，响应速度快，支持每秒数万次计数更新。

### 2.3.2 实操命令（实战重点）

```bash
# 1. 基础计数（自增1）：商品浏览量统计
INCR view:goods:1001  # 商品1001的浏览量+1
GET view:goods:1001   # 查询商品1001的浏览量

# 2. 批量自增/自减：文章点赞数+5，收藏数-1
INCRBY like:article:2001 5
DECRBY collect:article:2001 1

# 3. 临时计数（每日访问量，每天凌晨过期）
# 生成每日key（如20260419）
SET view:day:20260419 0 EX 86400
INCR view:day:20260419  # 每日访问量+1

# 4. 重置计数
SET view:goods:1001 0  # 将商品1001的浏览量重置为0
```

### 2.3.3 注意事项（生产避坑重点）

- 数据持久化：计数数据若需长期保存，需开启Redis持久化（AOF+RDB混合持久化，衔接第十二章），避免Redis宕机后计数丢失。

- 计数溢出：String数据结构支持的整数范围为64位（-9223372036854775808~9223372036854775807），一般场景可满足，若需更大计数，可结合Hash拆分计数（如按时间段拆分）。

- 高并发优化：高并发计数场景（如秒杀商品访问量），可开启Redis多线程IO（衔接第十二章），提升计数更新性能；同时使用Redis集群，分担计数压力。

### 2.3.4 面试考点

- 考点1：Redis实现计数器的优势是什么？为什么不用数据库计数？
答：优势：1. 原子操作（INCR等命令），避免并发计数错误；2. 内存操作，响应速度快，支持高并发；3. 操作简单，无需复杂SQL。数据库计数：磁盘操作，响应慢，高并发下易卡顿，且需加锁保证原子性，性能差。

- 考点2：Redis计数器如何保证数据不丢失？
答：开启Redis持久化（推荐RDB+AOF混合持久化），确保Redis宕机后，计数数据可从持久化文件中恢复；同时可定期将计数同步到数据库，做持久化备份。

## 2.4 会话存储场景（分布式系统必用）

核心背景：分布式系统中，用户会话（如登录状态、购物车）需在多台服务器间共享，传统的本地会话（如Tomcat会话）无法实现跨服务器共享，Redis可作为分布式会话存储容器，存储用户会话信息，实现会话共享，适配电商、后台管理系统等场景。

### 2.4.1 核心实现方案（衔接第六章String/Hash数据结构）

会话存储场景可根据会话复杂度选择数据结构：

- 简单会话（仅存储登录状态、用户ID）：使用String数据结构，key为会话ID（如session:123456），value为用户ID，设置过期时间（与会话有效期一致，如30分钟）。

- 复杂会话（存储用户ID、登录时间、权限信息等）：使用Hash数据结构，key为会话ID，field为会话字段（如user_id、login_time、permissions），value为字段值，便于单独更新会话信息。

核心流程：用户登录成功后，生成唯一会话ID，将会话信息存入Redis，设置过期时间；用户后续请求时，携带会话ID，从Redis中查询会话信息，验证登录状态；用户退出或会话过期时，删除会话信息。

### 2.4.2 实操命令（实战重点）

```bash
# 1. 简单会话存储（String）
SET session:123456 1001 EX 1800  # 会话ID=123456，用户ID=1001，过期时间30分钟

# 2. 复杂会话存储（Hash）
HSET session:654321 user_id 1002 login_time "2026-04-19 10:00:00" permissions "admin"
HGETALL session:654321  # 查询完整会话信息
HSET session:654321 login_time "2026-04-19 10:30:00"  # 更新登录时间
EXPIRE session:654321 1800  # 重置会话过期时间

# 3. 会话验证与删除
GET session:123456  # 验证会话是否存在（登录状态）
DEL session:123456  # 用户退出，删除会话

# 4. 批量查询过期会话（Redis6.2+新增命令）
EXPIREAT session:*  # 查看所有会话的过期时间
```

### 2.4.3 注意事项（生产避坑重点）

- 会话过期时间：需与业务会话有效期一致，同时可实现会话续约（如用户操作时，重置过期时间），避免用户正常操作时会话过期。

- 数据安全：会话信息可能包含敏感数据（如权限、用户信息），需开启Redis ACL权限控制（衔接第十二章），禁止未授权访问；同时可对会话数据进行加密存储。

- 高可用保障：会话存储是核心场景，需部署Redis集群（主从+哨兵），避免Redis宕机导致会话丢失，影响用户使用（衔接第八章集群）。

- 内存优化：会话数据会持续增长，需设置合理的过期时间，同时开启内存淘汰策略（如LRU），避免内存溢出（衔接第九章）。

### 2.4.4 面试考点

- 考点1：分布式系统中，Redis实现会话共享的优势是什么？
答：1. 跨服务器共享：解决分布式系统中本地会话无法共享的问题；2. 高性能：内存操作，响应速度快，不影响接口性能；3. 可扩展性强：支持Redis集群，适配高并发场景；4. 过期机制：自动清理过期会话，无需手动维护。

- 考点2：会话存储为什么选择Redis，而不选择数据库？
答：1. 性能差异：Redis是内存数据库，响应时间毫秒级，数据库是磁盘操作，响应慢；2. 过期机制：Redis支持自动过期，数据库需手动清理过期会话；3. 高并发：Redis支持高并发访问，数据库高并发下易卡顿。

## 2.5 消息队列场景（轻量级异步通信）

核心背景：生产环境中，需实现异步通信（如订单创建后发送通知、日志异步收集、任务异步执行），避免同步操作导致接口响应缓慢。Redis的List、Stream数据结构可实现轻量级消息队列，无需部署专门的消息队列（如RabbitMQ、Kafka），适配中小规模异步场景。

### 2.5.1 核心实现方案（衔接第六章List/Stream数据结构）

根据业务需求选择数据结构，两种方案各有适用场景：

- List实现简单消息队列（先进先出FIFO）：使用LPUSH（入队）、RPOP（出队）命令，实现消息的生产与消费；结合BLPOP（阻塞出队），避免消费者轮询，减少资源浪费。适用于简单异步场景（如日志收集、通知发送）。

- Stream实现高级消息队列（Redis5.0+新增）：支持消息持久化、消费者组、消息确认、重试机制，功能更完善，适用于复杂异步场景（如订单异步处理、任务调度），衔接第十二章Redis新特性拓展。

### 2.5.2 实操命令（实战重点）

```bash
# 方案1：List实现简单消息队列
# 生产者：消息入队（LPUSH从左侧入队，RPUSH从右侧入队）
LPUSH queue:notice "用户1001下单成功，发送通知"
LPUSH queue:notice "用户1002登录成功，发送欢迎通知"

# 消费者：消息出队（BLPOP阻塞出队，超时时间30秒，无消息时阻塞等待）
BLPOP queue:notice 30

# 方案2：Stream实现高级消息队列（Redis5.0+）
# 1. 生产者：发送消息（XADD）
XADD stream:order * order_id 3001 user_id 1001 status "created"  # *表示自动生成消息ID
XADD stream:order * order_id 3002 user_id 1002 status "created"

# 2. 消费者组：创建消费者组（XGROUP）
XGROUP CREATE stream:order group1 $  # $表示从最新消息开始消费

# 3. 消费者：从消费者组获取消息（XREADGROUP）
XREADGROUP GROUP group1 consumer1 COUNT 1 BLOCK 30000 STREAMS stream:order >

# 4. 消费者：确认消息消费完成（XACK）
XACK stream:order group1 1713528000000-0  # 消息ID

# 5. 查看未确认的消息（XPENDING）
XPENDING stream:order group1
```

### 2.5.3 注意事项（生产避坑重点）

- 消息持久化：开启Redis持久化（AOF+RDB），避免Redis宕机后消息丢失；Stream消息队列默认持久化，List消息队列需依赖Redis持久化。

- 消息丢失：List队列若消费者宕机，未消费的消息会留在队列中，重启后可继续消费；Stream队列支持消息确认机制，未确认的消息可重新分配给其他消费者，避免消息丢失。

- 适用场景：Redis消息队列适用于轻量级、中小规模异步场景；若需高吞吐、高可用（如秒杀消息、大规模日志收集），建议使用专门的消息队列（RabbitMQ、Kafka）。

- 消息堆积：定期监控消息队列长度，若出现消息堆积，需增加消费者数量，或优化消费逻辑，避免队列溢出。

### 2.5.4 面试考点

- 考点1：Redis如何实现消息队列？有哪些方案？
答：两种核心方案：1. List数据结构，使用LPUSH/RPOP/BLPOP命令，实现简单FIFO消息队列；2. Stream数据结构（Redis5.0+），支持消费者组、消息确认、重试，实现高级消息队列。

- 考点2：Redis消息队列与专门的消息队列（如RabbitMQ）有什么区别？
答：Redis消息队列：轻量级、部署简单、无需额外组件，适用于中小规模异步场景；缺点是吞吐率低、功能简单（如无复杂路由、延迟队列）。专门消息队列：功能完善、吞吐率高、支持复杂场景，缺点是部署复杂、运维成本高。

## 2.6 排行榜场景（高频业务）

核心背景：生产环境中，需实现各类排行榜（如商品销量排行、用户积分排行、文章点赞排行），要求支持实时更新、按分数排序、范围查询，Redis的ZSet（有序集合）数据结构完美适配该场景，ZSet的核心特性就是“有序、可排序、支持范围查询”。

### 2.6.1 核心实现方案（衔接第六章ZSet数据结构）

排行榜场景核心使用ZSet数据结构，key为排行榜名称（如rank:goods:sales、rank:user:points），member为参与排行的对象（如商品ID、用户ID），score为排序依据（如销量、积分、点赞数），利用ZSet的排序特性，实现实时排行榜。

核心优势：支持按score升序/降序排序、范围查询（如前10名、前100名）、实时更新（score变化时自动重新排序），性能高效。

### 2.6.2 实操命令（实战重点）

```bash
# 1. 添加排行数据（ZADD）：商品销量排行榜
ZADD rank:goods:sales 100 1001  # 商品1001，销量100
ZADD rank:goods:sales 200 1002  # 商品1002，销量200
ZADD rank:goods:sales 150 1003  # 商品1003，销量150

# 2. 更新排行数据（ZINCRBY）：商品1001销量+50
ZINCRBY rank:goods:sales 50 1001

# 3. 查看排行榜（降序，默认降序，ASC为升序）
# 查看前3名（带score）
ZREVRANGE rank:goods:sales 0 2 WITHSCORES
# 查看商品1001的排名（降序）
ZREVRANK rank:goods:sales 1001

# 4. 范围查询（查看销量100-200的商品）
ZRANGEBYSCORE rank:goods:sales 100 200 WITHSCORES

# 5. 删除排行数据
ZREM rank:goods:sales 1003
```

### 2.6.3 注意事项（生产避坑重点）

- score精度：ZSet的score支持64位浮点数，一般场景可满足，若需高精度排序（如积分精确到小数点后两位），需注意score的精度问题。

- 数据过期：若排行榜是临时的（如每日销量排行），可给ZSet设置过期时间，或每日凌晨删除旧数据，重新统计。

- 高并发优化：高并发更新排行榜（如秒杀商品销量），可开启Redis多线程IO（衔接第十二章），同时使用Redis集群，分担排序压力；避免频繁查询全量排行榜，优先查询TopN数据。

### 2.6.4 面试考点

- 考点1：Redis实现排行榜的核心数据结构是什么？核心优势是什么？
答：核心数据结构是ZSet（有序集合）；核心优势：1. 自动按score排序，无需手动维护排序逻辑；2. 支持范围查询、排名查询，操作高效；3. 支持原子更新（ZINCRBY），适配高并发场景。

- 考点2：ZSet的底层实现是什么？为什么适合做排行榜？
答：ZSet底层是“字典+跳表”（衔接第十一章源码）；字典用于存储“member→score”的映射，快速查询member的score；跳表用于按score排序，支持高效的范围查询和排名查询，因此适合做排行榜。

## 2.7 布隆过滤器场景（高并发去重）

核心背景：高并发场景中，需快速判断一个元素是否存在于集合中（如用户是否已参与活动、URL是否已爬取、订单是否已处理），若集合数据量巨大（如千万级、亿级），使用Redis普通集合（Set）会占用大量内存，布隆过滤器可在占用极小内存的前提下，快速判断元素是否存在，适配去重、防重复场景。

### 2.7.1 核心实现方案（衔接第六章Set数据结构+Redis扩展）

布隆过滤器是一种概率型数据结构，核心原理：通过多个哈希函数将元素映射到一个二进制数组中，判断元素是否存在时，检查对应的哈希位置是否全为1，全为1则可能存在（存在误判），有一个为0则一定不存在（无漏判）。

Redis中可通过RedisBloom扩展模块实现布隆过滤器，或用Bitmaps模拟简单布隆过滤器，生产环境中优先使用RedisBloom扩展，操作更便捷。

### 2.7.2 实操命令（实战重点，RedisBloom扩展）

```bash
# 1. 创建布隆过滤器（指定名称、预计元素数量、误判率）
BF.RESERVE bloom:activity:1001 100000 0.01  # 预计10万个元素，误判率1%

# 2. 向布隆过滤器中添加元素（用户ID）
BF.ADD bloom:activity:1001 1001
BF.ADD bloom:activity:1001 1002
BF.ADD bloom:activity:1001 1003

# 3. 判断元素是否存在（用户是否已参与活动）
BF.EXISTS bloom:activity:1001 1001  # 存在返回1（可能误判）
BF.EXISTS bloom:activity:1001 1004  # 不存在返回0（无漏判）

# 4. 查看布隆过滤器信息
BF.INFO bloom:activity:1001
```

### 2.7.3 注意事项（生产避坑重点）

- 误判率：布隆过滤器存在误判（无法避免），误判率越低，占用内存越大，需根据业务需求平衡（如活动去重误判率可设1%-5%）。

- 元素删除：布隆过滤器不支持元素删除（删除一个元素会影响其他元素的判断），若需支持删除，可使用计数布隆过滤器（RedisBloom扩展支持）。

- 适用场景：适用于“允许少量误判、无需删除元素”的去重场景，如活动防重复参与、URL去重、缓存穿透拦截（衔接2.1缓存场景）。

- 内存占用：布隆过滤器占用内存极小（如100万个元素，误判率1%，仅需约12MB内存），远优于普通Set集合。

### 2.7.4 面试考点

- 考点1：布隆过滤器的核心原理是什么？有什么特点？
答：核心原理：通过多个哈希函数将元素映射到二进制数组，判断元素是否存在时，检查对应哈希位置是否全为1；特点：1. 内存占用极小；2. 查询速度快；3. 存在误判（可控制）；4. 不支持元素删除；5. 无漏判（不存在的元素一定能判断出来）。

- 考点2：布隆过滤器在Redis中的应用场景有哪些？
答：1. 缓存穿透拦截：拦截不存在的key，避免频繁访问数据库；2. 活动防重复参与：判断用户是否已参与活动，避免重复参与；3. URL去重：爬虫场景中，判断URL是否已爬取；4. 订单去重：判断订单是否已处理，避免重复处理。

## 2.8 地理位置场景（LBS业务）

核心背景：LBS（地理位置）业务中，需实现基于地理位置的查询（如附近的人、附近的商家、距离计算），Redis的GEO（地理位置）功能（基于ZSet实现），可存储地理位置坐标，支持距离计算、范围查询，适配外卖、打车、社交等LBS场景。

### 2.8.1 核心实现方案（衔接第六章ZSet数据结构）

Redis的GEO功能底层基于ZSet实现，将经纬度坐标通过GeoHash算法转换为一个64位整数（score），member为地理位置标识（如商家ID、用户ID），通过ZSet的排序特性，实现地理位置相关查询。

核心功能：存储地理位置、计算两点距离、查询附近的元素、获取元素经纬度。

### 2.8.2 实操命令（实战重点）

```bash
# 1. 存储地理位置（GEOADD）：纬度、经度、标识、名称
GEOADD geo:merchant 116.403874 39.914885 1001 "北京朝阳店"
GEOADD geo:merchant 116.413874 39.924885 1002 "北京海淀店"
GEOADD geo:merchant 116.393874 39.904885 1003 "北京东城店"

# 2. 获取元素经纬度（GEOPOS）
GEOPOS geo:merchant 1001 1002

# 3. 计算两点距离（GEODIST）：单位m（米）、km（千米）、mi（英里）、ft（英尺）
GEODIST geo:merchant 1001 1002 km  # 计算北京朝阳店与海淀店的距离（千米）

# 4. 查询附近的元素（GEORADIUS）：根据经纬度查询附近商家，半径5千米，返回距离
GEORADIUS geo:merchant 116.403874 39.914885 5 km WITHDIST WITHLATLONG LIMIT 10

# 5. 根据元素查询附近的元素（GEORADIUSBYMEMBER）
GEORADIUSBYMEMBER geo:merchant 1001 5 km WITHDIST LIMIT 10

# 6. 删除地理位置（GEO本身不支持删除，需用ZREM）
ZREM geo:merchant 1003
```

### 2.8.3 注意事项（生产避坑重点）

- 坐标精度：GEO存储的经纬度精度为6位小数，可满足大部分LBS场景（如外卖、打车），若需更高精度，需结合其他地理位置工具。

- 删除操作：GEO功能本身不提供删除命令，需通过ZREM命令（因为底层是ZSet）删除地理位置元素，删除时需指定key和member。

- 范围查询优化：高并发LBS场景（如高峰期查询附近商家），可限制查询范围和返回数量（LIMIT），避免查询耗时过长；同时使用Redis集群，分担查询压力。

- 数据持久化：开启Redis持久化，避免Redis宕机后地理位置数据丢失，影响LBS业务正常运行。

### 2.8.4 面试考点

- 考点1：Redis的GEO功能底层实现是什么？核心原理是什么？
答：底层基于ZSet数据结构实现；核心原理：通过GeoHash算法将经纬度坐标转换为64位整数（作为ZSet的score），member为地理位置标识，利用ZSet的排序特性，实现距离计算、范围查询等功能。

- 考点2：Redis GEO适合哪些LBS场景？有什么局限性？
答：适合场景：附近的人、附近的商家、距离计算、地理位置排序（如外卖商家排序）；局限性：1. 精度有限（6位小数），不适合高精度场景；2. 不支持复杂的地理位置查询（如多边形查询）；3. 大数据量下查询性能会下降，需结合集群优化。

# 三、场景选型与性能优化（生产实战重点）

结合前十二章知识点与本章场景，梳理Redis场景选型原则与性能优化方案，帮助开发者在生产环境中合理选择场景、规避性能问题，提升Redis使用效率。

## 3.1 场景选型原则

- 数据结构优先：根据场景需求选择合适的数据结构（如缓存用String/Hash、排行榜用ZSet、消息队列用List/Stream），避免过度设计。

- 性能优先：高并发场景（如缓存、计数器）优先考虑Redis集群、多线程IO，确保性能达标；低并发场景可使用单节点Redis，降低运维成本。

- 数据安全优先：数据敏感场景（如会话、权限）开启ACL权限控制、数据加密、持久化；非敏感临时数据（如每日计数）可适当降低持久化级别。

- 成本优先：轻量级场景（如简单消息队列、小量计数）优先使用Redis，无需部署专门组件；大规模、高吞吐场景（如海量日志异步）可选择专门工具。

## 3.2 通用性能优化方案（衔接前序章节）

- 数据结构优化：避免使用大键（如超大String、超大Hash），大键会导致IO阻塞、内存碎片，可拆分大键（如将超大Hash拆分为多个小Hash）。

- 持久化优化：根据场景选择合适的持久化方案（如缓存场景用RDB，数据敏感场景用AOF+RDB混合持久化），避免持久化影响性能（如AOF刷盘策略设为everysec）。

- 集群优化：高并发、大数据量场景部署Redis集群，合理分配槽位，避免单节点压力过大；开启主从复制+哨兵，确保高可用。

- 内存优化：设置合理的过期时间、开启内存淘汰策略，避免内存溢出；开启内存池机制，减少内存碎片（衔接第十一章）。

- 命令优化：避免使用慢命令（如KEYS、HGETALL），慢命令会阻塞单线程；高并发场景避免频繁查询全量数据，优先查询部分数据（如排行榜查TopN）。
