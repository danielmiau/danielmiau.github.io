# 05-高可用：主从+哨兵+集群.md

前言：本文承接第四章Redis持久化机制，聚焦Redis高可用的三大核心方案——主从复制、哨兵机制、Redis Cluster集群。本章延续上一章的深度，从「核心原理、部署配置、工作流程、实战实操、优缺点对比、常见问题排查」六大维度，深入拆解Redis高可用的实现逻辑，适配中高级进阶与生产环境部署。

# 一、高可用核心认知（深入前置）

## 1.1 高可用的定义与核心目标

Redis高可用的核心定义是：**保证Redis服务在任何情况下（如主节点宕机、网络故障、硬件故障），都能持续提供稳定的读写服务，且数据不丢失、服务不中断**。

Redis作为缓存/数据库，是业务系统的核心依赖，一旦服务中断，会直接导致业务不可用，因此高可用是生产环境部署Redis的必备条件，其核心目标有3个：

- 服务可用性：99.99%以上的服务在线率，避免单点故障导致服务中断；

- 数据可靠性：主节点故障后，从节点能无缝接管，数据不丢失（结合第四章持久化机制）；

- 性能可扩展：支持横向扩展，分担主节点读写压力，应对高并发场景。

## 1.2 Redis高可用的三大核心方案

Redis高可用方案从简单到复杂，分为三个层级，适配不同业务规模，可单独使用或组合使用：

1. 主从复制：基础高可用方案，实现数据同步与读写分离，解决单点故障的基础问题；

2. 哨兵机制：在主从复制基础上，实现故障自动检测、自动切换，无需人工干预；

3. Redis Cluster集群：分布式高可用方案，实现数据分片、负载均衡，支持大规模部署（万级以上并发）。

关键：主从复制是基础，哨兵解决“自动故障切换”，集群解决“高并发、大数据量”的扩展问题，生产环境中，中小规模用“主从+哨兵”，大规模用“Redis Cluster”。

## 1.3 高可用与持久化的关联

第四章的持久化（RDB+AOF）是高可用的“数据保障”，高可用是持久化的“服务保障”，两者缺一不可：

- 持久化：保证“数据不丢失”，即使所有节点宕机，重启后可通过持久化文件恢复数据；

- 高可用：保证“服务不中断”，即使主节点宕机，从节点/新主节点能快速接管，避免业务不可用。

# 二、主从复制（基础高可用）

## 2.1 核心原理（深度拆解）

主从复制（Master-Slave Replication）是Redis最基础的高可用方案，核心逻辑是：**将一个Redis节点作为主节点（Master），多个节点作为从节点（Slave），主节点负责接收所有写命令（set、hset等），并将数据同步到所有从节点；从节点仅负责读命令（get、hget等），不允许写操作，实现读写分离与数据备份**。

核心细节拆解：

- 数据同步机制：主从复制分为“全量同步”和“增量同步”，首次同步（从节点刚启动）采用全量同步，后续同步采用增量同步，减少网络开销。

- 全量同步流程：从节点向主节点发送sync命令 → 主节点执行bgsave生成RDB文件，同时记录期间的写命令到缓冲区 → 主节点将RDB文件发送给从节点 → 从节点加载RDB文件，恢复全量数据 → 主节点将缓冲区的写命令发送给从节点，从节点执行命令，完成同步。

- 增量同步流程：主节点执行写命令后，会将命令记录到“复制积压缓冲区”（环形缓冲区） → 从节点定期向主节点发送ack消息，告知主节点自己已同步到的命令偏移量 → 主节点将从节点未同步的命令发送给从节点，从节点执行，完成增量同步。

- 读写分离：主节点负责写，从节点负责读，可将读请求分发到多个从节点，分担主节点读压力，提升并发能力。

## 2.2 部署配置（生产级实操）

主从复制部署简单，无需额外组件，仅需修改Redis配置文件，通常采用“一主多从”架构（1个主节点+2~3个从节点，避免单从节点故障），以下是Linux环境下的核心配置步骤：

### 2.2.1 主节点（Master）配置

主节点无需特殊配置，默认即可作为主节点，建议补充以下配置（redis.conf），优化同步性能：

```ini
# 主节点端口（默认6379，可修改）
port 6379
# 绑定IP（生产环境建议绑定内网IP，避免外部访问）
bind 192.168.1.100
# 开启持久化（结合第四章，保证数据可靠性）
appendonly yes
appendfsync everysec
save 3600 1
# 允许从节点连接（默认yes）
replica-allow-read-only yes
# 复制积压缓冲区大小（默认1MB，建议调整为10MB，避免增量同步失败）
repl-backlog-size 10mb
# 复制积压缓冲区过期时间（默认3600秒，无从节点连接时释放）
repl-backlog-ttl 3600
# 禁止主节点执行写命令时，从节点同步阻塞主节点（默认no，推荐yes）
repl-diskless-sync yes
```

### 2.2.2 从节点（Slave）配置

从节点需配置主节点的IP和端口，指定自己为从节点，核心配置（redis.conf）：

```ini
# 从节点端口（每个从节点端口不同，避免冲突）
port 6380
# 绑定IP（与主节点同内网）
bind 192.168.1.101
# 开启持久化
appendonly yes
appendfsync everysec
# 关键：指定主节点的IP和端口，配置后启动即为从节点
replicaof 192.168.1.100 6379
# 从节点只读（默认yes，禁止写操作，避免数据不一致）
replica-read-only yes
# 若主节点设置了密码，从节点需配置密码，否则无法同步
masterauth 123456
# 从节点优先使用RDB同步（默认yes，比AOF同步更快）
repl-diskless-sync yes
# 同步超时时间（默认60秒，可根据网络情况调整）
repl-timeout 120
```

补充：若已启动Redis，无需重启，可通过命令临时设置从节点（重启后失效）：slaveof 主节点IP 主节点端口（Redis 5.0+ 也可使用replicaof命令，功能一致）。

## 2.3 核心操作命令（实操高频）

```redis
# 主节点查看从节点列表（查看所有连接的从节点信息）
info replication

# 从节点设置主节点（临时生效，重启失效）
replicaof 192.168.1.100 6379

# 从节点取消主从关系（成为独立节点，临时生效）
replicaof no one

# 手动触发主节点全量同步（一般无需手动操作，异常时使用）
sync

# 从节点查看同步状态
info replication # 重点关注master_link_status（up=同步正常，down=同步异常）
```

## 2.4 优缺点深度分析

### 2.4.1 优点

- 部署简单：无需额外组件，仅需修改配置文件，成本低；

- 实现读写分离：分担主节点读压力，提升并发读性能；

- 数据备份：从节点是主节点的完整备份，主节点故障后，可手动切换从节点为主节点；

- 兼容性好：所有Redis版本都支持，适配中小规模业务。

### 2.4.2 缺点（核心痛点）

- 无自动故障切换：主节点宕机后，需人工干预切换从节点为主节点，期间服务中断（分钟级）；

- 主节点压力大：所有写命令都集中在主节点，高并发写场景下，主节点会成为瓶颈；

- 数据同步有延迟：增量同步存在毫秒级延迟，极端情况下，从节点数据与主节点不一致；

- 不支持横向扩展：所有从节点都存储全量数据，数据量大时，存储成本高，无法扩展。

# 三、哨兵机制（自动故障切换）

## 3.1 核心原理（深度拆解）

哨兵机制（Sentinel）是在主从复制基础上，实现“自动故障检测、自动故障切换”的高可用方案，核心逻辑是：**部署多个哨兵节点（通常3个），哨兵节点持续监控主节点和从节点的状态，当主节点宕机时，哨兵节点通过投票机制，自动选举一个从节点升级为主节点，同时更新其他从节点的主节点配置，实现服务无缝切换，无需人工干预**。

核心细节拆解：

- 哨兵节点的作用：监控（Monitor）、通知（Notify）、自动故障切换（Failover）。

- 监控机制：哨兵节点每隔1秒向主节点、从节点发送ping命令，检测节点是否存活；若节点在指定时间内（down-after-milliseconds）未响应，哨兵节点标记该节点为“主观下线”（SDOWN）。

- 故障确认：当多个哨兵节点（超过半数）都标记主节点为“主观下线”，则主节点被标记为“客观下线”（ODOWN），确认主节点真正故障。

- 故障切换流程：主节点客观下线后，哨兵节点通过投票机制（Raft算法）选举一个从节点 → 选举成功后，哨兵节点向该从节点发送slaveof no one命令，使其升级为主节点 → 哨兵节点向其他从节点发送replicaof 新主节点IP 新主节点端口命令，使其同步新主节点 → 哨兵节点更新自身配置，记录新主节点信息，完成故障切换。

- 哨兵集群：部署多个哨兵节点（至少3个，奇数个，避免投票平局），哨兵节点之间也会互相监控，确保哨兵节点自身的高可用，单个哨兵节点故障不影响整体功能。

## 3.2 部署配置（生产级实操）

哨兵机制部署需满足：1个主节点+2个从节点+3个哨兵节点（最小高可用架构），哨兵节点可部署在与主从节点不同的服务器，提升可靠性，核心配置步骤如下：

### 3.2.1 哨兵配置文件（sentinel.conf）

每个哨兵节点使用相同的配置文件（可修改端口区分），核心配置如下：

```ini
# 哨兵节点端口（每个哨兵端口不同，默认26379）
port 26379
# 绑定IP（内网IP，允许其他哨兵节点连接）
bind 192.168.1.100
# 监控主节点：sentinel monitor <主节点名称> <主节点IP> <主节点端口> <投票数量>
# 含义：监控名为mymaster的主节点，投票数量为2（超过2个哨兵标记主节点下线，才确认故障）
sentinel monitor mymaster 192.168.1.100 6379 2
# 主节点密码（若主节点设置了密码，必须配置，否则哨兵无法监控）
sentinel auth-pass mymaster 123456
# 节点下线判断时间（默认30000毫秒=30秒，建议改为10000毫秒=10秒，加快故障检测）
sentinel down-after-milliseconds mymaster 10000
# 故障切换超时时间（默认180000毫秒=3分钟，建议改为60000毫秒=1分钟）
sentinel failover-timeout mymaster 60000
# 故障切换时，最多有多少个从节点同时同步新主节点（默认1，避免同步压力过大）
sentinel parallel-syncs mymaster 1
# 禁止哨兵节点在故障切换期间，执行写命令（默认no，推荐yes）
sentinel deny-scripts-reconfig yes
```

### 3.2.2 启动顺序（关键）

哨兵机制启动必须遵循“先主从，后哨兵”的顺序，否则哨兵无法找到主节点，启动命令如下：

```bash
# 1. 启动主节点（后台启动）
redis-server /etc/redis/redis-master.conf &

# 2. 启动两个从节点（后台启动）
redis-server /etc/redis/redis-slave1.conf &
redis-server /etc/redis/redis-slave2.conf &

# 3. 启动三个哨兵节点（后台启动，每个哨兵单独启动）
redis-sentinel /etc/redis/sentinel1.conf &
redis-sentinel /etc/redis/sentinel2.conf &
redis-sentinel /etc/redis/sentinel3.conf 
```

## 3.3 核心操作命令（实操高频）

```redis
# 哨兵节点查看监控状态（核心命令，查看主从节点状态、哨兵节点数量）
sentinel info

# 查看指定主节点的详细信息（如mymaster）
sentinel master mymaster

# 查看指定主节点的从节点列表
sentinel slaves mymaster

# 手动触发故障切换（主节点正常时不建议使用，故障时可手动触发）
sentinel failover mymaster

# 查看哨兵节点列表
sentinel sentinels mymaster

# 移除指定从节点（主节点故障后，可移除故障从节点）
sentinel remove mymaster 192.168.1.101 6380
```

## 3.4 优缺点深度分析

### 3.4.1 优点

- 自动故障切换：主节点宕机后，哨兵自动完成故障检测、节点选举、主从切换，无需人工干预，服务中断时间缩短到秒级；

- 高可靠性：哨兵集群部署（奇数个），单个哨兵节点故障不影响整体功能，同时监控主从节点，确保故障检测准确；

- 兼容主从复制：基于主从复制实现，保留读写分离、数据备份的优势，无需修改主从节点配置；

- 部署简单：仅需新增哨兵节点，无需修改主从节点核心配置，成本适中。

### 3.4.2 缺点

- 不支持横向扩展：依然存在主节点写压力瓶颈，所有从节点存储全量数据，数据量大时存储成本高；

- 故障切换有短暂延迟：从主节点宕机到新主节点接管，存在秒级延迟，期间读请求可正常访问从节点，写请求会暂时中断；

- 配置复杂：相比主从复制，哨兵节点的配置更多，需要关注投票数量、超时时间等参数，容易配置不当导致故障切换失败；

- 不支持数据分片：所有节点存储全量数据，无法应对TB级大数据量场景。

# 四、Redis Cluster集群（分布式高可用）

## 4.1 核心原理（深度拆解）

Redis Cluster是Redis 3.0+ 推出的分布式高可用方案，核心逻辑是：**将Redis数据分片存储在多个节点（至少6个，3主3从），每个主节点负责一部分数据（哈希槽），从节点作为主节点的备份，实现数据分片、负载均衡、自动故障切换，解决主从+哨兵方案的“写压力瓶颈”和“存储扩展”问题**。

核心细节拆解（重点是哈希槽机制）：

- 哈希槽（Hash Slot）：Redis Cluster将所有数据划分为16384个哈希槽（0~16383），每个主节点负责一部分哈希槽（如3主节点，每个节点负责约5461个槽），数据存储时，通过key的哈希值取模16384，确定数据存储在哪个哈希槽，进而存储到对应的主节点。

- 节点架构：最小集群架构为3主3从，每个主节点对应至少1个从节点，主节点负责读写操作，从节点负责备份和故障接管；主节点之间互相通信，维护集群状态（通过Gossip协议）。

- 故障切换：当某个主节点宕机时，其对应的从节点会自动升级为主节点，接管该主节点的所有哈希槽；若主节点无从节点，集群会进入不可用状态（因此必须配置从节点）。

- 数据分片优势：每个主节点仅存储一部分数据，分散写压力和存储压力，支持横向扩展（新增主从节点，重新分配哈希槽即可）。

- 跨节点访问：当客户端访问的数据不在当前节点的哈希槽时，节点会返回“重定向”信息，告知客户端数据所在的节点IP和端口，客户端自动切换到目标节点访问（Redis 3.0+ 客户端支持自动重定向）。

## 4.2 部署配置（生产级实操）

Redis Cluster集群部署要求：至少6个Redis节点（3主3从），每个节点独立配置，不依赖哨兵节点，核心配置步骤如下（Linux环境）：

### 4.2.1 单个节点配置（redis.conf）

每个节点配置类似，重点修改端口、IP、集群相关配置，以主节点1（port 6379）为例：

```ini
# 节点端口（6个节点端口分别为6379、6380、6381、6382、6383、6384）
port 6379
# 绑定IP（内网IP）
bind 192.168.1.100
# 开启集群模式（核心配置，必须开启）
cluster-enabled yes
# 集群配置文件（自动生成，无需手动编辑，每个节点单独生成）
cluster-config-file nodes-6379.conf
# 集群节点超时时间（默认15000毫秒=15秒，建议改为10000毫秒）
cluster-node-timeout 10000
# 开启持久化（保证数据可靠性）
appendonly yes
appendfsync everysec
# 允许从节点升级为主节点（默认yes）
cluster-replica-validity-factor 10
# 每个主节点至少有1个从节点（默认1，符合3主3从架构）
cluster-migration-barrier 1
# 禁止集群自动创建主节点（默认no，推荐yes，避免误操作）
cluster-require-full-coverage no
```

补充：其他5个节点（3个主节点、3个从节点），仅修改port和cluster-config-file名称（如nodes-6380.conf），其余配置与上述一致。

### 4.2.2 集群启动与初始化（关键步骤）

```bash
# 1. 启动所有6个Redis节点（后台启动）
redis-server /etc/redis/cluster/6379.conf &
redis-server /etc/redis/cluster/6380.conf &
redis-server /etc/redis/cluster/6381.conf &
redis-server /etc/redis/cluster/6382.conf &
redis-server /etc/redis/cluster/6383.conf &
redis-server /etc/redis/cluster/6384.conf &

# 2. 初始化集群（仅执行一次，任意节点执行均可）
# 命令格式：redis-cli --cluster create 节点1IP:端口 节点2IP:端口 ... --cluster-replicas 1
# --cluster-replicas 1 表示每个主节点对应1个从节点
redis-cli --cluster create 192.168.1.100:6379 192.168.1.100:6380 192.168.1.100:6381 192.168.1.100:6382 192.168.1.100:6383 192.168.1.100:6384 --cluster-replicas 1

# 3. 确认集群初始化（输入yes，Redis会自动分配主从关系和哈希槽）
# 初始化成功后，会显示3个主节点、3个从节点的信息，以及每个主节点负责的哈希槽范围
```

### 4.2.3 集群扩容/缩容（补充实操）

生产环境中，可根据业务需求新增/删除节点，实现横向扩展，核心命令：

```bash
# 1. 新增主节点（先启动新节点，再加入集群）
redis-cli --cluster add-node 新节点IP:新节点端口 集群中任意节点IP:端口

# 2. 为新增主节点分配哈希槽（重新分配槽位，平衡数据）
redis-cli --cluster reshard 集群中任意节点IP:端口

# 3. 新增从节点（绑定到指定主节点）
redis-cli --cluster add-node 新从节点IP:端口 主节点IP:端口 --cluster-slave --cluster-master-id 主节点ID

# 4. 删除节点（先删除从节点，再删除主节点，删除主节点前需迁移其哈希槽）
redis-cli --cluster del-node 集群中任意节点IP:端口 要删除的节点ID
```

## 4.4 优缺点深度分析

### 4.4.1 优点

- 数据分片：将数据分散到多个主节点，解决主节点写压力和存储压力瓶颈，支持横向扩展（可扩展到数十个主节点）；

- 自动故障切换：主节点宕机后，从节点自动升级为主节点，无需人工干预，服务可用性高；

- 负载均衡：读写请求分散到多个主节点，读请求还可分发到从节点，提升并发处理能力，支持万级以上并发；

- 高可靠性：结合持久化和主从备份，数据不丢失，集群节点故障不影响整体服务（只要不是所有主节点同时故障）。

### 4.4.2 缺点

- 部署复杂：相比主从+哨兵，集群部署步骤更多，需要配置集群参数、初始化集群、分配哈希槽，维护成本高；

- 不支持跨节点事务：Redis Cluster不支持多key跨节点事务（因为多key可能存储在不同节点），仅支持单节点事务；

- 数据迁移复杂：扩容/缩容时，需要手动迁移哈希槽，迁移过程中可能存在短暂的数据不一致；

- 兼容性限制：部分Redis客户端不支持集群模式，需要升级客户端；部分Redis命令（如keys、flushall）在集群模式下使用受限。

# 五、三大高可用方案对比（生产选型核心）

|对比维度|主从复制|主从+哨兵|Redis Cluster集群|
|---|---|---|---|
|核心作用|数据备份、读写分离|自动故障切换、高可用|数据分片、负载均衡、分布式高可用|
|节点数量|1主N从（最少2个）|1主N从+3个哨兵（最少5个）|3主3从（最少6个）|
|故障切换|手动切换（服务中断久）|自动切换（秒级中断）|自动切换（秒级中断）|
|写压力分担|无（所有写请求集中在主节点）|无（所有写请求集中在主节点）|有（写请求分散到多个主节点）|
|数据分片|无（所有节点存储全量数据）|无（所有节点存储全量数据）|有（16384个哈希槽，分散存储）|
|部署复杂度|低（仅需修改主从配置）|中（需部署哨兵节点，配置参数多）|高（需配置集群、分配哈希槽）|
|维护成本|低（仅维护主从节点）|中（维护主从+哨兵）|高（维护集群节点、哈希槽迁移）|
|适用场景|小规模业务、测试环境、简单备份|中小规模业务、对高可用有要求、读写分离需求|大规模业务、高并发、大数据量、需要横向扩展|
# 六、生产环境高可用选型策略（实战重点）

## 6.1 中小规模业务（推荐：主从+哨兵）

场景：并发量1万以下，数据量GB级，对高可用有要求，无需大规模扩展，核心选型建议：

- 架构：1主2从+3个哨兵节点（部署在不同服务器，提升可靠性）；

- 配置优化：开启RDB+AOF组合持久化，哨兵故障检测时间设为10秒，故障切换超时时间设为1分钟；

- 核心优势：成本适中，部署难度不大，能实现自动故障切换，满足中小规模业务的高可用需求。

## 6.2 大规模业务（推荐：Redis Cluster集群）

场景：并发量1万以上，数据量TB级，需要横向扩展，分担写压力，核心选型建议：

- 架构：3主3从起步，根据业务需求逐步扩容（如6主6从）；

- 配置优化：开启AOF持久化（优先保证数据可靠性），集群节点超时时间设为10秒，每个主节点至少1个从节点；

- 核心优势：支持数据分片和负载均衡，能应对高并发、大数据量场景，可横向扩展，满足业务增长需求。

## 6.3 特殊场景选型

- 测试环境/临时缓存：仅使用主从复制（1主1从），降低部署成本；

- 金融类高可靠场景：主从+哨兵架构，增加从节点数量（1主3从），优化持久化配置（appendfsync everysec），确保数据零丢失；

- 轻量级高可用场景：Redis 6.0+ 可使用“主从+哨兵+ACL”组合，增强权限控制，提升安全性。

# 七、常见问题与排查（生产实战）

## 7.1 主从复制常见问题

- 问题1：从节点无法同步主节点数据？
排查：1. 主从节点网络是否通畅（ping命令测试）；2. 主节点是否开启replica-allow-read-only yes；3. 从节点masterauth配置是否正确（主节点有密码时）；4. 主节点复制积压缓冲区是否足够（repl-backlog-size）；解决：修复网络、检查配置、调整缓冲区大小，重启从节点重新同步。

- 问题2：从节点同步延迟过大？
排查：1. 主节点写压力过大（查看info stats）；2. 网络延迟过高；3. 从节点配置过低（CPU、内存不足）；解决：优化主节点写命令、提升网络带宽、升级从节点硬件配置。

## 7.2 哨兵机制常见问题

- 问题1：哨兵无法检测到主节点故障？
排查：1. 哨兵节点与主节点网络是否通畅；2. down-after-milliseconds配置是否过大；3. 主节点是否设置了密码，哨兵auth-pass配置是否正确；解决：修复网络、调整超时时间、配置正确密码。

- 问题2：故障切换失败？
排查：1. 哨兵节点数量不足（少于3个）；2. 投票数量配置不合理（sentinel monitor中的投票数）；3. 从节点同步状态异常；解决：增加哨兵节点数量、调整投票数、修复从节点同步问题。

## 7.3 Redis Cluster常见问题

- 问题1：集群初始化失败？
排查：1. 所有节点cluster-enabled yes配置是否开启；2. 节点端口是否对外开放；3. 节点之间网络是否通畅；解决：检查配置、开放端口、修复网络，重新初始化集群。

- 问题2：哈希槽分配不均，导致部分主节点压力过大？
排查：查看cluster nodes命令，确认每个主节点的哈希槽数量；解决：执行redis-cli --cluster reshard命令，重新分配哈希槽，平衡节点压力。

- 问题3：客户端无法连接集群，或出现重定向失败？
排查：1. 客户端是否支持集群模式（需带-c参数）；2. 集群节点IP是否为内网IP，客户端是否能访问；3. 集群状态是否正常（cluster info查看）；解决：使用支持集群的客户端、确保网络通畅、修复集群异常节点。

# 八、总结

1. Redis高可用的核心是“避免单点故障、保证服务不中断、数据不丢失”，三大方案层层递进：主从复制是基础，哨兵实现自动故障切换，集群实现分布式扩展；

2. 主从复制解决“数据备份、读写分离”，哨兵解决“自动故障切换”，集群解决“写压力瓶颈、大数据量扩展”；

3. 生产选型原则：中小规模用“主从+哨兵”，大规模用“Redis Cluster”，测试环境用“主从复制”；

4. 关键细节：主从复制的全量/增量同步、哨兵的故障检测与投票机制、集群的哈希槽与数据分片，是面试核心考点；

5. 高可用必须结合持久化机制，两者协同，才能真正实现“数据可靠、服务可用”。
