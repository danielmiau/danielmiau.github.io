# 一、位图（Bitmap）—— 极致省内存的布尔型数据结构

## 1.1 核心定义

Bitmap 并非独立的数据结构，而是基于String类型的按位操作扩展，本质是一串二进制位（bit）序列，每个bit仅存储0或1两种布尔状态，以极小的内存开销实现海量数据的状态标记。

## 1.2 基础使用命令

```redis
# 1. 设置指定偏移量的bit值（offset从0开始，对应唯一标识如用户ID）
setbit user:sign:202604 10 1   # 设置第10位为1（表示用户ID=10当日签到）
getbit user:sign:202604 10     # 获取第10位的值（1=已签到，0=未签到，不存在返回0）

# 2. 统计范围内为1的bit数量（核心场景：统计签到人数、活跃数）
bitcount user:sign:202604      # 统计所有为1的bit数量（当日总签到人数）
bitcount user:sign:202604 0 100 # 统计字节0~100范围内为1的bit数量（指定范围统计）

# 3. 位运算（多组状态对比，如多日签到交集/并集）
bitop AND sign:and user:sign:202604 user:sign:202603 # 两个月都签到的用户（交集）
bitop OR sign:or user:sign:202604 user:sign:202603   # 两个月内至少签到一次的用户（并集）
bitop XOR sign:xor user:sign:202604 user:sign:202603 # 两个月内仅签到一次的用户（异或）
```

## 1.3 底层原理

Bitmap 底层完全复用String的SDS（动态字符串）实现，本质是一串连续的二进制字节数组，无额外数据结构开销。

- 内存优势：1个bit存储1个状态，1亿用户仅需约12.5MB（1亿 ÷ 8 ÷ 1024 ÷ 1024 ≈ 11.9MB）；

- 扩容机制：与String一致，当偏移量超出当前SDS长度时，自动扩容SDS，保证连续内存存储；

- 核心特点：按位操作效率极高（O(1)），适合批量布尔状态管理，不适合存储复杂值。

## 1.4 常见使用场景

用户每日签到/连续签到记录、亿级用户在线状态标记（0=离线，1=在线）、日活/月活用户快速统计、布隆过滤器底层实现（结合多个Bitmap位运算）。

## 1.5 新手避坑要点

避免设置过大的偏移量（如offset=1亿），会导致SDS提前扩容，占用大量空闲内存；bitcount命令在大范围统计时效率略低，可分段统计后汇总。

# 二、超日志（HyperLogLog）—— 12KB统计亿级UV

## 2.1 核心定义

HyperLogLog（简称HLL）是基于概率算法的基数统计结构，核心作用是统计海量数据的去重数量（基数），无需存储原始数据，仅占用固定约12KB内存，误差率约0.81%，适合大数据量去重场景。

## 2.2 基础使用命令

```redis
# 1. 向HyperLogLog中添加元素（自动去重，重复元素不影响统计）
pfadd page:uv:home user1 user2 user3 user1  # 向首页UV统计中添加用户，user1重复会被忽略
pfadd page:uv:list user4 user5 user2        # 向列表页UV统计中添加用户

# 2. 统计基数（去重后的元素数量，核心命令）
pfcount page:uv:home       # 返回首页独立访客数（UV）
pfcount page:uv:list       # 返回列表页独立访客数

# 3. 合并多个HyperLogLog（求并集，统计总UV）
pfmerge page:uv:total page:uv:home page:uv:list # 合并首页和列表页的UV统计
pfcount page:uv:total      # 返回合并后的总独立访客数
```

## 2.3 底层原理

HyperLogLog 基于“伯努利试验”和“哈希分桶”原理，不存储原始元素，仅记录元素哈希值的前导0个数，通过概率估算实现基数统计。

- 内存固定：无论添加多少元素，均占用约12KB内存（由16384个6bit桶组成，16384×6÷8=12KB）；

- 误差率：标准误差率为1.04÷√16384≈0.81%，可通过合并多个HLL降低误差；

- 核心局限：只能统计基数，无法获取具体元素，不适合需要精准去重并获取元素的场景。

## 2.4 常见使用场景

网站/页面独立访客数（UV）统计、短视频/文章访问量去重统计、海量日志数据去重计数、大数据量下的近似基数估算（无需精准值）。

## 2.5 新手避坑要点

不可用于需要精准去重的场景（如用户付费记录统计）；pfmerge仅支持并集合并，不支持交集、差集运算；添加元素后无法删除单个元素，只能删除整个HLL（del命令）。

# 三、地理空间（GEO）—— 基于经纬度的位置计算

## 3.1 核心定义

GEO（Geospatial）是Redis基于ZSet实现的地理位置存储与计算结构，支持存储经纬度坐标，提供距离计算、范围查询等功能，专为LBS（地理位置服务）场景设计。

## 3.2 基础使用命令

```redis
# 1. 添加地理位置（经纬度+成员名，经度范围：-180~180，纬度范围：-85.05~85.05）
geoadd city:position 116.40 39.90 beijing 121.47 31.23 shanghai # 添加北京、上海坐标

# 2. 计算两个位置的直线距离（单位：m/km/mi/ft，默认m）
geodist city:position beijing shanghai km # 计算北京到上海的直线距离（单位：公里）

# 3. 按半径查询附近的位置（核心场景：附近的人/商家）
georadius city:position 116.40 39.90 100 km withdist withcoord # 查找北京周边100公里内的城市
georadiusbymember city:position beijing 100 km withdist # 以北京为中心，查找100公里内的城市

# 4. 获取指定成员的经纬度
geopos city:position beijing shanghai # 返回北京、上海的经纬度坐标

# 5. 删除指定地理位置成员（本质是删除ZSet中的元素）
zrem city:position beijing # GEO无专属删除命令，复用ZSet的zrem命令
```

## 3.3 底层原理

GEO 底层完全复用ZSet实现，无独立数据结构，核心是将经纬度坐标通过“GeoHash编码”转换为一个64位整数，作为ZSet的score，成员名为地理位置名称。

- 编码逻辑：GeoHash编码将二维经纬度转换为一维整数，相邻的地理位置编码相近，实现范围查询；

- 排序逻辑：ZSet按GeoHash编码后的score排序，可快速筛选出指定范围的地理位置；

- 核心优势：复用ZSet的高效排序和查询能力，无需额外维护地理位置索引，实现简单且高效。

## 3.4 常见使用场景

附近的人/商家推荐、打车/外卖距离计算、地理位置签到、区域内兴趣点查询（如周边加油站、餐厅）。

## 3.5 新手避坑要点

经纬度有范围限制，超出范围会报错；georadius查询半径过大会导致效率降低，建议合理设置半径；GEO不支持复杂的地理计算（如路线规划），仅支持直线距离和范围查询。

# 四、流（Stream）—— 可靠的消息队列专用结构

## 4.1 核心定义

Stream 是Redis 5.0+ 推出的消息队列专用数据结构，支持消息持久化、消费组、消息确认（ACK）、阻塞读取等功能，解决了List作为消息队列时“无持久化、无ACK、无消费组”的缺陷，是企业级消息队列的优选方案。

## 4.2 基础使用命令

```redis
# 1. 添加消息（核心命令，生成唯一消息ID：时间戳-序列号）
xadd order:stream * orderId 1001 userId 2001 status pending # *表示自动生成消息ID
xadd order:stream 1681800000-0 orderId 1002 userId 2002 status pending # 手动指定消息ID

# 2. 读取消息（简单读取，无消费组，适合单消费者）
xread count 2 streams order:stream 0 # 从消息ID=0开始，读取2条消息（0表示从头读）
xread block 10000 count 2 streams order:stream $ # 阻塞10秒，读取最新的2条消息（$表示末尾）

# 3. 创建消费组（核心，支持多消费者负载均衡）
xgroup create order:stream group1 $ # 创建消费组group1，从最新消息开始消费（$表示末尾）

# 4. 消费组读取消息（多消费者场景）
xreadgroup group group1 consumer1 count 2 streams order:stream > # consumer1从group1读取2条未消费消息
xreadgroup group group1 consumer1 count 2 streams order:stream 0 # 读取group1中所有未消费消息（包括已读取未ACK）

# 5. 确认消息消费成功（ACK，避免消息重复消费）
xack order:stream group1 1681800000-0 1681800001-0 # 确认指定消息ID的消息消费成功

# 6. 查看消费组信息
xinfo groups order:stream # 查看order:stream的所有消费组信息
xinfo consumers order:stream group1 # 查看group1的所有消费者信息

# 7. 删除消息/Stream
xdel order:stream 1681800000-0 # 删除指定消息ID的消息
del order:stream # 删除整个Stream（连带消费组、消息全部删除）
```

## 4.3 底层原理

Stream 底层采用“消息链表+消费组索引”的结构，消息按时间戳有序存储，支持持久化（写入RDB/AOF），消费组通过维护“未消费消息指针”和“消费者状态”，实现消息的可靠消费。

- 消息存储：消息以链表形式存储，每个消息包含ID、字段-值对，支持按ID范围查询；

- 消费组机制：每个消费组维护一个“最后交付消息ID”，记录当前消费进度；每个消费者维护自己的未ACK消息列表；

- 持久化：消息会同步写入AOF和RDB，重启Redis后消息不丢失，保证消息可靠性。

## 4.4 常见使用场景

企业级可靠消息队列（如订单状态流转、支付回调通知）、日志/事件流处理、分布式系统间的异步通信、任务分发（多消费者负载均衡）。

## 4.5 新手避坑要点

必须使用Redis 5.0+ 版本，低版本不支持Stream；消费消息后必须执行xack确认，否则消息会被重新分配给其他消费者，导致重复消费；避免创建过多消费组，会增加Redis内存占用和性能开销。

# 五、特殊类型：发布订阅（Pub/Sub）

## 5.1 核心定义

发布订阅（Publish/Subscribe）是Redis提供的简单消息通信模型，基于“频道（Channel）”实现消息的一对多分发，发布者发布消息到频道，所有订阅该频道的订阅者都会收到消息，无消息持久化机制。

## 5.2 基础使用命令

```redis
# 1. 订阅频道（订阅者）
subscribe chat:channel1 # 订阅chat:channel1频道，进入阻塞状态，等待接收消息
psubscribe chat:*       # 模式匹配订阅，订阅所有以chat:开头的频道

# 2. 发布消息（发布者）
publish chat:channel1 "hello redis" # 向chat:channel1频道发布消息，返回接收消息的订阅者数量

# 3. 取消订阅
unsubscribe chat:channel1 # 取消订阅chat:channel1频道
punsubscribe chat:*       # 取消模式匹配订阅的所有频道
```

## 5.3 底层原理

Pub/Sub 底层采用“频道-订阅者”的映射表实现，Redis维护一个全局字典，key为频道名称，value为订阅该频道的所有客户端列表。

- 发布机制：发布者发布消息时，Redis遍历该频道的订阅者列表，将消息逐一发送给每个订阅者；

- 核心局限：无消息持久化，订阅者离线后会丢失期间的消息；无消息ACK机制，无法确认订阅者是否成功接收消息；

- 与Stream区别：Pub/Sub 轻量简单，适合简单通知场景；Stream 可靠稳定，适合复杂消息队列场景。

## 5.4 常见使用场景

简单实时通知（如聊天室消息、系统公告）、配置变更广播、简单的日志实时推送（无需持久化）。

## 5.5 新手避坑要点

不适合需要可靠消息的场景（如订单通知）；订阅者必须在线才能接收消息，离线消息无法恢复；模式匹配订阅（psubscribe）会匹配所有符合规则的频道，注意避免匹配范围过大导致消息冗余。

# 六、特殊类型：Redis事务（Transaction）

## 6.1 核心定义

Redis 事务是一组命令的批量执行机制，保证“原子性”（要么全部执行，要么全部不执行），但不保证“隔离性”，不支持事务回滚，适合简单的批量操作场景。

## 6.2 基础使用命令

```redis
# 1. 开启事务（标记事务开始，后续命令加入事务队列）
multi

# 2. 向事务队列中添加命令（此时命令不执行，仅入队）
set user:1001 name zhangsan
hset user:1001 age 20
incr user:1001:score

# 3. 执行事务（一次性执行队列中的所有命令）
exec

# 4. 取消事务（清空事务队列，放弃执行）
discard

# 5. 乐观锁监控（监控key，若key被修改，事务执行失败）
watch user:1001 # 监控user:1001
multi
set user:1001 name lisi
exec # 若user:1001在watch后被其他客户端修改，exec会返回nil，事务执行失败
```

## 6.3 底层原理

Redis 事务通过“事务队列”和“原子执行”实现，开启事务后，所有命令会被加入队列，执行exec时，Redis会一次性执行队列中的所有命令，期间不会被其他客户端的命令打断。

- 原子性保障：若队列中某条命令语法错误，exec会直接放弃所有命令的执行；若命令逻辑错误（如对String执行List命令），错误命令会失败，其他命令正常执行（不回滚）；

- 乐观锁机制：watch命令通过监控key的版本号，实现乐观锁，若key被其他客户端修改，事务会执行失败，避免并发修改冲突；

- 核心局限：不支持事务回滚，错误命令无法撤销已执行的命令；不支持隔离级别，事务执行期间，其他客户端可读取到未提交的中间状态。

## 6.4 常见使用场景

简单的批量操作（如批量设置多个key-value）、结合watch实现分布式锁的乐观锁机制、需要保证多个命令原子执行的简单场景（如用户积分扣减+订单状态更新）。

## 6.5 新手避坑要点

Redis事务不支持回滚，需提前校验命令语法；watch监控的key数量不宜过多，否则会增加Redis性能开销；事务队列不宜过长，避免阻塞Redis主线程。

# 七、特殊类型：Lua脚本

## 7.1 核心定义

Lua脚本是Redis支持的嵌入式脚本语言，可在Redis服务器端执行一系列命令，实现复杂的原子操作，减少网络IO，提高执行效率，同时保证脚本内所有命令的原子性。

## 7.2 基础使用命令

```redis
# 1. 执行简单Lua脚本（直接执行）
eval "return redis.call('set', 'test:lua', 'hello')" 0 # 0表示脚本不使用参数
eval "return redis.call('get', KEYS[1])" 1 test:lua   # 1表示使用1个参数，KEYS[1]对应test:lua

# 2. 脚本缓存（避免重复编译，提高效率）
script load "return redis.call('get', KEYS[1])" # 返回脚本SHA1校验码（如：a1b2c3d4）
evalsha a1b2c3d4 1 test:lua # 通过校验码执行缓存的脚本

# 3. 查看脚本是否存在
script exists a1b2c3d4 # 返回1表示存在，0表示不存在

# 4. 杀死正在执行的脚本
script kill # 仅能杀死处于阻塞状态的脚本

# 5. 实战示例：原子性扣减库存（避免超卖）
eval "local stock = redis.call('get', KEYS[1]); if tonumber(stock) > 0 then return redis.call('decr', KEYS[1]) else return 0 end" 1 goods:stock:1001
```

## 7.3 底层原理

Redis 内置Lua解释器，执行Lua脚本时，会将脚本中的所有命令一次性提交给Redis内核，期间不会被其他客户端的命令打断，保证脚本内命令的原子性。

- 脚本执行机制：脚本执行时，Redis会进入“原子执行模式”，暂停处理其他客户端的命令，直到脚本执行完毕；

- 缓存机制：script load命令会将脚本编译后缓存到Redis中，后续通过evalsha执行，避免重复编译，提高效率；

- 核心优势：原子性、减少网络IO（一次请求执行多个命令）、灵活性（支持复杂逻辑判断）。

## 7.4 常见使用场景

分布式锁的原子释放（避免死锁）、秒杀/库存扣减的原子操作（避免超卖）、复杂数据聚合计算（减少网络传输）、批量命令的原子执行。

## 7.5 新手避坑要点

脚本不宜过长或过于复杂，否则会阻塞Redis主线程；避免在脚本中使用循环（如while），防止脚本执行超时；脚本中不可使用Redis的阻塞命令（如blpop），会导致整个Redis阻塞。

# 八、高级数据结构&特殊类型核心对比

|类型/结构|核心特点|底层实现（极简）|核心场景|核心避坑点|
|---|---|---|---|---|
|Bitmap|bit存储，极致省内存，布尔状态|String（SDS）|签到、日活统计、在线状态|避免过大偏移量|
|HyperLogLog|12KB固定内存，基数统计，有误差|概率算法+分桶|UV统计、海量去重计数|不适合精准去重场景|
|GEO|存储经纬度，支持距离/范围查询|ZSet（GeoHash编码）|附近的人、LBS服务|经纬度有范围限制|
|Stream|可靠消息队列，支持ACK、消费组|消息链表+消费组索引|企业级消息队列、事件流|必须ACK消息，避免重复消费|
