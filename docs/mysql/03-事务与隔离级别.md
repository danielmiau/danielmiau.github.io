# 一、事务简介

事务（Transaction）是数据库中一个不可分割的最小操作单元，是一组连续的SQL执行语句的集合，要么全部执行成功，要么全部执行失败，不会出现“部分成功、部分失败”的情况。简单来说，事务就是“要么都做，要么都不做”，核心目的是保证数据的一致性和完整性。

生活场景类比：银行转账操作，从A账户扣除1000元，向B账户增加1000元，这两个操作必须同时成功（A扣钱、B加钱），或同时失败（A不扣钱、B不加钱），若只执行其中一个操作，会导致数据不一致（钱凭空消失或凭空增加），这就是事务要解决的问题。

MySQL中事务的核心操作（实操必备）：

```sql
-- 1. 开启事务（默认自动提交关闭后生效）
START TRANSACTION;  -- 或 BEGIN;
-- 2. 执行SQL操作（可多个）
UPDATE account SET balance = balance - 1000 WHERE id = 1;  -- A账户扣钱
UPDATE account SET balance = balance + 1000 WHERE id = 2;  -- B账户加钱
-- 3. 提交事务（所有操作生效，数据持久化）
COMMIT;
-- 4. 回滚事务（若执行失败，撤销所有操作，恢复到事务开启前状态）
ROLLBACK;

-- 补充：查看事务自动提交状态
SHOW VARIABLES LIKE 'autocommit';  -- ON表示自动提交（默认），OFF表示手动提交
-- 关闭自动提交（适合手动控制事务）
SET autocommit = OFF;
-- 开启自动提交（恢复默认）
SET autocommit = ON;

```

关键说明：MySQL默认开启自动提交（autocommit=ON），即每执行一条SQL语句，就自动视为一个独立事务，执行成功后自动提交；若需手动控制事务（多SQL组合为一个事务），需先关闭自动提交，或用START TRANSACTION开启事务。

# 二、事务四大特性（ACID，核心重点）

事务的四大特性（ACID）是事务的核心定义，也是判断一个事务是否合规的标准，四大特性相互依赖、缺一不可，共同保证数据的一致性和可靠性。本节将逐一拆解每个特性的定义、作用及实操验证，结合案例帮助理解。

## 2.1 原子性（Atomicity）—— 事务不可分割

核心定义：事务是一个不可分割的最小操作单元，事务中的所有SQL操作，要么全部执行成功（COMMIT），要么全部执行失败（ROLLBACK），不会出现“部分执行成功、部分执行失败”的中间状态。

通俗理解：事务就像一个“打包好的操作”，要么完整执行，要么全部撤销，不存在“半完成”的情况。

### 实操案例（验证原子性）

```sql
-- 准备账户表
CREATE TABLE account (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,
    balance DECIMAL(10,2) NOT NULL DEFAULT 0.00
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 插入测试数据
INSERT INTO account (name, balance) VALUES ('A账户', 5000.00), ('B账户', 3000.00);

-- 开启事务，执行转账操作（故意让第二个SQL报错）
START TRANSACTION;
-- 1. A账户扣1000元（执行成功）
UPDATE account SET balance = balance - 1000 WHERE name = 'A账户';
-- 2. B账户加1000元（故意写错字段名，执行失败）
UPDATE account SET balanc = balance + 1000 WHERE name = 'B账户';

-- 此时事务未提交，执行ROLLBACK回滚
ROLLBACK;

-- 查看账户余额（验证原子性：A账户余额恢复为5000，B账户仍为3000，无中间状态）
SELECT * FROM account;

```

关键结论：由于第二个SQL执行失败，整个事务回滚，第一个SQL的操作也被撤销，体现了事务的原子性——不可分割、要么全成、要么全败。

## 2.2 一致性（Consistency）—— 数据状态稳定

核心定义：事务执行前后，数据库中的数据总状态保持一致，即数据从一个合法状态转换到另一个合法状态，不会出现非法数据。

通俗理解：事务执行前，数据符合业务规则；事务执行后，数据依然符合业务规则，不会出现“数据矛盾”。例如转账场景，执行前A+B的总余额为8000，执行后A+B的总余额仍为8000，不会出现总余额增加或减少的情况。

补充：一致性是事务的最终目标，原子性、隔离性、持久性都是为了保证一致性而存在的。

### 实操案例（验证一致性）

```sql
-- 开启事务，执行正常转账
START TRANSACTION;
UPDATE account SET balance = balance - 1000 WHERE name = 'A账户';  -- A: 5000→4000
UPDATE account SET balance = balance + 1000 WHERE name = 'B账户';  -- B: 3000→4000
COMMIT;

-- 查看总余额（执行前总余额8000，执行后总余额仍为8000，数据一致）
SELECT SUM(balance) FROM account;

```

## 2.3 隔离性（Isolation）—— 并发互不干扰

核心定义：多个事务同时并发执行时，一个事务的执行过程不会被其他事务干扰，每个事务都感觉不到其他事务的存在，仿佛自己是唯一执行的事务。

通俗理解：多个用户同时操作数据库（如同时转账、同时查询），每个用户的事务之间相互隔离，不会出现“一个事务读取到另一个事务未提交的数据”等问题。

关键说明：隔离性是解决事务并发问题的核心，MySQL通过“隔离级别”来控制隔离程度，不同隔离级别对应不同的并发问题解决能力（后续章节详细讲解）。

## 2.4 持久性（Durability）—— 数据永久保存

核心定义：事务一旦执行COMMIT提交后，其对数据的修改就会永久保存到数据库中，即使后续数据库发生崩溃（如断电、宕机），重启后数据依然存在，不会丢失。

通俗理解：事务提交后，数据就“落地”了，无论发生什么意外，修改后的状态都会被保留。

### 实操案例（验证持久性）

```sql
-- 执行事务并提交
START TRANSACTION;
UPDATE account SET balance = balance + 500 WHERE name = 'A账户';  -- A: 4000→4500
COMMIT;

-- 模拟数据库崩溃（关闭MySQL服务，再重启）
-- 重启后查看数据（A账户余额仍为4500，修改永久保存，体现持久性）
SELECT * FROM account;

```

## 2.5 ACID四大特性总结

- 原子性：解决“事务是否可分割”的问题（要么全成，要么全败）；

- 一致性：解决“数据是否合法”的问题（执行前后数据状态稳定）；

- 隔离性：解决“并发事务是否干扰”的问题（互不影响）；

- 持久性：解决“数据是否永久保存”的问题（提交后不丢失）。

核心关联：原子性是基础，隔离性是保障，持久性是结果，三者共同支撑一致性（最终目标）。

# 三、事务并发问题（核心难点）

当多个事务同时并发执行时，若没有合适的隔离机制，会引发一系列并发问题，破坏数据的一致性和隔离性。MySQL中最常见的事务并发问题有3种：脏读、不可重复读、幻读，本节将逐一拆解每种问题的定义、场景及危害，为后续学习隔离级别做铺垫。

前提说明：以下所有案例均基于「未设置隔离级别」（默认隔离级别为Repeatable Read，但需模拟并发场景），采用两个事务（事务A、事务B）同时执行的场景进行演示。

## 3.1 脏读（Dirty Read）—— 读取未提交数据

### 定义

事务A读取了事务B已经修改但尚未提交（COMMIT）的数据，若事务B后续执行ROLLBACK回滚，那么事务A读取到的数据就是“无效数据”（脏数据），这种情况称为脏读。

### 场景演示（脏读）

```sql
-- 初始数据：A账户余额4500，B账户余额4000
-- 事务A（读取数据）                事务B（修改数据）
1. START TRANSACTION;             1. START TRANSACTION;
2. SELECT balance FROM account    2. UPDATE account SET balance = balance - 800 
   WHERE name = 'A账户';          WHERE name = 'A账户';  -- A: 4500→3700（未提交）
   -- 此时事务A读取到的余额是3700（事务B未提交的数据）
3. 等待事务B操作                  3. ROLLBACK;  -- 事务B回滚，A账户余额恢复为4500
4. SELECT balance FROM account    -- 事务A再次读取，余额变为4500
   WHERE name = 'A账户';

```

### 危害

事务A基于“脏数据”进行后续操作（如计算、决策），会导致业务逻辑错误，例如事务A读取到A账户3700元，误以为余额不足，拒绝执行后续操作，而实际余额为4500元。

## 3.2 不可重复读（Non-Repeatable Read）—— 同一事务多次读取结果不一致

### 定义

事务A在同一个事务中，多次读取同一批数据，期间事务B修改并提交了该批数据，导致事务A多次读取的结果不一致，这种情况称为不可重复读。

关键区别：脏读是读取“未提交”的数据，不可重复读是读取“已提交”的数据，但多次读取结果不同（核心是“修改”）。

### 场景演示（不可重复读）

```sql
-- 初始数据：A账户余额4500
-- 事务A（多次读取）                事务B（修改并提交）
1. START TRANSACTION;             1. START TRANSACTION;
2. SELECT balance FROM account    2. UPDATE account SET balance = balance + 300 
   WHERE name = 'A账户';          WHERE name = 'A账户';  -- A: 4500→4800
   -- 第一次读取：4500             3. COMMIT;  -- 事务B提交，修改生效
3. 等待事务B操作                  
4. SELECT balance FROM account    -- 第二次读取：4800（与第一次不一致）
   WHERE name = 'A账户';

```

### 危害

破坏事务A的一致性，导致事务A内部逻辑混乱。例如事务A需要基于同一批数据进行多次计算，多次读取结果不同会导致计算结果错误。

## 3.3 幻读（Phantom Read）—— 同一事务多次查询行数不一致

### 定义

事务A在同一个事务中，多次执行相同的查询语句（如查询满足条件的行数），期间事务B插入或删除了满足该查询条件的数据并提交，导致事务A多次查询的行数不一致，这种情况称为幻读。

关键区别：不可重复读是“修改”导致的结果不一致，幻读是“插入/删除”导致的行数不一致（核心是“新增/删除”）。

### 场景演示（幻读）

```sql
-- 初始数据：account表有2条数据（A账户、B账户）
-- 事务A（多次查询行数）            事务B（插入数据并提交）
1. START TRANSACTION;             1. START TRANSACTION;
2. SELECT COUNT(*) FROM account;  2. INSERT INTO account (name, balance)
   -- 第一次查询：2行              VALUES ('C账户', 2000.00);
3. 等待事务B操作                  3. COMMIT;  -- 事务B提交，新增数据生效
4. SELECT COUNT(*) FROM account;  -- 第二次查询：3行（与第一次不一致，出现“幻影”数据）

```

### 危害

导致事务A对数据总量的判断错误，影响业务逻辑。例如事务A统计账户总数并进行分页操作，期间新增数据会导致分页错乱、统计结果错误。

## 3.4 并发问题总结（对比记忆）

|并发问题|核心原因|关键特征|危害|
|---|---|---|---|
|脏读|读取未提交的数据|数据无效（可能被回滚）|业务逻辑错误|
|不可重复读|读取已提交的修改数据|同一数据多次读取结果不同|事务内部逻辑混乱|
|幻读|读取已提交的新增/删除数据|同一查询多次行数不同|数据统计、分页错误|
核心结论：并发问题的本质是“多个事务同时操作同一批数据”，且缺乏有效的隔离机制。MySQL通过设置不同的“事务隔离级别”，来解决不同的并发问题（隔离级别越高，并发问题解决越彻底，但性能开销越大）。

# 四、四大事务隔离级别详解（实战重点）

为了解决事务并发问题，MySQL定义了四大事务隔离级别（从低到高），不同隔离级别对应不同的并发问题解决能力，同时也对应不同的性能开销。隔离级别越高，并发问题解决越彻底，但数据库的并发性能越低（因为需要更多的锁或其他机制来保证隔离）。

MySQL四大隔离级别（按隔离程度从低到高）：

1. 读未提交（Read Uncommitted, RU）

2. 读已提交（Read Committed, RC）

3. 可重复读（Repeatable Read, RR）—— MySQL默认隔离级别

4. 串行化（Serializable）

实操基础：查看当前事务隔离级别、修改隔离级别

```sql
-- 查看当前会话隔离级别
SELECT @@session.tx_isolation;
-- 查看全局隔离级别
SELECT @@global.tx_isolation;

-- 修改当前会话隔离级别（仅当前会话生效）
SET SESSION TRANSACTION ISOLATION LEVEL 隔离级别;
-- 修改全局隔离级别（需重启会话生效）
SET GLOBAL TRANSACTION ISOLATION LEVEL 隔离级别;

-- 示例：将当前会话隔离级别设置为读已提交
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

```

## 4.1 读未提交（Read Uncommitted, RU）—— 最低隔离级别

### 核心规则

允许事务读取其他事务“未提交”的数据，不做任何隔离控制。

### 并发问题解决能力

✅ 无任何并发问题解决能力，会出现：脏读、不可重复读、幻读（所有并发问题都存在）。

### 性能

最高（无需任何隔离机制，并发性能最好）。

### 适用场景

极少使用，仅适用于“对数据一致性要求极低、追求极致并发性能”的场景（如临时统计无关紧要的日志数据）。

## 4.2 读已提交（Read Committed, RC）—— 常用隔离级别

### 核心规则

禁止事务读取其他事务“未提交”的数据，只能读取“已提交”的数据（解决脏读）。

### 并发问题解决能力

✅ 解决脏读； ❌ 仍会出现：不可重复读、幻读。

### 核心原理

每次读取数据时，都会生成一个新的“ReadView”（读视图），只能看到已提交的事务数据，看不到未提交的事务数据（后续章节详解ReadView）。

### 性能

中等（需维护ReadView，性能略低于RU，但远高于RR和串行化）。

### 适用场景

最常用的隔离级别之一，适用于“对数据一致性有一定要求，但追求较高并发性能”的场景（如电商订单、用户登录等大部分业务场景），主流互联网公司大多采用此隔离级别。

### 实操验证（解决脏读）

```sql
-- 1. 设置当前会话隔离级别为读已提交
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 事务A（读取数据）                事务B（修改数据）
1. START TRANSACTION;             1. START TRANSACTION;
2. SELECT balance FROM account    2. UPDATE account SET balance = balance - 500 
   WHERE name = 'A账户';          WHERE name = 'A账户';  -- A: 4500→4000（未提交）
   -- 此时事务A读取到的仍是4500（看不到未提交数据，解决脏读）
3. 等待事务B操作                  3. COMMIT;  -- 事务B提交
4. SELECT balance FROM account    -- 读取到4000（已提交数据）
   WHERE name = 'A账户';

```

## 4.3 可重复读（Repeatable Read, RR）—— MySQL默认隔离级别

### 核心规则

禁止事务读取其他事务未提交的数据，且在同一个事务中，多次读取同一批数据的结果保持一致（解决脏读、不可重复读）。

### 并发问题解决能力

✅ 解决脏读、不可重复读； ❌ 理论上会出现幻读，但MySQL InnoDB引擎通过MVCC机制，实际避免了幻读。

### 核心原理

一个事务开启后，只生成一次ReadView（读视图），整个事务期间都使用这一个ReadView，因此多次读取同一批数据的结果保持一致（后续结合MVCC详解）。

### 性能

中等偏低（需维护MVCC机制和ReadView，性能低于RC，但高于串行化）。

### 适用场景

MySQL默认隔离级别，适用于“对数据一致性要求较高，并发性能要求适中”的场景（如金融转账、支付等核心业务场景）。

### 实操验证（解决不可重复读）

```sql
-- 1. 设置当前会话隔离级别为可重复读（默认就是，可省略）
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 事务A（多次读取）                事务B（修改并提交）
1. START TRANSACTION;             1. START TRANSACTION;
2. SELECT balance FROM account    2. UPDATE account SET balance = balance + 600 
   WHERE name = 'A账户';          WHERE name = 'A账户';  -- A: 4000→4600
   -- 第一次读取：4000             3. COMMIT;  -- 事务B提交
3. 等待事务B操作                  
4. SELECT balance FROM account    -- 第二次读取：仍为4000（结果一致，解决不可重复读）
   WHERE name = 'A账户';
5. COMMIT;  -- 事务A提交后，再次读取，才能看到4600
6. SELECT balance FROM account WHERE name = 'A账户';  -- 4600

```

## 4.4 串行化（Serializable）—— 最高隔离级别

### 核心规则

将所有事务按顺序执行（串行执行），禁止任何并发操作，一个事务执行完，另一个事务才能开始执行。

### 并发问题解决能力

✅ 解决所有并发问题（脏读、不可重复读、幻读），数据一致性最高。

### 核心原理

通过“表级锁”实现，事务执行时会锁定整个表，其他事务无法对该表进行任何操作（读、写），直到当前事务执行完成并释放锁。

### 性能

最低（完全禁止并发，适用于并发量极低的场景）。

### 适用场景

极少使用，仅适用于“对数据一致性要求极高、并发量极低”的场景（如银行核心转账、财务对账等，且需严格保证数据零错误）。

### 实操验证（解决所有并发问题）

```sql
-- 1. 设置当前会话隔离级别为串行化
SET SESSION TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- 事务A（执行查询）                事务B（执行插入）
1. START TRANSACTION;             1. START TRANSACTION;
2. SELECT COUNT(*) FROM account;  2. INSERT INTO account (name, balance)
   -- 事务A锁定account表           VALUES ('D账户', 3000.00);
                                  -- 事务B阻塞，等待事务A释放锁
3. COMMIT;  -- 事务A提交，释放锁    3. 锁释放后，事务B执行插入，提交

```

## 4.5 四大隔离级别总结（必记）

|隔离级别|解决的并发问题|未解决的并发问题|性能|适用场景|
|---|---|---|---|---|
|读未提交（RU）|无|脏读、不可重复读、幻读|最高|极低一致性要求场景|
|读已提交（RC）|脏读|不可重复读、幻读|中等|大部分业务场景（主流）|
|可重复读（RR）|脏读、不可重复读|理论幻读（InnoDB避免）|中等偏低|核心业务、金融场景（默认）|
|串行化|所有并发问题|无|最低|极高一致性、低并发场景|
# 五、InnoDB MVCC 底层实现原理（核心难点）

MVCC（Multi-Version Concurrency Control），即多版本并发控制，是InnoDB引擎实现事务隔离级别的核心机制，也是解决“并发读-写冲突”的关键。其核心思想是：**为每个数据行维护多个版本，不同事务读取不同版本的数据，从而实现并发读和写互不干扰**，既保证隔离性，又提升并发性能。

关键优势：MVCC实现了“读不加锁、写不阻塞读”，即一个事务在修改数据时，其他事务可以正常读取数据（读取旧版本），无需等待写事务提交，大幅提升数据库并发性能。

MVCC仅适用于 InnoDB 引擎，且支持的隔离级别为：读已提交（RC）和可重复读（RR）（读未提交无需MVCC，串行化用锁机制，不依赖MVCC）。

## 5.1 MVCC 核心组成部分

MVCC的实现依赖三大核心组件，三者协同工作，实现多版本数据的管理和读取：

1. 数据行的隐藏字段（InnoDB自动添加）；

2. undo log（回滚日志）；

3. ReadView（读视图）。

## 5.2 数据行的隐藏字段

InnoDB引擎会为表中的每一行数据，自动添加3个隐藏字段（用户无法直接查看），用于维护数据的多版本信息：

- DB_TRX_ID（事务ID）：记录修改该数据行的最后一个事务的ID（64位）；

- DB_ROLL_PTR（回滚指针）：指向该数据行的上一个版本（存储在undo log中），形成版本链；

- DB_ROW_ID（行ID）：若表没有主键或唯一非空索引，InnoDB会自动生成该字段，作为隐藏主键（自增）。

示例（数据行版本链示意）：

当一个数据行被多次修改时，会通过DB_ROLL_PTR形成一条版本链，每个版本都存储在undo log中，最新版本存储在表中：

```Plain Text
当前数据行（最新版本）
DB_TRX_ID = 100（最后修改的事务ID）
DB_ROLL_PTR → 指向 undo log 中的版本1

undo log 版本1（上一个版本）
DB_TRX_ID = 90（修改该版本的事务ID）
DB_ROLL_PTR → 指向 undo log 中的版本2

undo log 版本2（最早版本）
DB_TRX_ID = 80（修改该版本的事务ID）
DB_ROLL_PTR = NULL（无更早版本）

```

## 5.3 undo log（回滚日志）

### 核心定义

undo log（回滚日志）是InnoDB用于存储数据行“旧版本”的日志，当事务修改数据时，会先将数据的旧版本写入undo log，再修改表中的数据。undo log不仅是MVCC的核心组件，也是事务回滚（ROLLBACK）的基础。

### 核心作用

- 支持事务回滚：当事务执行ROLLBACK时，通过undo log中的旧版本数据，恢复到事务开启前的状态；

- 支持MVCC多版本读取：事务读取数据时，若当前版本不可见（被其他事务锁定或未提交），则通过undo log的版本链，找到可见的旧版本数据。

### undo log 类型

undo log分为两种类型，对应不同的操作：

1. INSERT undo log：记录插入操作的旧版本（插入的新数据，旧版本为空），事务提交后可直接删除（因为插入的数据只有当前事务可见，提交后其他事务无需读取该版本）；

2. UPDATE/DELETE undo log：记录更新、删除操作的旧版本，事务提交后不会立即删除，需等待所有依赖该版本的事务读取完成后，由InnoDB的purge线程（清理线程）统一清理。

## 5.4 ReadView（读视图）

ReadView（读视图）是MVCC的核心，本质是一个“当前活跃事务的快照”，用于判断当前事务能看到哪些版本的数据（即数据的可见性判断）。

ReadView中包含4个核心参数，用于可见性判断：

- m_ids：当前活跃的所有事务ID的集合（即正在执行、未提交的事务ID）；

- min_trx_id：m_ids中最小的事务ID；

- max_trx_id：当前数据库中即将生成的下一个事务ID（不是m_ids中的最大ID，而是全局下一个ID）；

- creator_trx_id：创建当前ReadView的事务ID（即当前执行读取操作的事务ID）。

## 5.5 MVCC 工作流程（核心逻辑）

MVCC的工作流程分为“数据修改”和“数据读取”两个部分，结合隐藏字段、undo log、ReadView协同工作，具体流程如下：

### 5.5.1 数据修改流程（事务写操作）

1. 事务开启，获取一个唯一的事务ID（TRX_ID）；

2. 修改数据时，先将数据的当前版本（旧版本）写入undo log，并记录该版本的DB_TRX_ID（当前事务ID）和DB_ROLL_PTR（指向之前的旧版本）；

3. 修改表中的数据，将数据行的DB_TRX_ID更新为当前事务ID，DB_ROLL_PTR指向undo log中的旧版本；

4. 事务提交后，数据行的最新版本保留在表中，undo log中的旧版本由purge线程后续清理。

### 5.5.2 数据读取流程（事务读操作）

事务读取数据时，会先创建ReadView，然后通过ReadView的可见性规则，判断数据行的当前版本是否可见；若不可见，则通过DB_ROLL_PTR遍历undo log的版本链，找到可见的旧版本。

ReadView可见性判断规则（核心必记）：

1. 若数据行的DB_TRX_ID = creator_trx_id → 可见（当前事务修改的数据，自己可见）；

2. 若数据行的DB_TRX_ID < min_trx_id → 可见（该版本由已提交的事务修改，所有事务都可见）；

3. 若数据行的DB_TRX_ID > max_trx_id → 不可见（该版本由未来的事务修改，当前事务无法看到）；

4. 若 min_trx_id ≤ DB_TRX_ID ≤ max_trx_id → 查看DB_TRX_ID是否在m_ids中：
        

    - 在m_ids中 → 不可见（该版本由未提交的活跃事务修改）；

    - 不在m_ids中 → 可见（该版本由已提交的事务修改）。

### 5.5.3 实操案例（MVCC工作流程演示）

```sql
-- 初始数据：A账户 balance = 4500，DB_TRX_ID = 0（初始版本），DB_ROLL_PTR = NULL
-- 事务A（TRX_ID = 100）            事务B（TRX_ID = 200）
1. START TRANSACTION;             1. START TRANSACTION;
2. UPDATE account SET balance = 4000 WHERE name = 'A账户';
   -- 操作：将旧版本（4500，TRX_ID=0）写入undo log，
   -- 表中数据更新为4000，DB_TRX_ID=100，DB_ROLL_PTR指向undo log旧版本
3. 未提交                         2. SELECT balance FROM account WHERE name = 'A账户';
                                  -- 操作：创建ReadView，m_ids = [100, 200]，min_trx_id=100，max_trx_id=201，creator_trx_id=200
                                  -- 可见性判断：数据行DB_TRX_ID=100，在m_ids中 → 不可见
                                  -- 遍历版本链：通过DB_ROLL_PTR找到undo log中的旧版本（4500，TRX_ID=0）
                                  -- 判断旧版本：DB_TRX_ID=0 < min_trx_id=100 → 可见，读取到4500
4. COMMIT;  -- 事务A提交，TRX_ID=100从m_ids中移除
                                  3. SELECT balance FROM account WHERE name = 'A账户';
                                  -- 操作：RC隔离级别会重新创建ReadView，m_ids = [200]，min_trx_id=200，max_trx_id=201
                                  -- 可见性判断：数据行DB_TRX_ID=100 < min_trx_id=200 → 可见，读取到4000
                                  -- （RR隔离级别不会重新创建ReadView，仍读取到4500）

```

## 5.6 MVCC 与隔离级别的关联（核心重点）

MVCC在不同隔离级别中的工作方式不同，核心区别在于“ReadView的创建时机”，这也是RC和RR隔离级别解决并发问题能力不同的根本原因：

1. 读已提交（RC）：**每次执行读取操作时，都会创建一个新的ReadView**。因此，每次读取都能看到已提交的最新数据，但同一事务中多次读取可能看到不同结果（不可重复读）；

2. 可重复读（RR）：**事务开启后，第一次执行读取操作时创建ReadView，整个事务期间都使用这一个ReadView**。因此，同一事务中多次读取的结果保持一致（解决不可重复读），且InnoDB通过版本链遍历，避免了幻读。

# 六、ReadView / undo log 机制详解（补充重点）

上一节讲解了ReadView和undo log在MVCC中的作用，本节进一步拆解两者的核心机制、工作细节，帮助学习者深入理解MVCC的底层逻辑，同时掌握相关面试高频考点。

## 6.1 ReadView 深度解析

### 6.1.1 ReadView 的创建时机（再强调）

ReadView的创建时机是区分RC和RR隔离级别的关键，务必牢记：

- RC隔离级别：**每次SELECT查询时，都创建一个新的ReadView**；

- RR隔离级别：**事务中第一次SELECT查询时，创建一个ReadView，后续所有SELECT都复用该ReadView**。

示例验证（RR隔离级别ReadView复用）：

```sql
-- 事务A（TRX_ID=100）            事务B（TRX_ID=200，RR隔离级别）
1. START TRANSACTION;             1. START TRANSACTION;
2. UPDATE account SET balance = 4000 WHERE name = 'A账户';  -- 未提交
                                  2. SELECT balance FROM account;  -- 第一次查询，创建ReadView，读取到旧版本4500
3. COMMIT;  -- 事务A提交
                                  3. SELECT balance FROM account;  -- 复用ReadView，仍读取到4500
                                  4. COMMIT;  -- 事务B提交
                                  5. SELECT balance FROM account;  -- 重新创建ReadView，读取到4000

```

### 6.1.2 ReadView 的销毁时机

ReadView的生命周期与创建它的事务绑定：

- 事务提交或回滚后，该事务创建的ReadView会被销毁；

- RR隔离级别中，事务期间复用的ReadView，在事务结束后统一销毁。

## 6.2 undo log 深度解析

### 6.2.1 undo log 的存储方式

undo log存储在InnoDB的表空间（ibdata1）中，分为两个部分：

1. undo segment（undo段）：用于存储undo log的物理空间，每个事务会分配一个独立的undo segment；

2. undo log page（undo页）：undo segment由多个undo页组成，每个undo页存储多个undo log记录。

### 6.2.2 undo log 的清理机制（purge线程）

undo log中的旧版本数据不会立即删除，因为可能有其他事务需要读取这些旧版本（MVCC多版本读取）。InnoDB通过专门的purge线程（后台清理线程），定期清理“不再被任何事务引用”的undo log数据。

清理条件（undo log可被清理的前提）：

- 修改该undo log对应数据的事务已提交；

- 所有依赖该undo log旧版本的事务（即所有ReadView中包含该事务ID的事务）已结束（提交或回滚）。

### 6.2.3 undo log 与事务回滚的关联

事务执行ROLLBACK时，InnoDB会根据当前事务的TRX_ID，遍历undo log的版本链，找到该事务修改前的数据版本，将表中的数据恢复到修改前的状态。

示例（事务回滚与undo log）：

```sql
-- 事务A（TRX_ID=100）
1. START TRANSACTION;
2. UPDATE account SET balance = 4000 WHERE name = 'A账户';  -- 旧版本（4500）写入undo log
3. UPDATE account SET balance = 3500 WHERE name = 'A账户';  -- 旧版本（4000）写入undo log，版本链：3500→4000→4500
4. ROLLBACK;  -- 事务回滚
-- 执行逻辑：遍历undo log版本链，依次恢复数据，最终恢复到4500

```

## 6.3 ReadView 与 undo log 协同工作总结

ReadView负责“判断数据版本是否可见”，undo log负责“存储数据的旧版本，形成版本链”，两者协同工作，实现MVCC的核心功能：

1. 事务修改数据时，通过undo log记录旧版本，形成版本链；

2. 事务读取数据时，通过ReadView判断当前版本是否可见；

3. 若当前版本不可见，通过undo log的版本链，遍历找到可见的旧版本；

4. 事务结束后，ReadView被销毁，undo log中无用的旧版本被purge线程清理。
