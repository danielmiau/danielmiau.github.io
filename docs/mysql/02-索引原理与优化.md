# 02-索引原理与优化

前言：本章为MySQL学习文档的第二章，聚焦索引的核心原理与优化技巧，是MySQL进阶学习的关键内容。索引作为MySQL提升查询性能的核心手段，贯穿后续事务、锁机制、性能优化等章节，学好索引能从根本上解决查询效率问题。本章内容涵盖索引底层数据结构（B+Tree详解）、核心索引类型（聚簇/非聚簇/联合索引）、最左匹配原则、索引失效场景及避坑技巧、索引设计最佳实践，全程搭配图解思路和实操案例，既兼顾理论深度，又贴合生产实战，帮助学习者从“会用索引”提升到“用好索引”，为后续高性能数据库开发和运维筑牢基础。

# 一、索引简介

索引（Index）是MySQL中用于快速查询数据的一种数据结构，本质是“排好序的快速查找数据结构”。类比书籍的目录，书籍目录能帮助我们快速定位到目标章节，而MySQL索引能帮助数据库引擎快速定位到表中目标数据，避免全表扫描（逐行查找数据），从而大幅提升查询效率。

核心作用：

- 提升查询效率：将全表扫描的O(n)时间复杂度，优化为索引查询的O(log n)时间复杂度，数据量越大，索引的优势越明显；

- 降低数据库IO压力：无需遍历全表，减少磁盘读取次数，降低服务器资源消耗；

- 辅助排序：索引本身是排好序的，查询时若需排序，可直接利用索引顺序，避免额外排序操作。

注意事项：索引并非越多越好，索引会占用额外的磁盘空间，且在执行INSERT、UPDATE、DELETE操作时，需要同步维护索引（更新索引结构），会增加写操作的开销。因此，索引设计的核心是“平衡查询效率和写操作开销”，按需创建索引。

补充：查看表的索引：`SHOW INDEX FROM 表名;`（如`SHOW INDEX FROM user;`）；创建索引：`CREATE INDEX 索引名 ON 表名(字段名);`；删除索引：`DROP INDEX 索引名 ON 表名;`。

# 二、索引底层数据结构：B+Tree 详解（核心重点）

MySQL索引的底层数据结构主流是B+Tree（B+树），部分场景会用到Hash索引（如Memory存储引擎），但B+Tree索引适配绝大多数业务场景，是学习的核心。本节将从B+Tree的结构、特性、优势三个维度，拆解其底层原理，让学习者理解“索引为什么能提升查询效率”。

先明确两个基础概念：

- 节点：B+Tree的基本组成单元，分为根节点、叶子节点、非叶子节点（中间节点）；

- 页：MySQL中数据存储的最小单位（默认16KB），每个节点对应一个磁盘页，节点的分裂、合并均以页为单位，减少磁盘IO。

## 2.1 B+Tree 的结构（图解思路）

B+Tree是一种平衡多路查找树，结构分为三层，自上而下依次为：根节点 → 非叶子节点 → 叶子节点，每层节点之间通过指针关联，具体结构如下：

1. 根节点：位于最顶层，仅存储索引键和指向非叶子节点的指针，不存储具体数据（数据只存于叶子节点）；

2. 非叶子节点（中间节点）：仅存储索引键和指针，用于导航查找，指针指向子节点，每个节点可存储多个索引键和对应指针（数量由索引键大小决定）；

3. 叶子节点：位于最底层，存储索引键和对应的数据地址（或具体数据，取决于索引类型），所有叶子节点通过双向链表连接，便于范围查询（如`WHERE age BETWEEN 20 AND 30`）。

关键细节：B+Tree的高度通常为2-4层（因为每个节点可存储多个索引键，16KB的节点可存储上千个索引键），意味着查询任意数据，最多只需2-4次磁盘IO，而全表扫描可能需要上百次、上千次磁盘IO，这也是索引提升效率的核心原因。

## 2.2 B+Tree 的核心特性

- 平衡树：左右子树的高度差不超过1，保证查询时的磁盘IO次数稳定（不会出现极端情况）；

- 多路查找：每个节点可拥有多个子节点（通常是100-1000个，取决于索引键大小），相比二叉树（每个节点最多2个子节点），层数更少，磁盘IO更少；

- 数据集中存储：所有具体数据仅存于叶子节点，非叶子节点仅用于导航，减少非叶子节点的存储空间，提升索引查询效率；

- 叶子节点有序且联动：叶子节点按索引键排序，且通过双向链表连接，支持快速范围查询、排序查询，这是B+Tree相比B树、Hash索引的核心优势。

## 2.3 B+Tree 与其他数据结构的对比（避坑）

为什么MySQL选择B+Tree作为主流索引结构？对比以下三种常见数据结构，就能明确其优势：

|数据结构|优势|劣势|MySQL中的应用场景|
|---|---|---|---|
|B+Tree|层数少、磁盘IO少；支持范围查询、排序；数据集中，查询稳定|插入/删除时需维护树的平衡，略有开销|主流场景（InnoDB/MyISAM存储引擎），适配绝大多数查询|
|B树|结构简单，查询单个数据效率尚可|非叶子节点存储数据，占用空间大；层数多，磁盘IO多；不支持高效范围查询|几乎不用，被B+Tree替代|
|Hash索引|等值查询（如`WHERE id = 1`）效率极高，O(1)时间复杂度|不支持范围查询、排序；不支持模糊查询；Hash冲突会影响效率|Memory存储引擎（临时表），仅适用于纯等值查询场景|
结论：B+Tree完美适配MySQL的查询场景（等值查询、范围查询、排序查询），是平衡查询效率和存储开销的最优选择，也是后续所有索引类型的底层基础。

# 三、核心索引类型：聚簇索引 / 非聚簇索引 / 联合索引

基于B+Tree底层结构，MySQL将索引分为三大核心类型，不同类型的索引存储方式、查询逻辑、适用场景不同，核心区别在于“数据是否与索引存储在一起”，本节将逐一拆解，结合实操案例说明其用法。

## 3.1 聚簇索引（Clustered Index）

聚簇索引又称“聚集索引”，核心特点：**索引键与数据存储在一起**，叶子节点存储的是完整的行数据，而非数据地址。InnoDB存储引擎默认会以主键（PRIMARY KEY）作为聚簇索引，若表中没有主键，则会选择唯一非空索引作为聚簇索引；若既没有主键，也没有唯一非空索引，InnoDB会自动生成一个隐藏的聚簇索引（基于行号）。

### 3.1.1 聚簇索引的核心特性

- 一个表只能有一个聚簇索引（因为数据只能按一种顺序存储）；

- 叶子节点存储完整行数据，查询时找到索引键，即可直接获取对应数据，无需二次查找（“索引即数据”）；

- 数据按聚簇索引的顺序存储，因此范围查询、排序查询效率极高（无需额外排序，直接利用索引顺序）；

- 插入数据时，需按聚簇索引顺序插入，若插入的是无序主键（如雪花ID），会导致索引页分裂，增加写操作开销；若插入的是自增主键（如INT AUTO_INCREMENT），则插入效率更高（无需分裂页）。

### 3.1.2 实操案例（基于InnoDB）

```sql
-- 创建用户表，主键id作为聚簇索引
CREATE TABLE user (
    id INT PRIMARY KEY AUTO_INCREMENT,  -- 聚簇索引（默认）
    name VARCHAR(50) NOT NULL,
    age INT DEFAULT 0,
    mobile VARCHAR(11)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 插入数据（自增主键，插入顺序与聚簇索引顺序一致，效率高）
INSERT INTO user (name, age, mobile) VALUES 
('张三', 25, '13800138000'),
('李四', 28, '13900139000'),
('王五', 22, '13700137000');

-- 按聚簇索引查询（id），直接从叶子节点获取数据，效率极高
SELECT * FROM user WHERE id = 2;  -- 聚簇索引查询，无需二次查找

```

## 3.2 非聚簇索引（Non-Clustered Index）

非聚簇索引又称“二级索引”“辅助索引”，核心特点：**索引键与数据分开存储**，叶子节点存储的是聚簇索引键（而非完整行数据），查询时需先通过非聚簇索引找到聚簇索引键，再通过聚簇索引找到对应数据，这个过程称为“回表”。

补充：MyISAM存储引擎的所有索引都是非聚簇索引（包括主键索引），因为MyISAM的索引和数据是分开存储的；InnoDB的非聚簇索引，叶子节点存储的是聚簇索引键（主键），依赖聚簇索引实现查询。

### 3.2.1 非聚簇索引的核心特性

- 一个表可以有多个非聚簇索引（不受限制，只要不影响写操作性能）；

- 叶子节点存储聚簇索引键，不存储完整数据，因此非聚簇索引占用的磁盘空间比聚簇索引小；

- 查询流程：非聚簇索引 → 聚簇索引 → 行数据（回表），比聚簇索引多一次磁盘IO，查询效率略低；

- 适用场景：频繁用于查询条件，但不是主键的字段（如按name、age查询）。

### 3.2.2 实操案例（基于InnoDB）

```sql
-- 给user表的name字段创建非聚簇索引
CREATE INDEX idx_user_name ON user(name);

-- 按非聚簇索引查询（name），会触发回表
SELECT * FROM user WHERE name = '李四';
-- 查询流程：idx_user_name索引找到name='李四'对应的id（聚簇索引键）→ 通过聚簇索引id找到完整行数据

-- 优化：若查询的字段仅为聚簇索引键和非聚簇索引键，无需回表（覆盖索引）
SELECT id, name FROM user WHERE name = '李四';  -- 无需回表，效率提升

```

关键补充：覆盖索引（Covering Index）—— 若查询的字段全部包含在非聚簇索引中（包括聚簇索引键），则无需回表，直接从非聚簇索引获取数据，是提升非聚簇索引查询效率的核心技巧。

## 3.3 联合索引（Composite Index）

联合索引又称“复合索引”，是指基于多个字段创建的索引（如`CREATE INDEX idx_age_name ON user(age, name);`），底层仍为B+Tree结构，索引键按“第一个字段→第二个字段→...→第N个字段”的顺序排序，核心遵循“最左匹配原则”（后续章节详细讲解）。

### 3.3.1 联合索引的核心特性

- 联合索引的排序规则：先按第一个字段排序，第一个字段相同的情况下，再按第二个字段排序，以此类推；

- 可以替代多个单一字段索引，减少索引数量，降低写操作开销（如idx_age_name可替代idx_age和idx_name，但需满足最左匹配）；

- 查询效率取决于是否遵循最左匹配原则，遵循则能利用索引，不遵循则索引失效；

- 适用场景：频繁使用多个字段组合作为查询条件（如`WHERE age = 25 AND name LIKE '张%'`）。

### 3.3.2 实操案例

```sql
-- 给user表创建联合索引（age, name）
CREATE INDEX idx_age_name ON user(age, name);

-- 遵循最左匹配，能利用联合索引（查询条件包含第一个字段age）
SELECT * FROM user WHERE age = 25;  -- 利用索引
SELECT * FROM user WHERE age = 25 AND name = '张三';  -- 利用索引
SELECT * FROM user WHERE age = 25 AND name LIKE '张%';  -- 利用索引

-- 不遵循最左匹配，索引失效（查询条件不包含第一个字段age）
SELECT * FROM user WHERE name = '张三';  -- 索引失效，全表扫描

```

# 四、最左匹配原则（核心难点）

最左匹配原则是联合索引的核心规则，直接决定联合索引是否能被利用，其核心定义：**联合索引的查询条件，必须从最左边的字段开始，且不能跳过中间字段，否则索引失效**。简单来说，联合索引`(a, b, c)`，能匹配的查询条件是“a”“a+b”“a+b+c”，无法匹配“b”“c”“b+c”。

## 4.1 最左匹配原则的底层逻辑

联合索引的B+Tree，是按“第一个字段→第二个字段→第三个字段”的顺序排序的，类比字典排序（先按首字母，首字母相同按第二个字母，以此类推）。若查询时跳过第一个字段，就相当于在字典中跳过首字母查找，无法利用排序好的索引结构，只能进行全表扫描。

示例：联合索引`(age, name, mobile)`的排序逻辑：

1. 先按age升序排列；

2. age相同的情况下，按name升序排列；

3. age和name都相同的情况下，按mobile升序排列。

因此，只有查询条件包含age（最左字段），才能利用索引；若跳过age，直接按name查询，无法利用索引的排序结构，索引失效。

## 4.2 最左匹配原则的常见场景（实操重点）

以联合索引`idx_age_name_mobile (age, name, mobile)`为例，逐一分析不同查询条件是否能利用索引：

|查询条件|是否利用索引|说明|
|---|---|---|
|WHERE age = 25|是|匹配最左字段，利用索引|
|WHERE age = 25 AND name = '张三'|是|匹配最左两个字段，利用索引|
|WHERE age = 25 AND name = '张三' AND mobile = '13800138000'|是|匹配所有字段，利用索引（效率最高）|
|WHERE age = 25 AND mobile = '13800138000'|是（部分利用）|匹配最左字段age，mobile字段无法利用索引，仅age字段生效|
|WHERE name = '张三'|否|跳过最左字段age，索引失效，全表扫描|
|WHERE name = '张三' AND mobile = '13800138000'|否|跳过最左字段age，索引失效，全表扫描|
|WHERE age > 25 AND name = '张三'|是（部分利用）|age字段范围查询，name字段无法利用索引，仅age字段生效|
## 4.3 最左匹配原则的避坑技巧

- 1.  联合索引的字段顺序，应将“查询频率高、区分度高”的字段放在最左边（区分度：字段值不重复的比例，如id区分度100%，age区分度低于id）；

- 2.  避免跳过中间字段查询，若频繁需要按“b+c”查询，可单独创建联合索引`(b, c)`，而非依赖`(a, b, c)`；

- 3.  范围查询（>、<、BETWEEN、LIKE %xxx）会导致后续字段无法利用索引，因此范围查询字段应放在联合索引的最右边；

- 4.  字段顺序不影响最左匹配（MySQL会自动调整查询条件顺序），如`WHERE name = '张三' AND age = 25`，MySQL会自动调整为`WHERE age = 25 AND name = '张三'`，仍能利用索引。

# 五、索引失效常见场景与避坑（实战重点）

创建索引后，若查询条件不当，会导致索引失效，无法发挥索引的作用，反而增加存储和写操作开销。本节总结生产中最常见的索引失效场景，结合案例说明原因和避坑方法，是索引优化的核心内容。

## 5.1 常见索引失效场景（必记）

### 场景1：函数或表达式操作索引字段

若对索引字段进行函数运算、表达式操作，MySQL无法利用索引，会触发全表扫描。

```sql
-- 索引：idx_user_age (age)
-- 失效场景：对age进行函数操作
SELECT * FROM user WHERE YEAR(age) = 25;  -- 索引失效（YEAR()函数操作索引字段）
SELECT * FROM user WHERE age + 1 = 26;    -- 索引失效（表达式操作索引字段）

-- 优化方案：避免对索引字段进行函数/表达式操作，调整查询条件
SELECT * FROM user WHERE age = 25;        -- 正常利用索引
SELECT * FROM user WHERE age = 26 - 1;    -- 表达式移到等号右边，利用索引

```

### 场景2：隐式类型转换

索引字段的类型与查询条件的值类型不匹配，MySQL会进行隐式类型转换，导致索引失效。最常见的场景：字符串类型索引字段，查询时用数字值。

```sql
-- 索引：idx_user_mobile (mobile)，mobile是VARCHAR(11)类型
-- 失效场景：查询条件用数字（隐式转换为字符串）
SELECT * FROM user WHERE mobile = 13800138000;  -- 索引失效（隐式类型转换）

-- 优化方案：查询条件的值类型与索引字段一致
SELECT * FROM user WHERE mobile = '13800138000';  -- 正常利用索引

```

### 场景3：模糊查询以%开头

字符串模糊查询时，若以`%`开头（如`LIKE '%张三'`），索引失效；若以`%`结尾（如`LIKE '张三%'`），索引有效；若前后都有`%`（如`LIKE '%张三%'`），索引失效。

```sql
-- 索引：idx_user_name (name)
-- 失效场景
SELECT * FROM user WHERE name LIKE '%张三';  -- 以%开头，索引失效
SELECT * FROM user WHERE name LIKE '%张三%'; -- 前后都有%，索引失效

-- 有效场景
SELECT * FROM user WHERE name LIKE '张三%';  -- 以%结尾，索引有效

-- 优化方案：若需模糊匹配前后都有%，可使用全文索引（InnoDB 8.0+支持）
CREATE FULLTEXT INDEX idx_user_name_full ON user(name);
SELECT * FROM user WHERE MATCH(name) AGAINST('张三');

```

### 场景4：OR连接的查询条件，部分字段无索引

若OR连接的多个查询条件中，有一个字段没有索引，则整个查询会触发全表扫描，所有索引失效。

```sql
-- 索引：idx_user_age (age)，name无索引
-- 失效场景：OR连接age（有索引）和name（无索引）
SELECT * FROM user WHERE age = 25 OR name = '张三';  -- 索引失效，全表扫描

-- 优化方案：给所有OR连接的字段创建索引，或拆分查询
-- 方案1：给name创建索引
CREATE INDEX idx_user_name ON user(name);
SELECT * FROM user WHERE age = 25 OR name = '张三';  -- 正常利用索引

-- 方案2：拆分查询
SELECT * FROM user WHERE age = 25
UNION ALL
SELECT * FROM user WHERE name = '张三';

```

### 场景5：联合索引不遵循最左匹配原则

如前文所述，联合索引必须从最左字段开始查询，跳过中间字段会导致索引失效（具体案例见第四章）。

### 场景6：NULL值判断（视情况失效）

若索引字段允许NULL值，查询`WHERE 字段 IS NULL`可能会使索引失效（取决于MySQL优化器判断），建议给索引字段设置默认值（如字符串默认''，数值默认0），避免NULL值。

```sql
-- 索引：idx_user_age (age)，age允许NULL
-- 可能失效场景
SELECT * FROM user WHERE age IS NULL;  -- 可能索引失效，全表扫描

-- 优化方案：给age设置默认值0，避免NULL值
ALTER TABLE user MODIFY COLUMN age INT DEFAULT 0;
SELECT * FROM user WHERE age = 0;  -- 正常利用索引

```

## 5.2 索引失效排查技巧（实战必备）

若怀疑索引失效，可使用`EXPLAIN`关键字分析查询语句，查看是否利用索引（重点看`type`和`key`字段）。

```sql
-- 用EXPLAIN分析查询语句
EXPLAIN SELECT * FROM user WHERE name LIKE '%张三';

-- 关键字段解读：
-- type：查询类型，ALL表示全表扫描（索引失效），ref、range表示利用索引；
-- key：实际使用的索引，NULL表示未使用索引；
-- rows：预计扫描的行数，行数越多，查询效率越低。

```

排查步骤：执行`EXPLAIN` → 查看`key`是否为目标索引 → 若为NULL，说明索引失效 → 对照上述场景，排查查询条件是否存在问题。

# 六、索引设计最佳实践（生产实战）

索引设计的核心是“平衡查询效率和写操作开销”，并非索引越多越好，也不是盲目创建索引。本节结合生产场景，总结索引设计的最佳实践，帮助学习者设计出高效、合理的索引，避免常见误区。

## 6.1 核心设计原则

- 1.  按需创建索引：只给“频繁用于查询、排序、分组”的字段创建索引，避免给“频繁更新、插入”的字段创建索引（如日志表的操作时间字段，若频繁插入，索引会增加写开销）；

- 2.  优先使用联合索引替代多个单一索引：若多个字段频繁组合查询，创建联合索引（如`idx_age_name`），比创建两个单一索引（`idx_age`、`idx_name`）更节省空间，且查询效率更高；

- 3.  联合索引字段顺序：“查询频率高、区分度高”的字段放在最左边，范围查询字段放在最右边；

- 4.  避免冗余索引：若已创建联合索引`(a, b)`，则无需再创建单一索引`(a)`（联合索引的最左字段可单独利用索引）；

- 5.  控制索引数量：一张表的索引数量建议不超过5个，过多索引会导致插入、更新、删除操作变慢，且占用大量磁盘空间；

- 6.  避免使用字符串作为主键：字符串主键的区分度虽高，但索引存储和查询效率低于自增主键（INT AUTO_INCREMENT），建议用自增主键或雪花ID作为主键；

- 7.  利用覆盖索引：查询时尽量只查询需要的字段，若查询字段能被索引覆盖（包含在索引中），可避免回表，提升查询效率；

- 8.  避免过度索引：对于数据量小的表（如少于1000行），无需创建索引（全表扫描效率更高，索引反而增加开销）。

## 6.2 不同场景的索引设计案例

### 案例1：用户表（user）设计

业务场景：频繁按id查询（主键）、按mobile查询（唯一）、按age+name组合查询，很少更新mobile、age字段。

```sql
CREATE TABLE user (
    id INT PRIMARY KEY AUTO_INCREMENT,  -- 聚簇索引（自增主键，高效）
    name VARCHAR(50) NOT NULL,
    age INT DEFAULT 0,
    mobile VARCHAR(11) NOT NULL UNIQUE,  -- 唯一索引（替代单一索引，保证唯一性）
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 创建联合索引（age, name），适配age+name组合查询
CREATE INDEX idx_age_name ON user(age, name);

```

### 案例2：订单表（order）设计

业务场景：频繁按order_id查询（主键）、按user_id查询（关联用户）、按create_time范围查询，order_id为雪花ID（BIGINT）。

```sql
CREATE TABLE `order` (
    order_id BIGINT PRIMARY KEY,  -- 聚簇索引（雪花ID，适配分布式场景）
    user_id INT NOT NULL,
    order_amount DECIMAL(10,2) NOT NULL,
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP,
    status TINYINT DEFAULT 0,  -- 0-待支付，1-已支付，2-已取消
    -- 关联用户表，创建非聚簇索引
    INDEX idx_user_id (user_id),
    -- 联合索引（create_time, status），适配按时间范围+状态查询
    INDEX idx_create_time_status (create_time, status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

```

## 6.3 常见设计误区（避坑必看）

- 误区1：给所有字段创建索引 → 导致写操作开销剧增，磁盘空间浪费；

- 误区2：联合索引字段顺序随意排列 → 导致最左匹配无法利用，索引失效；

- 误区3：忽略覆盖索引 → 频繁查询`SELECT *`，导致回表，查询效率低；

- 误区4：主键用字符串 → 索引查询效率低，且插入时容易导致索引页分裂；

- 误区5：数据量小时创建索引 → 全表扫描效率高于索引查询，索引反而增加开销。
