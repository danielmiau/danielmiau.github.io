# 一、锁机制简介

锁（Lock）是MySQL用于控制多个事务并发访问同一资源（表、数据行）的一种机制，核心作用是**防止并发事务之间的操作冲突**，保证数据的一致性和完整性。简单来说，锁就像“资源占用标识”，一个事务获取锁后，其他事务需等待该事务释放锁，才能操作对应的资源，从而避免并发修改、读取导致的数据错乱。

核心应用场景：当多个事务同时操作同一批数据时（如同时修改同一行数据、同时删除同一表的数据），通过锁机制控制操作顺序，避免出现“数据覆盖”“脏写”等问题。例如，两个事务同时给同一用户转账，锁机制会保证两个转账操作依次执行，不会出现“扣钱重复、加钱遗漏”的情况。

MySQL锁的核心分类（按不同维度）：

1. 按锁的粒度（锁定范围）：表锁、行锁、意向锁；

2. 按锁的兼容性（是否允许其他事务获取同一锁）：共享锁、排他锁；

3. 按锁的实现方式（并发控制思路）：乐观锁、悲观锁。

关键说明：不同存储引擎对锁的支持不同——InnoDB引擎支持表锁、行锁、意向锁，是主流的锁机制实现；MyISAM引擎仅支持表锁，不支持行锁，因此并发性能较差，目前已基本被InnoDB替代。本章所有内容均基于InnoDB引擎讲解。

# 二、核心锁类型（按粒度分类）—— 表锁、行锁、意向锁

按锁定的资源范围（粒度）划分，MySQL最核心的锁类型是表锁和行锁，意向锁是InnoDB为了协调表锁和行锁而引入的辅助锁，三者协同工作，平衡并发性能和锁开销。

## 2.1 表锁（Table Lock）—— 粒度最大的锁

### 核心定义

表锁是锁定整个数据表的锁，获取表锁后，事务可以操作表中的所有数据行，其他事务需等待该锁释放后，才能对该表执行任何操作（读、写）。表锁的粒度最大，实现简单，锁开销小，但并发性能极低（完全限制表级别的并发）。

### 核心特性

- 锁定范围：整个数据表，不区分具体数据行；

- 锁开销：小（无需遍历数据行，直接锁定整个表）；

- 并发性能：极低（同一时间只能有一个事务持有表锁，其他事务阻塞）；

- 适用场景：适合批量操作（如批量更新、批量删除），或表数据量极小的场景（如配置表），可减少锁的开销。

### 实操案例（表锁的使用）

```sql
-- 准备测试表（InnoDB引擎）
CREATE TABLE product (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,
    stock INT NOT NULL DEFAULT 0
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 插入测试数据
INSERT INTO product (name, stock) VALUES ('手机', 100), ('电脑', 50), ('平板', 80);

-- 手动获取表锁（读锁/写锁）
-- 1. 获取表读锁（共享锁）：允许其他事务读，禁止写
LOCK TABLE product READ;
-- 执行读操作（允许）
SELECT * FROM product;
-- 执行写操作（禁止，会阻塞）
UPDATE product SET stock = 90 WHERE id = 1;
-- 释放表锁
UNLOCK TABLES;

-- 2. 获取表写锁（排他锁）：禁止其他事务读、写
LOCK TABLE product WRITE;
-- 执行写操作（允许）
UPDATE product SET stock = 90 WHERE id = 1;
-- 执行读操作（允许，当前事务）
SELECT * FROM product;
-- 其他事务执行读/写操作（禁止，会阻塞）
-- 释放表锁
UNLOCK TABLES;

```

关键注意：InnoDB引擎中，手动获取表锁后，当前事务只能操作被锁定的表，无法操作其他表；且释放表锁时，会释放当前事务持有的所有表锁（UNLOCK TABLES 是全局释放）。

## 2.2 行锁（Row Lock）—— 粒度最小的锁

### 核心定义

行锁是锁定数据表中某一行或多行数据的锁，获取行锁后，事务仅能操作被锁定的数据行，其他事务可以正常操作表中未被锁定的数据行。行锁的粒度最小，并发性能极高，但锁开销大（需遍历数据行，定位到具体行）。

关键说明：InnoDB的行锁是基于“索引”实现的——若查询语句使用了索引（聚簇索引、非聚簇索引），则锁定对应的数据行；若查询语句未使用索引（全表扫描），则InnoDB会自动升级为表锁，导致并发性能下降（这是行锁使用的核心坑点）。

### 核心特性

- 锁定范围：数据表中的某一行或多行数据；

- 锁开销：大（需通过索引定位数据行，维护行锁状态）；

- 并发性能：极高（仅锁定需要操作的行，其他行可正常并发操作）；

- 适用场景：适合高频并发操作（如电商下单、用户转账），且操作的数据行分散的场景。

### 实操案例（行锁的使用与坑点）

```sql
-- 场景1：使用索引，触发行锁（正常情况）
-- 事务A                          事务B
1. START TRANSACTION;             1. START TRANSACTION;
2. UPDATE product SET stock = 90  2. UPDATE product SET stock = 49 
   WHERE id = 1;  -- id是主键（聚簇索引），锁定id=1的行   WHERE id = 2;  -- 锁定id=2的行，正常执行（不阻塞）
3. 未提交                         3. COMMIT;  -- 事务B执行成功
4. COMMIT;  -- 事务A执行成功

-- 场景2：未使用索引，行锁升级为表锁（坑点）
-- 事务A                          事务B
1. START TRANSACTION;             1. START TRANSACTION;
2. UPDATE product SET stock = 89  2. UPDATE product SET stock = 79
   WHERE name = '手机';  -- name无索引，全表扫描，升级为表锁   WHERE id = 3;  -- 被表锁阻塞，无法执行
3. 未提交                         
4. COMMIT;  -- 事务A提交，释放表锁，事务B才能执行

```

避坑技巧：使用行锁时，务必保证查询语句使用索引（主键、唯一索引、普通索引均可），避免全表扫描导致行锁升级为表锁，降低并发性能。

## 2.3 意向锁（Intention Lock）—— 表锁与行锁的协调者

### 核心定义

意向锁是InnoDB引擎为了协调表锁和行锁而引入的“辅助锁”，本质是表级锁，用于表示“某事务正在锁定表中的某些行”，或“即将锁定表中的某些行”，避免表锁和行锁之间的冲突。

核心作用：当一个事务要获取表锁时，无需遍历所有数据行判断是否有行锁（开销极大），只需判断该表是否有意向锁即可——若有，则说明表中存在行锁，表锁需等待；若无，则可直接获取表锁。

### 意向锁的类型（两种）

1. 意向共享锁（IS锁）：事务即将获取某行的共享锁（读锁），先给表添加IS锁；

2. 意向排他锁（IX锁）：事务即将获取某行的排他锁（写锁），先给表添加IX锁。

### 核心特性

- 意向锁是表级锁，不影响数据的读写操作（仅用于锁协调）；

- 意向锁自动添加、自动释放：事务获取行锁时，InnoDB自动给表添加对应意向锁；事务提交/回滚时，自动释放意向锁；

- 用户无需手动操作意向锁，由InnoDB引擎自动管理。

### 实操案例（意向锁的自动协调）

```sql
-- 事务A（获取行排他锁，自动添加IX锁）
1. START TRANSACTION;
2. UPDATE product SET stock = 88 WHERE id = 1;  -- 自动给product表添加IX锁，锁定id=1行
3. 未提交

-- 事务B（获取表读锁，判断意向锁）
1. START TRANSACTION;
2. LOCK TABLE product READ;  -- 检测到product表有IX锁，阻塞，等待事务A释放锁
3. 事务A COMMIT;  -- 释放行锁和IX锁
4. 事务B获取表读锁，执行查询：SELECT * FROM product;

```

关键结论：意向锁的核心价值是“减少锁判断的开销”，协调表锁和行锁的并发，避免表锁和行锁之间的冲突，是InnoDB行锁高效工作的基础。

## 2.4 三种锁的对比（必记）

|锁类型|锁定范围|锁开销|并发性能|适用场景|手动操作|
|---|---|---|---|---|---|
|表锁|整个表|小|极低|批量操作、小表|可手动（LOCK TABLE）|
|行锁|单行/多行|大|极高|高频并发、分散操作|自动（基于索引）|
|意向锁|整个表|极小|无影响|协调表锁与行锁|自动（InnoDB管理）|
# 三、共享锁 & 排他锁（按兼容性分类）

按锁的兼容性（即多个事务能否同时持有同一资源的锁）划分，MySQL的锁分为共享锁和排他锁，两者是实现事务隔离级别的核心，也是并发控制的基础。核心原则：**共享锁与共享锁兼容，共享锁与排他锁不兼容，排他锁与排他锁不兼容**。

## 3.1 共享锁（Shared Lock，S锁）—— 读锁

### 核心定义

共享锁又称“读锁”，是用于读取数据的锁。事务获取某资源的共享锁后，只能对该资源执行读操作（SELECT），不能执行写操作（INSERT、UPDATE、DELETE）；同时，其他事务也可以获取该资源的共享锁，共同读取数据（即“多事务并发读”）。

### 核心特性

- 兼容性：与共享锁兼容（多个事务可同时持有同一资源的S锁），与排他锁不兼容（一个事务持有S锁，其他事务无法持有X锁）；

- 操作权限：仅允许读操作，禁止写操作；

- 手动获取：通过`SELECT ... FOR SHARE`（MySQL 8.0+）或`SELECT ... LOCK IN SHARE MODE`（MySQL 5.x）获取；

- 自动释放：事务提交或回滚时，自动释放S锁。

### 实操案例（共享锁的使用）

```sql
-- 事务A（获取共享锁，读数据）
1. START TRANSACTION;
2. SELECT * FROM product WHERE id = 1 FOR SHARE;  -- 获取id=1行的S锁
3. 未提交（持有S锁）

-- 事务B（获取共享锁，读数据）
1. START TRANSACTION;
2. SELECT * FROM product WHERE id = 1 FOR SHARE;  -- 可正常获取S锁，并发读（兼容）
3. 未提交

-- 事务C（获取排他锁，写数据）
1. START TRANSACTION;
2. UPDATE product SET stock = 87 WHERE id = 1;  -- 需获取X锁，与S锁不兼容，阻塞
3. 事务A、B COMMIT;  -- 释放S锁
4. 事务C获取X锁，执行更新操作

```

## 3.2 排他锁（Exclusive Lock，X锁）—— 写锁

### 核心定义

排他锁又称“写锁”，是用于修改数据的锁。事务获取某资源的排他锁后，既能对该资源执行读操作，也能执行写操作；同时，其他事务无法获取该资源的任何锁（无论是S锁还是X锁），只能等待该事务释放X锁。

### 核心特性

- 兼容性：与任何锁都不兼容（一个事务持有X锁，其他事务无法持有S锁或X锁）；

- 操作权限：允许读、写操作（当前事务）；

- 自动获取：执行INSERT、UPDATE、DELETE操作时，InnoDB会自动给操作的数据行添加X锁；

- 手动获取：通过`SELECT ... FOR UPDATE`获取；

- 自动释放：事务提交或回滚时，自动释放X锁。

### 实操案例（排他锁的使用）

```sql
-- 事务A（获取排他锁，写数据）
1. START TRANSACTION;
2. UPDATE product SET stock = 86 WHERE id = 1;  -- 自动获取id=1行的X锁
3. 未提交（持有X锁）

-- 事务B（获取共享锁，读数据）
1. START TRANSACTION;
2. SELECT * FROM product WHERE id = 1 FOR SHARE;  -- 与X锁不兼容，阻塞

-- 事务C（获取排他锁，写数据）
1. START TRANSACTION;
2. UPDATE product SET stock = 85 WHERE id = 1;  -- 与X锁不兼容，阻塞

-- 事务A COMMIT;  -- 释放X锁
-- 事务B、C依次获取锁，执行操作

```

## 3.3 共享锁与排他锁的兼容性总结（必记）

|当前锁\请求锁|共享锁（S锁）|排他锁（X锁）|
|---|---|---|
|共享锁（S锁）|兼容（可共存）|不兼容（阻塞）|
|排他锁（X锁）|不兼容（阻塞）|不兼容（阻塞）|
核心结论：读操作（S锁）可以并发执行，写操作（X锁）会阻塞所有读、写操作，这是MySQL保证并发数据一致性的核心逻辑——避免“读-写冲突”“写-写冲突”。

# 四、乐观锁与悲观锁实战（按实现方式分类）

乐观锁与悲观锁是两种不同的并发控制思路，并非MySQL内置的锁类型，而是基于MySQL的锁机制或其他方式实现的“并发控制策略”。两者适用于不同的并发场景，核心区别在于“是否默认认为并发冲突会发生”。

## 4.1 悲观锁（Pessimistic Lock）—— 假设冲突会发生

### 核心思想

悲观锁的核心思想是“悲观地认为，并发操作一定会发生冲突”，因此在操作数据前，先获取锁，锁定资源，禁止其他事务操作，直到当前事务完成操作并释放锁。

实现方式：基于MySQL内置的锁机制（行锁、表锁、共享锁、排他锁）实现，本质是通过“提前加锁”来避免冲突。

### 核心特性

- 优点：并发冲突控制效果好，能有效避免脏写、不可重复读等问题，数据一致性高；

- 缺点：锁开销大，会导致事务阻塞，降低并发性能；若锁使用不当，容易引发死锁。

### 实战场景（悲观锁的应用）

适用场景：并发量高、写操作频繁、冲突概率高的场景（如电商下单、库存扣减、转账等核心业务）。

```sql
-- 场景：电商库存扣减（悲观锁实现，避免超卖）
-- 事务A（用户1下单，扣减库存）
1. START TRANSACTION;
2. -- 获取行排他锁，锁定id=1的商品（避免其他事务同时扣减）
   SELECT * FROM product WHERE id = 1 FOR UPDATE;
3. -- 检查库存
   IF (SELECT stock FROM product WHERE id = 1) > 0 THEN
       UPDATE product SET stock = stock - 1 WHERE id = 1;  -- 扣减库存
       -- 后续执行下单逻辑（创建订单等）
   END IF;
4. COMMIT;  -- 释放锁

-- 事务B（用户2下单，扣减库存）
1. START TRANSACTION;
2. SELECT * FROM product WHERE id = 1 FOR UPDATE;  -- 若事务A未提交，此处阻塞
3. -- 事务A提交后，获取锁，检查库存，执行扣减
4. COMMIT;

```

## 4.2 乐观锁（Optimistic Lock）—— 假设冲突不会发生

### 核心思想

乐观锁的核心思想是“乐观地认为，并发操作不会发生冲突”，因此在操作数据前，不获取锁，直接执行操作，仅在提交事务时，检查数据是否被其他事务修改过——若未被修改，则提交成功；若已被修改，则回滚事务，重新尝试。

实现方式：无需依赖MySQL内置锁，通过“版本号机制”或“时间戳机制”实现，核心是“提交时校验数据一致性”。

### 核心特性

- 优点：无锁开销，并发性能高，不会导致事务阻塞；

- 缺点：无法完全避免冲突，若冲突概率高，会导致频繁回滚，影响业务效率；需手动实现版本校验逻辑。

### 实战场景（乐观锁的应用）

适用场景：并发量低、写操作少、冲突概率低的场景（如用户资料修改、商品信息编辑等）。

```sql
-- 步骤1：给表添加版本号字段（乐观锁核心）
ALTER TABLE product ADD COLUMN version INT NOT NULL DEFAULT 1;

-- 场景：商品信息编辑（乐观锁实现）
-- 事务A（编辑商品名称）
1. START TRANSACTION;
2. -- 查询商品信息（获取当前版本号）
   SELECT id, name, version FROM product WHERE id = 1;  -- 此时version=1
3. -- 编辑商品名称（模拟业务操作）
   UPDATE product 
   SET name = '高端手机', version = version + 1  -- 版本号+1
   WHERE id = 1 AND version = 1;  -- 提交时校验版本号（核心）
4. -- 判断更新行数，若为0，说明数据已被修改，回滚；否则提交
   IF ROW_COUNT() = 0 THEN
       ROLLBACK;  -- 回滚，重新尝试
   ELSE
       COMMIT;  -- 提交成功，version变为2
   END IF;

-- 事务B（同时编辑商品名称）
1. START TRANSACTION;
2. SELECT id, name, version FROM product WHERE id = 1;  -- 此时version=1（事务A未提交）
3. UPDATE product 
   SET name = '旗舰手机', version = version + 1
   WHERE id = 1 AND version = 1;  -- 事务A已提交，version变为2，此处更新行数为0
4. ROLLBACK;  -- 回滚，重新查询最新数据，再次尝试
```

关键说明：乐观锁的核心是“版本号校验”——每次修改数据时，版本号+1，提交时判断版本号是否与查询时一致，一致则提交，不一致则回滚，从而避免并发修改冲突。

## 4.3 乐观锁与悲观锁对比（实战选型必看）

|锁类型|核心思想|实现方式|优点|缺点|适用场景|
|---|---|---|---|---|---|
|悲观锁|假设冲突会发生|MySQL内置锁（行锁、X锁等）|一致性高，避免冲突|锁开销大，易阻塞、死锁|高并发、写频繁、冲突高（下单、转账）|
|乐观锁|假设冲突不会发生|版本号/时间戳机制|无锁开销，并发高|冲突时需回滚，一致性一般|低并发、写少、冲突低（资料编辑）|
# 五、死锁（核心难点与实战重点）

死锁（Deadlock）是MySQL锁机制中最常见的问题之一，指两个或多个事务相互持有对方需要的锁，且都无法释放自己的锁，导致所有事务都处于阻塞状态，无法继续执行，最终只能通过外力（如数据库自动检测、手动终止）解决。

通俗类比：两个人面对面过窄桥，都走到桥中间，都要求对方后退，自己才能前进，双方都不后退，最终陷入僵持——这就是死锁。

## 5.1 死锁产生的必要条件（必记）

死锁的产生必须同时满足以下4个条件，缺一不可（只要破坏其中一个条件，就能避免死锁）：

1. 互斥条件：资源（锁）具有排他性，一个事务持有锁后，其他事务无法同时持有该锁；

2. 请求与保持条件：事务已经持有一个锁，又去请求另一个被其他事务持有的锁，且不释放自己已持有的锁；

3. 不可剥夺条件：事务持有的锁，无法被其他事务强制剥夺，只能由持有锁的事务主动释放；

4. 循环等待条件：多个事务之间形成循环等待关系（如事务A持有锁1，请求锁2；事务B持有锁2，请求锁1）。

## 5.2 死锁产生的常见场景（实操演示）

死锁最常见的场景是“两个事务，交叉获取不同资源的锁”，以下结合实操案例演示死锁的产生过程：

```sql
-- 测试表：product（id=1、2、3），account（id=1、2）
-- 事务A                          事务B
1. START TRANSACTION;             1. START TRANSACTION;
2. -- 持有product表id=1的X锁     2. -- 持有account表id=1的X锁
   UPDATE product SET stock = 85 WHERE id = 1;     UPDATE account SET balance = 5000 WHERE id = 1;
3. -- 请求account表id=1的X锁     3. -- 请求product表id=1的X锁
   UPDATE account SET balance = 4900 WHERE id = 1;     UPDATE product SET stock = 84 WHERE id = 1;
-- 此时：事务A持有product.id=1的X锁，请求account.id=1的X锁；
-- 事务B持有account.id=1的X锁，请求product.id=1的X锁；
-- 双方相互等待，形成死锁，均无法执行，也无法释放锁。

```

关键总结：死锁的核心是“交叉锁请求”，多个事务按不同顺序获取锁，最终形成循环等待。

## 5.3 死锁的排查方案（实战必备）

当数据库出现死锁时，需快速排查死锁原因、定位死锁事务，然后终止死锁事务，恢复数据库正常运行。MySQL提供了多种死锁排查工具和命令，以下是最常用的3种方式：

### 5.3.1 方式1：查看死锁日志（最核心）

MySQL会记录死锁相关信息到错误日志中，通过查看错误日志，可获取死锁事务的SQL语句、锁信息等，定位死锁原因。

```sql
-- 1. 查看错误日志路径（MySQL配置）
SHOW VARIABLES LIKE 'log_error';

-- 2. 查看死锁日志（直接读取错误日志文件，或通过以下命令快速查看）
SHOW ENGINE INNODB STATUS;

```

关键解读：执行`SHOW ENGINE INNODB STATUS;`后，找到“LATEST DETECTED DEADLOCK”部分，里面包含死锁事务的ID、SQL语句、持有锁、请求锁等信息，可直接定位死锁原因。

### 5.3.2 方式2：查看当前运行的事务

通过查看当前活跃事务，找到阻塞的事务，进而定位死锁事务。

```sql
-- 查看当前所有事务
SELECT * FROM INFORMATION_SCHEMA.INNODB_TRX;

-- 关键字段解读：
-- trx_id：事务ID（用于终止事务）
-- trx_state：事务状态（LOCK WAIT表示阻塞）
-- trx_wait_lock_id：等待的锁ID
-- trx_query：事务执行的SQL语句

```

### 5.3.3 方式3：终止死锁事务（解决办法）

排查到死锁事务后，通过事务ID终止死锁事务（通常终止执行时间较短、影响较小的事务），释放锁，解除死锁。

```sql
-- 1. 找到死锁事务的trx_id（通过方式2获取）
-- 2. 终止事务（释放锁）
KILL 事务ID;  -- 示例：KILL 12345;

```

## 5.4 死锁的避免策略（实战重点）

死锁的核心是“无法提前预测，但可以提前避免”，结合死锁的4个必要条件，总结以下5个实战避免策略（优先使用前3个，覆盖大部分场景）：

1. 策略1：统一锁获取顺序（破坏循环等待条件）—— 最有效
        所有事务获取锁的顺序保持一致，例如：所有事务都先获取product表的锁，再获取account表的锁；先获取id小的数据行锁，再获取id大的数据行锁，避免交叉锁请求。`-- 正确示例（统一锁顺序）
    -- 事务A                          事务B
1. START TRANSACTION;             1. START TRANSACTION;
2. UPDATE product SET stock = 85 WHERE id = 1;     UPDATE product SET stock = 84 WHERE id = 1;  -- 先获取product锁
3. UPDATE account SET balance = 4900 WHERE id = 1;     UPDATE account SET balance = 4800 WHERE id = 1;  -- 再获取account锁
4. COMMIT;                         4. COMMIT;
`

2. 策略2：减少锁持有时间（破坏请求与保持条件）事务中，尽量减少锁的持有时间：先执行非锁操作（如查询、计算），最后执行锁操作（如更新、删除）；事务执行完成后，立即提交，避免长时间持有锁。

3. 策略3：避免一次性获取多个锁（破坏请求与保持条件）
        尽量避免一个事务同时获取多个锁，若必须获取多个锁，可拆分为多个小事务，分批次执行，减少锁的冲突概率。

4. 策略4：使用乐观锁替代悲观锁（破坏互斥条件）
        在冲突概率低的场景，用乐观锁（版本号机制）替代悲观锁，避免加锁操作，从根本上减少死锁的可能。

5. 策略5：设置锁超时时间（破坏不可剥夺条件）
       通过设置锁超时时间，当事务等待锁超过指定时间后，自动释放已持有的锁，避免无限等待。MySQL默认锁超时时间为50秒，可通过以下命令修改：`-- 设置锁超时时间（单位：秒）
   SET GLOBAL innodb_lock_wait_timeout = 30;  -- 全局生效，重启会话生效
   SET SESSION innodb_lock_wait_timeout = 30;  -- 当前会话生效
   `
