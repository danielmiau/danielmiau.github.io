# 一、高可用与主从复制核心概念

在讲解具体技术之前，先明确两个核心概念，理解“为什么需要主从复制”“高可用的核心目标是什么”，为后续学习奠定基础。

## 1.1 高可用（High Availability）

### 核心定义

MySQL高可用，指数据库系统在规定时间内（如99.99%以上）能够正常提供服务，尽可能减少因硬件故障、软件异常、网络中断等问题导致的业务中断时间，核心目标是**减少停机时间、保障数据一致性、提升系统稳定性**。

通俗理解：生产环境中，单台MySQL服务器一旦出现故障（如硬件损坏、系统崩溃），所有依赖该数据库的业务都会中断，造成巨大损失；高可用架构就是通过“多节点部署”，让一台节点故障时，其他节点能快速接管服务，实现“无感知切换”，避免业务中断。

### 高可用的核心指标

- 可用性：通常用“停机时间”衡量，如99.99%的可用性意味着每年停机时间不超过52.56分钟；

- 数据一致性：故障切换后，新的主节点数据与原主节点数据保持一致，避免数据丢失或错乱；

- 切换效率：故障发生后，能快速完成节点切换，尽可能减少业务中断时间（理想状态下秒级切换）。

## 1.2 主从复制（Master-Slave Replication）

### 核心定义

主从复制是MySQL实现高可用的基础技术，指将一台MySQL服务器（主节点，Master）的数据，实时同步到一台或多台MySQL服务器（从节点，Slave），主节点负责处理写操作（INSERT、UPDATE、DELETE），从节点负责处理读操作（SELECT），实现“读写分离”和“数据备份”双重目标。

### 主从复制的核心价值（为什么需要主从复制）

1. 读写分离：分担主节点压力，主节点专注写操作，从节点承接读操作（如报表查询、高频读请求），提升系统并发处理能力；

2. 数据备份：从节点是主节点的实时备份，当主节点故障时，可快速将从节点切换为主节点，避免数据丢失；

3. 高可用支撑：基于主从复制搭建多节点架构，是实现故障切换、集群部署的基础；

4. 业务隔离：从节点可用于数据统计、日志分析等非核心业务，避免影响主节点的核心业务性能。

关键说明：主从复制的核心是“数据同步”，所有写操作都在主节点执行，然后同步到从节点；读操作可在从节点执行，实现负载均衡。本章所有内容均基于InnoDB引擎讲解，且主从节点MySQL版本保持一致（避免版本差异导致同步失败）。

# 二、主从复制原理与同步流程（核心重点）

MySQL主从复制的核心原理是“基于二进制日志（binlog）的异步同步”，主节点将写操作记录到binlog，从节点读取binlog并执行相同的操作，从而实现主从数据一致。整个同步流程分为3个核心步骤，涉及3个关键线程（主节点2个、从节点1个）。

## 2.1 核心线程（必记）

主从复制的正常运行，依赖3个核心线程的协同工作，缺一不可：

1. 主节点：二进制日志线程（binlog dump thread）
        负责将主节点的写操作（INSERT、UPDATE、DELETE）记录到二进制日志（binlog）中，同时监听从节点的请求，将binlog日志内容发送给从节点。

2. 从节点：IO线程（IO thread）
        负责连接主节点，请求主节点的binlog日志，将主节点发送的binlog日志写入到从节点的中继日志（relay log）中。

3. 从节点：SQL线程（SQL thread）
        负责读取中继日志（relay log）中的内容，解析并执行其中的SQL语句，将主节点的写操作同步到从节点，保证主从数据一致。

补充说明：中继日志（relay log）是从节点的“临时日志”，作用是缓存主节点的binlog日志，避免因网络波动、主节点压力大导致的同步中断，SQL线程可异步读取中继日志执行，不影响IO线程获取binlog。

## 2.2 完整同步流程（图文逻辑拆解）

主从复制的完整流程可分为5个步骤，清晰易懂，结合线程协同理解：

1. 主节点执行写操作：用户在主节点执行INSERT、UPDATE、DELETE等写操作，主节点的binlog dump线程将该操作记录到二进制日志（binlog）中，标记为“未同步”状态；

2. 从节点请求binlog：从节点的IO线程主动连接主节点，请求主节点发送最新的binlog日志（指定起始位置）；

3. 主节点发送binlog：主节点的binlog dump线程接收请求后，将未同步的binlog日志内容发送给从节点的IO线程；

4. 从节点写入中继日志：从节点的IO线程将接收的binlog日志内容，写入到本地的中继日志（relay log）中，并记录binlog的同步位置（避免重复同步）；

5. 从节点执行同步操作：从节点的SQL线程实时读取中继日志中的SQL语句，解析并执行，将写操作同步到从节点，执行完成后，标记该部分binlog为“已同步”，主从数据保持一致。

关键总结：整个同步流程的核心是“binlog日志的传递与执行”，主节点负责“记录日志”，从节点负责“获取日志、执行日志”，3个线程协同工作，实现数据同步。同步延迟是主从复制的常见问题（后续会讲解优化方案），本质是从节点IO线程获取日志、SQL线程执行日志需要时间。

## 2.3 主从复制实操配置（基础版）

以下是单主单从（1主1从）的基础配置步骤，基于MySQL 8.0版本，实操性极强，适合入门练习：

### 前提准备

- 两台MySQL服务器（主节点：192.168.1.100，从节点：192.168.1.101），版本一致（均为8.0+）；

- 两台服务器网络互通，关闭防火墙（或开放MySQL端口3306）；

- 主节点已开启binlog日志（核心前提）。

### 步骤1：配置主节点（Master）

```sql
-- 1. 编辑MySQL配置文件（my.cnf 或 my.ini）
[mysqld]
server-id = 100  -- 主节点唯一ID（必须唯一，建议用IP后三位）
log_bin = /var/lib/mysql/mysql-bin  -- 开启binlog日志，指定日志存储路径
binlog_format = ROW  -- binlog格式（推荐ROW，基于行复制，数据一致性更高）
expire_logs_days = 7  -- binlog日志保留7天，避免占用过多磁盘空间
innodb_flush_log_at_trx_commit = 1  -- 保证binlog日志实时写入磁盘，避免数据丢失

-- 2. 重启MySQL服务（使配置生效）
systemctl restart mysqld  #  CentOS/RHEL系统
service mysql restart     #  Ubuntu系统

-- 3. 登录主节点MySQL，创建同步用户（给从节点授权）
CREATE USER 'repl'@'192.168.1.101' IDENTIFIED BY 'Repl@123456';  -- 允许从节点IP连接
GRANT REPLICATION SLAVE ON *.* TO 'repl'@'192.168.1.101';  -- 授予同步权限
FLUSH PRIVILEGES;  -- 刷新权限

-- 4. 查看主节点binlog状态（记录关键信息，用于从节点配置）
SHOW MASTER STATUS;
-- 执行结果会显示：File（binlog文件名）、Position（同步起始位置），例如：
-- File: mysql-bin.000001, Position: 156, Binlog_Do_DB: , Binlog_Ignore_DB: 

```

### 步骤2：配置从节点（Slave）

```sql
-- 1. 编辑MySQL配置文件（my.cnf 或 my.ini）
[mysqld]
server-id = 101  -- 从节点唯一ID（必须与主节点不同）
relay_log = /var/lib/mysql/relay-bin  -- 开启中继日志，指定存储路径
read_only = 1  -- 从节点设置为只读（避免从节点被误写，超级用户可例外）
super_read_only = 1  -- 禁止超级用户写操作（增强只读安全性）

-- 2. 重启MySQL服务（使配置生效）
systemctl restart mysqld

-- 3. 登录从节点MySQL，配置主从同步
CHANGE MASTER TO
MASTER_HOST='192.168.1.100',  -- 主节点IP
MASTER_USER='repl',  -- 主节点创建的同步用户
MASTER_PASSWORD='Repl@123456',  -- 同步用户密码
MASTER_LOG_FILE='mysql-bin.000001',  -- 主节点binlog文件名（步骤1.4获取）
MASTER_LOG_POS=156;  -- 主节点binlog同步起始位置（步骤1.4获取）

-- 4. 启动从节点同步进程
START SLAVE;

-- 5. 查看从节点同步状态（核心检查）
SHOW SLAVE STATUS\G;
-- 关键判断：两个线程状态均为Yes，说明同步正常
-- Slave_IO_Running: Yes （IO线程正常运行）
-- Slave_SQL_Running: Yes （SQL线程正常运行）

```

### 步骤3：测试主从同步

```sql
-- 1. 主节点执行写操作
INSERT INTO product (name, stock) VALUES ('耳机', 30);  -- 假设product表已存在

-- 2. 从节点执行读操作，查看数据是否同步
SELECT * FROM product;  -- 若能查询到新增的耳机数据，说明同步成功

```

避坑技巧：配置过程中，若出现同步失败（Slave_IO_Running或Slave_SQL_Running为No），可查看从节点错误日志（show variables like 'log_error'），常见原因：主从server-id重复、同步用户权限不足、binlog文件名/位置错误、主从数据不一致。

# 三、异步 / 半同步 / 全同步复制（核心区别）

根据“主节点写操作完成后，是否等待从节点同步完成”，MySQL主从复制分为三种模式，三种模式的核心区别在于“数据一致性”和“性能”的权衡，适用于不同的生产场景。

## 3.1 异步复制（Asynchronous Replication）—— 默认模式

### 核心原理

异步复制是MySQL主从复制的默认模式，主节点执行完写操作后，只需将操作记录到binlog日志中，无需等待从节点获取binlog、执行同步操作，即可返回“操作成功”给客户端。从节点的同步操作是异步进行的，主节点不感知从节点的同步状态。

### 核心流程

客户端 → 主节点执行写操作 → 主节点记录binlog → 返回成功给客户端 → 从节点异步获取binlog、执行同步。

### 核心特性

- 优点：主节点性能最高，无需等待从节点同步，写操作响应速度快，适合高并发写场景；

- 缺点：数据一致性最差，若主节点执行完写操作后崩溃，而从节点尚未同步该操作，会导致数据丢失；

- 适用场景：对数据一致性要求不高，追求高并发写性能的场景（如日志存储、非核心业务数据）。

## 3.2 半同步复制（Semi-Synchronous Replication）—— 折中模式

### 核心原理

半同步复制是对异步复制的优化，主节点执行完写操作后，需等待**至少一个从节点**的IO线程将binlog日志写入中继日志（relay log），并返回“确认信号”给主节点，主节点才会返回“操作成功”给客户端；若超过指定时间（默认10秒）未收到从节点的确认信号，自动降级为异步复制。

### 核心流程

客户端 → 主节点执行写操作 → 主节点记录binlog → 等待至少一个从节点IO线程写入中继日志并确认 → 返回成功给客户端 → 从节点SQL线程执行同步。

### 核心特性

- 优点：数据一致性中等，避免了异步复制的数据丢失问题（至少有一个从节点备份了binlog）；性能介于异步和全同步之间，是生产环境中最常用的模式；

- 缺点：主节点写操作响应速度比异步复制慢（需等待从节点确认）；若所有从节点都故障，主节点会降级为异步复制，仍有数据丢失风险；

- 适用场景：对数据一致性有一定要求，同时追求较高并发性能的场景（如电商核心业务、用户中心）。

### 实操：开启半同步复制

```sql
-- 1. 主节点开启半同步复制（需安装半同步插件）
INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';  -- 安装插件
SET GLOBAL rpl_semi_sync_master_enabled = 1;  -- 开启主节点半同步
SET GLOBAL rpl_semi_sync_master_timeout = 10000;  -- 超时时间（10秒，单位：毫秒）
RESTART SLAVE;  -- 重启同步进程

-- 2. 从节点开启半同步复制
INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';  -- 安装插件
SET GLOBAL rpl_semi_sync_slave_enabled = 1;  -- 开启从节点半同步
STOP SLAVE IO_THREAD;  -- 停止IO线程
START SLAVE IO_THREAD;  -- 重启IO线程，使配置生效

-- 3. 查看半同步复制状态
-- 主节点查看
SHOW VARIABLES LIKE 'rpl_semi_sync_master%';
-- 从节点查看
SHOW VARIABLES LIKE 'rpl_semi_sync_slave%';

```

## 3.3 全同步复制（Fully Synchronous Replication）—— 高一致性模式

### 核心原理

全同步复制是数据一致性最高的模式，主节点执行完写操作后，需等待**所有从节点**的SQL线程执行完同步操作（将binlog中的SQL语句执行完毕，数据同步完成），并返回“确认信号”给主节点，主节点才会返回“操作成功”给客户端。

### 核心流程

客户端 → 主节点执行写操作 → 主节点记录binlog → 等待所有从节点IO线程获取binlog、SQL线程执行同步并确认 → 返回成功给客户端。

### 核心特性

- 优点：数据一致性最高，主节点写操作完成后，所有从节点数据均已同步，无数据丢失风险；

- 缺点：主节点性能最差，写操作响应速度极慢（需等待所有从节点同步完成）；若有一个从节点同步缓慢或故障，会阻塞主节点的所有写操作；

- 适用场景：对数据一致性要求极高，不追求高并发性能的场景（如金融交易、核心账务数据）。

## 3.4 三种复制模式对比（必记）

|复制模式|核心逻辑|数据一致性|主节点性能|数据丢失风险|适用场景|
|---|---|---|---|---|---|
|异步复制|主节点不等待从节点同步，直接返回成功|最低|最高|高|非核心业务、高并发写、对一致性要求低|
|半同步复制|主节点等待至少一个从节点写入中继日志|中等|中等|低|核心业务、兼顾一致性与性能（最常用）|
|全同步复制|主节点等待所有从节点同步完成|最高|最低|无|金融、账务等对一致性要求极高的场景|
# 四、GTID 复制（进阶特性）

GTID（Global Transaction Identifier，全局事务ID）是MySQL 5.6及以上版本引入的进阶复制特性，用于替代传统的“基于binlog文件名+位置”的复制方式，核心优势是“简化主从复制配置、便于故障切换、避免重复同步”。

## 4.1 GTID 核心定义

GTID是一个全局唯一的事务ID，格式为：`server-uuid:transaction-id`，其中：

- server-uuid：主节点的唯一标识符（每个MySQL节点的uuid唯一，存储在auto.cnf文件中）；

- transaction-id：主节点上执行的事务序号（从1开始递增，每个事务对应一个唯一序号）。

通俗理解：每个事务在主节点执行时，都会被分配一个全局唯一的GTID，从节点同步时，只需记录“已执行的GTID”，即可知道哪些事务已同步、哪些未同步，无需手动指定binlog文件名和位置。

## 4.2 GTID 复制的核心优势（对比传统复制）

1. 简化配置：从节点配置主从同步时，无需指定binlog文件名和位置，只需开启GTID，指定主节点IP、用户名、密码即可；

2. 便于故障切换：主节点故障后，从节点可快速找到未同步的事务，无需手动定位binlog位置，切换效率更高；

3. 避免重复同步：GTID全局唯一，从节点不会重复执行同一事务，减少同步错误；

4. 支持多主复制：GTID天然支持多主复制架构（如环形复制），便于搭建复杂的高可用集群。

## 4.3 GTID 复制实操配置

基于前一章的单主单从架构，修改配置开启GTID复制（MySQL 8.0版本）：

### 步骤1：配置主节点（Master）

```sql
-- 编辑my.cnf配置文件，新增GTID相关配置
[mysqld]
server-id = 100
log_bin = /var/lib/mysql/mysql-bin
binlog_format = ROW
expire_logs_days = 7
innodb_flush_log_at_trx_commit = 1
# GTID相关配置
gtid_mode = ON  -- 开启GTID模式
enforce_gtid_consistency = ON  -- 强制GTID一致性（避免非GTID事务）
log_slave_updates = ON  -- 允许从节点将同步的事务记录到自己的binlog（用于级联复制）

-- 重启MySQL服务
systemctl restart mysqld

-- 查看GTID状态
SHOW VARIABLES LIKE 'gtid%';

```

### 步骤2：配置从节点（Slave）

```sql
-- 编辑my.cnf配置文件，新增GTID相关配置
[mysqld]
server-id = 101
relay_log = /var/lib/mysql/relay-bin
read_only = 1
super_read_only = 1
# GTID相关配置
gtid_mode = ON
enforce_gtid_consistency = ON
log_slave_updates = ON

-- 重启MySQL服务
systemctl restart mysqld

-- 登录从节点，配置GTID同步（无需指定binlog文件名和位置）
CHANGE MASTER TO
MASTER_HOST='192.168.1.100',
MASTER_USER='repl',
MASTER_PASSWORD='Repl@123456',
MASTER_AUTO_POSITION = 1;  -- 开启GTID自动定位（核心）

-- 启动从节点同步进程
START SLAVE;

-- 查看同步状态（GTID相关字段）
SHOW SLAVE STATUS\G;
-- 关键字段：
-- Retrieved_Gtid_Set：从节点已获取的GTID集合
-- Executed_Gtid_Set：从节点已执行的GTID集合
-- 两个集合一致，说明同步正常

```

## 4.4 主从故障切换（基于GTID）

主节点故障后，基于GTID可快速将从节点切换为主节点，步骤如下（实操性强）：

```sql
-- 步骤1：确认主节点故障（无法连接），登录从节点，停止同步进程
STOP SLAVE;

-- 步骤2：确认从节点已同步所有事务（Executed_Gtid_Set 与主节点一致）
SHOW SLAVE STATUS\G;  -- 查看Executed_Gtid_Set

-- 步骤3：将从节点切换为主节点（取消只读模式）
SET GLOBAL read_only = 0;
SET GLOBAL super_read_only = 0;

-- 步骤4：通知应用程序，将数据库连接地址切换为新主节点（192.168.1.101）

-- 步骤5：若原主节点恢复，将其配置为新主节点的从节点（可选）
-- 登录原主节点，配置为从节点
CHANGE MASTER TO
MASTER_HOST='192.168.1.101',
MASTER_USER='repl',
MASTER_PASSWORD='Repl@123456',
MASTER_AUTO_POSITION = 1;
START SLAVE;

```

避坑技巧：故障切换时，务必确认从节点已同步所有事务（Executed_Gtid_Set与原主节点一致），避免数据不一致；切换后，及时更新应用程序的数据库连接配置，避免连接失败。

# 五、读写分离架构设计（实战重点）

读写分离是基于主从复制实现的核心架构，核心思想是“主节点负责写操作，从节点负责读操作”，通过分担读压力，提升系统并发处理能力，是生产环境中解决MySQL性能瓶颈的常用方案。

## 5.1 读写分离核心架构

读写分离的架构分为3层，从上到下依次为：应用层、中间件层、数据库层，各层职责清晰：

1. 应用层：业务系统，发送读、写请求（需区分读请求和写请求）；

2. 中间件层：读写分离中间件（如MyCat、Sharding-JDBC、ProxySQL），核心职责是“路由请求”——将写请求转发到主节点，将读请求转发到从节点（可实现负载均衡）；

3. 数据库层：主从复制架构（1主多从），主节点处理写请求，多个从节点处理读请求，实现负载分担。

补充说明：若不使用中间件，也可在应用层实现读写分离（如通过代码判断请求类型，转发到对应节点），但这种方式灵活性差、维护成本高，生产环境中优先使用中间件。

## 5.2 常用读写分离中间件对比（选型必看）

|中间件|核心特点|优势|缺点|适用场景|
|---|---|---|---|---|
|MyCat|基于Java开发，支持读写分离、分库分表，功能强大|配置简单、功能全面，支持多种分库分表策略，社区活跃|性能一般，Java虚拟机占用内存较大|中小规模业务、需要同时实现读写分离和分库分表|
|Sharding-JDBC|基于Java开发，嵌入应用程序，无独立部署节点|性能优秀（无中间件转发开销），开发灵活，支持分布式事务|需在应用层集成，维护成本较高，不支持非Java应用|Java开发的中大型业务、对性能要求高|
|ProxySQL|基于C++开发，独立部署，支持读写分离、负载均衡、故障切换|性能极高，支持多种数据库，故障切换灵活，支持动态配置|配置复杂，学习成本高，分库分表功能较弱|大规模业务、对性能要求极高，主要实现读写分离|
## 5.3 读写分离实操（基于MyCat，简单易上手）

以下是基于MyCat实现读写分离的基础配置步骤，假设已搭建1主1从架构（主：192.168.1.100，从：192.168.1.101）：

### 步骤1：安装MyCat

```bash
# 1. 下载MyCat（对应MySQL版本）
wget https://github.com/MyCATApache/Mycat-Server/releases/download/Mycat-server-1.6.7.6/Mycat-server-1.6.7.6-linux.tar.gz

# 2. 解压安装
tar -zxvf Mycat-server-1.6.7.6-linux.tar.gz -C /usr/local/

# 3. 启动MyCat
/usr/local/mycat/bin/mycat start

```

### 步骤2：配置MyCat（核心）

```xml
-- 1. 编辑schema.xml（配置逻辑库、数据节点）
<?xml version="1.0"?>
<!DOCTYPE mycat:schema SYSTEM "schema.dtd">
<mycat:schema xmlns:mycat="http://io.mycat/">
    <schema name="test_db" checkSQLschema="false" sqlMaxLimit="100" dataNode="dn1"/>
    <dataNode name="dn1" dataHost="localhost1" database="test"/>  -- test是实际数据库名
    <dataHost name="localhost1" maxCon="1000" minCon="10" balance="1"  -- balance=1表示读负载均衡
              writeType="0" dbType="mysql" dbDriver="jdbc" switchType="1"  slaveThreshold="100">
        <heartbeat>select user()</heartbeat>
        -- 主节点配置
        <writeHost host="hostM1" url="192.168.1.100:3306" user="root" password="123456">
            -- 从节点配置
            <readHost host="hostS1" url="192.168.1.101:3306" user="root" password="123456"/>
        </writeHost>
    </dataHost>
</mycat:schema>

-- 2. 编辑server.xml（配置MyCat登录用户）
<user name="mycat" defaultAccount="true">
    <property name="password">123456</property>
    <property name="schemas">test_db</property>  -- 关联schema.xml中的逻辑库
</user>

```

### 步骤3：测试读写分离

```sql
-- 1. 连接MyCat（端口：8066）
mysql -u mycat -p123456 -h 127.0.0.1 -P 8066

-- 2. 执行写操作（会转发到主节点）
INSERT INTO product (name, stock) VALUES ('音箱', 20);

-- 3. 执行读操作（会转发到从节点）
SELECT * FROM product;

-- 4. 验证：查看从节点是否同步了写操作，确认读请求来自从节点
-- 从节点执行：SELECT * FROM product;  -- 有新增的音箱数据
-- 主节点执行：SHOW GLOBAL STATUS LIKE 'Com_insert';  -- 有写操作记录
-- 从节点执行：SHOW GLOBAL STATUS LIKE 'Com_select';  -- 有读操作记录

```

避坑技巧：读写分离的核心问题是“主从同步延迟”——主节点写操作完成后，从节点需一定时间同步，此时读请求可能读取到旧数据。解决方案：核心读请求（如用户个人信息）可路由到主节点；非核心读请求（如商品列表）路由到从节点，或增加同步延迟监控。

# 六、分库分表方案与中间件（进阶实战）

当业务数据量达到千万级、亿级时，单库单表会出现性能瓶颈（查询慢、写入慢、索引失效），此时需要通过“分库分表”将数据拆分到多个数据库、多个表中，分担单库单表的压力，提升系统性能。分库分表分为“分库”和“分表”两种方式，可单独使用，也可组合使用。

## 6.1 分库分表核心概念

- 分表：将一个大表拆分为多个小表（同库内），例如：将user表拆分为user_0、user_1、user_2，每个表存储部分用户数据；

- 分库：将一个大数据库拆分为多个小数据库（不同节点），例如：将test_db拆分为test_db_0、test_db_1，每个数据库存储部分业务数据；

- 分片键：用于拆分数据的字段（如用户ID、订单ID），核心是“将数据均匀拆分到各个分片”，避免数据倾斜。

## 6.2 分库分表方案（核心两种）

分库分表的核心方案分为“水平拆分”和“垂直拆分”，两种方案适用场景不同，实际生产中常组合使用。

### 6.2.1 水平拆分（Horizontal Sharding）—— 按行拆分

核心原理：将一个大表的**行数据**按分片键拆分到多个小表（或多个数据库），每个小表的结构与原表一致，仅存储部分行数据。例如：按用户ID取模，将用户ID为1-10000的存储到user_0表，10001-20000的存储到user_1表。

### 核心特性

- 优点：拆分后单表数据量减小，查询、写入性能提升；可无限扩展（增加分片数量）；

- 缺点：查询跨分片数据时，需要聚合多个分片的结果（复杂度高）；需解决分片键选择、数据倾斜问题；

- 适用场景：表数据量极大（千万级以上），行数据独立，查询多基于分片键的场景（如用户表、订单表）。

### 常见水平拆分策略

1. 取模拆分：按分片键（如用户ID）取模，例如：user_id % 4 = 0 → user_0，user_id %4 =1 → user_1（均匀拆分，最常用）；

2. 范围拆分：按分片键的范围拆分，例如：订单时间2024年1月 → order_202401，2024年2月 → order_202402（适合按时间查询的场景）；

3. 哈希拆分：对分片键进行哈希计算，根据哈希结果拆分（均匀性更好，避免数据倾斜）。

### 6.2.2 垂直拆分（Vertical Sharding）—— 按列拆分

核心原理：将一个大表的**列数据**按业务逻辑拆分到多个小表（或多个数据库），每个小表存储部分列数据，共同组成原表的完整数据。例如：将user表拆分为user_base（存储用户基本信息：id、name、phone）和user_extend（存储用户扩展信息：avatar、address、signature）。

### 核心特性

- 优点：拆分后单表字段减少，索引体积减小，查询性能提升；可按业务模块拆分，便于维护；

- 缺点：查询完整数据时，需要关联多个表（join操作）；拆分逻辑依赖业务，灵活性差；

- 适用场景：表字段过多（如包含大字段blob、text），不同字段的访问频率差异大的场景（如商品表、用户表）。

## 6.3 分库分表中间件（选型实战）

分库分表的实现依赖中间件，中间件负责“数据分片、路由请求、聚合结果”，无需手动管理分片逻辑，常用中间件与读写分离中间件有重叠，核心推荐3种：

### 6.3.1 MyCat

核心优势：支持水平拆分、垂直拆分，同时支持读写分离，配置简单，社区活跃，适合中小规模业务；支持多种拆分策略（取模、范围、哈希），可实现跨分片join、分页查询。

缺点：性能一般，不支持分布式事务（需额外实现），大规模业务场景下压力较大。

### 6.3.2 Sharding-JDBC

核心优势：基于Java开发，嵌入应用程序，性能优秀（无中间件转发开销）；支持水平拆分、垂直拆分、读写分离，支持分布式事务（SAGA、TCC模式）；拆分策略灵活，可自定义分片规则。

缺点：需在应用层集成，维护成本较高，仅支持Java应用，学习成本中等。

### 6.3.3 Sharding-Proxy

核心优势：独立部署，支持多种数据库（MySQL、PostgreSQL），不依赖应用程序语言；支持水平拆分、垂直拆分、读写分离，支持分布式事务；配置简单，对应用程序透明（无需修改代码）。

缺点：性能比Sharding-JDBC略差（有中间件转发开销），大规模业务场景下需集群部署。

## 6.4 分库分表避坑要点（生产必备）

1. 分片键选择：优先选择查询频繁、分布均匀的字段（如用户ID、订单ID），避免选择频繁修改、分布不均的字段（如性别、状态），防止数据倾斜；

2. 避免跨分片查询：尽量避免查询多个分片的数据（如跨分片join、跨分片分页），若必须跨分片，需优化查询逻辑（如异步聚合）；

3. 数据扩容：提前规划分片数量，选择支持动态扩容的中间件（如Sharding-JDBC、Sharding-Proxy），避免扩容时数据迁移复杂；

4. 分布式事务：分库分表后，跨分片的事务需通过中间件或第三方组件（如Seata）实现，保证数据一致性；

5. 监控运维：搭建分库分表监控体系，监控各分片的性能、数据量、同步状态，及时发现数据倾斜、性能瓶颈。
