# 12-Sentinel 限流与高级规则配置

## 本章概述

本章是微服务高可用容错体系中**流量防护的核心章节**，承接上一章Sentinel熔断降级内容。熔断降级属于「故障事后容错」，用于服务出现异常、超时、报错后的兜底防护，而限流属于「流量事前防护」，是抵御流量突增、恶意请求、并发峰值的第一道屏障，二者组合可实现**事前限流、事后熔断、异常降级**的完整服务容错闭环。本章核心目标是从零掌握Sentinel限流核心原理、四大限流算法、底层滑动窗口机制、基础限流规则全场景实战，彻底理解QPS限流、线程数限流的区别与落地场景，同时吃透流控模式、流控效果的核心逻辑，为后续高级限流规则、规则持久化、集群限流实战打下坚实基础。全文兼顾原理深度、生产实操与面试考点，适配日常开发、线上落地、面试复盘全场景。

# 1. Sentinel 限流核心原理与基础规则

## 1.1 限流的核心概念与作用

### 1.1.1 限流的定义与目标（保护系统资源，防止流量突增导致崩溃）

**限流定义**：限流是一种**前置流量防护手段**，指系统通过预设规则，对单位时间内的请求数量、并发线程数进行限制，当入口流量超过系统承载能力时，自动拦截多余请求、拒绝访问或排队缓冲，避免瞬时大流量冲击拖垮服务。

在微服务架构中，服务器CPU、内存、数据库连接、线程池、带宽等资源都是有限的，突发秒杀流量、爬虫恶意刷量、接口频繁请求、上游服务流量异常等场景，都会瞬间耗尽系统资源，导致服务卡顿、雪崩、集群瘫痪。

**限流核心目标**：

- **保护核心资源**：限制超额流量，保证服务器、数据库、中间件资源不被耗尽

- **保障服务可用**：牺牲部分超额请求，保证核心业务正常运行，避免整体服务崩溃

- **抵御恶意流量**：拦截爬虫、刷接口、高频恶意请求，提升系统稳定性与安全性

- **削峰填谷**：针对瞬时峰值流量，通过缓冲、预热方式平滑流量曲线

### 1.1.2 限流与熔断降级的关系（前置防护手段，减少故障触发）

很多开发者容易混淆限流、熔断、降级三者，这里通过**执行时机、作用场景、核心定位**彻底区分，也是面试高频考点：

**1、核心定位区别**

- **限流**：**事前防护**，针对【流量过大】，流量没超标正常放行，超标直接拦截，从源头避免服务压力过载

- **熔断**：**事中止损**，针对【服务故障】，下游服务异常、超时、报错比例过高时，切断调用链路

- **降级**：**事后兜底**，限流、熔断、业务异常触发后，返回兜底数据，保障用户体验

**2、联动关系**

限流是熔断降级的前置屏障：通过限流拦截超额流量，大幅降低服务压力，**从源头减少服务故障概率，间接减少熔断降级的触发次数**。三者配合实现完整容错闭环：**流量超限→限流拦截、服务故障→熔断止损、异常拦截→降级兜底**。

### 1.1.3 限流的常见算法（计数器、滑动窗口、令牌桶、漏桶）

限流算法是流量防护的底层核心，Sentinel底层基于**滑动窗口**实现，同时适配各类流量场景，四种主流算法的原理、优缺点、适用场景如下，面试高频考察对比：

**1、固定计数器算法（简单计数器）**

原理：将时间划分为固定周期（1秒、1分钟），每个周期独立统计请求数，超过阈值则限流，周期结束计数器清零。

优缺点：实现简单、性能极高；存在**临界突刺问题**，例如1秒最后10ms和下1秒前10ms分别涌入大量流量，单秒未超限，但20ms内流量翻倍，极易打垮服务。

适用场景：简单低并发场景，微服务生产基本不用。

**2、滑动窗口算法（Sentinel核心算法）**

原理：将固定时间窗口拆分为多个细小时间粒度，时间随请求实时滑动，淘汰过期统计数据，实时统计最新流量。

优缺点：解决固定计数器的临界突刺问题，统计精准、流量平滑、适配高并发场景；略有少量计算开销。

适用场景：Sentinel默认底层算法，微服务生产限流标准方案。

**3、漏桶算法**

原理：请求进入桶内缓存，以**固定速率匀速流出**处理，桶满则丢弃请求。

优缺点：强制流量匀速，彻底削峰；无法应对流量突发预热，突发流量会被直接拦截，不适合秒杀、活动峰值场景。

**4、令牌桶算法**

原理：系统以固定速率生成令牌存入桶中，请求获取令牌方可执行，无令牌则限流；支持囤积令牌应对突发流量。

优缺点：支持流量突发、适配峰值场景、灵活度高；实现相对复杂。

Sentinel的**WarmUp预热限流**底层基于令牌桶思想实现。

## 1.2 Sentinel 限流的底层实现

### 1.2.1 滑动窗口统计机制（时间窗口、统计粒度）

Sentinel所有的QPS统计、限流判断、熔断判断，全部依赖**滑动时间窗口机制**，这是Sentinel精准流量控制的核心底层。

**核心结构**：

- **总时间窗口**：用户配置的统计时长，如1s、10s

- **子窗口粒度**：将总窗口均等拆分，默认1s总窗口拆分为2个500ms子窗口，粒度越细，统计越精准

- **数据存储**：每个子窗口独立存储请求总数、成功数、异常数、响应耗时、QPS等指标

**滑动机制**：随着时间推移，自动淘汰过期的子窗口数据，保留最新的完整窗口数据，实时计算当前QPS、异常比例，避免传统固定窗口的统计误差。

**生产优势**：统计精度高、无流量临界盲区、适配瞬时峰值流量，限流熔断触发精准，极少出现误判、漏判。

### 1.2.2 流量统计与限流决策流程

Sentinel客户端接收请求后，会执行一套完整的**流量统计+限流决策链路**，全程内存计算、无IO阻塞，性能极高：

**完整执行流程**：

1. **资源匹配**：请求进入后，匹配对应的接口资源名称、链路资源

2. **规则加载**：读取当前资源已生效的限流规则（内存缓存，无需频繁查控制台）

3. **数据统计**：基于滑动窗口，实时累加当前请求数、并发线程数

4. **阈值判断**：对比实时QPS/线程数与规则阈值，判断是否超限

5. **流量处理**：未超限则放行执行业务逻辑；超限则根据流控效果（快速失败/预热/排队）执行拦截

6. **数据上报**：异步将监控数据上报至控制台，用于可视化展示

**核心特性**：限流决策在客户端内存完成，不依赖控制台，**控制台仅用于配置下发和数据展示**，即使控制台宕机，客户端限流规则依旧永久生效。

### 1.2.3 限流规则的匹配与生效逻辑

Sentinel限流规则遵循**精准匹配、就近生效、优先级固定**的机制：

- **匹配依据**：以**资源名称**为唯一匹配Key，注解埋点、接口路径、Feign链路资源一一对应

- **规则优先级**：同资源多条规则同时生效，只要满足任意一条限流规则，即触发拦截

- **生效范围**：规则下发后实时缓存至客户端内存，无需重启服务，秒级生效

- **失效逻辑**：控制台删除规则后，客户端同步清空内存规则，流量恢复正常放行

**避坑要点**：资源名不匹配是限流规则不生效的第一大原因，自定义埋点资源名必须与控制台配置资源名完全一致。

## 1.3 基础限流规则配置实战

### 1.3.1 QPS限流规则配置

**QPS限流定义**：限制**每秒请求处理数量**，是最常用的限流方式，适用于绝大多数接口流量防护场景，针对请求频率做限制。

**一、规则核心参数说明**

- **资源名**：需要限流的接口/方法资源标识，唯一匹配规则

- **阈值类型**：选择QPS模式

- **单机阈值**：单实例每秒最大允许请求数，超过即触发限流

- **流控模式**：控制规则匹配的流量范围

- **流控效果**：超限后的流量处理方式

**二、三大流控模式详解与区别**

流控模式决定「统计哪部分流量」，是生产选型核心依据：

1. **直接限流（默认）**
   原理：统计**当前资源自身的所有请求**，只要该接口QPS超限就拦截。
        

适用场景：独立接口限流、普通查询接口防护，90%通用场景。
      

2. **关联限流**
   原理：监控**关联资源的流量**，关联资源超限，限流当前资源。
       

场景举例：下单接口关联支付接口，支付接口压力过大时，限流下单接口，保护核心支付链路。
      

3. **链路限流**
   原理：只统计**指定入口链路**的流量，其他链路调用不受限制。
        

场景举例：某个公共工具接口，允许内部业务调用，限制外部爬虫高频调用。
      

**三、三大流控效果配置与适用场景**

1. **快速失败（默认）**
   逻辑：流量超限立刻拦截，直接抛出限流异常，执行降级兜底。
        

适用场景：常规接口、瞬时突发流量，追求响应速度。
      

2. **Warm Up（预热限流）**
   逻辑：基于令牌桶原理，系统启动后逐步释放流量，缓慢提升QPS阈值，避免服务冷启动被瞬时流量打垮。
        

适用场景：服务重启、项目上线、数据库预热后需要逐步承载流量的场景。
        
核心参数：预热时长，设置系统从最低阈值恢复到最大阈值的时间。
      

3. **排队等待**
   逻辑：超额请求不直接拦截，而是排队等待，匀速处理请求，超时未处理则限流失败。
        

适用场景：订单提交、表单提交等**不允许丢请求、允许延迟处理**的业务。
      

**四、完整配置与效果验证步骤**

1. 启动微服务，确保服务成功注册至Sentinel控制台

2. 访问测试接口，生成簇点链路资源

3. 控制台选择对应资源，新增流控规则：QPS阈值设为5，流控模式直接限流，流控效果快速失败

4. 使用Postman/JMeter高频并发请求接口

5. 每秒请求超过5次后，自动触发限流，返回预设兜底信息，验证生效

### 1.3.2 线程数限流规则配置

**线程数限流定义**：限制**当前接口的最大并发线程数**，不限制请求频率，专门用于解决接口耗时过长、线程堆积阻塞问题。

**一、线程数阈值设置与适用场景**

单机阈值代表该接口同时允许的最大并行请求线程数，超过阈值直接限流。

**核心适用场景**：

- 慢接口、耗时接口（数据库批量查询、文件导出、复杂计算）

- 容易出现线程阻塞、连接超时的接口

- 防止单个慢接口耗尽服务线程池，导致全局接口不可用

**二、线程数限流与QPS限流的核心区别（面试高频）**

|对比维度|QPS限流|线程数限流|
|---|---|---|
|限制维度|每秒请求次数（请求频率）|同时并发线程数（并发数量）|
|适用接口|快速响应接口、查询接口|慢接口、阻塞接口、耗时任务接口|
|核心作用|防高频刷量、瞬时流量峰值|防线程堆积、线程池耗尽、服务阻塞|
|场景举例|用户列表、详情查询|批量导出、报表统计、大数据查询|
**三、高并发场景下的线程数保护效果验证**

1. 编写模拟耗时接口，线程休眠1秒模拟业务阻塞

2. 控制台为该资源配置线程数限流，阈值设置为10

3. 开启20个线程并发请求接口

4. 结果验证：最多10个线程正常执行，超额请求直接触发限流兜底，线程不会堆积，服务线程池稳定

**生产最佳实践**：**快接口用QPS限流，慢接口用线程数限流**，核心服务可同时配置双重限流，全方位防护。

# 2. 热点参数限流实战

普通QPS限流、线程数限流是**接口全局统一限流**，针对整个接口的所有请求均等限制，但在真实高并发场景中，往往存在**个别参数引发的流量热点**，例如秒杀单个商品、热门热搜话题被集中访问，普通全局限流无法精准防护。本章讲解的热点参数限流，是Sentinel独有的精细化限流能力，可针对接口中的**指定请求参数**做独立流量控制，精准防刷、防热点击穿，是秒杀、热门场景的核心防护方案。

## 2.1 热点参数限流的应用场景

### 2.1.1 什么是热点参数？（高并发场景下的高频请求参数）

**热点参数**指在短时间内被**超高频率重复携带、集中请求**的接口入参，是流量高度聚合的参数维度。这类参数对应的资源会瞬间产生超大流量，远超普通业务流量，极易造成单点击穿、数据库压力过载。

热点参数的核心特征：

- 流量高度集中：90%以上的流量集中在少数几个参数值上

- 请求频率极高：瞬时QPS远超普通业务请求

- 风险极高：单一热点参数可直接打垮接口、数据库、缓存集群

### 2.1.2 典型场景：秒杀商品ID、热门话题ID防刷

热点参数限流是互联网高并发业务的刚需能力，两大核心落地场景如下：

**1、秒杀商品防刷（最核心场景）**

秒杀活动中，所有用户的请求都携带**同一个秒杀商品ID**，该商品ID即为热点参数。若使用普通接口限流，要么全局限流导致普通商品无法访问，要么阈值过大无法防护秒杀热点流量，热点参数限流可单独限制该秒杀商品的访问频次，精准防护。

**2、热门话题/热点资讯防刷**

热搜榜单、热门新闻、爆款短视频会被用户集中刷新访问，对应的**话题ID、内容ID**成为热点参数，恶意用户可通过脚本高频刷取热点内容，造成服务器、缓存、数据库压力骤增，热点参数限流可精准拦截高频热点请求。

**3、其他延伸场景**

单个店铺爆单、单个用户高频操作、优惠券批量领取等单点流量集中场景。

### 2.1.3 热点参数限流与普通限流的区别

普通限流（QPS/线程数）是**粗粒度全局限流**，热点参数限流是**细粒度维度限流**，二者核心区别是面试与生产选型重点，对比如下：

| 对比维度 | 普通QPS/线程数限流                           | 热点参数限流                               |
| -------- | -------------------------------------------- | ------------------------------------------ |
| 限流维度 | 整个接口全局流量，所有请求一视同仁           | 接口中的**指定参数值**，精准控制单点流量   |
| 粒度精度 | 粗粒度，无法区分普通流量和热点流量           | 细粒度，精准识别热点参数，只限制高频参数   |
| 业务影响 | 热点流量超限会导致整个接口降级，影响所有用户 | 仅限制热点参数流量，普通参数请求正常放行   |
| 适用场景 | 接口整体流量均匀、无明显热点                 | 流量高度集中、存在单点热点参数的高并发场景 |

## 2.2 热点参数限流规则配置

热点参数限流无需额外引入依赖，基于Sentinel原生能力实现，支持控制台可视化配置，可精准控制指定参数的QPS阈值、统计时长，同时支持特殊参数白名单配置。

### 2.2.1 热点参数规则参数说明（资源名、参数索引、阈值、统计时长）

热点参数限流核心配置参数全部为生产常用核心字段，详细释义如下：

- **资源名**：需要做热点限流的接口资源名称，必须与簇点链路中的资源名完全一致

- **参数索引**：接口请求参数的下标索引，从0开始。例如接口参数为 `getGoods(Long goodsId)`，goodsId为第0个参数，索引填0

- **单机阈值**：单实例中，**单个参数值**每秒允许的最大请求次数

- **统计时长（秒）**：热点参数流量的统计时间窗口，默认1秒，可根据业务调整

**核心逻辑**：在指定统计时长内，同一个参数值的请求次数超过单机阈值，立即触发该参数的限流，其他参数不受影响。

### 2.2.2 单机阈值与集群阈值配置

**1、单机阈值**

针对单服务实例的限流阈值，适用于单机部署、测试环境、非集群生产环境。例如单机阈值设置为10，代表单个实例下，同一个参数每秒最多允许10次请求。

**2、集群阈值**

针对分布式集群部署场景，全局控制整个集群内单个参数的总请求量，避免多实例叠加流量击穿服务。集群阈值需要开启Sentinel集群限流模式，统一管控全实例流量。

**生产配置原则**：集群总阈值 = 单机阈值 × 集群实例数 × 容错系数（0.8），防止流量不均衡导致局部超限。

### 2.2.3 参数例外项配置（特殊参数单独设置阈值）

热点参数限流支持**参数白名单/例外阈值配置**，用于适配特殊业务场景：部分参数为核心业务参数、内部测试参数，需要放宽限流阈值，不跟随通用参数规则限制。

**配置方式**：在控制台热点规则中配置「例外项」，指定特殊参数值，并单独设置更高的阈值或不限流。

**典型使用场景**：

- 官方活动置顶商品ID，需要放宽限流阈值，保证活动流量正常访问

- 内部测试账号、测试参数，需要豁免限流，方便测试调试

### 2.2.4 规则配置与效果验证（模拟高频参数请求）

**完整实操步骤（可直接落地）**

**第一步：编写测试接口**

```java
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class HotParamController {

    /**
     * 热点参数限流测试接口
     * 参数goodsId：商品ID，作为热点统计参数（索引0）
     */
    @GetMapping("/hot/goods/detail")
    public String getGoodsDetail(@RequestParam("goodsId") Long goodsId) {
        // 模拟商品查询业务逻辑
        return "商品详情查询成功，商品ID：" + goodsId;
    }
}
```

**第二步：控制台配置热点规则**

1. 访问接口生成簇点链路资源

2. 新增热点参数限流规则：资源名选 `/hot/goods/detail`、参数索引0、单机阈值5、统计时长1秒

3. 不配置例外项，默认所有参数统一阈值

**第三步：效果验证**

1. 高频请求同一热点参数：`/hot/goods/detail?goodsId=1001`，每秒请求超过5次

2. 热点参数1001触发限流，返回默认降级提示

3. 同时请求普通参数：`/hot/goods/detail?goodsId=2001`，请求正常放行

**验证结论**：仅高频热点参数被限流，普通参数不受影响，精准实现单点热点防护。

## 2.3 热点参数限流的实现原理

### 2.3.1 热点参数的LRU缓存统计机制

Sentinel热点参数限流底层基于**LRU最近最少使用缓存**实现参数流量统计，避免无限存储海量参数数据导致内存溢出。

**核心机制**：

- Sentinel会维护一个固定容量的LRU缓存，用于存储接口参数对应的流量统计数据（请求数、QPS）

- 缓存默认容量有限，仅保留**近期高频访问的热点参数**

- 长期低频、未访问的参数会被自动淘汰，释放内存空间

**设计价值**：高并发场景下参数值海量多变，LRU缓存可保证内存占用可控，避免内存溢出，兼顾统计精准度与服务性能。

### 2.3.2 热点参数的统计与限流决策流程

完整执行链路（请求进入到限流判断全过程）：

1. **参数解析**：拦截接口请求，根据配置的参数索引，提取目标请求参数

2. **缓存查询**：查询LRU缓存中该参数的流量统计数据

3. **数据更新**：基于滑动窗口机制，实时累加该参数当前统计窗口的请求QPS

4. **阈值判断**：对比单个参数实时QPS与配置阈值

5. **流量处理**：超限则触发限流拦截，未超限则正常放行执行业务

6. **缓存维护**：定时淘汰低频参数，更新热点参数排序

### 2.3.3 热点参数限流的性能影响与优化建议

热点参数限流基于内存缓存+滑动窗口统计，性能损耗极低，但超高并发场景仍需优化，规避潜在性能问题。

**生产优化建议**：

- **合理设置统计时长**：常规业务使用1秒统计窗口，超长耗时业务可调整为2-3秒，减少统计计算频率

- **限制LRU缓存容量**：避免缓存过大占用内存，默认容量可满足绝大多数业务

- **搭配本地缓存使用**：热点参数接口优先查询本地缓存，减少数据库查询，配合限流实现双层防护

- **禁止多参数同时热点限流**：单个接口仅对核心热点参数配置限流，多参数统计会增加性能开销

**性能结论**：正常业务场景下，热点参数限流**几乎无性能损耗**，是高并发热点防护的最优方案。

# 3. 系统规则限流与防护

前面讲解的接口限流、热点参数限流都属于**应用层局部防护**，仅针对单个接口、单个参数做流量控制。而系统规则限流是Sentinel提供的**服务器全局防护能力**，从系统CPU、负载、线程数、响应时间、总QPS等全局维度监控系统状态，当整体系统资源过载时，统一拦截所有入口流量，防止服务整体崩溃，是微服务最后的兜底防护屏障。

## 3.1 系统规则的核心作用

### 3.1.1 系统保护与接口限流的区别（全局维度的系统状态保护）

**接口限流**：微观维度，针对**单个接口资源**做流量控制，保护单个接口不被打垮，不影响服务整体运行。

**系统规则限流**：宏观维度，针对**整个服务实例**的系统资源、全局流量做控制，不区分具体接口，当系统整体过载时，统一拦截所有入口请求，保护服务器整体资源。

**核心差异**：接口限流是「治局部」，系统限流是「保全局」。

### 3.1.2 系统规则的适用场景（防止整体系统过载崩溃）

系统规则是服务的**最后一道保命屏障**，适用于所有高并发、大流量、资源敏感场景：

- 瞬时超大流量导致CPU飙升、负载过高，系统即将卡死

- 大量慢接口导致服务线程池耗尽、请求堆积

- 多接口同时流量爆发，单个接口限流无法兜底，系统整体资源不足

- 秒杀、活动峰值流量场景，兜底防止服务整体雪崩

### 3.1.3 系统规则的触发条件与优先级

**触发条件**：系统实时指标（CPU负载、平均RT、总线程数、入口QPS）任意一项超过预设阈值，立即触发全局限流。

**执行优先级**：**系统规则限流 > 接口限流 > 熔断降级**

请求进入服务后，优先校验系统全局规则，系统过载直接拦截，无需执行接口级规则判断，最大程度节省系统资源。

## 3.2 系统规则配置与参数说明

Sentinel系统规则提供五种全局防护维度，覆盖服务器硬件资源、请求性能、并发流量全维度监控，所有规则可在控制台一键配置。

### 3.2.1 系统负载保护规则（基于CPU负载）

**规则原理**：监控服务器**系统CPU负载**，当机器负载超过设定阈值，触发全局限流，拒绝所有新请求。

**参数说明**：阈值为系统1分钟平均负载，Linux服务器可通过top命令查看负载数据。

**生产配置**：4核服务器阈值设为3，8核服务器阈值设为6，预留资源余量，避免CPU打满。

**适用场景**：防止CPU密集型业务导致服务器满载、卡死、无法响应。

### 3.2.2 平均RT保护规则（基于平均响应时间）

**规则原理**：统计服务所有入口请求的**平均响应耗时**，当全局接口平均RT超过阈值，判定系统拥堵，触发全局限流。

**核心意义**：平均RT飙升代表服务线程阻塞、数据库卡顿、资源耗尽，此时继续放行请求只会加剧拥堵，必须拦截新请求止损。

**生产配置**：常规微服务平均RT阈值设置为500ms，超过即判定系统拥堵。

### 3.2.3 并发线程数保护规则（基于系统总线程数）

**规则原理**：限制服务**全局最大并发线程数**，当所有接口的总活跃线程数超过阈值，拦截新请求，防止线程池耗尽。

**核心价值**：解决慢接口堆积、线程泄露导致的服务线程池占满、新请求无法处理的问题。

**生产配置**：根据服务线程池大小设置，常规Spring服务阈值设置为200~300。

### 3.2.4 入口QPS保护规则（基于系统总请求量）

**规则原理**：限制服务**全局入口总QPS**，控制整个服务的每秒最大请求总量，防止超大流量冲击。

**场景适配**：用于服务整体流量封顶，配合网关限流实现双层流量防护，避免流量超限压垮集群。

### 3.2.5 规则配置与效果验证（模拟系统过载场景）

**实操验证步骤**：

1. 控制台新增系统规则：配置全局并发线程数阈值为50

2. 编写耗时接口，模拟业务阻塞

3. 开启100个线程并发请求接口，制造线程堆积

4. 当全局活跃线程数超过50，系统触发全局限流，所有新请求被拦截

5. 停止压测，线程释放后，系统自动恢复正常放行

## 3.3 系统规则的设计与使用建议

### 3.3.1 系统规则阈值设置的最佳实践

系统阈值切忌设置过低，否则极易误杀正常流量，生产通用最优配置：

- **CPU负载阈值**：设置为CPU核心数的70%，预留系统基础运行资源

- **平均RT阈值**：根据业务P99耗时设置，比正常峰值耗时高出20%

- **全局线程数阈值**：小于服务最大线程池数量的80%，预留应急线程

- **入口QPS阈值**：根据服务器压测极限QPS的80%设置，预留流量缓冲空间

### 3.3.2 系统规则与接口限流的配合使用

生产环境必须采用**接口限流+热点限流+系统限流**的三层防护体系：

1. **第一层（精细防护）**：热点参数限流，拦截单点热点恶意流量

2. **第二层（常规防护）**：接口QPS/线程数限流，控制单个接口流量上限

3. **第三层（全局兜底）**：系统规则限流，保障服务整体不崩溃

三层防护层层兜底，既保证流量精准管控，又保证服务极限稳定性。

### 3.3.3 系统规则的生效条件与注意事项

**核心注意事项（生产避坑）**：

- **禁止滥用系统规则**：系统规则是兜底手段，不适合日常精细化流量控制，优先使用接口限流
- **阈值不可过低**：过低的系统阈值会导致正常业务流量被拦截，影响用户体验
- **集群环境单独配置**：系统规则是单机维度，集群部署时需根据单实例性能统一配置
- **优先级最高**：一旦触发系统限流，所有接口规则失效，全局拦截请求，需谨慎配置

# 4. 授权规则（黑白名单）配置

前面章节讲解的限流、熔断、系统防护规则，均是**基于请求流量、系统状态的被动防护**，无法对请求来源进行精准管控。Sentinel 授权规则是**基于请求来源的主动访问控制**，通过识别请求的来源IP、服务名称等标识，实现黑名单拦截、白名单放行，主要用于保护内部服务、拦截恶意调用、实现服务间权限隔离。本章将完整讲解授权规则的场景、实战配置、底层原理与自定义拓展方案。

## 4.1 授权规则的核心作用

### 4.1.1 什么是授权规则？（基于来源IP/服务名的访问控制）

**Sentinel授权规则**，也叫访问控制规则，是Sentinel提供的**来源维度权限管控能力**。不同于流量限流（控制请求数量）、熔断降级（控制异常流量），授权规则的核心是**控制“谁能访问、谁不能访问”**。

授权规则核心管控维度：

- **来源IP**：识别客户端真实IP，对指定IP进行放行或拦截

- **来源服务名**：微服务架构中，识别调用方服务名称，实现服务级别的权限访问控制

其核心本质是：在请求进入资源逻辑之前，拦截校验**请求来源身份**，根据黑白名单规则决定是否放行请求，属于前置权限防护机制。

### 4.1.2 白名单与黑名单的适用场景（内部服务保护、防恶意请求）

授权规则分为**白名单模式**和**黑名单模式**，两种模式场景完全相反，生产中按需选型：

**1、黑名单模式（拒绝指定来源）**

规则逻辑：默认放行所有请求，仅拦截黑名单中配置的IP/服务。

适用场景：

- 拦截恶意爬虫、高频刷接口的恶意IP

- 下线、废弃服务的非法调用拦截

- 临时封禁异常调用节点，规避故障扩散

**2、白名单模式（仅放行指定来源）**

规则逻辑：默认拦截所有请求，仅放行白名单内的IP/服务，安全性更高。

适用场景：

- 内部核心服务保护，仅允许指定微服务调用，禁止外部网关、第三方直接访问

- 测试环境服务隔离，仅允许测试机器IP访问

- 核心交易、支付服务的调用方严格管控，杜绝非法调用

### 4.1.3 授权规则与其他防护规则的配合使用

在生产完整防护体系中，授权规则需与限流、熔断、系统规则联动使用，形成**权限校验 → 流量控制 → 故障熔断 → 系统兜底**的完整链路：

1. **第一层级：授权规则优先执行**：先校验请求来源，拦截非法IP、非法服务调用，从源头过滤无效请求

2. **第二层级：限流规则拦截**：合法来源请求，再进行QPS、热点参数、线程数流量管控

3. **第三层级：熔断降级容错**：正常流量出现异常、超时，触发熔断降级

4. **第四层级：系统规则兜底**：系统资源过载时全局拦截，保障服务不崩溃

**核心优势**：通过授权规则提前拦截恶意、非法请求，大幅减少无效流量进入业务链路，降低后续限流、熔断规则的压力，提升系统整体稳定性。

## 4.2 授权规则配置实战

### 4.2.1 白名单规则配置（允许指定来源访问）

**实战需求**：配置服务白名单，仅允许本机IP（127.0.0.1）访问接口，其他所有IP请求全部拦截。

**控制台配置步骤**：

1. 进入Sentinel控制台对应服务页面，点击【授权规则】→【新增授权规则】

2. 选择**资源名**：需要保护的接口全路径

3. **授权模式**：选择白名单

4. **阈值内容**：填写放行IP 127.0.0.1（多个IP用英文逗号分隔）

5. 配置完成后点击保存，规则实时生效

**规则说明**：白名单模式下，只有阈值内的来源IP可以正常访问接口，其余所有来源请求直接被Sentinel拦截，抛出授权异常。

### 4.2.2 黑名单规则配置（拒绝指定来源访问）

**实战需求**：配置黑名单，拦截恶意IP 192.168.1.100，禁止该IP访问所有接口。

**控制台配置步骤**：

1. 新增授权规则，选择对应接口资源名

2. **授权模式**：选择黑名单

3. **阈值内容**：填写拦截IP 192.168.1.100

4. 保存规则，实时生效

**规则说明**：黑名单模式下，仅拦截配置的IP，其余所有IP正常访问，适用于精准封禁恶意节点。

### 4.2.3 来源IP/服务名的获取与传递方式

Sentinel默认支持**来源IP自动解析**，微服务跨服务调用场景下，需要手动传递**调用方服务名**，否则无法实现服务级别的黑白名单管控。

**1、来源IP获取方式**

Sentinel自动从HttpServletRequest中解析客户端真实IP，无需手动编码，原生支持。

**2、服务名来源传递方式（微服务必备）**

跨服务Feign调用时，通过请求头传递当前服务名称，下游服务通过请求头识别调用方身份。

核心实现：统一请求头常量，所有微服务调用自动携带 `service-name` 请求头。

### 4.2.4 规则配置与效果验证（模拟不同来源的请求）

**白名单验证流程**：

1. 本机127.0.0.1访问接口：请求正常放行，业务正常执行

2. 更换其他IP访问接口：请求被拦截，返回 **Request origin not allowed** 授权异常

**黑名单验证流程**：

1. 正常IP访问：请求正常放行

2. 黑名单IP访问：请求直接拦截，抛出授权异常

**生产异常说明**：授权拦截属于Sentinel框架级拦截，优先级最高，不会进入业务逻辑，性能损耗极低。

## 4.3 授权规则的实现原理

### 4.3.1 来源信息的传递与解析（RequestOriginParser接口）

Sentinel实现来源解析的核心拓展接口为 **RequestOriginParser**，是授权规则的底层核心。

**接口作用**：自定义解析请求来源，支持从IP、请求头、参数中获取调用方标识，适配微服务服务名识别场景。

**默认实现**：Sentinel默认实现类优先解析客户端IP作为请求来源。

**微服务拓展需求**：默认IP解析无法识别调用服务名，需要自定义实现该接口，从请求头中读取调用方服务名称，实现服务级授权管控。

### 4.3.2 授权规则的匹配与决策流程

Sentinel授权规则完整执行链路：

1. **请求拦截**：请求进入Sentinel拦截器，触发前置权限校验

2. **来源解析**：通过RequestOriginParser解析当前请求的来源标识（IP/服务名）

3. **规则匹配**：匹配当前资源的授权黑白名单规则

4. **决策执行**

   - 白名单：来源在列表内→放行，否则拦截

   - 黑名单：来源在列表内→拦截，否则放行

5. **结果返回**：拦截则抛出授权异常，放行则进入后续流量、业务逻辑

### 4.3.3 自定义来源解析器的实现方式

生产微服务场景必备拓展，实现**基于服务名的授权管控**，完整可运行代码如下：

```java
import com.alibaba.csp.sentinel.adapter.spring.webmvc.callback.RequestOriginParser;
import org.springframework.stereotype.Component;
import org.springframework.util.StringUtils;

import javax.servlet.http.HttpServletRequest;

/**
 * 自定义Sentinel请求来源解析器
 * 优先从请求头获取调用方服务名，无则使用IP
 */
@Component
public class CustomRequestOriginParser implements RequestOriginParser {

    // 自定义请求头key，用于传递调用方服务名
    private static final String SERVICE_ORIGIN_HEADER = "service-origin";

    @Override
    public String parseOrigin(HttpServletRequest request) {
        // 1. 优先获取请求头中的服务来源
        String serviceOrigin = request.getHeader(SERVICE_ORIGIN_HEADER);
        if (StringUtils.hasText(serviceOrigin)) {
            return serviceOrigin;
        }
        // 2. 无服务头则返回客户端IP
        return getIpAddr(request);
    }

    /**
     * 获取客户端真实IP
     */
    private String getIpAddr(HttpServletRequest request) {
        String ip = request.getHeader("x-forwarded-for");
        if (ip == null || ip.length() == 0 || "unknown".equalsIgnoreCase(ip)) {
            ip = request.getHeader("Proxy-Client-IP");
        }
        if (ip == null || ip.length() == 0 || "unknown".equalsIgnoreCase(ip)) {
            ip = request.getHeader("WL-Proxy-Client-IP");
        }
        if (ip == null || ip.length() == 0 || "unknown".equalsIgnoreCase(ip)) {
            ip = request.getRemoteAddr();
        }
        return ip;
    }
}

```

**使用说明**：将该类注入Spring容器后，Sentinel自动生效，支持服务名、IP双维度来源解析，可直接配置服务级黑白名单。

# 5. Sentinel 规则持久化配置

默认情况下，Sentinel所有限流、熔断、授权、系统规则均存储在**服务内存**中，服务重启、宕机后所有配置会全部丢失，无法满足生产环境高可用要求。本章重点讲解生产主流的**Nacos规则持久化方案**，实现规则永久存储、动态更新、重启不丢失，是Sentinel生产落地的必备配置。

## 5.1 规则持久化的必要性

### 5.1.1 内存规则的局限性（服务重启后规则丢失）

Sentinel默认内存规则存在致命短板，完全不适合生产环境：

- **重启丢失**：微服务重启、发布迭代、机器重启后，所有控制台配置的规则全部清空，需要重新手动配置

- **集群不一致**：集群多实例部署时，规则仅对当前连接实例生效，多实例规则无法同步，出现防护不一致问题

- **无版本回溯**：规则修改后无法回滚，误配置会直接影响线上业务

- **运维成本极高**：多服务、多环境需要重复配置，人工操作极易出错

### 5.1.2 生产环境下规则持久化的需求

生产环境对流量防护规则有三大核心诉求：**永久存储、动态更新、统一管控**，而内存模式完全无法满足：

- 服务发布、重启后规则不丢失，无需人工重新配置

- 集群所有实例规则统一同步，避免局部防护失效

- 支持规则动态修改、实时生效，无需重启服务

- 支持规则版本管理、出错可快速回滚

### 5.1.3 规则持久化的主流方案（文件、Nacos、Apollo等）

Sentinel官方支持多种持久化方案，生产常用三种方案对比如下：

| 持久化方案      | 优点                                              | 缺点                               | 适用场景                     |
| --------------- | ------------------------------------------------- | ---------------------------------- | ---------------------------- |
| 本地文件持久化  | 无需中间件、配置简单                              | 集群不同步、动态更新弱、无版本控制 | 单机测试、简单项目           |
| **Nacos持久化** | 实时动态更新、集群同步、版本管理、适配SpringCloud | 依赖Nacos注册配置中心              | **绝大多数生产项目（推荐）** |
| Apollo持久化    | 精细化版本、灰度发布、权限管控更强                | 组件重、配置复杂                   | 大型复杂微服务集群           |

## 5.2 基于Nacos的规则持久化配置

Nacos是SpringCloud生态标配组件，基于Nacos实现Sentinel规则持久化，是目前**企业生产标准方案**，支持全类型规则持久化与动态热更新。

### 5.2.1 依赖引入与配置文件修改

**第一步：引入Maven依赖**

所有需要持久化规则的微服务，新增Sentinel-Nacos适配依赖：

```xml
<!-- Sentinel规则Nacos持久化依赖 -->
<dependency>
    <groupId>com.alibaba.csp</groupId>
    <artifactId>sentinel-datasource-nacos</artifactId>
    <version>1.8.6</version>
</dependency>
```

**第二步：yml配置文件新增持久化配置**

```yaml
spring:
  cloud:
    sentinel:
      # 关闭默认内存规则（可选，开启持久化后推荐关闭）
      datasource:
        # 流控规则持久化
        flow:
          nacos:
            server-addr: 127.0.0.1:8848  # Nacos地址
            username: nacos
            password: nacos
            data-id: ${spring.application.name}-flow-rules
            group-id: SENTINEL_GROUP
            rule-type: flow  # 规则类型：流控
        # 熔断降级规则持久化
        degrade:
          nacos:
            server-addr: 127.0.0.1:8848
            username: nacos
            password: nacos
            data-id: ${spring.application.name}-degrade-rules
            group-id: SENTINEL_GROUP
            rule-type: degrade  # 规则类型：熔断降级
        # 授权规则持久化
        authority:
          nacos:
            server-addr: 127.0.0.1:8848
            username: nacos
            password: nacos
            data-id: ${spring.application.name}-authority-rules
            group-id: SENTINEL_GROUP
            rule-type: authority  # 规则类型：授权
        # 系统规则持久化
        system:
          nacos:
            server-addr: 127.0.0.1:8848
            username: nacos
            password: nacos
            data-id: ${spring.application.name}-system-rules
            group-id: SENTINEL_GROUP
            rule-type: system  # 规则类型：系统
```

**配置说明**：分别为流控、熔断、授权、系统四类规则单独配置Nacos数据源，实现全覆盖持久化。

### 5.2.2 流控规则、熔断规则、系统规则的持久化配置

Nacos持久化规则不支持控制台直接保存，需要**在Nacos控制台手动创建配置文件**，对应上述data-id和group-id。

支持同时配置多种规则，Nacos配置优先级高于本地内存规则。

### 5.2.3 规则在Nacos中的配置格式说明

所有Sentinel规则在Nacos中统一使用**JSON数组格式**，以下为生产通用完整模板。

**1、流控规则JSON模板**

```json
[
    {
        "resource": "/test/flow",
        "limitApp": "default",
        "grade": 1,
        "count": 5,
        "strategy": 0,
        "controlBehavior": 0,
        "clusterMode": false
    }
]
```

**参数释义**：grade=1(QPS)、count=阈值、strategy=流控模式、controlBehavior=流控效果

**2、熔断降级规则JSON模板**

```json

    {
        "resource": "/test/degrade",

        "count": 0.4,

        "minRequestAmount": 10
    }
]
```

**参数释义**：grade=2(异常比例)、count=异常阈值、timeWindow=熔断时长、minRequestAmount=最小请求数

**3、授权规则JSON模板**

```json
[
    {
        "resource": "/test/authority",
        "limitApp": "127.0.0.1",
        "strategy": 0
    }
]
```

**参数释义**：strategy=0白名单、1黑名单，limitApp为黑白名单内容

### 5.2.4 配置验证：服务重启后规则自动加载

**验证步骤**：

3. 进入Sentinel控制台，可看到自动同步的所有规则

4. 重启服务多次，规则不丢失，验证持久化生效

## 5.3 规则持久化的高级特性

### 5.3.1 规则动态更新（Nacos配置变更自动同步到Sentinel）

Nacos核心优势：**配置实时热更新**，无需重启服务。

修改Nacos中的Sentinel规则配置并发布，微服务会实时监听配置变更，自动刷新内存中的Sentinel规则，秒级生效，支持线上动态调参、紧急限流扩容、故障快速防护。

### 5.3.2 多环境规则隔离（不同环境配置不同规则）

通过Nacos的**命名空间+分组**实现多环境隔离：

- 开发环境Namespace：dev，配置宽松测试规则

- 测试环境Namespace：test，配置模拟生产规则

- 生产环境Namespace：prod，配置严格防护规则

彻底解决多环境规则混淆、测试规则污染生产环境的问题。

### 5.3.3 规则的版本管理与回滚

Nacos自带配置版本管理功能：

- 所有规则修改自动记录历史版本

- 线上规则配置错误、业务异常时，可一键回滚到历史稳定版本

- 支持配置对比，快速定位规则修改差异

极大提升线上运维安全性，避免误操作导致的生产故障。

### 5.3.4 生产环境规则持久化最佳实践

汇总企业生产落地标准规范，规避所有常见坑点：

1. **统一命名规范**：规则DataId统一使用 `服务名-规则类型-rules`，便于统一管理

2. **环境严格隔离**：开发、测试、生产使用独立Nacos命名空间，禁止混用规则

3. **禁止控制台手动配置**：生产所有规则统一在Nacos配置，控制台仅做查看，防止重启丢失

4. **规则备份留存**：核心流量、熔断规则做好版本备份，重大变更先灰度后全量发布

5. **集群统一配置**：集群所有实例共享Nacos规则，保证全集群防护策略一致

6. **开启配置监听日志**：便于排查规则更新、生效异常问题

# 6. Sentinel 限流高级特性与生产级优化

前面章节已经掌握了Sentinel基础限流、热点参数限流、系统防护、授权黑白名单、Nacos规则持久化等核心能力。本章聚焦**生产高级特性与落地优化**，讲解WarmUp冷启动限流、排队等待削峰、链路限流等高阶流控模式，铺垫集群限流核心架构，同时汇总生产90%以上的限流异常排查方案，并整理本章高频面试真题，彻底打通Sentinel限流从「会用」到「生产稳用、面试能答」的闭环。

## 6.1 限流效果与模式的进阶使用

Sentinel基础QPS限流默认采用「快速失败」模式，仅适用于常规平稳流量场景。在服务冷启动、流量突峰、链路隔离等复杂生产场景中，需要配合**WarmUp预热、排队等待、链路限流**三种高级特性，实现精细化、场景化流量防护。

### 6.1.1 Warm Up模式的原理与适用场景（冷启动限流）

**1、核心概念**

Warm Up（预热限流）是Sentinel专为**服务冷启动、资源预热**设计的限流效果。普通快速失败模式下，服务刚启动就会直接放开最大阈值流量，而服务刚启动时JVM未预热、缓存未加载、数据库连接未初始化、代码未JIT编译，处理性能极差，瞬时大流量极易直接打垮重启后的服务。

Warm Up模式会**渐进式释放流量**，从低阈值缓慢提升至设定的最大阈值，给系统预留预热时间。

**2、底层实现原理**

Warm Up底层基于**令牌桶算法**实现，核心逻辑：

- 系统启动初期，令牌生成速率极低，限流阈值很小，只放行少量请求

- 随着预热时间推移，令牌生成速率线性递增

- 达到预设预热时长后，阈值提升至配置的最大QPS阈值，恢复正常流量承载能力

Sentinel通过「冷却因子」控制流量增长坡度，避免流量瞬间冲高，完美解决冷启动雪崩问题。

**3、核心配置参数**

- **单机阈值**：服务预热完成后允许的最大QPS

- **预热时长**：流量从最低阈值提升至最大阈值的过渡时间，生产常规配置3~10秒

**4、适用场景**

- 服务重启、版本发布、滚动更新后的流量接入

- 缓存清空、数据库重启后的业务恢复场景

- 大促活动开启、系统初始化后的流量预热场景

**5、生产避坑**

预热时长不宜过长，否则会导致正常流量长时间被拦截；也不宜过短，失去预热效果，常规业务推荐配置5秒。

### 6.1.2 排队等待模式的原理与适用场景（削峰填谷）

**1、核心概念**

排队等待（匀速器）模式是Sentinel实现**削峰填谷**的核心能力，区别于快速失败的直接拦截，该模式会让超额请求进入队列排队，以**固定匀速**处理请求，彻底抹平流量峰值。

**2、底层原理**

基于**漏桶算法**实现，核心逻辑：

- 无论请求流量多么突发密集，系统统一以固定速率处理请求

- 超额请求进入等待队列，依次执行

- 超过超时时间未处理的请求，才会触发限流拦截

**3、核心配置参数**

- **单机阈值**：每秒匀速处理的最大请求数

- **超时时间**：请求最大排队等待时长，超时则限流失败

**4、适用场景**

核心适用于**不允许丢请求、允许延迟处理**的业务场景：

- 订单提交、支付回调、表单提交等核心交易业务

- 消息消费、异步任务处理场景

- 瞬时流量波动极大，但业务必须保证请求不丢失的场景

**5、优缺点总结**

优点：完美削峰填谷，保护服务线程池稳定，无请求丢失；缺点：会增加请求响应耗时，不适合极速查询类接口。

### 6.1.3 链路限流模式的使用场景与注意事项

**1、链路限流核心概念**

链路限流是Sentinel的**精细化链路防护**模式，区别于直接限流的全流量拦截，链路限流可以**只限制指定调用链路的流量**，其他链路调用不受任何影响。

简单来说：同一个接口，A入口调用限流，B入口调用正常放行。

**2、核心使用场景**

- **公共接口防刷**：底层公共工具接口被多个业务调用，允许内部业务链路正常调用，限制外部爬虫、非法链路的高频调用

- **链路隔离**：核心业务链路不限流，非核心运营、统计链路限流，保障核心业务资源优先占用

- **故障隔离**：某个上游服务异常刷流量，仅拦截该链路流量，不影响其他正常调用方

**3、核心注意事项（生产高频坑点）**

- **必须开启链路埋点**：链路限流依赖入口资源埋点，未配置入口资源会导致规则永久不生效

- **关闭调用链收敛**：SpringCloud环境下默认链路收敛会导致链路限流失效，需要手动关闭相关配置

- **资源名严格匹配**：入口资源、目标资源名称必须与控制台配置完全一致

- **优先级高于直接限流**：同接口同时配置直接限流和链路限流，链路限流规则优先生效

### 6.2 集群限流配置（基础铺垫）

前面所有限流规则均为**单机限流**，在单实例部署场景下可以正常使用，但在微服务集群多实例部署场景中，单机限流存在严重的流量击穿问题，集群限流是生产集群环境的必备进阶方案。

### 6.2.1 集群限流的应用场景（多实例服务的全局限流）

**1、单机限流的致命问题**

例如：服务部署3个实例，单机QPS阈值设置为100，集群总理论阈值为300。但网关负载均衡大概率出现**流量分配不均**，大量流量集中到某一个实例，导致单实例超限限流，整体集群流量未超标却出现误限流。

**2、集群限流核心作用**

集群限流实现**全局统一计数、全局限流管控**，以集群整体流量为维度统计阈值，而非单实例维度，彻底解决集群流量分配不均导致的误限流、防护失效问题。

**3、适用场景**

- 微服务集群多实例部署场景

- 秒杀、大促等高并发峰值场景，需要精准控制全局总流量

- 需要严格管控接口全局QPS，不允许局部击穿的核心业务

### 6.2.2 集群限流的架构模式（客户端模式/服务端模式）

Sentinel集群限流提供两种架构模式，适配不同生产场景：

**1、客户端模式（Client模式）**

所有服务实例平等，通过**客户端分片计数**实现全局流量统计，无需独立服务节点，无中心化架构。

优点：部署简单、无单点故障；缺点：流量统计存在微小误差，适配普通业务。

**2、服务端模式（Server模式）**

独立部署Sentinel集群限流Server节点，所有客户端实例统一上报流量、申请令牌，由Server统一分配流量、统计全局QPS。

优点：统计精准、管控统一，适配超高并发核心业务；缺点：需要维护独立Server节点，有极小中心化单点风险，可集群部署规避。

### 6.2.3 集群限流的优势与注意事项

**1、核心优势**

- **精准限流**：以集群全局流量为准，彻底解决单机限流分配不均问题

- **流量可控**：可精准控制整个服务集群的最大承载流量

- **适配高并发**：秒杀、活动峰值场景必备，避免流量击穿防护

**2、生产注意事项**

- 集群限流依赖网络通信，需要保证服务集群内网互通

- 需配合Nacos持久化统一集群规则，保证所有实例配置一致

- 超高并发场景建议使用Server集群模式，提升稳定性

- 集群限流阈值需要合理拆分，避免全局阈值过小导致大面积限流

## 6.3 常见问题与排查

汇总生产环境Sentinel限流、持久化、规则生效的高频问题，提供**问题现象+根因+解决方案**全套排查方案，覆盖95%以上生产异常。

### 6.3.1 限流规则不生效问题排查（资源名不匹配、规则未下发）

**问题现象**：控制台配置限流规则，接口超高QPS请求依旧不触发限流。

**常见根因与解决方案**：

1. **资源名不匹配（最高频）**
   根因：控制台配置资源名、接口实际簇点链路资源名大小写、路径不一致。
        

解决方案：优先查看Sentinel簇点链路，复制真实资源名配置规则。
      

2. **规则未真正下发**
   根因：控制台保存失败、服务与控制台心跳中断、网络不通。
        

解决方案：重启服务、重新保存规则，查看服务日志是否加载规则成功。
      

3. **请求未进入Sentinel拦截器**
   根因：静态资源、异常路径、自定义拦截器优先拦截请求。
        

解决方案：排查Web拦截器优先级，保证Sentinel拦截器正常执行。
      

4. **阈值设置过大**
   根因：测试QPS未达到配置阈值，误以为规则不生效。
        

解决方案：调低阈值本地压测验证。
      

### 6.3.2 热点参数限流不触发问题排查

**问题现象**：高频请求同一参数，热点参数限流无效果。

1. **参数索引配置错误**：索引从0开始，参数顺序、数量不匹配直接失效。修正索引与接口参数对应关系。

2. **参数类型不匹配**：热点参数仅支持基本数据类型，自定义对象参数无法统计。

3. **统计时长过大**：时间窗口设置过长，短时间高频请求未达到阈值。

4. **规则被全局限流覆盖**：接口配置了全局QPS限流，优先触发普通限流，掩盖热点限流效果。

### 6.3.3 规则持久化失败/不更新问题排查

**问题现象**：Nacos配置修改后，Sentinel规则不更新、重启服务不加载规则。

1. **Nacos配置信息错误**：data-id、group-id、服务地址、账号密码配置错误。严格核对yml配置与Nacos配置。

2. **JSON格式错误**：Nacos规则JSON语法错误、字段缺失，导致解析失败。通过日志查看JSON解析异常。

3. **未引入持久化依赖**：缺失sentinel-datasource-nacos依赖，无法监听Nacos配置。

4. **缓存未刷新**：本地内存缓存未更新，重启服务或手动触发配置刷新。

### 6.3.4 系统规则不生效问题排查

**问题现象**：CPU负载、线程数超标，系统规则不触发全局拦截。

1. **阈值设置过于宽松**：阈值高于系统极限，无法触发防护。根据服务器配置合理调低阈值。

2. **仅监控未触发**：系统规则是兜底规则，流量未达到临界拥堵状态不会触发。

3. **多规则优先级冲突**：接口限流提前拦截请求，系统指标未超标，无法触发系统规则。

## 6.4 面试高频题

### 6.4.1 Sentinel有哪些限流规则？分别适用什么场景？

**参考答案（面试标准作答）**：

Sentinel限流规则分为**基础限流、高级限流、全局限流**三大类，覆盖全生产场景：

1. **QPS限流**：限制接口每秒请求数，适用于普通查询接口、防高频刷量，是最通用的限流方式。

2. **线程数限流**：限制接口并发线程数，专门用于慢接口、阻塞接口，防止线程池耗尽。

3. **热点参数限流**：针对单个请求参数做精细化限流，适用于秒杀商品、热门话题等单点热点流量场景。

4. **链路限流**：针对指定调用链路限流，实现公共接口链路隔离，保护核心业务。

5. **系统规则限流**：全局维度监控CPU、线程数、RT，服务过载时全局兜底防护，防止整体崩溃。

### 6.4.2 热点参数限流的原理是什么？如何实现？

**参考答案（面试标准作答）**：

热点参数限流是Sentinel精细化限流方案，核心原理是**基于LRU缓存+滑动窗口统计**，实现单参数维度的流量管控。

核心实现流程：

1. 拦截接口请求，根据配置的参数索引提取目标请求参数；

2. 通过LRU缓存存储每个参数的流量统计数据，自动淘汰低频参数，控制内存占用；

3. 基于滑动窗口实时统计单个参数的QPS；

4. 单参数QPS超过阈值则精准拦截该参数请求，不影响其他正常参数流量。

主要用于秒杀、热点资讯等高并发单点流量场景，解决普通限流无法精准防护热点流量的问题。

### 6.4.3 如何实现Sentinel规则的持久化？为什么要做持久化？

**参考答案（面试标准作答）**：

**1、为什么需要持久化**

Sentinel默认规则存储在服务内存中，服务重启、宕机后规则全部丢失，集群多实例规则无法同步，运维成本高、线上稳定性差，生产环境必须做持久化。

**2、主流持久化方案**

生产首选**Nacos持久化**，适配SpringCloud生态，支持动态热更新、版本管理、多环境隔离、集群统一同步。

**3、实现步骤**

1. 引入sentinel-datasource-nacos持久化依赖；

2. 配置Nacos服务地址、dataId、groupId、规则类型；

3. 在Nacos控制台编写对应JSON格式的限流、熔断、授权规则；

4. 服务自动监听Nacos配置，实现规则加载、动态更新、重启不丢失。

### 6.4.4 系统规则的作用是什么？和接口限流有什么区别？

**参考答案（面试标准作答）**：

**1、系统规则作用**

系统规则是Sentinel的**全局兜底防护机制**，从服务器CPU负载、平均响应时间、全局并发线程数、总QPS维度监控系统状态，当服务整体资源过载、即将崩溃时，全局拦截所有入口请求，保障服务不宕机。

**2、与接口限流的核心区别**

- **维度不同**：接口限流是微观单接口防护，系统限流是宏观全服务防护；

- **作用范围不同**：接口限流只限制单个资源，系统限流全局所有请求；

- **触发条件不同**：接口限流看请求流量，系统限流看系统资源状态；

- **优先级不同**：系统规则优先级高于接口限流，是最后一道保命屏障。
