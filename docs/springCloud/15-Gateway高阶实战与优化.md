# 1. 网关统一鉴权与Token校验

在微服务架构中，若每个业务服务单独实现登录校验、Token 解析、权限判断，会存在代码冗余、规则不统一、维护成本高、安全标准不一致等严重问题。Gateway 作为微服务**唯一统一入口**，是实现全网关统一鉴权的最佳载体。通过网关前置鉴权，可以在请求进入业务服务之前完成身份认证与权限拦截，实现**前置防护、统一管控、业务解耦**的生产架构。

## 1.1 网关鉴权的核心价值与实现思路

### 1.1.1 网关统一鉴权的优势（集中管控、避免服务重复实现）

网关统一鉴权是微服务架构的**最佳安全实践**，对比业务服务分散鉴权，具备以下核心生产优势：

- **能力集中管控**：所有认证、授权、拦截规则统一在网关层配置，无需每个服务重复开发鉴权逻辑，统一安全标准，避免各服务权限规则混乱。

- **提前拦截防护**：非法请求、未登录请求、无权限请求在网关层直接拦截，不转发至后端业务服务，极大减轻后端服务运算压力，避免无效资源消耗。

- **业务解耦**：业务服务无需关注认证逻辑，只专注业务实现，提升开发效率，代码更简洁、更易维护。

- **动态灵活可配**：支持动态配置白名单、权限规则、拦截策略，无需修改业务代码即可调整安全规则，适配灰度、临时开放、紧急拦截等场景。

- **统一日志溯源**：所有鉴权失败、非法访问日志统一在网关层记录，便于安全审计、问题排查、攻击溯源。

### 1.1.2 常见鉴权方案对比（Session/Cookie vs JWT Token vs OAuth2）

微服务网关场景下，主流三种鉴权方案各有优劣，结合网关特性做选型对比，帮助理解**为什么生产首选JWT Token鉴权**。

|鉴权方案|实现原理|优点|缺点|适用场景|
|---|---|---|---|---|
|**Session/Cookie**|服务端存储Session，客户端通过Cookie传递会话ID|实现简单、兼容性好|服务端有状态，分布式场景需Session共享，跨域适配差、不安全、扩展性极低|单体项目、老旧项目，不适用微服务网关|
|**JWT Token**|无状态令牌，用户信息加密存储在Token中，网关验签认证|**无状态、分布式友好、跨域适配、轻量化、高性能**|Token无法主动失效，需配合缓存实现注销|主流微服务、前后端分离项目、网关统一鉴权（生产首选）|
|**OAuth2.0**|第三方授权协议，基于认证中心颁发令牌|安全性高、支持第三方登录、标准化授权|实现复杂、成本高、冗余度高|第三方授权、多系统单点登录、企业级复杂权限系统|
**生产结论**：常规微服务网关统一鉴权，优先选择 **JWT Token** 方案，兼顾性能、扩展性、开发成本；复杂单点登录场景选用 OAuth2.0。

### 1.1.3 网关鉴权的执行时机（pre过滤器阶段）

网关鉴权逻辑**必须在过滤器Pre前置阶段执行**，这是生产固定规范，核心原理与原因如下：

- Gateway 过滤器 Pre 阶段：请求转发至后端服务**之前**执行；

- 鉴权属于**流量拦截逻辑**，必须在请求转发、参数处理、业务执行前完成校验；

- 若鉴权失败，直接终止请求链路，不执行后续过滤器、不转发后端服务，实现前置拦截。

**执行顺序规范**：鉴权过滤器需设置**高优先级**，仅低于跨域、日志过滤器，保证优先拦截非法流量，避免无效链路执行。

## 1.2 基于JWT的统一Token校验实现

### 1.2.1 JWT Token的生成与校验原理

#### 1、JWT核心结构

JWT 由三部分组成：Header（头部）、Payload（载荷/用户信息）、Signature（签名），三段通过 Base64 编码拼接，用 . 分隔。

- **Header**：存储加密算法、Token类型；

- **Payload**：存储用户ID、账号、角色、过期时间等业务信息（非敏感明文）；

- **Signature**：基于密钥加密生成的签名，用于防篡改校验。

#### 2、生成原理

用户登录成功后，认证中心根据用户信息+密钥，加密生成 JWT Token 返回前端，前端后续所有请求携带 Token。

#### 3、网关校验原理

网关拦截请求，获取请求头中的 Token，通过**统一密钥验签**：校验签名合法性、判断Token是否过期、解析用户信息，全部通过则放行，否则直接拦截。

**核心优势**：JWT无状态，网关无需查询数据库，仅通过本地验签即可完成认证，性能极高。

### 1.2.2 自定义全局鉴权过滤器开发（GlobalFilter）

网关统一鉴权采用**全局过滤器 GlobalFilter** 实现，对全网关所有请求生效，无需逐个路由配置，符合统一管控思想。

#### 1、引入JWT核心依赖

```xml
<!-- JWT令牌解析依赖 -->
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-api</artifactId>
    <version>0.11.5</version>
</dependency>
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-impl</artifactId>
    <version>0.11.5</version>
    <scope>runtime</scope>
</dependency>
<dependency>
    <groupId>io.jsonwebtoken</groupId>
    <artifactId>jjwt-jackson</artifactId>
    <version>0.11.5</version>
    <scope>runtime</scope>
</dependency>

```

#### 2、统一JWT配置参数

在yml中配置密钥、Token请求头名称，统一管理，便于后期修改。

```yaml
# JWT配置
jwt:
  # 自定义密钥，生产环境务必更换复杂密钥
  secret: SpringCloudGatewayJwtSecretKey2026
  # Token请求头名称
  header: Authorization

```

#### 3、全局鉴权过滤器完整代码（可直接上线）

```java
import io.jsonwebtoken.*;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.cloud.gateway.filter.GlobalFilter;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.core.annotation.Order;
import org.springframework.http.HttpStatus;
import org.springframework.http.server.reactive.ServerHttpRequest;
import org.springframework.http.server.reactive.ServerHttpResponse;
import reactor.core.publisher.Mono;

/**
 * 网关全局统一鉴权过滤器
 * Pre阶段执行，完成Token校验、身份认证
 */
@Configuration
public class AuthGlobalFilter {

    @Value("${jwt.secret}")
    private String jwtSecret;

    @Value("${jwt.header}")
    private String jwtHeader;

    // 高优先级，优先执行鉴权拦截
    @Bean
    @Order(-500)
    public GlobalFilter authFilter() {
        return (exchange, chain) -> {
            ServerHttpRequest request = exchange.getRequest();
            ServerHttpResponse response = exchange.getResponse();

            // 1、获取请求头中的Token
            String token = request.getHeaders().getFirst(jwtHeader);

            // 2、Token为空，未登录，拦截返回401
            if (token == null || token.isEmpty()) {
                response.setStatusCode(HttpStatus.UNAUTHORIZED);
                return response.setComplete();
            }

            try {
                // 3、解析并校验Token（验签、过期校验）
                Claims claims = Jwts.parserBuilder()
                        .setSigningKey(jwtSecret.getBytes())
                        .build()
                        .parseClaimsJws(token)
                        .getBody();

                // 4、解析用户信息，存入请求头，透传给后端服务
                String userId = claims.getSubject();
                request.mutate().header("X-User-Id", userId).build();

            } catch (ExpiredJwtException e) {
                // Token过期异常
                response.setStatusCode(HttpStatus.UNAUTHORIZED);
                return response.setComplete();
            } catch (SignatureException e) {
                // 签名错误、Token篡改
                response.setStatusCode(HttpStatus.FORBIDDEN);
                return response.setComplete();
            } catch (Exception e) {
                // 其他Token异常
                response.setStatusCode(HttpStatus.UNAUTHORIZED);
                return response.setComplete();
            }

            // 5、校验通过，放行请求
            return chain.filter(exchange);
        };
    }
}

```

### 1.2.3 Token校验逻辑实现（解析、验签、过期判断）

上述代码完整实现了生产级 Token 全量校验逻辑，拆解核心校验流程：

1. **空值校验**：判断请求头是否携带 Token，未携带直接 401 未登录；

2. **签名校验**：通过密钥校验 Token 签名，防止 Token 被篡改、伪造；

3. **过期校验**：框架自动校验 Token 过期时间，过期抛出 ExpiredJwtException 异常；

4. **合法性校验**：拦截格式错误、非法 Token；

5. **信息透传**：解析出用户ID等核心信息，封装到自定义请求头，透传给后端所有微服务，后端无需重复解析。

### 1.2.4 校验失败的响应处理（返回401/403状态码与提示信息）

原生网关拦截仅返回状态码，无前端友好提示，生产环境需**自定义统一异常响应体**，提升前端交互体验。

#### 自定义JSON响应工具方法

```java
/**
 * 网关统一返回JSON错误信息
 */
private Mono<Void> writeErrorResponse(ServerHttpResponse response, Integer code, String msg) {
    response.getHeaders().add("Content-Type", "application/json;charset=UTF-8");
    String result = "{\"code\":" + code + ",\"msg\":\"" + msg + "\"}";
    return response.writeWith(Mono.just(response.bufferFactory().wrap(result.getBytes())));
}

```

#### 替换原有状态码返回逻辑

```java
// Token为空
if (token == null || token.isEmpty()) {
    return writeErrorResponse(response, 401, "用户未登录，请先登录");
}

// Token过期
catch (ExpiredJwtException e) {
    return writeErrorResponse(response, 401, "Token已过期，请重新登录");
}

// 签名篡改
catch (SignatureException e) {
    return writeErrorResponse(response, 403, "非法Token，禁止访问");
}

```

## 1.3 权限控制与白名单配置

全网关统一鉴权后，登录接口、注册接口、静态资源、健康检查接口会被强制拦截，因此需要配置**接口白名单**，同时实现基于角色的接口级权限管控，适配精细化业务场景。

### 1.3.1 接口白名单配置（无需鉴权的接口）

采用**配置文件动态配置白名单**，无需改代码，支持快速新增、删除放行接口，生产最常用方案。

#### 1、yml白名单配置

```yaml
# 网关白名单配置（无需鉴权接口）
gateway:
  white-list:
    urls:
      - /api/user/login
      - /api/user/register
      - /actuator/health
      - /static/**

```

#### 2、代码读取白名单并放行

```java
import org.springframework.boot.context.properties.ConfigurationProperties;
import org.springframework.stereotype.Component;
import java.util.List;

@Component
@ConfigurationProperties(prefix = "gateway.white-list")
public class GatewayWhiteListProperties {
    private List<String> urls;

    // getter & setter
    public List<String> getUrls() {
        return urls;
    }
    public void setUrls(List<String> urls) {
        this.urls = urls;
    }
}

```

#### 3、过滤器中增加白名单放行逻辑

```java
// 获取当前请求路径
String path = request.getPath().value();
// 白名单接口直接放行，无需鉴权
if (whiteListProperties.getUrls().contains(path)) {
    return chain.filter(exchange);
}

```

### 1.3.2 角色权限校验（Token中携带角色信息，控制接口访问）

JWT 的 Payload 可存储用户角色、权限标识，网关解析后实现**前置权限拦截**，无权限用户直接拦截，不进入业务服务。

#### 1、登录生成Token时存入角色信息

```java
// 存入用户角色
claims.put("role", "admin");

```

#### 2、网关解析角色并校验权限

```java
// 解析角色信息
String role = claims.get("role", String.class);
// 管理员接口权限校验
if (path.startsWith("/api/admin/") && !"admin".equals(role)) {
    return writeErrorResponse(response, 403, "无管理员权限，访问拒绝");
}

```

### 1.3.3 接口级权限控制实现

生产进阶方案：通过配置文件绑定**接口与角色映射关系**，实现动态接口权限控制，无需硬编码。

```yaml
gateway:
  permission:
    # 接口路径: 允许访问的角色
    mapping:
      /api/admin/**: admin
      /api/user/info: user,admin

```

网关根据请求路径匹配对应角色列表，校验当前用户角色是否在允许列表中，实现精细化接口级权限管控。

### 1.3.4 鉴权过滤器的执行顺序控制（优先于业务过滤器）

鉴权过滤器必须保证**全网关业务过滤器中优先级最高**，核心原因：

- 非法请求提前拦截，无需执行日志、参数处理、脱敏等后续过滤器，节省性能；

- 避免未登录数据泄露、无效链路执行。

**生产固定优先级**：跨域(-1000) > 日志(-800) > 鉴权(-500) > 业务过滤器(0及以上)

## 1.4 鉴权场景的常见问题与优化

### 1.4.1 Token传递与刷新机制（Token过期、刷新Token）

#### 1、常见问题

JWT 无状态特性导致 Token 过期后无法主动续期，直接强制登录用户体验差。

#### 2、生产优化方案

- **双Token机制**：AccessToken（短期有效，1小时）+ RefreshToken（长期有效，7天）；

- AccessToken过期后，前端携带RefreshToken请求网关刷新接口，网关校验通过后颁发新Token；

- 刷新接口加入白名单，无需鉴权，实现无感续期。

### 1.4.2 异步请求/跨域请求下的Token携带问题

#### 问题原因

前端异步请求、跨域 OPTIONS 预检请求不会携带 Token，导致网关误拦截。

#### 解决方案

- 网关过滤器中**直接放行OPTIONS预检请求**，不做鉴权校验；

- 前端配置 axios 允许跨域携带自定义请求头，保证正式请求正常携带 Token。

```java
// 放行OPTIONS预检请求
if ("OPTIONS".equalsIgnoreCase(request.getMethod().name())) {
    return chain.filter(exchange);
}

```

### 1.4.3 鉴权性能优化（Token缓存、验签优化）

网关高并发场景下，频繁解析、验签 Token 存在性能损耗，生产优化方案：

- **Token解析结果缓存**：将合法Token对应的用户信息缓存至Redis，有效时间与Token过期时间一致，减少重复验签解析；

- **异步非阻塞校验**：保证鉴权逻辑全程异步，不阻塞WebFlux线程池；

- **提前拦截无效流量**：白名单快速放行、非法Token快速拦截，减少后续运算。

### 1.4.4 敏感接口防刷与防重放攻击

#### 1、防重放攻击

在Token中加入随机 jti 唯一标识，网关缓存已使用的 jti，拒绝重复 Token 请求，防止请求被劫持重复调用。

#### 2、接口防刷

基于用户ID+接口路径做 Redis 限流，限制单用户单位时间内访问频次，实现网关层接口防刷，保护敏感支付、信息修改接口。

# 2. 网关层级限流、熔断整合Sentinel

微服务架构中，流量是不可控的，突发高并发、恶意刷量、服务超时异常都会导致服务雪崩、系统瘫痪。传统单服务限流只能防护单体服务，无法拦截网关入口的超大流量。Gateway作为微服务唯一入口，结合Sentinel实现**网关全局流量防护**，可以在流量进入业务服务之前完成限流、熔断、降级，从源头保护整个微服务集群，是生产环境必备的高可用保障方案。

## 2.1 网关限流的必要性与实现方式

### 2.1.1 网关限流与服务限流的区别（入口防护 vs 服务内防护）

微服务体系中存在两层限流体系：**网关层限流**与**业务服务内部限流**，二者层级、作用范围、防护场景完全不同，生产环境需配合使用，形成双层防护体系。

| 对比维度 | 网关层限流（Gateway+Sentinel）             | 服务内部限流（业务服务Sentinel）           |
| -------- | ------------------------------------------ | ------------------------------------------ |
| 拦截层级 | 微服务统一入口，请求最前置拦截             | 单个业务服务内部，请求已进入服务           |
| 防护范围 | 全局所有服务、所有接口，集群级防护         | 仅防护当前单体服务，服务隔离防护           |
| 拦截时机 | 路由转发前拦截，无资源消耗                 | 请求进入服务后拦截，已占用线程、连接资源   |
| 核心作用 | 抵挡洪峰流量、恶意流量、全局限流，保护集群 | 服务过载保护、接口粒度限流、防止单服务雪崩 |
| 优先级   | 优先执行，前置拦截                         | 后置兜底防护                               |

### 2.1.2 网关限流的核心优势（提前拦截，减少后端压力）

网关限流是**性价比最高的流量防护方案**，核心生产优势集中在前置拦截、资源节省、全局管控：

- **零消耗拦截**：在网关层直接拦截超限请求，不建立后端连接、不占用业务服务线程池、不执行业务逻辑，极大节省后端服务器资源；

- **全局统一管控**：所有流量统一管控，无需每个服务单独配置限流规则，规则统一、维护简单、无遗漏；

- **防刷防攻击**：可基于IP、接口、用户维度全局限流，拦截恶意刷量、爬虫攻击，从入口筑牢安全防线；

- **避免服务雪崩**：突发流量直接拦截，不会导致单个服务被打垮，进而连锁影响整个集群；

- **粒度灵活可控**：支持全局限流、服务级限流、接口分组限流，适配各种业务场景。

### 2.1.3 Sentinel对Gateway的支持（网关流控规则）

Sentinel 专门为 Spring Cloud Gateway 提供了**专属网关流控模块**，区别于普通服务的资源限流，适配网关路由、API分组特性，是官方标准化网关流量防护方案。

Sentinel 网关核心能力支持：

- 两种限流模式：**路由维度限流**（按路由ID限流）、**API分组限流**（自定义接口分组统一限流）；

- 多种限流策略：QPS限流、IP限流、用户维度限流、黑名单限流；

- 多种限流效果：快速失败、排队等待、预热限流（Warm Up）；

- 支持网关熔断降级、自定义兜底响应、规则动态下发、规则持久化；

- 完美适配网关异步非阻塞模型，无性能损耗。

## 2.2 Sentinel 网关流控规则配置

### 2.2.1 Gateway流控规则参数说明（资源名、阈值、模式、策略）

Sentinel网关流控规则核心参数是配置限流的关键，所有控制台配置、代码配置均基于以下参数，吃透参数可适配所有限流场景。

- **资源名**：限流的唯一标识，路由模式为路由ID，API分组模式为自定义分组名称；

- **限流模式**：分为 ROUTE（路由模式）、API_GROUP（API分组模式）两种；

- **阈值类型**：QPS（每秒请求数，生产最常用）、线程数（并发线程数）；

- **阈值**：最大允许的QPS/线程数，超出即触发限流；

- **限流策略**：直接限流（全局统一阈值）、IP限流（单IP阈值）、用户限流、黑名单限流；

- **限流效果**：快速失败、排队等待、预热模式（Warm Up）。

### 2.2.2 基于路由ID的限流配置

路由ID限流是**最简单、最常用**的网关限流方式，以网关路由ID为资源维度，对整个路由下的所有接口统一限流。

#### 适用场景

单个微服务整体流量管控，限制某一个服务的全局QPS，避免单服务流量过高影响其他服务。

#### 控制台配置步骤

1. 登录Sentinel控制台，进入【网关流控规则】；

2. 新增规则，限流模式选择「路由模式」；

3. 资源名选择对应网关路由ID；

4. 阈值类型选择QPS，设置阈值（如100）；

5. 限流效果选择快速失败，保存即刻生效。

#### 规则说明

配置QPS=100后，该路由下所有接口每秒最大允许100次请求，超出请求直接触发限流，返回默认兜底提示。

### 2.2.3 基于API分组的限流配置（多接口统一限流）

路由限流粒度为整个服务，无法适配部分接口特殊限流场景，**API分组限流**支持自定义接口集合，将多个零散接口划分为一个分组，统一配置限流规则，粒度更精细。

#### 适用场景

核心敏感接口（支付、下单、登录）单独限流、多服务通用接口统一限流、高风险接口统一防护。

#### 配置流程

1. Sentinel控制台【API管理】新增分组，自定义分组名称；

2. 添加匹配规则，支持路径匹配、正则匹配，绑定需要统一限流的接口；

3. 在【网关流控规则】中选择API分组模式，选中对应分组配置QPS阈值；

4. 保存规则，实现多接口统一限流。

### 2.2.4 限流效果配置（快速失败、排队等待、Warm Up）

Sentinel网关提供三种限流效果，适配不同业务流量场景，生产需根据业务特性合理选择。

#### 1、快速失败（默认）

流量超出阈值后，**立即拒绝请求**，返回限流兜底信息，适用于接口无积压、实时性要求高的场景，如查询接口、列表接口。

#### 2、排队等待

超出阈值的请求进入队列排队，设置最大等待时间，超时则失败。适用于**允许短暂积压、实时性一般**的接口，如异步提交、日志上报接口，避免瞬时流量抖动导致大量失败。

#### 3、Warm Up 预热限流（冷启动限流）

系统重启后，服务处于冷启动状态，资源未完全初始化，瞬时高并发容易打垮服务。Warm Up模式会**渐进式提升QPS阈值**，预热时间内缓慢放开流量，预热结束后恢复正常阈值。

**生产核心场景**：服务重启、凌晨流量低谷后突发高峰、大促预热场景，是生产避坑关键配置。

## 2.3 网关熔断降级实现

限流是拦截**正常过量流量**，熔断是防护**异常服务**。当后端服务出现慢调用、频繁报错、超时等问题时，网关触发熔断，直接降级返回兜底数据，停止转发请求到故障服务，避免故障扩散，实现服务自愈。

### 2.3.1 网关熔断的触发条件（慢调用、异常比例）

Sentinel网关熔断规则基于**统计窗口内的异常指标**触发，生产两大核心触发条件：

#### 1、慢调用比例熔断

设置最大响应超时时间，统计窗口内慢调用占比超过阈值，触发熔断。适用于服务卡顿、响应缓慢场景。

示例：超时时间500ms、比例阈值0.5，10秒窗口内超过50%的请求响应超时，立即熔断。

#### 2、异常比例熔断

统计窗口内，服务报错（500、业务异常）请求占比超过阈值，触发熔断。适用于服务报错、逻辑异常场景。

#### 3、异常数熔断

统计窗口内异常请求总数达到阈值，触发熔断，适用于瞬时大量报错场景。

### 2.3.2 熔断降级规则配置与自定义回调

#### 1、控制台熔断规则配置

1. 进入Sentinel控制台【熔断降级规则】，选择网关路由资源；

2. 选择熔断策略（慢调用比例/异常比例/异常数）；

3. 设置阈值、统计窗口时长、熔断时长、最小请求数；

4. 保存规则，动态生效。

#### 2、自定义熔断回调（生产必备）

默认熔断返回系统英文提示，用户体验差，生产需自定义网关熔断、限流统一兜底响应。

```java
import com.alibaba.csp.sentinel.adapter.gateway.sc.callback.GatewayCallbackManager;
import com.alibaba.csp.sentinel.adapter.gateway.sc.exception.SentinelGatewayBlockExceptionHandler;
import org.springframework.context.annotation.Configuration;
import org.springframework.http.HttpStatus;
import org.springframework.http.MediaType;
import org.springframework.web.reactive.function.server.ServerResponse;
import reactor.core.publisher.Mono;

import javax.annotation.PostConstruct;

/**
 * 网关限流熔断自定义兜底配置
 */
@Configuration
public class SentinelGatewayFallbackConfig {

    @PostConstruct
    public void initBlockHandler() {
        // 注册自定义限流熔断回调
        GatewayCallbackManager.setBlockHandler((exchange, ex) -> {
            // 自定义JSON兜底响应
            String result = "{\"code\":429,\"msg\":\"系统繁忙，请稍后再试！\"}";
            return ServerResponse.status(HttpStatus.TOO_MANY_REQUESTS)
                    .contentType(MediaType.APPLICATION_JSON)
                    .bodyValue(result);
        });
    }
}

```

### 2.3.3 网关层的服务熔断与Feign降级的配合使用

生产高可用架构采用**双层熔断降级体系**，网关熔断+Feign服务降级配合使用，实现全方位容错。

- **网关层熔断**：宏观防护，后端服务整体异常时熔断，停止所有流量转发，保护整个服务，拦截粒度大；

- **Feign降级**：微观兜底，单个接口调用失败、超时、异常时触发，返回接口级兜底数据，不影响服务其他接口；

**执行顺序**：网关优先熔断拦截（服务整体故障） → 正常流量进入服务 → 接口异常触发Feign降级，双层兜底，杜绝大面积报错。

### 2.3.4 熔断降级的响应处理（返回兜底数据/提示页面）

网关熔断降级支持两种生产级响应方案，适配不同业务场景：

- **统一提示文案**：普通查询类接口，返回「系统繁忙，请稍后重试」通用提示，极简兜底；

- **静态兜底页面**：页面访问请求，转发至静态降级页面，提升用户体验；

- **兜底默认数据**：非核心展示类接口，返回缓存默认数据，实现无感降级。

操作流程：启动网关服务 → 访问任意接口激活客户端连接 → 控制台识别网关服务 → 动态配置规则 → 实时下发生效。

### 2.4.3 网关流控规则持久化（基于Nacos）

**默认痛点**：Sentinel控制台规则默认存储在内存，网关重启、控制台重启后规则全部丢失，生产绝对不可用。

**生产解决方案**：整合Nacos实现规则持久化，规则存储在Nacos配置中心，永久保存、动态刷新、集群共享。

核心配置：开启Sentinel Nacos数据源，将限流、熔断规则同步至Nacos，实现规则持久化与热更新。

### 2.4.4 限流熔断常见问题排查（规则不生效、误拦截）

#### 1、规则不生效常见原因

- 未开启网关Sentinel适配开关，配置未加载；

- 路由ID、API分组名称与规则资源名不匹配；

- 规则阈值设置过大，未达到触发条件；

- 网关未成功连接Sentinel控制台，客户端未注册。

#### 2、正常流量被误拦截

- 统计窗口设置过小，流量波动导致误触发；

- Warm Up预热时间配置不合理，冷启动阶段阈值过低；

- IP限流规则配置错误，单IP阈值过低。

#### 3、排查方案

查看Sentinel控制台实时监控、网关日志、规则加载日志，核对资源名、阈值、统计窗口参数，快速定位问题。

# 3. 网关日志收集与请求监控

网关作为微服务统一入口，汇聚了所有服务的请求流量，网关日志与监控是**全链路问题排查、性能分析、故障溯源、安全审计**的核心依据。原生网关日志格式混乱、信息缺失、无结构化输出，生产环境必须自定义规范化日志体系，结合监控组件实现可视化可观测能力。

## 3.1 网关日志的核心价值与分类

### 3.1.1 访问日志（请求路径、耗时、状态码、来源IP）

访问日志是网关基础日志，记录每一次请求的全量基础信息，核心字段包含：请求ID、客户端IP、请求路径、请求方式、响应状态码、请求耗时、目标服务、响应大小。主要用于日常请求轨迹溯源、接口访问统计、异常请求定位。

### 3.1.2 错误日志（异常信息、堆栈、请求上下文）

专门收集网关层异常、路由异常、过滤器异常、请求转发异常，包含完整异常堆栈信息+当前请求上下文（请求参数、请求头、用户信息）。核心价值是快速定位网关报错、服务转发失败、过滤器执行异常等线上故障。

### 3.1.3 性能日志（请求耗时分布、峰值流量）

统计接口请求耗时区间分布、每秒峰值流量、慢请求占比，用于**性能瓶颈分析、接口优化、容量评估、压测分析**，是网关性能优化的核心数据依据。

### 3.1.4 日志收集的目标（问题排查、性能分析、审计）

- **故障排查**：快速定位请求失败、接口404/500、超时、拦截异常问题；

- **性能分析**：统计慢接口、高耗时请求，指导接口优化；

- **安全审计**：记录所有访问行为，用于攻击溯源、非法访问审计；

- **运维统计**：流量统计、接口热度统计、服务调用量统计。

## 3.2 网关请求日志收集实现

### 3.2.1 自定义日志过滤器开发（记录请求前后信息）

通过全局过滤器实现全网关请求日志统一收集，在Pre阶段记录请求入参信息，Post阶段记录响应信息与耗时，完整覆盖请求全生命周期。

```java
import org.springframework.cloud.gateway.filter.GlobalFilter;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.core.annotation.Order;
import reactor.core.publisher.Mono;
import java.util.UUID;

/**
 * 网关全局日志过滤器
 * 统一收集请求、响应、耗时、链路信息
 */
@Configuration
public class GatewayLogFilter {

    @Bean
    @Order(-1000)
    public GlobalFilter gatewayLogGlobalFilter() {
        return (exchange, chain) -> {
            // 生成全局链路ID
            String traceId = UUID.randomUUID().toString().replace("-", "");
            // 存入上下文，全局透传
            exchange.getAttributes().put("TRACE_ID", traceId);

            // 记录请求开始信息
            long startTime = System.currentTimeMillis();
            String path = exchange.getRequest().getPath().value();
            String method = exchange.getRequest().getMethod().name();
            String ip = exchange.getRequest().getRemoteAddress().getHostString();

            // 前置请求日志
            System.out.printf("【网关请求】TraceId:%s，IP:%s，路径:%s，方法:%s，时间:%d%n",
                    traceId, ip, path, method, startTime);

            // 后置响应日志
            return chain.filter(exchange).then(Mono.fromRunnable(() -> {
                long cost = System.currentTimeMillis() - startTime;
                int status = exchange.getResponse().getStatusCode().value();
                System.out.printf("【网关响应】TraceId:%s，状态码:%d，耗时:%dms%n",
                        traceId, status, cost);
            }));
        };
    }
}

```

### 3.2.2 日志格式规范（JSON格式、包含TraceId）

生产环境禁止使用普通文本日志，必须采用**JSON结构化日志**，方便ELK、 Loki等日志系统解析检索，核心规范：

- 每条日志必须携带 **TraceId**，实现全链路检索；

- 统一包含时间戳、服务名、IP、请求路径、状态码、耗时、异常信息；

- 字段命名统一、无冗余内容，适配日志检索聚合。

### 3.2.3 日志级别控制与脱敏处理（敏感信息隐藏）

#### 1、日志级别控制

生产环境网关日志级别设置为 INFO，关闭DEBUG冗余日志，减少日志输出量、提升性能；开发环境开启DEBUG便于调试。

#### 2、敏感信息脱敏

日志必须对手机号、身份证、密码、Token等敏感参数脱敏，避免数据泄露，在日志过滤器中添加参数脱敏逻辑，脱敏后再打印日志。

### 3.2.4 日志输出配置（控制台/文件/远程日志系统）

- **开发环境**：控制台输出，格式简洁，便于本地调试；

- **测试/生产环境**：关闭控制台输出，输出至日志文件，滚动分割，避免单文件过大；

- **线上生产**：对接ELK、Loki远程日志系统，实现日志集中收集、持久化、检索、分析。

- 流量突增/突降超过阈值，触发异常告警；

- 网关响应耗时骤增、服务熔断触发告警；

- 网关离线、端口异常、心跳中断告警。

## 3.4 链路追踪集成（基础铺垫）

### 3.4.1 TraceId传递与网关层链路埋点

网关作为请求入口，是**全链路追踪的起点**，网关层生成全局唯一TraceId，贯穿前端、网关、所有微服务、数据库操作，实现一次请求全链路溯源。

核心实现：网关过滤器生成TraceId，存入请求头，转发至后端所有服务，后端服务复用该TraceId，保证全链路日志唯一关联。

### 3.4.2 与SkyWalking/OpenTelemetry的集成方式

主流链路追踪框架均可无缝适配Gateway：

- **SkyWalking**：无侵入集成，通过agent探针自动采集网关链路信息，无需改代码；

- **OpenTelemetry**：标准化埋点，统一链路追踪协议，适配多框架、多语言。

### 3.4.3 网关链路追踪的关键指标与优化

链路追踪核心观测指标：网关转发耗时、服务响应耗时、网络耗时、过滤器执行耗时。通过链路数据可精准定位**是网关瓶颈、网络瓶颈还是业务服务瓶颈**，为性能优化提供精准依据。

优化要点：精简网关过滤器链路、减少网关同步阻塞操作、优化路由匹配规则，降低网关层链路耗时。

# 4. 高并发场景网关性能优化

Spring Cloud Gateway 基于 WebFlux + Netty 异步非阻塞模型，原生性能远优于传统 Zuul 网关，但默认配置仅适用于测试、低并发场景。在秒杀、大促、高吞吐业务场景下，若不做针对性调优，会出现吞吐量上不去、延迟升高、线程繁忙、连接耗尽、超时激增等问题。本节从**瓶颈分析→底层内核优化→配置优化→集群架构优化**全链路讲解生产级性能调优方案。

## 4.1 网关性能瓶颈分析

### 4.1.1 常见性能瓶颈点（Netty线程池、过滤器链、请求转发）

Gateway 高并发场景下90%的性能问题，集中在三大核心瓶颈，是调优的核心切入点：

- **Netty线程池瓶颈**：Boss、Worker线程默认配置过小，高并发下线程占满、任务堆积、请求排队，导致响应延迟飙升、吞吐量封顶；同时自定义过滤器存在同步阻塞代码，破坏异步模型，造成线程卡死。

- **过滤器链瓶颈**：过滤器数量过多、冗余重复逻辑、执行顺序混乱、自定义过滤器存在IO阻塞/循环耗时操作，单次请求链路耗时拉长，并发能力下降。

- **请求转发瓶颈**：HTTP连接池参数不合理、空闲连接过少、连接复用率低、握手频繁、后端服务超时配置不匹配，导致大量新建连接、连接超时、连接泄漏。

### 4.1.2 性能测试与瓶颈定位（压测工具、监控分析）

性能优化的前提是**精准定位瓶颈**，禁止盲目调参。生产标准定位流程如下：

#### 1、压测工具

使用 JMeter、Postman、wrk 对网关真实接口压测，模拟高QPS流量，采集吞吐量、延迟、错误率指标。

#### 2、监控指标定位

- **线程监控**：Netty Worker线程繁忙率、Reactor线程池队列堆积；

- **连接监控**：活跃连接数、新建连接数、连接超时数、连接关闭数；

- **耗时监控**：网关转发耗时、后端响应耗时、过滤器执行耗时；

- **系统监控**：CPU、内存、GC频率、上下文切换次数。

#### 3、瓶颈判定标准

CPU 打满、线程队列堆积 = 线程池配置不足或存在阻塞；连接数暴涨 = 连接池配置过小；单次请求耗时高 = 过滤器或业务接口瓶颈。

### 4.1.3 高并发场景下的性能挑战（QPS、延迟、吞吐量）

大促、秒杀等高并发场景，网关面临三大核心挑战：

- **QPS突增**：瞬时流量百倍上涨，默认配置无法承载，极易触发限流、超时；

- **延迟抖动**：流量高峰下线程排队、连接竞争、GC频繁，导致接口延迟波动极大；

- **吞吐量封顶**：受限于线程池、连接池、路由匹配效率，网关吞吐量存在上限，无法横向压榨单机性能。

## 4.2 Netty与Reactor性能优化

Gateway 底层基于 **Netty + Reactor 异步非阻塞模型**，内核参数直接决定网关单机性能上限，是高并发调优的核心。

### 4.2.1 Netty线程池配置优化（boss/worker线程数）

#### 1、线程角色说明

- **Boss线程**：负责接收客户端连接、建立TCP握手，任务轻，无需过大配置；

- **Worker线程**：负责所有请求读写、编解码、业务分发，是核心工作线程，直接影响并发能力。

#### 2、生产最优配置

```yaml
spring:
  cloud:
    gateway:
      httpclient:
        netty:
          # Boss线程数，默认1，生产保持1即可
          boss-pool-size: 1
          # Worker线程数，推荐配置为CPU核心数*2，高并发可适当上调
          worker-pool-size: 16

```

#### 3、优化原理

默认Worker线程数极小，高并发下线程快速耗尽，导致请求排队延迟；合理放大Worker线程数，可最大化利用CPU多核能力，提升吞吐。**禁止过度配置**，线程过多会导致上下文切换频繁、CPU飙升。

### 4.2.2 连接池配置优化（最大连接数、空闲连接超时）

网关转发依赖HTTP连接池，连接池配置不合理是**高并发超时、连接泄漏、吞吐量低**的头号原因。

```yaml
spring:
  cloud:
    gateway:
      httpclient:
        # 开启连接池复用
        pool:
          enabled: true
          # 每个后端服务最大连接数
          max-connections: 200
          # 最大空闲连接
          max-idle-connections: 100
          # 空闲连接超时，释放无效连接，避免连接占用
          idle-timeout: 30000
          # 连接等待超时
          acquire-timeout: 5000

```

**最佳实践**：高并发场景适当放大最大连接数，缩短空闲超时，及时回收闲置连接，避免连接耗尽与连接内存泄漏。

### 4.2.3 缓冲区大小配置与内存优化

Netty缓冲区用于读写请求数据，缓冲区过小导致频繁扩容、拷贝，过大导致内存浪费、大流量下内存飙升。

- 优化自适应缓冲区大小，避免频繁内存分配与回收；

- 限制单请求最大报文长度，防止超大请求导致OOM内存溢出；

- 关闭不必要的内存拷贝，开启零拷贝特性。

### 4.2.4 Reactor调度器配置优化

Gateway 基于 Reactor 响应式调度，默认调度器线程数较小，高并发下容易任务堆积。生产可自定义调度器参数，优化异步任务执行效率，保证所有请求处理逻辑均在异步线程执行，杜绝阻塞主线程。

## 4.3 网关配置优化

### 4.3.1 路由配置精简与匹配效率优化

网关路由匹配逻辑为**顺序遍历匹配**，路由越多、规则越复杂，匹配耗时越高。

- 下线废弃、无效路由，减少匹配遍历次数；

- 尽量使用精准路径匹配，减少正则模糊匹配（正则匹配性能远低于精准匹配）；

- 合并重复规则、统一路由前缀，简化匹配逻辑。

### 4.3.2 过滤器链优化（减少不必要的过滤器）

每一个过滤器都会增加单次请求链路耗时，高并发下累积损耗极大。

- 删除冗余、废弃、重复功能的过滤器；

- 通用逻辑全局统一，个性化逻辑按需绑定局部过滤器；

- 健康检查、OPTIONS预检、静态资源跳过所有业务过滤器；

- **核心禁止**：自定义过滤器中出现同步IO、数据库查询、远程调用、循环耗时操作。

### 4.3.3 断言匹配顺序优化（高频路由优先匹配）

网关路由按照**配置从上到下顺序匹配**，匹配成功即终止遍历。

**生产黄金规则**：访问量越高、越核心的路由，越放在配置靠前位置，减少低流量路由的无效遍历匹配，大幅提升整体匹配效率。

### 4.3.4 静态资源请求优化（缓存、CDN配合）

静态资源（图片、JS、CSS、静态页面）占用网关大量流量，是性能浪费的重点。

- 静态资源全部迁移至CDN，不经过网关转发；

- 网关开启静态资源缓存，设置合理缓存过期时间；

- 对静态资源路径直接放行，跳过鉴权、限流、日志等所有过滤器链路。

## 4.4 高并发场景最佳实践

### 4.4.1 网关集群部署与负载均衡（多实例部署）

单机网关存在性能上限与单点故障，生产高并发场景**必须集群部署**。通过Nginx/LVS做前置负载均衡，将流量分发至多台网关实例，实现流量分摊、故障冗余、高可用。

### 4.4.2 网关水平扩容策略

- 流量平稳期：固定节点数，保证稳定性；

- 大促高峰期：基于监控指标自动扩容，动态增加网关实例；

- 流量低谷：缩容释放资源，节省服务器成本。

### 4.4.3 限流熔断与性能优化的配合使用

性能调优是**提升上限**，限流熔断是**保下限**，二者必须配合：

- 性能优化最大化单机吞吐能力；

- 限流熔断拦截超限流量、故障流量，保护网关不被打垮；

- WarmUp预热限流配合冷启动优化，避免重启瞬间流量冲击。

### 4.4.4 压测验证与性能调优步骤

生产标准调优流程：压测摸底 → 定位瓶颈 → 单项调优 → 复测对比 → 固化参数 → 上线观测。每次只调整一项参数，对比压测数据，避免多变量干扰，确保调优有效。

# 5. 网关生产级避坑与最佳实践

## 5.1 常见问题与排查

### 5.1.1 鉴权过滤器执行顺序问题（被其他过滤器绕过）

#### 问题现象

部分请求绕过鉴权逻辑，未登录也可访问接口，存在严重安全漏洞。

#### 根因分析

自定义过滤器Order优先级设置不合理，日志、跨域、路径处理过滤器优先级高于鉴权过滤器，部分前置过滤器直接终止请求链路，导致鉴权逻辑未执行。

#### 解决方案

固定鉴权过滤器Order=-500，仅低于跨域、日志过滤器，保证全网业务逻辑中**最先执行鉴权拦截**。

### 5.1.2 限流规则误拦截问题（配置错误、阈值设置不合理）

#### 常见原因

- 统计窗口时间过短，流量小幅波动触发误限流；

- IP限流阈值过低，内网服务调用、自测流量被拦截；

- API分组规则匹配错误，误伤正常接口；

- 未配置白名单，健康检查、监控接口被限流。

#### 解决与避坑

合理设置统计窗口、区分内网外网限流策略、核心监控接口加入白名单、压测验证阈值合理性。

### 5.1.3 高并发下的请求超时与连接泄漏问题

#### 问题根因

- 连接池最大连接数不足，高并发下无可用连接，请求排队超时；

- 空闲连接未及时回收，大量无效连接占用资源；

- 异常请求未正常释放连接，导致**连接泄漏**，最终连接耗尽。

#### 解决方案

优化连接池参数、统一异常处理、保证所有场景连接正常释放、定时清理空闲连接。

### 5.1.4 日志收集导致的性能损耗问题

#### 问题现象

开启全量日志打印后，网关QPS大幅下降、CPU升高、GC频繁。

#### 根因

同步日志打印、超大报文日志输出、频繁字符串拼接、未脱敏大参数打印，占用大量CPU与IO资源。

#### 优化方案

- 生产使用**异步日志**，杜绝同步控制台打印；

- 限制单条日志长度，禁止打印超大请求体；

- 脱敏敏感参数，精简无效日志；

- 分级日志输出，INFO级别运行，关闭DEBUG日志。

## 5.2 安全防护与防攻击

### 5.2.1 请求头安全防护（XSS、CSRF防护）

网关统一添加安全响应头，拦截恶意脚本攻击、跨站请求伪造攻击：

- X-Content-Type-Options: nosniff 防止MIME类型嗅探；

- X-Frame-Options: DENY 防止页面被嵌套钓鱼；

- Content-Security-Policy 限制资源加载来源，防御XSS攻击。

### 5.2.2 请求参数校验与恶意请求拦截

在网关层前置拦截非法参数、空参数、超长参数、恶意脚本参数，避免无效请求进入后端服务，从源头防御注入攻击。

### 5.2.3 IP黑名单与防刷机制

基于Redis实现网关层IP黑名单机制，对频繁请求、恶意刷量、攻击IP自动封禁，支持临时封禁、永久封禁、自动解封，实现网关原生防刷能力。

### 5.2.4 敏感接口的二次验证

针对支付、提现、密码修改、手机号变更等高敏感接口，在网关层实现**二次Token校验、验证码校验、设备校验**，多层防护，杜绝越权访问与恶意操作。

## 5.3 面试高频题

### 5.3.1 如何实现网关的统一鉴权？有哪些注意事项？

#### 参考答案

通过 Gateway **全局GlobalFilter** 实现全网关统一鉴权，在Pre前置阶段拦截所有请求，校验请求头中的JWT Token，完成空值校验、签名验签、过期校验、用户信息解析、权限判断，白名单接口直接放行。

**核心注意事项**：

- 鉴权过滤器优先级必须合理，保证优先执行，避免被其他过滤器绕过；

- 必须放行OPTIONS预检请求、白名单接口、健康检查接口；

- Token校验全程异步非阻塞，禁止同步阻塞逻辑；

- 统一异常响应，返回标准化JSON错误信息；

- 做好Token缓存优化、防重放、防篡改防护。

### 5.3.2 Gateway如何整合Sentinel实现限流熔断？和服务内限流有什么区别？

#### 参考答案

Gateway引入sentinel-spring-cloud-gateway-adapter专属网关依赖，配置Sentinel控制台地址，即可实现网关层流量防护，支持路由维度、API分组维度限流，配置熔断规则实现慢调用、异常比例降级，自定义兜底响应。

**核心区别**：

- 网关限流是**全局前置防护**，在请求入口拦截，不消耗后端资源，保护整个微服务集群；

- 服务限流是**单体后置防护**，请求进入服务后拦截，仅保护当前服务；

- 生产架构采用网关全局限流+服务接口限流双层防护，保障高可用。

### 5.3.3 高并发场景下如何优化Gateway的性能？

#### 参考答案

从内核、配置、架构、流量四层优化：

1. **内核优化**：调优Netty线程池、HTTP连接池、Reactor调度器参数，适配高并发；

2. **配置优化**：精简路由规则、优化断言匹配顺序、删减冗余过滤器、杜绝阻塞代码；

3. **日志优化**：开启异步日志、精简日志输出、参数脱敏，减少IO损耗；

4. **架构优化**：网关集群部署、水平扩容、前置CDN分担静态流量；

5. **流量防护**：合理配置限流熔断、预热限流，保护网关稳定性。

### 5.3.4 网关日志收集需要注意哪些问题？如何避免性能损耗？

#### 参考答案

日志收集核心注意点：必须结构化JSON格式、全局透传TraceId、敏感信息脱敏、分级输出、区分环境日志策略、避免超大日志打印。

**性能避坑方案**：

- 生产关闭同步控制台日志，使用异步日志框架；

- 限制单条日志大小，禁止打印完整大报文请求体；

- 精简过滤器日志逻辑，避免频繁字符串拼接；

- 静态资源、健康检查接口减少无效日志输出；

- 线上严格控制日志级别，关闭DEBUG冗余日志。
